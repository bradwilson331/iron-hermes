# Testing

IronHermes uses the Rust standard test harness (`cargo test`) across a Cargo workspace. There are 215 source files containing inline unit test modules (`#[cfg(test)]`) and 108 standalone integration test files under `tests/` directories. Snapshot testing uses `insta`; static-grep invariant tests use `include_str!` macros against source files.

## Test Framework and Setup

The project is a pure Rust workspace — there is no separate JavaScript test runner. All testing goes through Cargo.

**Test libraries (declared in `[workspace.dependencies]` in `Cargo.toml`):**

| Library | Version | Role |
|---|---|---|
| `insta` | `1` (yaml feature) | Snapshot assertions for TUI frames and CLI output |
| `wiremock` | `0.6` | HTTP mock server for integration tests |
| `assert_cmd` | `2` | CLI binary integration testing |
| `predicates` | `3` | Assertion predicates for `assert_cmd` |
| `tracing-test` | `0.2` | Captures `tracing` output in unit tests |
| `tempfile` | `3` | Temporary directories and files in tests |

**Required setup before running tests:**

1. Install the stable Rust toolchain:
   ```bash
   rustup toolchain install stable
   ```
2. Install `cargo-insta` for snapshot review (needed only when accepting new snapshots):
   ```bash
   cargo install cargo-insta --locked
   ```
3. Install dependencies and build the workspace (done automatically by `cargo test`):
   ```bash
   cargo build --workspace
   ```

## Running Tests

### Full workspace

Runs all unit and integration tests across every crate in the workspace:

```bash
cargo test --workspace --all-features
```

### Single crate

```bash
cargo test -p ironhermes-cli
cargo test -p ironhermes-agent
cargo test -p ironhermes-core
```

### Single integration test file

```bash
cargo test -p ironhermes-cli --test invariants_21_7
cargo test -p ironhermes-agent --test budget_ordering_grep
cargo test -p ironhermes-agent --test transcript_no_unwrap_lint
```

### Snapshot tests

Run snapshot tests and review any pending changes interactively:

```bash
# Run and reject any unreferenced snapshots (mirrors CI behaviour)
cargo insta test --unreferenced=reject --workspace

# Accept new/changed snapshots interactively
cargo insta review
```

### CI gates script (local)

The Phase 21.7 gates script runs four targeted invariant checks and can be executed locally from the workspace root:

```bash
bash scripts/ci-gates.sh
```

## Writing New Tests

### File naming conventions

| Type | Location | Pattern |
|---|---|---|
| Unit tests | Inline in source file | `#[cfg(test)]` module at bottom of `.rs` file |
| Integration tests | `crates/<crate-name>/tests/` | `<descriptive_name>.rs` |
| Snapshot tests | `crates/<crate-name>/tests/snapshots/` | Auto-generated by `insta` |

### Unit test pattern

Place unit tests in the same source file as the code under test:

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn my_unit_test() {
        // ...
    }
}
```

### Integration test pattern

Create a file under `crates/<crate-name>/tests/`. The test has access to the crate's public API:

```rust
// crates/ironhermes-agent/tests/my_integration_test.rs

#[test]
fn my_scenario() {
    // Use public API of ironhermes_agent
}
```

### Static-grep invariant tests

Many invariant tests use `include_str!` to load source files and assert structural properties at compile time. This pattern prevents CI regressions without needing a running process:

```rust
const MAIN_RS: &str = include_str!("../src/main.rs");

#[test]
fn invariant_example() {
    let count = MAIN_RS.matches("some_required_call(").count();
    assert_eq!(count, 3, "Expected exactly 3 call sites; found {}", count);
}
```

### Snapshot tests (insta)

Snapshot tests require the `test-support` feature flag on the relevant crate. After the first run, review and accept snapshots with `cargo insta review`. Committed snapshots live in `crates/<crate-name>/tests/snapshots/`.

```rust
#![cfg(feature = "test-support")]

#[test]
fn snap_my_frame() {
    let output = render_something();
    insta::assert_snapshot!(output);
}
```

### Async tests

Use the `#[tokio::test]` attribute for async test functions:

```rust
#[tokio::test]
async fn my_async_test() {
    // ...
}
```

## CI Integration

Tests run automatically on every push and pull request targeting the `develop` or `main` branches via `.github/workflows/ci.yml`.

### CI jobs

| Job | Trigger | Command |
|---|---|---|
| `phase-21-7-gates` | push / PR to `develop`, `main` | `bash scripts/ci-gates.sh` |
| `insta-snapshots` | push / PR to `develop`, `main` | `cargo insta test --unreferenced=reject --workspace` |
| `workspace-tests` | push / PR to `develop`, `main` | `cargo test --workspace --all-features` |
| `lint` | push / PR to `develop`, `main` | `cargo fmt --all -- --check` + `cargo clippy --workspace --all-targets --all-features -- -D warnings` |

### Phase 21.7 CI gates

`scripts/ci-gates.sh` enforces four invariants that correspond to AI-SPEC evaluation dimensions:

| Gate | ID | What it checks |
|---|---|---|
| Gate 1 | E-05 | `BudgetHandle` uses only `SeqCst` ordering — no `Ordering::Relaxed` |
| Gate 2 | E-08 | Transcript writer never calls `.unwrap()` or `.expect()` — errors are fire-and-forget |
| Gate 3 | E-09 | Three-site wiring parity for `AgentSubagentRunner::new`, `register_delegate_task_tool`, and execute-code registration |
| Gate 4 | D-12 | Gateway and `main.rs` do not read a per-request `yolo` field |

All CI jobs run on `ubuntu-latest` with `Swatinem/rust-cache@v2` for dependency caching.

### Environment variable: `RUSTFLAGS`

CI sets `RUSTFLAGS="-D warnings"` globally. All compiler warnings are treated as errors in CI. Run locally with the same flag to catch issues before pushing:

```bash
RUSTFLAGS="-D warnings" cargo test --workspace --all-features
```
