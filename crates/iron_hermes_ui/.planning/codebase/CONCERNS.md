# Codebase Concerns

**Analysis Date:** 2026-05-02

## Tech Debt

**Scaffold placeholder — entire app is unimplemented:**
- Issue: `src/main.rs` (all 38 lines) is the default Dioxus scaffold generated by `dx new`. The `Hero` component renders links to Dioxus documentation and an auto-generated header SVG. No application logic has been written.
- Files: `src/main.rs`
- Impact: The app has no relation to the IronHermes product yet. Every feature visible in the design prototype still needs to be built from scratch.
- Fix approach: Replace the `Hero` component with the actual application shell described in the design prototype (`warp2ironhermes/project/`). Start with `src/main.rs` and expand into a proper `src/` module hierarchy.

**Single-file architecture has no room to grow:**
- Issue: The entire codebase lives in one file (`src/main.rs`). No modules, no component files, no service layer.
- Files: `src/main.rs`
- Impact: Any implementation of the design will immediately require a proper directory structure (`src/components/`, `src/state/`, `src/api/`, etc.). Adding code directly to `main.rs` will create an unmaintainable monolith.
- Fix approach: Establish a module structure before writing feature code. Minimum viable layout: `src/main.rs`, `src/app.rs`, `src/components/mod.rs`, `src/state.rs`.

**`dioxus` declared with no features enabled:**
- Issue: `Cargo.toml` line 10 declares `dioxus = { version = "0.7.1", features = [] }`. The `web` feature is wired through a Cargo feature flag (`[features] web = ["dioxus/web"]`) but the default feature activates it. This is workable but non-obvious and will silently compile with zero platform support if `--no-default-features` is passed.
- Files: `Cargo.toml`
- Impact: Low risk now, but will cause confusing compile failures if anyone builds without defaults. Also, `desktop` and `mobile` feature flags are declared but the corresponding `dioxus/desktop` and `dioxus/mobile` features are not listed in the dependency, so those builds may fail or produce wrong binaries.
- Fix approach: Add explicit features to the `dioxus` dependency line, e.g. `features = ["web", "desktop", "mobile"]`, and remove the indirection through the package-level feature flags unless a single-binary multi-target build is intended.

**`assets/tailwind.css` is an empty file:**
- Issue: `assets/tailwind.css` is 0 bytes. The `main.rs` stylesheet link at line 6 loads it, so nothing from Tailwind is applied in the built app.
- Files: `assets/tailwind.css`, `tailwind.css` (root-level, also present)
- Impact: All styling that depends on Tailwind utility classes will be absent at runtime. The design prototype relies heavily on a custom CSS design system; without Tailwind output, layout and spacing utilities will be missing.
- Fix approach: Run `dx serve` (which auto-generates Tailwind output per `README.md`) or wire the root-level `tailwind.css` input file correctly via `Dioxus.toml` using `[application] tailwind_input` and `tailwind_output` keys.

**Duplicate `tailwind.css` files in two locations:**
- Issue: A `tailwind.css` exists at the project root (Dioxus Tailwind input) and at `assets/tailwind.css` (Dioxus Tailwind output, currently empty). The relationship is implicit — there is no `tailwind_input`/`tailwind_output` configuration in `Dioxus.toml`.
- Files: `tailwind.css`, `assets/tailwind.css`, `Dioxus.toml`
- Impact: Confusion over which file to edit, and the auto-build may silently skip generation. New contributors will not know which file is the source of truth.
- Fix approach: Add to `Dioxus.toml`:
  ```toml
  [application]
  tailwind_input = "tailwind.css"
  tailwind_output = "assets/tailwind.css"
  ```

**`assets/main.css` uses generic web fonts, not the IronHermes design system:**
- Issue: `assets/main.css` uses `'Segoe UI', Tahoma, Geneva, Verdana, sans-serif` — standard Windows system fonts. The design prototype uses `Ioskeley Mono` (a Berkeley Mono look-alike) with a full 16-weight woff2 family stored in `warp2ironhermes/project/ironhermes/fonts/`.
- Files: `assets/main.css`, `warp2ironhermes/project/ironhermes/colors_and_type.css`
- Impact: The app will render with the wrong font family, breaking the entire visual identity. The design system is monospace-only; using a proportional sans-serif is a category mismatch.
- Fix approach: Copy the `IoskeleyMono-*.woff2` font files to `assets/fonts/` and port the `@font-face` declarations and CSS custom properties from `warp2ironhermes/project/ironhermes/colors_and_type.css` into the project's CSS layer.

**`Dioxus.toml` is empty except for a title:**
- Issue: `Dioxus.toml` contains only `[application]` (no values) and `[web.app] title = "iron_hermes_ui"`. All advanced configuration (tailwind paths, bundle identifiers, app icon, etc.) is absent.
- Files: `Dioxus.toml`
- Impact: The app title is `iron_hermes_ui` (the Rust crate name) rather than a user-facing product name. Any deployment or distribution will require filling this out.
- Fix approach: Set `title = "IronHermes"`, configure tailwind paths, and add bundle metadata before any release build.

## Known Bugs

**Token context budget display shows hardcoded values:**
- Symptoms: The prototype `StatusBar` component in `warp2ironhermes/project/app/shell.jsx` (lines 30-49) receives `tokens={{ used: 12300, max: 128000 }}` as hardcoded props from `app.jsx` line 218. This data is fake; there is no live token tracking.
- Files: `warp2ironhermes/project/app/shell.jsx`, `warp2ironhermes/project/app/app.jsx`
- Trigger: Always — the values never change during any session.
- Workaround: N/A for prototype. When implementing in Dioxus, a real token counter must be wired from the API response metadata.

**`pulseScanner._t` timer stored as function property — non-idiomatic and leaks in Dioxus:**
- Symptoms: `app.jsx` line 144-148 stores a `setTimeout` ID as `pulseScanner._t`, a JavaScript-ism (attaching state to a function object). This pattern has no equivalent in Rust/Dioxus and must not be copied.
- Files: `warp2ironhermes/project/app/app.jsx` lines 144-148
- Trigger: Every time `submit()` is called, a new timeout races against the previous one.
- Workaround: In Dioxus, use `use_signal` to hold the timer state and cancel via `spawn` task handles.

## Security Considerations

**Warp2IronHermes handoff zip committed to repository:**
- Risk: `warp2ironhermes-handoff.zip` (7.8 MB) is tracked in git at the repository root. Zip files in version control are opaque blobs — they are not diffable, can contain sensitive data, and bloat the repository permanently once committed.
- Files: `warp2ironhermes-handoff.zip`
- Current mitigation: None. The `.gitignore` does not exclude it.
- Recommendations: Add `warp2ironhermes-handoff.zip` to `.gitignore`. The zip contents are already fully extracted into `warp2ironhermes/` — the zip itself is redundant. If provenance must be preserved, use Git LFS or an external artifact store.

**`warp2ironhermes/` design files committed alongside production code:**
- Risk: The handoff bundle contains HTML prototypes that load React and Babel from unpkg CDN via `<script>` tags with `integrity` hashes (`Warp × IronHermes.html` lines 17-19). These CDN scripts will only be served from `app.ironhermes.dev` in production if someone accidentally deploys the design files. The Babel standalone build also processes arbitrary JSX client-side, which is a significant script execution surface.
- Files: `warp2ironhermes/project/Warp × IronHermes.html`
- Current mitigation: Files are not referenced by the Dioxus build.
- Recommendations: Move `warp2ironhermes/` to a separate non-deployed directory or list it explicitly in build exclusions. Confirm `Dioxus.toml` asset paths never include it.

**No authentication or authorization layer exists:**
- Risk: The prototype references API key configuration (`STATUS_TEXT` in `app.jsx` lines 25-36 shows `OpenRouter`, `Anthropic`, and `OpenAI` keys as "configured"). When the real app is built, these keys will need to be handled server-side and must never be embedded in WASM client bundles.
- Files: `warp2ironhermes/project/app/app.jsx` lines 25-36
- Current mitigation: No backend exists yet; this is a prototype-only concern.
- Recommendations: Plan a server-function boundary (`dioxus/server` feature) before any API key usage is implemented. API keys must live in server functions only, never in `web` or `desktop` feature code.

**`.DS_Store` files not excluded globally:**
- Risk: `warp2ironhermes-handoff.zip` and `.DS_Store` are both untracked in git status. The `.gitignore` excludes `.DS_Store` at the root but not recursively for all subdirectories.
- Files: `.gitignore`
- Current mitigation: Root-level `.DS_Store` is excluded.
- Recommendations: Change `.DS_Store` to `**/.DS_Store` in `.gitignore` to cover nested directories.

## Performance Bottlenecks

**Scanner animation runs at 100 ms tick unconditionally:**
- Problem: The `Scanner` component (`warp2ironhermes/project/app/shell.jsx` lines 5-27) uses `setInterval(..., 100)` — 10 re-renders per second — for a purely cosmetic animation. When `active=false` the interval is cleared, but startup always fires it once.
- Files: `warp2ironhermes/project/app/shell.jsx`
- Cause: Interval-driven re-renders trigger the full React subtree diff for the status bar on every tick.
- Improvement path: In Dioxus, implement using CSS `@keyframes` animation on the scanner cells instead of signal-driven re-renders. This offloads animation to the GPU compositor and eliminates the per-tick VDOM diff entirely.

**All fake shell and agent responses use `setTimeout` (no real async):**
- Problem: `runShell` and `runAgent` in `app.jsx` (lines 163-185) use `setTimeout` with hardcoded delays (600ms, 400ms, 1400ms). These represent real network round-trips in the final app.
- Files: `warp2ironhermes/project/app/app.jsx`
- Cause: Prototype convenience — no real backend.
- Improvement path: Replace with `use_resource` (for one-shot fetches) or `use_coroutine` (for streaming SSE responses) when implementing in Dioxus.

## Fragile Areas

**`warp2ironhermes/project/Warp × IronHermes.html` filename contains a special character:**
- Files: `warp2ironhermes/project/Warp × IronHermes.html`
- Why fragile: The filename contains a Unicode multiplication sign (`×`, U+00D7) and spaces. This causes issues with shell globbing, some CI systems, Windows filesystems, and HTTP servers that don't percent-encode paths. It also requires quoting in every shell command that references it.
- Safe modification: When copying assets into the Dioxus project, rename to `warp-ironhermes.html` or similar ASCII-safe equivalent.
- Test coverage: None.

**`CommandPalette` initializes `useState` inside a conditional render path:**
- Files: `warp2ironhermes/project/app/shell.jsx` lines 214-268
- Why fragile: `CommandPalette` returns `null` on line 215 (`if (!open) return null`) *before* calling `React.useState(0)` on line 219. This violates the Rules of Hooks — hooks must not be called after an early return. This is a latent bug in the prototype that will cause React warnings and potential state corruption when `open` toggles.
- Safe modification: Move `const [active, setActive] = React.useState(0)` above the `if (!open) return null` guard before porting this component to Dioxus.
- Test coverage: None.

**No error boundary or fallback UI:**
- Files: `src/main.rs`
- Why fragile: The Dioxus app has no error handling at any level. A panic in any component will crash the entire WASM runtime with no user-visible recovery path.
- Safe modification: Wrap top-level routes in Dioxus `ErrorBoundary` components once the router is introduced.
- Test coverage: None.

## Scaling Limits

**Single WASM binary for all platforms (web/desktop/mobile):**
- Current capacity: Feature flags in `Cargo.toml` expose `web`, `desktop`, and `mobile` targets, but all share one `main.rs` entry point with no platform-specific code paths.
- Limit: As UI complexity grows, platform-specific behaviors (mobile tab bar, desktop window chrome, web browser address bar) will require conditional compilation (`#[cfg(feature = "web")]`) or separate entry points. Currently there is no structure for this.
- Scaling path: Introduce `src/platform/web.rs`, `src/platform/desktop.rs`, `src/platform/mobile.rs` modules early, gated behind Cargo feature flags.

## Dependencies at Risk

**No dependency pinning — `dioxus = "0.7.1"` accepts patch bumps:**
- Risk: The semver specifier `"0.7.1"` in `Cargo.toml` will accept `0.7.x` patch releases automatically. Dioxus 0.7 is a major API overhaul (noted in `AGENTS.md`: "`cx`, `Scope`, and `use_state` are gone"). Patch releases in an actively developed framework can still introduce breaking changes in practice.
- Impact: A future `cargo update` could silently pull in an incompatible patch.
- Migration plan: Pin to `"=0.7.1"` in `Cargo.toml` and update deliberately. Add `Cargo.lock` to version control (currently absent from tracked files per git status output).

**`Cargo.lock` not committed:**
- Risk: `Cargo.lock` is absent from the tracked files listed in git status. For a binary application (not a library), `Cargo.lock` should be committed to ensure reproducible builds.
- Impact: Every `cargo build` on a fresh clone may produce a different dependency graph.
- Migration plan: Run `cargo build` to generate `Cargo.lock` and commit it. Remove `/target` from `.gitignore` is already correct — ensure `Cargo.lock` is not in `.gitignore`.

## Missing Critical Features

**No routing:**
- Problem: The app has a single component (`Hero`). The design prototype shows at minimum three distinct views: desktop shell, web shell, and mobile tabbed shell. No `Router` or `Route` enum exists.
- Blocks: Multi-page navigation, deep-linking, and platform-adaptive layout branching.

**No state management:**
- Problem: No `use_signal`, `use_context_provider`, or global state layer exists. The prototype's state (blocks, messages, input mode, palette state) all lives in a single React component; the Dioxus equivalent needs to be architected before implementation begins.
- Blocks: Any interactive component, real-time updates, and AI response streaming.

**No backend / server functions:**
- Problem: The design prototype uses `setTimeout` to simulate all AI and shell responses. No `dioxus/server` feature, no `#[server]` functions, and no HTTP client dependency exist in `Cargo.toml`.
- Blocks: Real AI responses, shell command execution, token tracking, and personality switching.

**No tests:**
- Problem: Zero test files exist. No `#[cfg(test)]` blocks in `src/main.rs`. No integration test directory (`tests/`). No test runner configuration.
- Blocks: Any confidence in correctness during implementation. The Dioxus test utilities (`dioxus::dioxus_core::VNode` snapshot testing) are not set up.

- What's not tested: The HTML prototype in `warp2ironhermes/project/` is the visual specification for the app. There are no Playwright, Puppeteer, or screenshot-diff tests against it.
