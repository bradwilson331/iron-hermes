<deviations>
**D-01 cascade-order deviation (PLANNER DECISION — applies to this plan only):**

CONTEXT.md D-01 specifies the relative cascade order of the three ported CSS files: `main.css` → `design-tokens.css` → `warp-ih.css`. D-01 does NOT explicitly fix where `tailwind.css` goes in that chain, and the Phase 1 code currently emits Tailwind LAST.

RESEARCH.md (Pattern 2, Pitfall 1, Open Question 2) and PATTERNS.md ("Cascade-Aware <link> Ordering" shared pattern) both recommend loading `tailwind.css` FIRST, before any of the three ported files, so Tailwind v4 preflight (which resets body font-family, margin, line-height) loses the cascade against the prototype's body styles. This is consistent with D-01's relative ordering of the three ported files (which is preserved verbatim).

**Decision:** Honor the research recommendation. The final `<head>` order is:

1. `FAVICON` (icon, not stylesheet)
2. `TAILWIND_CSS` (preflight loses cascade — Pitfall 1)
3. `MAIN_CSS` (project-specific base reset — D-01 pos 1 within ported files)
4. `DESIGN_TOKENS_CSS` (ANSI tokens, fonts, body styles — D-01 pos 2)
5. `WARP_IH_CSS` (Warp shell tokens + .wh-* classes — D-01 pos 3, last-wins for `--w-bg-*`)

The relative order of the three ported files (`main.css` → `design-tokens.css` → `warp-ih.css`) matches D-01 verbatim. Tailwind moves to position 2 (was last in Phase 1) so its preflight does not override prototype body styles.

If the user overrides this and prefers strict D-01 + Tailwind-last, the alternative is to add `@layer base { html, body { font-family: var(--font-body); background: var(--bg); color: var(--fg); } }` to `assets/main.css`. This adds 4 lines of cascade-fighting CSS for no behavioural benefit; the planner does not recommend it.
</deviations>
