## Warnings

### WR-01: `mod` declaration order is neither alphabetical nor dependency-ordered

**File:** `src/main.rs:3-6`
**Issue:** Current order is `mod app; mod components; mod state; mod platform;` — `app` depends on `components`, so it precedes its dependency, and `state` / `platform` are out of alpha order. Rust does not require declaration ordering, but inconsistency here will make future ordering decisions ambiguous as modules are added.
**Fix:** Pick a convention and document it in CONVENTIONS.md. Recommended (alphabetical):

```rust
mod app;
mod components;
mod platform;
mod state;

use app::App;
```

### IN-04: `Cargo.toml` lacks `[lints.clippy]` table to enforce `clippy.toml` rules at `cargo build`

**File:** `Cargo.toml` (whole file)
**Issue:** `clippy.toml` configures `await-holding-invalid-types` — the project's most important runtime-safety guardrail (CLAUDE.md line 88). Without a `[lints.clippy]` table elevating `await_holding_invalid_type` to `deny`, `cargo build` will not fail when a developer writes `signal.read().await`; only `cargo clippy` warns. If CI does not gate on clippy, the rule is effectively unenforced. This is a hygiene gap squarely in Phase 1's scope.
**Fix:**

```toml
[lints.clippy]
await_holding_invalid_type = "deny"
```

Consider adding to Phase 1 scope or filing as a follow-up before any async code lands.
