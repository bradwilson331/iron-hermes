## Implementation Decisions

### Phase Scope & Split

SKILL-09 is M-effort (~150 LOC) with user-facing UX decisions (how users set missing env vars, how activation responds) that deserve a dedicated discussion and verification pass.
- **D-02:** Phase 07.2 requirements addressed: `SKILL-05`, `SKILL-06`, `SKILL-07`, `SKILL-08`. These REQ-IDs are already mapped to Phase 07.2 in `REQUIREMENTS.md`'s traceability table. `SKILL-09` is currently mapped to Phase 07.2 there as well — it must be re-mapped to Phase 07.3 when that phase is created (follow-up housekeeping, not in this plan).

### SKILL-05: `platforms` Filter
- **D-04:** Add `platforms: Option<Vec<String>>` to `SkillFrontmatter`. Skills with a present `platforms` list are included only if the current OS appears in the list. Skills without the field (or with `platforms: []`) load on all platforms (spec default).
- **D-05:** Platform string mapping is **strict, no aliases** (matches agentskills.io spec and hermes-agent `_PLATFORM_MAP`):
  - `"macos"` → matches `cfg!(target_os = "macos")`
  - `"linux"` → matches `cfg!(target_os = "linux")`
  - `"windows"` → matches `cfg!(target_os = "windows")`
  - Any other platform string is unknown and treated as a non-match. A skill that only lists unknown platforms is filtered out.
- **D-06:** Platform check runs in `load_with_paths()` **after** YAML parse but **before** the `seen_names` dedup check. This ensures a filtered-out skill does not reserve its name slot against higher-priority paths.
- **D-07:** Tests required: (a) skill with matching platform loads, (b) skill with non-matching platform is skipped, (c) skill with no platforms field loads (default behavior preserved), (d) skill with mixed platforms list including current OS loads, (e) unknown platform string treated as non-match.

### SKILL-06: Extended Frontmatter Fields
- **D-08:** Add to `SkillFrontmatter`:
  - `compatibility: Option<String>` — plain string environment hint.
  - `allowed_tools: Option<Vec<String>>` with `#[serde(rename = "allowed-tools")]`.
  - `metadata: Option<serde_yaml::Value>` — **opaque** blob storing everything under `metadata:` including `metadata.hermes.*` extensions.
- **D-09:** Metadata representation is **opaque** (`serde_yaml::Value`), NOT a typed `HermesMetadata` struct. Rationale: forward-compatible with future hermes-agent extensions without code changes; matches the "parse leniently" divergence rule from Phase 07.1 D-09. Access to metadata sub-fields (e.g., `tags`, `category`, `requires_toolsets`) is via helper methods if ever needed by downstream code — not required in this phase.
- **D-10:** `allowed_tools` is **parsed and stored but NOT enforced** at activation time. Enforcement is a future security feature with its own design discussion. Parsing ensures hermes-agent skills load without losing information.
- **D-11:** `SkillRecord` gains matching fields so downstream code can access them without re-parsing the SKILL.md file:
  - `platforms: Option<Vec<String>>` (for introspection/debugging)
  - `compatibility: Option<String>`
  - `allowed_tools: Option<Vec<String>>`
  - `metadata: Option<serde_yaml::Value>`
- **D-12:** Catalog output (`catalog_text()`) is **UNCHANGED** in this phase. New fields do NOT leak into the system prompt. Catalog format stays `"- name: description"` per Phase 7 D-04.

### SKILL-07: Name + Description Validation Rules
- **D-13:** Validation rules enforced:
  - Name regex: `^[a-z0-9]([a-z0-9-]*[a-z0-9])?$` (lowercase alphanumeric + hyphens, no leading/trailing hyphens).
  - Additional consecutive-hyphen check: reject names containing `--`.
  - Name length: 1–64 characters.
  - Description length: 1–1024 characters.
  - Directory name must match skill name (case-insensitive) at load time.
- **D-14:** **Validation policy split by severity:**
  - **Strict reject (skill dropped):** invalid name regex OR name length outside 1–64 OR consecutive hyphens in name. `parse_skill_md` returns `None`. Logged via `tracing::warn!` (upgraded from current `debug!`) with file path + reason.
  - **Warn-but-load:** description length outside 1–1024. Logged via `tracing::warn!`. Skill still loads with the description as-is.
  - **Warn-but-load:** directory name does not match skill name (case-insensitive). Logged via `tracing::warn!`. Skill still loads.
- **D-15:** **Validation execution location:**
  - Name regex + consecutive-hyphen check + name length + description length → inside `parse_skill_md()` after successful YAML parse. Keeps the function pure and testable without filesystem.
  - Directory-name-match check → inside `load_with_paths()` since it requires the on-disk directory path.
- **D-16:** Tests required:
  - valid name accepted;
  - invalid name (uppercase, underscore, starts with hyphen, ends with hyphen, consecutive hyphens) rejected;
  - name too long rejected;
  - name empty/too short rejected;
  - description too long warn-loads with correct content;
  - dir-name mismatch warn-loads;
  - dir-name case-insensitive match succeeds silently.

### SKILL-08: SkillsConfig in config.yaml
- **D-17:** Add a new `SkillsConfig` struct to `crates/ironhermes-core/src/config.rs`:
  ```rust
  pub struct SkillsConfig {
      pub enabled: bool,             // default true
      pub extra_paths: Vec<PathBuf>, // appended AFTER the 3 defaults
  }
  ```
  with `impl Default` returning `{ enabled: true, extra_paths: vec![] }` so existing `config.yaml` files preserve current behavior.
- **D-18:** Add `pub skills: SkillsConfig` to the top-level `Config` struct. Because `Config` already uses `#[serde(default)]`, existing configs without a `skills:` section parse without change.
- **D-19:** `extra_paths` semantics: **appended AFTER** the 3 hardcoded defaults (`cwd/.ironhermes/skills/`, `~/.ironhermes/skills/`, `~/.agents/skills/`). First-path-wins dedup still applies across the combined list, so defaults retain priority over extras.
- **D-20:** `enabled: false` → `SkillRegistry` returns an empty registry without scanning any paths. Downstream code (PromptBuilder catalog injection, cron `resolve_skill_context`, etc.) must tolerate an empty registry — Phase 7 already handles this case.
- **D-21:** `SkillRegistry::load` gains a new constructor variant that accepts a `&Config` (or `&SkillsConfig`) so CLI/gateway/cron wiring can pass the user's config. The existing `load(cwd)` becomes a thin wrapper that uses `SkillsConfig::default()` for backward compat (tests and any callers that don't yet have `Config`).
- **D-22:** Tests required:
  - default config loads from the 3 defaults;
  - `extra_paths` are appended (verify order);
  - `extra_paths` respect first-path-wins dedup when they re-declare a skill that already exists in a default path;
  - `enabled: false` → empty registry;
  - `config.yaml` round-trip (serialize + deserialize).

### Project Conventions
- **D-23:** **New dependencies:** verify `regex` is already a dep of `ironhermes-core` (likely yes via other crates). If not, add it for SKILL-07 — no other new deps required. `serde_yaml` is already in use for current SKILL.md parsing, so opaque `metadata` storage comes free.
- **D-24:** Follow existing `ironhermes-core` patterns: keep modules in `skills.rs` (no new file splits); tests stay in the same file's `#[cfg(test)] mod tests` block; adhere to atomic structured changes per PLAN.
- **D-25:** No behavior changes to the Phase 7 `SkillsTool` (`ironhermes-tools/src/skills_tool.rs`). New frontmatter fields become available via `SkillRecord` if downstream code wants them; `list/view/activate` semantics remain exactly as in Phase 7.
- **D-26:** No changes to `PromptBuilder` catalog injection (`ironhermes-agent/src/prompt_builder.rs`). New frontmatter fields do NOT leak into the system prompt.

### Claude's Discretion
- Exact ordering of validation checks within `parse_skill_md()` (YAML-parse first, then name regex, then consecutive-hyphen check, then length checks — pick a natural order).
- Whether to extract a dedicated `validate_skill_name(&str) -> Result<(), &'static str>` helper or inline the checks.
- Regex compilation strategy: `once_cell::sync::Lazy` vs `LazyLock` (Rust ≥1.80) vs compile-per-call. Use whatever is idiomatic for the existing codebase — pick what's already in use elsewhere in `ironhermes-core`.
- Error message wording in `tracing::warn!` calls — must be unambiguous, identify the SKILL.md path, and state the rule that was violated.
- Whether to implement the `SkillRegistry::load` Config-aware constructor as a new method (`load_with_config`) or a required parameter (breaking change).
- Whether `SkillRecord` stores `compatibility` trimmed vs raw — spec hints at 1–500 chars but this phase only parses + stores.
- Internal field visibility on new `SkillRecord` fields (pub vs pub(crate) — pick what matches existing fields).

## Existing Code Insights

### Reusable Assets
- **`serde_yaml::Value`** — already in the `serde_yaml` dep used by current SKILL.md parsing. Opaque `metadata` storage is free.
- **`SkillFrontmatter: Deserialize`** — extending it is a pure addition; existing SKILL.md files without the new fields continue to parse (Option types default to None).
- **`tracing::warn!` / `debug!`** — already used in `skills.rs::load_with_paths` for parse errors and dedup. Warn-but-load violations follow the same style; upgrade the log level from `debug!` to `warn!` for validation issues.
- **Phase 7 test harness** — 14 unit tests in `skills.rs::tests`. New tests for SKILL-05..08 extend that same module so coverage stays local.
- **Top-level `#[serde(default)]` on `Config`** — zero-migration addition of the new `skills: SkillsConfig` field.

### Established Patterns
- **Additive frontmatter via serde defaults** — `Option<T>` fields plus `#[serde(default)]` where needed means existing config/skill files load without modification.
- **First-path-wins dedup in `load_with_paths`** — via `seen_names: HashSet<String>`. **Platform filter (D-06) must run BEFORE this dedup** to avoid a filtered-out skill blocking a lower-priority path from taking its name slot.
- **Warn-on-issue, load-anyway** — `tracing::debug!` today in `load_with_paths` for duplicates, missing files, and invalid YAML. Upgrading to `warn!` for validation issues matches the existing idiom.
- **Config crate-level `#[serde(default)]`** — means `SkillsConfig` only needs its own `Default` impl; no migration required.

### Integration Points
- `SkillRegistry::load(cwd: &Path) -> Self` — current entry point called from `run_single`/`run_chat`/`run_gateway`. Needs a Config-aware variant per D-21 (new method or new parameter — planner's call).
- `SkillRegistry::load_with_paths(&[PathBuf]) -> Self` — stays as the underlying mechanism; the new Config-aware constructor computes the combined path list and delegates here.
- `parse_skill_md(content: &str) -> Option<(SkillFrontmatter, String)>` — signature unchanged, but behavior becomes stricter (returns None on name-level violations, logs warn on description-length violations before returning Some).
