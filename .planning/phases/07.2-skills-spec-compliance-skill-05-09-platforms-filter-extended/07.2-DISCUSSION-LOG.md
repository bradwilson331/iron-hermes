## Validation policy (SKILL-07): severity

| Option | Description | Selected |
|--------|-------------|----------|
| Strict reject on name, warn on rest | Invalid name regex / length / consecutive hyphens → drop skill. Description length and dir-name mismatch → warn-but-load. Matches agentskills.io "non-critical issues should warn-but-load" guidance. | ✓ |
| Warn-but-load everything | All violations emit `tracing::warn!` and the skill still loads. Most permissive for migrating existing hermes-agent skills, but agent may see skills with broken references. | |
| Strict reject everything | Any violation → skill is silently dropped. Safest for production; risk of silently losing skills with minor mistakes. | |

**User's choice:** Strict reject on name, warn on rest

---

## Validation policy (SKILL-07): execution location

| Option | Description | Selected |
|--------|-------------|----------|
| Name regex + length at parse; dir-match at load | `parse_skill_md()` validates frontmatter-local rules (name, description). `load_with_paths()` validates the one cross-file rule (directory name matches skill name). Clean separation — parse stays pure and testable without filesystem. | ✓ |
| All validation at load time | `parse_skill_md` stays permissive (just YAML parse). All validation happens in `load_with_paths` so tests can bypass validation by calling parse directly. More complexity in load. | |

**User's choice:** Name regex + length at parse; dir-match at load

## Metadata typing (SKILL-06): storage

| Option | Description | Selected |
|--------|-------------|----------|
| Opaque `serde_yaml::Value` | Store as `Option<serde_yaml::Value>`. Lazy access via helper methods. Forward-compatible — new hermes-agent extensions parse without code changes. Matches "parse leniently" rule from 07.1 D-09. | ✓ |
| Typed `HermesMetadata` struct | Define `struct HermesMetadata { tags, category, requires_toolsets, fallback_for_toolsets, config, ... }`. Compile-time safety and IDE autocomplete. Risk: every new hermes-agent field needs a code change. | |
| Hybrid: typed common + opaque extras | Typed fields for tags/category/requires_toolsets + opaque `extras: serde_yaml::Value` catchall. Balance of safety and flexibility; more code than opaque, less than fully typed. | |

**User's choice:** Opaque `serde_yaml::Value`

## Metadata typing (SKILL-06): `allowed-tools` handling

| Option | Description | Selected |
|--------|-------------|----------|
| Parse and store, don't enforce | Add `allowed_tools: Option<Vec<String>>` (with `#[serde(rename = "allowed-tools")]`) to `SkillFrontmatter` + `SkillRecord`. Do NOT enforce at activation time — that is a future security feature. Ensures hermes-agent skills parse cleanly now. | ✓ |
| Parse, store, and enforce at activation | Same as above, plus filter the `ToolRegistry` passed to skill-loaded context when `allowed_tools` is present. Full semantic implementation — adds complexity, testing surface, and design discussion about WHERE enforcement happens. | |
| Ignore entirely | `serde` silently drops unknown fields. Simplest but loses information and violates SKILL-06's spirit. | |

**User's choice:** Parse and store, don't enforce
