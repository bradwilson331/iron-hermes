# Phase 07.2 Plan 01: SKILL-06 Extended SkillFrontmatter and SkillRecord Summary

**One-liner:** Extended `SkillFrontmatter` and `SkillRecord` with agentskills.io + hermes-agent backward-compat fields (`compatibility`, `allowed_tools` via kebab-case rename, opaque `metadata` as `serde_yaml::Value`), with destructuring construction site and full test coverage.

## What Was Built

Extended `crates/ironhermes-core/src/skills.rs` to add three new optional fields to `SkillFrontmatter` and mirror them on `SkillRecord`. This gives downstream code access to extended frontmatter without re-parsing SKILL.md files, and ensures hermes-agent skills with `metadata.hermes.*` extensions parse losslessly per Phase 07.1 D-09 ("parse leniently").

## Decisions Made

1. **Field ordering:** New fields appended at end of each struct (after `license`, after `path`) to minimize diff noise and avoid reordering existing fields.
2. **`serde(default)` on all three:** Ensures existing SKILL.md files without the new fields continue to parse without error ‚Äî backward compat is free with `Option<T>` + `serde(default)`.
3. **Opaque `serde_yaml::Value` for metadata:** Per D-09 ‚Äî forward-compatible with future `metadata.hermes.*` extensions without typed schema changes.
4. **Destructuring pattern at construction site:** Explicit about which frontmatter fields flow to `SkillRecord` and which are dropped (`version`, `author`, `license`), giving plan 07.2-02 a clean insertion point for the `platforms` field.
