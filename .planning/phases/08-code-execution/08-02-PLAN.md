Define `RegistryDispatch` — an adapter that implements `ironhermes_exec::ToolDispatch` by wrapping `Arc<ToolRegistry>`. This breaks the circular dependency (ironhermes-exec does not depend on ironhermes-tools; the adapter lives in ironhermes-tools).

```rust
use std::sync::Arc;
use async_trait::async_trait;
use ironhermes_core::{ExecConfig, ToolSchema};
use ironhermes_exec::{Sandbox, SandboxConfig, SandboxResult, ToolDispatch};
use serde_json::json;
use tracing::debug;

use crate::registry::{Tool, ToolRegistry};

/// Adapter: bridges ToolRegistry::dispatch into the ToolDispatch trait
/// that ironhermes-exec expects. Lives here to avoid circular crate deps.
struct RegistryDispatch {
    registry: Arc<ToolRegistry>,
}

#[async_trait]
impl ToolDispatch for RegistryDispatch {
    async fn dispatch(&self, tool_name: &str, args: serde_json::Value) -> anyhow::Result<String> {
        self.registry.dispatch(tool_name, args).await
    }
}
```

**NOTE on the self-reference problem:** `ExecuteCodeTool` needs `Arc<ToolRegistry>` but is registered INTO the `ToolRegistry`. The solution is a two-registry approach:
- Build the main registry with all tools EXCEPT execute_code.
- Wrap it in `Arc<ToolRegistry>` — this is the "inner" registry that the RPC dispatch uses.
- Create a second `ToolRegistry` that clones the inner one's tools and adds `ExecuteCodeTool`.
- OR simpler: `ExecuteCodeTool` receives an `Arc<ToolRegistry>` that is a snapshot of other tools. The `execute_code` tool itself dispatches through this snapshot, which does NOT include `execute_code` — naturally preventing recursion (D-07 already excludes it from ALLOWED_TOOLS, but this is defense-in-depth).

The cleanest pattern for the CLI:
```rust
// In CLI main.rs (Task 2 handles this):
let mut registry = build_registry();
registry.register_memory_tool(memory_store.clone());
registry.register_cronjob_tool(job_store.clone());
registry.register_skills_tool(skill_registry.clone(), active_skills);
// Snapshot for RPC dispatch (all tools except execute_code)
let rpc_registry = Arc::new(std::mem::take(&mut registry));
// Rebuild with execute_code added
let mut final_registry = ToolRegistry::new();
// Re-register defaults + shared tools... OR:
// Clone approach: just create ExecuteCodeTool with rpc_registry, register into a fresh registry
```

Actually, the simplest approach: since `ToolRegistry` stores `Box<dyn Tool>` which is not Clone, we need a different pattern. The cleanest is:

**Use a lazy Arc pattern:** `ExecuteCodeTool` holds an `Arc<tokio::sync::OnceCell<Arc<ToolRegistry>>>` that gets filled after the full registry is wrapped in Arc. But this adds complexity.

**Simplest viable approach:** Build a SEPARATE `ToolRegistry` containing only the RPC-safe tools, wrap it in Arc, and give that to `ExecuteCodeTool`. This is straightforward and naturally excludes execute_code and terminal:

```rust
// In CLI main.rs:

rpc_registry.register_defaults();  // includes terminal — we need to NOT include it
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Agent LLM -> ExecuteCodeTool | LLM-generated Python code passed to sandbox |
| ExecuteCodeTool -> Sandbox | Tool formats args and delegates to exec crate |
| RPC dispatch -> ToolRegistry | Sandbox tool calls dispatched to real tools |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-09 | Elevation of Privilege | execute_code.rs RPC registry | mitigate | Separate RPC registry built with ONLY D-07 tools. `terminal` and `execute_code` structurally absent — cannot be called even if ALLOWED_TOOLS check were bypassed. Defense in depth with Plan 01's ALLOWED_TOOLS constant. |
| T-08-10 | Elevation of Privilege | Recursive execute_code | mitigate | execute_code excluded from RPC registry (not registered) AND excluded from ALLOWED_TOOLS in rpc_server.rs. Double barrier. |
| T-08-11 | Spoofing | LLM injecting malicious Python | accept | Per D-02: no filesystem/network restrictions. The LLM is the operator's agent — it generates code on behalf of the user. Env stripping (Plan 01) prevents secret exfiltration. |
| T-08-12 | Information Disclosure | Tool results returned to Python | accept | RPC tool results (e.g., file contents) are returned to the Python script which returns them as stdout to the LLM. This is by design — the agent requested the code execution. |
</threat_model>
