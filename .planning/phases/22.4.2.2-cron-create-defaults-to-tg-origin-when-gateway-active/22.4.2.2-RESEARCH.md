## Overview

Phase 22.4.2.2 restores the v1.x ergonomic where `hermes cron create` (invoked without `--deliver`) automatically routes job output to the configured Telegram chat when the TG gateway is active and has exactly one authorized chat. The Phase 22.4.2.1 regression fix (Plan 02) correctly wired `deliver="origin"` jobs through `dispatch_delivery` and `resolve_delivery_target`; this phase makes `cmd_create` produce such jobs by default instead of always writing `deliver="local"`. Scope is strictly one plan (~5 tasks): add `OriginDecision` enum + `telegram_default_origin` helper to `ironhermes-core::config`, modify `cmd_create` in `ironhermes-cli::cron` to consult the helper when `--deliver` is absent, add behavioral tests in a new `cron_default_deliver.rs` integration-test file, and append one static-grep INV to the existing `invariants_22_4.rs` ledger (62 → 63 INVs).

### Q4: `OriginDecision` enum — confirmed shape and placement

**Recommended API** (matches CONTEXT D-07, confirmed against Config patterns):

```rust
// In crates/ironhermes-core/src/config.rs, before the Config impl block

pub enum OriginDecision {
    None,
    Single(JobOrigin),                       // whitelist len == 1
    Multi { whitelist: Vec<String> },        // whitelist len > 1; caller emits eprintln
}
```

**`Config::telegram_default_origin` method body:**
```rust
pub fn telegram_default_origin(&self) -> OriginDecision {
    let Some(tg) = self.gateway.platforms.get("telegram") else {
        return OriginDecision::None;
    };
    if !tg.enabled {
        return OriginDecision::None;
    }
    match tg.whitelist.len() {
        0 => OriginDecision::None,
        1 => OriginDecision::Single(JobOrigin {
            platform: "telegram".to_string(),
            chat_id: tg.whitelist[0].to_string(),
            chat_name: None,
            thread_id: None,
        }),
        _ => OriginDecision::Multi {
            whitelist: tg.whitelist.iter().map(|id| id.to_string()).collect(),
        },
    }
}
```

**Crate dependency note (OPEN QUESTION):** `ironhermes-core` must be able to reference `ironhermes_cron::JobOrigin` to embed it in `OriginDecision::Single`. If `ironhermes-core` does NOT currently depend on `ironhermes-cron` (to avoid circular deps), there are two options:
1. Define a local `TelegramOrigin { platform: String, chat_id: String }` in `ironhermes-core::config` and have the CLI-side caller convert it to `JobOrigin`.
2. Define `OriginDecision` in `ironhermes-cron` (where `JobOrigin` lives) and have `ironhermes-core::config` expose a simpler helper that returns raw data.

See Open Question Q-OQ1 for resolution path.

fn cmd_create(
    name: String,
    schedule: String,
    prompt: String,
    deliver: Option<String>,   // was: String with default_value="local"
    skills: Vec<String>,

## Open Questions

### Q-OQ1: Crate dependency — can `ironhermes-core` import `ironhermes_cron::JobOrigin`?

**What we know:** `ironhermes-core` is a foundational crate. `ironhermes-cron` currently imports from `ironhermes-core` (confirmed: `store.rs` line 3: `use ironhermes_core::get_hermes_home`). Adding `ironhermes_cron` as a dependency of `ironhermes-core` would create a circular dependency.

**What's unclear:** Whether `OriginDecision::Single` needs to embed `JobOrigin` directly or can use a simpler data structure.

**Recommendation:** Define `OriginDecision` in `ironhermes-cron` (where `JobOrigin` lives), NOT in `ironhermes-core`. Then add a free function `pub fn telegram_default_origin(config: &Config) -> OriginDecision` in `ironhermes-cron` (e.g., in a new `config_ext.rs` or in `delivery.rs`). The CLI crate (`ironhermes-cli`) depends on both crates and can call this function. Alternatively, define a thin `TelegramOriginHint` struct in `ironhermes-core` that contains only raw strings, and have the CLI crate convert it to `JobOrigin`. **Planner must check `ironhermes-core/Cargo.toml` for existing `ironhermes-cron` dependency.**

### Q-OQ2: `cmd_create` visibility for integration tests

**What we know:** `cmd_create` is `fn` (private) in `cron.rs`. Integration tests in `crates/ironhermes-cli/tests/` are separate compilation units and cannot access private functions.

**What's unclear:** Whether the test author wants to call `cmd_create` directly or test through observable side effects (written jobs.json).

**Recommendation:** Test through `JobStore` side effects — set `IRONHERMES_HOME` to temp dir, call the binary via `std::process::Command` OR extract a `pub(crate) fn resolve_cron_deliver(deliver_flag: Option<String>, config: &Config) -> (String, Option<JobOrigin>)` helper that IS accessible via `pub` and call it directly from the integration test for the 5 behavioral cases, then verify `store.add_job` with the results. This avoids binary spawning overhead. **Planner decides.**
