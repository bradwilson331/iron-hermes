# Phase 22.4.2.2: Cron create defaults to TG origin when gateway active - Context

**Gathered:** 2026-04-26
**Status:** Ready for planning
**Source:** Direct context capture from user — design locked in conversation immediately following Phase 22.4.2.1 completion. UAT confirmed Plan 02 dispatch fires correctly when `deliver=telegram:<chat>`, but existing CLI-created jobs default to `deliver=local`, so users see no Telegram delivery despite the regression fix landing.

<domain>
## Phase Boundary

Restore the v1.x ergonomics where `hermes cron create` (with no `--deliver` flag) routes the job's output back to Telegram when a TG gateway is configured. The user's UAT after Phase 22.4.2.1 confirmed:

- `Ctrl+C` graceful shutdown works (Plan 03 verified live).
- The TG dispatch path (Plan 02) fires correctly for jobs with `deliver=telegram:<chat>` (unit-test-confirmed), but
- Three pre-existing weather jobs in `~/.ironhermes/cron/jobs.json` show `"deliver": "local"` and `"origin": null`, so output lands only in `~/.ironhermes/cron/output/<id>/` — never in Telegram.
- The user's recollection: in v1.x, jobs created from the CLI when a TG gateway was running DID deliver to TG by default. That UX has regressed.

This phase restores the default. **Existing jobs are not migrated** — the user fixes those one-time via `hermes cron edit <id> --deliver telegram:<chat_id>` or via the equivalent `cron edit` flow once Phase 22.4.2.2 lands.

**In scope (1 plan):**
- `cmd_create` in `crates/ironhermes-cli/src/cron.rs:138-?` — when `--deliver` is NOT explicitly passed, compute the default from `~/.ironhermes/config.yaml::gateway.platforms.telegram` per the rules in `<decisions>`.
- New helper `Config::telegram_default_origin(&self) -> Option<Origin>` in `crates/ironhermes-core/src/config.rs` (or wherever the existing config struct lives — researcher confirms exact module).
- Unit tests for the helper covering all four edge cases (TG disabled, whitelist empty, whitelist len==1, whitelist len>1).
- Integration test that `cmd_create` writes a job with `deliver="origin"` and `origin={platform: "telegram", chat_id: "<whitelist[0]>"}` when TG is enabled with single-chat whitelist.
- One INV-22.4.2.2-01 in `crates/ironhermes-cli/tests/invariants_22_4.rs` proving the helper is wired into `cmd_create`.
- Documentation note in CLI `--help` text and in test directions: enabling TG in config.yaml AFTER cron jobs already exist will affect FUTURE `cron create` calls only; existing jobs keep their `deliver` field.

**Out of scope:**
- `cmd_edit` — leave unchanged. Editing a job without `--deliver` keeps the existing deliver field. (User confirmed.)
- Migration of existing `deliver=local` jobs — manual via `cron edit`.
- Multi-chat broadcast (deliver to whitelist[0..N]) — too aggressive, would surprise users.
- Webhook auto-routing — not in scope.
- Discord/Slack origin auto-routing — Telegram is the only platform with a runtime gateway today (per `gateway.platforms.{telegram}` in config.yaml).
- Renaming the `local` deliver target — keep semantic taxonomy intact.
- Refactoring the `Origin` struct — additive only.

</domain>

<decisions>
## Implementation Decisions

### Default-routing taxonomy

- **D-01:** **Use option A1 — set `origin` field, `deliver = "origin"`.** The CLI populates `job.origin = Some(Origin { platform: "telegram", chat_id: "<whitelist[0]>" })` and `job.deliver = "origin"` when the TG gateway is active and the whitelist has exactly 1 entry. This routes through the existing `origin` resolution path at `crates/ironhermes-gateway/src/runner.rs:803-807` (already extracts origin chat_id) AND through Phase 22.4.2.1 Plan 02's match-arm dispatch via `dispatch_delivery`. Reuses the locked Phase 22.4.2.1 D-07 taxonomy without introducing a new code path. Semantically clean: the job carries its origin, future `cron edit` works as expected, and the `Origin` struct is the same one the gateway uses for non-cron messages.

- **D-02:** **Detection signal: `~/.ironhermes/config.yaml::gateway.platforms.telegram.enabled == true` AND `whitelist` non-empty.** Loaded via the existing config loader (researcher locks the function — likely `Config::load()` from `crates/ironhermes-core/src/config.rs` per the existing pattern at `agent_wiring.rs:34`). The CLI process reads the config at `cmd_create` start (lazy, NOT at CLI process startup — keeps `hermes --help` and other no-config commands fast).

### Default-routing rules

- **D-03:** **Multi-chat whitelist (len > 1) → fall back to `local` AND `eprintln!` a hint.** Do NOT silently pick `whitelist[0]` when the user has multiple authorized chats — the wrong chat could receive private cron output. The hint goes to stderr (so it doesn't break stdout pipelines that read created-job JSON) with the exact text:

  ```
  hermes cron create: Telegram gateway has multiple authorized chats — defaulting to deliver=local.
                      Pass --deliver telegram:<chat_id> to route to a specific chat (whitelist: [7018949547, 12345678]).
  ```

  Print the actual whitelist in the hint so the user can copy a chat_id. Do NOT abort or fail the command — `local` is a valid default; the user just doesn't get auto-routing.

- **D-04:** **Explicit `--deliver` flag is ALWAYS respected.** When the user passes `--deliver local`, `--deliver telegram:<id>`, `--deliver origin` (uncommon for CLI but allowed), or `--deliver webhook:<url>`, the helper is NOT consulted and no eprintln fires. The default-computation only runs when the flag is absent.

- **D-05:** **TG section missing OR `enabled: false` → keep current `local` default.** No eprintln. Silent fallback to existing v2.0 behavior. This preserves the no-config and TG-disabled UX. **UAT/Test directions MUST call out the implication:** when a user enables TG in config.yaml AFTER existing cron jobs were created, the existing jobs are NOT modified — only future `cron create` calls auto-route. (User explicitly confirmed this requirement.)

### Edit-flow contract

- **D-06:** **`cmd_edit` is UNCHANGED.** `--deliver` flag passed → update the deliver field. `--deliver` flag absent → leave deliver field untouched. The default-computation helper is NOT invoked from `cmd_edit`. This preserves the principle of least surprise: editing a name or schedule shouldn't silently re-route deliveries.

### Helper API

- **D-07:** **New helper:** `pub fn telegram_default_origin(&self) -> Option<Origin>` on the `Config` struct (or a free function in `ironhermes-core::config` if that's the existing shape — researcher locks). Returns:
  - `None` when TG section missing, `enabled: false`, OR whitelist empty
  - `None` when whitelist len > 1 (with a paired side-effect: caller must invoke a sibling helper `telegram_multi_chat_hint(&self) -> Option<String>` or accept that the helper returns a structured `OriginDecision` enum — researcher chooses)
  - `Some(Origin { platform: "telegram".into(), chat_id: <whitelist[0].to_string()> })` when len == 1

  Recommended: one helper returns an enum:
  ```rust
  pub enum OriginDecision {
      None,                    // TG disabled or no whitelist
      Single(Origin),          // whitelist len == 1, auto-routed
      Multi { whitelist: Vec<String> },  // whitelist len > 1, caller emits hint
  }
  ```
  Researcher confirms or proposes a simpler API.

### Test discipline (mirrors Phase 22.4.2.1 D-12 / D-13)

- **D-08:** **Behavioral test in the crate that owns the fix.** New file: `crates/ironhermes-cli/tests/cron_default_deliver.rs`. Tests:
  - `tg_enabled_single_chat_routes_to_origin` — config has TG enabled + whitelist=[12345]; `cmd_create` (no `--deliver`) writes job with `deliver="origin"`, `origin.platform="telegram"`, `origin.chat_id="12345"`.
  - `tg_enabled_multi_chat_falls_back_to_local` — config has whitelist=[12345, 67890]; `cmd_create` writes `deliver="local"`, `origin=null`; stderr captured contains the hint substring.
  - `tg_disabled_falls_back_to_local` — config has `enabled: false`; `cmd_create` writes `deliver="local"`, `origin=null`; NO eprintln fires.
  - `tg_section_missing_falls_back_to_local` — config has no `gateway.platforms.telegram` section; `cmd_create` writes `deliver="local"`, `origin=null`; NO eprintln fires.
  - `explicit_deliver_flag_skips_helper` — config has TG enabled + single chat; user passes `--deliver local`; job written with `deliver="local"` and helper is NOT invoked (assert no eprintln, assert origin is null).

### Documentation surface (D-05 follow-through)

- **D-11:** **Update CLI help / docs.** The `--deliver` flag's help text in `cron.rs` (around `cron.rs:36-37` per Phase 22.4.2.1 CONTEXT) MUST be updated to mention the auto-routing default. Suggested:
  ```
  /// Delivery target for job output. When omitted: defaults to "origin" routing
  /// to the configured Telegram chat if the gateway has exactly one authorized
  /// chat in config.yaml's whitelist; otherwise defaults to "local". Pass
  /// "local", "origin", "telegram:<chat_id>", or "webhook:<url>" to override.
  #[arg(long)]
  deliver: Option<String>,  // changed from default_value="local" to Option<String>
  ```

  **Note:** This changes the field type from `String` (with default "local") to `Option<String>`. The plan's executor must handle the None case by invoking the helper. The default-value attribute is REMOVED. Researcher confirms whether other callers of `cmd_create` (programmatic, not CLI) rely on the default-value attribute — if so, fall back to keeping `default_value = "local"` and detecting the literal "local" string in `cmd_create` (less clean but backward-compatible).

- **D-12:** **UAT instructions in plan SUMMARY.** When the plan ships, the SUMMARY.md MUST include a "Live verification" subsection that explicitly states:
  - "Newly-enabled TG configs auto-route subsequent `cron create` calls only. Existing jobs in `~/.ironhermes/cron/jobs.json` are NOT migrated — fix individually with `hermes cron edit <id> --deliver telegram:<chat_id>`."
  - "Test the default by: (1) ensure `~/.ironhermes/config.yaml` has `gateway.platforms.telegram.enabled: true` and a single-entry whitelist, (2) run `hermes cron create --name 'test' --schedule '*/5 * * * *' --prompt 'echo hi'` (no `--deliver` flag), (3) inspect the resulting job in `~/.ironhermes/cron/jobs.json`: `deliver` MUST be `"origin"`, `origin.platform` MUST be `"telegram"`, `origin.chat_id` MUST match the whitelist entry."
  - "Multi-chat fallback test: temporarily add a second whitelist entry, run the same `cron create`, observe stderr hint, verify created job has `deliver="local"`, `origin=null`."

### Claude's Discretion

- Helper name (`telegram_default_origin` vs `default_origin_for_cron` vs `cron_default_origin`) — recommend `telegram_default_origin` since it's TG-specific today and may extend per-platform later.
- `OriginDecision` enum vs two-helper API — recommend the enum (one source of truth, testable as a unit, future-extensible if Discord/Slack join).
- Whether to move `Origin` struct's `chat_id` from `String` to `serde_json::Value` for typed chat IDs — out of scope; keep `String` (matches existing `Origin` shape).
- Whether to also auto-populate `origin.thread_id` from the gateway's "topic" config if any — researcher checks if `whitelist` entries are tuples or just chat IDs in the YAML. If just chat IDs, `thread_id` stays None.
- Whether `cmd_edit --deliver origin` (without specifying chat_id) should trigger the same helper for backward-compat ergonomics — recommend NO: `--deliver origin` requires `origin` to already be set on the job, otherwise error out. Don't conflate the create-time default with edit-time semantics. (Defer if user disagrees.)
- Whether to add an env var override (`IRONHERMES_CRON_DEFAULT_DELIVER`) for power users — recommend deferral, no signal yet.

<specifics>
## Specific Ideas

- **The user explicitly said "they were going to TG in the past."** This phase exists because Phase 22.4.2.1 Plan 02 was correct but the default routing was the silent regression. Researcher should NOT spend time hunting for a code-side regression in `cron create`'s default — the v2.0 default-value attribute (`default_value = "local"`) is the artifact, the user wants it changed.
- **Single-chat whitelist is the COMMON case** for the user's setup (`whitelist: [7018949547]`). The multi-chat fallback exists for safety, not as the primary path.
- **The eprintln hint MUST include the actual whitelist values.** Don't print just `"multiple chats configured"` — print the chat IDs so the user can copy/paste into `--deliver telegram:<id>`.
- **`Config::load()` may panic or error on missing file** — the helper must handle this: missing config = `OriginDecision::None`. No-config users see no eprintln, no auto-routing — same as v2.0 today.
- **`Origin.chat_id` is `String` everywhere** in the codebase (per `runner.rs:803-807` extraction). The whitelist entry (which is parsed from YAML as `i64` likely) must be `to_string()`'d before being stored as `Origin.chat_id`.
- **The new test file `cron_default_deliver.rs`** is a CLI-crate integration test — it can either invoke `cmd_create` programmatically (cleanest) or shell out to `cargo run --bin hermes`. Researcher recommends programmatic — fast, deterministic, no spawn overhead.
- **One regression test per fix is mandatory** — without it, a future refactor will silently break the default again. Five tests cover all four config edge cases plus the explicit-flag bypass.

</specifics>

<deferred>
## Deferred Ideas

- **Multi-chat broadcast** (deliver to whitelist[0..N]) — too aggressive, would surprise users. Defer until a user explicitly asks.
- **Per-platform default routing** (Discord, Slack) — only TG has a runtime gateway today. When other platforms ship, extend `OriginDecision` to take a platform parameter.
- **`cmd_edit` auto-routing on missing `--deliver`** — explicitly out of scope per D-06. Users editing a job without `--deliver` keep their existing deliver field.
- **Migration of existing `deliver=local` jobs** to the new default — out of scope. User fixes manually with `cron edit`.
- **Env var override (`IRONHERMES_CRON_DEFAULT_DELIVER`)** — power-user feature. No current signal.
- **Webhook auto-routing** — depends on a "configured webhook URL" concept that doesn't exist in `config.yaml::gateway.platforms` today.
- **Renaming the `local` deliver target** — keep semantic taxonomy intact.
- **Threading `--dry-run` through `cmd_create`** to preview the computed default — useful but separate UX phase.
- **Auto-creating the missing TG section in config.yaml** when the user runs `hermes cron create` without TG enabled — out of scope; no auto-config-mutation in CLI commands.
