## Summary

Phase 21.4 performs a systematic gap analysis of IronHermes's memory subsystem against the hermes-agent reference. The codebase audit reveals five concrete gaps spanning two categories: (1) lifecycle hook wiring gaps where hooks exist in the trait/manager but are never invoked at runtime, and (2) config/CLI surface gaps where features exist in hermes-agent but are absent from IronHermes.

The good news: the core memory data path (add/replace/remove, frozen snapshot, capacity headers, security scanning, file locking) is fully implemented and correct. The MemoryProvider trait has all 11 hooks. The MemoryManager correctly fans writes to the mirror. MEM-06 (frozen snapshot) is already correctly implemented — `load_memory` is called exactly once per session in all three entry points.

The gaps are in the wiring layer: hooks that exist on the trait are not being invoked because the `MemoryManager` handle is not passed through to the components that fire them (AgentLoop for `queue_prefetch`, context engine builders for `on_pre_compress`). The config schema and CLI are also missing the `memory_enabled`/`user_profile_enabled` toggles and `memory status`/`memory off` subcommands.

**Primary recommendation:** Produce GAP-ANALYSIS.md as Wave 0, then fix the 5 gaps in a single plan wave (they are independent and low-risk changes). All fixes are additive — no existing API breaks.

## Gap Catalogue

This is the primary research output. Each gap maps to a concrete fix.

### GAP-1 (Critical): `queue_prefetch` never fires in CLI chat mode

**What:** `AgentLoop` has `with_memory_manager()` and fires `queue_prefetch` on natural-end break (agent_loop.rs:542-564). But `run_agent_turn` (main.rs:950-1040) constructs `AgentLoop` without calling `.with_memory_manager(memory_manager)`.

**Evidence:** [VERIFIED: crates/ironhermes-cli/src/main.rs:974-993] — `AgentLoop::new(...)` builder chain in `run_agent_turn` has no `.with_memory_manager(...)` call. `memory_manager` is in scope in `run_chat` (line 536) and passed to `prompt_builder` but never passed down into `run_agent_turn`'s signature or into the `AgentLoop` it builds.

**hermes-agent behavior:** Prefetch before each turn (behavior #2 of 6).

**Fix:** Add `memory_manager: Arc<tokio::sync::Mutex<MemoryManager>>` parameter to `run_agent_turn`, then call `.with_memory_manager(memory_manager.clone())` on the `AgentLoop` builder chain. Wire it at all three call sites in `run_chat`.

**Test:** Static-grep regression test asserting `run_agent_turn` signature contains `memory_manager` parameter. Behavioral test: mock provider records `queue_prefetch` calls, verify 1 call after natural-end break.

### GAP-2 (Critical): `on_pre_compress` never fires in any code path

**What:** Both context engine types (`ContextEngine` at context_engine.rs:80-173, `SummarizingEngine` at summarizing_engine.rs:125-270) have `with_memory_manager()` and fire `on_pre_compress` before destructive compression. But `build_context_engine` (engine_factory.rs) does not receive a memory_manager and thus never calls `.with_memory_manager(...)` on the engines it builds. `attach_context_engine` (agent_wiring.rs:43-68) also does not accept or forward a memory_manager.

**Evidence:** [VERIFIED: crates/ironhermes-agent/src/engine_factory.rs] — no `memory_manager` parameter, no `with_memory_manager` call. [VERIFIED: crates/ironhermes-agent/src/agent_wiring.rs:43-68] — `attach_context_engine` signature has no memory_manager parameter.

**hermes-agent behavior:** The reference does not explicitly list `on_pre_compress` as one of the 6 automatic behaviors, but it corresponds to PRMT-16 ("Memory is flushed to disk before compression to prevent data loss") — a safety requirement.

**Fix:** Add `memory_manager: Option<Arc<tokio::sync::Mutex<MemoryManager>>>` to `attach_context_engine` signature. Pass it to `build_context_engine`. In `build_context_engine`, call `.with_memory_manager(mgr)` on the engine before returning. Update all callers of `attach_context_engine` to pass `Some(memory_manager.clone())` where available, `None` for cron/subagent paths.

**Test:** Static-grep regression asserting `attach_context_engine` passes memory_manager to engine. Mock provider records `on_pre_compress` calls, verify they fire when compression threshold is exceeded.

### GAP-3 (Critical): `queue_prefetch` never fires in gateway mode

**What:** gateway/handler.rs constructs `AgentLoop` (line 469) without `.with_memory_manager(...)`. The `memory_manager` is in scope on `self` (field set at line 56) and is passed to `prompt_builder` (line 382-388) but not to the `AgentLoop`.

**Evidence:** [VERIFIED: crates/ironhermes-gateway/src/handler.rs:469-498] — `AgentLoop::new(...)` builder chain has no `.with_memory_manager(...)`.

**Fix:** In `handle_agent_request` (or wherever the `AgentLoop` is constructed in handler.rs), add `.with_memory_manager(self.memory_manager.clone().unwrap_or_else(...))` or conditionally wire it when `self.memory_manager.is_some()`.

**Test:** Gateway handler integration test verifying `agent.has_memory_manager()` when memory_manager is set on the handler.

### GAP-4 (Major): `memory_enabled` and `user_profile_enabled` config toggles missing

**What:** `MemoryConfig` (config.rs:136-153) has only `provider: String` and `mirror_provider: Option<String>`. The hermes-agent reference config shows:

```yaml
memory:
  memory_enabled: true
  user_profile_enabled: true
  memory_char_limit: 2200
  user_char_limit: 1375
```

IronHermes has no `memory_enabled` or `user_profile_enabled` fields.

**Evidence:** [VERIFIED: crates/ironhermes-core/src/config.rs:136-153] — `MemoryConfig` struct definition confirmed.

**Fix (D-07/D-08):**
1. Add `memory_enabled: bool` (default `true`) and `user_profile_enabled: bool` (default `true`) to `MemoryConfig` with `#[serde(default = "...")]`.
2. In `build_memory_manager` (factory.rs): early-return a no-op if `!config.memory_enabled` — return `None` or a disabled sentinel (exact shape TBD by planner per Claude's Discretion).
3. In `load_memory` (prompt_builder.rs): skip `format_for_system_prompt(MemoryTarget::User)` when `user_profile_enabled=false`. Requires passing the config flags to the builder (or checking them in the manager).
4. In `MemoryTool`: reject writes to `MemoryTarget::User` with a clear error when `user_profile_enabled=false`.

**Note on "disabled sentinel" shape:** The cleanest approach is to make `build_memory_manager` return `Option<Arc<...>>` and propagate `None` through all consumers (prompt_builder skips if `None`, tool rejects if `None`). This is backward-compatible since all consumers already guard on `if let Some(ref mgr)`. The planner should confirm this approach.

**Test:** Config round-trip test for new fields. `build_memory_manager` returns None when `memory_enabled=false`. Prompt builder produces no memory slot when manager is None. Memory tool returns disabled-target error.

### on_session_end — Partial Gap (Minor/Info)

**What:** `MemoryManager::on_session_end` exists and is correctly implemented. In `run_chat`, there is a comment at line 873: "on_session_end requires MemoryEntries; skip with debug log" — this is in the `ExitCleanly` ctrl-c path only, where the session exits before the turn completes and no `MemoryEntries` snapshot is readily available.

**Gap scope:** `on_session_end` is NOT called on clean session termination in any code path. The gateway runner's session expiry and the CLI's clean exit both skip it.

**Severity:** Minor — the file provider's `on_session_end` is a no-op anyway, and the data is persisted immediately on each write. External providers may rely on this for fact-extraction (behavior #4 of 6 in hermes-agent). This is a gap but lower priority than the critical wiring gaps above.

**Fix option:** Call `on_session_end` in the clean-exit path of `run_chat` and in the gateway session-end path. Requires capturing `to_memory_entries()` before exit. The `run_single` path naturally ends and could call it after `agent.run()` completes.

## Common Pitfalls

### Pitfall 1: Passing `memory_manager` to `run_agent_turn` vs fresh manager per turn

**What goes wrong:** If a fresh `build_memory_manager` is called inside `run_agent_turn`, each turn gets a different provider instance with a different snapshot. The frozen-snapshot contract breaks.

**Why it happens:** The `MemoryStore` snapshot is captured at `load_from_disk()`. Building a new provider re-reads disk and re-captures a snapshot that may include mid-session writes — violating MEM-06.

**How to avoid:** Pass the session-scoped `memory_manager` (constructed once at session start) down into `run_agent_turn` as a parameter. Never construct a new manager inside the turn loop.

### Pitfall 2: `attach_context_engine` signature change is a cross-crate break

**What goes wrong:** `attach_context_engine` is called from `main.rs` (CLI) and `handler.rs` (gateway). Adding a `memory_manager` parameter breaks both callers.

**How to avoid:** Either (a) make the parameter `Option<Arc<...>>` with callers passing `Some(mgr)` or `None`, or (b) add a separate `attach_context_engine_with_memory` variant. Option (a) is cleaner. Both callers already have the manager in scope, so they can pass `Some(memory_manager.clone())`.

### Pitfall 3: Disabling memory globally via `memory_enabled=false` creates None-valued Arc

**What goes wrong:** All consumers currently assume `memory_manager` is `Arc<Mutex<MemoryManager>>` (non-optional). If `build_memory_manager` returns `None`, all downstream code needs updating.

**How to avoid:** Change the factory return type to `Option<Arc<...>>` and update all consumers to guard with `if let Some(ref mgr)`. Most consumers already pattern-match on the optional field in `PromptBuilder` and `AgentLoop`, so the change is mechanical. The main.rs call sites need `let memory_manager = build_memory_manager(...).await?` updated to handle `None`.

### Pitfall 4: `hermes memory off` must not corrupt config keys added in Phase 21.4

**What goes wrong:** If `memory off` saves a config with only the legacy fields (`provider`, `mirror_provider`), the new `memory_enabled` / `user_profile_enabled` fields get silently dropped on save.

**How to avoid:** Load config first (`Config::load()`), mutate only `memory.provider` and `memory.mirror_provider`, then save. Since `Config` has `#[serde(default)]` on all fields, a round-trip preserves unknown fields.

### How to wire memory_manager to context engine (fix for GAP-2)

```rust
// In attach_context_engine (agent_wiring.rs):
pub fn attach_context_engine(
    agent: AgentLoop,
    config: &Config,
    resolver: &ProviderResolver,
    session_id: impl Into<String>,
    hooks: Option<Arc<HookRegistry>>,
    tracker: Option<Arc<PressureTracker>>,
    context_length: usize,
    memory_manager: Option<Arc<TokioMutex<MemoryManager>>>,  // ADD
) -> AgentLoop {
    let engine = build_context_engine(
        config,
        &config.agent.context_engine,
        resolver,
        context_length,
        config.agent.compression_threshold,
        sid.clone(),
        hooks,
        Some(tracker.clone()),
    );
    // After building engine, attach memory_manager:
    let engine = if let Some(mgr) = memory_manager {
        engine.with_memory_manager(mgr)  // ADD
    } else {
        engine
    };
    agent
        .with_context_engine(engine, context_length)
        .with_pressure_tracker(tracker)
        .with_session_id(sid)
}
```

   - What we know: `MemoryManager::to_memory_entries()` returns the live entries. This is async. The ctrl-c exit path (main.rs:873) is inside a sync context.
   - What's unclear: Whether calling `to_memory_entries().await` in the ctrl-c handler is feasible or requires `tokio::runtime::Handle::current().block_on(...)`.
