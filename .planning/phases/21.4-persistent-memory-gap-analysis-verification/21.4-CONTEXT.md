<decisions>
## Implementation Decisions

### Gap analysis scope
- **D-01:** Primary comparison target is the REFERENCE-hermes-agent-memory.md document in the phase directory plus the user-provided description of hermes-agent's provider lifecycle (6 automatic behaviors: inject provider context, prefetch before turn, sync after response, extract on session end, mirror writes, add provider tools).
- **D-02:** MEM-06 (frozen snapshot in prompt assembly) is pulled into this phase's scope. It was previously mapped to Phase 15 but is a core memory behavior that belongs in the gap analysis. REQUIREMENTS.md traceability table should be updated to reflect the move.

### Gap resolution strategy
- **D-03:** Audit + fix in this phase. Produce a structured GAP-ANALYSIS.md report first (with severity ratings per gap), then create plans to close each gap with code changes and tests. Gaps too large to fix in this phase get documented as follow-up phases.
- **D-04:** Known suspected gap areas: (1) provider lifecycle hook wiring completeness across all code paths, (2) config/CLI surface parity with hermes-agent.

### Verification method
- **D-05:** Code-level audit (grep + read to verify wiring exists) plus automated tests (`cargo test` to confirm existing tests pass, new tests written for each fix). No manual UAT required — this is a library/infrastructure phase.
- **D-06:** GAP-ANALYSIS.md produced before any fixes. Structure: feature-by-feature comparison table with columns [Feature | hermes-agent behavior | IronHermes status | Gap severity | Resolution]. This becomes the verification baseline — after fixes, the report is updated with "CLOSED" status per gap.

### Config toggles
- **D-07:** Add `memory.memory_enabled` (bool, default `true`) and `memory.user_profile_enabled` (bool, default `true`) to config.yaml schema. When `memory_enabled` is false, the entire memory subsystem is skipped (no provider construction, no memory tool registration, no prompt injection). When `user_profile_enabled` is false, USER.md store is skipped but MEMORY.md still works.
- **D-08:** Toggles checked at the factory/wiring level — `build_memory_manager` in main.rs respects the flags. Prompt builder's `load_memory` skips the disabled target. Memory tool rejects writes to a disabled target with a clear error.

### CLI subcommands
- **D-09:** `hermes memory status` — shows: active provider name, provider availability, MEMORY.md size/capacity, USER.md size/capacity, entry counts, mirror status (if configured). Follows the `colored` crate output pattern established by `cron.rs` and `memory_setup.rs`.
- **D-10:** `hermes memory off` — sets `memory.provider` to `"file"` (built-in default) and disables any external provider. Does NOT set `memory_enabled=false` (that would disable all memory). This matches hermes-agent behavior where "off" means "no external provider, built-in still works."

### MEM-06: Frozen snapshot
- **D-11:** Memory snapshot is captured once during prompt assembly at session start (or session reset). The `load_memory` call in prompt_builder.rs already reads from the provider — verify this only fires once and is not re-read on subsequent turns. Mid-session `memory add/replace/remove` writes persist to disk immediately but the active system prompt is not mutated.
- **D-12:** Verify the frozen-snapshot contract holds across all three entry points: `run_chat`, `run_single`, `run_gateway`. Each should call `load_memory` exactly once at session start, not per-turn.

### Claude's Discretion
- Exact severity thresholds for gap classification (critical/major/minor/info)
- Order of gap fixes (may depend on dependency graph discovered during audit)
- Whether to split fixes into multiple plans or handle as a single plan (depends on gap count/complexity)
- Exact output format for `hermes memory status` (specific column widths, colors)
- Whether `hermes memory off` prompts for confirmation or acts immediately

<specifics>
## Specific Ideas

- The user provided a detailed description of hermes-agent's 6 automatic provider behaviors: (1) inject provider context into system prompt, (2) prefetch relevant memories before each turn, (3) sync conversation turns after each response, (4) extract memories on session end, (5) mirror built-in memory writes to external provider, (6) add provider-specific tools. Each of these maps to a MemoryProvider trait hook — the gap analysis should verify each is wired end-to-end.
- `hermes memory off` in hermes-agent disables the external provider but keeps built-in memory active. This is NOT the same as `memory_enabled=false` which would disable everything.
- GAP-ANALYSIS.md should be the first artifact — produced by code audit before any fixes. It becomes the living verification baseline as gaps are closed.

</specifics>
