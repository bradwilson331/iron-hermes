# Phase 21.4: Code Review Report

**Reviewed:** 2026-04-20T13:42:00Z
**Depth:** standard
**Files Reviewed:** 13
**Status:** issues_found

## Summary

This review covers the persistent memory subsystem wiring across CLI, gateway, and agent layers -- the files touched by Phase 21.4's GAP-1 through GAP-6 closure work plus the underlying memory factory, manager, prompt builder, and config types. The codebase is well-structured with thorough test coverage (static-grep invariant tests, round-trip persistence tests, mock mirror providers, etc.). The GAP closures are correctly wired end-to-end.

Five warnings were identified: a hardcoded compression config in the gateway handler that diverges from the user's config.yaml settings, a missing `on_session_end` call in the gateway path (GAP-6 incomplete), lock contention risk in `load_memory`, a ctrl-c exit path that bypasses session cleanup, and a test-only tempdir memory leak. Four informational items note code duplication, unsafe env var usage in tests, and minor consistency opportunities.

## Warnings

### WR-01: Gateway handler hardcodes compression config instead of reading from config

**File:** `crates/ironhermes-gateway/src/handler.rs:129-134`
**Issue:** `maybe_compress_gateway` constructs `ContextStats` with hardcoded `protect_first_n: 3` and `protect_last_tokens: 20_000.min(self.context_length / 4)` instead of reading from `self.config.compression.protect_first_n` and `self.config.compression.protect_last_tokens`. If a user changes these values in `config.yaml`, the gateway hygiene pass will ignore them while the agent-side compression respects them -- creating inconsistent behavior between the two compression paths.
**Fix:**
```rust
let stats = ContextStats {
    context_length: self.context_length,
    estimated_tokens: estimated,
    protect_first_n: self.config.compression.protect_first_n,
    protect_last_tokens: self.config.compression.protect_last_tokens
        .min(self.context_length / 4),
    compression_count: 0,
    prior_summary: None,
};
```

### WR-02: Gateway `run_agent` does not call `on_session_end` (GAP-6 incomplete for gateway)

**File:** `crates/ironhermes-gateway/src/handler.rs:315-564`
**Issue:** GAP-6 was closed for `run_single` (line 483-489 in main.rs) and `run_chat` (line 986-992 in main.rs), but the gateway `run_agent` method in handler.rs never calls `on_session_end` after the agent completes. For providers that rely on session-end hooks (e.g., flushing in-memory state to durable storage), gateway sessions will never trigger this lifecycle event. This is especially relevant because Telegram sessions are long-lived and the gateway lacks a natural "session end" signal -- but the per-request completion point (after `agent_result` is processed, around line 552) is the closest equivalent and should at minimum fire best-effort.
**Fix:**
```rust
// After line 552, before the final Ok(())
if let Some(ref mgr) = self.memory_manager {
    let mgr_lock = mgr.lock().await;
    let entries = ironhermes_core::memory_provider::MemoryEntries::default();
    if let Err(e) = mgr_lock.on_session_end(&session_id_str, &entries).await {
        tracing::debug!(error = %e, "on_session_end failed in gateway run_agent (best-effort)");
    }
}
```

### WR-03: `load_memory` holds MemoryManager tokio::Mutex lock across multiple awaits

**File:** `crates/ironhermes-agent/src/prompt_builder.rs:398-437`
**Issue:** The `load_memory` method acquires `mgr.lock().await` on line 403 and holds the guard across three separate `.await` calls (`format_for_system_prompt` at lines 405, 410, and `system_prompt_block` at line 418). While `tokio::sync::Mutex` is designed to be held across `.await` points, this means any other task trying to write to memory (via `handle_tool_call`) or read from it (via `prefetch`) will be blocked for the entire duration of the three sequential async calls. In the common case (prompt construction at session start), this is likely benign because no concurrent writes are happening yet. However, if `load_memory` is ever called mid-session or if providers have non-trivial `format_for_system_prompt` implementations (network calls, DB queries), this could cause noticeable contention.
**Fix:** Restructure to acquire the lock per-call rather than holding it across all three:
```rust
pub async fn load_memory(&mut self) {
    if self.skip_context_files { return; }
    let mem_content = if let Some(ref mgr) = self.memory_manager {
        let mut mem_parts = Vec::new();
        {
            let guard = mgr.lock().await;
            if let Some(block) = guard.format_for_system_prompt(MemoryTarget::Memory).await {
                mem_parts.push(block);
            }
        }
        if self.user_profile_enabled {
            let guard = mgr.lock().await;
            if let Some(block) = guard.format_for_system_prompt(MemoryTarget::User).await {
                mem_parts.push(block);
            }
        }
        // ... same pattern for system_prompt_block
```

### WR-04: `run_chat` ctrl-c ExitCleanly path bypasses `on_session_end` and memory flush

**File:** `crates/ironhermes-cli/src/main.rs:908-917`
**Issue:** The `ExitCleanly` branch in the `tokio::select!` loop (line 908) calls `std::process::exit(0)` which immediately terminates the process. This skips the `on_session_end` call at line 986-992 and any pending memory flushes. The comment on line 913 acknowledges this: "memory flush on interrupted-exit skipped -- no flush_to_disk API". However, `on_session_end` is the designed hook for providers to perform end-of-session cleanup, and it IS available. The `std::process::exit(0)` prevents it from running. Similarly, the `Err(ReadlineError::Interrupted)` ExitCleanly path at line 951-955 also calls `std::process::exit(0)`.
**Fix:** Instead of `std::process::exit(0)`, set a flag and break out of the loop to reach the normal cleanup path:
```rust
CtrlCDecision::ExitCleanly => {
    chat_cancel_token.cancel();
    println!("{}", "Goodbye!".dimmed());
    tui.cleanup_on_exit();
    let _ = state_store.end_session(&session_id, "interrupted");
    // Fall through to normal cleanup instead of process::exit
    break 'turn None;
}
```
Then handle the outer loop break to reach the existing `on_session_end` block.

### WR-05: Test helper `primary_file()` leaks tempdir via `std::mem::forget`

**File:** `crates/ironhermes-agent/src/memory/manager.rs:516-523`
**Issue:** The `primary_file()` test helper calls `std::mem::forget(tmp)` to prevent the `TempDir` from being dropped and deleted. This intentionally leaks the temporary directory on every test invocation. While this prevents use-after-free in tests, it accumulates leaked temp directories that are never cleaned up. Over many test runs (especially in CI), this can consume significant disk space.
**Fix:** Store the `TempDir` in a `Box::leak` or return it alongside the `SharedProvider` so it stays alive for the test duration but documents the intent more clearly. Alternatively, use `Arc` wrapping:
```rust
fn primary_file() -> (SharedProvider, Arc<tempfile::TempDir>) {
    let tmp = Arc::new(tempfile::TempDir::new().unwrap());
    let mem_dir = tmp.path().join("memories");
    let mut store = MemoryStore::new(mem_dir);
    store.load_from_disk().ok();
    (Arc::new(Mutex::new(store)), tmp)
}
```

## Info

### IN-01: Duplicated provider dispatch logic between `build_memory_provider` and `build_tokio_provider`

**File:** `crates/ironhermes-agent/src/memory/factory.rs:46-136` and `214-297`
**Issue:** The `build_memory_provider` function (lines 46-136) and `build_tokio_provider` function (lines 214-297) contain nearly identical match arms for each provider type (file, sqlite, duckdb, grafeo), differing only in the mutex flavor (`std::sync::Mutex` vs `tokio::sync::Mutex`). The code comments acknowledge this (lines 168-174) and note that Plan 20-02's Task 02 will migrate `build_memory_provider` to `tokio::sync::Mutex`, collapsing both into a single path. This is a known tech debt item, not a bug.
**Fix:** Tracked for Plan 20-02 Task 02. No immediate action needed.

### IN-02: `unsafe` `set_var` usage in test code

**File:** `crates/ironhermes-agent/src/memory/factory.rs:362`, `crates/ironhermes-agent/src/prompt_builder.rs:637`, `crates/ironhermes-cli/tests/chat_memory_persistence.rs:26`
**Issue:** Multiple test files use `unsafe { std::env::set_var(...) }` which is marked unsafe as of Rust 1.66 and will eventually be a hard error under the Rust 2024 edition. The tests use serialization locks (`env_lock()`, `ENV_MUTEX`) to prevent data races, which is the correct mitigation. The `unsafe` blocks are properly justified with comments. This is informational -- the current approach works but should be revisited when the codebase migrates to Rust 2024 edition.
**Fix:** When migrating to Rust 2024 edition, consider using `temp_env` crate or a test harness that provides isolated environments.

### IN-03: `memory_cmd.rs` uses `MemoryStore` directly instead of `MemoryManager`

**File:** `crates/ironhermes-cli/src/memory_cmd.rs:48-49`
**Issue:** The `handle_memory_status` function constructs a `MemoryStore` directly (`MemoryStore::new(hermes_home.clone())`) instead of going through the `MemoryManager`/factory path. This means the status display always reads from the file-based store regardless of the configured provider. If a user has `provider: sqlite` in their config, `hermes memory status` will show the file store's state, not the sqlite store's state. The `entries()` method on line 52 is also specific to `MemoryStore`.
**Fix:** Use `build_memory_manager` to construct the appropriate provider:
```rust
let mgr = ironhermes_agent::memory::factory::build_memory_manager(&config.memory)
    .await?;
if let Some(mgr) = mgr {
    let guard = mgr.lock().await;
    let entries = guard.to_memory_entries().await;
    // ... display entries
}
```

### IN-04: `run_single` and `run_chat` pass empty `MemoryEntries` to `on_session_end`

**File:** `crates/ironhermes-cli/src/main.rs:485-486` and `988-989`
**Issue:** Both `run_single` and `run_chat` call `on_session_end` with `MemoryEntries::default()` (an empty entries struct) rather than the actual session's memory entries. The comment says "providers use their own internal state" which is true for the current file-based provider, but external providers that rely on the `entries` parameter to reconcile state will receive empty data. This is a design limitation noted for awareness.
**Fix:** Before calling `on_session_end`, fetch current entries via `mgr_lock.to_memory_entries().await` and pass those instead of the default.
