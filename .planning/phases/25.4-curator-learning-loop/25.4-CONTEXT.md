## Phase Boundary

Land the **session-to-skill Curator + Learning Loop** subsystem on top of Phase 25.3's foundations (trajectory crate, Workspace newtype, RESERVED_ROLE_NAMES["curator"]). The phase delivers:

1. **Heuristic gate** — `tool_calls >= 3` AND at least one impactful tool (terminal, write_file, patch_file, mcp_call) AND `duration >= 30s`. Fast, no LLM cost.
2. **LLM educational-value scoring** — passing trajectories go through the Phase 26 cascade as the new `"curator"` RESERVED_ROLE; LLM returns 1-10 score; >=7 promotes to skill emission. Operator routes via `model.roles["curator"]` or `auxiliary.curator` (cheap model — Haiku/llama).
3. **Skill emission** — Curator authors a Phase 19 SKILL.md and writes it to `<workspace>/skills/<slug>/SKILL.md` (workspace-active) or `~/.ironhermes/skills/<slug>/SKILL.md` (global fallback). Frontmatter: `name`, `category`, `allowed_tools`, `env_requirements`. NO curator-only fields — must be indistinguishable from hand-authored skills.
4. **Two trigger paths:**
   - **Auto-fire** on `/new`, `/clear`, gateway idle-timeout — runs **heuristic gate only**, no LLM cost.
   - **On-demand** `/curate <session?>` slash + `hermes curate <session>` CLI — runs the **full pipeline** (heuristic + LLM scoring + skill emission). User can curate any session retroactively.
5. **Gateway idle-timeout sweep** — `cfg.gateway.session_idle_minutes` (default 30). Background tokio task in `GatewayRunner::start` sweeps `SessionStore`; for any `SessionKey` idle >= threshold, fires the heuristic curator on its trajectories. Sibling to the existing WAL checkpoint / cleanup / cron-tick tasks.
6. **`AnyClientCuratorHandle`** — port verbatim from Phase 25.2's `AnyClientSummarizationHandle` (any_client.rs:256). Add `CuratorClientHandle` trait in `ironhermes-core`; impl in `ironhermes-agent`.

**Out of scope (deferred or carried from 25.3):**
- RL training pipeline that consumes trajectories.jsonl (deferred — separate future phase).
- Daemon-based curator sweep across all surfaces (REPL + single-shot + Telegram). 25.4 only sweeps gateway via idle-timeout.
- Curator dedup against existing skills (embedding-search or substring-match) — adds vector-store dependency. Defer.
- Curator-only frontmatter extension (`source: curator`, `source_session_id`, `educational_score`) — kept out of v1 to keep curator skills indistinguishable from hand-authored. Reconsider only if curator generates high enough volume that operators need a `--curated` filter.
- Per-workspace skill registry isolation (skills runtime stays single-registry).

</domain>

## Implementation Decisions (carried from Phase 25.3 CONTEXT.md)

### Curator + Learning Loop (`tools/curator.py` Rust port — but see RESEARCH.md correction)

> NOTE: 25.3 RESEARCH.md (2026-05-03) discovered that the upstream Python `agent/curator.py` is a **skill-consolidation orchestrator**, NOT a session-to-skill creator. The IronHermes Curator is a **greenfield Rust design**. The CURATOR_PROMPT must be authored from scratch — there is no Python prompt to copy verbatim. /gsd-discuss-phase 25.4 should treat this as the load-bearing open decision.

- **D-C-1:** **Two triggers:** (a) end-of-session auto-fire on `/new`, `/clear`, gateway idle-timeout — runs **heuristic gate only** (no LLM cost); (b) on-demand `/curate <session?>` slash + `hermes curate <session>` CLI — runs the **full pipeline** (heuristic + LLM scoring + skill generation). User can curate any session retroactively.
- **D-C-2:** **Heuristic gate + LLM educational-value score >= 7.** Heuristic: `tool_calls >= 3` AND at least one impactful tool (terminal, write_file, patch_file, mcp_call) AND `duration >= 30s`. If passes, build trajectory summary + send through Phase 26 cascade as a NEW RESERVED_ROLE `"curator"`. LLM returns 1-10 score; >=7 promotes. Operator routes curator calls to a cheap model (Haiku/llama) via `model.roles["curator"]` or `auxiliary.curator` — same cascade as `summarization` (Phase 25.2 D-13).
- **D-C-3:** **Generated skills use the existing Phase 19 SKILL.md format verbatim.** Curator is just another skill author. Output to `<workspace>/skills/<slug>/SKILL.md` (project-local) when a Workspace is active; falls back to `~/.ironhermes/skills/<slug>/SKILL.md` (global) otherwise. Frontmatter: `name`, `category`, `allowed_tools`, `env_requirements`. No curator-only fields. Clean integration with Phase 21.8 install/update/uninstall.
- **D-C-4:** **Gateway end-of-session = idle-timeout cutoff.** New `cfg.gateway.session_idle_minutes` config (default 30). Background tokio task in `GatewayRunner::start` sweeps `SessionStore` periodically; for any `SessionKey` idle >= threshold, fires the heuristic curator on its trajectories. Gateway users get automatic curation without manual `/curate`. CLI `/new` and `/clear` keep their per-session triggers.

### Claude's Discretion (curator-related, carried from 25.3)

- **CURATOR_PROMPT authorship** — 25.3 RESEARCH.md confirms there is no upstream prompt to copy. /gsd-discuss-phase 25.4 should propose the prompt: must explain the heuristic criteria that already passed, accept a trajectory summary, return JSON `{"score": N, "reason": "..."}`, define "educational value" as "would a developer benefit from a SKILL.md documenting this workflow?". Lock-test (byte-equality) is written AFTER the prompt is authored.

- **Curator concurrency** — `Arc<Semaphore>(1)` to serialize curator LLM calls so cost is bounded and predictable when multiple sessions reach idle-timeout simultaneously. Recommended in 25.3 CONTEXT.md; 25.4 planner should confirm.

### Established Patterns

- **`AnyClient*Handle` trait pair** — trait defined in `ironhermes-core` (cycle break), impl in `ironhermes-agent` over `ProviderResolver`.
- **`tokio::select!` background task with `interval.tick()` + `cancel.cancelled()`** — standard sweep shape from existing GatewayRunner siblings.
- **Cross-surface wireup parity tax** — every new field threaded through `CommandContext` requires updates in `main.rs::build_cmd_ctx`, `tui_rata::build_command_context`, AND `gateway::handler::handle_command` (via `GatewayRunner::set_*` plumbing). CuratorHandle pays this.

### Integration Points

- **Curator dispatch** — slash command `/curate` lives in `commands/handlers.rs`. CLI subcommand `hermes curate <session>` is a separate `Subcommand` arm in `cli_args.rs`. Idle-timeout sweep is a `tokio::spawn` task in `GatewayRunner::start` sibling to existing WAL/cleanup/cron tasks.
- **Skill emission** — Curator writes to `<workspace>/skills/<slug>/SKILL.md` (when `Workspace` is `Some`) or `~/.ironhermes/skills/<slug>/SKILL.md` (global) and registers via the existing Phase 19 `SkillRegistry` path. Skills runtime is single-registry; the workspace-vs-global decision is purely about WRITE destination, not READ resolution.
- **Trajectory consumption** — Curator reads via `TrajectoryReader` (provided by 25.3 ironhermes-trajectory crate). For a given session_id, returns the `Vec<TrajectoryEntry>` for heuristic gate evaluation and summary construction.
