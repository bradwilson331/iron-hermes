# Phase 07.1: Skill System Gap Analysis — Context

**Gathered:** 2026-04-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Produce a comprehensive gap-analysis document (`07.1-GAP-REPORT.md`) that compares the ironhermes skills system shipped in Phase 7 against the full Hermes Skills System specification — the agentskills.io open standard, Python `hermes-agent`'s reference implementation (~6,136 LOC across 5 modules), and the self-improvement requirements (SELF-08, SELF-10) currently parked in v2.

**This phase produces documentation, not code.** It does, however, append new `SKILL-05..SKILL-NN` entries to `.planning/REQUIREMENTS.md` so that subsequent phases can claim them via `/gsd-plan-milestone-gaps` or `/gsd-insert-phase`.

**Output informs:** roadmap follow-on phases for v1.1 and v2 of the skills system.

**Out of scope:** Implementing any of the identified gaps. Modifying Phase 7 code. Changing `.planning/ROADMAP.md` phase list (beyond what the new REQ-IDs imply for future planning).

</domain>

<decisions>
## Implementation Decisions

### Deliverable Shape
- **D-01:** Audit-only phase. The only code changes permitted are (a) writing `07.1-GAP-REPORT.md` and (b) appending new `SKILL-*` entries to `.planning/REQUIREMENTS.md`. No Rust source files are touched.
- **D-02:** Document path: `.planning/phases/07.1-skill-system-gap-analysis-does-current-code-cover-hermes-ski/07.1-GAP-REPORT.md`.

### Audit Scope (all four capability groups in-scope)
- **D-03:** **Spec compliance basics** — Audit coverage of: `platforms` filter, extended frontmatter fields (`compatibility`, `allowed-tools`, `metadata`, `metadata.hermes.*`), SKILL.md validation rules (name regex, description length 1–1024, directory-matches-name), directory depth limits, name-collision precedence handling.
- **D-04:** **Configuration & env vars** — Audit coverage of: `SkillsConfig` block in `config.yaml` for custom scan paths, external/configurable skill directories (beyond the three hard-coded paths from D-02 of Phase 7), secure env-var setup flow for skills that need API keys, `metadata.hermes.config` prompt mechanism (skill declares required config keys, hermes prompts user at first use).
- **D-05:** **Skills Hub & lifecycle** — Audit coverage of: multi-source hub (git repos, local paths, remote tarballs), update lifecycle (fetch / refresh / diff / install / remove — Python `skills_sync.py`, `skills_hub.py`), trust levels (verified / community / local), hub state tracking (`.hub/` directory equivalent).
- **D-06:** **Security & agent-managed skills** — Audit coverage of: security scanning of skill content (Python `skills_guard.py` — prompt injection detection, malicious pattern detection, allow/deny lists), `skill_manage` tool (agent creates/edits/deletes its own skills — SELF-08/SELF-10 territory), slash-command integration (skills invoked via `/skill-name` in chat, not just tool calls).
- **D-07:** **Self-improvement capabilities from REQUIREMENTS.md v2** are in scope as part of the parity bar: SELF-08 (procedural knowledge stored in SKILL.md directories) and SELF-10 (agent saves multi-step procedures as reusable skills after 5+ tool calls).

### Parity Bar & Source of Truth
- **D-08:** Full parity bar — audit against the union of: (1) agentskills.io open standard, (2) Python `hermes-agent` reference (`~/code/hermes-agent/tools/skill_manager_tool.py`, `skills_guard.py`, `skills_hub.py`, `skills_sync.py`, `skills_tool.py`), (3) ironhermes REQUIREMENTS.md v2 self-improvement items (SELF-08, SELF-10).
- **D-09:** **Divergence resolution: open standard wins, but ironhermes must remain backward-compatible with hermes-agent skills.** When agentskills.io and Python `hermes-agent` specify different field names, layouts, or behaviors, the audit records the gap against the **agentskills.io** version and flags the hermes-agent extension as "extension, parse leniently". The rule for downstream phases: parsers must tolerate hermes-agent skill files without erroring, even if hermes-specific fields are ignored.
- **D-10:** Phase 7's CONTEXT.md (`D-01`..`D-09`) is the existing locked state. The audit does not revisit those decisions — it only identifies what was out of scope for Phase 7 and what should be closed in follow-ons.

### Evidence Standard
- **D-11:** **File:line citations + test coverage** for every gap. For each audited capability, the report records: (a) "present" / "partial" / "absent", (b) file path and line range in current ironhermes code where the feature lives (or "absent"), (c) which tests exercise it (from `cargo test` output or `grep #\[test\]`), (d) the delta vs. the spec.
- **D-12:** The audit MUST actually run `grep`, read files, and run `cargo test` to produce evidence. No assumption-based claims. If a finding cannot be backed by a citation, it is marked "unverified" and the gap is recorded as "needs investigation".

### Report Structure
- **D-13:** **Capability matrix + per-gap dossier** format. Structure:
  1. Header: phase metadata, summary counts (N capabilities audited, M gaps found, breakdown by severity).
  2. Capability matrix: one table row per audited capability with columns `Capability | Status (present/partial/absent) | Severity | Effort | Milestone | Recommended REQ-ID | Recommended Phase`.
  3. Per-gap dossier: one `###` section per gap, containing:
     - **Spec quote** — exact text from agentskills.io spec or pointer to Python file:line.
     - **Current state** — ironhermes file:line evidence (or "absent").
     - **Test coverage** — which tests (if any) exercise the feature.
     - **Delta** — what's missing, precisely.
     - **Recommended scope** — size estimate, proposed REQ-ID, proposed phase shape.
  4. Summary appendix: recommended roadmap additions grouped by milestone (v1.1 / v2 / backlog).

### Scoring
- **D-14:** Each gap receives three orthogonal scores:
  - **Severity:** `BLOCKER | HIGH | MEDIUM | LOW`
    - BLOCKER = breaks agentskills.io compliance or hermes-agent backward compat
    - HIGH = user-visible capability gap with workflow impact (e.g., no update lifecycle)
    - MEDIUM = nice-to-have parity feature
    - LOW = cosmetic or deep internal
  - **Effort:** `S | M | L | XL`, derived from Python LOC proxy: S < 200 LOC, M 200–800, L 800–2000, XL > 2000. Effort is advisory, not binding.
  - **Milestone:** `v1.1 | v2 | backlog`, derived from severity + strategic value. Matches the existing v1.1/v2 split in REQUIREMENTS.md. BLOCKER and most HIGH → v1.1. Self-improvement features (SELF-08/10) → v2. LOW → backlog.

### REQUIREMENTS.md Mutation
- **D-15:** For each confirmed gap, mint a new `SKILL-05`, `SKILL-06`, ... requirement entry and **append it** to the appropriate section of `.planning/REQUIREMENTS.md`:
  - BLOCKER / HIGH → `v1.1 Requirements` → `Skills System` section.
  - MEDIUM / nice-to-have → end of `v1.1 Requirements` → `Skills System` section with a note.
  - Self-improvement parity → `v2 Requirements` → `Self-Improvement (Advanced)` section (extending SELF-08/SELF-10).
  - Also append to the phase-mapping table at the bottom of REQUIREMENTS.md with `Pending` status.
- **D-16:** Do **not** call `/gsd-insert-phase` or otherwise mutate `.planning/ROADMAP.md` from within this phase. Roadmap expansion is a separate, human-reviewed step after the report lands.

### Phase 7 Re-verification
- **D-17:** The audit does **not** re-verify Phase 7's 10/10 pass. Phase 7's VERIFICATION.md is authoritative for the capabilities it claims. The audit only records: "Phase 7 covered SKILL-01..SKILL-04 and delivered X, Y, Z" — then focuses entirely on what is *outside* that scope.

### Reuse of Existing Research
- **D-18:** The audit reuses and cites `.planning/research/SKILLS.md` (the prior research doc dated 2026-04-08) as a starting point for the agentskills.io spec summary, but cross-checks its claims against the actual Python source and the current ironhermes code. Where the two disagree, source code wins.

### Claude's Discretion
- Exact section ordering within the per-gap dossier (D-13 defines required fields, not strict order).
- Wording/phrasing of severity justifications.
- Whether to include a small "quick wins" highlight box at the top for gaps that are S-effort + HIGH-severity.
- How to format file:line evidence (e.g., backticks, blockquote, table cells) — pick what reads cleanly.
- Whether to run `cargo test --workspace` for full coverage numbers or rely on `grep #\[test\]` counts — pick whichever gives a faster, citable result.
- How to handle "duplicate" gaps where the same underlying feature shows up under two capability groups — mention once, cross-reference from the other.

</decisions>

<specifics>
## Specific Ideas

- The phase name itself enumerates the 14 capability areas — that list IS the audit checklist. The audit must address each of: progressive disclosure, SKILL.md format, platform-specific skills, conditional activation, secure env var setup, config settings, directory structure, external skill dirs, agent-managed skills via skill_manage, Skills Hub with multiple sources, security scanning, trust levels, update lifecycle, slash command integration.
- Progressive disclosure and SKILL.md format are already ✓ in Phase 7 per VERIFICATION.md — the audit should still mention them with "covered by Phase 7 (SKILL-01..04)" and move on.
- The "effort S/M/L/XL" proxy from Python LOC is a rough upper bound — Rust implementations are frequently shorter. The audit records both the Python LOC and a brief sanity-check note on whether the Rust port is likely smaller.
- The audit should flag any current Phase 7 code that would need to change (not just be added to) in order to support a gap — those carry "migration risk" and should be called out.
- Favor citing the **most specific** hermes-agent source (e.g., `skills_guard.py:128-190` for a prompt injection heuristic) over general module references. Downstream planners need to know exactly where to look.

</specifics>
