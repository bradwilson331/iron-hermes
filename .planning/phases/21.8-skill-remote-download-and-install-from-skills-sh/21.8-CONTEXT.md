# Phase 21.8: Skill Remote Download and Install from skills.sh — Context

**Gathered:** 2026-04-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Port the install/update path from the open-source `skills` CLI (TypeScript, at `/Users/twilson/code/skills`) into ironhermes' Rust skill framework, specifically targeting `https://skills.sh` as the remote source. Closes the **Skills Hub** clause of the v2.0 Active requirement *Skill framework* (PROJECT.md:41) — the current skill framework is local-only and needs a working remote fetch path.

**In scope (Phase 21.8):**
- `hermes skills install <name>` — pull from skills.sh blob API, validate, extract, update local lock
- `hermes skills list` — read installed skills from lock file + filesystem
- `hermes skills remove <name>` — remove + update lock
- `hermes skills update [<name>]` — diff local snapshot hash against skills.sh, re-install if changed
- `~/.ironhermes/skills-lock.json` v1 lock file (SHA-256 `computedHash` + `snapshotHash`)
- Mandatory security primitives: terminal escape stripping, YAML-only frontmatter, path traversal guards, pre-install audit endpoint

**Out of scope for this phase:**
- GitHub/GitLab/arbitrary-git source types — skills.sh-only for v1
- Multi-agent symlink fan-out — ironhermes is its own agent
- `.well-known/agent-skills` provider for arbitrary hosts
- Telemetry endpoint (`add-skill.vercel.sh/t`) — opt-out by default
- Publish flows (push to skills.sh / GitHub)
- Registering `ironhermes` agent type upstream in the skills CLI types.ts
- Hot-reload into live agent session (violates PRMT-02 cache-safety)

</domain>

<decisions>
## Implementation Decisions

### 19.1 skills.sh adapter fate
- **D-01:** Rip out + replace. Delete `SkillsShSource` entirely from `ironhermes-hub` (its `registry.json` assumption is wrong — skills.sh serves a blob API at `/api/download`, not a registry endpoint). `GitHubSource` + `WellKnownSkillSource` from Phase 19.1 stay as-is.

### Crate layout
- **D-02:** Extend `ironhermes-hub`. Add new modules `blob.rs`, `lock.rs`, `sanitize.rs` (and whatever else the planner decomposes) inside the existing `ironhermes-hub` crate. Reuse `HubSource` trait, `CoreSkillScanner`, and the install pipeline. No new workspace crate.

### CLI surface
- **D-03:** Keep `hermes skills install|list|remove|update` (plural). Consistent with `sessions`, `models`, `memory` in the existing CLI. The Phase 19.1 bare-name routing in `cmd_install` already targets skills-sh; we swap the implementation beneath.
- **D-04:** Rename `hermes skills uninstall` → `hermes skills remove`. Aligns with reference TypeScript CLI verb. Keep `uninstall` as a hidden alias for one release cycle so users transitioning from 19.1 aren't broken.

### Auth
- **D-05:** Reuse `GitHubAuth::resolve()` from Phase 19.1 for GitHub Trees API calls. Precedence: anonymous → `HERMES_GITHUB_TOKEN`/`GITHUB_TOKEN` env var → `gh auth token` subprocess fallback (matches D-03 from 19.1-CONTEXT.md).

### Install flow
- **D-06:** Mirror reference three-step flow exactly:
  1. Fetch tree SHA + file list via GitHub Trees API (`api.github.com/repos/.../git/trees/<ref>?recursive=1`)
  2. Pull frontmatter snippets from `raw.githubusercontent.com` to discover skill names
  3. Fetch full blob from `https://skills.sh/api/download/<owner>/<repo>/<slug>` (path-based, per reference `blob.ts:276`), where `<slug>` = `toSkillSlug(frontmatterName)`. Response is JSON `{files: [{path, contents}], hash}` — plain strings, **not a tarball**.
  - **Corrected 2026-04-22 (research O-1):** earlier draft used query-string form (`?repo=&path=&ref=`); reference TS uses path-based form with slug. Query-string form silent-404s.
- **D-07:** Base URL override via `SKILLS_DOWNLOAD_URL` env var (matches reference `src/blob.ts`).
- **D-08:** Per-HTTP-call fetch timeout = 10 seconds (matches reference `FETCH_TIMEOUT`).
- **D-09:** No git clone fallback. If blob API fails, hard-fail — do not introduce `git2` or shell out to `git`.

### Lock file
- **D-10:** File location: `~/.ironhermes/skills-lock.json` (under `HERMES_HOME`, same resolver used by other hub files).
- **D-11:** Port schema exactly from reference `local-lock.ts`:
  ```jsonc
  {
    "version": 1,
    "skills": [
      {
        "name": "...",
        "source": "skills-sh",
        "identifier": "...",
        "repoPath": "...",       // SKILL.md path inside the source repo
        "snapshotHash": "...",   // skillsComputedHash from /api/download response
        "computedHash": "...",   // SHA-256 over sorted folder contents
        "installedAt": "2026-04-22T..."
      }
    ]
  }
  ```
- **D-12:** `skills[]` sorted alphabetically by name; the hashed region of the file must be timestamp-free for merge-clean diffs. `installedAt` sits outside the hash calculation (metadata only).
- **D-13:** `computedHash` = SHA-256 over the installed folder, files sorted alphabetically by path. For each file, feed `hash.update(path)` then `hash.update(content)` with **no separator bytes** between or after (matches reference `local-lock.ts:108-114` exactly). Re-verified on `list` to detect tampering; mismatch surfaces as a warning but does not block listing.
  - **Corrected 2026-04-22 (research O-2):** earlier draft specified `\n` separator between path + content; reference TS uses no separators. `\n` would produce silent hash drift vs server-returned `skillsComputedHash`.
- **D-14:** `snapshotHash` = `skillsComputedHash` returned by `/api/download`. Used as the version key for `update` diff: local snapshot ≠ remote snapshot → re-install.
- **D-15:** Manifest migration: Phase 19.1's `~/.ironhermes/skills/.hub/*.json` per-skill manifest files are superseded by the new lock file. On first run of 21.8 code, migrate any existing hub manifests into the new lock file format, then delete the `.hub/` directory. Idempotent — safe to re-run.

### Security primitives (non-negotiable)
- **D-16:** Terminal escape stripping. Port `stripTerminalEscapes` from reference `sanitize.ts`. Strip CSI / OSC / DCS / PM / APC / simple ESC / C1 / raw control chars from every skills.sh-sourced string before terminal output (CWE-150 defense). Applied to: `SkillMeta.description`, `SkillMeta.name`, version strings, error messages relayed from the server.
- **D-17:** YAML-only frontmatter guard. Explicitly reject `---js` and any non-YAML frontmatter delimiter in fetched `SKILL.md` files. Current `parse_skill_md` uses `serde_yaml`; add an explicit delimiter check (`---` only) before handing content to the parser. Any skill with a non-YAML delimiter fails install with a typed error.
- **D-18:** Path traversal guards. Port `sanitizeName`, `sanitizeSubpath`, `isPathSafe`, `isContainedIn` from reference `sanitize.ts`. Validate every skill name + subpath from the remote manifest (not just what the user typed) before any filesystem write. Use Rust's `path-clean` crate or manual normalize + prefix check; never trust fetched paths.
- **D-19:** Pre-install audit endpoint. Port the `add-skill.vercel.sh/audit` query (matches reference behavior) to fetch partner-risk data pre-install. Display risk summary to user, proceed unless `--skip-audit` flag is passed. Audit failure (endpoint down / rate-limited) is soft-fail: log, proceed with install. Network-call-free bypass via `--skip-audit` for CI/air-gapped use.
- **D-20:** Temp dir containment. All extraction work happens under `std::env::temp_dir()`. Before any cleanup `remove_dir_all`, re-validate the temp path is contained within `temp_dir()` — defense against symlink swaps between extract + cleanup.

### UX
- **D-21:** Progress output: multi-line headlines on stdout + detail via `tracing::info!`:
  ```
  Resolving skills.sh/<name>...
  Discovering skills in <owner>/<repo>...
  Downloading <size> bytes...
  Scanning for threats...
  Installed '<name>' [community] — hash: <12-char>
  ```
  No spinners/progress bars for v1 (avoids `indicatif` dep). `--verbose` flag unlocks extra detail lines; default is the 5-line shape above.
- **D-22:** User-Agent header: `ironhermes/<ver> (via openclaw)` — behaves as `openclaw` agent type toward skills.sh for v1 (gets tailored agent metadata from server-side if any). Upstream PR adding `ironhermes` to the skills CLI `AgentType` union is deferred.

### Post-install activation
- **D-23:** Next-session-only. Install lands on disk at `~/.ironhermes/skills/<category>/<name>/`; skill becomes visible at next agent startup via `SkillRegistry::load_with_paths`. No hot-reload into the live `SkillRegistry` — the frozen-snapshot prompt pattern (Phase 15) and MEM-06 cache-safety contract forbid mid-session mutation. Output line on success: `Installed. Restart the agent or start a new session to use <name>.`

### Error handling
- **D-24:** Fail-fast with typed `HubError` + one transient retry. Retry exactly once on 5xx / connection timeout / DNS failure (applies to all three hops: Trees API, raw.githubusercontent, skills.sh). After retry exhaustion, hard fail with exit code 1 and a typed message. Typed error variants: `Network`, `NotFound`, `Parse`, `Sha Mismatch`, `ScanHit`, `PathTraversal`, `Audit`. Nothing is written to the skills dir on failure; quarantine is cleaned up.

### Remote behavior contract
- **D-25:** Atomic install pipeline: fetch → quarantine → scan → audit → atomic `rename` to final location → lock file update. Failed installs leave no partial state. Lock file write uses tmp + rename (re-use `save_config_atomic` pattern from `skills_cmd.rs`).

### Claude's Discretion
- Exact Rust type shapes for `ParsedSource`, `BlobSkill`, `SkillSnapshotFile`, `TreeEntry`, `RepoTree` — planner-drafted from the reference TypeScript types.
- Tar extraction strategy (`tempfile::tempdir` + `flate2::read::GzDecoder` + `tar::Archive`) — choose the idiomatic Rust composition.
- Module decomposition within `ironhermes-hub` (`blob.rs` vs splitting blob/trees/raw — planner choice).
- Port of `toSkillSlug(name)` — must match server-side exactly, verified via a golden-test vector against reference output.
- Test harness layout: reuse existing `wiremock` dev-dep for HTTP mocking; `tempfile` for filesystem fixtures.
- Whether `sanitize.rs` is a pub module or crate-private helper (planner call).

</decisions>

<specifics>
## Specific Ideas

- **Primary user story:** `hermes skills install ascii-art` should Just Work — bare name → skills.sh resolution → download → install → visible on next agent session. No interactive prompts by default.
- **The reference implementation is authoritative for behavior, not code.** Port semantics, not literal lines. Use idiomatic Rust (Result + typed errors, not exceptions; owned types + borrow where sensible; Iterator adaptors over imperative loops where it clarifies).
- **Lock file diff is merge-clean.** The hashed region (sorted `skills[]` with no timestamps in the hashed payload) means two developers installing different skills produce diffs that merge without conflict. `installedAt` is per-entry metadata, not inside the hashed region.
- **Security is table stakes, not a stretch goal.** D-16..D-20 all land in v1. The reference implementation treats skills.sh content as adversarial by default; ironhermes inherits that posture.
- **`toSkillSlug` must match exactly.** A single off-by-one in the slug function produces silent 404s against skills.sh. Golden test vector required: feed the reference implementation's canonical test cases, assert identical output.
- **`openclaw` agent ride (D-22) is temporary.** Capture a deferred idea: upstream PR adding `ironhermes` to the skills CLI `AgentType` union. Not blocking this phase; future phase can register properly.
- **The 19.1 SkillsShSource removal is a breaking change in `ironhermes-hub` public API.** That's acceptable — the broken adapter isn't used anywhere that ships to users yet (integration test only). Verify no call sites outside this crate before deleting.
- **Error messages reach the terminal.** Every `HubError::Typed.message` + `suggestion` must flow through `stripTerminalEscapes` before `eprintln!` (D-16 applies to server-originated error strings too, not just happy-path metadata).

</specifics>
