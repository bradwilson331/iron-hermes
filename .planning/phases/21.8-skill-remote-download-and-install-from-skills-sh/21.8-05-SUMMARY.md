# Phase 21.8 Plan 05: Wave 4 — End-to-End Integration Testing Summary

Wave 4 lands the final regression harness for phase 21.8. Three new `ironhermes-hub` integration-test files exercise the three-hop blob pipeline against wiremock (happy-path + retry/404/path-traversal + D-14 opaque hash round-trip + D-13 local tamper detection + D-22 User-Agent + D-07 URL override), plus audit-endpoint soft-fail semantics (timeout / 5xx / non-JSON / skip bypass) and idempotent 19.1 → 21.8 lock-file migration. One `ironhermes-cli` subprocess-level integration file drives the full `skills install` → `skills list` → `skills remove` → `skills uninstall` (alias) round-trip against a wiremock backend. SkillsShBlobSource prod constructor gains env-driven GITHUB_API_BASE + GITHUB_RAW_CONTENT_BASE overrides + an https_only relaxer for http-scheme overrides, so the subprocess tests can drive production code against wiremock without bypassing the blob adapter's normal construction path.

20 new integration tests; 1252/1254 full-workspace tests pass (2 remaining failures are pre-existing ironhermes-core model-metadata failures deferred per plan 21.8-04's deferred-items.md).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 — Missing critical functionality] SkillsShBlobSource::new not env-overrideable for all three hops**

- **Found during:** Task 2 subprocess round-trip test
- **Issue:** Plan calls for a full CLI subprocess round-trip test against wiremock. But `cmd_install` builds its source via `SkillsShBlobSource::new(gh.clone())` — the prod constructor — which hard-codes `github_api_base = "https://api.github.com"` and `raw_content_base = "https://raw.githubusercontent.com"` AND sets `https_only(true)`. Only `download_base_url` is D-07 env-overrideable. The subprocess test cannot re-target hops 1 and 2 at wiremock without either (a) changing the CLI to use the test constructor (wrong — test/prod divergence), (b) adding env overrides to the prod constructor, or (c) skipping the round-trip entirely (fails revision round 1 W5 option (a)).
- **Fix:** Added `GITHUB_API_BASE` and `GITHUB_RAW_CONTENT_BASE` env resolvers as siblings to the existing `SKILLS_DOWNLOAD_URL` (D-07) resolver. `SkillsShBlobSource::new` reads all three; if any returns an `http://` URL, https_only is relaxed. Production (default) URIs keep HTTPS-only enforcement. Subprocess integration tests set the three env vars to a single wiremock origin and the prod constructor just works.
- **Files modified:** `crates/ironhermes-hub/src/blob.rs`
- **Commit:** `28d5187`

**2. [Rule 1 — Bug] Plan references wrong CARGO_BIN_EXE env name**

- **Found during:** Task 2 writing the subprocess test
- **Issue:** Plan uses `env!("CARGO_BIN_EXE_hermes")` but the binary is named `ironhermes` per `crates/ironhermes-cli/Cargo.toml` `[[bin]] name = "ironhermes"`. Cargo populates `CARGO_BIN_EXE_{binary-name}`; `CARGO_BIN_EXE_hermes` would fail at compile-time with "environment variable not defined".
- **Fix:** Used `env!("CARGO_BIN_EXE_ironhermes")` (correct name).
- **Files modified:** `crates/ironhermes-cli/tests/skills_cmd_integration.rs`
- **Commit:** `28d5187`

**3. [Rule 3 — Blocking] Test identifiers didn't match tree fixture**

- **Found during:** Task 1 first test run (`no_retry_on_404` and `path_traversal_blocked` failed with NotFound at hop 1)
- **Issue:** Initial test identifiers `foo/bar/ghost-skill` and `foo/bar/sneaky` caused installer to look for `ghost-skill/SKILL.md` and `sneaky/SKILL.md` in the tree. The `sample_tree_json` fixture only contains `ascii-art/SKILL.md` → hop 1 returns NotFound before the intended error path (404 at hop 3, path traversal at hop 3) is reached.
- **Fix:** Aligned identifiers with the tree fixture: all tests now use `foo/bar/ascii-art`. The lock-entry assertion in `no_retry_on_404` was updated to check that `ascii-art` is NOT in the lock (install failed mid-pipeline).
- **Files modified:** `crates/ironhermes-hub/tests/skills_sh_blob_adapter.rs`
- **Commit:** `ac35d25`
