# Phase 21.8: Skill Remote Download and Install from skills.sh — Research

**Researched:** 2026-04-22
**Domain:** Rust port of a TypeScript CLI install/update pipeline; HTTPS blob fetch + atomic filesystem pipeline + merge-clean lockfile; adversarial-input hardening.
**Confidence:** HIGH for reference-TS ports (code is in-tree and readable). MEDIUM for skills.sh server contract (not independently verifiable; reference TS is the only authoritative source). HIGH for existing ironhermes Rust surface (all extend points were read).

## Summary

Phase 21.8 replaces the broken `SkillsShSource` registry adapter with a new blob-API adapter that mirrors the reference TypeScript CLI at `/Users/twilson/code/skills`. The reference exists, is readable, and is the authoritative source for every behavior. The Rust port is an exercise in re-expressing known code using `reqwest` + `serde` + `sha2` + `tar` + `regex` — no new crates are strictly required (even `path-clean` is unnecessary; `std::path::Component` handles it).

The meat of the work breaks down as follows:
1. **Delete** `crates/ironhermes-hub/src/skills_sh.rs` plus its `SkillsShSource::new(gh.clone())` call sites in `crates/ironhermes-cli/src/skills_cmd.rs:136` and `crates/ironhermes-tools/src/skills_tool.rs:382`; delete the `tests/skills_sh_adapter_test.rs` integration test (its `registry.json` assumption is incorrect).
2. **Create** three new modules inside `ironhermes-hub`: `blob.rs` (three-hop fetcher), `lock.rs` (`skills-lock.json` v1), `sanitize.rs` (terminal escape stripper + path guards + slug function + metadata cleaner).
3. **Extend** `HubError`/`HubErrorKind` with `ShaMismatch`, `ScanHit`, `PathTraversal`, `Audit` variants (Network/NotFound/Parse/Io already exist).
4. **Rework** `cmd_install` / `cmd_list` / `cmd_update` in `skills_cmd.rs` to read `SkillLock` instead of `HubManifest`; rename `Uninstall` → `Remove` with `uninstall` as a hidden clap alias.
5. **Write** a one-way migration from `.hub/lock.json` (19.1 `HubManifest`) into `~/.ironhermes/skills-lock.json` (21.8 `SkillLock`), triggered idempotently on first touch.
6. **Carry over** every decision from CONTEXT.md (D-01..D-25) without re-litigating.

**Primary recommendation:** Structure this phase as five waves — Wave 0 (test infra + sanitize.rs as pure functions), Wave 1 (blob.rs + lock.rs), Wave 2 (install pipeline rewrite in installer.rs), Wave 3 (CLI rework + migration), Wave 4 (integration test + end-to-end validation). Wave 0 and the sanitize.rs piece are the lowest-risk highest-leverage starting point because (a) the security primitives are pure functions, (b) golden-test vectors against the reference TS can be generated immediately, (c) everything downstream depends on their correctness.

**CRITICAL CORRECTION TO CONTEXT.md D-06 / D-07 / handoff.md:** The CONTEXT.md text says the blob URL is `https://skills.sh/api/download?repo=<owner>/<repo>&path=<subpath>&ref=<sha>`. **This is wrong.** The reference at `/Users/twilson/code/skills/src/blob.ts:276` uses a **path-based URL**: `${DOWNLOAD_BASE_URL}/api/download/${owner}/${repo}/${slug}` — no query string, no `ref`, no `path` parameter. The `slug` is `toSkillSlug(skill.name)` (derived from the SKILL.md frontmatter `name`, not the folder path). The planner must code to the real shape observed in blob.ts; it is the wire contract. This mismatch would have caused silent 404s in every install attempt. Flagged as O-1 in Open Questions — the CONTEXT.md wording should be treated as aspirational/incorrect; blob.ts is authoritative.

### Pattern 1: Three-Hop Retry Wrapper

**What:** A single retry wrapper function that applies to every HTTP call in the blob pipeline.

**When to use:** Any time we call `reqwest::get` in `blob.rs` or `audit.rs`. Keeps D-24 "exactly one retry on transient" logic in one place, not scattered per-call.

**Example (Rust sketch):**
```rust
// Source: derived from D-24 + reqwest docs [VERIFIED reqwest exposes status().is_server_error() and
// builder.timeout()]
async fn with_one_retry<F, Fut, T>(op: F) -> Result<T, HubError>
where
    F: Fn() -> Fut,
    Fut: std::future::Future<Output = Result<T, HubError>>,
{
    match op().await {
        Ok(v) => Ok(v),
        Err(e) if is_transient(&e) => {
            tracing::warn!("transient error, retrying once: {}", e);
            op().await  // one retry, no backoff
        }
        Err(e) => Err(e),
    }
}

fn is_transient(e: &HubError) -> bool {
    match e {
        HubError::Reqwest(re) => re.is_timeout() || re.is_connect() ||
            re.status().map_or(false, |s| s.is_server_error()),
        HubError::Typed { kind: HubErrorKind::Network, .. } => true,
        _ => false,
    }
}
```

### Pattern 3: Pure-Function Core + I/O Shell

**What:** All security primitives (`strip_terminal_escapes`, `sanitize_metadata`, `sanitize_name`, `sanitize_subpath`, `is_path_safe`, `is_contained_in`, `to_skill_slug`, `compute_folder_hash`) are **pure functions** with no network or filesystem access. I/O lives in `blob.rs`, `audit.rs`, `lock.rs`, and `installer.rs`.

**Why:** Pure functions are trivially testable (plain `#[test]`, no `tokio::test`, no wiremock, no tempfiles). Test suite stays fast; security invariants are easy to fuzz.

### Anti-Patterns to Avoid

- **Recomputing folder hash inside installer.rs at commit time.** The hash must be computed over the **written-to-disk** folder AFTER extraction completes, not from the in-memory `files[]` vector. Writing and then re-reading catches write-time file corruption. [VERIFIED: reference `local-lock.ts::computeSkillFolderHash` walks the directory with `readdir` after install, not before.]
- **Treating `installedAt` as a versionable field.** It MUST NOT participate in the hash. A developer re-installing the same skill on a different machine must produce a byte-identical `skills[]` hashed region so git merges cleanly (D-12). Sort + serialize without `installedAt` in the hash region; include it only in the serialized output.
- **Hardcoding `https://skills.sh` anywhere except `constants.rs`.** `SKILLS_DOWNLOAD_URL` override must work (D-07). Reference precedent at `blob.ts:44`: `const DOWNLOAD_BASE_URL = process.env.SKILLS_DOWNLOAD_URL || 'https://skills.sh';`.
- **Relying on YAML parser failures to catch `---js`.** `serde_yaml::from_str` on an empty or non-YAML-looking block may return `Ok(Value::Null)` or swallow input silently. The delimiter check must be **explicit string-level** before YAML parsing, not left to serde. See sanitize.rs sketch below.
- **Calling `remove_dir_all` on a temp path without re-validating containment.** D-20 is non-negotiable: compute `canonicalize()` on the temp path and assert `starts_with(env::temp_dir().canonicalize())` before removal. Symlinks can be swapped between extraction and cleanup on multi-user boxes.
- **Partial install state on failure.** If ANY step fails — audit, fetch, frontmatter check, path sanitization, scan, rename, lock write — NOTHING has landed in `skills_root`. The pipeline must operate on a quarantine dir and atomic-rename only after every gate passes.
- **Re-running the download inside the retry wrapper for errors that aren't transient.** `HubErrorKind::NotFound` (404 on the skill slug), `PathTraversal` (bad skill name), `ScanHit` (community skill with threat pattern) are permanent — retrying them just spams the server. `is_transient()` must be narrow.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| HTTP client with timeout + retry | custom `hyper` wrappers | Reuse `reqwest::Client` already built in `github.rs:71-80` with `.timeout(Duration::from_secs(10))` per call (D-08) | Rust HTTP ecosystem is mature; reimplementing means missing TLS/cookie/redirect details |
| Terminal escape stripping | custom char-by-char state machine | `regex` with 6 compiled patterns (sketch below) | CSI + OSC + DCS/PM/APC + simple ESC + C1 + control chars is a known surface; hand-rolling risks missing a class |
| Path traversal check | custom string ops | `std::path::Path::components()` with match on `Component::ParentDir` etc. | Already proven in `tarball.rs::validate_bundle_rel_path` — same exact pattern |
| SHA-256 streaming hasher | custom `Digest` impl | `sha2::Sha256` + `hex::encode` | Already used in `installer.rs:60` for `bundle_content_hash` — same idiom |
| YAML frontmatter parsing | custom lexer | Explicit delimiter check on `---\n` then `serde_yaml::from_str` | `parse_skill_md` in `ironhermes-core/src/skills.rs:192` already does this; extend, don't reimplement |
| Atomic file writes | rename-in-place logic | Existing `save_config_atomic` pattern in `skills_cmd.rs:103` (tmp + rename) | Proven under crash; symmetric with `HubManifest::save` in `manifest.rs:42` |
| Slug URL encoding | custom percent-encoding | `url::Url` already in workspace (used by well_known.rs) | `url` crate is a workspace dep; no new imports |
| HTTP request body JSON | manual string building | `reqwest::RequestBuilder::json()` + `resp.json::<T>()` | Already used extensively in github.rs |
| ISO-8601 timestamps | manual string format | `chrono::Utc::now().to_rfc3339()` | Already used in `ManifestEntry.installed_at`; matches reference TS `new Date().toISOString()` |

**Key insight:** The reference TypeScript pushes nearly every concern to standard Node APIs (`fetch`, `createHash`, `normalize`, `resolve`). The Rust port has direct equivalents in `reqwest`, `sha2`, `std::path::Path`. Zero new crates are required, and introducing `path-clean` would duplicate what `std::path::Component` already provides. The only reason to reach outside workspace deps is if a reviewer wants a stronger audit trail on the escape-stripping regex, in which case `strip-ansi-escapes` could be considered — but it doesn't cover DCS/PM/APC.

## Common Pitfalls

### Pitfall 1: Silent `toSkillSlug` drift from the server

**What goes wrong:** The server-side slug function (running on skills.sh) is the authoritative URL encoder. If our Rust `to_skill_slug` has a one-character difference — e.g., Unicode NFC vs NFD normalization, whitespace-class interpretation, or hyphen collapsing order — every install produces a silent 404. The user gets "not found" with no hint that the slug was miscomputed.

**Why it happens:** The reference TS implementation uses JavaScript regex semantics (`/[\s_]+/g`, `/[^a-z0-9-]/g`, `/-+/g`, `/^-|-$/g`) with JavaScript's default ASCII-first matching. Rust's `regex` crate is Unicode-aware by default — the `\s` class matches more characters (e.g., U+00A0) than JS's `\s`. This can produce different slugs for non-ASCII input.

**How to avoid:**
1. Use `regex::Regex::new(r"[\s_]+")` with `.unicode(false)` OR explicitly enumerate replacement chars.
2. Generate a golden-vector test using the actual reference implementation (run `node -e "import('./src/blob.ts').then(m => console.log(m.toSkillSlug('<input>')))"` — or use `bun src/blob.ts`), capture 20+ inputs covering ASCII, Unicode spaces, emoji, mixed case, leading/trailing dashes, and assert bit-identical output.
3. **Pin the exact algorithm**: `name.to_lowercase()` → replace `[\s_]+` with `-` → remove `[^a-z0-9-]` → collapse `-+` to `-` → trim leading/trailing `-`.

**Warning signs:** Install fails with `HubErrorKind::NotFound` on a skill that appears in the skills.sh UI; `curl https://skills.sh/api/download/<owner>/<repo>/<manual-slug>` succeeds with the manually-computed slug but fails with our slug.

### Pitfall 2: Lock file hash includes `installedAt` by accident

**What goes wrong:** Developer A installs skill X on Monday. Developer B installs skill X on Tuesday. Both commit `skills-lock.json`. Git merge produces a conflict on the timestamp despite identical content.

**Why it happens:** Serializing the full `SkillLockEntry` (including `installedAt`) into the hash input, or into the on-disk JSON sort order that feeds the hash.

**How to avoid:** Keep `installedAt` **outside the hash calculation**. The `computedHash` is derived from the skill folder on disk (recursively sorted files with `path + "\n" + content + "\n"` — see Hash Algorithm section below), NOT from the lock file contents. The lock file just *records* the hash. The merge-cleanliness comes from sorting `skills[]` alphabetically by name before serialization (D-12) and using stable `serde_json::to_string_pretty`. `installedAt` is a per-entry string that changes — but since entries don't overlap across branches (different skills installed on different branches), git sees non-overlapping JSON object keys and merges cleanly.

**Warning signs:** `git merge develop` produces conflicts in `skills-lock.json` even when two branches add entirely different skills.

### Pitfall 3: Partial install after mid-pipeline failure

**What goes wrong:** Network succeeds (files fetched), audit succeeds, frontmatter parses OK, but the scan step blocks (community + scan-hit). The quarantine tempdir remains on disk; the lock file was not updated; BUT some code path already touched `skills_root/<category>/<name>/` because the atomic rename was done before the scan.

**Why it happens:** Reordering the pipeline steps during a "refactor to simplify" pass. The reference pipeline order is non-negotiable: **fetch → quarantine → scan → audit → atomic rename → lock update**. Any reordering (e.g., moving rename before scan, or lock-write before rename) breaks atomicity.

**How to avoid:** Encode the order as a linear function call chain in `installer.rs::install`, not as a set of independent steps. Add a comment: `// PIPELINE ORDER LOCKED: fetch → quarantine → scan → audit → rename → lock (D-25).`

**Warning signs:** After a failed install, `ls ~/.ironhermes/skills/` shows the partially-installed folder.

### Pitfall 4: Symlink swap between quarantine write and cleanup

**What goes wrong:** On a multi-user machine, an attacker races the install pipeline: right after our `tempfile::tempdir()` creates `/tmp/skills-abc123/`, the attacker symlinks `/tmp/skills-abc123/` to `/home/user/.ssh/`. On install failure, our code calls `remove_dir_all(tempdir_path)` — which follows the symlink and deletes `~/.ssh`.

**Why it happens:** `tempfile::TempDir::path()` returns the path as it was created; it's NOT re-checked against symlink attacks at drop time. The `Drop` impl calls `remove_dir_all` naively.

**How to avoid:** Before ANY `remove_dir_all` on a temp path:
```rust
let temp_root = std::env::temp_dir().canonicalize()?;
let actual = path_to_remove.canonicalize()?;
if !actual.starts_with(&temp_root) {
    // Refuse: something has swapped our temp dir for a real filesystem location
    return Err(HubError::Typed {
        kind: HubErrorKind::PathTraversal,
        message: format!("temp path escaped temp_dir: {}", actual.display()),
        ..
    });
}
std::fs::remove_dir_all(&actual)?;
```

**Warning signs:** Failed install deletes user data; `tracing` logs show `remove_dir_all` on a path that isn't under `/tmp`.

### Pitfall 5: YAML parser silently accepts `---js`

**What goes wrong:** A skill author uses `---js` at the top of SKILL.md (the reference TS gray-matter behavior allowed this, which permitted `eval()`-based RCE). The YAML parser sees `---js\n{...}\n---` and tries to parse `{...}` — but many YAML parsers accept JSON-like content as valid YAML. Install proceeds.

**Why it happens:** `serde_yaml` accepts `---` followed by any text up to the closing `---`. It doesn't enforce that the opening delimiter be exactly `---\n`.

**How to avoid:** Before calling `serde_yaml::from_str`, explicitly check the first few bytes:
```rust
fn strict_yaml_delimiter(content: &str) -> Result<(), HubError> {
    // The reference accepts: ^---\r?\n
    let first_line_end = content.find('\n').ok_or_else(||
        typed(HubErrorKind::Parse, "SKILL.md has no newline"))?;
    let first_line = &content[..first_line_end];
    let first_line = first_line.trim_end_matches('\r');
    if first_line != "---" {
        return Err(typed(HubErrorKind::Parse,
            format!("SKILL.md frontmatter must use YAML-only `---` delimiter, got: {:?}", first_line)));
    }
    Ok(())
}
```
Call `strict_yaml_delimiter` in the blob.rs frontmatter parse path. `ironhermes-core::parse_skill_md` also implicitly enforces this via `trimmed.strip_prefix("---")?` (line 199) followed by `.strip_prefix('\n')` — so anything like `---js\n` fails the second `strip_prefix`. **Verify by adding a test** that `---js\nname: x\n---\nbody` returns `None` from `parse_skill_md`.

**Warning signs:** A SKILL.md starting with `---js` or `---javascript` installs successfully and writes to disk.

### Pitfall 6: Retry wrapper re-triggers audit / re-parses frontmatter

**What goes wrong:** The retry wrapper is placed too high in the stack and wraps the whole `fetch_skill` flow. On a transient 5xx during the `/api/download` call, the retry re-runs the tree fetch, the frontmatter fetch, AND another audit call. This spams the audit endpoint and double-bills GitHub API quota.

**How to avoid:** Wrap each HTTP call individually — `with_one_retry(|| fetch_tree(...))`, not `with_one_retry(|| entire_install_pipeline())`. See Pattern 1 above.

### Example 1: `to_skill_slug` (ports `blob.ts:55-62`)

```rust
// Source: /Users/twilson/code/skills/src/blob.ts:55-62 [VERIFIED]
// Ported to Rust with ASCII-only regex classes to match JS semantics exactly.
use regex::Regex;
use std::sync::LazyLock;

static WS_UNDERSCORE: LazyLock<Regex> =
    LazyLock::new(|| Regex::new(r"[ \t\n\r\f\v_]+").unwrap()); // ASCII \s + underscore
static NON_SLUG: LazyLock<Regex> =
    LazyLock::new(|| Regex::new(r"[^a-z0-9-]").unwrap());
static COLLAPSE_DASH: LazyLock<Regex> =
    LazyLock::new(|| Regex::new(r"-+").unwrap());

pub fn to_skill_slug(name: &str) -> String {
    let lower = name.to_lowercase();
    let step1 = WS_UNDERSCORE.replace_all(&lower, "-").into_owned();
    let step2 = NON_SLUG.replace_all(&step1, "").into_owned();
    let step3 = COLLAPSE_DASH.replace_all(&step2, "-").into_owned();
    step3.trim_matches('-').to_string()
}
```

**Note on Unicode:** `String::to_lowercase()` in Rust IS Unicode-aware (e.g., "İ" → "i̇"). JS `String.prototype.toLowerCase()` with default locale is also Unicode-aware. Behavior should match for all practical inputs; if a skill name contains Turkish İ or German ß, both implementations produce the same lowercase form per Unicode default-case folding. ### Example 2: `strip_terminal_escapes` (ports `sanitize.ts:19-52`)

```rust
// Source: /Users/twilson/code/skills/src/sanitize.ts:19-52 [VERIFIED verbatim]
use regex::Regex;
use std::sync::LazyLock;

// CSI: ESC[ ... final byte
static CSI_RE: LazyLock<Regex> = LazyLock::new(||
    Regex::new(r"\x1b\[[\x30-\x3f]*[\x20-\x2f]*[\x40-\x7e]").unwrap()
);
// OSC: ESC] ... terminator (BEL or ESC\)
static OSC_RE: LazyLock<Regex> = LazyLock::new(||
    Regex::new(r"\x1b\].*?(?:\x07|\x1b\\)").unwrap()
);
// DCS, PM, APC: ESC P|^|_ ... ESC\
static DCS_PM_APC_RE: LazyLock<Regex> = LazyLock::new(||
    Regex::new(r"\x1b[P^_].*?\x1b\\").unwrap()
);
// Simple ESC + single printable char
static SIMPLE_ESC_RE: LazyLock<Regex> = LazyLock::new(||
    Regex::new(r"\x1b[\x20-\x7e]").unwrap()
);
// C1 control codes
static C1_RE: LazyLock<Regex> = LazyLock::new(||
    Regex::new(r"[\x80-\x9f]").unwrap()
);
// Raw controls except \t (0x09) and \n (0x0a)
static CONTROL_RE: LazyLock<Regex> = LazyLock::new(||
    Regex::new(r"[\x00-\x06\x07\x08\x0b\x0c\x0d-\x1a\x1c-\x1f\x7f]").unwrap()
);

pub fn strip_terminal_escapes(s: &str) -> String {
    let s = OSC_RE.replace_all(s, "");        // longest match first
    let s = DCS_PM_APC_RE.replace_all(&s, "");
    let s = CSI_RE.replace_all(&s, "");
    let s = SIMPLE_ESC_RE.replace_all(&s, "");
    let s = C1_RE.replace_all(&s, "");
    let s = CONTROL_RE.replace_all(&s, "");
    s.into_owned()
}

pub fn sanitize_metadata(s: &str) -> String {
    static NL: LazyLock<Regex> = LazyLock::new(|| Regex::new(r"[\r\n]+").unwrap());
    let clean = strip_terminal_escapes(s);
    NL.replace_all(&clean, " ").trim().to_string()
}
```

**Rust regex-crate compatibility note:** Rust `regex` supports non-greedy `*?` and the `.` class matches any char except `\n` by default. To get the TS `[\s\S]` semantics, we use `.*?` with the default setting plus `DOTALL`-equivalent via `(?s)` mode flag if needed. For OSC/DCS, which span newlines, add `(?s)`:

```rust
static OSC_RE: LazyLock<Regex> = LazyLock::new(||
    Regex::new(r"(?s)\x1b\].*?(?:\x07|\x1b\\)").unwrap()
);
static DCS_PM_APC_RE: LazyLock<Regex> = LazyLock::new(||
    Regex::new(r"(?s)\x1b[P^_].*?\x1b\\").unwrap()
);
```
**Verify** with a golden-vector test including an OSC sequence that contains embedded `\n`. [VERIFIED: `regex` crate docs confirm `(?s)` flag enables DOTALL semantics]

```rust
// TS uses string-prefix; Rust uses std::path::Path::starts_with (path-component-aware).
use std::path::{Path, PathBuf};

pub fn is_path_safe(base: &Path, target: &Path) -> std::io::Result<bool> {
    // Don't require both paths to exist — sanity-check lexically after resolving.
    // Fall back to lexical normalization if canonicalize fails (target not yet written).
    let norm_base: PathBuf = base.canonicalize().unwrap_or_else(|_| base.to_path_buf());
    let norm_target: PathBuf = target.canonicalize().unwrap_or_else(|_| {
        // Lexical fallback: resolve ".." components against base.
        normalize_lexical(target)
    });
    Ok(norm_target.starts_with(&norm_base))
}

fn normalize_lexical(p: &Path) -> PathBuf {
    let mut out = PathBuf::new();
    for comp in p.components() {
        use std::path::Component::*;
        match comp {
            ParentDir => { out.pop(); }
            CurDir => {}
            Normal(s) => out.push(s),
            RootDir | Prefix(_) => out.push(comp.as_os_str()),
        }
    }
    out
}
```

### Example 6: `compute_folder_hash` (ports `local-lock.ts:100-115`)

**CRITICAL — EXACT SHA-256 INPUT FORMAT.** The reference TS is at `local-lock.ts:108-114`:

```javascript
// Source: /Users/twilson/code/skills/src/local-lock.ts:105-114 [VERIFIED]
```

**Translation to Rust:**
```rust
// Source: /Users/twilson/code/skills/src/local-lock.ts:100-115 [VERIFIED]
use sha2::{Digest, Sha256};
use std::path::{Path, PathBuf};

pub fn compute_folder_hash(skill_dir: &Path) -> Result<String, HubError> {
    let mut files = collect_files_sorted(skill_dir)?;
    // files is Vec<(String /* rel path, forward-slash-normalized */, Vec<u8> /* bytes */)>
    files.sort_by(|a, b| a.0.cmp(&b.0));  // localeCompare → Rust str cmp is byte-wise,
                                          // which matches JS localeCompare for ASCII paths.
                                          // For non-ASCII paths, may diverge — add golden test.
    let mut hasher = Sha256::new();
    for (rel, content) in &files {
        hasher.update(rel.as_bytes());   // NO newline separator between path and content
        hasher.update(content);           // NO newline separator between files
    }
    Ok(hex::encode(hasher.finalize()))
}

fn collect_files_sorted(base: &Path) -> Result<Vec<(String, Vec<u8>)>, HubError> {
    let mut out = Vec::new();
    walk(base, base, &mut out)?;
    Ok(out)
}

fn walk(base: &Path, dir: &Path, out: &mut Vec<(String, Vec<u8>)>) -> Result<(), HubError> {
    for entry in std::fs::read_dir(dir)? {
        let entry = entry?;
        let path = entry.path();
        let name = entry.file_name();
        let name_str = name.to_string_lossy();
        if entry.file_type()?.is_dir() {
            if name_str == ".git" || name_str == "node_modules" { continue; }
            walk(base, &path, out)?;
        } else if entry.file_type()?.is_file() {
            let rel = path.strip_prefix(base).unwrap()
                .to_string_lossy().replace('\\', "/");
            let content = std::fs::read(&path)?;
            out.push((rel, content));
        }
        // Skip symlinks entirely (defense-in-depth).
    }
    Ok(())
}
```

**⚠️ IMPORTANT — the reference algorithm has NO separator bytes between path and content, and NO trailing newline.** The CONTEXT.md D-13 says "files sorted alphabetically, `\n` separator between path + content" — this is **incorrect** relative to the reference. The reference calls `hash.update(file.relativePath)` and `hash.update(file.content)` consecutively with **nothing in between**. Adding a `\n` would produce a different hash than the reference. **Flag as O-2 below** — if CONTEXT.md's D-13 wording is authoritative (user intent), then we diverge from reference on purpose and the server-side `computedHash` will never match our value, which breaks the hash-drift detection on `update`. Recommend: **match reference exactly** (no separator), and update CONTEXT.md D-13 post-research.

2. **`computedHash` separator bytes.** CONTEXT.md D-13 says `\n` separator; reference TS uses no separator. What we know: reference `local-lock.ts:108-114` calls `hash.update(path)` then `hash.update(content)` back-to-back. What's unclear: whether the server-side `skillsComputedHash` uses the same format (and thus whether our `computedHash` must match the server-computed value for the same files). **Recommendation:** Match reference exactly (no separator). If the server uses a different algorithm, it'll show up as `computedHash != snapshotHash` on first install — this is **expected** because they're documenting different things (local folder hash vs. server snapshot). No mismatch alarm. Update CONTEXT.md D-13 wording accordingly.

3. **Pre-install audit response shape.** The reference TS `fetchAuditData` returns `Record<skillSlug, PartnerAudit>`. The field `risk` is an enum ("safe" | "low" | "medium" | "high" | "critical" | "unknown"). What's unclear: exact UI for displaying risk (one line per risk level? color coding?). **Recommendation:** For v1, print one line per skill: `Audit: <skill> — risk: <level> (alerts: <n>)`; skip color. `--skip-audit` bypass per D-19.

4. **Migration of `install_path` in old manifest.** The 19.1 `HubManifest` stores the absolute install path per-skill. The 21.8 `SkillLockEntry` doesn't have an explicit install path field (it's derived from `skills_root + category + name`). What's unclear: should we preserve the 19.1 path during migration, or trust the derivation? **Recommendation:** Re-derive at load time; if the derived path doesn't exist on disk after migration, log a warning and keep the entry in the lock file (user can reinstall). Do not store the path in the lock file — it makes moving `HERMES_HOME` brittle.

5. **`extras` passthrough for forward compat.** Phase 19.1 `ManifestEntry` has a `#[serde(flatten)] extras` bag for unknown fields. Should `SkillLockEntry` do the same? **Recommendation:** YES — keep the `extras: HashMap<String, serde_json::Value>` with `#[serde(flatten, default)]` so future schema additions don't break old code. Matches the existing WARN-BUT-LOAD pattern.

6. **Should the `SkillsShBlobSource` implement `HubSource::search`?** Reference TS's `find.ts` queries skills.sh. ironhermes 19.1 `cmd_search` already routes `--source skills-sh` through a `HubSource::search`. For 21.8, what does `search` look like when the server exposes `/api/download` but no explicit search endpoint? **Recommendation:** Return `Ok(vec![])` for now. Users can still `install <name>` directly. Defer a real `search` impl to a later phase unless the skills.sh team publishes a search API. Flag as a known gap.

### Known Threat Patterns for `reqwest` + JSON file fetch

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Terminal escape injection via server-controlled description/error | Tampering, Elevation-of-privilege (screen spoofing) | `strip_terminal_escapes` on every server-originated string before `eprintln!`/`println!` (D-16). |
| `---js` frontmatter → RCE via gray-matter eval | Elevation-of-privilege | Strict `---\n` delimiter check before `serde_yaml::from_str` (D-17). `serde_yaml` has no eval path, but matching ref behavior for defense in depth. |
| Path traversal via malicious `path` in blob JSON | Tampering | `sanitize_subpath` on every `files[].path` before `skill_dir.join(&safe)`; re-validate with `is_path_safe(skill_dir, resolved)` after join (D-18). |
| Symlink swap between extract and cleanup | Tampering, Denial-of-service (deletes user data) | `assert_temp_contained` with `canonicalize()` before `remove_dir_all` (D-20). |
| Zip/tar bomb — oversized blob response | Denial-of-service | Cap HTTP response body size via `reqwest`'s `.bytes_limit()` if available, or stream with size accounting. Reuse 50MB cap from `tarball.rs::MAX_EXTRACTED_BYTES`. [VERIFIED: constant exists] |
| SSRF via user-supplied `SKILLS_DOWNLOAD_URL` | Info-disclosure | Document that `SKILLS_DOWNLOAD_URL` is trusted user config; do NOT apply SSRF guard (it's an intentional override for testing). Well-known source SSRF guard still applies to its domain. |

| Race condition on lock file concurrent write | Tampering | Atomic tmp + rename (D-25); matches `save_config_atomic` pattern. Does NOT protect against two processes racing — document as "one CLI at a time." |
| GitHub token leak in `tracing` logs | Info-disclosure | Existing `GitHubAuth::Debug` impl redacts. `#[tracing::instrument(skip(auth))]` on all fetchers. [VERIFIED: pattern used in existing `github.rs` — skip in instrument macro] |
| Audit endpoint returning malicious JSON | Tampering, Injection | `sanitize_metadata` on every rendered field; JSON parse rejects non-object responses. Parse errors are soft-fail (no install block). |
