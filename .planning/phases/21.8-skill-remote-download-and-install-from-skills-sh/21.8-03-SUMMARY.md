# Phase 21.8 Plan 03: Wave 2 — Installer Rewire + Audit + Migration Summary

Wave 2 reworks `installer.rs` to write `SkillLock` instead of `HubManifest`, threads a `skip_audit: bool` parameter through `install()` / `update()` / `uninstall()`, inserts the D-19 soft-fail audit endpoint call between fetch and quarantine, gates every `remove_dir_all` on a quarantine path with `sanitize::assert_temp_contained`, raises `HubErrorKind::ShaMismatch` when the post-rename `compute_folder_hash` differs from the server-returned `snapshot_hash`, and ships the idempotent one-way 19.1 → 21.8 `migrate_from_hub_manifest` helper.

## What Shipped

### `crates/ironhermes-hub/src/audit.rs` (NEW, ~200 lines)

Pre-install audit endpoint client per D-19. `fetch_audit(&reqwest::Client, owner_repo, &[&slug]) -> Option<AuditData>` wraps a `tokio::time::timeout(Duration::from_secs(3))` around a `GET {SKILLS_AUDIT_URL}/audit?source=<owner/repo>&skills=<joined slugs>`. Every error path (timeout, 5xx, non-JSON, HTTP/DNS) returns `None` and emits a `tracing::warn!` with the error message passed through `sanitize::strip_terminal_escapes` — the installer NEVER blocks because audit is down.

| Type | Shape |
|------|-------|
| `PartnerAudit` | `{ risk: String, alerts: u64, score: Option<f64>, analyzed_at: Option<String> }` |
| `AuditData` | `HashMap<String, PartnerAudit>` keyed by skill slug |

`SKILLS_AUDIT_URL` env var overrides the default base URL (`https://add-skill.vercel.sh`), used by tests and operators running a private mirror.

### `crates/ironhermes-hub/src/installer.rs` (REWRITE, ~735 lines)

**Pipeline order locked (D-25):**

```
1. migrate       crate::lock::migrate_from_hub_manifest()  -- idempotent
2. fetch         source.fetch(identifier)
                 bundle.snapshot_hash.clone()              -- plan 02 contract, Step 1a
3. audit         crate::audit::fetch_audit(...)            -- soft-fail, skipped if skip_audit=true
4. quarantine    write_bundle_to_dir(quarantine.path(), &bundle)
                 -- sanitize::sanitize_subpath BEFORE validate_bundle_rel_path (D-18)
5. scan          enforce_trust_gate(trust, &verdict)
                 -- cleanup_quarantine_safely on Err (D-20 gate)
6. rename        atomic_move(quarantine, final_path)
7. verify        compute_folder_hash(&final_path) vs bundle.snapshot_hash
                 -- mismatch -> HubErrorKind::ShaMismatch + cleanup_final_path_safely
8. lock write    SkillLock::load_or_default -> add_or_replace -> save_atomic
```

**New `skip_audit: bool` parameter** on `install()`, `update()`. `uninstall()` doesn't need it but still runs `migrate_from_hub_manifest()`. All three CLI call sites in `skills_cmd.rs` pass `false` (plan 04 wires `--skip-audit`).

**D-20 cleanup gate:** `cleanup_quarantine_safely(&Path)` canonicalizes the input and asserts it resolves under `std::env::temp_dir()` before `remove_dir_all`. Non-temp paths log a `tracing::warn` and are skipped — never followed through a symlink swap. `cleanup_final_path_safely(&Path)` uses `is_path_safe(skills_root, final_path)` instead (final installs live under `skills_root`, not `temp_dir`).

**Drift detection for `update()`** uses a new crate-private `bundle_folder_hash(&SkillBundle) -> String` that applies the identical D-13 no-separator algorithm `lock::compute_folder_hash` uses on disk — so the comparison against `SkillLockEntry::computed_hash` is byte-for-byte valid. The pre-21.8 `bundle_content_hash` (which uses a `0x00` separator) is retained for backward compatibility of `InstallOutcome::content_hash` but is no longer on the drift-detection path.

**17 new unit tests** in `installer::tests` cover: bundle hash determinism/sort/diff, category parsing, `extract_owner_repo` for github / skills-sh / https / well-known identifiers, `extract_repo_path`, `write_bundle_to_dir` with sanitize_subpath rejection, `cleanup_quarantine_safely` refusing non-temp paths and removing temp paths, `atomic_move`, `copy_dir_recursive`.

### `crates/ironhermes-hub/src/lock.rs` (EXTENDED)

Added `MigrationOutcome { NothingToMigrate, Migrated(usize) }` + `pub fn migrate_from_hub_manifest() -> anyhow::Result<MigrationOutcome>`:

- Guard 1: if `skills-lock.json` already has ≥1 entry → `NothingToMigrate` (idempotent).
- Guard 2: if `.hub/lock.json` doesn't exist → `NothingToMigrate` (no source).
- Otherwise: deserialize old `HubManifest`, map each `ManifestEntry` → `SkillLockEntry` (`content_hash` → `computed_hash`, empty `snapshot_hash` for backfill on next update, `repo_path` = first file path), `save_atomic` the new lock, then `remove_file` the old `.hub/lock.json` and `rmdir .hub/` if empty.
- Race safety: tmp+rename via `save_atomic` prevents torn writes; if two processes race, the second observes `NothingToMigrate`. Migration failure leaves both files in place.

3 new migration tests under `lock::migration_tests`: migrates + deletes old, byte-identical second run, no-source noop.

### `crates/ironhermes-hub/src/manifest.rs` (MODIFIED)

Added module-level doc note marking the file DEPRECATED in 21.8 and retained only for migration reads. `#[deprecated]` attribute on `HubManifest::save` pointing callers to `SkillLock::save_atomic`. Load path (`HubManifest::load_or_default`) stays callable so `migrate_from_hub_manifest` can read the old schema. Existing roundtrip tests use `#[allow(deprecated)]` on the tests mod.

let outcome2 = migrate_from_hub_manifest().unwrap();

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 — Missing critical functionality] drift detection used algorithmically incompatible hashes**
- **Found during:** Task 2 verification (integration tests failed)
- **Issue:** The plan's `update()` pseudocode compared `bundle_content_hash` (0x00-separated) against `SkillLockEntry::computed_hash` (D-13 no-separator). These two hashes CANNOT be equal for the same bundle — meaning drift detection was always-true and `update_noop_when_hash_matches` could never return `AlreadyInstalled`.
- **Fix:** Added `bundle_folder_hash(&SkillBundle) -> String` in `installer.rs`. This is a crate-private helper mirroring `lock::compute_folder_hash` exactly (sort by forward-slash-normalized path, hash `path_bytes ‖ content_bytes` with NO separator, no separator between files). Swapped the drift check to compare `bundle_folder_hash(&bundle)` vs `entry.computed_hash`. Byte-identical content now correctly yields `AlreadyInstalled`.
- **Files modified:** `crates/ironhermes-hub/src/installer.rs`
- **Commit:** `90b07c1`

**2. [Rule 1 — Bug] UpdateOutcome.old_hash algorithmic drift against integration test**
- **Found during:** Task 2 verification (`update_detects_hash_drift_and_replaces`)
- **Issue:** The old `UpdateOutcome.old_hash` was `manifest.content_hash` (bundle_content_hash). After the SkillLock swap, `UpdateOutcome.old_hash = entry.computed_hash` (folder walk hash) — the integration test still asserted equality with `InstallOutcome.content_hash` from the prior install call, which is `bundle_content_hash`. Always failed.
- **Fix:** Test reads `old_hash` from `SkillLock::load_or_default().get("tenor-gif").computed_hash` between install and update — apples-to-apples.
- **Files modified:** `crates/ironhermes-hub/tests/update_uninstall_test.rs`
- **Commit:** `90b07c1`

**3. [Rule 3 — Blocking] `with_audit_url` closure pattern from plan sample can't wrap `.await`**
- **Found during:** Task 1 (writing `audit::tests::happy_path_returns_some`)
- **Issue:** The plan's sample `with_audit_url<F: FnOnce() -> T>` closure wrapper could not be held across the `fetch_audit(...).await` call without creating a nested `block_on` inside a `#[tokio::test]` runtime — which panics.
- **Fix:** Rewrote as an RAII `AuditUrlGuard { prev, _lock }` struct with `Drop` restoring the env var. The MutexGuard lifetime extends across every `.await` between construction and drop, so the async test body can freely `await`.
- **Files modified:** `crates/ironhermes-hub/src/audit.rs`
- **Commit:** `f04e690`
