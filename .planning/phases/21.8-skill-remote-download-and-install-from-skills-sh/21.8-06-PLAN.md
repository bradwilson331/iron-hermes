<objective>
Unblock `hermes skills install <name>` against live skills.sh by reframing the post-install hash check from a strict equality failure into an advisory log. The current implementation violates D-14 (which declares `snapshotHash` opaque and forbids client-side recomputation) by raising `HubErrorKind::ShaMismatch` whenever the server's snapshotHash algorithm diverges from our D-13 no-separator folder hash â€” which is every real install, per UAT.

**Purpose:** close the UAT blocker so live installs reach the D-21 5th line + D-23 restart message, without touching D-13 (client-authoritative drift detection, already unit-tested with hand-computed vectors) or the `snapshotHash` round-trip plumbing (D-14 opaque contract).

**Output:**
- `installer.rs` â€” Step 7 in both `install()` and `update()` becomes an advisory compare (log-only; install proceeds unconditionally to lock-write).
- `error.rs` â€” doc comment on `HubErrorKind::ShaMismatch` narrowed to drift-detection-only semantics (matches the doc that was already there; now the code matches).
- `skills_sh_blob_adapter.rs` â€” new integration test exercising the divergence path end-to-end.
- `update_uninstall_test.rs` â€” new integration test confirming update() tolerates server/client divergence.
- No D-13 algorithm changes. No new workspace dependencies. No new CLI flags. No changes to `cmd_install` / `cmd_update` / `cmd_list` â€” plan 04 wiring stays as-is.
</objective>
