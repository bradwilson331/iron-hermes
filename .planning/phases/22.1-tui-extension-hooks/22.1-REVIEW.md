# Phase 22.1: Code Review Report

**Reviewed:** 2026-04-17T12:00:00Z
**Depth:** standard
**Files Reviewed:** 6
**Status:** issues_found

## Summary

The Phase 22.1 TUI extension hooks system is well-structured with clear separation of concerns: pure types in `extension.rs`, keybinding logic in `keybindings.rs`, command dispatch in `commands.rs`, and I/O isolated to `render.rs`. The extension-first dispatch chain with `catch_unwind` panic containment is a solid defensive pattern. Widget height capping and content truncation address the T-22.1-01 security concern properly.

Key concerns: (1) the pre-readline keybinding poll in `main.rs` silently consumes non-matching key events, which could cause user-visible input loss; (2) the `reserved` row count is cached at construction and never updated when widgets are added/removed at runtime via `TuiEvent`, causing a desync between the render loop's dynamic `reserved_rows()` and the value used by `prepare_prompt_with_reserve`; (3) the `status.hint` style override is accepted but silently dropped (dead code path); (4) `print_help()` in `main.rs` is dead code.

## Warnings

### WR-01: Cached `reserved` row count desyncs from runtime widget mutations

**File:** `crates/ironhermes-cli/src/tui/render.rs:170-205`
**Issue:** `TuiHandle.reserved` is computed once at construction from the initial widget set and cached in the struct field (line 170). However, the render loop dynamically processes `TuiEvent::UpdateWidget` and `TuiEvent::RemoveWidget` events that can add or remove widgets, changing the effective reserved row count. The render loop correctly calls `reserved_rows(&widgets)` on each tick (line 388), but callers of `tui.reserved_row_count()` (used in `prepare_prompt_with_reserve` and `finish_prompt_with_reserve` at main.rs lines 712-714) still get the stale cached value. This means the prompt positioning and DECSTBM scroll region set by the REPL loop will be wrong after any runtime widget add/remove.
**Fix:** Either expose the current reserved count via a `watch` channel that the render loop updates when widgets change, or use an `AtomicU16` shared between the render loop and TuiHandle:
```rust
// In TuiHandle struct:
reserved: Arc<std::sync::atomic::AtomicU16>,

// In render_loop after processing RemoveWidget/UpdateWidget:
reserved_atomic.store(reserved_rows(&widgets), std::sync::atomic::Ordering::Relaxed);

// In reserved_row_count():
pub fn reserved_row_count(&self) -> u16 {
    self.reserved.load(std::sync::atomic::Ordering::Relaxed)
}
```

### WR-02: Pre-readline keybinding poll silently eats non-matching key events

**File:** `crates/ironhermes-cli/src/main.rs:699-710`
**Issue:** When `crossterm::event::poll` returns true, `crossterm::event::read()` consumes the event from the buffer. If the event does not match any keybinding (lines 701-705), execution falls through to the comment on line 707: "Key didn't match any binding -- it's consumed and lost." While the comment acknowledges this is "acceptable for modifier-key combos," this also consumes non-modifier keys (e.g., a plain letter key typed just before `poll` runs) because `crossterm::event::poll` does not distinguish key types. In practice, a race between the user typing a character and the poll check could drop that character before rustyline ever sees it.
**Fix:** Re-inject the unmatched event back into the input stream, or only consume the event after confirming it matches a binding. Since crossterm does not support "peek without consume," an alternative is to only poll when extensions actually have Idle keybindings registered:
```rust
let has_idle_bindings = !keybinding_registry.help_entries().is_empty();
if has_idle_bindings && crossterm::event::poll(std::time::Duration::ZERO).unwrap_or(false) {
    // ... existing logic
}
```
This at least avoids the poll entirely when no extensions register keybindings (the current default).

### WR-03: `AboveStatus` widget rendering can underflow `row_cursor` to 0 and overwrite content

**File:** `crates/ironhermes-cli/src/tui/render.rs:613-631`
**Issue:** The `AboveStatus` widget rendering loop decrements `row_cursor` upward from `scanner_row - 1`. The guard `if row_cursor > 0 { row_cursor -= 1; }` prevents underflow past 0, but it does not prevent writing to row 0 itself (the top of the terminal). If a malicious or buggy extension provides a large number of AboveStatus widgets (or widgets with content exceeding their declared height), the render could overwrite application output in the scrollable content area. The `MAX_WIDGET_HEIGHT` cap mitigates this somewhat, but with multiple AboveStatus widgets the cumulative height can still exceed the available space above the scanner.
**Fix:** Add a lower-bound check that stops rendering when `row_cursor` would enter the scroll region:
```rust
let scroll_top = 0u16; // or a defined minimum safe row
for w in above_status.iter().rev() {
    let h = w.height.min(MAX_WIDGET_HEIGHT);
    let lines: Vec<&str> = w.content.lines().collect();
    let line_count = lines.len().min(h as usize);
    for i in (0..line_count).rev() {
        if row_cursor == 0 { break; } // Stop before overwriting scroll region
        // ... render line ...
    }
    if row_cursor == 0 { break; }
}
```

### WR-04: `status.hint` style override is accepted but silently dropped (dead code path)

**File:** `crates/ironhermes-cli/src/tui/render.rs:534-537`
**Issue:** The `hint_override` value is extracted from `style_overrides` (line 506), matched in the `if let Some(color)` block (line 534), and then immediately dropped with `let _ = color;` and a comment saying "accepted, T-22.1-05: handled gracefully." This is effectively dead code -- the style override is advertised to extensions via `StyleOverrides` but has no effect. An extension author who sets `"status.hint"` would get no visible change with no error or warning.
**Fix:** Either implement the color override (apply `colored::Color` from the string), or remove `"status.hint"` from the documented style slot names and add a `tracing::debug!` when an unsupported slot is provided so extension authors get feedback.

## Info

### IN-01: `print_help()` function in main.rs is dead code

**File:** `crates/ironhermes-cli/src/main.rs:1170-1174`
**Issue:** The `print_help()` function is defined but never called anywhere. The `/help` command is handled via `dispatch_command` -> `core_dispatch` -> `format_help`, and the function's own doc-comment acknowledges this: "Phase 22.1: delegate to format_help with no extensions..." No callers exist in the codebase.
**Fix:** Remove the function or annotate with `#[allow(dead_code)]` if retained for backward compatibility. The compiler may already warn about this.

### IN-02: Backward-compatibility wrappers `reset_scroll_region`, `prepare_prompt`, `finish_prompt`, and `redraw` are dead code

**File:** `crates/ironhermes-cli/src/tui/render.rs:289-292, 320-322, 349-352, 681-684`
**Issue:** Four functions are marked `#[allow(dead_code)]` as backward-compatibility wrappers for the zero-extension Phase 21 API. They delegate to the `_with_reserve`/`_with_extensions` variants. If no callers remain, these add maintenance burden with no value.
**Fix:** If the Phase 21 API surface is no longer needed externally, remove these wrappers. If kept for potential future use, consider marking them `#[cfg(test)]` instead of `#[allow(dead_code)]`.

### IN-03: `_extensions` parameter unused in `format_help`

**File:** `crates/ironhermes-cli/src/tui/commands.rs:88`
**Issue:** The `_extensions` parameter (prefixed with underscore to suppress the unused-variable warning) is accepted by `format_help` but never used. Extensions that register custom commands are not listed in the help output. This means extension-provided commands are invisible to `/help`.
**Fix:** Iterate over extensions and call a hypothetical `help_entries()` or `commands()` method to list extension-provided commands in the help output. Alternatively, document that extension command help is a TODO for a future phase.

### IN-04: `context_matches` does not handle `Always` as `active_ctx`

**File:** `crates/ironhermes-cli/src/tui/keybindings.rs:66-72`
**Issue:** `context_matches` handles the case where the *binding* context is `Always` (matches everything), but if the *active* context were ever `Always`, an `Idle`-only or `InFlight`-only binding would not match even though the system is semantically in "both" states. Currently `match_key` is only called with `Idle` or `InFlight` as the active context, so this is not a bug today, but it is a subtle contract gap that could cause confusion if a future caller passes `Always` as the active context.
**Fix:** Add a match arm or doc-comment clarifying that `active_ctx` should only ever be `Idle` or `InFlight`:
```rust
/// NOTE: `active_ctx` should be `Idle` or `InFlight` only.
/// `Always` as an active context is not supported and will not match
/// Idle-only or InFlight-only bindings.
```
