# Phase 22.1: TUI Extension Hooks - Context

**Gathered:** 2026-04-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Create a Rust extension mechanism for the CLI TUI so that external code (plugins, custom builds, future crates) can add widgets, keybindings, layout sections, command handlers, and style overrides — the Rust equivalent of hermes-agent's subclassable `CliManager`.

**In scope (Phase 22.1):**
- CLI-02: TuiExtension trait with methods mirroring the 5 Python hooks
- Hybrid architecture: trait objects (static contract) + mpsc message bus (dynamic updates) + command registry (extension-first dispatch)
- Slot-based layout system for extension widgets
- Keybinding registration with context-aware dispatch (Idle/InFlight/Always)
- process_command() hook point with extension-first priority chain
- Style override mechanism via `_build_tui_style_dict()` equivalent

**Out of scope:**
- Core slash command router (SKILL-12/13/14 — separate phase)
- ACP adapter (CLI-03..08 — Phase 22.2)
- Full ratatui migration or new TUI dependencies
- Actual extension implementations (this phase builds the hooks, not extensions that use them)

</domain>

<decisions>
## Implementation Decisions

### Extension architecture
- **D-01:** Hybrid three-layer architecture:
  1. **Trait objects** for the static contract — `TuiExtension` trait with methods for widgets, initial layout, help text, style overrides
  2. **Message bus** via tokio `mpsc` channel for the dynamic contract — extensions send `TuiEvent::UpdateWidget(id, data)` to TuiHandle for live updates
  3. **Command registry** with extension-first priority — `process_command()` checks extensions before falling back to core slash commands, allowing extensions to shadow built-in commands
- **D-02:** Reuse existing channel pattern — add a `TuiEvent` enum and new `mpsc::UnboundedSender` on TuiHandle, consistent with the `watch::channel` pattern already used for `ActivityState` and `StatusLineState` in `render.rs`. No new dependencies.

### Widget & layout model
- **D-03:** Slot-based layout. Define named slots (`AboveStatus`, `BelowScanner`, `StatusRight`). Extensions register widgets into slots via the trait. The render loop queries each slot and composites. DECSTBM scroll region adjusts dynamically based on `3 + extra_rows` for extension widgets.
- **D-04:** String-based Widget struct — `Widget { id: String, content: String /* ANSI-colored */, height: u16 }`. Extensions produce rendered content; the render loop handles cursor positioning and clearing. Stays on crossterm+colored primitives per Phase 21 D-18.

### Keybinding registration
- **D-05:** Idle-only dispatch as the primary mode. Extensions register `Keybinding { key: KeyEvent, description: String, when: KeyContext }` where `KeyContext` is `Idle | InFlight | Always`. A pre-readline key check reads one crossterm event — if it matches a registered binding, dispatch it and skip readline for that tick. If not, feed it to readline normally. No conflict with rustyline during editing.

### Command processing
- **D-06:** Hook point only in this phase. `TuiExtension::process_command(&self, cmd: &str, args: &[&str]) -> Option<CommandResult>` is the extension hook. Dispatch chain: try extensions first (shadowing), then core fallback. The core fallback is a stub (handles `/quit`, `/exit`, returns "unknown command" for everything else) until SKILL-12/13/14 wire the real slash command router.

### Claude's Discretion
- Exact `TuiEvent` enum variants beyond `UpdateWidget`
- Whether `KeyContext::Always` bindings use a separate signal handler or integrate into the existing ctrl-c state machine
- Whether style overrides use a `HashMap<String, Style>` or a dedicated `StyleDict` struct
- Error handling strategy for extension panics (catch_unwind wrapper or trust extensions)
- Whether to include a `TuiExtension::name()` method for debug logging and conflict resolution

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `TuiHandle` (render.rs) — Already holds `watch::Sender` channels and a shutdown token. Natural place to add `mpsc::UnboundedSender<TuiEvent>` and `Vec<Box<dyn TuiExtension>>`
- `render_loop` (render.rs) — Already ticks at 100ms, queries activity/status channels. Extension widget rendering plugs into the same loop
- `StatusLineState` (status_line.rs) — The `StatusRight` slot can append extension pills using the existing `rotate_pill_colors` function
- `DoubleCtrlCState` (double_ctrl_c.rs) — `KeyContext::Always` bindings may need to integrate with or sit alongside this state machine

### Established Patterns
- Watch channels for state broadcast (activity_tx, status_tx) — extension event bus follows same pattern with mpsc
- Pure-function submodules with I/O isolated to render.rs — extension types should follow same split
- DECSTBM scroll region (rows-3 reserved) — must become dynamic (rows-3-N where N = extension rows)
- `#[cfg(test)] new_for_tests()` pattern on TuiHandle — extension tests should use same approach

### Integration Points
- `TuiHandle::new()` — Must accept optional `Vec<Box<dyn TuiExtension>>` or have `register_extension()` method
- `render_loop` / `redraw()` — Must query extension widgets per slot and composite into frame
- `run_chat` REPL loop — Pre-readline keybinding check inserts before `rl.readline()` call
- `run_chat` command dispatch — `/` prefixed input routes through extension process_command chain before existing `/quit` handling

</code_context>

<specifics>
## Specific Ideas

- The hybrid architecture mirrors how many Rust GUI frameworks work: static trait for definition, channels for runtime updates. Think `iced` or `druid` patterns but at crossterm level.
- Extension-first command dispatch allows plugins to override `/help` with richer output, shadow `/quit` with cleanup hooks, etc. The stub fallback keeps this phase self-contained without pulling in SKILL-12/13/14.
- The `Widget.id` field enables `TuiEvent::UpdateWidget(id, data)` — extensions can update their widgets dynamically without re-registering the full widget set each frame.
- `KeyContext::InFlight` bindings are rare but enable use cases like "press Escape to cancel" as an alternative to ctrl-c.

</specifics>

<deferred>
## Deferred Ideas

- **Actual extension implementations** — This phase builds the hook mechanism. Concrete extensions (e.g., a timer widget, a custom status pill) would be separate work.
- **Full slash command router (SKILL-12/13/14)** — The `process_command()` core fallback is a stub. Real routing comes in the slash command phase.
- **Style theming system** — `_build_tui_style_dict()` creates the override mechanism, but a full theme config file (user-selectable palettes) is future work.
- **ratatui migration** — Stays out of scope per Phase 21 D-16/D-18. The slot/widget system is designed to be compatible if ratatui is adopted later.
