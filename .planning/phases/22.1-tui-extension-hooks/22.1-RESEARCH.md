# Phase 22.1: TUI Extension Hooks - Research

**Researched:** 2026-04-17
**Domain:** Rust trait-based extension system for crossterm TUI
**Confidence:** HIGH

## Summary

Phase 22.1 implements CLI-02: the TUI extension hooks that allow external code to add widgets, keybindings, layout sections, command handlers, and style overrides to the CLI's bottom-bar TUI. This is the Rust equivalent of hermes-agent's subclassable `CliManager` Python hooks (`_get_extra_tui_widgets()`, `_register_extra_tui_keybindings()`, `_build_tui_layout_children()`, `process_command()`, `_build_tui_style_dict()`).

The existing TUI architecture (Phase 21) is clean and well-structured: a `TuiHandle` struct holds `watch::Sender` channels for activity and status state, a render loop ticks at 100ms (10fps), and DECSTBM scroll regions reserve the bottom 3 rows (prompt + scanner + status bar). The extension system layers on top of this without new dependencies -- using `tokio::sync::mpsc` for the event bus (already used in 5+ other modules in the codebase), `crossterm::event::KeyEvent` for keybinding registration (already in deps), and a `TuiExtension` trait for the static contract.

**Primary recommendation:** Implement a three-layer hybrid architecture: (1) `TuiExtension` trait with methods for widgets, keybindings, commands, and style overrides; (2) `mpsc::UnboundedSender<TuiEvent>` on TuiHandle for runtime dynamic updates; (3) command registry with extension-first dispatch chain. All within `crates/ironhermes-cli/src/tui/` using existing crossterm 0.28.1 + colored primitives only.

### Pattern 1: TuiExtension Trait (Static Contract)
**What:** A trait defining the five hook methods that extensions implement. Trait objects (`Box<dyn TuiExtension>`) stored in a Vec on TuiHandle.
**When to use:** Every extension implements this trait. TuiHandle queries all registered extensions during render and command dispatch.
**Example:**
```rust
// Source: Design from CONTEXT.md D-01 through D-06
pub trait TuiExtension: Send + Sync {
    /// Unique name for debug logging and conflict resolution.
    fn name(&self) -> &str;

    /// Return widgets to render in named layout slots.
    /// Called once at registration time; dynamic updates go through TuiEvent.
    fn widgets(&self) -> Vec<(LayoutSlot, Widget)> {
        Vec::new()
    }

    /// Return keybindings this extension wants to register.
    fn keybindings(&self) -> Vec<Keybinding> {
        Vec::new()
    }

    /// Attempt to handle a slash command. Return Some(CommandResult) if handled,
    /// None to pass to next extension or core fallback.
    fn process_command(&self, cmd: &str, args: &[&str]) -> Option<CommandResult> {
        let _ = (cmd, args);
        None
    }

    /// Return style overrides. Keys are style slot names (e.g., "status.mode",
    /// "scanner.lit"), values are ANSI color strings or style descriptors.
    fn style_overrides(&self) -> StyleOverrides {
        StyleOverrides::default()
    }
}
```

### Pattern 2: Message Bus (Dynamic Contract)
**What:** `mpsc::UnboundedSender<TuiEvent>` added to TuiHandle. Extensions clone the sender and emit events to update widgets at runtime without re-registering.
**When to use:** When extension state changes mid-session (e.g., a timer widget updating every second, a progress bar advancing).
**Example:**
```rust

pub enum TuiEvent {

    /// widget in its slot and redraws on the next tick.

    /// Remove a widget by ID.
    RemoveWidget { id: String },
    /// Flash a transient message in the hint area (auto-clears after N ticks).
    FlashHint { message: String, duration_ticks: u16 },
}
```

### Pattern 3: Command Registry (Extension-First Dispatch)
**What:** When the user types a `/` prefixed command, the dispatch chain tries each extension's `process_command()` first (in registration order). If all return `None`, fall through to the core stub handler.
**When to use:** Every REPL loop iteration where input starts with `/`.
**Example:**
```rust
// Source: CONTEXT.md D-06
pub enum CommandResult {
    /// Command handled successfully, display this output.
    Handled(String),
    /// Command handled, no output to display.
    Silent,
    /// Command recognized but failed.
    Error(String),
}

// In run_chat REPL loop, replacing the current match block:
fn dispatch_command(
    extensions: &[Box<dyn TuiExtension>],
    cmd: &str,
    args: &[&str],
) -> CommandResult {
    // Extension-first: try each extension in order
    for ext in extensions {
        if let Some(result) = ext.process_command(cmd, args) {
            return result;
        }
    }
    // Core fallback stub
    match cmd {
        "quit" | "exit" | "q" => CommandResult::Handled("Goodbye!".to_string()),
        "clear" => CommandResult::Handled("Conversation cleared.".to_string()),
        "status" => CommandResult::Handled("Status...".to_string()),
        "help" => CommandResult::Handled(format_help(extensions)),
        _ => CommandResult::Error(format!("Unknown command: /{}", cmd)),
    }
}
```

### Pattern 4: Slot-Based Layout with Dynamic DECSTBM
**What:** Named layout slots (AboveStatus, BelowScanner, StatusRight) where extensions place widgets. The render loop counts total extension rows and adjusts the DECSTBM scroll region from `rows-3` to `rows-3-N` where N is the total height of extension widgets.
**When to use:** During render_loop initialization and on every resize event.
**Example:**
```rust

pub enum LayoutSlot {
    /// Widget rendered above the status bar
    AboveStatus,
    /// Widget rendered below the scanner row
    BelowScanner,
    /// Widget appended to the status line (pill-style)
    StatusRight,
}
```

### Pattern 5: Pre-Readline Keybinding Check
**What:** Before calling `rl.readline()`, do a non-blocking `crossterm::event::poll(Duration::ZERO)` + `crossterm::event::read()`. If the event matches a registered keybinding for the current context (Idle/InFlight), dispatch it and skip readline for that iteration. If not, the event is lost to readline's own event loop.
**When to use:** At the top of the REPL loop, before `prepare_prompt()` and `rl.readline()`.

**CRITICAL DESIGN NOTE:** rustyline 15.0.0 puts the terminal into raw mode during `readline()` and handles its own key events. The pre-readline check must happen BEFORE `readline()` is called, using crossterm's `poll(Duration::ZERO)` to check for pending events. This only works for keys pressed between readline calls (during idle at prompt). For InFlight bindings, the key check integrates into the existing `tokio::select!` loop where ctrl-c is already handled.

```rust
// Source: CONTEXT.md D-05, crossterm 0.28.1 docs
#[derive(Debug, Clone, PartialEq, Eq)]
pub enum KeyContext {
    /// Only active when at the prompt (not in-flight)
    Idle,
    /// Only active during agent turn execution
    InFlight,
    /// Active in both states
    Always,
}

#[derive(Debug, Clone)]
pub struct Keybinding {
    pub key: crossterm::event::KeyEvent,
    pub description: String,
    pub when: KeyContext,
    // The action is dispatched by name -- the extension handles it
    // via a callback or message
    pub action_name: String,
}
```

### Anti-Patterns to Avoid
- **DO NOT use crossterm EventStream:** Requires the `event-stream` feature flag (not enabled) and adds futures-core dependency. Per Phase 21 D-16/D-18, no new dependencies. Use synchronous `poll(Duration::ZERO)` + `read()` instead.
- **DO NOT put raw mode management in extension code:** rustyline manages raw mode. Extensions should never call `enable_raw_mode()` or `disable_raw_mode()`.
- **DO NOT block the render loop:** Extension trait methods must be non-blocking. Widget content is pre-rendered strings. Any async work should go through the mpsc event bus.
- **DO NOT break the existing REPL flow:** The extension hooks layer on top of the existing `run_chat` loop. The `/quit`, `/exit`, `/clear`, `/status`, `/help` commands still work; extensions just get first crack at dispatch.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Async event stream | Custom async crossterm wrapper | Synchronous poll(Duration::ZERO) + read() | Keeps it simple, no new deps, sufficient for pre-readline check |
| Channel-based state broadcast | Custom pub/sub system | tokio::sync::mpsc::unbounded_channel | Battle-tested, already used in 5+ modules in codebase |
| Terminal event types | Custom key representation | crossterm::event::KeyEvent, KeyCode, KeyModifiers | Already in deps, handles platform differences |
| Color/style output | Custom ANSI escape builder | colored crate macros | Already used throughout, consistent styling |

**Key insight:** The entire extension system can be built with zero new dependencies. Every primitive needed (channels, key events, ANSI colors, cursor positioning) is already in the workspace dep graph.

## Common Pitfalls

### Pitfall 1: DECSTBM Scroll Region Miscalculation
**What goes wrong:** Adding extension widget rows without updating the scroll region causes output to overwrite the extension widgets, or the prompt row shifts incorrectly.
**Why it happens:** The current code hardcodes `rows.saturating_sub(3)` in 5+ places. Extension widgets add N more reserved rows, making it `rows.saturating_sub(3 + N)`.
**How to avoid:** Extract the reserved-rows calculation into a single function `reserved_rows(extension_widgets: &[Widget]) -> u16` that returns `3 + total_extension_height`. Call this everywhere the hardcoded 3 appears: TuiHandle::new (initial DECSTBM), render_loop resize handler, prepare_prompt, finish_prompt, reset_scroll_region.
**Warning signs:** Text scrolling over the status bar or widgets being invisible.

### Pitfall 2: Rustyline vs Crossterm Event Conflict
**What goes wrong:** Both rustyline and crossterm try to read terminal events. Reading a key event with crossterm before readline consumes it -- readline never sees it.
**Why it happens:** The terminal input buffer is shared. A `crossterm::event::read()` that consumes a regular character (e.g., 'a') means rustyline won't get that keystroke.
**How to avoid:** Only poll for events at specific safe points: (a) before readline starts (Idle keybindings), (b) during the tokio::select! loop when agent is in-flight (InFlight keybindings). For Idle bindings, check if the key matches a registered binding FIRST -- if not, do NOT consume it (this is tricky: crossterm's `read()` is consuming). Consider only checking for modifier-key combinations (Ctrl+X, Alt+X) that rustyline wouldn't handle, or use `poll()` to peek then only `read()` if there's an event AND it matches.
**Warning signs:** User types a character but it doesn't appear in the readline prompt.

### Pitfall 3: Extension Panic Poisoning the Render Loop
**What goes wrong:** If an extension's `widgets()` or `process_command()` panics, it could take down the render task or the REPL loop.
**Why it happens:** Rust panics unwind the current thread/task by default.
**How to avoid:** Wrap extension trait method calls in `std::panic::catch_unwind()` at the dispatch boundary. Log the panic and skip the faulting extension. This is Claude's discretion per CONTEXT.md.
**Warning signs:** TUI suddenly stops rendering or REPL crashes without a clear error.

### Pitfall 4: Widget ID Uniqueness Collisions
**What goes wrong:** Two extensions register widgets with the same ID. UpdateWidget events go to the wrong widget.
**Why it happens:** No namespace enforcement on widget IDs.
**How to avoid:** Prefix widget IDs with the extension name: `"{extension_name}:{widget_id}"`. Enforce uniqueness at registration time with a HashSet check.
**Warning signs:** Widget content appearing in wrong slot or flickering between two extensions' content.

### Pitfall 5: Style Override Ordering
**What goes wrong:** Multiple extensions override the same style key. Last-wins is unpredictable if registration order isn't documented.
**Why it happens:** HashMap insertion with same key silently replaces.
**How to avoid:** Document that extensions registered later have higher priority. Log a warning when a style key is overridden. Or: first-wins policy where early registrations take precedence.
**Warning signs:** Style inconsistencies depending on extension load order.

### Pitfall 6: Pre-Readline Key Check Timing
**What goes wrong:** The pre-readline key check with `poll(Duration::ZERO)` misses key events because they arrive during readline's execution.
**Why it happens:** Idle keybindings only work between readline calls. During readline, rustyline owns the terminal.
**How to avoid:** Accept this limitation for Idle keybindings. Document that Idle bindings only fire between prompt displays (not while user is typing). For InFlight bindings, integrate into the existing `tokio::select!` loop alongside ctrl-c handling.
**Warning signs:** Key bindings that work inconsistently or never fire.

## Code Examples

### Example 1: TuiHandle Extension Registration
```rust
// Source: Synthesized from CONTEXT.md D-01, D-02 + existing TuiHandle pattern in render.rs
impl TuiHandle {
    pub fn new_with_extensions(
        initial_status: StatusLineState,
        extensions: Vec<Box<dyn TuiExtension>>,
    ) -> Self {
        let (activity_tx, activity_rx) = watch::channel(ActivityState::Idle);
        let (status_tx, status_rx) = watch::channel(initial_status);
        let (event_tx, event_rx) = tokio::sync::mpsc::unbounded_channel::<TuiEvent>();
        let shutdown = CancellationToken::new();

        // Collect initial widgets from all extensions
        let mut widgets: HashMap<String, (LayoutSlot, Widget)> = HashMap::new();
        for ext in &extensions {
            for (slot, widget) in ext.widgets() {
                let prefixed_id = format!("{}:{}", ext.name(), widget.id);
                widgets.insert(prefixed_id, (slot, widget));
            }
        }

        // Calculate dynamic reserved rows
        let extra_rows: u16 = widgets.values()
            .filter(|(slot, _)| *slot != LayoutSlot::StatusRight)
            .map(|(_, w)| w.height)
            .sum();

        // Set DECSTBM with dynamic reserve
        if stderr().is_tty() && let Ok((_cols, rows)) = size() {
            let scroll_end = rows.saturating_sub(3 + extra_rows);
            if scroll_end > 0 {
                let mut out = stderr();
                let _ = write!(out, "\x1b[1;{}r", scroll_end);
                let _ = out.flush();
            }
        }

        let task = tokio::spawn(async move {
            render_loop_with_extensions(
                activity_rx, status_rx, event_rx,
                widgets, shutdown.clone(),
            ).await;
        });

        Self {
            activity_tx, status_tx, event_tx: Some(event_tx),
            shutdown, task: Some(task),
        }
    }

    /// Get a clone of the event sender for extensions to use at runtime.
    pub fn event_sender(&self) -> Option<tokio::sync::mpsc::UnboundedSender<TuiEvent>> {
        self.event_tx.clone()
    }
}
```

### Example 2: Pre-Readline Keybinding Check (Idle)
```rust
// Source: CONTEXT.md D-05, crossterm 0.28.1 poll/read API
use crossterm::event::{self, Event, KeyEvent, poll};
use std::time::Duration;

fn check_idle_keybindings(
    keybindings: &[Keybinding],
) -> Option<String> {
    // Non-blocking check: is there a pending key event?
    if poll(Duration::ZERO).unwrap_or(false) {
        if let Ok(Event::Key(key_event)) = event::read() {
            // Only match Idle and Always bindings
            for binding in keybindings {
                if (binding.when == KeyContext::Idle || binding.when == KeyContext::Always)
                    && binding.key.code == key_event.code
                    && binding.key.modifiers == key_event.modifiers
                {
                    return Some(binding.action_name.clone());
                }
            }
            // Key didn't match any binding -- it's consumed and lost.
            // This is acceptable for modifier-key combos (Ctrl+X, etc.)
            // but problematic for regular characters. Consider restricting
            // Idle keybindings to modifier-key combos only.
        }
    }
    None
}
```

## Open Questions

1. **Crossterm poll/read thread safety with rustyline**
   - What we know: crossterm::event::poll and read are global operations on the terminal. rustyline also reads from the terminal during readline().
   - What's unclear: Whether calling poll(Duration::ZERO) just before rl.readline() can race with rustyline's initialization.
   - Recommendation: Test empirically. If races occur, limit Idle keybindings to the InFlight path only (where rustyline is not active).

2. **KeyContext::Always integration with ctrl-c state machine**
   - What we know: ctrl-c is handled via tokio::signal::ctrl_c() in the select! loop. Always bindings need to work both at prompt and in-flight.
   - What's unclear: Whether Always bindings should be checked in both the pre-readline path AND the select! loop, creating two dispatch points.
   - Recommendation: Yes, check in both paths. Document that Always bindings fire from whichever path is active. This matches the semantic intent.

3. **Extension lifecycle: when are extensions registered?**
   - What we know: CONTEXT.md says TuiHandle::new() accepts extensions or has register_extension().
   - What's unclear: Whether extensions can be registered after TuiHandle creation (dynamic registration).
   - Recommendation: Start with constructor-time registration only (Vec<Box<dyn TuiExtension>> passed to new()). Dynamic registration adds complexity (needs Arc<Mutex<Vec<...>>>) and no immediate use case.
