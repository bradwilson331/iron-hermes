# Phase 22.1 -- UI Design Contract (Terminal UI)

> Visual and interaction contract for TUI extension hooks. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Note:** This is a Rust terminal UI project using crossterm + colored for ANSI output. All "visual" specifications are expressed in terminal rows, columns, and ANSI color names -- not pixels, CSS, or web components.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Rust terminal UI -- no web frontend) |
| Preset | not applicable |
| Component library | none (crossterm 0.28.1 + colored crate) |
| Icon library | Unicode block characters (U+2588..U+2591 for scanner) |
| Font | User's terminal monospace font (not controlled by application) |

---

## Spacing Scale (Terminal Rows & Columns)

This phase uses terminal row counts, not pixel spacing. All values are integer row/column counts.

| Token | Value | Usage |
|-------|-------|-------|
| reserved-base | 3 rows | Existing bottom reserve: prompt row + scanner row + status bar row |
| widget-row | 1 row per widget.height unit | Each extension widget declares its height in rows |
| reserved-dynamic | 3 + N rows | Total reserve = base + sum of extension widget heights (non-StatusRight slots only) |
| slot-gap | 0 rows | No gap between adjacent slot regions -- widgets render contiguously |
| pill-separator | 3 chars (" . ") | Dot separator between status pills (space-dot-space), existing Phase 21 pattern |

Exceptions:
- StatusRight slot widgets are appended inline to the status bar (column-based, not row-based). They consume horizontal space, not vertical rows.
- Widget content is truncated to terminal width (columns) to prevent line wrapping. No horizontal scrolling.

## Typography

Not applicable. Terminal UI renders in the user's configured monospace font at the terminal's configured size. The application does not control font face, size, weight, or line height.

**Text conventions for extension-generated content:**
| Role | Convention |
|------|-----------|
| Widget content | Pre-rendered ANSI strings via `colored` crate. Extensions own their own formatting. |
| Command output | Plain text or ANSI-colored. Printed to stdout (scrollable content area). |
| Error messages | Red via `.red()` colorize method. |
| Hint/help text | Dimmed via `.dimmed()` colorize method. |

## Color

The existing Phase 21 TUI establishes a terminal ANSI color palette. Extensions MUST use these same colors for visual consistency.

| Role | ANSI Color | Usage |
|------|-----------|-------|
| Dominant surface | Terminal default background | Scrollable content area (user's terminal theme controls this) |
| Scanner lit cell | bright_cyan | Knight Rider scanner active cell (distance 0) |
| Scanner trail | cyan, cyan.dimmed | Scanner trailing cells (distance 1, 2) |
| Scanner background | dimmed | Scanner inactive cells (distance 3+) |
| Status pill 1 | cyan | First status pill (mode) |
| Status pill 2 | magenta | Second status pill (model) |
| Status pill 3 | green | Third status pill (provider) |
| Status pill 4 | yellow | Fourth status pill (tokens) |
| Status pill 5+ | dimmed | Fifth+ pills and hint text |
| In-flight label | dimmed | "Streaming" / "Running:" text |
| Tool name | yellow | Tool name during ToolCall activity |
| Error output | red | Command errors, extension errors |
| Success output | green | Command success confirmations |

**Extension color rules:**
- Extensions adding StatusRight pills inherit the `rotate_pill_colors` rotation (continue from pill index 4+).
- AboveStatus and BelowScanner widget content uses extension-chosen colors via the `colored` crate. No restriction beyond: do not use bright_cyan (reserved for scanner lit cell).
- Style overrides via `style_overrides()` use `HashMap<String, String>` where keys are style slot names and values are ANSI color names from the `colored` crate palette.

**Style slot names for overrides:**

| Slot Name | Default | What It Controls |
|-----------|---------|-----------------|
| `scanner.lit` | bright_cyan | Scanner lit cell color |
| `scanner.trail` | cyan | Scanner trailing cell color |
| `scanner.bg` | dimmed | Scanner background cell color |
| `status.separator` | dimmed | Dot separator between pills |
| `status.hint` | dimmed | Hint text at end of status bar |

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Not applicable (extension hooks are an API, not a user-facing feature with a CTA button) |
| Empty state -- no extensions registered | No visible change. TUI renders identically to Phase 21 (3-row reserve, no extension widgets). Zero-extension is the default state. |
| Empty state -- extension widget with no content | Widget slot is skipped (0 height contribution). Extension's `widgets()` returns empty Vec. |
| Error -- extension panic | Log: `"tui: extension '{name}' panicked in {method}() -- skipping"` (tracing::warn). User sees no disruption; faulting extension is silently bypassed for that frame/dispatch. |
| Error -- widget ID collision | Log: `"tui: widget id '{ext_name}:{widget_id}' conflicts with existing widget -- ignoring"` (tracing::warn at registration time). Later registration is dropped. |
| Error -- unknown command | `"Unknown command: /{cmd}. Type /help for available commands."` (printed to stdout) |
| Help -- command list header | `"Available commands:"` |
| Help -- extension command format | `"  /{cmd}  {description}"` (2-space indent, 2-space gap before description) |
| Help -- keybinding list header | `"Keybindings:"` |
| Help -- keybinding format | `"  {key_display}  {description}  ({context})"` where context is "idle", "in-flight", or "always" |
| Command success (Silent) | No output printed. Used for commands that produce side effects only (e.g., toggling a widget). |
| Command success (Handled) | Output string printed to stdout as-is. Extension controls formatting. |
| Command error | `"Error: {message}"` printed to stderr in red. |

## Layout Slots

Three named slots where extensions place widgets. This is the "component inventory" for the terminal UI.

| Slot | Position | Orientation | Rendering Rule |
|------|----------|-------------|---------------|
| AboveStatus | Between scrollable content and scanner row | Vertical (stacked rows) | Rendered bottom-up from `rows - 3 - N` where N is cumulative widget height. Each widget occupies `widget.height` rows. Content is cleared and redrawn each frame. |
| BelowScanner | Between scanner row and status bar row | Vertical (stacked rows) | Rendered top-down from `scanner_row + 1`. Shifts status bar down. |
| StatusRight | Appended to end of status bar | Horizontal (inline) | Widget content appended after last pill with pill separator. Content truncated to fit remaining terminal width. Height is always 1 (ignored). |

**DECSTBM adjustment formula:**
```
reserved_rows = 3 + sum(AboveStatus widget heights) + sum(BelowScanner widget heights)
scroll_end = terminal_rows.saturating_sub(reserved_rows)
DECSTBM escape: \x1b[1;{scroll_end}r
```

**Widget struct contract:**
```
Widget {
    id: String,        // Unique within extension; auto-prefixed to "{ext_name}:{id}"
    content: String,   // Pre-rendered ANSI-colored string. One line per row.
    height: u16,       // Number of terminal rows this widget occupies. Min 1.
}
```

**Content rendering rules:**
- Multi-row widgets: content is split on `\n`. Each line is rendered on a separate row.
- Lines longer than terminal width are truncated (no wrapping).
- Each widget row is `Clear(ClearType::CurrentLine)` before `Print(content_line)` to prevent stale glyphs.

## Interaction Contract

### Keybinding Dispatch

| Context | When Active | Dispatch Point | Integration |
|---------|-------------|---------------|-------------|
| Idle | At the REPL prompt, between readline calls | Pre-readline: `crossterm::event::poll(Duration::ZERO)` + `read()` before `rl.readline()` | Only modifier-key combos (Ctrl+X, Alt+X) to avoid consuming regular characters from rustyline |
| InFlight | During agent turn execution | Inside `tokio::select!` loop alongside ctrl-c handler | Full key events available since rustyline is not active |
| Always | Both Idle and InFlight | Checked in both dispatch points | Same modifier-key restriction as Idle applies (superset of both paths) |

**Consumed-key policy:** If `poll(Duration::ZERO)` returns true and `read()` returns a KeyEvent that does NOT match any registered Idle/Always binding, the event is consumed and lost. To minimize data loss, Idle keybindings MUST be restricted to modifier-key combinations (Ctrl+key, Alt+key) that rustyline would not normally process.

### Command Dispatch Chain

```
User types "/{cmd} {args...}"
  |
  v
For each extension (in registration order):
  ext.process_command(cmd, args) -> Option<CommandResult>
    |
    Some(result) -> dispatch complete, use result
    None -> try next extension
  |
  v
All extensions returned None:
  Core fallback stub:
    "quit" | "exit" | "q" -> exit chat
    "clear" -> clear conversation
    "status" -> show status
    "help" -> format_help (includes extension commands)
    _ -> "Unknown command: /{cmd}"
```
