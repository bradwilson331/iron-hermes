# Phase 26.3: chromiumoxide-user-data-dir — Research

**Researched:** 2026-05-05
**Domain:** chromiumoxide browser configuration / Rust codebase wiring
**Confidence:** HIGH

---

## Summary

Phase 26.3 is an urgent surgical fix: every call to `BrowserSession::spawn()` in
`crates/ironhermes-tools/src/browser_session.rs` builds a `CdpBrowserConfig` without
setting `user_data_dir`. When `user_data_dir` is `None`, chromiumoxide 0.9.1 falls
back to a process-scoped temporary directory (effectively `/tmp/chromiumoxide-runner`
or equivalent). That directory is destroyed when the browser closes, so **no
cookies, localStorage, IndexedDB, or login sessions survive across agent turns**.
This prevents any real-world authenticated browser automation (the core use case for
the browser toolset).

The fix is a 4-file change: add `user_data_dir: Option<String>` to `BrowserConfig`
in `ironhermes-core`, wire it into the `CdpBrowserConfig` builder in
`ironhermes-tools/src/browser_session.rs`, add `browser-profile` to
`ensure_home_dirs()` in the CLI, and add a unit test confirming the field round-trips
through serde and is passed to chromiumoxide.

`ironagent-tools-api` contains a byte-identical copy of `browser_session.rs` and must
also be updated, but neither `ironhermes-agent` nor `iron_hermes_ui` depends on it —
both depend on `ironhermes-tools`. The `ironagent-tools-api` crate is in the workspace
but is not consumed by any primary binary as of Phase 26.3.

**Primary recommendation:** Add `user_data_dir: Option<String>` to `BrowserConfig`
with default `None`; set it to `Some($HERMES_HOME/browser-profile)` inside
`BrowserSession::spawn()` when `config.user_data_dir` is `None` (computed from
`get_hermes_home()`). The operator may override via config.yaml
`browser.user_data_dir`. The path is created by `ensure_home_dirs()`.

### Pattern 1: Resolving the effective user_data_dir in spawn()

The project uses a "env > config > computed default" precedence pattern throughout
(see `find_chromium_binary` for the exact idiom). For `user_data_dir`, the pattern
is simpler — only config or computed default:

```rust
// Source: adapted from find_chromium_binary precedence pattern in browser_session.rs
use ironhermes_core::constants::get_hermes_home;

let user_data_dir: PathBuf = match config.user_data_dir.as_deref() {
    Some(p) if !p.is_empty() => PathBuf::from(p),
    _ => get_hermes_home().join("browser-profile"),
};

let cdp_cfg = builder
    .user_data_dir(&user_data_dir)
    // ... existing args
    .build()?;
```

`std::fs::create_dir_all` is NOT called in `spawn()` — chromiumoxide creates the
directory itself if it does not exist. `ensure_home_dirs()` pre-creates it so the
directory is visible even before the first browser spawn.

### Pattern 2: Adding field to BrowserConfig

Following the Phase 25.1 D-18 pattern (every BrowserConfig field has `#[serde(default)]`):

```rust
// Source: crates/ironhermes-core/src/config.rs BrowserConfig
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(default)]
pub struct BrowserConfig {
    pub headed: bool,
    pub no_sandbox: bool,
    pub allowed_domains: Vec<String>,
    pub allowed_schemes: Vec<String>,
    pub chromium_path: Option<String>,
    pub timeout_seconds: u64,
    /// Phase 26.3: persistent browser profile directory.
    /// None = use $HERMES_HOME/browser-profile (default).
    /// Set explicitly to override (e.g., "/tmp/ephemeral-profile" for stateless browsing).
    pub user_data_dir: Option<String>,
}

impl Default for BrowserConfig {
    fn default() -> Self {
        Self {
            // ... existing defaults unchanged
            user_data_dir: None,  // None = use HERMES_HOME/browser-profile
        }
    }
}
```

`None` is the serde default; pre-26.3 `config.yaml` files that omit `browser.user_data_dir`
parse cleanly.

### Pattern 3: ensure_home_dirs() extension

```rust
// Source: crates/ironhermes-cli/src/main.rs ensure_home_dirs()
fn ensure_home_dirs() -> Result<()> {
    let home = ironhermes_core::get_hermes_home();
    for sub in &[
        "cron",
        "sessions",
        "logs",
        "hooks",
        "memories",
        "skills",
        "workspace",
        "subagent-transcripts",
        "browser-profile",  // Phase 26.3: persistent chromium profile
    ] {
        std::fs::create_dir_all(home.join(sub))
            .with_context(|| format!("Failed to create {}/{}", home.display(), sub))?;
    }
    Ok(())
}
```

### Anti-Patterns to Avoid

- **Calling `std::fs::create_dir_all` inside `spawn()`:** `ensure_home_dirs()` owns
  directory scaffolding; spawn() should not duplicate that responsibility. Chromiumoxide
  also creates the dir itself when it does not exist.
- **Hardcoding `/tmp/chromiumoxide-runner`:** The default should always be computed
  from `get_hermes_home()` so profile isolation (Phase 24 profiles) works correctly —
  each profile gets its own `browser-profile` subdir under its HERMES_HOME.
- **Adding `user_data_dir` path validation to `config_validate.rs`:** The existing
  `chromium_path` field is not validated at load time either (it is checked at spawn
  time). Consistent policy: path fields are validated at use, not at config load.
- **Adding a wizard prompt for `user_data_dir`:** Browser toolset is disabled by default
  (`tools.toolsets.browser.enabled = false` per Phase 25.1 D-04). Wizard prompts for
  opt-in features add friction without benefit. Operator sets `browser.user_data_dir`
  in config.yaml only if they want to override the sensible default.
- **Updating `ironagent-tools-api` only without `ironhermes-tools`:** The live
  dependency is `ironhermes-tools`. Updating only the API crate fixes nothing.

## Common Pitfalls

### Pitfall 1: Profile Sharing Race Condition (Multiple Processes)
**What goes wrong:** If two ironhermes processes (e.g., gateway + CLI) both try to
open the same `$HERMES_HOME/browser-profile`, Chrome will refuse to launch the second
instance with a "profile already in use" error.
**Why it happens:** Chrome's profile lock file (`SingletonLock`) prevents concurrent
access.
**How to avoid:** For Phase 26.3, accept this limitation and document it. The typical
deployment has one agent process at a time. If concurrent access becomes a real problem
in a future phase, use profile per-session suffixes. Do NOT attempt to solve this in
26.3.
**Warning signs:** `chromium launch failed: ... could not create singleton lock` in
logs when gateway and CLI run simultaneously.

### Pitfall 2: Profile directory belongs to wrong HERMES_HOME when profiles are used
**What goes wrong:** When `--profile work` pivots HERMES_HOME to
`~/.ironhermes/profiles/work/`, the computed default `get_hermes_home().join("browser-profile")`
correctly returns `~/.ironhermes/profiles/work/browser-profile`. This is the desired
behavior — but only if `get_hermes_home()` is called AFTER the profile pivot in main.rs.
**Why it happens:** `get_hermes_home()` reads `IRONHERMES_HOME` env var which is set
by the profile pivot. If `spawn()` captured the home at import time (e.g., via a
lazy_static), it would get the wrong dir.
**How to avoid:** Call `get_hermes_home()` inside `spawn()` at call time, not at
startup. The current implementation already does this (no lazy_static in browser_session.rs).

### Pitfall 3: user_data_dir vs chromium_path — different semantics
**What goes wrong:** Treating `user_data_dir` like `chromium_path` (authoritative when
set, no fallthrough). For `chromium_path`, explicit `None` means autodiscover. For
`user_data_dir`, explicit `None` should mean "use the sensible default" (HERMES_HOME/browser-profile),
NOT "let chromiumoxide pick /tmp".
**How to avoid:** The resolution logic in `spawn()` MUST always produce a concrete path
(not pass `None` to chromiumoxide). See Pattern 1 above.

### Pitfall 4: Tests that assume ephemeral profile
**What goes wrong:** Existing `browser_integration.rs` tests navigate to wiremock pages
and check page content. If the tests run with a persistent profile, a cached login or
cookie from a prior test run could cause unexpected state.
**How to avoid:** Integration tests should set `IRONHERMES_BROWSER_TEST_DISABLE` or
use an explicit temp dir for the profile when testing. The existing test suite already
uses `IRONHERMES_BROWSER_TEST_DISABLE` to skip when chromium is unavailable — same
escape hatch can gate profile override in tests.

```rust
// Source: pattern derived from existing browser_session.rs + chromiumoxide docs
pub async fn spawn(config: &BrowserConfig) -> anyhow::Result<Self> {
    use ironhermes_core::constants::get_hermes_home;

    let binary = find_chromium_binary(config.chromium_path.as_deref())
        .ok_or_else(|| anyhow::anyhow!("Chromium/Chrome binary not found. ..."))?;

    debug!(binary = %binary.display(), headed = config.headed, no_sandbox = config.no_sandbox,
           "Phase 25.1: spawning chromium");

    // Phase 26.3: resolve persistent profile directory.
    let user_data_dir: std::path::PathBuf = match config.user_data_dir.as_deref() {
        Some(p) if !p.is_empty() => std::path::PathBuf::from(p),
        _ => get_hermes_home().join("browser-profile"),
    };

    let mut builder = CdpBrowserConfig::builder()
        .chrome_executable(binary)
        .launch_timeout(Duration::from_secs(config.timeout_seconds))
        .user_data_dir(&user_data_dir);  // Phase 26.3

    // ... rest of builder unchanged
```

## Open Questions

1. **Should `ensure_home_dirs()` in the agent's `app_runtime_factory.rs` also add `browser-profile`?**
   - What we know: `ensure_home_dirs()` is in `ironhermes-cli/src/main.rs` and called at CLI startup. The agent's factory (`ironhermes-agent/src/app_runtime_factory.rs`) does not call `ensure_home_dirs()`.
   - What's unclear: Whether gateway mode (which does NOT go through the CLI `main.rs`) also scaffolds home dirs.
   - Recommendation: chromiumoxide creates the directory if absent, so this is not blocking. The CLI fix is sufficient. A future phase can add scaffolding to the gateway path if needed.

2. **Should the integration tests be updated to use a temp profile dir?**
   - What we know: The existing `browser_integration.rs` tests skip when chromium is unavailable. They do not explicitly set `user_data_dir`.
   - What's unclear: Whether test isolation requires an ephemeral profile.
   - Recommendation: Update integration tests to use `tempfile::tempdir()` as the `user_data_dir` in the test `BrowserConfig`. This ensures tests don't accumulate stale cookies across runs.
