# Phase 26.3 Plan 01: Persistent chromiumoxide user_data_dir Summary

**One-liner:** Add `user_data_dir: Option<String>` to `BrowserConfig` and wire it into both `BrowserSession::spawn()` copies, defaulting to `$HERMES_HOME/browser-profile`, closing the cookie-loss bug that destroyed login state on every `browser_close`.

## What Was Built

Surgical 4-file change that closes the persistent browser profile bug structurally:

1. **`crates/ironhermes-core/src/config.rs`** — Added `pub user_data_dir: Option<String>` as the last field of `BrowserConfig` (line 546 post-format). Default impl sets it to `None`. Extended `browser_config_default_matches_d18` to assert the new default. Added 3 new unit tests (UDD-01, UDD-02, UDD-03). File grew by 45 lines.

2. **`crates/ironhermes-tools/src/browser_session.rs`** — Inserted user_data_dir resolution block in `spawn()` between `.launch_timeout(...)` and `if config.headed`. Resolution: `Some(non-empty)` → `PathBuf::from(p)`; `None` or `Some("")` → `get_hermes_home().join("browser-profile")`. `builder.user_data_dir(&user_data_dir)` always called with a concrete `PathBuf`. File grew by 9 lines.

3. **`crates/ironagent-tools-api/src/browser_session.rs`** — Identical change applied; zero drift verified by `diff`.

4. **`crates/ironhermes-cli/src/main.rs`** — Appended `"browser-profile"` as 9th entry in `ensure_home_dirs()` subdir array. Added `home_dirs_includes_browser_profile` test (UDD-04). File grew by 20 lines.
