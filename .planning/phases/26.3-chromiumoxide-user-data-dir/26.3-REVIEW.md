# Phase 26.3: Code Review Report

**Reviewed:** 2026-05-05T15:13:57Z
**Depth:** standard
**Files Reviewed:** 4
**Status:** issues_found

## Summary

This phase adds a `user_data_dir` field to `BrowserConfig` (config.rs), wires it into `BrowserSession::spawn()` in two parallel crates (`ironhermes-tools` and `ironagent-tools-api`), and scaffolds `browser-profile/` in `ensure_home_dirs()` (main.rs). The two `browser_session.rs` files are byte-for-byte identical — this duplication is a structural concern but is pre-existing. The Phase 26.3 logic itself is straightforward.

Two blockers were found: a security bypass via the `IRONHERMES_BROWSER_TEST_DISABLE` escape hatch that activates on any set value (including `0`), and a path traversal vulnerability where operator-supplied `user_data_dir` values are never sanitized before being passed to chromium as `--user-data-dir`. Three warnings cover a missing `user_data_dir` creation step, an ambiguous scheme-extraction edge case carried over from pre-existing code, and a duplicated test with identical assertions.

## Critical Issues

### CR-01: `IRONHERMES_BROWSER_TEST_DISABLE` escape hatch activates on any value, including `"0"`

**File:** `crates/ironhermes-tools/src/browser_session.rs:168`
**Also:** `crates/ironagent-tools-api/src/browser_session.rs:168`

**Issue:** The production escape hatch that disables chromium discovery checks only whether the variable is *set*, not whether it equals `"1"`. `std::env::var("IRONHERMES_BROWSER_TEST_DISABLE").is_ok()` returns `true` for any exported value — including `IRONHERMES_BROWSER_TEST_DISABLE=0`, `=false`, or `=no`. Any operator who exports this variable with a falsy value (e.g., in a shell profile that sets it to `0` to "disable the disable") will silently find their browser toolset non-functional in production: `find_chromium_binary` always returns `None`, causing every `browser_navigate` call to fail with a misleading "Chromium binary not found" error even when Chrome is installed. The comment says "MUST NOT be set in production" but this is only a comment-level guard.

**Fix:**
```rust
// Before
if std::env::var("IRONHERMES_BROWSER_TEST_DISABLE").is_ok() {
    return None;
}

// After — only disable when the value is exactly "1" (matches the documented contract)
if std::env::var("IRONHERMES_BROWSER_TEST_DISABLE").as_deref() == Ok("1") {
    return None;
}
```

Apply the same fix to `crates/ironagent-tools-api/src/browser_session.rs:168`.

### CR-02: Operator-supplied `user_data_dir` path is not sanitized — path traversal into arbitrary locations

**File:** `crates/ironhermes-tools/src/browser_session.rs:65-69`
**Also:** `crates/ironagent-tools-api/src/browser_session.rs:65-69`

**Issue:** The operator-supplied `browser.user_data_dir` string is converted directly to a `PathBuf` and passed to `builder.user_data_dir()` with no validation:

```rust
let user_data_dir: std::path::PathBuf = match config.user_data_dir.as_deref() {
    Some(p) if !p.is_empty() => std::path::PathBuf::from(p),
    _ => ironhermes_core::get_hermes_home().join("browser-profile"),
};
builder = builder.user_data_dir(&user_data_dir);
```

A value such as `../../../etc` or `/root/.ssh` is accepted silently. Chromium will create or write profile state (Cache, History, Cookies, Local Storage) at that arbitrary location. In a multi-tenant or containerised deployment where `config.yaml` is world-writable or operator-editable by untrusted users, this is a data-overwrite/exfiltration vector. At minimum the path should be resolved to an absolute path and either checked against `get_hermes_home()` as a base, or emitted as an error when it is a relative path.

**Fix:**
```rust
let user_data_dir: std::path::PathBuf = match config.user_data_dir.as_deref() {
    Some(p) if !p.is_empty() => {
        let resolved = std::path::PathBuf::from(p);
        // Reject relative paths — they resolve relative to chromium's cwd, not HERMES_HOME.
        if resolved.is_relative() {
            anyhow::bail!(
                "browser.user_data_dir '{}' must be an absolute path",
                p
            );
        }
        resolved
    }
    _ => ironhermes_core::get_hermes_home().join("browser-profile"),
};
builder = builder.user_data_dir(&user_data_dir);
```

Apply the same fix to `crates/ironagent-tools-api/src/browser_session.rs`.

## Warnings

### WR-01: `browser-profile/` directory is scaffolded but not created before chromium spawns when the CLI is not the entry point

**File:** `crates/ironhermes-cli/src/main.rs:532-534`

**Issue:** `ensure_home_dirs()` is called only by the CLI `main()` function. The `browser-profile/` directory is scaffolded there (Phase 26.3 UDD-04). However, `BrowserSession::spawn()` lives in `ironhermes-tools` and `ironagent-tools-api` and is used by both the CLI *and* the gateway / embedded server paths. When the browser toolset is activated via the gateway or UI server (not the CLI), `ensure_home_dirs()` is never called. Chromium will attempt to use the default `$HERMES_HOME/browser-profile` directory but it may not exist, causing a launch failure. The fix belongs in `BrowserSession::spawn()` itself so directory creation is guaranteed regardless of entry point.

**Fix:**
```rust
// In BrowserSession::spawn(), before builder.user_data_dir():
if let Err(e) = std::fs::create_dir_all(&user_data_dir) {
    // Treat as non-fatal warning — chromium may still succeed if the dir
    // was created by a concurrent process, or fail with a more specific error.
    tracing::warn!(
        path = %user_data_dir.display(),
        error = %e,
        "Phase 26.3: failed to pre-create user_data_dir; chromium launch may fail"
    );
}
```

Apply the same fix to `crates/ironagent-tools-api/src/browser_session.rs`.

### WR-02: `extract_host` strips subdomains, allowing subdomain-bypass of the domain allowlist

**File:** `crates/ironhermes-tools/src/browser_session.rs:152-156`
**Also:** `crates/ironagent-tools-api/src/browser_session.rs:152-156`

**Issue:** `extract_host` returns the full hostname including subdomains (e.g. `evil.example.com`). The allowlist comparison at line 133 is an exact string match against each entry in `allowed_domains`. This is correct for exact-match semantics, but the doc comment at line 535 in `config.rs` says "Exact-match (no wildcards in v2.1)". The real risk is the inverse: if an operator intends `allowed_domains: ["example.com"]` to permit `www.example.com`, it will not — and if they add `example.com` hoping it is a suffix match, only bare `example.com` URLs (without a `www.`) will pass. This is a usability hazard that causes silent blocking of valid navigations. More critically, `"example.com"` in the allowlist will never match `"www.example.com"` but also will never accidentally allow `"evil-example.com"` — so the current exact-match logic is correct but the config comment at line 535 ("Exact-match (no wildcards)") should warn operators that subdomains are not covered by a bare apex entry. The mismatch between operator expectation and actual behavior will cause legitimate navigations to be silently blocked, not tracked as errors, making debugging difficult.

This is pre-existing behavior (not introduced in Phase 26.3), but Phase 26.3 is the right time to enforce clarity since `user_data_dir` adds a new persistent-profile surface that may encourage operators to tighten their allowlists.

**Fix:** Add a note in the `BrowserConfig::allowed_domains` doc comment:
```rust
/// D-15: domain allowlist for browser_navigate. Empty = allow all hosts.
/// Exact hostname match — subdomains are NOT covered by the apex entry.
/// To allow both example.com and www.example.com, list both explicitly:
///   allowed_domains: ["example.com", "www.example.com"]
pub allowed_domains: Vec<String>,
```

### WR-03: Two tests in each `browser_session.rs` are duplicates with identical setup and assertions

**File:** `crates/ironhermes-tools/src/browser_session.rs:263-308`
**Also:** `crates/ironagent-tools-api/src/browser_session.rs:263-308`

**Issue:** `find_chromium_binary_returns_none_when_browser_path_points_to_absent_file` (line 263) and `find_chromium_binary_returns_none_when_browser_path_set_to_absent_file_strict` (line 287) are identical in every meaningful way: same env setup, same call, same assertion. The only difference is the failure message string. This doubles test runtime cost (both hold the global `env_lock`) and makes it harder to determine which invariant each test is meant to protect. One of the two should be removed, or their distinct intent should be expressed through different inputs.

**Fix:** Remove `find_chromium_binary_returns_none_when_browser_path_points_to_absent_file` (the first, less-specifically named test) and keep the `_strict` variant which has the better comment. Apply to both crate copies.

## Info

### IN-01: `user_data_dir` stored as `Option<String>` rather than `Option<PathBuf>`

**File:** `crates/ironhermes-core/src/config.rs:546`

**Issue:** `user_data_dir` is typed as `Option<String>` while `chromium_path` (line 541) and `extra_paths` in `SkillsConfig` (line 755) use `PathBuf`/`Option<String>` respectively. Using `String` means callers must construct a `PathBuf` at each use site. Since the field's entire purpose is to hold a filesystem path, `Option<PathBuf>` would be more idiomatic and eliminate the repeated `std::path::PathBuf::from(p)` conversions in both `browser_session.rs` files. However, `String` does not cause a functional bug — this is a code-quality concern.

**Fix:** Change the field type to `Option<PathBuf>` and update the serde derive to use `#[serde(default)]` (already present on the struct). Callers would then use `config.user_data_dir.as_ref().map(PathBuf::as_path)` instead of `config.user_data_dir.as_deref()`.

### IN-02: Exact code duplication between `ironhermes-tools` and `ironagent-tools-api` `browser_session.rs`

**File:** `crates/ironhermes-tools/src/browser_session.rs` and `crates/ironagent-tools-api/src/browser_session.rs`

**Issue:** Both files are byte-for-byte identical (confirmed via `diff`). Any bug fixed in one file must be manually mirrored to the other — this review itself found that both require the same CR-01 and CR-02 fixes. This duplication is pre-existing and not introduced by Phase 26.3, but Phase 26.3 extended the shared logic (`user_data_dir` resolution) and the fixes must be applied twice. The codebase should consolidate this into a shared `ironhermes-browser-session` crate or re-export from one crate to the other.

**Fix:** Extract `browser_session.rs`, `find_chromium_binary`, and `extract_host` into a dedicated `ironhermes-browser` crate and depend on it from both `ironhermes-tools` and `ironagent-tools-api`. This eliminates the dual-maintenance burden and ensures security fixes propagate automatically.
