# Phase 27.1.4.1: gateway-fallback-gap - Context

**Gathered:** 2026-05-12
**Status:** Ready for planning

<domain>
## Phase Boundary

Close the provider fallback coverage gaps across the IronHermes agent execution paths. Three production `AgentLoop` construction sites — the gateway cron runner, the agent subagent runner, and the CLI batch runner — build `AgentLoop` without wiring `.with_fallback()`. When the main LLM provider is unavailable, these paths fail hard with no retry. A full-workspace audit confirmed the other execution paths (handler.rs main conversation, iron_hermes_ui agent path, CLI interactive, CLI TUI) have fallback wiring but all use a triple-nested `if let` that fails silently with no log when the fallback provider is misconfigured.

Deliverables:
1. **Extract a shared helper** — `wire_fallback_if_configured(agent, &resolver) -> AgentLoop` in `ironhermes-agent` that encapsulates the if-let chain + `tracing::warn!` and is used at all sites.
2. **Wire fallback at 3 new sites** — cron (`runner.rs`), subagent delegation (`subagent_runner.rs`), CLI batch (`batch/runner.rs`).
3. **Add warn! to existing silent sites** — `handler.rs` and `iron_hermes_ui/server/state.rs` both silently skip fallback on misconfiguration; replace their inline if-let chains with the shared helper.
4. **Config/docs consistency** — `cli-config.yaml.example` is missing `learning:` and `tools:` sections; README needs fallback configuration documented. Any config change must be reflected across all paths and docs.
5. **Invariant tests** — Static assertions for the 3 new wiring sites, modeled on `invariants_22_4.rs`.

</domain>

<decisions>
## Implementation Decisions

### D-12 Revisit
- **D-01:** The original D-12 decision ("cron uses fixed provider, NO fallback") was an **oversight**, not intentional design. The cron path should have the same fallback resilience as the main conversation path. Remove the "NO fallback" comment and wire it identically to `handler.rs:931-942`.

### Scope — All 3 Gaps
- **D-02:** All three missing sites are in scope for this phase:
  - `crates/ironhermes-gateway/src/runner.rs` — cron job execution (primary gap)
  - `crates/ironhermes-agent/src/subagent_runner.rs` — subagent delegation
  - `crates/ironhermes-cli/src/batch/runner.rs` — CLI batch runner
- **D-03:** The existing `handler.rs` and `iron_hermes_ui/server/state.rs` inline if-let blocks also get refactored to use the shared helper (adds warn! logging, reduces duplication).

### Shared Helper
- **D-04:** Extract `wire_fallback_if_configured(agent: AgentLoop, resolver: &ProviderResolver) -> AgentLoop` into `crates/ironhermes-agent/src/` (likely `any_client.rs` or a new `fallback.rs`). All 5 sites use this function. It:
  1. Calls `resolver.resolve_for_main().fallback_providers.first()`
  2. If present: attempts `build_client(resolver, fb_name, &fb_endpoint.default_model)`
  3. On success: chains `.with_fallback(fb_client)` and returns the agent
  4. On any failure (provider not found, client build error): emits `tracing::warn!("fallback provider '{}' could not be resolved — running without fallback", fb_name)` and returns the agent unchanged
- **D-05:** Fallback provider source for all new sites: `resolver.resolve_for_main().fallback_providers.first()` — same list as the main conversation path. No cron-specific or batch-specific fallback config.

### Failure Escalation
- **D-06:** When a cron job fails on the main provider AND the fallback (double failure): deliver the failure message to the cron job's configured delivery target. This is consistent with how `complete_job_run` handles agent errors today — no new escalation behavior is needed; the existing error path already delivers to the target.

### Test Coverage
- **D-07:** Add static invariant tests for all 3 new wiring sites, modeled on `invariants_22_4.rs:516`. Each test uses `include_str!` on the source file and asserts it contains `.with_fallback(`. Tests live in `crates/ironhermes-cli/tests/` (for batch) and a new or existing invariants file for gateway and agent.

### Config Consistency Rule (Phase-Wide Invariant)
- **D-08:** Any config change in this phase MUST be reflected across ALL of:
  - `crates/ironhermes-cli/src/` (CLI)
  - `crates/ironhermes-gateway/src/` (gateway)
  - `crates/iron_hermes_ui/src/` (UI server)
  - `cli-config.yaml.example`
  - `README.md`
  Planner must include a verification step confirming all paths are consistent.

### Config File Gaps
- **D-09:** `cli-config.yaml.example` is missing:
  - A `learning:` section documenting the keys that `setup_wizard.rs` writes (e.g., `learning.skill_generation_enabled`, others — researcher to diff `Config::default()` + wizard output against the example)
  - The `tools:` section needs verification/completion against what the setup wizard configures
  - The model section defaults should be reviewed — the active config currently uses `gemma4:latest` / `ollama` provider locally; the example should document both cloud (OpenRouter/Anthropic) and local (Ollama) configurations clearly
- **D-10:** Researcher should diff `Config::default()` serialized output + what `setup_wizard.rs::apply_minimum_viable_answers` writes against `cli-config.yaml.example` to find all missing/stale fields.

### README Documentation
- **D-11:** `README.md` must be updated to document:
  - The `fallback_providers` configuration field with a concrete working example (cloud primary + local Ollama fallback is the canonical use case)
  - Example commands for `hermes config set providers.openrouter.fallback_providers[0] local-llama`
  - Any new config fields added in this phase

### Claude's Discretion
- Whether to put `wire_fallback_if_configured` in `any_client.rs` (alongside `build_main_client`) or a new `fallback.rs` module — either is fine; prefer the location that minimizes pub re-exports needed.
- Whether the CLI batch site (`batch/runner.rs`) needs its own `ProviderResolver` construction or can reuse a passed-in resolver — planner should check the existing batch runner setup to determine the right pattern.
- Exact warn! message wording — follow the pattern already established by any existing warn! messages in `any_client.rs`.

## Existing Code Insights

### Reusable Assets
- `build_main_client(&resolver)` in `any_client.rs` — already imported in all 3 gap files; the new helper will live alongside it
- `build_client as build_provider_client` — already used in handler.rs and iron_hermes_ui for building the fallback client; new helper uses the same call
- `resolver.resolve_for_main().fallback_providers.first()` — the exact chain already working in handler.rs (line 934); new helper extracts this

### Established Patterns
- **Triple-nested if-let pattern** (current, fragile): `if let Some(fb_name) = ... { if let Some(fb_endpoint) = ... { if let Ok(fb_client) = ... { agent = agent.with_fallback(fb_client); } } }` — replace with the shared helper call
- **`tracing::warn!` for provider resolution failures** — used throughout `any_client.rs` and `engine_factory.rs`; follow the same format
- **Static invariant test pattern** (`invariants_22_4.rs:514-519`): `let source = include_str!("../../path/to/file.rs"); assert!(source.contains(".with_fallback("), "...")` — copy exactly
- **`AgentLoop::new(...).with_*()` builder chain** — builder returns `Self`, so `wire_fallback_if_configured` can accept and return `AgentLoop` by value

### Integration Points
- All 3 gap sites already import `build_main_client` and have a `ProviderResolver` in scope — the helper only needs `resolver: &ProviderResolver` as input
- `batch/runner.rs` — check whether it constructs its own `ProviderResolver` or receives one; this determines where `resolver` comes from for the helper call
- The `lib.rs` re-export in `ironhermes-agent` must be updated to expose `wire_fallback_if_configured` under the same pub path as `build_main_client`

</code_context>

<specifics>
## Specific Ideas

- **Ollama as the canonical fallback example**: The user runs `gemma4:latest` via Ollama locally as a fallback provider. The README and yaml.example should use a local Ollama custom provider as the primary example for `fallback_providers` (cloud primary → local fallback). This is the most common real-world use case.
- **D-12 comment removal**: When fixing `runner.rs`, remove the "D-12: cron uses fixed provider, NO fallback" comment entirely and replace it with a comment like "Wire fallback via shared helper — same as handler.rs main path."
- **invariants_22_4.rs line 516** is the exact test model: check that the `SOURCE` string contains `.with_fallback(` and use a descriptive failure message naming the PROV-07 requirement.

</specifics>

<deferred>
## Deferred Ideas

- **Cron-specific fallback_providers config field** — user chose to reuse `main_endpoint.fallback_providers` rather than add a separate cron-specific field. If cron needs a different provider later, add `cron.fallback_provider` in a future phase.
- **Per-subagent fallback config** — subagents inherit the main resolver's fallback for now. Independent fallback routing per child agent is a future capability.
- **CLI batch fallback per-worker isolation** — batch runs parallel agents; this phase wires the same fallback to each worker. Finer-grained per-worker provider assignment is out of scope.
