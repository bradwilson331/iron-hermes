# Phase 27.1.4.1: Code Review Report

**Reviewed:** 2026-05-12
**Depth:** standard
**Files Reviewed:** 12
**Status:** issues_found

## Summary

The phase adds a single shared helper `wire_fallback_if_configured(agent, resolver) -> AgentLoop`
in `ironhermes-agent/src/any_client.rs` (re-exported from `lib.rs`), wires it at three production
`AgentLoop` construction sites that previously had no fallback (gateway cron `runner.rs`, agent
`subagent_runner.rs`, CLI `batch/runner.rs`), and refactors two pre-existing inline triple-nested
`if let` fallback blocks (`handler.rs`, `iron_hermes_ui/src/server/state.rs`) onto the same helper.

The refactor is **behaviorally faithful**: the helper reproduces the original `resolve_for_main()
-> fallback_providers.first() -> resolve() -> build_client(default_model) -> with_fallback()` chain
exactly, and additionally upgrades the previously-silent failure paths (provider name not found,
client build error) to emit `tracing::warn!`. No correctness regression, security gap, or data-loss
risk was found. The `mut`/shadowing pattern is correct at all five call sites, and the helper
returning `AgentLoop` by value composes cleanly with the surrounding `.with_*()` builders (order is
irrelevant since `with_fallback` only sets a field).

The notable defects are in the **test coverage** introduced for this phase: the three new
`invariants_27_1_4_1.rs` files are static-grep gates that match `".with_fallback("`, a string that
appears in the touched production files **only inside a trailing `//` comment** — not in any executed
code. The gates therefore do not verify the wiring they claim to lock, and the new helper itself has
zero behavioral test coverage.

## Warnings

### WR-01: Static-grep regression gates match a comment, not the actual wiring

**Files:**
- `crates/ironhermes-agent/tests/invariants_27_1_4_1.rs:9-15`
- `crates/ironhermes-cli/tests/invariants_27_1_4_1.rs:9-15`
- `crates/ironhermes-gateway/tests/invariants_27_1_4_1.rs:9-15`

**Issue:** All three new invariant tests assert `SOURCE.contains(".with_fallback(")`. But the
production code in the three corresponding source files (`subagent_runner.rs:210`,
`batch/runner.rs:258`, `gateway/runner.rs:1144`) calls `wire_fallback_if_configured(agent, ...)`
— the literal string `.with_fallback(` appears **only** in the trailing
`// chains .with_fallback() via the shared helper — PROV-07` comment on each of those lines. As a
result:
- The gate passes for the wrong reason (it is matching a comment).
- The gate would still pass if someone deleted the actual `wire_fallback_if_configured(...)` call
  but left the comment behind — i.e. it does not catch the regression it exists to catch.
- The gate would *fail* if someone tidied up the comment while leaving correct code in place — a
  false positive.

A static-grep gate that does not gate the thing it documents is worse than no gate: it provides
false assurance. (Note: `handler.rs` and `state.rs` happen to still contain a real `.with_fallback`
elsewhere via `AgentLoop` builders / tests, so they are partly insulated, but the three *new* gates
above are not.)

**Fix:** Match the actual wiring symbol used by the production code, e.g.:
```rust
assert!(
    SUBAGENT_SOURCE.contains("wire_fallback_if_configured("),
    "PROV-07: subagent_runner.rs must call wire_fallback_if_configured() on the child \
     AgentLoop so provider fallback fires on primary model failure. See phase 27.1.4.1."
);
```
Apply the same change to all three files (and update the assertion message). Optionally also keep a
check on `crates/ironhermes-agent/src/any_client.rs` that the helper still calls `.with_fallback(`.

### WR-02: New `wire_fallback_if_configured` helper has no behavioral test coverage

**File:** `crates/ironhermes-agent/src/any_client.rs:173-200`

**Issue:** This phase introduces a new public function that is now the single point of fallback
wiring for five call sites, yet `any_client.rs`'s `#[cfg(test)] mod tests` adds no test for it, and
the only tests added by the phase are the three (broken — see WR-01) static-grep gates. The helper
has at least three distinct branches worth covering:
1. `fallback_providers` empty → agent returned unchanged (no fallback set).
2. `fallback_providers.first()` names an unknown provider → `warn!` emitted, agent unchanged.
3. `fallback_providers.first()` resolves and builds → `agent.with_fallback(...)` chained.
None of these are exercised. A regression that, say, swapped `resolve_for_main()` for `resolve()`
with a wrong arg, or dropped the `.with_fallback()` chain, would compile and ship silently.

**Fix:** Add unit tests in `any_client.rs::tests`. `AgentLoop` already exposes the inspection
points used by `fallback_tests` in `agent_loop.rs` (`fallback_client.is_none()` etc.) — build a
`Config` with a `custom_providers` entry, set `providers.<main>.fallback_providers`, build a
`ProviderResolver`, call `wire_fallback_if_configured`, and assert the returned `AgentLoop` has a
fallback client; then a second test with no fallback configured asserts it stays `None`.

## Info

### IN-01: README claims `fallback_providers` is "tried in order" but only `.first()` is honored

**File:** `README.md:109-110` (also `cli-config.yaml.example:253`)

**Issue:** The README section updated by this phase says: *"The `fallback_providers` list is tried
in order. The swap is one-shot per session; if the fallback also fails the error is returned to the
caller."* But the helper (`any_client.rs:178`) reads only `main_endpoint.fallback_providers.first()`,
and `AgentLoop` holds exactly one `fallback_client` (`agent_loop.rs:155`) with one-shot activation —
so a `fallback_providers` list of length ≥ 2 silently ignores every entry after the first. This is a
pre-existing limitation, not introduced here, but the phase's README edits restate and reinforce the
inaccurate "tried in order" claim. The `cli-config.yaml.example` comment ("one-shot swap on
401/403/404/429/5xx") is more accurate but still uses the plural `fallback_providers: ["local-llama"]`
shape without noting only the first is used.

**Fix:** Either (a) clarify the docs — "only the first entry of `fallback_providers` is currently
used; a multi-tier cascade is not yet implemented", or (b) file a follow-up to actually iterate the
list. At minimum, drop the "tried in order" wording from `README.md` until it is true.

### IN-02: Redundant double resolve of the fallback endpoint

**File:** `crates/ironhermes-agent/src/any_client.rs:178-182`

**Issue:** `wire_fallback_if_configured` calls `resolver.resolve(fb_name)` to obtain
`fb_endpoint.default_model`, then calls `build_client(resolver, fb_name, &fb_endpoint.default_model)`
— and `build_client` (`any_client.rs:142-147`) calls `resolver.resolve(provider)` a second time on
the same name. Harmless (resolution is a cheap `HashMap::get`) but the double lookup is dead weight
and slightly obscures intent.

**Fix:** Either call `AnyClient::from_endpoint(fb_endpoint)` directly with the already-resolved
`&ResolvedEndpoint` (it already defaults `model` to `default_model`), or pass the resolved endpoint
into a model-aware constructor. Minor — leave if you prefer the symmetry with the other `build_*`
helpers.

### IN-03: `wire_fallback_if_configured` will panic if called with a resolver lacking a main provider

**File:** `crates/ironhermes-agent/src/any_client.rs:177`

**Issue:** The helper is `pub` and unconditionally calls `resolver.resolve_for_main()`, which
`panic!`s when the main provider is absent (`provider.rs:414-421`). Every current call site passes a
resolver built via `ProviderResolver::build`, which validates the main provider exists, so this
cannot fire today — but the doc comment on the new public fn does not mention the panic precondition,
and a future caller could trip it. Pre-existing pattern (`resolve_for_main` panics by design), noted
only for documentation completeness.

**Fix:** Add a `# Panics` line to the doc comment: "Panics if the resolver has no main provider —
construct the resolver via `ProviderResolver::build` (which validates this) before calling."

### IN-04: `cli-config.yaml.example` documents `learning:` keys that are not part of the typed `Config`

**File:** `cli-config.yaml.example:287-299`

**Issue:** The new `learning:` block is documented with a comment that correctly states these keys
"are NOT part of the typed `Config` struct (serde ignores them)" and only survive the raw-YAML
wizard path. That's accurate, but an operator who uncomments and edits `learning:` by hand (rather
than running the wizard) gets keys that the typed loader silently drops — there is no validation
feedback. The `tools:` block, by contrast, matches `ToolsConfig::default()` exactly and is fine.

**Fix:** Strengthen the comment to say "edit these only via `hermes setup` / the wizard — hand-edits
to `learning:` are silently ignored by the config loader", so the footgun is explicit. No code change.
