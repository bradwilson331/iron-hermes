## Summary

Phase 27.1.4.1 is a pure Rust refactor with no new dependencies, no new config fields, and no external services. It closes three production gaps where `AgentLoop` is constructed without `.with_fallback()`, and simultaneously improves two existing sites that have fallback but log nothing when provider resolution fails silently.

The work is entirely mechanical: extract one shared helper function in `ironhermes-agent`, call it at five sites (3 new + 2 refactor), add three static invariant tests, and update two documentation artifacts. The codebase already has all the raw materials — `build_client` (aliased as `build_provider_client`), `resolver.resolve_for_main().fallback_providers.first()`, and the `AgentLoop::with_fallback()` builder method are all in production use at `handler.rs:934-941` and `state.rs:181-188`.

The D-10 config gap audit (below) found one concrete gap: `cli-config.yaml.example` is missing the `learning:` section entirely. The `tools:` section is also missing from the example file. The README fallback section exists at line 55 and already has a `local-llama` example but lacks `hermes config set` commands and the Ollama-primary use case.

**Primary recommendation:** Place `wire_fallback_if_configured` in `any_client.rs` alongside `build_main_client` and `build_client`. This minimizes new re-exports (the function lives where the other factory fns already live), and `lib.rs` already re-exports the entire `any_client` factory group. The batch runner already has `resolver` in scope at line 238 as a local variable — no constructor changes needed.

### Recommended Module Location

`wire_fallback_if_configured` belongs in `crates/ironhermes-agent/src/any_client.rs`, placed after the existing factory functions (`build_client`, `build_main_client`, `build_role_client`). The function is public and must be re-exported in `crates/ironhermes-agent/src/lib.rs`.

### Pattern 1: The Shared Helper Function

**What:** A free function that wraps the existing triple-nested if-let pattern plus a `tracing::warn!` on misconfiguration.

**Signature** (derived from D-04 and current handler.rs pattern):
```rust
// Source: derived from crates/ironhermes-gateway/src/handler.rs:931-942 [VERIFIED]
pub fn wire_fallback_if_configured(
    mut agent: AgentLoop,
    resolver: &ProviderResolver,
) -> AgentLoop {
    let main_endpoint = resolver.resolve_for_main();
    if let Some(fb_name) = main_endpoint.fallback_providers.first() {
        if let Some(fb_endpoint) = resolver.resolve(fb_name) {
            match build_client(resolver, fb_name, &fb_endpoint.default_model) {
                Ok(fb_client) => {
                    agent = agent.with_fallback(fb_client);
                }
                Err(_) => {
                    tracing::warn!(
                        "fallback provider '{}' could not be resolved — running without fallback",
                        fb_name
                    );
                }
            }
        } else {
            tracing::warn!(
                "fallback provider '{}' could not be resolved — running without fallback",
                fb_name
            );
        }
    }
    agent
}
```

**When to use:** At every AgentLoop construction site that has a `ProviderResolver` in scope.

### Anti-Patterns to Avoid

- **Silent fallback failure:** The two existing sites (`handler.rs`, `state.rs`) drop the `Err` from `build_client` silently with no log. The shared helper must emit `tracing::warn!` in both the provider-not-found and client-build-failed branches.
- **Carrying primary model to fallback:** The pattern `build_client(resolver, fb_name, &fb_endpoint.default_model)` is correct — `default_model` comes from the fallback provider's own endpoint, not from the primary. This is documented in `cli-config.yaml.example` line 255-257.
- **Putting helper in a new crate:** All callers already depend on `ironhermes-agent`. A new crate would add dependency edges for no reason.

## Common Pitfalls

### Pitfall 1: warn! only on one branch, not both failure branches

**What goes wrong:** The helper warns when `build_client` returns `Err`, but silently skips when `resolver.resolve(fb_name)` returns `None` (provider name not found).
**Why it happens:** The triple-nested if-let pattern makes it easy to only add `warn!` at the deepest level.
**How to avoid:** The helper needs TWO warn! call sites: one for `resolver.resolve(fb_name) == None` and one for `build_client(...) == Err`.
**Warning signs:** Test the case where `fallback_providers: ["typo-provider"]` in config — should emit a warn! at startup.

### Pitfall 2: Using primary model string for fallback client

**What goes wrong:** `build_client(resolver, fb_name, &primary_endpoint.default_model)` — the model string comes from the primary, not the fallback.
**Why it happens:** The primary model is the one in scope when you start writing the helper.
**How to avoid:** The correct call is `build_client(resolver, fb_name, &fb_endpoint.default_model)` — get `fb_endpoint` from `resolver.resolve(fb_name)` first.
**Warning signs:** When Ollama is the fallback and the primary uses `anthropic/claude-sonnet-4` — Ollama will 404 with an Anthropic model name.

### Pitfall 3: Forgetting to add wire_fallback_if_configured to lib.rs re-exports

**What goes wrong:** Gateway and CLI callers get `unresolved import` errors because the function is `pub` in `any_client.rs` but not re-exported from the crate root.
**Why it happens:** `lib.rs` uses explicit `pub use any_client::{...}` lists, not a glob re-export.
**How to avoid:** Add `wire_fallback_if_configured` to the `pub use any_client::{...}` block in `lib.rs` as part of the same commit that adds the function.

### Pitfall 4: Cargo.toml dependency missing for gateway test file

**What goes wrong:** New `crates/ironhermes-gateway/tests/invariants_27_1_4_1.rs` added but the `[[test]]` table in `Cargo.toml` is not updated (if the gateway crate uses explicit test declarations rather than auto-discovery).
**Why it happens:** Some crates in the workspace use explicit `[[test]]` entries; others rely on Cargo's default test discovery.
**How to avoid:** Check each crate's `Cargo.toml` `[[test]]` entries before adding a new test file. If Cargo auto-discovers (default), no change needed.

### Pitfall 5: batch/runner.rs uses a cloned resolver, not a reference

**What goes wrong:** `resolver` is moved into the `join_set.spawn(async move {...})` closure after first use, making it unavailable for the `wire_fallback_if_configured` call.
**Why it happens:** `build_main_client(&resolver)` is called at line 238 INSIDE the spawn loop, then `client` is moved into the closure. `resolver` itself stays in the outer scope.
**How to avoid:** `wire_fallback_if_configured(agent, &resolver)` can be called INSIDE the `join_set.spawn` closure the same way `build_main_client(&resolver)` is already called there. `resolver` is borrowed, not moved, by `build_main_client`. This is fine.
