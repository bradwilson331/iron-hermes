Complete action list after Plan 01 (12 actions total):
1. walk          — CMD_MOVE#1#{x}#{y}#{speed}#0\n  (direction sets x/y; forward=y+25, backward=y-25, left=x-25, right=x+25; speed 2–10 clamped)
2. stop          — CMD_MOVE#0#0#0#0#0\n
3. read_battery  — CMD_POWER\n  (response: CMD_POWER#v1#v2\n → "Battery: v1V / v2V (OK|LOW)")
4. read_distance — CMD_SONIC\n  (response: CMD_SONIC#dist\n → "Distance: dist cm")
5. relax_servos  — CMD_RELAX\n  (toggle — two calls re-enable servos)
6. rotate        — CMD_MOVE#1#0#0#5#±10\n then sleep(|degrees|*20ms) then CMD_MOVE#0#0#0#0#0\n; ROTATE_MS_PER_DEGREE=20
7. head_pan      — CMD_HEAD#0#angle\n  (angle clamped ±90°; HEAD_PAN_MAX=90)
8. head_tilt     — CMD_HEAD#1#angle\n  (angle clamped ±90°; HEAD_TILT_MAX=90)
9. buzzer_on     — CMD_BUZZER#1\n
10. buzzer_off   — CMD_BUZZER#0\n
11. led          — CMD_LED#{r}#{g}#{b}\n  (r/g/b clamped 0–255; sets all 8 LEDs to solid color)
12. led_off      — CMD_LED#0\n  (sets led_mode=0 → dedicated off path; NOT CMD_LED#0#0#0\n)

Blocked commands (return Ok("Action '...' is blocked — not permitted via hexapod_tcp. Never send this command.")):
- CMD_CALIBRATION — would move servos to uncalibrated positions; risk of hardware damage
- CMD_SERVOPOWER  — would cut servo power mid-stance; robot drops; CMD_RELAX is the safe alternative
- CMD_LED_MOD modes 2–5 (chase, blink, breathing, rainbow) — asynchronous animation; may interfere with solid-color led control; blocked pending async-safe design

Connection details: HEXAPOD_IP env var (required), port 5002, one fresh TCP connection per call.

    ## Overview
    Brief description: the hexapod_tcp tool sends TCP commands to the Freenove hexapod robot at HEXAPOD_IP:5002. One fresh TCP connection per call. Fire-and-forget actions return "OK"; sensor actions return a formatted string. All errors return Ok("Error: ...") strings visible inline to the agent.

    ## Connection
    - Env var: HEXAPOD_IP (required — robot's IP address)
    - Port: 5002
    - Protocol: raw TCP, newline-terminated commands
    - Pattern: one connection per call, no persistent socket

    ## Actions and Wire Format
    Table with columns: Action | Parameters | Wire Command | Returns | Notes
    Populate all 12 rows accurately from the interfaces block above. Include the walk direction breakdown as a sub-note. For rotate, note the sleep-then-stop sequence.

    ## Parameter Ranges and Clamping
    - speed (walk): 2–10 (values outside range are clamped, not rejected)
    - degrees (rotate): negative values allowed (negative=counterclockwise); capped internally at ±3600 to prevent runaway rotation; ROTATE_MS_PER_DEGREE = 20 (calibration constant — tune on real robot)
    - angle (head_pan, head_tilt): ±90° (clamped; HEAD_PAN_MAX = HEAD_TILT_MAX = 90)
    - r, g, b (led): 0–255 per channel (clamped)

    ## Blocked Commands
    The following are explicitly blocked by hexapod_tcp and will return a blocked error string. Never attempt to call these:
    - CMD_CALIBRATION: moves servos to uncalibrated positions; risk of hardware damage
    - CMD_SERVOPOWER: cuts servo power mid-stance causing the robot to drop; use relax_servos instead
    - CMD_LED_MOD modes 2–5 (chase, blink, breathing, rainbow): asynchronous animations that may interfere with led control; blocked pending async-safe design

    ## Error Strings
    All errors are returned as Ok("Error: ...") — the agent sees them inline in the tool result:
    - Missing env var: "Error: HEXAPOD_IP env var not set — cannot connect to robot"
    - Connection failed: "Error: cannot connect to robot at {ip}:5002 — is HEXAPOD_IP set and the robot powered on?"
    - Read timeout (sensor commands): "Error: read timed out after 3s waiting for robot response"

    ## Session End Behavior
    When the IronHermes session ends, hexapod_tcp automatically sends stop then relax_servos (best-effort, errors silently swallowed). The robot will halt and relax even if the agent does not explicitly call stop.

    Do NOT include fenced code blocks with Rust source. Do NOT include content about IronHermes internals unrelated to controlling the robot. Keep the doc focused: an agent reading only this file should know exactly what to send and when.
