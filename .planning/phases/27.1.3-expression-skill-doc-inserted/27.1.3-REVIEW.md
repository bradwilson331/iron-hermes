## Summary

Phase 27.1.3 added two new actions (`led`, `led_off`) to `hexapod_tcp.rs` and created the `skills/hexapod/SKILL.md` protocol reference document. ## Warnings

### WR-01: Missing wire-assembly test — a regression in `CMD_LED` formatting would go undetected

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:453-457`

**Issue:** Tests 21–24 verify the constant value of `CMD_LED`, the clamping arithmetic in isolation, and the allowlist pass-through — but no test verifies the fully assembled LED wire string (e.g. `"CMD_LED#255#128#0\n"`). The format is built inline at line 457:

```rust
let wire = format!("{CMD_LED}#{r}#{g}#{b}\n");
```

If `CMD_LED` were accidentally changed to `"CMD_LED#"` (trailing `#`) or the format string were edited, the assembled wire would be malformed (`CMD_LED##255#128#0\n`) and no existing test would catch it. Tests 23 and 24 exercise the clamping arithmetic and the allowlist respectively but neither round-trips through `format!` with the constant.

**Fix:** Add a pure unit test that asserts the fully assembled wire string:

```rust
#[test]
fn test_led_wire_assembly() {
    let r = 255i64.clamp(0, 255);
    let g = 128i64.clamp(0, 255);
    let b = 0i64.clamp(0, 255);
    let wire = format!("{CMD_LED}#{r}#{g}#{b}\n");
    assert_eq!(wire, "CMD_LED#255#128#0\n");
    // Also verify black
    let wire_black = format!("{CMD_LED}#0#0#0\n");
    assert_eq!(wire_black, "CMD_LED#0#0#0\n");
}
```

### WR-02: Stale doc comment on `execute()` omits 7 of 12 allowed actions

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:291-295`

**Issue:** The doc comment on `execute()` reads:

```
/// Only `walk | stop | read_battery | read_distance | relax_servos` are forwarded to
/// the hardware.
```

The allowlist was expanded to 12 actions across phases 27.1.1–27.1.3 and this comment was not updated. It omits `rotate`, `head_pan`, `head_tilt`, `buzzer_on`, `buzzer_off`, `led`, and `led_off`. A future maintainer reading only the doc comment would have an incorrect picture of which actions are permitted.

**Fix:** Update the doc comment to enumerate all 12 allowed actions:

```rust
/// Only `walk | stop | read_battery | read_distance | relax_servos |
/// rotate | head_pan | head_tilt | buzzer_on | buzzer_off | led | led_off`
/// are forwarded to the hardware.
```

## Info

### IN-01: SKILL.md "8 LEDs" claim is an unverifiable hardware assertion

**File:** `skills/hexapod/SKILL.md:42`

**Issue:** The action table states: "sets all 8 LEDs to solid color". The `led` implementation sends `CMD_LED#{r}#{g}#{b}\n` — the LED count is not encoded in the wire format and is purely a hardware property of the specific Freenove model in use. No source in the codebase (constants, comments, research notes referenced by D-02/D-03) establishes "8" as the correct count. If the robot model changes or the claim is wrong, the skill doc misleads the agent.

**Fix:** Qualify the claim or source it:

```markdown
| led | r, g, b | `CMD_LED#{r}#{g}#{b}\n` | `"OK"` | sets all robot LEDs to solid color; each channel clamped to 0–255 |
```

Or add a parenthetical: "(8 LEDs on the Freenove hexapod)" so the reader knows it is model-specific, not protocol-defined.
