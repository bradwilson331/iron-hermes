**LED control added to hexapod_tcp: led(r,g,b) sets robot LEDs via CMD_LED#{R}#{G}#{B}\n with per-channel clamping; led_off sends CMD_LED#0\n (server's dedicated mode-0 off path)**

## Accomplishments

- Added `CMD_LED` ("CMD_LED") and `CMD_LED_OFF` ("CMD_LED#0\n") wire-protocol constants with doc comments distinguishing them from the naive color-zero form
- Implemented `led` execute arm: extracts r/g/b args as i64 (default 0), clamps each to 0..=255 independently, builds `format!("{CMD_LED}#{r}#{g}#{b}\n")`, fire-and-forget send
- Implemented `led_off` execute arm: sends `CMD_LED_OFF` constant unchanged — server mode-0 triggers `color_wipe([0,0,0])` dedicated off path
- Extended outer allowlist match arm with `| "led" | "led_off"` (continuous pattern, single arm)
- Updated schema action enum (12 values) and added r, g, b integer properties with minimum: 0, maximum: 255

## Decisions Made

- CMD_LED_OFF = `"CMD_LED#0\n"` (server mode-0) not `"CMD_LED#0#0#0\n"` (would set color to black instead of activating server's dedicated off path) — matches D-03 exactly; test 22 asserts both equality and inequality
- RGB clamped via `i64::clamp(0, 255)` (silent clamp, no error) — consistent with head_pan/head_tilt angle clamping pattern from Phase 27.1.2 D-04
