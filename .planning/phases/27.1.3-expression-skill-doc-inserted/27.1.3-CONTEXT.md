<decisions>
## Implementation Decisions

### LED Control (HXP-NAV-02)

- **D-01:** Two actions: `led` and `led_off` — follows the `buzzer_on`/`buzzer_off` pattern established in Phase 27.1.2. Do NOT collapse into a single `led(r,g,b)` where 0,0,0 = off.
- **D-02:** Wire format for `led`: `CMD_LED#{R}#{G}#{B}\n`. The Freenove server initializes `led_mode = '1'` (solid color) by default, so this command takes effect immediately — no mode-set preamble needed. `data[1..3]` = R, G, B (0–255 integers).
- **D-03:** Wire format for `led_off`: `CMD_LED#0\n`. This sets `led_mode = '0'` which triggers `color_wipe([0,0,0])` — the server's dedicated off path. Do NOT use `CMD_LED#0#0#0\n` (that would try to set color, not activate off mode).
- **D-04:** R, G, B params are integers 0–255. Add `minimum: 0`, `maximum: 255` constraints to the JSON schema properties. The tool should clamp or validate before sending — TBD by planner (follow the head angle clamping pattern from D-06/D-07 in 27.1.2).
- **D-05:** Both `led` and `led_off` are fire-and-forget. Return `"OK"` on success; return `Ok("Error: ...")` on TCP failure. No response expected from robot.

### Skill Doc (HXP-DOC-01)

- **D-06:** File path is `skills/hexapod/SKILL.md` — NOT `DESCRIPTION.md`. The requirement HXP-DOC-01 names `DESCRIPTION.md` to describe the content purpose, not the filename. IronHermes skill discovery looks for `SKILL.md`.
- **D-07:** Format: Full IronHermes SKILL.md with YAML frontmatter (`name`, `description`, `metadata.hermes.requires_toolsets: [robotics]`). Auto-activates when the robotics toolset is active — agent sees it without explicit load.
- **D-08:** Content scope is **protocol-complete**: all CMD_* wire strings, 12-action list with params and ranges, blocked commands with reason, calibration constants (`ROTATE_MS_PER_DEGREE = 20`), parameter ranges (head clamp ±90°, degree cap 3600, RGB 0–255). This doc replaces the inline `DESCRIPTION` const in `hexapod_tcp.rs` as the agent's canonical source of truth — the const can stay for tool schema display but the skill doc is the authoritative reference.
- **D-09:** Blocked commands section must list: `CMD_CALIBRATION`, `CMD_SERVOPOWER`, `CMD_LED_MOD` modes 2–5 (chase/blink/breathing/rainbow), with clear reason for each block (per D-16 from Phase 27.1.1).

### Claude's Discretion

- RGB validation approach (clamp vs. reject) — planner decides, follow head-angle clamping precedent.
- Exact markdown structure inside SKILL.md body — organize for scannability; tables for wire formats, lists for actions.

</decisions>

<specifics>
## Specific Ideas

- LED off uses `CMD_LED#0\n` (mode 0 = dedicated off path on server), not `CMD_LED#0#0#0\n` — important distinction, test this explicitly
- The Freenove server runs LED commands in a separate thread (`self.led_thread`) — means the command is fire-and-forget from the TCP perspective; server doesn't send a response
- `led_index(0x7f, R, G, B)` on the server side controls all 8 LEDs (0x7f = bitmask for all); this is transparent to IronHermes — we just send the wire command
- Skill doc should be useful standalone — agent reading only `skills/hexapod/SKILL.md` should be able to invoke any of the 12 actions without reading any other file

</specifics>
