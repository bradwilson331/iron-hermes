## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking] localStorage helpers route through `js_sys::Reflect` instead of `web_sys::Storage`**
- **Found during:** Task 1 implementation, before RED commit
- **Issue:** Plan 02's `<action>` step 12 specifies `web_sys::window()?.local_storage().ok().flatten()` for the WASM-only helpers, but the `Storage` feature is NOT enabled on `web-sys` in `Cargo.toml` at the base commit `45739caf`. Plan 01 (running in the same wave-1 parallel worktree) adds that feature, but Plan 02's `depends_on: []` means it must compile against the bare base — and the orchestrator's success criteria explicitly forbids Cargo.toml modifications ("Plan 01's territory").
- **Fix:** Reach `window.localStorage` through `js_sys::Reflect::get(&window, "localStorage")` instead of `web_sys::Window::local_storage`, then call `getItem` / `setItem` via `js_sys::Function`. Functionally equivalent — both code paths target the same DOM `Storage` object. Behaviour matches the RESEARCH §Security T-DESERIALIZE mitigation: corrupt blobs silently fall back to `T::default()` via `.ok()` rather than panicking. The js_sys path remains correct after Plan 01's merge (presence of the Storage feature doesn't break Reflect-based access).
- **Files modified:** `crates/iron_hermes_ui/src/ui_prefs.rs`
- **Commit:** `bf76fe6f`

**2. [Rule 3 — Blocking] `pub mod ui_prefs;` registered in `main.rs`, not `lib.rs`**
- **Found during:** Task 1
- **Issue:** Plan 02 step 15 says "Add `pub mod ui_prefs;` to `lib.rs`" but `iron_hermes_ui` is a **binary** crate — there is no `src/lib.rs`. The existing `mod state`, `mod app`, etc. all live in `src/main.rs`.
- **Fix:** Registered `mod ui_prefs;` in `src/main.rs:9` alongside the existing module declarations.
- **Files modified:** `crates/iron_hermes_ui/src/main.rs`
- **Commit:** `bf76fe6f`

**3. [Rule 3 — Blocking] Un-gated `mod mocks` from `#[cfg(any(test, feature = "demo"))]` to plain `mod mocks`**
- **Found during:** Task 1
- **Issue:** The existing `mod mocks` declaration was `#[cfg(any(test, feature = "demo"))]`, but CONTEXT D-04 mandates that stub data is the PRODUCTION path for the 10 visual screens in this phase. Leaving the gate would make `crate::mocks::stub_data::*` unreachable in default builds and the Plan 08 consumers would fail to compile.
- **Fix:** Removed the cfg gate. The pre-existing fixtures inside `mocks/` (e.g., `personalities::REPLIES`, `shell_outputs::*`, `agent_steps::run_agent_steps`) keep their own `#[cfg(any(test, feature = "demo"))]` gates where present, so removing the module-level gate doesn't expose anything that wasn't already in the `feature = "demo"` build.
- **Files modified:** `crates/iron_hermes_ui/src/main.rs`
- **Commit:** `bf76fe6f`
