# Phase 26.2.1: new-web-ui-with-wheel-menu — Research

**Researched:** 2026-05-13
**Domain:** Dioxus 0.7 web UI port — SVG wheel + multi-screen SPA + WebSocket streaming + theming + localStorage persistence
**Confidence:** HIGH

## Summary

This phase replaces the existing Warp-style Dioxus shell with a new wheel-menu-driven SPA ported verbatim from a Claude-Design HTML/CSS handoff bundle. The backend (`src/server/ws.rs`, `src/server/api.rs`, `src/protocol.rs`) is preserved byte-for-byte; the existing WebSocket client pattern in `src/components/warp_hermes.rs` (using `dioxus_fullstack::use_websocket` + `recv_raw` / `send_raw` + manual JSON serialization) is the proven pattern to mirror.

Three screens (Chat, Sessions, Settings) are live-wired; nine (Agents, Skills, Models, Memory, Soul, Tools, Schedules, Gateway, Office) are pixel-faithful visual stubs sourced from `src/mocks/stub_data.rs`; a 13th (Providers) is a visual stub reached from the Settings screen. The existing Warp shell moves behind a `legacy-shell` Cargo feature (default OFF) by renaming `src/components/shell/` → `src/components/shell_legacy/` and module-gating it.

The wheel is 12 SVG wedges plus a hub launch button, pre-rendered into RSX as a static structure with reactive `class:"is-active"` bindings driven by a `Signal<WheelState>` context. Drag/resize uses pointer events with `setPointerCapture`. All UI prefs (theme, accent, wheel position/size, density, scanlines, breadcrumb, footer, rail) persist to browser localStorage under `ih.ui.*` keys. CSS is copied verbatim from the bundle into `assets/`.

**Primary recommendation:** Mirror the existing `warp_hermes.rs` WebSocket pattern. Translate `wheel-v2.js` SVG-builder into a single static RSX `svg!` block with reactive class bindings (option (a) in CONTEXT discretion). Module-gate `legacy-shell` at the file system level (rename to `shell_legacy/`). Keep persistence keys flat (`ih.ui.wheel`, `ih.ui.theme`, `ih.ui.tweaks`) and gate writes during initial hydration to avoid clobbering stored prefs with defaults.

**Why compile-time `#[cfg(...)]` over runtime `if cfg!(...)`:** Compile-time gating means the OFF shell isn't compiled into the WASM binary at all — smaller payload, no risk of stale `legacy-shell` code reaching runtime. Runtime branching would compile both shells into every build.

### Pattern 2: Wheel SVG in RSX (option (a) — recommended)

[VERIFIED: wheel-v2.js geometry constants lines 26-37; wheel.css class names lines 121-148]

The wheel SVG is **static structure** with **reactive class bindings**. Geometry is computed once at module level (or via `const`/`fn` helpers) — only `is-active` classes and the `--wheel-size` CSS variable change per render.

```rust
// crates/iron_hermes_ui/src/components/hermes_app/wheel.rs
use dioxus::prelude::*;
use crate::state::{WheelWedge, WheelState};

// Wheel geometry — exact values from wheel-v2.js
pub const SIZE:     f64 = 380.0;
pub const R_OUTER: f64 = 190.0;  // SIZE / 2
pub const R_INNER: f64 = 78.0;
pub const R_LABEL: f64 = 148.0;
pub const R_GLYPH: f64 = 118.0;
pub const RING_GAP: f64 = 14.0;
pub const RING_W:   f64 = 4.0;
pub const PAD:      f64 = 22.0;
pub const VB:       f64 = SIZE + PAD * 2.0;   // 424
pub const N: usize = 12;                       // 12 wedges per app.html WHEEL_SECTIONS
pub const STEP: f64 = 360.0 / N as f64;
pub const MIN_SIZE: f64 = 240.0;
pub const MAX_SIZE: f64 = 640.0;

fn polar(ang_deg: f64, r: f64) -> (f64, f64) {
    let a = (ang_deg - 90.0).to_radians();
    (a.cos() * r, a.sin() * r)
}

fn wedge_path(ang_a: f64, ang_b: f64, r_inner: f64, r_outer: f64) -> String {
    let (x1, y1) = polar(ang_a, r_outer);
    let (x2, y2) = polar(ang_b, r_outer);
    let (x3, y3) = polar(ang_b, r_inner);
    let (x4, y4) = polar(ang_a, r_inner);
    format!(
        "M {x1} {y1} A {r_outer} {r_outer} 0 0 1 {x2} {y2} \
         L {x3} {y3} A {r_inner} {r_inner} 0 0 0 {x4} {y4} Z"
    )
}

#[component]
pub fn Wheel() -> Element {
    let mut wheel = use_context::<Signal<WheelState>>();
    let mut active_screen = use_context::<Signal<crate::state::Screen>>();

    // Tooltip + cursor mode local state
    let mut tooltip_open = use_signal(|| false);
    let mut tooltip_resize = use_signal(|| false);
    let mut tooltip_xy = use_signal(|| (0.0_f64, 0.0_f64));
```

### Pattern 3: Pointer Drag + Resize (Dioxus 0.7 — signal-borrow safe)

[VERIFIED: wheel-v2.js drag handlers lines 386-465; clippy.toml await-holding rules]

```rust
// In wheel.rs — drag the rim to move the shell
#[derive(Clone, Copy)]
struct DragState { start_client: (f64, f64), start_pos: (f64, f64), pointer_id: i32, dist: f64 }

#[derive(Clone, Copy)]
struct ResizeState { start_client: (f64, f64), start_size: f64, pointer_id: i32 }

// onpointerdown on `.wheel-rim`:
onpointerdown: move |e: PointerEvent| {
    if e.trigger_button() != Some(MouseButton::Primary) { return; }
    let c = e.client_coordinates();
    let pos = wheel.read().position;
    dragging.set(Some(DragState {
        start_client: (c.x, c.y),
        start_pos: pos,
        pointer_id: e.pointer_id(),
        dist: 0.0,
    }));
    // Add setPointerCapture via web_sys — Dioxus event doesn't expose it natively.
    // (Optional — pointermove on window still fires without capture; capture is
    // primarily for ensuring pointerup fires on the same element.)
    e.stop_propagation();
}

// Bind pointermove + pointerup at window scope via use_effect on mount.
// Reason: pointermove only fires on the element it started on unless captured.
// We dispatch from the window so we keep tracking even if the cursor leaves
// the wheel's bounding box during a drag.
use_effect(move || {
    use wasm_bindgen::closure::Closure;
    use wasm_bindgen::JsCast;
    use web_sys::PointerEvent as WebPointerEvent;

    let Some(window) = web_sys::window() else { return; };

    let move_cb = Closure::<dyn FnMut(WebPointerEvent)>::new(move |e: WebPointerEvent| {
        // PATTERN: read once into local, drop, then signal-write — never
        // hold the GenerationalRef across the .set() call.
        if let Some(d) = *dragging.read() {
            let dx = e.client_x() as f64 - d.start_client.0;
            let dy = e.client_y() as f64 - d.start_client.1;
            let dist = (dx*dx + dy*dy).sqrt().max(d.dist);
            let mut new_x = d.start_pos.0 + dx;
            let mut new_y = d.start_pos.1 + dy;
            // Margin clamp — match wheel-v2.js lines 409-412
            let margin = 12.0;
            let w = web_sys::window().and_then(|w| w.inner_width().ok()).and_then(|v| v.as_f64()).unwrap_or(1920.0);
            let h = web_sys::window().and_then(|w| w.inner_height().ok()).and_then(|v| v.as_f64()).unwrap_or(1080.0);
            let shell_size = wheel.read().size;  // brief read
            new_x = new_x.max(margin).min(w - shell_size - margin);
            new_y = new_y.max(margin).min(h - shell_size - margin);
            // Update wheel position
            wheel.with_mut(|s| s.position = (new_x, new_y));
            // Track max distance for click-suppression
            dragging.with_mut(|d_opt| if let Some(d) = d_opt { d.dist = dist; });
        }
        if let Some(r) = *resizing.read() {
            let dx = e.client_x() as f64 - r.start_client.0;
            let dy = e.client_y() as f64 - r.start_client.1;
            let delta = (dx + dy) / 2.0;
            let new_size = (r.start_size + delta).clamp(MIN_SIZE, MAX_SIZE);
            wheel.with_mut(|s| s.size = new_size);
        }
    });
    window.add_event_listener_with_callback("pointermove", move_cb.as_ref().unchecked_ref()).ok();
    let move_cb_boxed = Box::leak(Box::new(move_cb)); // keep alive for lifetime of effect

    let up_cb = Closure::<dyn FnMut(WebPointerEvent)>::new(move |_e: WebPointerEvent| {
        dragging.set(None);
        resizing.set(None);
    });
    window.add_event_listener_with_callback("pointerup", up_cb.as_ref().unchecked_ref()).ok();
    let _ = Box::leak(Box::new(up_cb));
});
```

**Signal-borrow safety contract** (clippy.toml lines 1-8): NEVER hold `GenerationalRef`, `GenerationalRefMut`, or `WriteLock` across `.await`. Pointer handlers are synchronous JS callbacks — the read/write pattern above is safe because every `.read()` borrow drops at the end of its block before any other signal interaction, and there are no `.await` points inside the closures.

```rust
// crates/iron_hermes_ui/src/components/hermes_app/screens/chat.rs
use dioxus::prelude::*;
use crate::protocol::{ChatRequest, ChatStreamEvent};

#[component]
pub fn ScreenChat() -> Element {
    let mut active_screen = use_context::<Signal<crate::state::Screen>>();
    let mut session_id = use_signal(|| "pending".to_string());
    let mut bubbles = use_signal(Vec::<ChatBubble>::new);   // user + assistant rows
    let mut streaming_id = use_signal(|| Option::<u64>::None);
    let mut next_id = use_signal(|| 1u64);
    let mut input_text = use_signal(String::new);
    let mut tokens = use_signal(|| (0u32, 128_000u32));

    let mut ws = dioxus_fullstack::use_websocket(move || {
        crate::server::ws::ws_chat(
            dioxus_fullstack::WebSocketOptions::new().with_automatic_reconnect(),
        )
    });

    // Receive loop — mirror warp_hermes.rs:145-410
    use_future(move || async move {
        loop {
            let state = ws.connect().await;
            if ws.is_err() { continue; }

            loop {
                match ws.recv_raw().await {
                    Ok(dioxus_fullstack::Message::Text(t)) => {
                        let Ok(event) = serde_json::from_str::<ChatStreamEvent>(&t) else { continue };
                        match event {
                            ChatStreamEvent::Delta { text } => {
                                let sid = *streaming_id.read();
                                if let Some(id) = sid {
                                    let mut bs = bubbles.write();
                                    if let Some(b) = bs.iter_mut().find(|b| b.id == id) {
                                        b.text.push_str(&text);
                                    }
                                } else {
                                    let id = { let n = next_id(); next_id.set(n+1); n };
                                    streaming_id.set(Some(id));
                                    bubbles.write().push(ChatBubble::assistant(id, text));
                                }
                            }
                            ChatStreamEvent::ToolCallStart { name, args } => {
                                let sid = *streaming_id.read();
                                if let Some(id) = sid {
                                    let mut bs = bubbles.write();
                                    if let Some(b) = bs.iter_mut().find(|b| b.id == id) {
                                        b.tool_rows.push(ToolRow { name, args, done: false, success: false });
                                    }
                                }
                            }
                            ChatStreamEvent::ToolCallEnd { name, success } => {
                                let sid = *streaming_id.read();
                                if let Some(id) = sid {
                                    let mut bs = bubbles.write();
                                    if let Some(b) = bs.iter_mut().find(|b| b.id == id) {
                                        for row in b.tool_rows.iter_mut().rev() {
                                            if row.name == name && !row.done {
                                                row.done = true;
                                                row.success = success;
                                                break;
                                            }
                                        }
                                    }
                                }
                            }
                            ChatStreamEvent::Finished { total_tokens } => {
                                tokens.set((total_tokens, 128_000));
                                streaming_id.set(None);
                            }
                            ChatStreamEvent::Error { message } => {
                                let id = { let n = next_id(); next_id.set(n+1); n };
                                bubbles.write().push(ChatBubble::error(id, message));
                                streaming_id.set(None);
                            }
                        }
                    }
                    Ok(dioxus_fullstack::Message::Close { .. }) => {
                        streaming_id.set(None);
                        break;  // outer loop reconnects
                    }
                    Err(_) => break,
                    _ => continue,
                }
            }
        }
    });

    // Submit handler: Enter (no Shift) → send. Shift+Enter → newline (browser default).
    let send = move || {
        let text = input_text.read().clone();
        if text.trim().is_empty() { return; }
        let sid = session_id.read().clone();

        // Push user bubble locally
        let id = { let n = next_id(); next_id.set(n+1); n };
        bubbles.write().push(ChatBubble::user(id, text.clone()));
        input_text.set(String::new());

        let req = ChatRequest { session_id: sid, message: text };
        let json = serde_json::to_string(&req).unwrap_or_default();
        spawn(async move {
            let _ = ws.send_raw(dioxus_fullstack::Message::Text(json)).await;
        });
    };

    rsx! {
        section {
            class: "screen is-active",   // is-active driven by parent screen_router
            id: "screen-chat",
            "data-screen-label": "01 Chat",
            // ... header + .chat-mini + .chat-stream + .chat-input-pill from UI-SPEC
        }
    }
}
```

**Hydration gate pattern — avoid clobbering stored prefs:**

```rust
// In HermesApp root:
let prefs = use_signal(UiPrefs::default);
let hydrated = use_signal(|| false);

// Step 1: on mount, read from localStorage exactly once.
use_effect(move || {
    if *hydrated.read() { return; }
    if let Some(p) = ui_prefs::read_json::<UiPrefs>(ui_prefs::KEY_TWEAKS) {
        prefs.set(p);
    }
    hydrated.set(true);
});

// Step 2: write on change, but ONLY after hydration completes.
use_effect(move || {
    if !*hydrated.read() { return; }
    let p = prefs.read().clone();
    ui_prefs::write_json(ui_prefs::KEY_TWEAKS, &p);
});

use_context_provider(|| prefs);
```

This ordering prevents the initial default `UiPrefs::default()` from overwriting stored prefs before they're read. The `hydrated` flag flips after the read, so the first write effect fires only with the user's actual prefs.

### Anti-Patterns to Avoid

- **❌ Hold `signal.read()` across `.await`** — clippy.toml will flag this; runtime panics on the second read/write inside the await.
- **❌ Use `cx`, `Scope`, `use_state`** — Dioxus 0.6 APIs; removed in 0.7. [VERIFIED: AGENTS.md line 1]
- **❌ Inject the wheel as raw HTML via `dangerous_inner_html`** — loses reactivity; can't bind `is-active` class to a signal.
- **❌ Embed `wheel-v2.js` via `document::Script`** — JS would manipulate the DOM independently of Dioxus's VDOM, causing desync. Was a CONTEXT-acknowledged option (b) but rejected.
- **❌ `<a href="#chat">` for screen nav** — would trigger browser hash change + reload semantics. CONTEXT anti-pattern explicit.
- **❌ Add `Routable` enum for screens** — deferred to 26.2.17. The class-toggle pattern is canonical for this phase.
- **❌ Edit `src/server/ws.rs`** — CONTEXT D-02 forbids any backend mutation.
- **❌ Compute wheel geometry inside `rsx!` body without extracting helper fns** — RSX expressions are evaluated on every render; pre-compute angles into closures or `let` bindings.
- **❌ Skip the hydration-gate** — writing `UiPrefs::default()` to localStorage on first mount would silently overwrite the user's saved prefs.

## Common Pitfalls

### Pitfall 1: SVG attribute names with hyphens fail to compile in RSX

**What goes wrong:** `rsx! { circle { stroke-width: "1" } }` does not compile — `stroke-width` is not a valid Rust identifier (hyphens are subtraction).

**Why it happens:** Dioxus 0.7's `rsx!` macro parses attributes as identifiers. SVG and many HTML attributes (`data-*`, `aria-*`, `stroke-width`, `text-anchor`) contain hyphens.

**How to avoid:** Quote hyphenated attribute names as raw strings:

```rust
rsx! { circle { "stroke-width": "1", "text-anchor": "middle", "data-i": "{i}" } }
```

**Warning signs:** Compile errors like `expected identifier, found '-'` inside an `rsx!` block — always inside SVG generation code.

### Pitfall 2: Pointer drag stops when cursor leaves wheel bbox

**What goes wrong:** `onpointermove` on the SVG `path#wheel-rim` only fires while the pointer is over that element. Fast drags fly the cursor outside, drag freezes.

**Why it happens:** Pointer events bubble through the DOM only while the pointer is over an element. Without `setPointerCapture` or window-level binding, the move event series breaks the moment the cursor crosses the SVG boundary.

**How to avoid:** Bind `pointermove` + `pointerup` at the `window` level inside a `use_effect`, not on the SVG element. This mirrors what `wheel-v2.js` does (lines 402, 417). See Pattern 3 above. Alternative: call `e.target.set_pointer_capture(e.pointer_id())` on pointerdown — but this requires a `web_sys` cast that's awkward in Dioxus event handlers.

**Warning signs:** Drag works for short movements but freezes when the user drags quickly across the screen.

### Pitfall 3: `body { overflow: hidden }` clips the resize ring outside the wheel

**What goes wrong:** The resize ring orbits at radius `R_OUTER + RING_GAP = 204` — OUTSIDE the rim. If `body { overflow: hidden }` (which is set on `.app` per screens.css line 414) clips elements beyond viewport bounds AND the wheel sits near a corner, the resize ring may be visually clipped or, worse, become un-hittable when out of the body's content box.

**Why it happens:** `.app { overflow: hidden }` clips its descendants. The wheel's `.wheel-shell` is `position: fixed` (per wheel.css line 9), which escapes the `.app` container's stacking context — fixed-positioned elements are clipped by the viewport, not by ancestor `overflow: hidden`. So in practice the ring is fine UNLESS the wheel is dragged close to a viewport edge.

**How to avoid:** The drag clamp (Pattern 3) already constrains the shell to viewport - shell_size - margin. Ensure the clamp uses `shell.offsetWidth` (which includes the ring orbit radius in the SVG viewBox padding `PAD = 22` — but NOT the full `RING_GAP = 14`). Conservative fix: extend the drag margin from 12 to `12 + RING_GAP + 7 ≈ 33` so the resize ring nodes stay on-screen. CONTEXT Claude's Discretion called this out — recommend implementing the conservative margin.

**Warning signs:** Resize ring partially hidden when wheel is near a viewport edge; resize cursor not appearing when hovering near corners.

### Pitfall 4: Initial `--wheel-size: 220px` from prototype vs. `MIN_SIZE = 240` clamp

**What goes wrong:** The `app.html` prototype sets `--wheel-size: 220px` inline (line 32); the `wheel-v2.js` resize clamp is `MIN_SIZE = 240` (line 426). The very first resize gesture snaps the wheel up to 240, causing a visible jump.

**Why it happens:** Prototype-author oversight; the 220 default predates the 240 floor.

**How to avoid:** Set the default in `WheelState::default()` to 240.0 (the UI-SPEC Revision 1 fix already records this). Do not honor the prototype's 220 — treat it as a pre-port artifact.

**Warning signs:** First resize gesture causes a visible 20px jump.

### Pitfall 5: `use_websocket` cannot be re-instantiated per screen mount/unmount

**What goes wrong:** If `dioxus_fullstack::use_websocket(...)` is called inside `ScreenChat`, then switching screens (unmounting the chat component) tears down the connection. The next mount creates a fresh socket, losing turn state.

**Why it happens:** Dioxus 0.7's `use_websocket` hook is scoped to the component instance — when the component unmounts, the hook drops, taking the connection with it.

**How to avoid:** Place `use_websocket` at the `HermesApp` root component (which never unmounts), provide the socket handle + a shared message stream as context via `use_context_provider(...)`. ScreenChat reads the context, mounts/unmounts freely. This is a departure from `warp_hermes.rs` where the WS lives at the shell root — but since `warp_hermes.rs` also never unmounts (it's the root), the same pattern applies: root-level placement.

**Warning signs:** Reconnect / "websocket connect failed" notices appear every time the user switches screens.

### Pitfall 6: Tab switch fires a stale on-click handler with the old `session_id`

**What goes wrong:** Phase 26.2 D-09 EventHandler<()> close button with stop_propagation. If the wedge click and the screen rendering both write to `WheelState` and `Screen` signals, multiple effect cycles can fire in the same tick.

**Why it happens:** Multiple signal writes in one synchronous handler are coalesced; effects re-run on the next render. But if a handler reads `wheel.active_wedge` BEFORE the same handler writes it, you can get a stale read.

**How to avoid:** Write to `wheel.active_wedge` FIRST, then immediately compute target and set `active_screen` from the new state in the same handler — Pattern 2 shows this. Do not split into two separate `use_effect` cycles unless you intend to give the user a frame of visual feedback.

**Warning signs:** Clicking a wedge sometimes activates the previously-active screen; intermittent off-by-one wedge selection.

### Pitfall 7: Hash deep-linking (`#chat`, `#sessions`) — out of scope but easy to break

**What goes wrong:** The `app.html` prototype uses `history.replaceState(null, '', '#' + id)` (line 1582) to reflect screen state in the URL hash. The Dioxus port intentionally omits this (CONTEXT anti-pattern: no `<a href="...">`, no router). But if a developer adds `<a href="#chat">` somewhere, the browser navigates and the page un-hydrates.

**Why it happens:** Browsers handle anchor links as soft navigation; the WASM client gets no warning.

**How to avoid:** Never use `<a href="#...">` in the new shell. Use `button` elements with `onclick` handlers that set `active_screen.set(Screen::X)`. Verify in code review.

**Warning signs:** Clicking a "link" within the app causes a brief flash or full re-mount.

### Pitfall 8: `web-sys` features omitted at compile time = silent runtime panics

**What goes wrong:** Calling `web_sys::window().unwrap().local_storage()` without the `Storage` feature enabled compiles fine (the function exists in `web-sys` but is gated). Result is a method-not-found error wrapped in a returned `Result::Err`.

**Why it happens:** `web-sys` ships every browser API as opt-in features. Without `Storage`, `HtmlElement`, `DomTokenList`, `CssStyleDeclaration` features, the methods used in Patterns 5–6 won't be available.

**How to avoid:** Add the features to `Cargo.toml` upfront (see Pattern 1 Cargo delta above). If `cargo check` passes but `local_storage()` returns `Err`, check the feature list.

**Warning signs:** Prefs reset every page load (writes failing silently); theme not applying.

### Pitfall 9: Wheel-rail position drifts when wheel resizes

**What goes wrong:** The wheel-rail (`#wheel-rail`) needs to position itself directly under the wheel (per app.html line 1625-1647: `rail.style.top = (r.bottom + 18) + 'px'`). When the wheel resizes or drags, the rail must reposition.

**Why it happens:** In the prototype, a `ResizeObserver` + `MutationObserver` re-runs `positionWheelRail()`. In Dioxus, the rail must subscribe to `WheelState` and reposition declaratively.

**How to avoid:** Compute rail style inline from `WheelState`:

```rust
let wheel = use_context::<Signal<WheelState>>().read();
let left = wheel.position.0;
let top  = wheel.position.1 + wheel.size + 18.0;
let width = wheel.size.max(280.0);
let style = format!("left: {left}px; top: {top}px; width: {width}px;");
rsx! { aside { class: "wheel-rail", id: "wheel-rail", style: "{style}", /* ... */ } }
```

The signal-driven style means the rail repositions on every WheelState change without observers. The CSS rules (`body.has-rail #wheel-rail { display: flex !important; }`) handle visibility.

**Warning signs:** Rail floats in a static position while the wheel moves; rail visible on non-chat screens.

## Open Questions

1. **Should the new shell use `use_drop` to clean up window event listeners on unmount?**
   - What we know: `HermesApp` is the root and never unmounts, so practically the cleanup never runs.
   - What's unclear: If 26.2.16 adds desktop/mobile features that swap shells at runtime, leaked window listeners could accumulate.
   - Recommendation: Use `use_drop` for window-level `pointermove`/`pointerup` listeners as a hygiene practice. Cost is one extra closure storage signal; benefit is forward-compat.

2. **Where should the Tweaks panel "host persistence" deferred decision (26.2.13) leave a seam?**
   - What we know: localStorage write is the canonical write path in 26.2.1.
   - What's unclear: Whether to introduce a `PrefsBackend` trait now so 26.2.13 can swap implementations without touching call sites.
   - Recommendation: Defer the trait — premature abstraction. The `write_json(key, &val)` helper signature is already a natural seam.

3. **How should the "wheel resize handle outside body overflow" edge case be tested?**
   - What we know: Pitfall 3 calls out the risk; the mitigation is a wider drag margin.
   - What's unclear: Whether automated tests can catch this or if it's manual UAT only.
   - Recommendation: Manual UAT step in verification — "drag the wheel to each viewport corner and confirm the resize ring remains hoverable." Add this to the UAT script. Automated visual regression would require headless browser + screenshot diff, which is out of scope.

4. **Does the new shell need to honor URL hash deep-links (`#chat`, `#sessions`) on first load?**
   - What we know: CONTEXT explicitly says "no `Routable` enum, no URL changes" (anti-pattern list).
   - What's unclear: Whether reading `location.hash` once on mount (without writing it on change) is in-spirit-of the constraint.
   - Recommendation: Skip hash-reading in 26.2.1. The Tweaks panel's "active screen" memory is implicit via the wheel state's `active_wedge` field (persisted to localStorage). Users land on whatever wedge they last activated. Hash routing arrives in 26.2.17.

5. **Should slash command auto-complete (from `list_slash_commands()`) appear in 26.2.1?**
   - What we know: `CommandRouter` exists server-side; `list_slash_commands()` server fn returns a `Vec<SlashCommandInfo>`. CONTEXT D-20 says slash commands route via the same WebSocket.
   - What's unclear: Whether the input pill in Chat should pop a typeahead list when the user types `/`.
   - Recommendation: NO typeahead in 26.2.1. Mirror the prototype (no typeahead in `app.html`). Slash commands work — they're just unsuggested. A typeahead is its own visual decision and would expand scope. Defer to a future phase.

### Known Threat Patterns for Dioxus/WASM/WebSocket Stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| XSS via injected message content rendered as HTML | Tampering | Dioxus rsx! escapes text by default. NEVER use `dangerous_inner_html` for chat content. Markdown rendering (if added later) needs a sanitizer — but 26.2.1 renders plaintext text, so no risk. |
| CSRF on server fn calls | Spoofing | Dioxus fullstack uses same-origin + cookies (server framework's default). No new auth surface. |
| WebSocket message size DoS | Denial of Service | Existing `ws_chat` doesn't impose a max frame size, but the keepalive + close-on-error already in place (Phase 26.1) prevents indefinite blocking. Out of scope to add limits here. |
| localStorage tampering by browser extensions | Tampering | UI prefs are non-secret. Worst case: a malicious extension flips theme or wheel size. No mitigation needed. |
| Theme injection via accent picker writing arbitrary CSS via `style.set_property` | Tampering | The accent picker writes ONLY from a closed enum `AccentColor` (Teal/Orange/Green/Violet/Amber). No user-supplied strings ever reach `set_property`. |
| Path traversal in `asset!` macro | Tampering | `asset!` is compile-time and only resolves paths under `/assets/`. Compile-time constraint. |
| Storage quota exhaustion via wheel position writes | DoS | Each write is < 100 bytes; localStorage quota is ≥ 5MB. Not a real risk. |
