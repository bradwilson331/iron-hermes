### 8. Chat — Round Trip
expected: On the Chat screen, the chat-mini header shows the 8-char session id prefix and status READY. Typing "hello, what model are you?" + Enter renders a user bubble immediately and an assistant bubble below streaming deltas character-by-character; status flips to STREAMING then back to READY. Shift+Enter inserts a literal newline in the user bubble. `/clear` empties the chat buffer. `/research weather` streams an assistant bubble with tool-progress rows underneath during tool execution (is-running → is-done is-success). Toggling DevTools "Offline" then back online triggers automatic WS reconnect (Phase 26.1 `with_automatic_reconnect`).
result: issue
reported: "no - it takes a while for the system to respond and this does not appear to be a builtin command *Context cleared.* I have reset the conversation state. I am ready for your next request, taking a completely fresh start! How can I assist you now?"
severity: major
notes: The `/clear` slash command is NOT being intercepted by the server-side CommandRouter (Plan 06 D-20 contract). Evidence — the assistant returned a natural-language reply ("*Context cleared.* I have reset the conversation state…") which is the LLM's verbal interpretation of the literal string `/clear`, not the buffer-clearing side-effect. Plan 06 D-20 specifies slash commands flow as plain text frames and the server CommandRouter handles them; the `ws_chat` endpoint is supposed to detect the `/` prefix BEFORE forwarding to the model. Either CommandRouter doesn't register `/clear` for the ws_chat handler in this build, or the slash-command detection branch is bypassed for the new HermesApp client. Latency ("takes a while") is a separate non-blocking observation. Shift+Enter, /research, and WS reconnect were not exercised because the tester stopped at the /clear failure.

### 9. Sessions — Switch & Delete

Clicking a row's "×" close button removes the row from the list via optimistic local delete; `evt.stop_propagation()` prevents accidental row-select on close. NOTE: server-side delete is deferred to 26.2.12 — after reload the closed session reappears (documented in VERIFICATION.md deferred items).
result: issue
reported: "shows different sessions but all reference the current session."
severity: major
notes: Sessions list renders multiple distinct rows (good — list_sessions server fn is firing). But every row appears to reference the current session — ambiguous between (a) per-row data is correct but the visible session-id / ACTIVE-pill column displays the *current* session id on every row instead of each row's own id, OR (b) clicking a non-current row does not actually switch SessionIdContext / Screen::Chat — the chat-mini header keeps showing the previously-active session. Either way the row's onclick contract from screens/sessions.rs is not delivering the documented behavior. Close-button optimistic-delete and stop_propagation were not exercised because the tester stopped at the switch failure.
