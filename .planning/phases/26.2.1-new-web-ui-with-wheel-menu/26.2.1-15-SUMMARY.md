## Round-5 Scope Expansion (2026-05-14)

**User report:** "scanlines animation is scrolling on the webpage even though the removal has been 'completed'. Only the controls have been removed from the webpages, which did not affect the animation."

**Root cause:** The visible scrolling-line effect was sourced from `.scan-bar` — a 2px teal horizontal line animating `translateY(0vh → 100vh)` over 7s on infinite loop — defined in `site.css:85–97` and mounted unconditionally by `HudChrome` (`hud_chrome.rs:20`). It was introduced by Plan **26.2.1-03** as one of three "decorative full-viewport overlays" alongside `hud-grid` and `hud-vignette`. The class was never called `scanlines` in code, comments, or commit history — the visual is identical, but the implementation name is a synonym.

Plan 15's round-4 `scanlines_feature_is_fully_removed` test was a textual `include_str!` grep for the literal strings `.scanlines` / `no-scanlines` / `pub scanlines: bool`. By construction, it could not catch a synonym. All 5 build gates were green, and the user-visible regression nonetheless persisted.

**Round-5 fix (4 commits, no new plan file per D-26.2.1-15-D):**

| Commit | Change |
|--------|--------|

| `74fe99c6` | fix(26.2.1-15): remove `<div class="scan-bar">` from HudChrome; module doc "three overlays" → "two" |
| `17e43f2e` | test(26.2.1-15): extend `scanlines_feature_is_fully_removed` with 3 new asserts (`.scan-bar`, `scan-bar-move`, `class: "scan-bar"`) |

**Lesson learned (logged for future phases):**

Textual-grep removal guards are necessary but not sufficient. They block re-introduction by the same literal name; they cannot block a synonym or a future rename of the same visual effect. A stronger guard would either (a) structurally match the CSS pattern (fixed-position full-viewport overlay with translateY animation) or (b) run a wasm-bindgen-test that renders HudChrome and asserts no full-viewport-animated element is present. Round-5 chose option (a) deferred and stayed with the existing textual pattern per D-26.2.1-15-E to keep the guard suite consistent. The structural / runtime-DOM guard is recorded here as a follow-on improvement and may be picked up by phase 26.2.12 or a later validation-pattern phase.
