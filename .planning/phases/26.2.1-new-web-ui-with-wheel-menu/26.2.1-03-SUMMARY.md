# Phase 26.2.1 Plan 03: HermesApp Root Composer + HUD Chrome + ScreenRouter Summary

One-liner: HermesApp now owns four context-provided signals (`UiPrefs`, theme String, `WheelState`, `Screen`), gates localStorage hydration through a one-shot effect (RESEARCH Pattern 5), and mounts always-on HUD chrome + breadcrumb + sys-meta + 1Hz UTC clock footer plus a class-toggle ScreenRouter that runs all 13 placeholder screens simultaneously (RESEARCH Pattern 7).

## Signals Provided + Hydration Order

`HermesApp` (`crates/iron_hermes_ui/src/components/hermes_app/mod.rs`) declares **four root signals** and provides each via `use_context_provider` in this exact order:

1. `Signal<UiPrefs>` — bare provide (`use_context_provider(|| prefs)`); type is unique in the tree.
2. `Signal<String>` (theme) — wrapped as `ThemeContext(theme)` (B-03 newtype, D-26) so Plan 06's `SessionIdContext(Signal<String>)` does not type-collide with the theme signal.
3. `Signal<WheelState>` — bare provide; unique type.
4. `Signal<Screen>` — bare provide; unique type.

A fifth local signal `hydrated: Signal<bool>` drives the hydration gate.

**Hydration order (single one-shot `use_effect`):**

1. Early-return if `hydrated` is already `true`.
2. Read `ih.ui.tweaks` → `UiPrefs::deserialize` → if `Some(p)`, `prefs.set(p)` (else default stays).
3. Read `ih.ui.theme` (raw string) → if `Some(t)`, `theme.set(t)` (else `"slate-dark"` default stays).
4. Read `ih.ui.wheel` → `WheelState::deserialize` → if `Some(ws)`, `wheel_state.set(ws)` (else `WheelState::default()` `(24.0, 24.0) / 240.0` default stays).
5. `hydrated.set(true)` — only after this flip do the three persist-on-change effects begin writing.

Three persist effects (one per key) follow the canonical signal-borrow safety pattern: read into a local clone, drop the borrow at the `;`, then call the side-effecting `ui_prefs::write_*` helper. Each effect early-returns when `hydrated` is still `false`, so the initial `Default` values never clobber stored blobs (RESEARCH Pitfall 5).

## Breadcrumb / AppFooter Screen-Label Match Arms

Both `breadcrumb.rs` and `app_footer.rs` define a local `match` on `Screen` to derive the rightmost crumb / `SCREEN ` field. The map is **identical between the two files** and locked to the plan's `<interfaces>` table:

```rust
match s {
    Screen::Chat       => "CHAT",
    Screen::Sessions   => "SESSIONS",
    Screen::Agents     => "AGENTS",
    Screen::Skills     => "SKILLS",
    Screen::Models     => "MODELS",
    Screen::Memory     => "MEMORY",
    Screen::Soul       => "SOUL",       // off-wheel (D-10)
    Screen::Tools      => "TOOLS",
    Screen::Schedules  => "SCHED",      // off-wheel (D-10); short-name locally
    Screen::Gateway    => "GATEWAY",
    Screen::Office     => "OFFICE",     // off-wheel (D-10)
    Screen::Settings   => "SYSTEM",     // matches WheelWedge::Settings.label() per D-10
    Screen::Providers  => "PROVIDERS",
}
```

Keeping the map local to each file (rather than adding a method to `Screen` in `state.rs`) preserves the plan's contract that Plan 02's `state.rs` stays minimal — the breadcrumb is the only consumer in this plan and the footer is the only other consumer.

## Classes Emitted by HudChrome

`HudChrome` (`hud_chrome.rs`) renders eight always-mounted, decorative, `aria-hidden="true"` divs. CSS is supplied by `assets/site.css` (copied verbatim from the design bundle by Plan 01). The classes — in render order — are:

| Order | Element | Purpose |
| ----- | ------- | ------- |
| 1     | `div.hud-grid` | Full-viewport grid overlay |
| 2     | `div.hud-vignette` | Edge vignette darkening |
| 3     | `div.scanlines` | Repeating scanline texture |
| 4     | `div.scan-bar` | Sweeping CRT scan bar |
| 5     | `div.corner.tl` | Top-left bracket |
| 6     | `div.corner.tr` | Top-right bracket |
| 7     | `div.corner.bl` | Bottom-left bracket |
| 8     | `div.corner.br` | Bottom-right bracket |

`scanlines` and `scan-bar` are listed before the corners (plan-spec order). The `app.html` source uses a slightly different ordering (`hud-grid`, `hud-vignette`, `scan-bar`, `scanlines`, corners) but every class is absolutely-positioned with explicit `z-index`, so render order is visually equivalent — the plan's acceptance criteria checks each class literal independently, not the order.

Plan 05 will add body-class toggle effects (`no-scanlines`, `no-breadcrumb`, `no-footer`) that hide individual chrome elements via CSS rules in `site.css` — no DOM churn, the elements stay mounted.

## Location of the Plan 04 / Plan 05 Insertion Markers

Inside `HermesApp`'s rsx! body, immediately after `app_footer::AppFooter {}` and before the closing `}`:

```rust
rsx! {
    hud_chrome::HudChrome {}
    breadcrumb::Breadcrumb {}
    sys_meta::SysMeta {}
    div { class: "app", id: "app",
        screen_router::ScreenRouter {}
    }
    app_footer::AppFooter {}

    // Plan 04 inserts: wheel::Wheel {} wheel_rail::WheelRail {}
    // Plan 05 inserts: tweaks_panel::TweaksPanel {}

```

The submodule declarations `pub mod wheel;`, `pub mod wheel_rail;`, and `pub mod tweaks_panel;` are **deliberately absent** from `hermes_app/mod.rs`; Plans 04 and 05 add them alongside their respective implementations so each plan keeps cargo check green within its own worktree.

## ScreenRouter Mount Order

`screen_router.rs` mounts the 13 placeholders in this order (plan-locked, ScreenRouter is the only file Plans 06/07/08 do **not** touch):

```
Chat, Sessions, Agents, Skills, Models, Memory, Soul, Tools, Schedules, Gateway, Office, Settings, Providers
```

Each placeholder receives `is_active: cur == Screen::X` where `cur = *active.read()` (borrow dropped before rsx). Per RESEARCH Pattern 7, every `<section class="screen">` is always rendered; only the active one gets the `is-active` class.
