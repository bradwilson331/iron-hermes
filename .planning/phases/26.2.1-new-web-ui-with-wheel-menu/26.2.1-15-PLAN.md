<objective>
Remove the scanlines feature entirely from the iron_hermes_ui crate as the closure path for GAP-26.2.1-07-R3.

Background — Plans 10, 13, and 14 (Branch c) each attempted to make the scanlines toggle engage by escalating the CSS hide rule. Plan 10 added `body.no-scanlines .scanlines { display: none }`. Plan 13 added `!important`. Plan 14 escalated to `html body.no-scanlines .scanlines { display: none !important; visibility: hidden !important; opacity: 0 !important; }` — a (0,2,2)-specificity triple-guard that should be unbeatable by any normal CSS mechanism. None of these engaged in UAT. The live DOM diagnostic in Plan 14 Task 1 confirmed `<body class="density-dense no-scanlines on-chat">` lands correctly and `.scanlines` exists as a direct body child, yet the overlay still paints. After three failed fix attempts, the user decision (2026-05-14) is: stop trying to fix the toggle, remove the scanlines feature entirely.

Purpose: ship Phase 26.2.1 with a clean, fully-removed scanlines surface. The gap GAP-26.2.1-07-R3 is closed by REMOVAL, not by FIX. The disposition is recorded as `GAP-26.2.1-07-R3-FEATURE-REMOVAL` in UAT.md so future readers can audit the four-attempt history.

Output: 7 source files plus UAT.md, with zero residual scanlines references; localStorage-compatible serde migration so existing users do not see a hydration regression; one replacement static-grep test that prevents accidental re-introduction.

Scope — SCANLINES ONLY (D-26.2.1-15-B). The plan does NOT touch BREADCRUMB / FOOTER / RAIL / DENSITY toggles, the theme picker, the AccentColor picker, the wheel, chat, sessions, agents, models, providers, gateway, system, soul, schedules, office, or any other UI feature. ## Diagnostic / History Note

Plan 14's Task 1 captured a live DOM diagnostic with these results (re-stated here for traceability):

- D1 — `document.body.classList` includes `no-scanlines` when toggle is OFF (branch (a) — class-not-landing — RULED OUT)
- D2 — the `<div class="scanlines">` element exists as a direct body child with exactly one class `scanlines` (branch (b) — selector-mismatch — RULED OUT)
- D3 — no competing `!important` rule for `.scanlines` was found in any of the 9 CSS files; computed style after toggle showed `display: block` (cascade-side mystery — branch (c) selected)

Plan 14's Branch (c) escalation (`html` prefix + visibility + opacity triple-guard, all `!important`) at specificity (0,2,2) was, in principle, unbeatable. Round-3 UAT confirmed it STILL DID NOT engage. The remaining hypotheses (uninvestigated and now irrelevant per the removal decision) are:

- (h1) The visible "scanlines" effect may not be painted by the `.scanlines` element at all — it could be a `body::before` / `body::after` pseudo-element with a `repeating-linear-gradient` defined in one of the imported CSS files that the Plan 14 diagnostic did not inspect.
- (h2) A Dioxus / WASM rendering quirk where the element is re-mounted with a fresh stylesheet attachment that bypasses the cascade we are debugging.
- (h3) The user's browser may have a cached stylesheet that does not include Plan 14's escalation, but Plan 13's hard-reload test rules that out.

None of these need to be resolved. The user decision is to remove the feature, not to fix it.
