## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] Models screen Dioxus 0.7 keys-on-first-node rule**
- **Found during:** Task 1 (models.rs)
- **Issue:** The first draft of `models.rs` emitted two sibling `<div>`s inside an inner `rsx! { ... }` block (one for the section label, one for the grid), each with its own `key:`. Dioxus 0.7 warns: "Keys are only allowed on the first node in the block" (deprecated `_::Warning` symbol surfaced by clippy).
- **Fix:** Wrapped both children in a single keyed parent `<div class="model-family-group" key="{family}">`. The inner section-label + grid no longer carry keys; the wrapper carries the single per-family key. Inner ModelCards keep their per-model `key: "{m.id}"`.
- **Files modified:** `crates/iron_hermes_ui/src/components/hermes_app/screens/models.rs`
- **Commit:** `5948cab5` (fix is squashed into the Task 1 commit since it was caught during pre-commit verification)
- **New class introduced:** `.model-family-group` is a structural wrapper not in `app.html` / `screens.css`. It carries no styles in the bundle CSS, so it renders as a no-op block-level container — visually identical to the prototype which uses two sibling divs directly under `.screen`. Documented here per the plan's "If they differ in the prototype, use the prototype's names" rule.

### Interface Shape Note (NOT a deviation — already correct in Plan 02 output)

The plan's `<interfaces>` block declares `ModelStub::context_window` as `u32`, but the actual Plan 02 output in `src/mocks/stub_data.rs` types it as `&'static str` (values like `"128k"`, `"200k"`, `"8k"` — the prototype displays the K-suffixed strings directly). Plan 08 consumed the actual Plan 02 shape (`&'static str`), so no shape changes were needed in `stub_data.rs`. The interface-spec drift was in the plan documentation, not the code; this was caught while reading both before writing models.rs and resolved by consuming the real shape.
