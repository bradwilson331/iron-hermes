# Phase 26.2.1: new-web-ui-with-wheel-menu - Context

**Gathered:** 2026-05-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the existing Warp-style Dioxus shell in `crates/iron_hermes_ui/` with a new wheel-menu-driven SPA ported from a Claude-Design handoff bundle delivered at `crates/iron_hermes_ui/filestoimport/ironhermes-design-system/`. The handoff bundle includes:

- A single-page HTML prototype (`project/app.html`) with **12 screens** rendered as `<section class="screen">` panels (one active at a time)
- A **wheel menu** primitive (SVG, draggable + resizable, 10 wedges) implemented in `project/wheel-v2.js` + `project/wheel.css` — runtime-positioned in the upper-left by default
- **HUD chrome**: scanlines, hud-grid, hud-vignette, sweeping scan-bar, corner brackets (TL/TR/BL/BR), breadcrumb (`NODE HERMES-7 › BRIDGE › CHAT`), top-right sys-meta (BUILD/UPTIME/OP), bottom app-footer
- **5 themes** in `project/tokens.css` (oklch color space) — `slate-dark` (default), `slate-light`, `iron-dark`, `terminal-dark`, `parchment-light`
- A **Tweaks panel** (gear FAB → slide-up drawer) with runtime toggles for accent / wheelSize / scanlines / breadcrumb / footer / density / rail
- Per-screen styles in `project/screens.css`; component styles in `project/components.css`; base in `project/site.css` (which defines `--teal: #39c5cf`)
- JetBrains Mono via Google Fonts as the body face in the prototype; Ioskeley Mono also shipped in `project/assets/fonts/`

The phase delivers an **MVP+** slice: the new shell + theming + wheel + tweaks panel renders, **three** screens are live-wired to existing backend (Chat over the existing `/api/ws/chat` WebSocket; Sessions via the existing `StateStore`; Settings reading the existing `ConfigSummary`), and the remaining **nine** screens are pixel-faithful **visual stubs** that render the prototype layout with mocked data. The existing Warp-shell components are **feature-gated** behind a `legacy-shell` Cargo feature (off by default, kept compiling as a fallback) — they are NOT deleted in this phase.

**Out of scope (deferred to 26.2.2+):**
- Live wiring for Agents / Skills / Models / Memory / Soul / Tools / Schedules / Gateway / Office / Providers screens (each becomes its own follow-up phase or grouped follow-up)
- Settings **write-back** (mutating `config.yaml` from the UI) — Settings is read-only in this phase
- Deleting the `legacy-shell` feature/module (waits for UAT sign-off in a later phase)
- A "host-side" persistence for the Tweaks panel (`config.yaml` round-trip) — this phase uses browser-side persistence only
- Tailwind v4 conversion — CSS is ported verbatim from the bundle
- Desktop / mobile targets — web only, same as Phase 25.5
- Any change to the chat WebSocket protocol contract (`ws_chat` signature stays `Websocket<String, String>`)

</domain>

<decisions>
## Implementation Decisions

### Replacement Strategy
- **D-01: Replace, don't augment.** The new wheel-driven SPA becomes the **default** rendered UI when `iron_hermes_ui` is built without extra features. The current Warp-style shell (13 components in `src/components/shell/` + `src/components/warp_hermes.rs`) is moved behind a `legacy-shell` Cargo feature, default OFF. Both shells share the same backend (`src/server/`) and the same protocol types (`src/protocol.rs`). Planner decides whether the cfg-gate lives at the module level (`#[cfg(feature = "legacy-shell")] mod shell;`) or at the `App` root branch.
- **D-02: Backend is untouched.** Zero edits to `src/server/ws.rs`, `src/server/api.rs`, `src/server/state.rs`, or `src/protocol.rs`. The new shell consumes the existing contracts as-is. New server functions for currently-stubbed screens (Agents/Tools/Schedules/Soul/Memory/Providers/Gateway/Office/Skills) are explicitly deferred to follow-up phases.

### Scope — MVP+
- **D-03: Three screens live-wired in 26.2.1.** Chat, Sessions, Settings.
  - **Chat** — full WebSocket round-trip (streaming assistant deltas, tool-call progress rows per `chat-progress` markup in app.html, slash command echo).
  - **Sessions** — list / switch / delete via the existing session server functions from Phase 25.5 (`Platform::Web` session keys, `agent:main:web:{chat_id}` pattern).
  - **Settings** — **read-only** display of model / provider / context_length / memory_enabled / active theme / wheel prefs, fed by the existing `ConfigSummary` server function (extended by Phase 26.4 with `memory_enabled`). Write-back deferred.
- **D-04: Nine screens are pixel-faithful visual stubs.** Agents, Skills, Models, Memory, Soul, Tools, Schedules, Gateway, Office. Each stub:
  - Ports the HTML layout from the corresponding `<section id="screen-…">` block in `app.html` verbatim into RSX
  - Sources data from a single `src/mocks/stub_data.rs` module (planner-defined shape; mirror what app.html hardcodes)
  - Includes no server function calls — pure client-side render
  - This makes the visual port "done" so follow-up phases only plug in data sources
- **D-05: Providers is on the wheel but its screen is NOT in app.html.** `wheel-v2.js` exposes a `providers` wedge that targets `screens/Providers.tsx`. The HTML prototype `app.html` has no `#screen-providers` section. Planner adds a Providers screen as a 13th screen, layout derived from `project/screens/Providers.tsx` (the React component reference). It is a visual stub in 26.2.1 (same rule as D-04).

### Design Source of Truth
- **D-06: `project/app.html` is the canonical structural spec.** Port screen layouts, HUD chrome positions, wheel mounting, breadcrumb / sys-meta / footer markup directly from `app.html` into RSX. When `app.html` and the React/TSX references (`project/screens/*.tsx`, `project/components/*.jsx`) disagree, **app.html wins** for layout — TSX wins for component-internal logic.
- **D-07: CSS is copied verbatim from the bundle into `crates/iron_hermes_ui/assets/`.** Five files: `tokens.css`, `site.css`, `wheel.css`, `screens.css`, `components.css`. Linked from the root `App` via `document::Link { rel: "stylesheet", href: asset!("/assets/tokens.css") }` etc. No conversion to Tailwind or typed CSS-in-Rust in this phase. The existing `assets/warp-ih.css` stays put (it's only used by the now-feature-gated legacy shell). The bundle's `colors_and_type.css` (the older ANSI-derived token sheet) is NOT copied — `tokens.css` supersedes it for this UI.
- **D-08: JetBrains Mono via Google Fonts is the body face.** Matches the prototype (`app.html` line 10). Ioskeley Mono webfonts also ship in `assets/fonts/` (copied from `project/assets/fonts/`) for offline / theme variants. Planner decides whether to preload Ioskeley or let it lazy-load.
- **D-09: Asset copies.** Copy the design bundle's `project/assets/fonts/` and `project/assets/ih-shield-caduceus-transparent-256.png` (the chat assistant avatar) into `crates/iron_hermes_ui/assets/`. Other bundle assets (uploaded images, screenshots in `project/uploads/`) are NOT copied.

### Wheel Menu — `wheel-v2.js` Parity
- **D-10: Wheel structure** — 10 SVG wedges around a hub, exactly the section list in `wheel-v2.js`:

      chat, agents, models, tools, skills, memory, sessions, providers, gateway, settings

  (Soul, Schedules, Office are screens but NOT on the wheel — they are reachable via the Settings screen or a separate nav surface. Planner decides the reachability path for those three.)
- **D-11: Wheel interactions** — exact parity with v2.1:
  - **Hover wedge** → activate (`is-active` class on wedge / label / glyph; tooltip follows cursor on the rim)
  - **Click hub** → launch the currently-active wedge (set `is-active` on the matching `<section class="screen">`)
  - **Click wedge** → launch that wedge directly
  - **Drag rim** (the wheel circumference annulus) → translate the entire `.wheel-shell` (position: fixed, free placement)
  - **Drag SE-handle** (the floating orange resize ring orbiting outside the rim) → resize via `--wheel-size` CSS variable
  - **Esc** → close any open overlay (planner decides if "open overlay" applies — there are none in app.html besides the Tweaks panel)
- **D-12: Wheel state primitive** — `WheelState { position: (f64, f64), size: f64, active_wedge: WheelWedge }` lives in a Dioxus context provided at the `App` root (planner decides `use_signal` vs `use_context_provider`). Position and size are persisted to **browser localStorage** via `web_sys::window().local_storage()` under a single key (e.g. `ih.ui.wheel`). No server-side persistence in this phase.

### Theming
- **D-13: Default theme = `slate-dark`**, accent = teal (`#39c5cf` per `site.css` line 21, oklch equivalent `oklch(76% 0.14 210)` per `tokens.css`).
- **D-14: All five themes ship in 26.2.1** — slate-dark, slate-light, iron-dark, terminal-dark, parchment-light. Theme switching is a simple `data-theme` attribute write on `<html>` (Dioxus signal → `document.documentElement.setAttribute`). Settings exposes the picker (read + write — the theme write is the **one** Settings write-back allowed this phase since it's purely client-side state, no `config.yaml` mutation).
- **D-15: Theme persistence** — same localStorage mechanism as the wheel (key e.g. `ih.ui.theme`).

### Tweaks Panel (MVP-grade)
- **D-16: Tweaks panel ships in 26.2.1** — gear FAB at bottom-right, slide-up panel mirroring app.html's `.tweaks` block. Runtime toggles for: accent, wheelSize, scanlines, breadcrumb, footer, density, rail. State persisted via the same `ih.ui.*` localStorage keys. The `window.APP_TWEAKS` JSON object in `app.html` becomes a typed Dioxus `Signal<UiPrefs>` context.
- **D-17: "Rail" toggle** — `body.has-rail.on-chat #screen-chat { padding-left: 360px; }` rule in app.html implies a wheel-rail variant (wheel docked to a vertical rail on the chat screen). Implement the CSS rule + the toggle; the actual rail rendering matches `body.has-rail #wheel-rail { display: flex !important; }` from app.html. Planner inspects `app.html` around lines 100–110 for the exact contract.

### Chat Wiring
- **D-18: Reuse `#[get("/api/ws/chat")] ws_chat()`** from `src/server/ws.rs` byte-for-byte. The endpoint stays `Websocket<String, String>` with the existing keepalive (`KEEPALIVE_INTERVAL`) and close-handshake plumbing.
- **D-19: Streaming render contract** — each text frame arriving over the WebSocket appends to the current assistant bubble's text content. Tool-call progress (the `chat-progress` rows in app.html lines 397–401) is driven by structured messages in the existing protocol (server emits `[tool] web_search · 14 results across 9 domains`-style frames; planner reads `src/protocol.rs` for the exact `ChatMessage`/`MessageEvent` variants from Phase 25.5).
- **D-20: Slash commands** — surface `/research`, `/clear`, `/quit`, etc. by reusing the Phase 25.5 / 26.4 path through `CommandRouter` server function. The Chat input box has the same submit-on-Enter / Shift+Enter newline rule as the Warp shell's `input_box.rs` (Phase 22.3 D-15).

### Sessions Wiring
- **D-21: Reuse existing session server functions** from Phase 25.5 / 26.2. List / switch / delete operate on the existing `StateStore` via `Platform::Web` session keys (`agent:main:web:{chat_id}`).
- **D-22: Visual layout** — port `<section id="screen-sessions">` from app.html (line 416) verbatim. The Tab struct from Phase 26.2 (now in `src/state.rs` with `session_id` field) is reused; tab close button rules from Phase 26.2 D-09 (EventHandler-based, stop_propagation) carry forward.

### Settings Wiring (read-only)
- **D-23: Reuse `ConfigSummary` server function** as widened in Phase 26.4 (model_name, provider_name, context_length, memory_enabled). 26.2.1 reads these fields and renders them in the Settings screen layout (port `<section id="screen-settings">` from app.html line 1356).
- **D-24: The Settings screen also surfaces the UI prefs** — theme picker, accent picker, wheel toggles, density toggle — which write to localStorage immediately (the only Settings "writes" in this phase). All `config.yaml`-bound fields are read-only with a "(coming soon)" or disabled affordance per D-03.

### Legacy Shell — Feature Gate
- **D-25: `legacy-shell` Cargo feature, default OFF.** All 13 existing Warp-style components move behind this feature. Planner decides:
  - Whether to move source files to `src/components/shell_legacy/` (rename) or keep them in place with `#[cfg(feature = "legacy-shell")]` at every module decl
  - Whether the `App` root uses `if cfg!(feature = "legacy-shell") { LegacyShell {} } else { NewShell {} }` or two separate binary targets
  - The simpler default (recommended): rename `src/components/shell/` → `src/components/shell_legacy/`, gate the whole module, gate `WarpHermes` and its consumers similarly, branch in `app.rs`.
- **D-26: Types in `src/state.rs` are shared.** `Block`, `BlockEntry`, `CommandLine`, `Token`, `ToolCall`, `Message`, `Tab`, `PaletteItem`, `ShellSettings`, `Personality`, `TokenBudget` — these are NOT gated, both shells use them (the new shell will need some new types alongside, planner adds them). Mock-fixture functions (`demo_messages()` etc.) move to `src/mocks/` if not already there and are accessible to either shell.

### Out-of-scope reminders (do not implement in 26.2.1)
- New server functions for the 9 stubbed screens
- Mutating `config.yaml` from the UI
- Removing or replacing `src/protocol.rs`
- Cleaning up the legacy shell code (the gate is enough; deletion comes later)
- Desktop / mobile feature gates (web-only, same as Phase 25.5)
- Routing via `Routable` enum (the SPA pattern from app.html uses class-toggled sections — keep that pattern; Dioxus router can come later)

### Claude's Discretion
- **CSS strategy refinement** — copy the 5 `.css` files verbatim is recommended; if a file references a path that doesn't resolve (e.g. `assets/fonts/...` vs `fonts/...`), the planner rewrites the path, not the styles.
- **Wheel SVG generation** — `wheel-v2.js` builds the SVG dynamically. Planner picks one of: (a) translate the JS SVG-builder into a Dioxus `rsx!` block that emits the same SVG nodes, (b) keep the SVG construction in a small JS interop module loaded via `document::Script`, or (c) pre-render the SVG once (10 wedges are static) and hand-write it in RSX. (a) is recommended for full Dioxus reactivity.
- **HUD scan-bar animation** — CSS keyframes are allowed here (this is web, not the TUI; the "no animation except knight-rider" rule from the design system README applies to the terminal, not to this web port).
- **Persistence keys** — picker is planner's call; suggested prefix `ih.ui.` (`ih.ui.theme`, `ih.ui.wheel`, `ih.ui.tweaks`).
- **`MessageEvent` extraction for streaming** — planner reads `src/protocol.rs` and decides whether the new Chat screen reuses `Phase 25.5 D-01` server-function shape exactly or restructures (recommended: reuse, since Phase 26.1 / 26.2 already locked the websocket lifecycle).
- **Stub data shape** — define `src/mocks/stub_data.rs` with hardcoded `Vec<…>` for each stubbed screen, sourced verbatim from the hardcoded values in `app.html` and `project/screens/*.tsx`. Keep them in one module so follow-up phases can grep-and-replace one site per screen.
- **Soul / Schedules / Office reachability** — three screens that exist in `app.html` but are NOT on the wheel. Planner picks how to reach them (Settings sub-nav, breadcrumb deep link, or a secondary menu).
- **Wheel SE-handle accessibility** — the orange resize ring orbits *outside* the rim per `wheel.css`. Planner verifies the hitbox doesn't conflict with `body { overflow: hidden }` or scroll surfaces.

</decisions>
