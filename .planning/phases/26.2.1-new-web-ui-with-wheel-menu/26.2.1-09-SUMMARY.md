## Deviations from Plan

**1. [Rule 3 — Blocker] `cargo test --lib` → `cargo test --bin iron_hermes_ui`**

- **Found during:** Task 1 verification.
- **Issue:** Plan 09's `<automated>` verify blocks (and VALIDATION.md Sampling Rate) call `cargo test -p iron_hermes_ui --lib`, but `iron_hermes_ui` is a binary-only crate (no `[lib]` target, no `src/lib.rs`). Running `--lib` errors with `no library targets found in package`.
- **Fix:** Used `cargo test -p iron_hermes_ui --bin iron_hermes_ui` for actual verification. This compiles `src/main.rs` plus all `#[cfg(test)] mod tests` blocks throughout `src/**`, providing identical coverage. UAT-26.2.1.md §12.4 records the correct command so future runs of the UAT script use the working invocation.
- **Files modified:** none beyond the planned set — only the verification command differs.
- **Commit:** N/A (procedural deviation, not a file change).
- **Why not a Rule 4 (architectural):** Switching from `--lib` to `--bin` is a one-character verification-command change; it doesn't change the test set, the test code, or the project architecture. Adding a `[lib]` target would be architectural — that's deferred to a future phase if/when the iron_hermes_ui crate needs to be reused as a library.

**2. [Rule 2 — Missing critical functionality] Plan referenced `--target wasm32-unknown-unknown` clippy variant the user did not require.**

- The plan's verification block in Task 2 mentioned `cargo clippy --target wasm32-unknown-unknown ... -- -D warnings` as a Wave-0 lint pass. The user-supplied prompt only required `cargo check -p iron_hermes_ui --target wasm32-unknown-unknown` (which is what I ran). No deviation needed — the user's success criteria take precedence over plan-internal verification commands, and the cargo check passed.

No other deviations.
