## Listener Lifetime — `use_drop` (NOT `Box::leak`)

The window-level pointer listeners are installed inside `#[cfg(target_arch = "wasm32")]` using the PATTERNS-blessed idiom:

```rust
type PointerListener = Closure<dyn FnMut(WebPointerEvent)>;
let mut move_slot: Signal<Option<PointerListener>> = use_signal(|| None);
let mut up_slot:   Signal<Option<PointerListener>> = use_signal(|| None);

use_effect(move || {
    // ... build callbacks, register on window, stash in slots ...
});

use_drop(move || {
    if let Some(cb) = move_slot.write().take() {
        let _ = window.remove_event_listener_with_callback("pointermove", ...);
    }
    if let Some(cb) = up_slot.write().take() {
        let _ = window.remove_event_listener_with_callback("pointerup", ...);
    }
});
```

`Box::leak` does NOT appear anywhere in the code (verified by grep — only the doc-comment negation references it).

## Drag Clamp Margin

`DRAG_MARGIN = 33.0` (= 12 + RING_GAP + 7) — the conservative value from RESEARCH Pitfall 3, ensuring the orange resize ring nodes never cross the viewport edge. Applied inside the wasm32 pointermove handler:

```rust
let max_x = (win_w - shell_size - DRAG_MARGIN).max(DRAG_MARGIN);
let max_y = (win_h - shell_size - DRAG_MARGIN).max(DRAG_MARGIN);
let new_x = (d.start_pos.0 + dx).clamp(DRAG_MARGIN, max_x);
let new_y = (d.start_pos.1 + dy).clamp(DRAG_MARGIN, max_y);
```

### Minor Auto-fixes Applied (Rule 2)

**1. Compile error — wrong `MouseButton` path**
- **Found during:** Task 1 `cargo check`
- **Issue:** The plan-suggested `dioxus::events::MouseButton` does not exist in Dioxus 0.7.7
- **Fix:** Used `dioxus::html::input_data::MouseButton` (the canonical path)
- **Files modified:** `wheel.rs`
- **Commit:** `8893d7a7`

**2. RSX warning — keys allowed only on first sibling**
- **Found during:** Task 1 `cargo check`
- **Issue:** Loop body emitted two `text` siblings; both had `key:` attrs (`key: "g-{i}"` and `key: "l-{i}"`), and Dioxus 0.7 only allows the key on the FIRST node in a block
- **Fix:** Kept `key: "{i}"` on the first (glyph) text; removed the redundant key from the second (label) text
- **Files modified:** `wheel.rs`
- **Commit:** `8893d7a7`

**3. Dead-code warnings on wasm32-only items**
- **Found during:** Task 2 host clippy run
- **Issue:** `MIN_SIZE`, `MAX_SIZE`, `DRAG_MARGIN` and `DragState`/`ResizeState` field reads live ONLY inside the `#[cfg(target_arch = "wasm32")]` block. On the host (server) build target those references are excluded, so clippy's dead-code analysis fires
- **Fix:** Added `#[allow(dead_code)]` to each affected item with documentation explaining the wasm32-only consumption pattern; tests in `#[cfg(test)] mod tests` exercise the constants on host so the values themselves can't drift undetected
- **Files modified:** `wheel.rs`
- **Commit:** `0af740ab`

**4. `clippy::type_complexity` on Signal<Option<Closure<dyn FnMut(...)>>>**
- **Found during:** Task 2 wasm32 clippy run
- **Issue:** wasm32 target lints the listener_slot type annotations as too complex
- **Fix:** Factored the closure type into a local `type PointerListener = Closure<dyn FnMut(WebPointerEvent)>` alias and used `Signal<Option<PointerListener>>` in the bindings
- **Files modified:** `wheel.rs`
- **Commit:** `0af740ab`

### Task Sequencing Adjustment

The plan's Task 1 acceptance criteria included `cargo test -p iron_hermes_ui --lib wheel::`. The crate is a `[[bin]]` not a `[lib]`, so tests run via `--bin iron_hermes_ui` (not `--lib`). All 5 wheel tests pass under this invocation. Additionally, Task 1 needed to declare `pub mod wheel;` in `hermes_app/mod.rs` for the tests to discover the module — the plan technically deferred that to Task 2, but tests cannot run without the declaration. The declaration moved into Task 1 commit `8893d7a7`; Task 2 commit `41f3d85f` added `pub mod wheel_rail;` alongside it and the rsx-tree mounts.
