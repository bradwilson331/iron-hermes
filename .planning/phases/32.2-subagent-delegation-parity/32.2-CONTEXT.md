# Phase 32.2: subagent-delegation-parity - Context

**Gathered:** 2026-05-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 32.2 closes the parity gap between IronHermes's `delegate_task` implementation and the Python hermes-agent delegation feature set. Discovered while writing cron-job documentation: several features described in the canonical hermes-agent docs are missing from the Rust implementation.

Five capability areas:

1. **Nested orchestration** ŌĆö `role` parameter (`leaf` | `orchestrator`) + `max_spawn_depth` config + `orchestrator_enabled` kill switch
2. **Complete blocked-tools list** ŌĆö `clarify` and `send_message` added to the structurally-excluded set in `build_child_registry`
3. **Batch oversize ŌåÆ tool error** ŌĆö return `Err` when batch size exceeds `max_concurrent_children` (no silent truncation)
4. **Config key rename + per-call max_iterations** ŌĆö hard rename `subagent:` ŌåÆ `delegation:` in config.yaml; expose `max_iterations` per-call in the schema
5. **`/agents` tree view** ŌĆö thread `parent_id` through `SubagentInfo` and build a parentŌĆōchild tree in `SubagentRegistry`; wire tree display in all surfaces (tui_rata, gateway, web)

**Does NOT deliver:** diagnostic timeout dump file (deferred), batch processing changes beyond oversize guard, MemoryProvider delegation hooks (deferred to Phase 34b+).

</domain>

<decisions>
## Implementation Decisions

### Nested Orchestration (role + depth)

- **D-01:** `role` parameter is added to the `delegate_task` schema with two values: `"leaf"` (default ŌĆö current flat behavior, cannot delegate further) and `"orchestrator"` (retains `delegate_task` in child registry PLUS whatever toolsets were specified ŌĆö same toolsets-as-leaf rule).

- **D-02:** `max_spawn_depth` config field (default `1` = flat). At depth 1, `role="orchestrator"` is a **no-op** ŌĆö the child is silently downgraded to leaf and a `tracing::warn!` is emitted. Raise to `2` to allow orchestrators to spawn leaf grandchildren; `3` for three levels. Cap at 3. This mirrors the Python hermes-agent default behavior.

- **D-03:** `orchestrator_enabled: false` global kill switch in `delegation:` config forces every child to leaf regardless of the `role` parameter. No error ŌĆö silent downgrade + warn.

- **D-04:** `build_child_registry` receives the current depth and `max_spawn_depth`. When depth >= max_spawn_depth OR orchestrator_enabled=false, the `delegate_task` tool is excluded from orchestrator children's registry (same structural-exclusion pattern as today's leaf logic). `SubagentRunner::run_child` receives `child_depth: u32` to thread depth through the spawn chain.

### Blocked Tools (Complete Set)

- **D-05:** `clarify` and `send_message` are added to the structurally-excluded set in `build_child_registry`, blocked for **ALL** children regardless of role (leaf and orchestrator). Rationale: children have no `clarify_callback` channel (no user-interaction surface) and no platform session to send_message through. Python confirms: `_NEVER_PARALLEL_TOOLS = frozenset({"clarify"})` and `send_message` is platform-dependent. Full blocked set: `delegate_task` (leaf only; re-added for orchestrators up to depth limit), `skills`, `execute_code`, `cronjob`, `clarify`, `send_message`.

### Batch Oversize ŌåÆ Tool Error

- **D-06:** When `tasks.len() > max_concurrent_children`, `execute_batch` returns `Err` immediately with a descriptive message citing the configured limit and the config key to change. Remove the current `tasks_array.iter().take(max_batch)` truncation + `tracing::warn!` behavior. This matches the Python docs: "Batches larger than the limit return a tool error rather than being silently truncated."

### Config Key Rename + Schema Additions

- **D-07:** Hard rename `subagent:` ŌåÆ `delegation:` in config.yaml. No backward-compat alias ŌĆö the old key is dead. Update `SubagentConfig` struct field names: `max_subagents` ŌåÆ `max_concurrent_children`, `timeout_secs` ŌåÆ `child_timeout_seconds`. Add new fields: `max_spawn_depth: u32` (default 1), `orchestrator_enabled: bool` (default true).

- **D-08:** `max_iterations` is exposed as a per-call override in the `delegate_task` JSON schema (single-task and per-task-in-batch). Falls back to `delegation.max_iterations` config (default 50). The value is already threaded to `runner.run_child(max_iterations)` ŌĆö this is a schema exposure only.

- **D-09:** `delegation.max_concurrent_children` config (renamed from `max_subagents`, default 3) remains the semaphore ceiling. The schema description and DELEGATION.md doc are updated to reflect the new name.

### /agents Tree View

- **D-10:** `SubagentInfo.parent_id: Option<String>` field is populated in `AgentSubagentRunner::run_child` ŌĆö currently a `// v2.1+` TODO at line 185. When spawning an orchestrator child, the parent subagent's `subagent_id` is threaded down through `run_child` and stored in the new `SubagentInfo.parent_id`.

- **D-11:** `SubagentRegistry` grows a `build_tree()` method that returns nodes grouped by parent_id chain. Root nodes (parent_id = None) are top-level; their children nest underneath. Used by all three display surfaces.

- **D-12:** Three surfaces wire up the tree:
  - **tui_rata** (`/agents` overlay): render as indented text tree (ASCII `Ōö£ŌöĆŌöĆ`/`ŌööŌöĆŌöĆ`) with per-node: id (short 8-char prefix), task_summary (80-char clip), uptime, status pill
  - **gateway** (`/agents` slash command text response): plain-text indented tree ŌĆö same content, newline-separated
  - **web** (API surface): structured JSON with `children: []` nesting in the `/agents` response shape

### Claude's Discretion

- Exact `child_depth` threading approach (field on `AgentSubagentRunner` vs parameter to `build_child_registry`) ŌĆö pick cleanest fit given borrow checker constraints
- Whether to emit `cost`/`token` rollups per subtree in the tree view (Python docs mention per-branch cost rollup ŌĆö implement if it doesn't require significant new instrumentation, otherwise stub with zeroes)
- Diagnostic timeout dump (write `~/.ironhermes/logs/subagent-timeout-<session>-<ts>.log` when child times out with zero API calls) ŌĆö implement if it falls naturally out of the timeout path, otherwise defer

<specifics>
## Specific Ideas

- **Role is per-call, depth is per-install config.** The `role` parameter in a `delegate_task` call is the caller's intent; `max_spawn_depth` in config is the operator's safety ceiling. A call with `role="orchestrator"` on a default install (max_spawn_depth=1) silently degrades to leaf ŌĆö the operator controls whether orchestration is allowed at all.
- **Config rename is a breaking change.** Users MUST update `~/.ironhermes/config.yaml` from `subagent:` to `delegation:`. Add a helpful error message at startup if `subagent:` key is detected: "ŌÜĀ Config key 'subagent:' is deprecated and no longer supported. Rename it to 'delegation:' in your config.yaml."
- **Python hermes-agent blocked set for reference:** `delegate_task` (role-conditional), `clarify`, `send_message`, `memory` (write-blocked, read allowed), `code_execution`. IronHermes mapped: `delegate_task` (role-conditional), `clarify`, `send_message`, `skills`, `execute_code`, `cronjob`.

</specifics>
