## build_subagent_info Helper Extraction

**Was extraction required?** Yes.

`run_child` in production requires a live `AnyClient`, `ProviderResolver`, and async `AgentLoop` — not drivable from a synchronous unit test without heavy mocking. The plan explicitly provided this as an escape hatch: "if run_child is hard to drive, extract a `fn build_subagent_info(...)` pure helper and test it directly."

The extracted helper is `pub(crate) fn build_subagent_info(subagent_id, task_summary, caller_subagent_id, cancel, transcript_path) -> SubagentInfo` in `subagent_runner.rs`. Production `run_child` continues to use the inline struct literal (satisfying the acceptance criterion `grep -c "parent_id: self.caller_subagent_id.clone()"`). The helper is separate and tested directly in `test_run_child_propagates_caller_id_to_subagent_info`.

## Stable Ordering Guarantee

**Field:** `info.started_at` (`std::time::Instant`)

`SubagentRegistry::build_tree()` and `children_of()` both call `.sort_by_key(|n| n.info.started_at)`. `Instant` implements `Ord` so the sort is total and deterministic for distinct timestamps.

**Ties:** Two nodes with the exact same `started_at` value are practically impossible — `Instant::now()` resolution is nanosecond on macOS/Linux and consecutive calls always differ. In the event of a tie (e.g. constructed with a fixed `Instant` in tests), the sort is stable (Rust's `sort_by_key` is stable), so insertion order into the sort input is preserved. The sort input is a `Vec` collected from `HashMap::values()`, which has non-deterministic iteration order. If deterministic tie-breaking ever matters, add a secondary sort on `id` (lexicographic) — not needed today.

## Example: 3-node tree → tree_summary output

**Setup:** root agent `sub_root` spawns two children `sub_child_a` and `sub_child_b`. `sub_child_b`'s cancel token is triggered before `tree_summary()` is called.

```rust
// Registration order: root, child_a, child_b (child_b pre-cancelled)
registry.register(SubagentInfo { id: "sub_root", parent_id: None, ... });
registry.register(SubagentInfo { id: "sub_child_a", parent_id: Some("sub_root"), ... });
registry.register(SubagentInfo { id: "sub_child_b", parent_id: Some("sub_root"), cancel: cancelled_token, ... });

let entries = handle.tree_summary();
```

**Resulting `Vec<SubagentTreeEntry>` (pre-order depth-first, root before children):**

```
[
  SubagentTreeEntry {
    id: "sub_root",
    task_summary: "orchestrate the pipeline",
    uptime: Duration::from_secs(42),
    status: "running",
    parent_id: None,
    depth: 0,
  },
  SubagentTreeEntry {
    id: "sub_child_a",
    task_summary: "fetch and parse data",
    uptime: Duration::from_secs(30),
    status: "running",
    parent_id: Some("sub_root"),
    depth: 1,
  },
  SubagentTreeEntry {
    id: "sub_child_b",
    task_summary: "write output files",
    uptime: Duration::from_secs(25),
    status: "killed",       // cancel token was triggered
    parent_id: Some("sub_root"),
    depth: 1,
  },
]
```

Children are sorted by `started_at` (earliest first), so `sub_child_a` before `sub_child_b` if it started first. The root appears at `depth: 0`; both children at `depth: 1`. Plan 05 uses this flat `Vec<SubagentTreeEntry>` to render the ASCII tree via depth + parent_id relationships.
