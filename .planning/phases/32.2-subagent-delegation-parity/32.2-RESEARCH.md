### Locked Decisions

**Nested Orchestration (role + depth)**

- D-05: `clarify` and `send_message` added to structural-exclusion set in `build_child_registry`, blocked for ALL children regardless of role.

## Summary

Phase 32.2 is a pure Rust refactor-and-extension phase — no new external dependencies, no new crates. All five capability areas operate within the existing crate graph (`ironhermes-core`, `ironhermes-tools`, `ironhermes-agent`, `ironhermes-cli`). The work is well-scoped: the codebase has clear TODO markers for `parent_id` threading (line 185 of `subagent_runner.rs`), existing structural patterns for tool exclusion (`match tool_name.as_str()` arms in `build_child_registry`), and a clean `SubagentConfig` struct that needs field additions and renames.

The highest-risk area is the **config key rename** (D-07): `config.subagent` is read at 31 distinct call sites in `main.rs` alone, plus the `SubagentConfig` struct is referenced in `ironhermes-core`, `ironhermes-tools`, `ironhermes-agent`, and `ironagent-tools-api`. The rename must be done atomically across all four crates. A helpful startup error message for the old `subagent:` key is required per the CONTEXT specifics. The existing serde `#[serde(default)]` annotation on `SubagentConfig` means forward-compat (config with no `delegation:` section) works automatically.

The second area requiring care is the **`/agents` tree view** — the current `SubagentListSnapshot` trait (in `ironhermes-core`) exposes only `list_summary() -> Vec<(String, String, Duration)>`. The tree view requires `parent_id` to be visible at the trait boundary. The cleanest approach is to add a new `tree_summary()` method to the trait returning a richer type, OR to extend the existing tuple to include `Option<String>` (parent_id). A decision on this shape drives both the `build_tree()` method and the three render surfaces.

**Primary recommendation:** Plan as five independent waves in dependency order: (1) Config rename, (2) Blocked tools addition, (3) Oversize error, (4) Nested orchestration, (5) Tree view — each wave is a self-contained diff that compiles and passes tests before the next begins.

### Pattern 2: Depth-Conditional Orchestrator Re-Addition

**What:** When `role = "orchestrator"`, `delegate_task` must be re-added to the child's allowed set BEFORE the structural-exclusion pass. The cleanest approach is to expand `allowed_tools` before the match loop.

```rust
// Pseudocode — exact implementation is Claude's discretion (D-04 note)
fn build_child_registry(
    allowed_tools: &[String],
    memory_manager: Option<SharedMemoryManager>,
    child_cwd: &Path,
    role: ChildRole,              // NEW: Leaf | Orchestrator
    child_depth: u32,             // NEW
    config: &SubagentConfig,      // NEW: reads max_spawn_depth + orchestrator_enabled
) -> anyhow::Result<ToolRegistry> {
    // Orchestrator re-add: only when role=orchestrator AND not depth-exceeded AND not kill-switched
    let effective_tools: Vec<String> = if matches!(role, ChildRole::Orchestrator)
        && child_depth < config.max_spawn_depth
        && config.orchestrator_enabled
    {
        let mut t = allowed_tools.to_vec();
        if !t.iter().any(|s| s == "delegate_task") {
            t.push("delegate_task".into());
        }
        t
    } else {
        if matches!(role, ChildRole::Orchestrator) {
            tracing::warn!("role=orchestrator ignored: depth={} max_spawn_depth={} orchestrator_enabled={}",
                child_depth, config.max_spawn_depth, config.orchestrator_enabled);
        }
        allowed_tools.to_vec()
    };
    // ... existing match loop on effective_tools ...
}
```

### Pattern 4: `build_tree()` — Pure HashMap Transformation

**What:** `SubagentRegistry` already holds `active: HashMap<SubagentId, SubagentInfo>`. `build_tree()` is a pure data transformation.

```rust
// Proposed implementation — pure Rust, no new deps
pub struct SubagentTreeNode {
    pub info: SubagentInfo,
    pub children: Vec<SubagentTreeNode>,
}

impl SubagentRegistry {
    pub fn build_tree(&self) -> Vec<SubagentTreeNode> {
        // 1. Collect roots (parent_id = None)
        // 2. For each root, recursively find children by parent_id match
        // 3. Return sorted Vec<SubagentTreeNode>
        let mut roots: Vec<SubagentTreeNode> = self.active.values()
            .filter(|info| info.parent_id.is_none())
            .map(|info| SubagentTreeNode {
                info: info.clone(),
                children: self.children_of(&info.id),
            })
            .collect();
        roots.sort_by_key(|n| n.info.started_at);
        roots
    }

    fn children_of(&self, parent_id: &str) -> Vec<SubagentTreeNode> {
        let mut children: Vec<SubagentTreeNode> = self.active.values()
            .filter(|info| info.parent_id.as_deref() == Some(parent_id))
            .map(|info| SubagentTreeNode {
                info: info.clone(),
                children: self.children_of(&info.id),
            })
            .collect();
        children.sort_by_key(|n| n.info.started_at);
        children
    }
}
```

### Pattern 5: ASCII Tree Rendering (tui_rata)

**What:** Standard indented tree format used in git / ls-tree style display.


### Anti-Patterns to Avoid

- **Don't use `serde(alias)` for the old `subagent:` key**: D-07 explicitly says no backward-compat alias. Use startup detection + helpful error instead.
- **Don't re-add `delegate_task` AFTER the structural exclusion match loop**: It must happen BEFORE (expand the tools list first, then the match loop runs on the expanded list). Otherwise the existing `"delegate_task" => {}` arm silently strips it.
- **Don't pass `parent_id` as a new parameter to `build_child_registry`**: `parent_id` is a property of the running subagent (known in `AgentSubagentRunner::run_child`), not of the registry build. Build the registry first, then create `SubagentInfo` with `parent_id`.
- **Don't forget `ironagent-tools-api` crate**: It has a parallel copy of `build_child_registry` and `DelegateTaskTool` in `crates/ironagent-tools-api/src/delegate_task.rs`. Both copies need identical changes. [VERIFIED: codebase grep — `max_subagents` appears in both]
- **Don't break the test `test_batch_truncates_to_max_subagents`**: It currently asserts truncation behavior. Replace with a test that asserts `Err` on oversize.

## Common Pitfalls

### Pitfall 1: Orchestrator Re-Addition After Exclusion

**What goes wrong:** Adding `delegate_task` to the allowed list AFTER the match loop runs means the existing `"delegate_task" => {}` arm has already silently stripped it.
**Why it happens:** The expansion logic is conceptually "add the tool back," but structurally the match loop runs first.
**How to avoid:** Expand `effective_tools` BEFORE passing to the match loop. The match loop then sees `delegate_task` in the list and hits the arm — so the arm must be made conditional: only strip when the effective role is leaf.
**Warning signs:** `test_no_recursive_delegation` passes but orchestrators still can't spawn children.

### Pitfall 2: Missing `ironagent-tools-api` Updates

**What goes wrong:** `crates/ironagent-tools-api/src/delegate_task.rs` is a parallel copy of the tools crate's `delegate_task.rs`. Changes to `ironhermes-tools` are not automatically mirrored there.
**Why it happens:** The codebase has two copies — `ironhermes-tools` and `ironagent-tools-api` — both referencing `SubagentConfig.max_subagents`.
**How to avoid:** Search for `max_subagents` across all crates before declaring completion. Both copies need identical changes to `build_child_registry` and `execute_batch`.
**Warning signs:** `cargo build` passes on one crate but fails on the other.

### Pitfall 3: `config.subagent` → `config.delegation` — 31 Call Sites in main.rs Alone

**What goes wrong:** Rust rename via grep-and-replace misses one site; compiler catches it but the diff is large.
**Why it happens:** The config field is accessed as `config.subagent.max_subagents`, `config.subagent.timeout_secs`, etc. across run_chat, run_single, run_gateway code paths.
**How to avoid:** Use `cargo check` to find all compilation errors after the struct rename; fix systematically. Write a grep invariant test: `assert!(source.contains("config.delegation."), "config.subagent must not appear post-rename")`.
**Warning signs:** `error[E0609]: no field subagent on type Config` at multiple sites.

### Pitfall 4: `SubagentListSnapshot` Trait — Tree Data Not Exposed

**What goes wrong:** The tree view needs `parent_id` but `SubagentListSnapshot::list_summary()` returns `Vec<(String, String, Duration)>` — no parent_id in the tuple. The trait is defined in `ironhermes-core`, the impl in `ironhermes-agent`. Adding `build_tree()` to the trait means the tuple must be extended OR a new trait method added.
**Why it happens:** The trait was designed for flat listing only.
**How to avoid:** Add a new method `tree_summary() -> Vec<SubagentTreeEntry>` to the trait with a default impl that calls `list_summary()` and returns flat nodes (no children), OR add `parent_id: Option<String>` as the 4th element of the tuple. Either approach avoids breaking existing impls.
**Warning signs:** Compile error in `SubagentRegistryHandle` impl because new trait method not implemented.

### Pitfall 5: Batch Oversize Test Breakage

**What goes wrong:** The existing test `test_batch_truncates_to_max_subagents` asserts truncation (2 tasks from a 4-task batch). After D-06, this behavior changes to an Err. The test will fail.
**Why it happens:** The test validates the old behavior that is being replaced.
**How to avoid:** Replace the test body: `test_batch_oversize_returns_err` — assert the call returns `Err` containing the config key name.
**Warning signs:** `test_batch_truncates_to_max_subagents` fails during `cargo test`.

### Pitfall 6: `child_depth` Threading — Borrow Checker Constraint

**What goes wrong:** Adding `child_depth: u32` to `SubagentRunner::run_child` changes the trait signature, which requires updating all impls (including `MockRunner` and test stubs).
**Why it happens:** `SubagentRunner` is a trait with multiple impls across test code.
**How to avoid:** Use Claude's Discretion guidance — consider whether `child_depth` goes on the trait method signature OR on `AgentSubagentRunner` as a field set before spawning. A field approach avoids breaking the trait. The CONTEXT says "pick cleanest fit given borrow checker constraints."
**Warning signs:** Multiple `E0061: this function takes N arguments but M were supplied` errors across test files.

## Open Questions

1. **Tree view trait extension approach**
   - What we know: `SubagentListSnapshot` in `ironhermes-core` returns flat `Vec<(String, String, Duration)>` — no `parent_id`.
   - What's unclear: Should we (a) add `tree_summary()` as a new trait method with default flat impl, (b) extend the tuple to 4 elements `(id, summary, uptime, parent_id)`, or (c) add `parent_id` only to the `SubagentRegistry::build_tree()` return type and bypass the trait for tree rendering?
   - Recommendation: Option (a) — add `tree_summary()` with a default implementation calling `list_summary()`. This keeps existing impls (test fakes in `cmd_agents_and_stop.rs`) working without changes.

2. **`child_depth` propagation approach**
   - What we know: CONTEXT says "Claude's Discretion — field on `AgentSubagentRunner` vs parameter to `build_child_registry`." Changing `SubagentRunner::run_child` signature breaks all test stubs.
   - What's unclear: Whether `AgentSubagentRunner` knows its own current depth at construction time or only at `run_child` call time.
   - Recommendation: Add `current_depth: u32` as a field on `AgentSubagentRunner`. When spawning a child runner, clone the runner and increment the depth field. This avoids trait signature changes entirely.

3. **Web API `/agents` surface**
   - What we know: No files in `website/` contain `SubagentInfo`, `list_summary`, or an `/agents` route (grep found no results).
   - What's unclear: D-12 specifies structured JSON with `children: []` for the web surface. Is this a new endpoint to create, or a stub that already exists?
   - Recommendation: Investigate `website/` API route registration before planning the web surface task. If no endpoint exists, create it as part of this phase. If it exists as a stub, wire `build_tree()`.
