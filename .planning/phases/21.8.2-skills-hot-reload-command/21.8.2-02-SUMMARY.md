# Phase 21.8.2 Plan 02: Command Foundation — SkillsReload Variants + with_skill_registry Wiring Summary

**One-liner:** CommandResult::SkillsReload + SkillActivated variants added to core; with_skill_registry wired at all 4 CommandContext construction sites closing the long-broken /skills→"No skills loaded." gap.

## What Was Built

### Task 1: SkillsReload + SkillActivated variants + cmd_skills reload arm

- Added `CommandResult::SkillsReload` variant to `ironhermes-core/src/commands/mod.rs` (parallel to McpReload)
- Added `CommandResult::SkillActivated { name: String, body: String }` variant (for SKILL-13 fallback in Plan 03)
- Updated `cmd_skills` to accept `args: &[&str]` and dispatch on `args[0] == "reload"`:
  - With registry: returns `CommandResult::SkillsReload`
  - Without registry: returns `Output("Skills not configured.")`
  - No args / "list" / other: preserves existing catalog behavior
- Updated dispatch table call site: `"skills" => cmd_skills(args, ctx)`
- 6 new unit tests: 2 variant existence tests in mod.rs, 4 cmd_skills behavior tests in handlers.rs

### Task 2: Wire with_skill_registry at all 4 CommandContext sites + exhaustiveness arms

- **main.rs `build_cmd_ctx` helper**: Added `skill_registry: Option<Arc<SkillRegistry>>` parameter; wired `.with_skill_registry()` inside helper body — covers both `run_chat` dispatch sites (prompt-time at ~1493, mid-turn at ~1807)
- **main.rs run_chat**: Changed `let skill_registry` → `let mut skill_registry`; fixed `set_skill_registry` to use `.clone()`; passed `Some(skill_registry.clone())` at both build_cmd_ctx call sites
- **main.rs match blocks**: Added `SkillsReload` + `SkillActivated` arms to both match blocks:
  - Turn-end dispatch (~1538): `unreachable!()` placeholder (Plan 03 replaces)
  - Mid-turn dispatch (~1860): dimmed notice (mirrors mid-turn McpReload pattern)
- **tui_rata/app.rs**: Added `skill_registry: Option<Arc<ironhermes_core::SkillRegistry>>` to both `AppDeps` and `App` structs; forwarded in `App::new`
- **tui_rata/event_loop.rs**: Added `skill_registry: Some(skill_registry.clone())` to `AppDeps` literal in `build_app_deps`
- **tui_rata/commands.rs**: Added `.with_skill_registry()` to `build_command_context`; added `SkillsReload` + `SkillActivated` arms to `map_core_to_slash_outcome`
- **tui/extension.rs**: Added `SkillsReload` + `SkillActivated` variants to classic TUI `CommandResult` (needed for exhaustiveness in tui/commands.rs map_core_to_tui)
- **tui/commands.rs**: Added exhaustive arms for `SkillsReload` + `SkillActivated` in `map_core_to_tui`; pass through to extension::CommandResult variants
- **gateway/handler.rs**: Added `.with_skill_registry()` to `handle_slash_command` CommandContext chain; added `SkillsReload` + `SkillActivated` arms to `match core_result` using `with_rate_limit_retry + send_message`
- **5 new static-grep invariant tests**: 4 in `ironhermes-cli/tests/skill_registry_wiring.rs`, 1 in `ironhermes-gateway/tests/skill_registry_wiring.rs`

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Classic TUI extension::CommandResult also needed new variants**
- **Found during:** Task 2 build
- **Issue:** `crates/ironhermes-cli/src/tui/commands.rs:119` `map_core_to_tui` is exhaustive on `CoreCommandResult`. When the new variants were added to core, this function needed arms for them. The plan only mentioned `tui_rata/commands.rs` but the classic TUI `tui/commands.rs` (used by `dispatch_command` called from `run_chat`) also has an exhaustive match.
- **Fix:** Added `SkillsReload` + `SkillActivated` variants to `tui/extension.rs` `CommandResult` enum; added pass-through arms in `tui/commands.rs` `map_core_to_tui`.
- **Files modified:** `crates/ironhermes-cli/src/tui/extension.rs`, `crates/ironhermes-cli/src/tui/commands.rs`

**2. [Rule 1 - Bug] Gateway send_text method doesn't exist on PlatformAdapter**
- **Found during:** Task 2 build
- **Issue:** The plan's Edit 5 snippet used `adapter.send_text(event, msg).await` but `PlatformAdapter` only has `send_message(&chat_id, &text, None)`.
- **Fix:** Used `with_rate_limit_retry(|| adapter.send_message(&event.chat_id, msg, None)).await` matching the existing gateway pattern.
- **Files modified:** `crates/ironhermes-gateway/src/handler.rs`

**3. [Rule 1 - Bug] skill_registry.clone() passed to build_cmd_ctx without Some() wrapper**
- **Found during:** Task 2 build
- **Issue:** `build_cmd_ctx` expects `Option<Arc<SkillRegistry>>` but `skill_registry` in run_chat is `Arc<SkillRegistry>` (not Option). Both call sites needed `Some(skill_registry.clone())`.
- **Fix:** Wrapped both call sites with `Some(...)`.
- **Files modified:** `crates/ironhermes-cli/src/main.rs`

**4. [Rule 2 - Adaptation] Static-grep test lowered from >= 2 to >= 1 for main.rs with_skill_registry count**
- **Found during:** Task 2 test run
- **Issue:** The plan's test expected `>= 2` occurrences of `.with_skill_registry(` in main.rs (one for run_chat, one for run_single), but since Phase 21.7-11 extracted both into the shared `build_cmd_ctx` helper, there's only 1 occurrence. The plan was written assuming separate call sites.
- **Fix:** Lowered assertion to `>= 1` with comment explaining that build_cmd_ctx covers both dispatch sites.
- **Files modified:** `crates/ironhermes-cli/tests/skill_registry_wiring.rs`
