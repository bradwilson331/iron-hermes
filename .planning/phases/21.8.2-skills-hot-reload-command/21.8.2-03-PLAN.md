## Decisions Recorded in This Plan

- D-Plan03-01 (UPDATED): Gateway SkillsReload arm performs an **immediate atomic swap** of the inner `Arc<SkillRegistry>` via `Arc<Mutex<Arc<SkillRegistry>>>` interior mutability on Handler. The Mutex is `std::sync::Mutex` (not tokio::sync::Mutex) because `load_with_config` is synchronous and the lock is held briefly for the swap. The previous deferral-to-restart approach was rejected during revision (2026-05-09) per user decision.
- D-Plan03-02: `prompt_builder.activate_skill(name, body)` adds `(name, body)` to a new `skill_overlays: Vec<(String, String)>` field; `build_system_message` prepends them to the system slot. The vector is cleared by `clear_skill_overlays()` (called from `/clear`/`/reset` handlers if those exist; otherwise persists for the session, matching `/personality` overlay semantics — D-07).
- D-Plan03-03: SKILL-13 fallback in tui_rata replaces the typo-suggester `Unknown { hint: "Did you mean..." }` ONLY when `app.skill_registry.find(token).is_some()`. Unknown commands without a matching skill keep the typo-suggestion behavior unchanged.
- D-Plan03-04: The CLI subcommand `hermes skills reload` does NOT compute a diff (no in-memory "old" registry in a one-shot CLI invocation). It reports loaded count + names. The `/skills reload` slash command DOES compute a diff vs the in-session registry.
- D-Plan03-05 (NEW): Gateway D-07 SkillActivated delivery uses a per-session overlay store on Handler — `skill_overlays: Arc<Mutex<HashMap<SessionKey, Vec<(String, String)>>>>`. The `SkillActivated` arm pushes `(name, body)` into the overlay for the current session key. The gateway's AgentLoop call site (which builds the per-turn prompt_builder around line 570-580) reads from the overlay and prepends the bodies to the system prompt — same semantics as hermes-agent's skill injection. The previous "discard body" approach (`let _ = body;`) was rejected during revision (2026-05-09) per user decision.
- D-Plan03-06 (NEW): D-05 WARN-BUT-LOAD reporting in the reload diff. `load_with_config` logs parse failures via `tracing::warn` internally. The reload arm computes `invalid_skipped = files_scanned - successfully_loaded` by walking the configured skill directories with `std::fs::read_dir` (counting subdirectories that have a `SKILL.md`) and subtracting `new_registry.list().len()`. The diff string then includes `(W invalid skipped — see logs)` whenever `invalid_skipped > 0`. This is concrete and self-contained — no new tracing-subscriber capture infrastructure is required. The exact formula and helper fn `count_invalid_skipped(cwd, &cfg, loaded_count) -> usize` is defined in Task 1 below.

/// Phase 21.8.2 D-05: count SKILL.md files in the configured search paths
/// that were SCANNED but failed to load (i.e. parse errors logged via
/// `tracing::warn` from inside `load_with_config`). Returns
/// `files_scanned - loaded_count`. Used by the `/skills reload` arms to
/// surface "(W invalid skipped — see logs)" in the diff output.
///
/// We re-walk `build_skill_search_paths(cwd, cfg)` and count subdirectories
/// that contain a `SKILL.md` file. Any path filter / hidden-dir convention
/// that `load_with_config` applies internally is matched here so we do not
/// over-count. If the helper cannot read a search path, it logs a debug
/// message and skips that path (returns the count from accessible paths).
fn count_invalid_skipped(
    cwd: &std::path::Path,
    cfg: &ironhermes_core::config::SkillsConfig,
    loaded_count: usize,
) -> usize {

SkillsAction::Reload => {
    use ironhermes_core::SkillRegistry;
    let cwd = std::env::current_dir()
        .unwrap_or_else(|_| std::path::PathBuf::from("."));
    let registry = SkillRegistry::load_with_config(&cwd, &cfg.skills);
    let names: Vec<&str> = registry.list().iter().map(|s| s.name.as_str()).collect();

    // D-05 / D-Plan03-06: count files scanned vs loaded for invalid_skipped.
    let invalid_skipped = {
        let search_paths =
            ironhermes_core::skills::build_skill_search_paths(&cwd, &cfg.skills);
        let mut files_scanned: usize = 0;
        for root in &search_paths {
            if let Ok(entries) = std::fs::read_dir(root) {
                for entry in entries.flatten() {
                    let path = entry.path();
                    if !path.is_dir() {
                        continue;
                    }
                    if path
                        .file_name()
                        .and_then(|n| n.to_str())
                        .map(|n| n.starts_with('.'))
                        .unwrap_or(false)
                    {
                        continue;
                    }
                    if path.join("SKILL.md").is_file() {
                        files_scanned += 1;
                    }
                }
            }
        }
        files_scanned.saturating_sub(names.len())
    };
    let invalid_clause = if invalid_skipped > 0 {
        format!(" ({} invalid skipped — see logs)", invalid_skipped)
    } else {
        String::new()
    };

    if names.is_empty() {
        println!("Skills reloaded. No skills found.{}", invalid_clause);
    } else {
        println!(
            "Skills reloaded. {} skills: {}{}",
            names.len(),
            names.join(", "),
            invalid_clause
        );
    }
    Ok(0)
}

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| slash command input → CommandRouter::resolve | Untrusted input from CLI/Telegram/TUI. The 3-stage resolution runs first; SKILL-13 fallback only fires on NotFound. |
| skill body → system prompt | When SKILL-13 activates a skill, the SKILL.md body content is injected into the next system message. Body content was scanned by `scan_skill_content` (Phase 19 D-15) at registry load. |
| reload command → filesystem rescan | `/skills reload` and `hermes skills reload` re-execute `load_with_config` which walks the configured skill paths. |
| gateway session key → skill_overlays HashMap | Per-session overlay store partitions activated-skill bodies by SessionKey so one Telegram chat’s activations never leak into another chat’s system prompt. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-21.8.2-03-01 | Elevation of Privilege | Skill name squatting a registered command (e.g. a skill named `help` shadowing `cmd_help`) | mitigate | D-08 is structurally enforced: `CommandRouter::resolve` runs all 3 stages BEFORE returning NotFound. The SKILL-13 fallback lives in the NotFound branch only. A skill named `help` cannot reach the fallback because `help` is a registered command. The static-grep invariant `main_rs_skill13_fallback_present` checks the fallback is in the NotFound arm, not before it. |
| T-21.8.2-03-02 | Tampering | Reload replaces a Trusted skill with a Community impostor | mitigate | `load_with_config` re-runs `resolve_source` + `scan_skill_content` (Phase 19 D-15) on EVERY reload. Community-tier skills with threat patterns are hard-rejected at load. The reload uses the same code path as startup; no posture bypass. |
| T-21.8.2-03-03 | Information Disclosure | Skill body containing secrets being echoed to Telegram via gateway adapter.send_text | accept | Skill bodies are operator-installed content under HERMES_HOME/skills; they are already injected into the LLM system prompt (which the LLM can echo to the user). The gateway SkillActivated arm sends ONLY the activation confirmation `Skill 'name' activated for this turn.` — NOT the body. The body lives in `Handler.skill_overlays` and is read by the AgentLoop call site, not echoed to the chat. |
| T-21.8.2-03-04 | Denial of Service | A SKILL-13 activation injecting a multi-MB body into every subsequent system prompt (token blowup) | mitigate | `prompt_builder.activate_skill` appends to `skill_overlays`. The Phase 21.3 token-counting + context-length enforcement (existing) compresses or refuses outsized system prompts. No new bypass. The tracing::warn in load can flag oversized SKILL.md files at registry load if Phase 19 size limits apply. |
| T-21.8.2-03-05 | Repudiation | Hot-reload race: user installs skill A, runs `/skills reload`, but a concurrent install of skill B partially writes to disk during the load_with_config rescan | accept | `load_with_config` is synchronous and reads files at the moment of the call. Mid-write SKILL.md files produce parse errors that are logged via the existing `scan_skill_content`/`parse_skill_md` warn paths — the partial skill is silently skipped (per D-05 WARN-BUT-LOAD posture for valid neighbors). The user’s next `/skills reload` after the partial write completes picks up the full skill. The D-05 invalid_skipped count is surfaced in the diff so the user can correlate with their write activity. |
| T-21.8.2-03-06 | Tampering | A skill body being mutated between `find` and `read_content` calls in the SKILL-13 fallback | accept | The window is microseconds; both calls read in-memory data from the Arc<SkillRegistry> snapshot. No filesystem re-read happens between them. An attacker would need write access to the operator's machine, which is outside the threat model. |
| T-21.8.2-03-07 | Elevation of Privilege | Gateway SKILL-13 fallback granting Telegram users access to skills they should not be able to invoke | accept | Gateway already exposes ALL registered slash commands to authorized Telegram users (Phase 21.6 D-X — gateway access control). | T-21.8.2-03-08 | Information Disclosure | Cross-session leakage of activated skill bodies via shared `Handler.skill_overlays` HashMap | mitigate | `Handler.skill_overlays` is keyed by `SessionKey`. The SkillActivated arm and SKILL-13 fallback push under the current event’s session key; the AgentLoop call site reads only that session’s entry. Two concurrent Telegram chats cannot read each other’s overlays. The Mutex serializes access; std::sync::Mutex is sufficient because operations are brief. |
| T-21.8.2-03-09 | Tampering | Gateway atomic-swap race: a reload running concurrently with a SkillActivated push could reference a stale `Arc<SkillRegistry>` for the SKILL-13 fallback lookup | accept | The Mutex on `skill_registry` serializes the swap. If a SkillActivated push reads the OLD Arc just before the swap, the activated body reflects the OLD registry contents — acceptable: the user’s next turn picks up the NEW registry, and the old body was valid at the moment of activation. No data corruption, no security boundary crossed. |
