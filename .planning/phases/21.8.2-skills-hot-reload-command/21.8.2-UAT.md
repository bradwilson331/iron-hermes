## Gaps

### G-01: System-role messages from `/skills reload` don't auto-scroll TUI/CLI viewport
- **Found in test:** 4
- **Severity:** low (UX papercut — feature works, scroll behavior wrong)
- **Surface:** rata TUI AND classic CLI REPL — confirmed cross-surface in tests 4 and 9. Telegram unaffected.
- **Symptom:** After typing `/skills reload`, the reload diff is appended to the transcript as a `Role::System` message but the viewport does not scroll to reveal it. The user has to send a real agent message (any prompt) before the new System line becomes visible.
- **Likely root cause:** In `crates/ironhermes-cli/src/tui_rata/app.rs::apply_slash_outcome` the `SkillsReload` arm pushes a System-role transcript message but does not flag the viewport for auto-scroll the way an agent-turn message does. (Same pattern probably exists for the classic-CLI side in `main.rs` `run_chat`.) The `SkillActivated` arm inherits the same gap.
- **Hypothesis to verify:** check `apply_slash_outcome` SkillsReload arm vs. how the existing `Output` / `McpReload` arms flag scroll. If they call a `scroll_to_bottom()` (or set a `dirty` flag for the next render tick) and SkillsReload skips that call, that's the fix.
- **Suggested fix scope:** 1-line wire-up in `apply_slash_outcome` SkillsReload + SkillActivated arms (rata TUI) and the parallel `match` arm in classic CLI `run_chat`. No new tests required beyond a manual recheck of test 4.
- **Routing:** Gap-closure plan candidate for `/gsd-plan-phase --gaps 21.8.2` after UAT completes.
