# Phase 21.8.2: skills hot reload command - Context

**Gathered:** 2026-05-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Add three connected capabilities to the IronHermes skills system:

1. **`/skills reload` hot-reload command** — Re-scan all skill search paths and atomically swap the `SkillRegistry` mid-session, so newly installed skills are available without restarting. This directly overrides Phase 21.8 D-23 ("next-session-only activation"), which explicitly deferred this capability to this phase.
2. **SKILL-13 dynamic skill activation** — After the CommandRouter's 3-stage resolution finds no match, fall back to `SkillRegistry` lookup. If a skill with that name exists, inject its SKILL.md body as system context for the next LLM call. Resolves the long-standing SKILL-13 backlog item.
3. **Title Case / spaces validation fix** — Auto-normalize skill names containing spaces or Title-Case characters at load time (in-memory only), so oh-my-claudecode skills (e.g., "Command Development", "Hook Development") load instead of being hard-rejected.

**In scope:**
- `/skills reload` slash command (all platforms: CLI, gateway/Telegram, TUI, web)
- `hermes skills reload` CLI subcommand
- Full rescan: builtin + `~/.ironhermes/skills/` + `optional-skills/` paths
- Immediate effect: system prompt + tool registry rebuild before next LLM call
- Diff output on reload (N added, N removed, N changed)
- SKILL-13: dynamic CommandRouter fallback → skill activation → system context injection
- Title Case / spaces normalization: `name.to_lowercase().replace(' ', "-")` in-memory

**Out of scope:**
- File-system watch / inotify / auto-reload on disk change (must be explicit command)
- Renaming skill directories on disk (normalization is in-memory only)
- Reloading MCP servers (that's `/reload-mcp`)
- Adding a new trust tier for hot-reloaded skills
- Telemetry for skill activations

</domain>

<decisions>
## Implementation Decisions

### Hot Reload Command Surface

- **D-01:** The slash command is `/skills reload` — a subcommand of the existing `/skills` dispatcher, not a new top-level command. Available across ALL delivery paths: CLI REPL, gateway (Telegram), TUI, web. The `hermes skills reload` CLI subcommand also exists for scripting use (reports diff and exits).
- **D-02:** Reload rescans ALL configured search paths — builtin skills dir + `~/.ironhermes/skills/` + `optional-skills/` — matching startup `load_with_paths` behavior exactly. No partial-path reload.
- **D-03:** Effect is **immediate**: the `Arc<SkillRegistry>` is atomically swapped before the command returns. The prompt builder reads from the registry at call time, so the next LLM call automatically uses the updated registry. No `/reset` required.

### Reload Output

- **D-04:** Output is diff format: names of added, removed, and changed skills, plus total count. Example: `Skills reloaded: 3 added (ascii-art, code-review, translator), 1 removed (old-tool), 0 changed. Total: 12 skills.`
- **D-05:** Invalid SKILL.md files encountered during reload → WARN-BUT-LOAD (consistent with Phase 19 D-15 posture for Trusted skills). Each warning reported in the reload output. Valid skills always load; one bad skill does not abort the reload.

### SKILL-13 Dynamic Activation

- **D-06:** Implementation is a **dynamic fallback in CommandRouter** — after the existing 3-stage resolution (exact match → alias match → prefix match) finds no registered command, check `SkillRegistry` for a skill whose name matches the command token. If found, activate. Unknown commands become skill lookups automatically with no new registered command needed.
- **D-07:** Activation = inject the skill's SKILL.md prompt body as **system context addition** for the next LLM call. Same semantics as hermes-agent's skill injection. The injected content prepends to the system prompt for that turn.
- **D-08:** **Registered commands always win.** The skill fallback fires only if all 3 resolution stages return nothing. A skill named `reset` or `help` cannot shadow a registered command.
- **D-09:** All platforms — CLI, gateway (Telegram), TUI, web. The CommandRouter is already platform-agnostic; the fallback adds no platform-specific code.

### Title Case / Spaces Validation Fix

- **D-10:** Normalize at load time: `name_candidate.to_lowercase().replace(' ', "-")`. Handles both cases the validator rejects: spaces ("Command Development" → "command-development") and Title-Case characters in otherwise-valid names ("Code-Review" → "code-review"). Applied to the name read from SKILL.md frontmatter **before** `validate_skill_name` runs.
- **D-11:** **In-memory only** — the directory on disk is never renamed or modified. Registry load is always read-only with respect to the filesystem. The normalized name is what appears in `/skills list` and what the SKILL-13 fallback resolves against.
- **D-12:** Emit `tracing::warn` for each normalized name so the operator knows normalization occurred: `SkillRegistry: normalized skill name "Command Development" → "command-development" at path <path>`.

### Claude's Discretion

- **Hot-swap mechanism:** Whether to use `Arc<RwLock<SkillRegistry>>`, `arc-swap`, or thread a freshly constructed `Arc<SkillRegistry>` through `CommandContext` for the atomic swap. The constraint is that `load_with_paths` is synchronous, so the reload can be done inline in the command handler without async complexity — planner picks the approach that requires the fewest call-site changes.
- **CommandResult variant for skills reload:** Whether the reload returns a new `CommandResult::SkillsReload(diff)` variant (parallel to `CommandResult::McpReload`) that the REPL loop processes, or whether the registry swap + diff computation is done synchronously inside the handler and returns `CommandResult::Output(diff_text)`. Either is fine; `McpReload` is the reference pattern.
- **SKILL-13 `CommandResult` variant:** Whether skill activation returns a new `CommandResult::SkillActivated { name, body }` that the REPL loop injects into the next system prompt, or whether the body is immediately folded into the `CommandContext` for the next turn. Planner picks the approach consistent with how `/personality` session overlays work.

### Folded Todos

- **SKILL-13 (Slash command integration):** Long-standing backlog item since v1.1. Resolved by D-06/D-07/D-08/D-09 — dynamic CommandRouter fallback with system context injection, all platforms, registered commands win.
- **SkillRegistry Title Case name validation rejects Title Case skills:** Live regression — observed in production session rejecting "Command Development", "Skill Development", "Plugin Settings", "Plugin Structure", "Writing Hookify Rules", "Hook Development", "MCP Integration", "Agent Development" (all oh-my-claudecode skills from `~/.agents/skills/`). Resolved by D-10/D-11/D-12.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `cmd_reload_mcp` (handlers.rs:1106) — exact reference pattern for a reload command: returns `CommandResult::McpReload` which the REPL loop handles asynchronously. The skills reload should follow this same pattern or return `CommandResult::Output(diff)` if the swap is synchronous.
- `SkillRegistry::load_with_paths(&[paths])` — synchronous; can be called inline in a command handler. No async needed.
- `validate_skill_name` (skills.rs) — the function to patch for D-10/D-11.

### Established Patterns
- `Arc<SkillRegistry>` is currently constructed once at startup and shared immutably. Hot-reload requires either `Arc<RwLock<SkillRegistry>>` wrapping or an `ArcSwap`-style atomic pointer swap. The existing `Arc<SkillRegistry>` in `CommandContext` is the target.
- CommandRouter 3-stage resolution is the correct extension point for SKILL-13 (D-06) — add a 4th fallback stage after stage 3 that checks `skill_registry`.
- WARN-BUT-LOAD for Trusted skills (Phase 19 D-15) is the existing scan posture — D-05 reload must not be stricter.

### Integration Points
- `/skills reload` handler wires into the existing `/skills` subcommand dispatch in `handlers.rs:cmd_skills`.
- SKILL-13 fallback wires into `CommandRegistry::dispatch` in `registry.rs` as a 4th resolution stage.
- Title Case fix is a single-function patch in `skills.rs:validate_skill_name` (or the name-extraction call site just before it).
- The `hermes skills reload` CLI subcommand adds a new arm in `skills_cmd.rs` parallel to `install`, `list`, `update`, `remove`.

</code_context>

## Specific Ideas

- **Live rejection log (2026-05-09 session):** The Title Case fix is urgent — the following skills are being hard-rejected in the user's live sessions from `~/.agents/skills/` (oh-my-claudecode): "Command Development", "Skill Development", "Plugin Settings", "Plugin Structure", "Writing Hookify Rules", "Hook Development", "MCP Integration", "Agent Development". Normalization → "command-development", "skill-development", "plugin-settings", "plugin-structure", "writing-hookify-rules", "hook-development", "mcp-integration", "agent-development".
- **Reload diff naming:** The diff output should name added/removed skills specifically, not just count them, since users want to confirm that the skill they just installed is in the "added" list.
- The `optional-skills/` directory at the project root is a valid third search path (gets `SkillSource::Official` trust per D-05 of skills.rs trust resolution). It must be included in reload rescans alongside builtin and installed paths.
