# Phase 21.8.2: skills-hot-reload-command — Research

**Researched:** 2026-05-09
**Domain:** Rust skills system — hot-reload, CommandRouter extension, name normalization
**Confidence:** HIGH (all findings verified by direct codebase inspection)

## Summary

Phase 21.8.2 adds three connected capabilities to the IronHermes skills system: a `/skills reload` hot-swap command, a SKILL-13 dynamic CommandRouter fallback for skill activation, and a Title Case / spaces normalization fix for oh-my-claudecode skill names. All three changes are greenfield additions to existing well-understood infrastructure.

The research confirms all key architectural questions from the brief. The hot-swap mechanism is straightforward: `SkillRegistry` is constructed once at startup as `Arc<SkillRegistry>` and cloned into several holders. Because `load_with_config` is synchronous, the reload can be done inline in the handler without async complexity. The simplest approach is a `CommandResult::SkillsReload(diff)` variant that the REPL loop handles, parallel to the existing `McpReload` pattern, but with the registry swap done via an `Arc::clone` replacement inside a `Mutex`-wrapped pointer — no `arc-swap` crate is available in the workspace.

The SKILL-13 fallback location is `CommandRouter::resolve` returning `ResolveResult::NotFound` — the dispatch call site in `main.rs` already matches on `CommandResult::PassThrough` for the normal "pass to agent" path. The skill fallback should be inserted at the dispatch call site (in `main.rs` REPL loop and gateway handler), not inside `CommandRouter::resolve` itself, because `resolve` has no access to `skill_registry`.

The Title Case fix is a one-line normalization inserted in `parse_skill_md` just before `validate_skill_name` is called, matching the D-10 decision exactly.

**Critical gap discovered:** `ctx.skill_registry` is `None` at all current dispatch sites. The `cmd_skills` handler (line 732) guards on it, but no `CommandContext` construction site calls `.with_skill_registry(...)`. This must be fixed in the same phase — the `/skills reload` result type needs to propagate the new `Arc<SkillRegistry>` back to `CommandContext` callers so the diff and subsequent `/skills` calls work correctly.

**Primary recommendation:** Use `CommandResult::SkillsReload(SkillsReloadDiff)` parallel to `McpReload`; store the reloadable registry pointer as `Arc<Mutex<Arc<SkillRegistry>>>` at startup; wire `with_skill_registry` at all four `CommandContext` construction sites.

## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** The slash command is `/skills reload` — a subcommand of the existing `/skills` dispatcher, not a new top-level command. Available across ALL delivery paths: CLI REPL, gateway (Telegram), TUI, web. The `hermes skills reload` CLI subcommand also exists for scripting use (reports diff and exits).
- **D-02:** Reload rescans ALL configured search paths — builtin skills dir + `~/.ironhermes/skills/` + `optional-skills/` — matching startup `load_with_config` behavior exactly. No partial-path reload.
- **D-03:** Effect is **immediate**: the `Arc<SkillRegistry>` is atomically swapped before the command returns. The prompt builder reads from the registry at call time, so the next LLM call automatically uses the updated registry. No `/reset` required.
- **D-04:** Output is diff format: names of added, removed, and changed skills, plus total count. Example: `Skills reloaded: 3 added (ascii-art, code-review, translator), 1 removed (old-tool), 0 changed. Total: 12 skills.`
- **D-05:** Invalid SKILL.md files encountered during reload → WARN-BUT-LOAD (consistent with Phase 19 D-15 posture for Trusted skills). Each warning reported in the reload output. Valid skills always load; one bad skill does not abort the reload.
- **D-06:** Implementation is a **dynamic fallback in CommandRouter** — after the existing 3-stage resolution (exact match → alias match → prefix match) finds no registered command, check `SkillRegistry` for a skill whose name matches the command token. If found, activate. Unknown commands become skill lookups automatically with no new registered command needed.
- **D-07:** Activation = inject the skill's SKILL.md prompt body as **system context addition** for the next LLM call. Same semantics as hermes-agent's skill injection. The injected content prepends to the system prompt for that turn.
- **D-08:** **Registered commands always win.** The skill fallback fires only if all 3 resolution stages return nothing. A skill named `reset` or `help` cannot shadow a registered command.
- **D-09:** All platforms — CLI, gateway (Telegram), TUI, web.
- **D-10:** Normalize at load time: `name_candidate.to_lowercase().replace(' ', "-")`. Applied to the name read from SKILL.md frontmatter **before** `validate_skill_name` runs.
- **D-11:** **In-memory only** — the directory on disk is never renamed or modified.
- **D-12:** Emit `tracing::warn` for each normalized name.

### Claude's Discretion

- **Hot-swap mechanism:** Whether to use `Arc<RwLock<SkillRegistry>>`, `arc-swap`, or thread a freshly constructed `Arc<SkillRegistry>` through `CommandContext` for the atomic swap. The constraint is that `load_with_paths` is synchronous, so the reload can be done inline in the command handler without async complexity — planner picks the approach that requires the fewest call-site changes.
- **CommandResult variant for skills reload:** Whether the reload returns a new `CommandResult::SkillsReload(diff)` variant (parallel to `CommandResult::McpReload`) that the REPL loop processes, or whether the registry swap + diff computation is done synchronously inside the handler and returns `CommandResult::Output(diff_text)`. Either is fine; `McpReload` is the reference pattern.
- **SKILL-13 `CommandResult` variant:** Whether skill activation returns a new `CommandResult::SkillActivated { name, body }` that the REPL loop injects into the next system prompt, or whether the body is immediately folded into the `CommandContext` for the next turn.

- File-system watch / inotify / auto-reload on disk change
- Disk-side rename of Title Case directories
- Telemetry for skill activation events

### Anti-Patterns to Avoid

- **Adding arc-swap as a new workspace dep:** Not available in the workspace; `Arc<Mutex<Arc<SkillRegistry>>>` achieves the same atomic swap with zero new dependencies.
- **Making `load_with_config` async:** It is synchronous by design. Do not wrap it in `tokio::spawn` or `block_in_place`; call it inline in the handler or REPL arm.
- **Inserting SKILL-13 fallback inside `CommandRouter::resolve`:** `resolve()` is a pure name-resolution function with no context access. The fallback must live at the call site.
- **Normalizing by mutating on-disk directories:** D-11 is in-memory only. Never call `fs::rename` from `parse_skill_md` or `try_register_skill_from_dir`.
- **Calling `with_skill_registry` before it's needed:** Wire it at the 4 `CommandContext` construction sites but do not add it to the `CommandDef` registry (skills are not commands).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Registry diff computation | Custom hash map diffing | `HashSet::difference` on `registry.list()` name sets | Two `collect::<HashSet>()` calls + difference is idiomatic Rust |
| Name normalization regex | New regex for spaces/case | `str::to_lowercase()` + `str::replace(' ', "-")` | The existing `SKILL_NAME_RE` only needs to validate after normalize; no new regex needed |
| Skill body injection | New prompt assembly infrastructure | `SkillRegistry::read_content(name)` + prepend to system slot | `read_content` already reads the SKILL.md body from disk |
| Hot-swap pointer | `arc-swap` crate | `Arc<Mutex<Arc<SkillRegistry>>>` | No new dep; std Mutex is sufficient since load is sync and rare |

## Common Pitfalls

### Pitfall 1: SkillsReload variant — REPL loop needs `skill_registry` as mutable local

**What goes wrong:** `skill_registry` is cloned from `runtime_bundle.skill_registry` as an immutable `Arc<SkillRegistry>` before the REPL loop. Replacing it mid-loop requires it to be a `let mut skill_registry`.

**Why it happens:** The `McpReload` arm does not replace a local — it calls async reload on `mgr`. The skills reload must replace the local `Arc` pointer itself.

**How to avoid:** Declare `let mut skill_registry = runtime_bundle.skill_registry.clone();` so the reload arm can reassign. Also call `prompt_builder.set_skill_registry(new_arc.clone())` to update the prompt builder for the next LLM call (D-03).

**Warning signs:** `ctx.skill_registry` in the rebuilt CommandContext still points to the old registry after reload.

### Pitfall 2: parse_skill_md path not available for D-12 warn

**What goes wrong:** `parse_skill_md` receives only `content: &str`, not the file path. D-12 requires logging the path in the normalization warning.

**Why it happens:** The function signature at `skills.rs:192` predates the need for path logging.

**How to avoid:** Two options:
- Option A: Add `path: Option<&Path>` parameter to `parse_skill_md` (changes test call sites).
- Option B: Detect normalization after `parse_skill_md` returns, inside `try_register_skill_from_dir` where `skill_md_path` is in scope. Compare `frontmatter.name` before vs after normalization at the call site.

Option B avoids the signature change and is consistent with the existing pattern (the dir-name-mismatch warn at `skills.rs:549` is also in `try_register_skill_from_dir`). [VERIFIED: skills.rs:484-597]

### Pitfall 3: SKILL-13 fallback triggers on non-slash input

**What goes wrong:** The REPL loop input path passes ALL non-slash-prefixed text to the agent directly. The SKILL-13 fallback must only fire for slash-prefixed input that resolved to `NotFound`.

**Why it happens:** The dispatch path only runs when input starts with `/`. The `NotFound` branch inside `router.resolve` is reached only for unrecognized `/command` tokens.

**How to avoid:** Insert the skill-fallback check only in the `ResolveResult::NotFound` branch, not before or after `router.resolve`. Verify the cmd_token extraction strips the leading `/` before calling `registry.find(cmd_token)`.

### Pitfall 4: Normalization creates consecutive-hyphen names

**What goes wrong:** A skill name like "Code--Review" would normalize to "code--review", which `validate_skill_name` rejects due to the `contains("--")` check.

**Why it happens:** Multiple consecutive spaces in a skill name ("Code  Review") become "code--review" after `replace(' ', "-")`.

**How to avoid:** After `to_lowercase().replace(' ', "-")`, additionally collapse consecutive hyphens: `.split('-').filter(|s| !s.is_empty()).collect::<Vec<_>>().join("-")`. This ensures "Code  Review" → "code-review" rather than "code--review".

### Pitfall 5: build_command_context in tui_rata has no skill_registry

**What goes wrong:** The ratatui TUI's `App` struct does not have a `skill_registry` field, so `build_command_context` at `tui_rata/commands.rs:514` cannot wire it.

**Why it happens:** `App` was built incrementally; skill_registry was never added to it.

**How to avoid:** Add `pub skill_registry: Option<Arc<SkillRegistry>>` to `App` in `tui_rata/app.rs` alongside existing registry fields. Populate it at `App::new` or equivalent from `runtime_bundle.skill_registry`. Then add `ctx = ctx.with_skill_registry(skill_registry.clone())` in `build_command_context`.
