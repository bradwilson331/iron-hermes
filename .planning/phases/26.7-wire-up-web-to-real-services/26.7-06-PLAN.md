<failure_mode_recap>
<!-- One-paragraph distillation of the diagnosis. Executor should not need to re-read the debug session. -->

Dioxus fullstack dispatches each `#[post]` / `#[get]` server fn invocation through an
internal per-connection `tokio::task::LocalSet` so server fns can hold `!Send` futures
(WASM compat). A `LocalSet` is a single-thread executor context. `tokio::task::block_in_place`
requires the multi-thread runtime to have a SPARE worker thread to park onto — inside a
LocalSet there is no spare thread, so the call panics with the exact message observed:
`"can call blocking only when running on the multi-threaded runtime"`. `tokio::task::spawn_blocking`
is the documented escape: it moves the closure to the dedicated blocking thread pool
(which lives on the global multi-thread runtime, NOT inside the LocalSet), and from
inside a blocking-pool thread `block_in_place` is legal.

Why the read path works: `api_agents_list` (api.rs:355-373) calls
`state.subagent_registry.read().await.list()` — natively async, never enters
block_in_place. No bridge needed.
</failure_mode_recap>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser/WASM → server fn (write) | `POST /api/agents/{kill,interrupt,prune}` same-origin |
| Server fn async context → blocking thread pool | spawn_blocking closure bridges Dioxus per-connection LocalSet to the global multi-thread runtime's blocking pool |
| Blocking thread → ShrikeService → SubagentRegistry | tokio RwLock writes via block_in_place + block_on (legal once in a blocking thread) |
