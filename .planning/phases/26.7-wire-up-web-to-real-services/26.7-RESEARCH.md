**Researched:** 2026-05-17
**Domain:** Dioxus 0.7 fullstack — server functions, AppState extension, service registry enumeration
**Confidence:** HIGH

## Summary

Phase 26.7 wires five stubbed screens (Tools, Agents, Memory, Skills, Models) to live server data. The foundational pattern is already proven: `sessions.rs` demonstrates the complete `use_server_future(api::fn)?` + `match resource() { Some(Ok(v)) => v, _ => Vec::new() }` idiom. Three of the five screens use server functions that already exist in `api.rs`; two (Memory and Skills) need new functions, and one (Models) needs a new function plus a new accessor on `resolver`.

**Primary recommendation:** Wire Tools first (trivial — one call swap), then Agents (write ops add local state complexity), then Skills and Models together (both need new server fns but are structurally similar), then Memory last (requires understanding `to_memory_entries()` async API).

**Critical findings:**

1. `SkillRegistry` is already on `AppState` — accessible via `state.runtime_bundle.skill_registry` (no `AppState` extension needed for skills).
2. `ModelRegistry` is NOT directly on `AppState`, but `ProviderResolver` (which IS on `AppState` as `state.resolver`) exposes `resolver.model_registry()` returning `&ModelRegistry`. `ModelRegistry::all_models()` enumerates all static + cache entries — no `AppState` extension needed.
3. `MemoryManager` is already on `AppState` as `state.memory_manager: Option<Arc<tokio::sync::Mutex<MemoryManager>>>`. The method `to_memory_entries()` is `async` and acquires the inner tokio Mutex.
4. The four agent server functions already exist in `api.rs` as `api_agents_kill`, `api_agents_interrupt`, `api_agents_prune`, `api_agents_status` — but their naming differs from what the CONTEXT.md originally described (`get_agents_status` / `kill_agent` / etc.). The component must call the actual function names.
5. `MemoryEntries` is `HashMap<MemoryTarget, Vec<String>>` — each entry in the Vec is a raw text block, not a structured key/value pair. The Memory screen must parse these or display them as opaque text blobs.

### Kill button: inline confirm pattern
Requires a local `use_signal(bool)` per card — cannot be a global signal. Pattern:
```rust
#[component]
fn AgentCard(agent: AgentInfo, resource: ...) -> Element {
    let mut confirm_kill = use_signal(|| false);
    // First click: set confirm_kill to true (shows "KILL?")
    // Second click within 3s: call api_agents_kill
    // Timeout: reset confirm_kill to false
    // ...
}
```

The 3-second timeout requires `use_effect` + `tokio::time::sleep` spawned in WASM. In Dioxus 0.7 WASM, use `gloo_timers::future::TimeoutFuture` or `wasm_bindgen_futures::spawn_local` + `wasm_bindgen::JsValue` — OR use `spawn` from dioxus which calls `spawn_local` internally. Standard pattern: `spawn(async move { /* timeout */ })`.

### Interrupt button: "..." feedback
```rust
let mut interrupting = use_signal(|| false);
// onclick: set interrupting=true, spawn async call, reset to false after 500ms
```

### Prune button
Screen-level action, no per-card state. Call `api_agents_prune(None)`, then restart the list resource.

### resource.restart() pattern
Confirmed from `sessions.rs` — resource is `Resource<T>` from `use_server_future`. Call `resource.restart()` to trigger re-fetch. The resource must be stored in a variable before the match guard so it can be passed into event handlers.

## Screen 3: Memory

### Current state
`memory.rs` calls `stub_data::memory_entries()` returning `Vec<MemoryEntryStub { scope, key, value_preview, updated }`.

### MemoryManager API surface (VERIFIED from source)
`MemoryManager::to_memory_entries()` is `async` — it acquires `self.primary.lock().await` and calls `p.to_memory_entries()`.

Returns `MemoryEntries { entries: HashMap<MemoryTarget, Vec<String>> }`.

`MemoryTarget` is an enum with two variants:
- `MemoryTarget::Memory` — corresponds to MEMORY.md (agent facts)
- `MemoryTarget::User` — corresponds to USER.md (user profile)

Each `Vec<String>` entry is a **raw text block string**, not a structured key/value pair. The strings are raw markdown bullet lines like `"Operator prefers monospace output for code blocks"`. There is no `key`, `updated_at`, or `scope` field in the underlying storage.

### MemoryInfo type (new, defined in api.rs)
```rust
#[derive(Serialize, Deserialize, Clone, Debug, PartialEq)]
pub struct MemoryEntry {
    pub store: String,   // "memory" | "user"
    pub content: String, // raw text block from Vec<String>
    pub index: usize,    // position within the store's Vec — used as key in RSX
}

#[derive(Serialize, Deserialize, Clone, Debug, PartialEq)]
pub struct MemoryInfo {
    pub entries: Vec<MemoryEntry>,
}
```

### Server function (new)
```rust
#[get("/api/memory")]
pub async fn get_memory() -> Result<MemoryInfo> {
    let state = crate::server::state::global_app_state();
    let Some(ref mgr) = state.memory_manager else {
        return Ok(MemoryInfo { entries: Vec::new() });
    };
    let entries = mgr.lock().await.to_memory_entries().await;
    let mut out = Vec::new();
    for (target, items) in &entries.entries {
        let store = match target {
            ironhermes_core::memory_store::MemoryTarget::Memory => "memory",
            ironhermes_core::memory_store::MemoryTarget::User => "user",
        };
        for (idx, content) in items.iter().enumerate() {
            out.push(MemoryEntry {
                store: store.to_string(),
                content: content.clone(),
                index: idx,
            });
        }
    }
    Ok(MemoryInfo { entries: out })
}
```

**Async note:** `to_memory_entries()` on `MemoryManager` requires `async` — it locks the inner `tokio::sync::Mutex`. Signal borrow rules: do NOT hold any signal read/write lock across this `.await`. Since this is a server fn (not component code), the clippy borrow rule does not apply, but standard async hygiene applies.

### Component changes
- The existing two-panel layout stays. Left panel: `entries` where `store == "memory"`. Right column top panel: `entries` where `store == "user"` (replaces static `dl` block). Right column bottom panel (Provider): remains static stub.
- `MemoryEntryRow` takes `MemoryEntry` instead of `MemoryEntryStub`. Display `entry.content` as the body, `entry.store.to_uppercase()` as the tag. No `key`, no `updated_at` available from the real API.
- Client-side filter: filter on `entry.content` prefix/substring match, case-insensitive (as per UI-SPEC).

### Field gap: no `updated_at` in real API
The stub has `updated` timestamps. `MemoryStore` does not store per-entry timestamps in the `Vec<String>` structure. The `mem-ts` div can either be omitted or show a static placeholder like `"—"`. Planner must decide; recommended: omit `mem-ts` div for entries that have no timestamp.

### Server function (new) — SYNC read path
`SkillRegistry::list()` is sync. `active_skills` requires a std::sync::Mutex lock (not tokio). No `.await` needed.

```rust
#[get("/api/skills")]
pub async fn list_skills() -> Result<Vec<SkillInfo>> {
    let state = crate::server::state::global_app_state();
    let registry = &state.runtime_bundle.skill_registry;
    let active = state.runtime_bundle.active_skills.lock()
        .map_err(|e| ServerFnError::new(format!("active_skills lock poisoned: {e}")))?;
    let active_names: std::collections::HashSet<&str> =
        active.iter().map(|r| r.name.as_str()).collect();

    let out = registry.list().iter().map(|r| {
        let category = match r.source {
            ironhermes_core::skills::SkillSource::Builtin => "bundled",
            ironhermes_core::skills::SkillSource::Official => "official",
            ironhermes_core::skills::SkillSource::Trusted => "trusted",
            ironhermes_core::skills::SkillSource::Community => "installed",
            ironhermes_core::skills::SkillSource::SelfCreated => "self-created",
        };
        SkillInfo {
            name: r.name.clone(),
            description: r.description.clone(),
            category: category.to_string(),
            enabled: active_names.contains(r.name.as_str()),
        }
    }).collect();
    Ok(out)
}
```

## Reference Patterns

### Pattern 1: use_server_future with suspension
Source: `crates/iron_hermes_ui/src/components/hermes_app/screens/sessions.rs` lines 33–51

```rust
// Line 33: The `?` suspends the component until first resolution.
let sessions_resource = use_server_future(crate::server::api::list_sessions)?;

// Lines 48–51: Loading guard — None = loading, Some(Err) = error, Some(Ok) = data.
let sessions: Vec<crate::server::api::SessionInfo> = match sessions_resource() {
    Some(Ok(v)) => v,
    _ => Vec::new(),
};
```

### Pattern 2: resource.restart() after write op
The `sessions_resource` variable is a `Resource<T>` handle. Call `sessions_resource.restart()` to trigger a new server fn call and re-render. Must hold this handle in a variable before the match guard because the match guard borrows the value.

```rust
let resource = use_server_future(crate::server::api::list_sessions)?;
// ... later in an event handler:
resource.restart();
```

### Pattern 3: Error state rendering (D-13)
No custom error component. Inline text inside the existing container:

```rust
// Inside the grid/row-list container:
if let Some(Err(_)) = resource() {
    div {
```

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---|---|---|---|
| Server-client data bridge | Custom REST endpoints | Dioxus `#[get]`/`#[post]` server fns | Handles serialization, CSRF, WASM/server split automatically |
| Async state in components | Manual futures | `use_server_future` | Handles suspension, caching, and restart |
| Signal borrow across await | Cloning before await | Hold signal value in local variable, drop lock at `;` | clippy.toml enforces this; violation = runtime panic |
| tokio Mutex in WASM | Any lock in component code | `use_server_future` runs on server; only client signal state in WASM | MemoryManager's tokio Mutex is fine in server fn, never in component |

## Common Pitfalls

### Pitfall 1: Wrong agent function names
**What goes wrong:** CONTEXT.md used names like `get_agents_status()`, `kill_agent()` that do not exist. Actual names are `api_agents_status`, `api_agents_kill`, `api_agents_interrupt`, `api_agents_prune`.
**Why it happens:** CONTEXT.md was written before Phase 32.3 implemented these functions with a different naming convention.
**How to avoid:** Always import from `crate::server::api::api_agents_*` not `get_agents_status`.
**Warning signs:** `error[E0425]: cannot find function` at compile time.

### Pitfall 2: No list-all agents endpoint
**What goes wrong:** Component needs `Vec<AgentInfo>` but only `api_agents_status(id)` exists — it queries a SINGLE agent.
**Why it happens:** Phase 32.3 built query-by-id; the list view was a stub with no server fn.
**How to avoid:** Add `api_agents_list()` server fn as part of this phase.
**Warning signs:** `api_agents_status` requires an `id` parameter — there is no overload without it.

### Pitfall 3: MemoryEntries has no timestamps or structured keys
**What goes wrong:** Planner assumes `MemoryEntry` has `key`, `updated_at`, `scope` fields matching the stub. They do not exist.
**Why it happens:** The stub was designed around the UI prototype, not the real `MemoryStore` API which stores raw text blocks.
**How to avoid:** `MemoryInfo.entries[].content` is a raw string. No `key` or `updated_at`. The `mem-ts` div must be omitted or show `"—"`.
**Warning signs:** Rust compile error trying to access `.key` or `.updated_at` on `MemoryEntry`.

### Pitfall 4: SkillRecord has no `version` field
**What goes wrong:** Planner includes `version: String` in `SkillInfo` and tries to access `r.version` on `SkillRecord`.
**Why it happens:** `version` is in `SkillFrontmatter` but is NOT mirrored onto `SkillRecord` during load.
**How to avoid:** Omit `version` from `SkillInfo`, or grep skills.rs to confirm: `pub struct SkillRecord` has no `version` field.
**Warning signs:** `error[E0609]: no field 'version' on type 'SkillRecord'`.

### Pitfall 5: ModelStub uses `&'static str` for `family`; ModelInfo uses `String`
**What goes wrong:** The `families` dedup loop in `models.rs` collects `&'static str` references from `ModelStub`. When switching to `ModelInfo` with `family: String`, the loop must use `String` and `contains` must compare `&String`.
**Why it happens:** Static lifetime on stub fields doesn't carry over to owned types.
**How to avoid:** Change `Vec<&'static str>` to `Vec<String>` in the families loop. Use `families.contains(&m.family)` (compares `&String` to `&String`).

### Pitfall 6: Signal borrow held across async spawn in Agents screen
**What goes wrong:** Reading a signal inside an async closure for the 3-second kill timeout causes clippy to reject the code (GenerationalRef held across await).
**Why it happens:** `use_signal` returns a handle; reading it borrows the arena; spawning async captures the borrow.
**How to avoid:** Clone the needed values OUT of the signal before the spawn. Use `let id = agent.id.clone(); spawn(async move { /* id is owned */ })`.

### Pitfall 7: `MemoryManager::to_memory_entries()` requires two `.await`s
**What goes wrong:** Code calls `mgr.lock().await` but forgets the second `.await` on `to_memory_entries()`.
**Why it happens:** The method is `pub async fn to_memory_entries()` on `MemoryManager` — it internally locks `self.primary`.
**How to avoid:** `mgr.lock().await.to_memory_entries().await`.
