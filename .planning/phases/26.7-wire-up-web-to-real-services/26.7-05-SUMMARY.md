# Phase 26.7 Plan 05: Agents Screen Wire-Up Summary

Live Agents screen wired to SubagentRegistry via new flat-list `api_agents_list` server fn, with KILL? two-click inline confirm (3-second gloo_timers timeout per-card), INTERRUPT 500ms feedback, and PRUNE ENDED screen-level action.

## What Was Built

### Task 1 (prior commit 27363b09)

`AgentInfo` struct added to `api.rs`:
- Fields: `id`, `task_summary`, `uptime_secs`, `status`, `parent_id`
- Derives: `Serialize, Deserialize, Clone, Debug, PartialEq`

`api_agents_list()` server fn at `GET /api/agents/list`:
- Acquires tokio RwLock READ guard in a scoped block, calls `SubagentRegistry::list()`, drops guard
- Maps `SubagentInfo` -> `AgentInfo` (computes `uptime_secs = started_at.elapsed().as_secs()`)
- Returns flat `Vec<AgentInfo>` per user choice (not tree)

### Tasks 2+3 (commit e179b414)

`ScreenAgents` component:
- Fetches via `use_server_future(crate::server::api::api_agents_list)?`
- `agents_list` and `load_error` computed before `rsx!` block (clippy.toml borrow discipline)
- Error row: `"Could not load agents — check server connection."` in danger color inside `.grid.wide`
- PRUNE ENDED button: `spawn(async move { api_agents_prune(None).await; agents_resource.restart(); })`
- `+ NEW AGENT` button retained as static visual affordance

`AgentCard` component:
- Props: `agent: AgentInfo`, `agents_resource: Resource<Result<Vec<AgentInfo>, CapturedError>>`
- Per-card signals: `armed: Signal<bool>`, `killing: Signal<bool>`, `interrupting: Signal<bool>`
- Avatar letter derived from first char of `agent.id`
- Status pill: `.pill.green` with `agent.status.to_uppercase()`
- Card meta: `"{agent.status} · {agent.uptime_secs}s"`

KILL button (inline two-click confirm):
- First click: `armed.set(true)` + spawns `TimeoutFuture::new(3_000)` to reset
- Second click within 3s: `armed.set(false); killing.set(true); spawn(kill + restart)`
- `disabled: killing()` prevents double-fire during in-flight POST

INTERRUPT button:
- `interrupting.set(true)` → POST → `TimeoutFuture::new(500)` → `interrupting.set(false)`
- Label shows `"..."` while in-flight, reverts to `"INTERRUPT"` after 500ms

CHAT button: static with no onclick (deferred per UI-SPEC and out-of-scope).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] CapturedError vs ServerFnError prop type**
- **Found during:** Task 2 first cargo check
- **Issue:** Plan specified `Resource<Result<Vec<AgentInfo>, ServerFnError>>` as the prop type for `AgentCard.agents_resource`, but Dioxus 0.7's `use_server_future` actually returns `Resource<Result<T, CapturedError>>` — different error type, compile error E0308.
- **Fix:** Added `use dioxus::CapturedError;` import and changed prop type to `Resource<Result<Vec<crate::server::api::AgentInfo>, CapturedError>>`.
- **Files modified:** `crates/iron_hermes_ui/src/components/hermes_app/screens/agents.rs`
- **Commit:** e179b414

**2. Tasks 2+3 combined into single commit**
- **Reason:** Both tasks modify only `agents.rs`. Splitting would mean Task 2 renders `AgentCard` with placeholder buttons (KILL/INTERRUPT with no handlers) but also passes `agents_resource` as a prop — which is only needed for the write ops in Task 3. Implementing them atomically is cleaner and matches the note in `<parallel_execution>`: "if a task adds endpoints and UI handlers in one logical step, batch them in a single commit."
