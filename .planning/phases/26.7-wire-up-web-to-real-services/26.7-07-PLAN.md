```rust
impl Drop for RegistrationGuard {
    fn drop(&mut self) {
        // Phase 26.7-07: dispatch unregister_internal via tokio::spawn instead of
        // block_in_place + block_on. block_in_place panics inside a single-thread
        // executor (e.g. Dioxus fullstack per-connection LocalSet), which is exactly
        // where this Drop can run when an awaiting ws connection task polls the
        // subagent future's terminal Poll::Ready (see 26.7-06 SUMMARY for the
        // independent-panic evidence at 2026-05-18 00:10:52).
        //
        // Handle::current() returns the underlying multi-thread runtime even inside
        // a LocalSet, so handle.spawn(async ...) dispatches the cleanup as a fresh
        // task on a worker thread where arc.write().await is legal. The JoinHandle
        // is dropped (fire-and-forget) since Drop is sync and cannot await.
        if let Some(arc) = self.registry.upgrade() {
            if let Ok(handle) = tokio::runtime::Handle::try_current() {
                let id = self.id.clone();
                handle.spawn(async move {
                    arc.write().await.unregister_internal(&id);
                });
            }
            // If no tokio runtime is current (e.g. drop during global teardown
            // outside a runtime context), silent no-op. This is rarer than the
            // upgrade()-None case below but possible during shutdown.
        }
        // If upgrade() is None the registry's Arc has already been dropped
        // (session shutdown). Silent no-op — there is nothing to unregister.
    }
}
```

WHY this is correct (executor MUST understand before editing — these are correctness
constraints, not commentary):

  - `Handle::try_current()` returns `Result<Handle, TryCurrentError>`. Inside a tokio
    task on any flavor, it returns Ok(Handle). Outside a runtime entirely (e.g. during
    process global-dtor teardown after the runtime has been shut down), it returns Err
    and we silent-no-op.
  - `handle.spawn(future)` (https://docs.rs/tokio/latest/tokio/runtime/struct.Handle.html#method.spawn)
    schedules the future on the runtime the handle belongs to. Inside a Dioxus
    fullstack per-connection LocalSet, `Handle::current()` returns the underlying
    multi-thread runtime that hosts the LocalSet — NOT the LocalSet itself — so
    the spawned task runs on a worker thread where `arc.write().await` is legal.
    See tokio's runtime-handle model docs.
  - The future captures `arc: Arc<RwLock<...>>` and `id: SubagentId`, both
    `Send + 'static`. No borrowed data crosses the spawn boundary.
  - The returned `JoinHandle` is dropped immediately. Tokio JoinHandle's Drop detaches
    the task — it does NOT abort it. So the cleanup runs to completion.
  - Do NOT use `tokio::task::spawn_blocking`. That would still require an inner
    `block_on` to drive `arc.write().await` (since spawn_blocking takes a sync
    closure), reintroducing the dual-bridge complexity. `handle.spawn(async)` is
    strictly simpler.
  - Do NOT use `tokio::spawn(future)` directly. `tokio::spawn` inside a LocalSet
    dispatches to whichever runtime hosts the current task; `Handle::current().spawn`
    is the documented explicit form, and `Handle::try_current` adds the no-runtime
    guard that the previous code lacked.
  - Do NOT widen the eventual-consistency change beyond what's necessary. Any test
    that currently relies on `drop(guard); assert!(registry empty)` will need a
    yield, but that is a test-level fix, not a code change here. The plan does not
    modify any tests; if a test breaks under cargo test, FAIL the build and ask the
    orchestrator — do NOT silently add `yield_now` calls into tests.
  - Do NOT remove or modify `SubagentRegistry::unregister_internal` — its signature
    and visibility are unchanged. Only the call site moves from sync-bridge to
    async-spawn.
  - PRESERVE the existing comment about "If upgrade() is None the registry's Arc has
    already been dropped (session shutdown). Silent no-op — there is nothing to
    unregister." It still applies and remains accurate.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Subagent future (any thread) → RegistrationGuard::drop | Drop runs synchronously on whichever thread polls the future's terminal `Poll::Ready` |
| Drop → multi-thread runtime via Handle::try_current().spawn | Spawned async task lifts the cleanup off the dropping thread |
| Spawned task → SubagentRegistry::unregister_internal | Async write lock acquisition (`arc.write().await`) on the multi-thread runtime |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-26.7-07-01 | Denial of Service | spawned cleanup tasks pile up if many RegistrationGuards drop in a burst | accept | Each task does one `arc.write().await` + one HashMap remove. Even 10k concurrent drops would complete in milliseconds on the runtime. The multi-thread runtime's task queue is bounded by available memory; no realistic DoS path. |
| T-26.7-07-02 | Tampering | Eventual-consistency window: external observer sees subagent in registry briefly after Drop | accept | Window is one runtime poll cycle (~microseconds). The Agents screen `agents_resource.restart()` on user action is the only observer; the spawned task completes before the next list query. No race-significant readers. |
| T-26.7-07-03 | Information Disclosure | Spawned task panic message in logs | accept | `unregister_internal` is infallible (HashMap remove). The only panic source removed is the `block_in_place` itself, which is now gone. No new panic surface. |
| T-26.7-07-04 | Tampering | JoinHandle dropped (detached) — cleanup not awaited | accept | tokio JoinHandle::drop only detaches, never aborts. Cleanup runs to completion on the multi-thread runtime. Documented in the Drop comment. |
| T-26.7-07-SC | Tampering | npm/pip/cargo installs | accept | NO new package installs. Zero Cargo.toml changes. `tokio::runtime::Handle::try_current` and `Handle::spawn` are stable since tokio 1.0. |
| T-26.7-07-05 | Elevation of Privilege | Spawned task has same auth context as caller | accept | Single-tenant local app. Async task inherits process credentials. No privilege change. |
