# Phase 26.7 — Security

> Per-phase security contract: threat register, accepted risks, and audit trail for the
> 5-screen wire-up from `stub_data::*` to live server functions on the IronHermes web UI.

---

## Scope

Five plans wiring the IronHermes web UI screens (Tools, Memory, Skills, Models, Agents)
from stub data to live server functions backed by real runtime services. Primary files
audited:

- `crates/iron_hermes_ui/src/server/api.rs`
- `crates/iron_hermes_ui/src/components/hermes_app/screens/tools.rs`
- `crates/iron_hermes_ui/src/components/hermes_app/screens/memory.rs`
- `crates/iron_hermes_ui/src/components/hermes_app/screens/skills.rs`
- `crates/iron_hermes_ui/src/components/hermes_app/screens/models.rs`
- `crates/iron_hermes_ui/src/components/hermes_app/screens/agents.rs`

---

## Trust Boundaries

| Boundary | Description | Data Crossing |
|----------|-------------|---------------|
| Browser/WASM → server fn (read) | Same-origin `GET /api/{tools,memory,skills,models,agents/list}` via Dioxus fullstack mount | Read-only metadata + memory text blocks |
| Browser/WASM → server fn (write) | Same-origin `POST /api/agents/{kill,interrupt,prune}` | Agent ID string |
| Server fn → AppState services | In-process tokio Mutex/RwLock + sync Arc access to `MemoryManager`, `SkillRegistry`, `ModelRegistry`, `SubagentRegistry`, `ToolRegistry` | In-memory state snapshots |

## Audit Notes

**T-26.7-05-02 (partial transfer to upstream):** The declared mitigation relies on
`ShrikeService::kill` returning `killed:false` for unknown IDs. This upstream Phase 32.3
behavior is not directly verifiable in this phase's code. What IS verified: `agents.rs:190`
uses `let _ = ...await` (result discarded, no panic path). The no-panic guarantee is
satisfied in code under audit; upstream idempotency is a transfer to Phase 32.3.

**T-26.7-02-03 mutex guard drop:** The chain
`mgr.lock().await.to_memory_entries().await` at `api.rs:160` is a single expression
ending at `;`. The `MutexGuard` is consumed by `.to_memory_entries()` on the same
expression and is not stored across a second `.await`. Scoping correct per plan.

**Dioxus 0.7 type deviation (informational, no security impact):** Plan 05 specified
`Resource<Result<Vec<AgentInfo>, ServerFnError>>` as the `AgentCard` prop type, but the
implementation uses `Resource<Result<Vec<AgentInfo>, CapturedError>>` (`agents.rs:112`).
This was an auto-corrected compile error documented in `26.7-05-SUMMARY.md`. The error
type change does not affect any threat mitigation.
