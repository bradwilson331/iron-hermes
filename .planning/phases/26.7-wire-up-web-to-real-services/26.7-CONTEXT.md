**Gathered:** 2026-05-17

<domain>
## Phase Boundary

Phase 26.7 wires 5 visually-complete-but-stubbed web UI screens to real server data, replacing `stub_data::*` calls with `use_server_future` + `#[get]` server functions backed by real runtime services.

**In scope ŌĆö 5 screens:**
1. **Tools** ŌĆö server fn `list_tools()` already exists in `api.rs`; just wire to the screen
2. **Agents** ŌĆö server fns `get_agents_status()`, `kill_agent()`, `interrupt_agent()`, `prune_agents()` already exist; wire all four (read + write ops)
3. **Memory** ŌĆö new server fn needed; read-only display of MEMORY.md + USER.md contents from `MemoryManager`
4. **Skills** ŌĆö new server fn needed; read list from `SkillRegistry` (add to `AppState` if not present)
5. **Models** ŌĆö new server fn needed; read list from `ModelRegistry` (add to `AppState` if not present)

**Out of scope (deferred):**
- Office, Soul, Schedules, Providers screens ŌĆö remain on stub_data
- Gateway screen ŌĆö explicitly deferred to the Discord/Slack phase (Phase 34), when real adapter status will exist
- Memory write ops (add/delete entries) ŌĆö read-only in this phase
- Mock layer cleanup (`mocks/agent_steps.rs`, `mocks/run_shell`, `mocks/personalities`) ŌĆö already dead code; leave for a cleanup phase

</domain>

<decisions>
## Implementation Decisions

### Scope
- **D-01:** Wire exactly 5 screens: Tools, Agents, Memory, Skills, Models. The other 5 stubbed screens (Office, Gateway, Soul, Schedules, Providers) remain on stub_data.
- **D-02:** Gateway screen is explicitly deferred to the Discord/Slack phase ŌĆö do not add a gateway status endpoint in this phase. When Discord/Slack adapters land, the gateway screen gets wired then.

### Server function strategy
- **D-03:** Follow the established Sessions screen pattern exactly: `#[get("/api/...")]` attribute + `use_server_future(api::fn_name)?` in the component with `?` for suspension. No deviation from this pattern.
- **D-04:** New server fns for Memory, Skills, and Models belong in `crates/iron_hermes_ui/src/server/api.rs` alongside the existing fns. Do NOT create new files for these.
- **D-05:** Type shapes for new server fns: owned-`String` types defined in `api.rs` (same pattern as `SessionInfo`, `ConfigSummary`, `ToolInfo`). Do NOT modify the `&'static str` stub types in `stub_data.rs` ŌĆö those remain as-is for the deferred screens. New types: `MemoryInfo`, `SkillInfo`, `ModelInfo` (names subject to planner refinement, but owned-String fields are locked).

### Agents screen
- **D-06:** Wire all four existing agent server fns: `get_agents_status()` (read), plus `kill_agent()`, `interrupt_agent()`, `prune_agents()` (writes). These already exist in `api.rs` ŌĆö no new endpoints needed, just component wiring.

### Memory screen
- **D-07:** Memory screen is read-only in this phase. Display the current contents of MEMORY.md and USER.md (or equivalent MemoryProvider output) via a new `GET /api/memory` server fn. No add/delete form UI.
- **D-08:** Data source is `MemoryManager` on `AppState`. If the read path requires awaiting the async `Mutex`, handle it inline in the server fn (same pattern as `list_tools()` which does `registry.read().await`).

### Skills screen
- **D-09:** New `GET /api/skills` server fn reading from `SkillRegistry`. If `SkillRegistry` is not currently on `AppState`, add it as a field during this phase (same pattern as `runtime_bundle`, `memory_manager`, `subagent_registry` ŌĆö constructed in `AppState::init()`, stored as `Arc`).

### Models screen
- **D-10:** New `GET /api/models` server fn reading from `ModelRegistry`. If `ModelRegistry` is not on `AppState`, add it as a field during this phase (same pattern as D-09). Researcher should check `Phase 21.3` (ModelRegistry, Phase 21.3 plans) for how ModelRegistry is currently accessed.

### AppState extension
- **D-11:** If SkillRegistry or ModelRegistry need to be added to `AppState`, follow the existing field pattern: `Arc<T>` field, constructed in `AppState::init()`, no `OnceLock` wrapping needed (global state already uses `OnceLock<AppState>`).

### Tools screen
- **D-12:** Tools screen wiring is the simplest ŌĆö `list_tools()` already exists. The screen currently uses `stub_data::tools()`. Replace with `use_server_future(api::list_tools)?` and map `Vec<ToolInfo>` to the existing RSX layout. `ToolInfo` already has `name` + `description` fields that match what the stub provides.

### Loading and error states
- **D-13:** Loading state: when `use_server_future` returns `None`, render the same layout with an empty list (no special skeleton needed). Error state: when it returns `Some(Err(...))`, render a single error message row inside the screen's existing container. Do NOT add custom error UI components ŌĆö inline text is sufficient.

### Claude's Discretion
- Exact field set for `MemoryInfo` (what fields from MEMORY.md/USER.md to expose ŌĆö key, value, store name, capacity, etc.) ŌĆö researcher determines from MemoryManager API
- Exact field set for `SkillInfo` (name, description, category, enabled state, etc.) ŌĆö researcher determines from SkillRegistry API
- Exact field set for `ModelInfo` (name, provider, context_length, etc.) ŌĆö researcher determines from ModelRegistry API; Phase 21.3 is the reference
- Whether `ModelRegistry` and `SkillRegistry` are async or sync (affects whether server fn uses `.await`)
- Order of sub-items within each screen's RSX (preserve existing stub_data ordering where possible)

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `use_server_future(api::fn_name)?`: the one-liner that replaces every stub call ŌĆö returns `Option<Result<T, ServerFnError>>`, suspends via `?`
- `api::ToolInfo { name: String, description: String }`: already defined and serializable ŌĆö Tools screen wiring needs zero new types
- `api::get_agents_status()`, `kill_agent()`, `interrupt_agent()`, `prune_agents()`: all exist in `api.rs`, just not called from the Agents component

### Established Patterns
- Server fn attribute: `#[get("/api/route")]` returns `Result<T>` (shorthand for `Result<T, ServerFnError>`)
- Type requirements: `#[derive(Serialize, Deserialize, Clone, Debug, PartialEq)]` on every response type
- Loading guard: `match resource() { Some(Ok(v)) => v, _ => Vec::new() }` ŌĆö used in Sessions, carry forward
- Signal borrow: drop `.read()` / `.write()` at `;` before any `.await`; never hold across async boundary

### Integration Points
- New server fns plug into `AppState` via `crate::server::state::global_app_state()` ŌĆö same access pattern as all existing fns
- Screens receive data via `use_context` (for shared signals like `active_screen`) plus `use_server_future` for their own data fetch
- `AppState::init()` in `state.rs` is where new `Arc<SkillRegistry>` / `Arc<ModelRegistry>` fields would be constructed

</code_context>
