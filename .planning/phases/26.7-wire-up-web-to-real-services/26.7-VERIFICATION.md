### 1. Tools Screen Live Data

**Test:** Navigate to the Tools screen in the running web UI.
**Expected:** Tool cards render with real names and descriptions sourced from the running tool registry. The grid is empty (not a placeholder error) when no tools are registered. ENABLE ALL / DISABLE ALL buttons are present but clicking them has no effect.
**Why human:** Cannot verify live data render without running the Dioxus fullstack server. The `use_server_future` call is correctly wired in code, but the data flow from registry to rendered card requires browser observation.

### 2. Memory Screen Live Data and Client Filter

**Test:** Navigate to the Memory screen. Verify both panels render. Type in the filter input.
**Expected:** Left panel (Entries) shows real MEMORY.md text blocks. Right-top panel (User Profile) shows USER.md entries. Each row has an em-dash in the `mem-ts` cell. Typing in the filter narrows visible rows in the left panel without a page reload. If MemoryManager is disabled, both panels show empty row-lists (not an error).
**Why human:** Filter input behavior and MemoryManager data flow require browser interaction; the client-side filter oninput handler cannot be verified by grep.

### 3. Skills Screen Live Data and Toggle State

**Test:** Navigate to the Skills screen.
**Expected:** Skill cards render with real names, categories, and enabled state. Cards with enabled=true show the `tgl on` CSS class (visually on). The `ALL · {count}` tab shows the actual live count. Sub-copy shows correct loaded/enabled counts.
**Why human:** CSS class application to .tgl based on enabled bool requires browser rendering to confirm. SkillRegistry data flow needs server observation.

### 4. Models Screen Family Grouping and Default Marker

**Test:** Navigate to the Models screen.
**Expected:** Models are grouped by inferred family (Anthropic (Claude), OpenAI, Google (Gemini), etc.). The configured default model (from config.model.default) shows the `DEFAULT` pill in teal. Context windows display as human-readable strings (e.g. "200k", "1M"). The EDIT button per card is present but clicking has no effect.
**Why human:** Family grouping and DEFAULT pill placement depend on live ModelRegistry data and config state.

### 5. Agents Screen KILL?/INTERRUPT/PRUNE Interactions

**Test:** Navigate to the Agents screen with at least one active subagent running. Click KILL on an agent card once, then wait 3 seconds without clicking again. Click KILL a second time within 3 seconds of the first click. Click INTERRUPT. Click PRUNE ENDED.
**Expected:** First KILL click: button label changes to "KILL?". After 3 seconds without second click: label reverts to "KILL". Second click within 3 seconds: agent is killed and the list refreshes. INTERRUPT click: button shows "..." for ~500ms then reverts to "INTERRUPT". PRUNE ENDED: list refreshes and any ended agents are removed.
**Why human:** Two-click KILL confirm with 3-second gloo_timers timeout, INTERRUPT 500ms feedback, and list refresh after prune all require live browser interaction with a running SubagentRegistry.
