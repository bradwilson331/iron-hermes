# Phase 25.1: Built-in Browser Tools - Context

**Gathered:** 2026-04-30
**Status:** Ready for planning
**Source:** Discuss-phase with user picking all 4 Recommended defaults; secondary decisions locked with codebase-grounded defaults

<domain>
## Phase Boundary

Eleven new built-in tools that give the agent browser-automation capability via the Chrome DevTools Protocol. The tools share a single AgentLoop-scoped browser singleton (lazy-spawned on first use), expose an LLM-friendly accessibility-tree snapshot for non-vision interactions, and route `browser_vision` through Phase 26's auxiliary `vision` role — making Phase 25.1 the first real consumer of that just-landed infrastructure.

The 11 tools (LLM-visible names):
1. `browser_back` — Navigate back in history
2. `browser_click` — Click element by ref ID (refs come from `browser_snapshot`)
3. `browser_close` — Close session, free resources
4. `browser_console` — Read console log entries OR evaluate JS in the page (gated)
5. `browser_get_images` — List all `<img>` elements + their src/alt
6. `browser_navigate` — Navigate to a URL (allowlist-validated)
7. `browser_press` — Press a keyboard key (Enter, Tab, Escape, etc.)
8. `browser_scroll` — Scroll the page up/down/by-pixels
9. `browser_snapshot` — Text-based accessibility-tree snapshot with stable ref IDs
10. `browser_type` — Type text into an input field by ref ID
11. `browser_vision` — Screenshot + analyze via the auxiliary vision role (PROV-06)

This phase covers exactly these 11 tools — no more, no fewer. All in a new `browser` toolset that ships DISABLED by default (Phase 25 D-20 pattern: high-blast-radius toolsets opt-in).

**Out of scope:**
- Multi-tab / multi-page management (single-page model in v2.1).
- File downloads through the browser (handled by `web_read` for known URLs).
- Cookie / session-storage persistence across agent sessions.
- Browser extension installation.
- WebDriver / Playwright fallback backends — chromiumoxide only.
- Form auto-fill from saved profiles.
- Network interception / request modification.
- PDF rendering / printing.
- Geolocation / camera / mic permissions handling beyond defaults.
- Mobile emulation / device profiles.

</domain>

<decisions>
## Implementation Decisions

### Backend & Runtime

- **D-01: Backend is `chromiumoxide` (pure Rust Chrome DevTools Protocol client).** Adds two workspace dependencies: `chromiumoxide` (the client) and `futures` (already present). No Node.js, no Python, no `chromedriver` external process. Chromium-only — Firefox/WebKit are out of scope. Chromium binary discovery via the `is_available()` prereq mechanism (see D-04).

- **D-02: Headless is the default.** `browser.headed: false` in config; flipping to `true` runs a visible window for debugging. The headless flag is a global per-agent-session setting, NOT per-tool-call. Default chromiumoxide flags include `--disable-gpu` (headless-stable), `--no-first-run`, `--no-default-browser-check`. NO `--no-sandbox` by default — operators on Docker/sandboxed environments who need it set `browser.no_sandbox: true` explicitly.

- **D-03: One browser per AgentLoop instance, lazy-spawned on first `browser_*` call.** A single `BrowserSession` struct lives behind `Arc<Mutex<Option<BrowserSession>>>` on the AgentLoop (mirrors `Arc<Mutex<Vec<String>>>` for todo state from Phase 25 D-13). First `browser_*` tool call constructs it; subsequent calls reuse. `browser_close` terminates the chromiumoxide handler and sets the Option back to None (next call re-spawns). On AgentLoop drop, the browser is killed via the chromiumoxide handler's drop impl.

### Toolset Membership & Defaults

- **D-04: New `browser` toolset.** `Tool::toolset()` returns `"browser"` for all 11 browser tools — they're a single semantic group. Phase 25 D-01's six-toolset enumeration expands to seven: `web | code | memory | agent | skills | session | browser`. Phase 25's `DEFAULT_TOOLSETS` (`["memory", "session", "agent", "skills"]`) does NOT include `browser` — operators run `hermes toolset enable browser` to opt in.

- **D-05: `is_available()` returns true iff a Chromium binary is discoverable.** Implementation walks (in order):
  1. `BROWSER_PATH` env var (if set, must point to an executable file)
  2. `CHROMIUM_PATH` env var
  3. `which chromium-browser` / `which chromium` / `which google-chrome` / `which chrome` on PATH
  4. macOS-specific: `/Applications/Google Chrome.app/Contents/MacOS/Google Chrome`, `/Applications/Chromium.app/Contents/MacOS/Chromium`
  5. Linux-specific: `/usr/bin/chromium`, `/usr/bin/chromium-browser`, `/usr/bin/google-chrome`, `/snap/bin/chromium`
  6. Windows: `%PROGRAMFILES%\Google\Chrome\Application\chrome.exe` and Edge fallback
  Returns false if none found. The `prerequisites()` method (Phase 25 D-09) declares `Prerequisite { kind: "binary_present", name: "chromium-or-chrome", description: "Chromium or Google Chrome browser binary on PATH or at a standard install location", required: true }`.

- **D-06: `browser_vision` has an additional prerequisite — the auxiliary `vision` role OR the main provider must support multimodal.** `browser_vision::prerequisites()` adds `Prerequisite { kind: "config_field", name: "auxiliary.vision OR multimodal-capable main provider", required: true }`. `is_available()` for `browser_vision` checks both the chromium binary AND the resolver: `resolver.resolve_role("vision").is_some() || resolver.resolve_for_main().model_metadata.as_ref().map_or(false, |m| m.supports_vision)`. The other 10 tools only require chromium.

### Vision Routing (PROV-06 first consumer)

- **D-07: `browser_vision` calls `resolver.resolve_role("vision")` (Phase 26 D-07 cascade).** Cascade resolution order: per-task `vision: { provider, model }` → `auxiliary: { provider, model }` → main provider. If `resolve_role` returns None, fall back to main. Phase 26 wired the cascade infrastructure but had zero consumers (vision/session_search/skills_hub/mcp_helper greenfield); Phase 25.1 is the first real consumer of `vision`. Closes the Phase 26 SC-2 "vision wired" gap retroactively.

- **D-08: Screenshot capture is full-page PNG.** chromiumoxide's `Page::screenshot(...)` with `clip: full_page`. Captured bytes converted to base64 inline-image content for the multimodal LLM call. NO viewport clipping in v2.1 — full page so vision sees scroll context. NO PDF rendering.

- **D-09: `browser_vision` accepts an optional `prompt: String` argument** ("What's the price of the highlighted item?"). Default prompt when omitted: "Describe what's visible on this page in detail, including any interactive elements, text content, and visual structure." The screenshot + prompt go to the resolved vision-role endpoint via the existing `AnyClient` infrastructure.

### Element Reference IDs (browser_click, browser_type)

- **D-10: Numeric refs produced by `browser_snapshot` and consumed by interaction tools.** Each `browser_snapshot` call walks the accessibility tree and assigns sequential integer IDs (1, 2, 3, ...) to interactive elements (links, buttons, inputs, textareas, selects, role=button/link/textbox). The output text format includes the ref inline: `[1] button "Sign In"`, `[2] textbox "Email"`. Refs are STABLE for the lifetime of one snapshot call but INVALIDATED on the next snapshot call (counter resets to 1). Tools that take a `ref` argument validate it against the most recent snapshot's ref-table; using a stale ref returns a clear error: `"ref 5 not found in current snapshot — call browser_snapshot first"`.

- **D-11: Refs are NOT URL-stable, NOT page-stable.** Navigating, scrolling, or any DOM mutation between snapshot and click can invalidate the underlying element. browser_click catches chromiumoxide's element-not-found error and returns a structured `error: "element_stale"` envelope so the LLM knows to re-snapshot. NO automatic re-snapshot on stale ref.

### Console & JS Evaluation

- **D-12: `browser_console` has two modes via a `mode` argument:** `mode: "log"` (default) reads accumulated console.log/warn/error entries since last call (or page load); `mode: "eval"` evaluates a JS expression in the page context and returns the result. Two modes inside one tool keeps the LLM-visible surface compact (one tool, two args) without splitting into `browser_console_log` / `browser_console_eval`.

- **D-13: `mode: "eval"` is gated behind the existing yolo/approval flow from Phase 21.7.** Identical mechanism to `terminal` and `execute_code`: when `mode == "eval"` AND yolo is OFF, the tool emits an `approval_needed` envelope (Phase 17 D-15 / Phase 21.7 approval format) showing the JS expression. Yolo ON skips the prompt. Read-only `mode: "log"` is NEVER gated — pure observation.

- **D-14: JS eval results are JSON-serializable only.** chromiumoxide's `Page::evaluate(expr)` returns a `serde_json::Value`. Functions, DOM nodes, undefined, and circular refs serialize to `null` with a warning string in the result envelope. Strings/numbers/bools/arrays/objects pass through.

### URL Allowlist & Domain Safety

- **D-15: `browser.allowed_domains: Vec<String>` config field — empty list = allow all.** When non-empty, `browser_navigate` validates the target URL's host against the list (exact match, no wildcards in v2.1). Rejection returns a structured error with the configured allowlist in the message. Wildcards (`*.example.com`) deferred to a future "Browser Sandboxing" phase. The allowlist applies ONLY to `browser_navigate`; in-page `browser_click` that triggers navigation is NOT validated (would require subscribing to chromiumoxide nav events; deferred).

- **D-16: `is_provider_url_safe`-style validation IS NOT applied to browser_navigate.** Phase 12's `is_provider_url_safe` requires HTTPS-or-localhost — that's appropriate for LLM API endpoints, NOT for general web browsing where HTTP is common. `browser_navigate` accepts any well-formed URL (scheme http/https/file/data — operator-restrictable via `browser.allowed_schemes: ["http", "https"]`, defaulting to `["http", "https"]`).

### File Layout

- **D-17: One file per tool, flat in `crates/ironhermes-tools/src/`** matching the existing `web_read.rs` / `web_search.rs` / `terminal.rs` / etc. pattern from Phase 17 / Phase 25:
  ```
  crates/ironhermes-tools/src/
    browser_back.rs
    browser_click.rs
    browser_close.rs
    browser_console.rs
    browser_get_images.rs
    browser_navigate.rs
    browser_press.rs
    browser_scroll.rs
    browser_snapshot.rs
    browser_type.rs
    browser_vision.rs
    browser_session.rs        # NEW: shared BrowserSession struct, Arc<Mutex<Option<...>>>
  ```
  All 11 tool structs implement `Tool` trait. `BrowserSession` is the lazy-spawned chromiumoxide handler shared across tool instances via `with_browser_session(...)` builder (mirrors `with_intercepts(...)` from Phase 25 D-16).

### Default Configuration Shape

- **D-18: New `browser:` config block** (Phase 25/26 plain-String pattern):
  ```yaml
  browser:
    headed: false                    # default headless; true for debugging
    no_sandbox: false                # set true on Docker / restricted environments
    allowed_domains: []              # empty = all hosts allowed
    allowed_schemes: ["http", "https"]
    chromium_path: null              # null = autodiscover (D-05)
    timeout_seconds: 30              # default per-operation timeout
  ```
  All fields `#[serde(default)]` for backward compat. `browser` toolset enabled in `tools.toolsets.browser.enabled: false` (default). All new types use plain Strings per Phase 25 D-25.

### Schema Cache & Banner

- **D-19: Toolset-cache-break banner emits on `hermes toolset enable browser` per Phase 25 D-04.** No new banner mechanism — inherits the existing one. Format unchanged: `[toolset: browser] enabled — schema cache will rebuild on next LLM call`.

### Test Strategy

- **D-20: Three integration tests are mandatory:**
  1. `browser_navigate_then_snapshot_returns_refs` — Spawn binary in a tempdir IRONHERMES_HOME with browser toolset enabled; navigate to a wiremock-hosted HTML page with a known button; call browser_snapshot; assert the response contains the expected `[N] button "Submit"` line. Validates D-10 ref scheme end-to-end via the subprocess pattern from Phase 24/25/26.
  2. `browser_click_with_stale_ref_returns_structured_error` — Navigate, snapshot, navigate to a different page (invalidates the ref), click with the now-stale ref; assert the error envelope contains `"element_stale"` per D-11.
  3. `browser_vision_routes_to_auxiliary_vision_role` — Configure `auxiliary.vision: { provider: openai, model: gpt-4o }` with two wiremock servers (main + aux); call browser_vision; assert the multimodal request hits the aux server, not main. End-to-end PROV-06 vision wireup validation. This test depends on Phase 26's resolve_role cascade.

- **D-21: Schema-exclusion test mandatory (Phase 25 D-26 Test 2 carry-forward):** `browser_tools_excluded_when_chromium_missing` — Spawn registry with `BROWSER_PATH` and `CHROMIUM_PATH` UNSET and a mocked `which` that returns nothing; assert all 11 browser_* schemas are NOT in `get_definitions()` output. Confirms D-05 prereq gating works for the new toolset.

- **D-22: NO live network in tests by default.** All integration tests use wiremock-hosted local HTML — no real-world browsing during `cargo test`. The Chromium binary is detected at test time via the same D-05 `is_available()` path; if no Chromium is present in CI, tests are SKIPPED with a clear log line, NOT failed (mirrors the `web_search` test pattern when FIRECRAWL_API_KEY is unset).

<specifics>
## Specific Ideas

- **Snapshot output format**: text outline, NOT JSON. Each line is `[<ref>] <role> "<accessible-name>"` with optional `<state>` annotation. Hierarchical structure shown via 2-space indentation. Example:
  ```
  [1] heading "Welcome to Acme"
  [2] navigation
    [3] link "Home"
    [4] link "About"
  [5] main
    [6] heading "Sign In"
    [7] textbox "Email" required
    [8] textbox "Password" required password
    [9] button "Sign In"
  [10] contentinfo
    [11] link "Privacy"
  ```
  The LLM gets a compact view of the page structure with stable refs for the rest of the call.

- **`browser_navigate` argument shape**: `{ "url": "https://example.com", "wait_until": "domcontentloaded" }`. `wait_until` defaults to `"domcontentloaded"`; alternatives: `"load"`, `"networkidle"`. Returns `{ "status": 200, "url": "https://example.com/redirected", "title": "Example" }`.

- **`browser_click` argument shape**: `{ "ref": 9 }`. Optional `{ "modifiers": ["ctrl"] }` for ctrl-click.

- **`browser_type` argument shape**: `{ "ref": 7, "text": "user@example.com" }`. Optional `{ "submit": false }` — when true, presses Enter after typing.

- **`browser_press` argument shape**: `{ "key": "Enter" }`. Names match standard JS KeyboardEvent.key.

- **`browser_scroll` argument shape**: `{ "direction": "down", "amount": "page" }`. amount: `"page"` | `"half"` | `<integer pixels>`.

- **`browser_get_images` output**: array of `{ "src": "...", "alt": "...", "ref": <N>, "bbox": {x, y, w, h} }` for the LLM's visual inspection of the DOM.

- **`browser_close` returns** `{ "closed": true }` — minimal confirmation; subsequent browser_* calls re-spawn lazily.

- **Stale-ref error envelope** (D-11):
  ```json
  {
    "error": "element_stale",
    "ref": 5,
    "hint": "ref 5 not found in current snapshot — call browser_snapshot first"
  }
  ```

- **`browser_console mode: eval` approval envelope** (D-13) follows the Phase 17 D-15 / Phase 21.7 shape verbatim — matches `terminal` approvals so the operator UX is consistent.

- **chromiumoxide version**: pin to a major-version-stable line (e.g., `chromiumoxide = "0.7"`). Cargo.toml entry is the only Phase 25.1 dependency addition. No Node, no Python, no chromedriver.

- **ROADMAP goal text** (currently `[Urgent work - to be planned]`) should be updated to: `"Eleven built-in browser-automation tools (browser_back/click/close/console/get_images/navigate/press/scroll/snapshot/type/vision) backed by chromiumoxide, sharing one AgentLoop-scoped browser session, with browser_vision routing through the Phase 26 auxiliary vision role. Toolset disabled by default; chromium binary required."` — plan-phase or the user can update via `/gsd-add-phase` style edit.

<deferred>
## Deferred Ideas

- **Multi-tab / multi-page management** — single Page in v2.1. Re-open if real workflows need parallel tabs.
- **Cookie / session-storage persistence across agent sessions** — chromiumoxide spawns a fresh user-data-dir per session today. Persistent profile is a future "Browser Identity" phase.
- **Browser extensions / userscripts** — security-heavy; defer to a future "Browser Sandboxing" phase.
- **Wildcard domain allowlists (`*.example.com`)** — D-15 limits to exact-match in v2.1. Re-open when the operator UX demands it.
- **WebDriver / Playwright fallback backends** — D-01 commits to chromiumoxide-only.
- **Network interception / request modification** — chromiumoxide supports it; intentionally NOT exposed to the LLM in v2.1 (huge security surface).
- **PDF rendering / printing** — separate concern; out of REQUIREMENTS.md.
- **Geolocation / camera / mic permissions handling** — defaults are "deny all"; explicit allow is a future phase.
- **Mobile emulation / device profiles** — chromiumoxide supports it; not in v2.1 scope.
- **Vision provider selection per call** (e.g., `browser_vision { provider: anthropic }`) — D-07 says always use the auxiliary `vision` role + main fallback. Per-call override defeats the cache story.
- **Auto-snapshot on navigate** — would simplify the LLM flow but doubles per-navigate token cost. LLM explicitly calling `browser_snapshot` is the discipline.
- **Auto-retry on stale ref** — D-11 explicitly says "no automatic re-snapshot". The LLM decides.
- **Full-page PDF capture as alternative to vision** — not in v2.1; defer to "Document Extraction" phase if it materializes.
- **Browser developer tools panel control** — out of scope.
