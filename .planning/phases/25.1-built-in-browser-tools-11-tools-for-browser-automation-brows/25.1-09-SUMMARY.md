# Phase 25.1 Plan 09: registry + AgentLoop + CLI wiring Summary

One-liner: All 11 browser_* tools registered in ToolRegistry via register_browser_tools, AgentLoop holds the shared session Arc via with_browser_session, and all three CLI entry points (run_single, run_chat, run_gateway) construct and wire the Arc identically.

## What Was Built

### Task 1: register_browser_tools in ToolRegistry

`crates/ironhermes-tools/src/registry.rs` gained:
- `register_browser_tools(session, resolver)` — registers all 11 browser_* tools with cloned Arc per tool. browser_vision gets an additional `resolver` Arc for D-06 vision capability checks and a `NoOpVisionHandle` as the default vision client (real wiring deferred to AnyClientVisionHandle in a future plan).
- Unit test `register_browser_tools_registers_all_11` asserting exactly 11 browser_* tools register.

`crates/ironhermes-tools/src/browser_vision.rs` gained:
- `pub struct NoOpVisionHandle` implementing `VisionClientHandle` — returns a clear error message if called without the real agent-side vision client wired. This avoids the registry needing to know about `ironhermes-agent`.

### Task 2: AgentLoop browser_session field + with_browser_session builder

`crates/ironhermes-agent/src/agent_loop.rs` gained:
- `browser_session: Option<Arc<Mutex<Option<BrowserSession>>>>` field (D-03: initialized to None).
- `pub fn with_browser_session(self, session: Arc<...>) -> Self` builder method (D-17: mirrors with_memory_manager shape).
- `agent_loop_with_browser_session_sets_field` — tokio test asserting the field is populated.
- `agent_loop_browser_session_wiring_invariants` — source-grep test locking field, builder, and None init.

### Task 3: CLI + Gateway wiring

**main.rs** — 3 wiring sites:
- `run_single` (line ~674): constructs Arc, calls `registry.register_browser_tools(...)`, adds `.with_browser_session(...)` to AgentLoop builder.
- `run_chat` (line ~1169): same Arc construction + register_browser_tools; passes `Some(browser_session.clone())` to both `run_agent_turn` call sites. `run_agent_turn` extended with new `browser_session: Option<Arc<...>>` param.
- `run_gateway` (line ~2118): same Arc construction + register_browser_tools; passes via `runner.set_browser_session(browser_session.clone())`.

**handler.rs** — `GatewayMessageHandler` gains `browser_session` field, `set_browser_session` setter, and the per-request AgentLoop builder calls `agent.with_browser_session(sess.clone())`.

**runner.rs** — `GatewayRunner` gains `browser_session` field, `set_browser_session` setter, and `build_gateway_handler` threads it to the handler.

**INV-25.1** regression test: `inv_25_1_browser_session_wired_in_all_entry_points` — checks ≥3 `register_browser_tools(`, ≥2 `.with_browser_session(`, ≥1 `set_browser_session(` in main.rs.

### make_test_agent_loop helper

Does NOT exist in agent_loop.rs. The existing tests use an inline builder pattern (construct `AnyClient::ChatCompletions(LlmClient::new(...))` + `Arc::new(RwLock::new(ToolRegistry::new()))` + `AgentLoop::new(...)`). The new browser_session tests follow this same inline pattern.

### AgentLoop drop semantics (T-25.1-04)

AgentLoop holds the shared `browser_session: Option<Arc<Mutex<Option<BrowserSession>>>>`. When AgentLoop drops, this Arc's reference count decrements. The tools (registered via register_browser_tools) each hold their own clone. When all tool clones drop with the registry, the Mutex drops, and `BrowserSession::drop` kills the handler task. No leaked handler tasks.

### Note for Plan 11 integration tests

- Use `run_single` (single-shot) for browser_* subprocess coverage — it has the simplest AgentLoop lifecycle and no REPL state.
- `browser_vision_routes_to_auxiliary_vision_role` will need to construct `AnyClientVisionHandle` with a mock resolver pointing at a wiremock aux server, then call `register_browser_tools` with that handle (or extend `register_browser_tools` to accept an `Arc<dyn VisionClientHandle>` override). The `VisionClientHandle` trait boundary makes this straightforward to mock.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] resolver type mismatch: ProviderResolver vs Arc<ProviderResolver>**
- **Found during:** Task 3 cargo build
- **Issue:** `register_browser_tools` signature requires `Arc<ProviderResolver>`, but `resolver` in main.rs entry points is an owned `ProviderResolver`.
- **Fix:** Wrapped with `std::sync::Arc::new(resolver.clone())` at all 3 call sites.
- **Files modified:** `crates/ironhermes-cli/src/main.rs`
- **Commit:** 3d06687

**2. [Rule 1 - Bug] browser_session out of scope in run_agent_turn**
- **Found during:** Task 3 cargo build
- **Issue:** The run_chat AgentLoop builder lives inside `run_agent_turn` helper function (called per-REPL-turn), not directly in `run_chat`. The `browser_session` Arc was out of scope.
- **Fix:** Added `browser_session: Option<Arc<...>>` parameter to `run_agent_turn`; passed `Some(browser_session.clone())` from both call sites in `run_chat`.
- **Files modified:** `crates/ironhermes-cli/src/main.rs`
- **Commit:** 3d06687

### Architectural Deviation (gateway pattern — plan-anticipated)

**Gateway uses GatewayRunner.set_browser_session, not direct with_browser_session in main.rs**
- **Reason:** `run_gateway` uses `GatewayRunner` which holds a shared registry and builds AgentLoop per-request in `handler.rs`. There is no direct AgentLoop builder in `run_gateway` body.
- **Chosen path:** Add `browser_session` field + `set_browser_session` setter to both `GatewayRunner` and `GatewayMessageHandler`; wire via `build_gateway_handler`. The per-request handler calls `agent.with_browser_session(...)`.
- **Impact:** INV-25.1 test checks `set_browser_session` presence (≥1) instead of `with_browser_session` (≥3). Plan's proposed INV test (≥3 `with_browser_session` in main.rs) adjusted to match reality.
- **Files modified:** `crates/ironhermes-gateway/src/handler.rs`, `crates/ironhermes-gateway/src/runner.rs`

**NoOpVisionHandle added to browser_vision.rs (public, non-test)**
- **Reason:** `register_browser_tools` needs a default `Arc<dyn VisionClientHandle>` without importing `ironhermes-agent`. The test-only `NoOpVisionClient` was not accessible from registry.rs.
- **Fix:** Added `pub struct NoOpVisionHandle` with an `anyhow::bail!` impl that gives a clear error message. Future plan can provide a `register_browser_tools_with_vision(session, resolver, vision_client)` variant for the real handle.
- **Files modified:** `crates/ironhermes-tools/src/browser_vision.rs`

## Known Stubs

**NoOpVisionHandle** — `crates/ironhermes-tools/src/browser_vision.rs` (line ~234).

`browser_vision.execute()` will return an error if called with the `NoOpVisionHandle` (the default from `register_browser_tools`). This is intentional: `browser_vision` requires the real `AnyClientVisionHandle` from `ironhermes-agent`. Plan 10 or 11 should wire the real handle via an extended `register_browser_tools_with_vision` variant or by passing `Arc<dyn VisionClientHandle>` directly. The other 10 browser_* tools are fully operational.
