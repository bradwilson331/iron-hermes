# Phase 25.1 Plan 18: GAP-7 Gateway-Handler Follow-Up Summary

GAP-7 follow-up closure: Plan 17 closed the in-core invariant + ORDER BY id
ASC pieces but never reached the gateway hot path. This plan adds
`AgentResult.appended` and replaces the buggy gateway role-filter with
direct persistence of the round-trip output, fixing GAP-7 end-to-end for
production.

## ROOT CAUSE (post-mortem of plan 17 over-claim)

Plan 17's `must_haves.truths[1]` claimed:

> "After a multi-turn browser-tool conversation in gateway/Telegram mode...
> the persisted session history reloaded into the next agent loop turn
> passes the same invariant check — i.e. tool messages are persisted
> alongside the assistant messages that triggered them, in correct order,
> with no orphans"

This was unverified end-to-end. Plan 17's deterministic regression tests
were:

| Test | What it covered | What it missed |
|------|-----------------|----------------|
| `tool_pair_round_trip.rs` (3 tests) | StateStore add_message + get_messages round-trip | Skipped the gateway handler entirely; constructed Vec<ChatMessage> directly and persisted it raw |
| `client_invariant.rs` (2 tests) | Pre-send invariant check at LlmClient layer | Did not exercise agent_loop.run() — just the client call |

The actual production path is:
`gateway::handler::run_agent` → `agent.run(messages)` → role-filter persist
to `SessionStore`. The filter at `handler.rs:642-646` was the bug; both
plan 17 test suites bypassed it.

Additionally, the `ORDER BY id ASC` fix landed in plan 17 was correct but
**inert for the gateway case**. Per `crates/ironhermes-gateway/src/session.rs`
D-02 comment: "Per D-02: on restart, fresh session starts — old data is
query-only via StateStore." Gateway sessions live in
`GatewaySession::messages` (in-memory) and never re-hydrate from SQLite
on restart. The SQL ordering matters for `query` (audit, `/sessions list`)
but never for the live agent loop.

## Why this design (compression safety)

A simpler approach — `result.messages.into_iter().skip(input_count)` — was
considered and rejected. `pre_chat_compress` (called inside `run()` before
the LLM call) mutates the input vec arbitrarily via
`engine.compress(messages, stats)` — it can shrink the vec by replacing
N messages with 1 summary. After compression, `result.messages.len()` no
longer equals `input_count + appended.len()`, so `skip(input_count)` would
either over-skip (losing new messages) or under-skip (re-persisting
prior assistants).

The append-only side channel is invariant under compression: `appended`
only ever grows when we push to it at the assistant/tool sites, regardless
of what compression does to the messages vec.
