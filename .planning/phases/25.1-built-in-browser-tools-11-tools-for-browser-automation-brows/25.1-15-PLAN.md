Close GAP-5 from `25.1-UAT.md`: when the user asks the agent to "close the browser", the model (openai/gpt-4o-mini via openrouter) responds "I don't have the capability to open or close browsers" — yet `hermes toolset list` shows browser_close as available (11/11 ✓), `browser_navigate` works fine in the same session, and the registry IS registering all 11 tools (Plan 09's `register_browser_tools_registers_all_11` test passes).

This means there's a path-divergence between three views of "is browser_close available":
- (a) `toolset list` — 11/11 ✓
- (b) `Tool::is_available()` per tool — identical to other 10 (`find_chromium_binary(None).is_some()`)
- (c) the actual `tool_schemas: Vec<ToolSchema>` sent to the LLM provider in `AgentLoop::run()` line 636 — UNCONFIRMED

The hypothesis is one of:
1. `browser_close.description` ("Close the browser session and free chromium resources. The next browser_* call will lazy-respawn a new session.") doesn't trigger the model's mapping from "close the browser" → tool call. Other 10 tools (especially browser_navigate "Navigate the browser session to a URL") have descriptions that more clearly hook the model. **Easy fix if confirmed.**
2. `browser_close.schema()` produces malformed JSON the provider strips silently (chrome-empty-properties shape). **Schema dump will reveal.**
3. Some filter (toolset filter, is_available filter, enabled_tools filter) is excluding browser_close at request time but not at toolset-list time — **shouldn't be possible structurally because all 11 tools share the same toolset() and is_available() impl, but worth proving.**

The plan splits into two tasks: **diagnose first, fix second.** The diagnostic task pins down which hypothesis is true; the fix task addresses the confirmed cause. The plan deliberately writes the fix task to be flexible enough to handle any of the three hypotheses, with clear instructions on how to adapt.

Purpose: Restore browser_close model invocation so test 10 (close idempotency + session re-spawn) becomes verifiable end-to-end.

Output: A diagnostic dump that proves which hypothesis was correct, plus a targeted fix to either the description (most likely), the schema (less likely), or the filter logic (least likely). The fix is a small change; the value is in the diagnosis preventing wasted-effort guesswork.

     {error: 'domain_blocked'|'scheme_blocked', ...} envelope on rejection."

The schema for browser_close has `"properties": {}` (empty object) — some LLM providers or schema validators are STRICT about empty properties + required arrays. This is hypothesis 2 territory. Compare with browser_navigate which has populated properties.

The description "Close the browser session and free chromium resources" uses jargon-adjacent language ("chromium resources") that the model may not associate with "close the browser". A clearer description like "Close the browser. Shuts down chromium and ends the current browser session." has the trigger word "close" + "browser" prominently. This is hypothesis 1 territory.
