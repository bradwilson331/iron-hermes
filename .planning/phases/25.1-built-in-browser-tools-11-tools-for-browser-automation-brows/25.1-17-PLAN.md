Close GAP-7 from `25.1-UAT.md`: when the gateway/Telegram path runs a multi-turn conversation that includes a prior browser-tool round-trip, the second user turn's streaming chat completion request is rejected by OpenAI with a 400 Bad Request:

> "An assistant message with 'tool_calls' must be followed by tool messages responding to each 'tool_call_id'. The following tool_call_ids did not have response messages: call_3PlsiYtJbAmwCtvtxjcIhI3G" (param: messages.[3].role)

The UAT root-cause analysis (definitively confirmed by the {err:#} diagnostic at agent_loop.rs:793 captured in the user's 2026-05-01T13:01:23Z log) names this as a SESSION-HISTORY PERSISTENCE BUG. Session 1 ends in-memory at 7 messages (system, user, asst_A, tool_A, asst_B, tool_B, asst_final); session 2 restores at 6 — exactly one tool message missing. Browser tool results (especially `browser_snapshot`) are large enough to cross the threshold where the bug manifests; smaller `web_read` results stay below it (proven by the 9-turn web_read non-streaming round-trip that succeeded at 13:51:02-19Z with no failures).

The fix has three layers:

1. **Add a strict invariant function with single source of truth** — `validate_tool_call_pairing(messages: &[ChatMessage]) -> Result<(), String>` in `crates/ironhermes-core/src/types.rs`. The semantics MUST be the strict OpenAI semantics, NOT a set-difference check: each assistant block opens a `pending_ids` set; the next non-tool message (assistant/user/system) closes that set — if pending_ids is still non-empty when the closing message arrives, the call is invalid. This catches the case where assistant_A emits ["c1"], assistant_B follows immediately, then a tool message `tool_call_id=c1` appears late — which OpenAI rejects (and which a flat HashSet across the entire array would falsely accept). After the new function lands, `crates/ironhermes-agent/src/tool_pair.rs::check_orphan_invariant` becomes a thin wrapper that delegates to it — a single source of truth for the pairing walk, with the existing `ContextError::OrphanedToolPair` mapping preserved for backward compatibility with Phase 18 D-16's call sites.

2. **Call the invariant pre-send at both client.rs sites** — `chat_completion` (non-streaming, around the existing `let request = ChatRequest { ... }` block) and `chat_completion_stream` (line 117). On Err, log the orphan id at WARN with `error.message`, return the error so agent_loop's retry loop sees a non-retriable diagnostic instead of looping on a 400. This converts an opaque provider 400 into a deterministic, locally-named failure — the change that would have prevented the user's 3-retry cascade.

3. **Fix the persistence root bug at lib.rs:462 — `ORDER BY timestamp ASC` → `ORDER BY id ASC`.** Source-level evidence pins the root cause to a same-tick timestamp collision: `unix_now()` returns millisecond-resolution f64 (`Utc::now().timestamp_millis() as f64 / 1000.0` at lib.rs:807); back-to-back `add_message` calls within one millisecond produce identical `timestamp` values; `ORDER BY timestamp ASC` ties non-deterministically across same-tick rows. Meanwhile the messages table has `id INTEGER PRIMARY KEY AUTOINCREMENT` (lib.rs:71), so `id` is strictly monotonic per insert across the entire table; the `WHERE session_id = ?1` clause filters to one session, so within the result set `id` ordering equals per-session insert order — a sufficient and clearer guarantee. We pick `ORDER BY id ASC` over `ORDER BY timestamp ASC, id ASC` because id-alone is sufficient and the smaller, clearer fix; the tiebreaker form is "safe-in-depth" but a strictly weaker statement of intent.

Once the persistence bug is fixed, the pre-send invariant becomes a belt-and-braces guard: it catches any future regression before it reaches the provider as a 400.

    ```rust
    /// Phase 25.1 GAP-7: validate the OpenAI strict tool-pair invariant before
    /// any LLM request goes on the wire.
    ///
    /// SEMANTICS (strict, NOT flat-set-difference):
    ///   Walk messages in order. Maintain `pending_ids: HashSet<String>` of
    ///   tool_call ids the most-recent assistant block emitted. Then:
    ///   - On a NEW assistant message:
    ///       * If `pending_ids` is non-empty → Err (previous assistant's tool
    ///         calls were never answered; OpenAI rejects this even if a
    ///         matching tool message appears LATER in the array).
    ///       * Otherwise, replace `pending_ids` with the ids from this
    ///         message's tool_calls (or empty if no tool_calls).
    ///   - On a TOOL message:
    ///       * If `tool_call_id` is missing → Err.
    ///       * If `tool_call_id` not in `pending_ids` → Err (orphan tool, e.g.
    ///         the id belonged to a prior assistant whose pending set was
    ///         already cleared by an interjecting assistant).
    ///       * Otherwise remove the id from `pending_ids`.
    ///   - On a USER or SYSTEM message:
    ///       * If `pending_ids` is non-empty → Err (a non-tool message
    ///         interjected before all tool results arrived).
    ///   - At END of the array:
    ///       * If `pending_ids` is non-empty → Err.
    ///
    /// Returns Err with a human-readable diagnostic naming the offending id and
    /// (where applicable) the message-array index. The WARN log at the call
    /// site surfaces this diagnostic so debugging is one-line.
    ///
    /// Why strict and not flat-set: a flat `HashSet<String>` across the whole
    /// array would falsely accept `[asst[c1], asst[no-calls], tool(c1)]` even
    /// though OpenAI's invariant is "answered BEFORE the next assistant", and
    /// the provider rejects this sequence. The flat semantics is what the prior
    /// `check_orphan_invariant` used; we fix it here and re-route that wrapper
    /// to delegate so there is a SINGLE source of truth for pairing logic
    /// across both crates.
    pub fn validate_tool_call_pairing(messages: &[ChatMessage]) -> Result<(), String> {
        use std::collections::HashSet;
        let mut pending: HashSet<String> = HashSet::new();
        for (idx, msg) in messages.iter().enumerate() {
            match msg.role {
                Role::Assistant => {
                    if !pending.is_empty() {
                        // close-out failure: previous assistant's tool calls were never answered
                        let example = pending.iter().next().cloned().unwrap_or_default();
                        return Err(format!(
                            "orphan tool_call_id '{example}' (and {} other(s)) at messages[{idx}]: a new assistant message arrived before tool messages answered the prior assistant block",
                            pending.len().saturating_sub(1)
                        ));
                    }
                    pending.clear();
                    if let Some(ref calls) = msg.tool_calls {
                        for c in calls {
                            pending.insert(c.id.clone());
                        }
                    }
                }
                Role::Tool => {
                    let id = msg.tool_call_id.as_ref().ok_or_else(|| {
                        format!("tool message at messages[{idx}] has no tool_call_id")
                    })?;
                    if !pending.remove(id) {
                        return Err(format!(
                            "orphan tool_call_id '{id}' at messages[{idx}]: tool message has no preceding (still-pending) assistant.tool_calls entry"
                        ));
                    }
                }
                Role::User | Role::System => {
                    if !pending.is_empty() {
                        let example = pending.iter().next().cloned().unwrap_or_default();
                        return Err(format!(
                            "orphan tool_call_id '{example}' (and {} other(s)) at messages[{idx}]: a {:?} message arrived while tool calls were still pending",
                            pending.len().saturating_sub(1),
                            msg.role
                        ));
                    }
                }
            }
        }
        if let Some(id) = pending.iter().next() {
            return Err(format!(
                "orphan tool_call_id '{id}': assistant.tool_calls entry has no following tool message before end-of-history"
            ));
        }
        Ok(())
    }
    ```
