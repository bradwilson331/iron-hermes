## Summary

Phase 25.1 adds 11 browser-automation tools to IronHermes backed by the `chromiumoxide` Rust crate (pure CDP, no Node/Python). All 11 tools share a single `BrowserSession` struct held behind `Arc<Mutex<Option<BrowserSession>>>` on `AgentLoop`, lazy-spawned on first use. The tools form a new `browser` toolset that ships disabled by default, requiring an explicit `hermes toolset enable browser` and a discoverable Chromium binary.

The implementation follows every existing project pattern exactly: one `.rs` file per tool flat in `crates/ironhermes-tools/src/`, a `with_browser_session(...)` builder method mirroring `with_intercepts(...)`, `is_available()` returning false when no Chromium binary is found (schema-excluded until prereq met), and `browser_console mode:"eval"` gated through the same `should_prompt_for_approval()` helper that gates `terminal` and `execute_code`. The `browser_vision` tool is Phase 25.1's most architecturally interesting piece: it is the **first real consumer** of the Phase 26 `resolve_role("vision")` cascade, constructing an `AnyClient` via `build_role_client(resolver, "vision")` and sending a multimodal request using the already-existing `ContentPart::ImageUrl` type in `ironhermes_core::types`.

**Primary recommendation:** Follow the `with_intercepts` builder pattern verbatim; reuse `ContentPart::ImageUrl { image_url: ImageUrl { url: "data:image/png;base64,...", detail: None } }` for the vision payload; use `page.execute(AccessibilityGetFullAXTreeParams::default())` via raw CDP for the accessibility tree; pin chromiumoxide to `"0.9"` (current: 0.9.1).

### Browser Launch Pattern

```rust
// Source: Context7 /mattsse/chromiumoxide — "Launch Browser Instance"
use chromiumoxide::browser::{Browser, BrowserConfig};
use futures::StreamExt;

let config = BrowserConfig::builder()
    // headless is default; .with_head() for headed mode
    .no_sandbox()              // only when browser.no_sandbox: true
    .chrome_executable(path)   // only when custom path resolved
    .arg("--disable-gpu")
    .arg("--no-first-run")
    .arg("--no-default-browser-check")
    .launch_timeout(Duration::from_secs(config.browser.timeout_seconds))
    .build()?;

let (browser, mut handler) = Browser::launch(config).await?;

// Handler MUST be driven on a separate tokio task (CDP websocket pump).
let _handler_task = tokio::spawn(async move {
    while let Some(h) = handler.next().await {
        if h.is_err() { break; }
    }
});
```

**Drop semantics:** [VERIFIED: Context7] `browser.close().await?` sends the CDP Browser.close command. Dropping `Browser` without calling `close()` also terminates the chromium process via the handler's drop impl. For Phase 25.1's `browser_close` tool, call `browser.close().await` explicitly; for `AgentLoop` drop path, rely on the handler task drop (the handler ends when the channel closes and chromium exits naturally).

### Custom Binary Path

[VERIFIED: Context7] `.chrome_executable(path)` on `BrowserConfig::builder()` accepts a `PathBuf` or `impl Into<PathBuf>`.

```rust
// D-05 binary discovery (implementation sketch for is_available / BrowserSession::launch):
fn find_chromium_binary(config: &BrowserConfig) -> Option<PathBuf> {
    // 1. BROWSER_PATH env var
    if let Ok(p) = std::env::var("BROWSER_PATH") {
        let path = PathBuf::from(&p);
        if path.is_file() { return Some(path); }
    }
    // 2. CHROMIUM_PATH env var
    if let Ok(p) = std::env::var("CHROMIUM_PATH") {
        let path = PathBuf::from(&p);
        if path.is_file() { return Some(path); }
    }
    // 3. config.browser.chromium_path
    if let Some(ref p) = config.chromium_path {
        let path = PathBuf::from(p);
        if path.is_file() { return Some(path); }
    }
    // 4. PATH search (which)
    for name in &["chromium-browser", "chromium", "google-chrome", "chrome"] {
        if let Ok(p) = which::which(name) { return Some(p); }
    }
    // 5. Platform-specific hard paths
    #[cfg(target_os = "macos")]
    for p in &[
        "/Applications/Google Chrome.app/Contents/MacOS/Google Chrome",
        "/Applications/Chromium.app/Contents/MacOS/Chromium",
    ] {
        let path = PathBuf::from(p);
        if path.is_file() { return Some(path); }
    }
    #[cfg(target_os = "linux")]
    for p in &[
        "/usr/bin/chromium", "/usr/bin/chromium-browser",
        "/usr/bin/google-chrome", "/snap/bin/chromium",
    ] {
        let path = PathBuf::from(p);
        if path.is_file() { return Some(path); }
    }
    // Windows handled similarly with %PROGRAMFILES%
    None
}
```

[ASSUMED] `which` crate is not in workspace deps. `which::which()` must either be added or replaced with a manual PATH walk using `std::env::var("PATH").split(':').find_map(...)`. **Recommendation:** do not add `which` crate — implement a minimal PATH search inline to avoid new workspace dep. The existing codebase avoids unnecessary deps per Phase 21 D-18 pattern.

## Accessibility Tree → Ref Table → Click Flow

### browser_snapshot Implementation Sketch

[VERIFIED: Context7 CDP raw execute pattern; ASSUMED: AXNode field names follow CDP spec]

```rust
// Rough implementation of browser_snapshot execute():
async fn snapshot_page(session: &mut BrowserSession) -> anyhow::Result<String> {
    // Clear previous ref table
    session.ref_table.clear();

    // Option A: CDP Accessibility.getFullAXTree (preferred)
    use chromiumoxide_cdp::cdp::browser_protocol::accessibility::GetFullAxtreeParams;
    let ax_result = session.page
        .execute(GetFullAxtreeParams::builder().build())
        .await?;
    let nodes = ax_result.nodes;  // Vec<AXNode>

    let mut counter: u64 = 0;
    let mut output = String::new();

    for node in &nodes {
        // role comes from node.role.value (AXValue)
        let role = node.role.as_ref()
            .and_then(|r| r.value.as_ref())
            .and_then(|v| v.as_str())
            .unwrap_or("unknown");

        let name = node.name.as_ref()
            .and_then(|n| n.value.as_ref())
            .and_then(|v| v.as_str())
            .unwrap_or("");

        // Assign ref only to interactive elements (D-10)
        let is_interactive = matches!(role, "button" | "link" | "textbox" |
            "checkbox" | "radio" | "combobox" | "menuitem" | "tab" |
            "searchbox" | "spinbutton" | "slider");

        if is_interactive {
            counter += 1;
            // Store BackendNodeId for later click/type dispatch
            if let Some(backend_id) = node.backend_dom_node_id {
                session.ref_table.insert(counter, backend_id.to_string());
            }
            output.push_str(&format!("[{}] {} \"{}\"\n", counter, role, name));
        } else {
            // Non-interactive structural nodes shown without ref
            output.push_str(&format!("{} \"{}\"\n", role, name));
        }
    }

    Ok(output)
}
```

**Option B fallback (if GetFullAxtreeParams not generated):**

```rust
// JS-based accessibility walker — works on any chromiumoxide version
let ax_json: serde_json::Value = session.page
    .evaluate(r#"
        (function walkAX(node, depth) {
            if (!node) return [];
            const role = node.role || '';
            const name = node.name || '';
            const id = node.backendDOMNodeId || 0;
            return [{role, name, id, depth}]
                .concat(...(node.children || []).map(c => walkAX(c, depth+1)));
        })(window.__ax_tree__, 0)
    "#)
    .await?.into_value::<serde_json::Value>()
    .unwrap_or(serde_json::Value::Array(vec![]));
```

[ASSUMED] The exact field names on `AXNode` structs depend on the generated chromiumoxide_cdp bindings. The CDP spec defines: `nodeId`, `backendDOMNodeId`, `role` (AXValue), `name` (AXValue), `children` (Vec<AXNodeId>). Implementor must check actual generated types.

### browser_click: Ref → Click

```rust
async fn click_ref(session: &mut BrowserSession, ref_id: u64) -> anyhow::Result<String> {
    let backend_node_id = session.ref_table.get(&ref_id)
        .ok_or_else(|| anyhow::anyhow!(serde_json::json!({
            "error": "element_stale",
            "ref": ref_id,
            "hint": format!("ref {} not found in current snapshot — call browser_snapshot first", ref_id)
        }).to_string()))?
        .clone();

    // Use CDP DOM.resolveNode to get runtime object, then call click()
    // OR use JS via evaluate to click by backendNodeId
    let result = session.page.evaluate(
        format!("document.querySelector('[data-ref-id=\"{}\"]').click()", backend_node_id)
    ).await;

    match result {
        Ok(_) => Ok(r#"{"clicked": true}"#.to_string()),
        Err(e) if is_stale_element_error(&e) => {
            Ok(serde_json::json!({
                "error": "element_stale",
                "ref": ref_id,
                "hint": format!("ref {} not found in current snapshot — call browser_snapshot first", ref_id)
            }).to_string())
        }
        Err(e) => Err(e),
    }
}
```

[ASSUMED] The precise approach for clicking by `BackendNodeId` in chromiumoxide — options are: (1) `element.click()` on an `Element` retrieved by selector, (2) CDP DOM.focus + CDP Input.dispatchMouseEvent via `page.execute()`, or (3) `page.evaluate(js_that_queries_by_backend_node_id)`. The cleanest approach is `page.find_element(selector).await?.click().await?` but requires a CSS selector. For ref-table lookup, consider storing both a CSS selector and the BackendNodeId at snapshot time. This is an **open question** for the implementor — the planner should note it as a Wave 0 task to verify chromiumoxide's click-by-BackendNodeId API.

### Call Flow

// 3. Assemble multimodal message (ContentPart::ImageUrl already exists in types.rs)

    role: Role::User,
    content: Some(MessageContent::Parts(vec![
        ContentPart::Text { text: prompt },
        ContentPart::ImageUrl {

                url: data_url,
                detail: None,
            }
        },
    ])),
    tool_calls: None,
    tool_call_id: None,

    let result: serde_json::Value = session.page
        .evaluate(&expression)
        .await
        .map(|r| r.into_value::<serde_json::Value>().unwrap_or(serde_json::Value::Null))

## Open Questions / Risks (RESOLVED)

### OQ-1: chromiumoxide BackendNodeId Click API [HIGH RISK]

**RESOLVED:** data-ironhermes-ref CSS attribute injection per Plan 25.1-05 — at snapshot time, browser_snapshot injects `data-ironhermes-ref="N"` on each interactive element via JS, then browser_click/type use `page.find_element("[data-ironhermes-ref=\"N\"]")` (CSS selector path, no BackendNodeId click API needed).

**What we know:** CDP `Accessibility.getFullAXTree` returns `AXNode` objects with `backendDOMNodeId` fields. CDP `DOM.resolveNode` can resolve a `BackendNodeId` to a runtime object.

**What's unclear:** Does chromiumoxide expose a direct `click_by_backend_node_id(id)` method on `Page`, or must we use `page.execute(CDP params)` to dispatch a mouse event? The `page.find_element(selector)?.click()` path requires a CSS selector, not a BackendNodeId.

**Recommendation:** At implementation time, check the generated chromiumoxide types for `Input.dispatchMouseEvent` and `DOM.getBoxModel` — these allow clicking by coordinates fetched from a BackendNodeId. Alternatively, at snapshot time, store a JS-queryable selector alongside the BackendNodeId (e.g., `document.querySelector('[data-axref="N"]')` after injecting `data-axref` attributes via JS).

### OQ-2: AnthropicClient multimodal support [MEDIUM RISK]

**RESOLVED:** Plan 25.1-01 task 3 adds Image ContentBlock + convert_image_url_to_block helper to anthropic_client.rs. The conversion from `ContentPart::ImageUrl { url: "data:image/png;base64,..." }` to Anthropic's `{"type":"image","source":{"type":"base64","media_type":"image/png","data":"..."}}` ships as a Wave 0 prerequisite of Plan 25.1-08 (browser_vision).

**What we know:** `ContentPart::ImageUrl` exists in `types.rs`. `AnyClient::chat_completion` passes `ChatMessage` slices to either `LlmClient` or `AnthropicClient`.

**What's unclear:** `anthropic_client.rs` (38KB) may or may not convert `ContentPart::ImageUrl` with a `data:` URL into Anthropic's `{"type":"image","source":{"type":"base64",...}}` format. OpenAI ChatCompletions accepts `image_url` directly; Anthropic does not.

**Recommendation:** Wave 0 task — inspect `anthropic_client.rs` message serialization. If `MessageContent::Parts` with `ImageUrl` is not handled, add the conversion as part of this phase before implementing `browser_vision`.

### OQ-3: Console log accumulation API [LOW RISK]

**RESOLVED:** Plan 25.1-06 implements browser_console with the JS console-override injection approach — after each browser_navigate, inject a `console.log/warn/error` override that buffers entries into `window.__ironhermes_console_buffer__`; mode:"log" reads and clears the buffer. No async Stream lifetime juggling.

**What we know:** CDP `Runtime.consoleAPICalled` event exists. chromiumoxide supports event listeners.

**What's unclear:** Whether the `page.event_listener::<EventConsoleAPICalled>()` approach requires holding a Stream across multiple tool calls (which is hard with `&mut` borrows across async boundaries), or whether a buffer approach (injecting JS console override at page load) is more practical.

**Recommendation:** Use JS console override injection: after each `browser_navigate`, inject a console override that buffers entries into `window.__ironhermes_console_buffer__`. The `browser_console mode:"log"` tool reads and clears this buffer. Simpler, no async Stream lifetime issues.

### OQ-4: which crate vs inline PATH walk [LOW RISK]

**RESOLVED:** Plan 25.1-02 implements `find_chromium_binary()` with an inline PATH walk (`std::env::var("PATH").split(separator).find_map(...)`) — no new workspace crate dep.

**What we know:** `which` crate is not in workspace deps. Phase 25.1 binary discovery requires PATH search.

**Recommendation:** Implement inline: `std::env::var("PATH").unwrap_or_default().split(':').find_map(|dir| { let p = PathBuf::from(dir).join(name); p.is_file().then_some(p) })`. No new crate dep.

### OQ-5: BrowserSession in ironhermes-tools creates circular dep risk [LOW RISK]

**RESOLVED:** Plan 25.1-02 keeps BrowserSession entirely within ironhermes-tools; AgentLoop holds `Arc<tokio::sync::Mutex<Option<BrowserSession>>>` via `with_browser_session(...)` builder (Plan 25.1-09). ironhermes-agent already depends on ironhermes-tools — no new edge added, no cycle.

**What we know:** `BrowserSession` holds `Browser` and `Page` from `chromiumoxide`. Only `ironhermes-tools` needs the `chromiumoxide` dep.

**Recommendation:** Keep `BrowserSession` entirely within `ironhermes-tools`. The `AgentLoop` field type becomes `Option<Arc<tokio::sync::Mutex<ironhermes_tools::browser_session::BrowserSession>>>` — this requires `ironhermes-agent` to depend on `ironhermes-tools` (it already does).
