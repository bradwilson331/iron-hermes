<objective>
Close GAP-3 (D-15 / T-25.1-01 — URL allowlist not enforced) and GAP-4 (D-13 / T-25.1-02 — yolo+approval gate bypassed for browser_console mode='eval'). Both UAT failures share one architectural root cause:

**The runtime `Config` loaded by `main.rs` is not threaded into `BrowserNavigateTool` or `BrowserConsoleTool`.** Both tools call `ironhermes_core::config::Config::load()` from disk inside `execute()` — see `browser_navigate.rs:79-81` and `browser_console.rs:166-168`:

```rust
// browser_navigate.rs (BUG):
let cfg = ironhermes_core::config::Config::load().unwrap_or_default().browser;
// browser_console.rs (BUG):
let yolo = ironhermes_core::config::Config::load()
    .map(|c| c.autonomous.yolo).unwrap_or(false);
```

The same flaw exists in `ensure_session()` helpers in both files — they call `Config::load().unwrap_or_default().browser` so when load fails, BrowserSession spawns with `BrowserConfig::default()` (empty allowed_domains, default schemes). This explains why Plan 02's unit tests pass (5/5 against `validate_navigation_url`) but real-world enforcement is broken: the function works perfectly with the BrowserConfig it receives — it's just receiving the wrong (default) BrowserConfig.

Both tools' unit tests pass (Plan 02: 5/5 for validate_navigation_url; Plan 07: 7/7 for browser_console gating logic) — the issue is exclusively at the tool→Config wiring boundary, not in the gate or allowlist logic itself.

Symptoms aligned with the suspected root cause:
- Empty `allowed_domains: []` in `BrowserConfig::default()` means "allow all" per D-15 → wikipedia.org navigates fine despite operator config
- Default `autonomous.yolo: true` (or whatever the default is — verify during execution; if false then `Config::load().unwrap_or_default()` returning a default still wouldn't bypass the gate, in which case the bug is that `Config::load()` succeeds but loads a different config file path than the CLI did — same architectural fix)

**The fix:** Add `config: Arc<ironhermes_core::config::Config>` to both `BrowserNavigateTool` and `BrowserConsoleTool`. Wire it through `register_browser_tools_with_vision`. Pass `Arc::new(config.clone())` from all 3 CLI entry points (`run_single`, `run_chat`, `run_gateway`) — the runtime config is already in scope at all 3 sites.

Purpose: Restore the SSRF mitigation (T-25.1-01) and the arbitrary-JS mitigation (T-25.1-02). Both are major security controls that are currently silently bypassed.

Output: A 4-file change with compile-time enforcement (the new param is required) plus a workspace-level integration test that exercises the full Config→tool→behavior path.
</objective>
