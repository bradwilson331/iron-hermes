## HIGH

### H-01: Env-var leakage on test panic — `BROWSER_PATH` / `CHROMIUM_PATH` survive a failed assertion

**File:** `crates/ironhermes-cli/src/toolset_cmd.rs:521-624` (all three new tests)

**Issue:** Each test mutates process-wide environment variables and clears
them via a trailing `unsafe { std::env::remove_var(...) }` AFTER the assertions.
If any assertion panics (e.g., a future regression makes `available_count != 0`),
the env var is left set and bleeds into:
1. Subsequent tests in the same process that share the env_lock (line 406).
2. Other crates' tests when run via `cargo test --workspace` — the env_lock at
   `toolset_cmd.rs:406` only synchronises within this module; no
   cross-crate coordination exists.

The `unwrap_or_else(|p| p.into_inner())` poison-recovery at lines 522, 576,
635 deliberately swallows the poisoned-mutex signal that would otherwise warn
the next test something panicked under the lock. This compounds the leak.

**Fix:** wrap the env mutation in a `Drop` guard so cleanup runs even on
panic:

```rust
struct EnvGuard { keys: Vec<&'static str> }
impl Drop for EnvGuard {
    fn drop(&mut self) {
        for k in &self.keys {
            unsafe { std::env::remove_var(k); }
        }
    }
}

#[test]
fn browser_row_reports_zero_when_chromium_unavailable() {
    let _g = env_lock().lock().unwrap_or_else(|p| p.into_inner());
    let _env = EnvGuard { keys: vec!["BROWSER_PATH", "CHROMIUM_PATH"] };
    unsafe {
        std::env::remove_var("CHROMIUM_PATH");
        std::env::set_var("BROWSER_PATH", "/dev/null/nonexistent-chromium-for-test");
    }
    // ... assertions ... no manual remove_var at end.
}
```

The same guard fixes the green-arm test at `:634` which sets `CHROMIUM_PATH`
without a defensive prior `remove_var("BROWSER_PATH")` in the failure path.

## MEDIUM

### M-01: Non-deterministic diagnostic — HashSet iteration order leaks into the orphan-id message

**File:** `crates/ironhermes-core/src/types.rs:420, 445, 455`

**Issue:** When `pending` contains multiple unanswered tool_call_ids, the
diagnostic surfaces `pending.iter().next().cloned()` as the "example" id.
`HashSet` iteration order is non-deterministic across runs, so two identical
failures will print different example ids. This makes log-grep brittle (the
UAT-style `grep "call_3PlsiYtJbAmwCtvtxjcIhI3G"` only fires for one specific
run) and hampers debugging when the user reports "the orphan id changes
between attempts".

**Fix:** sort or pick the smallest id deterministically:

```rust
let mut ids: Vec<&String> = pending.iter().collect();
ids.sort();
let example = ids.first().map(|s| s.as_str()).unwrap_or("");
```

Or surface ALL pending ids: `pending.iter().sorted().join(", ")`.

### M-02: "(and 0 other(s))" diagnostic when only one id is pending

**File:** `crates/ironhermes-core/src/types.rs:422-423, 447-448`

**Issue:** `format!("orphan tool_call_id '{example}' (and {} other(s))", pending.len().saturating_sub(1))`
prints `(and 0 other(s))` when `pending.len() == 1`. This is grammatically
ugly and slightly confusing in logs ("0 other(s)" suggests we expected more).

**Fix:** branch on the count:

```rust
let extra = pending.len() - 1;
let suffix = match extra {
    0 => String::new(),
    1 => " (and 1 other)".to_string(),
    n => format!(" (and {n} others)"),
};
return Err(format!("orphan tool_call_id '{example}'{suffix} at messages[{idx}]: ..."));
```

### M-03: Diagnostic loses the orphan id at the agent-crate boundary

**File:** `crates/ironhermes-agent/src/tool_pair.rs:177-182`

**Issue:** `check_orphan_invariant` maps `Err(String)` → `ContextError::OrphanedToolPair`
(a unit variant). The diag string is logged at WARN inside the wrapper but
the caller (e.g., `summarizing_engine`, `context_engine`) only sees the
unit-variant Err. Programmatic callers cannot recover the offending id; only
log scrapers can. This is acceptable for backward compat with Phase 18 D-16
but means GAP-7's "name the orphan id" guarantee only holds at the WARN-log
layer, not the typed-error layer.

**Fix:** add a payload to the variant if any caller will need to surface it
to UI/UAT:

```rust
ContextError::OrphanedToolPair { diagnostic: String }
```

(Out-of-scope for this plan; flagged so a future caller migration is on the
roadmap.)

### M-04: Single-char `'x'` substring assertion is fragile

**File:** `crates/ironhermes-agent/tests/client_invariant.rs:53, 89`

**Issue:** `assert!(err_text.contains('x'), ...)` is the second guard on each
integration test. The orphan id was hand-picked as `"x"` (line 18), and the
diagnostic does include it — but `'x'` is one character: any future change
that makes the error format include literal "x" anywhere (e.g., "extra",
"prefix", a hex byte) will let a regression where the id is dropped slip
through. Also, the diag includes role names like `"User"` which don't have
'x', but the wrapper diag at `tool_pair.rs:179` says
`"tool-pair invariant violated"` which also has no 'x' — so the test only
passes today because the core diag literally substitutes `id`, not because
the assertion is robust.

**Fix:** use a more specific id and check substring:

```rust
let orphan_id = "call_TEST_ABCDEF";
// ... build history with this id ...
assert!(err_text.contains(orphan_id), "diagnostic must reference orphan id; got: {err_text}");
```

### L-03: `build_toolset_show_view` returns empty `member_summary` silently

**File:** `crates/ironhermes-cli/src/toolset_cmd.rs:215`

**Issue:** the `ToolsetRow` returned by `build_toolset_show_view` always sets
`member_summary: String::new()` with the comment "not used for show". A
future caller that consumes both the tuple list AND the row's summary will
get empty output without warning.

**Fix:** either compute the summary (cheap; 11 names tops) so the row is

### L-04: `state_store_get_messages_orders_by_id_not_timestamp` relies on WAL allowing concurrent connections

**File:** `crates/ironhermes-state/tests/tool_pair_round_trip.rs:219-227`

**Issue:** the inversion test opens a sibling `rusqlite::Connection` on the
same DB file while `StateStore` holds an open connection. This works
because `StateStore::init_schema` enables WAL (`crates/ironhermes-state/src/lib.rs:45`)
and sets `busy_timeout(5s)` on both connections. If a future change disables
WAL or changes the lock timeout, this test could become flaky on slow CI.

**Fix:** non-blocking — accept as-is, but add a comment noting the WAL
dependency, or expose a `#[cfg(test)] fn raw_conn(&mut self) -> &mut Connection`
on `StateStore` so the test uses the same connection (no WAL dependency).

### WARNING

#### W-01: `inv_25_1_browser_session_wired_in_all_entry_points` in `main.rs` still asserts `>= 3` — will not catch a future deletion of the rata wiring

**File:** `crates/ironhermes-cli/src/main.rs:2770-2774`

**Issue:** The pre-existing INV test in `main.rs` asserts `reg_count >= 3`
(run_single + run_chat + run_gateway). Plan 25.1-19 adds a 4th call site in
`tui_rata/event_loop.rs`, which `main.rs`'s test does NOT scan (it uses
`include_str!("main.rs")`). If the rata wiring is deleted in a future plan,
`inv_25_1_browser_session_wired_in_all_entry_points` in `main.rs` will still
pass (3 remaining sites satisfy `>= 3`) and will not catch the regression.

The new `inv_25_1_gap8_browser_tools_wired_in_rata_chat` test in `event_loop.rs`
covers the rata side, so the regression net does catch the deletion — but only
if `event_loop.rs` is in scope for the test run. If someone deletes the test
alongside the wiring (the classic "delete both together" anti-pattern), no
cross-file test fires.

**Fix:** Update the `main.rs` INV test to assert `>= 4` now that there are
four entry points, and update its doc-comment to list all four:

```rust
// - register_browser_tools must appear in run_chat, run_single, run_gateway,
//   AND run_chat_ratatui::build_app_deps (Phase 22 D-04 + GAP-8, plan 25.1-19)
assert!(
    reg_count >= 4,
    "Phase 25.1 D-04: register_browser_tools_with_vision MUST be called in \
     run_single, run_chat, run_gateway, AND run_chat_ratatui::build_app_deps \
     (Phase 22 D-04 mirror-pattern invariant + GAP-8 closure); got {} calls",
    reg_count
);
```

Note: `main.rs`'s test only counts occurrences within `main.rs` itself (via
`include_str!("main.rs")`), so it cannot enforce the rata call site even with
an updated threshold. The real fix is a workspace-level INV test that scans
all entry-point source files. Updating the threshold is still useful because it
documents intent and will catch regressions in `main.rs`'s own three sites more
precisely. The stronger fix (workspace-level scan) is a follow-up.

### NIT

#### N-01: Double `create_dir_all` call for the same directory list in `build_app_deps`

**File:** `crates/ironhermes-cli/src/tui_rata/event_loop.rs:133-137`

**Issue:** `build_app_deps` calls `std::fs::create_dir_all` for the
`["cron", "sessions", "logs", "hooks", "memories", "skills", "workspace",
"subagent-transcripts"]` list inline at line 133-136, then immediately calls
`ensure_home_dirs(&hermes_home)` at line 137 — which does the exact same
`create_dir_all` loop over the same list. Both calls succeed because
`create_dir_all` is idempotent, so there is no correctness defect. But the
inline loop is dead code: every directory it creates is unconditionally
recreated by `ensure_home_dirs` one line later.

**Fix:** Delete the inline loop at lines 133-136; the `ensure_home_dirs` call
at line 137 is sufficient:

```rust
// Before (lines 133-137):
for sub in &["cron", "sessions", "logs", ...] {
    std::fs::create_dir_all(hermes_home.join(sub))?;
}
ensure_home_dirs(&hermes_home)?;

// After:
ensure_home_dirs(&hermes_home)?;
```

#### N-02: `inv_25_1_browser_session_wired_in_all_entry_points` in `main.rs` counts comment lines via `source.matches(...)` without filtering — the self-invalidating grep-gate trap applies

**File:** `crates/ironhermes-cli/src/main.rs:2764-2788`

**Issue:** The test at line 2764 reads `let source = include_str!("main.rs")`
and runs `source.matches(".with_browser_session(").count()` without filtering
comment lines. Both the plan-19 PLAN.md and the plan-19 INV test in
`event_loop.rs` (line 643) explicitly call out the self-invalidating-grep-gate
trap and filter comment lines. The `main.rs` test does not, meaning:

1. A future developer who adds a comment line containing `.with_browser_session(`
   (e.g., a doc-comment example) would increment the count without adding real
   wiring.
2. The test's own assertion string `"with_browser_session MUST be on AgentLoop
   builder chain"` contains `.with_browser_session(` — though in a string
   literal, not on its own source line, so it doesn't trigger this specific
   instance. But the pattern is fragile.

The companion `inv_25_1_browser_config_threaded_through_all_entry_points` test
at line 2798 DOES filter comments (lines 2801-2804). The two tests in the same
module are inconsistent.

**Fix:** Add comment-line filtering to `inv_25_1_browser_session_wired_in_all_entry_points`:

```rust
let source = include_str!("main.rs");
let non_comment: String = source.lines()
    .filter(|line| !line.trim_start().starts_with("//"))
    .collect::<Vec<_>>()
    .join("\n");
let with_count = non_comment.matches(".with_browser_session(").count();
```
