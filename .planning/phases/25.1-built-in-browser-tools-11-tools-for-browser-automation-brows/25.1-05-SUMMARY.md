## Key Implementation Notes

### OQ-1 Fix: data-ironhermes-ref attribute injection

At snapshot time, `SNAPSHOT_WALKER_JS` does two things in sequence:
1. **Clear pass**: `document.querySelectorAll('[data-ironhermes-ref]')` → removes all existing attributes (D-10 stale ref invalidation + T-25.1-03b mitigation — attacker-injected fake refs are cleared first).
2. **Walk pass**: DFS from `document.body`, assigns `counter += 1` and `el.setAttribute('data-ironhermes-ref', String(counter))` for each interactive element.

The Rust side populates `ref_table` with `N → "[data-ironhermes-ref=\"N\"]"` — the exact CSS selector string plan 06 passes to `page.find_element(selector).await?.click()`.

### chromiumoxide page.evaluate() serialization

`page.evaluate(expr).await` returns `EvaluationResult`. Calling `.into_value()` on the result yields `Result<serde_json::Value>`. For the array returned by `SNAPSHOT_WALKER_JS`, this works correctly: JS `return out;` where `out` is an Array of plain objects serializes cleanly through CDP's JSON bridge.

The `ref` field name in the JS object (e.g., `{ ref: counter, role, name, depth }`) is not a reserved word in JSON — only in Rust. Rust-side hand-decoding via `entry_val.get("ref").and_then(|v| v.as_u64())` avoids any serde rename complexity.

### GET_IMAGES_JS argument passing

`chromiumoxide::Page::evaluate()` accepts a single JS expression string with no separate argument mechanism. To pass `startCounter` (the current max ref in ref_table), the JS is formatted as:

```rust
let js = format!("({})({})", GET_IMAGES_JS.trim(), start_counter);
```

This wraps the `function(startCounter) { ... }` body (which is already written as a named-parameter function expression without the self-calling `()`) in an extra `()(N)` invocation. The JS constant defines the function body; Rust inlines the current counter value at call time.

### Snapshot depth calculation note (for plan 11 integration tests)

Real-world pages rarely exceed depth 8-10 in the interactive-element tree. The walker increments `nextDepth` only when crossing into an interactive or structural element boundary — not for every DOM level. A page with nav > ul > li > a structure will show `a` at depth 2 (nav=0, li children of nav increments to 1, a children of li increments to 2). The `depth.min(20)` cap in Rust prevents runaway indentation on pathological DOMs.

### Note for plan 06 (browser_click / browser_type)

`ref_table` values are CSS selector strings in the format `"[data-ironhermes-ref=\"N\"]"`. Plan 06 should use:

```rust
let selector = sess.ref_table.get(&ref_id)
    .ok_or_else(|| anyhow::anyhow!(json!({"error":"element_stale","ref":ref_id,...}).to_string()))?;
let element = sess.page.find_element(selector.as_str()).await?;
element.click().await?;
```

The `page.find_element(css_selector)` → `Element::click()` path is the clean chromiumoxide API — no CDP BackendNodeId click uncertainty.
