## Key Implementation Notes

### chromiumoxide version
Pinned to `version = "0.9"` (Cargo resolves to 0.9.1). Added as a direct dep to ironhermes-tools only — no other crate needs the chromiumoxide type system. Not added to workspace deps since single-consumer.

### base64 workspace dep
base64 0.22 was already in `[workspace.dependencies]` in the root Cargo.toml. Only the per-crate `{ workspace = true }` re-declaration was needed in ironhermes-tools/Cargo.toml.

### Stub module warnings
No warnings introduced by the empty stub modules. Rust 2024 edition with empty module files compiles cleanly. The single pre-existing warning (`DEFAULT_SAFE_TOOLS` unused in delegate_task.rs) is unchanged.

### ImageSource::Url branch
Added support for `http://` / `https://` URLs producing `ImageSource::Url { url }` even though RESEARCH.md notes that browser_vision only sends `data:` URLs. This is included per the plan action specification and serves as a future-proof path. Whether the Anthropic API actually accepts URL sources depends on the API version in use — if it doesn't, the request will fail at the API level, not at the serialization level. This is documented for the browser_vision plan (25.1-08).

### OQ-2 Resolution
The User arm in `adapt_messages` now handles all three `MessageContent` variants:
- `Text(t)` → single Text block (existing behavior preserved)
- `Parts(parts)` → maps each part: Text → Text block, ImageUrl → Image block via `convert_image_url_to_block()`
- `None` → empty Text block (existing behavior preserved)

The `convert_image_url_to_block()` helper is the security mitigation for T-25.1-OQ2: strict media-type allowlist (`image/png`, `image/jpeg`, `image/gif`, `image/webp`) rejects anything else with `tracing::warn`, ensuring malformed URLs never crash the LLM call.
