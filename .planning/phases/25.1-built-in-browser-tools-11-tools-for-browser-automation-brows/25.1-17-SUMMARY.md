# Phase 25.1 Plan 17: GAP-7 Streaming-400 Closure (Tool-Pair Invariant + Persistence Fix) Summary

GAP-7 closure: strict tool-pair invariant in `ironhermes-core`, refactor `tool_pair::check_orphan_invariant` to delegate, add pre-send guards at both client.rs sites, and fix the same-millisecond-timestamp persistence bug at `lib.rs:468` (was 460) by switching `ORDER BY timestamp ASC` to `ORDER BY id ASC`.

## AUDIT FINDINGS (Task 2a read_first confirmations)

Source-level evidence verifying the UAT root-cause analysis named fault class (A) and ruling out (B) and (C):

| # | Confirmation | Evidence |
|---|--------------|----------|
| 1 | `id INTEGER PRIMARY KEY AUTOINCREMENT` | `crates/ironhermes-state/src/lib.rs:71` — strictly monotonic per-insert across the messages table. |
| 2 | `unix_now()` is millisecond-resolution f64 | `crates/ironhermes-state/src/lib.rs:806-808` — `Utc::now().timestamp_millis() as f64 / 1000.0`. Same-tick `add_message` calls collide on this f64. |
| 3 | `get_messages` does NOT silently drop on JSON parse | `StoredMessage.tool_calls` is `Option<String>` (raw JSON kept verbatim, lib.rs:143). The query at lib.rs:468 propagates rusqlite errors via `?`; deserialization happens later at the consumer layer (and was never the failure point — class (B) ruled out). |

Class (B) deserialization-drop ruled out: get_messages returns raw JSON in `tool_calls`. Any malformed value would surface upstream during `serde_json::from_str`, not be silently dropped here.

Class (C) cache-coherency ruled out: gateway/Telegram restore path calls `StateStore::get_messages` directly; there is no in-memory cache layered on top of the read.

Therefore class (A) — same-millisecond timestamp ties under `ORDER BY timestamp ASC` — is the only remaining root cause. Fix is the 1-line ORDER BY swap.

## FIX APPLIED

```diff
-    "SELECT id, session_id, role, content, tool_call_id, tool_calls, tool_name, \
-     timestamp, token_count, finish_reason \
-     FROM messages WHERE session_id = ?1 ORDER BY timestamp ASC"
+    "SELECT id, session_id, role, content, tool_call_id, tool_calls, tool_name, \
+     timestamp, token_count, finish_reason \
+     FROM messages WHERE session_id = ?1 ORDER BY id ASC"
```

**Rationale (`id ASC` over `timestamp ASC, id ASC`):**

- `id INTEGER PRIMARY KEY AUTOINCREMENT` (lib.rs:71) is strictly increasing per insert across the entire table.
- The `WHERE session_id = ?1` filter narrows the result set to a single session, so within the result, id-order equals per-session insert order.
- `timestamp ASC, id ASC` would also work and is "safe-in-depth" — but it is a strictly weaker statement of intent and may invite future refactors to "trust the wall clock". `id ASC` alone is sufficient AND explicit about WHAT is being ordered (insert sequence, not wall-clock time).
