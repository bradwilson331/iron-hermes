<objective>
Close GAP-6 from `25.1-UAT.md`: `ironhermes toolset list` always reports the browser row as `11/11 ✓` regardless of chromium availability. Plan 13 fixed `find_chromium_binary` to be authoritative on `BROWSER_PATH`/`CHROMIUM_PATH` (returns `None` when the env var points at a non-existent path), but the fix never reaches the display path because `cmd_toolset_list` builds a `ToolRegistry` via `register_defaults()`, which only registers the 7 non-browser tools — browser tools require `Arc<Config>` (added by plan 14) and are registered separately via `register_browser_tools`. Consequently `registry.list_unavailable()` returns no browser_* names, and `build_toolset_rows` always counts `available_count = 11` for the browser row no matter what `find_chromium_binary` would say.

CRITICAL: `cmd_toolset_show` (toolset_cmd.rs:135-189) has the EXACT SAME bug — it also calls `register_defaults()` (line 142), derives `unavailable_names` from `registry.list_unavailable()` (lines 148-152), and computes `members[].avail = !unavailable_names.contains(tool_name)` (line 157). Without fixing both call sites, `ironhermes toolset list` would report `0/11` while `ironhermes toolset show browser` would still report `11/11 ✓` for the same absent chromium — a user diagnosing chromium availability would get inconsistent answers from two CLI subcommands. The fix MUST cover BOTH paths.

The fix is option-(a) from the UAT GAP-6 `missing` block — the smallest possible surface change. We introduce ONE private helper `browser_chromium_unavailable() -> bool` inside `toolset_cmd.rs` that wraps `ironhermes_tools::browser_session::find_chromium_binary(None).is_none()`, and call it from BOTH `build_toolset_rows` (browser branch) AND `cmd_toolset_show` (when `validated == "browser"`). When the helper returns `true`, the calling code treats every browser_* member as unavailable with prereq label `chromium` for THAT row/show only. The existing per-row `available_count`/`member_summary` machinery and the per-member `(name, avail, prereq_str)` tuple machinery downstream of that decision are reused unchanged.

Purpose: close the gap-2/gap-6 chain so BOTH `toolset list` AND `toolset show browser` are finally honest answers to "do I have browser tools right now?" — a prerequisite for users to debug their setup before reaching for an agent session, and a contract that the two subcommands agree.

Output: surgical patch to `toolset_cmd.rs` adding (1) `browser_chromium_unavailable` helper, (2) browser-branch override in `build_toolset_rows`, (3) browser-branch override in `cmd_toolset_show`, plus two unit tests locking the contract for both paths.
</objective>
