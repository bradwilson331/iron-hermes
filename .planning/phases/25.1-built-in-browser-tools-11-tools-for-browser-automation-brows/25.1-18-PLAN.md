<objective>
Close the gateway-handler residue of GAP-7 left open after plan 25.1-17. Plan 17
landed `validate_tool_call_pairing`, pre-send guards, and `ORDER BY id ASC`,
but never reached the gateway hot path: per D-02, gateway sessions live in
`GatewaySession::messages` (in-memory) and never re-hydrate from SQLite. The
actual orphan-creation site was a Role filter at `handler.rs:642-646` that
dropped every `Role::Tool` message and re-persisted prior-turn assistants —
producing `[sys, user, assistant_tc1, assistant_tc2, ...]` after each
multi-turn browser-tool turn, two consecutive assistants with pending
tool_calls and no answering tool messages between them.

The fix tracks newly-appended messages explicitly in `AgentResult.appended`
inside `agent_loop::run()` at the assistant + tool_result push sites, and
the gateway handler persists `result.appended` directly. This is also
compression-safe: `pre_chat_compress` mutates the input vec, but `appended`
contains exactly the round-trip output of THIS run regardless. Transient
pressure-tier system advisories (`agent_loop.rs:738` and `:576`) are
deliberately not tracked in `appended`, so they don't survive into next
turn's history.

Live verification: 4-turn browser-tool conversation against a real provider
completed cleanly, multi-browser conversations work end-to-end (operator
confirmed). Closes GAP-7 for production.
</objective>
