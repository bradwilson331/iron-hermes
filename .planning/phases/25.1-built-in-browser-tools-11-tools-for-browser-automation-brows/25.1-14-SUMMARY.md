# Phase 25.1 Plan 14: GAP-3 + GAP-4 Config Threading Summary

**One-liner:** Constructor-injected `Arc<Config>` replaces `Config::load()` in both browser tools, restoring D-15 URL allowlist enforcement (T-25.1-01 SSRF) and D-13 yolo+approval gate (T-25.1-02 arbitrary JS) that were silently bypassed by disk-reload anti-pattern.

## Objective Achieved

Closed GAP-3 (D-15 / T-25.1-01 — URL allowlist not enforced) and GAP-4 (D-13 / T-25.1-02 — yolo+approval gate bypassed). Both share one architectural root cause: `BrowserNavigateTool` and `BrowserConsoleTool` called `Config::load()` from disk inside `execute()` and `ensure_session()`, ignoring the runtime Config loaded by `main.rs`.

## Architectural Decision: Constructor Injection vs Execute-Time Reload

The bug was an architectural anti-pattern: tools calling `Config::load()` inside `execute()` means:

1. A different config file path may be loaded than what the CLI used at startup
2. `unwrap_or_default()` silently falls back to `BrowserConfig::default()` (empty `allowed_domains` = allow all; default `autonomous.yolo: false`) when load fails
3. The operator's config is bypassed entirely — security controls become inert

Constructor injection with `Arc<Config>` fixes all three:
1. The exact same Config instance loaded by `main.rs` reaches the tool — no path divergence
2. No fallback path — the operator's values are always used
3. The new required parameter is a compile-time invariant — any future call site that forgets it fails to build

`ensure_session()` was refactored to take `&BrowserConfig` as a parameter (rather than converting to a method) — this produces a smaller diff and avoids a lifetime complexity around `MutexGuard` and `&self`.

## Note for Future Integration Test Authors

The new `Arc<Config>` parameter shape for all browser tool constructors:

```rust
BrowserNavigateTool::new(session: Arc<Mutex<Option<BrowserSession>>>, config: Arc<Config>)
BrowserConsoleTool::new(session: Arc<Mutex<Option<BrowserSession>>>, config: Arc<Config>)
register_browser_tools_with_vision(session, resolver, vision_client, config: Arc<Config>)
```

For integration tests needing specific allowed_domains or yolo values:
```rust
let mut config = Config::default();
config.browser.allowed_domains = vec!["example.com".to_string()];
let tool = BrowserNavigateTool::new(session, Arc::new(config));
```
