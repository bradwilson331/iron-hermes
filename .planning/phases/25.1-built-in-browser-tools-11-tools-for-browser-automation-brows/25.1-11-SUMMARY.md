## D-20 Integration Tests (test file: `crates/ironhermes-tools/tests/browser_integration.rs`)

1. **`browser_navigate_then_snapshot_returns_refs`** (D-10 end-to-end)
   - Subprocess runs `ironhermes` with `IRONHERMES_HOME=<tempdir>` and a wiremock server serving HTML containing `<button>Submit</button>`
   - Asserts response contains `[N] button "Submit"` line — D-10 ref-line format

2. **`browser_click_with_stale_ref_returns_structured_error`** (D-11 stale ref envelope)
   - navigate → snapshot → navigate again (different page) → click(stale_ref)
   - Asserts response envelope contains `"element_stale"` per D-11

3. **`browser_vision_routes_to_auxiliary_vision_role`** (D-07 aux-server routing)
   - Two wiremock servers (main + aux) configured via `auxiliary.vision: { provider: openai, model: gpt-4o-mini }`
   - Mirrors `provider_integration.rs::auxiliary_routes_to_separate_model` ProviderResolver setup
   - Asserts the multimodal request hits the **aux server**, not main — closes Phase 26's SC-2 "vision wired" gap retroactively

All three tests use a `chromium_available()` guard that prints `"SKIP: ..."` and returns early when the Chromium binary is absent (D-22 skip-not-fail; mirrors `web_search` `FIRECRAWL_API_KEY` pattern).

## Plan 09 Deviation Closure (OQ-5)

Plan 09 left `NoOpVisionHandle` in the production registry-construction path because adding
`ironhermes-agent` as a dependency of `ironhermes-tools` would create a cycle. The
intended resolution (handle-trait pattern, Phase 20 MemoryManagerHandle precedent) is
now landed:

- The `VisionClientHandle` trait is defined in `ironhermes-tools::browser_vision`.
- The real implementation `AnyClientVisionHandle` lives in `ironhermes-agent::any_client`
  and depends only on `ironhermes-tools` for the trait.
- `ironhermes-tools` exposes a new `register_browser_tools_with_vision(Arc<dyn VisionClientHandle>)`
  method on `ToolRegistry`. The bare `register_browser_tools` (which still uses
  `NoOpVisionHandle`) is retained for tests that don't need vision routing.
- All three CLI entry points (`run_single`, `run_chat`, `run_gateway`) construct an
  `AnyClientVisionHandle` from the resolver and pass it through. `NoOpVisionHandle` no
  longer reaches production code paths.
- The Plan 09 invariant test `inv_25_1_browser_session_wired_in_all_entry_points` is
  updated to count `register_browser_tools_with_vision` calls.
