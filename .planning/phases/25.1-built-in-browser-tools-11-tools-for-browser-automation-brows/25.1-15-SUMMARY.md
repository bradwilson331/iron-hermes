### Diagnosis Conclusion

| Hypothesis | Description | Status |
|---|---|---|
| H1: Description jargon | "free chromium resources" / "lazy-respawn" — no user-facing verbs | **CONFIRMED root cause** |
| H2: Schema malformation | `properties: {}` silently stripped by provider | **FALSIFIED** — schema is present and well-formed |
| H3: Filter divergence | `is_available` / `toolset` / `prerequisites` diverge | **FALSIFIED** — identical filter signals |

The schema for `browser_close` reaches the LLM intact. The LLM simply cannot map "close the browser" → `browser_close` because the description phrase "Close the browser session and free chromium resources. The next browser_* call will lazy-respawn a new session." contains no synonym verbs (quit, exit, shut down) and buries "session" before "close", reducing relevance for the model's tool-selection heuristic.

## Task 2: Fix — Description Rewrite

### Change Applied (`crates/ironhermes-tools/src/browser_close.rs`)

**Before:**
```rust
fn description(&self) -> &str {
    "Close the browser session and free chromium resources. \
     The next browser_* call will lazy-respawn a new session."
}
```

**After:**
```rust
fn description(&self) -> &str {
    "Close the browser. Shuts down the chromium browser session and \
     frees its resources. Use this when the user asks to close, quit, exit, \
     or shut down the browser. The next browser_navigate or other browser_* \
     call will automatically respawn a fresh session."
}
```

**Rationale:**
- Starts with "Close the browser." — the exact phrase users type
- Explicitly enumerates synonym verbs: close, quit, exit, shut down
- Retains technical accuracy (chromium session teardown, lazy respawn)
- Both "close" and "browser" appear prominently in the first sentence

### Regression Test Added

```rust
#[test]
fn description_uses_explicit_close_verb_for_llm_mapping() {
    let t = BrowserCloseTool::new(dummy_session());
    let desc = t.description().to_lowercase();
    assert!(desc.contains("close"), "description MUST contain 'close': {:?}", t.description());
    assert!(desc.contains("browser"), "description MUST contain 'browser': {:?}", t.description());
}
```

Guards against future regressions where someone rewrites the description back to jargon-only language.

## Architectural Lesson

**Tool descriptions are part of the LLM-visible API surface and need natural-language affordances, not just technical accuracy.**

A technically correct description ("Close the browser session and free chromium resources") that omits the user-facing verbs the model sees in chat ("close", "quit", "exit") is functionally invisible to the model. The model's tool-selection heuristic matches natural-language intent against the description string — if the description only uses noun-heavy jargon, the match fails and the model concludes no capability exists, even though the tool is registered and available.

This is the same problem as writing an API that works but has no documentation users can search for. The fix is always the same: lead with user-facing language, enumerate synonyms, keep the technical details in a supporting clause.
