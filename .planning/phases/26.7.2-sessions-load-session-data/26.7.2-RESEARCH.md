## Summary

Phase 26.7.2 wires session history into the Chat screen. There are three
parallel deliverables: (1) a new `GET /api/session-messages` server fn that
slices the last 50 messages from `StateStore::get_messages`; (2) a
`use_effect` in `ScreenChat` that fires on every `session_id` change to
clear and repopulate `Signal<Vec<ChatBubble>>`; and (3) enrichment of
`SessionInfo` with a `last_message: Option<String>` field and client-side
relative timestamp formatting in `SessionRow`.

All three deliverables are confined to `api.rs`, `chat.rs`, and `sessions.rs`
— no other files change. The server data layer (`ironhermes-state`) is
read-only: `StateStore::get_messages(session_id)` returns `Vec<StoredMessage>`
(all messages, id ASC, no limit) and the server fn slices the tail. The
existing `stored_to_chat_message` private fn in `state.rs` shows exactly how
to convert role strings to `ChatBubble` kinds.

The biggest implementation risk is signal borrow discipline in the D-06
`use_effect`: the `session_id` borrow must be cloned into an owned `String`
before `spawn(async move { ... })`, and the spawned future must not capture
any live signal borrows. The pattern is established in both `agents.rs`
(`use_effect` with synchronous `.restart()`) and `mod.rs` (`spawn(async move
{ create_session().await })` inside `use_effect`).

**Primary recommendation:** Use query-param style (`/api/session-messages?id=`),
`use_resource` (not `use_server_future`) for the fetch primitive, define a
new `ChatMessage` struct in `api.rs` with `role: String` and `content: String`
fields, and implement relative timestamps as a `#[cfg(target_arch = "wasm32")]`
inline helper using `js_sys::Date::now()`.

## Key Research Findings (answering the 8 specific questions)

### Q1: Dioxus 0.7 server fn routing — query params vs path params

[VERIFIED: codebase grep of api.rs]

Every `#[get]` route in `api.rs` uses **query param** style. The function
parameter is automatically extracted from the query string by the
dioxus-fullstack macro. Examples:

```rust
// Existing pattern — parameter extracted from ?id=sub_xxx query string
#[get("/api/agents/status")]
pub async fn api_agents_status(id: String) -> Result<serde_json::Value> { ... }
```

There are **no path-parameterized routes** (e.g., `#[get("/api/foo/{id}")]`)
anywhere in `api.rs`. Path parameter syntax is a Dioxus fullstack macro
feature but is not used in this codebase; the established convention is
query params. Use:

```rust
#[get("/api/session-messages")]
pub async fn get_session_messages(id: String) -> Result<Vec<ChatMessage>> { ... }
```

Called client-side as `get_session_messages(sid.clone()).await`.

### Q2: use_resource vs use_server_future for the history fetch

[VERIFIED: agents.rs UAT-2 hotfix comment lines 34-58]

**Use `use_resource`, not `use_server_future`.**

The agents screen switched from `use_server_future(...)?` to `use_resource`
because the `?` operator on `use_server_future` propagates a `RenderError`
when the resource is in the loading (`None`) state, causing the component to
return early **before** any `use_future` / `use_effect` hooks below it can
register. In Dioxus 0.7, hooks must register in the same order on every
render. The `?` early-return breaks this invariant whenever the resource
re-enters loading state (e.g., after `.restart()`).

However, the history fetch for D-06 is **not** a polling resource that
`.restart()` — it is triggered imperatively from inside `use_effect` via
`spawn(async move { ... })`. The distinction matters:

- For the **session-messages fetch** (called once per `session_id` change
  from inside `use_effect`): use a plain `async move` block inside `spawn`,
  not `use_resource` at all. There is no resource to restart; the fetch is a
  fire-and-populate pattern.
- `use_resource` would be appropriate if you wanted a reactive handle, but
  the `use_effect` + `spawn` approach is simpler and matches the `create_session`
  bootstrap in `mod.rs` lines 137-150.

The `use_server_future` problem does NOT apply here because the D-06
`use_effect` does not use `?` propagation.

### Q3: session_id initialization timing — race analysis

[VERIFIED: mod.rs lines 123, 137-150, 326]

The `session_id` signal is initialized to `"pending"` and asynchronously set
to a real UUID key by the `create_session()` call in a `use_effect` at
`mod.rs:137`. This effect fires **once** on mount.

When a user clicks a session row in `ScreenSessions`, `on_select` calls
`session_id.set(sid)` — this overwrites whatever value `session_id` holds at
that moment (could be `"pending"` if the initial create has not resolved yet,
or the UUID from `create_session` if it has).

**Race analysis for D-06:**

The D-06 `use_effect` in `ScreenChat` subscribes to `session_id` by calling
`session_id()` (call syntax). It fires whenever `session_id` changes. There
are three cases:

1. **Initial mount (`"pending"`):** `session_id` starts as `"pending"`. The
   D-06 effect will fire immediately on mount because `session_id()` is
   evaluated. **Must skip** when `session_id == "pending"` to avoid issuing
   a fetch for a non-existent session.

2. **create_session resolves (sets real UUID):** The `create_session()` effect
   sets `session_id` to a newly-created session key. The D-06 effect will
   fire again. **Must skip** history fetch for a brand-new session — it has
   zero messages and `get_session_messages` will return `[]`. Fetching is
   harmless but wastes a round-trip. The recommended guard is to check
   `session_id.starts_with("agent:main:web")` and skip if the session was
   just created (the `create_session` server fn always creates a fresh
   session, so it will have 0 messages). A simpler guard: always fetch, just
   skip populating if the returned `Vec` is empty.

3. **User selects an existing session:** `on_select` sets a real session id.
   D-06 fires, fetches history, populates bubbles. This is the intended path.

**Recommended guard pattern:**

```rust
use_effect(move || {
    let sid = session_id_ctx();          // subscribe; owned String copy
    if sid == "pending" {
        return;                          // not yet initialized — skip
    }
    // ... clear bubbles, spawn fetch
});
```

The `create_session` case (new UUID, 0 messages) is handled gracefully:
the fetch will return an empty `Vec<ChatMessage>`, nothing is prepopulated,
and no divider is inserted. D-07 (empty = ok) is satisfied.

### Q4: ChatMessage vs StoredMessage — what to return from the server fn

[VERIFIED: state.rs stored_to_chat_message lines 539-560; lib.rs StoredMessage struct]

The `StoredMessage` type lives in `ironhermes-state` which is a
`cfg(not(target_arch = "wasm32"))` server-only dep. It cannot be used on the
WASM client side.

`ChatBubble` is defined in `chat.rs` (client-side component file) and is also
not suitable as a wire type.

**Define a new `ChatMessage` struct in `api.rs`** (alongside the existing
`SessionInfo`, `AgentInfo`, etc.) that derives `Serialize + Deserialize`:

```rust
#[derive(Serialize, Deserialize, Clone, Debug, PartialEq)]
pub struct ChatMessage {
    pub role: String,    // "user" | "assistant" | "tool" | "system"
    pub content: String, // empty string when content is None in StoredMessage
}
```

The server fn maps `StoredMessage → ChatMessage`. The client maps
`ChatMessage → ChatBubble`.

**Role → ChatBubble mapping:**

From `stored_to_chat_message` in `state.rs` (lines 540-546), the role strings
are: `"system"`, `"user"`, `"assistant"`, `"tool"`. Only two map to visible
bubbles:

| `StoredMessage.role` | `ChatBubble` kind | Notes |
|---------------------|-------------------|-------|
| `"user"` | `ChatBubbleKind::User` | Show |
| `"assistant"` | `ChatBubbleKind::Assistant` | Show |
| `"tool"` | skip | Tool result rows are not renderable as standalone bubbles |
| `"system"` | skip | System prompt — not shown in chat history |

The server fn should filter to only `"user"` and `"assistant"` messages before
slicing to `HISTORY_LIMIT`, since tool and system messages are not renderable
as `ChatBubble` items. Otherwise the "last 50" cap would be consumed by
invisible messages.

**content field:** `StoredMessage.content` is `Option<String>`. For
`"assistant"` messages with tool calls, `content` may be `None` (the response
was a tool call with no text). Map `None` to empty string `""`. The client
can skip rendering empty-content bubbles if desired, but this is a display
detail for the planner.

**tool_calls are NOT rendered:** The `StoredMessage.tool_calls` field (a
JSON-encoded array) is intentionally excluded from `ChatMessage`. The
historical view shows text only, consistent with D-01 (same bubble components,
no special treatment of tool history).

### Q5: js_sys::Date for relative timestamps — availability and pattern

[VERIFIED: Cargo.toml line 40; state.rs lines 292-306; app_footer.rs lines 83-98]

`js-sys = "0.3"` is already in `Cargo.toml` as an **unconditional** dependency
(not gated behind `#[cfg(not(target_arch = "wasm32"))]`). It is available on
both WASM and native builds.

The established wasm-gating pattern (from `state.rs:292-306` and
`app_footer.rs:83-98`):

```rust
#[cfg(target_arch = "wasm32")]
fn format_relative_time(unix_secs: f64) -> String {
    let now_ms = js_sys::Date::now();          // f64 milliseconds since epoch
    let then_ms = unix_secs * 1000.0;          // started_at is SECONDS, multiply by 1000
    let diff_secs = ((now_ms - then_ms) / 1000.0) as i64;
    match diff_secs {
        s if s < 60      => "just now".to_string(),
        s if s < 3600    => format!("{} min ago", s / 60),
        s if s < 86400   => format!("{} hr ago", s / 3600),
        s if s < 604800  => format!("{} days ago", s / 86400),
        _                => format!("{} wk ago", diff_secs / 604800),
    }
}

#[cfg(not(target_arch = "wasm32"))]
fn format_relative_time(_unix_secs: f64) -> String {
    "—".to_string()   // native/server build stub
}
```

Note: `js_sys::Date::now()` returns milliseconds (matches `Date.now()` in JS).
`started_at` is in **seconds** (see Q7 below). Multiply `started_at * 1000.0`
before subtracting.

There is **no need** to call `js_sys::Date::new(&JsValue::from_f64(...))` —
`js_sys::Date::now()` is sufficient since we only need current time and the
stored timestamp arithmetic.

### Q6: last_message preview — role filtering and content handling

[VERIFIED: lib.rs StoredMessage struct; state.rs role_str fn lines 881-888]

The CONTEXT.md specifics say: "last assistant message where possible, falling
back to last message of any role if the session ends on a user turn."

The `get_messages` call returns ALL messages ordered by `id ASC`. To find the
best preview message, iterate in reverse and pick:

1. First `"assistant"` message found (scanning from end) — preferred
2. If no assistant message, first `"user"` message found
3. Skip `"tool"` and `"system"` messages entirely

Content is `Option<String>`. An assistant message with tool calls may have
`content = None` (tool-call-only response). In that case, fall back to a
synthesized string like `"[tool call]"` or skip to the next candidate.

Truncate to 80 chars server-side with `chars().take(80).collect::<String>()`.

```rust
fn extract_last_message(msgs: &[StoredMessage]) -> Option<String> {
    // Prefer last assistant message, fall back to last user message.
    let preview = msgs.iter().rev()
        .find(|m| m.role == "assistant" && m.content.is_some())
        .or_else(|| msgs.iter().rev().find(|m| m.role == "user" && m.content.is_some()));
    preview.and_then(|m| m.content.as_ref()).map(|c| {
        let s: String = c.chars().take(80).collect();
        s
    })
}
```

This runs server-side inside `list_sessions()`, which already fetches sessions
from the store. It requires an additional `get_messages(session.id)` call per
session, which adds N extra SQLite reads for N sessions (up to 100). This is
acceptable for the initial implementation but should be noted as a potential
optimization target.

### Q7: started_at — seconds or milliseconds

[VERIFIED: ironhermes-state/src/lib.rs line 877-878]

```rust
fn unix_now() -> f64 {
    Utc::now().timestamp_millis() as f64 / 1000.0   // SECONDS (with sub-second precision)
}
```

`started_at` is stored as **fractional seconds** (e.g., `1716124800.432`).

In `api.rs::list_sessions`, the existing mapping confirms:
```rust
created_at: format!("{}", session.started_at as i64),  // cast to integer seconds
```

For `js_sys::Date`, you must multiply by 1000 to get milliseconds:
```rust
let then_ms = unix_secs * 1000.0;
```

**Do NOT** use `js_sys::Date::new(&JsValue::from_f64(started_at * 1000.0))`
unless you need a `Date` object for formatting. `js_sys::Date::now() - (started_at * 1000.0)`
gives the elapsed milliseconds directly.

### Q8: Signal borrow safety in the D-06 use_effect

[VERIFIED: mod.rs lines 137-150, agents.rs lines 318-325, CLAUDE.md signal borrow rules]

The pattern for async work inside `use_effect` is:

```rust
use_effect(move || {
    let sid = session_id_ctx();          // call-syntax: subscribes + returns owned String
    // No .read() borrow held past this point.
    if sid == "pending" { return; }

    // Clear bubbles synchronously (write borrow scoped to this block).
    bubbles.write().clear();
    streaming_id.set(None);             // cancel any in-flight streaming indicator

    // Spawn async fetch — no signal borrows inside the future.
    spawn(async move {
        match crate::server::api::get_session_messages(sid).await {
            Ok(msgs) => {
                // Populate bubbles.
                let mut bs = bubbles.write();
                let mut id_counter = *next_id.read();
                for msg in msgs {
                    // map ChatMessage → ChatBubble
                    ...
                    id_counter += 1;
                }
                next_id.set(id_counter);
                // Insert divider bubble if history was non-empty.
            }
            Err(_) => { /* D-07: silent failure = empty stream */ }
        }
    });
});
```

Key constraints:
- `session_id_ctx()` (call syntax, not `.read()`) returns an owned `String` —
  no `GenerationalRef` borrow is held. This is the safe pattern per
  `agents.rs:319` (`let cur = subagent_events();`).
- `bubbles.write().clear()` completes synchronously; the `WriteLock` is
  dropped at the `;` before `spawn`.
- Inside `spawn(async move { ... })`, signal writes via `.write()` are safe
  because each `.write()` call acquires and releases the lock atomically —
  no single `WriteLock` is held across an `.await` point.
- `next_id` inside the async block: read and set in two separate calls
  (`*next_id.read()` → copy → `next_id.set(...)`) — no borrow spans an await.

The `use_effect` body itself is **not async** (Dioxus 0.7 constraint confirmed
by `agents.rs` pattern). Async work must go inside `spawn(...)`.

### Pattern 1: Server fn with query param (established)

```rust
// Source: api.rs lines 343-349 (api_agents_status)
#[get("/api/session-messages")]
pub async fn get_session_messages(id: String) -> Result<Vec<ChatMessage>> {
    const HISTORY_LIMIT: usize = 50;
    let state = crate::server::state::global_app_state();
    let all_msgs = state
        .state_store
        .lock()
        .unwrap()
        .get_messages(&id)
        .map_err(|e| ServerFnError::new(format!("get_messages failed: {e}")))?;

    // Filter to renderable roles only, take last HISTORY_LIMIT.
    let out: Vec<ChatMessage> = all_msgs
        .iter()
        .filter(|m| m.role == "user" || m.role == "assistant")
        .rev()
        .take(HISTORY_LIMIT)
        .rev()
        .map(|m| ChatMessage {
            role: m.role.clone(),
            content: m.content.clone().unwrap_or_default(),
        })
        .collect();

    Ok(out)
}
```

### Pattern 4: Relative timestamp (D-05)

```rust
// In sessions.rs SessionRow RSX — replaces the raw created_at display.
// Pattern established by state.rs:now_time() and app_footer.rs:now_time_utc()

#[cfg(target_arch = "wasm32")]
fn format_relative(unix_secs_str: &str) -> String {
    let unix_secs: f64 = unix_secs_str.parse().unwrap_or(0.0);
    let now_ms = js_sys::Date::now();
    let then_ms = unix_secs * 1000.0;
    let diff_secs = ((now_ms - then_ms) / 1000.0) as i64;
    match diff_secs {
        s if s < 5       => "just now".to_string(),
        s if s < 60      => format!("{}s ago", s),
        s if s < 3600    => format!("{}m ago", s / 60),
        s if s < 86400   => format!("{}h ago", s / 3600),
        s if s < 604800  => format!("{}d ago", s / 86400),
        _                => format!("{}w ago", diff_secs / 604800),
    }
}

#[cfg(not(target_arch = "wasm32"))]
fn format_relative(_: &str) -> String { "—".to_string() }
```

Note: `js_sys` is already imported for wasm (unconditional dep). No
`use wasm_bindgen::JsValue;` needed for `Date::now()`.

### Anti-Patterns to Avoid

- **Holding `.read()` across spawn:** `session_id.read().clone()` must be
  called and dropped before `spawn`. Do NOT write `spawn(async move { ... session_id.read() ... .await ... })`.
- **Using `use_server_future(...)?` with hooks below it:** If the history
  resource were wired as `use_server_future`, `?` would cause early return on
  loading, breaking hook ordering for the `use_effect`. Use `spawn` inside
  `use_effect` instead.
- **`StoredMessage` in the wire type:** `ironhermes-state` is a server-only
  dep. Define `ChatMessage` in `api.rs` so it compiles on both sides.
- **Including tool/system messages in the wire response:** They are not
  renderable as `ChatBubble` and would waste the 50-message cap.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Query param extraction | Custom URL parsing | `#[get]` macro param | Dioxus fullstack handles this automatically |
| Relative time library | Moment-style crate | Inline `js_sys::Date::now()` arithmetic | Zero new deps; logic is 10 lines |
| Async in use_effect | Make use_effect async | `spawn(async move { ... })` inside sync effect | Dioxus 0.7 design: effects are sync |
| Signal access in async | `Arc<Mutex<Signal>>` | Clone before spawn, write via `.write()` in spawn | Dioxus signals are arena-allocated, not Arc'd |

## Common Pitfalls

### Pitfall 1: Borrow of session_id held across spawn
**What goes wrong:** `let guard = session_id_ctx.read(); spawn(async move { use guard })` — `GenerationalRef` is not `Send`, compile error.
**Why it happens:** `.read()` returns a `GenerationalRef` tied to arena lifetime.
**How to avoid:** `let sid = session_id_ctx();` (call syntax returns owned `String`). Drop before `spawn`.
**Warning signs:** Compiler error "does not implement Send" or clippy warning about borrow across await.

### Pitfall 2: Infinite re-render loop from use_effect clearing bubbles
**What goes wrong:** `use_effect` calls `bubbles.write().clear()`, which marks `bubbles` dirty, which triggers a re-render, which re-runs `use_effect`, which clears again — infinite loop.
**Why it happens:** `use_effect` in Dioxus 0.7 re-runs when any signal it reads changes. If the effect both reads AND writes the same signal, it loops.
**How to avoid:** The D-06 `use_effect` subscribes to `session_id`, NOT to `bubbles`. Writing to `bubbles` inside the effect does not re-trigger it (the effect only subscribes to signals it reads via `signal()` call syntax). Confirmed safe by inspecting `mod.rs` pattern: the receive loop writes `bubbles` without re-triggering `use_effect`.
**Warning signs:** Browser freezes, CPU pegged, rapid `[26.7.x]` log spam in DevTools.

### Pitfall 3: Divider inserted for brand-new sessions with 0 history
**What goes wrong:** A divider bubble appears even when `msgs.is_empty()`, creating a visual artifact above the empty chat stream.
**Why it happens:** Divider insertion logic not guarded on `!msgs.is_empty()`.
**How to avoid:** Only push the divider when `!msgs.is_empty()`. Check before populating.

### Pitfall 4: started_at units in relative timestamp
**What goes wrong:** Times show "3 million seconds ago" instead of "2 days ago".
**Why it happens:** `js_sys::Date::now()` returns **milliseconds**; `started_at` is **seconds**. Direct subtraction without `* 1000.0` produces wrong diff.
**How to avoid:** Always convert: `then_ms = unix_secs * 1000.0`. Divide the ms diff by `1000.0` to get seconds for the match branches.

### Pitfall 5: get_messages for last_message in list_sessions is N+1
**What goes wrong:** `list_sessions` returns up to 100 sessions; adding a `get_messages` call per session = 100 extra SQLite reads per page load.
**Why it happens:** There is no `last_message` column in the sessions table — it must be derived.
**How to avoid:** Acceptable for this phase (50-100 sessions, fast local SQLite). Document as a known performance trade-off. If profiling shows it's slow, a future phase can add a `last_message_preview TEXT` column updated by a trigger.

### Pitfall 6: create_session race — history fetch for "pending"
**What goes wrong:** D-06 effect fires when `session_id == "pending"`, issues a fetch with `id = "pending"`, which either 404s or returns `[]`, creating a spurious network call and potentially clearing bubbles unnecessarily.
**How to avoid:** Guard: `if sid == "pending" { return; }` as the first statement in the `use_effect` body.

## Open Questions (RESOLVED)

1. **`ChatBubbleKind::Divider` match exhaustiveness**
   - What we know: `ChatBubbleKind` is matched in `chat.rs` in one place (the RSX bubble loop).
   - What's unclear: Are there any other `match b.kind { ... }` sites that will become non-exhaustive after adding `Divider`?
   - Recommendation: Planner must grep for `ChatBubbleKind::` before adding the variant. Confirmed: chat.rs has 2 match sites (class selector + avatar selector). Both need a `Divider` arm.

2. **`last_message` display in SessionRow**
   - What we know: CONTEXT.md says Claude's discretion on whether to show role prefix.
   - What's unclear: Whether to show "You: ..." / "Hermes: ..." prefix or just the raw content.
   - Recommendation: Strip the prefix for brevity. The sub-text in `SessionRow` already shows message count; the preview should be the content only.

3. **`streaming_id.set(None)` on session switch**
   - What we know: `streaming_id` tracks the currently-streaming bubble. If a session switch happens mid-stream, `streaming_id` would still hold the old bubble's id.
   - What's unclear: Should the D-06 effect also call `streaming_id.set(None)` to cancel the visual streaming indicator?
   - Recommendation: Yes — call `streaming_id.set(None)` in the synchronous part of the effect (before spawn) to reset the STREAMING indicator in the header bar. This prevents stale "STREAMING" status after switching sessions.

## Security Domain

This phase touches no authentication, session management, or cryptography.
The history endpoint accepts a `session_id: String` query param with no auth
gate — consistent with all other server fns in `api.rs` which also have no
per-request auth (the server fn surface is local-only per CLAUDE.md "No
external services").

The only security-relevant consideration is that `session_id` is passed
directly to `state_store.get_messages(id)` via a rusqlite parameterized query
(`params![session_id]`). SQL injection is not possible by construction (same
pattern as all existing `state_store` calls).
