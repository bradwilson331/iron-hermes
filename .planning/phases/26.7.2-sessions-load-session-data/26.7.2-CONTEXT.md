# Phase 26.7.2: Sessions load session data - Context

**Gathered:** 2026-05-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 26.7.2 makes clicking a session row in ScreenSessions load that session's conversation history into the Chat screen. Today, `on_select` sets `SessionIdContext` and navigates to `Screen::Chat` — but `Signal<Vec<ChatBubble>>` starts empty regardless of which session was selected. This phase adds a `GET /api/sessions/{id}/messages` server fn and wires ScreenChat to populate its bubble stream with the last N messages when the session ID changes.

**In scope — 2 deliverables:**
1. **Server fn** — `GET /api/sessions/{id}/messages` returning `Vec<ChatMessage>` (last 50 messages, capped server-side).
2. **Chat screen wiring** — `use_effect` in ScreenChat watches `session_id` changes: clears `Signal<Vec<ChatBubble>>`, fetches history, prepopulates with loaded messages + inserts a divider bubble at the boundary.
3. **Session row enrichment** — `SessionInfo` gains `last_message: Option<String>` (last message preview, truncated to ~80 chars server-side) and client-side relative timestamp formatting ("2 days ago") replacing the raw Unix timestamp string.

**Out of scope (deferred):**
- Session deletion server fn (client-side optimistic hide still the pattern — Phase 26.2.1 deferred item)
- Pagination / infinite scroll of history beyond the N=50 cap
- Session search / filtering
- `+ NEW SESSION` button wiring (already stubbed, own phase)

</domain>

<decisions>
## Implementation Decisions

### History display in Chat
- **D-01:** Historical messages render using the **same `ChatBubble` components** as live messages (`chat-msg.user` / `chat-msg.assistant`). No visual differentiation for individual historical bubbles.
- **D-02:** A **divider row** is inserted between the loaded history and the start of new live input. The divider is a thin rule or label element (e.g. `"─── Earlier ───"`) rendered as a special `ChatBubble` variant or inline RSX element in ScreenChat's bubble list. Exact visual treatment is Claude's discretion — matches the existing design system's separator conventions.
- **D-03:** History clears on `/clear` command — same as live bubbles. Historical messages are prepopulated into `Signal<Vec<ChatBubble>>`; `/clear` wipes the whole signal (no distinction between historical and live bubbles after population).

### Session row enrichment
- **D-04:** `SessionInfo` gains a `last_message: Option<String>` field — the last message text, truncated to ~80 characters server-side in `list_sessions()`. Shown in `SessionRow` sub-text replacing or alongside the existing `message_count` sub-copy.
- **D-05:** `created_at` is formatted client-side as **relative time** ("2 days ago", "just now") in the `SessionRow` RSX. The raw Unix timestamp string from `SessionInfo.created_at` is parsed via JS `Date` in the wasm target. Exact implementation (direct `js_sys::Date`, a small helper fn, or inline `web_sys` call) is Claude's discretion.

### Load trigger + navigation
- **D-06:** History loading is triggered by a **`use_effect` on `session_id` in ScreenChat**. When `session_id` changes: (1) clear `Signal<Vec<ChatBubble>>`, (2) spawn a fetch to the new sessions messages endpoint, (3) populate the signal with loaded bubbles. Navigation to Chat happens first; loading is async after.
- **D-07:** While history is loading (after `session_id` changes, before messages arrive), the `.chat-stream` shows **empty** — same as a fresh session. No loading indicator. Consistent with 26.7 D-13 (no custom loading UI).
- **D-08:** A `session_id` change **always clears and reloads** — whether switching to an existing session, a new session, or after `/clear`. The `use_effect` fires on every `session_id()` change; clearing the bubble signal first ensures no stale history from the previous session leaks.

### Continuation semantics
- **D-09:** Loaded sessions are **fully continuable**. The chat-input-pill stays active; sending a new message after loading history continues the session server-side. `state.rs::build_messages_for_turn` already uses `session_id` to fetch persisted messages — the chat history visible in the UI mirrors what the server will prepend to the next turn's context.
- **D-10:** The history endpoint loads the **last 50 messages** (capped as a constant in the server fn, e.g. `const HISTORY_LIMIT: usize = 50`). This is a UI display cap only — the agent's `build_messages_for_turn` has its own context-window management independent of this display limit.

### Claude's Discretion
- Exact divider element shape (thin `<hr>`, a `div.chat-divider` with a center label, or a ChatBubble variant with `is-divider` class). Prefer CSS-only styling; matches existing `.section-label` or `.pill` design token conventions.
- Whether `last_message` shows the role prefix ("You: …" / "Hermes: …") or strips it. Check `StateStore::get_messages` return type to see if role is available.
- Field name for the new server fn param — `/api/sessions/{id}/messages` or `/api/session-messages?id={id}` per Dioxus `#[get]` routing conventions (check existing parameterized routes in api.rs).
- Whether relative timestamp formatting uses `js_sys::Date` directly or a small helper in a `utils.rs` module.

</decisions>

## Existing Code Insights

### Reusable Assets
- `Signal<Vec<ChatBubble>>` at `hermes_app/mod.rs:123` — already provided via context; ScreenChat reads it. The D-06 `use_effect` writes to this same signal to prepopulate history.
- `SessionIdContext` (B-03 newtype) at `hermes_app/mod.rs:326` — already consumed by ScreenChat at `chat.rs:126`. The `use_effect` subscribes to this signal.
- `use_server_future` + `.restart()` — established in Phase 26.7.1 Agents screen; same pattern for the history fetch.
- `gloo_timers::future::TimeoutFuture` — already used in agents.rs; available for any loading delays if needed.

### Established Patterns
- **Signal-borrow discipline**: `GenerationalRef` must drop before any `.await`. The `use_effect` must collect `session_id.read().clone()` into an owned `String` and drop the borrow before spawning the async fetch.
- **`use_effect` for cross-signal reactivity**: `use_effect(move || { let sid = session_id_ctx(); ... })` — calling the signal with `()` subscribes to changes. This is the D-06 idiom (same as agents.rs:272-276 for `subagent_events`).
- **`spawn` for async inside `use_effect`**: `use_effect` bodies cannot be `async`; async work must be wrapped in `spawn(async move { ... })`.
- **`#[get]` parameterized routes**: Check `api.rs` for any existing parameterized `#[get]` routes for the D-10 endpoint URL form.

### Integration Points
- `hermes_app/mod.rs` → `ScreenChat` → `Signal<Vec<ChatBubble>>`: prepopulate here for D-06/D-08
- `api.rs::list_sessions()` → `SessionInfo`: add `last_message` field here for D-04
- `state.rs::StateStore::get_messages(session_id)` → new server fn reads this for D-10 history fetch
- `sessions.rs::SessionRow` → sub-text `span.row-sub`: update for D-04 preview + D-05 relative timestamp

<specifics>
## Specific Ideas

- The divider label should feel like the existing `section-label` convention used in sessions.rs (`div.section-label` with `"Sessions · {count}"`). A similar element with `"─── Earlier ───"` or `"// HISTORY"` fits the terminal-style design language.
- The `HISTORY_LIMIT = 50` constant is a starting point; researcher should verify the existing `StateStore::get_messages` signature to confirm whether it already supports a limit parameter or returns all messages (and we slice client-side or server-side).
- `SessionInfo.last_message` should be the last **assistant** message where possible (most useful preview), falling back to last message of any role if the session ends on a user turn. Researcher to confirm from `get_messages` return shape.

</specifics>
