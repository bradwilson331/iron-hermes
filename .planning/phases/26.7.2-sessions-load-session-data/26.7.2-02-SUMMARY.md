## Decisions Made

- `next_id` is provided via context from HermesApp (deviation Rule 2 — missing critical functionality). Without this, history bubble IDs allocated in ScreenChat's async spawn would collide with IDs allocated by the WS receive loop in HermesApp. The WS loop reads `*next_id.read()` and calls `next_id.set(n+1)` — sharing a single signal via context ensures monotonicity.
- Signal bindings `bubbles`, `streaming_id`, `next_id` declared as `let mut` in ScreenChat. Rust 2021 requires `mut` on the binding to call `.write()` / `.set()` even though Dioxus signals have interior mutability. This is consistent with mod.rs which uses `let mut` for the same signals.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing Critical] Expose next_id via context in HermesApp**
- **Found during:** Task 2 (use_effect history load implementation)
- **Issue:** The plan's reference pattern called `*next_id.read()` inside the spawn, but `next_id` was not provided via context — it only lived as a local signal in HermesApp. ScreenChat had no way to access it, and without it, IDs allocated in the history effect would start from an arbitrary value and collide with live WS receive loop IDs.
- **Fix:** Added `use_context_provider(|| next_id)` in HermesApp mod.rs and `let mut next_id = use_context::<Signal<u64>>()` in ScreenChat, matching the pattern of the 5 existing context providers.
- **Files modified:** `crates/iron_hermes_ui/src/components/hermes_app/mod.rs`, `crates/iron_hermes_ui/src/components/hermes_app/screens/chat.rs`
- **Verification:** cargo check --features web exits 0; next_id correctly shared between HermesApp and ScreenChat
- **Committed in:** `97ef1e25` (Task 2 commit)

## Issues Encountered

- Signal bindings in ScreenChat required `let mut` declarations for `bubbles`, `streaming_id`, `next_id` — Rust compiler rejects calling `.write()` / `.set()` on immutable bindings. Fixed by adding `mut` to all three. This is not a deviation (it is correct Rust) but required a compile-fix iteration.
