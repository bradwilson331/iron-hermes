Create `providers/memory-grafeo/src/lib.rs` implementing MemoryProvider:

```rust
pub struct GrafeoMemoryProvider {
    db: grafeo::Database, // or whatever the actual type is per grafeo docs
    // Internal cache for frozen snapshot pattern
    snapshot: HashMap<MemoryTarget, Vec<String>>,
}
```

The implementation follows the same pattern as SqliteMemoryProvider:
- `new(db_path)`: Opens/creates grafeo database, initializes schema
- `load_from_disk()`: Query all memory entry nodes, group by target, populate snapshot cache
- `add(target, content)`: Security scan. Check capacity. Create MemoryEntry node with content and target properties. Create HAS_ENTRY edge from target root node. Return success JSON.
- `replace(target, old_text, new_content)`: Security scan. Find entry nodes for target where content CONTAINS old_text. Single match -> update node content. Multiple/zero -> error.
- `remove(target, old_text)`: Find matching entry node. Delete node and its edges.
- `format_for_system_prompt(target)`: Read from snapshot cache (frozen). Format with capacity header per D-13.
- `to_memory_entries()`: Return snapshot cache as MemoryEntries.
- Lifecycle hooks (initialize, prefetch, sync_turn, on_session_end, shutdown): Same no-op pattern as other providers.

Per D-02: Metadata keys become edge labels. For the basic implementation, the primary edge is HAS_ENTRY linking a target root node (Memory or User) to each MemoryEntry node. Additional relationship edges (like INTEREST, PREFERENCE) are a future enhancement — the base implementation stores facts as nodes with a single relationship type.

Add comprehensive unit tests with a tempdir-based grafeo database.

If grafeo's API differs significantly from assumptions (e.g., no embedded file-based DB, no node/edge operations as expected), document the actual API in comments and adapt the implementation accordingly. The key requirement is that it implements MemoryProvider correctly — the internal storage mechanism is Claude's discretion per CONTEXT.md.
