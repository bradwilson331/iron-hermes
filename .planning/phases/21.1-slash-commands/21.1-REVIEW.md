# Phase 21.1: Code Review Report

**Reviewed:** 2026-04-17T19:42:00Z
**Depth:** standard
**Files Reviewed:** 8
**Status:** issues_found

## Summary

The Phase 21.1 slash command router implementation is well-structured. The `CommandRouter` with three-stage resolution (exact, alias, prefix), `CommandDef` builder pattern, `PlatformFilter` system, and centralized `build_registry()` are solid architectural choices. The extension-first dispatch chain in the CLI with panic containment is a good defensive pattern.

However, the review identified one critical issue (magic-string coupling for control flow in main.rs), several warnings around the `/stop` command not actually cancelling anything, a silent discard of gateway slash command attachments, and an inconsistency in help output formatting between CLI and core. Test coverage is strong across all reviewed files.

## Critical Issues

### CR-01: Magic-string coupling for Quit/ClearSession control flow

**File:** `crates/ironhermes-cli/src/main.rs:755-760`
**Issue:** The REPL loop detects `Quit` and `ClearSession` semantics by comparing the `CommandResult::Handled(output)` string against hardcoded magic values (`"Goodbye!"`, `"Conversation cleared."`, `"Conversation cleared. Starting fresh."`). If any handler changes its output text -- even just capitalisation or punctuation -- the REPL silently stops exiting or clearing the session. This is a fragile coupling that can produce silent logic errors. The code comments on lines 750-754 acknowledge this limitation but do not mitigate it.
**Fix:** Add `Quit` and `ClearSession` variants to the TUI `CommandResult` enum (in `extension.rs`) and update `map_core_to_tui` to preserve the semantic distinction:
```rust
// In crates/ironhermes-cli/src/tui/extension.rs:
pub enum CommandResult {
    Handled(String),
    Silent,
    Error(String),
    Quit,           // New
    ClearSession,   // New
}

// In crates/ironhermes-cli/src/tui/commands.rs map_core_to_tui():
CoreCommandResult::Quit => CommandResult::Quit,
CoreCommandResult::ClearSession => CommandResult::ClearSession,

// In main.rs REPL:
CommandResult::Quit => { println!("Goodbye!"); break; }
CommandResult::ClearSession => { messages.truncate(1); println!("Conversation cleared."); continue; }
```

## Warnings

### WR-01: `/stop` command reports status but never cancels the agent

**File:** `crates/ironhermes-core/src/commands/handlers.rs:77-83`
**Issue:** `cmd_stop` reads `agent_running` to decide its output message but never sets it to `false` or cancels any `CancellationToken`. On the CLI side (`main.rs`), the `agent_running` flag is set/cleared around the agent turn, and the `CancellationToken` is managed by ctrl-c. The `/stop` command effectively does nothing -- it only prints a status string. A user issuing `/stop` during a gateway turn or future async CLI turn will see "Stopping agent..." but the agent continues running.
**Fix:** Either (a) accept `CancellationToken` in `CommandContext` and cancel it in `cmd_stop`, or (b) rename the output to clearly indicate it is not yet functional (matching the `todo_stub` pattern for unimplemented commands):
```rust
// Option (a): add cancel_token to CommandContext and wire it
fn cmd_stop(ctx: &CommandContext) -> CommandResult {
    if ctx.agent_running.load(Ordering::SeqCst) {
        if let Some(token) = &ctx.cancel_token {
            token.cancel();
        }
        CommandResult::Output("Stopping agent...".to_string())
    } else {
        CommandResult::Output("No agent running.".to_string())
    }
}
```

### WR-02: Gateway slash commands silently discard multimodal attachments

**File:** `crates/ironhermes-gateway/src/handler.rs:309-311`
**Issue:** In `handle_with_multimodal`, when `event.content.starts_with('/')`, the method calls `handle_slash_command` which does not receive the `processed: ProcessedAttachments` parameter. If a user sends a slash command with an attached image (e.g., `/compress` with a photo), the attachment is silently dropped. For commands that fall through to `run_agent` via `PassThrough` or `NotFound`, the attachments are reconstructed as empty `ProcessedAttachments` (lines 265-266, 287-288) instead of forwarding the original.
**Fix:** Pass `processed` through to `handle_slash_command` and forward it to `run_agent` on `PassThrough`/`NotFound` paths:
```rust
async fn handle_slash_command(
    &self,
    event: &MessageEvent,
    adapter: Arc<dyn PlatformAdapter>,
    cancel: CancellationToken,
    processed: ProcessedAttachments, // Add parameter
) -> Result<()> {
    // ... in PassThrough and NotFound branches:
    return self.run_agent(event, adapter, cancel, processed).await;
}
```

### WR-03: `cmd_new` ignores the `args` parameter (session name)

**File:** `crates/ironhermes-core/src/commands/handlers.rs:63-67`
**Issue:** The `/new` command is registered with `args_hint("[name]")` (registry.rs line 18), indicating it accepts an optional session name argument. However, `cmd_new` ignores `_args` entirely and always returns the same hardcoded message. The session name is never used for naming or any purpose.
**Fix:** Either use the args to set a session name in the result, or remove the `args_hint` from the registry to avoid misleading users:
```rust
fn cmd_new(args: &[&str], _ctx: &CommandContext) -> CommandResult {
    let name = if args.is_empty() { None } else { Some(args.join(" ")) };
    CommandResult::NewSession {
        message: match name {
            Some(n) => format!("Starting new session: {}", n),
            None => "Conversation cleared. Starting fresh.".to_string(),
        },
    }
}
```

### WR-04: Help text formatting inconsistency between core handler and CLI adapter

**File:** `crates/ironhermes-core/src/commands/handlers.rs:171-176` vs `crates/ironhermes-cli/src/tui/commands.rs:162-165`
**Issue:** The core `cmd_help` handler (handlers.rs:171) formats commands as `"/{name}"` padded to 14 chars, while the CLI `format_help` (commands.rs:162) formats as `"/{name}"` padded to 13+1 chars (using `/{:<13}`). These produce different column alignments. Additionally, the core handler uses `"TOOLS AND SKILLS"` (handlers.rs:160) while the CLI adapter uses `"TOOLS & SKILLS"` (commands.rs:154). Since gateway users call `cmd_help` via the core handler and CLI users call `format_help` via the adapter, these two paths produce visually inconsistent output for the same `/help` command.
**Fix:** Unify by having `cmd_help` delegate to a shared formatting function, or have the CLI's `format_help` be the canonical implementation used by both paths. At minimum, make the category label and column widths consistent.

### WR-05: `cmd_stop` reads `agent_running` with `SeqCst` ordering but CLI sets it outside the command path

**File:** `crates/ironhermes-core/src/commands/handlers.rs:78` and `crates/ironhermes-cli/src/main.rs:791,855`
**Issue:** On the CLI, `/stop` can only be entered at the readline prompt, which means the agent is never running when the command is dispatched (the REPL is single-threaded: readline blocks, then agent runs, then readline again). The `agent_running` check on line 78 will always be `false` on CLI, making `/stop` always report "No agent running." On the gateway (handler.rs:187), `agent_running` is hardcoded to `false`. This means `/stop` never reports the agent as running on any platform.
**Fix:** For the gateway, wire the actual per-session running state into the `CommandContext`. For the CLI, acknowledge that `/stop` is only meaningful during in-flight turns (ctrl-c handles that) and either route it through the ctrl-c path or mark it as a stub.

## Info

### IN-01: `todo_stub` match arms reference alias names that will never match

**File:** `crates/ironhermes-core/src/commands/handlers.rs:225-226,244-245,249`
**Issue:** The `todo_stub` match includes patterns like `"bg"`, `"set-home"`, `"reload_mcp"`, and `"gateway"`. These are alias names, not command names. The `dispatch()` function matches on `cmd.name` (line 17), so after resolution the `cmd.name` will be the canonical name (e.g., `"background"`, `"sethome"`, `"reload-mcp"`, `"platforms"`). The alias arms are dead code -- they will never match.
**Fix:** Remove the unreachable alias arms from `todo_stub`:
```rust
"background" => "No background session manager",  // remove "bg"
"sethome" => "No home channel concept",            // remove "set-home"
"reload-mcp" => "No MCP reload",                   // remove "reload_mcp"
"platforms" => "No platform status display",        // remove "gateway"
```

### IN-02: `print_help()` function in main.rs is unused in the REPL path

**File:** `crates/ironhermes-cli/src/main.rs:1189-1197`
**Issue:** The standalone `print_help()` function constructs its own `CommandRouter` and calls `format_help`. It is not called anywhere in the `run_chat` loop (which uses `dispatch_command` -> `cmd_help` or `format_help`). It appears to be leftover code from before the router integration, or reserved for a `--help` CLI flag that does not exist.
**Fix:** If unused, remove it to reduce dead code. If intended for future use, add an `#[allow(dead_code)]` annotation with a comment explaining the intended use.

### IN-03: Category label for ToolsAndSkills has two different string representations

**File:** `crates/ironhermes-core/src/commands/handlers.rs:160` and `crates/ironhermes-cli/src/tui/commands.rs:154`
**Issue:** Core uses `"TOOLS AND SKILLS"` while CLI adapter uses `"TOOLS & SKILLS"`. This is a minor copywriting inconsistency.
**Fix:** Pick one canonical form (e.g., `"TOOLS & SKILLS"`) and use it consistently, ideally via a `Display` impl on `CommandCategory`.

### IN-04: `build_registry` called multiple times without caching

**File:** `crates/ironhermes-cli/src/main.rs:665` and `crates/ironhermes-cli/src/main.rs:1192`
**Issue:** `build_command_registry()` (aliased from `build_registry`) is called in `run_chat` to construct the `CommandRouter`, and `print_help()` constructs its own via `build_cli_router()`. The gateway handler also constructs one in `GatewayMessageHandler::new()`. While `build_registry` is cheap (returns a `Vec` of static data), it is called redundantly. This is not a bug but a minor code quality observation.
**Fix:** No action needed for correctness. If consolidation is desired, consider making the registry a `once_cell::sync::Lazy<Vec<CommandDef>>`.
