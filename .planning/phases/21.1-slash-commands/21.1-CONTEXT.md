<decisions>
## Implementation Decisions

### Router location
- **D-01:** Router lives in `ironhermes-core` — the leaf crate accessible from CLI, gateway, agent, and ACP. Command definitions, `resolve_command()`, dispatch table, and `CommandContext` all in core. No circular deps.

### Command set
- **D-02:** Full hermes-agent slash command parity. Implement ALL commands from the hermes-agent Slash Commands Reference. The researcher MUST read the hermes-agent Python source at `/Users/twilson/code/hermes-agent` to understand each command's behavior. Each command must be wired to a working Rust function, or explicitly marked as TODO with a note about what infrastructure is missing.
- **D-03:** Platform tags on commands. Each command declares which platforms it's available on (cli, gateway, acp, all). The router filters by current platform context. Platform tags from hermes-agent reference:
  - **CLI-only:** /skin, /tools, /toolsets, /browser, /config, /cron, /skills, /platforms, /paste, /image, /statusbar, /plugins, /verbose (configurable)
  - **Messaging-only:** /sethome, /update, /restart, /approve, /deny, /commands
  - **Universal (CLI + messaging):** /new, /reset, /status, /stop, /model, /provider, /personality, /fast, /retry, /undo, /compress, /title, /resume, /usage, /insights, /reasoning, /voice, /rollback, /snapshot, /background, /plan, /reload-mcp, /reload, /yolo, /debug, /help, /<skill-name>
  - **CLI + ACP:** /quit, /clear (session control)

### Alias & prefix matching
- **D-04:** Shortest-unique-prefix matching. Any unambiguous prefix resolves: `/h` → `/help`, `/mod` → `/model`, `/per` → `/personality`. Ambiguous prefixes (e.g., `/s` could be /stop, /status, /search, /sessions, /skills, /snapshot) return an error listing candidates.
- **D-05:** Explicit aliases + prefix matching. Commands can declare explicit aliases. Resolution order: exact match → alias match → prefix match. `/snap` resolves to `/snapshot` via alias. `/set-home` resolves to `/sethome` via alias. Matches hermes-agent behavior.
- **D-06:** Built-in commands first, then skills. `resolve_command()` checks built-in commands (exact → alias → prefix) first. If no match, treat as a `/<skill-name>` invocation. Skills are never shadowed by prefix matches.

### Integration pattern
- **D-07:** Replace both existing implementations. Delete hardcoded commands from CLI `dispatch_command` (tui/commands.rs) and gateway `handle_slash_command` (handler.rs). Both platforms call the unified router. CLI's `TuiExtension::process_command()` extension-first dispatch chain stays — extensions get first crack, then the core router.
- **D-08:** Intercept before AgentLoop, in caller. CLI's REPL loop and gateway's message handler check for `/` prefix and call the router BEFORE passing to AgentLoop. Unknown commands (no built-in, no alias, no prefix, no skill match) pass through to the agent as normal messages.
- **D-09:** CommandContext struct. Commands receive a `CommandContext` with references to shared state: config, StateStore, SkillRegistry, session info, platform identifier. Each platform constructs its own CommandContext. Commands check for `Option<T>` on deps that may not be available.

### Claude's Discretion
- Exact `CommandContext` fields and which deps are Optional vs required
- Whether command handlers are functions or trait objects
- Internal module structure within ironhermes-core (e.g., `commands/` submodule)
- How `/model --global` flag parsing works (inline vs separate arg parser)
- Whether running-agent guards (/model blocked while agent active, /stop bypasses) use an AtomicBool or a more structured mechanism
- Error message format for ambiguous prefixes

### Folded Todos
- "Slash command integration (SKILL-13)" (2026-04-17) — Router + command implementation. This is the primary scope of Phase 21.1.

</decisions>

## Existing Code Insights

### Reusable Assets
- `dispatch_command()` (tui/commands.rs) — Extension-first dispatch chain with panic containment via `catch_unwind`. The dispatch PATTERN stays (extension first, core fallback), but the core fallback gets replaced by the unified router.
- `format_help()` (tui/commands.rs) — Help text generator with keybinding registry integration. Will need to pull from the router's command registry instead of hardcoded strings.
- `handle_slash_command()` (gateway/handler.rs) — Gateway command handlers for /stop, /new, /clear, /help. Logic moves to unified router.
- `CommandResult` enum (tui/extension.rs) — `Output(String)`, `Handled`, `Error(String)`, `Quit`, `Clear`. Reusable as the router's return type.
- `SkillRegistry` (ironhermes-core/skills.rs) — For `/<skill-name>` catch-all resolution.

### Established Patterns
- Extension-first dispatch: `TuiExtension::process_command()` checked before core commands (Phase 22.1)
- Platform enum already exists in `types.rs` with 16 variants — use for platform tags
- `Arc<ToolRegistry>` shared across turns — similar pattern for command registry

### Integration Points
- CLI REPL: `/` prefix check in `run_chat` main loop → call router
- Gateway handler: `event.content.starts_with('/')` check in `handle_message` → call router
- ACP (Phase 22.2): Will call same router with `Platform::Local` or a new ACP variant
- `dispatch_command` in tui/commands.rs: Extension chain stays, core fallback replaced by router call

## Specific Ideas

- The hermes-agent reference shows `/model` has complex behavior: provider switching (`/model zai:glm-5`), custom endpoints (`/model custom:model`), named custom providers (`/model custom:local:qwen`), auto-detect (`/model custom`), and `--global` persistence flag. The Rust implementation should match this behavior where the backing infrastructure exists (provider resolution is Phase 12, already complete).
- Some commands may need TODO stubs because their backing infrastructure doesn't exist yet in IronHermes:
  - `/voice` — no TTS/voice infrastructure
  - `/rollback` / `/snapshot` — no filesystem checkpoint system
  - `/background` — no background session spawning
  - `/plan` — no bundled plan skill
  - `/insights` — no analytics/cost tracking
  - `/sethome` — no home channel concept yet
  - `/update`, `/restart` — no self-update/service management
  - The researcher should confirm which of these have backing code and which don't
- Running-agent guards from hermes-agent: some commands are blocked while the agent is active (/model), while others bypass the guard (/stop, /approve, /deny). The router needs a guard mechanism.
- `/compress [focus topic]` can reuse the existing `ContextCompressor` from Phase 18.
