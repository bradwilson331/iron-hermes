## Fixed Issues

### CR-01: Magic-string coupling for Quit/ClearSession control flow

**Files modified:** `crates/ironhermes-cli/src/tui/extension.rs`, `crates/ironhermes-cli/src/tui/commands.rs`, `crates/ironhermes-cli/src/main.rs`
**Commit:** bbb6545
**Applied fix:** Added `Quit` and `ClearSession(String)` variants to the TUI `CommandResult` enum. Updated `map_core_to_tui()` to map `CoreCommandResult::Quit` to `CommandResult::Quit` and `CoreCommandResult::ClearSession`/`NewSession` to `CommandResult::ClearSession(msg)` instead of converting to magic strings. Updated the REPL loop in main.rs to match on the new enum variants directly, eliminating all magic string comparisons. Updated 4 tests to assert on the new variant types.

### WR-01: `/stop` command reports status but never cancels the agent

**Files modified:** `crates/ironhermes-core/src/commands/handlers.rs`
**Commit:** 2d4a7b3
**Applied fix:** Added documentation comments explaining why `/stop` is effectively a stub on both CLI (single-threaded REPL means agent is never running when command is dispatched) and gateway (agent_running not yet wired). Added a TODO comment for wiring CancellationToken. Updated output messages to clearly communicate the limitation: the "running" branch notes that cancellation token is not yet wired, and the "idle" branch suggests using Ctrl-C. Updated tests to use pattern matching on message content rather than exact string equality.

### WR-02: Gateway slash commands silently discard multimodal attachments

**Files modified:** `crates/ironhermes-gateway/src/handler.rs`
**Commit:** d3e7c50
**Applied fix:** Added `processed: ProcessedAttachments` parameter to `handle_slash_command()`. Updated `handle_with_multimodal()` to pass `processed` through to `handle_slash_command()`. Updated `PassThrough` and `NotFound` branches to forward the original `processed` attachments to `run_agent()` instead of constructing empty `ProcessedAttachments`. The text-only `MessageHandler::handle()` path constructs empty attachments as before since no multimodal data exists on that path.

### WR-03: `cmd_new` ignores the `args` parameter (session name)

**Files modified:** `crates/ironhermes-core/src/commands/handlers.rs`
**Commit:** 97b57cb
**Applied fix:** Updated `cmd_new` to use the `args` parameter instead of ignoring it. When args are provided, the session name is included in the `NewSession` message (e.g., "Starting new session: my-project"). When args are empty, the original default message is preserved.

### WR-04: Help text formatting inconsistency between core handler and CLI adapter

**Files modified:** `crates/ironhermes-core/src/commands/handlers.rs`
**Commit:** 8aa268d
**Applied fix:** Aligned the core `cmd_help` handler formatting to match the CLI `format_help` implementation. Changed category label from `"TOOLS AND SKILLS"` to `"TOOLS & SKILLS"`. Changed column formatting from `{:<14}` (with manual `format!("/{}", cmd.name)`) to `/{:<13}` to match the CLI's column width calculation. Added a comment referencing the CLI source for future maintainers.

### WR-05: `cmd_stop` reads `agent_running` with `SeqCst` ordering but CLI sets it outside the command path

**Files modified:** `crates/ironhermes-core/src/commands/handlers.rs`
**Commit:** 2d4a7b3
**Applied fix:** Combined with WR-01 fix. The documentation comments added to `cmd_stop` explicitly describe the single-threaded REPL limitation that causes `agent_running` to always be false on CLI, and the hardcoded false on gateway. The TODO for wiring CancellationToken covers both the cancellation gap (WR-01) and the state visibility gap (WR-05).
