# Phase 21.1: Slash Commands - Research

**Researched:** 2026-04-17
**Domain:** Platform-agnostic slash command router in Rust (ironhermes-core)
**Confidence:** HIGH

## Summary

This phase builds a unified slash command router in `ironhermes-core` that replaces the hardcoded command dispatch in both the CLI (`dispatch_command` / `core_dispatch` in tui/commands.rs) and the gateway (`handle_slash_command` in handler.rs). The router implements full hermes-agent command parity (~38 commands from the Python `COMMAND_REGISTRY` plus `/<skill-name>` catch-all), alias resolution, shortest-unique-prefix matching, platform availability filtering, and a running-agent guard mechanism.

The hermes-agent Python source at `/Users/twilson/code/hermes-agent/hermes_cli/commands.py` provides the authoritative `COMMAND_REGISTRY` with 38 `CommandDef` entries organized into 5 categories (Session, Configuration, Tools & Skills, Info, Exit). Each command declares `cli_only`, `gateway_only`, aliases, args hints, and subcommands. The `resolve_command()` function does exact + alias lookup, while prefix matching is implemented separately in `cli.py:process_command()` at lines 4020-4058. The running-agent guard in the gateway (`gateway/run.py:1735-1852`) allows `/stop`, `/new`, `/queue`, and `/status` to bypass the guard while all other commands are queued as pending messages.

**Primary recommendation:** Build a `CommandRouter` struct in `ironhermes-core::commands` that owns a `Vec<CommandDef>` registry, provides `resolve_command()` with three-stage resolution (exact -> alias -> prefix), and dispatches to `async fn` handlers returning `CommandResult`. Each command declares platform availability via a `PlatformFilter` enum. The router integrates into the existing extension-first dispatch chain: TUI extensions get first crack, then the core router, then `/<skill-name>` catch-all.

### Pattern 1: Command Registry (from hermes-agent COMMAND_REGISTRY)

**What:** A static registry of `CommandDef` structs defining all slash commands, built at startup.
**When to use:** Always -- this is the single source of truth for command definitions.

The hermes-agent Python source defines 38 commands in `COMMAND_REGISTRY` (hermes_cli/commands.py:46-142). Each `CommandDef` has:
- `name`: canonical name without slash (e.g. "background")
- `description`: human-readable description
- `category`: "Session", "Configuration", "Tools & Skills", "Info", "Exit"
- `aliases`: tuple of alternative names (e.g. ("bg",) for background)
- `args_hint`: argument placeholder (e.g. "<prompt>", "[name]")
- `cli_only`: bool -- only available in CLI
- `gateway_only`: bool -- only available in gateway/messaging

[VERIFIED: hermes-agent/hermes_cli/commands.py lines 28-142]

### Pattern 2: Three-Stage Resolution (resolve_command)

**What:** Command resolution with exact -> alias -> prefix fallback.
**When to use:** Every time a `/` prefixed message is received.

The hermes-agent implements this in two layers:
1. `resolve_command()` in commands.py (line 162-167): exact name + alias lookup via `_COMMAND_LOOKUP` dict [VERIFIED]
2. Prefix matching in `cli.py:process_command()` (lines 4020-4058): finds all commands starting with the typed prefix, prefers exact match, then shortest unique match, then reports ambiguity [VERIFIED]

For IronHermes, these should be unified into a single `resolve_command()`:

```rust
pub enum ResolveResult {
    /// Exact or alias match found
    Exact(CommandDef),
    /// Unique prefix match found
    PrefixMatch(CommandDef),
    /// Multiple commands match the prefix
    Ambiguous(Vec<&'static str>),
    /// No built-in match; may be a skill name
    NotFound,
}

impl CommandRouter {
    pub fn resolve_command(&self, input: &str, platform: &Platform) -> ResolveResult {
        let name = input.trim_start_matches('/').to_lowercase();

        // Stage 1: Exact match (filtered by platform)
        if let Some(cmd) = self.lookup.get(name.as_str()) {
            if cmd.is_available_on(platform) {
                return ResolveResult::Exact(cmd.clone());
            }
        }

        // Stage 2: Alias match
        if let Some(cmd) = self.alias_lookup.get(name.as_str()) {
            if cmd.is_available_on(platform) {
                return ResolveResult::Exact(cmd.clone());
            }
        }

        // Stage 3: Prefix match (only platform-available commands)
        let candidates: Vec<_> = self.commands.iter()
            .filter(|c| c.is_available_on(platform))
            .filter(|c| c.name.starts_with(&name)
                || c.aliases.iter().any(|a| a.starts_with(&name)))
            .collect();

        match candidates.len() {
            0 => ResolveResult::NotFound,
            1 => ResolveResult::PrefixMatch(candidates[0].clone()),
            _ => {
                // Prefer shortest unique match (hermes-agent behavior)
                let min_len = candidates.iter().map(|c| c.name.len()).min().unwrap();
                let shortest: Vec<_> = candidates.iter()
                    .filter(|c| c.name.len() == min_len)
                    .collect();
                if shortest.len() == 1 {
                    ResolveResult::PrefixMatch(shortest[0].clone())
                } else {
                    let names: Vec<&str> = candidates.iter()
                        .map(|c| c.name)
                        .collect();
                    ResolveResult::Ambiguous(names)
                }
            }
        }
    }
}
```

### Pattern 3: Running-Agent Guard

**What:** Mechanism to block/bypass certain commands while the agent is actively running.
**When to use:** Gateway and CLI when an agent turn is in-flight.

From `gateway/run.py:1735-1852`, the running-agent guard works as follows [VERIFIED]:
- When `session_key in self._running_agents`:
  - `/stop`: Hard-kills the agent, clears running state, returns confirmation
  - `/new` (alias `/reset`): Interrupts agent, clears pending, dispatches reset
  - `/status`: Dispatches directly (bypasses guard)
  - `/queue` (alias `/q`): Queues prompt for next turn without interrupting
  - Photo messages: Queued without interrupt (Telegram-specific)
  - All other messages/commands: Interrupt agent, queue as pending

For the CLI (`cli.py`), `_agent_running` is a bool that:
- Is set to `True` when agent loop starts (line 7563)
- Is set to `False` when agent completes (line 7569)
- Most commands check it for queuing behavior (e.g., `/queue` shows different message when agent running)

**Recommendation:** Use an `AtomicBool` (or `Arc<AtomicBool>`) for the running-agent flag, passed into `CommandContext`. Commands that bypass the guard: `/stop`, `/new`/`/reset`, `/status`, `/queue`. Commands that are blocked while agent is running: everything else returns "Agent is running. Use /stop to interrupt." [ASSUMED]

### Pattern 5: Integration With Extension Chain

**What:** The router becomes the core fallback in the extension-first dispatch chain.
**When to use:** CLI only (gateway calls router directly).

From Phase 22.1 D-06 and the existing code in tui/commands.rs [VERIFIED]:

```
CLI dispatch chain:
  1. TUI Extensions (process_command) -- first crack, can shadow built-ins
  2. CommandRouter.resolve_command() -- core router (replaces core_dispatch)
  3. SkillRegistry.find() -- /<skill-name> catch-all
  4. Unknown command error

Gateway dispatch chain:
  1. Running-agent guard (bypass /stop, /new, /status, /queue)
  2. CommandRouter.resolve_command() -- core router
  3. Quick commands (user-defined, NOT in scope for 21.1)
  4. Plugin commands (NOT in scope)
  5. SkillRegistry.find() -- /<skill-name> catch-all
  6. Pass through to agent as normal message
```

### Anti-Patterns to Avoid
- **Inline match arms for commands:** The current CLI uses `match cmd { "quit" => ..., "clear" => ... }` with 4 hardcoded commands. The gateway uses `match command { "/start" => ..., "/new" => ... }` with 4 hardcoded commands. Both must be replaced by the router, not extended.
- **Separate command lists per platform:** The hermes-agent has `GATEWAY_KNOWN_COMMANDS` derived from `COMMAND_REGISTRY`. IronHermes should have a single registry with platform filtering, not separate lists.
- **Recursive prefix expansion:** The Python CLI's prefix matching calls `self.process_command(full_cmd)` recursively. The Rust router should resolve-then-dispatch in one pass to avoid recursion.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Prefix matching | Custom trie or regex | Linear scan with `starts_with` | Command list is ~45 items; linear scan is O(n) with n<50, far simpler than a trie and perfectly fast [ASSUMED] |
| Alias resolution | Separate alias map maintained manually | Derive from `CommandDef.aliases` at build time | Single source of truth, matches hermes-agent pattern [VERIFIED: commands.py] |
| Help text generation | Hardcoded help strings | Generate from `CommandDef.description` + `args_hint` | Auto-updates when commands change [VERIFIED: hermes-agent pattern] |
| Platform filtering | Per-platform command lists | `PlatformFilter` enum on each `CommandDef` | Single registry, no synchronization needed [VERIFIED: hermes-agent cli_only/gateway_only pattern] |

**Key insight:** The hermes-agent derives ALL secondary data structures (COMMANDS dict, COMMANDS_BY_CATEGORY, SUBCOMMANDS, GATEWAY_KNOWN_COMMANDS) from the single `COMMAND_REGISTRY`. IronHermes must follow the same pattern -- one `Vec<CommandDef>` as the source of truth, everything else derived.

## Common Pitfalls

### Pitfall 1: Circular Dependencies from Router Location
**What goes wrong:** Placing command handlers in `ironhermes-core` that need types from `ironhermes-agent` or `ironhermes-tools` creates circular deps.
**Why it happens:** Command handlers for `/compress`, `/model`, `/stop` need AgentLoop, ProviderResolver, CancellationToken etc.
**How to avoid:** Command definitions and `resolve_command()` live in core. Command _handlers_ receive dependencies via `CommandContext` (trait objects or `Arc<dyn T>`). Handler implementations can live in core for simple cases (return a string) or the caller can match on the resolved command and execute platform-specific logic. The `CommandResult` enum bridges the gap.
**Warning signs:** `ironhermes-core/Cargo.toml` gaining dependencies on agent or tools crates.

### Pitfall 2: Prefix Matching Infinite Recursion
**What goes wrong:** The Python CLI's prefix matching calls `self.process_command(full_cmd)` recursively. If the expanded command doesn't match an exact branch, it loops.
**Why it happens:** The prefix expander reconstructs a command string and re-dispatches.
**How to avoid:** Resolve-then-dispatch in one pass. `resolve_command()` returns the resolved `CommandDef`, and the caller dispatches directly on the result. No re-entry.
**Warning signs:** `resolve_command()` calling itself or the dispatch function calling `resolve_command()` in a loop.
[VERIFIED: hermes-agent cli.py:4040-4052 has explicit recursion guard]

### Pitfall 3: Extension Shadowing Confusion
**What goes wrong:** A TUI extension claims a command (e.g., `/help`) but the router also handles it, leading to inconsistent behavior.
**Why it happens:** The extension-first dispatch chain (Phase 22.1 D-06) lets extensions shadow built-in commands, but only in CLI.
**How to avoid:** The dispatch chain is: extensions -> router -> skills. The router is ONLY called when no extension claimed the command. Gateway skips extensions entirely and calls router directly.
**Warning signs:** Same command behaving differently in CLI vs gateway when no extension is registered.
[VERIFIED: tui/commands.rs dispatch_command() extension-first pattern]

### Pitfall 4: Platform Filter Mismatch
**What goes wrong:** A CLI-only command accidentally available in gateway, or a gateway-only command appearing in CLI help.
**Why it happens:** Platform filtering not applied consistently at both resolution and help-generation time.
**How to avoid:** `resolve_command()` takes a `Platform` parameter and filters candidates. `help_text()` also takes a `Platform` parameter. Both use the same `CommandDef.is_available_on()` method.
**Warning signs:** `/approve` showing up in CLI `/help` output, or `/tools` working in Telegram.

### Pitfall 5: Running-Agent Guard Race Conditions
**What goes wrong:** User sends `/model` while agent is running. Without a guard, the model changes mid-turn and the next API call uses different credentials.
**Why it happens:** The running-agent flag and command dispatch are checked on different threads/tasks.
**How to avoid:** The guard check happens at the dispatch entry point (before resolution). An `AtomicBool` is sufficient since the check is "is agent running? yes/no" without complex state. Commands that bypass the guard (`/stop`, `/new`, `/status`, `/queue`) are checked explicitly.
**Warning signs:** Model or config changes taking effect during an active agent turn.
[VERIFIED: gateway/run.py:1735-1852 guard pattern]

### Pitfall 6: CommandResult Mismatch With Existing Types
**What goes wrong:** The router returns a different `CommandResult` type than the existing `tui::extension::CommandResult`, requiring conversion layers.
**Why it happens:** Phase 22.1 already defined `CommandResult { Handled(String), Silent, Error(String) }` in the CLI crate.
**How to avoid:** The core router should define its own `CommandResult` (possibly with additional variants like `Quit`, `ClearSession`, `PassThrough`). The CLI adapter maps core `CommandResult` to TUI `CommandResult`. Keep the mapping thin.
**Warning signs:** A single `CommandResult` type trying to serve both core routing semantics and TUI rendering needs.

```rust
impl CommandRouter {
    pub fn resolve(&self, input: &str, platform: &Platform) -> ResolveResult {
        let name = input.trim_start_matches('/').split_whitespace().next()
            .unwrap_or("").to_lowercase();

        // Stage 1: exact match
        if let Some(def) = self.by_name.get(name.as_str()) {
            if def.is_available_on(platform) {
                return ResolveResult::Exact(def);
            }
        }

        // Stage 2: alias match
        if let Some(def) = self.by_alias.get(name.as_str()) {
            if def.is_available_on(platform) {
                return ResolveResult::Exact(def);
            }
        }

        // Stage 3: prefix match (platform-filtered)
        let candidates: Vec<&CommandDef> = self.commands.iter()
            .filter(|c| c.is_available_on(platform))
            .filter(|c| c.name.starts_with(&name)
                || c.aliases.iter().any(|a| a.starts_with(&name)))
            .collect();

        match candidates.len() {
            0 => ResolveResult::NotFound,
            1 => ResolveResult::PrefixMatch(candidates[0]),
            _ => {
                // Shortest-match preference (hermes-agent behavior)
                let min_len = candidates.iter()
                    .map(|c| c.name.len()).min().unwrap();
                let shortest: Vec<_> = candidates.iter()
                    .filter(|c| c.name.len() == min_len).collect();
                if shortest.len() == 1 {
                    return ResolveResult::PrefixMatch(shortest[0]);
                }
                ResolveResult::Ambiguous(
                    candidates.iter().map(|c| c.name).collect()
                )
            }
        }
    }
}
```

## Open Questions

1. **Handler Architecture: Functions vs Trait Objects**
   - What we know: Each command needs a handler. Simple commands return a string. Complex commands need async access to state.
   - What's unclear: Whether handlers should be `fn` pointers stored in `CommandDef`, a match-on-name dispatch, or `Box<dyn CommandHandler>` trait objects.
   - Recommendation: Use `match` on `cmd.name` in a `dispatch()` method. This avoids boxing overhead, keeps all dispatch visible in one place, and is the pattern hermes-agent uses (elif chain in process_command). Trait objects add complexity with no clear benefit for a fixed command set.

2. **Core CommandResult vs TUI CommandResult**
   - What we know: TUI has `CommandResult { Handled(String), Silent, Error(String) }`. Core router needs additional semantics: `Quit`, `ClearSession`, `ModelChanged`, `PassThroughToAgent`.
   - What's unclear: Whether to extend the existing enum or create a new one.
   - Recommendation: New enum in core with richer variants. CLI adapter maps to TUI `CommandResult`. Gateway adapter maps to response string. Keeps each layer's concerns clean.

3. **Where Do Handler Implementations Live?**
   - What we know: D-01 says router lives in `ironhermes-core`. But some handlers need `AgentLoop`, `MemoryManager`, `TuiHandle` etc.
   - What's unclear: How much logic can live in core vs must live in the caller crate.
   - Recommendation: Core provides `resolve_command()` + `CommandDef` + `CommandContext` + `CommandResult`. Core also provides simple handlers (pure-data commands like /help, /config display). Complex handlers (those needing agent/platform state) are dispatched by the caller (CLI's `run_chat` or gateway's `handle_message`) which matches on the resolved command and calls platform-specific logic. This keeps core as a leaf crate.
