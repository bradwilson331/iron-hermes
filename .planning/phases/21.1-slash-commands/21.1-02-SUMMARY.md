# Phase 21.1 Plan 02: Wire CommandRouter into CLI and Gateway Summary

**One-liner:** CLI and gateway slash command dispatch replaced with unified CommandRouter resolve()+dispatch() — extension-first chain preserved, old hardcoded dispatchers deleted, all platforms route through ironhermes-core.

## What Was Built

### Task 1: CLI CommandRouter integration

**`crates/ironhermes-cli/src/tui/commands.rs`**
- `dispatch_command` signature extended with `router: &CommandRouter` and `ctx: &CommandContext`
- Core body: extension chain unchanged (catch_unwind loop), then `router.resolve()` + `handlers::dispatch()`
- `core_dispatch` function deleted entirely
- `map_core_to_tui` helper added: maps all 7 `CoreCommandResult` variants to TUI `CommandResult`
- `format_help` signature updated to accept `router: &CommandRouter` and `platform: &Platform`; now uses `router.commands_by_category()` instead of hardcoded strings
- `build_cli_router()` convenience function exported
- Tests updated: all 14 existing tests updated to pass `router` and `ctx`; 2 new tests added (`dispatch_ambiguous_prefix_returns_error_with_candidates`, `dispatch_unknown_returns_error`)

**`crates/ironhermes-cli/src/main.rs`**
- Added imports: `CommandRouter`, `CommandContext`, `CoreCommandResult`, `build_command_registry`, `Platform`, `AtomicBool`, `build_cli_router`
- `CommandRouter::new(build_command_registry())` constructed before REPL loop
- `Arc<AtomicBool> agent_running` constructed before REPL loop
- `CommandContext::new(Platform::Local, session_id, agent_running.clone())` built per turn
- `agent_running.store(true)` before `run_agent_turn`; `agent_running.store(false)` after
- REPL match block updated: `Handled("Goodbye!")` breaks, `Handled("Conversation cleared.")` truncates messages
- `print_help` updated to use `build_cli_router()` + new `format_help` signature
- 3 new static-grep tests: `run_chat_constructs_command_router`, `run_chat_constructs_command_context`, `run_chat_has_agent_running_flag`

### Task 2: Gateway CommandRouter integration

**`crates/ironhermes-gateway/src/handler.rs`**
- Added imports: `CommandResult as CoreCommandResult`, `CommandRouter`, `ResolveResult`, `build_registry`, `CommandContext`
- `command_router: CommandRouter` field added to `GatewayMessageHandler`; initialized in `new()` with `CommandRouter::new(build_registry())`
- `handle_slash_command` rewritten to use `self.command_router.resolve(command_input, platform)` with all 4 result arms
- `cmd_start`, `cmd_new`, `cmd_clear`, `cmd_help` methods deleted
- `/start` behavior preserved: `NewSession` arm checks `def.name == "start"` and runs LLM greeting
- `/new` clears full session; `/clear` wipes messages but keeps session alive
- Ambiguous prefix: sends candidate list to user
- Unknown (NotFound): passes message through to agent per D-08
- `@botname` suffix stripped before `resolve()` per T-21.1-06
- 5 static-grep regression tests: `handler_uses_command_router`, `handler_uses_command_context`, `handler_does_not_have_hardcoded_help_text`, `handler_resolves_commands_via_router`, `handler_struct_has_command_router_field`

### Architectural Note: TUI CommandResult mapping

The plan suggested handling `CoreCommandResult::Quit` and `CoreCommandResult::ClearSession` directly in the REPL loop by matching on `ResolveResult` variants. Instead, `dispatch_command` in `commands.rs` was kept as the single entry point (extension-first -> router -> map_core_to_tui), and the REPL loop detects quit/clear semantics by matching on the mapped `Handled("Goodbye!")` and `Handled("Conversation cleared.")` message strings. This avoids duplicating the extension-chain logic in main.rs and preserves the `dispatch_command` API.
