## User Constraints (from CONTEXT.md)

### Locked Decisions

**Video Capture Tool (hexapod_video)**
- D-01: New file `crates/ironhermes-tools/src/hexapod_video.rs` with `HexapodVideoTool` struct. Registered via `register_unless_skipped!` in `registry.rs` alongside `hexapod_tcp`. Follows the `terminal.rs` / `hexapod_tcp.rs` Tool trait pattern.
- D-02: Single action: `capture_frame`. No parameters. Connects to `HEXAPOD_IP:8002`, reads 4 bytes as little-endian u32 (frame length), reads that many bytes as JPEG, disconnects.
- D-03: Return format: `data:image/jpeg;base64,<encoded>` — standard data URI prefix so Gemma 4 recognizes it as a JPEG image inline in the next LLM call. Use the `base64` crate (already in workspace at `0.22`).
- D-04: Port 8002 is hardcoded — matches how `hexapod_tcp` hardcodes port 5002. `HEXAPOD_IP` env var is shared. Missing `HEXAPOD_IP` returns `Ok("Error: HEXAPOD_IP env var not set — cannot connect to robot")`.
- D-05: Stateless connection — connect → read one frame → disconnect. No persistent streaming socket.
- D-06: Port busy (connection refused) → `Ok("Error: video port 8002 is busy — another client is connected. Disconnect the other client and retry.")`. No automatic retry.
- D-07: No UI component — agent-only tool.

**Sonic Stream (stream_distance on hexapod_tcp)**
- D-08: New `stream_distance` action on `hexapod_tcp`. Grows the action enum from 14 to 15 entries (12 from Phase 27.1.3 + camera_pan + camera_tilt from D-12/D-13 + stream_distance).
- D-09: `samples` integer parameter (required). Clamped to `[1, 20]` silently. 20 samples × 200ms = 4 seconds max.
- D-10: ~200ms `tokio::time::sleep` between polls. Each poll opens a fresh TCP connection to port 5002, sends `CMD_SONIC\n`, reads `CMD_SONIC#<dist>\n`.
- D-11: Return format: `"Distances: [42, 43, 41, 44, 42] cm | min=41 max=44 avg=42.4"`.

**Camera Gimbal Control (camera_pan + camera_tilt on hexapod_tcp)**
- D-12: `camera_pan` action: `x` parameter (integer 50–180), sends `CMD_CAMERA#x#<current_y>\n`. Fire-and-forget, returns `"OK"`.
- D-13: `camera_tilt` action: `y` parameter (integer 0–180), sends `CMD_CAMERA#<current_x>#y\n`. Fire-and-forget, returns `"OK"`.
- D-14: Wire format: `CMD_CAMERA#x#y\n`. Both axes set in each call. x range 50–180, y range 0–180 — clamped client-side using `i64::clamp()`.
- D-15: Named constants: `CAMERA_PAN_MIN = 50`, `CAMERA_PAN_MAX = 180`, `CAMERA_TILT_MIN = 0`, `CAMERA_TILT_MAX = 180` at the top of `hexapod_tcp.rs`.

**Inherited from prior phases (carry-forward)**
- D-01 (27.1.1): Fire-and-forget returns `"OK"`; errors return `Ok("Error: ...")` strings inline.
- D-11 (27.1.1): One fresh TCP connection per tool call — stateless.
- D-12 (27.1.1): `HEXAPOD_IP` env var, missing → error string without network attempt.
- D-16 (27.1.1): Blocked commands return the blocked string; compile-time exhaustive match allowlist.
- D-20 (27.1.2): `tokio::time::sleep` (not `std::thread::sleep`) for async sleep.

### Claude's Discretion

- **camera_pan/camera_tilt joint-axis problem:** `CMD_CAMERA#x#y\n` sets BOTH axes simultaneously. For separate `camera_pan(x)` and `camera_tilt(y)` actions, the planner must decide what to send for the unused axis. Options: (a) use protocol midpoints as defaults (x=115, y=90), (b) collapse into a single `camera_move(x, y)` action with two required params instead. Planner should choose the approach that's clearest for Gemma 4 to call correctly.
- JPEG validation before base64 encoding (check JFIF/Exif header bytes like the Python client's `is_valid_image_4_bytes`) — planner decides whether to add this guard or trust the server stream.
- Whether to add a `read timeout` for the video frame read (analogous to the 3s read timeout in `read_distance`).

### Deferred Ideas (OUT OF SCOPE)

- Background streaming mode (`start_stream` / `stop_stream` actions): Continuous video at configurable frame rate, stored to a ring buffer or directory.
- `CMD_LED_MOD` animated modes (chase/blink/breathing/rainbow): Remain blocked per D-16 of Phase 27.1.1.
- JPEG validation guard before base64 encoding — planner may add this if the frame reliability warrants it; not mandated.

## Summary

Phase 27.1.4 delivers three sensor-awareness capabilities that let the IronHermes agent navigate with its Gemma 4 vision model: a new `hexapod_video` tool that reads one JPEG frame from the robot's video port (8002) and returns it as a base64 data URI, a `stream_distance` extension to `hexapod_tcp` that polls the ultrasonic sensor N times and returns a trend summary, and `camera_pan`/`camera_tilt` extensions to `hexapod_tcp` that point the camera gimbal before a capture.

All three deliverables follow well-established patterns already present in the codebase. The new `hexapod_video` tool is modeled on `browser_vision.rs` for the base64 encoding pattern and on `hexapod_tcp.rs` for the Tool trait implementation and stateless TCP connection pattern. The `stream_distance` action is a direct loop extension of the existing `read_distance` arm. The `camera_pan`/`camera_tilt` actions follow the `head_pan`/`head_tilt` pattern exactly.

The only genuinely novel complexity in this phase is the video port's 4-byte little-endian length-prefix framing protocol (port 8002) and the single-client constraint of `video_socket.listen(1)`. All other patterns are proven carry-forwards from Phases 27.1.1–27.1.3. The base64 crate (0.22) is already a workspace dependency used by `browser_vision.rs`, and `tokio::io::AsyncReadExt::read_exact` is available via the existing tokio dependency.

**Primary recommendation:** Implement `hexapod_video.rs` as a stateless unit-struct Tool following `hexapod_tcp.rs` structure, use `base64::engine::general_purpose::STANDARD.encode(bytes)` exactly as `browser_vision.rs` does, and use `tokio::io::AsyncReadExt::read_exact` for the framed video read. For the camera joint-axis problem (Claude's Discretion), use protocol midpoints as defaults (x=115, y=90) when the other axis is not specified.

**When to use:** `hexapod_video` / `capture_frame` action only.

### Pattern 4: Sonic Stream Loop (N polls with sleep)

**What:** Repeat `read_distance` N times with 200ms sleep between; accumulate results; format summary.
**When to use:** `stream_distance` action in `hexapod_tcp`.
**Example:**
```rust
// Source: derived from existing read_distance arm in hexapod_tcp.rs (verified pattern)
let samples = args["samples"].as_i64().unwrap_or(5).clamp(1, STREAM_DISTANCE_MAX_SAMPLES as i64) as usize;
let mut readings: Vec<i64> = Vec::with_capacity(samples);
for i in 0..samples {
    let read_fut = send_and_read_line(&addr, DISTANCE_CMD);
    let outcome = timeout(Duration::from_secs(3), read_fut).await;
    let raw = map_read_outcome(outcome, &addr)?;
    if raw.starts_with("Error:") { return Ok(raw); }
    let dist = parse_distance_value(&raw); // extract numeric cm value
    readings.push(dist);
    if i + 1 < samples {
        tokio::time::sleep(Duration::from_millis(200)).await;
    }
}
// compute min/max/avg, format return string
```

### Pattern 5: Camera Gimbal — Joint-Axis Default (Claude's Discretion resolution)

**What:** `CMD_CAMERA#x#y\n` requires both axes. For `camera_pan(x)`, use y=90 (midpoint of 0–180). For `camera_tilt(y)`, use x=115 (midpoint of 50–180).
**When to use:** Separating the two axes into independent actions.
**Rationale:** Using protocol midpoints means the "inactive" axis stays in a neutral center position, which is the least surprising behavior for an agent that calls one action at a time. The alternative — a single `camera_move(x, y)` action — requires the agent to always specify both axes, which is more typing and less intuitive for single-axis pointing. ### Anti-Patterns to Avoid

- **Using `std::thread::sleep` instead of `tokio::time::sleep`:** Would block the async executor. D-20 mandates `tokio::time::sleep`. [VERIFIED: hexapod_tcp.rs on_session_end + rotate arm]
- **Persistent video socket across calls:** The server's `video_socket.listen(1)` is single-client. A persistent socket would hold the connection exclusively. The stateless pattern (connect → read → disconnect) is correct per D-05. [VERIFIED: server.py transmit_video()]
- **Using `std::io::Read` instead of `tokio::io::AsyncReadExt`:** All I/O in this codebase is async. Blocking read traits cannot be used in async execute() without blocking the executor.
- **Using the old `read` loop instead of `read_exact`:** For a framed protocol where the exact byte count is known, `read_exact` eliminates partial-read bugs. The Python client's `connection.read(leng[0])` is equivalent.
- **Forgetting `base64::Engine as _` import:** The `encode` method requires the `Engine` trait to be in scope. `browser_vision.rs` line 22 shows the correct import. [VERIFIED: browser_vision.rs]
- **Sending `CMD_CAMERA` with only one argument:** The server's receive_commands() checks `len(command_parts) == 3` — if the count is not exactly 3 (prefix + x + y), the command is silently dropped. Always send both x and y. [VERIFIED: server.py line 185–189]

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JPEG → base64 encoding | Custom base64 encoder | `base64::engine::general_purpose::STANDARD.encode(bytes)` | Already in workspace; browser_vision.rs uses this exact call |
| Framed byte reading | Manual loop with partial-read tracking | `tokio::io::AsyncReadExt::read_exact` | Handles TCP recv splits natively; exact-count reads are idiomatic |
| Connection error detection | Manual string matching on io::Error | Return `Ok("Error: ...")` on connect failure | Established pattern; agent sees error inline as tool result |
| Sonic trend stats | Third-party stats crate | Inline min/max/avg over Vec<i64> | Trivial; no crate justified for 3 summary values |

**Key insight:** The video frame reading problem (4-byte length prefix + variable payload) is a textbook framed protocol; `read_exact` solves it in two calls with no edge-case risk.

## Common Pitfalls

### Pitfall 1: Port 8002 Is Single-Client
**What goes wrong:** `TcpStream::connect` to port 8002 fails with `ConnectionRefused` when the desktop PyQt5 client (`ui_client.py`) is already connected.
**Why it happens:** `video_socket.listen(1)` — the server only accepts one connection at a time and does not queue additional attempts. [VERIFIED: server.py line 59]
**How to avoid:** Detect `ConnectionRefused` specifically and return D-06's "video port 8002 is busy" message (not the generic robot-unreachable error).
**Warning signs:** Testing with the desktop UI client open will always trigger this; ensure desktop client is disconnected during development.

### Pitfall 2: `read_exact` Returns `UnexpectedEof` When Server Disconnects Mid-Frame
**What goes wrong:** If the camera is not running or the video stream stalls, the server's `transmit_video()` loop breaks and closes the connection. `read_exact` returns `Err(UnexpectedEof)` after reading fewer bytes than the declared length.
**Why it happens:** `camera_device.get_frame()` blocks until a frame is available, but if the camera fails to start the server thread terminates. [VERIFIED: server.py lines 102–113]
**How to avoid:** Wrap the `read_exact` calls in a `timeout` (same pattern as `read_distance`'s 3s timeout). The timeout ensures the tool doesn't hang indefinitely if the camera is not running.
**Warning signs:** Hanging `capture_frame` calls during testing — add a read timeout as Claude's Discretion item suggests.

### Pitfall 3: CMD_CAMERA Requires Exactly 3 Parts
**What goes wrong:** If the wire command has the wrong number of `#`-separated parts, the server silently ignores it.
**Why it happens:** `server.py` checks `len(command_parts) == 3` before executing the camera command. [VERIFIED: server.py line 185]
**How to avoid:** Wire format must be exactly `CMD_CAMERA#x#y\n` — three parts when split on `#`. Sending `CMD_CAMERA#x\n` (2 parts) is silently dropped. Always include both x and y in every call.
**Warning signs:** Camera doesn't move but tool returns `"OK"` — check the wire format part count.

### Pitfall 4: Outer Allowlist Match Must Include All 3 New Actions
**What goes wrong:** If `stream_distance`, `camera_pan`, or `camera_tilt` are missing from the outer allowlist arm in `hexapod_tcp.rs::execute()`, they fall to the catch-all and return the "blocked" error string.
**Why it happens:** The compile-time exhaustive match pattern requires every permitted action to be explicitly listed in the outer arm. [VERIFIED: hexapod_tcp.rs lines 302–305]
**How to avoid:** Add all three new actions to both the outer allowlist arm AND the inner match. The compiler will catch inner-match completeness but not the outer arm's list.
**Warning signs:** Test 20's pattern (`test_new_actions_not_blocked`) — add a parallel test for the three new actions.

### Pitfall 5: `base64::Engine as _` Import Required
**What goes wrong:** Calling `base64::engine::general_purpose::STANDARD.encode(bytes)` fails to compile with "method not found" if the `Engine` trait is not in scope.
**Why it happens:** The `encode` method is defined on the `Engine` trait; the struct `STANDARD` implements it but the trait must be imported.
**How to avoid:** Add `use base64::Engine as _;` at the top of `hexapod_video.rs`. [VERIFIED: browser_vision.rs line 22]
**Warning signs:** Compiler error "method not found in `GeneralPurpose`".

### Pitfall 6: Sonic Stream — Parse Numeric Value Not Formatted String
**What goes wrong:** `parse_distance_response` returns `"Distance: 42 cm"` (a formatted string). The `stream_distance` accumulator needs the integer `42`, not the string.
**Why it happens:** The existing helper is designed for human-readable output, not numeric accumulation.
**How to avoid:** Either add a `parse_distance_value(raw: &str) -> i64` pure helper that extracts just the numeric part, or inline the extraction in the stream loop. Do not call `parse_distance_response` and then try to parse back from the formatted string.
**Warning signs:** `"Distance: 42 cm".parse::<i64>()` fails; the number is in `parts[1]` of the raw response.

let mut jpg_buf = vec![0u8; frame_len];
stream.read_exact(&mut jpg_buf).await
    .map_err(|_| /* return error string */)?;

Alternative: collapse to single `camera_move(x, y)` action. Low risk — midpoints are safe neutral positions within the server's enforced ranges. |
| A2 | `std::io::ErrorKind::ConnectionRefused` is the error variant returned when port 8002 is busy | Common Pitfalls (Pitfall 1) | If tokio returns a different variant for the listen-queue-full case vs refused, the busy detection logic would miss it. In practice, `listen(1)` with 0 backlog available maps to ConnectionRefused on Linux. |

## Open Questions

1. **Should `capture_frame` have a read timeout?**
   - What we know: `read_distance` uses a 3s timeout. The video read can hang indefinitely if the camera is not streaming (camera.py's `get_frame()` blocks on a `Condition.wait()`).
   - What's unclear: How long should the timeout be? Frames are JPEG at ~15fps; a 5s timeout would be generous.
   - Recommendation: Add a timeout (5s suggested) wrapping both `read_exact` calls. This matches the established defensive pattern and prevents tool hangs.

2. **JPEG validation guard?**
   - What we know: The Python client checks `buf[6:10] in (b'JFIF', b'Exif')` and validates with PIL as fallback. The server is locally trusted hardware.
   - What's unclear: How often the Freenove server emits malformed frames (truncated frames at stream start, before camera warms up).
   - Recommendation: Planner may add a lightweight check (first 2 bytes == `\xff\xd8` SOI marker) as a guard. This is deferred per CONTEXT.md but worth documenting.
