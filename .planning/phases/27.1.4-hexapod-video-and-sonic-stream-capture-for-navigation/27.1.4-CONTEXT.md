# Phase 27.1.4: Hexapod Video and Sonic Stream Capture for Navigation - Context

**Gathered:** 2026-05-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Three deliverables that give the IronHermes agent sensor awareness for navigation:

1. **Video capture** (new tool) — New `hexapod_video` tool with a single `capture_frame` action. Connects to the Freenove server's video port (8002), reads one JPEG frame using the 4-byte LE length-prefix protocol, and returns a base64 data URI that Gemma 4 can consume inline as a multimodal tool result.

2. **Sonic stream** (extend existing tool) — New `stream_distance` action on `hexapod_tcp`. Polls `CMD_SONIC` N times (agent-specified, clamped to 20 max) with ~200ms between samples, returns all readings plus min/max/avg summary. Agent (Gemma 4) uses the trend to detect approaching obstacles.

3. **Camera gimbal control** (extend existing tool) — New `camera_pan` and `camera_tilt` actions on `hexapod_tcp` using `CMD_CAMERA#x#y\n`. Agent points the camera before capturing a frame. x range: 50–180, y range: 0–180 (server-enforced, clamped client-side).

The Freenove repo remains read-only protocol reference. No Python server code is modified.

</domain>

<decisions>
## Implementation Decisions

### Video Capture Tool (hexapod_video)

- **D-01:** New file `crates/ironhermes-tools/src/hexapod_video.rs` with `HexapodVideoTool` struct. Registered via `register_unless_skipped!` in `registry.rs` alongside `hexapod_tcp`. Follows the `terminal.rs` / `hexapod_tcp.rs` Tool trait pattern.
- **D-02:** Single action: `capture_frame`. No parameters. Connects to `HEXAPOD_IP:8002`, reads 4 bytes as little-endian u32 (frame length), reads that many bytes as JPEG, disconnects.
- **D-03:** Return format: `data:image/jpeg;base64,<encoded>` — standard data URI prefix so Gemma 4 recognizes it as a JPEG image inline in the next LLM call. Use the `base64` crate (already in workspace at `0.22`).
- **D-04:** Port 8002 is hardcoded — matches how `hexapod_tcp` hardcodes port 5002. `HEXAPOD_IP` env var is shared (same robot, same IP). Missing `HEXAPOD_IP` returns `Ok("Error: HEXAPOD_IP env var not set — cannot connect to robot")`.
- **D-05:** Stateless connection — connect → read one frame → disconnect. No persistent streaming socket. Each `capture_frame` call opens a fresh `TcpStream` to port 8002.
- **D-06:** Port busy (connection refused) → `Ok("Error: video port 8002 is busy — another client is connected. Disconnect the other client and retry.")`. Agent decides whether to retry. No automatic retry.
- **D-07:** No UI component — agent-only. Tool result is consumed by Gemma 4 in the next model call. No TUI panel or display logic in this phase.

### Sonic Stream (stream_distance on hexapod_tcp)

- **D-08:** New `stream_distance` action on `hexapod_tcp`. Grows the action enum from 14 to 15 entries (12 from Phase 27.1.3 + camera_pan + camera_tilt from D-12/D-13 + stream_distance).
- **D-09:** `samples` integer parameter (required). Clamped to `[1, 20]` silently — 20 samples × 200ms = 4 seconds max. Clamping follows the `head_pan`/`head_tilt` clamp pattern (silent, no error).
- **D-10:** ~200ms `tokio::time::sleep` between polls. Each poll opens a fresh TCP connection to port 5002, sends `CMD_SONIC\n`, reads `CMD_SONIC#<dist>\n` — follows the existing `read_distance` pattern exactly.
- **D-11:** Return format: `"Distances: [42, 43, 41, 44, 42] cm | min=41 max=44 avg=42.4"`. All individual readings visible for trend analysis; stats calculated as integers for min/max and one decimal for avg.

### Camera Gimbal Control (camera_pan + camera_tilt on hexapod_tcp)

- **D-12:** `camera_pan` action: `x` parameter (integer 50–180), sends `CMD_CAMERA#x#<current_y>\n`. Fire-and-forget, returns `"OK"`.
- **D-13:** `camera_tilt` action: `y` parameter (integer 0–180), sends `CMD_CAMERA#<current_x>#y\n`. Fire-and-forget, returns `"OK"`.
- **D-14:** Wire format: `CMD_CAMERA#x#y\n`. Both axes are set in each call (server requires both). x range 50–180, y range 0–180 — clamped client-side using `i64::clamp()`, same pattern as `head_pan`/`head_tilt`.
- **D-15:** Named constants: `CAMERA_PAN_MIN = 50`, `CAMERA_PAN_MAX = 180`, `CAMERA_TILT_MIN = 0`, `CAMERA_TILT_MAX = 180` at the top of `hexapod_tcp.rs` alongside existing constants.

### Inherited (carry-forward — do NOT re-litigate)

- **D-01 (27.1.1):** Fire-and-forget returns `"OK"`; errors return `Ok("Error: ...")` strings inline.
- **D-11 (27.1.1):** One fresh TCP connection per tool call — stateless. (Video tool follows this too.)
- **D-12 (27.1.1):** `HEXAPOD_IP` env var, missing → error string without network attempt.
- **D-16 (27.1.1):** Blocked commands return the blocked string; compile-time exhaustive match allowlist.
- **D-20 (27.1.2):** `tokio::time::sleep` (not `std::thread::sleep`) for async sleep.

### Claude's Discretion

- **camera_pan/camera_tilt joint-axis problem:** `CMD_CAMERA#x#y\n` sets BOTH axes simultaneously. For separate `camera_pan(x)` and `camera_tilt(y)` actions, the planner must decide what to send for the unused axis. Options: (a) use protocol midpoints as defaults (x=115, y=90), (b) collapse into a single `camera_move(x, y)` action with two required params instead. Planner should choose the approach that's clearest for Gemma 4 to call correctly.
- JPEG validation before base64 encoding (check JFIF/Exif header bytes like the Python client's `is_valid_image_4_bytes`) — planner decides whether to add this guard or trust the server stream.
- Whether to add a `read timeout` for the video frame read (analogous to the 3s read timeout in `read_distance`).

</decisions>

<specifics>
## Specific Ideas

- **Gemma 4 as the vision model**: The `data:image/jpeg;base64,<data>` data URI is the standard format that Gemma 4's multimodal inputs expect. The agent calls `capture_frame`, gets the data URI back as the tool result string, and the next LLM call includes it as an image in the multimodal message. No special IronHermes changes needed — the tool result flows naturally through the existing agent loop.
- **Python client wire format** (from `Client.py`): `stream_bytes = connection.read(4)` then `leng = struct.unpack('<L', stream_bytes[:4])` then `jpg = connection.read(leng[0])`. The Rust equivalent: `read_exact(&mut len_buf)` → `u32::from_le_bytes(len_buf)` → `read_exact(&mut jpg_buf)`.
- **Video port is single-client**: The Freenove server's `video_socket.listen(1)` only accepts one connection at a time. The existing PyQt5 client (`ui_client.py`) occupies port 8002 when running. The agent should not attempt video capture while the desktop UI client is connected.
- **Sonic stream use case**: Agent calls `stream_distance(samples=10)`, Gemma 4 sees the trend (e.g., distances decreasing from 80→45 cm), decides to stop or redirect before the next walk command.
- **Camera gimbal vs head**: `CMD_CAMERA` controls the camera's own servo mount; `CMD_HEAD` controls the robot's head servos. They are independent. The camera gimbal can be pointed while the head is looking elsewhere.

</specifics>

<deferred>
## Deferred Ideas

- **Background streaming mode** (`start_stream` / `stop_stream` actions): Continuous video at configurable frame rate, stored to a ring buffer or directory. Bigger scope — not needed for single-frame navigation decisions.
- **`CMD_LED_MOD` animated modes** (chase/blink/breathing/rainbow): Remain blocked per D-16 of Phase 27.1.1. Could be a future expression phase.
- **JPEG validation guard** before base64 encoding — planner may add this if the frame reliability warrants it; not mandated.

</deferred>
