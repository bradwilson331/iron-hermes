# Phase 27.1.4: Code Review Report

**Reviewed:** 2026-05-12T00:00:00Z
**Depth:** standard
**Files Reviewed:** 5
**Status:** issues_found

## Summary

Reviewed the Phase 27.1.4 implementation: `hexapod_tcp` Phase-2 extensions (stream_distance, camera_pan, camera_tilt), the new `hexapod_video` capture tool, module registration in `lib.rs` and `registry.rs`, and the updated hexapod skill reference. The architecture is sound — allowlist-before-env-var ordering is consistent, the frame bounds check in `hexapod_video` is correct, and registry wiring is complete.

Four warnings require attention before this ships: a silent false-LOW battery report on malformed voltage strings, a silently-corrupted `stream_distance` aggregate on malformed distance data, an unreachable code comment that misrepresents a reachable (if unlikely) division-by-zero, and a test isolation gap where the per-module `ENV_LOCK` mutex does not prevent races between `hexapod_tcp` and `hexapod_video` test modules running concurrently in the same binary.

No critical security vulnerabilities were found. The `unsafe` env-var mutations are test-only and correctly gated behind `#[cfg(test)]`.

## Warnings

### WR-01: `parse_battery_response` silently reports LOW on malformed voltage strings

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:145-152`
**Issue:** When the robot's `CMD_POWER` response contains present-but-non-numeric voltage fields (e.g., `CMD_POWER#err#err\n`), `s1.parse::<f32>().unwrap_or(0.0)` and `s2.parse::<f32>().unwrap_or(0.0)` silently substitute `0.0`. Because `0.0 < BATTERY_LOW_V1 (5.5)` and `0.0 < BATTERY_LOW_V2 (6.0)`, the function returns `"Battery: 0V / 0V (LOW)"` — indistinguishable from a genuine low-battery reading. The error branch on line 151 only fires for missing fields, not for unparseable ones. An LLM receiving `(LOW)` may make incorrect follow-up decisions (land the robot, abort the mission) on what is actually a parse failure.

**Fix:**
```rust
pub(crate) fn parse_battery_response(raw: &str) -> String {
    let parts: Vec<&str> = raw.trim().split('#').collect();
    match (parts.get(1), parts.get(2)) {
        (Some(s1), Some(s2)) => {
            let v1 = match s1.parse::<f32>() {
                Ok(v) => v,
                Err(_) => return format!(
                    "Error: unexpected battery response from robot (non-numeric voltage): {:?}",
                    raw.trim()
                ),
            };
            let v2 = match s2.parse::<f32>() {
                Ok(v) => v,
                Err(_) => return format!(
                    "Error: unexpected battery response from robot (non-numeric voltage): {:?}",
                    raw.trim()
                ),
            };
            let status = if v1 < BATTERY_LOW_V1 || v2 < BATTERY_LOW_V2 { "LOW" } else { "OK" };
            format!("Battery: {v1}V / {v2}V ({status})")
        }
        _ => format!("Error: unexpected battery response from robot: {:?}", raw.trim()),
    }
}
```

### WR-02: `stream_distance` silently substitutes `0` for unparseable distance readings

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:555-559`
**Issue:** When a `CMD_SONIC` response has a present-but-non-numeric distance field, `.parse::<i64>().ok().unwrap_or(0)` silently inserts `0` cm into the `readings` vector. A `0` distance is a physically valid sensor reading (obstacle touching the sensor), so it is impossible to distinguish a real zero from a parse failure in the aggregated output. The min/max/avg statistics are then silently corrupted. On repeated failures, the output could be `"Distances: [0, 0, 0] cm | min=0 max=0 avg=0.0"` with no indication that data collection failed.

**Fix:** Return an error on parse failure rather than substituting 0:
```rust
let dist = match parts.get(1).and_then(|s| s.trim().parse::<i64>().ok()) {
    Some(d) => d,
    None => return Ok(format!(
        "Error: unexpected distance response from robot: {:?}", raw.trim()
    )),
};
```

### WR-03: Division by zero in `stream_distance` is possible if early-return is added in future

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:567-568`
**Issue:** The average is computed as:
```rust
let avg = readings.iter().copied().sum::<i64>() as f64
    / readings.len() as f64;
```
The code also has `let min = readings.iter().copied().min().unwrap_or(0)` (line 565) and `let max = ... .unwrap_or(0)` (line 566), which defensively handle the empty-vec case. But the division on line 568 has no such guard — `readings.len()` would be `0` if `readings` is empty, causing a divide-by-zero (in Rust, `f64` division by zero produces `inf`/`NaN`, not a panic, but `format!("{:.1}", f64::NAN)` yields `"NaN"` in the output). Currently the `samples` clamp to `[1, 20]` prevents an empty `readings` vec in the non-error path. However the inconsistency with the guarded `min`/`max` calls (which defend against empty) and the unguarded `len()` division is fragile — adding WR-02's fix (early return on parse failure mid-loop) preserves the invariant, but a future refactor that changes control flow could silently produce `NaN` output.

**Fix:** Add a guard consistent with the min/max pattern:
```rust
let avg = if readings.is_empty() {
    0.0
} else {
    readings.iter().copied().sum::<i64>() as f64 / readings.len() as f64
};
```

### WR-04: Per-module `ENV_LOCK` mutex does not prevent cross-module env-var races

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:616-620` and `crates/ironhermes-tools/src/hexapod_video.rs:234-237`
**Issue:** Both modules define a separate `static ENV_LOCK: Mutex<()>`. This serializes concurrent env-var mutation only within each module's test suite. When `cargo test` runs tests across both modules in the same process (the default with multiple threads), `hexapod_tcp` tests that call `remove_var("HEXAPOD_IP")` can race with `hexapod_video` tests that depend on `HEXAPOD_IP` being set (or vice versa). The comment in both files correctly identifies this: "Run the full test binary with RUST_TEST_THREADS=1 to avoid races with other modules." But this constraint is not enforced at the code level — a CI configuration change or `cargo nextest` run could silently introduce flaky tests. Note: `unsafe { std::env::set_var(...) }` and `unsafe { std::env::remove_var(...) }` are correctly `unsafe` as of Rust 1.81 (they are unsafe in a multithreaded process).

**Fix:** Hoist the env-var mutex to a crate-level `once_cell::sync::Lazy<Mutex<()>>` (or `std::sync::OnceLock`) shared across all test modules, and document it in a `tests/common/mod.rs` helper. Alternatively, add a `#[serial_test::serial]` attribute (if the `serial_test` crate is available) to all env-var-mutating tests in both modules. As a minimum, add a workspace-level `.cargo/config.toml` setting:
```toml
[test]
# Required: env-var mutations in hexapod tests are not cross-module-safe
# when tests run in parallel across modules.
# Remove only after env-var isolation is implemented at the crate level.
# RUST_TEST_THREADS = "1"
```
This makes the constraint machine-enforced rather than documentation-only.

### IN-01: `rotate` action returns env-var error for `degrees=0` when `HEXAPOD_IP` is unset

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:347-424`
**Issue:** The `degrees == 0` no-op short-circuit (line 424) fires only after `HEXAPOD_IP` has been read at line 349. The comment at line 341 ("HEXAPOD_IP is only read for the five permitted actions") and the description of `rotate(0)` as a no-op are slightly misleading: when `HEXAPOD_IP` is unset, `rotate(0)` returns an env-var error rather than `"OK"`. The test at line 862-865 acknowledges this ("But execute() reads HEXAPOD_IP before the inner match") and works around it by setting the env var to a non-routable address. This is not a behavioral bug in production (where `HEXAPOD_IP` is always set before any robotics actions), but it is an observable inconsistency and could confuse future test authors.

**Fix:** Move the `degrees == 0` check before the env-var read, consistent with the stated design intent:
```rust
"rotate" => {
    let degrees = args["degrees"].as_i64().unwrap_or(0);
    if degrees == 0 {
        return Ok("OK".to_string()); // no-op before env-var read
    }
    let ip = match env::var("HEXAPOD_IP") { ... };
    // ...
}
```
This would require restructuring the outer match to defer the env-var read per-action rather than once at the top of the allowed-actions arm — a broader refactor that is likely out of scope, but the inconsistency should be documented.

### IN-02: `CAMERA_PAN_DEFAULT` comment documents midpoint as 115 but the range is asymmetric

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:69`
**Issue:** The comment reads `// midpoint of 50–180`. The arithmetic midpoint of [50, 180] is (50 + 180) / 2 = 115, which is correct. However, "midpoint" in a servo context often implies the neutral/center position, not the arithmetic mean of the clamping range. The SKILL.md documents this as the "protocol midpoint" and it matches what `camera_pan` action sends as the y-default. No behavioral issue — this is a comment precision note.

**Fix:** Clarify the comment to distinguish protocol-neutral from arithmetic-midpoint if those ever diverge:
```rust
/// Midpoint of camera pan range [50, 180] = (50+180)/2 = 115. Sent as x-default in camera_tilt.
pub(crate) const CAMERA_PAN_DEFAULT: i64 = 115;
```
