## Summary

Phase 25.2 adds **one** LLM-visible tool, `web_extract`, in `crates/ironhermes-tools/src/web_extract.rs`. It is a dispatcher: classify URL → branch (YouTube skill / PDF / default web backend) → normalize to Markdown → sanitize (base64 + secret-URL) → run a 4-tier summarization pipeline gated on char count. The 28 D-decisions in CONTEXT.md are locked; this research resolves the residual implementation contracts.

Most of the heavy infrastructure already exists in the codebase: SSRF (`is_safe_url`), Phase 26 `resolve_role` cascade (`provider.rs:420`), skill registry with `find()` + `read_content()` (`skills.rs:729-740`), `pdf-extract` workspace dep (already at `0.10`, NOT `0.7` as CONTEXT.md assumed), `tokio::sync::Semaphore`, `wiremock 0.6` dev-dep, `regex` workspace dep, the `Tool` trait + `register_defaults()` pattern, and the `extract_content_local` HTML→Markdown helper (`web_read.rs:114`) earmarked for the `web_local.rs` refactor. The hermes-agent Python reference implementation (`/Users/twilson/code/hermes-agent/tools/web_tools.py`) is locally available and is the authoritative source for tier sizing, prompt shapes, base64 strip semantics, and provider response normalization.

**Primary recommendation:** Lift the Python `web_extract_tool` semantics verbatim into Rust, swap the SDK calls (Exa `exa_py` / Firecrawl SDK / Tavily `httpx`) for hand-rolled `reqwest` clients matching `web_read.rs::fetch_with_firecrawl`, swap the Python `_resolve_web_extract_auxiliary` for a `SummarizationClientHandle` trait (mirrors Phase 25.1's `VisionClientHandle` to avoid the `tools→agent` circular dep), and reuse `validate_url_async` + `extract_content_local` (refactored to `web_local.rs`) without modification.

### Pattern 1: Hand-Rolled Provider Client

**What:** Use `reqwest::Client::builder()` per call (matching `web_read.rs::fetch_with_firecrawl` at line 171), not a global lazy client. Each provider gets its own `fetch_with_<provider>` async fn returning `anyhow::Result<NormalizedDoc>`.

**When to use:** All three of Firecrawl, Exa, Tavily (D-06).

**Example (Firecrawl, verbatim from existing code, source of truth):**

```rust
// Source: /Users/twilson/code/ironhermes/crates/ironhermes-tools/src/web_read.rs:171
async fn fetch_with_firecrawl(url: &str) -> anyhow::Result<String> {
    let api_key = std::env::var("FIRECRAWL_API_KEY")
        .map_err(|_| anyhow::anyhow!("FIRECRAWL_API_KEY not set"))?;
    let timeout_secs = Config::load().map(|c| c.web.timeout_secs).unwrap_or(30);
    let client = reqwest::Client::builder()
        .timeout(Duration::from_secs(timeout_secs))
        .build()?;
    let response = client
        .post("https://api.firecrawl.dev/v1/scrape")
        .bearer_auth(&api_key)
        .json(&json!({ "url": url, "formats": ["markdown"] }))
        .send()
        .await
        .map_err(|e| anyhow::anyhow!("Firecrawl request failed: {}", e))?;
    // ... status check, JSON parse, normalize ...
}
```

### Pattern 4: Bounded Semaphore for Nested Fan-Out

**What:** A single `Arc<Semaphore>` created at tool-call time covers BOTH outer (per-URL) and inner (per-chunk) fan-out. Each await of an aux-LLM call acquires a permit.

**Idiomatic Rust:**

```rust
use std::sync::Arc;
use tokio::sync::Semaphore;

let sem = Arc::new(Semaphore::new(config.extract.max_parallel_summaries));
let mut handles = Vec::new();
for url in urls {
    let sem_c = sem.clone();
    handles.push(tokio::spawn(async move {
        // Outer permit: held for duration of this URL's full pipeline (D-17).
        let _permit = sem_c.acquire().await.expect("semaphore not closed");
        process_url(url).await
    }));
}
let results = futures::future::join_all(handles).await;
```

For chunked summaries inside one URL (tier 3), the same `sem` Arc is captured by the chunk-processing closure. **Caveat (D-17 open question):** if many URLs each enter tier 3, all chunk fan-outs share the budget — this is intentional (single-budget contract) but may serialize chunk processing within a URL when other URLs hold permits. Plan-phase should validate via integration test 4 (synthetic 600K) that chunk parallelism still works when other slots are idle. [VERIFIED pattern: tokio docs + standard Rust idiom]

### Pattern 5: Skill Dispatch From a Tool

**What:** The `SkillRegistry` exposes `find(name) -> Option<&SkillRecord>` (skills.rs:729) and `read_content(name) -> Option<String>` (skills.rs:735). There is **no** runtime "execute skill" entry point — skills are content (Markdown bodies + helper scripts) that the LLM is expected to follow.

**Implication for D-10 (YouTube):** The current Phase 19 skills runtime does NOT execute skill bodies; it serves them to the LLM as text. The PRD's "skill returns Markdown" semantic does NOT match how IronHermes skills work today.

**Recommended implementation:** `web_extract::youtube::dispatch(url)` runs the skill's helper script (`scripts/fetch_transcript.py`) directly via `tokio::process::Command`, then formats the JSON output as Markdown. The skill's `SKILL_DIR` is `record.path.parent()` — the directory containing the SKILL.md file (resolvable via `SkillRegistry::find("youtube-content").map(|r| r.path.parent())`).

```rust
// Pseudocode:
let registry = SkillRegistry::load(cwd); // or shared instance
let record = registry.find("youtube-content")
    .ok_or_else(|| anyhow!("youtube-content skill not installed"))?;
let skill_dir = record.path.parent().expect("SKILL.md has parent");
let script = skill_dir.join("scripts/fetch_transcript.py");
let output = tokio::process::Command::new("python3")
    .arg(&script)
    .arg(url)
    .arg("--text-only")
    .arg("--timestamps")
    .output().await?;
let json_or_text = String::from_utf8_lossy(&output.stdout).to_string();
// Format as Markdown per the schema in CONTEXT <specifics>.
```

This deviates from CONTEXT.md's wording ("the existing youtube-content skill via the Phase 19 skills framework") because the framework as built does not support tool-driven dispatch. Plan-phase should treat this as a **research-recommended adjustment to D-10** and surface as an Open Question. The alternative (exposing a thin `pub async fn execute_skill_helper(name, script_relpath, args) -> Result<String>` in `ironhermes-core::skills`) is also viable but adds a new public API surface to a stable module.

### Anti-Patterns to Avoid

- **Calling `Config::load()` per URL:** Load once at tool-call entry, share via `&Config` reference. (`web_read.rs` does it once per call — match that.)
- **Holding `MutexGuard` across `.await`:** `WebExtractTool` is stateless; no shared `Mutex`. The browser_session.rs `Arc<Mutex<...>>` pattern does NOT apply here. [VERIFIED: CONTEXT §code_context confirms statelessness]
- **Building a new `reqwest::Client` per chunk:** Build per URL (as `web_read.rs` does per call), reuse across the URL's fetch + LLM-call lifecycle if possible.
- **Unbounded `tokio::spawn` over the URL list:** Always combine with the Semaphore — D-15 is non-negotiable.
- **Calling skill body as LLM text from a tool:** Wrong layer. The tool runs the skill's helper script; the SKILL.md body is for the LLM.
- **Using `pdf-extract = "0.7"`:** Wrong version — workspace has `0.10`. See Standard Stack §verified versions.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| HTML → Markdown conversion | Custom DOM walker | `htmd::convert` (already in tree, used by `extract_content_local`) | Edge cases: tables, code blocks, nested lists, entity decoding. |
| HTML parsing | Regex on HTML | `scraper::Html` (already in tree) | Regex on HTML is famously broken; `scraper` uses html5ever. |
| PDF text extraction | byte-level PDF spec parsing | `pdf_extract::extract_text_from_mem` (workspace dep) | PDF is a complex stream format; pdf-extract handles encrypted streams, fonts, encoded text. |

### Target 7: `tokio::sync::Semaphore` patterns for nested concurrency

**Idiomatic single-budget pattern (D-15, D-17):**
```rust
let sem = Arc::new(Semaphore::new(max_parallel));

// Outer: per-URL fan-out
let outer_handles = urls.into_iter().map(|url| {
    let sem = sem.clone();
    let summarizer = summarizer.clone();
    tokio::spawn(async move {
        let _outer_permit = sem.acquire_owned().await.expect("sem");
        let raw = fetch_and_normalize(&url).await?;
        // Tier router decides whether to call inner fan-out:
        if raw.len() > TIER3_THRESHOLD {
            // Inner: per-chunk fan-out, SAME sem (D-15)
            let chunks = chunk(raw, CHUNK_SIZE);
            let inner_handles = chunks.into_iter().map(|chunk| {
                let sem = sem.clone();
                let summarizer = summarizer.clone();
                tokio::spawn(async move {
                    let _inner_permit = sem.acquire_owned().await.expect("sem");
                    summarizer.summarize_call(...).await
                })
            }).collect::<Vec<_>>();
            // ... collect, synthesize ...
        }
        Ok(...)
    })
}).collect::<Vec<_>>();
```
**Why `acquire_owned()`:** Permit lives across `await` points and the spawned tasks; `acquire()` returns a borrow tied to the semaphore's lifetime, which is awkward in spawned tasks. `acquire_owned()` requires `Arc<Semaphore>` and yields a `OwnedSemaphorePermit` that drops naturally. [VERIFIED: tokio docs + standard idiom]

**D-17 granularity validation:** Per-URL granularity means an outer permit is held for the WHOLE URL pipeline (fetch + normalize + summarize). With `max_parallel = 4`, four URLs can be in tier 3 simultaneously, each with its own chunk fan-out attempting to acquire from the same exhausted budget — chunks **serialize** when other URLs hold permits.

This is exactly the contract D-15 wants ("ONE budget for the whole call"), but performance-wise it can stall: if all 4 outer permits are held by URLs in tier 3, NO inner chunk can summarize until an outer permit releases. **Recommendation:** Plan-phase should run integration test 4 (synthetic 600K) with 1 vs 3 vs 8 URLs to validate behavior. If stalling is bad, the per-LLM-call alternative (each LLM call acquires/releases) is a single-line change but breaks the "one budget" contract. **[Open Question §3: confirm or amend D-17.]**

[VERIFIED: tokio docs + standard idiom; performance verification deferred to integration test]

## Common Pitfalls

### Pitfall 1: `pdf-extract` blocks the runtime

**What goes wrong:** `pdf_extract::extract_text_from_mem(&bytes)` is a synchronous CPU/IO-bound call. On a large PDF (say 30 MB), it can block the tokio worker for seconds, stalling other futures.

**Why it happens:** `pdf-extract` parses the PDF synchronously; tokio workers expect futures to yield.

**How to avoid:** Wrap in `tokio::task::spawn_blocking`:
```rust
let bytes_owned = bytes.to_vec();
let text = tokio::task::spawn_blocking(move || pdf_extract::extract_text_from_mem(&bytes_owned))
    .await
    .map_err(|e| anyhow!("PDF extract task panicked: {}", e))?
    .map_err(|e| anyhow!("PDF extract failed: {}", e))?;
```
**Warning signs:** Other tool calls in the same session getting starved during a large-PDF extraction; `tracing` shows long stalls between span enter/exit on adjacent tasks.

### Pitfall 2: SSRF check happens before classification, but PDF byte-fetch is a SECOND fetch

**What goes wrong:** If the PDF route fetches bytes via raw `reqwest` (D-09 local fallback), the redirect path can resolve to a private IP that wasn't validated.

**Why it happens:** SSRF was checked once at dispatch; the PDF byte-fetch hits a different URL (final after redirect).

**How to avoid:** Apply the post-redirect re-validation pattern from `web_read.rs:272-279` to the PDF byte-fetch path too. This is non-negotiable per D-18.

### Pitfall 3: Provider env var present but invalid (auth error)

**What goes wrong:** `FIRECRAWL_API_KEY` is set but expired/invalid. Backend chain logic (`if env::var(...).is_ok()`) selects Firecrawl; it returns 401; the URL fails entirely.

**Why it happens:** D-04 chain selects by env-var **presence**, not validity. There's no preflight ping.

**How to avoid:** Per-call backend chain MUST fall through on Firecrawl error → try Exa → try Tavily → try local. This is more aggressive than the `web_read.rs` pattern which only falls back to local on Firecrawl failure. Plan-phase: explicit chain logic, NOT short-circuit on first match.

### Pitfall 4: Base64 image regex catastrophic backtracking on adversarial input

**What goes wrong:** A page with megabytes of `data:image/png;base64,...` and irregular whitespace can pathologically slow a backtracking regex.

**Why it happens:** The pattern `data:image/[a-zA-Z0-9+.-]+;base64,[A-Za-z0-9+/=\n]+` is greedy.

**How to avoid:** The Rust `regex` crate uses RE2 (NFA-based, no backtracking) by default — this pitfall is theoretical, not actual. Verify by NOT using `regex::bytes::Regex` with backtracking flags. Just use `Regex::new(...)` from `LazyLock`. [VERIFIED: regex crate is RE2-based]

### Pitfall 5: Skill helper script not on PATH or wrong Python

**What goes wrong:** YouTube dispatch calls `python3 .../fetch_transcript.py URL`; system has no `python3` or has Python without `youtube-transcript-api`.

**Why it happens:** Skill helpers depend on operator setup; the tool can't ensure them.

**How to avoid:** Capture stderr; on non-zero exit, return `error: "youtube_skill_helper_failed: <stderr last line>"` per D-02. Surface a setup hint in the error text directing operator to `pip install youtube-transcript-api`. Don't crash the tool call.

### Pitfall 6: `Vec<ExtractionResult>` ordering vs URL input ordering

**What goes wrong:** With `tokio::spawn` over the URL list, tasks complete in arbitrary order. If we naively `.collect()`, the result `Vec` order doesn't match input order.

**Why it happens:** Tokio scheduling is non-deterministic.

**How to avoid:** Tag each spawn with the input index, sort results by index before returning. Or use `futures::stream::iter(urls).enumerate().map(...).buffer_unordered(N)` then sort. This matters because the LLM expects positional correspondence: input `urls[0]` → output `results[0]`.

### Pitfall 7: Semaphore deadlock if outer permit exhaustion blocks inner permits

**What goes wrong:** All 4 outer permits are held by tier-3 URLs; their inner chunk fan-outs all wait on the same semaphore. They CAN make progress (chunks acquire/release individually) but only one chunk per outer permit at a time.

**Why it happens:** D-15's "single budget" contract.

**How to avoid:** This is by design (D-15) but can be observed as poor parallelism. **Plan-phase MUST add an integration test** with 4 large URLs to validate that the semaphore eventually drains and no deadlock occurs. This is not actually a deadlock (permits are short-lived per LLM call) but a serialization that should be measured. See Open Question §3.

### Pitfall 8: Wrong skill name lookup

**What goes wrong:** `SkillRegistry::find("youtube_content")` (underscore, from PRD) returns `None` because the registry uses the `name:` frontmatter field which is `"youtube-content"` (hyphen).

**Why it happens:** PRD copy-paste error; CONTEXT D-10 flagged it for verification.

**How to avoid:** Use `"youtube-content"` (hyphen) in the dispatch call. Verified above (Target 4). Add a unit test asserting the lookup succeeds.

### Pitfall 9: Reqwest default timeout is None (infinite)

**What goes wrong:** Without `.timeout(Duration::from_secs(N))`, a slow provider hangs the call indefinitely.

**Why it happens:** `reqwest::Client::default()` has no timeout.

**How to avoid:** Always `.timeout(Duration::from_secs(config.web.timeout_secs))` per `web_read.rs:182` precedent. For LLM calls, the agent crate's `AnyClient` wrapper handles timeouts (matches Phase 25.1).

### Pitfall 10: Tier-4 refuse must still populate `url`

**What goes wrong:** Returning `ExtractionResult { url: "".into(), ... }` for the refuse case loses positional correspondence.

**Why it happens:** Easy to forget under "refuse" semantic.

**How to avoid:** Always populate `url` in EVERY `ExtractionResult` — the refuse envelope per CONTEXT `<specifics>` shows `url: "https://example.com/giant"`. Same for SSRF-blocked, secret-redacted, and per-URL extraction errors.

### Tier router (D-11/D-12)

```rust
async fn summarize_by_tier(
    doc: NormalizedDoc,
    use_llm: bool,
    extract_cfg: &ExtractConfig,
    web_cfg: &WebConfig,
    sem: Arc<Semaphore>,
    summarizer: Arc<dyn SummarizationClientHandle>,
) -> anyhow::Result<String> {
    let n = doc.content.chars().count();
    if n > extract_cfg.refuse_threshold_chars {
        anyhow::bail!("content_too_large");  // tier 4 — caller maps to ExtractionResult.error
    }
    if !use_llm {
        // D-12: short-circuit, smart-truncate
        return Ok(truncate_content(&doc.content, web_cfg.max_content_chars));
    }
    if n < extract_cfg.summary_tier2_threshold_chars {
        return Ok(doc.content);  // tier 1
    }
    if n < extract_cfg.summary_tier3_threshold_chars {
        // tier 2: single-pass summary
        let _permit = sem.acquire().await?;
        return summarizer.summarize_call(SUMMARY_SYSTEM.into(), build_user(&doc).into(), 20_000).await;
    }
    // tier 3: chunked + synthesize
    chunked_summarize(doc, extract_cfg, sem, summarizer).await
}
```

### Skill dispatch (YouTube, recommended adjustment)

```rust
async fn dispatch_youtube(
    url: &str,
    registry: &SkillRegistry,
) -> anyhow::Result<NormalizedDoc> {
    let record = registry.find("youtube-content")
        .ok_or_else(|| anyhow!("youtube-content skill not installed"))?;
    let skill_dir = record.path.parent()
        .ok_or_else(|| anyhow!("skill record has no parent dir"))?;
    let script = skill_dir.join("scripts/fetch_transcript.py");
    if !script.exists() {
        anyhow::bail!("youtube_skill_helper_missing");
    }
    let output = tokio::process::Command::new("python3")
        .arg(&script)
        .arg(url)
        .arg("--text-only")
        .arg("--timestamps")
        .output()
        .await
        .map_err(|e| anyhow!("youtube helper spawn failed: {}", e))?;
    if !output.status.success() {
        let stderr = String::from_utf8_lossy(&output.stderr);
        anyhow::bail!("youtube_skill_helper_failed: {}",
            stderr.lines().last().unwrap_or("(no stderr)"));
    }
    let body = String::from_utf8_lossy(&output.stdout).to_string();
    Ok(NormalizedDoc {
        url: url.into(),
        title: extract_h1_or_empty(&body),
        content: body,
        metadata: serde_json::json!({}),
    })
}
```

### §3 — D-17 Semaphore granularity

**Recommendation:** Add as `extract.max_total_aux_calls_per_invocation` with a generous default (e.g., 200) + `tracing::warn!` near 80% of cap. Plan-phase to decide.

**What's unclear:** Whether plan-phase should silently consume `workspace = true` (use 0.10) or amend the CONTEXT decision.

### §6 — Exa returns plain text, not Markdown

**What we know:** Exa's `/contents` API returns `text` field as plain text. The `format` parameter doesn't exist.

**What's unclear:** Whether to (a) prepend a Markdown header and treat as content (current Python ref impl pattern) OR (b) treat Exa output as a degraded source (rank below Tavily).

**Recommendation:** Option (a) — prepend `# {title}\nSource: {url}\n\n` to Exa text and treat as Markdown. Note the inconsistency in tool description so the LLM knows. Plan-phase to confirm.
