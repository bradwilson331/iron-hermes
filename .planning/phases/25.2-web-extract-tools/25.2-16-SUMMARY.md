# Phase 25.2 Plan 16: Gap-closure for UAT Issue 9 (secret leak in agent warning) Summary

**One-liner:** Both R1 (prompt-side) and R2 (sanitizer-side) root causes of UAT Issue 9 closed atomically: new `redact_secrets_in_url` transform sibling of Plan 04's predicate, wired through `ExtractionResult.url` echo on error AND success paths plus all 4 URL-field warn! sites in `fetch_web_with_chain` plus the pdf.rs warn! site, and a new `Secret handling:` rule appended to `DEFAULT_AGENT_IDENTITY` shaping the LLM to redact rather than echo credential-looking tokens — defense-in-depth across the trained-reflex layer AND the tool-output layer.

## What Was Built

### Task 1 — `redact_secrets_in_url` primitive (TDD: RED → GREEN)

Added a new public function `pub fn redact_secrets_in_url(url: &str, extras: &[String]) -> String` to `crates/ironhermes-tools/src/web_extract/sanitize.rs` as a sibling of the existing `contains_secret` predicate. Both functions consume the same `SECRET_URL_PATTERNS` const (9 entries, unchanged) and the same `&[String]` operator-extension list — single source of truth. Empty value spans skip redaction to avoid false positives.

**Implementation strategy chosen:** percent-decode-once → byte-scan over the decoded form → splice `***` for each matched value span up to the next `&` or `#` (or end-of-string). Fast-path: clean URLs (those that don't trigger `contains_secret`) are returned bytewise-equal with zero allocation. Dirty URLs emit the decoded form (encoding fidelity traded for redaction completeness).

The 8 behavior-locked tests written in the RED commit (`8513d45`) before the implementation:
- `redact_simple_token_query_param` — `?token=sk-fake-abc123` → `?token=***`
- `redact_handles_multiple_secrets_in_one_url` — `?token=abc&api_key=def&safe=ok` → `?token=***&api_key=***&safe=ok`
- `redact_case_insensitive_key_match` — `?TOKEN=ABC` → `?TOKEN=***` (key case preserved)
- `redact_handles_percent_encoded_param_name` — `?token%3Dabc` redacts the `abc` value (decode-aware)
- `redact_respects_extra_patterns` — operator-supplied `["x_custom_secret="]` extension list honored
- `redact_returns_input_unchanged_on_clean_url` — bytewise equality on clean URLs
- `redact_bearer_value_in_query_string` — `?h=Bearer%20sk-abc` redacts the `sk-abc` value
- `secret_url_patterns_const_count_unchanged` — `SECRET_URL_PATTERNS.len() == 9` lock

All 8 pass GREEN (`f116e77`). All 11 pre-existing sanitize tests continue to pass — total 19/19.

### Task 2 — Wire the redactor through web_extract.rs and pdf.rs

`crates/ironhermes-tools/src/web_extract.rs` (commit `9e237fb`):
- Imported `redact_secrets_in_url` alongside `contains_secret` and `strip_base64_images`
- **(A)** `process_one_url` step 1: secret-rejection branch now wraps the URL in `redact_secrets_in_url(url, &cfg.extract.redact_url_patterns)` before passing to `ExtractionResult::error(...)`. The model now receives the redacted form (e.g. `https://example.com/?api_key=***`) in the error envelope, not the cleartext.
- **(B)** `process_one_url` end: defensive `extraction.url = redact_secrets_in_url(&extraction.url, &cfg.extract.redact_url_patterns)` before the final return. Catches operator-extended pattern lists matching backend-returned URLs (post-redirect, normalized) that the pre-gate did not see.
- **(C)** `fetch_web_with_chain`: introduced `let url_for_log = redact_secrets_in_url(url, &cfg.extract.redact_url_patterns);` at the top of the function body and rewired all 4 URL-field warn! sites (Firecrawl/Exa/Tavily fall-through messages plus the final `backend_chain_exhausted` warn) to log `url_for_log` instead of raw `url`. The `url_for_log` count is exactly 5 in the file (1 declaration + 4 warn! sites).
- **(D)** `execute()`'s `URL task panicked` warn at line 207 left untouched — the message logs the join error, not the URL.
- **(G)** New integration-style test `process_one_url_redacts_secret_in_error_envelope` constructs the secret-rejection envelope shape and asserts `envelope.url` does NOT contain the literal `sk-or-v1-fakekeyabc123` token while still preserving the `api_key` parameter name for operator readability.

`crates/ironhermes-tools/src/web_extract/pdf.rs` (same commit):

### Task 3 — Append "Secret handling" rule to DEFAULT_AGENT_IDENTITY

`crates/ironhermes-agent/src/prompt_builder.rs` (commit `b64c067`):

Appended a "Secret handling:" multi-line section to `DEFAULT_AGENT_IDENTITY` (slot 1, durable, cacheable). The section contains three rules:

1. **NEVER echo the literal secret value** — the model must refer to it as `'the redacted credential'` or `'the [param_name] value'` rather than quoting it.
2. **Tools have built-in redaction — call them anyway** — closes the failure mode where the LLM short-circuits tool dispatch on secret-looking URLs. Tools (`web_extract`, `web_read`, `web_search`, `browser_navigate`) have their own gates; the LLM should let them run.
3. **Paraphrase the parameter name** — when the model must mention that a URL contains a secret, name the parameter without quoting the value (`'your URL contains an api_key parameter'`, not `'your URL contains api_key=sk-abc123'`).

**Raw-string-literal hazard:** the plan's example used double-quotes inside the prose, which would terminate the `r#"..."#` literal early. Used single-quotes inside the prose; the three required acceptance substrings ("`Secret handling:`", "`NEVER echo the literal secret value`", "`call them anyway`") remain byte-exact and pass all grep gates.

`TOOL_USE_GUIDANCE` intentionally NOT modified. Keeping the rule in slot 1 (Identity) means it persists across every subagent / personality overlay and benefits from durable-slot caching (Phase 18 D-04).

New unit test `default_agent_identity_includes_secret_handling_rule` locks all three required substrings against future prompt-tuning regressions.
