## Trust Boundaries

Boundaries are deduplicated across the 17 plan threat models.

| Boundary | Description | Data Crossing |
|----------|-------------|---------------|
| LLM tool call args (`urls`, `use_llm_processing`) → `WebExtractTool::execute` | JSON shape validated by serde; missing fields use defaults; partial-success per D-02 | Untrusted URL list, optional bool |
| URL string → SSRF gate (`validate_url_async` / `is_safe_url`) | Pre-fetch validation; post-redirect re-validation when `final_url != url` | URL host/IP literal |
| URL string → `contains_secret` / `redact_secrets_in_url` (sanitize.rs) | Percent-decoded substring match against const + operator patterns; no regex backtracking | URL with potential credential params |
| Untrusted page bytes → `scraper` html5ever / htmd Markdown | Browser-grade parser; threat surface inherited from web_read.rs (Phase 25.1) | HTML / Markdown |
| Untrusted page Markdown → `strip_base64_images` | RE2 NFA regex with bounded character classes; no backtracking DoS | Markdown bytes |
| URL → `tokio::process::Command` arg (youtube.rs) | URL passed verbatim as `arg(url)` — no shell, no `format!()` interpolation | URL string only |
| Skill helper `scripts/fetch_transcript.py` → execute as agent | Operator-installed script; trust delegated to Phase 19 SKILL-09 (pending) | Stdout transcript |
| URL → Firecrawl / Exa / Tavily POST (TLS, bearer auth) | Pre-fetch SSRF; provider follows redirects server-side | URL + provider response |
| URL → reqwest GET → `pdf_extract::extract_text_from_mem` | Pre-fetch + post-redirect SSRF; 50 MB byte cap; spawn_blocking + 30s timeout | PDF bytes (≤ 50 MB) |
| LLM tool args → Aux LLM (`AnyClientSummarizationHandle`) | Phase 26 cascade: `model.roles["summarization"]` → `auxiliary.summary` → main; bounded by `Arc<Semaphore>` (default 4 permits) | Page Markdown → summary |
| Operator YAML config → `ExtractConfig` | Typed serde deserialization; `#[serde(default)]` for backward-compat; bounded fields | Config knobs |
| Slash command `/toolset enable <name>` → `RegistryToolsetSession` | Sync trait bridged via `block_in_place + block_on`; `validate_name` regex + `check_known` allow-list; in-memory registry only — NO config.yaml writes (D-06) | Toolset name string |
| `process_one_url` → `ExtractionResult.url` echo (success + error) | Output crosses tool→LLM boundary; URL redacted via `redact_secrets_in_url` before exit | Possibly-redacted URL |
| Tracing `warn!` sites in web_extract.rs / pdf.rs | Secondary leak surface; URL fields use `url_for_log` (redacted) | Redacted URL string |
| `tokio::spawn` chunk fan-out → shared `Arc<Semaphore>` | Single-budget contract (D-15): outer per-URL + inner per-chunk share permits | LLM aux call frequency |
| `youtube.rs:81` → `tracing` (yt-dlp stderr) | OUT OF SCOPE — residual risk T-25.2-16-residual-yt; logged to follow-up | Helper-script stderr |

## Amendment — 2026-05-03 (post-UAT wireup gap close-out)

User UAT replay on 2026-05-03 found that `/toolset` returned the
"toolset session handle not configured" fallback string in the
ratatui REPL **and** Telegram, despite the 2026-05-02 audit verifying
T-25.2-15-01 and T-25.2-15-02 mitigations as CLOSED. The previous
audit's evidence was scoped to `crates/ironhermes-cli/src/main.rs`
(the surface Plan 15's parity guard checks) and missed two
production CommandContext-construction sites where the handle was
not threaded through:

- `crates/ironhermes-cli/src/tui_rata/commands.rs:446`
  (`build_command_context` — default REPL since Phase 22.4)
- `crates/ironhermes-gateway/src/handler.rs:254`
  (`handle_command` — per-message Telegram dispatch)

In both surfaces `ctx.toolset_session` was `None`, so
`cmd_toolset` (handlers.rs:782) short-circuited to the documented
fallback string. That fallback is a **deny-by-default** path — no
unauthorized mutation is possible — so this is a **functional gap
masquerading as a security guard**, not a security vulnerability.
But because T-25.2-15-01 and T-25.2-15-02 register their mitigations
as gating mechanisms reached *through* the handle, the previous
audit's "verified" claim was over-broad: the gates exist in the
production impl but were unreachable from those two surfaces.

### Fix commits

| Commit | Surface | Effect on threat register |
|--------|---------|---|
| `61ba493 fix(25.2-15): wire ToolsetSessionHandle into ratatui REPL` | ratatui REPL | T-25.2-15-01 (`validate_name` + `check_known`) and T-25.2-15-02 (D-06 no config.yaml writes) gates are now reachable from the default REPL surface. |
| `3f25ac5 fix(25.2-15): wire ToolsetSessionHandle into Telegram gateway` | Telegram | Same — gates now reachable from per-message Telegram dispatch via `GatewayMessageHandler.toolset_session` threaded through `GatewayRunner::set_toolset_session` → `build_gateway_handler`. |

The `Slash command /toolset enable <name> → RegistryToolsetSession`
trust boundary (line 39 of the Trust Boundaries table) now applies

production CommandContext-construction call site across the workspace,
not just `crates/ironhermes-cli/src/main.rs`. Logged as a follow-up
candidate for a future hardening plan.
