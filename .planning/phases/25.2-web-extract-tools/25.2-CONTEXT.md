# Phase 25.2: web-extract-tools — Context

**Gathered:** 2026-05-01
**Status:** Ready for planning
**Source:** Discuss-phase. User supplied a comprehensive PRD ported from `hermes-agent/tools/web_tools.py` covering tool surface, backends, format dispatch (HTML/PDF/YouTube), tiered LLM summarization, security, and the public Rust signature. <domain>
## Phase Boundary

A **single new tool** `web_extract` (LLM-visible name) that unifies multi-format web content extraction behind one dispatcher. Routes URLs by detected type (HTML/Markdown via configured backend, direct PDFs to a PDF→Markdown converter, YouTube to the existing `youtube-content` skill), normalizes everything to Markdown, sanitizes (base64 strip, secret-in-URL redaction), and runs a tiered LLM summarization pipeline gated on content size — all returned as a `Vec<ExtractionResult>` so the LLM can pass multiple URLs in one call.

Public Rust signature (locked):
```rust
pub struct ExtractionResult {
    pub url: String,
    pub title: String,
    pub content: String, // Normalized Markdown or summary
    pub error: Option<String>,
}

pub async fn web_extract(
    urls: Vec<String>,
    format: Option<String>,       // "markdown" (default) | "html"
    use_llm_processing: bool,     // false short-circuits all aux-LLM tiers
    min_length: usize,            // skip URLs whose extracted content is shorter
) -> Vec<ExtractionResult>;
```

The phase delivers exactly one tool — `web_extract` — joining the existing `web` toolset alongside `web_read` and `web_search`. Tooling for the new external providers (Exa, Tavily) ships as private submodules of this tool; they are NOT exposed as separate tools to the LLM in v2.1.

**Out of scope:**
- Multi-page crawling / spidering (would be a future "Web Crawl" phase using Firecrawl `/crawl`).
- Sitemap / robots.txt parsing.
- Site-specific scrapers (Twitter, Reddit, Hacker News specialty extractors).
- Asset downloading (binaries, archives, images for storage).
- WebSocket / streaming endpoints.
- JS-rendered SPA extraction via the Phase 25.1 `BrowserSession` (browser-backed extraction is browser_snapshot's job; if a page needs JS, the LLM uses browser_navigate + browser_snapshot, not web_extract).
- Per-URL custom selectors / XPath / CSS-selector extraction (declarative selector mode is deferred).
- Schema-driven structured output (Firecrawl `/extract` JSON-schema mode) — defer to a future "Structured Web Extraction" phase if real workflows demand it.
- Translation / language-detection on extracted content.
- Audio transcription beyond YouTube (general media STT defer to v2.2 Voice phase).

</domain>

<decisions>
## Implementation Decisions

### Tool Surface & Dispatch

- **D-01: Single dispatcher tool `web_extract`.** NOT split into multiple narrow tools (no `web_extract_links`, `web_extract_meta`, etc.). One LLM-visible tool, multi-URL input, internal classifier routes per URL. Compact LLM surface; multi-URL batching reduces turn-count when the agent fans out across sources.

- **D-02: Public signature is fixed (see <domain>).** `Vec<String>` input, `Vec<ExtractionResult>` output. Per-URL failures populate `ExtractionResult.error` rather than aborting the call — partial-success semantics are mandatory.

- **D-03: URL classification happens at dispatch time before any network call.** Three branches:
  1. **YouTube** — host matches `youtube.com` / `youtu.be` / `m.youtube.com` / `music.youtube.com` → D-09 skill dispatch.
  2. **PDF** — URL suffix `.pdf` (case-insensitive, ignoring query string) OR (after fetch) `Content-Type: application/pdf` → D-08 PDF handler.
  3. **Default web** — everything else → D-04 backend chain.

  Classifier is pure-string for YouTube + suffix-PDF; the content-type fallback for PDF requires a HEAD request OR detection on the response of the default-web fetch (defer the implementation choice to plan-phase, but the classifier interface MUST allow re-routing mid-fetch when content-type comes back as PDF).

### Backend Strategy (default web handler)

- **D-04: Backend chosen by `_get_backend()` based on env-var presence, priority: Firecrawl → Exa → Tavily → local.** Mirrors `web_read.rs` Firecrawl-first pattern verbatim. The local backend reuses `extract_content_local` (scraper + htmd) from `web_read.rs` — DRY constraint: NO copy-paste; refactor into a shared `crates/ironhermes-tools/src/web_local.rs` module exposed to both `web_read` and `web_extract`.

- **D-05: Backend-failure fallback returns truncated raw content (first 5K chars), NEVER a generic error.** Per PRD §4: "ensure the agent receives *some* context rather than a generic error." On total failure (raw fetch also fails), `ExtractionResult.error` is populated and `content` is empty — the LLM gets a structured signal without losing the call.

- **D-06: All three external providers (Firecrawl + Exa + Tavily) ship in this phase, env-gated.** Hand-rolled `reqwest` clients per provider; NO new SDK deps (matches `web_read.rs` Firecrawl pattern). Each provider's response is normalized to the common Markdown schema (D-07) inside its backend module before returning to the dispatcher.

  Provider env vars (all optional, presence drives priority):
  - `FIRECRAWL_API_KEY` — already used by `web_read` / `web_search`
  - `EXA_API_KEY` — NEW
  - `TAVILY_API_KEY` — NEW

### URL Normalization & Output Schema

- **D-07: All output normalized to Markdown schema `{url, title, content, metadata}`.** `ExtractionResult` exposes `url`, `title`, `content`; `metadata` is dropped at the public boundary (researcher: confirm whether to surface metadata as a JSON sub-field on `content` or extend the struct — defer the decision to plan-phase but flag it as open).

- **D-08: Base64 `data:image/...;base64,...` regex-stripped post-extract.** Single regex pass after Markdown normalization, before tiered summarization. Pattern: `data:image/[a-zA-Z0-9+.-]+;base64,[A-Za-z0-9+/=\n]+` replaced with `[image stripped]`. Lives in `sanitize::strip_base64_images` for unit-testability.

### PDF Handling

- **D-09: PDF backend = Firecrawl primary + `pdf-extract` 0.7 local fallback.** Mirrors `web_read.rs` two-backend pattern. Firecrawl's `/scrape` already extracts PDFs to Markdown when called with the same payload shape; local fallback uses `pdf-extract` 0.7 (pure Rust, no system deps, single-binary deployment preserved). When Firecrawl is unset OR returns non-Markdown for a PDF URL, the bytes are fetched directly via `reqwest`, validated against SSRF (D-19), and passed to `pdf_extract::extract_text()`. Title extraction: filename from URL path's last segment (sans `.pdf`); document title only if Firecrawl provides it.

### YouTube Handling

- **D-10: YouTube URLs dispatch to the existing `youtube-content` skill via the Phase 19 skills framework.** The skill ships in-tree at `./skills/media/youtube-content/SKILL.md`. Tool calls into `crates/ironhermes-core/src/skills.rs` runtime to execute the skill with the URL as input; skill returns transcript + title + description as Markdown. On transcripts-disabled videos, skill must return available metadata as fallback (per PRD §2.A).

  **Researcher MUST verify:** the canonical `name:` field in `./skills/media/youtube-content/SKILL.md` frontmatter (directory uses hyphens; the PRD reference `youtube_content` uses an underscore — confirm the dispatch key matches the SKILL.md frontmatter, not the directory name).

  The tool itself contains zero transcript-fetching code — implementation lives entirely in the skill, which can be replaced or disabled by operators without a binary release.

### Tiered Content Pipeline (LLM summarization)

- **D-11: Char-tier processing matches PRD §3 verbatim:**
  | Tier | Char range | Action |
  |---|---|---|
  | 1 | `< 5,000` | Return full Markdown (no LLM call) |
  | 2 | `5,000 – 500,000` | Single-pass aux-LLM summary |
  | 3 | `500,000 – 2,000,000` | Chunked: 100K-segment parallel summaries → final synthesis pass |
  | 4 | `> 2,000,000` | Refuse with `error: "content_too_large"`, content empty |

  Thresholds operator-overridable via the new `extract:` config block (D-22).

- **D-12: `use_llm_processing: false` short-circuits ALL aux-LLM tiers.** When the flag is false: tier 1 still returns raw Markdown; tiers 2/3 fall back to web_read-style truncation at `config.web.max_content_chars`; tier 4 still refuses (token-conscious operator still gets the size signal). Preserves a fully-deterministic, no-cost path.

### Aux-LLM Routing (Phase 26 cascade — second consumer)

- **D-13: Aux model resolved via `resolver.resolve_role("summarization")` from Phase 26 D-07.** Cascade order: per-task `summarization: { provider, model }` → `auxiliary: { provider, model }` → main provider. Phase 25.1's `browser_vision` was the **first** consumer of `resolve_role`; web_extract is the **second**. NO new aux config block — the existing `auxiliary:` block in `~/.ironhermes/config.yaml` is the single source of truth.

  When `resolve_role("summarization")` returns `None`, the cascade falls through to the main provider per Phase 26 D-07 contract — web_extract does NOT need its own None-handling logic.

- **D-14: Cache-awareness flag.** `summarization` role config changes are schema-altering for the model context (different aux model = different cost/latency curve). Phase 25 D-04's toolset-cache-break banner does NOT cover provider role changes. Flag for **Phase 27 (cache enforcement)** to surface a mid-session warning when the operator mutates `auxiliary.summary` while a session is live. Out of scope for 25.2; documented here so the v2.1 architectural-principle-#2 audit sees the dependency.

### Concurrency & Synthesis

- **D-15: Bounded `tokio::sync::Semaphore` controls all aux-LLM concurrency.** Single semaphore covers BOTH multi-URL parallelism AND chunk-summary parallelism within a single URL (one budget for the whole call). Default permits = 4 (overridable via `extract.max_parallel_summaries`). Prevents fan-out → 200 concurrent aux requests when the LLM passes 50 URLs each above 500K chars.

- **D-16: Synthesis pass is a single LLM call** over all chunk summaries joined with explicit chunk markers (`---\n[Chunk N/M]\n` between segments). Synthesis uses the same `resolve_role("summarization")` endpoint as the per-chunk passes — no separate "synthesizer" role.

- **D-17: Multi-URL parallelism uses the same Semaphore.** Each URL's full pipeline (fetch + extract + summarize tiers) is one semaphore-permit unit, NOT broken down per-LLM-call. Researcher: validate this is the right granularity vs per-LLM-call permits — log as a plan-phase open question if performance modeling suggests otherwise.

### Security

- **D-18: SSRF validation via existing `ironhermes_core::ssrf::is_safe_url` before any HTTP fetch.** Same hook as `web_read.rs`. Post-redirect re-validation when reqwest follows redirects (Phase 12 contract). Applies to PDF byte fetches, default-web fetches, and YouTube skill input (skill is responsible for its own outbound validation but the URL we hand it is pre-validated).

- **D-19: Secret-in-URL redaction rejects URLs containing common secret patterns BEFORE backend dispatch.** Patterns (extensible via `extract.redact_url_patterns`): case-insensitive match on `token=`, `api[_-]?key=`, `password=`, `secret=`, `bearer\s+`, `auth=`, `access[_-]?token=`. Pattern list lives as a `const &[&str]` in `sanitize::SECRET_URL_PATTERNS` for grep-ability. On match: tool returns `ExtractionResult { error: Some("url_contains_secret"), ... }` for that URL; no network call is made.

### Toolset Membership & Defaults

- **D-20: `web_extract` joins the existing `web` toolset alongside `web_read` and `web_search`.** `Tool::toolset()` returns `"web"`. `DEFAULT_TOOLSETS` already includes `web` (Phase 25 D-01) — new operators get the tool automatically. The `use_llm_processing: false` flag (D-12) is the per-call cost-gating mechanism; no toolset-level default-disabled is needed because the tool's no-cost path is reachable by the LLM itself.

- **D-21: No new toolset banner mechanism.** Schema-cache rebuild on tool registry changes inherits Phase 25 D-04 verbatim. Mid-session aux-routing changes are flagged for Phase 27 (D-14).

### Configuration

- **D-22: New `extract:` config block in `crates/ironhermes-core/src/config.rs`:**
  ```yaml
  extract:
    max_parallel_summaries: 4         # Semaphore permits (D-15)
    summary_chunk_chars: 100000       # Tier-3 chunk size (D-11)
    refuse_threshold_chars: 2000000   # Tier-4 cutoff (D-11)
    summary_tier2_threshold_chars: 5000      # Tier 1→2 boundary
    summary_tier3_threshold_chars: 500000    # Tier 2→3 boundary
    redact_url_patterns:              # D-19 — appends to const default set
      - "token="
      - "api_key="
      - "bearer "
  ```
  All fields `#[serde(default)]` for backward compat. The `summarization` role lives under the existing Phase 26 `auxiliary:` block (D-13), NOT here. Per-URL cost knobs (the `use_llm_processing: false` flag) are tool-call args, NOT config.

### Test Strategy

- **D-26: Mandatory integration tests (8 total) — wiremock-backed, NO live network:**

  4. `web_extract_summary_tier_thresholds` — synthetic content at 4K / 50K / 600K / 2.5M chars; asserts (i) tier 1 raw, (ii) tier 2 single aux call, (iii) tier 3 chunk-count + synthesis call counts, (iv) tier 4 refuse with `error: "content_too_large"`.
  5. `web_extract_use_llm_processing_false_skips_all_aux_calls` — 600K-char content with the flag false; asserts zero aux LLM calls and truncated output at `config.web.max_content_chars`.
  6. `web_extract_summarization_role_resolves_via_phase26_cascade` — configure `auxiliary.summary: { provider, model }` with two wiremock servers (main + aux); assert summary calls hit the aux server. End-to-end Phase 26 PROV-06 second-consumer validation.
  7. `web_extract_secret_in_url_redacted` — URL with `?token=abc` fails fast with `error: "url_contains_secret"`; assert zero network calls.
  8. `web_extract_multi_url_partial_failure_returns_per_url_errors` — 3 URLs, middle wiremock returns 500; assert `Vec[0].content` populated, `Vec[1].error` populated, `Vec[2].content` populated. Partial-success contract (D-02).

Test asserts `web_extract` schema is ALWAYS in `get_definitions()` output regardless of env-var state. Contrasts with `web_search` (Firecrawl-required) and `browser_*` (Chromium-required) gating models.

## Specific Ideas

- **`web_extract` argument shape (locked):**
  ```json
  {
    "urls": ["https://arxiv.org/abs/2401.12345.pdf", "https://youtu.be/abc"],
    "format": "markdown",
    "use_llm_processing": true,
    "min_length": 200
  }
  ```

- **`ExtractionResult` JSON shape (per-URL):**
  ```json
  {
    "url": "https://example.com/article",
    "title": "How to extract the web",
    "content": "# How to extract the web\n\n...",
    "error": null
  }
  ```

- **Refuse envelope (tier 4):**
  ```json
  {
    "url": "https://example.com/giant",
    "title": "",
    "content": "",
    "error": "content_too_large"
  }
  ```

- **Secret-redaction error envelope (D-19):**
  ```json
  {
    "url": "https://example.com?token=REDACTED",
    "title": "",
    "content": "",
    "error": "url_contains_secret"
  }
  ```

- **YouTube dispatch result shape:** the skill returns Markdown like:
  ```markdown
  # Video Title

  **Channel:** Author Name
  **Description:** ...

  ## Transcript

  [00:00] Speaker text...
  ```
  web_extract uses the first `# ` heading as `title`; the rest goes into `content`. If transcripts are disabled, the skill returns metadata-only Markdown without the `## Transcript` section.

- **Synthesis prompt shape (D-16):** simple — "Summarize the following N chunk summaries into one coherent overview. Preserve key facts, numbers, and named entities. Output Markdown."

- **`min_length` semantics:** when extracted content is shorter than `min_length` chars, the result still populates `content` (so the agent sees the snippet) but adds a `metadata.warning: "content_below_min_length"` marker. Does NOT fail the URL. PRD §5 leaves this ambiguous; this is the conservative interpretation. Researcher: validate against the Python source.

- **`format: "html"` semantics:** when set, the local backend skips htmd Markdown conversion and returns raw HTML in `content`. Firecrawl/Exa/Tavily backends still return Markdown (their APIs don't reliably surface raw HTML). Document the inconsistency in the schema description so the LLM knows.

- **chromiumoxide / browser_session NOT used.** Even though Phase 25.1 is available, web_extract stays HTTP-only. JS-rendered pages are the LLM's responsibility (it can call browser_navigate + browser_snapshot if needed). This keeps web_extract fast, stateless, and binary-singleton-friendly.
