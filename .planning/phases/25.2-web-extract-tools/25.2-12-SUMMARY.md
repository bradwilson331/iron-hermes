## Decisions Made

| Decision | Rationale | Outcome |
| -------- | --------- | ------- |
| Inline Firecrawl→Exa→Tavily→Local chain in `fetch_web_with_chain` (not `select_backend()`-driven match) | The enum-based selector returns ONE backend; the dispatcher needs to TRY the first, fall through to the second on Err, and emit `warn!("Firecrawl failed for {url}: {err}; trying Exa")` between each step. Inline `if let Ok(...) = …` per provider is the only structure that captures both behaviors. `select_backend()` remains useful for single-shot tests. | ✓ Good |
| `tokio::spawn` per URL instead of `futures::join_all` | Per-URL `JoinHandle` lets us detect panics via `h.await` Err arm and synthesize an `ExtractionResult::error` for that URL without dropping the entire call — Pitfall 6 ordering preserved by tagging tasks with `(idx, result)` and sorting before serialize. | ✓ Good |
| `Config::load().unwrap_or_default()` | Tool must NEVER break on a malformed config.yaml — that's an operator boot-time problem, not a per-call problem. Default `ExtractConfig` carries the documented thresholds (max_parallel_summaries=4, refuse_threshold_chars=2_000_000, etc.). | ✓ Good |
| Defer end-to-end smoke test to Plan 14 | Plan's NOTE block explicitly defers; the smoke needs a real `AnyClientSummarizationHandle` from `ironhermes-agent` which Plan 12 cannot construct without crossing the dep boundary. Compile-time signature-lock test stands in for it locally. | ✓ Good |
| Compile-time `register_web_extract_tool_signature_locked` test | Cheaper than a runtime invocation test; catches signature drift at the build step. The test body is just a function-pointer cast — zero runtime cost. | ✓ Good |
| Use Rust 2024 let-chains for the `is_some_and(reroute_for_pdf) && let Some(bytes) = …` collapse | Workspace edition is 2024; let-chains stable; idiomatic and clippy-clean. Avoids the `collapsible_if` warning a nested `if … { if let Some(...) = … }` would produce. | ✓ Good |

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Replaced `.entered()` with async-block + `.instrument(span)` in `tiers.rs` and `chunked.rs`**

- **Found during:** First `cargo check -p ironhermes-tools` after writing `WebExtractTool::execute`.
- **Issue:** Plan 11's `tiers.rs` and `chunked.rs` introduced `let _span = info_span!(…).entered()` bindings whose `EnteredSpan` is `!Send`. Plan 12's `WebExtractTool::execute` wraps the per-URL pipeline in `tokio::spawn(async move { … route_tiers(...).await … })`, and tokio's spawn requires `Future + Send + 'static`. The `_span` binding was held across the inner `sem.acquire().await` and `client.summarize_call(...).await` points, so the entire spawned future became `!Send` and the build failed with `error: future cannot be sent between threads safely`.
- **Fix:** In both files, wrapped the function body in `async move { … }.instrument(outer_span).await` and changed the inner `_aux_span = info_span!(...).entered()` / `_syn_span = info_span!(...).entered()` to `info_span!(...)` + `.instrument(aux_span)` on the awaited future. Span attachment is preserved (tracing observes the same scope), but the `EnteredSpan` is never bound across an await, so the future remains `Send`. Logic is byte-identical otherwise.
- **Files modified:** `crates/ironhermes-tools/src/web_extract/summary/tiers.rs`, `crates/ironhermes-tools/src/web_extract/summary/chunked.rs`.
- **Verification:** All 9 pre-existing summary tier+chunked tests still pass; the broader `cargo test -p ironhermes-tools --lib` reports `286 passed; 0 failed`. `cargo check -p ironhermes-tools` clean.
- **Committed in:** `149c9a1` (Task 1 commit — atomic with the WebExtractTool::execute impl that exposed the bug, since they must land together to keep `develop` green).
