## PDF → Markdown backend

| Option | Description | Selected |
|--------|-------------|----------|
| Firecrawl `/scrape` (already extracts PDFs) | Zero new workspace deps. Loses local fallback when FIRECRAWL_API_KEY is unset. | |
| `pdf-extract` 0.7 pure-Rust | Pure-Rust, no system deps, works offline. Single workspace dep added. | |
| `pdfium-render` (Chromium PDFium bindings) | Highest fidelity but requires shipping the PDFium shared library or relying on Chromium availability — breaks single-binary. | |
| **Both: Firecrawl primary + `pdf-extract` local fallback** | Mirrors web_read.rs Firecrawl-first / scraper-fallback pattern verbatim. Maximum coverage; one extra dep. | ✓ |

**User's choice:** Both — Firecrawl primary + `pdf-extract` local fallback.
**Notes:** Chosen for symmetry with the existing web_read.rs dual-backend pattern. The local fallback ensures the tool keeps working when no API keys are configured — important for first-run UX and CI environments without billable accounts. `pdf-extract` 0.7 is pure-Rust, preserving the IronHermes single-binary deployment constraint (PROJECT.md §Constraints).

## Toolset placement & default-enabled state

| Option | Description | Selected |
|--------|-------------|----------|
| Extend `web` toolset, default-enabled | Sits next to web_read/web_search. Auto-enabled in DEFAULT_TOOLSETS. Operators get it for free. Cost guardrails are per-call (`use_llm_processing: false` short-circuits LLM tiers). Same precedent as web_read which can also call billable Firecrawl. | ✓ |
| New `extract` toolset, default-disabled | Phase 25 D-20 high-blast-radius precedent (browser, code). Operators opt in via `hermes toolset enable extract`. Justified if LLM tiers are considered too expensive to be on by default. | |
| Extend `web` toolset, default-disabled | Sits in `web` for grouping but ships off until the operator opts in. Likely too disruptive given web_read/web_search already ship enabled. | |

**User's choice:** Extend `web` toolset, default-enabled.
**Notes:** The per-call `use_llm_processing: false` flag is the runtime cost gate; operators don't need a toolset-level off switch when the LLM itself can choose the no-cost path. Treats web_extract as a peer of web_read rather than a high-blast-radius newcomer like browser_*.

## Claude's Discretion

The following decisions were made by Claude based on codebase patterns and PRD inference, NOT explicitly answered by the user. They are captured in CONTEXT.md but are open to objection before plan-phase:

- **D-04 DRY refactor:** Extracting `extract_content_local` from `web_read.rs` into a shared `web_local.rs` module instead of duplicating the function. Drives a small refactor of web_read.rs that wasn't in the PRD scope.
- **D-15 Single shared Semaphore:** Multi-URL parallelism AND chunk-summary parallelism share one `tokio::sync::Semaphore` budget rather than separate budgets. Prevents fan-out blowup when the LLM passes many URLs each with large content. The user's PRD did not specify the granularity.
- **D-17 Per-URL semaphore granularity:** Each URL's full pipeline (fetch + extract + summarize tiers) is one permit unit, NOT broken down per-LLM-call. Flagged as an open question for plan-phase performance modeling.
- **D-19 Secret-URL pattern set:** The specific regex set (`token=`, `api[_-]?key=`, `password=`, `secret=`, `bearer\s+`, `auth=`, `access[_-]?token=`) is Claude's interpretation of "block URLs containing sensitive tokens" from PRD §4. Operator-extensible via `extract.redact_url_patterns`.
- **D-22 Config block layout:** New top-level `extract:` block name and field structure inferred from PRD requirements; no canonical config name was specified.
- **D-23 Internal module structure:** `dispatch / backends / pdf / youtube / summary / sanitize` submodule layout chosen to mirror Phase 25.1 file-per-tool conventions while keeping web_extract as a single tool implementation file. Alternatives (e.g., one file per backend) were considered but rejected as over-engineering for v2.1.
- **D-26 Eight integration tests:** Test list size and scope inferred from Phase 25.1's three-test minimum + Phase 26 cascade test pattern. The user did not specify a test budget.
- **D-27 web_extract is ALWAYS available:** Schema-exclusion test inverts the Phase 25.1 pattern because the local fallback (scraper+htmd) needs no env var. This is a deliberate divergence from web_search.rs's strict env-gating; if the user wants stricter availability, raise before plan-phase.
- **D-28 ROADMAP goal text:** Drafted to capture the PRD as a one-paragraph summary; plan-phase will write it into ROADMAP.md.
