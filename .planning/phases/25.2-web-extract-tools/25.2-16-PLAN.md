<objective>
Close UAT Issue 9 (secret leak in agent warning) by landing all four sub-fixes from
the diagnoser's `.planning/debug/issue9-secret-leak-in-agent-warning.md` report
**atomically** in a single plan: (1) add the missing `redact_secrets_in_url` transform
primitive (Plan 04 only shipped a `contains_secret` predicate); (2) use the redactor
on the `ExtractionResult.url` echo path in `web_extract.rs`; (3) audit & redact every
`warn!(... url ...)` log site in the web_extract module tree where a URL field is
passed as a `{}` argument (web_extract.rs:322/332/342/375 and pdf.rs:50); (4) append
a "Secret handling" rule to `DEFAULT_AGENT_IDENTITY` so the LLM is shaped to redact
rather than echo secret-looking tokens it sees in user input.

The diagnoser confirmed two cooperating root causes:

- **R1 (primary):** The IronHermes system prompt has ZERO guidance on secret handling.
  `DEFAULT_AGENT_IDENTITY`, `TOOL_USE_GUIDANCE`, and docker/SOUL.md all lack any
  redaction rule. With no shaping, the model falls back to its trained reflex: refuse
  + warn + quote the offending token verbatim. That is exactly what UAT Test 9
  reproduced: `"⚠️ Hold on! That URL contains what appears to be an API key
  (sk-or-v1-fakekeyabc123) in the query string."` — the warning literally echoed the
  secret.

- **R2 (structural):** Plan 04's sanitizer is a PREDICATE (`contains_secret -> bool`),
  not a REDACTOR. No `redact_secrets_in_url(s) -> String` primitive exists. The
  sanitizer only fires inside `web_extract` tool execution at web_extract.rs:236-237
  — and the LLM short-circuits before calling the tool, so it never runs. Even on the
  success path, `ExtractionResult.url` round-trips the original (unsanitized) URL back
  to the model, so the leak surface is wider than the failed test demonstrates.

This plan addresses BOTH cooperating root causes. R1 is the system-prompt rule (Task 3).
R2 is the redactor primitive + use sites (Tasks 1 + 2). Both must land together — landing
just one leaves the other half of the leak surface open.

**Defense-in-depth:** Even after the system-prompt rule lands, a misbehaving model could
still echo a secret. The redactor primitive guarantees that even if the model parrots
back what `web_extract` told it about a URL, the value the model SEES from the tool
output is already redacted. Two independent layers, neither alone sufficient.

**Scope decision (B-1 audit, post-revision-1):** `youtube.rs:81` was originally listed
as a leak site. Re-read of the actual code shows the warn! site logs
`output.status` and `stderr` from the yt-dlp helper, NOT a URL field directly:
```rust
warn!("youtube-content helper exited {}: {}", output.status, stderr);
```
The URL CAN appear inside `stderr` if yt-dlp echoes it on failure, but that is
downstream of this plan's scope (it is the helper's responsibility to redact its
own error output, or this plan would need to redact `stderr` content rather than
a URL field). This plan **drops youtube.rs from `files_modified`** to keep scope
honest and focused on URL-field log sites. The residual stderr-leak risk in
youtube.rs is logged in the SUMMARY's "Next Phase Readiness" as a follow-up
debug task.

Output: One new public function in sanitize.rs (with 8 tests), one rewrite of the
ExtractionResult.url echo path in web_extract.rs (with one integration-style test),
5 warn! log site rewrites across 2 files (web_extract.rs + pdf.rs), one
multi-line append to DEFAULT_AGENT_IDENTITY in prompt_builder.rs.
