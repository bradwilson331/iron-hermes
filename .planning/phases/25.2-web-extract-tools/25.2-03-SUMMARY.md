# Phase 25.2 Plan 03: D-13 SummarizationClientHandle in ironhermes-core::provider Summary

**One-liner:** Defined the `SummarizationClientHandle` async trait in `ironhermes-core::provider` (Send + Sync + dyn-compatible) and re-exported it from `lib.rs` so `WebExtractTool` (in `ironhermes-tools`) can hold `Arc<dyn SummarizationClientHandle>` for the Phase 26 `resolve_role("summarization")` cascade without forming the forbidden `tools → agent` circular dependency.

## What Was Built

### Trait definition (CONTEXT D-13)

A new public trait at the bottom of `crates/ironhermes-core/src/provider.rs` (between the `impl ProviderResolver` block and the `#[cfg(test)] mod tests` block):

```rust
#[async_trait::async_trait]
pub trait SummarizationClientHandle: Send + Sync {
    async fn summarize_call(
        &self,
        system_prompt: String,
        user_prompt: String,
        max_tokens: u32,
    ) -> anyhow::Result<String>;
}
```

The trait is preceded by a 25-line doc comment that:

- Cites the `VisionClientHandle` (browser_vision.rs:51) as the canonical sibling pattern.
- Explains the cycle-break rationale (`tools` → `core` ← `agent` is acyclic; `tools` → `agent` would not be).
- Codifies the cascade contract: implementer MUST call `ProviderResolver::resolve_role("summarization")` first; on `None`, MUST fall back to `ProviderResolver::resolve_for_main`. The consumer never sees `None`.
- Documents the parameter semantics: `system_prompt`, `user_prompt`, `max_tokens` (with tier-2 ≈ 20K / tier-3-chunk ≈ 5K / tier-3-synthesis ≈ 20K caller guidance).
- Documents the error-propagation semantics: `Err` becomes per-URL `ExtractionResult.error` per D-02 partial-success contract.

### Compile-only dyn-compatibility test

Added inside the existing `#[cfg(test)] mod tests` block in `provider.rs` (after the Phase 26 Plan 02 cascade tests):

```rust
#[test]
fn summarization_client_handle_is_dyn_compatible() {
    fn _accepts(_: std::sync::Arc<dyn super::SummarizationClientHandle>) {}
}
```

Locks `Arc<dyn SummarizationClientHandle>` acceptance into a regression test — if the trait ever loses `Send + Sync` (e.g., a future refactor adds an associated non-`Send` type), this test fails to compile and surfaces the breakage before downstream plans hit it.

### Re-export from lib.rs

## Key Decisions

- **Trait insertion site:** chose immediately AFTER `impl ProviderResolver { ... }` (closing `}` at line 453 in the original file) and BEFORE the `// Tests` section header. Rationale: keeps the trait visible at a top-level section break, mirrors the file's other section-header convention (`// =====`) for `ResolvedEndpoint`, `ProviderResolver`, and now `SummarizationClientHandle`. The dyn-compatibility unit test is the LAST test in the existing `#[cfg(test)] mod tests` block, so the trait + its test stay close.
- **`#[async_trait::async_trait]` fully-qualified form** chosen over adding `use async_trait::async_trait;` at the top of `provider.rs`. Rationale: (a) `async-trait` is already a workspace dep declared in `crates/ironhermes-core/Cargo.toml:21`, no Cargo.toml edit; (b) keeps the provider.rs import block byte-identical with prior plans, minimizing merge friction with any concurrent edits to the same file; (c) the trait sits in its own section, so the one-line FQN is locally readable.
- **Re-export style:** appended `SummarizationClientHandle` to the existing `pub use provider::{ProviderResolver, ResolvedEndpoint};` brace-list rather than adding a sibling `pub use provider::SummarizationClientHandle;` line. Rationale: matches Plan 25.2-02's pattern (which inserted `ExtractConfig` into the existing `pub use config::{...}` brace-list at line 21-25); preserves the file's combined-brace convention; both approaches were enumerated as Option A / Option B in the plan, and Option A keeps the `provider::` re-exports contiguous.
- **Doc-comment depth:** wrote a 25-line doc comment (vs the plan's minimum-shape ~6-line block) per Rule 2 critical-correctness — without the cascade-contract documentation, future implementers in `ironhermes-agent` could write a non-cascading `summarize_call` (e.g., direct main-provider call) and silently break the per-task `summarization` role override that Phase 26 D-07 + Plan 25.2-02 D-13 designed for. The doc explicitly enumerates the implementer's two-level fallback obligation.
