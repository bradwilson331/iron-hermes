# Phase 25.2 Plan 06: D-02 ExtractionResult + D-06 Firecrawl Backend Summary

**One-liner:** Defines the canonical `ExtractionResult` struct (url/title/content/error) used by every later backend, and implements the Firecrawl scrape backend with SSRF pre-validation, bearer-auth, and a wiremock-friendly endpoint override — mirrors `web_read.rs::fetch_with_firecrawl` shape verbatim, only differing in return type.

## Accomplishments

- `pub struct ExtractionResult` exported from `crate::web_extract` — the canonical per-URL extraction outcome that Plans 07–13 will all return; derives `Debug + Clone + Serialize + Deserialize` so it can cross crate boundaries cleanly.
- `ExtractionResult::error(url, code)` constructor builds the partial-success error envelope with empty `title` + empty `content` and `error: Some(code)` — used downstream by the dispatcher for D-19 secret-redaction and D-11 tier-4 refuse paths.
- `pub async fn fetch_with_firecrawl(url: &str) -> Result<ExtractionResult>` implemented in `web_extract::backends::firecrawl`:
  - **SSRF pre-validation FIRST** (line 1 of fn body) via `crate::web_local::validate_url_async` — T-25.2-03 mitigation enforced structurally.
  - **POST `https://api.firecrawl.dev/v1/scrape`** with `bearer_auth(&FIRECRAWL_API_KEY)` and `{ "url": url, "formats": ["markdown"] }` body — verbatim match for web_read.rs:184.
  - **Config::load fallback to 30s timeout** — `.timeout(Duration::from_secs(timeout_secs))` set on the reqwest client per T-25.2-10 DoS mitigation.
  - **D-07 Option B inline Markdown header** prepended: `# {title}\nSource: {url}\n\n` (or `Source: {url}\n\n` when title is empty) — matches web_read.rs:159 precedent.
  - **Returns Err on backend failure** so the Plan 13 dispatcher can fall through Firecrawl → Exa → Tavily → Local per D-04; only the dispatcher converts a final Err into `ExtractionResult.error`.
- `FIRECRAWL_ENDPOINT_OVERRIDE` env var hook via `resolve_endpoint()` helper — Plan 14's wiremock-backed firecrawl_path test will set this to a wiremock base URL without going to production Firecrawl.
- 3 unit tests pass (2 ExtractionResult + 1 endpoint helper); cargo build clean; clippy clean on the new module surface.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Critical] `#[allow(dead_code)]` added to `FirecrawlMetadata.status_code` field**

- **Found during:** Task 2 (after first cargo build)
- **Issue:** The plan's `<interfaces>` block specified the `status_code: Option<u16>` field on `FirecrawlMetadata`, mirroring `web_read.rs:35-37`, but Plan 06's `fetch_with_firecrawl` does not consume the field (web_read.rs's >=400-status guard is out of scope here per the plan's <action> block, which copies only the simpler shape). Without `#[allow(dead_code)]`, this would have emitted a new compiler warning on a field that is intentionally future-reserved. Pre-existing crate baseline already has 1 `dead_code` warning (`DEFAULT_SAFE_TOOLS` in delegate_task.rs documented in `deferred-items.md`), so adding a second silent dead_code warning would obscure the inventory.
- **Fix:** Added `#[allow(dead_code)] // Reserved for future >=400 status detection (web_read.rs:174 pattern).` to the `status_code` field. Schema parity preserved without polluting the warning baseline.
- **Files modified:** `crates/ironhermes-tools/src/web_extract/backends/firecrawl.rs`
- **Verification:** `cargo build -p ironhermes-tools --tests 2>&1 | grep -E 'firecrawl|status_code'` returns zero matches; only the pre-existing `result_str` warning in `tests/hub_search_test.rs:167` remains (already in `deferred-items.md`).
- **Committed in:** `e95515d` (Task 2 commit, batched with the backend impl since the attribute is part of the same atomic struct definition)
