# Phase 25.2 Plan 08: D-04 / D-18 Local Pure-Rust Backend Summary

**One-liner:** Implements `fetch_local_content(url, web_cfg) -> LocalFetchOutcome` — the always-available D-04 fallback that GETs the URL via reqwest, runs SSRF pre-validation + post-redirect re-validation (D-18), detects `application/pdf` Content-Type to enable Plan 13's mid-fetch PDF reroute (D-03) without a second network hop, and otherwise hands the bytes to the shared `web_local::extract_content_local` helper for HTML→Markdown conversion.

## Accomplishments

- `pub async fn fetch_local_content` exported from `web_extract::backends::local` — the Plan 13 dispatcher's terminal D-04 fallback (`Firecrawl → Exa → Tavily → Local`). No env-var prereq → matches D-27 ("local fallback is unconditionally available").
- `pub struct LocalFetchOutcome { result, content_type, raw_bytes }` exposes the three pieces of state Plan 13 needs to decide HTML-vs-PDF routing without re-fetching:
  - `result: ExtractionResult` — populated with Markdown + extracted title on the HTML path; empty (no error) on the PDF path so the dispatcher knows to delegate to `pdf::extract_pdf_bytes(raw_bytes)`.
  - `content_type: Option<String>` — surfaced verbatim from the `Content-Type` response header so the dispatcher can call `dispatch::reroute_for_pdf(&content_type)` (Plan 25.2-05 helper).
  - `raw_bytes: Option<Vec<u8>>` — the bytes already on hand from the body read; lets the PDF handler re-use them with zero extra HTTP traffic.
- **D-18 SSRF defence in depth:** `validate_url_async(url)` runs as the first statement of the fn body (T-25.2-03 mitigation enforced structurally). On reqwest's default-follow-redirects path, `final_url != url` triggers a second `validate_url_async(&final_url)` — closes the post-redirect SSRF bypass that web_read.rs:142-150 mitigates.
- **D-03 mid-fetch PDF detection:** `application/pdf` Content-Type (with tolerant primary-token parse for `application/pdf; charset=binary`-style variants) returns `LocalFetchOutcome` with empty `result.content` and `Some(bytes)` in `raw_bytes` — the dispatcher (Plan 13) gets a structurally distinguishable PDF signal without a HEAD round-trip.
- **D-05 backend-failure contract honoured:** HTML extraction failures return `Err(anyhow!("local HTML→Markdown extraction failed: ..."))` so the Plan 13 dispatcher can apply the 5K-truncated raw fallback per D-05.
- 2 helper unit tests pass:
  - `extract_h1_pulls_title_from_markdown` — confirms `# Hello\n...` → `"Hello"`
  - `extract_h1_returns_empty_when_no_header` — confirms no-header markdown → `""`
- Plan 14's wiremock-backed `web_extract_single_url_local_fallback_returns_markdown` will exercise the full network path; that integration coverage is the canonical contract for `fetch_local_content`'s real I/O behaviour.
- `cargo check -p ironhermes-tools` clean (only the pre-existing `DEFAULT_SAFE_TOOLS` dead_code warning that's already in `deferred-items.md`).
- `cargo clippy -p ironhermes-tools --lib --no-deps` produces zero warnings touching `local.rs` (9 pre-existing warnings in unrelated files — same baseline Plan 06 / Plan 07 documented).
