# Phase 25.2 Plan 11: Tiered Summarization Pipeline (D-11..D-17) Summary

**One-liner:** Replaced the Wave-0 stubs in `web_extract/summary/{tiers,chunked}.rs` with the full D-11..D-17 char-tier dispatch pipeline — `route_tiers` is the 4-branch decision tree (raw <5K / single-pass aux 5K-500K / chunked synthesis 500K-2M / refuse >2M), and `summarize_chunked` provides the tier-3 path (parallel `tokio::spawn` over `extract_cfg.summary_chunk_chars` segments, in-order collection, single synthesis LLM call) — both consume `Arc<dyn SummarizationClientHandle>` (Plan 25.2-03) and share one `Arc<Semaphore>` (D-15 single-budget) with the multi-URL outer fan-out that Plan 12 will hold.

## What Was Built

### Tier Router (`tiers.rs`)

A new `pub async fn route_tiers(content, use_llm_processing, extract_cfg, web_cfg, client, sem) -> Result<String>`:

1. **Tier 4 first (D-11 + D-12 — tier-bypass mitigation):** if `n > extract_cfg.refuse_threshold_chars` (default 2M) → `Err("content_too_large")` regardless of `use_llm_processing`. The tier-4 check happens BEFORE the use_llm short-circuit so an operator cannot bypass the 2M cap by passing `use_llm_processing: false`.
2. **Tier 1 (raw):** if `n < extract_cfg.summary_tier2_threshold_chars` (default 5K) → `Ok(content.to_string())` — no LLM call.
3. **D-12 short-circuit:** if `!use_llm_processing` and we got here (5K ≤ n ≤ 2M) → `Ok(truncate_content(content, web_cfg.max_content_chars))`.
4. **Tier 2 (single-pass):** if `n ≤ extract_cfg.summary_tier3_threshold_chars` (default 500K) → acquire one sem permit, `client.summarize_call(SUMMARY_SYSTEM, content, TIER2_MAX_TOKENS=20_000)`.
5. **Tier 3 (chunked + synthesis):** delegate to `chunked::summarize_chunked(content, extract_cfg, client, sem)`.

Plus 5 module-pub constants (`SUMMARY_SYSTEM`, `SYNTHESIS_SYSTEM`, `TIER2_MAX_TOKENS`, `TIER3_CHUNK_MAX_TOKENS`, `TIER3_SYNTHESIS_MAX_TOKENS`) used by `chunked.rs` and (later) Plan 12 telemetry.

Tracing spans: `web_extract.summary.tier{chars}`, `web_extract.summary.aux_call{tier, max_tokens}` — wired in for Plan 12's instrumentation hooks.

### Chunked Tier (`chunked.rs`)

A new `pub async fn summarize_chunked(content, extract_cfg, client, sem) -> Result<String>`:

1. **Split:** `split_into_chunks(content, extract_cfg.summary_chunk_chars)` produces N owned `String` segments. UTF-8 safe via `chars()` iteration (each char appended to the current chunk, swap-on-threshold via `std::mem::take`).
2. **Spawn:** for each chunk, `tokio::spawn` an async task that `sem.acquire_owned().await`s a permit, then calls `client.summarize_call(prompt, chunk_str, TIER3_CHUNK_MAX_TOKENS=5_000)`. Permits drop at task end.
3. **Collect IN ORDER:** sequentially `handles[i].await` and prepend each summary with `---\n[Chunk {i+1}/{total}]\n` (D-16 marker).
4. **Synthesize:** acquire a fresh sem permit (after chunk permits released), call `client.summarize_call(SYNTHESIS_SYSTEM, joined, TIER3_SYNTHESIS_MAX_TOKENS=20_000)`.

Tracing spans: `web_extract.summary.chunked{chunks}`, `web_extract.summary.synthesis{chunk_count}`.

## Key Decisions

- **Cross-task dependency forced single-commit landing.** The plan's split into Task 1 (tiers) + Task 2 (chunked) presumed they were independent enough to commit separately. They are not — tiers.rs contains a hard-coded `chunked::summarize_chunked(...)` call that requires chunked.rs to define the function. Committing tiers.rs alone would break `develop` mid-Wave 3, jeopardizing the parallel Plan 25.2-09/10 merges. Single atomic commit preserves a green main; documented as Rule 3 deviation #1.
- **Sequential `handles[i].await` collection (not `join_all`).** Each `tokio::spawn` runs concurrently (limited by sem permits), so all chunks execute in parallel — but collection happens sequentially in index order so the `[Chunk N/M]` markers stay correct without a sort. Avoids pulling in `futures` crate (zero new workspace deps preserved).
- **Synthesis acquires its own permit AFTER chunks complete.** All chunk permits drop before the synthesis acquire — guarantees no permit-deadlock even at `max_parallel_summaries=1` (single permit) where all 6 chunks would serialize but the synthesis pass still gets its turn.
- **Owned `String` chunks via `mem::take`.** `split_into_chunks` returns `Vec<String>` (not `Vec<&str>`) because the spawned tasks need `'static` futures — owned strings move into the closure cleanly.
- **System prompt re-import via `super::tiers::{...}`.** chunked.rs imports `SUMMARY_SYSTEM` (unused but explicit `let _ =` suppresses dead-code from D-16 design intent doc), `SYNTHESIS_SYSTEM`, and the two TIER3 constants from tiers.rs — keeps prompt strings as a single source of truth.
- **Tests use `std::sync::Mutex`** for the in-test handle's call-counter (not `tokio::sync::Mutex`) — short critical sections, no `.await` inside locks, matches the pattern from Plan 25.2-03's dyn-compatibility test.
