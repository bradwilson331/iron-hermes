# Phase 25.6 Research: Replicate CLI/Web Wiring

**Date:** 2026-05-04  
**Mode:** Standard research (codebase-first, no new external dependencies)

## Question

How do we give web sessions the same tool/guardrail/hook/MCP wiring as CLI, while preventing long-term drift and preserving startup/per-session lifecycle boundaries?

## Findings

1. **CLI wiring is duplicated across four entry points** (`run_single`, `run_chat`, `run_gateway`, `tui_rata::build_app_deps`) with the same registration sequence and hook setup pattern.
2. **Web UI currently re-initializes app state per WebSocket connection** (`ws.rs` calls `AppState::init()` inside `on_upgrade`) which violates app-scoped lifecycle decisions.
3. **Web API surfaces are partially stubbed** (`list_sessions`, `list_tools`) and do not consume the startup `AppState` runtime.
4. **`ironhermes-agent` already depends on `ironhermes-tools`, `ironhermes-hooks`, `ironhermes-state`**, making it the right in-process home for shared factory logic (per D-02).
5. **No new external package is required**; this is an extraction + wiring-parity phase, not a dependency-selection phase.

## Recommended Architecture

- Introduce a shared **app-scoped runtime factory** in `ironhermes-agent` that builds the parity bundle (tool registry wiring + guardrails + hook registry listeners + retry-queue drain + MCP startup bridge + skill registry wiring + browser/web_extract parity hooks).
- Keep **per-session/per-turn** concerns in consumers (CLI/web) via explicit turn/session context creation (`session_id`, persistence, cancellation, PromptBuilder turn wiring).
- Migrate both CLI and web to consume the same factory output, then lock parity with static and behavioral regression tests.

## Risks + Mitigations

- **Risk:** behavior drift while extracting duplicated wiring.  
  **Mitigation:** add ordering/parity regression tests at factory + CLI/web callsites.
- **Risk:** startup blocking due to hooks/MCP setup changes.  
  **Mitigation:** preserve existing background MCP startup semantics and current retry-queue drain behavior.
- **Risk:** web accidentally recreates app runtime per connection.  
  **Mitigation:** enforce startup-injected `AppState` usage in WS/API handlers and add tests.

## Decision

Proceed with a three-plan implementation:
1) shared factory in `ironhermes-agent`,
2) CLI migration to factory,
3) web migration to factory + startup-scoped state usage.
