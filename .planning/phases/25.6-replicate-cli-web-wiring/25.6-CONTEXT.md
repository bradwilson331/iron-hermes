# Phase 25.6: replicate-cli-web-wiring - Context

**Gathered:** 2026-05-04
**Status:** Ready for planning

<domain>
## Phase Boundary

Replicate the full CLI tool-wiring sequence into the web UI bootstrap so web sessions have the same tool access and runtime guardrails as CLI sessions, while extracting shared initialization into a reusable in-process factory.

This phase covers parity wiring for:
- file tools
- web search/read/extract
- browser tools
- memory_recall and related memory wiring
- skills
- delegate_task
- cronjob
- execute_code
- MCP registration/discovery
- BlocklistGuardrail
- HookRegistry (JSONL + webhook listeners)

It also covers removing copy-pasted wiring drift by introducing a shared factory used by both CLI and web.

Out of scope:
- introducing a separate localhost proxy backend process for this phase
- introducing a new `ironagent-tools-api` crate in this phase

</domain>

<decisions>
## Implementation Decisions

### Shared Factory Boundary
- **D-01:** Phase 25.6 uses a **shared in-process factory** (no network hop) to provide CLI-equivalent wiring for web.
- **D-02:** The shared factory lives in **`ironhermes-agent`** for this phase.
- **D-03:** The factory output must preserve CLI load-bearing registration order and behavior (tool registration, guardrails, hooks, MCP integration) so web and CLI do not drift.

### Lifecycle Scope
- **D-04:** Use a **hybrid lifecycle model**.
- **D-05:** **App-scoped**: tool registry, hook registry, MCP manager, skill registry, and browser session handle.
- **D-06:** **Per-session / per-turn**: `session_id`, message persistence, cancellation token, and turn-specific AgentLoop execution context.
- **D-07:** Web runtime should use startup-initialized shared state rather than recreating full app state per WebSocket connection.

### Hook and Guardrail Parity
- **D-08:** Mirror CLI behavior exactly for hook/guardrail wiring.
- **D-09:** Load `HooksConfig`, add `BlocklistGuardrail` when `blocked_tools` is non-empty, and set `error_detail` on the registry.
- **D-10:** Build `HookRegistry`, attach JSONL listener when enabled, attach webhook listeners when configured, and drain retry queue at startup.

### Prompt and Session Parity
- **D-11:** Web uses the same PromptBuilder-based turn setup path as CLI (provider/context/memory/skills).
- **D-12:** Persist session and message data to `StateStore` with `Platform::Web`, keeping web session namespace separate from CLI/Telegram.
- **D-13:** Keep slash/API behavior aligned with existing `CommandRouter` contracts.
- **D-14:** Web must receive the same high-level tool surface as CLI, including browser and web_extract parity wiring.

### the agent's Discretion
- Exact helper function names and internal struct layout for the shared factory return type.
- Whether to expose one composite builder or a small set of composable builders, as long as D-01..D-14 remain true.
- Exact placement of parity assertions/tests that lock wiring presence and ordering.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `build_registry_with_process_registry` and repeated wiring blocks in CLI provide the concrete parity blueprint.
- Existing registration calls (`register_delegate_task_tool`, `register_cronjob_tool`, `register_browser_tools_with_vision`, `register_skills_tool`, `register_web_extract_tool`, `register_execute_code_tool_with_process_registry`) can be reused via shared factory output.
- Existing hook setup pattern (`HooksConfig` -> `BlocklistGuardrail` -> `HookRegistry` listeners -> retry drain) is already stable and should be reused directly.

### Established Patterns
- Wiring order parity is treated as a correctness contract, not cosmetic refactor territory.
- Gateway-style shared infra + per-request/per-session runtime layering already exists and maps well to web server concurrency.
- Web uses a separate platform/session namespace (`Platform::Web`) while sharing backend components.

### Integration Points
- Shared factory call sites: CLI (`run_single`, `run_chat`, `run_gateway`, `tui_rata::build_app_deps`) and web (`AppState::init()`).
- WebSocket execution should consume startup-managed shared state and per-session context, not re-bootstrap full state each connection.
- API endpoints and WS handler must both operate on the same initialized app-scoped runtime artifacts.

</code_context>

<specifics>
## Specific Ideas

- Keep parity tests that assert presence/order of key registration calls across CLI and web wiring call sites.
- Treat the shared factory as a parity lock to prevent future drift as new tools are added.
- Preserve current web additive model: web is another surface, not a replacement for CLI/TUI.

</specifics>

<deferred>
## Deferred Ideas

- `localhost` proxy backend model (`/api` forwarding to `http://127.0.0.1:9119`) deferred to a future phase; it introduces a new service boundary and is outside 25.6 scope.
- New `ironagent-tools-api` crate deferred; phase 25.6 proceeds with in-process shared factory in `ironhermes-agent`.

</deferred>
