# Phase 21.5 Plan 01: Factory Config Loading and SQLite Arc Refactor Summary

**Factory loads provider-specific JSON config from $HERMES_HOME/<name>.json with graceful fallbacks; SQLite conn refactored to Arc<Mutex<Connection>> for tokio::spawn compatibility**

## Accomplishments
- Factory now reads `$HERMES_HOME/<provider_name>.json` and passes parsed config to `initialize()` for each provider
- Missing config files silently default to `Value::Null` (provider uses built-in defaults)
- Malformed JSON logs a tracing warning and falls back to `Value::Null`
- SqliteMemoryProvider conn field refactored from `Mutex<Connection>` to `Arc<Mutex<Connection>>` enabling `tokio::spawn` in Plan 02's `sync_turn`
- All existing tests pass unchanged after Arc refactor (Arc derefs through to Mutex)

## Files Created/Modified

- `providers/memory-sqlite/src/lib.rs` - Changed `conn` field from `Mutex<Connection>` to `Arc<Mutex<Connection>>`, updated constructor

## Decisions Made
- `load_provider_config` kept as module-private helper since only factory.rs calls it
- Per-arm config loading pattern: each provider match arm calls `load_provider_config(&hermes_home, "name")` independently rather than a single binding before the match
- Arc<Mutex<Connection>> chosen over tokio::sync::Mutex because rusqlite Connection is synchronous — std::sync::Mutex is the correct choice, Arc enables cloning into spawn closures
