# Phase 21.5: Code Review Report

**Reviewed:** 2026-04-21T12:00:00Z
**Depth:** standard
**Files Reviewed:** 10
**Status:** issues_found

## Summary

The Phase 21.5 memory provider plugin system introduces three pluggable memory backends (SQLite with FTS5, DuckDB columnar, Grafeo graph) behind the `MemoryProvider` trait, a factory with feature-gated construction, a `MemoryManager` write-path dispatcher with mirror support, and agent loop integration for provider-declared tools (e.g., `memory_recall`). The architecture is well-structured with frozen-snapshot patterns, security scanning, parameterized queries, and capacity enforcement consistently applied across all providers.

Two critical issues were found: a panicking byte-index truncation in the DuckDB recall snippet builder, and unescaped LIKE/ILIKE wildcard characters in SQL pattern construction across DuckDB and SQLite providers. Five warnings address missing error handling, a shared sequence across tables, and inconsistent wildcard escaping. Three informational items note code duplication opportunities.

## Critical Issues

### CR-01: Panicking byte-slice truncation on multi-byte UTF-8 content (DuckDB bridge)

**File:** `providers/memory-duckdb/src/bridge.rs:441`
**Issue:** The snippet builder uses `&content[..100]` which indexes by byte offset, not character offset. If the 100th byte falls in the middle of a multi-byte UTF-8 character (common with non-ASCII text, emoji, CJK), this will panic at runtime with `byte index 100 is not a char boundary`. The Grafeo provider correctly handles this at line 179 using `char_indices().nth(100)`, but DuckDB does not.
**Fix:**
```rust
let snippet = if content.len() > 100 {
    let end = content.char_indices()
        .nth(100)
        .map(|(i, _)| i)
        .unwrap_or(content.len());
    format!("{}...", &content[..end])
} else {
    content.clone()
};
```

### CR-02: LIKE pattern injection via unescaped wildcards in DuckDB replace/remove and SQLite replace/remove

**File:** `providers/memory-duckdb/src/bridge.rs:277,360`
**File:** `providers/memory-sqlite/src/lib.rs:484,565`
**Issue:** The `handle_replace` and `handle_remove` functions in both DuckDB and SQLite build LIKE patterns as `format!("%{}%", old_text)` without escaping SQL LIKE wildcard metacharacters (`%`, `_`). If user-supplied `old_text` contains `%` or `_`, the LIKE match will be broader than intended -- `%` matches any sequence and `_` matches any single character. This can cause the wrong entry to be matched and replaced/removed, leading to data corruption. Notably, the `handle_recall` function in DuckDB bridge (line 425) correctly escapes these characters with `query.replace('%', "\\%").replace('_', "\\_")`, but this escaping was not applied to replace/remove.
**Fix:** Apply the same escaping in all LIKE pattern constructions:
```rust
// DuckDB bridge.rs handle_replace (line 277) and handle_remove (line 360):
let escaped = old_text.replace('%', "\\%").replace('_', "\\_");
let pattern = format!("%{}%", escaped);

// SQLite lib.rs replace (line 484) and remove (line 565):
let escaped = old_text.replace('%', "\\%").replace('_', "\\_");
let pattern = format!("%{}%", escaped);
```

## Warnings

### WR-01: DuckDB queue_prefetch does not escape LIKE wildcards

**File:** `providers/memory-duckdb/src/bridge.rs:489`
**Issue:** `handle_queue_prefetch` builds a pattern as `format!("%{}%", query)` without escaping `%` and `_`. While this is a fire-and-forget cache-warming query with no user-visible result, it means a query containing `%` will scan more rows than intended, and the warm-up will not match the pattern used by `handle_recall` (which does escape). This is inconsistent and could mask issues during debugging.
**Fix:**
```rust
let escaped = query.replace('%', "\\%").replace('_', "\\_");
let pattern = format!("%{}%", escaped);
```

### WR-02: DuckDB conversation_facts shares memory_facts_seq leading to ID gaps

**File:** `providers/memory-duckdb/src/schema.rs:15`
**Issue:** The `conversation_facts` table uses `DEFAULT nextval('memory_facts_seq')` -- the same sequence as `memory_facts`. This means inserts into `conversation_facts` (via `on_pre_compress`) consume IDs from the `memory_facts` sequence, causing non-contiguous ID gaps in `memory_facts`. While not a correctness bug today (IDs are used for ordering, and `created_at` provides time ordering), this couples two independent tables and could cause confusion during debugging or if ID-based pagination is ever added.
**Fix:** Create a dedicated sequence for `conversation_facts`:
```sql
CREATE SEQUENCE IF NOT EXISTS conversation_facts_seq START 1;
CREATE TABLE IF NOT EXISTS conversation_facts (
    id          BIGINT DEFAULT nextval('conversation_facts_seq'),
    ...
);
```

### WR-03: DuckDB conversation_facts table has no PRIMARY KEY

**File:** `providers/memory-duckdb/src/schema.rs:14-19`
**Issue:** The `conversation_facts` table declares `id BIGINT DEFAULT nextval(...)` but does not mark it as `PRIMARY KEY`. This means the table has no primary key, which prevents efficient single-row lookups and could allow duplicate IDs if rows are inserted via other paths. The `memory_facts` table correctly uses `PRIMARY KEY`.
**Fix:**
```sql
CREATE TABLE IF NOT EXISTS conversation_facts (
    id          BIGINT PRIMARY KEY DEFAULT nextval('conversation_facts_seq'),
    content     VARCHAR NOT NULL,
    category    VARCHAR DEFAULT 'general',
    created_at  TIMESTAMP NOT NULL DEFAULT current_timestamp
);
```

### WR-04: MemoryManager.handle_tool_call signature mismatch with MemoryProvider trait for memory_recall

**File:** `crates/ironhermes-agent/src/memory/manager.rs:92-148`
**Issue:** `MemoryManager::handle_tool_call` calls `p.handle_tool_call(name, args.clone())` on the primary provider and then attempts to infer `(action, target, content)` for mirror dispatch via `infer_action_target_content`. For `memory_recall`, `infer_action_target_content` returns `None` (because `memory_recall` is not add/replace/remove), so the mirror is not called -- this is correct. However, `handle_tool_call` on the trait takes `&mut self` while the manager only holds `&self` -- the lock provides interior mutability via `Mutex<dyn MemoryProvider>`, but the `memory_recall` path through `handle_tool_call` on DuckDB/Grafeo/SQLite is read-only and only needs `&self`. The `&mut self` requirement forces the manager to take an exclusive write lock for read-only recall queries, serializing reads behind writes unnecessarily. This is a design concern rather than a bug, but it limits concurrency.
**Fix:** Consider splitting the trait method into `handle_tool_call_mut` for write operations and a `handle_tool_query` for read-only operations like recall. Alternatively, document that this serialization is intentional and acceptable for the current single-user workload.

### WR-05: SQLite on_pre_compress creates tables on every call without caching

**File:** `providers/memory-sqlite/src/lib.rs:286-304`
**Issue:** `on_pre_compress` spawns a blocking task that runs `CREATE TABLE IF NOT EXISTS` and `CREATE VIRTUAL TABLE IF NOT EXISTS` plus a trigger creation on every invocation. While SQLite handles `IF NOT EXISTS` efficiently, this DDL is executed on every compression event. The schema should be created once during `initialize` or `new` (like the main `memory_facts` schema), not on every hook invocation.
**Fix:** Move the `conversation_extracts` schema creation into the `CREATE_SCHEMA` constant in `providers/memory-sqlite/src/schema.rs` so it is created during provider construction.

## Info

### IN-01: Duplicated helper functions across all three providers

**File:** `providers/memory-sqlite/src/lib.rs:677-698`
**File:** `providers/memory-duckdb/src/lib.rs:418-440`
**File:** `providers/memory-grafeo/src/lib.rs:745-767`
**Issue:** `char_count` and `format_with_commas` are identically implemented in all three providers. These are pure utility functions with no provider-specific behavior.
**Fix:** Consider extracting these into a shared utility module (e.g., `ironhermes-core::memory_util`) to eliminate the three-way duplication.

### IN-02: Duplicated handle_tool_call dispatch logic across providers

**File:** `providers/memory-sqlite/src/lib.rs:145-186`
**File:** `providers/memory-duckdb/src/lib.rs:155-195`
**File:** `providers/memory-grafeo/src/lib.rs:263-303`
**Issue:** All three providers override `handle_tool_call` with nearly identical dispatch logic for `memory_recall`, `memory_add`, `memory_replace`, `memory_remove`. The only difference is the `memory_recall` branch which calls a provider-specific `recall` method. The base trait already provides a default `handle_tool_call` implementation. The providers could leverage the default for the add/replace/remove arms and only override the recall-specific dispatch.
**Fix:** Refactor to call the default trait implementation for non-recall tool names, or introduce a `recall(&self, query, limit) -> MemoryResult` method on the trait with the dispatch handled once in the default impl.

### IN-03: Grafeo extract_entity_triples uses naive heuristic with potential false positives

**File:** `providers/memory-grafeo/src/lib.rs:718-742`
**Issue:** The `extract_entity_triples` function splits on `.` and `\n`, then searches for patterns like `" is "`, `" has "`, etc. This produces false positives on common English sentences (e.g., "This is a test" yields subject="This", relation="is", object="a test"). The function is used in `sync_turn` and `on_pre_compress` to populate the knowledge graph. While documented as "heuristic patterns", the noise level may reduce the utility of the entity graph over time.
**Fix:** Consider adding a minimum subject/object word count filter, or filtering out common stop-word subjects like "this", "it", "that", "there". This is a quality improvement, not a bug.
