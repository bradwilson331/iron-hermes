### Inversion Analysis (Structured Reasoning)

Three potential failure modes investigated:

1. **Agent loop intercept untested at runtime with real manager**: The positive path (manager configured, schemas injected) is verified by static-grep regression test and code inspection, but no integration test exercises the full runtime flow. Mitigated by correct code placement and pattern matching the existing session_search intercept.

2. **DuckDB on_pre_compress silent data loss**: The conversation_facts table creation uses `.ok()` which swallows errors. However, the table DDL is also in CREATE_SCHEMA (schema.rs:14) ensuring creation at init time. The `.ok()` is a safety net for edge cases, consistent with the D-07 fire-and-forget contract.

3. **Grafeo entity extraction heuristic limitations**: The `extract_entity_triples` function uses simple keyword matching (7 relation words). Complex sentences may not extract well. This is acceptable -- CONTEXT.md explicitly lists entity extraction approach as "Claude's Discretion" and the heuristic approach was chosen for zero dependencies.

### Confirmation Bias Counter

1. **Partially met**: D-09 DuckDB says "pre-computes time-windowed aggregates" but implementation does a COUNT warmup query. Acceptable -- D-09 implementation strategy is at Claude's Discretion per CONTEXT.md.
2. **Test coverage gap**: SQLite `test_on_pre_compress_indexes_messages` only asserts `result.is_ok()`, does not verify messages actually indexed. Acceptable -- fire-and-forget with tokio::spawn makes synchronous verification impractical.
3. **Uncovered error path**: Unicode input to FTS5 MATCH not tested. Low risk -- double-quote wrapping in `sanitize_fts_query` handles it safely.
