# Phase 21.5: Memory Provider Plugin - Context

**Gathered:** 2026-04-20
**Status:** Ready for planning

<domain>
## Phase Boundary

Make memory providers deliver on the "plugin" promise — implementing provider-specific tools, hook overrides, config loading, and differentiation that make each provider genuinely useful beyond basic add/replace/remove storage. The trait surface exists (Phase 20), wiring is verified (Phase 21.4), but providers currently use only no-op defaults for all advanced hooks.

**In scope:**
- Factory config loading: read `$HERMES_HOME/<provider>.json` and pass as `&Value` to `initialize()`
- Unified `memory_recall` tool: same schema across all providers, per-provider implementation (FTS5, graph walk, analytical query)
- Hook implementations for all 3 external providers: `queue_prefetch`, `on_pre_compress`, `system_prompt_block`, `sync_turn`
- Provider-specific differentiation via hook behavior and recall strategy
- Non-blocking `sync_turn` via `tokio::spawn` pattern

**Out of scope:**
- Runtime plugin discovery / dynamic loading (per PROJECT.md:52)
- Provider-specific CLI subcommands beyond existing `hermes memory status/off` (Phase 21.4 delivered those)
- Overriding base `memory add/replace/remove` tool behavior (additive only)
- LLM-assisted extraction in `on_pre_compress` (requires auxiliary client — not yet implemented)
- Broadcast `on_memory_write` to multiple subscribers (deferred in Phase 20)
- New provider backends beyond existing three (SQLite, Grafeo, DuckDB)

</domain>

<decisions>
## Implementation Decisions

### Provider config loading
- **D-01:** Factory reads `$HERMES_HOME/<provider_name>.json` (using the provider's `name()` return value) and passes as `&Value` to `initialize()`. If file doesn't exist, passes `Value::Null` (provider uses defaults). Replaces the current hardcoded `Value::Null` in factory.rs.
- **D-02:** Providers read their own env vars directly (e.g., `std::env::var("MY_API_KEY")`) inside `initialize()`. Secrets never flow through the `&Value` config blob. This matches hermes-agent's pattern where secrets stay in `.env` and the provider resolves them.

### Provider-specific tools
- **D-03:** All providers implement a unified `memory_recall` tool with the same schema: `memory_recall(query: String, limit: Option<u32>)` defaulting to 5 results. Each provider implements recall differently (SQLite: FTS5 keyword search, Grafeo: graph traversal, DuckDB: analytical/temporal query).
- **D-04:** `memory_recall` is ADDITIVE — it does not replace or modify the existing `memory add/replace/remove` tools. Those continue to be handled by the MemoryManager's default `handle_tool_call` impl. Providers override `get_tool_schemas()` to return `memory_recall` alongside the defaults, and override `handle_tool_call()` to dispatch `memory_recall` while delegating other names to the default.
- **D-05:** `memory_recall` returns ranked results with relevance snippets. Result format: `[{content, target, relevance_score, snippet}]` serialized as JSON string in the tool response.

### Hook implementations
- **D-06:** All three external providers implement `queue_prefetch`, `on_pre_compress`, `system_prompt_block`, and `sync_turn`. The file-based `MemoryStore` keeps no-op defaults (it has no indexing layer to benefit from these hooks).
- **D-07:** `sync_turn` uses `tokio::spawn` fire-and-forget pattern for any I/O work. The hook itself returns immediately; a spawned task handles indexing/persistence. Errors are logged via `tracing::warn!`, never propagated. This matches hermes-agent's "daemon thread" contract.
- **D-08:** `on_pre_compress` extraction strategy is provider-specific (no enforced common approach):
  - **SQLite:** Full-text indexes the raw messages being discarded (FTS5 insert)
  - **Grafeo:** Extracts entity-relationship triples from message content
  - **DuckDB:** Extracts structured facts with timestamps for analytical queries
- **D-09:** `queue_prefetch` pre-warms provider state for the next `prefetch()` call:
  - **SQLite:** Warms FTS5 cache with likely query terms from the last turn
  - **Grafeo:** Pre-traverses relationship paths relevant to recent entities
  - **DuckDB:** Pre-computes time-windowed aggregates for recent topics
- **D-10:** `system_prompt_block` content is provider-specific (no common template):
  - **SQLite:** Surfaces recent recall hits or frequently accessed entries
  - **Grafeo:** Surfaces key entity relationships and connection summaries
  - **DuckDB:** Surfaces detected patterns, frequency insights, temporal trends

### Provider differentiation
- **D-11:** SQLite differentiator: **Full-text search recall.** FTS5 powers `memory_recall` with ranked keyword search, snippet generation, and proximity scoring. `sync_turn` indexes each conversation turn for future retrieval. Best for keyword-based recall across large memory stores.
- **D-12:** Grafeo differentiator: **Relationship recall.** Graph traversal powers `memory_recall` with multi-hop relationship queries. `sync_turn` extracts entity-relationship triples. `on_pre_compress` saves relationship networks from discarded messages. Best for "how does X relate to Y?" queries.
- **D-13:** DuckDB differentiator: **Analytical recall.** Columnar storage powers `memory_recall` with time-range queries, frequency analysis, and pattern detection. `sync_turn` logs structured metrics/facts. Best for "how often/when/trends" style queries.

### Claude's Discretion
- Exact FTS5 query construction for SQLite `memory_recall` (reuse `SearchFilter` pattern from Phase 13, or custom)
- Entity extraction approach for Grafeo `sync_turn` and `on_pre_compress` (regex-based, heuristic, or keyword)
- DuckDB schema for structured facts (columns, types, indexes)
- `system_prompt_block` wording and format for each provider
- Whether `queue_prefetch` implementations share a common trait helper or are fully independent
- `memory_recall` result formatting (exact JSON shape within the MemoryResult string envelope)

</decisions>

<specifics>
## Specific Ideas

- The user provided the complete hermes-agent Memory Provider Plugin guide as a reference. The Rust implementation should match the behavioral contract (non-blocking sync_turn, provider-specific hooks, unified tool schema for LLM) while using Rust idioms (tokio::spawn instead of daemon threads, Arc<Mutex> instead of threading.Lock).
- `memory_recall` as a unified interface is the key insight — the LLM doesn't need to know which backend is active; it always calls `memory_recall(query, limit)` and gets ranked results. This keeps the tool schema stable across provider swaps.
- Provider-specific `system_prompt_block` is where each provider's unique value is most visible to the LLM — SQLite shows keyword-relevant recent entries, Grafeo shows relationship maps, DuckDB shows temporal patterns. This makes provider choice meaningful for the user experience.
- The factory config loading fix (D-01) unblocks everything else — without it, providers can't read their per-provider JSON config at startup.

</specifics>
