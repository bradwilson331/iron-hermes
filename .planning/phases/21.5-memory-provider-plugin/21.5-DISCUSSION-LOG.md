## Provider Config Loading

| Option | Description | Selected |
|--------|-------------|----------|
| Factory loads JSON (Recommended) | Factory reads $HERMES_HOME/<provider>.json, passes as &Value to initialize(). Providers parse what they need. | ✓ |
| Provider self-loads in initialize | Factory passes only hermes_home; each provider reads its own <name>.json inside initialize(). | |
| Both paths | Factory loads and passes the JSON if it exists. Provider can also self-load extra files. | |

**User's choice:** Factory loads JSON
**Notes:** Consistent with the existing initialize(session_id, hermes_home, &Value) signature.

---

| Option | Description | Selected |
|--------|-------------|----------|
| Providers read env directly (Recommended) | Matches hermes-agent pattern: secrets stay in .env, provider reads os::var() in initialize(). | ✓ |
| Factory injects env vars into Value | Factory reads ConfigField.env_var for each secret field and merges into the JSON blob. | |

**User's choice:** Providers read env directly
**Notes:** Secrets never flow through the Value config blob.

## Provider-specific Tools

| Option | Description | Selected |
|--------|-------------|----------|
| SQLite: memory_search | FTS5 full-text search over memory entries. | |
| Grafeo: memory_relate | Query relationships between memory entries (multi-hop graph traversal). | |
| DuckDB: memory_analyze | Analytical queries over memory facts (aggregations, time-series). | |
| All providers: memory_recall | Semantic/keyword recall tool shared across all providers. Each implements differently. | ✓ |

**User's choice:** All providers: memory_recall
**Notes:** Unified interface — LLM sees one tool regardless of backend.

---

| Option | Description | Selected |
|--------|-------------|----------|
| Query + limit (Recommended) | Simple: memory_recall(query, limit=5). Provider decides how to match. Returns ranked results. | ✓ |
| Query + target + mode | Richer: memory_recall(query, target, mode, limit). More control for the LLM. | |
| Just query | Minimal: memory_recall(query). Provider returns top-5. No tuning knobs. | |

**User's choice:** Query + limit
**Notes:** Simple interface, provider has discretion on matching strategy.

---

| Option | Description | Selected |
|--------|-------------|----------|
| Additive (Recommended) | memory_recall is NEW alongside existing memory add/replace/remove. Base tools stay as-is. | ✓ |
| Full override | Providers override get_tool_schemas entirely — returning custom versions of all tools. | |
| Additive now, override later | This phase adds memory_recall only. Future phase lets providers customize write tools. | |

**User's choice:** Additive
**Notes:** Keeps scope tight. Existing tool behavior unchanged.

## Hook Implementations

| Option | Description | Selected |
|--------|-------------|----------|
| queue_prefetch | Pre-warm results for next turn's prefetch() call. | ✓ |
| on_pre_compress | Extract insights from messages about to be discarded. | ✓ |
| system_prompt_block | Inject provider-specific block into system prompt. | ✓ |
| sync_turn | Index conversation after each completed turn. | ✓ |

**User's choice:** All four hooks
**Notes:** Makes providers genuinely useful beyond basic storage.

---

| Option | Description | Selected |
|--------|-------------|----------|

**User's choice:** tokio::spawn
**Notes:** Matches hermes-agent's daemon thread contract and existing queue_prefetch pattern.
