## Summary

Phase 21.5 fills in all the "no-op" hook bodies and adds the `memory_recall` tool to the three external memory providers (SQLite, Grafeo, DuckDB). The trait surface (`MemoryProvider` with 11 hooks) and the wiring (`MemoryManager` dispatch, agent_loop `queue_prefetch` fire, `context_engine` `on_pre_compress` fire) are all in place from Phase 20. What does not yet exist: real implementations of the 4 advanced hooks for each provider, the `memory_recall` tool schema and dispatch, and the config JSON loading in `factory.rs`.

There are two independent work streams: (1) the factory config loading fix (D-01), which is a single localized change to two spots in `factory.rs`, and (2) the provider hook + tool implementations across three provider crates. Both streams can be planned as separate waves.

The most architecturally tricky piece is `memory_recall` dispatch. The existing `MemoryTool` is registered under the name `"memory"` and only handles `add/replace/remove`. The `memory_recall` schema returned by `get_tool_schemas()` must appear in the LLM's tool list and must route to `manager.handle_tool_call("memory_recall", args)`. Currently the `agent_loop.run()` only adds `session_search` as a dynamic schema — it does not add memory manager schemas. A new wiring step is required: extend `agent_loop.run()` to also add schemas from `memory_manager.get_tool_schemas()` and intercept calls to names in that set, routing them through `manager.handle_tool_call()`.

**Primary recommendation:** Implement in three waves: (1) factory config loading, (2) provider `memory_recall` tool + hook impls, (3) agent_loop wiring to expose `memory_recall` to the LLM.

### Pattern 3: sync_turn Fire-and-Forget (D-07)

**What:** `sync_turn` returns immediately; a `tokio::spawn` task does the actual work.

**When to use:** All three external providers.

**Example:**
```rust
// Source: agent_loop.rs:554 — the queue_prefetch tokio::spawn pattern
async fn sync_turn(&self, session_id: &str, entries: &MemoryEntries) -> anyhow::Result<()> {
    // Capture what we need before the spawn (Arc clones, owned values)
    let conn = self.conn.clone(); // Arc<Mutex<Connection>> for SQLite
    let session_id = session_id.to_string();
    let entries = entries.clone();

    tokio::spawn(async move {
        // Actual indexing work here
        let guard = conn.lock().await;
        if let Err(e) = index_turn_entries(&guard, &session_id, &entries) {
            tracing::warn!(error = %e, session_id = %session_id, "sync_turn indexing failed");
        }
    });
    Ok(()) // Return immediately
}
```

[VERIFIED: tokio::spawn fire-and-forget pattern at agent_loop.rs lines 554-562]

**Critical constraint:** `sync_turn` signature takes `&self` (immutable reference). The spawned task cannot borrow `self`. All resources needed must be cloned into the spawn closure.

**DuckDB-specific:** The DuckDB bridge uses `mpsc::Sender` which is `Clone + Send`. The spawn closure can own a cloned sender and send a command to the worker thread without blocking the async caller.

**SQLite-specific:** `conn: Mutex<Connection>` is NOT Clone. Provider must store `conn` as `Arc<Mutex<Connection>>` for the spawn to clone the Arc. Currently `SqliteMemoryProvider.conn` is `Mutex<Connection>` (not Arc). Either wrap in Arc or send via a mpsc channel like DuckDB.

**Grafeo-specific:** `GrafeoDB` must be checked for Clone/Arc-ability. The provider may need to restructure similarly.

### Anti-Patterns to Avoid

- **Blocking async in sync_turn:** Never call `.lock().unwrap()` on a tokio Mutex inside `sync_turn`'s async body without spawning. The hook is called while the manager holds the primary lock — deadlock risk.
- **Non-Arc conn in tokio::spawn:** `Mutex<Connection>` is not Clone. Must be `Arc<Mutex<Connection>>` to share across a spawn boundary.
- **FTS5 unescaped queries:** Raw user query strings passed to FTS5 MATCH will error on special characters (`"`, `*`, parentheses). Sanitize first.
- **Propagating hook errors:** All hook implementations (sync_turn, queue_prefetch, on_pre_compress) must log errors and return `Ok(())`. The agent loop and context engine swallow errors from these hooks but log them — providers must not bubble panics.
- **session_search shadow:** Do NOT name any provider tool `session_search` — `MemoryManager::new` enforces this and will reject the provider at construction time.

## Common Pitfalls

### Pitfall 1: sync_turn conn Borrowing
**What goes wrong:** `SqliteMemoryProvider.conn` is `Mutex<Connection>` (not Arc). Spawning a task that needs the connection will fail to compile — `Mutex<T>` is not Clone.
**Why it happens:** Phase 17 wrapped `Connection` in `Mutex` for `Sync` compliance, but didn't need Arc because all callers were synchronous. `tokio::spawn` requires `'static` closure, which means owned data.
**How to avoid:** Change `conn: Mutex<Connection>` to `conn: Arc<Mutex<Connection>>` in `SqliteMemoryProvider`. This is a field type change that requires updating all existing `self.conn.lock()` callsites to also work through an Arc (which they will — Arc<Mutex<T>> derefs to same lock API).
**Warning signs:** Compiler error "cannot be sent between threads safely" or "`Connection` does not implement `Clone`".

### Pitfall 2: DuckDB fire-and-forget has no respond channel
**What goes wrong:** Trying to use `mpsc::sync_channel` for fire-and-forget DuckDB operations adds unnecessary synchronization. The `sync_turn` / `on_pre_compress` hooks must return immediately, not wait for the bridge.
**Why it happens:** Copy-pasting the existing `bridge_add` pattern which uses a respond channel.
**How to avoid:** Add new "fire-and-forget" DuckDbCommand variants that have no `respond` field. The worker thread processes them and logs errors internally.
**Warning signs:** Hooks block on `rx.recv()`, defeating the fire-and-forget contract.

### Pitfall 3: memory_recall Not Appearing in LLM Tool List
**What goes wrong:** Provider implements `get_tool_schemas()` correctly, but the LLM never calls `memory_recall` because the schema was never added to `tool_schemas` in `agent_loop.run()`.
**Why it happens:** The current `run()` only dynamically adds `session_search`. Memory provider schemas are not yet injected.
**How to avoid:** Add memory manager schema injection to `agent_loop.run()` alongside the session_search block. Also add name-based intercept in `execute_tool_call()`.
**Warning signs:** Tests pass in isolation but `memory_recall` never appears in a real agent session.

### Pitfall 4: FTS5 Query Syntax Errors
**What goes wrong:** FTS5 `MATCH` throws `SqliteFailure` when the query contains special FTS5 syntax characters (`"`, `(`, `)`, `*`, `NOT`, `AND`, `OR` as literal text).
**Why it happens:** FTS5 has its own query language. User-provided search terms may contain these characters.
**How to avoid:** Sanitize query before passing to MATCH. Simplest: tokenize on whitespace, wrap each token in double quotes, join with space. E.g., `"cats" "dogs"` is safe FTS5.
**Warning signs:** `memory_recall` returns errors for queries with punctuation or boolean keywords.

### Pitfall 5: factory.rs has TWO identical stubs
**What goes wrong:** Fixing only `build_memory_provider` but not `build_tokio_provider` — providers constructed through the manager path (which all runtime code uses since Phase 20-02) still get `Value::Null`.
**Why it happens:** The factory has two provider-kind dispatch functions: `build_memory_provider` (legacy, std::sync::Mutex) and `build_tokio_provider` (current, tokio::sync::Mutex). Both have `let provider_config = serde_json::Value::Null;` at lines 42 and 212.
**How to avoid:** Fix BOTH. Extract `load_provider_config()` helper called by both.
**Warning signs:** Config changes in `sqlite.json` have no effect at runtime.

### Pitfall 6: Grafeo edge creation between sessions
**What goes wrong:** `sync_turn` extracts entities and creates nodes/edges, but across sessions the same entity gets new nodes instead of connecting to existing ones.
**Why it happens:** Naive implementation creates new nodes on every call without checking if an entity node already exists.
**How to avoid:** Before creating an entity node, check `db.find_nodes_by_property(PROP_ENTITY_NAME, entity_text)`. If found, reuse the existing node ID for edge creation.
**Warning signs:** Grafeo node count grows unboundedly; graph traversal finds no relationships.

## Open Questions

1. **Grafeo edge API**
   - What we know: `GrafeoMemoryProvider` uses `create_node`, `set_node_property`, `iter_nodes`, `delete_node`
   - What's unclear: Whether Grafeo has `create_edge(from_id, to_id, label)` or a different API for edge creation
   - Recommendation: Read `grafeo` crate source or docs before planning Grafeo hook implementations. Entity extraction via edges may need a different storage model if edges aren't supported as expected.

2. **sync_turn semantics for conversation indexing**
   - What we know: `sync_turn` is called with `(session_id, &MemoryEntries)` where MemoryEntries is current memory state
   - What's unclear: D-11 says "indexes each conversation turn" — but the hook doesn't receive conversation text, only memory entries
   - Recommendation: Implement as "refresh search index from current memory entries" rather than "index conversation text". This is safe and actionable with the current interface.

3. **AgentLoop::execute_tool_call signature**
   - What we know: Tool dispatch happens in `run()` via `execute_tool_call` or an inline match block
   - What's unclear: Whether execute_tool_call is a separate method that can receive the memory_tool_names set
   - Recommendation: Add `memory_provider_tool_names: HashSet<String>` as a field on AgentLoop, set in `run()`, read in dispatch. Mirrors the `state_store` pattern exactly.

4. **System prompt block timing**
   - What we know: `system_prompt_block()` is called by `MemoryManager::system_prompt_block()` which is called by `PromptBuilder`
   - What's unclear: When this is called relative to `sync_turn` — if sync_turn hasn't run yet, system_prompt_block will surface stale data
   - Recommendation: `system_prompt_block` should surface data from the last completed sync_turn. SQLite can query the most recently indexed entries; DuckDB can query recent analytical aggregates.
