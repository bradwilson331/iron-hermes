# Phase 21.3 — UI Design Contract

> Visual and interaction contract for terminal/TUI output surfaces.
> This phase has no web frontend. "UI" means: CLI stdout formatting,
> in-session slash command response text, and status line field accuracy.
> Generated by gsd-ui-researcher; verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (terminal output — no component library) |
| Preset | not applicable |
| Component library | colored crate (ANSI color), crossterm (cursor/render) |
| Icon library | none — ASCII/Unicode glyphs only (·, ─, →) |
| Font | terminal default — no font selection |

**Source:** RESEARCH.md Standard Stack + codebase — `colored` crate already used in `cron.rs`, `skills_cmd.rs`, and `status_line.rs`.

---

## Spacing Scale

Terminal output uses character-based layout, not pixel spacing. Declared in columns:

| Token | Value | Usage |
|-------|-------|-------|
| indent-1 | 2 spaces | All row content under a section header |
| indent-2 | 4 spaces | Nested content (sub-fields, continuation lines) |
| label-col | 16 chars | Left-padded label column in detail views (`{:<16}`) |
| name-col | 20 chars | Name column in list table (`{:<20}`) |
| status-col | 12 chars | Status column in list table (`{:<12}`) |
| separator | 70 chars | `─`.repeat(70) for section dividers in list commands |
| separator-detail | 50 chars | `─`.repeat(50) for detail/info views |

Exceptions: none.

**Source:** Measured directly from `cron.rs` — `"─".repeat(70)` in `cmd_list`, `"─".repeat(50)` in `cmd_status`/`render_job_details`; label column `{:<14}` in `cron.rs` detail view. Phase 21.3 uses 16-char label col to accommodate `"Max output:"`, `"Tool use:"`, `"Tokenizer:"` labels.

---

## Typography

Terminal text only. No font sizes — all terminal default. Role maps to ANSI style:

| Role | Style | Usage |
|------|-------|-------|
| Section header | `.bold().cyan()` | `"Model Metadata"`, `"Known Models"` — one per command |
| Table header row | `.bold()` (no color) | Column names: `NAME`, `CONTEXT`, `TOKENIZER`, `CAPABILITIES` |
| Key name (label) | `.dimmed()` | `"Context:"`, `"Tokenizer:"`, `"Capabilities:"` in detail views |
| Value — normal | unstyled | Context length numbers, tokenizer names, capability flags |
| Value — model name | `.yellow()` | Model canonical IDs and aliases (matches cron `job.name.yellow()`) |
| Value — good/active | `.green()` | Capabilities that are true: `"vision"`, `"tool_use"`, `"streaming"` |
| Value — inactive | `.dimmed()` | Capabilities that are false: `"no reasoning"`, `"no vision"` |
| Value — warning | `.yellow()` | Unknown model fallback warning: `"unknown — using 128K default"` |
| Error prefix | `.red().bold()` | `"Error:"` prefix on stderr (matches cron pattern `"Error".red().bold()`) |
| Count footer | `.dimmed()` | `"N model(s) total"` / `"Cache last updated: ..."` |
| Progress/status | `.dimmed()` | `"Fetching from models.dev..."`, `"Fetching from OpenRouter..."` |

**Source:** Directly observed from `cron.rs` (`job.name.yellow()`, `"scheduled".green()`, `"paused".yellow()`, `"Error".red().bold()`, footer `.dimmed()`), `skills_cmd.rs` (unstyled values), and `status_line.rs` (color palette: cyan/magenta/green/yellow/dimmed).

## Color

Terminal ANSI colors only. No hex values — terminal palette.

| Role | ANSI Color | Usage |
|------|------------|-------|
| Dominant | terminal default (unstyled) | All value text, table body rows |
| Secondary | `.dimmed()` | Labels, footers, timestamps, progress messages |
| Accent | `.cyan().bold()` | Section headers only: `"Known Models"`, `"Model Info"`, `"Fetch Complete"` |
| Model name highlight | `.yellow()` | Model canonical IDs in tables and detail views |
| Capability active | `.green()` | True capability flags in detail view |
| Error / destructive | `.red().bold()` | Error prefix on stderr only |

Accent reserved for: section header lines (`"Known Models".bold().cyan()`, `"Model Info".bold().cyan()`, `"Fetch Complete".bold().cyan()`, `"Models Cache".bold().cyan()`). Never applied to value text or table body.

**Source:** Direct pattern match from `cron.rs` — `"Scheduled Jobs".bold().cyan()`, `"Cron Status".bold().cyan()`, `"Cron Job".bold().cyan()`.

## Output Surface Contracts

### Surface 1: `hermes models list`

Lists all models in the static table plus any disk-cached entries.

```
Known Models                                    ← .bold().cyan()
──────────────────────────────────────────────────────────────────────  ← "─"×70
  NAME                 CONTEXT    OUTPUT     TOKENIZER      CAPABILITIES
  claude-sonnet-4      200.0K     64.0K      cl100k_base    vision tool_use streaming
  claude-haiku-4       200.0K     64.0K      cl100k_base    vision tool_use streaming
  claude-opus-4        200.0K     32.0K      cl100k_base    vision tool_use reasoning streaming
  gpt-4o               128.0K     16.4K      o200k_base     vision tool_use streaming
  gpt-4o-mini          128.0K     16.4K      o200k_base     vision tool_use streaming
  ...
──────────────────────────────────────────────────────────────────────
  47 model(s) total · Cache: not loaded                                 ← .dimmed()
```

Rules:
- NAME column: 20 chars, model name in `.yellow()`
- CONTEXT and OUTPUT columns: 10 chars each, formatted via `format_token_count()` (e.g., `200.0K`, `1.0M`)
- TOKENIZER column: 14 chars, unstyled
- CAPABILITIES column: space-separated active capability names in `.green()`; if none active, `"none"` in `.dimmed()`
- Footer line: count + cache status on one dimmed line, separated by ` · `
- Empty state: `"  No models loaded."` in `.dimmed()`

**Source:** Follows `cmd_list` in `cron.rs` — section header, `─`×70 rule, column headers `.bold()`, name `.yellow()`, footer `.dimmed()`.

### Surface 2: `hermes models info <model>`

Detail view for a single model. Accepts canonical ID, alias, or provider-prefixed form.

```
Model Info                                      ← .bold().cyan()
──────────────────────────────────────────────────────────────────────  ← "─"×50
  Canonical ID:     claude-sonnet-4             ← label .dimmed(), value .yellow()
  Context:          200.0K tokens               ← label .dimmed(), value unstyled
  Max output:       64.0K tokens                ← label .dimmed(), value unstyled (or "unknown" .dimmed())
  Tokenizer:        cl100k_base                 ← label .dimmed(), value unstyled
  Vision:           yes                         ← label .dimmed(), "yes" .green() / "no" .dimmed()
  Tool use:         yes
  Reasoning:        no
  Streaming:        yes
  Source:           static table                ← label .dimmed(), "static table" / "disk cache (fetched YYYY-MM-DD HH:MM UTC)" unstyled
  Aliases:          claude-sonnet-4-20250514, anthropic/claude-sonnet-4   ← .dimmed() if no aliases
```

Rules:
- Label column: 18 chars (`{:<18}`)
- "yes" capability values: `.green()`
- "no" capability values: `.dimmed()`
- Source line shows `"disk cache (fetched YYYY-MM-DD HH:MM UTC)"` when metadata came from cache
- If model not found: `"Model not found: <id>. Run \`hermes models fetch\` to update cache."` on stderr as `.red().bold()` prefix + unstyled message

**Source:** Follows `render_job_details` in `cron.rs` — section header, `─`×50 rule, label+value columns, state color-coding (green/yellow/dimmed).

### Surface 3: `hermes models fetch`

Fetches from models.dev and OpenRouter, saves to disk cache.

```
Fetching model metadata...                      ← .dimmed()
  Querying models.dev...                        ← .dimmed()
  models.dev: 247 models received               ← unstyled
  Querying OpenRouter (OPENROUTER_API_KEY set)... ← .dimmed()
  OpenRouter: 183 models received               ← unstyled
Fetch Complete                                  ← .bold().cyan()
──────────────────────────────────────────────────────────────────────
  Cache saved: ~/.ironhermes/models-cache.json  ← .dimmed()
  Models updated: 312 entries                   ← .dimmed()
  Fetched at: 2026-04-19 07:38 UTC              ← .dimmed()
```

Rules:
- Progress lines use `.dimmed()` to avoid visual noise during fetch
- `"Fetch Complete"` header only printed on success
- On partial success (one API fails): print warning line `"  Warning: OpenRouter fetch failed — <error>"` in `.yellow()` before the success header
- On total failure: `"Error: <message>"` in `.red().bold()` to stderr, exit non-zero

**Source:** Follows `cmd_tick` in `cron.rs` — progress lines `.dimmed()`, per-item lines `.yellow()`, completion line `.dimmed()`.

### Surface 4: `/models refresh` (in-session slash command)

Returns `CommandResult::Output(text)` — displayed inline in the REPL, no ANSI color (slash command output is plain text rendered by the REPL loop).

```
Fetching model metadata...
  models.dev: 247 models received
  OpenRouter: 183 models received
Fetch complete. 312 entries saved to cache.
```

Rules:
- No ANSI color codes — `CommandResult::Output` text is rendered verbatim by the REPL
- Maximum 5 lines of output — do not flood the REPL
- On error: `CommandResult::Error("Fetch failed: <reason>. Check network and OPENROUTER_API_KEY.")` — the REPL renders errors distinctly

**Source:** Follows `cmd_stop` / `cmd_status` handlers in `handlers.rs` — plain string in `CommandResult::Output`, no ANSI coloring.

---

### Surface 5: `/models info <model>` (in-session slash command)

Returns `CommandResult::Output(text)`.

```
claude-sonnet-4
  Context:    200,000 tokens
  Max output: 64,000 tokens
  Tokenizer:  cl100k_base
  Vision: yes  Tool use: yes  Reasoning: no  Streaming: yes
  Source: static table
```

Rules:
- No ANSI color codes
- Compact 6-line format — not the full 50-char-separator detail view
- Numbers formatted with commas for readability (e.g., `200,000` not `200.0K`) — easier to compare against model docs
- If model not found: `CommandResult::Error("Model not found: <id>. Run /models refresh to update cache.")`

**Source:** Follows `cmd_status` / `cmd_profile` in `handlers.rs` — multi-line plain `CommandResult::Output`.

### Surface 6: Status line — `tokens_used` / `tokens_limit` accuracy

No new visual elements. The existing `StatusLineState` struct and `render_status_line()` function handle display. This surface contract defines the data that must be supplied:

```
Chat · claude-sonnet-4 · anthropic · 14.2K/200.0K (7%) · ctrl+c cancel · /help commands
                                      ↑     ↑
                             tokens_used  tokens_limit (from resolved ModelMetadata.context_length)
```

Rules:
- `tokens_limit` must be populated from `ResolvedEndpoint.context_length()` — never hardcoded `128_000`
- `format_token_count()` already handles K/M formatting — no changes needed to render function
- Percentage calculation already handles zero-limit guard — no changes needed
- `model_short` field should use the canonical model ID (without provider prefix) — strip `provider/` prefix before setting

**Source:** `status_line.rs` — `StatusLineState`, `render_status_line()`, `format_token_count()` — all existing, no new rendering code needed.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA (`hermes models fetch`) | `"Fetching model metadata..."` |
| Empty state — no models | `"  No models loaded."` |
| Empty state — no cache | `"  Cache: not loaded"` (in footer of `hermes models list`) |
| Error — model not found (CLI) | `"Model not found: <id>. Run \`hermes models fetch\` to update cache."` |
| Error — model not found (slash cmd) | `"Model not found: <id>. Run /models refresh to update cache."` |
| Error — fetch failed | `"Fetch failed: <reason>. Check network and OPENROUTER_API_KEY."` |
| Unknown model fallback warning (tracing) | `"unknown model {model_id}, using default 128K context"` (D-15 — tracing::warn, not printed to stdout) |
| Cache stale notice | not displayed automatically — shown only in `hermes models list` footer as `"Cache: fetched YYYY-MM-DD"` |
| Fetch success | `"Fetch complete. {N} entries saved to cache."` |
| Capability true | `"yes"` (slash command) / capability name in `.green()` (CLI list) |
| Capability false | `"no"` (slash command) / omitted from list / `"no"` `.dimmed()` in detail view |

**No destructive actions in this phase.** All operations are read-only (list/info) or additive writes (fetch appends/overwrites cache). No confirmation prompts needed.

**Source:** Copywriting patterns follow `cron.rs` (`"Job created"`, `"Job removed"`, `"Cancelled."`) and `handlers.rs` (`"No agent is currently running."`, `"not yet available"`).

## Implementation Notes for Executor

1. `models_cmd.rs` must import `colored::Colorize` and follow the exact same pattern as `cron.rs` — `println!` for stdout, `eprintln!` for errors, colored strings via `.bold().cyan()` / `.yellow()` / `.green()` / `.dimmed()` / `.red().bold()`.

2. All formatting functions that produce strings for unit-testing must follow the `render_job_details` pattern — return `String` from a pure inner function, call `print!("{}", result)` from the outer command handler.

3. `format_token_count()` from `status_line.rs` must be used for all token/context length display values — `200.0K`, `1.0M`, etc. Do not duplicate formatting logic.

4. Slash command handlers (`cmd_models` in `handlers.rs`) return plain `CommandResult::Output(String)` — no ANSI escape codes in the string. The REPL renders it.

5. `tokens_limit: 128_000` hardcodes in `main.rs` at lines 489 and 990 are replaced with `resolved_endpoint.context_length()` — the status line visual does not change, only the data source changes.
