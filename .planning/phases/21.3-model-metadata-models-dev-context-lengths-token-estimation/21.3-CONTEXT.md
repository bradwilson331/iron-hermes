# Phase 21.3: Model Metadata & models.dev — Context Lengths, Token Estimation - Context

**Gathered:** 2026-04-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the hardcoded `DEFAULT_CONTEXT_LENGTH = 128_000` with a model-aware metadata system, improve token estimation from the crude `text.len() / 4` heuristic to proper BPE tokenization via tiktoken-rs, and wire accurate model metadata through all consumers (AgentLoop, ContextCompressor, PressureTracker, StatusLine). Add `hermes models` CLI subcommand and `/models` slash command for fetching metadata from models.dev and OpenRouter APIs.

In scope: `ModelMetadata` struct and static lookup table, canonical ID + alias map, tiktoken-rs integration, `hermes models list/fetch/info` CLI subcommand, `/models` slash command, disk cache auto-loading, wiring metadata through ProviderResolver to all consumers.

Out of scope: Pricing/cost tracking, model benchmarking, dynamic model switching mid-session, model recommendation engine.

</domain>

<decisions>
## Implementation Decisions

### Model metadata source
- **D-01:** Static lookup table (`HashMap<&str, ModelMetadata>`) compiled into the binary with known models (Claude family, GPT family, Llama family, Gemini, Mistral, etc.). Updated each release. This is the primary source — fast, no network, works offline.
- **D-02:** Manual API fetch supplements the static table. Users can run `hermes models fetch` or `/models refresh` to query external APIs and cache results to `~/.ironhermes/models-cache.json`. The disk cache is auto-loaded at startup and merged over the static table (cache entries override static entries for the same canonical ID).
- **D-03:** API fetch uses a fallback chain: try models.dev first (no auth needed, broader cross-provider coverage), fall back to OpenRouter `/models` API if models.dev doesn't have the model (requires `OPENROUTER_API_KEY`).
- **D-04:** `hermes models` CLI subcommand for setup/browsing (`list`, `fetch`, `info <model>`) plus `/models` slash command for in-session refresh (`/models refresh`, `/models info <model>`). Both share the same underlying metadata service.
- **D-05:** Model ID normalization uses a canonical ID + alias map. Metadata is stored under canonical IDs (e.g., `claude-sonnet-4`). An alias map resolves provider-prefixed names (`anthropic/claude-sonnet-4`), versioned names (`claude-sonnet-4-20250514`), and legacy names to their canonical form. Lookup: exact match → alias map → strip `provider/` prefix → fallback to DEFAULT_CONTEXT_LENGTH with warning.
- **D-06:** Resolution precedence for context_length: user config.yaml `context_length` > disk cache entry > static table entry > `DEFAULT_CONTEXT_LENGTH` (128K) with tracing::warn. User config always wins.

### Token estimation accuracy
- **D-07:** Replace the crude `text.len() / 4 + 1` heuristic with proper BPE tokenization via the `tiktoken-rs` crate. This gives accurate token counts for compression threshold decisions, pressure warnings, and status line display.
- **D-08:** Encoding selection is model-metadata-driven. `ModelMetadata` includes a `tokenizer` field (e.g., `"cl100k_base"`, `"o200k_base"`). The token estimator uses the encoding specified by the active model's metadata. Fallback to `cl100k_base` for models with unknown tokenizer.
- **D-09:** tiktoken is used everywhere — all `estimate_tokens`, `estimate_message_tokens`, and `estimate_messages_tokens` callsites in `context_compressor.rs` are updated to use the tiktoken-backed estimator. No dual heuristic/accurate split. Consistent accuracy across all consumers.

### Metadata scope & fields
- **D-10:** `ModelMetadata` struct fields: `context_length: usize`, `max_output_tokens: Option<usize>`, `tokenizer: String` (tiktoken encoding name), `capabilities: ModelCapabilities` (struct with bool fields for `vision`, `tool_use`, `reasoning`, `streaming`).
- **D-11:** Pricing metadata (input/output cost per token) is deferred to a future cost-tracking phase. Not included in `ModelMetadata` for now.

### Wiring & resolution flow
- **D-12:** `ModelMetadata` struct and static lookup table live in `ironhermes-core` (alongside existing `Config`, `constants.rs`, `types.rs`). Available to all workspace crates.
- **D-13:** `tiktoken-rs` is added as a dependency of `ironhermes-core`. A `TokenEstimator` struct in core wraps encoding selection + counting. Single encoding initialization at startup. Available to all consumer crates.
- **D-14:** `ProviderResolver` carries metadata: `resolve_for_main()` returns a `ResolvedEndpoint` that includes the resolved `ModelMetadata` for the target model. Consumers (`AgentLoop`, `ContextCompressor`, `PressureTracker`, `StatusLineState`) receive `context_length` from the resolved endpoint instead of hardcoding `128_000`.
- **D-15:** When a model isn't found in the static table, disk cache, or config, log a `tracing::warn!("unknown model {model_id}, using default 128K context")` and use `DEFAULT_CONTEXT_LENGTH`. User can resolve by running `hermes models fetch` or setting `context_length` in config.yaml.

### Claude's Discretion
- Exact models included in the static lookup table (which model families, how far back in versions)
- models.dev API response parsing and field mapping
- OpenRouter /models API response parsing (schema is already known from `OPENROUTER_MODELS_URL`)
- Disk cache format and merge strategy details
- `TokenEstimator` API design (method signatures, caching of encoding instances)
- Whether to lazy-init tiktoken encodings or eagerly load at startup
- Exact `hermes models` subcommand UI/output format (follow existing `hermes memory`/`hermes sessions` patterns)
- `/models` slash command handler details and integration with CommandRouter

</decisions>

<specifics>
## Specific Ideas

- The static lookup table should cover at minimum: Claude family (haiku/sonnet/opus across 3.5, 4, 4.5, 4.6), GPT family (gpt-4o, gpt-4o-mini, gpt-4.1, o3, o4-mini), Llama family (3.1, 3.3, 4), Gemini (2.0, 2.5), Mistral/Mixtral, DeepSeek, and Qwen models.
- The alias map needs to handle OpenRouter's `provider/model` prefix format since that's the default provider.
- tiktoken-rs `cl100k_base` covers Claude and GPT-4 models; `o200k_base` covers GPT-4o+ models. Most open-source models use sentencepiece but cl100k_base is a reasonable approximation.
- The disk cache at `~/.ironhermes/models-cache.json` should include a `fetched_at` timestamp per entry so users can see how fresh their cache is.

</specifics>

<deferred>
## Deferred Ideas

- **Pricing/cost tracking** — Track input/output cost per token in ModelMetadata, display session cost in status line. Separate phase.
- **Dynamic model switching mid-session** — `/model <name>` re-resolves metadata and updates all consumers. Currently model is frozen at session start.
- **Model recommendation engine** — Suggest models based on task type (long context → use 1M model, vision → use vision-capable model).
- **Tokenizer accuracy for open-source models** — tiktoken cl100k_base is an approximation for sentencepiece-based models (Llama, Mistral). A future phase could add sentencepiece-rs for exact counts.
- **Auto-refresh cache on staleness** — Automatically fetch from APIs when cache is older than N days. Adds network dependency on startup.

</deferred>
