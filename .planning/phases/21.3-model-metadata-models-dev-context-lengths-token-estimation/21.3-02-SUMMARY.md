**D-06 config-aware context_length on ResolvedEndpoint, tiktoken BPE replacing text.len()/4 heuristic, all 128K hardcoded sites eliminated**

## Accomplishments
- ResolvedEndpoint now carries model_metadata and config_context_length fields with context_length() implementing D-06 full precedence (config.yaml > metadata > 128K default)
- ProviderResolver::build() populates model_metadata from ModelRegistry lookup on all endpoints and stores config_context_length from config.model.context_length
- context_compressor.estimate_tokens() now delegates to global_estimate_tokens (tiktoken BPE) instead of text.len()/4+1 heuristic
- attach_context_engine parameterized with caller-provided context_length (removed hardcoded DEFAULT_CONTEXT_LENGTH constant)
- All four 128_000 hardcoded sites in main.rs replaced with resolver-derived context_length
- Gateway handler constructor and runner resolve context_length from model metadata instead of hardcoding 128K
- Global token estimator initialized from model's tiktoken encoding in CLI (run_single, run_chat) and gateway startup paths
- warm_tiktoken_singletons() called at CLI startup to eagerly initialize BPE tables

## Decisions Made
- D-06 precedence implemented as `context_length()` method on ResolvedEndpoint: user config.yaml always wins, then model metadata, then DEFAULT_CONTEXT_LENGTH (128K) as last resort
- ModelRegistry gets Debug+Clone derives to satisfy ProviderResolver's `#[derive(Debug, Clone)]`
- Hysteresis test recalibrated for tiktoken: wider threshold band (0.05 vs 0.01) and dynamic filler sized from context_length to be robust against both tiktoken and heuristic counting
- Global estimator initialization placed in run_single, run_chat, and gateway build_handler to cover all three entry paths
