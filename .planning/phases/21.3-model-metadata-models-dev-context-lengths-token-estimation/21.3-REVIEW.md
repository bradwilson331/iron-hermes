# Phase 21.3: Code Review Report

**Reviewed:** 2026-04-19T18:42:00Z
**Depth:** standard
**Files Reviewed:** 16
**Status:** issues_found

## Summary

Phase 21.3 introduces model metadata (static table + disk cache from models.dev and OpenRouter APIs), tiktoken-based BPE token estimation replacing the `text.len()/4+1` heuristic, and a `context_length()` method on `ResolvedEndpoint` with D-06 precedence (config > metadata > 128K default). The implementation is well-structured with clear separation of concerns across `model_metadata.rs` (registry + static table + alias map), `models_cache.rs` (disk persistence + API fetch), `token_estimator.rs` (tiktoken BPE singletons), and `provider.rs` (precedence resolution). The wiring into CLI (`main.rs`, `models_cmd.rs`), gateway (`handler.rs`, `runner.rs`), and agent (`agent_wiring.rs`, `context_compressor.rs`) is consistent -- all three entry points initialize the global estimator and resolve context_length from model metadata.

One critical string-slicing bug was found in the context compressor that can panic on multi-byte UTF-8 input. Three warnings cover dead code in the alias resolution path, a `std::ptr::eq` comparison that may produce incorrect results, and the `OnceLock`-based global estimator silently ignoring encoding changes after first initialization.

## Critical Issues

### CR-01: Byte-index string slice panics on multi-byte UTF-8 input

**File:** `crates/ironhermes-agent/src/context_compressor.rs:142`
**Issue:** `&text[..200]` performs a byte-index slice on a `String`. If the text contains multi-byte UTF-8 characters (emoji, CJK, accented characters -- all common in LLM tool output), this panics with `byte index 200 is not a char boundary`. The guard on line 140 checks `text.len() > 500` (byte length), which can be true for strings with fewer than 200 Unicode characters. For example, a string of 170 CJK characters is 510 bytes but only 170 chars -- slicing at byte 200 will split a multi-byte character and panic. This is a runtime crash in production when the context compressor triggers on conversations containing non-ASCII content.
**Fix:**
```rust
// Replace line 142:
format!("{}... [truncated, {} chars total]", &text[..200], text.len()),

// With a char-boundary-safe truncation:
let truncation_point = text.char_indices()
    .nth(200)
    .map(|(i, _)| i)
    .unwrap_or(text.len());
format!("{}... [truncated, {} chars total]", &text[..truncation_point], text.len()),
```

## Warnings

### WR-01: `std::ptr::eq` comparison on borrowed metadata may not work across registries

**File:** `crates/ironhermes-cli/src/models_cmd.rs:252`
**Issue:** `std::ptr::eq(*m, metadata)` compares pointer addresses to find the canonical entry matching a looked-up metadata reference. After `merge_cache()` has been called, the `metadata` reference from `registry.lookup(model)` may point into the cache HashMap, while `all_models()` iterates both the static table and cache -- meaning the same logical entry could exist at two different addresses if lookup resolved via alias. The `ptr::eq` check could then fail to find the canonical ID, falling through to `unwrap_or(model)`. This produces incorrect output (displaying the user's input instead of the canonical ID) rather than a crash, but it is a logic error.
**Fix:** Instead of pointer comparison, use a dedicated method on `ModelRegistry` to resolve a model ID to its canonical form, or compare by value (model ID string matching) rather than by reference identity.
```rust
// Replace lines 249-253 with value-based canonical resolution:
let canonical = model; // default
// Add a `resolve_canonical(model_id) -> &str` method to ModelRegistry
// that returns the canonical ID after alias + prefix resolution.
```

### WR-02: Dead alias-resolution code block does nothing

**File:** `crates/ironhermes-cli/src/models_cmd.rs:256-277`
**Issue:** The loop at lines 262-274 iterates `alias_models` and compares metadata fields, but the inner `if` block on lines 265-270 contains only a comment -- it never pushes to the `aliases` Vec. The result is that `aliases` is always empty when passed to `render_model_info`. This is dead code that suggests incomplete implementation. The comment on lines 275-277 acknowledges this limitation but the loop should either be removed or completed.
**Fix:** Remove the dead loop (lines 256-274) and the `aliases` variable, passing an empty slice directly to `render_model_info`. Or implement alias extraction by exposing the alias map from `ModelRegistry`. Keeping a no-op loop with an empty block confuses future maintainers.

### WR-03: Global token estimator silently ignores encoding mismatch after first init

**File:** `crates/ironhermes-core/src/token_estimator.rs:62-64`
**Issue:** `init_global_estimator` uses `OnceLock::get_or_init`, so the first call wins and subsequent calls with a different encoding are silently ignored. If a user switches models mid-session (e.g., from a cl100k_base model to an o200k_base model via `/model` command), the token estimator will continue using the first model's encoding for the entire process lifetime. This produces incorrect token counts that could cause premature or missed context compression. The doc comment says "Safe to call multiple times" which is technically true (no panic) but misleading about the behavioral semantics.
**Fix:** Either (a) document this limitation clearly and ensure callers understand first-init-wins, or (b) replace `OnceLock` with an `std::sync::RwLock<Option<TokenEstimator>>` that supports re-initialization when the model encoding changes. At minimum, add a `tracing::warn!` when a second call passes a different encoding:
```rust
pub fn init_global_estimator(encoding: TiktokenEncoding) {
    let existing = GLOBAL_ESTIMATOR.get();
    if let Some(est) = existing {
        if est.encoding() != encoding {
            tracing::warn!(
                current = ?est.encoding(),
                requested = ?encoding,
                "global token estimator already initialized with different encoding; ignoring"
            );
        }
        return;
    }
    GLOBAL_ESTIMATOR.get_or_init(|| TokenEstimator::new(encoding));
}
```

## Info

### IN-01: `cmd_info` source detection checks raw model ID against cache before alias resolution

**File:** `crates/ironhermes-cli/src/models_cmd.rs:234`
**Issue:** The `source` variable is determined by `cache.entries.contains_key(model)` using the raw user input before alias resolution. If the user passes an alias like "claude-sonnet-4-20250514", the cache lookup may miss because the cache stores the canonical form "claude-sonnet-4". The source would then incorrectly report "static table" even when the data came from the disk cache.
**Fix:** Perform source detection after alias/canonical resolution, or normalize the model ID before the cache lookup.

### IN-02: `/models info` slash command handler always reports "Source: static table"

**File:** `crates/ironhermes-core/src/commands/handlers.rs:333`
**Issue:** The `/models info <model>` slash command handler hardcodes `"Source: static table"` on line 333, even though it loads and merges the disk cache on lines 298-299. If the metadata came from the cache, the reported source is inaccurate. This is a minor display issue limited to the interactive slash command.
**Fix:** Check whether the model was found in the cache vs static table and report the source accurately, similar to how `cmd_info` in `models_cmd.rs` attempts to do.

### IN-03: `tiktoken-rs` version not pinned to workspace

**File:** `crates/ironhermes-core/Cargo.toml:24`
**Issue:** `tiktoken-rs = "0.11.0"` is the only dependency in `ironhermes-core/Cargo.toml` that is not managed via `workspace = true`. All other dependencies use workspace-level version management. This inconsistency means a workspace-wide version bump would miss this dependency.
**Fix:** Add `tiktoken-rs` to the workspace `[dependencies]` table and reference it here as `tiktoken-rs = { workspace = true }`.

### IN-04: `encode_with_special_tokens` used instead of `encode_ordinary`

**File:** `crates/ironhermes-core/src/token_estimator.rs:43-46`
**Issue:** `encode_with_special_tokens` counts special tokens (like `<|endoftext|>`) as single tokens rather than encoding them as regular text. For user-generated content that happens to contain strings matching special token patterns, this could undercount tokens. `encode_ordinary` is typically more appropriate for counting tokens in arbitrary user text. The practical impact is low since these patterns are rare in natural text, but it is technically less accurate for estimation purposes.
**Fix:** Consider using `encode_ordinary` instead, which treats all text as regular content without special token recognition.
