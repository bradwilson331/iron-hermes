## User Constraints (from CONTEXT.md)

### Locked Decisions

**Model metadata source**
- D-01: Static lookup table (`HashMap<&str, ModelMetadata>`) compiled into the binary with known models (Claude family, GPT family, Llama family, Gemini, Mistral, etc.). Updated each release. Primary source — fast, no network, works offline.
- D-02: Manual API fetch supplements the static table. Users can run `hermes models fetch` or `/models refresh` to query external APIs and cache results to `~/.ironhermes/models-cache.json`. The disk cache is auto-loaded at startup and merged over the static table (cache entries override static entries for the same canonical ID).
- D-03: API fetch uses a fallback chain: try models.dev first (no auth needed, broader cross-provider coverage), fall back to OpenRouter `/models` API if models.dev doesn't have the model (requires `OPENROUTER_API_KEY`).
- D-04: `hermes models` CLI subcommand for setup/browsing (`list`, `fetch`, `info <model>`) plus `/models` slash command for in-session refresh (`/models refresh`, `/models info <model>`). Both share the same underlying metadata service.
- D-05: Model ID normalization uses a canonical ID + alias map. Metadata is stored under canonical IDs (e.g., `claude-sonnet-4`). An alias map resolves provider-prefixed names (`anthropic/claude-sonnet-4`), versioned names (`claude-sonnet-4-20250514`), and legacy names to their canonical form. Lookup: exact match → alias map → strip `provider/` prefix → fallback to DEFAULT_CONTEXT_LENGTH with warning.
- D-06: Resolution precedence for context_length: user config.yaml `context_length` > disk cache entry > static table entry > `DEFAULT_CONTEXT_LENGTH` (128K) with tracing::warn. User config always wins.

**Token estimation accuracy**
- D-07: Replace the crude `text.len() / 4 + 1` heuristic with proper BPE tokenization via the `tiktoken-rs` crate.
- D-08: Encoding selection is model-metadata-driven. `ModelMetadata` includes a `tokenizer` field (e.g., `"cl100k_base"`, `"o200k_base"`). The token estimator uses the encoding specified by the active model's metadata. Fallback to `cl100k_base` for models with unknown tokenizer.
- D-09: tiktoken is used everywhere — all `estimate_tokens`, `estimate_message_tokens`, and `estimate_messages_tokens` callsites in `context_compressor.rs` are updated to use the tiktoken-backed estimator. No dual heuristic/accurate split.

**Metadata scope & fields**
- D-10: `ModelMetadata` struct fields: `context_length: usize`, `max_output_tokens: Option<usize>`, `tokenizer: String` (tiktoken encoding name), `capabilities: ModelCapabilities` (struct with bool fields for `vision`, `tool_use`, `reasoning`, `streaming`).
- D-11: Pricing metadata deferred to future phase. Not included in `ModelMetadata`.

**Wiring & resolution flow**
- D-12: `ModelMetadata` struct and static lookup table live in `ironhermes-core`. Available to all workspace crates.
- D-13: `tiktoken-rs` added as a dependency of `ironhermes-core`. A `TokenEstimator` struct in core wraps encoding selection + counting. Single encoding initialization at startup.
- D-14: `ProviderResolver` carries metadata: `resolve_for_main()` returns a `ResolvedEndpoint` that includes the resolved `ModelMetadata` for the target model. Consumers receive `context_length` from the resolved endpoint instead of hardcoding `128_000`.
- D-15: When a model isn't found in the static table, disk cache, or config, log a `tracing::warn!("unknown model {model_id}, using default 128K context")` and use `DEFAULT_CONTEXT_LENGTH`.

## Summary

Phase 21.3 replaces two hardcoded values — `DEFAULT_CONTEXT_LENGTH = 128_000` and the `text.len() / 4 + 1` token estimation heuristic — with a model-aware metadata system. The work has three well-scoped tracks that can be planned as sequential waves: (1) create the `ModelMetadata` struct and static lookup in `ironhermes-core`, (2) replace token estimation with `tiktoken-rs`, (3) wire metadata through all consumers and add the `hermes models` CLI subcommand + `/models` slash command.

The codebase investigation reveals exactly four hardcoded `128_000` sites: two `with_compression(128_000, ...)` calls and two `tokens_limit: 128_000` assignments in `main.rs`, plus one `context_length: 128_000` default in `AgentLoop::new()`, and one `DEFAULT_CONTEXT_LENGTH = 128_000` constant in `agent_wiring.rs`. The `attach_context_engine` function is the critical choke point — it hard-codes `DEFAULT_CONTEXT_LENGTH` from `agent_wiring.rs`. Once `ProviderResolver` carries `ModelMetadata`, all consumers can receive accurate context lengths via the already-resolved endpoint.

The tiktoken-rs crate is at v0.11.0 [VERIFIED: cargo registry], supports `cl100k_base` and `o200k_base` singletons for zero-cost repeated access, and initialization is synchronous. models.dev provides an open JSON endpoint (`https://models.dev/api.json`) with `limit.context` and `limit.output` fields per model but no tokenizer field. OpenRouter's `/models` endpoint provides `context_length`, `architecture.tokenizer`, and `top_provider.max_completion_tokens` — the richer source for tokenizer info.

**Primary recommendation:** Build `ModelMetadata` + static table in `ironhermes-core` first, add tiktoken-rs there, update `ProviderResolver::resolve_for_main()` to attach metadata, then thread the resolved `context_length` through `attach_context_engine` and `StatusLineState`. The CLI subcommand and slash command are additive and can be a final wave.

### Pattern 1: ModelMetadata Static Lookup Table

**What:** A `HashMap<&'static str, ModelMetadata>` keyed by canonical model ID, plus an alias map resolving versioned/provider-prefixed names to canonical IDs.

**When to use:** At startup for all context_length and tokenizer lookups. No network, no I/O.

**Example:**
```rust
// Source: codebase pattern (SkillRegistry, CommandRouter — HashMap<String, T> lookups with normalized keys)
pub struct ModelRegistry {
    /// canonical ID -> metadata
    table: HashMap<&'static str, ModelMetadata>,
    /// alias -> canonical ID (provider/model, model-YYYYMMDD, legacy names)
    aliases: HashMap<&'static str, &'static str>,
    /// disk cache overlay (loaded at startup, overrides static entries)
    cache: HashMap<String, ModelMetadata>,
}

impl ModelRegistry {
    pub fn lookup(&self, model_id: &str) -> Option<&ModelMetadata> {
        // 1. exact match in cache
        if let Some(m) = self.cache.get(model_id) { return Some(m); }
        // 2. exact match in static table
        if let Some(m) = self.table.get(model_id) { return Some(m); }
        // 3. alias map -> canonical -> static table
        if let Some(&canonical) = self.aliases.get(model_id) {
            return self.table.get(canonical);
        }
        // 4. strip provider prefix (e.g. "anthropic/claude-sonnet-4" -> "claude-sonnet-4")
        if let Some((_, bare)) = model_id.split_once('/') {
            if let Some(m) = self.table.get(bare) { return Some(m); }
            if let Some(&canonical) = self.aliases.get(bare) {
                return self.table.get(canonical);
            }
        }
        None
    }
}
```

### Pattern 2: TokenEstimator with Singletons

**What:** A `TokenEstimator` struct that holds a reference to the tiktoken encoding for the active model, using singletons to avoid repeated initialization.

**When to use:** Replace all `estimate_tokens` / `estimate_message_tokens` / `estimate_messages_tokens` callsites in `context_compressor.rs`.

**Example:**
```rust
// Source: tiktoken-rs 0.11.0 docs.rs — singleton pattern
use tiktoken_rs::{cl100k_base_singleton, o200k_base_singleton};

pub struct TokenEstimator {
    encoding_name: String,
}

impl TokenEstimator {
    pub fn new(encoding_name: &str) -> Self {
        Self { encoding_name: encoding_name.to_string() }
    }

    pub fn count_tokens(&self, text: &str) -> usize {
        let bpe = match self.encoding_name.as_str() {
            "o200k_base" => o200k_base_singleton().read(),
            _ => cl100k_base_singleton().read(), // cl100k_base as fallback
        };
        bpe.encode_with_special_tokens(text).len()
    }
}

// Drop-in replacement for estimate_tokens():
pub fn estimate_tokens(text: &str) -> usize {
    GLOBAL_ESTIMATOR.count_tokens(text)
}
```

**Key detail:** The singleton functions (`cl100k_base_singleton()`, `o200k_base_singleton()`) return an `Arc<RwLock<CoreBPE>>`. Call `.read()` to get the guard, then call methods on it. This is the preferred pattern for repeated calls — avoids re-initializing the BPE tables on every count.

### Pattern 3: ResolvedEndpoint Carries ModelMetadata

**What:** Extend `ResolvedEndpoint` in `provider.rs` with an `Option<ModelMetadata>` field, populated by `ProviderResolver` during `build()`.

**When to use:** All consumers receive `context_length` and tokenizer info from the endpoint they already resolve.

**Example:**
```rust
// Source: codebase — provider.rs ResolvedEndpoint, agent_wiring.rs DEFAULT_CONTEXT_LENGTH

    pub api_key: Option<String>,
    pub api_mode: ApiMode,
    pub default_model: String,

    pub model_metadata: Option<ModelMetadata>,  // NEW: D-14

        self.model_metadata

            .map(|m| m.context_length)
            .unwrap_or(DEFAULT_CONTEXT_LENGTH)
    }
}
```

### Anti-Patterns to Avoid

- **Re-creating BPE tables per call:** Never call `cl100k_base()` or `o200k_base()` (non-singleton) in hot paths — each call re-initializes BPE tables. Always use `_singleton()` variants.
- **Storing CoreBPE guard across await points:** The `RwLockReadGuard` from singletons is not `Send`. Count tokens synchronously; don't hold the guard across `.await`.
- **Making TokenEstimator async:** All tiktoken-rs functions are synchronous. Wrapping in `tokio::spawn_blocking` is unnecessary for typical message sizes.
- **Fetching models API on startup:** API fetch only occurs on explicit `hermes models fetch` or `/models refresh`. Never on startup or per-turn.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| BPE token counting | Custom char/byte heuristic | tiktoken-rs 0.11.0 singletons | BPE vocabulary is 100K+ entries; CL100K handles unicode normalization, special tokens, byte fallback — all edge cases for non-ASCII text |
| HTTP client for API fetch | Custom reqwest setup | Use existing `reqwest` workspace dep (same as web_read tool) | Already configured with rustls-tls, gzip, json; consistent with codebase |
| JSON deserialization of API responses | Manual string parsing | `serde_json` with typed structs | Typed deserialization catches schema changes; `#[serde(default)]` for new fields |
| Cache file path | Hardcoded string | `get_hermes_home().join("models-cache.json")` | Consistent with all other `~/.ironhermes/` paths in codebase |

**Key insight:** The token counting problem looks simple (`text.len() / 4`) but silently degrades for code, non-ASCII content, and compressed text. BPE tokenization accuracy matters specifically because all compression threshold decisions use token counts — an undercount delays compression, an overcount triggers it prematurely.

## Common Pitfalls

### Pitfall 1: RwLockReadGuard Not Send — Can't Hold Across Await

**What goes wrong:** Code holds `let guard = cl100k_base_singleton().read()` then calls `.await` on an async function. Compile error: `guard` is not `Send`.

**Why it happens:** tiktoken-rs singleton returns `Arc<RwLock<CoreBPE>>`; the read guard is not `Send` across thread boundaries.

**How to avoid:** Count tokens in a synchronous block, store the count as `usize`, then proceed with async code. Never hold the guard across an await point.

**Example fix:**
```rust
// WRONG: holds guard across await
let count = cl100k_base_singleton().read().encode_with_special_tokens(text).len();
some_async_fn().await;  // compile error if guard held

// RIGHT: count synchronously, drop guard before await
let token_count = {
    let bpe = cl100k_base_singleton();
    let guard = bpe.read();
    guard.encode_with_special_tokens(text).len()
};
some_async_fn().await;  // fine — guard dropped
```

### Pitfall 2: attach_context_engine Uses Its Own DEFAULT_CONTEXT_LENGTH

**What goes wrong:** After wiring `ModelMetadata` through `ResolvedEndpoint`, the two `with_compression(128_000, ...)` calls in `main.rs` are updated, but `attach_context_engine` in `agent_wiring.rs` still uses its local `DEFAULT_CONTEXT_LENGTH = 128_000` constant at line 55 and 62. The context engine is built with the wrong window size even though `with_compression` received the correct one.

**Why it happens:** `attach_context_engine` is a separate wiring function with its own constant, not reading from `main.rs`'s resolved value.

**How to avoid:** Add `context_length: usize` parameter to `attach_context_engine`. All callers (two in `main.rs`, one in `gateway`) pass the resolved context length. Remove the local constant.

**Warning signs:** Tests still pass with 128K but production with a 1M-context model shows premature compression at 64K tokens.

### Pitfall 3: OpenRouter model IDs Include Provider Prefix

**What goes wrong:** OpenRouter's API returns IDs like `"anthropic/claude-sonnet-4-20250514"`. Storing these directly as keys means lookups from `ModelConfig.default = "anthropic/claude-sonnet-4"` fail because the cache key is versioned.

**Why it happens:** OpenRouter uses `provider/model-YYYYMMDD` format; config uses `provider/model` without date.

**How to avoid:** The alias map (D-05) must cover both forms. When parsing OpenRouter responses: strip the `provider/` prefix, then check for date suffix (`-YYYYMMDD` pattern) and register both versioned and unversioned aliases pointing to the canonical key.

### Pitfall 4: tiktoken-rs Encoding Initialization Is Blocking

**What goes wrong:** First call to `cl100k_base_singleton()` parses the BPE vocabulary file from compiled-in data — this is synchronous and takes ~100ms on first call. If called inside a tokio `async fn` on the first request, it blocks the async runtime.

**Why it happens:** BPE vocab parsing is CPU-bound and happens lazily on first singleton access.

**How to avoid:** Initialize both `cl100k_base_singleton()` and `o200k_base_singleton()` eagerly during startup (before `tokio::main` async body, or in a `std::thread::spawn` initialization step). The singleton caches the result, so all subsequent calls are O(1).

**Warning signs:** First request to the agent after startup is noticeably slower than subsequent ones.

### Pitfall 5: models.dev Has No Tokenizer Field

**What goes wrong:** Plan assumes models.dev can be used as the sole source for tokenizer selection. It cannot — models.dev `api.json` has no tokenizer field.

**Why it happens:** models.dev is a general metadata database optimized for context windows and pricing, not tokenizer specifications.

**How to avoid:** Use the static lookup table as the authoritative source for `tokenizer` field. OpenRouter API (`architecture.tokenizer`) supplements the static table. models.dev only contributes `context_length` and `max_output_tokens` to the cache.

## Code Examples

### TokenEstimator initialization and usage
```rust
// Source: tiktoken-rs 0.11.0 docs.rs [CITED: https://docs.rs/tiktoken-rs/latest/tiktoken_rs/]
use tiktoken_rs::{cl100k_base_singleton, o200k_base_singleton};

/// Estimator backed by tiktoken BPE encoding.
/// Thread-safe: singletons use RwLock internally.
pub struct TokenEstimator {
    encoding: TiktokenEncoding,
}

#[derive(Clone, Copy)]
pub enum TiktokenEncoding {
    Cl100kBase,
    O200kBase,
}

impl TiktokenEncoding {
    pub fn from_name(name: &str) -> Self {
        match name {
            "o200k_base" => Self::O200kBase,
            _ => Self::Cl100kBase, // D-08: fallback to cl100k_base
        }
    }
}

impl TokenEstimator {
    pub fn new(encoding: TiktokenEncoding) -> Self {
        Self { encoding }
    }

    pub fn count(&self, text: &str) -> usize {
        match self.encoding {
            TiktokenEncoding::O200kBase => {
                let bpe = o200k_base_singleton();
                bpe.read().encode_with_special_tokens(text).len()
            }
            TiktokenEncoding::Cl100kBase => {
                let bpe = cl100k_base_singleton();
                bpe.read().encode_with_special_tokens(text).len()
            }
        }
    }
}
```

        let path = get_hermes_home().join("models-cache.json");
        if path.exists() {
            if let Ok(content) = std::fs::read_to_string(&path) {
                if let Ok(cache) = serde_json::from_str(&content) {
                    return cache;
                }
            }
        }
        Self::default()
    }

    pub fn save(&self) -> anyhow::Result<()> {
        let path = get_hermes_home().join("models-cache.json");
        std::fs::create_dir_all(path.parent().unwrap())?;
        let json = serde_json::to_string_pretty(self)?;
        std::fs::write(&path, json)?;
        Ok(())
    }
}

## Open Questions (RESOLVED)

1. **Where should TokenEstimator initialization happen?**
   - What we know: tiktoken-rs first call is blocking ~100ms; singletons cache the result
   - What's unclear: Whether to eagerly init at startup (in `main()` before async) or lazily on first use
   - Recommendation: Eagerly initialize both `cl100k_base_singleton()` and `o200k_base_singleton()` in `main()` sync setup, before `tokio::main`. This costs ~200ms at startup but ensures all async token counting is O(1).
   - **RESOLVED:** Eagerly initialize at startup. Plan 01 Task 2 implements `warm_tiktoken_singletons()` called early in `main()`. Plan 02 Task 2 wires the call in `main.rs` before async runtime entry. The `OnceLock<TokenEstimator>` global is initialized after model resolution via `init_global_estimator()`.

2. **Should `context_compressor.rs` functions become methods on a struct or stay free functions?**
   - What we know: Currently all three are free functions; `ContextCompressor` struct uses them internally
   - What's unclear: The `TokenEstimator` could be stored on `ContextCompressor` or accessed via a global
   - Recommendation: Use a `OnceLock<TokenEstimator>` global in `token_estimator.rs` (initialized from the resolved model's encoding at startup). Free functions in `context_compressor.rs` delegate to the global — minimal API change, all 3 callsites work unchanged.
   - **RESOLVED:** Free functions stay as free functions. Plan 01 Task 2 creates the `OnceLock<TokenEstimator>` global with `global_estimate_tokens()` public API. Plan 02 Task 2 updates `context_compressor.rs` to delegate `estimate_tokens()` to `global_estimate_tokens()`. Function signatures preserved — callers are unaffected.

3. **Gateway wiring for context_length**
   - What we know: `attach_context_engine` is called from both CLI (`main.rs`) and gateway
   - What's unclear: The gateway's `attach_context_engine` call site was not examined in this research
   - Recommendation: The plan should include reading `crates/ironhermes-gateway/src/` to find the gateway call site and ensure it also receives resolved `context_length`.
   - **RESOLVED:** Gateway handler examined during plan revision. `handler.rs` dispatches commands inside an async tokio context (line 200). `GatewayMessageHandler` already has a `context_length: usize` field (line 66, currently hardcoded to 128_000 at line 93). Plan 02 Task 2 updates the gateway `attach_context_engine` call site to pass `context_length` from resolved endpoint metadata. The gateway's `context_length` field will be populated from `resolver.resolve_for_main().context_length()` during handler construction.
