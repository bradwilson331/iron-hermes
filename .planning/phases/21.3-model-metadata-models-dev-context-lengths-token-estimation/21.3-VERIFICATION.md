### Gaps Summary

One gap found: **disk cache not auto-loaded at startup**.

The Plan 04 Task 1 step 4 explicitly called for loading `ModelsCache::load()` at startup in `run_single` and `run_chat` (or inside `ProviderResolver::build()`) and merging cached entries into the ModelRegistry used for context_length resolution. The executor did not implement this step. As a result:

- The cache infrastructure exists and works (D-02 fetch/save/load functions all tested and correct)
- CLI subcommands (`hermes models list/info`) correctly load and merge the cache
- Slash commands (`/models info`) correctly load the cache for display
- **But the runtime resolution chain** (ProviderResolver -> ResolvedEndpoint -> context_length()) **uses only the static table**, not the disk cache

This means if a user runs `hermes models fetch` to update metadata for a model not in the static table (or with a different context_length), that fetched data will NOT be used for context_length resolution during actual agent sessions. It will only be visible in `hermes models list` and `/models info`.

**Impact:** Low-medium. The static table covers 37 common models with correct context lengths. The gap only affects users who:
1. Use models not in the static table AND rely on fetched cache for correct context_length, OR
2. Need fetched context_length overrides for static table models

**Fix:** Add ~3 lines to `ProviderResolver::build()` to call `ModelsCache::load()` and `model_registry.merge_cache(cache.into_metadata_map())` before the endpoint population loop.
