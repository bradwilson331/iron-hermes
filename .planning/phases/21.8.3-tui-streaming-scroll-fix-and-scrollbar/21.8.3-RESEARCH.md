## Summary

Phase 21.8.3 is a narrow bug-fix and UX-enhancement phase operating entirely within two files: `crates/ironhermes-cli/src/tui_rata/app.rs` and `crates/ironhermes-cli/src/tui_rata/ui.rs`. The streaming-scroll bug is caused by two confirmed divergences between `transcript_line_count()` (the model) and `transcript_text()` (the renderer): (1) the model does not account for the role-prefix (`"You: "` / `"Hermes: "`) prepended to the first line of each message, and (2) the model may not correctly handle System messages.

**Critical finding:** The CONTEXT.md was written before Phase 22.4 Plan 22.4-17 landed. That plan changed `role_style()` to return `Some(Color::DarkGray)` for `Role::System` (documented in app.rs:854-874). The `transcript_text()` function therefore does NOT skip System messages any more — it renders them in dim DarkGray. D-07 ("skip System messages in `transcript_line_count`") must be re-evaluated: the correct fix may be to INCLUDE System messages in `transcript_line_count` (matching what `transcript_text()` now does), not skip them. The planner must verify this parity.

Phase 21.8.2-04 is already fully implemented: `scroll_to_bottom()` exists at app.rs:392-395 and is called from all 6 `apply_slash_outcome` arms. The submit() path at app.rs:742 still uses bare `self.auto_follow = true;` — D-09's unification is still pending.

**Primary recommendation:** Fix `transcript_line_count` to match `transcript_text` exactly — account for role-prefix on line 0 of each non-buffer message, and match the System-message treatment to the current `transcript_text` behavior (include, not skip). Add the Scrollbar widget to `ui.rs` using `render_stateful_widget` with a per-render `ScrollbarState`. Everything else (D-08, D-09, D-10, D-11, D-13..D-16) is mechanical.

### Pattern 1: Scrollbar Widget with Per-Render ScrollbarState

**What:** Build a `ScrollbarState` from existing scroll fields on every render; call `render_stateful_widget`. No new field on `App` needed.

**When to use:** D-05 specifies per-render recomputation. This is the recommended approach for the planner (lowest friction — no App field lifecycle to manage).

```rust
// Source: Context7 /ratatui/ratatui — "Draw Scrollbar with Stateful Widget"
use ratatui::layout::Margin;
use ratatui::widgets::{Scrollbar, ScrollbarOrientation, ScrollbarState};

fn render_transcript(frame: &mut Frame, app: &App, area: Rect) {
    let title = format!("Chat [{}]", app.scroll_indicator(area));
    let block = Block::default().borders(Borders::ALL).title(title);
    let text = app.transcript_text();
    let paragraph = Paragraph::new(text)
        .block(block)
        .wrap(Wrap { trim: false })
        .scroll((app.transcript_scroll, 0));
    frame.render_widget(paragraph, area);

    // D-01..D-05: Scrollbar always visible, inside border, right edge.
    // content_length = total lines; position = current scroll offset.
    let total = app.transcript_line_count(area.width as usize);
    let mut scrollbar_state = ScrollbarState::new(total)
        .position(app.transcript_scroll as usize);
    let scrollbar = Scrollbar::new(ScrollbarOrientation::VerticalRight);
    frame.render_stateful_widget(
        scrollbar,
        area.inner(Margin { vertical: 1, horizontal: 0 }),
        &mut scrollbar_state,
    );
}
```

**Key detail:** `area.inner(Margin { vertical: 1, horizontal: 0 })` trims the top/bottom border rows so the scrollbar track aligns with the content area. [VERIFIED: Context7 ratatui Scrollbar example uses `Margin`]

**ScrollbarState field types:** `position` and `content_length` are `usize` (not `u16`) as of ratatui. Cast `transcript_scroll as usize`. [VERIFIED: Context7 ratatui BREAKING-CHANGES note]

### Pattern 2: Role-Prefix Width Accounting in transcript_line_count

**What:** On the first line of each message (i == 0), the rendered width is reduced by the role-label prefix length. The fix makes `transcript_line_count` mirror `transcript_text`'s actual output.

```rust
// Source: Derived from app.rs transcript_text() at lines 781-821 [VERIFIED: codebase]
pub fn transcript_line_count(&self, width: usize) -> usize {
    let mut total = 0usize;
    for msg in &self.history {
        let (role_label, color) = role_style(msg);
        if color.is_none() {
            continue; // mirror transcript_text's let-else skip
        }
        let prefix_len = role_label.len() + 2; // ": " separator
        let body = render_message_body(msg);
        for (i, line) in body.lines().enumerate() {
            let effective_width = if i == 0 {
                width.saturating_sub(prefix_len).max(1)
            } else {
                width
            };
            total = total.saturating_add(wrapped_line_count(line, effective_width));
        }
    }
    if let Some(buf) = &self.assistant_buffer {
        let prefix_len = "Hermes".len() + 2; // "Hermes: "
        for (i, line) in buf.lines().enumerate() {
            let effective_width = if i == 0 {
                width.saturating_sub(prefix_len).max(1)
            } else {
                width
            };
            total = total.saturating_add(wrapped_line_count(line, effective_width));
        }
    }
    total
}
```

**Discretion note for planner:** The exact approach — using `role_label.len() + 2` vs hard-coding `"You: ".len()` / `"Hermes: ".len()` — is left to Claude's discretion (CONTEXT.md). Using `role_label` from `role_style()` is more robust and mirrors what `transcript_text()` uses.

### Pattern 5: Paused Indicator (D-11)

The indicator requires knowing how many new lines arrived since the user last had auto_follow enabled. Two implementation options for the planner (Claude's discretion):

**Option A — Separate counter field on App:**
Add `pub lines_since_paused: usize` to App. Increment it in `handle_stream_event::Delta` when `auto_follow == false`. Reset it in `scroll_to_bottom()`. Show in `scroll_indicator()` when `!auto_follow && (pending_rx.is_some() || assistant_buffer.is_some())`.

## Common Pitfalls

### Pitfall 1: D-07 System Message Handling — Post-22.4-17 State Mismatch

**What goes wrong:** CONTEXT.md D-07 says "skip System messages in `transcript_line_count` — symmetric to `transcript_text()` skipping them." But Phase 22.4 Plan 22.4-17 changed `role_style()` to return `Some(Color::DarkGray)` for `Role::System`, so `transcript_text()` now RENDERS System messages (in dim gray). The skip no longer exists.

**Why it happens:** CONTEXT.md was written before the 22.4-17 fix landed. The "Option B" fix for System rendering is now in production (app.rs:867-874). The `let Some(color) = color else { continue };` pattern in `transcript_text()` only skips if `color` is `None` — and no current `Role` variant returns `None`.

**How to avoid:** The fix for D-07 in `transcript_line_count` should INCLUDE System messages using the same `role_style()` lookup as `transcript_text()` does. Concretely: if `color.is_none()` then `continue`; otherwise count the lines. Since no role currently returns `None`, this means System IS counted — which fixes the previously-confirmed divergence (System was being counted with full width but rendered with a `"System: "` prefix).

**Warning signs:** If tests show System messages still causing a line-count discrepancy after the "fix", the `role_style()` return value for System changed again.

[VERIFIED: app.rs lines 867-874 — `Role::System` returns `Some(Color::DarkGray)`, not `None`]

### Pitfall 2: Scrollbar track overlapping border

**What goes wrong:** Rendering the Scrollbar at `area` directly places it on top of the border characters.

**Why it happens:** `area` includes the border columns/rows for the `Block`. The content area is `area.inner(Margin{vertical:1, horizontal:0})`.

**How to avoid:** Always use `area.inner(Margin { vertical: 1, horizontal: 0 })` as the render area for the Scrollbar. [VERIFIED: Context7 example]

### Pitfall 3: submit() D-09 — Verify No Callers Depend on Bare auto_follow

**What goes wrong:** Replacing `self.auto_follow = true;` with `self.scroll_to_bottom()` at app.rs:742 also zeros `transcript_scroll = 0`. Tests or callers that expect `transcript_scroll` to retain its value after `submit()` will break.

**Why it happens:** `scroll_to_bottom()` sets `transcript_scroll = 0` for symmetry; `reconcile_scroll` on the next render tick resets it to `max` anyway, so this is invisible in production. But any test that inspects `transcript_scroll` immediately after `submit()` without going through `reconcile_scroll` would see 0 instead of the prior value.

**How to avoid:** Verify the existing test `non_slash_submit_creates_pending_rx_and_pending_tx` (app.rs:1223) does not assert on `transcript_scroll`. It doesn't (only asserts `pending_rx`, `pending_tx`, and `history.len()`). Safe to proceed. [VERIFIED: app.rs:1223-1246]

### Pitfall 4: End key falling through to textarea input

**What goes wrong:** If the `End` key arm is not placed BEFORE the `_ => { let _ = self.textarea.input(key); }` catch-all, the key is consumed by the textarea widget instead of triggering `scroll_to_bottom()`.

**Why it happens:** The match arm order in `handle_key` is significant.

**How to avoid:** Insert the `End` arm immediately after the `PageDown` arm (app.rs:494), before the `Esc` arm. [VERIFIED: app.rs:493-503]

### Pitfall 5: wrapped_line_count with width=0 after prefix subtraction

**What goes wrong:** If `prefix_len >= width` (very narrow terminal), `effective_width = width.saturating_sub(prefix_len)` returns 0. `wrapped_line_count("...", 0)` returns 1 (defensive) — so this is safe, but the result will differ from what ratatui actually renders (which wraps differently at very narrow widths).

**Why it happens:** Edge case at terminal widths < 8 (less than `"Hermes: ".len()`).

**How to avoid:** Use `.max(1)` after `saturating_sub` (already shown in Pattern 2 above). Ratatui's word-wrap behavior at sub-prefix widths is unspecified anyway — this is an acceptable edge-case approximation. [ASSUMED]
