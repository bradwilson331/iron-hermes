## Summary

The phase delivers four targeted fixes: role-prefix-aware `transcript_line_count`, System-message inclusion in line counting, `scroll_to_bottom()` on `StreamEvent::Finished`, and a `scroll_to_bottom()` call in `submit()`. A scrollbar widget is also added to `ui.rs`. The core logic of each fix appears correct. However, there are several correctness and robustness issues: a width mismatch between the scrollbar state and the Paragraph scroll position (they use different effective widths), `scroll_down` never re-engages `auto_follow` directly and relies entirely on `reconcile_scroll` for this, an off-by-one in scrollbar `total` vs `max_scroll`, cursor overflow on wide terminals, and minor quality issues.

## Critical Issues

### CR-01: Scrollbar `total` and Paragraph `scroll` use mismatched effective widths — scrollbar thumb position is wrong

**File:** `crates/ironhermes-cli/src/tui_rata/ui.rs:68-70`

**Issue:** `render_transcript` passes `area.width as usize` to `transcript_line_count` for the scrollbar's `total`. But the Paragraph widget renders inside a bordered block, so its effective content width is `area.width - 2` (one column each for left and right borders). The `transcript_max_scroll` helper (called by `reconcile_scroll`) uses `area.width - 2` implicitly via `area.inner` semantics — but `render_transcript` does not. This means the scrollbar's `total` is computed for a wider width than the Paragraph wraps at. On content that wraps (i.e., lines near the column boundary), `total` is under-counted relative to the actual rendered row count, causing the scrollbar thumb to appear too low and the position ratio to be off. In the worst case it makes the thumb show "at bottom" when the user has not scrolled to the actual bottom.

The correct call should use `(area.width.saturating_sub(2)) as usize` (matching the bordered inner width that `Paragraph::wrap` actually uses).

**Fix:**
```rust
// ui.rs line 68 — match the bordered inner width that Paragraph uses
let total = app.transcript_line_count(area.width.saturating_sub(2) as usize);
```

Note: `scroll_indicator` and `reconcile_scroll` call `transcript_max_scroll(area)` which itself calls `transcript_line_count(area.width as usize)` — they share the same mismatch. If this CR is fixed in `ui.rs`, the same fix should be applied in `transcript_max_scroll`:

```rust
// app.rs line 426 — use the bordered inner width
pub fn transcript_max_scroll(&self, area: Rect) -> u16 {
    let total = self.transcript_line_count(area.width.saturating_sub(2) as usize) as u32;
    let visible = area.height.saturating_sub(2) as u32;
    total.saturating_sub(visible).min(u16::MAX as u32) as u16
}
```

## Warnings

### WR-01: `scroll_down` never re-engages `auto_follow` — relies silently on `reconcile_scroll` as an invisible contract

**File:** `crates/ironhermes-cli/src/tui_rata/app.rs:371-374`

**Issue:** `scroll_down` increments `transcript_scroll` but never sets `auto_follow = true`. Auto-follow is re-engaged only indirectly when `reconcile_scroll` notices `transcript_scroll >= max`. This is a silent contract: if `reconcile_scroll` is ever not called on the next render tick (e.g., during a test or a future refactor that skips the render path), scrolling down past the bottom leaves `auto_follow = false` indefinitely. The docstring says "auto-follow re-enables via `reconcile_scroll`" which confirms this dependency, but makes the behavior non-obvious and fragile. At minimum the intent should be made explicit.

**Fix:** Either make `scroll_down` check `>= max` itself (if area is available) and re-engage, or add a visible assertion/comment at `reconcile_scroll` call sites making the contract explicit. The safest mechanical fix is to add a note to scroll_down's doc that this is intentional and tested, but a structural fix would have `reconcile_scroll` always be called unconditionally in the event loop, guarded by a lint.

### WR-02: Scrollbar `ScrollbarState::new(total)` uses raw line count, not `max_scroll` — thumb position is semantically incorrect

**File:** `crates/ironhermes-cli/src/tui_rata/ui.rs:69-70`

**Issue:** `ScrollbarState::new(total).position(app.transcript_scroll as usize)` sets the scrollbar "content length" to the total number of wrapped lines, and the "position" to the current scroll offset. Ratatui's `ScrollbarState` interprets these as: thumb position = `position / total * track_height`. But `app.transcript_scroll` is clamped to `transcript_max_scroll` (= `total - visible_rows`), not to `total`. This means the thumb never reaches the bottom of the track during normal scrolling — it stops at `(total - visible) / total` fraction. The correct semantic is `ScrollbarState::new(max_scroll).position(transcript_scroll)`, where `max_scroll = total.saturating_sub(visible_rows)`. That makes the thumb reach the track bottom exactly when the user is at the bottom.

**Fix:**
```rust
// ui.rs render_transcript — use max_scroll as the content length
let total_lines = app.transcript_line_count(area.width.saturating_sub(2) as usize);
let visible_rows = area.height.saturating_sub(2) as usize;
let max_scroll = total_lines.saturating_sub(visible_rows);
let mut scrollbar_state = ScrollbarState::new(max_scroll)
    .position(app.transcript_scroll as usize);
```

### WR-03: `render_cursor` can overflow terminal width — cursor placed outside terminal bounds on wide content

**File:** `crates/ironhermes-cli/src/tui_rata/ui.rs:105-106`

**Issue:** `cursor_x` is computed as `area.x.saturating_add(col as u16).saturating_add(1)`. If `col` (the cursor column within the textarea) exceeds `area.width - 2` (the inner content width), the resulting `cursor_x` will exceed the terminal width. Ratatui's `Frame::set_cursor_position` does not clamp to terminal dimensions — out-of-bounds cursor positions can cause visual glitches or corrupt the cursor state on some terminals. This is reachable when the user types a long line that soft-wraps but tui-textarea reports the logical column, not the visual column.

**Fix:**
```rust
let max_cursor_x = area.x.saturating_add(area.width).saturating_sub(1);
let cursor_x = area.x.saturating_add(col as u16).saturating_add(1).min(max_cursor_x);
let cursor_y = area.y.saturating_add(row as u16).saturating_add(1);
frame.set_cursor_position(Position::new(cursor_x, cursor_y));
```

### WR-04: `wrapped_line_count` uses `char` count — not display column width — so wide Unicode characters (CJK, emoji) are miscounted

**File:** `crates/ironhermes-cli/src/tui_rata/app.rs:926-935`

**Issue:** `wrapped_line_count` counts `line.chars().count()` and divides by `width`. This assumes every Unicode scalar value occupies exactly one terminal column, which is false. CJK characters, emoji, and many other code points occupy two columns. A line of 40 CJK characters would be counted as 40 columns but rendered as 80 columns, causing `transcript_line_count` to under-count by half for such content. The result: `transcript_max_scroll` returns a value too small, `reconcile_scroll` snaps to the wrong bottom, and the user cannot scroll to the actual bottom of CJK-heavy responses.

The correct fix is to use a Unicode display-width library (e.g., `unicode_width::UnicodeWidthStr::width(line)`) instead of `chars().count()`.

**Fix:**
```rust
// In Cargo.toml (ironhermes-cli): unicode-width = "0.2"
// In app.rs:
use unicode_width::UnicodeWidthStr;

pub(crate) fn wrapped_line_count(line: &str, width: usize) -> usize {
    if line.is_empty() {
        return 1;
    }
    let cols = UnicodeWidthStr::width(line);
    if width == 0 {
        return 1;
    }
    (cols + width - 1) / width
}
```

## Info

### IN-01: `pending_skill_overlays` is initialized with `Vec::new()` in `App::new` but `AppDeps` also carries the field — the deps value is silently dropped

**File:** `crates/ironhermes-cli/src/tui_rata/app.rs:157-158, 359`

**Issue:** `AppDeps.pending_skill_overlays` is declared and documented as "pending activated-skill overlays" that the SkillActivated arm pushes to. However, `App::new` ignores `deps.pending_skill_overlays` and always initializes with `Vec::new()` (line 359). If a caller pre-seeds `AppDeps.pending_skill_overlays`, those entries are silently dropped. Currently only `test_deps()` seeds it with an empty vec, so there is no runtime impact. But the discrepancy between the AppDeps field declaration and its use is misleading and could cause a real bug if someone expects pre-seeded overlays to survive into the App.

**Fix:** Either forward the value (`pending_skill_overlays: deps.pending_skill_overlays`) or remove the field from `AppDeps` since it is always initialized empty.

### IN-02: `dispatch_or_submit` and `submit` both handle slash routing — duplicated logic with subtle difference

**File:** `crates/ironhermes-cli/src/tui_rata/app.rs:560-568, 759-769`

**Issue:** Both `dispatch_or_submit` (the primary Enter handler) and `submit` contain a slash-prefix check and call `dispatch_slash_blocking`. The `submit` version is documented as a "defensive re-check" but the duplication means any future change to slash routing logic must be applied in two places. The comment at line 756 ("Defensive re-check: slash input must never enter history as User") is accurate but the pattern is fragile — a future refactor that calls `submit` directly bypasses `dispatch_or_submit` and still works, but any divergence between the two checks would be silent. This is a maintainability concern rather than a current bug.

**Fix:** Extract the slash-check-and-dispatch into a single private method to eliminate the duplication.
