**Edit 2 — Scrollbar render appended to `render_transcript` (after line 62):**

```rust
    // D-01..D-05: Scrollbar always visible, inside border, right edge, default style.
    // ScrollbarState built per-render from authoritative App fields — no cached state.
    // area.inner(Margin{vertical:1, horizontal:1}) trims all four border cells so the
    // track renders at column width-2 (inside the right border) not on the border char.
    let total = app.transcript_line_count(area.width as usize);
    let mut scrollbar_state = ScrollbarState::new(total)
        .position(app.transcript_scroll as usize);
    let scrollbar = Scrollbar::new(ScrollbarOrientation::VerticalRight);
    frame.render_stateful_widget(
        scrollbar,
        area.inner(Margin { vertical: 1, horizontal: 1 }),
        &mut scrollbar_state,
    );
```

Insertion point: after `frame.render_widget(paragraph, area);`, before the closing `}` of `render_transcript`. Existing `Paragraph` render, block title, and scroll indicator are unchanged (D-04 coexistence).

## Deviations from Plan

### [Rule 1 - Bug] Margin horizontal corrected from 0 to 1

**Found during:** Task 2 GREEN — snapshot test still FAILED after adding Scrollbar

**Issue:** Plan's `Margin { vertical: 1, horizontal: 0 }` places the VerticalRight scrollbar track at column 79 (x + width - 1 = 0 + 80 - 1 = 79), which is the right border character position. The border char is already drawn there by the Block widget. The track painted by the Scrollbar would overlay the border char but our test checks column 78.

**Fix:** Changed to `Margin { vertical: 1, horizontal: 1 }` — inner area starts at x=1, width=78, so rightmost column = 1 + 78 - 1 = 78. Track renders at col 78, visually inside the border at col 79. This correctly places the scrollbar in the content area.

**Files modified:** `crates/ironhermes-cli/src/tui_rata/ui.rs`

**Commit:** `bf7cd0b0`

### [Rule 1 - Bug] Test row range tightened from 1..23 to 1..17

**Found during:** Task 1 RED — test passed unexpectedly with original row range 1..23

**Issue:** Bottom-border chars (`─`) from adjacent layout blocks (knight-rider, status, input) land at rows 17+ in column 78 of the full 24-row buffer, causing the pre-fix test to pass as if a scrollbar was present.

**Fix:** Narrowed assertion to rows `1u16..17` — well within the transcript content area (transcript height=19 means content rows 1..17), cleanly separated from all adjacent block border chars.

**Files modified:** `crates/ironhermes-cli/src/tui_rata/ui.rs`

**Commit:** `358c4c51`
