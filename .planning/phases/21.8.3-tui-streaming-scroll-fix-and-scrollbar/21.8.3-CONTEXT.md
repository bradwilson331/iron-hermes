# Phase 21.8.3: tui-streaming-scroll-fix-and-scrollbar — Context

**Gathered:** 2026-05-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix the rata-TUI's agent-turn streaming-scroll bug surfaced post-21.8.2 UAT and add a visible Scrollbar widget for transcript navigation feedback.

**The bug (confirmed via bisect against `triage/pre-plan-04` @ `29bf199d`):** During long streaming responses, the viewport stops following the assistant output before the actual bottom. Reproducible by asking for the Atlanta weather (or any 10+ line response) — the TUI cuts off mid-stream (humidity-line cutoff at `29bf199d`, temperature-line cutoff at `10ddb712`). Both states fail; plan 21.8.2-04 did not cause the regression. Root cause hypothesis (locked): `transcript_line_count()` (`crates/ironhermes-cli/src/tui_rata/app.rs:425-440`) under-counts wrapped rows because it ignores the role-prefix (`"You: "` / `"Hermes: "`) that `transcript_text()` (line 798) prepends to the first line of each message, AND it counts System messages that `transcript_text()` skips (line 785, `role_style` returns `None` for `Role::System`). The two divergences together cause `transcript_max_scroll()` to return a value that does not equal the rendered total, so `auto_follow=true` snaps `transcript_scroll = max` short of (or past) the real bottom.

Phase fixes the line-count parity for these two specific divergences, adds a `StreamEvent::Finished` snap-to-bottom safety net, unifies submit() with the `scroll_to_bottom()` helper introduced in plan 21.8.2-04, adds an End / Ctrl+End jump-to-bottom binding, adds a "paused (N new lines below)" indicator while the user is scrolled up mid-stream, and renders a ratatui `Scrollbar` widget on the right edge of the transcript pane.

**In scope:** `crates/ironhermes-cli/src/tui_rata/app.rs` (line-count fixes, helper reuse, key binding, paused indicator), `crates/ironhermes-cli/src/tui_rata/ui.rs` (Scrollbar widget rendering, ScrollbarState wiring). Tests live in `mod scroll_tests` (gated on `feature = "test-support"`).

**Out of scope:** Full audit of every `transcript_line_count` vs `transcript_text` divergence (only the two confirmed ones); property-based testing harness; mouse wheel sensitivity tuning; horizontal scrolling; accessibility/screen-reader cues; `scroll_to_top` / `End` interaction with non-streaming history navigation beyond simple jump.

</domain>

<decisions>
## Implementation Decisions

### Scrollbar Widget
- **D-01:** Render the ratatui `Scrollbar` widget on the **right edge, inside the transcript border**, using `ScrollbarOrientation::VerticalRight`. Border absorbs the column; no horizontal content reflow.
- **D-02:** Scrollbar is **always visible** — empty track when content fits one screen, populated when it overflows. No conditional rendering, no fade animation. Predictable layout matters more than minimalism.
- **D-03:** Use **default ratatui `Scrollbar::default()` style** — thin track, filled thumb, terminal-theme colors. No custom thumb/track characters. Zero risk of breaking on narrow / non-Unicode terminals.
- **D-04:** **Keep the existing border-title indicator** (`scroll_indicator()` at app.rs:397 — `"live"` / `"scroll N/M"`) AND add the scrollbar. Textual readout for accessibility/glance + visual bar for at-a-glance position. Redundant by design.
- **D-05:** **Live thumb tracking during streaming** — recompute `ScrollbarState` (position + content_length) on every render tick from `transcript_scroll` and `transcript_max_scroll(area)`. Same cost as the existing `reconcile_scroll` path already pays per-tick.

### Streaming-Follow Fix Scope
- **D-06:** **Narrow fix** — only repair the role-prefix accounting in `transcript_line_count` (account for `"You: "` / `"Hermes: "` prefix length on the first line of each non-System message; subsequent wrapped rows do not include the prefix). No speculative full audit of other potential divergences.
- **D-07:** **Also skip System messages in `transcript_line_count`** — symmetric to `transcript_text()` skipping them at line 785. Same bug class as D-06, separately confirmed in discussion. Two specific bugs locked, not a broad audit.
- **D-08:** **Add a snap-to-bottom safety net on `StreamEvent::Finished`** — defense-in-depth. After `commit_assistant_buffer` runs (app.rs:678), if `auto_follow == true` then explicitly call `scroll_to_bottom()` (or set `transcript_scroll = max` via reconcile). Cheap insurance against future line-count drift.
- **D-09:** **Reuse `scroll_to_bottom()` on the agent-turn path** — replace `submit()`'s bare `self.auto_follow = true;` (app.rs:742) with `self.scroll_to_bottom();`. Single source of truth for "snap to bottom" semantics. Acceptable because `reconcile_scroll` runs every render before draw, so the brief `transcript_scroll = 0` from the helper is invisible.

### Scroll UX
- **D-10:** **Add `End` and `Ctrl+End` key bindings** to `handle_key` (app.rs:462) — both call `self.scroll_to_bottom()`. Standard terminal mental model. Sits next to existing `PageUp`/`PageDown` (lines 493-494).
- **D-11:** **"Paused (N new lines below)" indicator** when `auto_follow == false` AND a stream is active (`pending_rx.is_some()` or `assistant_buffer.is_some()`). `N` is the count of lines added since auto_follow was last true. Reuses `scroll_indicator()` extension; not a new field — derive from existing state plus a tracked counter that resets on every `scroll_to_bottom()` / auto_follow re-engage.
- **D-12:** **Submit() continues to re-engage auto-follow** (preserves existing behavior at app.rs:742). Sending a new prompt is an explicit action — user expects to see the response. End / Ctrl+End is the manual override path.

### Test Strategy
- **D-13:** **Targeted unit tests in `mod scroll_tests`** (gated on `feature = "test-support"`, app.rs:1002-onward) covering: (a) long User message produces correct wrapped-line count accounting for `"You: "` prefix at narrow widths, (b) System message contributes 0 to `transcript_line_count` (parity with `transcript_text` skip), (c) regression test for the bug-as-experienced — long assistant_buffer growth + `auto_follow=true` keeps `reconcile_scroll` clamping `transcript_scroll = max` such that the last rendered line is visible.
- **D-14:** **Snapshot test via `ratatui::backend::TestBackend`** for the Scrollbar widget render at a fixed `(width, height)` with a known transcript_scroll/max. Captures the exact buffer cells. Standard ratatui pattern. Catches regressions in scrollbar position, character set, visibility.
- **D-15:** **Unit test on `handle_key` for `End` / `Ctrl+End`** — simulate `KeyEvent { code: KeyCode::End, modifiers: NONE }` and `KeyCode::End + CONTROL`, assert post-call `auto_follow == true` and `transcript_scroll == 0`. Pattern matches existing PageUp/PageDown coverage.
- **D-16:** **Manual UAT — Atlanta weather reproduction** is the bare-minimum end-to-end check. Same prompt the user used to surface the bug; confirm the final line of the response (e.g., the "Heads up" trailing paragraph) is visible without manual scroll. Documented in `21.8.3-UAT.md` post-implementation.

### Claude's Discretion
- The exact subtraction logic for the role-prefix in `transcript_line_count` (e.g., whether to compute prefix length from `role_style`'s label string + `": "` separator, or hard-code `"You: ".len()` / `"Hermes: ".len()`). Implementation detail.
- ScrollbarState lifecycle (per-render recomputation vs cached field on App). The planner picks the lowest-friction option.
- "N new lines below" counter mechanics (separate field on App vs derived from `transcript_max_scroll - transcript_scroll`). The planner decides based on edge cases (does it reset on resize? on user-initiated `scroll_down`?).
- Whether to also extend `scroll_to_top` symmetrically with `Home` / `Ctrl+Home` (the discussion focused on bottom-jump; symmetric top-jump is a low-friction tag-along but not locked).

</decisions>

<specifics>
## Specific Ideas

- **Bug-as-experienced is the regression test.** Atlanta weather prompt (or equivalent 10+ line response) — the user's screenshot is the single source of truth for "fixed". `21.8.3-UAT.md` documents this scenario verbatim; D-16 enforces it.
- **The two known divergences from `transcript_text` are the only confirmed bugs.** Don't speculatively add fixes for unverified ones (e.g., trailing-newline edge cases) without evidence. If a parity test is added later, it can drive new fixes phase-by-phase.
- **Scrollbar must NOT shift transcript content.** D-01's "inside-border-right" choice is specifically because the border absorbs the scrollbar column — no Layout change required.
- **Helper unification is safe but verify no other callers depend on submit() not zeroing transcript_scroll.** Audit for `submit()` callers; if any rely on the bare `auto_follow=true`, adjust during planning.
- **`triage/pre-plan-04` branch is preserved** at `29bf199d` if mid-implementation bisect is needed.

</specifics>
