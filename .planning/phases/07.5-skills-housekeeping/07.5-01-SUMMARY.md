# Phase 07.5 Plan 01: SKILL-06 allowed_tools Enforcement Summary

**One-liner:** Active skill tracking via shared Arc<Mutex<Vec<SkillRecord>>> with union-based allowed_tools enforcement in execute_tool_call, deactivate action, and full construction-site wiring.

## What Was Done

### Task 1: Active skills tracking + deactivate action (97dad98)

**AgentLoop (`agent_loop.rs`):**
- Added `active_skills: Arc<std::sync::Mutex<Vec<ironhermes_core::SkillRecord>>>` field
- Added `with_active_skills()` builder method following existing `with_hook_registry` pattern
- Added `active_skills()` getter for shared Arc access
- Default initialization with empty Vec (backward compatible)

**SkillsTool (`skills_tool.rs`):**
- Added `active_skills` field to struct, updated constructor to accept shared Arc
- Updated `handle_activate` to push SkillRecord into active_skills with dedup check
- Added `handle_deactivate` to remove named skill from active_skills
- Updated SKILLS_DESCRIPTION, schema enum, execute match, and error messages for deactivate
- Added `make_tool_with_skills_and_active` test helper
- Added 5 new tests: activate pushes, duplicate idempotent, deactivate removes, deactivate nonactive, deactivate missing name

**Registry (`registry.rs`):**
- Updated `register_skills_tool` signature to accept `active_skills` parameter
- Passes active_skills through to SkillsTool constructor

**CLI (`main.rs`):**
- Updated gateway `register_skills_tool` call to pass new active_skills Arc

### Task 2: Enforcement in execute_tool_call + wiring (46c893b)

**AgentLoop enforcement (`agent_loop.rs`):**
- Inserted SKILL-06 enforcement block BEFORE guardrail check (line 340 vs 389)
- Only enforces when at least one active skill has `allowed_tools: Some(...)` (D-04)
- Computes union of all restricting skills' allowed_tools lists (D-05)
- Skills tool ("skills") always inserted into allowed set (D-07)
- Blocked tools fire ToolCompleted{success:false} without ToolCalled (D-08)
- Actionable error message names allowed tools and suggests fix (D-09)
- Mutex poisoning recovery via `unwrap_or_else(|e| e.into_inner())` (T-07.5-03)

**Handler (`handler.rs`):**
- Added `active_skills` field to GatewayMessageHandler
- Initialized in constructor, passed via `.with_active_skills()` when building AgentLoop

**6 enforcement tests added:**
- `test_skill_enforcement_blocks_unlisted_tool` - blocked tool gets enforcement error
- `test_skill_enforcement_allows_listed_tool` - listed tool executes normally
- `test_skill_enforcement_inactive_means_all_allowed` - empty active_skills = unrestricted
- `test_skill_enforcement_skills_tool_always_allowed` - D-07 verified
- `test_skill_enforcement_union_of_multiple_skills` - union semantics verified
- `test_skill_enforcement_none_allowed_tools_ignored` - D-06 None vs Some semantics

## Decisions Made

1. **Mutex poisoning recovery**: Used `unwrap_or_else(|e| e.into_inner())` per T-07.5-03 to avoid panics in agent loop
2. **Skills tool always allowed**: D-07 ensures the agent can never be locked out of deactivating a restrictive skill
3. **Cron path unchanged**: execute_cron_job uses default empty active_skills (all tools allowed), no explicit wiring needed since AgentLoop::new() initializes with empty Vec
