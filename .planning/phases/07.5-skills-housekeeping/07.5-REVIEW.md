# Phase 07.5: Code Review Report

**Reviewed:** 2026-04-10T12:00:00Z
**Depth:** standard
**Files Reviewed:** 6
**Status:** issues_found

## Summary

Reviewed the skills housekeeping changes across the agent loop, CLI, gateway handler, gateway runner, tool registry, and skills tool. The skills activation/deactivation lifecycle and tool enforcement logic are well-structured with good test coverage. However, there is one critical bug where a string slice can panic on multi-byte UTF-8 boundaries, several warning-level issues around silent failures on poisoned mutexes and mismatched request IDs for hook event correlation, and minor informational items.

## Critical Issues

### CR-01: Potential panic on multi-byte UTF-8 boundary in args preview

**File:** `crates/ironhermes-agent/src/agent_loop.rs:320`
**Issue:** The tool progress preview truncates `args_str` by byte index (`&args_str[..100]`). If `args_str` contains multi-byte UTF-8 characters (e.g., from user input in non-Latin scripts, or JSON-escaped unicode), slicing at byte offset 100 will panic at runtime with `byte index 100 is not a char boundary`.
**Fix:**
```rust
let preview = if args_str.len() > 100 {
    match args_str.char_indices().nth(100) {
        Some((idx, _)) => format!("{}...", &args_str[..idx]),
        None => args_str.clone(),
    }
} else {
    args_str.clone()
};
```
Alternatively, use `args_str.get(..100).unwrap_or(args_str)` -- but this has the same byte-boundary problem. The `char_indices` approach is correct. Another option is `args_str.chars().take(100).collect::<String>()` followed by appending `"..."`.

## Warnings

### WR-01: MessageReceived and ResponseSent use different request_ids, breaking correlation

**File:** `crates/ironhermes-gateway/src/handler.rs:221,392`
**Issue:** In `run_agent`, a `request_id` is generated at line 221 for the `MessageReceived` hook event, but a completely different `uuid::Uuid::new_v4()` is generated at line 392 for the `ResponseSent` event. The purpose of `request_id` is to correlate request/response pairs in observability pipelines (JSONL logs, webhooks). Using different IDs breaks this correlation. The cron path in `runner.rs:618` correctly reuses the same `request_id` for both events.
**Fix:**
```rust
// Line 392: reuse the request_id from line 221 instead of generating a new one
let hook_event = ironhermes_hooks::HookEvent::new(
    &request_id,  // reuse, not &uuid::Uuid::new_v4().to_string()
    ironhermes_hooks::HookEventKind::ResponseSent { ... },
);
```
This requires hoisting the `request_id` binding to be available later in the function (it is currently scoped inside the `if let Some(ref registry)` block at line 220).

### WR-02: Poisoned mutex panic in cron skill pre-population

**File:** `crates/ironhermes-gateway/src/runner.rs:669`
**Issue:** `active_skills.lock().unwrap()` will panic if the mutex is poisoned. This is inconsistent with the poison-recovery pattern used in `agent_loop.rs:342` (`unwrap_or_else(|e| e.into_inner())`). A panic here would kill the entire cron tick task, preventing all future cron jobs from running.
**Fix:**
```rust
let mut skills_guard = active_skills.lock().unwrap_or_else(|e| e.into_inner());
```

### WR-03: Silent skill activation failure on poisoned mutex

**File:** `crates/ironhermes-tools/src/skills_tool.rs:76`
**Issue:** `handle_activate` uses `if let Ok(mut skills) = active_skills.lock()` which silently swallows a poisoned mutex error. The function returns `{"status": "ok", "name": ..., "content": ...}` even when the skill was NOT actually added to the active set. The LLM and any downstream enforcement logic will believe the skill is active, but it is not -- tool calls that should be restricted will remain unrestricted (or vice versa).
**Fix:** Either use the poison-recovery pattern or return an error status:
```rust
let mut skills = active_skills.lock().unwrap_or_else(|e| e.into_inner());
if !skills.iter().any(|s| s.name == canonical_name) {
    skills.push(record.clone());
}
```

### WR-04: Empty API key used silently in multiple paths

**File:** `crates/ironhermes-gateway/src/handler.rs:346`, `crates/ironhermes-gateway/src/runner.rs:637`
**Issue:** Both `handler.rs` and `runner.rs` call `config.resolve_api_key().unwrap_or_default()`, which substitutes an empty string when no API key is configured. This produces a confusing downstream error from the LLM provider (e.g., "401 Unauthorized") rather than a clear "no API key configured" message. The CLI path in `main.rs:489` correctly uses `.context("No API key configured...")` to fail with a clear message.
**Fix:**
```rust
let api_key = config
    .resolve_api_key()
    .context("No API key configured for cron/gateway agent")?;
```

## Info

### IN-01: CLI mode does not wire active_skills to AgentLoop

**File:** `crates/ironhermes-cli/src/main.rs:237,363`
**Issue:** In `run_single` and `run_agent_turn`, the `AgentLoop` is constructed without calling `.with_active_skills(...)`. This means skill-based tool enforcement (SKILL-06) does not apply when using the CLI. This may be intentional since the CLI does not register the skills tool either (only `register_defaults` is called), but it creates an asymmetry between gateway and CLI behavior.
**Fix:** If intentional, add a comment documenting the decision. If not, wire skills in the CLI path similarly to the gateway.

### IN-02: Duplicate guardrail checking logic in dispatch_with_hook

**File:** `crates/ironhermes-tools/src/registry.rs:166-188`
**Issue:** `dispatch_with_hook` contains its own inline guardrail checking loop (lines 166-188) that duplicates the logic in `check_guardrails` (lines 86-109). Now that `agent_loop.rs` uses the split `check_guardrails` + `execute_tool` API (D-05), the `dispatch_with_hook` method and its inline guardrail loop may be dead code in production. If no callers remain, it could be removed to reduce maintenance burden.
**Fix:** Verify whether `dispatch_with_hook` or `dispatch` are still called in production paths. If not, consider deprecating or removing them to avoid the duplicated guardrail logic drifting out of sync.
