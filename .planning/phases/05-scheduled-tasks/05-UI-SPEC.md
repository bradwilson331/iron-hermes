# Phase 5 — UI Design Contract: Scheduled Tasks

> Visual and interaction contract for CLI terminal output and agent tool JSON responses.
> This phase has no web frontend. The "UI" surfaces are: (1) terminal stdout/stderr via the `colored` crate, (2) agent tool JSON responses, (3) Telegram message delivery, and (4) local output files.
> Generated by gsd-ui-researcher.

| Property | Value |

## Spacing Scale

This scale maps to terminal output conventions, not CSS pixels.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 2 spaces | Inline field alignment inside a job row |
| sm | 4 spaces | Indented sub-fields in `cron get` output |
| md | 1 blank line | Separator between list items |
| lg | 2 blank lines | Separator between sections (e.g. after header) |
| separator | `─` × 40 | Section divider (matches existing `status` command pattern) |

Exceptions: Job list table uses fixed-width columns aligned to terminal width. No minimum touch target requirement (CLI, not touch UI).

**Source:** Established pattern in `crates/ironhermes-cli/src/main.rs` — `cmd_status()` uses `"─".repeat(40)` and 2-space indented fields.

## Typography

Terminal output uses the system monospace font. The following roles map to `colored` crate styling, not font sizes.

| Role | Styling | Weight Equivalent | Usage |
|------|---------|-------------------|-------|
| Section heading | `.bold().cyan()` | Bold | Command name headers (e.g. "Scheduled Tasks") |
| Field label | `.bold()` | Semibold | Field names in detail view (e.g. "Schedule:", "Deliver:") |
| Body / value | unstyled | Regular | Field values, job names, schedule expressions |
| Dimmed / meta | `.dimmed()` | Light | Stats, hints, secondary info (e.g. next run time) |
| Status: enabled | `.green()` | Regular | "enabled" state, "OK" checks |
| Status: disabled | `.yellow()` | Regular | "paused" state, warnings, missing optional config |
| Status: error | `.red()` | Regular | Error messages, failed checks |
| Highlight: job name | `.yellow()` | Regular | Job name in list view (matches tool name highlight pattern) |

**Source:** Established in `crates/ironhermes-cli/src/main.rs` — `cmd_version()`, `cmd_status()`, `print_check()`, `run_single()` tool progress output.

## Color

Terminal color palette using the `colored` crate. Maps to the 60/30/10 pattern as applied to terminal output density.

| Role | Colored Equivalent | Usage |
|------|--------------------|-------|
| Dominant (60%) | unstyled white/default | All body text, field values, job prompts, schedule expressions |
| Secondary (30%) | `.dimmed()` | Meta information: next_run, last_run, created_at, stats footer |
| Accent (10%) | `.cyan()` + `.yellow()` | Cyan: section headings and the IronHermes branding. Yellow: job names in list, action highlights, tool progress |
| Destructive | `.red()` | Error messages, blocked security scan results, failed delivery |
| Success | `.green()` | Enabled status, "OK" state indicators, successful create/update confirmations |

Accent reserved for:
- Section heading labels (cyan bold): "Scheduled Tasks", "IronHermes Cron"
- Job names in list output (yellow): the job name column only
- Tool progress indicator (yellow): "Running: cronjob..." pattern already established
- CTA prompts at end of create flow: "Job created." confirmation label

**Source:** CONTEXT.md discretion area (error message wording). Established color usage from `crates/ironhermes-cli/src/main.rs`.

## Copywriting Contract

### CLI Output Strings

| Element | Copy |
|---------|------|
| Primary CTA (create success) | `Job created: {name} ({id})` |
| List empty state heading | `No scheduled jobs.` |
| List empty state body | `Create one with: ironhermes cron create --name "..." --schedule "every 9am" --prompt "..."` |
| List header | `Scheduled Jobs` (bold cyan, followed by `─` × 40 separator) |
| Job enabled status | `enabled` (green) |
| Job paused status | `paused` (yellow) |
| Pause success | `Job paused: {name}` |
| Resume success | `Job resumed: {name}` |
| Update success | `Job updated: {name}` |
| Remove success | `Job removed: {name}` |
| Remove confirmation prompt | `Remove job "{name}"? [y/N] ` |
| Tick: no jobs due | `(no jobs due)` (dimmed) |
| Tick: running job | `Running job: {name}` (dimmed) |
| Tick: job complete | `Job complete: {name} — delivered to {target}` |
| Security scan blocked | `Blocked: cron prompt contains restricted pattern — {pattern_type}` (red) |
| Error: job not found | `Error: job "{id_or_name}" not found.` (red) |
| Error: invalid schedule | `Error: cannot parse schedule "{input}". Try: "every 2h", "0 9 * * *", "30m", or an ISO timestamp.` (red) |
| Error: delivery failed | `Warning: delivery to {target} failed — output saved to {path}` (yellow) |
| Silent suppression | (no output — `[SILENT]` marker consumed, platform delivery suppressed) |

### Agent Tool JSON Responses (cronjob tool)

The `cronjob` tool returns structured JSON. Field names mirror the Python reference implementation (`hermes-agent/tools/cronjob_tools.py`) for jobs.json compatibility.

| Action | Success Response Shape | Error Response Shape |
|--------|----------------------|---------------------|
| create | `{"status": "created", "job": {job_object}}` | `{"status": "error", "message": "..."}` |
| list | `{"status": "ok", "jobs": [...], "count": N}` | `{"status": "error", "message": "..."}` |
| get | `{"status": "ok", "job": {job_object}}` | `{"status": "error", "message": "job not found: {id}"}` |
| update | `{"status": "updated", "job": {job_object}}` | `{"status": "error", "message": "..."}` |
| pause | `{"status": "paused", "job_id": "{id}"}` | `{"status": "error", "message": "..."}` |
| resume | `{"status": "resumed", "job_id": "{id}", "next_run": "{iso}"}` | `{"status": "error", "message": "..."}` |
| run | `{"status": "triggered", "job_id": "{id}"}` | `{"status": "error", "message": "..."}` |
| remove | `{"status": "removed", "job_id": "{id}"}` | `{"status": "error", "message": "..."}` |

**Source:** CONTEXT.md D-03 (single tool, compressed action pattern). Claude's discretion: error message wording and tool response JSON structure.

### Destructive Actions

| Action | Confirmation Approach |
|--------|----------------------|
| `ironhermes cron remove {id}` | Interactive prompt: `Remove job "{name}"? [y/N] ` — default NO, requires explicit `y`. |
| `ironhermes cron remove --force {id}` | No prompt — skips confirmation for scripted use. |
| Agent `cronjob` tool with `action: "remove"` | No interactive confirmation — agent already obtained user intent. Tool returns `{"status": "removed", "job_id": "..."}`. |

### Telegram Delivery Message Format

Scheduled task output delivered to Telegram uses Markdown formatting (matching existing gateway message pattern):

```
[Job: {name}]
{agent_output_text}
```

If `[SILENT]` is the entire output, the message is suppressed — no Telegram message sent.

Output file always written to: `~/.ironhermes/cron/output/{job_id}/{timestamp}.md` regardless of delivery target.

**Source:** CONTEXT.md D-07 (delivery pattern, SILENT marker, file output path).

## CLI Interaction Patterns

These define the exact interaction contract for each `ironhermes cron` subcommand.

### `ironhermes cron list`

```
Scheduled Jobs
────────────────────────────────────────
  NAME            SCHEDULE        STATUS    NEXT RUN
  morning-brief   0 9 * * *       enabled   2026-04-09 09:00 UTC
  weekly-review   0 17 * * 5      paused    —
────────────────────────────────────────
  2 jobs total
```

- Name column: yellow
- Status "enabled": green. Status "paused": yellow.
- Next run when paused: `—` (em dash, unstyled)
- Footer count: dimmed

### `ironhermes cron get {id}`

```
Job: morning-brief
────────────────────────────────────────
  ID:         abc123
  Schedule:   0 9 * * *  (cron)
  Prompt:     Write a brief summary of today's priorities
  Deliver:    origin
  Skills:     focus, writing
  Status:     enabled  (green)
  Created:    2026-04-07 14:32 UTC  (dimmed)
  Next run:   2026-04-09 09:00 UTC  (dimmed)
  Last run:   2026-04-08 09:00 UTC  (dimmed)
```

- Section header: bold cyan
- Field labels: bold
- Values: unstyled
- Temporal fields (Created, Next run, Last run): dimmed

### `ironhermes cron create`

Flags-based, no interactive wizard:

```
ironhermes cron create \
  --name "morning-brief" \
  --schedule "every 9am" \
  --prompt "Write a daily summary" \
  --deliver origin \
  --skills focus,writing
```

Success output:
```
Job created: morning-brief (abc123)
  Schedule:  0 9 * * *  (interpreted from "every 9am")
  Next run:  2026-04-09 09:00 UTC
```

- `Job created:` label: bold cyan
- Schedule interpretation line shown only when input was natural language (not a raw cron expression)
- Next run: dimmed

### `ironhermes cron status`

Shows tick runner state and next due job:

```
Cron Status
────────────────────────────────────────
  Jobs:       3 total, 2 enabled, 1 paused
  Next due:   morning-brief in 14h 22m
  Tick lock:  free
  Output dir: ~/.ironhermes/cron/output/
```

### `ironhermes cron tick` (manual trigger)

```
Tick: checking 3 jobs...
  Running job: morning-brief
  Job complete: morning-brief — delivered to origin
Tick complete. 1 job ran, 2 skipped.  (dimmed)
```
