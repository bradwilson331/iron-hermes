## Whitelist handling alignment (revised per BLOCKER 4)

Verified canonical semantics from `crates/ironhermes-core/src/config.rs:731` (`Empty = deny all (D-12)`) and `crates/ironhermes-gateway/src/runner.rs:601-611` (Telegram runtime gate). The runner ALWAYS spawns the adapter when the token resolves — the adapter itself (Plans 03/04) enforces empty-whitelist-deny-all. This matches Telegram's existing behavior: Telegram is spawned even with an empty whitelist; messages are dropped at runner.rs:609 with `warn!("Whitelist is empty — denying all messages (D-12)")`. Discord/Slack mirror this: spawn-then-deny-inside-adapter, not skip-spawn.

This avoids an inconsistency where Discord/Slack would have stricter spawn gates than Telegram and would surprise operators whose intent was "config sections enable adapters; whitelist controls who can talk to them."
