<objective>
Migrate the two CLI agent entry points to `AgentRuntime::run_turn` (design §5
stage 3). `run_chat` (main.rs, budget ~1199, runtime_bundle ~1360,
`run_agent_turn` helper ~2294-2340 which builds the per-turn AgentLoop with
`.with_budget` ~2333) is long-lived multi-turn and reuses one never-reset handle
→ the same latent Stop100 latch as the gateway. `run_single` (budget ~737,
runtime_bundle ~770, AgentLoop ~838 with `.with_budget` ~839) is one-shot
(per-process == per-run, so never buggy) but migrates for consistency per §6.3.

After this plan: each entry point builds one `AgentRuntime::from_config`;
`run_agent_turn` (the run_chat per-turn helper) builds a `TurnRequest` and calls
`runtime.run_turn`; `run_single`'s single turn calls `run_turn`. Their
`BudgetHandle::new` + `.with_budget` are deleted.

This plan edits ONLY `crates/ironhermes-cli/src/main.rs`, which Plan 02 also edits
(`run_gateway` lives in the same file). It therefore depends on Plan 02 and runs
after it to avoid an edit conflict. `run_chat` / `run_single` / `run_gateway` are
disjoint functions, so the changes do not interact semantically. The TUI
(`event_loop.rs`) and its invariant test (`tests/invariants_22_4.rs`) are Plan 05's
territory and are NOT touched here.

Purpose: Fix the latent run_chat multi-turn latch; one owner per CLI session/run.
Output: Both CLI entry points route through `AgentRuntime::run_turn`.
</objective>
