<objective>
Give cron a distinct agent budget so scheduled turns do not compete with or drain
the interactive chat budget (design decision §6.4). Today `run_cron_job`
(cron-runner/src/runner.rs ~289) builds its own per-job `AgentLoop` from the
SHARED `ctx.tool_registry` / `ctx.config` / `ctx.hook_registry` and runs WITHOUT
any budget (`with_budget` is never called for cron). The gateway cron tick
(gateway/src/runner.rs ~858-872) constructs `CronRunnerContext` from the gateway's
own shared Arcs and calls `run_tick_loop`. Because cron currently has no budget,
there is no Stop100 latch to fix here — the work is to give cron its OWN budget
unit so a runaway scheduled job is bounded independently of interactive chat.

### Design constraint the executor MUST honor (per-job overrides)

Cron is NOT a simple `run_turn` channel: each job may override the model
(`job.model`, runner.rs ~256) and may scope the tool registry to a subset of
toolsets (`job.enabled_toolsets` → `reg.scope_to(names)`, ~279-287). A single
fixed `AgentRuntime::from_config` bakes in one client + one tool registry and
cannot express per-job model/registry scoping through `run_turn`. Therefore:

- DO give cron a SEPARATE `BudgetHandle` (sized from `config.agent.max_iterations`)
  that is distinct from the interactive budget, threaded onto the per-job AgentLoop
  via `.with_budget(...)` and shared with cron's subagent runner if/when cron spawns
  subagents (PROV-10 within the cron unit). This is the minimal faithful
  implementation of §6.4 and preserves per-job overrides.
- The cron budget must be reset per job run (a fresh job run starts full), mirroring
  the run_turn boundary semantics — call `reset()` at the top of each `run_cron_job`,
  or construct a fresh handle per job.
- If, after inspection, a per-job `AgentRuntime` is genuinely clean (build a runtime
  per job with the job's scoped registry + model client), that is acceptable and
  more consistent — but only if it does not regress per-job model/toolset scoping.
  PREFER the dedicated-budget approach unless the per-job runtime is demonstrably
  simpler; document the choice and the rationale in the SUMMARY.

This plan touches `gateway/src/runner.rs` (the cron tick context construction),
which Plan 02 also edits — hence wave 3, after Plan 02, to avoid an edit conflict.

Purpose: Isolate scheduled-turn budget from interactive budget (§6.4).
Output: Cron turns consume a separate budget; interactive budget is untouched by cron.
</objective>

| Boundary | Description |
|----------|-------------|
| cron schedule → run_cron_job | job-defined prompt + per-job model/toolset overrides cross into the cron agent |
| cron agent → shared tool registry | cron uses ctx.tool_registry (shared with gateway); budget must NOT be shared |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-28.1-14 | Denial of Service | runaway cron job drains the interactive chat budget | mitigate | cron gets a distinct BudgetHandle (§6.4); Task 3 proves independence |
| T-28.1-15 | Elevation of Privilege | per-job toolset scoping lost in migration → job accesses tools it should not | mitigate | preserve `scope_to(names)` exactly; per-job model override preserved |
| T-28.1-16 | Denial of Service | cron subagents drain interactive budget via shared registry delegate_task runner | accept (with documented follow-up) | top-level cron isolation delivered; subagent-level isolation flagged if registry sharing makes it costly |
| T-28.1-SC | Tampering | npm/pip/cargo installs | accept | no new dependencies introduced |
</threat_model>
