# Phase 28.1: AgentRuntime Channel Migration — Verification Report

**Phase Goal:** AgentRuntime is the single channel-facing agent API: every channel (Telegram gateway, web UI, CLI run_chat/run_single, TUI) builds one AgentRuntime and calls run_turn(TurnRequest) per top-level turn. No channel constructs BudgetHandles or assembles AgentLoops by hand; the run-boundary owns budget reset, permanently fixing the Stop100 latch class for current and future channels. Cron gets a separate runtime/budget so scheduled turns do not drain interactive chat.

### Human Verification Required

All automated code checks pass. The following items require live operator testing that cannot be performed in-session.

#### 1. Gateway Stop100 Latch Fix (Critical Path)

**Test:** Start `hermes gateway --token <tg-token>`. Drive a first Telegram conversation long enough to exhaust the per-turn budget (watch logs for Stop100 / pressure-tier advisories). Send a SECOND message after exhaustion.
**Expected:** Second message produces a normal multi-turn response (`turns_used > 0`), proving the `run_turn` budget reset. Previously would have returned `turns_used=0`.
**Why human:** Telegram E2E cannot be tested in-session; requires a live bot token and real Telegram client.

#### 2. CLI run_chat Stop100 Latch Fix

**Test:** Run `hermes` (interactive run_chat). Drive a first prompt to budget exhaustion (a long agentic task with many tool calls). Submit a second user turn.
**Expected:** Second turn starts with a full budget and executes normally (multi-turn response, not an immediate Stop100).
**Why human:** Interactive multi-turn budget exhaustion requires a live terminal REPL session.

#### 3. TUI Latch Fix + StreamEvent Forwarding

**Test:** Run `hermes` in ratatui TUI mode. Drive a budget-exhausting first turn. Send a second turn. Also exercise a tool-call-heavy task and confirm tool-call pills + tool-result indicators render.
**Expected:** Second turn starts fresh (latch fixed). Tool-call status pills and tool-result indicators still render (StreamEvent forwarding preserved through TurnRequest callbacks).
**Why human:** TUI rendering and multi-turn state cannot be E2E tested in-session.

#### 4. TUI AgentLoopHandle / /status Integration

**Test:** In the TUI, exercise the `/status` command (or equivalent AgentLoopHandle-backed surface).
**Expected:** Budget and status readings reflect the runtime's live state, not a stale/disconnected loop.
**Why human:** Requires live TUI session.

#### 5. Cron Budget Isolation Under Live Gateway

**Test:** With the gateway running and a cron job scheduled: (a) drive interactive chat to consume most of the budget, let the cron job fire — confirm the cron job runs with its own full budget; (b) let a cron job exhaust its budget — confirm interactive chat retains its full budget.
**Expected:** Cron and interactive budgets are completely independent; exhausting one never affects the other.
**Why human:** Requires live gateway with both Telegram token and a scheduled cron job.

#### 6. Active Skills Identity Sharing (Gateway)

**Test:** In the gateway, run `/skills activate <name>` then send a normal turn.
**Expected:** The activated skill is available in the next turn (active_skills identity-shared via runtime).
**Why human:** Requires live Telegram gateway and slash command interaction.

#### 7. Trajectory Writer Preservation (Gateway)

**Test:** Have a gateway conversation. Run `hermes session export <session_id>`.
**Expected:** Trajectory JSONL file exists under HERMES_HOME (per-session writer preserved through TurnRequest.trajectory_writer).
**Why human:** Requires live gateway session to create the trajectory writer entry.
