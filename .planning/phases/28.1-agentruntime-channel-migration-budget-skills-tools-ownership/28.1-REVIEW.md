# Phase 28.1: Code Review Report

**Reviewed:** 2026-05-21
**Depth:** standard
**Files Reviewed:** 21
**Status:** issues_found

## Summary

The migration to `AgentRuntime::run_turn` as the single channel-facing API is, on the
**production-code** side, well-executed and correct:

- `AgentRuntime::from_config` creates one `BudgetHandle` and clones it into the
  `AgentSubagentRunner` (PROV-10 parent/child sharing), and `run_turn` resets that
  shared counter *before* `AgentLoop::new` (verified `agent_runtime.rs:198` precedes
  `:202`). The Stop100 latch fix is real.
- All four interactive channels (gateway handler, web `run_web_turn`, CLI `run_single`
  / `run_agent_turn`, TUI `spawn_turn`) build a `TurnRequest` and delegate to
  `runtime.run_turn`. None hand-builds a `BudgetHandle` or calls `AgentLoop.with_budget`
  with a bare handle. The only `with_budget` in CLI production is `build_cmd_ctx`'s
  `.with_budget(Arc::new(budget))` for `CommandContext` observability — explicitly
  permitted and threaded from `runtime.budget().clone()`.
- Cron correctly uses a **fresh** `BudgetHandle::new` per job (cron-runner/runner.rs:185),
  independent of any interactive runtime budget (T-28.1-14).
- Per-turn wiring (stream/tool callbacks, cancel token, trajectory_writer,
  pressure_tracker, state_store, compression_count round-trip) is threaded cleanly
  through `TurnRequest` into `run_turn`.

The defects are concentrated in the **static-grep parity gates**, which the phase brief
flagged as the highest-risk surface. One gate has been silently neutered into a
self-satisfying tautology, and another now asserts a pre-migration pattern that the
migration deliberately removed — both give false green signal.

## Critical Issues

### CR-01: `inv_25_1_gap8_browser_tools_wired_in_rata_chat` is a self-satisfying grep gate (false green)

**File:** `crates/ironhermes-cli/src/tui_rata/event_loop.rs:845-855`
**Issue:** This gate claims to guard that BOTH AgentLoop builders in the TUI
(`build_app_deps` and `spawn_turn`) chain `.with_browser_session(...)`, asserting
`with_count >= 2`. After the Phase 28.1-05 migration **there are zero
`.with_browser_session(` calls in non-comment production code of `event_loop.rs`** —
`spawn_turn` builds a `TurnRequest` and calls `runtime.run_turn`; the runtime owns the
browser session internally. The test's comment-stripping filter only removes lines that
`trim_start().starts_with("//")`. The two surviving matches are the test's *own* source:

- line 847: `let with_count = non_comment.matches(".with_browser_session(").count();`
- line 851: the assertion message string `... MUST chain .with_browser_session(); \`

Neither line starts with `//`, so both survive the filter and the literal
`.with_browser_session(` inside them is counted. `with_count` evaluates to 2 by matching
the test's own assertion text — the gate passes regardless of production wiring. The
invariant it purports to enforce (browser tools reaching the turn) is no longer verified
at all. This is exactly the "weakened to pass" failure mode the brief asked to flag.

**Fix:** Rewrite the gate to assert the post-migration contract instead of a deleted
pattern, and exclude its own literals from the corpus. For example, assert the runtime
owns the browser session and `spawn_turn` delegates through `run_turn`:
```rust
// Strip comments AND the test module so assertion strings can't self-satisfy.
let prod = strip_comments_and_test_module(source);
assert!(
    prod.contains("runtime.run_turn(") && prod.contains("AgentRuntime::from_config"),
    "Phase 28.1-05: TUI spawn_turn must delegate to runtime.run_turn; the runtime \
     owns the browser session (verify build_app_runtime_bundle wires browser tools)."
);
// If browser-tool coverage must still be asserted, do it behaviorally (as the
// sibling `rata_bootstrap_registry_contains_all_11_browser_tools` test already does),
// not via a grep that can match its own message string.
```
Alternatively, delete this gate and rely on `rata_bootstrap_registry_contains_all_11_browser_tools`
(event_loop.rs:862) which is a real behavioral check.

## Warnings

### WR-01: `invariant_21_7_04` asserts a pre-migration pattern (`BudgetHandle` in 3 sites)

**File:** `crates/ironhermes-cli/tests/invariants_21_7.rs:77-91`
**Issue:** The gate asserts `MAIN_RS.matches("BudgetHandle").count() >= 3` with the
rationale "BudgetHandle must appear in all 3 registration sites (run_single, run_chat,
run_gateway)." The Phase 28.1 migration's explicit goal was to **remove** `BudgetHandle`
construction from those three functions (budget is now owned by `AgentRuntime`). The 8
remaining occurrences in `main.rs` are: 1 `use` import (line 8), 1 type annotation on the
`build_cmd_ctx` parameter (line 947), and 6 in comment prose / the test module
(lines 753, 754, 1298, 1300, 2393, 2417, 2418, 2475, plus the test-module body at
3114+). None are the "registration site" the assertion describes. The gate passes only on
stale comment/import text, contradicting the migration it ostensibly co-exists with. A
future cleanup that removes the now-misleading comments could flip this RED for the wrong
reason.

**Fix:** Retarget to the migration's actual invariant — that `main.rs` does NOT construct
`BudgetHandle` and routes through `run_turn`. This is already correctly enforced by
`main_rs_has_no_direct_budget_construction_or_with_budget` (main.rs:3129) and
`main_rs_all_entry_points_use_run_turn` (main.rs:3187). Either delete `invariant_21_7_04`
as superseded, or invert it:
```rust
// Budget is owned by AgentRuntime post-28.1; main.rs must NOT construct one.
let prod = strip_comments_and_tests(MAIN_RS);
assert!(!prod.contains("BudgetHandle::new("),
    "Phase 28.1: main.rs must not construct BudgetHandle; AgentRuntime owns it.");
```

### WR-02: `gap3_gateway_handler_wires_memory_manager` carries a false comment

**File:** `crates/ironhermes-cli/tests/run_chat_invariants.rs:253-256`
**Issue:** The inline comment states "run_gateway is the only AgentRuntime::from_config
call site in main.rs (run_chat / run_single use build_app_runtime_bundle directly)." This
is factually wrong after Phase 28.1-04: `AgentRuntime::from_config` appears at **3**
production call sites in `main.rs` (run_single:757, run_chat:1305, run_gateway:2421), and
`build_app_runtime_bundle` is no longer called directly by any of them. The assertion body
itself still holds (it greps for `from_config` + `memory_manager: memory_manager`, both
present), so this is doc rot rather than a broken gate — but the comment will mislead the
next maintainer reasoning about which sites exist.

**Fix:** Update the comment to "all three CLI entry points use `AgentRuntime::from_config`;
this gate confirms run_gateway threads the real memory_manager into it."

### WR-03: GAP-3 `memory_manager: memory_manager` substring match is over-broad

**File:** `crates/ironhermes-cli/tests/run_chat_invariants.rs:258-263`
**Issue:** The assertion checks `main_rs.contains("memory_manager: memory_manager")`. The
actual production code passes `memory_manager: memory_manager.clone()` at all three sites
(main.rs:762, 1310, 2426). The substring match happens to succeed, but it does not
distinguish run_gateway from run_single/run_chat, nor would it catch a regression where
run_gateway alone was switched to `memory_manager: None` while another site kept the
real handle. The gate is weaker than its docstring ("run_gateway must pass memory_manager
... not None") implies.

**Fix:** Scope the assertion to the run_gateway function body (the file already has a
brace-balanced `extract_function_body` helper in sibling `cli_tool_parity.rs`) and assert
the negative too:
```rust
let body = extract_function_body(&main_rs, "run_gateway");
assert!(body.contains("memory_manager: memory_manager")
    && !body.contains("memory_manager: None"),
    "GAP-3: run_gateway must thread the real memory_manager into from_config, not None.");
```

### WR-04: `api_sessions_and_tools_are_backed_by_real_state` uses a logically inert assertion

**File:** `crates/iron_hermes_ui/tests/server_runtime_parity.rs:36-39`
**Issue:** `assert!(!api.contains("Ok(vec![])") || !api.contains("TODO"))`. This passes if
*either* token is absent. The intent is presumably "the endpoints must not be empty
`Ok(vec![])` TODO stubs," but the `||` makes the guard near-unfalsifiable: it would only
fail if `api.rs` contained BOTH `Ok(vec![])` AND `TODO` simultaneously. A real stub that
returns `Ok(vec![])` with no `TODO` comment (or a `TODO` with no `Ok(vec![])`) passes
silently. Not a Phase 28.1 regression (pre-existing), but it provides false assurance for
the web API surface this phase touches.

**Fix:** Use AND for a stub guard, or assert the positive behavior directly:
```rust
assert!(!(api.contains("Ok(vec![])") && api.contains("TODO")),
    "list_sessions/list_tools must not be empty TODO stubs");
```

## Info

### IN-01: TUI keeps a second hand-built ToolRegistry distinct from the turn registry

**File:** `crates/ironhermes-cli/src/tui_rata/event_loop.rs:218-314`, `app.rs:77-81`
**Issue:** `build_app_deps` constructs a full `ToolRegistry` (terminal-with-process-registry,
execute_code, browser tools, MCP) and stores it on `App.registry`, while
`AgentRuntime::from_config` builds its own independent registry internally that actually
serves turns. The App-side registry is used only for slash dispatch and
`call_session_end_hooks`. This is documented as intentional (the Phase 28.1-05 DECISION
block, lines 422-428), and behavior is equivalent because both registries derive from the
same config — but it is duplicate construction (two MCP-less browser sessions, two
execute_code RPC sub-registries, etc.) and a latent drift hazard: a future tool wired into
only one registry would behave inconsistently between `/tools list` and actual turns. Worth
a follow-up to source the slash/hook registry from `runtime.registry()` for true identity
sharing (as run_gateway already does at main.rs:2440).

**Fix:** Where feasible, drop the hand-built TUI registry and use
`runtime.registry().clone()` for slash dispatch + session-end hooks, matching the gateway's
identity-sharing pattern. If the TUI genuinely needs process-registry-scoped terminal
tools the runtime doesn't build, document the divergence explicitly in `app.rs`.

### IN-02: `run_single` persists `result.messages` (Assistant-only) while web uses `result.appended`

**File:** `crates/ironhermes-cli/src/main.rs:908-912` vs `iron_hermes_ui/src/server/state.rs:227-230`
**Issue:** `run_single` iterates `result.messages` and persists only `Role::Assistant`
rows, whereas `run_web_turn` persists `result.appended` (all newly-appended rows). The two
channels use different `AgentResult` accessors for the same "persist this turn's output"
operation. This is likely pre-existing and not introduced by 28.1 (run_single is one-shot
batch and may intentionally skip tool rows), but the inconsistency is a maintenance trap:
a reader cannot tell from the migration whether the divergence is deliberate. Confirm
`result.appended` and `result.messages` semantics are documented on `AgentResult`.

**Fix:** Add a one-line comment at main.rs:909 explaining why run_single uses
`result.messages` (Assistant-only) rather than `result.appended`, or align it with the web
path if tool-row persistence is desired for batch sessions.

### IN-03: `for_tests()` JobStore temp dir keyed by PID can collide across test binaries

**File:** `crates/ironhermes-agent/src/agent_runtime.rs:342-347`
**Issue:** `AgentRuntime::for_tests()` opens a `JobStore` at
`temp_dir()/ironhermes_test_cron_{process_id}`. Within one test binary, parallel tests
share the same PID and therefore the same cron dir; if two tests both call `for_tests()`
and one mutates the JobStore, they can interfere. The comment claims "unique to the process
so parallel test runs don't collide" — that protects across separate `cargo test`
*processes*, not across threads within one process. Low risk today (the fixture is for
struct construction, not JobStore writes), but the comment overstates the guarantee.

**Fix:** Append a per-call nonce (e.g. an `AtomicU64` counter or `Uuid`) to the temp dir
name so concurrent fixtures in the same process get distinct stores.
