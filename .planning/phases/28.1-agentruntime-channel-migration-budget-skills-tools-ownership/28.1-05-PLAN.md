<objective>
Migrate the ratatui TUI to `AgentRuntime::run_turn` (design §5 stage 3). The TUI
is the most intricate non-gateway channel: `build_app_deps` (event_loop.rs)
builds a `BudgetHandle` (~199) sized from `cli.max_turns.unwrap_or(config.agent.max_turns)`
(NOTE: this is the one config-drift site — it reads `max_turns`, not
`max_iterations`; the migration fixes the drift because AgentRuntime sizes the
budget from `config.agent.max_iterations`), an App-level `Arc<AgentLoop>` (~440)
stored "for integrations", and `spawn_turn` (~809) builds a fresh per-turn
`AgentLoop` with `.with_budget` (~810). `AppDeps`/`App` (app.rs ~55-78, ~210-247)
carry the loose pieces (`agent_loop`, `budget`, `client`, `registry`, `max_turns`,
`fallback_client`) that `spawn_turn` reassembles each turn.

The App-level `Arc<AgentLoop>` has exactly ONE consumer: `commands.rs:590` wraps
it in `AgentLoopAdapter(app.agent_loop.clone())` to satisfy `AgentLoopHandle` for
an integration/status surface.

After this plan: `build_app_deps` builds one `AgentRuntime::from_config`; `App`
carries an `Arc<AgentRuntime>` (replacing the loose budget/client/registry/
max_turns/fallback pieces and the App-level loop); `spawn_turn` builds a
`TurnRequest` and calls `runtime.run_turn`; the `AgentLoopHandle` integration is
re-pointed (see Task 2).

This edits files under `crates/ironhermes-cli/src/tui_rata/` (disjoint from
main.rs which Plans 02/04 own), so it is file-parallel with Plan 04 within the
same crate. It depends on Plan 02 only for AgentRuntime API stability.

Purpose: Fix the latent TUI latch + the max_turns/max_iterations config drift;
one owner of the TUI agent.
Output: TUI routes every turn through `AgentRuntime::run_turn`.
</objective>
