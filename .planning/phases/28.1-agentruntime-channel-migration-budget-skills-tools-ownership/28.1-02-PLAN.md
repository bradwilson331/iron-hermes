<objective>
Migrate the Telegram gateway to `AgentRuntime::run_turn` (design §5 stage 2,
decision §6.3). This is the highest-value migration: it removes the `367eaa79`
band-aid, proves `run_turn` end-to-end against the most complex channel, and
establishes the pattern the remaining channels copy.

`run_gateway` (main.rs ~2467-2640) currently hand-builds a `BudgetHandle`
(~2540), a `subagent_runner` with a clone of it (~2547), and a `runtime_bundle`
via `build_app_runtime_bundle` (~2554), then threads everything into
`GatewayRunner` via setters. Per-turn, `handler.rs::run_agent` (~959-1082)
hand-assembles an `AgentLoop` with `.with_streaming/.with_tool_progress/.with_active_skills/.with_budget`
(+ the reset band-aid ~995-1006) `/.with_hook_registry/.with_memory_manager/.with_browser_session/.with_trajectory_writer`
+ `attach_context_engine`, then runs.

After this plan: `run_gateway` builds ONE `AgentRuntime` via `from_config`; the
runner and handler source their shared Arcs (tool_registry, hook_registry,
active_skills, browser_session, memory_manager, subagent_registry,
process_registry) FROM the runtime so slash/toolset/status paths and `run_turn`
operate on the same instances; `run_agent` builds a `TurnRequest` and calls
`runtime.run_turn`. The gateway-owned `BudgetHandle` and the `with_budget` /
band-aid `reset()` are gone.

Purpose: One owner of the gateway agent; permanent fix to the Stop100 latch via
the run-boundary reset; removes one of four copies of the wiring.
Output: Gateway routes every top-level turn through `AgentRuntime::run_turn`.
</objective>
