# Phase 28.1 Plan 06: Cron-Distinct Budget Summary

**One-liner:** Fresh `BudgetHandle::new(config.agent.max_iterations)` per cron job, wired via `.with_budget()` onto the per-job `AgentLoop`, isolating scheduled turns from the interactive chat budget (§6.4 / T-28.1-14).

## What Was Built

### Task 1

Added `use ironhermes_agent::budget::BudgetHandle` import to `cron-runner/src/runner.rs`. At the top of `run_cron_job`, before any agent construction, a fresh `BudgetHandle::new(ctx.config.agent.max_iterations)` is created. This handle is installed on the per-job `AgentLoop` via `.with_budget(cron_budget)` immediately after `AgentLoop::new(...)` and before `wire_fallback_if_configured`. The budget is a brand-new `Arc<AtomicUsize>` — not a clone of any gateway budget — so consuming it cannot affect interactive chat.

The per-job model override (`job.model`) and toolset scoping (`scope_to(names)`) are fully preserved — they operate on separate dimensions from the budget wiring.

### Task 2

No file changes required. The fresh-per-job approach means `CronRunnerContext` has no new field, so the three `CronRunnerContext { ... }` construction sites in `ironhermes-gateway/src/runner.rs` (lines ~857, ~1233, ~1787) and the `make_ctx` test helper in cron-runner required no updates. Build and test suite confirmed clean across both crates.

### Task 3

Added `cron_budget_is_independent_from_interactive_budget` unit test in `runner::tests`. The test:

1. Creates `cron_budget_job1 = BudgetHandle::new(max)` and `interactive_budget = BudgetHandle::new(max)`.
2. Drains `cron_budget_job1` to exhaustion (all `max` calls to `.consume()`).
3. Asserts `interactive_budget.remaining() == max` — unaffected.
4. Constructs `cron_budget_job2 = BudgetHandle::new(max)` — simulates second job — asserts it starts full.
5. Verifies `clone` of exhausted cron budget shares its `Arc` (expected cron-internal sharing); confirms `reset()` on cron is not visible to interactive.

Uses behavioral form (drain-and-compare-remaining) rather than `Arc::ptr_eq` since `BudgetHandle.remaining` is private.

## Design Decision: Fresh Handle vs. CronRunnerContext Field

The plan offered two options:
- **Option A (chosen):** Fresh `BudgetHandle::new(...)` per job inside `run_cron_job`.
- **Option B:** Add `cron_budget: BudgetHandle` field to `CronRunnerContext`, reset it per job.

**Rationale for Option A:** Genuinely simpler — zero construction site changes, zero field plumbing, and the per-job fresh-handle guarantee is structural (no reset call can be forgotten). Option B would require updating three gateway construction sites and the test helper. Both options achieve §6.4 isolation; Option A is demonstrably smaller diff with identical safety properties.

## Cron Subagent Budget Isolation (T-28.1-16)

**Accepted disposition.** If a cron job uses `delegate_task`, the delegate runner may be wired to the gateway's interactive budget (via the shared `ctx.tool_registry`). This plan delivers **top-level cron isolation** — the dominant cost — and flags subagent-level isolation as a follow-up. This matches the threat model's `accept (with documented follow-up)` disposition for T-28.1-16.
