### WR-01: `Finished` can append stale AI output from an older turn

**File:** `/Users/twilson/code/ironhermes/crates/iron_hermes_ui/src/components/warp_hermes.rs:231-257`  
**Issue:** On `ChatStreamEvent::Finished`, the code scans all blocks in reverse and picks the latest non-empty `Block::Ai`. If the current request ends without deltas (or disconnect/reset happened first), it can reuse an older AI block and append duplicate/stale content into `messages`.

**Fix:** Finalize using the active stream id only, then clear it.

```rust
let completed_id = streaming_block_id();
streaming_block_id.set(None);

if let Some(id) = completed_id {
    let ai_text = blocks.read().iter().find_map(|entry| {
        if entry.id != id {
            return None;
        }
        match &entry.block {
            Block::Ai { markdown, .. } if !markdown.is_empty() => Some(markdown.clone()),
            _ => None,
        }
    });

    if let Some(text) = ai_text {
        messages.write().push(Message {
            who: "hermes".into(),
            time: now_time(),
            body: text,
            tool: None,
        });
    }
}
```

### WR-02: Busy-gate can incorrectly reject a new request after turn completion

**File:** `/Users/twilson/code/ironhermes/crates/iron_hermes_ui/src/server/ws.rs:65-73, 155-160`  
**Issue:** Busy detection is `if in_flight_turn.is_some()`. In `tokio::select!`, a request frame can be processed before the `maybe_event` branch observes `None` from the completed channel and clears `in_flight_turn`, causing a false “Another request is already in progress” error.

**Fix:** Before busy check, opportunistically clear finished task state.

```rust
if let Some(turn) = in_flight_turn.as_ref() {
    if turn.handle.is_finished() {
        in_flight_turn = None;
    }
}

if in_flight_turn.is_some() {
    // send busy error
    continue;
}
```

### WR-03: Parity assertions are too broad and can mask lifecycle regressions

**File:** `/Users/twilson/code/ironhermes/crates/iron_hermes_ui/tests/websocket_lifecycle_parity.rs:48-55, 80-84`  
**Issue:** Checks like `ws.contains("continue;")` and `ws.contains("abort()")` are global-string matches and not anchored to the malformed-request/send-failure branches they are meant to guard. Tests may pass if those tokens appear elsewhere.

**Fix:** Anchor assertions to branch-local snippets (or regex over local context), e.g. assert the malformed-request block contains both `Invalid request:` and a nearby `continue;`, and the send-failure block contains `socket.send` error handling plus `turn.handle.abort()`.
