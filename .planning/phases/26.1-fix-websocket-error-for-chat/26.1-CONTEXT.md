# Phase 26.1: fix-websocket-error-for-chat - Context

**Gathered:** 2026-05-04
**Status:** Ready for planning

<domain>
## Phase Boundary

Stabilize Dioxus web chat transport so `/api/ws/chat` streams events live during an active turn, handles disconnects without orphaned work, and keeps client receive behavior resilient across reconnect boundaries.

This phase covers WebSocket lifecycle behavior only:
- server-side per-connection request/turn lifecycle in `crates/iron_hermes_ui/src/server/ws.rs`
- client-side receive/reconnect lifecycle in `crates/iron_hermes_ui/src/components/warp_hermes.rs`
- lifecycle regression locks in `crates/iron_hermes_ui/tests/websocket_lifecycle_parity.rs`

Out of scope:
- protocol redesign (`ChatRequest` / `ChatStreamEvent` shape changes)
- global runtime architecture changes (startup-scoped AppState from Phase 25.6 stays)
- UX redesign beyond disconnect/stream interruption handling

</domain>

<decisions>
## Implementation Decisions

### Server Turn Concurrency Model
- **D-01:** Per request, run the agent turn in a spawned task and forward websocket events concurrently while the turn is running.
- **D-02:** Enforce one in-flight turn per socket. If a second request arrives while one is active, return a structured `ChatStreamEvent::Error` and do not queue/cancel-switch.
- **D-03:** Keep the in-flight gate as explicit local lifecycle state in `ws_chat` (transport-owned), not hidden in `run_web_turn`.
- **D-04:** Terminal event ownership lives in the spawned turn task: emit exactly one terminal event (`Finished` or `Error`) there, then close sender to end forwarding.

### Disconnect Teardown Semantics
- **D-05:** On websocket send failure during forwarding, abort the in-flight turn task immediately and exit the connection loop.
- **D-06:** Do not attempt explicit close-frame writes on broken send path; end handler cleanly.
- **D-07:** Emit warn-level telemetry for disconnect aborts including session id, reason, and in-flight state.
- **D-08:** Malformed request payloads remain recoverable per-frame errors: send `ChatStreamEvent::Error` and continue receiving next frames.

### Client Reconnect Lifecycle
- **D-09:** Replace single-pass receive loop with an outer reconnect lifecycle that re-enters connect/receive after transient failures.
- **D-10:** Use Dioxus websocket automatic reconnect (`with_automatic_reconnect()`) only; no custom app-level backoff loop.
- **D-11:** On disconnect boundary, reset transient stream state only: `scanner_active = false` and `streaming_block_id = None`.
- **D-12:** Do not auto-resubmit last request after reconnect.

### UI Interruption Behavior
- **D-13:** Keep already-rendered partial AI text visible when disconnect occurs mid-stream.
- **D-14:** After reconnect, new submits start a new AI block; do not append into old interrupted partial block.
- **D-15:** Show a transcript-visible disconnect error block, but keep message generic/user-facing. Put technical error details in logs only.

### Claude's Discretion
None. User selected explicit implementation direction for all discussed areas.

</decisions>

## Existing Code Insights

### Reusable Assets
- `crates/iron_hermes_ui/src/server/ws.rs`: existing websocket route and callback wiring for `Delta`, `ToolCallStart`, `ToolCallEnd`, `Finished`, `Error` can be reused with ordering/lifecycle refactor.
- `crates/iron_hermes_ui/src/server/state.rs`: `run_web_turn` already supports stream/tool callbacks and can remain lifecycle execution core.
- `crates/iron_hermes_ui/src/components/warp_hermes.rs`: existing event-to-UI mapping logic for blocks/messages/tokens can stay, with reconnect control-flow hardening.

### Established Patterns
- Startup-scoped `global_app_state()` is already enforced by parity tests and must be preserved.
- Transport remains JSON-encoded `ChatStreamEvent` over Dioxus websocket with no protocol expansion.
- UI streaming state is intentionally represented by `scanner_active` + `streaming_block_id`; reset behavior should stay bounded to these transients.

### Integration Points
- Server lifecycle changes wire directly through `ws_chat` -> callback channel -> socket forwarding.
- Client lifecycle changes live in `use_websocket` + receive future loop in `WarpHermes`.
- New lifecycle regression guardrail belongs in `crates/iron_hermes_ui/tests/websocket_lifecycle_parity.rs` and should complement `server_runtime_parity.rs`.

</code_context>

<specifics>
## Specific Ideas

- Disconnect transcript message should remain short and user-facing (for example: "Connection interrupted. Reconnected; you can resend if needed."), while internal error detail stays in telemetry.
- Concurrent request behavior on one socket is explicit reject-with-error (not queue, not cancel-and-replace) for deterministic turn boundaries.

</specifics>
