# Phase 26.1 Research — WebSocket reset without closing handshake

**Date:** 2026-05-04  
**Scope:** `crates/iron_hermes_ui` WebSocket server route + client connection lifecycle  
**Mode:** Level 1/2 targeted implementation research (existing stack, behavior bug)

## Problem Statement

`/api/ws/chat` intermittently resets without a clean close handshake during chat streaming. The phase goal is to restore reliable Dioxus chat-to-LLM streaming and stabilize the WebSocket lifecycle.

## Current-State Findings

### 1) Server-side event delivery is effectively buffered-until-complete

In `src/server/ws.rs`, the handler currently:

1. Creates an internal `mpsc` channel
2. Calls `run_web_turn(...).await`
3. Only after that finishes, drains `rx` and sends events to socket

That ordering means `Delta`/`ToolCall*` events are not forwarded live while the turn runs, and write-path lifecycle is coupled to long-running agent execution.

### 2) Connection teardown path is abrupt

`socket.send(...)` failures break loops, but there is no explicit close-path normalization and no single lifecycle state for:

- malformed client message
- downstream agent failure
- socket write failure mid-turn
- reconnect boundary

### 3) Client receive loop exits permanently on first recv error

In `src/components/warp_hermes.rs`, the `use_future` receiver loop `break`s on `recv_raw()` error. Even with `with_automatic_reconnect()`, this code path can terminate the app-side stream task and stop receiving future events.

## Relevant Existing Contracts

- Server route exists: `#[get("/api/ws/chat")]` returning `Websocket<String, String>` in `src/server/ws.rs`
- Shared protocol types: `src/protocol.rs` (`ChatRequest`, `ChatStreamEvent`)
- Runtime wiring is startup-scoped (`global_app_state()`), already locked by `tests/server_runtime_parity.rs`

## Dioxus 0.7.1 Documentation Check

Context7 (`/dioxuslabs/dioxus/v0.7.1`) confirms the intended baseline pattern:

- fullstack `#[get]` websocket endpoint using `WebSocketOptions::on_upgrade(...)`
- client-side `use_websocket(...)` + `recv()`/`send()` loop

Reference pattern aligns with existing architecture, so this phase should **fix lifecycle ordering and teardown semantics** rather than introduce new transport/framework dependencies.

## Recommended Implementation Direction

1. **Server lifecycle refactor (no API shape change):**
   - Keep `/api/ws/chat` route and protocol unchanged
   - Stream `ChatStreamEvent` to socket as events are produced (during run, not after)
   - Centralize close/error handling so send failures and malformed messages terminate the turn cleanly

2. **Client lifecycle hardening:**
   - Keep `with_automatic_reconnect()`
   - Make receive task resilient across transient disconnects (re-enter receive cycle after reconnect)
   - Reset UI transient state (`scanner_active`, `streaming_block_id`) on disconnect/error boundaries

3. **Regression locking:**
   - Add parity tests for lifecycle invariants (server streaming order + reconnect loop persistence)

## Constraints / Non-Goals

- No protocol redesign (keep current `ChatRequest` and `ChatStreamEvent` enum)
- No new crate dependencies
- No global runtime architecture changes (Phase 25.6 startup-scoped AppState remains)

## Risks and Mitigations

- **Risk:** concurrent send + turn task can leak tasks on disconnect  
  **Mitigation:** abort/join turn task when socket write fails

- **Risk:** reconnection loop duplicates handlers  
  **Mitigation:** single receiver task with explicit outer reconnect loop; avoid nesting additional `use_future`

- **Risk:** regression in parity invariants from Phase 25.6  
  **Mitigation:** retain and extend static/runtime parity tests in `tests/`
