## Root Cause

`TypedWebsocket<String, String>` has two distinct serialization layers:

- **`recv()` (Stream/poll_next path):** Calls `JsonEncoding::decode` on each incoming
  frame, interpreting it as a JSON-encoded `String` value. A chat payload like
  `{"session_id":"...","message":"..."}` is a JSON object, not a JSON string literal, so
  deserialization fails. The prior Pong-skip hack swallowed these errors silently, dropping
  every chat message.

- **`send()` (Sink path):** Calls `JsonEncoding::to_bytes` which produces **binary**
  frames. The client's `recv_raw` loop only handles `Message::Text`, so all server events
  were invisibly dropped.

Note: Pong frames are already handled by `continue` inside the Stream's `poll_next` — they
never surfaced as errors. The errors were exclusively from chat message deserialization.

## Fix

Switched the entire server WebSocket loop to raw frame operations:

```rust
// Before (broken)
let msg = socket.recv().await          // JSON-decodes as String — fails for JSON objects
socket.send(json_string).await         // sends BINARY frame — client never sees it

// After (correct)
let raw = socket.recv_raw().await      // returns Message::Text(t) — direct text
socket.send_raw(Message::Text(json))   // sends TEXT frame — client recv_raw Text arm fires
```

Also added:
- `dotenvy::from_path(Config::env_path())` in `main.rs` to load `~/.ironhermes/.env`
  so `OPENROUTER_API_KEY` is available to the embedded agent
- `dotenvy = { workspace = true }` in `Cargo.toml`
- Server-side `WS_KEEPALIVE_INTERVAL = 5s` Ping (from `434e2ca`, confirmed working)
- Close frame on all teardown branches (from `e830039`/`e950d71`)
- Removed debug `console.log` from `warp_hermes.rs` submit handler

which were not needed once the core transport was working.
