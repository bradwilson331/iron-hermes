    In `crates/iron_hermes_ui/src/server/ws.rs`, replace the current sequential flow (await full `run_web_turn`, then drain rx) with a lifecycle-safe concurrent flow:

    1. Keep `#[get("/api/ws/chat")]` route and protocol types unchanged.
    2. For each received `ChatRequest`, create `mpsc::unbounded_channel<ChatStreamEvent>` and spawn the turn execution in a `tokio::spawn` task that emits events through callbacks.

    4. When run task ends, emit exactly one terminal event (`Finished` on success or `Error` on failure), drop sender, then finish the forwarding loop.
    5. On socket send failure, abort the spawned turn task and break out of connection loop (clean teardown path, no orphan task).
    6. Keep malformed JSON handling explicit: emit `ChatStreamEvent::Error { message: "Invalid request: ..." }` and continue receiving next frames.

  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser client -> `/api/ws/chat` | Untrusted websocket payloads cross into server parsing and agent execution |
| websocket handler -> AgentLoop runtime | Transport layer controls long-running compute + callback streaming |
| server event stream -> browser UI state | Untrusted/disrupted transport can induce inconsistent client state |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-26.1-01 | D (Denial of Service) | `src/server/ws.rs` turn lifecycle | mitigate | On socket send failure, abort in-flight turn task and terminate connection loop cleanly so disconnected clients cannot leave orphan compute loops. |
| T-26.1-02 | T (Tampering) | `src/server/ws.rs` request parsing | mitigate | Keep strict JSON parse gate for `ChatRequest`; malformed frames emit `ChatStreamEvent::Error` and do not crash handler state. |
| T-26.1-03 | I (Information Disclosure) | server->client error path | mitigate | Keep error path structured and bounded to websocket protocol events; avoid panic/backtrace leakage in transport channel. |
| T-26.1-04 | D (Denial of Service) | `warp_hermes.rs` client lifecycle | mitigate | Reconnect-resilient receive loop resets transient UI state on disconnect to prevent permanent stuck state and repeated runaway retries from user actions. |
</threat_model>
