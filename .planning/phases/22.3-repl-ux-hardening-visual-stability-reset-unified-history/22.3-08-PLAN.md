```rust
pub fn reset_terminal_visual(reserved: u16) {
    use std::io::Write as _;
    let mut out = stderr();
    if !out.is_tty() {
        return;
    }
    // CLR-1: erase scrollback + screen.
    let _ = write!(out, "\x1b[3J\x1b[H\x1b[2J");
    // CLR-2 + CLR-3: re-establish DECSTBM and re-anchor prompt.
    if let Ok((_cols, rows)) = size() {
        let scroll_end = rows.saturating_sub(reserved);
        if scroll_end > 0 {
            let _ = write!(out, "\x1b[1;{}r", scroll_end);
        }
        if let Some(bytes) = prompt_position_ansi(rows, reserved) {
            let _ = out.write_all(&bytes);
        }
    }
    let _ = out.flush();
}
```

**The streaming write site that produces unflushed stdout buffer** (main.rs:1679-1686):
```rust
.with_streaming(Box::new(move |delta| {
    print!("{}", delta);
    io::stdout().flush().ok();   // ← already flushes per-token, but the LAST token's flush could race the user typing /clear
    tui_stream.set_activity(ActivityState::Streaming);
}))
```

The streaming callback already flushes per delta. The race is: between the last `print!` of a token and the user typing `/clear` and `Enter`, stdout buffer state at the moment `reset_terminal_visual` is called is "drained" only if no other code path (e.g., the post-turn `println!("{} {}", "Hermes:".bold().cyan(), text)` at main.rs:1034) wrote unflushed bytes after streaming ended. The defensive fix is to flush at the top of `reset_terminal_visual` so we are independent of every upstream writer's buffering discipline.

**`std::io::Write` is already in scope** (line 408 of the function: `use std::io::Write as _;`). The trait must be in scope for `.flush()` to resolve on `std::io::stdout()`. The simplest pattern is to call `flush()` AFTER the existing `use` statement.
