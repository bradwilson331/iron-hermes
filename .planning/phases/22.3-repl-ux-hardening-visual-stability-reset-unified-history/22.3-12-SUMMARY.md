# Phase 22.3 Plan 12: Streaming-Discipline Static-Grep Lockdown Summary

**One-liner:** Three new static-grep regression gates (INV-22.3-07/08/09) in a sibling test file `invariants_22_3_streaming.rs` lock the Plan 22.3-11 streaming-discipline fix against reversion — helper existence + re-export + import shape, run_agent_turn callback uses helper (not legacy `print!`), DECSTBM/DECSC/DECRC bytes encapsulated in render.rs only.

## Objective

Make GAP-22.3-01's structural fix permanent. Future refactors that accidentally revert the streaming-discipline behavior (e.g., a developer reaches for `print!` to debug something and forgets to revert, or a refactor moves the helper without updating callers) will now fail CI before merge.

## Approach

Three new `#[test]` functions in a NEW sibling test file `crates/ironhermes-cli/tests/invariants_22_3_streaming.rs`:

|--------|---------------|
| INV-22.3-07 | `pub fn write_into_scroll_region(bytes: &[u8], reserved: u16)` exists in tui/render.rs, is re-exported via `pub use render::{...}` in tui/mod.rs, and is imported by main.rs via the exact brace-group form `use crate::tui::{reset_terminal_visual, write_into_scroll_region};` |

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] INV-22.3-09 escape-literal vs source-text mismatch**

- **Found during:** Task 1 first test run — `RENDER_RS.contains("\x1b[1;")` returned false despite `\x1b[1;` clearly being in render.rs source.
- **Issue:** `include_str!` loads source files as TEXT. A Rust escape literal `"\x1b[1;"` compiles to a single ESC byte (0x1b) followed by `[1;`, but the rendered source text in render.rs contains the 7-character ASCII sequence `\`, `x`, `1`, `b`, `[`, `1`, `;` — NOT an actual ESC byte. The positive-control assertion trivially failed because I was searching the source text for a byte value that doesn't appear in the source text.
- **Fix:** Switched all six search patterns in INV-22.3-09 from Rust escape literals (`"\x1b[1;"`) to raw string literals (`r"\x1b[1;"`) so the grep matches the source-form the way a developer running `grep -F '\x1b[1;' main.rs` would.
- **Files modified:** `crates/ironhermes-cli/tests/invariants_22_3_streaming.rs` (inside Task 1 commit, folded into `fa28f09`).
- **Intent preserved:** the invariant still locks the same encapsulation rule — DECSTBM/DECSC/DECRC bytes stay in render.rs, not main.rs — but now it actually tests the intended condition.
- **Applies lesson-learned from Plan 22.3-06** §"Specific search strings" about the Rust source-vs-runtime-value distinction.

**2. [Rule 1 - Bug] Module-docstring forbidden-pattern false positive**

- **Found during:** Task 1 acceptance grep — `grep -cE "vt100|expectrl|pty|tokio::test|tracing-test"` returned 1 instead of 0.
- **Cause:** The module-level docstring line `//! harness — static-grep only per CONTEXT D-03 + Phase 21 D-18.` contained the exact phrase `No vt100/expectrl/pty harness` (two lines above the match), and `pty` is one of the forbidden tokens. The acceptance criterion flagged it even though it was a self-documenting mention, not an import.
- **Fix:** Rewrote the docstring from literal crate names (`No vt100/expectrl/pty harness`) to prose (`no terminal-emulator harness, no pseudo-terminal spawner, no scripted-REPL crate`). Intent preserved, acceptance grep returns 0.
- **Files modified:** `crates/ironhermes-cli/tests/invariants_22_3_streaming.rs` (inside Task 1 commit, folded into `fa28f09`).

No other deviations — plan executed exactly as written.
