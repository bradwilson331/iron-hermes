## Concrete Defects (extracted from transcript)

### D-1 — Stdout bleed into slash-command output (first `/agents list`)

**Observed:**
```
Active subagents:failed to connect.imized] target(s) in 26.35s
```

The `/agents list` header line (`Active subagents:`) is immediately followed on
the SAME terminal row by what looks like the tail of a cargo build message
(`...imized] target(s) in 26.35s`) and a provider reconnect warning
(`failed to connect.`). Three writers contending for the same tty row without
a synchronization point.

**Root cause hypothesis:**
- Slash-command handlers print directly to `println!` / stdout.
- Subagent ticker prints via `ExternalPrinter` on a different task.
- Background processes (cargo output pipe? provider reconnect log?) also write raw bytes.
- None of them own the cursor or acquire a paint lock.

**Contract gap:** There is no "who may write to the tty, and when" discipline.

### D-2 — Subcommand typo produces unhelpful error (`/agents log`)

**Observed:**
```
/agents log
Error: Unknown /agents subcommand: log==
```

The trailing `==` is a garbled echo artifact (cursor-position bleed from the
preceding ticker output). The error message string itself appears correct in
the source — the corruption is terminal-level.

**Secondary issue:** no "did you mean `logs`?" suggestion.

### D-3 — `/agents logs <id>` fails when transcript path missing

**Observed:**
```
/agents logs subagent-2
Error: Cannot read transcript: No such file or directory (os error 2)
```

But later the same session:
```
/agents logs subagent-1
{"event":"tool_call","at":"2026-04-24T01:02:42.313869Z",...}
```

**Hypothesis:** TranscriptWriter (Plan 21.7-03) writes to
`$HERMES_HOME/subagent-transcripts/<session>/<subagent_id>.jsonl` but at least
one subagent's transcript file never gets flushed/created before the user
queries it. Likely race between `register(subagent_id)` → first `write_line()`
and the user running `/agents logs` in the narrow window between them.

Note: `subagent_id` here refers to the display alias (`subagent-2`) — handler
must resolve alias → full `sub_XXX` hex → transcript path. The alias-resolution
layer is where the mismatch hides.

### D-4 — ASCII banner leaking into `/agents list` output

**Observed (second `/agents list`):**
```
Active subagents:\| __ / -_) '_| '  \/ -_|_-<
- subagent-1 (sub_8e231073717b) ...
```

The fragment `\| __ / -_) '_| '  \/ -_|_-<` is the ironhermes ASCII banner
(figlet-style "hermes" rendering). It's printing on the same line as the
slash-command output. Confirms D-1 — some code path emits the startup banner
mid-session, and no paint serialization guards against it.

### D-5 — Prompt anchoring still non-deterministic

**User direct feedback:**
> "still very clunky and not enough for a pass"

Plan 21.7-12 moved ANSI cursor positioning onto the worker thread
(`prompt_position_ansi` + `PromptRequest.reserved_rows`) and added a render-loop
barrier. Both gated by unit tests (INV-21.7-14, INV-21.7-15). But the operator
still reports the input line is clunky — symptoms are likely:

- Ticker writes (`[subagent-N] Running: web_read`) hit the tty AFTER the worker
  positioned the cursor, scrolling the prompt again.
- Empty-prompt entry (Enter on a blank input) shows `^C — turn cancelled` on the
  row above and repaints the next prompt without a visible separator.
- Slash-command output does not reserve rows for itself.

**The deterministic property we proved in unit tests does not translate to the
observed UX** because the proof assumed no other writer would touch the tty
between `prepare` and `finish`. That assumption is false in production.

### D-6 — No clear/reset command

**User direct feedback:**
> "There also needs to be a clear command, to reset the prompt"

Today the user has no way to:

- Clear the scrollback + redraw the prompt at a known row.
- Re-anchor after a visual glitch without killing + restarting `hermes`.

Candidates: `/clear` (like shell), `/reset`, dedicated key binding (e.g., Ctrl-L
routed through rustyline config).

### D-7 — History recall does not include slash commands

**User direct feedback:**
> "recall must recall slash commands as well as prompts"

Readline up-arrow history likely only records chat prompts (fed through the
chat path) and skips slash commands (dispatched through the slash router before
hitting the chat history store). Unified history store needed — probably
persistent across restarts too.

## Suggested UI-SPEC Scope

These are inputs for `/gsd-ui-phase 22.3`, not commitments:

1. **TTY ownership contract** — exactly one writer at a time for the "live region"
   (prompt + status pill). Ticker, banner, slash-command output, provider logs
   must all go through the paint coordinator or to a scrollback-only channel.
2. **Slash-command output framing** — guarantee `/agents list` etc. print their
   own block without interleaving with ticker writes. Likely pause ticker /
   buffer it during slash dispatch, flush after.
3. **Unified history store** — single ring buffer capturing chat prompts AND
   slash commands, restored on restart. Up-arrow surfaces both. Configurable
   max size.
4. **Reset command** — `/clear` (or equivalent) that redraws the status line at
   a known row and wipes scrollback in a way that's safe under TUI scroll
   region (DECSTBM).
5. **Alias→transcript resolution hardening** — fix the alias path race in
   `/agents logs` so a just-registered subagent has a queryable transcript
   (D-3). May require TranscriptWriter to create an empty file on register,
   before the first line writes.
6. **Typo affordance** — "Unknown subcommand: X — did you mean Y?" for slash
   commands with close matches.
