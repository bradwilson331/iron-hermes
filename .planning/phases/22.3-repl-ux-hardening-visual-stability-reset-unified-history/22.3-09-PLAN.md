<objective>
Close `WR-03` from `22.3-REVIEW.md` by joining the worker `std::thread::JoinHandle` inside `ReplInputChannel::shutdown(self)`. Today the method sends `Command::Shutdown` then drops `self`. The worker's `Command::Shutdown` arm at `repl_input.rs:328-340` calls `rl.save_history(path)` before `break`-ing the loop, but on the spawning side, dropping the `JoinHandle` does NOT cancel or wait for the OS thread. Under the emergency-exit path (`std::process::exit(130)`) and under fast-shutdown sequences, the OS may schedule the worker thread to handle the Shutdown after the spawning thread has already returned from `run_chat`, leading to a process exit that clips the history save mid-write.

The fix joins the worker thread synchronously in `shutdown()`. This guarantees `save_history` completes before `shutdown()` returns. The change is local to the `shutdown` method and one supporting field declaration semantic (`Option::take` instead of bare drop).

**`Option::take` pattern** (used throughout the codebase to extract owned content from a `&mut Option<T>` field on `self`): `let handle = self.worker.take();` then `if let Some(h) = handle { ... }`. Because `shutdown(self)` consumes `self` by value (not `&mut self`), we have unique ownership and can use `let handle = self.worker.take();` even though `self` is consumed at the end of the function. (Note: `self.worker.take()` requires `self.worker: Option<...>` to be a field that can be taken; the field declaration confirms this.)

**send_command_only signature** — sends Command on the command channel, returns `Result<(), SendError<Command>>`. The leading `let _ =` is correct: if the channel is closed (because the worker already exited), the Shutdown send fails harmlessly and the join below will see the worker has already terminated.

**Why we still need the send-then-join order:**
- The worker's `cmd_rx.blocking_recv()` returns either `Some(Command::Shutdown)` (our explicit send) or `None` (channel closed when `self.cmd_tx` drops).
- If we ONLY join (without sending), the worker stays blocked on `cmd_rx.blocking_recv()` forever — because `self.cmd_tx` is moved into `self` and only dropped when the function returns.
- If we send-then-join, the worker observes the Shutdown explicitly, runs save_history, and breaks — then the join returns.
- The bare drop sequence in the OLD code is broken because `self.cmd_tx` is dropped at function return time, but by then the spawning thread has already returned and (under emergency exit) the process may have exited before the worker reads its channel.

    /// **Phase 22.3 WR-03 (review fix):** joins the worker thread after
    /// sending `Command::Shutdown` so `rl.save_history(path)` (the

    /// on `Drop` to release the command sender — correct in steady
    /// state, but unreliable under fast shutdown.
    ///

    /// `cmd_rx.blocking_recv()`. We must send Shutdown first so the
    /// worker observes the command and runs the save_history branch
    /// before terminating; only then is the join meaningful.
    pub fn shutdown(mut self) {
        let _ = self.cmd_tx.send_command_only(Command::Shutdown);
        if let Some(handle) = self.worker.take() {
            // Best-effort: if the worker panicked, join returns Err and we
            // cannot recover the history. There is no other path that could.
            let _ = handle.join();

        // Dropping `self` after the join releases the command sender; by
        // this point the worker has already exited, so the closed-channel
        // observation is moot.
    }
