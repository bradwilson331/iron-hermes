Close `GAP-22.3-01` from `22.3-HUMAN-UAT.md` — the BLOCKING live-TTY defect where the input prompt is overwritten by streaming token output, agent output is invisible until Ctrl+C, and slash commands cannot be used during streaming.

**Root cause (verified from RESEARCH.md + grep + read):**

The DECSTBM scroll region IS established at REPL boot (`tui/render.rs:188-196` writes `\x1b[1;{scroll_end}r` immediately when `TuiHandle::new_with_extensions` is called) and IS re-applied on resize (`tui/render.rs:518-529` in the render loop). So the scroll region is not the problem.

The problem is at the **streaming write callback** in `run_chat`'s `run_agent_turn` (main.rs:1679-1686):

```rust
.with_streaming(Box::new(move |delta| {
    print!("{}", delta);              // ← raw print, no cursor save
    io::stdout().flush().ok();        // ← flush, but cursor is wherever rustyline last left it
    tui_stream.set_activity(ActivityState::Streaming);
}))
```

When a mid-turn rustyline prompt is primed (`repl_input.request_prompt(...in_turn: true...)` at main.rs:1274), rustyline owns the prompt row at the bottom of the terminal. Each `print!("{}", delta)` writes from wherever the cursor currently is — which, after rustyline's most recent paint, is on or near the prompt row. The token bytes land on the prompt row, rustyline repaints over the prompt prefix, the token visually disappears or fragments, and the user sees the symptoms in the UAT transcript.

**The fix (Option A from CONTEXT D-15 / GAP-22.3-01 recommended_approach_options):**

Per-write cursor discipline. Every streaming write site:
1. Saves the cursor (`\x1b7` DECSC).
2. Moves the cursor to the bottom row OF THE SCROLL REGION (`\x1b[{scroll_end};1H`) — the row directly above the prompt — so the next `print!` lands inside the scroll region.
3. Writes the token bytes + flushes stdout. DECSTBM's hardware-scroll guarantees the bytes scroll up inside the scroll region without touching the prompt row.
4. Restores the cursor (`\x1b8` DECRC) to where rustyline left it.

This is encapsulated in a new `tui::render::write_into_scroll_region(bytes: &[u8], reserved: u16)` helper, used from both the streaming-token callback (high-frequency, per-delta) and the post-turn `Hermes:` label write (low-frequency).

**Why NOT alternate screen (Option B):** loses scrollback continuity. The user just spent Phases 21 / 22 hardening `/list` and persistent history; tossing scrollback for `/clear` semantics would regress that work. UI-SPEC explicitly preserves scrollback for `/clear` (CLR-1: erase scrollback so banner bleed self-heals, but this is a one-shot user action, not a perpetual mode change).

**Why NOT full TUI redraw (Option C):** would push toward ratatui complexity without ratatui (forbidden by D-03 / D-18); also would require a content ring buffer + redraw loop, violating CONTEXT D-13's "PaintCoordinator is NOT a new struct" rule.

```rust
/// Phase 22.3 GAP-22.3-01 closure / UI-SPEC OWN-2 (PaintCoordinator):
/// Write `bytes` into the DECSTBM scroll region without disturbing the
/// rustyline-owned prompt row. Used by the streaming-token callback in
/// `run_chat`'s `run_agent_turn` and by the post-turn `Hermes:` label.
///
/// Mechanism:
///   1. `\x1b7` (DECSC) — save the current cursor position (typically
///      the prompt row, where rustyline most recently positioned it).
///   2. `\x1b[{scroll_end};1H` — absolute CUP to the LAST row of the
///      scroll region (one row above the prompt). Subsequent writes
///      land inside the scroll region; DECSTBM's hardware-scroll moves
///      older content UP as new content arrives.
///   3. Write `bytes` to stdout, then flush.
///   4. `\x1b8` (DECRC) — restore the cursor to its pre-write position
///      (the prompt row). Rustyline's next paint sees the cursor
///      exactly where it left it; nothing visually shifts.
///
/// All escape sequences are emitted to STDOUT (same stream as the
/// payload) so the save / write / restore triple is atomic at the
/// stream level. No race against stderr writes by the render loop.
///
/// **Non-TTY fallback:** when `stdout` is not a tty (e.g., redirected
/// to a file by a CI runner or piped), the function writes `bytes` to
/// stdout + flush, with NO escape sequences — preserving the existing
/// non-interactive behavior so log captures stay clean.
///
/// **Tiny terminals / size() failure:** if `crossterm::terminal::size`
/// fails OR `scroll_end == 0` (terminal too small to have a scroll
/// region after subtracting `reserved`), the function falls back to
/// the same plain-stdout path as the non-TTY case. The escape
/// sequences are skipped and content still reaches the user.
///
/// Errors are silently dropped per existing render.rs discipline.
pub fn write_into_scroll_region(bytes: &[u8], reserved: u16) {
    use std::io::Write as _;
    let mut out = std::io::stdout();
    if !out.is_tty() {
        let _ = out.write_all(bytes);
        let _ = out.flush();
        return;
    }
    let Ok((_cols, rows)) = size() else {
        let _ = out.write_all(bytes);
        let _ = out.flush();
        return;
    };
    let scroll_end = rows.saturating_sub(reserved);
    if scroll_end == 0 {
        let _ = out.write_all(bytes);
        let _ = out.flush();
        return;
    }
    // DECSC: save cursor (typically the rustyline-owned prompt row).
    let _ = write!(out, "\x1b7");
    // CUP to the last row of the scroll region. DECSTBM hardware-scroll
    // pushes older content up as we write into this row.
    let _ = write!(out, "\x1b[{};1H", scroll_end);
    // Payload + flush.
    let _ = out.write_all(bytes);
    let _ = out.flush();
    // DECRC: restore cursor to pre-write position (the prompt row).
    let _ = write!(out, "\x1b8");
    let _ = out.flush();
}
```
