## Scroll-Region Geometry (Spacing/Layout)

The DECSTBM scroll region is set ONCE at TUI startup and recalculated dynamically on terminal resize (SIGWINCH). The reserved area at the bottom of the terminal has the following fixed layout, top-to-bottom within the reserved zone:

```
┌─── scroll region (rows 1 .. scroll_end) ──────────────────────┐
│  scrollable content: agent output, slash-command blocks,       │
│  ticker lines (painted via ExternalPrinter above prompt row)   │
└────────────────────────────────────────────────────────────────┘
 row: rows - reserved_row_count      ← PROMPT ROW (owned by rustyline)
 row: rows - reserved_row_count + 1  ← SCANNER ROW (knight-rider / idle)
 [row: rows - reserved_row_count + 2..N  ← BelowScanner widget slots (0 now)]
 row: rows - 1                       ← STATUS BAR (pills)
```

**Reserved row count formula:** `3 + sum(extension_widget_heights)`. With zero extensions currently wired, `reserved_row_count == 3`. This is the only source of truth; all saturating_sub calls MUST delegate to `TuiHandle::reserved_row_count()` (locked by INV-22.1-04).

**Slash-command output geometry:**

Slash-command output is written into the SCROLL REGION (above the prompt row), not into the reserved area. The block consists of:
- Header line: one line
- Body lines: variable
- Footer separator: one blank line

After slash-command output is written, the DECSTBM scroll region contains it and it scrolls naturally as more output arrives. The prompt row position does not change.

**Minimum terminal size:** 4 rows, 40 columns. Below this threshold the render loop skips all reserved-area writes (`rows < 4` guard already present in `prompt_position_ansi`). Slash-command output is not width-constrained but is clipped to terminal column width by the render loop's existing truncation.

**Column width rule for slash output:** Each slash-command output line is truncated at `terminal_cols - 1` columns (1-column right margin). No wrapping — truncate with no ellipsis (raw data display).

## Copywriting Contract

### Slash-Command Output Block Format

Every slash-command that produces visible output MUST use this block format:

```
{header_line}
{body_lines...}

```

Where:
- `{header_line}` is the first line of the command's output, rendered `.bold()`.
- `{body_lines...}` are subsequent lines, plain (no color).
- A single blank line follows the last body line to visually separate the block from ticker or the next prompt.

**`/agents list` header:** `Active subagents:` (bold). Body: one line per agent in format `- subagent-N (sub_XXXX) (summary…) — Xs`.

**`/agents logs <id>` header:** `Transcript: subagent-N (sub_XXXX)` (bold). Body: raw JSONL lines, no color.

**`/agents kill <id>` output:** `Cancelled subagent sub_XXXX.` (plain, one line, no header/footer).

**`/clear` output:** no body text. Terminal is cleared and redrawn. See Reset Command section.

### Empty States

| State | Copy |
|-------|------|
| No active subagents | `No active subagents.` (plain, one line) |
| No transcript yet for subagent | `Transcript not yet available for subagent-N. The subagent may not have started a tool call yet.` |
| Slash-command output empty | (not shown — handler returns Silent if truly empty) |

### Error States

All errors follow: `Error: {problem description}` on stderr (`.red()`), then optionally a second line with the suggestion (`.dimmed()`).

| Error | Copy |
|-------|------|
| Unknown top-level slash command | `Error: Unknown command: /{X}` followed by `— did you mean /{Y}?` (see Typo Affordance section) |
| Unknown `/agents` subcommand | `Error: Unknown /agents subcommand: {X}` followed by `— did you mean {Y}?` |
| `/agents kill` missing id | `Error: /agents kill <id>: missing id` |
| `/agents logs` missing id | `Error: /agents logs <id>: missing id` |
| Transcript file not found | `Error: Cannot read transcript for subagent-N. The file may not exist yet — try again in a moment.` (replaces raw OS error string) |
| Ambiguous alias prefix | `Error: Ambiguous id '{X}' — matches: {A}, {B}, …` |

**Tone rule:** Error copy states the problem and, where actionable, the next step. Do not expose raw OS error strings (e.g. `(os error 2)`) to the user. Wrap them: `Cannot read transcript for {alias}. {human-readable reason}.`

### Destructive Action Copywriting

| Action | Trigger | Confirmation approach |
|--------|---------|----------------------|
| `/agents kill <id>` | Explicit subcommand | No confirmation — action is already explicit. Output `Cancelled subagent {id}.` |
| `/clear` | Command or Ctrl-L | No confirmation required (scrollback wipe + redraw is reversible by scrolling). Execute immediately. |
| Double Ctrl-C exit | Second Ctrl-C within 1.5s during in-flight turn | No text prompt — existing `Goodbye!` is the only output (locked by Phase 21 D-12) |

## TTY Ownership Contract (PaintCoordinator)

### Problem Statement (D-1, D-4, D-5)

The UAT evidence shows three writers contending for the same terminal rows without synchronization:
1. Slash-command handlers (`println!` to stdout)
2. Subagent ticker (`ExternalPrinter` → rustyline buffered printer)
3. Background processes (cargo stdout pipe, provider reconnect log, ASCII banner)

### Ownership Rules

**Rule OWN-1: Single live-region owner.** The "live region" is the prompt row (row `rows - reserved_row_count`) and the reserved area below it (scanner row, status bar row). At any instant, exactly one owner controls writes to this region:

- **Owner A: TUI render loop** — owns the reserved area at all times EXCEPT during active readline.
- **Owner B: rustyline worker** — owns the prompt row during the inter-turn readline window (`readline_active == true`). The TUI render loop skips the status-row write (not the scanner or widget rows) during this window (INV-21.7-15, already implemented).

**Rule OWN-2: Scroll-region is for scrollable content.** All non-reserved output (agent responses, slash-command blocks, ticker lines) flows through the scroll region. The scroll region start is row 1; the scroll region end is `rows - reserved_row_count` (exclusive). Writers that use `println!` / `eprintln!` or `ExternalPrinter::println` are automatically constrained to this region by the DECSTBM hardware scroll guard.

**Rule OWN-3: Slash-command handlers are sync.** They call `println!` to stdout, which flows into the scroll region. This is correct — no change needed to the output path. The problem is that ticker writes interleave with the slash output. The fix is to buffer the ticker during slash dispatch (see Ticker Buffering below).

**Rule OWN-4: Banner, cargo output, and provider reconnect logs are stray writers.** They MUST be rerouted:
- **Banner (`print_banner`):** Already flushed before the async REPL enters its loop (GAP-5 fix, commit in Phase 21.2-09). The problem in D-4 is that `print_banner()` is being called again mid-session from some code path. The spec requires: `print_banner()` is called ONCE, before `TuiHandle::new()` initializes the scroll region, and never called again during the session.
- **Cargo stdout:** Background compile output (from `execute_code` / child processes) MUST be consumed via the ProcessRegistry's rolling buffer and surfaced only through `/stop` or watch-pattern notifications — never directly to the terminal during a live REPL session.
- **Provider reconnect logs (`tracing::warn!` / `eprintln!`):** Already gated by the interactive tracing filter (`error` level for interactive REPL) installed in Plan 21.2-09. No change required here; the `failed to connect.` bleed in D-1 is a consequence of the ticker/slash race, not a new source.

**Rule OWN-5: `ExternalPrinterHandle::println` is the ONLY path for concurrent output during readline.** Any code that writes to the terminal while the rustyline worker owns the prompt row MUST route through `ExternalPrinterHandle`. Direct `eprintln!` / `println!` during readline is forbidden for concurrent writers. Slash-command handlers are NOT concurrent with readline (they execute after a line is received and before the next `request_prompt`) — they may use `println!` directly.

### Ticker Buffering During Slash Dispatch

**Contract TB-1:** When a slash command is dispatched from the prompt-time branch (not the mid-turn branch), the ticker is NOT active because there is no in-flight agent turn. No buffering needed at prompt time.

**Contract TB-2:** When a slash command is dispatched from the mid-turn branch (`in_turn: true` select arm), the ExternalPrinter continues to fire. The correct behavior is that ticker lines continue to appear in the scroll region interleaved with slash output — this is acceptable because both writers target the scroll region via ExternalPrinter, and the DECSTBM hardware serializes concurrent writes at the TTY driver level. The visual artifact in D-1/D-2 is caused by the ANSI escape code corruption (the `==` in `Unknown /agents subcommand: log==`) from cursor-position escapes bleeding into the line, not from line interleaving per se.

**Contract TB-3:** The `==` corruption artifact in D-2 is a consequence of the cursor-position escape codes (`\x1b[6n` DSR response) from the worker's readline being emitted before the slash output line is fully written. This is fixed by OWN-1 (readline barrier) which is already implemented in Plan 21.7-12. No additional buffering mechanism is required.

## Slash-Command Output Block (SlashOutputBlock)

### Contract

**SL-1: Header is always the first line.** Every slash command that produces multi-line output MUST print the header as the first `println!` call, followed by body lines. Single-line outputs (e.g. "Cancelled subagent …") do not need a header/body split.

**SL-2: Block separator.** After the last body line, a single empty `println!()` call creates visual separation. This is a discipline rule for the handlers — not a wrapper function. (A wrapper function is optional; the planner may choose to add `slash_block_print!` helper but it is not required by this spec.)

**SL-3: No ANSI in transcript body.** `/agents logs` output is raw JSONL. It MUST NOT be colorized. The terminal is responsible for line wrapping. The handler already does this correctly.

**SL-4: Width clipping.** Each output line is written as-is to stdout. The scroll region's terminal-driver behavior clips at the right margin without ANSI corruption. No explicit clipping is required in handlers. (Contrast with the render loop's widget slot clipping which IS explicit — widget content goes through crossterm's cursor moves, not raw println.)

**SL-5: Ticker continues during mid-turn slash output.** The interleaving of `[subagent-N] Running: X` lines with slash output is ACCEPTABLE as long as each line is a complete terminal line (no partial-line corruption). OWN-5 + the existing ExternalPrinter path already guarantee this.

## Prompt Reset Command (D-6)

### Name and Binding

**Name:** `/clear` (primary). No alias. No keybinding for Ctrl-L in Phase 22.3 (deferred — adding a keybinding requires changes to the rustyline config and the pre-readline keybinding dispatch layer; out of scope).

**Rationale for name choice:** `/clear` matches shell convention and is already stubbed as `cmd_clear` in `handlers.rs` which returns `CommandResult::ClearSession`. Phase 22.3 repurposes this to a TTY-visual reset (screen wipe + scroll region redraw) rather than a session-history wipe.

**Disambiguating `/clear` from `/new`:**

| Command | Effect on conversation | Effect on terminal |
|---------|----------------------|-------------------|
| `/new` | Wipes conversation history (messages truncated to system message) | No visual reset |
| `/clear` | Does NOT wipe history | Full visual reset: clear scrollback, redraw prompt at known row |

This is the key design decision. `/clear` is purely visual. History is never mutated by `/clear`.

### Behavior Contract

**CLR-1: Scrollback wipe.** Emit `\x1b[3J` (erase scrollback) + `\x1b[H\x1b[2J` (home cursor + erase screen) to stderr before re-establishing the scroll region. This is safe under DECSTBM — the erase happens before DECSTBM is restored.

**CLR-2: Scroll region re-anchor.** After erase, re-emit `\x1b[1;{scroll_end}r` with the current `reserved_row_count` to re-establish DECSTBM. This is equivalent to what `TuiHandle::new_with_extensions` does at startup.

**CLR-3: Prompt re-anchor.** After DECSTBM is restored, emit `prompt_position_ansi(rows, reserved_row_count)` to position the cursor at the prompt row.

**CLR-4: History preserved.** `/clear` does NOT call `messages.truncate()`. Conversation context survives.

**CLR-5: Status bar redraws on next render tick.** The status bar is not explicitly repainted by `/clear` — the render loop's 100ms tick fires within ≤100ms and repaints it naturally. No explicit repaint call needed.

**CLR-6: No confirmation.** Execute immediately on `/clear`. No "Are you sure?" prompt.

**CLR-7: Output.** After the visual reset, the REPL issues the next `You: ` prompt. No intermediate text is printed. The effect is that the screen clears and the prompt appears at a known row.

**CLR-8: Implementation site.** `/clear` is handled by `cmd_clear` in `handlers.rs`, which returns a new `CommandResult` variant (e.g. `CommandResult::ResetTerminal`). The `run_chat` loop matches this variant and calls a new helper `reset_terminal_visual(reserved_row_count)` before `continue`-ing to the next prompt cycle.

## Unified History Store (D-7)

### Contract

**HIST-1: Single ring buffer.** Chat prompts and slash commands are stored in the same rustyline history. `repl_input.add_history(line)` is called for BOTH chat prompts AND slash commands (currently only called for chat prompts at main.rs:1213).

**HIST-2: Add slash commands to history.** At the slash-dispatch site (prompt-time branch, after the leading `/` is confirmed), call `repl_input.add_history(&input)` with the full slash command string (including the `/`). This mirrors the existing `repl_input.add_history(&input)` at main.rs:1213.

**HIST-3: Up-arrow recalls both.** Since both kinds of lines are stored in the same rustyline history, up-arrow in rustyline naturally surfaces both. No additional plumbing required.

**HIST-4: Persistence across restarts.** The `history_path: Option<PathBuf>` parameter of `ReplInputChannel::spawn` is currently ignored (the code has `let _ = history_path;`). Phase 22.3 activates it:
- Path: `$HERMES_HOME/repl_history` (resolved from `ironhermes_core::get_hermes_home().join("repl_history")`)
- Format: rustyline's native history file format (call `rl.save_history(&path)` on worker exit and `rl.load_history(&path)` on worker init)
- The file is created on first write; if it doesn't exist, load is a no-op (rustyline's `load_history` already handles missing files gracefully)

**HIST-5: Max size.** Default: 1000 entries. Configured by `rl.set_max_history_size(1000)` on worker init. No config-file key in Phase 22.3 — hardcoded default. (Config-file key is a deferred enhancement.)

**HIST-6: De-duplication.** Rustyline's native `HistoryDuplicates::Prev` policy: if the same line appears consecutively, only one entry is kept. Enabled via `rl.set_history_duplicates(rustyline::config::HistoryDuplicates::Prev)` on worker init. Non-consecutive duplicates are kept (standard shell behavior).

**HIST-7: Redaction.** Commands are stored; responses are NOT stored. Specifically:
- Stored: `You: use agents to research…` → stored as `use agents to research…` (without `You: ` prefix)
- Stored: `/agents list` (full slash command string)
- NOT stored: agent response text, slash-command output, ticker lines
- NOT stored: empty inputs (existing `if input.is_empty() { continue; }` guard prevents empty-line history)

**HIST-8: Mid-turn slash commands.** Slash commands dispatched from the mid-turn select arm are NOT added to history. History entries are only added from the prompt-time branch where the user typed a full line and received it back via `recv_line()`. The mid-turn arm receives the line and dispatches it immediately without calling `add_history`.

## Alias to Transcript Resolution (D-3)

### Root Cause

`/agents logs subagent-2` failed with `Cannot read transcript: No such file or directory (os error 2)` because `TranscriptWriter` creates the JSONL file lazily on the first write, not on `register`. In the narrow window between `register(subagent_id)` and the first `write_line()`, the file does not exist.

### Contract

**ALIAS-1: Transcript file created at registration time.** When `SubagentRegistry::register(subagent_id, task_summary, token)` is called, `TranscriptWriter` MUST create the transcript file immediately — before any event is written. This is a "touch" operation: open the file in append mode (`OpenOptions::new().create(true).append(true)`) and close it immediately. The file then exists for `stat()` / `read_to_string()` even if zero events have been written.

**ALIAS-2: Error copy for missing transcript.** If `std::fs::read_to_string` fails after this change, it is a genuine I/O error (disk full, permissions). The error copy changes from `Cannot read transcript: {os_error}` to `Cannot read transcript for {alias}: {human_readable_reason}`. The OS error string is included in parentheses for diagnostic value but never displayed as the primary message. Example: `Cannot read transcript for subagent-2: permission denied`.

**ALIAS-3: Transcript not yet available.** If `reg.transcript_path(&resolved)` returns `None` (alias resolved but path not registered), the message is: `Transcript not yet available for {alias}. The subagent may not have started a tool call yet.` This covers the case where registration happened but `with_transcript_scope` was not called on the runner.

**ALIAS-4: Alias resolution is already correct.** The `resolve_subagent_id` function in handlers.rs (landed in post-UAT commit e5a3dc7) correctly handles exact-id, positional alias (`subagent-N`), and prefix matching. No change to resolution logic.

## Typo Affordance (D-2)

### Contract

**TYPO-1: Levenshtein threshold.** If `levenshtein_distance(user_token, known_command) <= 2`, show a suggestion. Threshold 2 catches single-character mismatches, transpositions, and `log` vs `logs`. Threshold is applied to lowercase tokens (case-insensitive comparison).

**TYPO-2: Suggestion format.** Append to the error line:
- If exactly one candidate is within threshold: `— did you mean {Y}?` (`.dimmed()`)
- If 2-3 candidates are within threshold: `— did you mean one of: {A}, {B}?` (`.dimmed()`)
- If more than 3 candidates or zero candidates: no suggestion appended

**TYPO-3: Applies to `/agents` subcommands.** The known subcommand list for `/agents` is: `list`, `kill`, `logs`. A user typing `/agents log` gets: `Error: Unknown /agents subcommand: log — did you mean logs?`

**TYPO-4: Applies to top-level slash commands.** The known top-level commands are those registered in the `CommandRouter` (44 commands). A user typing `/agent` gets: `Error: Unknown command: /agent — did you mean /agents?`

**TYPO-5: Implementation.** A pure function `suggest_typo(input: &str, candidates: &[&str]) -> Option<Vec<String>>` returns the matching candidates. This function is testable without IO. The Levenshtein implementation MUST NOT add a new crate dependency — implement a simple O(n*m) matrix in ~20 lines of Rust (standard algorithm, no trait magic needed).

**TYPO-6: Error output path.** Suggestions are appended to the error string returned by `CommandResult::Error(msg)`. The existing `eprintln!("{}", format!("Error: {}", msg).red())` at main.rs:1159 prints the full string. No change to the output path.

## Visual Stability Invariants

Phase 22.3 EXTENDS the existing invariants from Plans 21.7-12 (INV-21.7-14/15) and Plan 22.1 (INV-22.1-04). All existing invariants must remain green. The following new invariants must hold:

### New Invariants

**INV-22.3-01: Slash-command output does not move the prompt row.**
After a prompt-time slash command produces output (`println!` to stdout), the next `You: ` prompt anchors at the same row as before the slash output. This is enforced structurally by DECSTBM: the scroll region pushes content up; the prompt row is outside the scroll region and does not shift.

**INV-22.3-02: Banner is printed exactly once per session, before `TuiHandle::new()`.**
`print_banner()` is called only at the top of `run_chat`, before `TuiHandle::new_with_extensions()` initializes the scroll region. It is never called after that. Verified by static grep: `grep -c "print_banner()" main.rs == 1` AND the single call site is before any `TuiHandle::new` call.

**INV-22.3-03: `/clear` does not mutate `messages`.**
The `ResetTerminal` variant handler in `run_chat` does NOT call `messages.truncate()` or modify the conversation history vector. Only `ClearSession` (from `/new`) may do so.

**INV-22.3-04: Slash commands are added to history at the prompt-time dispatch site.**
At the slash-dispatch site in the prompt-time branch, `repl_input.add_history(&input)` is called for slash-command lines. Verified by static grep: the `add_history` call appears AFTER the `if input.starts_with('/')` check and before the `continue` at the end of the slash dispatch match.

**INV-22.3-05: Transcript file exists at the moment the alias appears in `/agents list`.**
After `SubagentRegistry::register()`, the transcript file MUST exist on disk. Verified by a new unit test that calls `register()` and immediately asserts the file exists via `std::fs::metadata()`.

**INV-22.3-06: Mid-turn slash dispatch does not add history.**
The mid-turn `recv_line()` arm in `run_chat` does NOT call `repl_input.add_history()`. Only the prompt-time branch adds to history.

### Preserved Invariants (must stay green)

| Invariant | Source | What it checks |
|-----------|--------|---------------|
| INV-21.7-14 | Plan 21.7-12 | `PromptRequest.reserved_rows` field present + worker calls `prompt_position_ansi` |
| INV-21.7-15 | Plan 21.7-12 | `readline_active` in render.rs (≥2 refs) + `readline_active_handle()` in main.rs + barrier toggle |
| INV-22.1-04 | Plan 22.1-02 | `readline_active_handle()` present + `reserved_rows: Some(` at prompt site |
| INV-21.7-12 | Plan 21.7-11 | Mid-turn dispatch arm exists in run_chat |
| INV-21.7-13 | Plan 21.7-11 | ReplInputChannel spawned in run_chat |

## Component Specifications

### PaintCoordinator

The PaintCoordinator is NOT a new struct. It is the combination of:
1. The existing `TuiHandle.readline_active: Arc<AtomicBool>` barrier (Plan 21.7-12)
2. The existing `ExternalPrinterHandle` for concurrent output during readline
3. Rule OWN-1 through OWN-5 in this spec

No new Rust struct is required. The contract is enforced via discipline rules and static-grep invariant tests.

**New API surface required:** None beyond what is already present.

### SlashOutputBlock

**Implementation:** A helper macro or function `print_slash_block(header: &str, body_lines: &[String])` that:
1. `println!("{}", header.bold())` — header line
2. For each line in body_lines: `println!("{}", line)` — body lines
3. `println!()` — blank separator line

This is OPTIONAL as a function — handlers may inline the pattern. The planner should decide. The spec requires the output pattern; not the function.

### UnifiedHistoryStore

**Changes to `ReplInputChannel::spawn`:**
1. Accept `history_path: Option<PathBuf>` (already in signature, currently ignored)
2. On worker init: if path is Some, call `rl.load_history(&path)` (ignore error if file missing)
3. Set `rl.set_max_history_size(1000)` and `rl.set_history_duplicates(HistoryDuplicates::Prev)` on init
4. On `Command::Shutdown`: if path is Some, call `rl.save_history(&path)` (log error via eprintln if fails)

**Changes to `run_chat` in main.rs:**
1. Compute history path: `let history_path = Some(ironhermes_core::get_hermes_home().join("repl_history"))`
2. Pass to `ReplInputChannel::spawn(Some(history_path))`
3. At slash-dispatch site (prompt-time branch): add `repl_input.add_history(&input)` call alongside the existing one at main.rs:1213

### ResetCommand

**Changes to `handlers.rs`:**

**New helper `reset_terminal_visual(reserved: u16)` in render.rs or a new file:**
```rust
pub fn reset_terminal_visual(reserved: u16) {
    let mut out = std::io::stderr();
    if !out.is_tty() { return; }
    // 1. Erase scrollback + screen
    let _ = write!(out, "\x1b[3J\x1b[H\x1b[2J");
    // 2. Re-establish DECSTBM scroll region
    if let Ok((_cols, rows)) = crossterm::terminal::size() {
        let scroll_end = rows.saturating_sub(reserved);
        if scroll_end > 0 {
            let _ = write!(out, "\x1b[1;{}r", scroll_end);
        }
        // 3. Position cursor at prompt row
        if let Some(bytes) = prompt_position_ansi(rows, reserved) {
            let _ = out.write_all(&bytes);
        }
    }
    let _ = out.flush();
}
```
