# Phase 22.3: REPL UX hardening (visual stability + reset + unified history) - Context

**Gathered:** 2026-04-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Close the six concrete TTY-UX defects (D-1..D-7 minus D-5 — D-5 prompt anchoring was already addressed by Plan 21.7-12) captured verbatim in `22.3-UAT-EVIDENCE.md` against the live-TTY session that immediately followed the GAP-21.7-02 unit-level fix:

1. **D-1 / D-4 — TTY ownership discipline:** stop slash-command output, ticker writes, banner fragments, and provider reconnect logs from clobbering one another inside the same terminal row. Enforce the PaintCoordinator rules (OWN-1..5) from the UI-SPEC.
2. **D-2 — Typo affordance:** add Levenshtein-≤2 "did you mean …?" suggestions for unknown top-level slash commands and unknown `/agents` subcommands.
3. **D-3 — Alias→transcript race:** TranscriptWriter touches the JSONL file at `register()` time so `/agents logs <alias>` always finds something on disk.
4. **D-6 — `/clear` visual reset:** repurpose `cmd_clear` to a TTY-only redraw (scrollback wipe, DECSTBM re-anchor, prompt re-anchor) that **does NOT mutate `messages`** — `/new` keeps its existing message-truncate semantics.
5. **D-7 — Unified persistent history:** chat prompts AND prompt-time slash commands flow into the same rustyline ring, persisted to `$HERMES_HOME/repl_history`, replayable across restarts. Activates the `history_path` parameter that already exists in `repl_input.rs` but is currently ignored.

**In scope:** All implementation work needed to satisfy the 22.3-UI-SPEC.md contract end-to-end and re-pass the 22.3-UAT-EVIDENCE.md scenario on a live TTY.

**Out of scope (UI-SPEC-locked deferrals):**
- Ctrl-L keybinding for `/clear` (requires rustyline config + pre-readline keybinding plumbing)
- Config-file key for max history size (hardcoded 1000 in this phase)
- Redaction list for sensitive history entries (verbatim storage)
- New crate dependencies (Phase 21 D-18 reaffirmed)

<decisions>
## Implementation Decisions

### Plan structure & wave parallelization

- **D-01:** Two-wave plan structure. Wave 1 (foundations, parallel-safe): typo helper as a pure function, TranscriptWriter touch-on-register, history activation in `repl_input.rs`, new `CommandResult::ResetTerminal` variant added to both core (`commands/mod.rs`) and TUI (`tui/extension.rs`) enums plus the mapper at `tui/commands.rs:112-125`. Wave 2 (integration, sequential): `run_chat`/`run_single` integration — slash-side `add_history` call, `ResetTerminal` arm in the dispatch match, banner-once invariant, copy fixes for the locked error strings. **Why:** UI-SPEC components are pure-function-shaped at the boundary (typo, transcript touch) and only the integration site (main.rs around lines 1142-1213) requires sequential edits. 4-5 plans total maximises parallelism while keeping each plan reviewable.

### Banner bleed (D-1 / D-4) investigation

- **D-02:** Spawn a research probe BEFORE writing plans. The static-grep already shows `print_banner()` is called only once (main.rs:673), yet UAT D-4 captured `Active subagents:\| __ / -_) '_| '  \/ -_|_-<` mid-session — a banner fragment. Hypotheses to investigate: (a) `tracing::warn!` re-emitting cached banner bytes through a stderr scrollback redraw, (b) DECSTBM scroll region not wiping the row before reuse so old banner pixels persist, (c) a `print_banner`-equivalent path through a gateway/setup helper not visible to the grep, (d) terminal driver re-rendering scrollback after `\x1b[H` cursor moves issued by `prompt_position_ansi`. **Why:** The user explicitly chose investigation over "trust the spec" — landing INV-22.3-02 (single call site) without understanding the actual writer risks the bleed reappearing in re-UAT and forcing another phase. **Probe deliverable:** a 1-2 page findings note appended to `22.3-UAT-EVIDENCE.md` (or a sibling `22.3-BANNER-PROBE.md`) identifying the writer + recommended fix BEFORE the planner finalises the integration wave plan.

### Test strategy

- **D-03:** Static-grep regression invariants only — no vt100 / pty harness in this phase. The six INV-22.3-* invariants land as static-grep tests following the existing INV-21.7-14/15 + INV-22.1-04 pattern (regex over `main.rs` / `handlers.rs` / `repl_input.rs` / `subagent_runner.rs` source). Visual stability is proven by the live-TTY UAT re-run (D-04). **Why:** Matches Phase 21 D-18 (no new crate deps). vt100/expectrl would catch a different class of bugs but adds dev-deps, conflicts with the no-new-deps rule, and the user explicitly chose the simpler path. If post-UAT regression demand surfaces, the vt100 harness can be added as a follow-up phase under `--features tty-snapshot`.

### UAT re-verification scenario

- **D-04:** Re-run the exact 3-concurrent-subagent LoRA-research scenario captured in `22.3-UAT-EVIDENCE.md`. Apples-to-apples comparison against the recorded baseline. Acceptance: each of D-1..D-7 (minus D-5) no longer reproduces. The transcript belongs in `22.3-HUMAN-UAT.md`. **Why:** Structured baseline already exists; deviating loses the regression signal. Operator may add ad-hoc steps but the scripted baseline is the pass/fail gate.

### Component implementation sites (locked from UI-SPEC + scout)

- **D-05:** New `CommandResult::ResetTerminal` variant added to BOTH enums:
  - `crates/ironhermes-core/src/commands/mod.rs:131` (the canonical enum)
  - `crates/ironhermes-cli/src/tui/extension.rs:114` (the TUI re-export)
  - Mapper at `crates/ironhermes-cli/src/tui/commands.rs:112-125` adds `CoreCommandResult::ResetTerminal => CommandResult::ResetTerminal` arm
  - **Why:** Two-enum reality from Phase 21.1 / 22.1; the spec correctly forbids reusing `ClearSession` because `main.rs:1151` calls `messages.truncate(1)` on `ClearSession` — exactly what `/clear` MUST NOT do.

- **D-06:** `cmd_clear` at `crates/ironhermes-core/src/commands/handlers.rs:79` changes return value from `CommandResult::ClearSession` to `CommandResult::ResetTerminal`. `cmd_new` at handlers.rs:70 keeps returning `ClearSession` (current truncate semantics preserved for `/new`). This is the structural enforcement of UI-SPEC's "/clear ≠ /new" disambiguation.

- **D-07:** TranscriptWriter touch-on-register lives in `crates/ironhermes-agent/src/subagent_runner.rs` around line 150-154 (where `TranscriptWriter::open(path)` is currently lazy). The touch is a no-op `OpenOptions::new().create(true).append(true).open(&path)` immediately after `register(subagent_id)` in the registry, before any worker spawn. INV-22.3-05 verifies via a unit test that calls register and asserts `std::fs::metadata(&path).is_ok()`.

- **D-08:** History activation lives in `crates/ironhermes-cli/src/repl_input.rs:184` (`history_path: Option<PathBuf>` parameter, currently dropped at line 244 with `let _ = history_path;`). Replace the drop with `rl.load_history(&path)` on init and `rl.save_history(&path)` on `Command::Shutdown`. `rl.set_max_history_size(1000)` and `rl.set_history_duplicates(HistoryDuplicates::Prev)` also fire on init.

- **D-09:** `repl_input.add_history(&input)` for slash commands fires at the prompt-time slash-dispatch site in `run_chat` (main.rs around the `if input.starts_with('/')` check, before the `continue` at the end of the slash dispatch match). The existing call at main.rs:1213 (chat prompts) stays unchanged. INV-22.3-04 grep verifies the new call appears AFTER the slash check.

- **D-10:** Levenshtein helper `suggest_typo(input: &str, candidates: &[&str]) -> Vec<&'static str>` lives as a new pure function (likely in `crates/ironhermes-core/src/commands/` or a new `commands/typo.rs`). ~20 lines of standard O(n*m) DP matrix. No new crate. Called from `cmd_agents`'s `Some(other)` arm and from the unknown-top-level-command fallback in the CommandRouter (or in the relevant `todo_stub` handler — researcher confirms exact site).

- **D-11:** New helper `reset_terminal_visual(reserved: u16)` lives next to the existing render machinery (likely `crates/ironhermes-cli/src/tui/render.rs` — same module as `prompt_position_ansi`). Implementation per UI-SPEC `ResetCommand` example: stderr `\x1b[3J\x1b[H\x1b[2J` → re-emit DECSTBM `\x1b[1;{scroll_end}r` → re-anchor prompt via `prompt_position_ansi`. Called from the `CommandResult::ResetTerminal` arm in `run_chat`.

### Banner-once enforcement

- **D-12:** INV-22.3-02 implementation: static-grep ensures `print_banner()` appears only at the established call site (main.rs:673) and that this site occurs BEFORE any `TuiHandle::new_with_extensions` call in `run_chat`. The grep is in addition to whatever runtime fix the D-02 probe identifies. **Note:** Until the D-02 probe completes, the planner cannot lock the runtime fix — only the structural invariant.

### Scope of "PaintCoordinator"

- **D-13:** PaintCoordinator is NOT a new struct (UI-SPEC explicit). It is the combination of (a) the existing `TuiHandle.readline_active: Arc<AtomicBool>` barrier from Plan 21.7-12, (b) the existing `ExternalPrinterHandle`, and (c) the documented OWN-1..5 discipline rules. No new Rust type is introduced. Planner enforces via existing invariants + the new INV-22.3-* gates.

### Mid-turn vs prompt-time slash dispatch

- **D-14:** Mid-turn slash dispatch (the in-turn `select!` arm in `run_chat`) does NOT call `add_history`. INV-22.3-06 grep enforces this — only the prompt-time branch records history entries. **Why:** UI-SPEC HIST-8; mid-turn lines come from `recv_line()` already mid-flight and the user has already had a chance to commit them via the prompt-time path.

### Three-site sync

- **D-15:** Per existing Phase 22 / 22.1 / 21.7 pattern, any wiring change to `run_chat` that needs symmetry must also touch `run_single` (and `run_gateway` if relevant). For 22.3 specifically: `run_single` and `run_gateway` do NOT host an interactive REPL, so history activation, slash add_history, and ResetTerminal handling are `run_chat`-only. The three-site invariant does NOT apply here, but the planner must explicitly justify the `run_chat`-only scope in the relevant plan SUMMARY.

### Claude's Discretion

- Exact placement of `reset_terminal_visual` helper (render.rs vs a new file) — planner picks based on import graph
- Whether `suggest_typo` lives in `commands/typo.rs` or inline in `handlers.rs` — planner picks based on testability
- Exact regex anchors for the six INV-22.3-* static-grep tests — follow established INV-21.7-14/15 + INV-22.1-04 patterns
- Whether to use the optional `print_slash_block` helper macro or inline the header/body/blank pattern in each handler — UI-SPEC SL-2 leaves this to the planner
- Exact text of the BANNER-PROBE findings note format — researcher's call
