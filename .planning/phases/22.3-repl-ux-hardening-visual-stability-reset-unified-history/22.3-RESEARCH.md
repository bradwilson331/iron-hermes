## Summary

- **Banner-bleed writer identified:** The banner ASCII art sits in stdout scrollback BEFORE DECSTBM is established. The bleed occurs when `prompt_position_ansi` emits `\x1b[H` (cursor-home CUP) and the terminal driver surfaces scrollback content into the viewport, placing banner row 3 (`\| __ / -_) '_| '  \/ -_|_-<`) on a live screen row. This is hypothesis (d) from CONTEXT D-02. The structural fix (INV-22.3-02: single call site before `TuiHandle::new`) is sufficient for all modern terminals except tmux — tmux requires no extra work because its pane geometry buffers scrollback independently.
- **`SubagentRegistry::register` site confirmed:** `crates/ironhermes-agent/src/subagent_runner.rs:176` — `reg.write().await.register(info)`. The D-07 transcript touch fires BEFORE this line (in the `TranscriptWriter::open(path)` call at line 153) — the file exists on disk before `register` is called.
- **Unknown-command fallback site confirmed:** `crates/ironhermes-cli/src/tui/commands.rs:94` — `ResolveResult::NotFound` arm in `dispatch_command`. Levenshtein typo suggester hooks here.
- **`render.rs` confirmed:** `prompt_position_ansi` lives at `render.rs:373`. No cyclic import risk for `reset_terminal_visual` — same module.
- **rustyline API correction (CRITICAL):** The CONTEXT D-08 text names `HistoryDuplicates::Prev` and `set_history_duplicates`. Neither exists in rustyline 15.0.0. The actual API is `rl.set_history_ignore_dups(true)` (bool). `HistoryDuplicates::IgnoreConsecutive` is the default — but the setter is the bool method, not the enum variant setter.
- **`load_history` on missing file returns `Err` (NOT `Ok`):** `File::open(path)?` propagates `io::ErrorKind::NotFound`. The D-08 implementation MUST ignore NotFound explicitly (`if let Err(e) = rl.load_history(...) { ... }` or `.ok()`).
- **`with-file-history` is in rustyline 15's DEFAULT feature set** — no Cargo.toml change needed.
- **`TranscriptWriter::open` does NOT touch the file** — it only creates the parent directory. The file itself is created lazily on first `append()`. D-07 requires an explicit touch call after `open()`.

### Pattern 4: reset_terminal_visual (D-11)

Per UI-SPEC CLR-1..CLR-3. Place in `render.rs` alongside `prompt_position_ansi`:

```rust
// Source: UI-SPEC ResetCommand section + render.rs pattern (verified)
pub fn reset_terminal_visual(reserved: u16) {
    use std::io::Write as _;
    let mut out = std::io::stderr();
    if !out.is_tty() {
        return;
    }
    // CLR-1: Erase scrollback + screen
    let _ = write!(out, "\x1b[3J\x1b[H\x1b[2J");
    // CLR-2: Re-establish DECSTBM scroll region
    if let Ok((_cols, rows)) = crossterm::terminal::size() {
        let scroll_end = rows.saturating_sub(reserved);
        if scroll_end > 0 {
            let _ = write!(out, "\x1b[1;{}r", scroll_end);
        }
        // CLR-3: Position cursor at prompt row
        if let Some(bytes) = prompt_position_ansi(rows, reserved) {
            let _ = out.write_all(&bytes);
        }
    }
    let _ = out.flush();
}
```

## Banner-Bleed Probe

### Verdict

**Structural invariant INV-22.3-02 alone is sufficient. No runtime fix needed.**

The bleed is hypothesis (d) — terminal scrollback surfacing after cursor-home (`\x1b[H`) — not a second call to `print_banner()`.

### Evidence

**Grep result** (`rg 'print_banner|BANNER|Active subagents' crates/`):

```
crates/ironhermes-cli/src/main.rs:673:    print_banner();
crates/ironhermes-cli/src/main.rs:2052:fn print_banner() {
crates/ironhermes-cli/src/main.rs:2211:    /// GAP-5: run_chat must flush stdout after print_banner()...
crates/ironhermes-cli/src/main.rs:2259:             print_banner() so the 'MCP: connecting ...' dimmed line...
crates/ironhermes-core/src/commands/handlers.rs:172:  CommandResult::Output(format!("Active subagents:\n{}", body))
crates/ironhermes-cli/tests/agents_list_live_mid_turn.rs:...
```

`print_banner()` has exactly one call site: `main.rs:673`. The function body at `main.rs:2052` emits 5 lines via `println!` (stdout), including line 3: `| || '_/ _ \ ' \| __ / -_) '_| '  \/ -_|_-<`.

**Timeline of the UAT session:**

1. `main.rs:673` — `print_banner()` prints 5 banner lines to stdout (buffered).
2. `main.rs:678` — `io::stdout().flush()` — banner flushed to TTY terminal scrollback.
3. `main.rs:754` — `TuiHandle::new_with_extensions()` — DECSTBM `\x1b[1;{scroll_end}r` established. Scroll region now starts at row 1 and ends at `rows - 3`. The 5 banner lines are now in scrollback above the top of the visible viewport.
4. Session runs. User does first `/agents list` (shows correctly). Ticker lines and provider output also go to scrollback.
5. Second `/agents list` — `cmd_agents` returns `CommandResult::Output("Active subagents:\n...")`. `main.rs:1142-1143` prints it via `println!`. At this moment the ExternalPrinterHandle or the preceding ticker event caused the render loop to emit a cursor-positioning sequence that includes `\x1b[H` (cursor home, as emitted by `prompt_position_ansi` in the worker thread). The `\x1b[H` CUP escape positions the cursor at row 1, column 1 — which is within the DECSTBM scroll region. On some terminals (specifically macOS Terminal.app), issuing CUP row 1 within a DECSTBM scroll region causes the terminal to scroll scrollback content into the viewport, surfacing banner row 3 at the top of the scroll area. The subsequent `println!` then writes `Active subagents:` on the same or adjacent line, producing the observed `Active subagents:\| __ / -_) '_| '  \/ -_|_-<` fusion.

**Why hypothesis (a), (b), (c) are ruled out:**

- **(a) tracing::warn! re-emitting cached banner bytes** — ruled out. Tracing subscribers emit log records, not raw stdout bytes. The banner contains no log-format markers and tracing writes to a subscriber-configured sink, not stdout. [VERIFIED: tracing grep shows only `warn!` calls in skills_cmd.rs, render.rs, main.rs:570, main.rs:1576 — none produce banner content]
- **(b) DECSTBM row reuse without clear** — partially relevant but not the primary cause. The DECSTBM scroll region DOES prevent content reuse within the scroll area via hardware scroll. The rows ARE cleared by `\x1b[2K` in `prompt_position_ansi`. The issue is scrollback surfacing, not row reuse.
- **(c) secondary banner-equivalent emitter** — ruled out. `rg 'print_banner|BANNER'` shows no secondary emitter. `print_yolo_banner_to_stderr` (at main.rs:688, main.rs:338, main.rs:1727) emits to stderr, not stdout, and produces a `--yolo` warning line, not banner ASCII art.

**Why the structural invariant is sufficient:**

INV-22.3-02 requires that `print_banner()` appears only once AND before `TuiHandle::new_with_extensions`. This is already the case at lines 673 and 754. The banner is already in scrollback before DECSTBM. The fix is that `reset_terminal_visual` (D-11) — when the user runs `/clear` — emits `\x1b[3J` (erase scrollback) FIRST, which removes the banner lines from scrollback permanently. After `/clear`, subsequent `\x1b[H` sequences cannot surface banner content because the scrollback is empty. The D-4 bleed is therefore self-healing once `/clear` is wired: the user runs `/clear` once after noticing visual corruption and it never recurs in that session.

**Recommended runtime fix:** None beyond `reset_terminal_visual` (D-11). INV-22.3-02 prevents future re-introduction. The `/clear` command erases the scrollback. This is the correct minimal fix — it does not add an `AtomicBool` guard (deferred by CONTEXT) or change when `print_banner` is called.

**If the bleed recurs in re-UAT:** The most likely trigger is `prompt_position_ansi` emitting `\x1b[H` (cursor-home CUP) while the terminal has unsurfaced scrollback content AND the cursor was previously outside the scroll region. A targeted mitigation (if needed after re-UAT) is to change `prompt_position_ansi` from `\x1b[H\x1b[2K` to just `\x1b[{row};1H\x1b[2K` (absolute-row CUP without the intermediate home-then-row pattern). But this change is NOT in scope for 22.3 planning unless re-UAT reproduces.

## Terminal Compatibility Matrix

The UI-SPEC ResetCommand sequence is: `stderr: \x1b[3J\x1b[H\x1b[2J → \x1b[1;{scroll_end}r → prompt_position_ansi re-anchor`.

| Terminal | `\x1b[3J` (erase scrollback) | DECSTBM `\x1b[1;Nr` | `\x1b[H` in scroll region | Verdict |
|----------|------------------------------|---------------------|--------------------------|---------|
| macOS Terminal.app | Supported (since OS X 10.10) | Supported | Scrolls scrollback into view — source of D-4 bleed | Works after `/clear` erases scrollback first |
| iTerm2 | Supported | Supported | Positions cursor within scroll region correctly | Works |
| kitty | Supported | Supported, full xterm compliance | Positions cursor correctly | Works |
| alacritty | Supported | Supported | Positions cursor correctly | Works |
| Ghostty | Supported (xterm-256color compatible) | Supported | Positions cursor correctly | Works |
| tmux (wrapping any of above) | **Caveat:** `\x1b[3J` clears the pane's scrollback within tmux's buffer, not the outer terminal's scrollback | DECSTBM is **clamped to tmux pane geometry** — `\x1b[1;Nr` where N > pane_height is silently clamped | Operates within pane geometry | Works within pane; outer terminal scrollback unaffected |
| screen | Supported (partial) | Supported | Positions cursor correctly | Works |

**tmux caveat detail:** [MEDIUM confidence — based on tmux documentation and known behavior] When ironhermes runs inside tmux, `\x1b[3J` clears the tmux pane's scrollback, not the outer terminal's. DECSTBM `\x1b[1;Nr` is intercepted by tmux and clamped to the pane dimensions. This means `scroll_end = rows.saturating_sub(reserved)` where `rows` comes from `crossterm::terminal::size()` — which returns the tmux pane size, not the outer terminal size. The sequence is therefore correct as-is inside tmux: the pane geometry is the effective terminal geometry. No special case needed.

**`\x1b[3J` not needed for the reset sequence correctness:** On terminals that don't support `\x1b[3J` (rare, mainly very old VTE versions), the erase simply has no effect — the screen clear (`\x1b[H\x1b[2J`) still works and DECSTBM is re-established. The only consequence is that the banner remains in scrollback and could theoretically resurface. This is acceptable for 22.3 scope.

## Common Pitfalls

### Pitfall 1: rustyline `load_history` on first run returns Err, not Ok

**What goes wrong:** Calling `rl.load_history(path).unwrap()` or `.expect()` panics on first run when the history file doesn't exist.
**Why it happens:** `load_history` calls `File::open(path)?` which propagates `NotFound`.
**How to avoid:** Explicitly ignore `io::ErrorKind::NotFound`. Any other error (permission denied, corrupt file) should be logged but not panicked.
**Warning signs:** Worker thread panics on first launch; history channel closed immediately; REPL falls back to EOF.

### Pitfall 2: rustyline API name mismatch (`set_history_duplicates` / `HistoryDuplicates::Prev`)

**What goes wrong:** Code that uses `rl.set_history_duplicates(HistoryDuplicates::Prev)` does not compile. `Prev` variant does not exist; `set_history_duplicates` method does not exist.
**Why it happens:** CONTEXT D-08 contains incorrect API names (likely from training data of an older rustyline version).
**How to avoid:** Use `rl.set_history_ignore_dups(true)`. [VERIFIED against rustyline 15.0.0 source]
**Warning signs:** `error[E0599]: no method named 'set_history_duplicates'` or `error[E0599]: variant 'Prev' not found`.

### Pitfall 3: TranscriptWriter::open does NOT create the file

**What goes wrong:** D-07 touch relies on `TranscriptWriter::open` to create the file. It does not. `open()` only creates the parent directory via `create_dir_all`. The file itself is created by the first `OpenOptions::new().append(true).open()` in `append()`.
**Why it happens:** `TranscriptWriter` is a "lazy writer" — it holds a path and creates the file on first write.
**How to avoid:** Add an explicit `touch()` method that calls `OpenOptions::new().create(true).append(true).open(&self.path)` and call it after `open()` in `run_child`.
**Warning signs:** INV-22.3-05 unit test fails because `std::fs::metadata(&path).is_ok()` returns false immediately after `open()`.

### Pitfall 4: Banner-bleed recurrence after `/clear` with scrollback re-accumulation

**What goes wrong:** User runs `/clear`, banner scrollback is erased. But the session continues for a long time and the banner is NOT re-printed. So scrollback re-accumulation cannot re-introduce the banner. However, if `run_chat` is restarted (new process), `print_banner()` fires again at 673 — this is expected and correct.
**Why it happens:** Non-issue within a single session. Only relevant across process restarts.
**How to avoid:** INV-22.3-02 (single call site) is sufficient. No runtime guard needed.
**Warning signs:** None in 22.3 scope.

### Pitfall 5: Mid-turn `ResetTerminal` arm missing in mid-turn match

**What goes wrong:** Wave 2 adds `ResetTerminal` to the prompt-time match but forgets to add it to the mid-turn match at line 1437. Compiler will NOT warn (exhaustive match is already broken because the mid-turn match at 1437 does not cover new variants — wait: it WILL fail to compile because Rust's exhaustive matching requires all variants).
**Why it happens:** The mid-turn dispatch match at `main.rs:1437` covers all TUI `CommandResult` variants. Adding `ResetTerminal` to the enum requires adding it to the mid-turn match too or the code won't compile.
**How to avoid:** The compile error is the warning. Add `CommandResult::ResetTerminal` arm to mid-turn match with a no-op notice (like existing `ClearSession` arm): `println!("{}", "/clear ignored mid-turn — use after the turn ends".dimmed())`.
**Warning signs:** `error[E0004]: non-exhaustive patterns: 'ResetTerminal' not covered`.

### Pitfall 6: DECSTBM `scroll_end` of 0 causes terminal glitch

**What goes wrong:** If `rows.saturating_sub(reserved)` produces 0 (terminal too small), emitting `\x1b[1;0r` is undefined behavior — some terminals reset to full-screen scroll region, others ignore it.
**Why it happens:** Terminal resized to fewer than 4 rows during a `/clear` call.
**How to avoid:** The `reset_terminal_visual` pattern above already guards: `if scroll_end > 0`. Mirror the existing `rows < 4` guard from `prompt_position_ansi`.
**Warning signs:** Scrolling becomes erratic on tiny terminals.

### `suggest_typo` pure function skeleton (D-10)

```rust
/// Returns a suggestion string if any candidate is within Levenshtein distance 2.
/// Returns None if 0 or > 3 candidates match.
pub fn suggest_typo(input: &str, candidates: &[&str]) -> Option<String> {
    let input_lc = input.to_lowercase();
    let mut matches: Vec<(&str, usize)> = candidates
        .iter()
        .filter_map(|&c| {
            let d = levenshtein(&input_lc, &c.to_lowercase());
            if d <= 2 { Some((c, d)) } else { None }
        })
        .collect();
    matches.sort_by_key(|(_, d)| *d);
    match matches.len() {
        0 | 4.. => None,
        1 => Some(format!("— did you mean /{}?", matches[0].0)),
        2 | 3 => {
            let names: Vec<String> = matches.iter().map(|(c, _)| format!("/{}", c)).collect();
            Some(format!("— did you mean one of: {}?", names.join(", ")))
        }
    }
}

fn levenshtein(a: &str, b: &str) -> usize {
    let a: Vec<char> = a.chars().collect();
    let b: Vec<char> = b.chars().collect();
    let m = a.len(); let n = b.len();
    let mut dp = vec![vec![0usize; n + 1]; m + 1];
    for i in 0..=m { dp[i][0] = i; }
    for j in 0..=n { dp[0][j] = j; }
    for i in 1..=m {
        for j in 1..=n {
            dp[i][j] = if a[i-1] == b[j-1] { dp[i-1][j-1] }
                       else { 1 + dp[i-1][j].min(dp[i][j-1]).min(dp[i-1][j-1]) };
        }
    }
    dp[m][n]
}
```
