# Phase 22.3: Code Review Report

**Reviewed:** 2026-04-23T00:00:00Z
**Depth:** standard
**Files Reviewed:** 14

## Summary

Phase 22.3 delivers four interlocking changes: (1) `TranscriptWriter::touch()` pre-creates the JSONL file before `SubagentRegistry::register` to close the ENOENT race window, (2) `cmd_clear` now returns `CommandResult::ResetTerminal` (TTY visual reset) rather than `ClearSession`, cleanly disambiguating `/clear` from `/new`, (3) rustyline history is wired with `set_history_ignore_dups` / `load_history` / `save_history` and persisted to `$HERMES_HOME/repl_history`, and (4) slash commands are added to history at the prompt-time dispatch site.

The implementation is largely correct and well-tested. The INV-22.3 invariant tests are a good structural safeguard. No critical security issues were found. The warnings below describe logic errors or behavioral gaps that could cause user-visible regressions; the info items are code quality observations.

## Warnings

### WR-01: `transcript.rs` — `touch()` uses a sync blocking call inside an async context

**File:** `crates/ironhermes-agent/src/transcript.rs:143`
**Issue:** `TranscriptWriter::touch()` calls `std::fs::OpenOptions::new().open(...)` synchronously. The doc comment acknowledges this and argues "a single short syscall; safe inside async contexts because it does not block on user IO." That argument is sound under normal filesystem conditions, but it does not hold under slow or remote filesystems (NFS, SSHFS, Docker overlay), where a single `open()` for file creation can block the calling tokio thread for tens of milliseconds — stalling the entire executor thread. `touch()` is called from within `AgentSubagentRunner::run_child`, which is an async context managed by the tokio multi-thread scheduler. The phase constraint bars vt100/pty harness (D-03) and new crates (D-18), but the fix here requires neither: `tokio::fs::OpenOptions` is already used in `append()` in the same file.
**Fix:**
```rust
// Replace the sync touch() body with an async version, and make touch() async:
pub async fn touch(&self) {
    let _ = tokio::fs::OpenOptions::new()
        .create(true)
        .append(true)
        .open(&self.path)
        .await;
}
```
Then in `subagent_runner.rs` at the call site, `writer.touch().await`. The test `transcript_touch.rs` uses `#[test]` (sync), not `#[tokio::test]`, so the test would also need updating — but given D-03 prohibits a vt100/pty harness, the existing sync test can be kept as-is if `touch()` is kept sync and only the async wrapper is added. Alternatively, if the team accepts the current sync behavior as a documented trade-off, add a `#[allow(clippy::blocking_operation_on_async_context)]` annotation with an explicit comment explaining the bounded latency assumption.

### WR-02: `render.rs` — `reset_terminal_visual` writes to stderr before flushing stdout

**File:** `crates/ironhermes-cli/src/tui/render.rs:407-426`
**Issue:** `reset_terminal_visual` writes `\x1b[3J\x1b[H\x1b[2J` and the DECSTBM sequence to stderr, then flushes stderr. It does **not** flush stdout first. In `run_agent_turn`, the streaming callback writes tokens to stdout via `print!("{}", delta)` and calls `io::stdout().flush().ok()`. However, in the window between the last token and the user's `/clear`, unflushed stdout content could still be in the userspace buffer. If the user issues `/clear` during that window, the scrollback-erase on stderr fires before the pending stdout tokens are drained, resulting in those tokens being written to the freshly-cleared screen.
**Fix:**
```rust
pub fn reset_terminal_visual(reserved: u16) {
    use std::io::Write as _;
    // Flush stdout first so buffered streaming tokens land before the clear.
    let _ = std::io::stdout().flush();
    let mut out = stderr();
    if !out.is_tty() {
        return;
    }
    // ... rest of function unchanged
```

### WR-03: `repl_input.rs` — `shutdown()` does not await worker thread join; history save may be lost

**File:** `crates/ironhermes-cli/src/repl_input.rs:442-446`
**Issue:** `ReplInputChannel::shutdown(self)` sends `Command::Shutdown` and then drops `self`. Dropping `self` releases the command sender, which is correct. However the `JoinHandle` stored in `worker: Option<std::thread::JoinHandle<()>>` is dropped without being joined. The `Command::Shutdown` match arm in the worker calls `rl.save_history(path)` before `break`-ing. If the OS schedules the worker thread to handle the Shutdown command *after* the `JoinHandle` is dropped and the spawning thread proceeds — which is valid behavior — the history file may not be fully written before the process returns from `run_chat`. In practice, OS threads are not cancelled on `JoinHandle` drop (unlike tasks), so this is a latency concern rather than a data-loss certainty, but it's a correctness gap. In `run_chat`, the REPL exits normally and the OS will eventually flush, but under `std::process::exit(130)` (the emergency escape path) the thread may not complete its save.
**Fix:** Join the worker thread in `shutdown()`:
```rust
pub fn shutdown(self) {
    let _ = self.cmd_tx.send_command_only(Command::Shutdown);
    if let Some(handle) = self.worker {
        // The worker breaks its loop on Shutdown; join ensures save_history completes.
        // unwrap_or_default: if the thread panicked, we cannot recover history anyway.
        let _ = handle.join();
    }
}
```
Note that in the emergency `std::process::exit(130)` path in `main.rs`, there is no `repl_input.shutdown()` call, so history is already unguaranteed there. This fix covers the normal exit path where `run_chat` drops `repl_input` implicitly.

### WR-04: `handler.rs` (gateway) — `CoreCommandResult::ClearSession` arm treats `/clear` as a message history wipe

**File:** `crates/ironhermes-gateway/src/handler.rs:287-299`
**Issue:** The `handle_slash_command` method dispatches via `command_router.resolve()`, which calls `cmd_clear()` in `handlers.rs`. `cmd_clear()` now correctly returns `CoreCommandResult::ResetTerminal`. However, the `CoreCommandResult::ClearSession` arm (lines 287–299) still performs `session.clear()` and sends "History cleared." to the user. This arm is no longer reachable by `/clear` in the current code, but it remains as dead code that could be confusingly re-triggered if another command is ever mapped to `ClearSession`. More concretely, the arm's doc-comment says `// /clear: wipe messages but keep session alive` — this comment is now stale and misleading since `/clear` no longer routes here. Any reader following the comment to understand `/clear` semantics will be misled.
**Fix:** Update the comment (and consider whether `ClearSession` is still needed at all in the gateway, given no command currently emits it):
```rust
CoreCommandResult::ClearSession => {
    // NOTE: No built-in command currently emits ClearSession on the gateway.
    // This arm exists for forward-compatibility if a future command uses it.
    // /clear now returns ResetTerminal (Phase 22.3), which is handled below.
    {
        let mut store = self.session_store.write().await;
        if let Some(session) = store.get_mut(&session_key) {
            session.clear();
        }
    }
    with_rate_limit_retry(|| {
        adapter.send_message(&event.chat_id, "History cleared.", None)
    })
    .await?;
}
```

### WR-05: `main.rs` — `readline_barrier` is not cleared on `request_prompt` error

**File:** `crates/ironhermes-cli/src/main.rs:1082-1092`
**Issue:** The readline barrier is set to `true` before calling `repl_input.request_prompt(...)`. If `request_prompt` returns `Err` (channel closed), the code clears the barrier (`readline_barrier.store(false, ...)`) and then `break`s out of the REPL loop. That part is correct. However, if `recv_line().await` returns `None` (worker exited unexpectedly, line 1109-1113), the code maps it to `ReadlineError::Eof` and then falls into the `Err(Eof)` match arm at line 1584 — but at that point, `readline_barrier` was already cleared at line 1094 (`readline_barrier.store(false, Ordering::Relaxed)`), so this specific path is safe. There is a subtle ordering concern though: the barrier is stored `false` at line 1094 unconditionally after `recv_line().await`, regardless of whether the line is an error that sends us into a path where the render loop might briefly interfere. This is mostly fine, but worth noting that if `recv_line` returns `None` and is treated as `Eof`, the barrier should arguably remain `false` (which it does). No real bug here, but the implicit assumption that the barrier-off store at 1094 always fires before the `match` at line 1115 is correct because there is no `await` between lines 1094 and 1115. The code is safe as written — marking this warning for the edge case at the `request_prompt` Err path where the break fires before the outer REPL cleanup:

The real issue: in the emergency-exit path `std::process::exit(130)` on line 1295 and 1359, `readline_barrier` is left `true` if the exit fires while `readline_barrier` is set. This does not matter because the process exits, but the `tui.cleanup_on_exit()` call at those sites does attempt to write to stderr. The render loop is still alive at that moment (cleanup_on_exit cancels it, but there is a window). The barrier prevents the status-row write, which is actually the desired behavior during readline — so this is consistent. No fix needed, but the interaction should be documented.

**Fix (documentation only):** Add a comment near the emergency-exit `std::process::exit(130)` call:
```rust
// readline_barrier may still be true here; that is intentional — it prevents
// the render loop's status-row write from racing cleanup_on_exit's stderr writes.
tui.cleanup_on_exit();
std::process::exit(130);
```

### IN-04: `handlers.rs` — `truncate_ellipsis` checks `s.len()` (bytes) against `max` (chars)

**File:** `crates/ironhermes-core/src/commands/handlers.rs:277-285`
**Issue:** `truncate_ellipsis(s, max)` guards with `if s.len() <= max` where `s.len()` returns the byte count, not the char count. For ASCII-only input (task summaries generated from user prompts), this is correct. But if a task summary contains multi-byte UTF-8 characters, `s.len()` (bytes) may exceed `max` (chars) even when the string is short enough in char-count terms, causing unnecessary truncation. The function then correctly walks to a char boundary, so it will not panic — but the guard condition is semantically wrong.
**Fix:**
```rust
fn truncate_ellipsis(s: &str, max: usize) -> String {
    if s.chars().count() <= max {   // use char count, not byte count
        return s.to_string();
    }
    let mut idx = 0;
    let mut count = 0;
    for (i, _) in s.char_indices() {
        if count >= max {
            idx = i;
            break;
        }
        count += 1;
        idx = i + s[i..].chars().next().map(|c| c.len_utf8()).unwrap_or(0);
    }
    format!("{}\u{2026}", &s[..idx])
}
```
