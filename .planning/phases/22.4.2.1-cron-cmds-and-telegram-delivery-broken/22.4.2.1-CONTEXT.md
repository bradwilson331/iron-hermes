# Phase 22.4.2.1: Cron cmds and telegram delivery broken - Context

**Gathered:** 2026-04-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Restore three v1.1-validated capabilities that have regressed by milestone v2.0 Phase 22.4.2. Three distinct regressions, all confirmed by the user from runtime experience:

1. **`/cron` slash command** returns the stub `"/cron is not yet available. (No cron management UI)"` (`crates/ironhermes-core/src/commands/handlers.rs:1102`) instead of surfacing the cron jobs that the CLI subcommands already manage successfully. CLI subcommands (`hermes cron list/get/status/...`) work correctly — only the in-REPL slash UI is missing.

2. **Cron → Telegram delivery is not happening.** Scheduled jobs fire, the agent runs, `complete_job_run` records `success=true`, the `ResponseSent` hook fires — but the output is **never dispatched to the `deliver` target** (e.g. `telegram:<chat_id>`). `execute_cron_job` (`crates/ironhermes-gateway/src/runner.rs:781-938`) ends after `complete_job_run` with no platform dispatch step. The cron job's `deliver` field set at creation time is silently ignored. v1.1 validated multi-platform delivery; somewhere between v1.1 and now the delivery dispatch was removed or routed away.

3. **Ctrl+C does not gracefully shut down `hermes gateway`** — SIGINT handling in gateway mode is broken (the user has to kill the process forcibly). v1.0 validated "graceful shutdown" for the Telegram gateway.

**In scope (3 fixes):**
- Wire `/cron` slash to surface job list / status / get / pause / resume / run / remove (read-side mirroring the CLI surface; `create` / `edit` interactive flows stay deferred — those need multi-arg parsing the slash framework hasn't standardized yet)
- Restore cron-job → platform delivery in `execute_cron_job` so jobs with `deliver=telegram:<chat_id>` (or `origin`) actually call the gateway's TG send path
- Restore SIGINT graceful shutdown for `hermes gateway`

**Out of scope:**
- Cron CLI subcommands (`hermes cron list/...`) — already work, no fix needed
- Cron LLM-tool `cronjob_tool.rs` — not flagged as broken
- Cron scheduling / parsing / persistence (`ironhermes-cron` crate) — not flagged
- New cron features (interactive `/cron create` from slash, webhook delivery formatting, etc.) — defer
- A full audit of every gateway-side delivery path — targeted regression fixes only
- Persisting toggle state, snapshot frame additions, dispatcher unification — no relation to these three bugs

</domain>

<decisions>
## Implementation Decisions

### Scope shape

- **D-01:** **Three independent regressions, fixed in one phase.** All three are v1.1-validated capabilities that broke at some point during the v2.0 milestone work. Bundled because they share the same "make ironhermes gateway operationally usable again" theme; planner is free to ship them in three separate plans (likely Wave 1 = parallel, since the surfaces are disjoint: core::handlers vs gateway::runner vs gateway main-loop signal handling).

- **D-02:** **Fix targeted regressions, do not refactor.** Each fix should be the smallest change that restores the v1.1 behavior. No opportunistic refactors of the cron crate, the slash dispatcher, or the gateway runner. If the researcher finds a structural problem that explains the regression (e.g. the delivery dispatcher was deleted in a refactor), the fix should reintroduce the dispatcher locally — broader cleanup belongs in a follow-up phase.

- **D-03:** **Each regression gets a regression test.** All three were "working before" — the absence of regression coverage is why they slipped. Mandatory:
  - **`/cron` slash:** static-grep INV asserting the literal `"No cron management UI"` substring is ABSENT from the handlers.rs `todo_stub` match (or the entry is gone), plus a behavioral test that `dispatch(&cron_def, ...)` returns a `CommandResult::Output` containing job-list-shaped text when JobStore has 0 / 1 / N jobs.
  - **Cron → Telegram delivery:** behavioral test (gateway crate) that `execute_cron_job` with a job whose `deliver = "telegram:<chat_id>"` invokes the Telegram client's `send_message` path with the job's output. Use a fake Telegram client (mock at the trait/Arc<dyn> seam, NOT a wiremock HTTP fake — the regression is the missing call site, not the HTTP wiring).
  - **Ctrl+C gateway shutdown:** smoke test that spawns `hermes gateway` (or the gateway runner equivalent), sends SIGINT, asserts the process exits cleanly within ~3 seconds. Pattern source: Phase 21.2 Plan 11's `shutdown_all_returns_within_timeout_when_stdio_child_blocks` test (proves shutdown completes in 2.51s vs 5s outer bound).

### `/cron` slash command UI

- **D-04:** **Read-side only, mirror CLI subcommand surface.** The slash UI exposes the read-side cron operations available today via `hermes cron`: `list` (default no-arg call), `status`, `get <id>`, `pause <id>`, `resume <id>`, `run <id>`, `remove <id>`. The write-side interactive flows (`create`, `edit` with their multi-flag arg shape) stay as their CLI subcommand selves — no slash equivalent in this phase. Slash framework standardizes single-line `/cmd <space-split-args>` parsing; the multi-`--flag` shape of `cron create` doesn't fit cleanly without ad-hoc parsing.

- **D-05:** **Wire via `CommandContext.cron_store: Option<Arc<JobStore>>`** following the Phase 22.4.2 D-04 pattern. Mirrors `mcp_manager`, `memory_manager`, `state_store`, `provider_resolver`, etc. — additive, `Option<Arc<...>>` defaulted to None, `with_cron_store(...)` builder. Existing `CommandContext::new()` constructor stays unchanged. Both the gateway and the tui_rata REPL get the wire-up; the classic-tui REPL gains the same handler body for free via `core::handlers::dispatch`.

  **NOTE:** `JobStore` is currently behind `Arc<Mutex<JobStore>>` in the gateway runner (per runner.rs:783). The CommandContext field uses the same type to avoid lock-contention divergence: `cron_store: Option<Arc<Mutex<ironhermes_cron::JobStore>>>`. Researcher confirms whether this cross-crate type leak (ironhermes-core depending on ironhermes-cron's JobStore) introduces a circular dep — if yes, fall back to a `dyn CronJobReader` trait in `ironhermes-core::commands::context` (mirrors the `McpReloader` pattern from Phase 21.2 Plan 03's `core::commands/context.rs` to avoid `core` depending on `ironhermes-mcp`).

- **D-06:** **`cmd_cron` body returns formatted text matching the CLI rendering.** Reuse the existing CLI helper if exposed (e.g. `cron::render_job_list`, `cron::render_job_details` at `crates/ironhermes-cli/src/cron.rs:270`). If those helpers are private to the CLI binary, lift the pure-text formatting into `ironhermes-cron::format` (or a new `ironhermes-cron::display` module) so both the CLI and the slash handler share the same renderer. Avoid duplicating render logic in two places.

### Cron → platform delivery

- **D-07:** **Restore the delivery dispatch step inside `execute_cron_job`.** The fix lives at `crates/ironhermes-gateway/src/runner.rs:781-938`. After `complete_job_run` succeeds, dispatch the agent's `output` to `job.deliver` per the existing taxonomy:
  - `local` → no-op (delivery is the success log + JSONL)
  - `origin` → resolve the origin's platform + chat_id (already extracted at runner.rs:803-807) and dispatch via the gateway's existing send path
  - `telegram:<chat_id>` (or `platform:chat_id` general form) → dispatch via gateway's TG client (or whichever platform's client matches)
  - `webhook:<url>` → POST output (out of scope this phase if the webhook code path is also missing — flag and split into a follow-up)

  The TG dispatch path uses the gateway's existing `TgClient::send_message` (`telegram.rs:106`) — no new send code, just a missing call site. Researcher locates the `Arc<TgClient>` (or its trait sibling) the gateway runner already holds, and threads it into `execute_cron_job` via `GatewayRunner` if it isn't already passed.

- **D-08:** **Hook ordering preserved.** `MessageReceived` fires BEFORE the agent run (already correct, runner.rs:813-822). `ResponseSent` fires AFTER the agent run AND BEFORE the platform dispatch (current location at runner.rs:894-903 is correct — observability still captures the response even if delivery itself fails). Add a `MessageDelivered` fire-and-forget hook (or reuse `ResponseSent` with a `delivered: bool` field — researcher checks the existing HookEventKind variants) so future regressions of the dispatch step are observable in JSONL.

- **D-09:** **Delivery failure is non-fatal.** A failed TG send (network, rate-limit, invalid chat_id) does NOT mark the job as failed — `complete_job_run(success=true)` already happened. The dispatcher logs the error via `tracing::error!` and exits the function with `Ok(())`. Mirrors Phase 22 D-12's "single failing job does not panic the tick task" principle. If the user wants delivery failures to mark the job for retry, that's a future feature.

### Ctrl+C / SIGINT in gateway mode

- **D-10:** **Restore SIGINT graceful shutdown for `hermes gateway`.** Researcher locates the gateway main loop (likely `crates/ironhermes-gateway/src/runner.rs::GatewayRunner::start` per Phase 21.2 Plan 11 references) and inventories the SIGINT handler. Likely root causes (researcher confirms one):
  - tokio `signal::ctrl_c()` future is awaited, but the cancellation token never reaches the long-poll TG `get_updates` loop
  - signal handler is not installed in gateway mode (only in CLI mode via `rustyline` ctrl-c, which doesn't apply to gateway)
  - shutdown_all is awaited but the outer `JoinSet` never drains because a child task is detached without abort
  - per-process `SIGINT` is propagated to subprocesses (MCP stdio children per Phase 21.2 Plan 11) but the parent's own loop never observes the signal
  
  Fix is whichever of the above the researcher confirms; the test is `D-03` smoke test. Likely a small change (re-attach a `tokio::select!` arm for `tokio::signal::ctrl_c()` in the gateway runner's select loop, or thread a `CancellationToken` through that wasn't threaded before).

- **D-11:** **Shutdown timeout matches Phase 21.2 Plan 11's GAP-8 fix.** ~2 seconds for child processes (MCP stdio reaper via `kill_on_drop(true)` + 2s `JoinHandle` timeout); add a similar outer cap for the gateway's main loop. If the loop doesn't observe shutdown within ~5 seconds, force-exit with a log line. Don't hang waiting for a stuck `get_updates` long poll.
