# Phase 22.4.2.1: Cron cmds and Telegram delivery broken ŌĆö Research

**Researched:** 2026-04-26
**Domain:** Rust gateway runner, slash command dispatch, cron delivery routing
**Confidence:** HIGH ŌĆö all findings verified against live codebase and git history

---

## 1. Overview

Three v1.1-validated capabilities have regressed during v2.0 milestone work. All three are
*targeted regression fixes* ŌĆö the infrastructure is present, but call sites are wrong or missing:

1. **`/cron` slash UI** ŌĆö the `"cron"` arm in `todo_stub` (handlers.rs:1102) routes to
   `"/cron is not yet available. (No cron management UI)"`. The full `JobStore` API, CLI rendering
   helpers, and the sub-dispatch pattern (`cmd_agents`, handlers.rs:164ŌĆō286) all exist and are
   reusable. Fix = replace the `todo_stub` arm with a real `cmd_cron` function, add `cron_store`
   field to `CommandContext`, lift private CLI rendering helpers to `ironhermes-cron::display`, and
   wire the store at both `build_command_context` and the gateway `CommandContext` site.

2. **Cron ŌåÆ Telegram delivery not happening** ŌĆö `complete_job_run` in tick.rs ALREADY returns
   `Result<Option<DeliveryTarget>>`. The regression is at runner.rs:906 where the call site is
   `if let Err(e) = ironhermes_cron::complete_job_run(...)` ŌĆö this matches only the `Err` branch
   and silently drops the `Ok(Some(DeliveryTarget))` value. The dispatch step was never removed; it
   was *never written at the call site that matters*. The `execute_cron_job` function was introduced
   in commit `56f0ac5` (07.3-01) without consuming the `DeliveryTarget` return value, leaving a
   systematic dead code path. Fix = change `if let Err` to `match`, dispatch the `Some(target)` to
   `TelegramAdapter::send_message`, and thread `Arc<TelegramAdapter>` into `execute_cron_job`.

3. **Ctrl+C gateway shutdown broken** ŌĆö `GatewayRunner::start` (runner.rs:640ŌĆō650) has a
   `tokio::select!` block that already includes a `tokio::signal::ctrl_c()` arm AND a
   `self.cancel.cancelled()` arm. The SIGINT arm IS installed and fires correctly. The regression
   is that after the select fires, `self.cancel.cancel()` is called at line 668 ŌĆö but the
   per-chat worker tasks spawned at line 458 are detached (`tokio::spawn`, not added to `join_set`)
   and do not exit promptly because they are blocked on `sem_task.acquire().await` waiting for a
   semaphore permit on an in-flight agent turn. These detached tasks keep the tokio runtime alive.
   The `join_set.join_next()` drain at line 674 only covers poll-loop, WAL, and session-cleanup
   tasks ŌĆö NOT the per-chat workers. Fix = replace `tokio::spawn` at line 458 with
   `join_set.spawn` (requires moving `join_set` ownership into the dispatch closure), or abort all
   detached tasks via a secondary `JoinSet` that IS drained, or add `cancel_task.is_cancelled()`
   checks plus bounded semaphore acquisition with `tokio::time::timeout`. The `CancellationToken`
   field `self.cancel` is already in scope; the fix is mechanical.

**Primary recommendation:** All three fixes are small and targeted. The planner should schedule
them as three parallel Wave-1 plans with disjoint file surfaces. The largest engineering question
is Plan 03's worker-drain strategy ŌĆö recommend the secondary-JoinSet approach described below.

## 2. Regression Archaeology

### Regression 1: `/cron` stub (Plan 01)

The stub string `"No cron management UI"` was introduced in commit `13b7bc8` (the context capture
commit) ŌĆö meaning it was already present in the codebase before Phase 22.4.2. The stub dates to
Phase 21.1 (slash command framework introduction) when `cmd_cron` did not yet exist. It was never
graduated out of `todo_stub` because the slash framework and the `CommandContext` extension pattern
were not fully established until Phase 22.4.2. There is no single "regression commit" ŌĆö this is an
*omission* from Phase 21.1 that has persisted.

**Root cause:** `todo_stub` never had a `cmd_cron` body written. The CLI cron subcommands (Phase
05-03, commit `ef014c9`) work, but the slash path was a separate integration left incomplete.

### Regression 2: Cron ŌåÆ Telegram delivery (Plan 02)

**Git archaeology findings:**

Commit `ef014c9` (2026-04-08, `feat(05-03): delivery routing + tick runner + gateway integration`)
introduced `delivery.rs` with `resolve_delivery_target()` and added the 60s tick task to
`GatewayRunner::start`. The tick task used a placeholder `complete_job_run` call that did not
attempt delivery. This was explicitly noted as pending ("Full agent execution requires AgentLoop
integration").

Commit `56f0ac5` (2026-04-09, `feat(07.3-01): wire real AgentLoop + HookRegistry in cron tick
task`) replaced the placeholder with a real `execute_cron_job` helper. The comment in the diff
reads "D-03 / D-13: real output into complete_job_run; **delivery routing unchanged**". The phrase
"delivery routing unchanged" refers to the placeholder era (no routing), meaning routing was never
wired here.

**`complete_job_run` (tick.rs:94-122) signature (VERIFIED):**

```rust

    store: &Arc<Mutex<JobStore>>,
    job: &CronJob,
    output: &str,
    success: bool,
) -> Result<Option<DeliveryTarget>>
```

It already returns `Ok(Some(DeliveryTarget))` when `job.deliver` is not `"local"` and not silent.
The `DeliveryTarget` struct (delivery.rs) has `platform: String`, `chat_id: String`,
`thread_id: Option<String>`.

**The exact regression (VERIFIED at runner.rs:906-910):**

```rust

    error!("Failed to complete job run: {}", e);
    return Err(e);

```

`if let Err(e) = expr` is equivalent to `let _ = expr.ok(); return ...` ŌĆö the `Ok(Some(...))` is
unreachable from the current match arm. This is a silent dead-code path.

`send_message` was never called from runner.rs in any commit (`git log -p -S "send_message"` on
runner.rs finds zero matches). The dispatch was never present; it was assumed to be implicit in
`complete_job_run`'s return value but that value was never consumed.

### Regression 3: Ctrl+C gateway shutdown (Plan 03)

`git log -p -S "ctrl_c"` shows commit `e767ed9` (2026-04-02, `feat(02-03)`) introduced the
initial `GatewayRunner` with the `tokio::signal::ctrl_c()` select arm. It has been present ever
since ŌĆö the arm exists in the current code (runner.rs:644).

**Root cause (VERIFIED):** The SIGINT arm IS installed. The gateway *does* observe Ctrl+C and
enters the shutdown path. The problem is structural: per-chat worker tasks are spawned with bare
`tokio::spawn` (runner.rs:458) and are NOT added to `join_set`. When `self.cancel.cancel()` fires
(runner.rs:668), these tasks observe `cancel_task.is_cancelled()` only on the *next iteration*
check (runner.rs:491-493). If a task is blocked in `handler_task.handle_with_multimodal(...)` ŌĆö
an agent turn that can take seconds to minutes ŌĆö it will not exit promptly. The tokio runtime
stays alive until all spawned tasks complete, and since agent turns hold open HTTP connections,
the process appears hung to the user.

**The four candidate root causes from CONTEXT.md ŌĆö resolution:**
- "tokio signal::ctrl_c() future is awaited but cancellation token never reaches the long-poll
  get_updates loop" ŌåÆ **FALSE.** The poll loop IS wrapped in a select with `poll_cancel.cancelled()`
  (runner.rs:313-315) and `poll_cancel = self.cancel.clone()` (runner.rs:305).
- "signal handler is not installed in gateway mode" ŌåÆ **FALSE.** runner.rs:644 has the arm.
- "shutdown_all is awaited but outer JoinSet never drains because a child task is detached without
  abort" ŌåÆ **TRUE** ŌĆö this is the root cause. Worker tasks are detached via `tokio::spawn`.
- "per-process SIGINT propagated to subprocesses but parent loop never observes the signal" ŌåÆ
  **FALSE for the cancel signal.** The parent observes it fine; the issue is worker drain.

### Cross-Crate Dependency Analysis (Uncertainty 1 ŌĆö RESOLVED)

**`ironhermes-core` Cargo.toml does NOT list `ironhermes-cron` as a dependency (VERIFIED).**
`ironhermes-cron` depends on `ironhermes-core` (cron/Cargo.toml verified). Adding
`ironhermes-cron::JobStore` directly to `CommandContext` in `ironhermes-core` would create a

(McpReloader trait) and memory (MemoryManagerHandle trait) in earlier phases.

**Resolution (LOCKED):** Define a `CronJobReader` trait in `ironhermes-core::commands::context`
(exactly mirrors `McpReloader` defined at context.rs:28-37). The trait provides the read-side
methods the slash handler needs:

```rust

pub trait CronJobReader: Send + Sync {

    fn list_jobs_text(&self) -> String;
    /// Returns formatted detail text for /cron get <id>.

    /// Returns formatted status text for /cron status.

    /// Pause a job. Returns Ok(name) or Err(message).
    fn pause_job(&self, id_or_name: &str) -> Result<String, String>;
    /// Resume a job. Returns Ok(name) or Err(message).
    fn resume_job(&self, id_or_name: &str) -> Result<String, String>;
    /// Remove a job. Returns Ok(name) or Err(message).
    fn remove_job(&self, id_or_name: &str) -> Result<String, String>;
    /// Queue a manual run. Returns Ok(name) or Err(message).
    fn queue_run(&self, id_or_name: &str) -> Result<String, String>;
```

Builder: `pub fn with_cron_store(mut self, store: Arc<dyn CronJobReader>) -> Self`.

The concrete implementation (`CronJobReaderImpl` wrapping `Arc<Mutex<ironhermes_cron::JobStore>>`)

## 4. Plan 02: Cron ŌåÆ Platform Delivery

### Exact Regression (VERIFIED)

**File:** `crates/ironhermes-gateway/src/runner.rs:906-910`

Current broken code:

`complete_job_run` (tick.rs:94) returns `Result<Option<DeliveryTarget>>`. The `if let Err`
pattern silently drops `Ok(Some(target))`.

**Fix:** Replace with `match`-based dispatch:

```rust
match ironhermes_cron::complete_job_run(job_store, job, &output, true).await {
    Err(e) => {
        error!("Failed to complete job run: {}", e);
        return Err(e);
    }
    Ok(None) => {
        // local delivery or [SILENT] ŌĆö no platform dispatch
    }
    Ok(Some(target)) => {
        // Dispatch to platform
        if target.platform == "telegram" {
            if let Some(ref tg) = tg_client {
                if let Err(e) = tg.send_message(&target.chat_id, &output, target.thread_id.as_deref()).await {
                    error!("Delivery failed for job {} -> telegram:{}: {}", job.name, target.chat_id, e);
                    // D-09: delivery failure is non-fatal ŌĆö complete_job_run already succeeded
                }
            } else {
                warn!("Job {} has deliver=telegram but no TgClient wired", job.name);
            }
        } else {
            warn!("Unsupported delivery platform '{}' for job {} ŌĆö skipping (webhook deferred)", target.platform, job.name);
        }
    }
}
Ok(())
```

The same fix applies symmetrically to the error-path `complete_job_run` call at runner.rs:930-935
(which passes `success=false`). That call should also be `match`-dispatched; on the error path,
the `target` is available but D-09 says delivery failure of an already-failed job is fine ŌĆö either

## 5. Plan 03: Ctrl+C Gateway Shutdown

### Confirmed Root Cause (Uncertainty 7 ŌĆö VERIFIED)

The SIGINT arm IS installed and fires (runner.rs:644). The `CancellationToken` IS propagated to
the poll loop and dispatch loop. The root cause is **detached per-chat worker tasks**.

Per-chat workers are spawned at runner.rs:458:
```rust
tokio::spawn(async move {
    while let Some(queued_msg) = chat_rx.recv().await {
        let permit = match sem_task.acquire().await { ... };
        let result = handler_task.handle_with_multimodal(...).await;
        drop(permit);
        if cancel_task.is_cancelled() { break; }
    }
    queue_task.remove(&chat_id_task).await;
});
```

The `cancel_task.is_cancelled()` check at the bottom of the loop only fires *after* the current
`handle_with_multimodal(...)` call returns. Agent turns can take 10-60 seconds. During that time
the process appears hung. The `join_set.join_next()` drain at line 674 does NOT cover these
detached tasks.

**Fix strategy (LOCKED ŌĆö simplest compliant with D-11):**

Use a secondary `JoinSet<()>` for per-chat workers:

```rust
// In start(), before dispatch_future:
let mut worker_join_set: JoinSet<()> = JoinSet::new();

// Replace tokio::spawn(...) with:
worker_join_set.spawn(async move { ... });

// After the main select! and MCP shutdown, before self.cancel.cancel():
// Abort all running worker tasks with a bounded timeout
let abort_deadline = tokio::time::Instant::now() + std::time::Duration::from_secs(5);
while let Some(_) = tokio::time::timeout_at(abort_deadline, worker_join_set.join_next()).await.ok().flatten() {}
// Any remaining tasks are aborted when worker_join_set is dropped.
```

This gives in-flight agent turns up to 5 seconds to complete naturally before the `JoinSet` is
dropped (which aborts all remaining tasks). Mirrors Phase 21.2 Plan 11 GAP-8's 2s timeout pattern.

**Alternative:** Add a `tokio::time::timeout` around the `handle_with_multimodal` call inside the
worker to bound each turn independently. This is architecturally cleaner but requires changes to
`GatewayMessageHandler` ŌĆö out of scope for a targeted fix.

### CancellationToken reuse (Uncertainty 6 ŌĆö VERIFIED)

`GatewayRunner` has `cancel: CancellationToken` (runner.rs:62). This is the SAME token used by:
- Poll loop: `poll_cancel = self.cancel.clone()` (runner.rs:305)
- WAL task: `wal_cancel = self.cancel.clone()` (runner.rs:505)
- Cleanup task: `cleanup_cancel = self.cancel.clone()` (runner.rs:528)
- Cron tick: `tick_cancel = self.cancel.clone()` (runner.rs:545)
- Dispatch loop: `dispatch_cancel = self.cancel.clone()` (runner.rs:366)

Phase 21.2 Plan 11 (`shutdown_all_returns_within_timeout_when_stdio_child_blocks`) confirmed that
this token drives the MCP shutdown path as well (`mcp_manager.shutdown_all()` is called before
`self.cancel.cancel()` at runner.rs:661-665). **Plan 03 reuses this same token for worker drain.**
No new token needed.

### `shutdown_all` ordering (VERIFIED)

Existing shutdown sequence (runner.rs:652-676):
1. MCP shutdown (runner.rs:661-665) ŌĆö BEFORE cancel propagation
2. `self.cancel.cancel()` (runner.rs:668) ŌĆö propagates to all subtasks
3. `drop(msg_tx)` (runner.rs:671) ŌĆö closes polling channel
4. `join_set.join_next()` drain (runner.rs:674) ŌĆö drains poll+WAL+cleanup+cron tasks

Plan 03 adds step 2.5: drain `worker_join_set` with 5s timeout, between step 2 and step 3. This
preserves the existing ordering invariant (MCP first) locked in Phase 21.2 Plan 11.

### Failure Modes Each Plan Must Defend Against

- **Plan 01:** Silent regeneration of the stub if the `dispatch` match arm is detached from
  `cmd_cron` in a future refactor (e.g., someone re-adds `"cron"` to `todo_stub`). The INV grep
  for "No cron management UI" being absent prevents this.
- **Plan 02:** Silent removal of the dispatch step in a future refactor that reverts the `match`
  back to `if let Err` (exactly what produced this regression). The INV grep for BOTH
  `send_message` and the absence of `if let Err(...complete_job_run` prevents this.
- **Plan 03:** A future `tokio::select!` refactor dropping the `ctrl_c()` arm or moving worker
  spawns back to detached `tokio::spawn`. The INV greps for `worker_join_set` and `ctrl_c()`
  both present prevents this.

### Open Validation Questions

1. **`TgSendApi` trait introduction scope:** If Plan 02 introduces a new trait in
   `ironhermes-gateway::telegram` for testability, this is a new public API surface. Confirm with

   wrap at the `Arc<TelegramAdapter>` level using a constructor override). The concrete struct

   borrowing it from inside the dispatch loop closure requires either `RefCell<JoinSet>` or
