# Phase 22.4.2.1 Plan 02: Cron -> Telegram Delivery Dispatch Summary

**One-liner:** Restore cron job Telegram delivery by replacing the broken `if let Err` pattern with a `match`-consuming `Result<Option<DeliveryTarget>>` and wiring `Arc<dyn TgSendApi>` into `execute_cron_job`.

## What Was Built

The regression was that `complete_job_run` in tick.rs already returns `Result<Option<DeliveryTarget>>` but the call site at runner.rs:906 used `if let Err(e) = ...` — silently dropping `Ok(Some(target))`. The dispatch step was never written. This plan:

1. Adds a `TgSendApi` async trait + `impl TgSendApi for TelegramAdapter` in `gateway/src/telegram.rs` — provides the testable seam so `FakeTgClient` can capture send calls without HTTP.

2. Adds `dispatch_delivery(target, output, tg_client, job_name)` as a `pub` async helper in `runner.rs` — contains the `match` dispatch logic, extracted so integration tests can call it directly without a full agent run.

3. Adds `tg_client: Option<Arc<dyn TgSendApi>>` as the 8th parameter of `execute_cron_job`. The cron tick closure now clones `adapter_tick = adapter.clone()` and passes `Some(adapter_tick.clone() as Arc<dyn TgSendApi>)`.

4. Replaces both broken `if let Err(e) = ironhermes_cron::complete_job_run(...)` calls (success path + error path) with `match` blocks that dispatch `Ok(Some(target))` to `dispatch_delivery`.

5. Adds 3 behavioral tests in `cron_delivery.rs` covering: telegram dispatch fires, local no-op, and TG send failure is non-fatal (D-09).

6. Appends `invariant_22_4_2_1_02_delivery_call_site_present` to `invariants_22_4.rs` (now GREEN).
