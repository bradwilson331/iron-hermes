## Common Pitfalls

### Pitfall 1: Over-broad String Replacement
**What goes wrong:** A naive `s/TOOL-01/HXP-TOOL-01/g` run across the entire IronHermes `.planning/` tree rewrites the existing "Toolset management" TOOL-01..05 entries in REQUIREMENTS.md, PROJECT.md, ROADMAP.md, STATE.md, v2.0-MILESTONE-AUDIT.md, TOOLS-GAPS-RESEARCH.md, and milestones/v1.1-REQUIREMENTS.md.
**Why it happens:** IronHermes already has TOOL-01..05 in active use for a different domain.
**How to avoid:** Apply the HXP- prefix ONLY to the new content being written, not via global find-replace. The 27.1.1-CONTEXT.md verbatim copy is the only existing file that needs in-body rewriting, and it lives in a new file written from scratch.
**Warning signs:** If any grep after the operation finds `HXP-TOOL-01` in `ROADMAP.md` in a context that mentions "toolset management" or "Phase 25", the global-replace trap was hit.

### Pitfall 2: Forgetting the ROADMAP.md Active Scope Table Update
**What goes wrong:** The Active scope summary table at the top of REQUIREMENTS.md (`| Category | Reqs in v2.1 |`) is not updated, leaving a stale count that doesn't include the 16 HXP- reqs.
**Why it happens:** The table is at lines 11–22 of REQUIREMENTS.md, far from where the new section is inserted (line ~184).
**How to avoid:** The planner must include a task that adds the `| Hexapod Integration | HXP-TOOL-01..06, HXP-LOCO-01..05, HXP-NAV-01..04, HXP-DOC-01 |` row to the Active scope table AND adds 16 rows to the Traceability table.

### Pitfall 3: Wrong Depends-on for Sub-phases
**What goes wrong:** 27.1.1 is given `Depends on: Phase 27` (the source says "Nothing — first phase"), or 27.1.2 is given `Depends on: Phase 27.1` instead of `Phase 27.1.1`.
**Why it happens:** The source ROADMAP uses Phase 1/2/3 numbering; renaming requires re-anchoring the dependency chain.
**How to avoid:** Dependency chain is: 27.1.1 → Phase 27.1, 27.1.2 → Phase 27.1.1, 27.1.3 → Phase 27.1.2.

### Pitfall 4: Missing Phase Directory for 27.1.1
**What goes wrong:** The CONTEXT.md file is written without first creating the directory, causing a file-not-found error.
**How to avoid:** Directory creation is a prerequisite task (step 3 in order-of-operations).

### Pitfall 5: Source CONTEXT.md canonical_refs Paths
**What goes wrong:** The source 01-CONTEXT.md `<canonical_refs>` section references `.planning/REQUIREMENTS.md` as a relative path. After copy to IronHermes, this path remains correct (same directory structure) but the reader might be confused.
**How to avoid:** Leave the relative path as-is — it resolves correctly in IronHermes' identical directory layout.

### Pitfall 6: Markdown Fence Interactions
**What goes wrong:** The source 01-CONTEXT.md contains code-style backtick-fenced examples. These don't contain TOOL-NN / LOCO-NN patterns, so rewriting is safe; but a visual check during the write step is recommended.
**Warning signs:** None expected — confirmed by inspection that the 5 ID occurrences are all in prose/list context, not inside fenced blocks.
