# Phase 19.1: Code Review Report

**Reviewed:** 2026-04-15T22:15:00Z
**Depth:** standard
**Files Reviewed:** 25
**Status:** issues_found

## Summary

Phase 19.1 implements the IronHermes Skills Hub: hub source adapters (GitHub, well-known, skills.sh), a 5-step atomic install pipeline, trust resolution, CLI subcommands, and a read-only `hub_search` tool action.

Overall the implementation is well-structured with strong security mitigations. Path traversal protection in tarball extraction is thorough, D-08 trust recomputation is correctly implemented (never frozen in manifest), D-13 is properly enforced (no `hub_install` tool action), and credential handling in auth.rs properly redacts tokens in Debug output.

One critical SSRF bypass was found in the well-known adapter where `tarball_url` from untrusted index.json is fetched without validation. Several warnings address potential panics from string slicing, a blocking call inside an async context, and a missing path-traversal guard in `write_bundle_to_dir`.

## Critical Issues

### CR-01: SSRF via unvalidated `tarball_url` in well-known adapter

**File:** `crates/ironhermes-hub/src/well_known.rs:337-342`
**Issue:** The `fetch` method correctly validates the identifier URL against the SSRF guard (`parse_identifier` rejects loopback/private IPs and enforces HTTPS). However, it then reads `tarball_url` from the remote `index.json` and passes it directly to `fetch_tarball_bundle` without any validation. A malicious well-known server could set `tarball_url` to `http://169.254.169.254/latest/meta-data/` (AWS IMDS) or any internal service URL. The production `http` client has `https_only(true)` which mitigates plain HTTP SSRF, but does not prevent SSRF to internal HTTPS endpoints (e.g., `https://10.0.0.1/internal-api`). The `is_private_host` guard is only applied to the identifier, not to the tarball URL.

**Fix:**
```rust
// In WellKnownSkillSource::fetch, after extracting tarball_url:
let tarball_url = entry.tarball_url.ok_or_else(|| {
    typed(HubErrorKind::Parse, format!("index entry for '{skill_name}' has no tarball_url"))
})?;

// Validate tarball_url against the same SSRF guards
let tarball_parsed = Url::parse(&tarball_url).map_err(|e| {
    typed(HubErrorKind::InvalidIdentifier, format!("invalid tarball_url: {e}"))
})?;
if !self.test_mode {
    if tarball_parsed.scheme() != "https" {
        return Err(typed(HubErrorKind::InvalidIdentifier, "tarball_url must use HTTPS"));
    }
    if let Some(host) = tarball_parsed.host_str() {
        if is_private_host(host) {
            return Err(typed(
                HubErrorKind::InvalidIdentifier,
                format!("tarball_url has private/loopback host (SSRF guard): {host}"),
            ));
        }
    }
}
```

## Warnings

### WR-01: Potential panic from string slice on short hashes

**File:** `crates/ironhermes-cli/src/skills_cmd.rs:244`
**Issue:** The code `&outcome.content_hash[..12]` will panic if `content_hash` is shorter than 12 bytes. While `bundle_content_hash` always produces a 64-char hex SHA-256, a future change to the hash function or an unexpected empty string would cause an unrecoverable panic in the CLI. The same pattern appears at lines 403-404 with `old_hash[..12]` and `new_hash[..12]`.

**Fix:**
```rust
// Replace:
&outcome.content_hash[..12],
// With:
&outcome.content_hash.get(..12).unwrap_or(&outcome.content_hash),
```

### WR-02: `futures::executor::block_on` inside async-callable code path

**File:** `crates/ironhermes-cli/src/skills_cmd.rs:119`
**Issue:** `build_sources` calls `futures::executor::block_on(GitHubAuth::resolve(...))` which creates a blocking executor. If `build_sources` is ever called from within a tokio runtime (it is called from `cmd_install`, `cmd_search`, `cmd_update` which are all `async fn`), this will panic with "Cannot start a runtime from within a runtime" or cause a deadlock. Currently this works because the async functions call `build_sources` synchronously before any `.await`, but the `GitHubAuth::resolve` function spawns a tokio subprocess (`tokio::process::Command`), meaning this is calling tokio from within tokio via a blocking bridge -- a known footgun.

**Fix:**
```rust
// Make build_sources async and await directly:
async fn build_sources(cfg: &Config) -> Vec<Box<dyn HubSource + Send + Sync>> {
    let auth = GitHubAuth::resolve(
        cfg.skills.hub.github_token_env.as_deref(),
    ).await;
    // ... rest unchanged
}
```

### WR-03: `write_bundle_to_dir` does not validate file paths against traversal

**File:** `crates/ironhermes-hub/src/installer.rs:334-343`
**Issue:** While `extract_tarball_prefix` in `tarball.rs` validates paths against traversal before producing `BundleFile` structs, `write_bundle_to_dir` trusts `file.path` without re-validation. If a `SkillBundle` is constructed by any code path that bypasses `extract_tarball_prefix` (e.g., a future adapter), the `dir.join(&file.path)` call could write outside the quarantine directory. Defense-in-depth suggests validating at the write site as well.

**Fix:**
```rust
fn write_bundle_to_dir(dir: &Path, bundle: &SkillBundle) -> Result<(), HubError> {
    for file in &bundle.files {
        // Defense-in-depth: re-validate path even though tarball extraction already checked
        let _ = crate::tarball::validate_bundle_rel_path(&file.path)?;
        let dest = dir.join(&file.path);
        // Verify the resolved dest is still under dir (canonicalization guard)
        if !dest.starts_with(dir) {
            return Err(HubError::Typed {
                kind: HubErrorKind::Parse,
                message: format!("path escapes target directory: {}", file.path),
                suggestion: None,
                retry_after_s: None,
            });
        }
        if let Some(parent) = dest.parent() {
            std::fs::create_dir_all(parent)?;
        }
        std::fs::write(&dest, &file.bytes)?;
    }
    Ok(())
}
```

### WR-04: Trust recomputation in `cmd_list_impl` duplicates logic and may drift

**File:** `crates/ironhermes-cli/src/skills_cmd.rs:457-468`
**Issue:** `cmd_list_impl` manually re-implements trust resolution by checking `e.source == "github"` and splitting the identifier to extract `owner/repo`. This duplicates the logic in `resolve_source()` in `ironhermes-core/src/skills.rs` and `trust_level_for()` in `GitHubSource`. If the trust resolution logic changes (e.g., new source types, different identifier format), this inline version will drift. Additionally, the well-known and skills-sh branches hardcode `"community"` which is correct per D-07 but should use the shared `trust_level_str` helper for consistency.

**Fix:** Extract the trust recomputation into a shared function, or call into the adapter's `trust_level_for` method. At minimum, add a comment noting this must stay in sync with `resolve_source`.

### WR-05: `is_private_host` SSRF guard is incomplete for IPv6

**File:** `crates/ironhermes-hub/src/well_known.rs:214-233`
**Issue:** The `is_private_host` function checks `"::1"` for IPv6 loopback but does not check IPv6 link-local (`fe80::`) or IPv6 unique-local (`fc00::`/`fd00::`) addresses. An attacker could use `https://[fd00::1]/...` to bypass the SSRF guard and reach internal services. The function also does not handle bracket-wrapped IPv6 (`[::1]`) which URLs may contain.

**Fix:**
```rust
fn is_private_host(host: &str) -> bool {
    // Strip brackets from IPv6 addresses in URLs
    let host = host.trim_start_matches('[').trim_end_matches(']');

    if host == "localhost" || host == "127.0.0.1" || host == "::1" {
        return true;
    }
    // IPv6 loopback/link-local/unique-local
    let lower = host.to_lowercase();
    if lower.starts_with("fe80:") || lower.starts_with("fc00:") || lower.starts_with("fd00:") {
        return true;
    }
    // RFC-1918 ranges
    if host.starts_with("10.") || host.starts_with("192.168.") {
        return true;
    }
    if let Some(rest) = host.strip_prefix("172.") {
        if let Some(octet_str) = rest.split('.').next() {
            if let Ok(octet) = octet_str.parse::<u8>() {
                if (16..=31).contains(&octet) {
                    return true;
                }
            }
        }
    }
    false
}
```
