# Phase 19.1: Skills Hub — Research

**Researched:** 2026-04-14
**Domain:** Skills publishing/install hub with trust-gated security scanning (Rust port of hermes-agent `skills_hub.py`)
**Confidence:** HIGH

## Summary

Phase 19.1 adds a Skills Hub to IronHermes — publish is explicitly deferred per D-12, so v1 covers **install / search / update / uninstall / trust-management only**. Three source adapters (GitHub, well-known HTTP, skills.sh) map discovered skills to origin-derived trust labels that gate the Phase 19 D-15 scan enforcement (Community + scan-hit = hard-reject; Builtin/Official/Trusted + scan-hit = WARN-BUT-LOAD). Installs follow a five-step pipeline: fetch → quarantine → scan → atomic move → manifest, directly porting `~/code/hermes-agent/tools/skills_hub.py` (2691 LOC). Surface is CLI-only (`hermes skills …`) with a single read-only `hub_search` agent tool action — the agent helps users discover skills but never mutates filesystem state.

**Primary recommendation:** Build a new `ironhermes-hub` crate containing the `HubSource` trait + three adapter impls + install pipeline. Reuse the existing workspace async stack (reqwest 0.12 / tokio 1 / serde_yaml 0.9 / sha2 0.10 / tempfile 3). Add `flate2 = "1.1"` and `tar = "0.4"` for GitHub tarball extraction (avoid git2/gix — libgit2 build complexity on Windows, no net benefit when Git Trees + tarball APIs cover all needs). Wave 0 must add the `Trusted` variant to `SkillSource` (currently only Builtin/Official/Community) and plumb per-skill trust resolution through `load_with_paths`.

## User Constraints (from CONTEXT.md)

### Locked Decisions

#### Source Adapters
- **D-01:** Port all three reference adapters in v1 — `GitHubSource`, `WellKnownSkillSource` (reads `/.well-known/skills/index.json`), `SkillsShSource` (wraps GitHub via the skills.sh registry). Rationale: users migrating from hermes-agent keep install muscle memory; the adapter trait lets us add more later.
- **D-02:** GitHub default taps mirror hermes-agent exactly: `openai/skills`, `anthropics/skills`, `VoltAgent/awesome-agent-skills`, `garrytan/gstack`. Port as-is to avoid silent drift from reference. User can add extra taps via config.
- **D-03:** GitHub auth precedence: anonymous (public repos) → `HERMES_GITHUB_TOKEN` or `GITHUB_TOKEN` env var → `gh auth token` subprocess fallback. **No GitHub App support** — overkill for the single-operator PROJECT.md target and adds ~80 lines of JWT signing.
- **D-04:** Trusted-repo allowlist lives in `config.yaml` under `hub.trusted_repos: [owner/repo, ...]` with an **empty default**. User adds explicitly via `hermes skills trust add owner/repo` (writes config) or manual edit. No compiled-in allowlist.

#### Trust Assignment
- **D-05:** `SkillSource::Official` maps to skills bundled in the repo's `optional-skills/` directory at compile time (matches REQUIREMENTS SKILL-09). Anything installed via the Hub never becomes Official — Hub installs are at most Trusted (config allowlist) or Community (default).
- **D-06:** Trust is **per-source, never per-skill**. The adapter's `trust_level_for(identifier)` is authoritative. No per-skill overrides in config (avoids forgotten-override rot and attack surface). To elevate a single skill, promote its repo to `hub.trusted_repos`.
- **D-07:** Well-known HTTP adapter always returns Community regardless of origin. Rationale: any HTTPS origin can serve a well-known index; there's no central vetting surface. D-15 scan + Community = hard-reject on injection pattern is the security gate.
- **D-08:** Trust is **recomputed on every registry-load** by re-reading `hub.trusted_repos` from config. Never frozen in an install-time manifest. User can demote a repo and the stricter scan-enforcement tightens on next load without reinstall. Install-time metadata records source + identifier, not trust.

#### Install Lifecycle
- **D-09:** Install layout is **flat by category**: `~/.ironhermes/skills/<category>/<skill-name>/`. A `~/.ironhermes/skills/.hub/` directory holds install manifests (origin identifier, install hash, install timestamp). Matches reference `skills_hub.py` exactly. Trust is computed from config at runtime — never encoded in the directory tree.
- **D-10:** Update strategy is **manual only** via `hermes skills update [name]`. No auto-refresh at startup, no daily background check. Rationale: deterministic, scriptable, no surprise behavior changes between runs; matches reference `skills_sync` manual pattern. No lockfile in v1 — pinning can be added in a follow-up phase if reproducibility becomes a pain point.
- **D-11:** Install pipeline ports reference shape: fetch to quarantine → scan (Phase 19 D-14/D-16 + D-15 enforcement using source-derived trust) → atomic `shutil.move`-equivalent to final location → record manifest. Failed installs leave no partial state in `skills/`; quarantine is cleaned up.

#### Surface Scope
- **D-12:** Publish is **deferred**. v1 covers install / search / update / uninstall / trust-list management only. Rationale: smallest viable Hub, unblocks Phase 20 quickly, publish is ~300-500 LOC on top and needs PAT write-scope + repo bootstrap.
- **D-13:** Tool surface additions are **read-only**: add `hub_search(query, source?)` tool action only. `skills_list` and `skill_view` already exist (Phase 07); they naturally show installed Hub skills because installs land in the scanned directories. **No `hub_install` tool action** — installs are CLI-only per ROADMAP criterion 4 (agent is discovery-helpful but user gates mutations).
- **D-14:** CLI command shape: `hermes skills install <identifier>`, `hermes skills search <query> [--source github|well-known|skills-sh]`, `hermes skills update [name]`, `hermes skills uninstall <name>`, `hermes skills list` (installed + trust), `hermes skills trust add|remove <owner/repo>`. JSON output via `--format json` on read-only subcommands. No interactive prompts at install (CI-friendly); `--yes` flag for future destructive paths.

### Claude's Discretion
- Crate selection for HTTP / tarball / compression (within "port faithfully" constraint).
- Whether `ironhermes-hub` is a new crate vs. a module inside `ironhermes-tools`.
- Manifest file shape: single `lock.json` dict vs. per-skill files (reference uses single dict — recommendation: match reference).
- Error envelope wording for Hub-specific failures (reuse Phase 17 D-15 shape).
- Test harness choice for HTTP mocking (wiremock vs. mockito vs. httpmock).

### Deferred Ideas (OUT OF SCOPE)
- **Publish flows** (push to GitHub, mirror to skills.sh, generate manifest): deferred per D-12.
- **Lockfile / reproducible pinning** (`.hub/skills.lock`): defer until users report update-drift pain.
- **Auto-update check at startup**: explicitly rejected for v1 (D-10).
- **Per-skill trust overrides**: explicitly rejected (D-06).
- **GitHub App auth**: rejected (D-03).
- **`hub_install` tool action**: deferred per D-13.
- **Interactive install prompts**: deferred per D-14.
- **Mid-session Hub refresh without restart**: not in scope — Phase 19 frozen-snapshot rule applies.

## Architecture Patterns

### Recommended Project Structure

**Option A (recommended): new crate `ironhermes-hub`**

```
crates/
├── ironhermes-hub/
│   ├── Cargo.toml
│   └── src/
│       ├── lib.rs             # re-export HubSource, HubInstaller, HubManifest, adapters
│       ├── source.rs          # HubSource trait + SkillBundle + SkillMeta types
│       ├── github.rs          # GitHubSource adapter (DEFAULT_TAPS, Trees API, tarball fetch)
│       ├── well_known.rs      # WellKnownSkillSource adapter
│       ├── skills_sh.rs       # SkillsShSource adapter (delegates to GitHubSource)
│       ├── auth.rs            # GitHubAuth: env → gh-cli subprocess fallback
│       ├── installer.rs       # fetch→quarantine→scan→move→manifest pipeline
│       ├── manifest.rs        # HubManifest = HashMap<String, ManifestEntry> + JSON io
│       ├── paths.rs           # .hub/ / quarantine/ / index-cache/ resolution (uses HERMES_HOME)
│       └── error.rs           # HubError + structured-envelope mapping
```

**Rationale:**
- Keeps reqwest/tar/flate2 out of `ironhermes-core` (core stays lean and build-fast).
- `ironhermes-tools` depends on `ironhermes-hub` for `hub_search` only — one-way dep.
- `ironhermes-cli` depends on `ironhermes-hub` for install/update/uninstall/trust.
- Test isolation: Hub integration tests stay inside the crate.

**Option B (lighter): module `ironhermes_tools::hub`**
- Saves Cargo.toml overhead but pollutes tools deps with reqwest/tar/flate2 and complicates mock isolation. Not recommended.

### Pattern 1: HubSource Trait (port of Python ABC)

**What:** Each adapter implements fetch + search + trust resolution for one origin type.
**When to use:** Any new source (future: OCI registry, IPFS).

```rust
// Source: ports Python ABC from ~/code/hermes-agent/tools/skills_hub.py
use async_trait::async_trait;
use ironhermes_core::skills::SkillSource;

#[derive(Debug, Clone)]
pub struct SkillMeta {
    pub name: String,
    pub identifier: String,       // adapter-specific, stable; user pastes into `skills install`
    pub source_id: String,        // "github" | "well-known" | "skills-sh"
    pub description: Option<String>,
    pub version: Option<String>,
}

#[derive(Debug)]
pub struct SkillBundle {
    pub name: String,
    pub identifier: String,
    pub source_id: String,
    pub files: Vec<BundleFile>,   // relative path + bytes
    pub skill_md: String,         // SKILL.md content (for frontmatter parse)
    pub metadata: serde_json::Value,
}

#[async_trait]
pub trait HubSource: Send + Sync {
    fn source_id(&self) -> &str;
    fn trust_level_for(&self, identifier: &str) -> SkillSource;
    async fn search(&self, query: &str, limit: usize) -> Result<Vec<SkillMeta>, HubError>;
    async fn fetch(&self, identifier: &str) -> Result<SkillBundle, HubError>;
}
```

### Pattern 2: Install Pipeline (atomic, all-or-nothing)

**What:** Five steps that leave zero partial state on failure.
**When to use:** Every `hermes skills install` invocation.

```rust
pub async fn install(
    source: &dyn HubSource,
    identifier: &str,
    scanner: &dyn SkillScanner,    // wraps scan_skill_content
    config: &HubConfig,
) -> Result<InstallOutcome, HubError> {
    // 1. Fetch
    let bundle = source.fetch(identifier).await?;
    let content_hash = bundle_content_hash(&bundle); // sha256 over sorted (path, bytes)

    // 2. Quarantine (tempfile::TempDir scoped to this install — auto-cleans on drop if we bail)
    let quarantine = tempfile::tempdir_in(hub_paths::quarantine_dir()?)?;
    write_bundle(&quarantine, &bundle)?;

    // 3. Scan (applies trust-based enforcement from Phase 19 D-15)
    let trust = source.trust_level_for(identifier);
    let verdict = scanner.scan_bundle(&quarantine, &bundle)?;
    enforce_trust_gate(trust, &verdict)?;  // hard-reject iff Community && scan-hit

    // 4. Atomic move
    let (category, name) = parse_skill_name(&bundle)?;
    let final_path = skills_root()?.join(&category).join(&name);
    if final_path.exists() { return Err(HubError::AlreadyInstalled(name)); }
    fs::create_dir_all(final_path.parent().unwrap())?;
    fs::rename(quarantine.path(), &final_path)
        .or_else(|_| copy_then_remove(quarantine.path(), &final_path))?;

    // 5. Manifest
    let mut manifest = HubManifest::load_or_default()?;
    manifest.insert(ManifestEntry {
        name, source: source.source_id().into(), identifier: identifier.into(),
        content_hash, scan_verdict: verdict.summary(),
        install_path: final_path.clone(),
        files: bundle.files.iter().map(|f| f.path.clone()).collect(),
        installed_at: chrono::Utc::now(),
        updated_at: None,
        metadata: bundle.metadata.clone(),
    });
    manifest.save()?;

    Ok(InstallOutcome { /* ... */ })
}
```

### Pattern 3: Trust Resolution at Registry Load (D-06, D-08)

**What:** `load_with_paths` consults manifest + `hub.trusted_repos` to assign `SkillSource` per skill.
**When to use:** Every time `SkillRegistry::load_with_paths` runs (session start, per frozen-snapshot rule).

```rust
// In ironhermes-core/src/skills.rs (Wave 0 edit)
pub enum SkillSource {
    Builtin,    // compile-time bundle in skills/
    Official,   // optional-skills/ at compile time (D-05)
    Trusted,    // Hub install + origin in hub.trusted_repos (D-04, D-06)
    Community,  // Hub install, origin not on allowlist (D-07 default)
}

// During load, for each skill directory found under user skills root:
fn resolve_source(
    skill_dir: &Path,
    manifest: &HubManifest,
    trusted_repos: &HashSet<String>,
) -> SkillSource {
    match manifest.lookup_by_path(skill_dir) {
        Some(entry) => {
            // WellKnown: D-07 — always Community regardless of origin
            if entry.source == "well-known" { return SkillSource::Community; }
            // GitHub / skills.sh: check trusted_repos from config (re-read every load per D-08)
            let repo = entry.identifier.rsplit_once('/').map(|(owner_repo, _)| owner_repo);
            if repo.map_or(false, |r| trusted_repos.contains(r)) {
                SkillSource::Trusted
            } else {
                SkillSource::Community
            }
        }
        None => SkillSource::Builtin,  // fallback (shouldn't happen for user-scope skills)
    }
}
```

### Pattern 4: Structured Error Envelope (reuse Phase 17 D-15)

**What:** Hub failures surface to `hub_search` tool and CLI `--format json` as a consistent envelope.
**When to use:** Every user-facing Hub error path.

```json
{
  "error": "hub_search_failed",
  "kind": "rate_limited" | "network" | "not_found" | "auth_required" | "trust_rejected" | "scan_blocked",
  "message": "GitHub rate limit exceeded; retry after 2026-04-14T03:00:00Z",
  "suggestion": "Set HERMES_GITHUB_TOKEN or run 'gh auth login' to raise the rate limit to 5000/hr",
  "retry_after_s": 420
}
```

### Anti-Patterns to Avoid

- **Freezing trust in manifest at install time.** Violates D-08. Manifest stores origin `identifier`; trust is recomputed from config on every load.
- **Per-skill trust overrides in config.** Violates D-06. Promote the repo to `hub.trusted_repos` instead.
- **Writing directly into `~/.ironhermes/skills/<cat>/<name>/` during fetch.** Must go through quarantine first (D-11).
- **Using `git2`/`gix` to clone the whole repo.** GitHub Trees API + tarball download of a single path is faster and avoids libgit2 build hell on Windows.
- **Blocking `gh auth token` subprocess on the async executor thread.** Use `tokio::process::Command` or `spawn_blocking` with a short timeout (2-3 s).
- **Treating WellKnown trust as origin-dependent.** D-07 locks all WellKnown installs to Community.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Tarball extraction | Custom tar parser | `tar` crate 0.4 | Handles long names, sparse files, symlink safety |
| Gzip stream decode | Custom inflate | `flate2` crate 1.1 | Battle-tested zlib/miniz_oxide backend |
| HTTP retry/backoff | Loop with `sleep` | `reqwest` + manual single retry on 429/503 + `Retry-After` header | GitHub API sends explicit retry hints |
| Path traversal validation | Regex allowlist | Port Python `_validate_bundle_rel_path` semantics: reject absolute, `..`, drive letters, symlink escape | One well-tested helper; tar crate's `Entry::path()` is not auto-safe |
| Atomic cross-FS move | Custom retry | `fs::rename` with `copy_then_remove` fallback (the quarantine-to-skills move may cross filesystems on Linux when `$HOME` and `/tmp` differ) | Mirrors Python `shutil.move` semantics |
| SHA256 bundle hash | Hand-rolled update loop | `sha2::Sha256` fed (path bytes + 0x00 + content bytes) for each file in sorted order | Matches Python `bundle_content_hash` exactly |
| YAML frontmatter parse | Regex | `serde_yaml` (already used by SkillRegistry) | Reuse existing code path |
| `gh auth token` retrieval | Parse `gh` config files | Subprocess call with 2-second timeout | Matches reference; gh is the only supported gh-cli tool |
| GitHub JSON response typing | stringly-typed | `serde::Deserialize` structs with `#[serde(rename_all = "snake_case")]` | Typed errors when schema changes |
| CLI argument parsing | Manual `args()` | `clap` derive with nested subcommands (`Skills { subcommand: SkillsAction }`) | Already the convention |

**Key insight:** Every "simple" task in the install pipeline (tar, gzip, rename, hash, path safety) has a well-tested crate. The only truly novel code is the `HubSource` trait + adapter logic + manifest shape — everything else is glue.

## Common Pitfalls

### Pitfall 1: `SkillSource` enum missing `Trusted` variant
**What goes wrong:** `ironhermes-core/src/skills.rs` currently defines only `Builtin / Official / Community`. Phase 19.1 install of a `trusted_repos` skill cannot be labeled correctly.
**Why it happens:** Phase 19 intentionally left this for 19.1 (comment: "Phase 19.1 will set provenance before this point").
**How to avoid:** Wave 0 task: add `Trusted` variant, update match arms in `context_scanner` enforcement, update tests.
**Warning signs:** Compile errors in scan-enforcement match arms after adding the variant — good; that proves coverage.

### Pitfall 2: Cross-filesystem atomic rename fails
**What goes wrong:** `fs::rename` from `/tmp/...` to `$HOME/.ironhermes/...` returns `EXDEV` on some Linux setups (separate `/tmp` mount).
**Why it happens:** POSIX `rename(2)` is only atomic within one filesystem.
**How to avoid:** Place the quarantine under `~/.ironhermes/skills/.hub/quarantine/` (same FS as final dest) — Python reference already does this. Keep the fallback `copy_then_remove` for safety.
**Warning signs:** CI on Linux containers with bind-mounted `/tmp` will surface this if quarantine is misplaced.

### Pitfall 3: `gh auth token` subprocess hangs
**What goes wrong:** `gh` may prompt interactively (first-run OAuth flow) and block the tokio runtime.
**Why it happens:** Unconfigured `gh` on a fresh machine.
**How to avoid:** 2-second timeout via `tokio::time::timeout`; treat timeout as "no token, fall back to anonymous."
**Warning signs:** Intermittent hangs in CI without `HERMES_GITHUB_TOKEN` set.

### Pitfall 4: GitHub rate limits in tests
**What goes wrong:** Integration tests that hit real GitHub flake with 403 on CI.
**Why it happens:** Anonymous rate limit is 60/hr per IP; CI shares IPs.
**How to avoid:** Mock every HTTP call with `wiremock`. Reserve real-network tests for a separate `integration-live` feature flag, not run in CI.
**Warning signs:** Test failures that correlate with wall-clock time.

### Pitfall 5: WellKnown origin trust drift
**What goes wrong:** User assumes a "well-known" domain they control is trusted.
**Why it happens:** D-07 is counterintuitive — well-known is always Community regardless of origin.
**How to avoid:** Document prominently in CLI help text for `hermes skills install well-known:...`. Make the scan-rejection error message explicitly say "well-known origins always evaluate as Community; to mark this skill as Trusted, vendor it into a GitHub repo and add that repo to `hub.trusted_repos`."
**Warning signs:** User reports "why did my well-known install get rejected when I own the domain?"

### Pitfall 6: Manifest round-trip loses unknown fields
**What goes wrong:** A future IronHermes version adds `scan_verdict_version` field; older clients drop it on re-save.
**Why it happens:** Default serde strict deserialization.
**How to avoid:** Either (a) `#[serde(flatten)] extra: HashMap<String, serde_json::Value>` on ManifestEntry to preserve unknowns (WARN-BUT-LOAD pattern from 07.2 D-13), or (b) version the manifest and refuse to downgrade-save.
**Warning signs:** Data loss after rolling back an IronHermes version.

### Pitfall 7: tar crate path-traversal
**What goes wrong:** `tar::Archive::unpack()` resolves symlinks and can escape the destination.
**Why it happens:** Default behavior preserves symlinks.
**How to avoid:** Walk entries manually; validate each `entry.path()` against the quarantine root; reject absolute paths, `..` components, and symlinks that point outside the root. Port `_validate_bundle_rel_path` from Python reference.
**Warning signs:** Security audit flags "unpack" calls in code review.

### Pitfall 8: Manual clap nested enum for Skills subcommand
**What goes wrong:** `hermes skills install` and `hermes skills trust add` both parse but share no type — easy to drift.
**Why it happens:** clap derive nested enums require discipline.
**How to avoid:** Single `SkillsAction` enum with `Install { identifier: String }`, `Search { query: String, source: Option<SourceFlag> }`, `Trust { action: TrustAction }`, etc. Test each variant round-trips through `--help`.
**Warning signs:** Help text disagreement between CLI docs and actual output.

### GitHubAuth Resolution (D-03)

```rust
pub struct GitHubAuth { token: Option<String> }

impl GitHubAuth {
    pub async fn resolve(override_env: Option<&str>) -> Self {
        // 1. Explicit config override
        if let Some(var) = override_env {
            if let Ok(t) = std::env::var(var) { return Self { token: Some(t) }; }
        }
        // 2. HERMES_GITHUB_TOKEN, then GITHUB_TOKEN
        for var in &["HERMES_GITHUB_TOKEN", "GITHUB_TOKEN"] {
            if let Ok(t) = std::env::var(var) { return Self { token: Some(t) }; }
        }
        // 3. gh auth token fallback (2s timeout)
        if let Ok(Ok(out)) = tokio::time::timeout(
            std::time::Duration::from_secs(2),
            tokio::process::Command::new("gh").args(["auth", "token"]).output(),
        ).await {
            if out.status.success() {
                let t = String::from_utf8_lossy(&out.stdout).trim().to_string();
                if !t.is_empty() { return Self { token: Some(t) }; }
            }
        }
        Self { token: None }  // anonymous
    }
}
```

```rust
    pub fn load_or_default() -> anyhow::Result<Self> {
        let p = hub_paths::manifest_path()?;
        if !p.exists() { return Ok(Self::default()); }
        Ok(serde_json::from_str(&std::fs::read_to_string(p)?)?)
    }

    pub fn save(&self) -> anyhow::Result<()> {
        let p = hub_paths::manifest_path()?;
        std::fs::create_dir_all(p.parent().unwrap())?;
        let tmp = p.with_extension("json.tmp");
        std::fs::write(&tmp, serde_json::to_string_pretty(self)?)?;
        std::fs::rename(tmp, p)?;  // atomic replace
        Ok(())
    }
```

## Open Questions (RESOLVED)

1. **Should `hermes skills update` re-run scan-enforcement with current config, or only compare content hash?**

   - What's unclear: Is that redundant given D-16 (scan at every registry-load) already covers trust demotion?

### Known Threat Patterns for Rust/HTTP/tarball stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Path traversal via tar entry (`../../etc/passwd`) | Tampering | Manual validation of every `entry.path()` against quarantine root before unpack |
| Symlink escape (tar entry is a symlink to `/`) | Tampering | Reject symlink entries outright during walk |
| Zip-bomb / tar-bomb (decompressed size >> compressed) | DoS | Cap extracted bytes at 50 MB; cap entry count at 1000 |
| SSRF via well-known URL pointing to internal service | Information Disclosure | Reject private/loopback IPs; require HTTPS; `url` crate validation |
| Token leak in error logs | Information Disclosure | `secrecy::Secret` wrapper + tracing-filter for `Authorization` header |
| MITM on tarball download | Tampering | reqwest rustls-tls, TLS 1.2+; content_hash comparison on update |
| Malicious SKILL.md frontmatter (prompt injection) | Elevation of Privilege | `scan_skill_content` with `SKILL_THREAT_PATTERNS`; Community hard-reject (D-15) |
| Supply-chain: trusted repo owner rotates to malicious content | Tampering | D-08 recompute means user can `trust remove` and next load re-gates; no mitigation against silent content changes mid-install except content_hash comparison |
| GitHub rate-limit DoS | DoS | Exponential backoff on 429/503; surface `Retry-After` in structured error envelope |
