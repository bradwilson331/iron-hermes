## Summary

Wired trust-label resolution into SkillRegistry::load_with_config so installed skills get the correct SkillSource at registry-load time. Trust is never frozen in the install manifest — recomputed on every load from config.hub.trusted_repos (D-08).

## What Was Built

### Task 1: resolve_source() + load_with_config trust wiring

- HubManifestReadOnly / HubManifestEntryReadOnly in skills.rs — minimal read-only types to avoid crate cycle with ironhermes-hub
- load_hub_manifest() — reads .hub/lock.json, falls back to empty on error
- resolve_source() — 6-row decision table: optional-skills→Official (D-05), no manifest→Builtin, well-known→Community (D-07), skills-sh→Community, github+trusted→Trusted, github+untrusted→Community (D-06/D-08)
- Modified load_with_config to merge manifests from all search_paths, re-resolve every SkillRecord.source, and re-apply D-15 Community hard-reject
- 7 test_resolve_source_* unit tests covering all 4 outcomes

### Task 2: End-to-end integration tests

- crates/ironhermes-core/tests/trust_resolution_test.rs — 4 integration tests:
  - e2e_trust_recompute_registry_reload: flips trusted_repos → same skill changes Trusted→Community (D-08)
  - e2e_well_known_ignores_trusted_repos: well-known forces Community despite trusted_repos entry (D-07)
  - e2e_official_skill_warn_but_load: optional-skills/ → Official → WARN-BUT-LOAD on scan hit
  - e2e_no_manifest_entry_builtin: orphan skill → Builtin → WARN-BUT-LOAD
