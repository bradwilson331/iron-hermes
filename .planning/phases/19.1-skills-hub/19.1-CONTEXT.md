# Phase 19.1 тАФ Skills Hub тАФ Context

<domain>
Publishing skills to, and installing skills from, external sources. Installed skills are cloned-and-vendored into `~/.ironhermes/skills/` so the runtime works offline and is stable against upstream changes. Each installed skill carries an origin-derived `SkillSource` trust label (Builtin / Official / Trusted / Community) that gates the Phase 19 D-15 security-scan enforcement: hard-reject on scan hit for Community; WARN-BUT-LOAD for Builtin / Official / Trusted. Primary management surface is CLI; agent gets a read-only discovery tool action (`hub_search`) so it can help users find skills, but install mutations stay under user control.

**In scope (SKILL-08, SKILL-09):** Source adapters (GitHub, well-known, skills.sh), origin-based trust labeling, clone-and-vendor install, manual update/uninstall, trusted-repo config, read-only `hub_search` tool action.

**Out of scope (deferred):** Publish flows (push to GitHub, mirror to skills.sh); defer to Phase 19.2 or Phase 20 alongside `skill_manage` create/edit/delete.
</domain>

<decisions>

## Source Adapters

- **D-01:** Port all three reference adapters in v1 тАФ `GitHubSource`, `WellKnownSkillSource` (reads `/.well-known/skills/index.json`), `SkillsShSource` (wraps GitHub via the skills.sh registry). Rationale: users migrating from hermes-agent keep install muscle memory; the adapter trait lets us add more later.

- **D-02:** GitHub default taps mirror hermes-agent exactly: `openai/skills`, `anthropics/skills`, `VoltAgent/awesome-agent-skills`, `garrytan/gstack`. Port as-is to avoid silent drift from reference. User can add extra taps via config.

- **D-03:** GitHub auth precedence: anonymous (public repos) тЖТ `HERMES_GITHUB_TOKEN` or `GITHUB_TOKEN` env var тЖТ `gh auth token` subprocess fallback. **No GitHub App support** тАФ overkill for the single-operator PROJECT.md target and adds ~80 lines of JWT signing.

- **D-04:** Trusted-repo allowlist lives in `config.yaml` under `hub.trusted_repos: [owner/repo, ...]` with an **empty default**. User adds explicitly via `hermes skills trust add owner/repo` (writes config) or manual edit. No compiled-in allowlist.

## Trust Assignment

- **D-05:** `SkillSource::Official` maps to skills bundled in the repo's `optional-skills/` directory at compile time (matches REQUIREMENTS SKILL-09). Anything installed via the Hub never becomes Official тАФ Hub installs are at most Trusted (config allowlist) or Community (default).

- **D-06:** Trust is **per-source, never per-skill**. The adapter's `trust_level_for(identifier)` is authoritative. No per-skill overrides in config (avoids forgotten-override rot and attack surface). To elevate a single skill, promote its repo to `hub.trusted_repos`.

- **D-07:** Well-known HTTP adapter always returns Community regardless of origin. Rationale: any HTTPS origin can serve a well-known index; there's no central vetting surface. D-15 scan + Community = hard-reject on injection pattern is the security gate.

- **D-08:** Trust is **recomputed on every registry-load** by re-reading `hub.trusted_repos` from config. Never frozen in an install-time manifest. User can demote a repo and the stricter scan-enforcement tightens on next load without reinstall. Install-time metadata records source + identifier, not trust.

## Install Lifecycle

- **D-09:** Install layout is **flat by category**: `~/.ironhermes/skills/<category>/<skill-name>/`. A `~/.ironhermes/skills/.hub/` directory holds install manifests (origin identifier, install hash, install timestamp). Matches reference `skills_hub.py` exactly. Trust is computed from config at runtime тАФ never encoded in the directory tree.

- **D-10:** Update strategy is **manual only** via `hermes skills update [name]`. No auto-refresh at startup, no daily background check. Rationale: deterministic, scriptable, no surprise behavior changes between runs; matches reference `skills_sync` manual pattern. No lockfile in v1 тАФ pinning can be added in a follow-up phase if reproducibility becomes a pain point.

- **D-11:** Install pipeline ports reference shape: fetch to quarantine тЖТ scan (Phase 19 D-14/D-16 + D-15 enforcement using source-derived trust) тЖТ atomic `shutil.move`-equivalent to final location тЖТ record manifest. Failed installs leave no partial state in `skills/`; quarantine is cleaned up.

## Surface Scope

- **D-12:** Publish is **deferred**. v1 covers install / search / update / uninstall / trust-list management only. Rationale: smallest viable Hub, unblocks Phase 20 quickly, publish is ~300-500 LOC on top and needs PAT write-scope + repo bootstrap.

- **D-13:** Tool surface additions are **read-only**: add `hub_search(query, source?)` tool action only. `skills_list` and `skill_view` already exist (Phase 07); they naturally show installed Hub skills because installs land in the scanned directories. **No `hub_install` tool action** тАФ installs are CLI-only per ROADMAP criterion 4 (agent is discovery-helpful but user gates mutations).

- **D-14:** CLI command shape: `hermes skills install <identifier>`, `hermes skills search <query> [--source github|well-known|skills-sh]`, `hermes skills update [name]`, `hermes skills uninstall <name>`, `hermes skills list` (installed + trust), `hermes skills trust add|remove <owner/repo>`. JSON output via `--format json` on read-only subcommands. No interactive prompts at install (CI-friendly); `--yes` flag for future destructive paths.

</decisions>

<code_context>

## Existing assets to reuse

- `SkillSource` enum from 19-01 (Builtin / Official / Trusted / Community) already drives scan enforcement per D-15. Hub install sets the label at load time; `trust_level_for(identifier)` resolution lives in the adapter.
- `scan_skill_content` from 19-05 is the mandatory install-pipeline gate (D-11). Community + scan-hit = hard-reject (install fails; nothing moves out of quarantine). Builtin/Official/Trusted + scan-hit = WARN-BUT-LOAD (install succeeds with tracing warning).
- `load_with_paths` already scans `~/.ironhermes/skills/` on registry load. Hub installs land in this tree, so scanning is automatic тАФ no new scan call site needed for *registry load*, only for the *install-time* gate.
- `HERMES_HOME` resolution helper from 19-03 (`default_credential_dir` precedence: env тЖТ `dirs::home_dir()`). Hub uses the same `$HERMES_HOME/skills` pattern.
- `SkillsConfig` round-trip test harness in `config.rs` covers all new fields with serde defaults.

## New surface to add

- New crate or module: `ironhermes-hub` (adapter trait, three adapter impls, install pipeline, manifest read/write). Keeps `ironhermes-core` small and lets the hub crate pull in reqwest / git2 / tar without polluting core deps.
- Adapter trait (port of Python ABC):
  ```rust
  trait HubSource {
      fn source_id(&self) -> &str;
      fn trust_level_for(&self, identifier: &str) -> SkillSource;
      fn search(&self, query: &str, limit: usize) -> Result<Vec<SkillMeta>>;
      fn fetch(&self, identifier: &str) -> Result<SkillBundle>;
  }
  ```
- Install pipeline: `fetch тЖТ quarantine тЖТ scan тЖТ move тЖТ manifest`. Manifest file at `~/.ironhermes/skills/.hub/<skill>.json` records `{source, identifier, installed_at, hash}`.
- CLI subcommand tree under `hermes skills <verb>`.
- `hub_search` action in `SkillsTool` (read-only; no filesystem mutation).

## Patterns to follow

- Structured-error envelope (Phase 17 D-15) for `hub_search` tool responses and for CLI JSON output on errors.
- Atomic file ops per reference (quarantine + move, never write-in-place).
- WARN-BUT-LOAD divergence for parsing of manifest files (D-13 pattern тАФ unknown fields preserved in extras).

<deferred>

## Deferred to later phases

- **Publish flows** (push to GitHub, mirror to skills.sh, generate manifest): deferred per D-12. Candidate next phase: 19.2 or co-located with Phase 20 `skill_manage`.
- **Lockfile / reproducible pinning** (`.hub/skills.lock`): defer until users report update-drift pain. Manual `update` per D-10 is enough for v1.
- **Auto-update check at startup**: explicitly rejected for v1 (D-10). Revisit if discoverability becomes a problem.
- **Per-skill trust overrides**: explicitly rejected (D-06). If a user needs finer trust control, we'll add a new concept rather than overloading the allowlist.
- **GitHub App auth**: rejected (D-03). PAT + gh-cli cover the single-operator use case.
- **`hub_install` tool action** (agent-initiated installs): deferred per D-13. If users request "let the agent install skills," reconsider with an explicit confirmation gate.
- **Interactive install prompts** (CLI): deferred per D-14 to keep CI/scripting clean. `--yes` flag reserved for future destructive paths.
- **Mid-session Hub refresh without restart**: not in scope тАФ registry-load snapshot is frozen per Phase 19 frozen-snapshot rule; `update` takes effect next run.

</deferred>
