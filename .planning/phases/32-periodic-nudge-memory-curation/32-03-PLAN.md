turn counter (HashMap<String, u32>) using the same Arc<std::sync::Mutex<HashMap>>
interior-mutability pattern as the gateway. run_web_turn fires spawn_nudge_review via
tokio::spawn after agent.run() completes successfully.

the Dioxus server (no separate gateway process — see project memory). Without this wiring,
users on the web UI never get the periodic memory-review nudge.

nudge fire site mirroring the gateway pattern from Plan 32-02.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| web UI tokio::spawn -> MemoryManager | Nudge writes flow through MemoryTool -> MemoryStore::scan_content (Phase 17 security scanner); no bypass |
| std::sync::Mutex guard scope (nudge_turns) | Guard released before tokio::spawn; Mutex never held across await — enforced by inner-block scoping and clippy::await_holding_lock |
| web session -> nudge fork | Nudge AgentLoop is independent; cannot mutate AppState.state_store entries written during the active turn (snapshot is a Vec<ChatMessage> clone) |
| AppState -> global OnceLock | install_global_app_state writes once; nudge wiring does not alter the singleton invariant — nudge_turns mutates through Arc<Mutex<>>, not by replacing AppState |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-32-09 | Denial of Service | per-session HashMap unbounded growth | accept | One u32 per active web session; web sessions bounded by uuid-keyed entries; entry remains for the session lifetime — acceptable given small per-entry size (4 bytes + key string) and finite session count per host. No active eviction needed for MVP. |
| T-32-10 | Availability | std::sync::Mutex held across await in run_web_turn | mitigate | Counter mutation done in synchronous inner block; should_fire bool extracted; guard dropped before tokio::spawn. Verified by clippy::await_holding_lock in acceptance criteria. |
| T-32-11 | Denial of Service | web UI nudge recursion | accept | nudge_turns HashMap lives on AppState; spawn_nudge_review constructs an independent AgentLoop with a narrow ToolRegistry — it does not reference AppState or its nudge_turns map. Structurally impossible for the nudge to trigger another nudge. |
| T-32-12 | Information Disclosure | nudge snapshot leaks across sessions | mitigate | messages_snapshot is built from a per-session messages Vec returned by build_messages_for_turn(session_id, ...). The HashMap entry is keyed by session_id (String). Cross-session leakage requires session_id collision, which is uuid-keyed (~10^-37 probability). |
| T-32-13 | Tampering | prompt injection via memory review | mitigate | Inherited from Plan 32-01: nudge writes flow through MemoryManager::handle_tool_call -> MemoryStore::scan_content. No web-UI-specific bypass added. |
| T-32-14 | Availability | provider resolver failure on nudge client build | accept | build_main_client(&self.resolver)? propagates the error via `?` — same failure that would block the main turn. The nudge does not fire when the provider config is broken, which is correct behavior (no silent retries on bad config). |
| T-32-SC | Tampering | cargo installs | accept | No new external packages in this plan; pure intra-workspace changes to crates/iron_hermes_ui only |
</threat_model>
