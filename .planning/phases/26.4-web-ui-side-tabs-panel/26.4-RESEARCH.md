# Phase 26.4: web-ui-side-tabs-panel - Research

**Researched:** 2026-05-05
**Domain:** Dioxus 0.7 RSX component extension, CSS side panel tab strip
**Confidence:** HIGH

## Summary

Phase 26.4 is a pure UI extension — no new server functions, no new state primitives. It adds a two-tab strip (AGENT / INFO) inside the existing `AgentPanel` (`agent_panel.rs`) and a conditional INFO view that reads entirely from existing signals in `WarpHermes`. The implementation touches exactly three files: `agent_panel.rs` (new props + tab strip RSX + conditional content), `warp_hermes.rs` (new `active_side_tab` signal + new prop pass), and `warp-ih.css` (five new CSS classes). All data sources are already in scope in `WarpHermes` — model, provider, and context length come from `config_summary()` server future (already fetched); `session_id` and `tokens` are existing signals; `messages.read().len()` gives the count. Memory enabled (`config.memory.memory_enabled`) is NOT currently included in `ConfigSummary` and is NOT currently passed down to the UI — the planner must decide whether to add it to the server-side `ConfigSummary` struct or omit the memory row.

The implementation is self-contained. The only non-trivial decision is the `memory_enabled` source path. All other wiring follows directly from established patterns already used in `TitleBar` (Phase 26.2) and existing `AgentPanel`/`StatusBar` signal prop patterns.

**Primary recommendation:** Add `active_side_tab: Signal<usize>` in `WarpHermes`, pass `active_side_tab: active_side_tab` and `on_side_tab_click: on_side_tab_click` plus three read-only data props to `AgentPanel`. Use `ReadSignal<T>` for read-only reactive data, `EventHandler<usize>` for the tab click callback — exactly the patterns already in the codebase.

### Pattern 1: ReadSignal Prop for Reactive Read-Only Data

This is the established project pattern for parent→child reactive data (confirmed in `agent_panel.rs:21`, `command_palette.rs:26`, `status_bar.rs:23-24`, `block_stream.rs:19`).

```rust
// Source: VERIFIED in agent_panel.rs, status_bar.rs, block_stream.rs
#[component]
pub fn AgentPanel(
    messages: ReadSignal<Vec<Message>>,      // existing
    active_side_tab: ReadSignal<usize>,      // NEW
    on_side_tab_click: EventHandler<usize>,  // NEW
    session_id: ReadSignal<String>,          // NEW
    token_budget: ReadSignal<TokenBudget>,   // NEW
    model_label: String,                     // NEW — non-reactive, re-renders on parent re-render
    provider_label: String,                  // NEW
    context_length: u32,                     // NEW
    memory_enabled: bool,                    // NEW (if implemented)
) -> Element { ... }
```

**Note on `Signal<T>` → `ReadSignal<T>` coercion:** Dioxus 0.7 automatically coerces a `Signal<T>` at the call site into `ReadSignal<T>` when the prop type is `ReadSignal<T>`. This is the established pattern — `warp_hermes.rs` passes `messages: messages` (a `Signal<Vec<Message>>`) and `agent_panel.rs` receives it as `ReadSignal<Vec<Message>>`. [VERIFIED: project codebase]

## Common Pitfalls

### Pitfall 1: Signal Borrow Held Across Await
**What goes wrong:** Calling `.read()` or `.write()` on a signal and holding the result across a `.await` point — compile panic at runtime (clippy.toml enforces).
**Why it happens:** Dioxus uses arena-allocated signals; holding a borrow across await violates the arena's epoch model.
**How to avoid:** The INFO tab only reads signals synchronously in RSX expressions — no async code paths. `messages.read().len()` inside RSX is safe.
**Warning signs:** clippy lint `holding_invalid_reference` fires.

### Pitfall 2: Forgetting `context_length` in config_summary Destructure
**What goes wrong:** `cfg.context_length` exists in `ConfigSummary` but the current destructure at `warp_hermes.rs:387` only extracts `model` and `provider`. The executor must also extract `context_length` and pass it as a prop.
**How to avoid:** Update the destructure to a 3-tuple: `let (model_name, provider_name, context_length) = match config_summary() { Some(Ok(cfg)) => (cfg.model, cfg.provider, cfg.context_length), _ => ("loading...".to_string(), "...".to_string(), 0_u32) };`

### Pitfall 3: Using `::before` Instead of `::after` for Active Indicator
**What goes wrong:** TitleBar uses `::before` with `top: 0` for the active stripe. Copying it verbatim would put the indicator at the top of side tabs, not the bottom.
**How to avoid:** Side tab active indicator uses `::after` with `bottom: 0`. Explicitly specified in UI-SPEC.

### Pitfall 4: Active Tab Index Comparison
**What goes wrong:** Using `class: if i == active_side_tab() { "is-active" }` where `i` is from a `for` loop, but side tabs are not iterated — they are hardcoded (only 2 tabs). Iterating with wrong index gives wrong active state.
**How to avoid:** Because there are exactly 2 tabs with fixed labels (AGENT at 0, INFO at 1), use hardcoded comparisons: `class: if 0 == active_side_tab() { "is-active" }` on the AGENT button, `class: if 1 == active_side_tab() { "is-active" }` on the INFO button. No loop needed.

### Pitfall 5: `.wh-side` Already Has `display: flex; flex-direction: column`
**What goes wrong:** Adding `.wh-side-tabs` without `flex-shrink: 0` causes it to collapse to zero height as `.wh-side-scroll` (which has `flex: 1`) takes all space.
**How to avoid:** `.wh-side-tabs` needs `flex-shrink: 0` and a fixed `height: 32px` — already specified in UI-SPEC CSS. [VERIFIED: warp-ih.css:333-341 showing flex column layout]

## Memory Enabled — Source Decision

The CONTEXT.md D-07 specifies that memory-enabled state should be shown in the Config card. Investigation reveals:

- `MemoryConfig.memory_enabled: bool` exists in `ironhermes_core::config::MemoryConfig` [VERIFIED: config.rs:385]
- It is accessible server-side via `state.config.memory.memory_enabled` [VERIFIED: server/state.rs]
- `ConfigSummary` (server/api.rs) does NOT currently include `memory_enabled`
- `ShellSettings` context (state.rs:261-264) does NOT include it
- There is no existing signal in `WarpHermes` that carries this value

**Two options for the planner:**

**Option A (recommended): Add `memory_enabled: bool` to `ConfigSummary`**
- Add field to `ConfigSummary` struct in `server/api.rs`
- Populate it from `state.config.memory.memory_enabled` in `get_config_summary()`
- Extract it in the `warp_hermes.rs` `config_summary()` destructure
- Pass as `memory_enabled: bool` prop to `AgentPanel`
- Minimal surface area, no new server function, consistent with D-07

**Option B: Pass hardcoded default**
- Pass `memory_enabled: true` as a literal (default from `MemoryConfig::default()`)
- Simpler, but misleads the user if memory is actually disabled
- Not recommended
