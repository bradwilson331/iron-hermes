<decisions>
## Implementation Decisions

### INV-22.3-02 Assertion Update

- **D-01:** Replace the equality assertion `count == 1` on `print_banner();` with `count >= 1`. **Why:** the original `count == 1` was correct for Phase 22.3 (single `run_chat` call site) but became stale when Phase 22.4 added two ratatui-bound startup paths (per Phase 22.4 CONTEXT D-03), each of which legitimately needs to print the banner before `ratatui::init()`. Relaxing to `>= 1` accepts the legitimate sites without forcing a centralization that would harm Phase 22.4's startup ergonomics. **How to apply:** in `crates/ironhermes-cli/tests/invariants_22_3.rs`, the existing `assert_eq!(count, 1, ...)` becomes `assert!(count >= 1, ...)`.

- **D-02:** Strengthen the ordering assertion: **every** `print_banner();` call site MUST appear strictly before the `TuiHandle::new_with_extensions` call site. **Why:** the original test only checked the first call site via `MAIN_RS.find("print_banner();")` — which is positionally weak; it admits a future second call AFTER `new_with_extensions` so long as some first call exists before it. With three legitimate call sites we tighten the ordering to "all-before" so any future regression that places banner emission inside or after the TUI lifecycle trips the gate. **How to apply:** iterate every `print_banner();` match position via `MAIN_RS.match_indices("print_banner();")` (or a manual loop with `MAIN_RS[search_from..].find(...)`) and assert each absolute byte offset is `< tui_pos`.

- **D-03:** Use `TuiHandle::new_with_extensions` (qualified) as the TUI-init anchor string, not bare `new_with_extensions`. **Why:** the file contains test references to the bare name in INV-22.1-01 (`fn run_chat_uses_new_with_extensions`, plus assertion-message string literals); using the qualified form anchors on the actual call site at `main.rs:851` and stays robust if the test layout changes. **How to apply:** `MAIN_RS.find("TuiHandle::new_with_extensions")` instead of the existing `find("new_with_extensions")`.

- **D-04:** Rename the test function from `invariant_22_3_02_banner_called_exactly_once_before_tui_init` to `invariant_22_3_02_banner_called_at_least_once_strictly_before_tui_init` to reflect the relaxed-count + strengthened-ordering contract. **Why:** the old name encodes the `count == 1` contract that this phase explicitly retires. A test name that lies about behavior is itself a regression hazard. **How to apply:** rename the `#[test] fn` in the same file. No callers — Rust test discovery is by attribute.

- **D-05:** Update the assertion failure messages and the doc-comment block above the test to:
  - cite the new contract (count `>= 1`, every call site strictly before `TuiHandle::new_with_extensions`),
  - reference Plan 22.4-11's legitimate ratatui-arm additions and Phase 22.4 CONTEXT D-03 as the rationale for accepting more than one site,
  - preserve the existing reference to Plans 22.3-05 / RESEARCH §Banner-Bleed Probe (the structural reason banner-bleed is self-healing — `/clear`'s scrollback erase + banner-before-DECSTBM).

  **Why:** future regression hits should explain why three call sites are now legitimate so the next reader does not "fix" it back to `count == 1`. **How to apply:** rewrite the doc-comment and the two assertion messages in `invariants_22_3.rs` to match.

- **D-06:** Continue matching the call-site form `print_banner();` with the trailing semicolon, NOT bare `print_banner()`. **Why:** the bare form matches the function definition (`fn print_banner()` at `main.rs:2207`) and doc-comment references at `main.rs:2366` and `main.rs:2414`, inflating the count and producing false positives (per `22.3-06-SUMMARY.md` lesson). **How to apply:** the count check, the position iteration, and the position-of-first-call lookup all use the literal `"print_banner();"`.

### Scope Boundaries (Claude's Discretion)

- **Test-only edit.** No edits to `crates/ironhermes-cli/src/main.rs`. No changes to `print_banner()` itself. No call-site consolidation. No new dev-deps. No new test files.
- The `must_haves.truths` line in `22.3-06-PLAN.md` ("INV-22.3-02: print_banner() appears exactly once in main.rs AND strictly before TuiHandle::new_with_extensions") is historical documentation of what 22.3-06 shipped — it is **not** a contract to preserve. This phase deliberately supersedes that "exactly once" half.
- The runtime banner-bleed behavior captured in `22.3-UAT-EVIDENCE.md` (D-4) was determined self-healing per `22.3-RESEARCH.md` §Banner-Bleed Probe (single-or-more banner sites in scrollback before DECSTBM is established + `/clear`'s `\x1b[3J` scrollback erase). The relaxation to `count >= 1` does NOT regress this property — Phase 22.4's two extra sites both fire BEFORE `ratatui::init()` (per the inline `D-03 (Phase 22.4)` comments at `main.rs:254` and `main.rs:318`), so banner-in-scrollback-before-TUI-init still holds.
- No other Phase 22.3 invariants change. INV-22.3-01, 03, 04, 05, 06 stay byte-for-byte identical.

<specifics>
## Specific Ideas

- The "every X strictly before Y" iteration can be done with `MAIN_RS.match_indices("print_banner();").for_each(|(pos, _)| assert!(pos < tui_pos, ...))`, or with a manual `while let Some(rel) = MAIN_RS[search_from..].find(...)` loop. Either is acceptable — pick the form that produces the clearest failure message (which call site tripped the assertion).
- Keep the doc comment immediately above the test concise: a single paragraph plus a `// NOTE:` block explaining the semicolon match (D-06) and why three sites are legitimate (D-05).

</specifics>

<deferred>
## Deferred Ideas

- **Consolidating the three `print_banner();` call sites into a single helper** that fires unconditionally before any TUI init — not pursued. Phase 22.4 CONTEXT D-03 deliberately fires the banner inside the ratatui dispatch arms so the timing is local to that startup path; centralizing would either run the banner twice (once globally + once per arm) or introduce a flag-guard. Both are worse than the current state.
- **Re-running human UAT on banner-bleed** — not required. Phase 22.3 HUMAN-UAT already certified D-4 closed with the structural fix; this phase only relaxes the structural test, not the runtime behavior.
- **Updating the `must_haves.truths` line in `22.3-06-PLAN.md`** — explicitly NOT done. That file is the historical record of what Plan 22.3-06 shipped; rewriting it would falsify the audit trail.
