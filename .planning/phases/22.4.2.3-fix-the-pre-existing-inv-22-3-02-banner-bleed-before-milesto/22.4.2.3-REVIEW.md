# Phase 22.4.2.3: Code Review Report

**Reviewed:** 2026-04-27
**Depth:** standard
**Files Reviewed:** 1
**Status:** clean

## Summary

Single-file structural-test rewrite. The renamed test
`invariant_22_3_02_banner_called_at_least_once_strictly_before_tui_init`
correctly relaxes the count gate from `== 1` to `>= 1` and replaces the
"first call site before TUI init" check with a `match_indices` loop that
asserts EVERY `print_banner();` call site precedes
`TuiHandle::new_with_extensions` in source order.

I verified the four risk areas called out in the deliverable:

1. **Logic correctness of the new test.** The loop `for (banner_pos, _)
   in MAIN_RS.match_indices("print_banner();")` walks every call site.
   `tui_pos` is computed once via `find("TuiHandle::new_with_extensions")`,
   which returns the FIRST occurrence — that is the correct anchor because
   any banner site at or after even the first TUI-init occurrence is a
   bleed regression. A future regression that places `print_banner();`
   after the (first) `TuiHandle::new_with_extensions` site will trip with
   a clear message including the offending byte offset and the delta. A
   regression that REMOVES every banner site will trip the `count >= 1`
   gate. A regression that removes the TUI-init anchor will trip the
   `.expect()` on `find`. Verified against today's main.rs:
   - 3 `print_banner();` sites at lines 257, 321, 770
   - 1 `TuiHandle::new_with_extensions` site at line 851
   - All three banner sites precede the TUI anchor — test passes today,
     and `cargo test -p ironhermes-cli --test invariants_22_3` reports
     `6 passed; 0 failed` per the SUMMARY.

2. **Grep false-positive risk on the literal `print_banner();`.**
   Verified no false positives in main.rs. Grep against the live source
   shows the exact string appears only at the three call sites
   (main.rs:257, 321, 770). Doc-comment occurrences at main.rs:2366 and
   main.rs:2414 use `print_banner()` WITHOUT a trailing semicolon and the
   definition at main.rs:2207 reads `fn print_banner() {` — neither
   matches the search literal. The doc-comment in the test file itself
   does mention `print_banner()` but `include_str!` only loads `main.rs`,
   not the test file, so test-source self-reference is not a concern.
   Anchor-string false-positive risk is also OK: the qualified
   `TuiHandle::new_with_extensions` literal occurs once in main.rs at
   line 851 (the bare `new_with_extensions` substring also appears in
   the in-file doc tests at line 2244–2245, but the qualified form
   anchors correctly to line 851 and no positionally-earlier qualified
   match exists).

3. **D-4 banner-bleed coverage preserved.** D-4 requires that the banner
   reaches scrollback BEFORE DECSTBM is established. The pre-change test
   enforced this via `count == 1` plus a single ordering check. The new
   test enforces it via `count >= 1` plus a per-site ordering check —
   strictly stronger on the ordering axis (every site, not just the
   first) and equivalent on the "banner-must-exist" axis. The
   loosened-count axis is intentional and matches the legitimate
   plan-22.4-11 ratatui-arm sites (CONTEXT D-03). The doc-comment
   block (lines 35–62) documents the trade-off and cites Plans 22.3-05 /
   22.3-RESEARCH §Banner-Bleed Probe correctly.

4. **Assertion failure messages.** The `count >= 1` panic message,
   the `find().expect(...)` message on `tui_pos`, and the per-site
   ordering panic message all include enough state (count value,
   `banner_pos`, `tui_pos`, signed delta) to debug a future regression
   without re-running tests.

The other 5 invariant tests (22.3-01, 22.3-03, 22.3-04, 22.3-05, 22.3-06)
are byte-identical to the pre-change state apart from a whitespace-only
formatter change at lines 129–134 in 22.3-03 (`rustfmt`-style line wrap of
two `find().expect()` chains; no semantic change). This is acceptable for
a `cargo fmt`-touched file — flagged as Info IN-03 below for the record,
not as a defect.

No critical issues. No warnings. Three Info items below.
