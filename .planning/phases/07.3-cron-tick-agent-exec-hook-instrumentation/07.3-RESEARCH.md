## Summary

Phase 07.3 closes two critical integration gaps identified by the v1.1 audit. Both gaps live in a single ~50-line block inside `crates/ironhermes-gateway/src/runner.rs` (lines 368-424). The fix is surgical: the tick task closure needs four additional variable captures before `join_set.spawn`, then the per-job branch replaces 11 lines of stub code with ~25 lines of real AgentLoop construction and execution.

The `GatewayRunner` struct already holds every field needed for AgentLoop construction: `config`, `tool_registry`, `memory_store`, `hook_registry`, and `skill_registry` are all present. No struct changes are required — only clones added to the closure capture block. The `ironhermes-cron` crate does NOT need `ironhermes-hooks` as a dependency; the HookRegistry capture lives entirely in `runner.rs`.

`LlmClient` is a concrete struct (not a trait), so test isolation requires either a feature-gated echo implementation or a real LLM endpoint. The recommended approach is an `#[cfg(test)]` module that constructs a `GatewayRunner` with a real (but minimal) `ToolRegistry`, creates a `JobStore` with one due job, and validates via a test `HookRegistry` listener using `Arc<Mutex<Vec<HookEvent>>>`. For the LLM call itself, the integration test should use a real endpoint if available, or gate the test with `#[ignore]` and document the `IRONHERMES_TEST_LLM=1` env var pattern.

**Primary recommendation:** Inline AgentLoop construction directly in the tick task (no helper function needed — it's 12 lines and used once). Add four variable captures before `spawn`. The D-15/D-16 integration tests live in a new `crates/ironhermes-gateway/tests/cron_integration.rs` file.

### Q1: Exact AgentLoop Construction Template (from `handler.rs::run_agent`)

**Source:** `crates/ironhermes-gateway/src/handler.rs` lines 343-368 [VERIFIED: direct read]

The full construction chain in `run_agent` (Telegram path):

```rust
// handler.rs:343-368 — canonical template
let base_url = self.config.resolve_base_url();
let api_key = self.config.resolve_api_key().unwrap_or_default();
let max_turns = self.config.agent.max_turns;

let client = LlmClient::new(base_url, api_key, &model);

let stream_tx_clone = stream_tx.clone();
let stream_callback: StreamCallback = Box::new(move |delta: &str| {
    let _ = stream_tx_clone.try_send(delta.to_string());
});

let tool_tx_clone = tool_tx.clone();
let tool_callback: ToolProgressCallback = Box::new(move |name: &str, _args: &str| {
    let _ = tool_tx_clone.try_send(name.to_string());
});

let mut agent = AgentLoop::new(client, self.tool_registry.clone(), max_turns)
    .with_streaming(stream_callback)
    .with_tool_progress(tool_callback);

if let Some(ref registry) = self.hook_registry {
    agent = agent.with_hook_registry(registry.clone());
}

let agent_result = agent.run(messages).await;
```

**For the cron path**, streaming is NOT needed (no Telegram adapter to push chunks to). The construction simplifies to:

```rust
// Cron-path construction — no streaming, no tool_progress callback
let base_url = config_tick.resolve_base_url();
let api_key = config_tick.resolve_api_key().unwrap_or_default();
let max_turns = config_tick.agent.max_turns;
let model = config_tick.model.default.clone();

let client = LlmClient::new(base_url, api_key, &model);

let mut agent = AgentLoop::new(client, tool_registry_tick.clone(), max_turns);

if let Some(ref registry) = hook_registry_tick {
    agent = agent.with_hook_registry(registry.clone());
}

let agent_result = agent.run(messages).await;
```

**Types for each field:**
| Field | Type | Where it comes from |
|-------|------|---------------------|
| `base_url` | `String` | `config.resolve_base_url()` → `Config::resolve_base_url(&self) -> String` |
| `api_key` | `String` | `config.resolve_api_key()` → `Config::resolve_api_key(&self) -> Option<String>` |
| `max_turns` | `usize` | `config.agent.max_turns` |
| `model` | `String` | `config.model.default.clone()` |
| `client` | `LlmClient` | `LlmClient::new(base_url, api_key, &model)` |
| `tool_registry` | `Arc<ToolRegistry>` | `self.tool_registry` on `GatewayRunner` |
| `hook_registry` | `Option<Arc<HookRegistry>>` | `self.hook_registry` on `GatewayRunner` |

**Is there a factory/builder?** No. Construction is inlined in `run_agent`. The cron path should inline it too (single use, ~12 lines).

### Q9: Platform Label for Cron Hook Events

**Source:** `crates/ironhermes-hooks/src/event.rs`, `handler.rs`, `agent_loop.rs` [VERIFIED: direct read]

**Platform strings currently in use:**
- `"telegram"` — used in `handler.rs:223`, `handler.rs:391`, `agent_loop.rs:132/201` (the agent path uses "agent" placeholder)
- `"agent"` — used by `AgentLoop::run()` internal fires (lines 132, 201) — these are the "wrong" empty values from the duplicate-events issue
- `"cron"` — appears in `HookEventKind::SkillActivated { source: "cron" }` (event.rs test at line 161) — the schema already anticipates cron as a source identifier

**Recommendation: Use `"cron"` as the platform label.** Reasoning:
1. `HookEventKind::SkillActivated { source: "cron" }` already uses `"cron"` as the cron identifier in the schema — consistent
2. Downstream consumers (JSONL, webhooks) can filter `platform == "cron"` vs `platform == "telegram"` without ambiguity
3. Using the delivery platform (e.g., `"telegram"` for jobs delivered to Telegram) would make cron-triggered runs indistinguishable from direct Telegram messages in event logs — wrong semantics
4. `"scheduled"` is wordier with no benefit over `"cron"`

**chat_id recommendation:** Use `job.id` as fallback if `job.origin` is `None`. If `job.origin` is `Some(origin)`, use `origin.chat_id`. This gives every event a non-empty `chat_id` that correlates back to the specific job.

```rust

    .map(|o| o.chat_id.clone())
    .unwrap_or_else(|| job.id.clone());
```

## Architecture Patterns

### The Full Replacement Block (lines 369-424 in runner.rs)

The complete capture + spawn block after modification:

```rust
// --- 10. Cron tick task ---
if let Some(ref job_store) = self.job_store {
    let tick_cancel = self.cancel.clone();
    let job_store_tick = job_store.clone();
    let skill_registry_tick = self.skill_registry.clone();
    // NEW: four additional captures
    let hook_registry_tick = self.hook_registry.clone();
    let tool_registry_tick = self.tool_registry.clone();
    let memory_store_tick = self.memory_store.clone();
    let config_tick = self.config.clone();

    join_set.spawn(async move {
        let mut interval = tokio::time::interval(
            tokio::time::Duration::from_secs(60)
        );
        interval.set_missed_tick_behavior(tokio::time::MissedTickBehavior::Skip);
        loop {
            tokio::select! {
                _ = tick_cancel.cancelled() => {
                    info!("Cron tick task shutting down");
                    break;
                }
                _ = interval.tick() => {
                    match ironhermes_cron::run_tick_check(&job_store_tick).await {
                        Ok((due_jobs, result, _lock_guard)) => {
                            if result.jobs_run > 0 {
                                info!("Tick: {} jobs due", due_jobs.len());
                            }
                            for job in &due_jobs {
                                info!("Job due: {} ({})", job.name, job.id);

                                // Resolve skill content (per D-02: rename _full_input → full_input)
                                let full_input = if let Some(ref skill_reg) = skill_registry_tick {
                                    let skill_context = resolve_skill_context(skill_reg, &job.skills);
                                    if skill_context.is_empty() {
                                        job.prompt.clone()
                                    } else {
                                        format!("{}\n\n---\n\n{}", skill_context, job.prompt)
                                    }
                                } else {
                                    job.prompt.clone()
                                };

                                // D-12: derive chat_id from job origin or job id
                                let cron_chat_id = job.origin.as_ref()
                                    .map(|o| o.chat_id.clone())
                                    .unwrap_or_else(|| job.id.clone());

                                // D-04/D-06/D-07: fire MessageReceived with cron metadata
                                let request_id = uuid::Uuid::new_v4().to_string();
                                if let Some(ref registry) = hook_registry_tick {
                                    registry.fire(ironhermes_hooks::HookEvent::new(
                                        &request_id,
                                        ironhermes_hooks::HookEventKind::MessageReceived {
                                            platform: "cron".to_string(),
                                            chat_id: cron_chat_id.clone(),
                                            content_preview: ironhermes_hooks::event::preview(&full_input, 200),
                                        },
                                    ));
                                }

                                // D-09/D-10/D-11: build fresh AgentLoop per job
                                let base_url = config_tick.resolve_base_url();
                                let api_key = config_tick.resolve_api_key().unwrap_or_default();
                                let max_turns = config_tick.agent.max_turns;
                                let model = config_tick.model.default.clone();

                                // Build system message via PromptBuilder
                                let cwd = std::env::current_dir().unwrap_or_default();
                                let mut prompt_builder = PromptBuilder::new(&model, "cron")
                                    .load_context(&cwd);
                                if let Some(ref store) = memory_store_tick {
                                    prompt_builder.set_memory_store(store.clone());
                                }
                                if let Some(ref skill_reg) = skill_registry_tick {
                                    prompt_builder.set_skill_registry(skill_reg.clone());
                                }
                                let system_msg = prompt_builder.build_system_message();

                                // Build user message
                                let user_msg = ironhermes_core::ChatMessage {
                                    role: ironhermes_core::Role::User,
                                    content: Some(ironhermes_core::MessageContent::text(&full_input)),
                                    tool_calls: None,
                                    tool_call_id: None,
                                    name: None,
                                };
                                let messages = vec![system_msg, user_msg];

                                let client = LlmClient::new(base_url, api_key, &model);
                                let mut agent = AgentLoop::new(client, tool_registry_tick.clone(), max_turns);
                                if let Some(ref registry) = hook_registry_tick {
                                    agent = agent.with_hook_registry(registry.clone());
                                }

                                // D-03/D-14: run agent, use real output and real success flag
                                match agent.run(messages).await {
                                    Ok(result) => {
                                        info!("Cron agent completed for job {}, turns_used={}", job.name, result.turns_used);
                                        let output = result.final_response
                                            .unwrap_or_else(|| "[No response generated]".to_string());

                                        // D-04/D-06: fire ResponseSent with cron metadata
                                        if let Some(ref registry) = hook_registry_tick {
                                            registry.fire(ironhermes_hooks::HookEvent::new(
                                                &request_id,
                                                ironhermes_hooks::HookEventKind::ResponseSent {
                                                    platform: "cron".to_string(),
                                                    chat_id: cron_chat_id.clone(),
                                                    response_preview: ironhermes_hooks::event::preview(&output, 200),
                                                },
                                            ));
                                        }

                                        if let Err(e) = ironhermes_cron::complete_job_run(
                                            &job_store_tick, job, &output, true
                                        ).await {
                                            error!("Failed to complete job run: {}", e);
                                        }
                                    }
                                    Err(e) => {
                                        error!("Agent error for cron job {}: {}", job.name, e);
                                        let error_output = format!("[Agent error: {}]", e);
                                        if let Err(ce) = ironhermes_cron::complete_job_run(
                                            &job_store_tick, job, &error_output, false
                                        ).await {
                                            error!("Failed to complete job run after agent error: {}", ce);
                                        }
                                    }
                                }
                            }
                        }
                        Err(e) => {
                            error!("Tick error: {}", e);
                        }
                    }
                }
            }
        }
    });
    info!("Cron tick task started (60s interval)");
}
```

### Anti-Patterns to Avoid

- **Sharing `AgentLoop` across jobs:** Each job must get a fresh `AgentLoop` per D-09. Do not construct once outside the `for job in &due_jobs` loop.
- **Sharing `AgentLoop` across ticks:** The agent holds mutable state (compressor, etc.). Construct per-tick inside the loop.
- **Unwrapping `hook_registry_tick` without the Option guard:** The registry is `Option<Arc<...>>` — always use `if let Some(ref registry) = hook_registry_tick` (D-08).
- **Passing `hook_registry.clone()` directly from `self` inside `spawn`:** `self` is not `Send`. Use the pre-captured `hook_registry_tick` variable.
- **Adding streaming callbacks for cron:** Cron jobs are headless — there's no adapter to stream to. Do NOT add `.with_streaming(...)` or `.with_tool_progress(...)` in the tick path.

## Common Pitfalls

### Pitfall 1: Missing `use` Imports
**What goes wrong:** `AgentLoop`, `LlmClient`, `PromptBuilder` are not currently imported in `runner.rs`. The file only imports `ironhermes_core::{Config, MemoryStore, SkillRegistry}` and `ironhermes_tools::ToolRegistry`.
**Why it happens:** The tick task was a stub — no agent code was needed.
**How to avoid:** Add to the top of runner.rs: `use ironhermes_agent::{AgentLoop, LlmClient, PromptBuilder};` and `use ironhermes_core::{ChatMessage, MessageContent, Role};`
**Warning signs:** `E0433: failed to resolve: use of undeclared crate or module` on first compile.

### Pitfall 2: Borrow of `self` inside `spawn` Closure
**What goes wrong:** The `join_set.spawn(async move { ... })` closure moves its captures. If you try to access `self.hook_registry` inside the closure, the compiler rejects it because `self` is not `Send` (or not available after move).
**Why it happens:** The existing pattern already handles this — `job_store_tick` and `skill_registry_tick` are pre-cloned before the spawn. The new fields must follow the same pattern.
**How to avoid:** ALL new fields (`hook_registry_tick`, `tool_registry_tick`, `memory_store_tick`, `config_tick`) MUST be cloned BEFORE `join_set.spawn(async move {`.

### Pitfall 3: `ChatMessage` vs `Vec<ChatMessage>` input to `agent.run()`
**What goes wrong:** `agent.run()` takes `Vec<ChatMessage>` — a full message history. Passing a single `ChatMessage` or a plain `String` fails to compile.
**Why it happens:** The cron path is adapting a "single prompt" job into the messages-format expected by the OpenAI API.
**How to avoid:** Build `vec![system_msg, user_msg]` as shown in the architecture pattern above. The system message comes from `PromptBuilder::build_system_message()`.

### Pitfall 4: `LockGuard` Drop Timing
**What goes wrong:** The `_lock_guard` from `run_tick_check` holds a file lock. If dropped too early, another process could start a concurrent tick. If held too long, it blocks the next tick.
**Why it happens:** The existing code already handles this correctly (the `_lock_guard` is dropped at the end of the `Ok(...)` match arm). The replacement code must not change where `_lock_guard` is bound or dropped.
**How to avoid:** Keep `_lock_guard` bound in the `Ok((due_jobs, result, _lock_guard)) =>` match arm as-is. Do not move it into the `for job` loop.

### Pitfall 5: `AgentLoop::run()` fires `MessageReceived`/`ResponseSent` with `platform: "agent"`
**What goes wrong:** After wiring `with_hook_registry(...)`, the AgentLoop will fire its own `MessageReceived` at agent_loop.rs:131 with `platform: "agent"` and `chat_id: ""`. This is in addition to the explicit `MessageReceived` fire with `platform: "cron"` shown in the architecture pattern.
**Why it happens:** The AgentLoop was designed for the Telegram path where the gateway fires the real-platform events — the agent's internal fires were always redundant. Phase 07.4 fixes this globally.
**How to avoid:** Accept the duplicate for now (Phase 07.4 scope). The cron path will produce 2x `MessageReceived` and 2x `ResponseSent` per job run — one pair with `"cron"` metadata (correct) and one with `"agent"/""` (legacy). Document in plan as known Phase 07.4 residual.

Alternative: Do NOT fire the manual `MessageReceived`/`ResponseSent` in runner.rs (like the handler.rs pattern), and instead rely solely on AgentLoop's internal fires. This produces events with `platform: "agent"` and empty `chat_id` — wrong per D-12. The explicit cron-labelled fires are recommended.
