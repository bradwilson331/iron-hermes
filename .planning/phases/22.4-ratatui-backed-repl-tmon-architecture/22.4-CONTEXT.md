**Gathered:** 2026-04-24

<domain>
## Phase Boundary

Replace `hermes chat`'s custom crossterm + rustyline + raw-ANSI REPL (the ~3,126-LOC `crates/ironhermes-cli/src/tui/` module built across Phases 21 → 22.1 → 22.3) with a ratatui-driven REPL modelled after the `tmon` reference architecture at `/Users/twilson/code/tmon/`, landing as a side-by-side `tui_rata/` module that defaults on for interactive TTY sessions while preserving the classic path as an explicit opt-out for one cycle.

**In scope:**
- New `crates/ironhermes-cli/src/tui_rata/` module implementing a full ratatui App (alt-screen, mouse capture, tokio-select event loop, tui-textarea input, transcript Paragraph with scroll/auto-follow, pill-style status line, knight-rider activity row, double-Ctrl-C state machine).
- Full feature parity with the existing `run_chat` wiring: AgentLoop streaming, HookRegistry (JSONL + webhook), MCP manager + `/reload-mcp`, memory_manager + MemoryTool, SubagentRegistry + TranscriptWriter, ProcessRegistry + `/agents` + `/stop`, 49-command slash router, typo suggester, BlocklistGuardrail, cron/skills/execute_code tools, `--yolo` gate + banner, CancellationToken cascade.
- Workspace crossterm bump from `0.28` → `0.29` (atomic migration across all current call sites).
- Dual-layer testing: static-grep INV-22.4-* regression tests over wiring call sites, plus ratatui `TestBackend` + `insta` snapshot tests of canonical frames.

**Out of scope (this cycle):**
- Deletion of `crates/ironhermes-cli/src/tui/` or `crates/ironhermes-cli/src/repl_input.rs` — classic-tui stays compilable for one cycle; deletion is a follow-up phase (22.5) gated on clean ratatui UAT.
- `run_single` and `run_gateway` integration — both are non-interactive (no REPL surface); they remain on the current non-TUI code path. Matches Phase 22.3 D-15 `run_chat`-only precedent.
- Inline viewport mode (`ratatui::Viewport::Inline`) — explicitly rejected in D-01 in favor of alt-screen.
- Retirement or redesign of the Phase 22.1 `TuiExtension` trait for external consumers — the trait stays exported at the crate boundary for classic-tui; `tui_rata/` simply does not call its widget-slot surface (D-09). A future phase redesigns the plugin contract around ratatui widgets if/when an extension consumer materialises.
- `/logs` slash command or Ctrl-G toggle surfacing the `tui-logger` panel — the crate is in the floor allowlist, but whether to expose a panel + binding is deferred to the planner or a follow-up phase.
- Config-driven backend selection (`cli-config.yaml repl.backend: classic | ratatui`) — CLI flag + env var are sufficient for one cycle.

<decisions>
## Implementation Decisions

### Viewport & migration shape

- **D-01:** **Alternate screen (tmon-style).** `tui_rata::run_chat` calls `enable_raw_mode()` + `execute!(stdout, EnterAlternateScreen, EnableMouseCapture)` on entry; reverses on exit. Scrollback is lost on `/quit` by design — transcript scroll inside the alt-screen region gives the same functional outcome during the session. Rejected: inline viewport (loses bordered-Block chrome, mouse-capture gotchas); hybrid dump-on-exit (deferred — nice-to-have).
- **D-02:** **Side-by-side module at `crates/ironhermes-cli/src/tui_rata/`.** The existing `tui/` module stays compilable and functional for this phase. Deletion of `tui/` + `repl_input.rs` + the old `run_chat` code path + `write_into_scroll_region` helper + INV-21.7/22.1/22.3-* tests is a follow-up phase gated on clean live-TTY UAT of tui_rata/.
- **D-03:** **Ratatui default, `--classic-tui` escape hatch.** `hermes chat` defaults to `tui_rata::run_chat`. Opt-out via `--classic-tui` CLI flag OR `IRONHERMES_CLASSIC_TUI=1` env var (CLI flag wins if both present). `print_banner()` still fires BEFORE entering alt-screen so the banner lands in scrollback pre-TUI.
- **D-04:** **Auto-fall-back on non-TTY (IsTerminal gate).** Before raw-mode setup, check `std::io::IsTerminal` on both stdin AND stdout. If either is non-TTY (piped, CI, redirected), silently route to the classic-tui path. No warning print. Matches Phase 21.7 D-14 `non-TTY IsTerminal gate` pattern.

### Input widget & rustyline fate

- **D-05:** **`tui-textarea` crate owns character editing.** App state stores submitted-entry history separately; `tui-textarea::TextArea` handles cursor, character editing, selection, Ctrl-A/E/U/K/W word nav. No rustyline in the ratatui path — rustyline's `Editor::readline()` wants to own the terminal and is structurally incompatible with alt-screen + frame-owned rendering.
- **D-06:** **Arrow-key binding — classic REPL muscle memory.** Up/Down in input = history recall (prior submitted entries, dedupe-consecutive). PageUp/PageDown + mouse wheel = transcript scroll. Home/End within the input line (tui-textarea default). Esc clears the input buffer. tmon's transcript-scroll-on-Up/Down is rejected — users expect history recall from bash/ipython/rustyline.
- **D-07:** **History persistence — reuse Phase 22.3 contract.** Path: `$HERMES_HOME/repl_history`. Format: rustyline plain-text (one entry per line). Max 1000 entries (matches 22.3 D-08). Dedupe-consecutive (HistoryDuplicates::Prev semantics). New module `tui_rata/history.rs` reads the file on App::new, writes on Shutdown/quit. Zero migration: existing classic-tui users' history carries over unchanged.
- **D-08:** **Enter submits; Shift+Enter / Alt+Enter inserts newline.** Modern chat-REPL convention (Claude Code, ChatGPT, Slack). Multi-line input supported via tui-textarea; on-disk serialisation uses `\u{1F}` (unit-separator) to encode newlines within a single history-file line so the rustyline-compatible format (D-07) is preserved. Reversible at load time.

### Port or retire Phase 21/22.1 investments

- **D-09:** **Port keybindings + command dispatch; retire widget slot system.** `KeybindingRegistry` (`tui/keybindings.rs`, 318 LOC) and `process_command` dispatch chain from Phase 22.1 are ported into `tui_rata/` because slash routing depends on them. The widget-slot layout system (AboveStatus / BelowScanner / StatusRight) is RETIRED in the ratatui path — ratatui's native `Layout + Constraint` replaces it. The `TuiExtension` trait stays exported for classic-tui consumers; `tui_rata/` does NOT call `register_extension()` or the trait's widget methods. No in-tree consumers break; external consumers (none known) must migrate when 22.1 is eventually redesigned.
- **D-10:** **Port Phase 21 status pill line verbatim.** Dot-separated pills (`mode · model · provider · tokens/limit · hint`), rotating `cyan / magenta / green / yellow / dimmed` color cycle. New `tui_rata/status_line.rs` renders as a ratatui `Line` of styled `Span`s inside the bottom layout block. Pill content derivation logic lifts from classic-tui `tui/status_line.rs`. Signature Phase 21 aesthetic preserved.
- **D-11:** **Port Phase 21 knight-rider scanner.** 10-cell triangle-wave animation during in-flight turns, 100ms tick, color rotation matching Phase 21 D-06/D-07. `tui/knight_rider.rs::frame()` is a pure function — lift 1:1 into `tui_rata/knight_rider.rs`. Rendered as a dedicated row above the status pill line (new Length(1) Constraint).
- **D-12:** **Port Phase 21 double-Ctrl-C state machine verbatim.** First Ctrl+C cancels in-flight turn via `child_cancel_token.cancel()` + returns to prompt (preserves history); second within 1.5s persists session as `interrupted` and exits; third within 3s = emergency escape. Pure state machine from `tui/double_ctrl_c.rs` lifts without changes. Integrates with AgentLoop's CancellationToken cascade from Phase 21.7. Note: Ctrl+L — currently unbound in classic-tui; in tui_rata/ it stays unbound at the tui-textarea layer (tui-textarea does not consume Ctrl+L by default) and is available for a future `/clear` shortcut if needed.

### Event loop shape

- **D-16:** **Async `tokio::select!` over `crossterm::event::EventStream`.** The main loop composes:
  ```
  loop {
      tokio::select! {
          Some(event) = events.next() => app.handle_event(event),
          Some(stream_event) = stream_rx.recv() => app.handle_stream_event(stream_event),
          _ = ctrl_c.recv() => app.handle_ctrl_c(),
          _ = tick.tick() => app.on_tick(),  // 100ms for knight-rider animation
      }
      app.reconcile_scroll(transcript_area);
      terminal.draw(|frame| ui(frame, &app))?;
      if app.should_quit { break; }
  }
  ```
  Requires crossterm `event-stream` feature (D-13). Composes naturally with AgentLoop / MCP / HookRegistry / `tokio::signal` async call graph. Sync tmon-style `event::poll(30ms)` is NOT adopted — IronHermes's host is already deeply async and bridging the gap is more work than adopting EventStream.

### Streaming-token wiring

- **D-17:** **`tokio::sync::mpsc::UnboundedSender<StreamEvent>` per turn.** When a user submits, the main loop spawns `tokio::spawn(agent_loop.run_agent_turn(..., callback))` where callback is `Box::new(move |delta| { let _ = tx.send(StreamEvent::Delta(delta.to_string())); })`. Main loop's `tokio::select!` arm `stream_rx.recv()` pushes each `Delta` into the App's `assistant_buffer` and triggers an implicit redraw on the next loop iteration. Per-turn `Sender` is dropped when the turn finishes — channel close signals "no more deltas this turn". `StreamEvent` variants (canonical list): `Started`, `Delta(String)`, `ToolCall { name: String }`, `ToolProgress { name: String, phase: String }`, `ToolResult { name: String, ok: bool }`, `Finished`, `Error(String)`, `Cancelled`. Planner may add variants with justification. `Arc<Mutex<AppState>>` and `tokio::sync::watch<AppState>` alternatives are explicitly rejected (lock contention / latest-only semantics wrong for streaming text).

## Specific Ideas

- **tmon's scroll logic is the transcript UX reference.** Port the shape of `auto_follow` + `transcript_max_scroll` + `reconcile_scroll` + `scroll_indicator` — users always see either "live" or "scroll N/M" in the transcript title. Under alt-screen, transcript-within-region scroll replaces terminal-scrollback.
- **`write_into_scroll_region` (Plan 22.3-11) becomes OBSOLETE under tui_rata/** because ratatui's `terminal.draw()` owns every paint. The helper stays alive in classic-tui for one cycle; deleted in the follow-up that removes `tui/`.
- **PaintCoordinator (Phase 22.3 UI-SPEC OWN-1..5) is absorbed** — ratatui enforces single-owner rendering by construction. No runtime enforcement needed in tui_rata/; OWN-1..5 become trivially true.
- **Banner-bleed D-02 probe from 22.3** — not applicable to tui_rata/. Ratatui redraws the full frame each tick; there is no scrollback to bleed into during a session. The probe's findings stay relevant only to classic-tui.
- **`IRONHERMES_CLASSIC_TUI` env var spelling** matches existing `IRONHERMES_HOME` / `IRONHERMES_RATATUI` (rejected) conventions.
- **Max history file size 1000** matches Phase 22.3 D-08 verbatim.
- **Multi-line history encoding** — on-disk `\u{1F}` (unit-separator) round-trips single-line file format ↔ multi-line entries. Backward-compatible: entries written by classic-tui (no newlines) round-trip unchanged.
- **Role colors in transcript** (tmon §transcript_text) — User=Cyan, Assistant=Green. Map to IronHermes conventions: User=Cyan, Hermes=Green, Tool=Yellow, Subagent=Magenta, Error=Red. System messages hidden from transcript (tmon pattern).
- **Layout Constraint recommendations** — `Min(5)` transcript + `Length(1)` knight-rider (only visible during in-flight — collapse to Length(0) when idle) + `Length(1)` status pills + `Length(3)` input textarea. Bottom-row help/hint line can live inside the status pills line or below the input; planner picks.
