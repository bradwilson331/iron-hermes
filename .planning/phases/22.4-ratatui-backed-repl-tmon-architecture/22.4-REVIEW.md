# Phase 22.4: Code Review Report

**Phase:** 22.4 — ratatui-backed REPL (tmon architecture)
**Depth:** standard
**Reviewed:** 2026-04-24
**Diff base:** `497942c`

## Summary

Phase 22.4 introduces the `tui_rata` module alongside the classic `tui` module (D-02 side-by-side) and makes ratatui the default for `hermes chat` (D-03). The implementation is structurally solid — the 14-item D-18 parity wiring block is present, the 4-arm `tokio::select!` loop follows the tmon template correctly, and the double-ctrl-c state machine is a clean pure port. All 25 `invariants_22_4` grep gates target meaningful surfaces.

Several correctness and quality issues warrant attention before the classic path can be safely retired. No secrets, no injection vectors, no unsafe blocks, no unchecked external-input deserialization hot paths.

## Critical Issues

### CR-01: Duplicated home directory creation — two call sites

**File:** `crates/ironhermes-cli/src/tui_rata/event_loop.rs:125-130` and `301-307`

`build_app_deps` creates `$HERMES_HOME/{cron,sessions,logs,hooks,memories,skills,workspace,subagent-transcripts}` with an inlined `for sub in &[...]` loop on lines 126-129, then immediately calls `ensure_home_dirs(&hermes_home)?` on line 130 — which contains the **identical** loop (lines 302-305). The second call is a no-op in practice (directories already exist), but the inlined loop on 125-129 defeats the abstraction `ensure_home_dirs` was meant to provide. If a future author updates one list but not the other (e.g., adds a new subdir), the two fall out of sync silently.

**Fix:** Delete the inlined loop and rely on the helper.

### CR-02: StreamEvent variants `ToolProgress` / `ToolResult` are never emitted

**Files:** `crates/ironhermes-cli/src/tui_rata/event_loop.rs:437-469`, `crates/ironhermes-cli/src/tui_rata/app.rs:435-444`

`spawn_turn` emits only 5 of the 8 D-17 canonical variants: `Started`, `Delta`, `Finished`, `Error`, `Cancelled`. `ToolCall`, `ToolProgress`, and `ToolResult` are never sent. `AgentLoop` exposes `with_tool_progress(ToolProgressCallback)` (`agent_loop.rs:294`) which `spawn_turn` does not wire. The `App::handle_stream_event` arms for those three variants are dead code — the status line never shows tool-name pills mid-turn, breaking the advertised D-17 contract. The `snap_tool_call_activity_row` snapshot passes only because the test calls `app.handle_stream_event(StreamEvent::ToolCall { .. })` directly rather than exercising the end-to-end path.

**Fix:** Wire the callbacks inside `spawn_turn` before building the `AgentLoop`, or document that Phase 22.4 intentionally ships with 5 of 8 variants live and file a follow-up.

## Warnings

### WR-01: `block_in_place` panics on current-thread runtime

**File:** `crates/ironhermes-cli/src/tui_rata/app.rs:331-347`

`dispatch_slash_blocking` calls `tokio::task::block_in_place(...)`, which panics on a current-thread runtime. Works today because `#[tokio::main]` with workspace `tokio.features = "full"` implies multi-threaded. Any future caller using `LocalSet`, `#[tokio::main(flavor = "current_thread")]`, or a single-thread test harness hits a panic on every slash submission. Fallback `Err(_) => { self.status.hint = format!("slash (test): {input}"); }` silently swallows command output in tests.

**Fix:** Pass slash input through a `mpsc::UnboundedSender<String>` to the event-loop layer and dispatch there with a proper `.await`.

### WR-02: `wrapped_line_count` uses `chars().count()` — wrong for ratatui wrap

**File:** `crates/ironhermes-cli/src/tui_rata/app.rs:625-639`

Ratatui wraps by **display width** (via `unicode-width`). `chars().count()` is incorrect for CJK (2 cols), emoji (2 cols), zero-width joiners, combining marks. The scroll indicator understates row count; auto-follow can latch into a false "at bottom".

**Fix:** Use `unicode_width::UnicodeWidthStr::width` or document the ASCII-only assumption and lock it in invariants.

### WR-03: `invoke_handler` drops `CommandContext`

**File:** `crates/ironhermes-cli/src/tui_rata/commands.rs:103-154`

`build_command_context` constructs a `CommandContext` with `agent_running` and optionally `with_mcp_reloader(mgr.clone())`. This is passed to `invoke_handler(name, _ctx)` — unused. A `/quit` while a turn is in-flight does not block; `/reload-mcp` returns `CommandResult::McpReload` without actually invoking `ctx.mcp_reloader`. Comments acknowledge "not yet wired in Phase 22.4" but the typed surface silently signals "handled".

**Fix:** Either wire the context, or make the stub visible at the type level.

### WR-04: `spawn_turn` clones `app.agent_loop` only to discard it

**File:** `crates/ironhermes-cli/src/tui_rata/event_loop.rs:435`

`let _ = app.agent_loop.clone();` allocates an Arc clone bound to `_`, dropped at end of statement. The comment claims "keep Arc alive for future method-call integrations" — misleading, since `_` drops immediately and `app.agent_loop` is already kept alive by the struct.

**Fix:** Delete the line.

### WR-05: `expect("checked above")` pattern racy under future re-entry

**File:** `crates/ironhermes-cli/src/tui_rata/event_loop.rs:356-361`

Current nested-`if` form invites future refactors where `is_some()` and `take()` could race if the body `.await`s. Also leaves an edge case where `pending_tx` is `Some` but `cancel_child` is `None` — then `pending_tx` stays stuck forever.

**Fix:**
```rust
if let (Some(tx), Some(cancel)) = (app.pending_tx.take(), app.cancel_child.clone()) {
    spawn_turn(app, tx, cancel);
}
```

### WR-06: `transcript_max_scroll` discounts height borders but not width

**File:** `crates/ironhermes-cli/src/tui_rata/app.rs:208-212`

`visible = area.height.saturating_sub(2)` correctly discounts top+bottom borders, but `transcript_line_count(area.width as usize)` does NOT subtract left+right borders. For a 80-wide pane, wrap should be at width 78. Compounds WR-02.

**Fix:**
```rust
let inner_width = area.width.saturating_sub(2) as usize;
let total = self.transcript_line_count(inner_width) as u32;
```

### WR-07: `ChatMessage` role mutation pattern fragile

**File:** `crates/ironhermes-cli/src/tui_rata/app.rs:358-386`, `398-400`

Pattern: `let mut msg = ChatMessage::user(&text); msg.role = Role::System;` — when `ChatMessage::system` already exists. Breaks if `::user` ever populates role-specific fields that `::system` zeroes out.

**Fix:** Use `ChatMessage::system(&text)` directly.

### WR-08: Snapshot test coverage gap — classic-vs-ratatui dispatch

**File:** `crates/ironhermes-cli/tests/tui_rata_snapshots.rs:40-145`

Missing frames:
1. `should_use_classic_tui` gate logic (non-TTY stdin, env var, CLI flag) — the D-03 acceptance condition, only exercised by structural grep-gate.
2. Slash-dispatch round-trip.
3. `KeyEventKind::Release` filter render path.
4. `ToolProgress` (declared, not emitted — see CR-02).

**Fix:** Add `snap_slash_handled_help` and `snap_slash_unknown_typo_hint`. Add behavioral test for `should_use_classic_tui`.

### WR-09: `invariant_22_4_13` prefix-match pitfall

**File:** `crates/ironhermes-cli/tests/invariants_22_4.rs:159-161`

`register_execute_code_tool` is a prefix of `register_execute_code_tool_with_process_registry`. A rename to `_v2` still satisfies the gate. Same concern for `invariant_22_4_14_yolo_banner_pre_alt_screen` (checks presence, not order relative to `ratatui::init()`).

**Fix:** Match on the full method name.

### WR-10: `EventStream` not pinned — inconsistency

**File:** `crates/ironhermes-cli/src/tui_rata/event_loop.rs:343-372`

`let mut events = EventStream::new();` followed by `events.next()` in `select!`. Works because `EventStream: Unpin`. Neighbour `ctrl_c` is explicitly `tokio::pin!(ctrl_c)` — inconsistent.

**Fix:** Pin both (`tokio::pin!(events);`) or add a short comment justifying the asymmetry.

## Info

### IN-01: Knight-rider ANSI → ratatui roundtrip wasteful
`knight_rider::frame` returns ANSI-escaped `String`; `render_knight_rider` re-parses via `ansi-to-tui` at ~10Hz. Rewrite `frame` to return `Vec<Span<'static>>` directly. Drops the `ansi-to-tui` dep from this path.

### IN-02: `ReplHistory::load` silently drops empty middle lines
Skipping blank lines is fine for corrupt-file tolerance. Add a one-line doc comment.

### IN-03: `format!("{}", n)` stylistic
Prefer `n.to_string()`. Minor clippy lint.

### IN-04: `help_entries` returns refs
`Vec<(String, &str, &KeyContext)>` ties result lifetime to `&self`. Consumers storing tuples outside borrow must clone. Consider owned return.

### IN-05: Test-support feature gate can hide regressions
Unit tests under `#[cfg(all(test, feature = "test-support"))]` silently skip without `--features test-support`. Ensure CI runs with the feature.

### IN-06: `SlashOutcome::Unknown` discards source input
`input: _` drops the slash string captured by the router. Revisit when slash telemetry lands.

### IN-07: `ResetTerminal` / `McpReload` arms are silent no-ops
Users typing `/reset` or `/reload-mcp` get no feedback. Add at minimum a status-hint acknowledgement.

### IN-08: Yolo banner printed twice
`main.rs` calls `print_yolo_banner_to_stderr(true)` before `run_chat_ratatui`; `build_app_deps` calls it again as "D-18 parity defensive fire". With `--yolo` the banner prints twice. Prefer a single call site in `main.rs` and drop the defensive fire (which also runs after `ratatui::init()` enters raw mode — stderr may interleave poorly).

### IN-09: `drain_retry_queue` result discarded
Awaited without `?` or log. If it returns `Result`, add `if let Err(e) = ... { tracing::warn!(...) }`.

### IN-10: `build_mcp_manager` drops malformed YAML silently
`serde_yaml::from_value::<McpServerConfig>(val.clone()).ok()` — failed parse, server vanishes from registry with zero feedback. Add `tracing::warn!` before the `.ok()`.

### IN-11: `invariant_22_4_15` counts `.child_token()` — test-code inflates count
The count of `.child_token()` includes a test-only call (`app.rs:824`). A production-call deletion still satisfies the gate. Strip `#[cfg(test)]` regions before counting, or assert on the production call site directly.

### IN-12: Same role-mutation pattern as WR-07 in test helpers
`test_message` uses `ChatMessage::user(&text); msg.role = ...`. Consistent with WR-07 fix.
