Today `spawn_turn` in tui_rata/event_loop.rs emits only 5 (Started, Delta, Finished, Cancelled, Error). The ToolCall, ToolProgress, and ToolResult arms are defined in stream_events.rs and handled in app.rs:435-444, but `AgentLoop::with_tool_progress()` is NEVER chained on the per-turn AgentLoop builder inside spawn_turn — so the production code path cannot construct any of the three variants. The snapshot test `snap_tool_call_activity_row` passes only because it directly injects `App::handle_stream_event(StreamEvent::ToolCall{..})`, bypassing spawn_turn entirely.

Fix:
1. **Extend AgentLoop** — add a small, symmetric `ToolResultCallback = Box<dyn Fn(&str, bool) + Send + Sync>` type and a `with_tool_result(callback)` builder method in agent_loop.rs. The callback MUST fire at every existing `fire_hook(HookEventKind::ToolCompleted { success, .. })` site (6 sites: lines 875, 906, 967, 976, 994, 1022), passing `(tool_name, success_bool)` before or after the hook fires (either ordering is fine — both observe the same completion event).
2. **Wire in spawn_turn** — inside the tokio::spawn closure in tui_rata/event_loop.rs:437-468, chain `.with_tool_progress(...)` and `.with_tool_result(...)` onto the `AgentLoop::new(...)` builder. Both closures clone the existing per-turn `tx`/`tx_delta` sender and forward `(name, preview)` → `StreamEvent::ToolCall { name }` + `StreamEvent::ToolProgress { name, phase: preview }`, and `(name, ok)` → `StreamEvent::ToolResult { name, ok }`.
3. **Lock wiring** — append INV-22.4-26 + INV-22.4-27 to invariants_22_4.rs to static-grep-assert the wiring.
