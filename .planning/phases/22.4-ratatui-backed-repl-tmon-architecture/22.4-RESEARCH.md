## Summary

Phase 22.4 replaces the ~3,126-LOC classic crossterm+rustyline+raw-ANSI REPL with a side-by-side ratatui-backed module at `crates/ironhermes-cli/src/tui_rata/`. The build is **well-supported by the ecosystem** — ratatui 0.30.0, crossterm 0.29.0, tui-textarea 0.7.0, ansi-to-tui 8.0.1, and tui-logger 0.18.2 form a mutually compatible stack and are what the tmon reference already uses in production. The tmon reference at `/Users/twilson/code/tmon/src/main.rs` is a faithful template for the `App` struct shape, StreamEvent enum, Layout geometry, scroll reconciliation, and terminal setup/restore — but its sync `event::poll(30ms)` loop must be swapped for an async `tokio::select!` (D-16) because IronHermes's host is deeply async and bridges to `AgentLoop::run_agent_turn` via a `tokio::sync::mpsc::UnboundedSender<StreamEvent>` (D-17).

The most important divergences from tmon: (1) IronHermes gets `ratatui::init()` / `ratatui::restore()` free — ratatui 0.30 ships a panic hook that calls `disable_raw_mode` + `LeaveAlternateScreen` before the default panic printer fires (D-15 primary path confirmed); (2) arrow-key semantics are inverted from tmon — Up/Down = history recall (D-06), not transcript scroll (PageUp/PageDown + mouse wheel own that); (3) Enter submits, Shift+Enter / Alt+Enter inserts newline (D-08) — tui-textarea's default binds `Ctrl+M` + `Enter` to `insert_newline()` so we MUST intercept Enter before delegating to textarea.input(); (4) multi-line input serialises to `\u{1F}` on disk to preserve the rustyline plain-text history contract from Phase 22.3 D-08.

**Primary recommendation:** Adopt the D-14 stack verbatim (ratatui 0.30.0 / crossterm 0.29.0 / tui-textarea 0.7.0 / ansi-to-tui 8.0.1 / tui-logger 0.18.2). Use `ratatui::init()` / `ratatui::restore()` as the primary panic-safe terminal lifecycle path. Structure `tui_rata/` as submodules-per-concern following `tui/`'s shape. Port pure-function cores (pills, knight_rider, double_ctrl_c, keybindings) source-verbatim. Build the main loop as a 4-arm `tokio::select!` with a per-turn `UnboundedSender<StreamEvent>` dropped on turn completion.

## ratatui 0.30 API Confirmation (D-15 panic/restore)

[VERIFIED: Context7 `/ratatui/ratatui` + docs.rs/ratatui/0.30.0] The 0.30 release ships the following top-level helpers:

- `ratatui::init() -> DefaultTerminal` — enables raw mode, enters alternate screen, installs a panic hook that restores terminal state before the default panic printer fires, and returns a `DefaultTerminal` (type alias for `Terminal<CrosstermBackend<Stdout>>`). **Panics** on init failure.
- `ratatui::try_init() -> io::Result<DefaultTerminal>` — fallible variant.
- `ratatui::restore()` — disables raw mode, exits alternate screen. Returns unit (infallible-ish — errors logged).
- `ratatui::try_restore() -> io::Result<()>` — fallible variant.
- `ratatui::run(|terminal| { ... })` — convenience wrapper combining init + restore with automatic cleanup on panic or error. **Recommended for simple apps; NOT used here** because our event loop is too structured to fit inside a closure.

**Panic hook behaviour (verified):** "installs a panic hook that restores terminal state before panicking occurs. The panic hook installation requires calling this function AFTER any other panic hooks you've installed, ensuring proper restoration order." Translation: if we want additional panic behaviour (e.g., log to file), install OUR hook FIRST, then call `ratatui::init()` — ratatui's hook wraps ours and delegates to the default printer last.

**Top-level items confirmed present in 0.30.0:**
- `Frame`, `Terminal`, `DefaultTerminal`
- `backend::CrosstermBackend`
- `layout::{Constraint, Direction, Layout, Position, Rect}`
- `style::{Color, Modifier, Style}`
- `text::{Line, Span, Text}`
- `widgets::{Block, Borders, Paragraph, Wrap}`
- `Frame::set_cursor_position(Position)` — confirmed via tmon's live usage at `/Users/twilson/code/tmon/src/main.rs:623`

**Feature flags (0.30):** `crossterm_0_29` is the default; a `crossterm_0_28` feature exists for backward compat. We use the default — D-13 aligns the workspace crossterm version at 0.29.

**Primary D-15 path:**


## crossterm 0.28 → 0.29 Migration (D-13)

[VERIFIED: Context7 + GitHub CHANGELOG] The 0.28 → 0.29 bump is **remarkably small** per the official changelog — the only documented breaking change is `KeyModifiers` Display impl fix (adds `+` between modifiers). Everything else in the migration (EventStream, KeyEventKind, EnterAlternateScreen etc.) was present and unchanged from 0.28.

### Breaking changes (actual, from source-of-truth)

1. **`KeyModifiers` Display impl** — [CITED: github.com/crossterm-rs/crossterm/CHANGELOG.md] "Correctly fix KeyModifiers Display impl Properly adding `+` in between modifiers (#979)". Only matters if you format `KeyModifiers` for display (e.g., rendering `Ctrl+Shift+A`). No IronHermes call sites use this form — the existing `tui/keybindings.rs` matches `KeyModifiers` by discrim, not by Display.

### Non-breaking (but phase-relevant) facts

2. **`EventStream` is in 0.28 AND 0.29** — gated on `event-stream` feature flag in both. The feature flag MUST be enabled in Cargo.toml (`features = ["event-stream"]`). Adds `futures` dep transitively (already in IronHermes workspace via `futures = "0.3"`).

3. **`KeyEventKind::{Press, Release, Repeat}`** — all three variants exist in 0.28 and 0.29. On Unix terminals in raw mode, crossterm emits **only `Press`** events by default unless `PushKeyboardEnhancementFlags` is issued. On Windows, **all three** can fire. The existing `tui/keybindings.rs` already discriminates via `kind: KeyEventKind::Press` (line 134) — ported verbatim under D-09 this is already correct.

   **Action:** tui_rata's `handle_key` must filter `if key.kind != KeyEventKind::Press { return; }` at the top (matches tmon line 312). Without this filter, Windows users see duplicated keystrokes.

4. **`EnterAlternateScreen` / `LeaveAlternateScreen` / `EnableMouseCapture` / `DisableMouseCapture`** — import paths unchanged: `crossterm::event::{EnableMouseCapture, DisableMouseCapture}` and `crossterm::terminal::{EnterAlternateScreen, LeaveAlternateScreen}`. Same in 0.28 and 0.29.

5. **`enable_raw_mode()` / `disable_raw_mode()`** — unchanged. Path: `crossterm::terminal::{enable_raw_mode, disable_raw_mode}`.

### API surface

```rust
// Minimal integration
use tui_textarea::{TextArea, Input, Key, CursorMove, Scrolling};

let mut textarea = TextArea::default();
textarea.set_block(Block::bordered().title("Prompt"));
textarea.set_cursor_line_style(Style::default());  // disable default underline on cursor line
textarea.set_placeholder_text("Type your message...");

// Rendering inside ui(frame)
frame.render_widget(&textarea, input_chunk);

// Event handling - two paths:
// Path A: full delegate (lets textarea handle Enter → insert_newline internally)
textarea.input(key_event);   // accepts crossterm::event::KeyEvent directly

// Path B: intercept, decide, then delegate (REQUIRED for D-08 Enter-submit semantics)
match (key.code, key.modifiers) {
    (KeyCode::Enter, m) if m.contains(KeyModifiers::SHIFT) || m.contains(KeyModifiers::ALT) => {
        textarea.insert_newline();       // explicit multi-line input
    }
    (KeyCode::Enter, _) => {
        submit_current_input();          // D-08 bare Enter → submit
    }
    (KeyCode::Up, _) => history_prev(),  // D-06 — NOT passed to textarea
    (KeyCode::Down, _) => history_next(), // D-06
    (KeyCode::Esc, _) => textarea.select_all() /* then */ textarea.cut(),  // D-06 clear
    _ => { let _ = textarea.input(key); }   // everything else → textarea
}
```

### Confirmed default keybindings in tui-textarea

| Key | Default action |
|-----|----------------|
| Enter, Ctrl+M | `insert_newline()` |
| Backspace, Ctrl+H | delete char before cursor |
| Delete, Ctrl+D | delete char after cursor |
| Ctrl+A, Home | move to line start |
| Ctrl+E, End | move to line end |
| Ctrl+K | delete to end of line |
| Ctrl+U | delete to start of line (NOT undo — undo is on a different binding) |
| Ctrl+W, Alt+H | delete word before cursor |
| Alt+F, Ctrl+Right | move forward one word |
| Alt+B, Ctrl+Left | move back one word |
| Arrow Left/Right | char nav |

**D-06 conflict:** tui-textarea's default Up/Down is line-nav; D-06 wants Up/Down to be **history recall**. Resolution: intercept Up/Down BEFORE calling `textarea.input()` (Path B above). When the textarea is single-line (no newlines), Up/Down goes to history; when multi-line and cursor is not on first/last line, Up/Down should move within textarea. Recommended: **always intercept; Up/Down is always history recall** (matches bash/ipython/rustyline muscle memory per D-06). If user wants line-nav within a multi-line buffer, they can use Alt+Up / Alt+Down or click (mouse positioning).

### Cursor position for `frame.set_cursor_position`

`TextArea::cursor()` returns `(row, col)` inside the textarea's local coordinate space. To convert to absolute frame coordinates:

```rust
let (cursor_row, cursor_col) = textarea.cursor();
let abs_x = input_chunk.x + 1 + cursor_col as u16;  // +1 for block border
let abs_y = input_chunk.y + 1 + cursor_row as u16;
frame.set_cursor_position(Position::new(abs_x, abs_y));
```

### Multi-line history encoding (D-08)

tui-textarea's internal representation is `Vec<String>` (one entry per line). `TextArea::lines()` returns `&[String]`. To serialize for history persistence:

```rust
fn encode_entry(textarea: &TextArea) -> String {
    textarea.lines().join("\u{1F}")  // U+001F Unit Separator
}
fn decode_entry(line: &str) -> Vec<String> {
    line.split('\u{1F}').map(String::from).collect()
}
```

The `\u{1F}` character is ASCII C0 control code "Unit Separator" — guaranteed never to appear in user-typed input (it's not a printable character). Rustyline's plain-text history format is LF-delimited entries; we read each LF-delimited line and split on `\u{1F}` to restore multi-line entries. Backward-compatible: Phase 22.3-written entries have no `\u{1F}` so `split('\u{1F}')` yields a single-element vec — indistinguishable from a fresh single-line entry.

## tokio::select! Event Loop (D-16)

[VERIFIED: via CONTEXT.md skeleton + Context7 ratatui async-app examples]

### Canonical shape

```rust
use crossterm::event::EventStream;
use futures_util::StreamExt;
use tokio::{signal, time};

let mut events = EventStream::new();
let mut tick = time::interval(Duration::from_millis(100));  // knight-rider cadence
tick.set_missed_tick_behavior(time::MissedTickBehavior::Skip);  // don't burst after a long turn
let ctrl_c = signal::ctrl_c();
tokio::pin!(ctrl_c);

loop {
    tokio::select! {
        maybe_event = events.next() => match maybe_event {
            Some(Ok(event)) => app.handle_event(event, transcript_area),
            Some(Err(e)) => { tracing::warn!("terminal event error: {e}"); }
            None => { app.should_quit = true; }  // stream ended
        },
        Some(stream_event) = stream_rx.recv() => {
            app.handle_stream_event(stream_event);
        },
        _ = &mut ctrl_c => {
            app.handle_ctrl_c();
        },
        _ = tick.tick() => {
            app.on_tick();   // knight-rider advance
        },
    }
    app.reconcile_scroll(transcript_area);
    terminal.draw(|f| ui(f, &app))?;
    if app.should_quit { break; }
}
```

### Cancellation-safety audit

[CITED: tokio::select! docs + tokio::sync::mpsc::UnboundedReceiver::recv docs]

| Branch | Cancel-safe? | Notes |
|--------|------|-------|
| `events.next()` (EventStream → futures Stream::next) | YES | Standard async Stream impl; crossterm's EventStream is documented cancel-safe |
| `stream_rx.recv()` (tokio::sync::mpsc::UnboundedReceiver::recv) | YES | Explicitly documented cancel-safe |
| `&mut ctrl_c` (tokio::signal::ctrl_c future, pinned) | YES | Future is cancel-safe if not moved; pinning + `&mut` preserves this |
| `tick.tick()` (tokio::time::Interval::tick) | YES | Documented cancel-safe |

**Pitfall:** Do NOT call `tokio::signal::ctrl_c()` inside the loop — it returns a new future each time. Install once, pin, reuse via `&mut`. The CONTEXT skeleton at D-16 shows `ctrl_c.recv()` as if it were a channel — **this is shorthand**. The actual pattern is `tokio::pin!(ctrl_c)` + `&mut ctrl_c` in the select arm. Planner must use the correct form.

**Alternative:** `tokio::signal::unix::signal(SignalKind::interrupt())` returns a stream — `sig_stream.recv()` is cancel-safe and channel-like. More awkward on Windows. We stick with `signal::ctrl_c()` + pin.

### Draw cadence

The recommended shape is **redraw every iteration** (the canonical shape above). Rationale:

1. ratatui's `terminal.draw()` is **double-buffered** — the backend diffs against the previous frame and only sends dirty cells to crossterm. Zero-cost if state hasn't changed.
2. Skipping a draw loses responsiveness to stream events that mutate `assistant_buffer`.
3. tmon draws every iteration (line 644) — proven pattern at 30ms cadence.

tmon's 30ms is the `event::poll` timeout — irrelevant to us because `tokio::select!` is event-driven, not poll-based. Our cadence is driven by the 100ms `tick.tick()` (knight-rider), EventStream events (user input), and streaming deltas (50-200/sec). In steady-state this averages one redraw per 5-20ms, well within terminal frame budgets.

### Pitfalls specific to combining EventStream + tokio::signal

[VERIFIED: crossterm issue tracker + community patterns]

1. **EventStream creation placement** — Call `EventStream::new()` **after** `enable_raw_mode()` (inside / after `ratatui::init()`). Otherwise you may miss early events.
2. **EventStream requires `event-stream` feature** in crossterm Cargo.toml — D-13 must include `features = ["event-stream"]`.
3. **Ctrl-C on Unix** — raw-mode terminals do NOT auto-translate Ctrl-C to SIGINT by default when `EnterAlternateScreen` is active. `tokio::signal::ctrl_c()` will NOT fire for user keystrokes in raw mode. The Ctrl-C key arrives via `events.next()` as `KeyEvent { code: Char('c'), modifiers: CONTROL, kind: Press }`, and THAT is what triggers the double-Ctrl-C state machine (D-12). The `signal::ctrl_c()` arm is a safety net for SIGINT from parent processes (e.g., shell sending SIGTERM to the tty process group) — rare but the parity-list requires it.
   **Takeaway:** The double-Ctrl-C machine MUST be wired on the `Event::Key(...)` match arm, not on `ctrl_c.recv()`. The `ctrl_c` arm is belt-and-braces for non-keyboard SIGINT sources only.

## Streaming mpsc Bridge (D-17)

### Shape

```rust
// Types
use tokio::sync::mpsc::{UnboundedSender, UnboundedReceiver, unbounded_channel};

#[derive(Debug)]
enum StreamEvent {
    Started,
    Delta(String),
    ToolCall { name: String },
    ToolProgress { name: String, phase: String },
    ToolResult { name: String, ok: bool },
    Finished,
    Error(String),
    Cancelled,
}

// Per-turn channel
let (tx, rx) = unbounded_channel::<StreamEvent>();
app.pending_rx = Some(rx);

// Wrap AgentLoop's streaming callback
let tx_clone = tx.clone();
let stream_cb = Box::new(move |delta: &str| {
    let _ = tx_clone.send(StreamEvent::Delta(delta.to_string()));
});

// Spawn the turn
let cancel = app.cancel_parent.child_token();
tokio::spawn(async move {
    let _ = tx.send(StreamEvent::Started);
    let result = agent_loop.run_agent_turn(
        /* args including stream_cb, cancel */
    ).await;
    match result {
        Ok(_)                             => { let _ = tx.send(StreamEvent::Finished); }
        Err(e) if cancel.is_cancelled()   => { let _ = tx.send(StreamEvent::Cancelled); }
        Err(e)                            => { let _ = tx.send(StreamEvent::Error(e.to_string())); }
    }
    // tx drops here → channel closes → rx.recv() returns None on main loop
});
```

### Backpressure

Unbounded is acceptable because:
- Max burst: ~200 tok/sec × ~10-50 bytes each = ~2-10 KB/sec. A turn lasts ~10-60 sec. Worst case ~600 KB buffered if the main loop fell behind. Realistic: main loop drains every 5-20ms (see Draw cadence), so drain rate dwarfs produce rate.
- Bounded would need a sensible capacity. `unbounded_channel()` sidesteps the capacity-tuning problem at the cost of potential memory growth **only** if the main loop stalls — which would be a bug, not backpressure.
- The AgentLoop callback is called from within the turn's execution — if we used bounded and the main loop stalled, `tx.send().await` would block the turn. Worse outcome than memory growth.

**Decision confirmed:** Unbounded is correct for D-17.

### Channel-close-signals-turn-end

Pattern works: when `tx` drops (at end of the spawned task), `rx.recv()` returns `None`. Main-loop `select!` arm `Some(stream_event) = stream_rx.recv()` ignores `None` (because of the `Some(...) = ...` pattern — the arm is disabled when the channel closes). This is why we ALSO send `StreamEvent::Finished` / `Error` / `Cancelled` before drop — the app needs to know which terminal state, not just "no more events."

**Pitfall:** if the spawned task panics before sending a terminal variant, the channel closes with nothing sent. Wrap the spawn body in `tokio::task::spawn` + `.catch_unwind()` OR ensure the turn's Result mapping is the last statement before tx drops. Tmon does not handle this — IronHermes adds `Cancelled` + explicit `Error(String)` + `Finished` to cover all paths, then relies on channel-close as belt-and-braces.

## Panic Hook Implementation (D-15 primary + fallback)

### Primary path — ratatui::init() is sufficient

[VERIFIED: docs.rs/ratatui/0.30.0 + Context7]

`ratatui::init()` installs a panic hook that:
1. Saves current panic hook
2. Installs a hook that calls `ratatui::restore()` (disables raw mode, leaves alt-screen) BEFORE calling the saved hook
3. Returns a `DefaultTerminal`

**Effect:** Any panic anywhere in the program restores the terminal first, then lets the default printer show the panic message.

**Recommended code:**

```rust
// tui_rata/event_loop.rs
pub async fn run_chat_ratatui(/* args */) -> Result<()> {
    let mut terminal = ratatui::init();           // installs panic hook
    let _mouse_guard = enable_mouse_capture()?;   // RAII guard
    let result = run_app_inner(&mut terminal, /* args */).await;
    drop(_mouse_guard);                           // disables mouse on normal exit
    ratatui::restore();                           // disables raw + leaves alt-screen
    result
}

struct MouseCaptureGuard;
fn enable_mouse_capture() -> Result<MouseCaptureGuard> {
    execute!(std::io::stdout(), EnableMouseCapture)?;
    Ok(MouseCaptureGuard)
}
impl Drop for MouseCaptureGuard {
    fn drop(&mut self) {
        let _ = execute!(std::io::stdout(), DisableMouseCapture);
    }
}
```

The RAII guard ensures mouse is disabled on both panic-path AND normal-path. On panic, the guard's Drop runs (stack unwind) before ratatui's panic hook fires — the order is fine because DisableMouseCapture commutes with LeaveAlternateScreen.

## Security Surface Notes (for planner threat_model blocks)

### Threat 1: Terminal left in broken state on panic

**Category:** Denial of service / User experience catastrophic failure
**Mitigation [locked by D-15]:** `ratatui::init()`'s panic hook + RAII mouse guard. Fallback `std::panic::set_hook` wrapping default printer.
**Test:** Deliberately trigger `panic!()` mid-session; confirm terminal usable afterwards (`reset` not required).

### Threat 2: Malicious ANSI escapes in tool output / subagent transcripts

**Category:** Tampering / Information disclosure
**Vector:** A tool spawned by the agent (e.g., `terminal("cat malicious-file.txt")`) outputs bytes containing cursor-move, scroll-region, clear-screen, or OSC-8 hyperlink escapes. If we naively `ansi_to_tui::into_text()` these and render, we rely on ansi-to-tui to strip non-SGR escapes.

**[VERIFIED: docs.rs/ansi-to-tui]** ansi-to-tui handles **SGR (Select Graphic Rendition) only** — colors and modifiers. Non-SGR escape sequences (cursor positioning, screen clear, scrollback-delete, OSC-8, OSC-52 clipboard-write) are **silently stripped** during parsing rather than passed through. This is the safe default.

**Residual threat:** A tool could still output printable Unicode that exploits terminal-level ambiguity (RTL override, zero-width spaces for command-injection hiding). ratatui's Paragraph widget renders unicode glyphs as-is; the terminal emulator, not ratatui, is responsible for rendering them. This is out of scope for phase 22.4.

**Mitigation:** Document in planner threat_model: rely on ansi-to-tui's non-SGR-strip behaviour. Spike test: feed a fixture byte stream with `\x1b]52;c;<base64>\x07` (OSC-52 clipboard write) and confirm ansi-to-tui strips it. If the spike shows ansi-to-tui passes through non-SGR escapes — add an upfront byte-scrubber.

### Threat 3: `EnableMouseCapture` leaves terminal consuming mouse events

**Category:** Availability (user can't select text with mouse)
**Mitigation:** RAII `MouseCaptureGuard` (see §Panic Hook above) + `ratatui::restore()` double-path teardown.

### Threat 4: History file (D-07) as info-disclosure sink

**Category:** Information disclosure
**Vector:** `$HERMES_HOME/repl_history` contains raw user inputs — includes secrets if user pastes passwords/tokens. Already present in Phase 22.3 contract.
**Mitigation (out of scope — carried forward from 22.3):** File permissions should be 0600 on Unix. Phase 22.3 D-08 should have locked this; if not, raise in planner as a leak.

### Threat 5: Unvalidated ANSI in user-submitted input

**Category:** Tampering
**Vector:** User pastes text containing ANSI escapes into tui-textarea. Would those escapes round-trip into the transcript and then into prompts sent to the LLM?
**Mitigation [inherent to tui-textarea]:** tui-textarea stores input as `Vec<String>`; escape bytes would be stored as literal characters (e.g., `\x1b` as a single `char`). When the submitted string is rendered via `Line::from(String)` (not `IntoText`), escapes are not interpreted — they're displayed as glyphs (replacement characters for non-printables). Not a concern.

## Common Pitfalls

### Pitfall 1: tui-textarea 0.7 declares `ratatui ^0.29` — Cargo resolution uncertainty

**What goes wrong:** Cargo's caret operator on `0.y.z` treats minor bumps as breaking. `ratatui ^0.29` means `>=0.29.0, <0.30.0` — does **not** include 0.30.x. If tui-textarea 0.7's Cargo.toml hasn't been bumped, `cargo add ratatui@0.30 tui-textarea@0.7` could either (a) fail to resolve, (b) pull in BOTH ratatui 0.29 AND 0.30 (duplicate compilation, duplicate types, compilation failure when the app passes a 0.30 `Frame` to a 0.29-expecting widget).

**Why it happens:** Semantic-versioning convention in the Rust ecosystem says 0.y bumps are breaking until 1.0. Widget crates sometimes lag the framework's minor bumps.

**How to avoid:**
1. **Spike test first.** `cargo add ratatui@0.30 tui-textarea@0.7` in a scratch crate; `cargo tree -p ratatui` — single version ⇒ we're fine; two versions ⇒ fallback.
2. **Fallbacks (in priority order):**
   - `tui-textarea = { version = "0.7", features = [...] }` if a 0.30-compat release exists on crates.io (check `cargo search tui-textarea` for a version declaring `ratatui ^0.30`)
   - `tui-textarea-2 = "0.10.2"` — maintained fork
   - Git dependency on main branch
3. **Warning signs:** `cargo tree | grep ratatui` showing two versions; linker errors mentioning "Frame" defined in two crates.

### Pitfall 2: Dual tracing-subscriber registration

**What goes wrong:** `crates/ironhermes-cli/src/main.rs:181-184` installs `tracing_subscriber::fmt()` unconditionally. If tui_rata then calls `tui_logger::TuiTracingSubscriberLayer` → `registry().with(layer).init()` — you get the "global subscriber already set" panic.

**Why it happens:** `tracing_subscriber::fmt().init()` calls `set_global_default`, which can only be called once.

**How to avoid:** Resolve the D-03 / D-04 backend choice BEFORE calling the subscriber init. If ratatui path: use registry + tui_logger layer only. If classic path: keep the existing fmt() subscriber. Both paths still honour `RUST_LOG`.

### Pitfall 3: EnterAlternateScreen + println inside the alt-screen region

**What goes wrong:** Any `println!("...")` call INSIDE the alt-screen will land in the frame buffer and get overwritten on the next `terminal.draw()` — or worse, corrupt the frame. Phase 22.3 already fought this war (the `write_into_scroll_region` helper exists for this exact reason).

**Why it happens:** `println!` writes to stdout; ratatui's `terminal.draw()` also writes to stdout. They race.

**How to avoid:**
1. Audit all `println!` / `eprintln!` / `print!` calls in the run_chat code path. Migrate to `tracing::info!` / `tracing::error!` (which flows to tui-logger via D-14) or to an App state mutation + frame re-render.
2. The `--yolo` banner is `eprintln!` to stderr BEFORE alt-screen entry — fine, it lands in scrollback.
3. `print_banner()` fires BEFORE `ratatui::init()` (D-03) — fine.
4. Subagent progress ticker currently uses `ExternalPrinter.println` — NOT applicable under ratatui (external printer is a rustyline concept). Migrate the ticker to a StatusLine-pill update + tracing log entry.
5. **Warning signs:** Visual artifacts during a subagent run; log lines visibly flickering.

### Pitfall 4: Arrow-key conflict between tui-textarea and history recall

**What goes wrong:** D-06 wants Up/Down = history recall. tui-textarea's default Up/Down = line-nav within textarea. If you forward ALL keys to `textarea.input(key)` (the "easy path"), history recall breaks.

**How to avoid:** Intercept Up/Down before the `textarea.input(key)` call. See the `handle_key` template in §tmon Pattern Extraction. Alternative: set `textarea.set_ignored_keys(...)` if the API exists — check tui-textarea docs during implementation.

### Pitfall 5: Enter-submit vs Enter-newline (D-08)

**What goes wrong:** Same as above, but for Enter. tui-textarea's default Enter = `insert_newline()`. Without interception, Enter never submits — the user's input accumulates as multi-line forever.

**How to avoid:** Intercept `(KeyCode::Enter, NO_MODIFIERS)` → submit. Intercept `(KeyCode::Enter, SHIFT|ALT)` → `textarea.insert_newline()`. Warning sign: "Pressing Enter does nothing."

### Pitfall 6: tokio::signal::ctrl_c() called inside the loop

**What goes wrong:** `tokio::signal::ctrl_c()` creates a new future each call. Repeatedly awaiting a new future per iteration works but is wasteful. More importantly, if you wrap it in `_ = tokio::signal::ctrl_c() => ...` inside `select!`, the future is dropped at the end of each iteration — on some platforms the signal handler may be torn down/re-installed repeatedly.

**How to avoid:** Create ONCE outside the loop; pin once; reuse via `&mut pinned`:
```rust
let ctrl_c = tokio::signal::ctrl_c();
tokio::pin!(ctrl_c);
loop {
    tokio::select! {
        ...
        _ = &mut ctrl_c => { /* handle */ }
        ...
    }
}
```

### Pitfall 7: KeyEventKind::Press filter missing on Windows

**What goes wrong:** On Windows, crossterm emits `KeyEventKind::{Press, Release, Repeat}` for every keystroke. Without a `kind: Press` filter, typing "hello" produces 10 events (5 Press + 5 Release), and your textarea renders "hheelllloo".

**How to avoid:** First line of `handle_key`: `if key.kind != KeyEventKind::Press { return; }`. This is already in tmon line 312.

### Pitfall 8: Streaming callback thread-safety

**What goes wrong:** `AgentLoop::run_agent_turn`'s streaming callback is `Box<dyn Fn(&str) + Send + Sync>`. The callback captures `tx.clone()` and calls `tx.send(StreamEvent::Delta(...))`. If the callback runs on a different tokio task than the App's main task — which it does — you need `UnboundedSender`, which IS `Send + Sync`. 

**Watch out for:** If the callback closure captures an `Arc<Mutex<App>>`, you now have lock contention (see D-17 rejection).

### Pitfall 9: `terminal.clear()` vs `/clear` semantics

**What goes wrong:** CONTEXT §Claude's Discretion says "`ResetTerminal` maps to `terminal.clear()` + redraw." Under ratatui, `Terminal::clear()` clears the entire backend buffer — next `draw()` repaints from scratch. This is fine on alt-screen (no scrollback to lose). BUT if a future phase adds inline viewport, `terminal.clear()` would nuke scrollback above the viewport.

**How to avoid:** For alt-screen this phase, `terminal.clear()` is correct. Comment the call site with "alt-screen only — inline viewport would need different handling."

### Pitfall 10: EventStream lifetime and Send bounds

**What goes wrong:** `EventStream` wraps an internal mio-based reactor. If the App struct is `Send` but holds `EventStream` as a field, compilation may fail depending on futures-util version.

**How to avoid:** Keep `EventStream` local to the event loop function — don't store it on App. Pattern:
```rust
async fn run_app(terminal: &mut DefaultTerminal, app: &mut App) -> Result<()> {
    let mut events = EventStream::new();  // local; not on App
    loop { tokio::select! { ... } }
}
```
Matches tmon's (sync) shape.

### Async event loop

```rust
// Source: CONTEXT D-16 skeleton + cancel-safety audit
use crossterm::event::EventStream;
use futures_util::StreamExt;
use tokio::{signal, sync::mpsc::UnboundedReceiver, time};
use std::time::Duration;

async fn run_app_inner(
    terminal: &mut DefaultTerminal,
    app: &mut App,
    mut stream_rx: UnboundedReceiver<StreamEvent>,
) -> Result<()> {
    let mut events = EventStream::new();
    let mut tick = time::interval(Duration::from_millis(100));
    tick.set_missed_tick_behavior(time::MissedTickBehavior::Skip);
    let ctrl_c = signal::ctrl_c();
    tokio::pin!(ctrl_c);

    loop {
        let transcript_area = compute_transcript_area(terminal.size()?);
        tokio::select! {
            maybe_event = events.next() => {
                if let Some(Ok(ev)) = maybe_event { app.handle_event(ev, transcript_area); }
            },
            Some(se) = stream_rx.recv() => app.handle_stream_event(se),
            _ = &mut ctrl_c => app.handle_ctrl_c_signal(),
            _ = tick.tick() => app.on_tick(),
        }
        app.reconcile_scroll(transcript_area);
        terminal.draw(|f| ui(f, app))?;
        if app.should_quit { break; }
    }
    Ok(())
}
```

### Streaming bridge (per-turn channel)

```rust
// Source: D-17 + tmon spawn pattern adapted to tokio+AgentLoop
use tokio::sync::mpsc::unbounded_channel;

fn submit_turn(app: &mut App) -> Result<()> {
    let (tx, rx) = unbounded_channel::<StreamEvent>();
    app.pending_rx = Some(rx);
    let agent_loop = app.agent_loop.clone();
    let cancel = app.cancel_parent.child_token();
    app.cancel_child = Some(cancel.clone());
    let tx_delta = tx.clone();
    let stream_cb = Box::new(move |delta: &str| {
        let _ = tx_delta.send(StreamEvent::Delta(delta.to_string()));
    });
    tokio::spawn(async move {
        let _ = tx.send(StreamEvent::Started);
        let r = agent_loop.run_agent_turn(stream_cb, cancel.clone()).await;
        let terminal = match r {
            Ok(_)                          => StreamEvent::Finished,
            Err(_) if cancel.is_cancelled() => StreamEvent::Cancelled,
            Err(e)                         => StreamEvent::Error(e.to_string()),
        };
        let _ = tx.send(terminal);
        // tx drops → channel closes → App knows turn is done
    });
    Ok(())
}
```

```rust
use ratatui::{Frame, layout::{Constraint, Direction, Layout, Position, Rect}, widgets::*};
use ansi_to_tui::IntoText;

fn ui(frame: &mut Frame, app: &App) {

        .direction(Direction::Vertical)
        .constraints([
            Constraint::Min(5),      // transcript
            Constraint::Length(1),   // knight-rider (Length(0) when idle)
            Constraint::Length(1),   // status pills

        .wrap(Wrap { trim: false })

    frame.render_widget(transcript, chunks[0]);

        Paragraph::new(app.status_line.render_line()),
        chunks[2],

    // Input (tui-textarea)
    frame.render_widget(&app.textarea, chunks[3]);

    // Cursor inside input
    let (row, col) = app.textarea.cursor();
    frame.set_cursor_position(Position::new(
        chunks[3].x + 1 + col as u16,
        chunks[3].y + 1 + row as u16,
    ));
}
```
