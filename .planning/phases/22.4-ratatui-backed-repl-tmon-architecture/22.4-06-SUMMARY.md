## §Cursor Offset Decision

Plan 22.4-05 Task 1 constructs `TextArea::default()` without calling `ta.set_block(...)` — so there is no border surrounding the textarea widget. Therefore `render_cursor` uses:

```rust
let cursor_x = area.x.saturating_add(col as u16);  // +0, no border
let cursor_y = area.y.saturating_add(row as u16);  // +0, no border
```

If plan 22.4-07 or later adds a `Block` to the textarea (e.g., for a visible input border), the offsets must change to `area.x + 1 + col` and `area.y + 1 + row`. Plan 22.4-10 snapshot tests will catch any mismatch.

## Decisions Made

- **Paragraph lock (WARNING-07):** Transcript must use `Paragraph::new(...)` in v1 per RESEARCH Open Question §4 commitment. Any future swap to `tui-scrollview` requires updating the acceptance grep and re-snapshotting plan 22.4-10.
- **Knight-rider idle = blank Paragraph:** Rendering `Paragraph::new("")` when idle keeps the layout stable (same chunk count, no reflow) instead of hiding the chunk, which ratatui's `Layout::split` does not support dynamically.
- **No cursor border offset:** TextArea has no Block wrapper → cursor sits at `(area.x + col, area.y + row)`. Documented inline and in §Cursor Offset Decision above.
