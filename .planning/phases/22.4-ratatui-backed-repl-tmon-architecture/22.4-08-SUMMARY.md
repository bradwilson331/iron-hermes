### Task 2: `should_use_classic_tui` helper + dispatch branches + tracing gate

**Helper function** inserted before `#[tokio::main]`:

```rust
fn should_use_classic_tui(cli: &Cli) -> bool {
    use std::io::IsTerminal;
    if cli.classic_tui { return true; }
    if std::env::var("IRONHERMES_CLASSIC_TUI").map(|v| v == "1").unwrap_or(false) { return true; }
    if !std::io::stdin().is_terminal() || !std::io::stdout().is_terminal() { return true; }
    false
}
```

**Tracing gate** (Pitfall 2 ‚Äî prevents double-subscriber panic):

**1. [Rule 1 - Bug] `mod tui_rata` in main.rs causes `crate::cli_args` resolution failure**

- **Found during:** Task 2 first build attempt
- **Issue:** Plan instructed adding `mod tui_rata;` to `main.rs`. The `event_loop.rs` was authored (Plan 22.4-07) using `crate::cli_args::Cli` and `crate::yolo::...` ‚Äî these paths resolve correctly from the lib crate (`lib.rs` has `pub mod cli_args; pub mod yolo;`) but NOT from the binary crate where `crate::` is `main.rs` root.
- **Fix:** Did NOT add `mod tui_rata` to `main.rs`. Instead, used `ironhermes_cli::tui_rata::run_chat_ratatui` (lib re-export path). Constructed `ironhermes_cli::cli_args::Cli` at each call site to bridge the private `main::Cli` ‚Üí lib-accessible `cli_args::Cli`.
