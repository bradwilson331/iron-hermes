# Phase 27.1.4.1.1 Plan 01: Transport-failure classifier for provider fallback (PROV-07) Summary

**One-liner:** Transport-level failure detection via `is_transport_failure` helper wired into `classify_llm_error`, activating provider fallback on connection refused / DNS / timeout / connection reset errors (reqwest 0.12 verified allowlist).

## What Was Built

`AgentLoop::classify_llm_error` previously returned `(true, false)` for any error without an extractable HTTP status code — meaning local-Ollama-down produced `MAX_RETRIES` retries against the dead endpoint with no fallback switch. This plan closes that gap:

1. **`is_transport_failure(err_str: &str) -> bool`** — private helper placed immediately after `extract_http_status` in `agent_loop.rs`. Lowercases the full `{err:#}` chain once (D-01a) and checks a 7-needle conservative allowlist (D-01): `error sending request for url`, `connection refused`, `dns error`, `failed to lookup address`, `connection reset`, `operation timed out`, `tcp connect error`. All needles verified against pinned reqwest 0.12.28 / hyper-util 0.1.20 source.

2. **Single edit site in `classify_llm_error`** — `return (true, false)` in the `None` branch changed to `return (true, Self::is_transport_failure(&err_str))`. The existing `err_str` binding is reused; the `match code` block is unchanged; `run()` is unchanged.

3. **`test_classify_other_error` repurposed (D-02)** — input changed from `"Connection refused: failed to connect to LLM"` (which would have flipped to `(true, true)`) to `"unexpected end of JSON input"` (a clearly non-transport string that continues to assert `(true, false)`). Committed in the same atomic commit as the helper (atomicity required — Pitfall 3).

4. **6 new unit tests (D-02a + optional guard):**
   - `test_classify_transport_request_send_marker` — marquee Ollama-down case with full chain
   - `test_classify_transport_connection_refused` — `"tcp connect error"` + `"Connection refused"` needles in isolation
   - `test_classify_transport_connect_timeout` — `"tcp connect error"` + `"Operation timed out"` needles
   - `test_classify_transport_dns_failure` — `"dns error"` + `"failed to lookup address"` needles
   - `test_classify_transport_connection_reset` — `"connection reset"` needle alone (mid-stream reset via Kind::Body)
   - `test_classify_sse_read_timeout_not_transport` — Pitfall 1 regression guard; `"SSE stream read timed out after 60s"` asserts `!should_fallback`

5. **`crates/ironhermes-agent/tests/invariants_27_1_4_1_1.rs`** — Two static-grep tests using `include_str!("../src/agent_loop.rs")` that fail the build if `fn is_transport_failure(` or `"error sending request for url"` disappear from `agent_loop.rs`. Modeled on `invariants_27_1_4_1.rs`. No `use` imports, no dev-deps.

## Allowlist Notes

- Helper name chosen: `is_transport_failure` (plan recommendation, mirrors `extract_http_status` naming)
- `connection timed out` was **omitted** from the allowlist per RESEARCH Open Question 1 — for the Ollama use case `reqwest::ClientBuilder::connect_timeout` fires before the kernel-level TCP timeout; `"operation timed out"` covers the reqwest-controlled path. Can be added in a future patch if integration testing on Linux reveals the kernel-path fires.
- `run()` was **not modified** — `should_retry=true` is preserved in the transport branch so the no-fallback path (no `fallback_providers` configured) is behaviorally unchanged.
