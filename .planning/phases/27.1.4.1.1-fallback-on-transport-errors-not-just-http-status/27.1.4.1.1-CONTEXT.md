# Phase 27.1.4.1.1: fallback-on-transport-errors-not-just-http-status - Context

**Gathered:** 2026-05-12
**Status:** Ready for planning

<domain>
## Phase Boundary

`AgentLoop` provider fallback must fire on transport-level failures (provider unreachable / request never delivered / no response), not just on extractable HTTP status codes.

Today `AgentLoop::classify_llm_error(err)` does `format!("{err:#}")`, calls `extract_http_status`, and:
- if a status is found → matches `429|5xx → (true, true)`, `400|401|403|404 → (false, true)`, other codes → `(true, false)`
- if **no** status is found → returns `(true, false)` — retry only, no fallback

So a `Connection refused`, connect timeout, DNS failure, connection reset, or reqwest `error sending request for url` (none of which carry an HTTP status) means the loop retries the dead endpoint `MAX_RETRIES` (3) times and then errors — the configured `fallback_providers` chain, wired onto the `AgentLoop` in Phase 27.1.4.1, never activates. Real-world repro: local-Ollama primary + OpenRouter fallback with Ollama not running → gateway hammers `localhost:11434` 3× then errors instead of switching to OpenRouter.

**Fix (scoped):** In `classify_llm_error`, when `extract_http_status` returns `None`, run a new transport-failure detection helper over the same `format!("{err:#}")` string; on a match return `(true, true)` instead of `(true, false)`. No changes to `run()` — returning `should_fallback=true` is a no-op when `fallback_client` is `None`, so the no-fallback path is unchanged.

**Deliverables:**
1. `agent_loop.rs`: new private transport-failure detection helper (sits beside `extract_http_status`), consulted only when `extract_http_status` returns `None`.
2. `classify_llm_error`: in the `None` branch, return `(true, true)` when the helper matches, `(true, false)` otherwise.
3. Unit tests for connect-refused / connect-timeout / DNS-failure / connection-reset / reqwest `error sending request for url` error strings (each → `(true, true)`), plus a still-generic non-transport error → `(true, false)`.
4. Repurpose the existing `test_classify_other_error` so it no longer uses a transport string (see D-02).
5. New static-invariant grep test `crates/ironhermes-agent/tests/invariants_27_1_4_1_1.rs`, modeled on `invariants_27_1_4_1.rs`.
6. Do not regress the other existing `classify_*_error` tests (`test_classify_429_error`, `test_classify_401_error`, `test_classify_walks_anyhow_context_chain`, `test_classify_production_error_format`, `test_classify_500_error`, `test_classify_500_error`).

**Out of scope:** Any change to `run()`'s fallback/retry block; structured `reqwest::Error` downcasting (see Deferred); cron/subagent/batch wiring (closed in 27.1.4.1); the `extract_http_status` HTTP path (validated in Phase 12).

</domain>

<decisions>
## Implementation Decisions

### Transport-failure detection
- **D-01:** **Conservative substring allowlist** — not "any no-status error → fallback". The helper matches a specific set of transport/connectivity markers in `format!("{err:#}")`; anything it doesn't recognize keeps the current `(true, false)` (retry, no fallback). Starting allowlist (researcher to validate exact spellings against reqwest/hyper/std::io and OS errno strings):
  - `connection refused`
  - connect timeout — `operation timed out`, `connect timed out`, `timed out` (scoped to the connect context, not a general "timed out" anywhere)
  - DNS failure — `dns error`, `failed to lookup address`, `name resolution`, `failed to resolve`
  - `connection reset`
  - reqwest transport marker — `error sending request for url` (this is the hardest one to reproduce in a unit test and the one that hit production; it MUST be in the list)
- **D-01a:** Marker matching should be **case-insensitive** (lowercase the haystack once, compare lowercase needles) — OS errno strings vary in casing ("Connection refused" vs "connection refused"). `extract_http_status` stays as-is (fixed `(NNN Reason)` / `status: NNN` formats); only the new transport helper is case-insensitive.

### `should_retry` for transport errors
- **D-03:** Detected transport markers return `(should_retry = true, should_fallback = true)` — i.e. `(true, true)`. Rationale: when a fallback IS configured, `run()` checks fallback before retry so it switches immediately anyway; when no fallback is configured, behavior is **unchanged** — still retries `MAX_RETRIES` (3) times with 0.5s/1s/1.5s backoff then errors. Keeping `should_retry=true` means zero behavior change to the no-fallback path. Fail-fast (`(false, true)`) was considered and rejected as an unnecessary behavior change for this phase.

### Existing test collision
- **D-02:** `test_classify_other_error` currently feeds `"Connection refused: failed to connect to LLM"` and asserts `should_fallback == false` — that assertion is wrong once D-01 lands. **Repurpose it**: change the input to a clearly non-transport error string (e.g. `"unexpected end of JSON"` or `"invalid response body from provider"`) and keep its intent — a truly generic error → `(true, false)`, no fallback. Do not delete it; the "generic → no fallback" bucket still exists under the conservative allowlist (D-01).
- **D-02a:** Add a *new* test that feeds `"Connection refused"` (and the other transport markers) and asserts `(true, true)` — i.e. the behavior `test_classify_other_error` used to cover for that string is replaced, not lost.

### Static-invariant test
- **D-04:** New file `crates/ironhermes-agent/tests/invariants_27_1_4_1_1.rs` (no dev-deps, `include_str!` pattern, modeled on `invariants_27_1_4_1.rs`). It asserts **both**:
  1. `agent_loop.rs` contains the transport-detection helper's `fn` name (planner picks the name — e.g. `is_transport_failure` / `classify_transport_error`; whatever name ships, the test locks it).
  2. `agent_loop.rs` contains the literal `error sending request for url` — locks the one reqwest marker that's hardest to reproduce in a unit test so a refactor can't silently drop it.
  Failure messages should name PROV-07 and cite phase 27.1.4.1.1, matching the style of `invariants_27_1_4_1.rs`.

### Claude's Discretion
- Exact `fn` name for the transport-detection helper (D-04 locks whatever it ends up being).
- Exact final allowlist spellings — D-01 gives the starting set; researcher verifies against current reqwest/hyper/std::io error formats and adds variants if needed. The list MUST include `error sending request for url`, `connection refused`, a connect-timeout marker, a DNS marker, and a connection-reset marker.
- Whether the helper takes `&str` (the already-formatted `format!("{err:#}")` string, reused from `classify_llm_error`) or `&anyhow::Error` — prefer `&str` to reuse the existing format and keep it trivially unit-testable, matching `extract_http_status`'s `&str` signature.
- Whether to place the new transport tests in the existing `mod` alongside `test_classify_*` in `agent_loop.rs` or a small new `mod` — either is fine; co-locating with the other `classify_*` tests is the obvious choice.

</decisions>

## Existing Code Insights

### Reusable Assets
- `extract_http_status(err_str: &str) -> Option<u16>` (agent_loop.rs ~606) — the new transport helper mirrors its `&str` signature and sits right next to it; it's the gate (`None` → consult transport helper).
- `format!("{err:#}")` inside `classify_llm_error` — already produces the full anyhow-chain string; reuse the same value for the transport scan instead of re-formatting.
- `run()` fallback/retry block (agent_loop.rs ~877–907) — already does everything needed when `should_fallback=true`: takes the fallback client (one-shot), swaps `self.client`, sets `fallback_activated`, resets `retry_count`, `continue`. **No change required here** — that's the whole point of putting the fix in the classifier.

### Established Patterns
- **`(should_retry, should_fallback)` tuple contract** — `classify_llm_error` returns this; `run()` checks `should_fallback` *before* `should_retry`. New transport markers → `(true, true)`.
- **Static-invariant grep tests** — `include_str!("../src/<file>.rs")` + `assert!(SOURCE.contains("<literal>"), "<message naming the requirement + phase>")`, no dev-deps. `invariants_27_1_4_1.rs` (agent crate) is the closest model.
- **`{err:#}` for error classification** — anyhow alternate Display walks the full context chain; plain `{err}` only shows the outermost frame. The transport helper must operate on the `{err:#}` string, same as the HTTP-status scan, or wrapped transport errors will be missed.
- **`test_classify_*_error` unit tests** — construct `anyhow!("…")`, call `AgentLoop::classify_llm_error(&err)`, assert the tuple. New transport tests follow the same shape.

### Integration Points
- Single edit site in `classify_llm_error`: the `else` arm of `let Some(code) = Self::extract_http_status(&err_str) else { … }`. Currently `return (true, false);` → becomes `return (true, Self::is_transport_failure(&err_str));` (or equivalent), where the new helper returns `true` on an allowlist match.
- New test file `crates/ironhermes-agent/tests/invariants_27_1_4_1_1.rs` — sibling of `invariants_27_1_4_1.rs`; no Cargo.toml change needed (tests/ files are auto-discovered).

<specifics>
## Specific Ideas

- **`error sending request for url` is the marquee marker.** It's the reqwest error text that hit production (Ollama-down repro), and it's the hardest to reproduce faithfully in a unit test — so it gets belt-and-suspenders coverage: a unit test that feeds that exact string AND a static-grep invariant (D-04) that locks the literal into `agent_loop.rs`.
- **Conservative on purpose.** The user explicitly chose the allowlist over "any no-status error → fallback" — a malformed-response / serde error in the LLM call path should *not* trigger a provider switch, only genuine connectivity failures should.
- **`test_classify_other_error` keeps its job.** Don't delete it — repurpose its input to a non-transport string so the "generic error → no fallback" guarantee stays under test (D-02).

</specifics>

<deferred>
## Deferred Ideas

- **Structured `reqwest::Error` source-chain inspection** — instead of (or in addition to) substring matching, downcast the anyhow chain to `reqwest::Error` and check `is_connect()` / `is_timeout()` / `is_request()`. More robust than strings, but only works if the underlying `reqwest::Error` survives the anyhow wrapping (needs verification), and it's a bigger change than this phase warrants. Revisit if string-matching proves brittle across reqwest versions / platforms.
- **Fail-fast for transport errors when no fallback is configured** — making transport markers return `(false, true)` so a `connection refused` errors immediately instead of after ~3s of pointless retries. Rejected for this phase (D-03) as an unnecessary behavior change; reconsider if the wasted-retry latency ever matters.
- **Honoring fallback chains longer than one entry** — `run()` currently activates exactly one fallback (`fallback_client.take()` is one-shot). Walking `fallback_providers[1..]` on the fallback's own failure is a separate enhancement, untouched here.
