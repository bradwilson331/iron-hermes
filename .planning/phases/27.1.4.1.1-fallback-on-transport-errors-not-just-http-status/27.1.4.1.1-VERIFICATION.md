# Phase 27.1.4.1.1: Verification Report

**Phase Goal:** `AgentLoop` provider fallback fires on transport-level failures (provider unreachable / request never delivered / no response), not just on extractable HTTP status codes. In `classify_llm_error`, when `extract_http_status` returns `None`, detect transport-failure markers (Connection refused, connect timeout, DNS failure, connection reset, reqwest `error sending request for url`) and return `should_fallback=true` so the loop resets `retry_count` and switches to the fallback client; behavior unchanged when no fallback is configured. Add unit tests for connect-refused / connect-timeout / DNS / connection-reset error strings + a static-invariant grep test; do not regress the existing `classify_*_error` tests.

#### 1. Live end-to-end fallback activation

**Test:** Configure IronHermes with a local Ollama primary endpoint (e.g. `http://localhost:11434`) that is NOT running, and a valid `fallback_providers` entry (e.g. OpenRouter). Send a message and observe the provider-switch behavior.

**Expected:** The AgentLoop switches to the fallback provider after the first transport failure on the Ollama endpoint. It does NOT retry `MAX_RETRIES` (3×) against the dead Ollama endpoint before erroring out. The response is delivered via the fallback provider.

**Why human:** Live socket + process state required. The transport-failure classifier is fully covered by unit tests that feed synthetic error strings. However, the end-to-end path — real TCP connection refused by OS, reqwest wrapping it into an anyhow chain, that chain flowing through `classify_llm_error`, then `run()` actually calling `fallback_client.take()` and resetting `retry_count = 0; continue` — requires a running process with a dead primary endpoint.
