# Phase 27.1.4.1.1: fallback-on-transport-errors-not-just-http-status - Research

**Researched:** 2026-05-12
**Domain:** Rust error-chain string spellings — reqwest 0.12 / hyper-util / anyhow `{:#}` — for conservative transport-failure substring matching in `classify_llm_error`
**Confidence:** HIGH (all error-string claims verified against pinned source in `~/.cargo/registry`)

## Summary

This phase adds one private helper function and one edit site to `agent_loop.rs`, plus a new static-invariant test file. The research question is narrow: **what exact substrings does `format!("{err:#}")` contain for each failure mode?** Everything else (code shape, test pattern, integration point) is already precisely described in CONTEXT.md.

The anyhow `{:#}` alternate-Display format joins the full context chain with `": "` separators (verified in `anyhow-1.0.100/src/fmt.rs` line 13). A transport failure in the streaming path wraps as: `Streaming LLM call failed: Failed to send streaming request: error sending request for url (<url>): <hyper-util ConnectError msg>: <std::io::Error text>`. The reqwest `Kind::Request` Display always begins with `"error sending request"` and, when a URL is attached, appends `" for url (<url>)"` — making `"error sending request for url"` a reliable marker for the entire `Kind::Request` family.

The Conservative substring allowlist in D-01 is validated and refined below. Every marker is confirmed against pinned source. The `"error sending request for url"` string is exactly right and present in reqwest 0.12.28. Connect-phase timeout produces `"tcp connect error"` (not `"operation timed out"` — that is the request-level tower timeout). DNS failure produces `"dns error"` (from hyper-util `ConnectError::dns()`). Connection refused and reset appear as OS errno strings in the chain tail and are matched case-insensitively.

**Primary recommendation:** Implement `is_transport_failure(err_str: &str) -> bool` with case-insensitive matching on the verified allowlist. The single edit site in `classify_llm_error` changes `return (true, false)` to `return (true, Self::is_transport_failure(&err_str))`.

### The full `{err:#}` chain for streaming-path transport errors

The chain is built in this order (outermost to innermost):

```
"Streaming LLM call failed"           ← agent_loop.rs:1034 .context(...)
": Failed to send streaming request"  ← client.rs:164 .context(...)
": error sending request for url (http://localhost:11434/v1/chat/completions)"
                                       ← reqwest Kind::Request Display + url suffix
": <ConnectError msg>"                 ← hyper-util ConnectError Display (just the msg field)
": <std::io::Error text>"             ← OS errno string
```

For non-streaming path (`call_llm`), the outer context is `"LLM call failed"` (agent_loop.rs:1013) instead of `"Streaming LLM call failed"`, and the client wraps with `"Failed to send chat completion request"` (client.rs:99). The reqwest layer and below are identical.

### Verified marker spellings

#### 1. Connection refused

**std::io::Error Display text:**
- macOS: `"Connection refused (os error 61)"` — capital C
- Linux: `"Connection refused (os error 111)"` — capital C

**After hyper-util wrapping:**  ConnectError Display = `"tcp connect error"` (the `msg` field in `ConnectError`). The io::Error is the `source`, not part of the Display string — it appears in `{err:#}` because anyhow's chain expansion walks `std::error::Error::source()` recursively.

**After reqwest wrapping:** `"error sending request for url (<url>)"` (reqwest `Kind::Request` with URL attached)

**Full `{err:#}` contains:**
- `"error sending request for url"` (reqwest level) ← **primary marker, always present**
- `"tcp connect error"` (hyper-util ConnectError level)
- `"connection refused"` (std::io::Error level, lowercase after `.to_lowercase()`)

**Recommended allowlist match for this failure mode:** `"error sending request for url"` catches it. `"connection refused"` is a belt-and-suspenders addition. [VERIFIED: reqwest-0.12.28/src/error.rs line 231, 268; hyper-util-0.1.20/src/client/legacy/connect/http.rs line 947, 787]

#### 2. Connect timeout (TCP connect phase times out)

hyper-util (http.rs line 940-943) applies the connect timeout with `tokio::time::timeout(dur, connect).await` and on `Err(elapsed)` wraps it as:
```rust
Err(io::Error::new(io::ErrorKind::TimedOut, e))
```
This `io::Error` text is `"timed out"` or platform-specific — it is the Elapsed display wrapped in an io::Error. That io::Error is then wrapped by `ConnectError::m("tcp connect error")`.

**IMPORTANT DISTINCTION:** `"operation timed out"` is reqwest's own `TimedOut` sentinel (reqwest error.rs line 386) and is emitted by the **request-level tower timeout** (reqwest async_impl/client.rs line 3048/3056), NOT by the TCP connect-phase timeout. These are two different code paths.

**Full `{err:#}` for connect timeout contains:**
- `"error sending request for url"` ← still present (reqwest Kind::Request)
- `"tcp connect error"` (hyper-util level)
- `"timed out"` or OS timeout text (io::Error level)

**For request-level timeout (tower):** reqwest wraps as `Kind::Request` + source = `reqwest::TimedOut`, so `{err:#}` contains `"error sending request for url"` and `"operation timed out"`.

**Conclusion:** Both connect-phase and request-level timeouts surface `"error sending request for url"` at the reqwest level. Adding `"operation timed out"` as a separate allowlist entry is still correct but strictly redundant given `"error sending request for url"`. [VERIFIED: reqwest-0.12.28/src/error.rs lines 231, 268, 382-388; hyper-util-0.1.20/src/client/legacy/connect/http.rs lines 939-947]

#### 3. DNS resolution failure

hyper-util `ConnectError::dns()` (http.rs line 664-668):
```rust
fn dns<E>(cause: E) -> ConnectError {
    ConnectError::new("dns error", cause)
}
```
`ConnectError::Display` just writes `self.msg` — so it always displays exactly `"dns error"`.

The cause (from GaiResolver, which calls `getaddrinfo` in a thread pool) is a `std::io::Error`. On macOS the io::Error text is `"failed to lookup address information: nodename nor servname provided, or not known"`. On Linux it is typically `"failed to lookup address information: Name or service not known"` (glibc errno string).

**Full `{err:#}` for DNS failure contains:**
- `"error sending request for url"` ← reqwest level
- `"dns error"` ← hyper-util ConnectError level
- `"failed to lookup address information"` ← std::io::Error level (both platforms)
- Platform-specific suffix after the colon

**Recommended allowlist additions:** `"dns error"` is the most reliable (exact static string from hyper-util source). `"failed to lookup address"` is a belt-and-suspenders addition. [VERIFIED: hyper-util-0.1.20/src/client/legacy/connect/http.rs lines 664-668, 467; connect/dns.rs - GaiResolver confirmed as default resolver]

#### 4. Connection reset by peer

This occurs after a TCP connection is established but reset before or during the response. It is an `io::Error` wrapping through hyper (Kind::Io → `"connection error"` display in hyper 1.9.0/src/error.rs line 523) and then reqwest `Kind::Body` or `Kind::Request`.

**std::io::Error Display text:**
- macOS: `"Connection reset by peer (os error 54)"` — capital C
- Linux: `"Connection reset by peer (os error 104)"` — capital C

After case folding: `"connection reset by peer"`. The shorter `"connection reset"` matches both.

**Note:** When a connection reset happens after HTTP headers are received (mid-stream), reqwest may surface it as `Kind::Body` (`"request or response body error"`), not `Kind::Request`. In that case `{err:#}` does NOT contain `"error sending request for url"`. The `"connection reset"` marker covers this path. [VERIFIED: reqwest-0.12.28/src/error.rs line 232 (Kind::Body); hyper-1.9.0/src/error.rs line 523]

#### 5. `error sending request for url` — exact spelling and survival through wrapping

From reqwest-0.12.28/src/error.rs lines 227-272 (Display impl, read directly from source):

```rust
Kind::Request => f.write_str("error sending request")?,
// ...
if let Some(url) = &self.inner.url {
    write!(f, " for url ({url})")?;
}
```

When a URL is present (which it always is for `.send()` calls — reqwest attaches the URL via `if_no_url`), the full string is:

```
error sending request for url (http://localhost:11434/v1/chat/completions)
```

**Survival through anyhow wrapping:** Yes. `format!("{err:#}")` walks `StdError::source()` recursively and appends each cause with `": "`. The reqwest error becomes one node in the chain; its Display string `"error sending request for url (...)"` appears verbatim in the output. The substring `"error sending request for url"` survives regardless of what URL is attached. [VERIFIED: reqwest-0.12.28/src/error.rs lines 227-273; anyhow-1.0.100/src/fmt.rs lines 7-15]

### Downcast feasibility (deferred path, informational only)

The reqwest `Error` is wrapped in anyhow via `Into<BoxError>` → stored as `source: Option<BoxError>` inside anyhow's `ErrorImpl`. Downcasting via `err.chain().find_map(|e| e.downcast_ref::<reqwest::Error>())` would work if the reqwest error is in the source chain and not erased. However, because `anyhow::Context` wraps the error in a context struct (not a transparent newtype), the reqwest::Error is accessible via `downcast_ref` on a chain node. This is feasible but deferred per CONTEXT.md — the substring approach is sufficient and simpler for this phase. [ASSUMED — based on anyhow source structure; not exercised in tests for this phase]

## Common Pitfalls

### Pitfall 1: Matching `"timed out"` as a bare substring without scope
**What goes wrong:** `"timed out"` appears in error messages that are NOT transport failures (e.g. SSE stream read timeout: `"SSE stream read timed out after 60s"` — from client.rs:185). Matching it broadly would misclassify a streaming read timeout as a provider connectivity failure.
**Why it happens:** CONTEXT D-01 listed `"timed out"` as a potential marker; the warning was there but not fully qualified.
**How to avoid:** Use the more specific `"operation timed out"` (reqwest TimedOut sentinel) and `"tcp connect error"` (hyper-util connect phase) rather than a bare `"timed out"`. The `"error sending request for url"` marker is the safest high-level match.
**Warning signs:** A test feeding `"SSE stream read timed out after 60s"` returns `(true, true)`.

### Pitfall 2: Expecting `"connection refused"` to match without case-insensitive folding
**What goes wrong:** OS errno text starts with capital `"C"` — `"Connection refused"`. A case-sensitive `.contains("connection refused")` misses it.
**How to avoid:** Lowercase the entire `{err:#}` string once before matching all needles (as D-01a specifies).

### Pitfall 3: Forgetting that `test_classify_other_error` feeds a transport string
**What goes wrong:** The test currently uses `"Connection refused: failed to connect to LLM"` and asserts `should_fallback == false`. After the new helper is added, this will fail because `"connection refused"` is now an allowlist match. The planner must repurpose this test (D-02) before the helper is added — or do both changes atomically.
**How to avoid:** The plan must change `test_classify_other_error`'s input string in the same commit as adding the helper. A good replacement: `"unexpected end of JSON input"` or `"invalid response body from provider"`.

### Pitfall 4: Placing `include_str!` path wrong in invariants file
**What goes wrong:** The invariants file is at `crates/ironhermes-agent/tests/invariants_27_1_4_1_1.rs` and needs to reach `src/agent_loop.rs`. The relative path from the test file's directory is `"../src/agent_loop.rs"`.
**How to avoid:** Copy the pattern from `invariants_27_1_4_1.rs` exactly — it uses `include_str!("../src/subagent_runner.rs")`.

### Pitfall 5: Omitting the URL-suffix portion of the reqwest marker
**What goes wrong:** Matching `"error sending request"` (without `" for url"`) is technically sufficient, but it's less specific. The CONTEXT.md D-04 explicitly requires locking the literal `"error sending request for url"` — the longer string that includes the URL suffix. The static-invariant must contain this exact substring.
**How to avoid:** The allowlist entry and the static-invariant both use `"error sending request for url"` (with the URL portion).

## Architecture Patterns

### Helper function placement

The new `is_transport_failure(err_str: &str) -> bool` sits immediately beside `extract_http_status(err_str: &str) -> Option<u16>` — both are private `fn` items on `impl AgentLoop`, both take `&str` (the already-formatted `{err:#}` string), and neither takes `self`. This follows the existing pattern.

### Case-insensitive matching pattern (D-01a)

```rust
fn is_transport_failure(err_str: &str) -> bool {
    let lower = err_str.to_lowercase();
    TRANSPORT_MARKERS.iter().any(|m| lower.contains(m))
}
```

Or equivalently inline without a constant. The key is lowercasing once and then checking all needles.

## Open Questions

1. **`"connection timed out"` for Linux kernel TCP timeout**
   - What we know: Linux ETIMEDOUT (errno 110) produces `"Connection timed out"` at the os level; macOS uses ETIMEDOUT (errno 60) `"Operation timed out"`. The reqwest connect_timeout wraps via `io::Error::new(io::ErrorKind::TimedOut, elapsed)` — the elapsed Display is not the same as OS errno text.
   - What's unclear: In practice for the Ollama use case, `reqwest::ClientBuilder::connect_timeout` fires before the kernel-level TCP timeout. The kernel-level timeout string (`"connection timed out"`) may never be observed.
   - Recommendation: The current allowlist (`"operation timed out"` + `"tcp connect error"`) covers the reqwest-controlled paths. Add `"connection timed out"` only if integration testing reveals it. The planner can leave it out of the initial allowlist.
