# Phase 27.1.4.1.1: Code Review Report

**Reviewed:** 2026-05-12T15:38:32Z
**Depth:** standard
**Files Reviewed:** 2
**Status:** issues_found

## Summary

This phase adds a private `is_transport_failure` helper to `AgentLoop` and rewires
the `None` branch of `classify_llm_error` to return `(true, Self::is_transport_failure(&err_str))`
instead of the hardcoded `(true, false)`. The intent — fall back to a sibling provider
when the primary is unreachable (Ollama not running, DNS failure, connection refused) —
is sound, the implementation is small, and the new test coverage is thorough and
exercises each allowlist needle.

No correctness BLOCKERs were found: the helper is only invoked after `extract_http_status`
returns `None`, the markers are lowercased before matching, and the new tests pass through
the production `{err:#}` formatting path. However, the design has a brittleness/coupling
surface worth flagging, and the allowlist-vs-blocklist trade-off means several real
transport-error spellings will still silently fail to trigger fallback.

## Warnings

### WR-01: Allowlist-only transport detection — unrecognised transport errors silently never fall back

**File:** `crates/ironhermes-agent/src/agent_loop.rs:639-651`
**Issue:** `is_transport_failure` is a fixed 7-entry allowlist. Any transport-layer failure
whose formatted chain does not contain one of those exact substrings returns `false`, so
`classify_llm_error` yields `(true, false)`: the call is retried 3× against the dead primary
and then the whole turn errors out — the configured fallback provider is never tried. Real
spellings not covered include hyper-util's `"client error (Connect)"`, `"network is unreachable"`,
`"no route to host"`, `"broken pipe"`, `"connection aborted"`, TLS handshake failures
(`"received fatal alert"`, `"handshake failed"`), and reqwest's body-phase `"error reading a body from connection"`.
There is no runtime safety net behind the allowlist. The doc comment claims the needles are
"verified against pinned source (reqwest 0.12.28, hyper-util 0.1.20, anyhow 1.0.100)" — that
pinning is exactly what makes this fragile: a dependency bump that reworks an error string
silently regresses fallback with no compile error and no test failure (the static-grep gate
in `invariants_27_1_4_1_1.rs` only checks that the literals are *present in source*, not that
they still *match real errors*).
**Fix:** Consider broadening the allowlist to cover the common cases above, and/or add a
defensive catch-all: if the error chain contains `"os error"` or `"client error (Connect)"`
or any `std::io::ErrorKind::{ConnectionRefused, ConnectionReset, TimedOut, ...}` Display text,
treat it as a transport failure. If the conservative scope is deliberate (per the phase plan),
at minimum add a `// LIMITATION:` comment enumerating the *known-uncovered* spellings so a
future reader knows the gap is intentional, not an oversight.

### WR-02: `is_transport_failure` matches against the entire formatted error chain, including provider-returned response bodies

**File:** `crates/ironhermes-agent/src/agent_loop.rs:649-650` (and call site at `:661-663`)
**Issue:** `format!("{err:#}")` flattens every context frame, which on the no-HTTP-status path
can include an upstream provider's *error message body* (e.g. a 200-OK response from OpenRouter
carrying `{"error":{"message":"upstream connection reset"}}` that the client `bail!`s without a
status code, or any wrapped error where the provider relayed its own transport diagnostics as
text). Because the substring scan is unanchored and case-folded, a body containing
`"connection reset"`, `"dns error"`, etc. will be classified as a local transport failure and
trigger fallback even though *our* request was delivered and answered. This is a low-probability
mis-classification (it only fires when `extract_http_status` also returns `None`), and the
resulting behaviour — switching providers — is mostly benign, but it is a behavioural change
that the tests do not cover and the doc comment does not acknowledge.
**Fix:** Either accept and document the behaviour ("we also fall back if the chain *mentions*
a transport error, even one relayed by the provider"), or narrow the scan to the portion of the
chain that originates locally — e.g. only inspect the innermost reqwest/hyper frame rather than
the full flattened string. At minimum add a unit test pinning the chosen behaviour for a
provider-relayed-error string so it can't drift.

### WR-03: New code now depends on `extract_http_status` returning `None` for transport errors — a coupling the lax parser does not guarantee

**File:** `crates/ironhermes-agent/src/agent_loop.rs:606-626` (consumed at `:661`)
**Issue:** Before this phase, `is_transport_failure` did not exist and `extract_http_status`'s
loose parsing (first `(` followed by any 3 ASCII digits + a space → parsed as a status code)
was harmless on transport errors. Now the transport path is *gated* on that parser returning
`None`. If a transport error's formatted chain ever contains a parenthesised 3-digit-plus-space
token — e.g. a URL like `(http://host/v1/200 OK/...)`, a contrived port path, or a future hyper
error rendering like `(111 ConnectionRefused)` — `extract_http_status` returns `Some(111)`,
falls into the `_ => (true, false)` arm at line 672, and fallback silently never fires. The
risk is small with today's error spellings, but the new feature's correctness now rides on an
implementation detail of an unrelated, deliberately-lax helper.
**Fix:** Tighten `extract_http_status` to only accept a status code in a position consistent
with the two documented formats (e.g. require it to be the *first* `(`-delimited token *and*
followed by a recognised reason word, or anchor on the known `bail!` prefixes), or invert the
order: run `is_transport_failure` *first* and only fall through to status extraction if it
returns `false`. Add a regression test for a transport error whose chain contains a
parenthesised 3-digit token.

## Info

### IN-01: `to_lowercase()` allocates a full copy of the error chain on every classification

**File:** `crates/ironhermes-agent/src/agent_loop.rs:649`
**Issue:** `err_str.to_lowercase()` allocates a new `String` the size of the entire flattened
error chain on every failed LLM call, only to do a handful of `contains` checks. Out of v1
performance scope and not hot-path-critical (it runs only on errors), noted for completeness.
**Fix:** Use a case-insensitive substring match (e.g. iterate `err_str` with `eq_ignore_ascii_case`
windows, or the `memchr`/`aho-corasick` already in the tree if available) to avoid the
allocation, or accept it — error paths are cold.

### IN-02: Static-grep invariant gate gives false confidence about *behavioural* coverage

**File:** `crates/ironhermes-agent/tests/invariants_27_1_4_1_1.rs:7-27`
**Issue:** The two tests assert that the literal strings `fn is_transport_failure(` and
`error sending request for url` appear in `agent_loop.rs`. They do not — and cannot — verify
that the allowlist still matches real reqwest/hyper output after a dependency bump. A reviewer
skimming the test names ("locks transport-failure detection") may over-trust them. This mirrors
the established `invariants_22_4.rs` pattern, so it is consistent with the codebase, but the gap
is real (see WR-01).
**Fix:** Pair the grep gate with at least one behavioural test that constructs the actual error
type chain (e.g. perform a real connect to `127.0.0.1:1` / an unbound port, or a closed
`TcpListener`, and pass the resulting `reqwest::Error` wrapped in anyhow through `classify_llm_error`)
so a string-spelling regression in a dependency fails CI. If a live socket in tests is undesirable,
document in the invariant file that these gates are source-presence only, not behavioural.

### IN-03: Doc comment slightly overstates the caller contract

**File:** `crates/ironhermes-agent/src/agent_loop.rs:634-635`
**Issue:** The doc says "the caller should then return `(true, true)`" — accurate when the
helper returns `true`, but the actual call site (`return (true, Self::is_transport_failure(&err_str))`)
returns `(true, false)` otherwise; the prose reads as if `(true, true)` is unconditional. Minor.
**Fix:** Reword to "...the caller returns `(true, true)` when this returns `true`, and
`(true, false)` otherwise."
