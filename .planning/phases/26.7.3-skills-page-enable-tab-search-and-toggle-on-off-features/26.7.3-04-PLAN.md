<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 2: Browser UAT — D-02 (UPDATES disabled), D-08 (optimistic revert), D-09 (tab+search composition), live counter update</name>
  <what-built>
    Phase 26.7.3 wired three Skills page features:
    1. Five tabs (ALL / BUNDLED / INSTALLED / ENABLED / UPDATES) with live counts; UPDATES is grayed-disabled per D-02.
    2. Client-side search input AND-combined with the active tab (D-04/D-05/D-09).
    3. Per-skill toggle on the `.tgl` div that flips optimistically, persists via `toggle_skill` server fn (writes config.yaml + mutates in-process active_skills), and reverts + shows inline error on server failure (D-07/D-08).
    Header sub-copy "{count} loaded · {N} enabled" updates live from optimistic toggle state.
  </what-built>
  <how-to-verify>
    Prerequisites:
    1. Start `dx serve` from the iron_hermes_ui crate:
       `cd crates/iron_hermes_ui && dx serve --platform web`
       Wait for the "Server running at ..." line (typically http://127.0.0.1:8080).
    2. Open the served URL in a Chromium-based browser (Chrome/Edge/Brave) — DevTools network blocking is most reliable there.
    3. Open DevTools (Cmd+Opt+I on macOS / F12 elsewhere). Keep Network panel visible.

    Navigate to the Skills page (the "04 Skills" screen — click the Skills tab in the wheel menu or use the appropriate route).

    Scenario A — D-02 UPDATES tab disabled visual:
    - Locate the UPDATES tab (rightmost in the tab strip).
    - Observe: it MUST appear visually grayer/dimmer than the other four tabs (50% opacity per UI-SPEC).
    - Observe: label MUST be exactly "UPDATES · 0".
    - Click UPDATES — NOTHING should change. The grid should NOT reset, no tab should become active.
    - Hover the cursor — the cursor should remain default (no pointer cursor) because of `pointer-events:none`.
    - Record PASS/FAIL.

    Scenario B — D-09 Tab + Search AND composition:
    - Click BUNDLED tab → grid shows ONLY skills with `category == "bundled"`.
    - Without clearing search, type a query that matches the name or description of a NON-bundled skill (e.g., type the first 3-4 chars of a known INSTALLED skill name visible from the ALL view).
    - Observe: grid shows ZERO cards AND the literal text "No skills match." appears inline in the grid in gray.
    - Click INSTALLED tab WITHOUT clearing the search — grid should now show matching INSTALLED skills (search query did NOT reset).
    - Clear the search input — INSTALLED tab still shows only installed skills (tab did NOT reset).
    - Record PASS/FAIL.

    Scenario C — D-08 Optimistic revert + inline error:
    - In DevTools Network panel, find a previous `toggle_skill` request (toggle any skill once to populate). Right-click → "Block request URL" (or "Block request domain" if Block URL is too narrow). The exact menu wording varies by browser; the goal is to make subsequent POST to that endpoint fail at the network layer.
    - With the block active, click any skill's toggle.
    - Observe (within ~1 second):
       1. Toggle FIRST flips visually (e.g., card border goes teal if was-off → was-on).
       2. THEN reverts to its pre-click visual state.
       3. THEN the inline text "Toggle failed — try again." appears inside the card in red below the description.
    - Click the same toggle a second time (still blocked). The error text should clear momentarily as the new optimistic flip happens, then re-appear when the second request also fails.
    - Remove the network block. Click the toggle once more — it should now flip and persist (no revert, no error text).
    - Record PASS/FAIL.

    Scenario D — Live counter update on optimistic toggle:
    - Note the header sub-copy: `"{N1} loaded · {N2} enabled for default."`. Record N2.
    - Toggle one skill from on → off. Observe N2 decreases by 1 IMMEDIATELY (before any server confirmation).
    - Toggle the same skill from off → on. Observe N2 returns to original.
    - Record PASS/FAIL.

    Persistence sanity check (informational, not a separate scenario):
    - Stop the dev server (Ctrl+C). Inspect `~/.ironhermes/config.yaml` (or wherever `HERMES_HOME/config.yaml` resolves on your machine) — the `skills:` section should contain a `disabled:` key with the names of any skills currently in the off state.
    - Restart `dx serve` and reopen the Skills page. Toggle states must match the on-disk config (skills present in `disabled:` should render as off).

    Write the result to `.planning/phases/26.7.3-skills-page-enable-tab-search-and-toggle-on-off-features/26.7.3-04-UAT-RESULT.md` with the following four-line block (one line per scenario), each ending in "PASS" or "FAIL":

    ```

    D-09 Tab + Search AND composition: <PASS|FAIL — notes>
    Live counter update on toggle: <PASS|FAIL — notes>
    Persistence (informational): <observation>
    ```

    If any scenario is FAIL, describe what was observed vs expected. Do NOT mark the plan complete; instead, surface the failure to the user — gap-closure planning may be required.
