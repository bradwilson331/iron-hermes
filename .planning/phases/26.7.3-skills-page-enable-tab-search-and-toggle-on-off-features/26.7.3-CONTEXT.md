# Phase 26.7.3: Skills page - enable tab, search and toggle on-off features - Context

**Gathered:** 2026-05-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 26.7.3 makes three currently-inert UI elements in `ScreenSkills` functional:
1. **Tabs** — clicking ALL / BUNDLED / INSTALLED / ENABLED filters the displayed skill cards; live counts replace hardcoded numbers; UPDATES tab deferred.
2. **Search** — the existing search input filters the list client-side by name + description; AND-combined with the active tab.
3. **Toggle** — the `.tgl` div on each `SkillCard` gains an `onclick` handler that persists the change to `SkillsConfig.disabled: Vec<String>` in config YAML and hot-reloads `active_skills` in-process.

**In scope:**
- Tab filtering with live counts
- Client-side search (name + description, case-insensitive)
- Skill toggle server fn + config write + in-process hot-reload
- `disabled: Vec<String>` field added to `SkillsConfig` in `ironhermes-core`
- Optimistic toggle UI with inline error revert

**Out of scope (deferred):**
- UPDATES tab (no version-check mechanism exists; tab rendered disabled with count 0)
- TUI / gateway hot-reload wiring for the toggle (follow-on phase)
- IMPORT / NEW SKILL button wiring
- Session deletion, pagination, session search (26.7.2 deferred list)

</domain>

<decisions>
## Implementation Decisions

### Tab filtering
- **D-01:** The **INSTALLED** tab shows all skills where `category != "bundled"` — i.e., everything except Builtin-sourced skills. This includes "official", "trusted", "installed", and "self-created" categories.
- **D-02:** The **UPDATES** tab is rendered as grayed-out / disabled with a hardcoded count of 0. No version field is exposed on `SkillInfo` this phase. The tab is present in the UI but not interactive.
- **D-03:** Tab counts are **live** — computed from `skills_list` in the component after it loads. The hardcoded placeholder numbers (87, 55, 57) are replaced. Each tab label becomes `"TAB · {live_count}"`.

### Search behavior
- **D-04:** Filtering is **client-side, in-memory** using a `use_signal(|| String::new())` for the query string. No server round-trip or re-fetch on typing.
- **D-05:** The search matches **name + description** — case-insensitive substring match on `skill.name` and `skill.description`. Category is excluded from search (tabs already handle category filtering).

### Toggle persistence
- **D-06:** Add `disabled: Vec<String>` (opt-out model) to `SkillsConfig` in `crates/ironhermes-core/src/config.rs`. All skills are on by default; only explicitly disabled skill names are listed. This is consistent with the existing `enabled: bool` master switch (opt-out style).
- **D-07:** A new **`POST /api/skills/{name}/toggle`** (or equivalent `#[server]` fn) goes in `crates/iron_hermes_ui/src/server/api.rs`. It: (1) reads the current config, (2) adds or removes the skill name from `disabled`, (3) writes the config YAML back to disk, (4) re-filters `active_skills` in-process from the updated config. This is web UI only — TUI/gateway hot-reload is a follow-on phase.
- **D-08:** The toggle uses **optimistic UI** — the card's enabled state flips immediately in the component signal, the server fn fires in the background, and on failure the state reverts with an inline error message. Matches the kill/interrupt pattern in agents.rs.

### Filter composition
- **D-09:** Tab filter and search query are **AND-combined** — a skill must pass both the active tab filter and the search string to appear. Switching tabs does not clear search; typing in search does not reset the tab.
- **D-10:** When the filtered + searched list is empty, render an **inline empty-state message** inside `.grid` (e.g., `"No skills match."`) — same inline-text approach as the `load_error` state already in the component.

### Claude's Discretion
- Exact signal names for active tab state (`use_signal(|| "all")` or an enum variant) and search query string.
- Whether the toggle server fn uses `#[server]` or `#[post]` — check Dioxus 0.7 fullstack conventions; `#[server]` is preferred per iron_hermes_ui/CLAUDE.md patterns.
- Config write-back path: the server fn needs to know the config file location — follow the same pattern as how `AppState` loads config at startup (check `state.rs` or `app_runtime_factory.rs` for the config path resolution).
- The exact in-process hot-reload mechanism: after config write, either re-lock `active_skills` and recompute from config, or dispatch the existing `CommandResult::SkillsReload` equivalent if accessible from the server fn context.
- Empty-state message copy and styling — `"No skills match."` or similar, rendered as a `div` inside `.grid`.
- Whether the UPDATES tab button has `disabled` attribute or just a grayed CSS class.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `use_server_future(api::list_skills)?` already in `ScreenSkills` — `skills_list: Vec<SkillInfo>` is already loaded; tab filtering and search operate on this in-memory slice, no new fetch needed.
- `SkillInfo { name, description, category, enabled }` — `category` strings match `SkillSource` variants ("bundled", "official", "trusted", "installed", "self-created"); D-01 INSTALLED tab filters `category != "bundled"`.
- `.tgl` / `.tgl.on` CSS classes already exist and are rendered conditionally in `SkillCard` — adding `onclick` is the only UI change needed for the toggle affordance.
- `active_skills: Arc<Mutex<Vec<SkillRecord>>>` on `AppRuntimeBundle` — the toggle server fn locks this mutex and recomputes the active list after writing config.

### Established Patterns
- **Signal borrow discipline** (from clippy.toml): extract all signal reads before `rsx!`; no `GenerationalRef` held across await. For `use_signal(|| "all")` (active tab) and `use_signal(|| String::new())` (search query), call them with `()` to subscribe and drop immediately.
- **`use_memo` for derived state**: the filtered skills list (`tab_filtered` then `search_filtered`) should use `use_memo(move || { ... })` not a `use_signal` — memo recomputes when dependencies change, no manual invalidation needed.
- **`spawn` for async inside event handlers**: `onclick` closures cannot be `async`; the toggle server fn call must be wrapped in `spawn(async move { ... })`.
- **Optimistic update pattern** (agents.rs): (1) capture current state, (2) flip signal, (3) spawn async server call, (4) on `Err`, revert signal to captured state + set error signal.

### Integration Points
- `ScreenSkills` → `use_signal` for `active_tab` + `search_query` → `use_memo` for `filtered_skills` → `for skill in filtered_skills.read().iter()` in grid
- `SkillCard` → needs `mut enabled: Signal<bool>` or optimistic state owned by `ScreenSkills` and passed down — check whether to lift toggle state into a `HashMap<String, bool>` signal at `ScreenSkills` level (avoids per-card signal proliferation)
- Toggle server fn → `global_app_state()` → `runtime_bundle.active_skills` lock → recompute from updated `SkillsConfig.disabled`
- Config write-back → resolve config path via same mechanism used at startup (check `app_runtime_factory.rs` for config file path)

</code_context>

<specifics>
## Specific Ideas

- User wants this toggle to be the **foundation for cross-surface skill management** (web → gateway → TUI hot-reload). The `disabled: Vec<String>` config field must be designed to be readable by all surfaces, not web-UI-specific state. The config YAML is the source of truth; surfaces read from it on startup and after reload.
- The UPDATES tab should be clearly visually disabled (not just empty) — users shouldn't wonder if it's a bug.
- Filter composition is AND: tab + search both active simultaneously with no reset behavior between them.

</specifics>

<deferred>
## Deferred Ideas

- **Cross-surface hot-reload** (TUI + gateway) — the `disabled: Vec<String>` config field is being added now, but wiring TUI's `active_skills` and gateway's skill state to re-read the config on change is a follow-on phase. TUI/gateway currently have no runtime toggle UX.
- **UPDATES tab** — needs a version-check mechanism (compare `SkillRecord.version` against a remote catalog). No update-check infrastructure exists; deferred until skills hub / agentskills.io integration phase.
- **IMPORT / NEW SKILL button** — stubbed in the UI header; own phase.
- **Per-skill config values** (`SkillsConfig.config.<name>.*`) — editing per-skill config from the web UI; deferred.

</deferred>
