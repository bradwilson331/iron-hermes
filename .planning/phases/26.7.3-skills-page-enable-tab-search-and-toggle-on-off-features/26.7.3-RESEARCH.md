## Summary

Phase 26.7.3 activates three currently-inert UI elements in `ScreenSkills`: tab filtering, search, and the per-skill toggle. All three features operate on `skills_list` already loaded by `use_server_future(api::list_skills)` — no new server reads are needed for tab/search, which are pure in-memory derived state via `use_memo`. The toggle is the only write path and requires: (1) a new `#[server]` fn in `api.rs`, (2) a `disabled: Vec<String>` field added to `SkillsConfig`, (3) config write-back via the existing `Config::save()` method, and (4) in-process `active_skills` mutation by locking the `Arc<std::sync::Mutex<Vec<SkillRecord>>>` held on `AppRuntimeBundle`.

The key insight about `active_skills`: it is a **session-transient** list populated lazily when the agent calls `/skill activate` during a conversation. It is NOT pre-populated from config at startup. For the purposes of the Skills page display and toggle, the `enabled` state in `SkillInfo` is computed from whether the skill name appears in this runtime list. The toggle server fn must therefore also mutate this runtime list — not just write config — so the change is reflected immediately without restarting the server.

**Primary recommendation:** Lift all toggle and filter state into `ScreenSkills` using `use_signal` for `active_tab`, `search_query`, `toggle_states: HashMap<String, bool>`, and `toggle_errors: HashMap<String, String>`. Derive `filtered_skills` via `use_memo`. Pass `enabled`/`error_msg`/`on_toggle` as props to `SkillCard`. The toggle server fn follows the `#[server]` pattern (not `#[get]`/`#[post]`) and uses `global_app_state()` → `Config::load()` + `Config::save()` + lock `active_skills`.

### Pattern 1: Signal Borrow Discipline (CRITICAL — clippy-enforced)

**What:** All signal reads must complete before `rsx!` is entered. No `GenerationalRef` or `WriteLock` held across `.await`.

**When to use:** Every component that uses signals alongside async — which is every component in this phase.

**Example:**
```rust
// Source: iron_hermes_ui/CLAUDE.md + skills.rs existing pattern
#[component]
pub fn ScreenSkills(is_active: bool) -> Element {
    let skills_resource = use_server_future(crate::server::api::list_skills)?;

    // All signal reads BEFORE rsx! — extract to owned values
    let skills_list: Vec<SkillInfo> = match skills_resource() {
        Some(Ok(v)) => v,
        _ => Vec::new(),
    };
    let load_error = matches!(skills_resource(), Some(Err(_)));

    let active_tab = use_signal(|| "all");
    let search_query = use_signal(|| String::new());
    let mut toggle_states = use_signal(|| {
        skills_list.iter().map(|s| (s.name.clone(), s.enabled)).collect::<HashMap<_, _>>()
    });
    let toggle_errors: Signal<HashMap<String, String>> = use_signal(|| HashMap::new());

    // Extract all reads BEFORE rsx!
    let tab_val = active_tab();        // &'static str — Copy
    let query_val = search_query();    // String — Clone

    let count = skills_list.len();
    let enabled_count = toggle_states.read().values().filter(|&&v| v).count();
    // ... then rsx! { ... }
}
```

### Pattern 2: `use_memo` for Derived Filter State

**What:** `use_memo` recomputes when any captured signal changes. No manual invalidation needed.

**When to use:** Any derived value that depends on multiple signals — here, the AND-combined tab+search filter.

**Example:**
```rust
// Source: iron_hermes_ui/CLAUDE.md Dioxus 0.7 Specific Conventions
let filtered_skills = use_memo(move || {
    let tab = active_tab();
    let query = search_query().to_lowercase();
    skills_list.iter()
        .filter(|s| match tab {
            "bundled"   => s.category == "bundled",
            "installed" => s.category != "bundled",
            "enabled"   => s.enabled,      // server-sourced, not optimistic
            _           => true,           // "all"
        })
        .filter(|s| {
            query.is_empty()
                || s.name.to_lowercase().contains(&query)
                || s.description.to_lowercase().contains(&query)
        })
        .cloned()
        .collect::<Vec<_>>()
});
```

### Pattern 3: Optimistic Toggle with Spawn (agents.rs reference)

**What:** onclick cannot be `async`; use `spawn(async move { ... })`. Capture current state, flip, call server, revert on error.

**When to use:** Any write-path server fn called from a click handler.

**Example:**
```rust
// Source: agents.rs lines 498–554 (interrupt/kill pattern)
// Called from ScreenSkills, passed to SkillCard as on_toggle: EventHandler<()>
let on_toggle = move |name: String| {
    let current = *toggle_states.read().get(&name).unwrap_or(&false);
    // 1. Optimistic flip
    toggle_states.write().insert(name.clone(), !current);
    // 2. Clear prior error
    toggle_errors.write().remove(&name);
    // 3. Async server call
    spawn(async move {
        match crate::server::api::toggle_skill(name.clone()).await {
            Ok(_) => {}  // optimistic state is already correct
            Err(_) => {
                // 4. Revert + set error
                toggle_states.write().insert(name.clone(), current);
                toggle_errors.write().insert(
                    name.clone(),
                    "Toggle failed — try again.".to_string(),
                );
            }
        }
    });
};
```

### Pattern 4: `#[server]` for Write Server Fn

**What:** `#[server]` (not `#[get]` or `#[post]`) is the correct Dioxus 0.7 fullstack attribute for write operations that take a body argument. The locked pattern from Phase 26.7 D-03/D-04 uses `#[get]` for reads; writes use `#[server]`.

**Example:**
```rust
// Source: api.rs get_session_messages pattern (Phase 26.7.2)
#[server]
pub async fn toggle_skill(name: String) -> Result<(), ServerFnError> {
    let state = crate::server::state::global_app_state();
    // Load fresh config from disk
    let mut config = ironhermes_core::config::Config::load()
        .unwrap_or_default();
    // Mutate disabled list
    let disabled = &mut config.skills.disabled;
    if disabled.contains(&name) {
        disabled.retain(|n| n != &name);  // was disabled → enable it
    } else {
        disabled.push(name.clone());       // was enabled → disable it
    }
    // Write back
    config.save()
        .map_err(|e| ServerFnError::new(format!("Config save failed: {e}")))?;
    // Mutate in-process active_skills
    let mut skills = state.runtime_bundle.active_skills.lock()
        .map_err(|e| ServerFnError::new(format!("active_skills lock poisoned: {e}")))?;
    if disabled.contains(&name) {
        // Now disabled — remove from active set
        skills.retain(|s| s.name != name);
    } else {
        // Now enabled — the skill tool's handle_activate normally populates this;
        // for the toggle path, find the SkillRecord from the registry and push
        if let Some(record) = state.runtime_bundle.skill_registry.find(&name) {
            if !skills.iter().any(|s| s.name == name) {
                skills.push(record.clone());
            }
        }
    }
    Ok(())
}
```

### Pattern 5: `SkillCard` Prop Interface

**What:** Toggle state moves from `skill.enabled` (server-sourced, static after load) to a `Signal<bool>` owned by the parent and passed as a prop.

**Example:**
```rust
// Source: CONTEXT.md code_context + UI-SPEC Component Inventory
#[component]
fn SkillCard(
    skill: SkillInfo,
    enabled: Signal<bool>,      // optimistic state owned by ScreenSkills
    error_msg: Option<String>,  // None = no error
    on_toggle: EventHandler<()>,
) -> Element {
    let is_enabled = enabled();  // read before rsx!
    rsx! {
        div {
            class: "card",
            class: if is_enabled { "is-active" },
            // ... card content ...
            div {
                class: if is_enabled { "tgl on" } else { "tgl" },
                role: "switch",
                aria_checked: "{is_enabled}",
                onclick: move |_| on_toggle.call(()),
            }
            if let Some(ref err) = error_msg {
                div {
                    style: "color:var(--danger);font-size:var(--fs-12);",
                    "{err}"
                }
            }
        }
    }
}
```

### Anti-Patterns to Avoid

- **Holding signal `.read()` across `rsx!`:** Extract to owned value before `rsx!` — clippy will catch this at compile time, but it causes runtime panics in WASM if not caught.
- **Using `signal.write()` inside `rsx!`:** Write operations inside the render tree cause re-render loops. All mutations must be in event handlers or `spawn` closures.
- **Using `use_signal` for derived state:** Use `use_memo` for `filtered_skills` — `use_signal` would require manual invalidation.
- **Putting `toggle_states` inside `SkillCard`:** Per-card signal proliferation for N skills creates N signals. Lift to a `HashMap<String, bool>` at `ScreenSkills` level — one signal for all toggle state.
- **`Config::save()` vs `Config::save_to(Config::config_path())`:** Both are equivalent. `Config::save()` calls `get_hermes_home().join("config.yaml")` internally. Either form is correct. Prefer `config.save()` for brevity.
- **Recomputing `active_skills` from scratch:** Do not call `SkillRegistry::load_with_config()` again — this re-scans the filesystem and is expensive. Instead, lock the existing `Arc<Mutex<Vec<SkillRecord>>>` and `push`/`retain` directly.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Config file path resolution | Custom path logic | `Config::config_path()` / `Config::save()` | Already encapsulates `get_hermes_home().join("config.yaml")` |
| YAML serialization | Manual string formatting | `serde_yaml::to_string(self)?` via `Config::save_to()` | Handles escaping, field ordering, default omission |
| Client-side filter | Custom reactive state machine | `use_memo(move || ...)` | Dioxus 0.7 memo handles dependency tracking automatically |
| Async in event handler | `async` closure on `onclick` | `spawn(async move { ... })` | `onclick` cannot be `async` in Dioxus 0.7; `spawn` is the correct escape hatch |
| Active skills management | New in-memory data structure | Lock existing `runtime_bundle.active_skills: Arc<Mutex<Vec<SkillRecord>>>` | This Arc is shared between the server fn context and the agent loop; it IS the runtime source of truth |

**Key insight:** The `Config::save()` write path is already battle-tested across the codebase (Phase 24 profile isolation, Phase 19 skills config). The `active_skills` mutex is the same Arc the agent's `handle_activate` modifies — the toggle fn is just another writer on the same lock.

## Common Pitfalls

### Pitfall 1: active_skills is Session-Transient, Not Config-Driven at Startup

**What goes wrong:** Implementer assumes `active_skills` is pre-populated from `SkillsConfig.disabled` at startup, and designs the toggle to only write config. The web UI's "enabled" display correctly shows skills as enabled/disabled, but the agent's `execute_code` sandbox still loads the supposedly-disabled skill because `active_skills` was never mutated.

**Why it happens:** `active_skills` starts as an empty `Vec::new()` in `build_app_runtime_bundle`. It is populated lazily by `handle_activate` (the `/skill activate` tool action) during agent conversations. `SkillsConfig.disabled` is a new field being added this phase — no code yet reads it to pre-filter `active_skills` at startup.

**How to avoid:** The toggle server fn MUST lock `runtime_bundle.active_skills` and `push`/`retain` in addition to writing config. The config write ensures persistence across restarts; the in-process mutation ensures the current session sees the change immediately.

**Warning signs:** Toggle appears to work (config YAML has the entry) but `list_skills` still shows the skill as enabled after toggle, because the runtime `active_skills` set was not updated.

### Pitfall 2: Signal Read Held Across `rsx!` Boundary

**What goes wrong:** `let guard = toggle_states.read();` held in a binding that outlives `rsx!` entry — causes a `GenerationalRef` borrow panic at runtime in WASM.

**Why it happens:** Dioxus 0.7 signals use arena-based memory (`generational_box`). The `clippy.toml` enforces this but only catches cross-`.await` violations at compile time; RSX boundaries are a runtime concern.

**How to avoid:** Always extract signal reads to owned values before `rsx!`:
```rust
let is_enabled = enabled();        // bool — Copy — safe
let err = error_msg.clone();       // Option<String> — Clone — safe
```
Never pass `toggle_states.read().get(&name)` into RSX — clone the value first.

### Pitfall 3: `toggle_states` Not Seeded from `skills_list`

**What goes wrong:** `toggle_states` starts as `HashMap::new()` (empty). The first render shows all toggles as OFF because `toggle_states.read().get(&name)` returns `None`, and the fallback is `false`.

**Why it happens:** `use_signal(|| HashMap::new())` runs at component mount time, but `skills_list` is populated asynchronously by `use_server_future`. If `toggle_states` is initialized as empty, it will never be seeded with the server-sourced `enabled` values.

**How to avoid:** Per the UI-SPEC Component Inventory, `toggle_states` must be initialized from `skills_list` after it loads. Since `use_signal` initializer only runs once (at mount), the seed must either: (a) initialize lazily on first read (check if map is empty and populate from `skills_list`), or (b) use a `use_effect` to seed after load. Pattern (a) is simpler for this phase.

**Implementation:**
```rust
// After skills_list is extracted from skills_resource():
let toggle_states = use_signal(|| HashMap::<String, bool>::new());
// Seed on first non-empty load — use_effect approach
{
    let sl = skills_list.clone();
    use_effect(move || {
        if !sl.is_empty() && toggle_states.read().is_empty() {
            let mut map = toggle_states.write();
            for s in &sl {
                map.insert(s.name.clone(), s.enabled);
            }
        }
    });
}
```

### Pitfall 4: `serde(default)` Required for New Config Field

**What goes wrong:** Adding `disabled: Vec<String>` to `SkillsConfig` without `#[serde(default)]` breaks YAML parsing for all existing `config.yaml` files that predate this phase (they don't have a `disabled:` key).

**Why it happens:** `serde_yaml` will error on missing required fields.

**How to avoid:** Mark the new field with `#[serde(default)]` (consistent with `credential_dir`, `config`, and `hub` fields on the same struct). `Vec::new()` is the natural default (no skills disabled = all enabled).

**Existing pattern in `SkillsConfig`:**
```rust
#[serde(default)]
pub credential_dir: Option<PathBuf>,

#[serde(default)]
pub config: HashMap<String, HashMap<String, serde_yaml::Value>>,
```

### Pitfall 5: `use_memo` Closure Cannot Capture `skills_list` by Reference

**What goes wrong:** `use_memo(move || { ... skills_list.iter() ... })` — `skills_list` is a `Vec<SkillInfo>` owned by the component scope; it must be moved or cloned into the closure. Since `use_memo` captures for `'static`, the closure must own its data.

**How to avoid:** Either clone `skills_list` before the `use_memo` call, or restructure so the memo reads from a signal (if `skills_list` itself becomes a signal):
```rust
let skills_for_memo = skills_list.clone(); // clone before use_memo
let filtered_skills = use_memo(move || {
    // uses skills_for_memo — owned, no borrow of component scope
});
```

### Pitfall 6: ENABLED Tab Uses Server-Sourced `enabled`, Not Optimistic State

**What goes wrong:** Implementer uses `toggle_states.read()[&s.name]` for the ENABLED tab predicate. This means a skill just toggled-on (optimistically) immediately appears in the ENABLED tab before the server confirms — and a skill just toggled-off disappears from ENABLED immediately.

**Why it matters:** The UI-SPEC (Interaction Contracts §Filter composition) explicitly states the ENABLED tab predicate uses `skill.enabled` (the server-sourced value in `SkillInfo`), not the optimistic state. The optimistic state is only for the visual toggle affordance on the card itself.

**How to avoid:** Use `s.enabled` (from `SkillInfo` in `skills_list`) for the ENABLED tab predicate, and `*toggle_states.read().get(&s.name).unwrap_or(&s.enabled)` only for the card's `.tgl on` class and `is-active` class.

### Toggle Server Fn — Full Implementation

```rust
// Source: api.rs existing #[server] pattern (get_session_messages) + state.rs global_app_state()
#[server]
pub async fn toggle_skill(name: String) -> Result<(), ServerFnError> {
    let state = crate::server::state::global_app_state();

    // 1. Load fresh config from disk (not from AppState.config — that's the startup snapshot)
    let mut config = ironhermes_core::config::Config::load()
        .unwrap_or_default();

    // 2. Mutate disabled list (add/remove)
    let is_currently_disabled = config.skills.disabled.contains(&name);
    if is_currently_disabled {
        config.skills.disabled.retain(|n| n != &name);  // re-enable
    } else {
        config.skills.disabled.push(name.clone());       // disable
    }

    // 3. Write config back to disk
    config.save()
        .map_err(|e| ServerFnError::new(format!("Config save failed: {e}")))?;

    // 4. Mutate in-process active_skills (std::sync::Mutex — no .await)
    let mut skills = state.runtime_bundle.active_skills.lock()
        .map_err(|e| ServerFnError::new(format!("active_skills lock poisoned: {e}")))?;

    if is_currently_disabled {
        // Was disabled, now enabled — push to active_skills if found in registry and not already present
        if let Some(record) = state.runtime_bundle.skill_registry.find(&name).cloned() {
            if !skills.iter().any(|s| s.name == name) {
                skills.push(record);
            }
        }
    } else {
        // Was enabled, now disabled — remove from active_skills
        skills.retain(|s| s.name != name);
    }

    Ok(())
}
```

## Open Questions (RESOLVED)

1. **`list_skills` `enabled` field vs `toggle_states` seeding race**
   - What we know: `use_server_future` is async; `skills_list` may be `Vec::new()` on first render.
   - What's unclear: If `use_effect` for seeding runs before the future resolves, `skills_list` will be empty and seeding is a no-op. Subsequent renders will have `toggle_states` still empty.
   - Recommendation: Check `skills_list.is_empty()` inside the `use_effect` and only seed when non-empty. Because `use_effect` re-runs when dependencies change, and `skills_list` is derived from `skills_resource()` which will re-render on load completion, the effect will re-run when data arrives.

2. **`SkillRegistry::find()` return type**
   - What we know: `registry.list()` is used in `list_skills()`. `find()` is referenced in `skills_tool.rs` but not yet verified in the public API surface.
   - What's unclear: Whether `SkillRegistry::find(name)` returns `Option<&SkillRecord>` or `Option<SkillRecord>`. The `handle_activate` code uses `.clone()` suggesting it returns a reference.
   - Recommendation: The toggle fn should call `.cloned()` on the result to avoid lifetime issues in the server fn context.

3. **`Config::load()` inside server fn — startup config vs live config**
   - What we know: `AppState.config` is the config snapshot from startup (Arc<Config>). `Config::load()` reads fresh from disk each call.
   - What's unclear: If another process modifies the config between the toggle fn's `Config::load()` and `Config::save()`, the intermediate changes will be overwritten.
   - Recommendation: Acceptable for this phase — single-user, single-process assumption. Document in code. A mutex around config writes is a future concern.

### Input Validation for Toggle Fn

The `name: String` parameter in `toggle_skill` arrives as a Dioxus server fn body argument. It must be validated before use in `Config.skills.disabled`:

- **Length check:** Skill names are validated by `validate_skill_name()` in `skills.rs` (max 64 chars, lowercase alphanumeric + hyphens). The toggle fn should verify the name exists in `skill_registry.find(&name)` before mutating `disabled`. An unknown name that passes through still produces a harmless no-op (retain on empty list), but it's cleaner to fail fast.
- **No shell injection risk:** The name is only used as a string key in `Vec<String>` and YAML serialization — no shell, no SQL, no interpolation.
- **Config write atomicity:** `serde_yaml::to_string` + `std::fs::write` is not atomic. On crash between the two, the file could be empty. Acceptable for this phase; document as known limitation.

### Known Threat Patterns

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Arbitrary config key injection via `name` | Tampering | Verify `skill_registry.find(&name)` returns `Some` before mutating `disabled` |
| Config file corruption on write crash | Denial of Service | Write to temp file then rename (future hardening; acceptable for phase) |
