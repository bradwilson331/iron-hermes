<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → server fn | `name: String` arrives as a server fn body argument from the WASM client over HTTP |
| server fn → disk | `Config::save()` writes the YAML file at `$HERMES_HOME/config.yaml` |
| server fn → in-process state | `active_skills` Arc<Mutex<...>> is shared with the agent loop |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-26.7.3-02-01 | Tampering | toggle_skill input `name` | mitigate | Reject with `ServerFnError` if `SkillRegistry::find(&name)` returns `None` BEFORE any disk write or mutex lock. Verified by source-grep acceptance criterion that the `.find(&name)` check precedes `Config::load()` lexically. |
| T-26.7.3-02-02 | Tampering | config.yaml write path | accept | `Config::save()` resolves the path via `get_hermes_home().join("config.yaml")` — no user input flows into the path. Skill name is only written as a value inside the `disabled` Vec, never interpolated into the file path. |
| T-26.7.3-02-03 | Denial of Service | active_skills mutex poisoning | accept | `.lock().map_err(...)` surfaces poison as an error string; the agent loop is the only other writer and the toggle fn is short-lived. Single-operator deployment posture per PROJECT.md. |
| T-26.7.3-02-04 | Information Disclosure | error message leaking config path or internal state | mitigate | Error strings include only the skill name and the bubbled-up error text from `Config::save()` / `lock()` — no environment variables or arbitrary state. |
| T-26.7.3-02-05 | Tampering | config write atomicity | accept | `serde_yaml::to_string` + `std::fs::write` is not atomic; on crash the file could be empty. RESEARCH §Security explicitly documents this as out of scope for this phase. Future hardening = temp-file-rename pattern. |
| T-26.7.3-02-SC | Tampering | npm/pip/cargo installs | n/a | No new dependencies added in this plan (RESEARCH §"No New Dependencies"). |
</threat_model>
