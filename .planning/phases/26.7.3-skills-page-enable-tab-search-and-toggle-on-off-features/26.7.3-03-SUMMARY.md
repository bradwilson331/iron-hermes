## Signal Inventory

| Signal Name | Type | Purpose |
|---|---|---|
| `active_tab` | `Signal<&'static str>` (init: `"all"`) | Tracks the active tab; drives tab button class and use_memo filter |
| `search_query` | `Signal<String>` (init: `String::new()`) | Holds the search input value; drives use_memo filter |
| `toggle_states` | `Signal<HashMap<String, bool>>` (init: `HashMap::new()`) | Per-skill optimistic enabled state; seeded from skills_list via use_effect |
| `toggle_errors` | `Signal<HashMap<String, String>>` (init: `HashMap::new()`) | Per-skill inline error message; set on toggle_skill Err, cleared on next click |

## enabled_count_live vs count_enabled

- **`enabled_count_live`**: computed from `toggle_states.read().values().filter(|&&v| v).count()` — reads the optimistic HashMap; used in the header sub-copy so the count updates immediately on toggle click
- **`count_enabled`**: computed from `skills_list.iter().filter(|s| s.enabled).count()` — reads the server-sourced SkillInfo slice; used exclusively in the ENABLED tab label (`"ENABLED · {count_enabled}"`) per RESEARCH Pitfall 6

## SkillCard enabled Prop Confirmation

`SkillCard`'s `enabled` prop is declared as **plain `bool`** — not `Signal<bool>`. The parent `ScreenSkills` owns `toggle_states: Signal<HashMap<String, bool>>` and extracts a per-card `is_enabled: bool` snapshot (via pre-computed `card_data` vec) before passing it down. This avoids per-card signal proliferation (N signals for N skills) and keeps the optimistic state ownership at the screen level.

## Decisions Made

- **Pre-compute card_data before rsx!**: Dioxus 0.7 RSX macro does not permit `let` bindings inside `for` loop bodies. The plan's code snippet showed per-card `let is_enabled = ...` inside the for loop — this fails with `error: expected identifier`. Fixed by collecting `(SkillInfo, bool, Option<String>)` tuples into `card_data: Vec<...>` before entering `rsx!`, then iterating that owned vec.
- **`let mut on_toggle`**: The closure captures `toggle_states` and `toggle_errors` (both mutable signals) and calls `.write()` and `.read()` on them. Rust requires the closure binding to be `mut` so it can be called mutably. The plan specified `let on_toggle` which would fail with E0596.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] `let` bindings inside rsx! for loop not supported by Dioxus RSX macro**
- **Found during:** Task 2 (first build attempt)
- **Issue:** The plan's grid loop body contained `let is_enabled = ...`, `let err_msg = ...`, `let skill_name_for_toggle = ...` inside the `for skill in filtered_skills.read().iter()` loop within `rsx!`. Dioxus 0.7's RSX macro does not support `let` statements inside `for` loop bodies — compiler emits `error: expected identifier` at the `=` sign.
- **Fix:** Pre-compute all per-card data into `card_data: Vec<(SkillInfo, bool, Option<String>)>` before `rsx!` using a scoped block that drops all borrow guards at `}`. The `for` loop inside `rsx!` iterates the owned vec with `.cloned()` — no `let` bindings needed.
- **Files modified:** `crates/iron_hermes_ui/src/components/hermes_app/screens/skills.rs`
- **Verification:** `cargo build -p iron_hermes_ui --features web` exits 0 after fix
- **Committed in:** `72572506` (Task 2 commit)

**2. [Rule 1 - Bug] `let on_toggle` must be `let mut on_toggle`**
- **Found during:** Task 2 (second build attempt after fix 1)
- **Issue:** The closure capturing mutable signals (`toggle_states.write()`, `toggle_errors.write()`) requires the binding to be `mut` — compiler emits `E0596: cannot borrow on_toggle as mutable, as it is not declared as mutable`.
- **Fix:** Changed `let on_toggle = move |name: String|` to `let mut on_toggle = move |name: String|`.
- **Files modified:** `crates/iron_hermes_ui/src/components/hermes_app/screens/skills.rs`
- **Verification:** `cargo build -p iron_hermes_ui --features web` exits 0 after fix
- **Committed in:** `72572506` (Task 2 commit)
