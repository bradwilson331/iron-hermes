# Phase 27.1.4.2: hexapod-led-off-fails - Context

**Gathered:** 2026-05-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Two distinct bugs, both in `crates/ironhermes-tools/src/`:

**Bug 1 — Wrong wire protocol for `led_off`**
`hexapod_tcp.rs` defines `CMD_LED_OFF = "CMD_LED#0\n"` (line 58). On the Freenove server,
`CMD_LED` is the *color* channel (`CMD_LED#r#g#b\n`); passing `#0` is parsed as a color with
only one field, which is silently ignored or treated as black — *not* as lights-out. The
correct command is `CMD_LED_MOD#0\n`, sent on the *mode* channel. Confirmed against the
official Freenove client: `Freenove/Code/Client/Main.py` line 1063 sends
`cmd.CMD_LED_MOD + '#' + '0' + '\n'` in `lightsOut()`. Real-world symptom: LEDs do not
turn off when the `led_off` action is invoked on the physical robot.

**Bug 2 — Cross-module `ENV_LOCK` race**
`hexapod_tcp::tests` (line 620) and `hexapod_video::tests` (line 234) each declare their
own `static ENV_LOCK: Mutex<()>`. Because these are separate statics, tests in the two
modules run concurrently and race on the `HEXAPOD_IP` env var.
Failure observed: `test_capture_frame_passes_allowlist` (hexapod_video) calls
`remove_var("HEXAPOD_IP")`, but a concurrently-running hexapod_tcp test has already called
`set_var("HEXAPOD_IP", "127.0.0.255")`. The video test then reads the set value and
gets `"cannot connect to robot at 127.0.0.255:8002"` instead of the expected env-var error.

**Deliverables:**
1. `hexapod_tcp.rs`: change `CMD_LED_OFF` from `"CMD_LED#0\n"` to `"CMD_LED_MOD#0\n"`.
2. `hexapod_tcp.rs`: update `test_cmd_led_off_constant_value` — change assertion from
   `"CMD_LED#0\n"` to `"CMD_LED_MOD#0\n"` and remove the `assert_ne!` for the
   three-field form (it was guarding the wrong direction).
3. `lib.rs`: add a single `pub(crate) static ENV_LOCK: Mutex<()>` under `#[cfg(test)]`.
4. `hexapod_tcp.rs` tests: replace `use std::sync::Mutex; static ENV_LOCK: ...` with
   `use crate::ENV_LOCK;`.
5. `hexapod_video.rs` tests: same swap; also remove the stale comment (lines 232–233)
   that says "This mutex only protects against races within this module. Run with
   RUST_TEST_THREADS=1 to avoid races with other modules that also read HEXAPOD_IP."
6. New static-invariant test `crates/ironhermes-tools/tests/invariants_27_1_4_2.rs` that
   locks (a) `CMD_LED_MOD#0\n` is present in `hexapod_tcp.rs` and (b) `CMD_LED#0\n` is
   *absent* from `hexapod_tcp.rs` (so a refactor cannot silently revert Bug 1).

**Out of scope:** skills_tool.rs env-var mutations (different vars, self-contained lock,
no conflict); browser_session.rs unguarded env mutations (different vars, different concern,
deferred — see Deferred below); any change to the `led_off` allowlist or action dispatch
logic (already correct); `hexapod_video.rs` protocol changes (unrelated).

</domain>

<decisions>
## Implementation Decisions

### Bug 1 — CMD_LED_OFF constant

- **D-01:** Change `CMD_LED_OFF` from `"CMD_LED#0\n"` to `"CMD_LED_MOD#0\n"`. This is
  the exact string sent by `Freenove/Code/Client/Main.py:lightsOut()` and is the only
  form the Freenove server's `CMD_LED_MOD` handler accepts for mode 0 (lights-out).

- **D-01a:** `test_cmd_led_off_constant_value` currently asserts
  `assert_eq!(CMD_LED_OFF, "CMD_LED#0\n")` (wrong) and
  `assert_ne!(CMD_LED_OFF, "CMD_LED#0#0#0\n")` (vacuous guard). Repurpose it: assert
  `assert_eq!(CMD_LED_OFF, "CMD_LED_MOD#0\n")`. The `assert_ne!` can be removed — it
  was defending against the wrong constant family.

- **D-01b:** No other call sites or constants need updating. `CMD_LED_OFF` is referenced
  only by `led_off` (the action handler) and `test_cmd_led_off_constant_value`. The
  allowlist already permits `"led_off"` — no change there.

### Bug 2 — Shared crate-level ENV_LOCK

- **D-02:** Add to `lib.rs` under `#[cfg(test)]`:
  ```rust
  pub(crate) static ENV_LOCK: std::sync::Mutex<()> = std::sync::Mutex::new(());
  ```
  Use `std::sync::Mutex` (not tokio) — consistent with the type both modules currently use.

- **D-02a:** In `hexapod_tcp.rs` test module: remove the local `use std::sync::Mutex;`
  import and local `static ENV_LOCK` declaration; add `use crate::ENV_LOCK;`. All existing
  `ENV_LOCK.lock()` call sites are unchanged.

- **D-02b:** In `hexapod_video.rs` test module: same replacement. Additionally remove the
  stale comment at lines 232–233 explaining cross-module races (the fix makes it false).

- **D-02c:** The lock acquisition pattern `ENV_LOCK.lock().unwrap_or_else(|p| p.into_inner())`
  is correct (poison-tolerant) and must be preserved at all call sites — do not simplify to
  `.unwrap()`.

### Audit findings (full audit, D-03)

- **skills_tool.rs** (line 892): has its own module-local `ENV_LOCK` aliased as `StdMutex`.
  It protects `HERMES_TEST_MISSING_KEY`, `HERMES_TEST_PRESENT_KEY`, `HERMES_TEST_MIXED_KEY`,
  and `TENOR_API_KEY` — none of which overlap with `HEXAPOD_IP`. No cross-module conflict
  exists; **leave as-is**. Migrating it to the crate-level lock would create spurious
  serialization between unrelated test domains.

- **browser_session.rs** (lines 291–368): mutates `BROWSER_PATH`, `CHROMIUM_PATH`, `PATH`
  without any lock. These vars don't interact with `HEXAPOD_IP`. The races are
  browser_session-internal and not the subject of this phase. **Deferred** — see Deferred.

### Static-invariant test (D-04)

New file `crates/ironhermes-tools/tests/invariants_27_1_4_2.rs`. Pattern: `include_str!` +
`assert!(SOURCE.contains(...))`. Two assertions:
1. `hexapod_tcp.rs` contains `"CMD_LED_MOD#0\n"` — locks the correct off command.
2. `hexapod_tcp.rs` does NOT contain `"CMD_LED#0\n"` (plain, without `_MOD`) — locks out
   silent regression to the wrong constant. Use `assert!(!SOURCE.contains(...))`.
Both assertions should name PROV-HEXAPOD (or the relevant requirement) and cite phase
27.1.4.2 in their failure messages, matching the style of `invariants_27_1_4_1.rs`.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- All `ENV_LOCK.lock().unwrap_or_else(|p| p.into_inner())` call sites are syntactically
  identical in both modules — only the `use` path and `static` declaration change.
- `lib.rs` currently has no `#[cfg(test)]` block; add one near the end with the new static.
  `std::sync::Mutex` is in std, no import needed at the crate root beyond fully-qualifying it.

### Established Patterns
- **Module-local ENV_LOCK** (skills_tool.rs): uses `Mutex as StdMutex` alias to distinguish
  from tokio Mutex. The crate-level lock should use plain `std::sync::Mutex` (no alias needed)
  since it is only accessed from `#[cfg(test)]` contexts where no tokio Mutex is in scope.
- **Static invariant tests**: `include_str!("../src/<file>.rs")` + `assert!(... .contains(...))`
  with no dev-deps. Negative assertion (`assert!(! ... .contains(...))`) is new for this phase
  but is just standard Rust — no special pattern needed.

### Integration Points
- Single edit site in `hexapod_tcp.rs` for Bug 1: line 58 constant.
- Single test edit site for Bug 1: `test_cmd_led_off_constant_value`.
- Two modules for Bug 2: remove local static, add `use crate::ENV_LOCK;`.
- One lib.rs addition: `#[cfg(test)]` block with the new crate-level static.

</code_context>

<specifics>
## Specific Ideas

- **The negative assertion is the safety net.** `!SOURCE.contains("CMD_LED#0\n")` in the
  invariant test is what prevents a future refactor from silently reverting to the wrong
  constant. The positive assertion alone doesn't protect against e.g. a copy-paste that
  adds both the old and new strings.

- **Lock rename consideration.** To make the crate-level lock self-documenting, consider
  naming it `HEXAPOD_ENV_LOCK` instead of the generic `ENV_LOCK`. This makes the usage
  intent clear at the import site (`use crate::HEXAPOD_ENV_LOCK;`) and avoids any future
  confusion if other crate-level test locks are added. **Claude's discretion** — either
  name is acceptable; whatever ships, use it consistently across both modules.

- **Verify the full test suite passes.** After the fix, running
  `cargo test -p ironhermes-tools --lib` must show 0 failures, including
  `test_capture_frame_passes_allowlist` which was the observed flaky failure.

</specifics>
