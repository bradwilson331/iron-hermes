# Phase 27.1.4.2: Code Review Report

**Reviewed:** 2026-05-14T00:00:00Z
**Depth:** standard
**Files Reviewed:** 4

## Summary

The primary fix — replacing `CMD_LED#0\n` with `CMD_LED_MOD#0\n` for `led_off` — is correctly implemented. `CMD_LED_OFF` is set to `"CMD_LED_MOD#0\n"` at line 59, `led_off` sends it via `send_fire_and_forget`, and both invariant tests in the regression gate file will pass against the source text. The wire-protocol constants, battery/distance parsers, camera and buzzer handlers are all structurally sound.

However, there is one critical correctness bug: the `rotate(0)` test comment asserts the zero-degrees guard fires before the env-var read, but the actual implementation reads `HEXAPOD_IP` unconditionally before reaching the inner `degrees == 0` check. The test itself works around this silently by setting the env var to a non-routable address — masking the stated invariant. There are also three warnings: the `parse_battery_response` unwrap-or silently treats parse failures as 0 V (triggering a false LOW alert), the `stream_distance` loop swallows malformed distance values as 0 cm (skewing statistics), and the ENV_LOCK mutex is not consistently held in all async tests that modify `HEXAPOD_IP`.

## Critical Issues

### CR-01: `rotate(0)` zero-guard fires after env-var read, contradicting stated invariant

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:424-426`

**Issue:** The comment on test 16 (line 856–859) asserts "execute() must return Ok("OK") for degrees=0 before reading the env var." The source code does not satisfy this invariant. The outer `match action` arm at line 344–348 is entered for `"rotate"`, which immediately reads `HEXAPOD_IP` at line 350–357 before the inner `match action` reaches the `degrees == 0` short-circuit at line 425. The test masks this by setting `HEXAPOD_IP` to `"127.0.0.255"` and only asserting `result == "OK"` — a no-TCP-connection coincidence (the rotate(0) guard returns before any connect call), not evidence that the env-var read was skipped.

This matters for correctness: if the stated design invariant (D-12: HEXAPOD_IP read only for permitted actions, allowlist guards first) implies that a `rotate(degrees=0)` call with an unset `HEXAPOD_IP` should return `"OK"` rather than the env-var error, the current code returns the env-var error instead. This is a behavioral discrepancy from the documented contract.

**Fix:** Move the `degrees == 0` no-op guard to before the env-var read, as the comment implies:

```rust
"rotate" => {
    let degrees = args["degrees"].as_i64().unwrap_or(0);
    if degrees == 0 {
        return Ok("OK".to_string()); // no-op before any I/O
    }
    // D-12: read HEXAPOD_IP only when work will actually be done
    let ip = match env::var("HEXAPOD_IP") { ... };
    ...
}
```

Or, if the current behavior (env-var check runs first) is intentional, remove the misleading comment from the test and update the test to assert the env-var error when HEXAPOD_IP is unset.

## Warnings

### WR-01: `parse_battery_response` silently treats unparseable voltage fields as 0 V, triggering false LOW

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:147-148`

**Issue:** `s1.parse().unwrap_or(0.0)` and `s2.parse().unwrap_or(0.0)` silently substitute 0 V when a field is not a valid float. `0.0 < 5.5` is true and `0.0 < 6.0` is true, so a malformed response (e.g., garbled frame, unexpected protocol change) produces `Battery: 0V / 0V (LOW)` instead of an error. The LLM may then take incorrect follow-up actions (e.g., issuing an emergency stop or warning the user of a dead battery) based on a phantom reading. The `match` arm already handles the case where fields are missing by returning an error string — the parse-failure case should be treated the same way.

**Fix:**
```rust
(Some(s1), Some(s2)) => {
    let v1 = match s1.parse::<f32>() {
        Ok(v) => v,
        Err(_) => return format!(
            "Error: unexpected battery response from robot: {:?}", raw.trim()
        ),
    };
    let v2 = match s2.parse::<f32>() {
        Ok(v) => v,
        Err(_) => return format!(
            "Error: unexpected battery response from robot: {:?}", raw.trim()
        ),
    };
    let status = if v1 < BATTERY_LOW_V1 || v2 < BATTERY_LOW_V2 { "LOW" } else { "OK" };
    format!("Battery: {v1}V / {v2}V ({status})")
}
```

### WR-02: `stream_distance` substitutes 0 cm for unparseable distance readings, silently corrupting statistics

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:557-560`

**Issue:** The numeric parse in the `stream_distance` loop uses `.unwrap_or(0)` at line 559:

```rust
let dist = parts.get(1)
    .and_then(|s| s.trim().parse::<i64>().ok())
    .unwrap_or(0);
```

If any response frame is malformed (field missing, non-numeric), 0 is silently pushed into `readings`. Since a valid sensor reading of 0 cm is impossible for an ultrasonic sensor with a non-zero minimum range, a spurious 0 will drag down `min` and `avg` while appearing like a valid data point. The error case should be surfaced to the caller, consistent with how `raw.starts_with("Error:")` is handled two lines above.

**Fix:** Return an error on parse failure rather than substituting 0:
```rust
let dist = match parts.get(1).and_then(|s| s.trim().parse::<i64>().ok()) {
    Some(v) => v,
    None => return Ok(format!(
        "Error: unexpected distance response from robot: {:?}", raw.trim()
    )),
};
```

### WR-03: `ENV_LOCK` not held in `test_new_actions_not_blocked` (test 20), racing on `HEXAPOD_IP`

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:904-920`

**Issue:** Test 20 (`test_new_actions_not_blocked`) calls `unsafe { std::env::remove_var("HEXAPOD_IP") }` at line 907 under the `ENV_LOCK` guard. However, the test iterates over five actions including `"rotate"` with `degrees: 0`. Because the rotate handler reads `HEXAPOD_IP` before checking `degrees == 0` (see CR-01), the action will reach `env::var("HEXAPOD_IP")` — but with the env var removed, so it returns the error string. The test assertion only checks `!result.starts_with("Action '")`, so the env-var error satisfies the assertion and the test passes. However, the guard is held correctly here; this is a latent issue that interacts with CR-01.

Additionally, `test_new_actions_not_blocked_27_1_4` (line 986) acquires ENV_LOCK but the `stream_distance` action with `samples: 1` will attempt a real TCP connection after the env-var check passes (env var is unset, so it returns the error string — test is correct but fragile: any reorder that sets the env var in a concurrent test could cause a live TCP attempt to `127.0.0.255:5002` that blocks for the OS connect timeout before the async test framework cancels it).

**Fix:** Ensure all tests that call `execute()` with actions that reach the TCP layer either:
1. Unset `HEXAPOD_IP` and assert the env-var error, or
2. Set `HEXAPOD_IP` to a non-routable address and assert the connection-error string.

The current approach is mostly correct; the fragility comes from CR-01's ordering issue. Fixing CR-01 makes these tests fully reliable.

### IN-01: `invariants_27_1_4_2.rs` test 2 (`hexapod_tcp_does_not_use_cmd_led_plain_for_led_off`) is a source-text grep and can produce a false negative

**File:** `crates/ironhermes-tools/tests/invariants_27_1_4_2.rs:20-28`

**Issue:** The negative invariant asserts that `hexapod_tcp.rs` must not contain the string `CMD_LED#0\n` anywhere in source. This is a sound guard against the specific regression of reverting to the old command. However, `CMD_LED` is also used as a constant name (`pub(crate) const CMD_LED: &str`) and in comment text. A future contributor adding a comment like `// previously used CMD_LED#0\n` or adding a new test that asserts the old string is NOT used in a wire command would trip this invariant even if the implementation is correct. The grep is overly broad.

**Fix:** Narrow the grep to the implementation body only, or match on the constant definition pattern:
```rust
// Match only the wire string in the led_off constant or send call, not comments or tests
assert!(
    !HEXAPOD_TCP_SOURCE.contains("CMD_LED#0\\n"),
    ...
);
```
The current approach is already the right string, but documenting this caveat in a comment above the assertion would prevent future confusion.

### IN-02: Dead default branch in `build_walk_wire` is unreachable but undocumented as such

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:124`

**Issue:** The `_` arm of `build_walk_wire` is documented as "safe default: forward" in the comment. The function doc comment at line 116 states "execute() validates direction before calling this helper, so the default branch is a belt-and-suspenders guard." This is accurate, but the function is `pub(crate)` and nothing stops an internal caller from passing an unvalidated string. The fallback silently sends a forward motion command rather than returning an error, which could cause unexpected robot movement if called with an invalid direction outside of `execute()`.

**Fix:** Consider returning an `Option<String>` or `Result` from `build_walk_wire`, or panic with an explicit message, to make misuse visible:
```rust
_ => panic!("build_walk_wire: unexpected direction '{direction}' — execute() should have validated"),
```
Or document the current behavior explicitly and accept the silent forward fallback as intentional.
