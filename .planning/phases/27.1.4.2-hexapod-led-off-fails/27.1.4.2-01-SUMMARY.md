# Phase 27.1.4.2 Plan 01: Hexapod LED-Off Bugs Summary

**One-liner:** Fixed hexapod led_off wire command from wrong CMD_LED color channel to correct CMD_LED_MOD#0 mode channel, and resolved cross-module ENV_LOCK race by introducing a single crate-level shared mutex in lib.rs.

## What Was Built

### Bug 1 Fix — Correct LED Off Wire Protocol
`CMD_LED_OFF` was `"CMD_LED#0\n"` (color channel, silently ignored). Changed to `"CMD_LED_MOD#0\n"` (mode channel, triggers `color_wipe([0,0,0])` in the Freenove server's `CMD_LED_MOD` handler per `led.py`). The unit test `test_cmd_led_off_constant_value` was repurposed from a wrong positive assertion to a single correct one. No `CMD_LED#0\n` substring remains anywhere in `hexapod_tcp.rs`.

### Bug 2 Fix — Cross-Module ENV_LOCK Race
`hexapod_tcp::tests` and `hexapod_video::tests` each had their own `static ENV_LOCK: Mutex<()>`. Two separate statics race on `HEXAPOD_IP`, causing `test_capture_frame_passes_allowlist` to see a `set_var("HEXAPOD_IP", "127.0.0.255")` from a concurrent hexapod_tcp test. Fix: single `pub(crate) static ENV_LOCK` added to `lib.rs` under `#[cfg(test)]`; both modules now import it via `use crate::ENV_LOCK`. All 9 hexapod_tcp and 3 hexapod_video poison-tolerant lock sites (`unwrap_or_else(|p| p.into_inner())`) preserved unchanged.

### Task 3 — Static-Invariant Regression Gates
New file `crates/ironhermes-tools/tests/invariants_27_1_4_2.rs` contains two tests:
1. `hexapod_tcp_uses_cmd_led_mod_for_led_off` — positive: source must contain `CMD_LED_MOD#0\n`
2. `hexapod_tcp_does_not_use_cmd_led_plain_for_led_off` — negative: source must NOT contain `CMD_LED#0\n`

Both failure messages cite `PROV-HEXAPOD` and `phase 27.1.4.2`. Pattern follows `invariants_27_1_4_1_1.rs` exactly.
