### `crates/ironhermes-tools/src/lib.rs` (add crate-level test lock)

**Analog:** `crates/ironhermes-tools/tests/toolset_prereq.rs` (OnceLock/Mutex env-lock
pattern) and `hexapod_tcp.rs` existing `static ENV_LOCK: Mutex<()>` type.

The CONTEXT.md D-02 decision specifies using a plain `static` (not OnceLock) to match
the type both modules currently use. The name is `ENV_LOCK` per D-02 (or `HEXAPOD_ENV_LOCK`
per the Specific Ideas section — planner picks one and uses it consistently).

**Addition to end of lib.rs** (after line 36, the last `pub use` line):
```rust
// ---------------------------------------------------------------------------
// Crate-level test utilities
// ---------------------------------------------------------------------------

/// Shared env-var serialization lock for all hexapod test modules.
///
/// Replaces the per-module `static ENV_LOCK` in `hexapod_tcp::tests` and
/// `hexapod_video::tests`, which raced on `HEXAPOD_IP` across modules.
/// All call sites use: `ENV_LOCK.lock().unwrap_or_else(|p| p.into_inner())`
#[cfg(test)]
pub(crate) static ENV_LOCK: std::sync::Mutex<()> = std::sync::Mutex::new(());
```

Key constraints from D-02:
- Use `std::sync::Mutex` (not `tokio::sync::Mutex`) — matches existing call sites.
- `pub(crate)` visibility — accessible from `hexapod_tcp::tests` and
  `hexapod_video::tests` via `use crate::ENV_LOCK`.
- Wrapped in `#[cfg(test)]` — not compiled into production binary.
- No `use` import needed at the crate root; qualify fully as `std::sync::Mutex`.
