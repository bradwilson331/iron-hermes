# Phase 21.8.3.1: personality applied doesn't change the llm responses - Context

**Gathered:** 2026-05-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix the bug where `/personality <name>` in all real agent surfaces (TUI ratatui + Telegram gateway) shows "Personality applied" but the LLM receives the same unmodified system message — because `next_turn_personality_overlay` is set but never consumed when building the per-turn message history.

**In scope:** TUI fix, gateway fix, `/personality clear` mechanic, field rename for accuracy.
**Out of scope:** Web UI (iron_hermes_ui is mock-only, personality Signal already drives the status bar pill — no agent connection exists yet).

</domain>

<decisions>
## Implementation Decisions

### Scope
- **D-01:** Fix covers two real agent surfaces only: TUI (ratatui, `ironhermes-cli`) and Telegram gateway (`ironhermes-gateway`). The web UI (`iron_hermes_ui`) is mock-only with no LLM backend connection — its `Signal<Personality>` already drives the status bar pill correctly. No web UI changes in this phase.

### TUI Fix (event_loop.rs + app.rs + commands.rs)

- **D-02:** Rename `App.next_turn_personality_overlay: Option<String>` → `App.active_personality_overlay: Option<String>`. The old name implies single-turn consumption; the correct behavior is session-persistent until changed or cleared. Rename everywhere: struct field definition, `App::new` initializer, `handle_subsystem_mutator` setter.

- **D-03:** In `spawn_turn` (`event_loop.rs:717`), after cloning `app.history` into `messages_snapshot`, if `app.active_personality_overlay` is `Some(overlay_text)` and `messages_snapshot` is non-empty, append `"\n\n{overlay_text}"` to `messages_snapshot[0].content` (the system message clone). This modifies the local per-turn clone only — `app.history[0]` is never mutated. The overlay persists naturally because the field is never cleared after injection.

- **D-04:** Do NOT clear `active_personality_overlay` after each turn. The field stays `Some(text)` across all subsequent turns until the user applies a different personality (overwrites it) or runs `/personality clear` (sets it to `None`).

- **D-05:** In `handle_subsystem_mutator` (`commands.rs`), add `"clear"` as a special case in the `"personality"` arm — checked BEFORE the registry lookup. When `args[0] == "clear"`: set `app.active_personality_overlay = None` and return `SlashOutcome::Handled("Personality cleared.")`. This gives users a way to revert to default identity without restarting the session.

- **D-06:** The overlay injection format for the TUI path (where PromptBuilder is not called per-turn) mirrors PromptBuilder's slot 8 (SessionOverlay) behavior: the raw overlay text is appended after the durable system message content separated by `"\n\n"`. No header prefix like "Active personality:" — just the raw preset text (e.g., "Arrr! Respond as a salty sea pirate!...").

### Gateway Fix (handler.rs)

- **D-07:** Add `active_personality_overlay: Option<String>` field to `GatewayHandler` struct (`handler.rs`). Initialize to `None` in the constructor.

- **D-08:** Wire the gateway's command dispatch to update `self.active_personality_overlay` when `/personality <name>` succeeds. The gateway processes slash commands and receives a `CommandResult::Output(overlay_text)` — detect the personality apply case using the same heuristic as the TUI's `handle_subsystem_mutator`: `Output(text)` that doesn't start with "Available" / "Personality registry" / "No personalities". On match: `self.active_personality_overlay = Some(text)`. On `/personality clear`: `self.active_personality_overlay = None`.

- **D-09:** In the per-turn PromptBuilder setup (`handler.rs:757-789`), after `prompt_builder.load_skills()` and before `prompt_builder.build_system_message()`, call `prompt_builder.set_overlay(text.clone())` when `self.active_personality_overlay.is_some()`. This uses the existing `PromptBuilder::set_overlay()` API which places the text in slot 8 (SessionOverlay, ephemeral) — no format changes needed.

### Claude's Discretion

- Whether to surface "Active personality: {name}" in the gateway's confirmation reply when applying (the TUI already does this via `SlashOutcome::Handled(format!("Personality applied ({} chars). Active next turn.", ...))`) — gateway equivalent is whatever the gateway sends back as the command result text.
- Whether "clear" should appear in the `/personality` (no-args) list output as a hint — implementation detail; a trailing line "Use /personality clear to reset." is reasonable.
- Gateway command dispatch wiring location: the personality-specific logic may live in the same place as the TUI's `handle_subsystem_mutator`, or inline in the gateway's command processing path — researcher should identify the exact dispatch site.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `PromptBuilder::set_overlay(text: String)` / `clear_overlay()` — already exists in `prompt_builder.rs:164-172`; the gateway fix (D-09) uses this directly, no new API needed
- `handle_subsystem_mutator` pattern in `commands.rs:956` — established pattern for TUI-side command post-processing; the "personality" arm (line 1024) is where D-05 clear goes

### Established Patterns
- **Per-turn clone mutation**: `spawn_turn` already clones `app.history` into `messages_snapshot` (line 730) before passing to `agent.run()`. The D-03 overlay injection follows this pattern — mutate the local clone, never the canonical `app.history`.
- **Session-persistent overlay**: `app.fast_enabled`, `app.verbose_enabled` are Arc atomics that persist across turns without being cleared. `active_personality_overlay: Option<String>` follows the same session-persistent-state pattern.
- **Gateway per-turn PromptBuilder**: The gateway already builds `PromptBuilder` fresh each turn (handler.rs:757). Calling `set_overlay()` before `build_system_message()` fits this pattern exactly — same as `activate_skill()` (line 785).

### Integration Points
- TUI: `spawn_turn` (`event_loop.rs:717`) reads from `&App` — add `app.active_personality_overlay.as_deref()` read there
- Gateway: `GatewayHandler::handle_message()` or equivalent dispatch method — where the `/personality` command result is currently handled (researcher should confirm exact method name and line)
- Both: The `cmd_personality` core handler (`handlers.rs:586`) returns `CommandResult::Output(overlay_text)` unchanged — no changes to core handler needed

</code_context>

<specifics>
## Specific Ideas

- The user confirmed the bug is real: `/personality pirate` shows "Personality applied (N chars). Active next turn." but the LLM response style is unchanged — the overlay text never reaches the LLM.
- `app.rs:248` has a comment documenting the deferred consume: "Consumed (and cleared) by spawn_turn bootstrap (Plan 03 scope: set only; consume deferred)." — this comment should be updated to reflect D-03/D-04 (persistent, inject per-turn into clone).
- The gateway path was confirmed to also miss personality wiring: `handler.rs:84` has a comment "scope for this phase — matches /personality overlay semantics" but no field was ever added.
- `/personality clear` is the canonical clear mechanic. The "restart session" approach was explicitly rejected.

</specifics>
