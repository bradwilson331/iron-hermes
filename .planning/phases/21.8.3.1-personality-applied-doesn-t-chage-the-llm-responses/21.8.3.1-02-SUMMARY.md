# Phase 21.8.3.1 Plan 02: Gateway Personality Overlay Wiring Summary

Gateway personality overlay wiring with per-session Arc<Mutex<HashMap>> state, pre-dispatch clear intercept, Output-arm heuristic detection, and per-turn PromptBuilder set_overlay() injection.

## What Was Built

### Task 1: D-07 - active_personality_overlay field (struct + constructor)

Added `active_personality_overlay: Arc<std::sync::Mutex<std::collections::HashMap<SessionKey, String>>>` to `GatewayMessageHandler` struct with doc comment referencing D-07/D-08/D-09. Constructor initializes to empty HashMap (matching `skill_overlays` pattern). The field is private.

### Task 2: D-05/D-08/D-09 - Three insertions in handler.rs

**Insertion A (D-05 gateway analog):** Pre-dispatch check `def.name == "personality" && args.first() == Some(&"clear")` before `handlers::dispatch()`. On match: removes the session key from overlay map, sends `"Personality cleared."` via `with_rate_limit_retry`, returns `Ok(())` without calling core.

**Insertion B (D-08):** `CoreCommandResult::Output(text)` arm expanded with `is_personality_apply` detection (name == "personality" AND text NOT prefixed with "Available"/"Personality registry"/"No personalities"). On personality apply: inserts `text` into overlay map keyed by `session_key`, sends `"Personality applied (N chars). Active for this session."` confirmation instead of echoing raw overlay text.

**Insertion C (D-09):** Per-turn `PromptBuilder` setup now reads `self.active_personality_overlay` and calls `prompt_builder.set_overlay(overlay_text.clone())` after the `skill_overlays` loop and before `build_system_message()`. Overlay is re-applied every turn from the map; no explicit clearing between turns (entry absent = no personality active).

### Task 3: Core regression test

Added `personality_clear_at_core_returns_unknown_error` to `crates/ironhermes-core/tests/handlers_personality.rs`. Dispatches `&["clear"]` directly to `cmd_personality` via the standard `dispatch()` path and asserts `CommandResult::Error` is returned. Documents and locks the contract: "clear" is NOT a core-level case. Surface handlers (TUI pre-check from Plan 01, gateway pre-dispatch from Plan 02) MUST intercept before core. Guards against future refactors.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Interior mutability required for active_personality_overlay field**
- **Found during:** Task 2 — compilation failed with `error[E0594]: cannot assign to self.active_personality_overlay, which is behind a & reference`
- **Issue:** The plan specified `active_personality_overlay: Option<String>` and assumed `handle_slash_command` takes `&mut self`. In reality, the handler is wrapped in `Arc<GatewayMessageHandler>` (runner.rs:481) and `handle_slash_command` takes `&self`. Plain `Option<String>` cannot be mutated through `&self`.
- **Fix:** Changed field type to `Arc<std::sync::Mutex<std::collections::HashMap<SessionKey, String>>>` mirroring the existing `skill_overlays` pattern exactly. All three insertion sites updated to use lock/insert/remove/get pattern. Per-session keying (by SessionKey) prevents cross-user personality bleed in multi-user gateway deployments.
- **Files modified:** `crates/ironhermes-gateway/src/handler.rs` (field definition, constructor, all three insertion sites)
- **Commit:** `2f471bc9`
