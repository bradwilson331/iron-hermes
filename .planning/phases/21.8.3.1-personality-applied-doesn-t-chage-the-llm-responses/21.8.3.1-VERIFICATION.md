## Architectural Deviation: D-07 Field Type

**Plan 02 specified:** `active_personality_overlay: Option<String>` (private, plain field)

**Actual implementation:** `active_personality_overlay: Arc<std::sync::Mutex<std::collections::HashMap<SessionKey, String>>>`

**Reason:** `handle_slash_command` (and the per-turn message handler) take `&self`, not `&mut self`. Mutating a plain `Option<String>` field would require `&mut self`. The executor mirrored the pattern already established by `skill_overlays` (same type, same lock/get/insert/remove access pattern), which uses interior mutability for the same reason. This is a correct, non-regressive adaptation — the functional contract of D-07 through D-09 is fully satisfied:

- Per-session overlay is stored (keyed by `SessionKey`, preventing cross-user bleed — a security improvement over plain `Option<String>`)
- Clear removes the session entry (`overlays.remove(&session_key)`)
- Per-turn read uses `overlays.get(&key)` to retrieve the active overlay
- The executor documented this deviation in the field's doc comment at `handler.rs:153-155`

This deviation does NOT require an override — the goal is achieved and the implementation is more correct than the plan assumed.
