# Phase 21.8.3.1: personality-applied-doesn-t-chage-the-llm-responses ‚Äî Research

**Researched:** 2026-05-10
**Domain:** Rust / ratatui TUI + Telegram gateway ‚Äî personality overlay injection bug fix
**Confidence:** HIGH (all claims verified against live codebase)

## Summary

This is a narrow bug-fix phase with a fully explored codebase. The root cause is confirmed: `App.next_turn_personality_overlay` is written by `handle_subsystem_mutator` when `/personality <name>` succeeds, but `spawn_turn` never reads it ‚Äî the field is dead write. The per-turn `messages_snapshot` clone passes directly to `agent.run()` without the overlay appended to `messages_snapshot[0].content`. The gateway has no `active_personality_overlay` field at all; its `CoreCommandResult::Output(text)` arm for the personality command sends the overlay text back to the Telegram chat as a plain message and discards it.

The fix has two independent surfaces: (1) TUI `event_loop.rs` ‚Äî read the overlay field and mutate `messages_snapshot[0].content` before the `agent.run()` call; (2) Gateway `handler.rs` ‚Äî add the field, detect the personality `Output(text)` result in the command dispatch match, store it, and call `prompt_builder.set_overlay()` each turn. All supporting APIs (`PromptBuilder::set_overlay`, `PromptBuilder::clear_overlay`, `PersonalityRegistry`, `CommandContext.personality_overlay`) are already implemented and require zero changes.

**Primary recommendation:** All decisions are locked in CONTEXT.md (D-01..D-09). This phase has no discretionary architecture choices remaining ‚Äî it is a pure implementation of well-defined changes to four files plus a test addition.

### Locked Decisions

- **D-01:** Fix covers two real agent surfaces only: TUI (ratatui, `ironhermes-cli`) and Telegram gateway (`ironhermes-gateway`). The web UI (`iron_hermes_ui`) is mock-only with no LLM backend connection ‚Äî its `Signal<Personality>` already drives the status bar pill correctly. No web UI changes in this phase.
- **D-02:** Rename `App.next_turn_personality_overlay: Option<String>` ‚Üí `App.active_personality_overlay: Option<String>`. The old name implies single-turn consumption; the correct behavior is session-persistent until changed or cleared. Rename everywhere: struct field definition, `App::new` initializer, `handle_subsystem_mutator` setter.
- **D-03:** In `spawn_turn` (`event_loop.rs:717`), after cloning `app.history` into `messages_snapshot`, if `app.active_personality_overlay` is `Some(overlay_text)` and `messages_snapshot` is non-empty, append `"\n\n{overlay_text}"` to `messages_snapshot[0].content` (the system message clone). This modifies the local per-turn clone only ‚Äî `app.history[0]` is never mutated. The overlay persists naturally because the field is never cleared after injection.
- **D-04:** Do NOT clear `active_personality_overlay` after each turn. The field stays `Some(text)` across all subsequent turns until the user applies a different personality (overwrites it) or runs `/personality clear` (sets it to `None`).
- **D-05:** In `handle_subsystem_mutator` (`commands.rs`), add `"clear"` as a special case in the `"personality"` arm ‚Äî checked BEFORE the registry lookup. This gives users a way to revert to default identity without restarting the session.
- **D-06:** The overlay injection format for the TUI path (where PromptBuilder is not called per-turn) mirrors PromptBuilder's slot 8 (SessionOverlay) behavior: the raw overlay text is appended after the durable system message content separated by `"\n\n"`. No header prefix like "Active personality:" ‚Äî just the raw preset text.
- **D-07:** Add `active_personality_overlay: Option<String>` field to `GatewayHandler` struct (`handler.rs`). Initialize to `None` in the constructor.
- **D-08:** Wire the gateway's command dispatch to update `self.active_personality_overlay` when `/personality <name>` succeeds. Detect the personality apply case using the same heuristic as the TUI's `handle_subsystem_mutator`: `Output(text)` that doesn't start with "Available" / "Personality registry" / "No personalities". On match: `self.active_personality_overlay = Some(text)`. On `/personality clear`: `self.active_personality_overlay = None`.
- **D-09:** In the per-turn PromptBuilder setup (`handler.rs:757-789`), after `prompt_builder.load_skills()` and before `prompt_builder.build_system_message()`, call `prompt_builder.set_overlay(text.clone())` when `self.active_personality_overlay.is_some()`. This uses the existing `PromptBuilder::set_overlay()` API which places the text in slot 8 (SessionOverlay, ephemeral) ‚Äî no format changes needed.

### Pattern 3: Gateway personality detection in command dispatch

**What:** The gateway's command dispatch `CoreCommandResult::Output(text)` arm currently sends all Output to Telegram. The personality apply case must be intercepted before sending and stored.
**When to use:** The detection heuristic is the same three-string exclusion as the TUI's `handle_subsystem_mutator`.

```rust
// Source: handler.rs:407 (existing Output arm) + D-08 detection
CoreCommandResult::Output(text) => {
    // D-08: detect personality apply (same heuristic as TUI handle_subsystem_mutator)
    let is_personality_apply = def.name == "personality"
        && !text.starts_with("Available")
        && !text.starts_with("Personality registry")
        && !text.starts_with("No personalities");

    if is_personality_apply {
        self.active_personality_overlay = Some(text.clone());
        let confirm = format!("Personality applied ({} chars). Active for this session.", text.len());
        with_rate_limit_retry(|| adapter.send_message(&event.chat_id, &confirm, None))
            .await?;
    } else {
        with_rate_limit_retry(|| adapter.send_message(&event.chat_id, &text, None))
            .await?;
    }
}
```

[VERIFIED: handler.rs:407 confirmed `CoreCommandResult::Output(text)` arm; `def.name` is available in scope (line 399 `ResolveResult::Exact(def) | ResolveResult::PrefixMatch(def)`)]

**Note on `/personality clear` in gateway:** The "clear" arg is intercepted by `cmd_personality` in `handlers.rs` only if the core handler is taught "clear". Currently `cmd_personality` has no "clear" case ‚Äî it looks up "clear" as a preset name and returns `CommandResult::Error("Unknown personality: clear")`. The TUI handles "clear" in the post-router `handle_subsystem_mutator` BEFORE the core handler runs (D-05). For the gateway, the detection must similarly happen before or after the core dispatch.

**Gateway clear options (Claude's Discretion ‚Äî D-05 for gateway is not explicitly locked):**
- Option A: Check `args[0] == "clear"` inside the `CoreCommandResult::Error` arm (since core returns Error for unknown preset "clear") ‚Äî fragile, couples to error message text.
- Option B: Check `def.name == "personality" && args.first() == Some(&"clear")` BEFORE calling `dispatch()` ‚Äî consistent with TUI pattern.
- Option C: Add "clear" handling to `cmd_personality` in core (`handlers.rs`) ‚Äî returns `CommandResult::Output("Personality cleared.")` and gateway detects by gateway-side field clear. Cleanest but changes core behavior.

**Recommendation:** Option B ‚Äî add a pre-dispatch check in `handle_slash_command` for `def.name == "personality" && args[0] == "clear"` that sets `self.active_personality_overlay = None` and sends confirmation without calling core dispatch. Mirrors TUI D-05 pattern exactly.

### Anti-Patterns to Avoid

- **Mutating `app.history[0]` directly:** The overlay must only be injected into the per-turn `messages_snapshot` clone, never the canonical history. Mutating `app.history` would permanently corrupt the system message.
- **Clearing the overlay after each turn:** D-04 explicitly prohibits this. The overlay is session-persistent by design.
- **Using `PromptBuilder::clear_overlay()` in the gateway:** The overlay is re-set from `self.active_personality_overlay` on every turn; no explicit clear between turns is needed.
- **Routing "clear" through `cmd_personality` core handler:** The core handler does not know about "clear" ‚Äî it will return `Error("Unknown personality: clear")`. The clear mechanic is a surface-level concern handled in the post-router (TUI) or pre-dispatch (gateway).

## Common Pitfalls

### Pitfall 1: spawn_turn takes &App not &mut App

**What goes wrong:** The planner or executor tries to declare `messages_snapshot` as immutable (matching the existing `let messages_snapshot = app.history.clone()`) and then mutates it.
**Why it happens:** The existing line 730 uses `let messages_snapshot = ...` (immutable binding). Injection requires mutation.
**How to avoid:** Change line 730 to `let mut messages_snapshot = app.history.clone();` ‚Äî `spawn_turn` receives `app: &App` (not `&mut App`), so the `mut` applies to the local binding only. No signature change needed.
**Warning signs:** Compiler error "cannot borrow `messages_snapshot` as mutable, as it is not declared as mutable."

### Pitfall 2: messages_snapshot[0] panics on empty history

**What goes wrong:** If `app.history` is empty (no system message yet), indexing `messages_snapshot[0]` panics.
**Why it happens:** D-03 guards with `messages_snapshot.is_empty()` check but a careless implementation skips the guard.
**How to avoid:** Always check `!messages_snapshot.is_empty()` before accessing `messages_snapshot[0]`.
**Warning signs:** Runtime panic in `spawn_turn` on fresh sessions with no history.

### Pitfall 3: Gateway Output arm has no def.name available at the wrong scope

**What goes wrong:** The personality detection in the gateway `Output` arm needs `def.name` to avoid treating all `Output` text as personality overlays.
**Why it happens:** `def` is bound in the outer `match self.command_router.resolve(...)` at line 398.
**How to avoid:** `def.name` is in scope inside the `CoreCommandResult::Output` arm ‚Äî verified from the code structure at handler.rs:398-407. No restructuring needed.
**Warning signs:** Compiles but all Output commands are treated as personality overlays.

### Pitfall 4: set_overlay takes String not &str

**What goes wrong:** Calling `prompt_builder.set_overlay(&overlay_text)` fails to compile.
**Why it happens:** `PromptBuilder::set_overlay(text: String)` takes ownership.
**How to avoid:** Use `.clone()` when the field retains ownership: `prompt_builder.set_overlay(overlay_text.clone())`.
**Warning signs:** Compiler error "expected `String`, found `&String`."

### Pitfall 5: invariants_22_4.rs line 1316 reference

**What goes wrong:** After D-02 rename, the test at invariants_22_4.rs:1316 that references `handle_subsystem_mutator` may fail if it also greps for the old field name.
**Why it happens:** Invariant tests often use `include_str!` + string search against source.
**How to avoid:** Check line 1316 context ‚Äî confirmed it references the function name `handle_subsystem_mutator` only (not the field name), so the rename alone does not break it. However, if any invariant greps for `next_turn_personality_overlay`, it must be updated.
**Warning signs:** `cargo test -p ironhermes-cli --test invariants_22_4` fails with string-match assertion.

### Pitfall 6: Gateway "clear" goes through core which returns Error

**What goes wrong:** The gateway dispatches `/personality clear` to `cmd_personality` which tries to look up "clear" as a preset, returns `CommandResult::Error("Unknown personality: clear")`, and the gateway sends that error to Telegram ‚Äî no field is cleared.
**Why it happens:** The "clear" special case is in the TUI post-router but not in the core handler.
**How to avoid:** Add the "clear" pre-dispatch check in the gateway's slash handler BEFORE calling `dispatch()` ‚Äî identical pattern to TUI D-05.
**Warning signs:** Sending `/personality clear` in Telegram returns "Unknown personality: clear" instead of confirmation.
