# Phase 22.4.1 Plan 02: Bulk Session + Configuration invoke_handler Arms + INV-22.4-33 Summary

## Task 1 — 26 invoke_handler arms (commit `bacfe45`)

`crates/ironhermes-cli/src/tui_rata/commands.rs` edits (single insert):

The 26 arms were inserted as one contiguous block immediately after the Plan-01 `"memory" => CommandResult::Output(...)` arm and immediately before the preserved Phase 22.4 `// UAT Round 2 Gap 5 ...` comment block + `"agents"` arm. Two section comment banners separate the Session block (lines added in the locked registry order from registry.rs lines 22–65) from the Configuration block (registry.rs lines 71–98).

Each arm follows the locked D-08 stub-text format:

```rust
"<name>" => CommandResult::Output(
    "/<name> — <description verbatim from CommandDef>.\n\
     [Args: <args_hint>\n\]\
     Phase 22.4.1 stub: <owning-subsystem rationale>.".to_string()
),
```

The `Args:` line is omitted when the CommandDef has empty `args_hint` (e.g., `/history`, `/save`, `/retry`, `/undo`, `/stop`, `/status`, `/config`, `/provider`, `/statusbar`, `/verbose`, `/yolo`, `/fast`, `/debug` — all 5-arg-less arms).

**Owning-subsystem rationales** (sourced from RESEARCH Finding 2 enumeration):

Session 13:
- history → StateStore FTS5 session history (Phase 13 SESS-01)
- save → Session export (Phase 13 SESS-08)
- retry → AgentLoop last-message retry (Phase 21 session control)
- undo → AgentLoop message-pair removal (Phase 21 session control)
- title → StateStore session title (Phase 13 SESS-04)
- compress → ContextCompressor (Phase 18 PRMT-11)
- rollback → Session checkpoint rollback (Phase 21 session control)
- stop → CancellationToken cascade (Phase 21 D-14)
- background → SubagentRunner background delegation (Phase 21.7 D-09)
- btw → Ephemeral single-turn query (Phase 21 session control)
- queue → Turn queue (Phase 21.7 D-29)
- status → hermes status diagnostics (Phase 21.7 D-18)
- resume → StateStore session restore (Phase 13 SESS-04)

Configuration 13:
- config → config.yaml reader (Phase 23 CFG-02)
- provider → ProviderResolver current endpoint (Phase 12 PROV-01)
- prompt → System prompt override (Phase 15 PRMT-06)
- personality → SOUL.md overlay (Phase 15 PRMT-06/PRMT-07)
- statusbar → TUI status bar toggle (Phase 21 D-03)
- verbose → Verbose tool output toggle (Phase 21 CLI config)
- yolo → Dangerous command auto-approval (Phase 21.7 D-11)
- reasoning → Provider reasoning level (Phase 12 PROV-02)
- skin → Color theme (Phase 22.3 skin config)
- voice → TTS/voice output (future phase)
- model → ProviderResolver model switch (Phase 21.3)
- fast → Fast model preset toggle (Phase 21.3 model config)
- debug → Debug information toggle (Phase 21 CLI config)

**Preserved verbatim per D-06 / D-07 / Plan 01 hand-off:**
- All Plan-01 arms (mouse, mcp, sessions, memory) — byte-identical.
- Phase 22.4 D-06 arms (agents, skills) — byte-identical, location preserved.
- Generic `not yet wired in Phase 22.4` fallback (D-07) — byte-identical.
- `dispatch_slash`, `handle_mouse_slash`, `build_command_context`, `collect_known_command_names`, `render_help_router`, `map_core_to_slash_outcome` — untouched.

**GatewayOnly names DELIBERATELY OMITTED (RESEARCH Pitfall 6):**
- `approve`, `deny`, `sethome`, `start` — these CommandDefs have `.platform(GatewayOnly)`. The router's `PlatformFilter` rejects them on `Platform::Local`, so they would never reach `invoke_handler`. Adding arms for them would create dead code and weaken INV-33's name-tightly-bound contract.

## Task 2 — INV-22.4-33 invoke_handler_arms (commit `d8e5967`)

`crates/ironhermes-cli/tests/invariants_22_4.rs` edits (single append):

`invariant_22_4_33_invoke_handler_arms` appended after `invariant_22_4_34_dispatch_slash_no_strip_prefix`. The test:

1. Defines `expected_arms: &[&str]` containing all 26 names in registry-source order (13 Session + 13 Configuration).
2. Iterates each name, formats `"<name>" => CommandResult::Output(` as a needle, and asserts `TUI_RATA_COMMANDS.contains(&needle)` — fails with a per-name diagnostic if any arm is silently removed by a future refactor.
3. Counts `Phase 22.4.1 stub:` literal substrings via `.matches(...).count()` and asserts the total is `>= expected_arms.len()` (= 26).

**Why floor at 26 and not 30:** Plan-01 contributes 4 occurrences (mouse/mcp/sessions/memory). Flooring at 26 keeps the test robust to Plan-01's count without coupling INV-33 to Plan-01's deliverable. The stricter `>= 30` source-grep is in Plan 02's `<acceptance_criteria>`, not in the test.

**Why GatewayOnly names are omitted from `expected_arms`:** Same reason as Task 1 — RESEARCH Pitfall 6. INV-33's purpose is to grep-lock arms that `invoke_handler` actually reaches on `Platform::Local`. Including GatewayOnly names would either make the test pass against dead arms (false positive) or fail against the correct absence (false negative).

## Phase 22.4.1 Closure Status

The phase's three plans landed in sequence:

- **Plan 22.4.1-00**: Promoted `mouse`/`mcp`/`sessions`/`memory` into the core CommandRouter registry; INV-22.4-32 grep-lock.
- **Plan 22.4.1-01**: Retired the four `strip_prefix` fast-paths; replaced hand-built `render_help` with `render_help_router`; widened `invoke_handler` signature to `(name, _ctx, &CommandRouter)`; deleted 3 dead helpers; inverted INV-29/-31 + added INV-34.
- **Plan 22.4.1-02 (this plan)**: Added 26 dedicated arms for the high-traffic Session + Configuration surface; added INV-33.

After this plan, `tui_rata::dispatch_slash` is **shape-symmetric with classic-tui and gateway**: a single `app.command_router.resolve(input, &platform)` call with one post-router App-side hook for the only stateful command (`/mouse`), and the bulk of routing delegates to `invoke_handler`'s match table whose surface now matches the `Platform::Local`-reachable subset of `build_registry()`.

The generic `not yet wired in Phase 22.4` fallback now only fires for:
- **ToolsAndSkills** category bulk: `tools`, `toolsets`, `cron`, `reload`, `browser`, `plugins` (6 commands).
- **Info** category bulk: `commands`, `usage`, `models`, `insights`, `platforms`, `paste`, `update`, `snapshot`, `profile` (9 commands).

These 15 deferrals are scoped to a future cleanup phase per CONTEXT §Deferred Ideas — out of scope for Phase 22.4.1's "discoverable output for high-traffic surface" mission (D-05).
