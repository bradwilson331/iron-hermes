# Phase 22.4.1: tui_rata handler re-port — route dispatch_slash through CommandRouter and registry handlers - Context

**Gathered:** 2026-04-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Re-port `crates/ironhermes-cli/src/tui_rata/commands.rs::dispatch_slash` so that **every** visible-surface slash command resolves through the unified `ironhermes_core::commands::CommandRouter`, retiring the four ad-hoc `strip_prefix` fast-paths added in Plans 22.4-16 (`/mouse`) and 22.4-18 (`/mcp`, `/sessions`, `/memory`). After this phase, `tui_rata`'s slash dispatch shape is symmetric with `crates/ironhermes-cli/src/tui/commands.rs` (classic-tui) and `crates/ironhermes-gateway/src/handler.rs` (gateway): `dispatch_slash` is a pure router shell — `build_command_context` → `router.resolve` → `invoke_handler` → `map_core_to_slash_outcome` — with one localised post-router hook for `/mouse`'s state-mutation. As part of the same re-port, the `invoke_handler` match table is expanded to cover every command in the Session and Configuration registry categories (≈32 new arms) so the generic "(tui_rata: /X not yet wired in Phase 22.4 …)" fallback stops firing for the bulk of the user-typed surface.

**In scope:**

- `ironhermes-core::commands::registry::build_registry()` gains four new `CommandDef` entries: `mouse` (Configuration, `args_hint("[on|off]")`), `mcp` (ToolsAndSkills), `sessions` (Session), `memory` (ToolsAndSkills). Exact category placement: planner picks based on existing groupings.
- `tui_rata::commands::dispatch_slash` body is reduced to the shared router-shell shape. All four `strip_prefix` branches deleted. Single `match def.name` post-router hook intercepts `"mouse"` to call the existing `handle_mouse_slash(app, args)` for its side effect.
- `tui_rata::commands::invoke_handler` gains explicit `"<name>" => CommandResult::Output(…)` arms for every command in the **Session** and **Configuration** categories of the core registry that does not already have one. Stub bodies follow the multi-line shape established by the existing `/agents` and `/skills` arms (line 1 `/X — <description>`, optional `Args:` line from `args_hint`, closing `Phase 22.4.1 stub: …` marker).
- `tui_rata::commands::render_help` is replaced by a call into the classic-tui helper `tui::commands::render_help_text(router, …)` (or a shared lift into `ironhermes-core` if the function turns out to be tui-specific in shape — researcher's call). Single source of truth for `/help` across CLI + tui_rata + gateway.
- `crates/ironhermes-cli/tests/invariants_22_4.rs`:
  - **INV-22.4-29** (mouse fast-path "must be present") **inverted in place** to assert the fast-path is **absent**.
  - **INV-22.4-31** (handler-coverage `strip_prefix` "must be present") **inverted in place** to assert all three `input.strip_prefix("/mcp")` / `("/sessions")` / `("/memory")` lines are **absent** from `dispatch_slash`. The `"agents" => CommandResult::Output(` and `"skills" => CommandResult::Output(` sub-assertions are **preserved unchanged** (the /agents + /skills arms survive verbatim per D-06).
  - New **INV-22.4-32** (`router_membership`) — grep-locks the four new `CommandDef::new("mouse"|"mcp"|"sessions"|"memory", …)` entries in `crates/ironhermes-core/src/commands/registry.rs`.
  - New **INV-22.4-33** (`invoke_handler_arms`) — grep-locks every new `"<name>" => CommandResult::Output(` arm in `tui_rata/commands.rs::invoke_handler` (one assertion per Session + Configuration category command added in Plan 22.4.1-02).
  - New **INV-22.4-34** (`dispatch_slash_no_strip_prefix`) — asserts the `dispatch_slash` body contains zero `input.strip_prefix("/`-prefixed lines. Whole-function regex bite.
- `tui_rata` insta snapshots: re-baselined via `cargo insta accept` after Plan 22.4.1-02 lands, if any of the 8 canonical frames produce byte-diffs (likely only the `/help`-related frame, if one exists).

**Out of scope (this phase):**

- Real implementations for any of the new stub handlers (live `/mcp` enumeration, `/sessions` listing, `/memory` show, `/agents` subcommand parsing, `/skills` enumeration, `/personality` overlays, `/model` switching, `/yolo` toggle, `/verbose` toggle, etc.). Stub-text only — Phase 22.4.1 ships visible-output, not behavior.
- Any change to `ironhermes-core::commands::CommandContext` shape. `_ctx` stays unused in `invoke_handler` (D-03). Whichever future phase wires the first stateful handler can introduce ctx fields on demand.
- Any change to the four App-side helper functions (`handle_mcp_slash`, `handle_sessions_slash`, `handle_memory_slash`, `handle_mouse_slash`). Their bodies move from being called by `dispatch_slash` to being called by `invoke_handler` (for the three pure-text helpers) or by the `dispatch_slash` post-router mouse hook (for `handle_mouse_slash`). No new logic, no signature changes.
- A 9th insta snapshot frame seeding slash output into history (22.4-18 recommended follow-up #2). Stays deferred to a later phase.
- Promotion or removal of the generic `other => CommandResult::Output(format!("(tui_rata: /{other} not yet wired in Phase 22.4 …)"))` fallback. Stays verbatim as a safety net for any registry name not yet reached by Plan 22.4.1-02 (D-07). Removing it is a follow-up cleanup once the registry is locked.
- Changes to `/agents` and `/skills` invoke_handler arms (D-06). They are the template — touching them adds noise without payoff.
- Cross-crate refactor of `CommandResult` to add new variants (e.g., `ToggleMouseCapture(Option<bool>)`). Rejected in D-10 in favor of a localised post-router hook.
- Live-TTY operator UAT. The 22.4-UAT.md re-runs are tracked under Phase 22.4 closure, not 22.4.1.

## Implementation Decisions

### Routing strategy & handler ownership

- **D-01:** **Registry in core, handler bodies tui_rata-local.** Add `CommandDef::new(...)` entries for `mouse`, `mcp`, `sessions`, `memory` to `ironhermes-core::commands::registry::build_registry()`. Stub bodies for `mcp`, `sessions`, `memory` (and the ≈32 new Session + Configuration arms) live as `"<name>" => CommandResult::Output(...)` match arms in `tui_rata::commands::invoke_handler`. No `CommandContext` schema change. Cross-crate change is intentionally tiny (4 `CommandDef::new` lines + Plan-02 stubs all in one tui_rata file). Mirrors the existing `/agents` and `/skills` pattern from Plan 22.4-18.
- **D-02:** **Zero `strip_prefix` fast-paths in `dispatch_slash` after the re-port.** `dispatch_slash` becomes the pure router-shell shape used by classic-tui (`tui::commands::dispatch`) and gateway. `dispatch_slash` body shape (locked):
  ```rust
  pub async fn dispatch_slash(app: &mut App, input: &str) -> SlashOutcome {
      let platform = Platform::Local;
      match app.command_router.resolve(input, &platform) {
          ResolveResult::Exact(def) | ResolveResult::PrefixMatch(def) => {
              let ctx = build_command_context(app);
              match invoke_handler(def.name, &ctx).await {
                  Ok(result) => {
                      // D-10 post-router App-side hook for stateful /mouse.
                      if def.name == "mouse" {
                          let args = input
                              .strip_prefix(&format!("/{}", def.name))
                              .unwrap_or("")
                              .trim();
                          handle_mouse_slash(app, args)
                      } else {
                          map_core_to_slash_outcome(result)
                      }
                  }
                  Err(e) => SlashOutcome::Error(e.to_string()),
              }
          }
          ResolveResult::Ambiguous(candidates) => { /* unchanged */ }
          ResolveResult::NotFound => { /* typo suggester unchanged */ }
      }
  }
  ```
  INV-22.4-34 grep-locks the absence of any `input.strip_prefix("/...` pattern other than the post-router args extraction. The post-router args extraction uses `def.name` interpolation specifically so an INV grep for `strip_prefix("/mouse")`, `("/mcp")`, `("/sessions")`, `("/memory")` literal strings returns 0 hits in `dispatch_slash`'s body.
- **D-03:** **`invoke_handler` keeps its `_ctx: &CommandContext` arg unused.** Stub bodies are static text; `CommandContext` fields (`platform`, `args`, session metadata, etc.) are not needed at this scope. Drop the underscore prefix only when the first concrete handler reads from `ctx`. This defers the `CommandContext` schema decision to whichever future phase lands real subcommand parsing — Phase 22.4.1 does not commit to a ctx schema.
- **D-04 (superseded by D-09):** Two-plan wave was the original sequencing intent. **Superseded** by the scope expansion in D-05/D-08 — see D-09 for the locked three-plan sequencing.

### Scope: command set

- **D-05:** **In-scope command set.** Four new core-registry entries: `mouse`, `mcp`, `sessions`, `memory`. PLUS explicit `invoke_handler` stub arms for every command in the Session **and** Configuration categories of the core registry that does not already have a dedicated arm. Source of truth for the name list is `crates/ironhermes-core/src/commands/registry.rs::build_registry()` filtered by `category == Session | Configuration`. Today's Session list (per registry.rs grep, may shift if registry changes): `new`, `clear` (already wired), `history`, `save`, `retry`, `undo`, `title`, `compress`, `rollback`, `stop`, `agents` (already wired), `approve`, `deny`, `background`, `btw`, `queue`, `status`, `sethome`, `resume`, `start`. Today's Configuration list: `config`, `provider`, `prompt`, `personality`, `statusbar`, `verbose`, `yolo`, `reasoning`, `skin`, `voice`, `model`, `fast`. Subtract names that already have invoke_handler arms (`new`, `clear`, `agents`, plus the universal `quit`/`exit`/`help`/`reset`/`reload-mcp`/`skills`). Net new: ≈30 invoke_handler arms in Plan 22.4.1-02. Tools & subagents categories beyond the locked set stay on the generic fallback for one more cycle.
- **D-06:** **`/agents` and `/skills` invoke_handler arms left exactly as-is.** They are the template the new Plan 22.4.1-02 arms follow. INV-22.4-31's sub-assertions for `"agents" => CommandResult::Output(` and `"skills" => CommandResult::Output(` survive intact across the inversion. No cosmetic refactor (alphabetical ordering, prefix-string normalization) — strict scope discipline keeps the diff reviewable.
- **D-07:** **Generic fallback arm stays verbatim.** `other => CommandResult::Output(format!("(tui_rata: /{other} not yet wired in Phase 22.4 — see plan 22.4-07 §Handler Coverage)"))` continues to fire for any router-resolved command name without an explicit arm — covers Tools/Subagents/Memory categories not in the D-05 set. Removing it is a follow-up cleanup. INV-22.4-31's safety-net assertion (`"not yet wired in Phase 22.4"` literal) is preserved.
- **D-08:** **Stub-text format.** Every new `invoke_handler` arm follows the shape established by `/agents` and `/skills`:
  ```rust
  "<name>" => CommandResult::Output(
      "/<name> — <description from CommandDef>.\n\
       Args: <args_hint>\n\
       Phase 22.4.1 stub: full implementation deferred — <one-line rationale or which subsystem owns it>.".to_string()
  ),
  ```
  The `Args:` line is omitted when the CommandDef has no `args_hint`. The `Phase 22.4.1 stub:` marker is mandatory and is what INV-22.4-33 grep-locks per arm. Sourcing the description text from the CommandDef.description field (rather than re-typing) keeps tui_rata in sync if registry text drifts — the planner decides whether to copy verbatim or reference at runtime via `def.description`.

### Mouse handler special case

- **D-10:** **Post-router App-side hook in `dispatch_slash`** for `/mouse`'s state-mutation. CommandDef registered in core (D-05). `invoke_handler` returns a stub `CommandResult::Output(...)` for `"mouse"` (so /help discovers it and the stub text shows in dry-run paths). `dispatch_slash` post-router branches on `def.name == "mouse"`: it calls the existing `handle_mouse_slash(app, args)` for the side effect (crossterm capture toggle + AtomicBool mutation) and uses `handle_mouse_slash`'s return value as the SlashOutcome instead of mapping the invoke_handler stub. Smallest surface change; no new public CommandResult variant; App-state never crosses the crate boundary. Pre-emptive scaffolding for future stateful commands (D-10b alternative — typed dispatch table) is **rejected** for this phase as premature.
- **D-11:** **`handle_mouse_slash` argument parsing unchanged.** Continues to take `(app: &mut App, arg: &str)` where `arg` is the trimmed remainder after `/mouse`. The new call site in `dispatch_slash` extracts args via `input.strip_prefix(&format!("/{}", def.name)).unwrap_or("").trim()`. Note: this is the ONE remaining `strip_prefix` in dispatch_slash, but its target string is `def.name`-interpolated (not a literal `"/mouse"`), so INV-22.4-34's grep for literal `strip_prefix("/<name>")` lines returns 0 hits. The whole-body grep for `.strip_prefix("/")` will hit this once — the INV must scope its assertion narrowly (e.g., grep for the exact literal pattern `strip_prefix("/mouse")` etc., NOT a generic `strip_prefix("/`).
- **D-12:** **Hook fire site:** inside the `Exact(def) | PrefixMatch(def)` arm, AFTER `invoke_handler(def.name, &ctx).await` returns `Ok(_)`, BEFORE `map_core_to_slash_outcome`. Local to `dispatch_slash`, single-purpose, doesn't leak into invoke_handler's signature (which stays `async fn invoke_handler(name: &str, _ctx: &CommandContext) -> Result<CommandResult, anyhow::Error>` per D-03). On `invoke_handler` returning `Err(e)`, the existing `SlashOutcome::Error(e.to_string())` path is taken — the mouse hook only fires when invoke_handler succeeded, which is always for the stub case. Future stateful commands can be added by extending the `def.name == "mouse"` check into a small match.

### Claude's Discretion

- Exact category placement of the four new CommandDefs in `build_registry()`. Recommend: `mouse` → Configuration (matches `/voice` and `/skin` UI-toggle precedent), `mcp` → ToolsAndSkills (matches `/reload-mcp`), `sessions` → Session, `memory` → ToolsAndSkills (closest existing category — Memory category does not currently exist; introducing a new category for a stub would be premature). Planner picks; researcher may confirm via category enumeration.
- Whether to source stub-text descriptions verbatim from `CommandDef.description` (compile-time copy) or to look them up at runtime via `app.command_router.find(name)` from inside the invoke_handler arm (zero text duplication, slight indirection). Either is acceptable per D-08; the runtime lookup is more drift-resistant but requires invoke_handler to take a router handle — currently not threaded. Compile-time copy is simpler and matches the existing `/agents` + `/skills` shape.
- Whether `tui::commands::render_help_text` is reused directly via `pub use` re-export, or lifted to `ironhermes-core::commands::render_help_text`. Researcher confirms the function's body and dependencies before locking the location. If it transitively pulls in classic-tui state (e.g., `KeybindingRegistry` references), lift the pure-text core to ironhermes-core and keep the tui-specific keybindings render in classic-tui.
- The exact text of each new `Phase 22.4.1 stub: …` rationale clause. Planner picks per command; the rationale should reference the subsystem that owns full implementation (e.g., `/personality stub: SOUL.md overlay infrastructure exists in Phase 15 — wiring lands in a later phase`).
- Whether Plan 22.4.1-02's bulk arm-additions are written by hand (≈30 verbatim arms) or generated via a helper function (`stub_for(def)` taking a `CommandDef` and returning a `String`). The helper saves LOC at the cost of one indirection per arm. Either is acceptable; INV-22.4-33's grep-bites are insensitive to which approach is used so long as `"<name>" => CommandResult::Output(...)` appears literally in the source for each name.
- Whether the rename of INV-22.4-29's function name (the body inversion makes its current name `invariant_22_4_29_mouse_capture_paired_with_toggle` slightly misleading post-re-port, since the toggle pairing is now mediated by the registry/post-router-hook chain). Acceptable to rename to `invariant_22_4_29_mouse_via_registry` or leave as-is and add a docstring clarifying the post-re-port semantics.
- Whether INV-22.4-32 / -33 / -34 land in Plan 00 / 01 / 02 respectively (matching the source they bite) or all three land alongside INV-22.4-31's inversion in Plan 01. The first option has cleaner per-plan acceptance criteria; the second keeps all INV churn in one commit. Planner picks based on plan-frontmatter cohesion.
