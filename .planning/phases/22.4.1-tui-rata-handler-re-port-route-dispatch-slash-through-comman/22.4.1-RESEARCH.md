## Summary

Phase 22.4.1 is a pure refactor of `tui_rata/commands.rs` and `ironhermes-core/src/commands/registry.rs`. No new runtime behavior is introduced — every change is either (a) routing plumbing (retire four `strip_prefix` fast-paths in `dispatch_slash`, add four `CommandDef::new` entries to the core registry) or (b) stub-body expansion (add ~30 new `invoke_handler` arms for Session and Configuration categories). The phase also inverts two invariants (INV-22.4-29/-31) and adds three new invariants (INV-22.4-32/-33/-34).

All ten specific items flagged for researcher confirmation are resolved below with HIGH confidence. The most consequential finding is the **render_help path (D-13)**: `tui/commands.rs::format_help` takes `extensions: &[Box<dyn TuiExtension>]` and `keybinding_registry: Option<&KeybindingRegistry>` — both are classic-tui-only types. Direct reuse from tui_rata is NOT feasible without introducing a dependency on `crate::tui::extension` and `crate::tui::keybindings`. The correct path is to **lift a pure-text helper** that takes only `&CommandRouter` and `&Platform` into either `tui_rata/commands.rs` itself (inline) or a shared location. See Finding 1 for the full analysis and recommendation.

**Primary recommendation:** Planner should write a private `render_help_router(router: &CommandRouter, platform: &Platform) -> String` function directly in `tui_rata/commands.rs` (6–10 lines mirroring `format_help`'s core loop without the extension/keybinding args), replacing the current 22-line hand-built `render_help()` string. This avoids a cross-module dep while achieving the single-source-of-truth goal for router-driven `/help`.

## Research Finding 1: render_help reuse path (D-13) — VERIFIED

**Question:** Is `tui::commands::format_help` (the router-driven /help helper in classic-tui) directly reusable from tui_rata, or does it pull in classic-tui-only state?

**Actual function signature (line 157 of `crates/ironhermes-cli/src/tui/commands.rs`):** [VERIFIED: direct source read]

```rust
pub fn format_help(
    _extensions: &[Box<dyn TuiExtension>],
    keybinding_registry: Option<&KeybindingRegistry>,
    router: &CommandRouter,
    platform: &Platform,
) -> String
```

**Finding:** The function takes two classic-tui-only types:
1. `_extensions: &[Box<dyn TuiExtension>]` — requires `crate::tui::extension::TuiExtension`
2. `keybinding_registry: Option<&KeybindingRegistry>` — requires `crate::tui::keybindings::KeybindingRegistry`

The first parameter is unused (`_extensions` — only its length is never even checked in the body). The second is used only to append a `"Keybindings:\n"` section when `Some`. The pure-text command enumeration body (lines 165–183) uses only `router.commands_by_category(platform)` and `CommandCategory` variants — zero tui-specific types.

**Conclusion:** Direct `pub use` re-export from tui_rata is NOT feasible — it would require tui_rata to import classic-tui extension and keybinding types, creating a circular/unintended cross-module dependency. A lift to `ironhermes-core` is possible but overkill for one call site.

**Recommended locked path:** Write a private `render_help_router(router: &CommandRouter, platform: &Platform) -> String` function in `tui_rata/commands.rs`. Body is 6–10 lines identical to `format_help`'s inner loop but without the extension/keybinding parameters. This is a minimal, self-contained duplication that avoids cross-module coupling. The `invoke_handler` "help" arm calls this instead of `render_help()`.

**The classic-tui `format_help` function DOES serve as the exact template** for the body — copy its lines 165–183 verbatim, dropping only the extensions and keybinding blocks.

## Research Finding 4: CommandRouter shape parity check (D-02) — VERIFIED

[VERIFIED: direct reads of tui/commands.rs and gateway/handler.rs]

**Classic-tui dispatch shape** (`tui/commands.rs::dispatch_command`, line 40):
```rust
pub fn dispatch_command(
    extensions: &[Box<dyn TuiExtension>],
    cmd: &str,
    args: &[&str],
    router: &CommandRouter,
    ctx: &CommandContext,
) -> CommandResult {
    // Extension-first chain, then:
    match router.resolve(&full_input, &ctx.platform) {
        ResolveResult::Exact(def) | ResolveResult::PrefixMatch(def) => {
            let core_result = ironhermes_core::commands::handlers::dispatch(def, args, ctx, router);
            map_core_to_tui(core_result)
        }
        ResolveResult::Ambiguous(candidates) => { /* error */ }
        ResolveResult::NotFound => { /* typo suggest */ }
    }
}
```

**Gateway dispatch shape** (`handler.rs::handle_slash_command`, line 218):
```rust
match self.command_router.resolve(command_input, platform) {
    ResolveResult::Exact(def) | ResolveResult::PrefixMatch(def) => {
        let core_result = ironhermes_core::commands::handlers::dispatch(def, &args, &ctx, &self.command_router);
        match core_result { /* output/session handling */ }
    }
    // ... other arms
}
```

**D-02 locked tui_rata dispatch_slash shape** (from CONTEXT.md):
```rust
match app.command_router.resolve(input, &platform) {
    ResolveResult::Exact(def) | ResolveResult::PrefixMatch(def) => {
        let ctx = build_command_context(app);
        match invoke_handler(def.name, &ctx).await {
            Ok(result) => {
                if def.name == "mouse" { handle_mouse_slash(app, args) }
                else { map_core_to_slash_outcome(result) }
            }
            Err(e) => SlashOutcome::Error(e.to_string()),
        }
    }
    ResolveResult::Ambiguous(candidates) => { /* unchanged */ }
    ResolveResult::NotFound => { /* typo suggester unchanged */ }
}
```

**Parity assessment:** The locked shape is symmetric with classic-tui and gateway in the critical dimension: `resolve(input, &platform)` → `Exact | PrefixMatch` → handler → map-to-outcome. Two deviations are intentional and correct:
1. tui_rata uses `invoke_handler(def.name, &ctx)` (local stub table) rather than `handlers::dispatch(def, args, ctx, router)` (real handlers in core) — per D-01.
2. tui_rata has the post-router `/mouse` hook — gateway/classic-tui do not need it because they don't manage mouse capture state.

**No flag needed** — the deviations are by design per CONTEXT.md decisions.

```rust
// Private fn in tui_rata/commands.rs — mirrors format_help core loop
// (crates/ironhermes-cli/src/tui/commands.rs lines 165–183)
fn render_help_router(router: &CommandRouter, platform: &Platform) -> String {
    let mut out = String::from("Available commands:\n");
    for (category, cmds) in router.commands_by_category(platform) {
        out.push('\n');
        let cat_name = match category {
            CommandCategory::Session => "SESSION",
            CommandCategory::Configuration => "CONFIGURATION",
            CommandCategory::ToolsAndSkills => "TOOLS & SKILLS",
            CommandCategory::Info => "INFO",
            CommandCategory::Exit => "EXIT",
        };
        out.push_str(cat_name);
        out.push('\n');
        for cmd in cmds {
            out.push_str(&format!(
                "  /{:<13}{:<16}{}\n",
                cmd.name, cmd.args_hint, cmd.description
            ));
        }
    }
    out
}
// Then in invoke_handler:
"help" => CommandResult::Output(render_help_router(/* need router ref */)),
```

**Note on router access in invoke_handler:** The current signature is `async fn invoke_handler(name: &str, _ctx: &CommandContext)`. To call `render_help_router`, the function needs access to the router. Options:
1. Pass router as a third arg: `invoke_handler(def.name, &ctx, &app.command_router)` — requires signature change
2. Reconstruct router inline: wasteful and defeats purpose
3. Pass router through ctx: `ctx` doesn't carry router, and D-03 says not to change CommandContext

**Recommended:** Add `router: &CommandRouter` as a third parameter to `invoke_handler`. This is a minimal signature change (one parameter) that doesn't change `CommandContext`, doesn't violate D-03, and is trivially thread-safe (CommandRouter is immutable after construction). The call site becomes `invoke_handler(def.name, &ctx, &app.command_router).await`.

### Anti-Patterns to Avoid

- **Passing `&mut App` into invoke_handler:** Contradicts D-03 (keeps ctx unused) and would require a much larger signature change. The `/mouse` hook deliberately stays in dispatch_slash to avoid this.
- **Runtime router lookup in invoke_handler for description text:** `app.command_router.find(name)` from inside invoke_handler would require either the App ref or the router ref — neither is currently threaded through. Compile-time copy of description text is simpler (per CONTEXT.md Claude's Discretion).
- **Deleting the generic fallback arm before Plan 22.4.1-02 is fully committed:** The fallback must survive as a safety net for any registry command not yet covered. D-07 is explicit.
- **Using literal `strip_prefix("/mouse")` in the post-router args extraction:** D-11 requires `strip_prefix(&format!("/{}", def.name))` so INV-22.4-34's literal-substring grep returns 0 hits for "/mouse".

## Common Pitfalls

### Pitfall 1: Adding `mouse`/`mcp`/`sessions`/`memory` to registry before INV-22.4-34 lands
**What goes wrong:** Once the four commands are in the registry, `router.resolve("/mouse", Platform::Local)` returns `Exact` — the fast-path in dispatch_slash is bypassed before Plan 22.4.1-01 deletes it, but the code still works because the fast-path fires first (it precedes the router call). This is safe. However, if Plan 22.4.1-00 lands without Plan 22.4.1-01 quickly following, there's a window where the registry has `mouse` but `dispatch_slash` still uses the fast-path AND the router. Per D-09 sequencing, this is by design: "behavior is intentionally backward-compatible until Plan 22.4.1-01 lands."
**How to avoid:** Commit plans in order: 00 → 01 → 02.

### Pitfall 2: INV-22.4-34 grep matching the post-router args extraction
**What goes wrong:** INV-22.4-34 asserts that `dispatch_slash` body contains zero occurrences of `strip_prefix("/mouse")`, `strip_prefix("/mcp")`, etc. The post-router args extraction uses `strip_prefix(&format!("/{}", def.name))` — which does NOT match the literal `strip_prefix("/mouse")` string. If a developer writes the extraction as `strip_prefix("/mouse")` instead of the format, INV-22.4-34 will fail.
**How to avoid:** Use `input.strip_prefix(&format!("/{}", def.name))` exactly as specified in D-11.

### Pitfall 3: invoke_handler receiving `def.name` for aliases
**What goes wrong:** The CommandRouter resolves `/q` via the `quit` alias and returns `def.name == "quit"`. invoke_handler receives `"quit"`, not `"q"`. Aliases are transparent at the invoke_handler boundary. No issue for the new arms, but planners writing `"q"` arms in invoke_handler would create dead code.
**How to avoid:** Only write arms for canonical `def.name` values from the registry (the names enumerated in Finding 2 above).

### Pitfall 4: `mouse` platform filter too broad
**What goes wrong:** If `mouse` CommandDef uses `Platform::Universal` instead of `CliOnly`, `router.resolve("/mouse", Platform::Telegram)` returns `Exact(def)`. A gateway session hitting `/mouse` would call `invoke_handler("mouse", ...)` and return stub text — confusing for non-CLI users.
**How to avoid:** Set `.platform(CliOnly)` for the `mouse` CommandDef (researcher recommendation above).

### Pitfall 5: render_help_router missing CommandCategory import
**What goes wrong:** The `match category` block in `render_help_router` needs `CommandCategory` variants in scope. `tui_rata/commands.rs` already imports `ironhermes_core::commands::{CommandResult, CommandRouter, ResolveResult}` — `CommandCategory` must be added to this import.
**How to avoid:** Add `CommandCategory` to the existing use statement: `use ironhermes_core::commands::{CommandCategory, CommandResult, CommandRouter, ResolveResult};`

### Pitfall 6: GatewayOnly commands reaching invoke_handler
**What goes wrong:** On `Platform::Local`, `router.resolve("/approve", Platform::Local)` returns `NotFound` (GatewayOnly filter). So `approve`, `deny`, `sethome`, `start` will never reach `invoke_handler`. If Plan 22.4.1-02 adds arms for them "just in case," they are dead code and INV-22.4-33 should NOT assert their presence.
**How to avoid:** Do NOT add invoke_handler arms for GatewayOnly commands. INV-22.4-33 must only assert names that `Platform::Local` can resolve.

### Pitfall 7: `Platform` import needed for render_help_router
**What goes wrong:** `render_help_router` takes `platform: &Platform`. `tui_rata/commands.rs` already imports `ironhermes_core::types::Platform` (confirmed in source line 26). No new import needed.
**Warning sign:** Compiler error `use of undeclared type Platform` if the import was removed in an edit.

```rust
pub async fn dispatch_slash(app: &mut App, input: &str) -> SlashOutcome {
    let platform = Platform::Local;
    match app.command_router.resolve(input, &platform) {
        ResolveResult::Exact(def) | ResolveResult::PrefixMatch(def) => {
            let ctx = build_command_context(app);
            match invoke_handler(def.name, &ctx, &app.command_router).await {
                Ok(result) => {
                    if def.name == "mouse" {
                        let args = input
                            .strip_prefix(&format!("/{}", def.name))
                            .unwrap_or("")
                            .trim();
                        handle_mouse_slash(app, args)
                    } else {
                        map_core_to_slash_outcome(result)
                    }
                }
                Err(e) => SlashOutcome::Error(e.to_string()),
            }
        }
        ResolveResult::Ambiguous(candidates) => {
            let hint = format!(
                "Ambiguous command — matches: {}. Type /help for the list.",
                candidates.join(", ")
            );
            SlashOutcome::Unknown { input: input.to_string(), hint }
        }
        ResolveResult::NotFound => {
            let known = collect_known_command_names(&app.command_router);
            let stripped = input
                .trim_start_matches('/')
                .split_whitespace()
                .next()
                .unwrap_or("");
            let known_refs: Vec<&str> = known.iter().map(|s| s.as_str()).collect();
            let hint = match suggest_typo(stripped, &known_refs) {
                Some(candidate) => format!("Did you mean `/{candidate}`?"),
                None => "Type /help for the list of commands.".to_string(),
            };
            SlashOutcome::Unknown { input: input.to_string(), hint }
        }
    }
}
```

Note: `invoke_handler` signature changes to add `router: &CommandRouter` parameter to support `render_help_router`. This is the only signature deviation from CONTEXT.md D-03 (which says keep `_ctx` unused — adding `router` does not change that).
