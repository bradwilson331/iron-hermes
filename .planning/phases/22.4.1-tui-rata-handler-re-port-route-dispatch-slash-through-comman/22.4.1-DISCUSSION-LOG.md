## Routing strategy & handler ownership

### Q1: Where do /mcp, /sessions, /memory handler BODIES live after the re-port?

| Option | Description | Selected |
|--------|-------------|----------|
| (A) Registry in core, bodies tui_rata-local | Add CommandDef entries to ironhermes-core registry. Move existing stub bodies into invoke_handler match arms. No CommandContext signature change. Mirrors current /agents + /skills shape. | ✓ |
| (B) Bodies in core too — full migration | Move stub text into ironhermes-core handlers. Requires extending CommandContext to expose tui_rata App state. Cleanest dispatch shape but leaks tui_rata implementation into core. | |
| (C) Hybrid — split by purity | Pure-text stubs migrate to core; App-state-touching stubs stay tui_rata-local. Two patterns, harder to grep-lock as INV. | |

**User's choice:** (A) Registry in core, bodies tui_rata-local
**Notes:** Maps to D-01. Smallest cross-crate change (4 CommandDef::new lines). Consistent with existing /agents + /skills pattern from Plan 22.4-18.

## Mouse handler special case

### Q11: How does /mouse's state-mutation reach the App after the re-port?

| Option | Description | Selected |
|--------|-------------|----------|
| (1) Post-router App-side hook | Register CommandDef in core. invoke_handler returns stub Output. dispatch_slash post-router branches on def.name == "mouse" and calls handle_mouse_slash. Smallest core change. | ✓ |
| (3) New CommandResult::ToggleMouseCapture(Option<bool>) variant | Cleaner data flow but adds a new public CommandResult variant. Cross-crate ripple. | |
| (1b) Same as (1) but with a typed App-side dispatch table | Pre-emptive scaffolding for future stateful commands. More structure than (1). | |

**User's choice:** (1) Post-router App-side hook
**Notes:** Maps to D-10. No new CommandResult variant. App-state never crosses the crate boundary.

### Q12: Should /mouse's args parsing live in the router or stay in the App-side hook?

| Option | Description | Selected |
|--------|-------------|----------|
| App-side hook parses args | dispatch_slash hands raw `rest` to handle_mouse_slash. Zero change to existing handle_mouse_slash body. | ✓ |
| Router parses args, App-side hook receives parsed value | Couples /mouse to whatever args-shape the router commits to. Likely overkill. | |

**User's choice:** App-side hook parses args
**Notes:** Maps to D-11. Note: this leaves ONE strip_prefix in dispatch_slash (the post-router args extraction using `&format!("/{}", def.name)`) — INV-22.4-34 must scope its grep to LITERAL strings `strip_prefix("/mouse")` etc., not generic `strip_prefix("/`.

### Q13: Where in dispatch_slash does the post-router App-side hook fire?

| Option | Description | Selected |
|--------|-------------|----------|
| Inside the Exact/PrefixMatch arm, after invoke_handler returns | Local to dispatch_slash, single-purpose. INV-grep-lockable. | ✓ |

**User's choice:** Inside the Exact/PrefixMatch arm, after invoke_handler returns
**Notes:** Maps to D-12. Hook fires AFTER invoke_handler returns Ok(_), BEFORE map_core_to_slash_outcome.
