<objective>
Bulk-fill the `invoke_handler` match table with arms for every Platform::Local-reachable command in the Session and Configuration categories of the core registry that does not already have a dedicated arm. After this plan, the generic `not yet wired in Phase 22.4` fallback only fires for ToolsAndSkills, Info, and any future-added categories that haven't been bulk-armed yet — for the high-traffic Session + Configuration surface, every command produces discoverable output.

Purpose (per CONTEXT D-05, D-08, D-09): Phase 22.4.1 ships visible-output, not behavior. With Plan 22.4.1-01's pure-router shape locked, the remaining gap is the silent-fallback for the bulk of session-control and config-toggle commands. This plan closes that gap mechanically — 26 new arms following D-08's locked stub format. INV-22.4-33 grep-locks every arm by name plus the `Phase 22.4.1 stub:` marker count, so a future refactor cannot silently regress the discoverability surface.

```rust
"<name>" => CommandResult::Output(
    "/<name> — <description from CommandDef>.\n\
     Args: <args_hint>\n\
     Phase 22.4.1 stub: <one-line owning-subsystem rationale>.".to_string()
),
```

Rules:
- Line 1: `"/<name> — <description>.\n\` — `<description>` is the third arg of `CommandDef::new(name, description, category)` from `registry.rs` (verbatim copy).
- Line 2 (omitted when CommandDef has empty `args_hint`): `"     Args: <args_hint>\n\` — 5-space indent before `Args:`.
- Final line: `"     Phase 22.4.1 stub: <rationale>."` — 5-space indent, must literally contain `Phase 22.4.1 stub:` (NOT `Phase 22.4 stub:` — INV-33 grep-discriminates).
- Closing: `.to_string()`.
