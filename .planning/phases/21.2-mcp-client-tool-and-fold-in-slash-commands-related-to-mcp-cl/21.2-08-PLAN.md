<objective>
Close GAP-5 (high): when `ironhermes` starts with one or more `mcp_servers` configured, the banner (`Chat · ...` / `IronHermes vX.Y — Type /help for commands ...`) and the first `You:` prompt do not paint on the terminal until the user presses Enter. From the user's perspective the CLI appears frozen. This violates the D-07 contract "agent prompt appears immediately without blocking on MCP connection".

Root cause (from VERIFICATION.md:76-83 + source inspection of `crates/ironhermes-cli/src/main.rs`):
- `run_chat` calls `print_banner()` at line 552 (fresh function at line 1439).
- `spawn_mcp_background` at `main.rs:1398-1436` immediately `eprintln!`s the dimmed line `MCP: connecting to N server(s) in background...` from the main thread (line 1403-1406), THEN `tokio::spawn`s the actual discovery task.
- Between that point and the first `rl.readline` at line 839, multiple async `eprintln!` writes from the background task can interleave with whatever stdout data is buffered from the banner. With line-buffered stdout on a TTY the banner often paints, but with an MCP server configured the stderr interleave (and, separately, GAP-6's child-process-stderr leak) can corrupt or defer the paint until rustyline redraws on first keystroke.
- There is no explicit `io::stdout().flush()` between `print_banner()` and `rl.readline` in run_chat — the paint is left to the buffer's implicit flush, which loses a race with the MCP background task's `eprintln!`.

Fix: insert an EXPLICIT `io::stdout().flush().ok(); io::stderr().flush().ok();` pair at the LAST point inside `run_chat` before the REPL `loop { ... }` starts. This guarantees the banner and any synchronous pre-loop prints have hit the terminal before `rl.readline` is allowed to block on stdin. Because rustyline always repaints its prompt on `readline`, the banner is the only thing that benefits from the flush — but the banner is exactly what the user sees as "missing until keystroke".

Purpose: Restore D-07 "non-blocking startup" — the banner + prompt appears IMMEDIATELY when `ironhermes` starts, even with MCP servers configured. Pairs with GAP-6 (plan 09): once 09 contains the child's stderr, the async clobber path disappears entirely. GAP-5 alone closes the flush race; both together deliver the full D-07 promise.

Output: Two surgical edits inside `run_chat` (one flush pair right after `print_banner()`, one flush pair just before the REPL loop as belt-and-braces) + a new static-grep regression test locking the call order into source against future refactors.
</objective>
