<objective>
Close GAP-6 (medium): two related bugs hit the user at startup when MCP is configured:

1. **GAP-6a — WARN flood.** The `tracing_subscriber` init at `crates/ironhermes-cli/src/main.rs:135-141` uses:
   ```rust
   tracing_subscriber::fmt()
       .with_env_filter(
           tracing_subscriber::EnvFilter::from_default_env()
               .add_directive("ironhermes=info".parse().unwrap()),
       )
       .with_target(false)
       .init();
   ```
   With `ironhermes=info` as a forced directive, every startup emits 8-10 lines of WARN from `SkillRegistry`, `Cron`, `D-06 provider logging`, etc. — visually burying the REPL prompt. For interactive `hermes chat` (and bare `ironhermes` with no subcommand), the user expects a clean prompt, not a diagnostic dump. The `hermes gateway` / `hermes agent` operators DO want today's output.

2. **GAP-6b — child stderr leak.** `crates/ironhermes-mcp/src/transport.rs:23-34` spawns the stdio MCP child via `TokioChildProcess::new(tokio::process::Command::new(command).configure(...))` but never sets `.stderr(Stdio::piped())`. Tokio's default is to inherit the parent's stderr fd, so when `npx @modelcontextprotocol/server-filesystem` exits and prints `Usage: mcp-server-filesystem [allowed-directory] ...`, that text lands on the parent terminal — corrupting the `You:` prompt.

Fix:
- **GAP-6a:** In `main.rs`, change the tracing init so the interactive chat entry point defaults to `EnvFilter::new("error")` unless `RUST_LOG` is set. All other commands (`Gateway`, `Cron`, `Batch`, `Skills`, `Memory`, `Models`, `Mcp`, `Status`, `Doctor`, `Version`) keep today's `ironhermes=info` default. The branch happens on `&cli.command` BEFORE `tracing_subscriber` init — we read the clap parse once (it is idempotent), pick the filter, then call `.init()`.
- **GAP-6b:** In `transport.rs::connect_stdio`, add `cmd.stderr(std::process::Stdio::piped());` inside the existing `configure` closure. `TokioChildProcess::new` accepts this without any further code changes — rmcp's docs explicitly document piping stderr for this exact reason. Do NOT change the HTTP transport.

Purpose: Users running `ironhermes` or `hermes chat` get a clean prompt — no WARN flood, no child-process usage text clobbering the prompt. This plan hands GAP-5 (plan 08) a clean screen to flush into and delivers the full D-07 "non-blocking startup, clean prompt" contract together with plan 08.

Output: two files modified (`main.rs` tracing init branch; `transport.rs` stdio stderr pipe), two regression tests (one static-grep in main.rs, one functional + static-grep in transport.rs).

```rust
pub async fn connect_stdio(
    config: &McpServerConfig,
) -> Result<RunningService<RoleClient, ()>> {
    let command = config
        .command
        .as_ref()
        .ok_or_else(|| anyhow::anyhow!("stdio transport requires 'command' field"))?;

    let safe_env = build_safe_env(&config.env);
    let args = config.args.clone();

    use rmcp::transport::{ConfigureCommandExt, TokioChildProcess};
    let transport = TokioChildProcess::new(
        tokio::process::Command::new(command).configure(move |cmd| {
            for arg in &args {
                cmd.arg(arg);
            }
            // D-19: clear host env, then add only safe vars
            cmd.env_clear();
            for (k, v) in &safe_env {
                cmd.env(k, v);
            }
        }),
    )?;

    let client = ().serve(transport).await?;
    Ok(client)
}
```
Key facts:
- `tokio::process::Command` supports `.stderr(std::process::Stdio::piped())` with the same signature as `std::process::Command`.
- `rmcp::transport::ConfigureCommandExt` gives a `.configure(FnOnce(&mut Command))` closure hook — the closure is allowed to mutate any field of the command, including stderr.
- `TokioChildProcess::new` does NOT internally override stderr, so whatever we set in the closure wins. (rmcp docs: the transport reads stdin/stdout for MCP JSON-RPC; stderr is free.)

From `std::process::Stdio`:
```rust
pub struct Stdio { /* ... */ }
impl Stdio {
    pub fn piped() -> Stdio;
    pub fn null() -> Stdio;
}
```
`piped()` creates a new pipe for the child's stderr (reader stays in the parent as `Child::stderr: Option<ChildStderr>`). `TokioChildProcess` owns the child but exposes NO public accessor to read the stderr pipe — that's fine for GAP-6b, because the contract is "do not inherit parent fd". The bytes pile up in the kernel pipe buffer until the child exits; for a short-lived misbehaving child that's a drop in the bucket. Later work can spawn a reader task if we want `failure_reason` to include the captured stderr; that is not required for GAP-6b closure.

        /// identically to what connect_stdio does (env_clear + piped stderr),
        /// have it write to stderr, and assert the bytes land on the child's
        /// piped stderr handle (not on the parent's stderr fd).

        /// so the test has zero dependency on a live MCP server. The behavior
        /// under test is std/tokio's Stdio::piped contract — identical to what
        /// TokioChildProcess inherits from the configure closure.
        #[test]
        fn stdio_child_stderr_does_not_inherit_parent_fd() {
            use std::io::Read;
            use std::process::{Command, Stdio};

            // A POSIX-ish command that prints to stderr and exits. `sh -c` is
            // available on macOS + Linux CI; on Windows this test is gated out.
            #[cfg(unix)]
            let mut child = Command::new("sh")
                .arg("-c")
                .arg("printf 'usage: this-must-not-hit-parent-terminal\\n' 1>&2")
                .stderr(Stdio::piped())
                .stdout(Stdio::null())
                .spawn()
                .expect("failed to spawn sh for GAP-6b test");

            let mut child = Command::new("cmd")
                .args(["/C", "echo usage: this-must-not-hit-parent-terminal 1>&2"])
                .stderr(Stdio::piped())
                .stdout(Stdio::null())
                .spawn()
                .expect("failed to spawn cmd for GAP-6b test");

                .stderr
                .take()
                .expect("GAP-6b: Stdio::piped() must produce a reader handle on ChildStderr");

            let mut captured = String::new();
            stderr
                .read_to_string(&mut captured)
                .expect("failed to drain child stderr pipe");
            let _ = child.wait();

            assert!(
                captured.contains("usage: this-must-not-hit-parent-terminal"),
                "GAP-6b: child stderr bytes must be captured on the piped handle, not \
                 inherited by the parent. captured={captured:?}"
            );
        }
    }
