# Phase 21.2 Plan 11: Gap-Closure — Fix Ironhermes Gateway Ctrl+C Hang Summary

One-liner: GAP-8 closed — `ironhermes gateway` now exits within bounded time (2s/server) on Ctrl+C instead of hanging indefinitely when stdio MCP servers are connected, via kill_on_drop(true) in connect_stdio + tokio::time::timeout in McpManager::shutdown_all + GatewayRunner wire.

## What Was Built

### Task 1: Transport + server_task plumbing (commit ae2a185)

- `connect_stdio` + `connect_http` in `crates/ironhermes-mcp/src/transport.rs` signatures change to return `Result<(RunningService<RoleClient, ()>, Option<tokio::process::Child>)>`.
- Added `cmd.kill_on_drop(true)` inside the `configure` closure of `TokioChildProcess::new` — this is the load-bearing Option B guarantee: when rmcp's transport drops after the serve loop exits (or is cancelled), tokio fires SIGKILL at the OS level, so the stdio child cannot outlive its parent.
- `connect_stdio` returns `Ok((client, None))` for the Child handle under Option B (rmcp 1.5 owns the child internally). `connect_http` returns `None` always (no child process for HTTP).
- `run_server_task` + `connect_and_serve` accept a new `child_slot: Arc<tokio::sync::Mutex<Option<tokio::process::Child>>>` parameter threaded alongside the plan-10 `connected: Arc<AtomicBool>`.
- After successful transport connect, `connect_and_serve` parks `child_opt` in the slot (no-op today under Option B; load-bearing under future Option A upgrade).
- On retry, the outer loop in `run_server_task` calls `slot.take()` + `start_kill()` on any stale Child to prevent shutdown_all from seeing a dangling reference.
- New static-grep test `connect_stdio_sets_kill_on_drop` locks the kill_on_drop(true) contract against silent regression.

### Task 2: McpManager + GatewayRunner + main.rs wiring (commit 231a063)

- `McpManager.tasks` value tuple widens from `(JoinHandle, CancellationToken)` to `(JoinHandle, CancellationToken, Arc<tokio::sync::Mutex<Option<tokio::process::Child>>>)`. Both spawn sites (`start_all`, `start_all_and_wait`) allocate the slot.
- `McpManager::shutdown_all` rewritten:
  1. Cancel the task's `CancellationToken` (existing).
  2. Take the Child from `child_slot`; call `start_kill()` if present (no-op under Option B; load-bearing under future Option A).
  3. `tokio::time::timeout(Duration::from_secs(2), handle).await` — the key user-facing guarantee. On timeout, log warning + proceed. Ensures shutdown returns in ~2s/server regardless of child behavior.
  4. Unregister tools (existing GAP-4 path).
  5. Remove connected flag (existing GAP-7 path).
- `McpManager::Drop` impl widens tuple pattern to 3-tuple; still only cancels tokens (sync context can't await or kill).
- Existing tests updated to the 3-tuple insert pattern via `replace_all`.
- `crates/ironhermes-gateway/Cargo.toml`: adds `ironhermes-mcp = { path = "../ironhermes-mcp" }`.
- `GatewayRunner` gains `mcp_manager: Option<Arc<McpManager>>` field + `set_mcp_manager` setter (mirrors `set_memory_manager`). Initialized `None` in `new()`.
- `GatewayRunner::start` shutdown path: between the `tokio::select!` exit and `self.cancel.cancel()`, calls `mgr.shutdown_all().await` if mcp_manager is `Some`. This is BEFORE the propagation comment and BEFORE the JoinSet drain — enforced by source-grep regression test.
- `run_gateway` in `main.rs`: adds `runner.set_mcp_manager(mgr.clone())` after `set_active_skills` and before `runner.start()`.
- Two regression tests in `runner.rs`:
  - `gateway_runner_invokes_mcp_shutdown_all_on_cancel`: source-grep locks call site + ordering (shutdown_call < propagation_anchor).
  - `gateway_runner_has_set_mcp_manager_setter`: source-grep locks field + setter shape.
- One regression test in `manager.rs`:
  - `shutdown_all_returns_within_timeout_when_stdio_child_blocks`: spawns `sleep 300` stdio server (unix-gated), calls `shutdown_all` wrapped in an OUTER 5s timeout, asserts the outer timeout does NOT fire. Measured pass time: 2.51s (well under 5s; the 2s-per-server internal bound + reaper overhead).

## Decisions Made (Per Orchestrator Directive)

The orchestrator resolved a prior decision checkpoint (Option A vs Option B for `connect_stdio`) in favor of **Option B — plan-blessed fallback**:

- `connect_stdio` returns `Ok((RunningService, None))` for the Child handle.
- rmcp 1.5's `TokioChildProcess::new(Command)` owns the child internally; no supported constructor accepts a pre-spawned Child. Rather than trying to refactor around rmcp or implement dual-spawn, fall back to the plan-blessed simpler path.
- `cmd.kill_on_drop(true)` inside the configure closure gives OS-level SIGKILL guarantee when rmcp's transport drops.
- Paired with `tokio::time::timeout(Duration::from_secs(2), handle)` in `McpManager::shutdown_all`, this fully closes GAP-8 at the user-facing level (gateway exits in ~2s/server on Ctrl+C).
- The signature change (Option<Child>) and `child_slot` plumbing STILL land — so when rmcp later exposes a pre-spawned-Child constructor, the Option A upgrade becomes a one-line change inside `connect_stdio` with zero call-site changes.

Plan quote supporting this choice (from 21.2-11 plan body):
> *"If the executor lands on the simpler fallback (Ok((rmcp_client, None))), the tokio::time::timeout in Task 2 STILL closes GAP-8 at the user-facing level."*

## Known Stubs

None. Option B is a plan-blessed architectural choice, not a stub. `connect_stdio` parking `None` for the Child handle is the intentional design under rmcp 1.5; when rmcp exposes a pre-spawned-Child constructor, the upgrade to Option A is a one-line change inside `connect_stdio` — no stubs, no TODOs marking incomplete work.

The `child_slot` plumbing is live code, not a stub: `connect_and_serve` parks the Option into the slot unconditionally, and `shutdown_all` reads from the slot and calls `start_kill()` unconditionally. The fact that the Option is None today is a consequence of rmcp 1.5's API, not a placeholder.
