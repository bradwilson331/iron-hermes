# Phase 21.2: MCP Client Tool ŌĆö Context

**Gathered:** 2026-04-18
**Status:** Ready for planning

<domain>
## Phase Boundary

Port hermes-agent's MCP client infrastructure to IronHermes: connect to external MCP servers via stdio and HTTP/StreamableHTTP transports, discover their tools, register them into the ToolRegistry so the LLM can call them like built-in tools, and wire the MCP-related slash commands and CLI subcommands. Includes sampling support (server-initiated LLM requests).

**In scope (Phase 21.2):**
- New `ironhermes-mcp` crate: MCP client with stdio + HTTP/StreamableHTTP transports
- Per-server tokio tasks with reconnection (exponential backoff, 5 retries, max 60s)
- Tool discovery at startup, tool call dispatch via background tasks
- Sampling/createMessage support (MCP servers requesting LLM completions)
- Environment variable filtering for stdio subprocesses (security)
- Credential stripping in error messages returned to LLM
- Dynamic ToolRegistry API: `register_dynamic()` / `unregister_by_prefix()` behind `Arc<RwLock<ToolRegistry>>`
- Wire `/reload-mcp` and `/reload` slash command stubs (disconnect, re-read config, reconnect, re-discover)
- `hermes mcp add/remove/list/test/configure` CLI subcommands in ironhermes-cli
- `mcp_servers` config section in existing config.yaml with hermes-agent-compatible YAML schema
- `${ENV_VAR}` interpolation in config values
- Per-server `enabled_tools` filtering

**Out of scope:**
- OAuth authentication for MCP servers (defer)
- Dynamic tool discovery via `tools/list_changed` notifications (defer)
- Config file watching for auto-reload (defer ŌĆö manual `/reload-mcp` only)
- MCP server mode (`hermes mcp serve` / `mcp_serve.py`) ŌĆö separate concern

</domain>

<decisions>
## Implementation Decisions

### MCP client architecture
- **D-01:** New `ironhermes-mcp` crate. Dedicated crate for the MCP client, similar to ironhermes-hooks and ironhermes-exec. Depends on ironhermes-core for ToolSchema. Clean separation, independent compilation.
- **D-02:** Use the official mcp-rust-sdk crate. Provides ClientSession, transport abstractions, and MCP types. Matches the Python `mcp` package used by hermes-agent.
- **D-03:** Port core + sampling. Stdio + HTTP/StreamableHTTP transports, tool discovery, tool call dispatch, sampling/createMessage support, reconnection with backoff, env filtering, credential stripping. OAuth deferred.
- **D-04:** Tokio tasks per server. Each MCP server gets a long-lived `tokio::spawn` task managing connection lifecycle (connect, discover, serve, disconnect). Shutdown via `CancellationToken`. Fits existing tokio patterns in IronHermes.
- **D-05:** Exponential backoff reconnection matching hermes-agent: up to 5 retries, max 60s backoff. On exhaustion, mark server as failed and skip.

### Tool registration
- **D-06:** Tool naming: `server__tool` (double-underscore separator). e.g., `filesystem__read_file`, `github__create_issue`. Matches hermes-agent pattern. Clear provenance for the LLM.
- **D-07:** Tool discovery at startup + `/reload-mcp`. One-shot discovery when agent starts. `/reload-mcp` re-discovers. No dynamic `tools/list_changed` notification support.
- **D-08:** Per-server `enabled_tools` filtering via config. `mcp_servers.github.enabled_tools: [create_issue, list_repos]`. Only selected tools registered. Matches `hermes mcp configure` behavior.
- **D-09:** Rebuild registry on reload. On `/reload-mcp`: disconnect all MCP servers, remove their tools from registry, reconnect, re-discover, re-register. Same approach as hermes-agent's `_reload_mcp()`.
- **D-10:** Add mutable API to ToolRegistry. New `register_dynamic()` and `unregister_by_prefix(server_name)` methods. ToolRegistry behind `Arc<RwLock<ToolRegistry>>` for concurrent reads during dispatch, exclusive write on reload.
- **D-11:** Prepend `[MCP: server_name]` to tool description for LLM context. Gives the model provenance info alongside the `server__tool` naming.

### Slash commands and CLI
- **D-12:** `/reload-mcp` matches hermes-agent: disconnect all MCP servers, re-read config.yaml `mcp_servers` section, reconnect, re-discover tools, update ToolRegistry, return status summary of connected/failed servers. `/reload` is an alias with same behavior.
- **D-13:** Port all `hermes mcp` CLI subcommands in this phase: `add`, `remove`, `list`, `test`, `configure`. Users need interactive MCP server management, not just config file editing.
- **D-14:** CLI subcommands live in ironhermes-cli with a new `mcp_config.rs` module. Calls into ironhermes-mcp crate for connection testing and discovery.
- **D-15:** CommandContext carries `Option<Arc<McpManager>>` for `/reload-mcp` access. Follows existing CommandContext pattern from Phase 21.1 D-09.
- **D-16:** `hermes mcp test` connects to the server and lists discovered tools (connect + list only). No test tool call.

### Config format
- **D-17:** Match hermes-agent YAML schema exactly. Same `mcp_servers` config key structure: `command/args/env` for stdio, `url/headers` for HTTP, `timeout`, `connect_timeout`, `sampling`, `enabled_tools`, `auth` fields. Users can copy config between Python and Rust agents.
- **D-18:** `${ENV_VAR}` interpolation in config values. Resolve placeholders from environment (including .env file) before parsing server configs. Essential for secrets like API keys.
- **D-19:** Safe env filtering for stdio subprocesses matching hermes-agent: only pass `PATH`, `HOME`, `USER`, `LANG`, `LC_ALL`, `TERM`, `SHELL`, `TMPDIR` + `XDG_*` from host env, plus user-specified `env` vars from config.
- **D-20:** Credential stripping in error messages matching hermes-agent regex patterns: `ghp_*`, `sk-*`, Bearer tokens, `token=`, `key=`, `API_KEY=`, `password=`, `secret=`. Replace with `[REDACTED]` before returning to LLM.
- **D-21:** MCP config reads from the existing config.yaml using the existing config loader. Add `mcp_servers` as a new serde field to the existing Config struct.

### Claude's Discretion
- Internal module structure within ironhermes-mcp (e.g., `transport/`, `sampling/`, `security/`)
- Exact `McpManager` API surface and how it wraps per-server tasks
- Whether sampling callback uses a trait or closure pattern
- How `Arc<RwLock<ToolRegistry>>` migration affects existing `Arc<ToolRegistry>` callsites
- Error types within ironhermes-mcp (anyhow vs dedicated McpError enum)
- Default timeout values if not matching hermes-agent exactly (120s tool, 60s connect)

</decisions>

<specifics>
## Specific Ideas

- The Python MCPServerTask runs in a dedicated background asyncio event loop on a daemon thread. In Rust, this maps directly to `tokio::spawn` tasks since IronHermes already runs on tokio. No need for a separate event loop ŌĆö just spawn tasks on the existing runtime.
- The hermes-agent tool handler factory (`_make_tool_handler`) creates a sync wrapper that calls the async MCP session via `run_coroutine_threadsafe`. In Rust, the `Tool::execute()` is already async, so MCP tools can directly `await` the session call. Much cleaner.
- The `Arc<RwLock<ToolRegistry>>` migration is the biggest cross-cutting change ŌĆö every callsite that currently uses `Arc<ToolRegistry>` needs updating. The researcher should map all callsites.
- `hermes mcp add` has a "discovery-first" flow: connect to server, discover tools, let user select which to enable, save config. This is an interactive CLI wizard similar to `hermes memory setup` (Phase 20-03).
- The sampling callback maps naturally to a Rust async function/trait since IronHermes's LLM calls are already async. No need for `asyncio.to_thread()` like Python.

</specifics>

<deferred>
## Deferred Ideas

- **OAuth authentication** (`tools/mcp_oauth.py`) ŌĆö MCP servers with OAuth auth flows. Separate concern requiring credential storage, refresh token management, browser redirect handling.
- **Dynamic tool discovery** (`tools/list_changed` notifications) ŌĆö MCP servers can notify when their tool list changes. Requires message handler support in the SDK.
- **Config file watching** (`_check_config_mcp_changes`) ŌĆö Auto-reload MCP connections when config.yaml changes. Quality-of-life feature, manual `/reload-mcp` sufficient for now.
- **MCP server mode** (`mcp_serve.py`) ŌĆö Running IronHermes AS an MCP server (exposing its tools to other MCP clients). Entirely different direction from client support.

</deferred>
