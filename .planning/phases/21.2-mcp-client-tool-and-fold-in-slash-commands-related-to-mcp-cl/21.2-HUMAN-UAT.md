## Gaps

### GAP-1: `hermes mcp add` / `hermes mcp test` hang indefinitely when server misbehaves
source_test: 1
severity: high
status: resolved
closed_by: 21.2-06 (commit ad6f950)
evidence: |
  `ironhermes mcp add mcpfs` with command=npx, args=empty:
  - stderr: `sh: line 1: jsonrpc:2.0: command not found`  (npx REPL reading MCP JSON-RPC as shell commands)
  - wizard never returns; user forced to Ctrl-C
root_cause: |
  `crates/ironhermes-cli/src/mcp_config.rs:479-509` — `attempt_connect_and_list` calls
  `connect_stdio(config).await?` then `client.list_all_tools().await?` with NO timeout
  wrapper. `McpServerConfig.connect_timeout` (default 60s) is never applied at the
  `cmd_test` / `cmd_add` call site, so a child process that is alive but not responding
  to MCP initialize blocks forever.
fix_direction: |
  Wrap both the transport connect and the initial `list_all_tools()` call in
  `tokio::time::timeout(Duration::from_secs(config.connect_timeout), ...)`. On timeout,
  kill the child and return a clear "Timed out waiting for MCP initialize response
  after Ns. Check command and args." error.

### GAP-5: `hermes` REPL prompt does not render until first keystroke (D-07 violation)
source_test: 4 (live terminal, 2026-04-22T23:05Z)
severity: high
status: open
evidence: |
  After `ironhermes` starts with 1+ MCP servers configured, the dimmed
  `MCP: connecting to 1 server(s) in background...` line prints on stderr,
  but the banner (`Chat · anthropic/claude-sonnet-4 · ...`) and input prompt
  (`You:`) do not appear on the terminal until the user presses Enter. From
  the user's perspective the CLI looks frozen — the prompt paint is gated
  behind stdin activity. This violates the D-07 contract
  "agent prompt appears immediately without blocking on MCP connection".
root_cause_hypothesis: |
  The REPL read-loop in `crates/ironhermes-cli/src/main.rs` writes the banner
  and first `You:` prompt but does not flush stdout before `read_line`. With
  line-buffered stdout attached to a TTY the buffer usually flushes on its own,
  but async `eprintln!` writes from the MCP background task are interleaving
  and may be clobbering the prompt before it renders. A user keystroke triggers
  rustyline (or whatever line editor is in use) to redraw and the prompt
  finally appears.
fix_direction: |
  In `main.rs`, after the initial banner + `You:` prompt paint and before the
  first `read_line`, explicitly call `io::stdout().flush().ok();`. If a line
  editor is in use, consult its docs for the right "force redraw" hook. Pair
  with GAP-6's fix: if the child's stderr is also contained, there is no async
  clobber of the prompt. Add a regression test: start `ironhermes` with a
  dummy MCP server entry in a pty, read the first 500 ms of stdout, and assert
  the banner text appears.

### GAP-6: Tracing WARN floods startup; child MCP stderr leaks into parent terminal
source_test: 4 (live terminal, 2026-04-22T23:05Z)
severity: medium
status: open
evidence: |
  Fresh `ironhermes` startup dumped 8-10 lines of WARN tracing from
  `SkillRegistry` (invalid skill names, WARN-BUT-LOAD), cron jobs.json parse,
  and D-06 provider logging — visually burying the MCP startup messages and
  the REPL prompt. Separately, when the npx child MCP process printed its
  usage text to stderr (`Usage: mcp-server-filesystem [allowed-directory] ...`),
  that output landed in the parent terminal directly on top of the `You:` prompt
  line, corrupting it.
fix_direction: |
  (a) In `hermes` CLI init (main.rs near the `tracing_subscriber` setup),
      default the EnvFilter to `error` (or `warn` for the hermes crates only)
      when running interactive chat/REPL unless RUST_LOG is explicitly set.
      Keep the current default for `hermes agent` / gateway commands where
      operators expect diagnostics.
  (b) In `ironhermes-mcp/src/transport.rs` (or wherever stdio children are
      spawned), set `Stdio::piped()` on the child's stderr and spawn a
      per-server reader task that routes lines into a bounded buffer used
      for `failure_reason`. Do NOT inherit the parent stderr fd.
  (c) Regression: add a test that starts a child that prints to stderr,
      confirms that no parent-fd reaches stderr of the caller, and confirms
      that `failure_reason` on the server task result contains the captured
      text.

### GAP-7: `McpManager` reports server as `connected` even when child exited before initialize
source_test: 4 (live terminal, 2026-04-22T23:05Z) — observed "MCP: 0 tool(s) ready from 1 server(s)."
severity: high
status: open
evidence: |
  When the npx child exited shortly after launch (usage printed and exit
  because the directory argument was missing), the dimmed startup message
  in `spawn_mcp_background` (main.rs:1423) reported
  `MCP: 0 tool(s) ready from 1 server(s).` — the `names.len() == n` branch —
  instead of `MCP: all servers failed to connect.` or
  `MCP: 0 tool(s) ready. 1 server(s) failed.`. This means
  `McpManager::connected_server_names()` returned the server name even though
  it never completed `initialize` and never registered a tool. The "connected"
  bit is being flipped too early (likely at "process spawned" or "transport
  open" stage).
fix_direction: |
  In `crates/ironhermes-mcp/src/manager.rs` + `server_task.rs`, redefine the
  source of `connected_server_names` so a server is "connected" only after:
  (a) the rmcp `initialize` handshake completed AND
  (b) `list_all_tools` returned without error.
  Move the `connected = true` flag set from any early spawn/transport point
  to inside `connect_and_serve` after both steps succeed. If either step
  fails, set `failure_reason` and keep `connected = false`. Then the
  `else if names.len() < n` branch at main.rs:1414 will fire correctly
  ("MCP: 0 tool(s) ready. 1 server(s) failed."). Add a regression test that
  runs a dummy child which exits immediately and asserts
  `connected_server_names()` is empty and `failure_reason` is populated.
