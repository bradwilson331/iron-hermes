## Gaps

### GAP-1: `hermes mcp add` / `hermes mcp test` hang indefinitely when server misbehaves
source_test: 1
severity: high
evidence: |
  `ironhermes mcp add mcpfs` with command=npx, args=empty:
  - stderr: `sh: line 1: jsonrpc:2.0: command not found`  (npx REPL reading MCP JSON-RPC as shell commands)
  - wizard never returns; user forced to Ctrl-C
root_cause: |
  `crates/ironhermes-cli/src/mcp_config.rs:479-509` — `attempt_connect_and_list` calls
  `connect_stdio(config).await?` then `client.list_all_tools().await?` with NO timeout
  wrapper. `McpServerConfig.connect_timeout` (default 60s) is never applied at the
  `cmd_test` / `cmd_add` call site, so a child process that is alive but not responding
  to MCP initialize blocks forever.
fix_direction: |
  Wrap both the transport connect and the initial `list_all_tools()` call in
  `tokio::time::timeout(Duration::from_secs(config.connect_timeout), ...)`. On timeout,
  kill the child and return a clear "Timed out waiting for MCP initialize response
  after Ns. Check command and args." error.

### GAP-2: `hermes mcp add` silently persists broken entries on connection failure
source_test: 1
severity: medium
evidence: |
  `ironhermes mcp add @modelcontextprotocol/server-filesystem` with wrong args:
  - red line: "Failed to connect: connection closed: initialize response"
  - immediately followed by: "MCP server added: @modelcontextprotocol/server-filesystem (0 tool(s) enabled)"
  Entry now lives in config.yaml with no tools and a broken command — no confirmation
  prompt, no retry, no abort path.
root_cause: |
  `crates/ironhermes-cli/src/mcp_config.rs:247-252` — connection error falls through to
  `(vec![], 0)` with a code comment "Still allow adding — user may fix server config
  later". The wizard proceeds to `save_mcp_server_config(&name, &server_config)?` with
  no further user input.
fix_direction: |
  Prompt after failure: "Connection failed. [R]etry / [S]ave anyway / [A]bort?"
  (default Abort). Retry re-runs `attempt_connect_and_list` against the same (or
  re-prompted) config; Abort exits without writing config.yaml.

### GAP-4: Name sanitization mismatch (CR-01 from code review) will compound once GAP-1/2 are fixed
source_test: 1 (blocked by GAP-1 from being observed)
severity: high
evidence: |
  Review finding CR-01: `unregister_by_prefix` in `ironhermes-mcp/src/manager.rs:117,152`
  uses the raw `server_name`, but tools are registered under sanitized names
  (`-`/`.`/`@`/`/` → `_`). Any server whose name contains these characters (e.g.
  `@modelcontextprotocol/server-filesystem`) will double-register tools on every
  `/reload-mcp` and never unregister cleanly.
fix_direction: |
  Store the sanitized prefix (`make_prefixed_name(&server_name, "")`) alongside the
  raw name in the manager's task map, and pass the sanitized prefix to
  `unregister_by_prefix`. Alternatively, reject server names that aren't already
  valid Rust-identifier-ish in `hermes mcp add` and in config loading.
