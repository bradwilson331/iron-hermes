## Summary

Phase 21.2 ports hermes-agent's MCP client infrastructure (~1100 lines Python) to Rust. The work has three major tracks: (1) a new `ironhermes-mcp` crate implementing the MCP client with stdio/HTTP transports using the `rmcp` SDK, (2) a cross-cutting `Arc<RwLock<ToolRegistry>>` migration touching 7 files, and (3) CLI subcommands and slash command wiring in ironhermes-cli.

**Critical discovery:** D-02 names `mcp-rust-sdk`, but that crate is abandoned at v0.1.1 (December 2024). The official SDK is `rmcp` v1.5.0 (published 2026-04-16), maintained at `github.com/modelcontextprotocol/rust-sdk`. This is what must be used. The planner should flag this discrepancy and use `rmcp` throughout.

The Python architecture (dedicated background asyncio loop + `run_coroutine_threadsafe`) maps cleanly to Rust: each MCP server becomes a `tokio::spawn` task, connections live inside the task's `async with` equivalent (RAII via RunningService), and `Tool::execute()` is already async so no thread bridging is needed.

**Primary recommendation:** Use `rmcp = { version = "1.5", features = ["client", "transport-child-process", "transport-streamable-http"] }` and migrate all `Arc<ToolRegistry>` callsites to `Arc<RwLock<ToolRegistry>>` in a single atomic wave.

## Architecture Patterns

### Recommended Crate Structure
```

├── Cargo.toml
└── src/
    ├── lib.rs               # Public API: McpManager, McpServerConfig
    ├── manager.rs           # McpManager: orchestrates per-server tasks
    ├── server_task.rs       # McpServerTask: per-server tokio task lifecycle
    ├── transport.rs         # Stdio + HTTP transport setup via rmcp
    ├── registry.rs          # Tool registration helpers (register_dynamic, unregister_by_prefix)
    ├── sampling.rs          # SamplingHandler: rate-limit, model resolution, LLM callback
    ├── security.rs          # build_safe_env(), sanitize_error(), CREDENTIAL_PATTERN
    └── config.rs            # McpServerConfig serde struct, env interpolation
```

### Pattern 1: rmcp Client Connection (stdio)
**What:** Connect to a stdio MCP server via TokioChildProcess, list tools, keep session alive.
**When to use:** `command` key present in server config (no `url`).
**Example:**
```rust
// Source: github.com/modelcontextprotocol/rust-sdk README
use rmcp::{ServiceExt, transport::{TokioChildProcess, ConfigureCommandExt}};
use tokio::process::Command;

let client = ().serve(
    TokioChildProcess::new(Command::new("npx").configure(|cmd| {
        cmd.arg("-y").arg("@modelcontextprotocol/server-filesystem");
        cmd.env_clear(); // then add safe env vars only
    }))?
).await?;

let tools = client.list_all_tools().await?;
// client is RunningService — keep alive while task is running
```

### Pattern 2: rmcp Client Connection (HTTP/StreamableHTTP)
**What:** Connect to an HTTP MCP server via StreamableHttpClientTransport.
**When to use:** `url` key present in server config.
**Example:**
```rust
// Source: rmcp docs, transport module
use rmcp::transport::StreamableHttpClientTransport;

let transport = StreamableHttpClientTransport::from_uri(
    "https://my-mcp-server.example.com/mcp"
);
let client = ().serve(transport).await?;
```

### Pattern 3: Per-Server Tokio Task (McpServerTask)
**What:** Each server lives in a `tokio::spawn` task. CancellationToken controls lifecycle.
**When to use:** Always — one task per configured server.
**Example:**
```rust
// Rust maps directly to Python MCPServerTask — no separate event loop needed
// because IronHermes is already fully async
pub struct McpServerTask {
    name: String,
    client: Option<RunningService<RoleClient, ()>>,
    registered_tool_names: Vec<String>,
    cancel_token: CancellationToken,
}

// In McpManager::start_all():
for (name, cfg) in &self.configs {
    let cancel = CancellationToken::new();
    let task = tokio::spawn(run_server_task(name.clone(), cfg.clone(), cancel.clone(), registry.clone()));
    self.tasks.insert(name.clone(), (task, cancel));
}
```

### Pattern 4: Arc<RwLock<ToolRegistry>> Migration
**What:** Change all `Arc<ToolRegistry>` to `Arc<RwLock<ToolRegistry>>` so MCP can write during reload.
**When to use:** Required for D-10. All reads use `.read().await`, writes use `.write().await`.
**Example:**
```rust
// Before (existing pattern):
let registry = Arc::new(registry);
// dispatch call:
registry.dispatch(name, args).await?

// After migration:
let registry = Arc::new(RwLock::new(registry));
// read-side (agent loop, tool dispatch — most callsites):
registry.read().await.dispatch(name, args).await?
// write-side (MCP register/unregister — only McpManager):
registry.write().await.register_dynamic(tool);
```

### Pattern 5: MCP Tool as Box<dyn Tool>
**What:** Each discovered MCP tool becomes a `Box<dyn Tool>` with `execute()` calling the rmcp session.
**When to use:** Tool registration after discovery.
**Example:**
```rust
pub struct McpTool {
    prefixed_name: String,    // "filesystem__read_file"
    server_name: String,      // "filesystem"
    original_name: String,    // "read_file"
    description: String,      // "[MCP: filesystem] Read a file at path"
    schema: ToolSchema,
    // Sender to the server task's channel for dispatching calls
    call_tx: mpsc::Sender<McpCallRequest>,
}

#[async_trait]
impl Tool for McpTool {
    fn name(&self) -> &str { &self.prefixed_name }
    fn toolset(&self) -> &str { "mcp" }
    fn description(&self) -> &str { &self.description }
    fn schema(&self) -> ToolSchema { self.schema.clone() }
    async fn execute(&self, args: serde_json::Value) -> anyhow::Result<String> {
        // Send call request to server task, await response
        let (resp_tx, resp_rx) = oneshot::channel();
        self.call_tx.send(McpCallRequest { name: self.original_name.clone(), args, resp_tx }).await?;
        resp_rx.await?
    }
}
```

### Pattern 6: Exponential Backoff (matching hermes-agent)
**What:** Reconnect up to 5 times with exponential backoff, max 60s.
**Example:**
```rust
// Matches Python: retries=0, backoff=1.0, max_backoff=60, max_retries=5
let mut retries = 0u32;
let mut backoff = Duration::from_secs(1);
const MAX_RETRIES: u32 = 5;
const MAX_BACKOFF: Duration = Duration::from_secs(60);

loop {
    match connect_and_serve(&config, &registry, &cancel_token).await {
        Ok(()) => break, // clean shutdown
        Err(e) if cancel_token.is_cancelled() => break,
        Err(e) => {
            retries += 1;
            if retries > MAX_RETRIES {
                tracing::warn!("MCP server '{}' exhausted retries: {}", name, e);
                break;
            }
            tracing::warn!("MCP server '{}' disconnected, retry {}/{} in {}s", name, retries, MAX_RETRIES, backoff.as_secs());
            tokio::select! {
                _ = tokio::time::sleep(backoff) => {}
                _ = cancel_token.cancelled() => break,
            }
            backoff = (backoff * 2).min(MAX_BACKOFF);
        }
    }
}
```

### Anti-Patterns to Avoid
- **Building a separate tokio runtime for MCP:** Python needed a separate asyncio loop because it mixed sync and async. Rust/tokio is uniformly async — just `tokio::spawn`.
- **Cloning ToolRegistry for each turn:** With `Arc<RwLock<ToolRegistry>>`, reads should use short-lived `.read().await` guards, not clone the whole registry.
- **Holding RwLock write guard across await points:** Write lock must be acquired, mutations made, then guard dropped before any `.await`. Never hold write guard across an async boundary.
- **Passing `Arc<RwLock<ToolRegistry>>` to execute_code's rpc_registry:** The rpc_registry (safe subset) does not need MCP tools and should remain `Arc<ToolRegistry>` (read-only, no dynamic registration). Only the main registry needs RwLock.

## ToolRegistry Changes Required

**New methods to add to `ToolRegistry` (D-10):**

```rust
impl ToolRegistry {
    /// Register a tool dynamically (e.g., from MCP discovery).
    /// Used during startup and /reload-mcp.
    pub fn register_dynamic(&mut self, tool: Box<dyn Tool>) {
        self.tools.insert(tool.name().to_string(), tool);
    }

    /// Remove all tools whose name starts with `{prefix}__`.
    /// Called on reload to clear one server's tools before re-registering.
    pub fn unregister_by_prefix(&mut self, server_name: &str) -> usize {
        let prefix = format!("{server_name}__");
        let before = self.tools.len();
        self.tools.retain(|name, _| !name.starts_with(&prefix));
        before - self.tools.len()
    }
}
```

Note: `register()` already exists as `pub fn register(&mut self, tool: Box<dyn Tool>)`. The new `register_dynamic()` is functionally identical — the distinction is semantic (dynamic = runtime MCP registration vs. static = startup built-in registration). Could reuse `register()` directly.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| MCP protocol (JSON-RPC, initialization, tool list/call) | Custom JSON-RPC client | rmcp v1.5.0 | Protocol negotiation, type safety, transport lifecycle managed by SDK |
| stdio subprocess management | Manual process spawn + pipes | rmcp TokioChildProcess | Handles process lifecycle, stdin/stdout piping, cleanup on drop |
| HTTP StreamableHTTP transport | Custom reqwest-based streaming | rmcp StreamableHttpClientTransport | SSE session management, reconnection semantics built in |
| Env var interpolation | Custom regex replace | Direct `std::env::var` + regex sub | Simple enough to hand-roll (Python does it in 4 lines) |
| Credential stripping | Custom string scanner | Regex with the exact patterns from hermes-agent | Pattern already defined and tested |

**Key insight:** The entire MCP connection lifecycle (initialization handshake, capability negotiation, JSON-RPC framing) is handled by rmcp. The plan should never try to implement these directly.

## Common Pitfalls

### Pitfall 1: Wrong crate name (mcp-rust-sdk vs rmcp)
**What goes wrong:** Adding `mcp-rust-sdk` to Cargo.toml gets abandoned v0.1.1 with completely different API.
**Why it happens:** D-02 says "mcp-rust-sdk" but the official SDK publishes under the name `rmcp`.
**How to avoid:** Use `rmcp = "1.5"` in Cargo.toml, not `mcp-rust-sdk`.
**Warning signs:** Compilation errors for `ClientSession`, `StdioServerParameters` (Python SDK types that don't exist in either Rust crate).

### Pitfall 2: Holding RwLock write guard across await
**What goes wrong:** `let mut guard = registry.write().await; guard.register(tool); some_async_call().await;` — this holds the write lock across an async boundary, potentially deadlocking.
**Why it happens:** RwLock write guards are not Send when held across await in tokio.
**How to avoid:** Always drop the guard before any `.await`. Acquire write lock, mutate, drop immediately.
```rust
// Wrong:
let mut guard = registry.write().await;
guard.register_dynamic(tool);
do_async_thing().await; // DEADLOCK RISK

// Right:
{
    let mut guard = registry.write().await;
    guard.register_dynamic(tool);
} // guard dropped here
do_async_thing().await; // safe
```

### Pitfall 3: execute_code rpc_registry type confusion
**What goes wrong:** `rpc_registry` in ExecuteCodeTool is typed as `Arc<ToolRegistry>`. After migration, if you try to pass `Arc<RwLock<ToolRegistry>>` as the rpc_registry, everything breaks.
**Why it happens:** rpc_registry is a separate instance (safe subset), not the main registry.
**How to avoid:** Keep rpc_registry as `Arc<ToolRegistry>` (no lock needed — it's never mutated). Build it fresh before the main registry, then wrap main in `Arc::new(RwLock::new(...))` separately. Use a type alias if needed.

### Pitfall 4: Tool naming separator collision
**What goes wrong:** Naming tools `server__tool` (double-underscore) but some server names or tool names already contain double-underscore.
**Why it happens:** D-06 mandates double-underscore. The Python `_convert_mcp_schema` sanitizes hyphens and dots to underscores first.
**How to avoid:** Sanitize server names and tool names (replace `-` and `.` with `_`) BEFORE applying the double-underscore separator. Match Python: `safe_name = name.replace('-', '_').replace('.', '_')`.

### Pitfall 5: Sampling callback async boundary
**What goes wrong:** Python uses `asyncio.to_thread()` to offload the sync LLM call. In Rust, the LLM call is already async, so the callback can directly await it. However, if the sampling callback holds a reference to the server task or registry incorrectly, it can create circular Arc references.
**Why it happens:** Sampling handler needs access to the LLM client (via resolver/config).
**How to avoid:** Pass an `Arc<dyn Fn(...) -> BoxFuture>` sampling callback into McpManager at construction time. McpManager stores it and passes per-server to each task. No registry access in the sampling path.

### Pitfall 6: CommandContext add breaks existing callsites
**What goes wrong:** Adding `Option<Arc<McpManager>>` to CommandContext requires all existing `CommandContext::new()` calls to be updated.
**Why it happens:** Rust struct field additions are not backward-compatible without Default impls.
**How to avoid:** Add the field with a builder method `with_mcp_manager()` like the existing `with_skill_registry()` pattern. Existing callers (main.rs line 744) pass `CommandContext::new(...)` and need no update until they want MCP.

### Pitfall 7: MCP discovery blocking startup
**What goes wrong:** Connecting to MCP servers during startup blocks the user from interacting with the agent until all servers connect (or timeout).
**Why it happens:** `discover_mcp_tools()` in Python awaits all connections with a 120s outer timeout. Per-server connect_timeout is 60s.
**How to avoid:** Spawn a detached `tokio::spawn` task for discovery that updates the registry in the background. The agent can start immediately; MCP tools become available once discovery completes. Log a startup message that MCP is loading.

```rust
const SAFE_ENV_KEYS: &[&str] = &[
    "PATH", "HOME", "USER", "LANG", "LC_ALL", "TERM", "SHELL", "TMPDIR",

fn build_safe_env(user_env: &HashMap<String, String>) -> HashMap<String, String> {
```

### /reload-mcp Handler (D-12)
```rust
// Based on hermes-agent/cli.py _reload_mcp()
async fn cmd_reload_mcp(ctx: &CommandContext) -> CommandResult {
    let Some(mcp_manager) = &ctx.mcp_manager else {
        return CommandResult::Output("MCP not configured.".to_string());
    };
    let old_names: Vec<String> = mcp_manager.connected_server_names();

    // Shutdown + reconnect (same as Python: shutdown_mcp_servers + discover_mcp_tools)
    mcp_manager.reload().await;

    let new_names: Vec<String> = mcp_manager.connected_server_names();
    let tool_count = mcp_manager.registered_tool_count();

    // Build status summary (matches Python output)
    let added: Vec<_> = new_names.iter().filter(|n| !old_names.contains(n)).collect();
    let removed: Vec<_> = old_names.iter().filter(|n| !new_names.contains(n)).collect();

    let mut parts = vec![format!("{tool_count} tool(s) from {} server(s)", new_names.len())];
    if !added.is_empty() { parts.push(format!("Added: {}", added.iter().map(|s| s.as_str()).collect::<Vec<_>>().join(", "))); }
    if !removed.is_empty() { parts.push(format!("Removed: {}", removed.iter().map(|s| s.as_str()).collect::<Vec<_>>().join(", "))); }

    CommandResult::Output(format!("MCP reloaded. {}", parts.join(". ")))
}
```

### McpManager::reload() signature
```rust
impl McpManager {
    /// Shutdown all servers, re-read mcp_servers config, reconnect, re-discover.
    /// Acquires write lock on registry to unregister_by_prefix then register_dynamic.
```

## Open Questions

1. **rmcp feature flags — exact names**
   - What we know: rmcp has client, stdio, and HTTP transport features
   - What's unclear: Exact feature flag names in rmcp 1.5.0 Cargo.toml
   - Recommendation: Check `https://docs.rs/crate/rmcp/1.5.0/features` or `cargo add rmcp --features ...` before implementing Wave 1

2. **rpc_registry type in execute_code**
   - What we know: rpc_registry is `Arc<ToolRegistry>` and must NOT include MCP tools (safe subset only)
   - What's unclear: Should rpc_registry be changed to `Arc<RwLock<ToolRegistry>>` (no writes needed, just type consistency) or kept as `Arc<ToolRegistry>`?
   - Recommendation: Keep `Arc<ToolRegistry>` for rpc_registry (built separately, never mutated after construction). Add a type alias `type ReadOnlyRegistry = Arc<ToolRegistry>` for clarity if needed.

3. **McpManager circular dependency resolution**
   - What we know: CommandContext (in ironhermes-core) needs to reference McpManager (in ironhermes-mcp), but ironhermes-mcp depends on ironhermes-core
   - What's unclear: Whether to use trait object `Arc<dyn McpReloader + Send + Sync>` or put McpManager in ironhermes-tools (like MemoryManagerHandle)
   - Recommendation: Define `trait McpReloader: Send + Sync { async fn reload(&self); fn connected_names(&self) -> Vec<String>; fn tool_count(&self) -> usize; }` in ironhermes-core or ironhermes-tools. McpManager implements it in ironhermes-mcp.

4. **Background vs. blocking discovery at startup**
   - What we know: Python `discover_mcp_tools()` blocks the CLI startup until all servers connect (max 120s outer timeout)
   - What's unclear: Should Rust discovery block `run_chat` startup or run in background?
   - Recommendation: Spawn as background task, print "MCP: connecting to N servers..." and let agent start immediately. On completion, print summary. This is strictly better UX than blocking.
