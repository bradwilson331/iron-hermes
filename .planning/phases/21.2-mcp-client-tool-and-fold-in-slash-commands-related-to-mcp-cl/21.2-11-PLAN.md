Close GAP-8 (high): `ironhermes gateway` hangs on Ctrl+C once MCP servers are connected because `McpManager` is never wired into `GatewayRunner`'s shutdown path AND `McpManager::shutdown_all` cannot async-kill stdio child processes. VERIFICATION.md GAP-8 evidence (2026-04-22T23:45Z): operator pressed Ctrl+C in a running gateway with an active stdio MCP server; the runner logged `Ctrl+C received, initiating graceful shutdown` (runner.rs:561), `self.cancel.cancel()` fired at runner.rs:569, but the process hung indefinitely until `kill -9`. Root cause confirmed by source read at HEAD:

1. **`crates/ironhermes-cli/src/main.rs:1291`**: `let mcp_manager = build_mcp_manager(&config, registry.clone()).await;` produces an `Option<Arc<McpManager>>` but `run_gateway` NEVER passes it to `GatewayRunner` — contrast the `memory_manager` branch at line 1341-1343 which DOES wire in via `runner.set_memory_manager(mgr)`. So the runner has no knowledge of the MCP manager to shut down.
2. **`crates/ironhermes-gateway/src/runner.rs:25-37`**: `GatewayRunner` has fields for `memory_manager`, `job_store`, `hook_registry`, `skill_registry`, `active_skills`, `cancel` — NO `mcp_manager` field. The shutdown block at runner.rs:556-575 only fires `self.cancel.cancel()` and drains the JoinSet. There is no MCP manager to consult.
3. **`crates/ironhermes-mcp/src/manager.rs:144-163 (shutdown_all)` + `crates/ironhermes-mcp/src/manager.rs:257-265 (Drop)`**: `shutdown_all` cancels each task's `CancellationToken` and awaits the `JoinHandle` unconditionally (no timeout), but does NOT kill the stdio child process. `Drop` cancels tokens synchronously but cannot `await` anything. The rmcp `TokioChildProcess` transport closes the parent→child pipe on cancel, BUT the child may keep running (e.g., a process reading stdin with `stdin.read_to_end()` that blocks on EOF, or a hung Node runtime). Tokio's process reaper keeps the runtime alive until the child is reaped, so the whole process hangs.
4. **`crates/ironhermes-mcp/src/transport.rs:11-38 (connect_stdio)`**: Passes `tokio::process::Command` directly to `rmcp::transport::TokioChildProcess::new(...)`, which owns the spawned child internally. There is no external handle on the `tokio::process::Child`, so the manager cannot call `start_kill()` or `kill().await` on it from outside. This is the structural gap that makes the fix surgical: we need to spawn the child ourselves and hold a handle.

Fix (matches the four-part fix_direction in VERIFICATION.md):
- **(a) Transport refactor (Task 1):** `transport::connect_stdio` spawns the child itself via `tokio::process::Command::spawn()` (using `Stdio::piped()` for stdin/stdout so rmcp can still take them), then wraps the rmcp `TokioChildProcess` using the `ConfigureCommandExt` path OR — if rmcp requires a Command — keeps its Command-based API and ALSO spawns a parallel Child from the same Command config, returning both. The cleanest API at this layer: `connect_stdio` returns `Result<(RunningService<RoleClient, ()>, Option<tokio::process::Child>)>`. HTTP transport returns `Ok((service, None))`.
- **(b) Server-task plumbing (Task 1 cont.):** `run_server_task` accepts an additional `child_slot: Arc<tokio::sync::Mutex<Option<tokio::process::Child>>>` parameter. On successful `connect_stdio`, it parks `Some(child)` in the slot. On retry/disconnect, it clears the slot. The slot is shared with `McpManager::tasks` so shutdown_all can reach in and `start_kill()` the child.
- **(c) Manager hard-kill + bounded await (Task 2):** `McpManager::tasks` HashMap value grows from `(JoinHandle, CancellationToken)` to `(JoinHandle, CancellationToken, Arc<tokio::sync::Mutex<Option<tokio::process::Child>>>)`. `shutdown_all` now:
  1. `cancel.cancel()` (existing)
  2. `if let Some(mut ch) = child_slot.lock().await.take() { let _ = ch.start_kill(); }` (NEW — hard SIGKILL)
  3. `tokio::time::timeout(Duration::from_secs(2), handle).await` (NEW — bounded wait). On timeout, log warning + drop handle + proceed; on success, push the `ServerTaskResult`.
  4. Registry unregister (existing, line 157-161).
- **(d) Runner wiring (Task 2):** `GatewayRunner` grows `mcp_manager: Option<Arc<McpManager>>` field + `set_mcp_manager` setter. In `start()` shutdown path (between tokio::select! at line 566 and `self.cancel.cancel()` at line 569), await `mgr.shutdown_all()`. `run_gateway` in main.rs wires `runner.set_mcp_manager(mcp_manager.clone());` before `runner.start()`.
- **(e) Regression tests (both tasks):** `shutdown_all_returns_within_timeout_when_stdio_child_blocks` in manager.rs with an outer 5-second test-side timeout wrapping shutdown_all; `gateway_runner_invokes_mcp_shutdown_all_on_cancel` in runner.rs (OR a source-grep regression in runner.rs tests proving the `shutdown_all().await` call exists in the shutdown path — whichever is less fragile given the dispatch-loop dependencies; pick the grep fallback if a live harness would require a full Telegram adapter mock).

Purpose: Restore operator-sane Ctrl+C handling for `ironhermes gateway`. Without this fix, any production gateway deployment with an MCP server attached is un-shutdownable without SIGKILL — a severity-high UX regression and a deployment hazard (systemd will escalate to SIGKILL after its stop-timeout anyway, risking half-written state).

pub async fn connect_stdio(
    config: &McpServerConfig,
) -> Result<RunningService<RoleClient, ()>> {
    let command = config.command.as_ref().ok_or_else(...)?;
    let safe_env = build_safe_env(&config.env);
    let args = config.args.clone();

    use rmcp::transport::{ConfigureCommandExt, TokioChildProcess};
    let transport = TokioChildProcess::new(
        tokio::process::Command::new(command).configure(move |cmd| {
            for arg in &args { cmd.arg(arg); }
            cmd.env_clear();
            for (k, v) in &safe_env { cmd.env(k, v); }
        }),
    )?;
    let client = ().serve(transport).await?;
    Ok(client)
}

**Design note on the refactor:** rmcp's `TokioChildProcess::new` owns the child internally. To get an external handle, we need to either:
- **Option A (clean but may not be supported by rmcp 1.5):** use a rmcp API that accepts an already-spawned Child (e.g. `TokioChildProcess::from_child` — verify existence). If present, spawn the child ourselves and hand ownership to rmcp while keeping a Weak reference — NOT POSSIBLE because Child is not Clone.
- **Option B (used in this plan):** spawn the child ONCE, keep the `tokio::process::Child`, and use rmcp's lower-level transport constructors that accept stdin/stdout pipes directly. Inspect `rmcp::transport::child_process` module for APIs like `ChildProcessTransport::new(child_stdin, child_stdout)`. If that path is too deep to thread, fall back to:
- **Option C (pragmatic fallback used in this plan):** spawn the child ourselves with `Stdio::piped()` for stdin/stdout, take the `ChildStdin` and `ChildStdout` out of the Child, pass them to rmcp via a constructor that accepts `AsyncRead + AsyncWrite` (rmcp supports this via its `IoTransport` or similar — `rmcp::transport::async_rw::AsyncRwTransport` in newer versions). Then the manager's Child handle is a "stripped" Child (stdin/stdout taken) but still owns the process for SIGKILL.

If none of the above are exposed in rmcp 1.5, the executor should consult rmcp docs via Context7 MCP during Task 1 discovery and choose the cleanest supported approach. Minimum bar: `connect_stdio` must return an external `tokio::process::Child` handle alongside the `RunningService`.

    **Part B — rewrite `shutdown_all` to hard-kill + bounded-timeout.**

    ```rust
    /// Shutdown all running server tasks and unregister their tools.
    ///
    /// GAP-8: hard-kill stdio children via `Child::start_kill()` and bound the
    /// JoinHandle await to 2 seconds. Prior to this fix, a stdio child that
    /// ignored its parent-pipe EOF kept tokio's process reaper alive indefinitely,
    /// hanging `ironhermes gateway` on Ctrl+C.
    pub async fn shutdown_all(&self) -> Vec<ServerTaskResult> {
        use tokio::time::{timeout, Duration};

        let mut tasks = self.tasks.lock().await;
        let mut flags = self.connected_flags.lock().await;  // plan 10
        let mut results = Vec::new();

        for (name, (handle, cancel, child_slot)) in tasks.drain() {
            tracing::info!(server = %name, "Shutting down MCP server");

            // 1. Cancel the task's cancellation token (tells the serve loop to break)
            cancel.cancel();

            // 2. GAP-8: hard-kill the stdio child (if any). start_kill() is
            //    non-blocking — it sets the SIGKILL flag on the child. The
            //    subsequent timeout(...handle) does the actual bounded wait.
            if let Some(mut child) = child_slot.lock().await.take() {
                match child.start_kill() {
                    Ok(()) => tracing::debug!(server = %name, "Sent SIGKILL to MCP stdio child"),
                    Err(e) => tracing::warn!(
                        server = %name,
                        error = %e,
                        "Failed to SIGKILL MCP stdio child (may already be reaped)"
                    ),
                }
            }

            // 3. GAP-8: await the JoinHandle with a bounded 2-second ceiling.
            //    If the child ignores SIGKILL or the task is genuinely stuck,
            //    we log a warning and proceed — never blocking the gateway's
            //    Ctrl+C return.
            match timeout(Duration::from_secs(2), handle).await {
                Ok(Ok(result)) => results.push(result),
                Ok(Err(join_err)) => tracing::warn!(
                    server = %name,
                    error = %join_err,
                    "MCP server task panicked during shutdown"
                ),
                Err(_elapsed) => tracing::warn!(
                    server = %name,
                    "MCP server task did not join within 2s of cancel+SIGKILL; proceeding"
                ),
            }

            // 4. Unregister tools (existing — GAP-4 sanitized prefix per plan 07)
            let mut guard = self.registry.write().await;
            let removed = guard.unregister_by_prefix(&sanitize_server_name(&name));
            tracing::debug!(server = %name, removed, "Unregistered MCP tools");
            drop(guard);

            // 5. Clean the plan-10 connected flag
            flags.remove(&name);
        }
        results
    }
    ```
