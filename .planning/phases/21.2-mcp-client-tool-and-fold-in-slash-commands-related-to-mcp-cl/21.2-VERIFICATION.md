#### 2. hermes mcp test colored output

**Test:** With a server configured, run `hermes mcp test <name>` and verify the terminal output matches UI-SPEC: bold cyan heading, tool names in yellow, descriptions truncated at 80 cols, "Connected -- N tool(s) available" in green.
**Expected:** Correct column widths, color codes, and tool count displayed.
**Why human:** Colored terminal output and live connection cannot be verified programmatically. The static-grep tests only confirm sanitize_error is called, not visual rendering.

#### 3. /reload-mcp end-to-end in running session

**Test:** Start `hermes` chat with one or more MCP servers configured. Type `/reload-mcp`. Verify the output matches UI-SPEC format. Then kill one configured server and type `/reload-mcp` again — verify the "Failed: <name> (<error>)" partial failure line appears.
**Expected:** "MCP reloaded. N tool(s) from N server(s)." on success; "MCP reloaded. N tool(s) from N server(s). Failed: server_name (error)." on partial failure.
**Why human:** Requires live agent session with real MCP servers; the McpReload REPL arm and D-12 partial failure display verified by code inspection but not exercised interactively.

#### 4. Background startup messages on stderr

**Test:** Start `hermes` chat with mcp_servers configured. Observe stderr output before the first prompt.
**Expected:** "MCP: connecting to N server(s) in background..." appears immediately (dimmed), followed by "MCP: N tool(s) ready from N server(s)." or "MCP: N tool(s) ready. M server(s) failed." after connection attempts settle. Agent prompt appears without waiting for MCP.
**Why human:** Timing behavior (background spawn, non-blocking startup) and stderr rendering require a live session to confirm.
