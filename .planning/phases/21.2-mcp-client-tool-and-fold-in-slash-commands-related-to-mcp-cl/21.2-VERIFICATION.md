## Remaining Gaps (for /gsd-plan-phase 21.2 --gaps)

### GAP-5: `hermes` REPL prompt does not render until first keystroke (D-07 violation)
severity: high
source_test: 4 (live)
evidence: |
  After `ironhermes` starts with 1+ MCP servers configured, the dimmed
  `MCP: connecting to 1 server(s) in background...` line prints on stderr,
  but the banner (`Chat · ...`) and input prompt (`You: `) do not appear until
  the user presses Enter. The redraw is gated behind stdin activity instead of
  flushing at startup. This violates the D-07 contract "agent prompt appears
  immediately without blocking on MCP connection" — from the user's perspective
  the CLI looks frozen.
fix_direction: |
  Explicitly flush stdout after the banner and the initial input prompt render,
  or force an initial prompt paint in the REPL read-loop before the first
  `read_line` call. Likely a missing `io::stdout().flush()` or a rustyline
  `readline_with_initial` layout issue in the REPL setup in
  `crates/ironhermes-cli/src/main.rs` (search for where `You:` / banner is printed
  around the REPL loop entry).

### GAP-6: Tracing WARN floods startup + child MCP stderr leaks into parent terminal
severity: medium
source_test: 4 (live)
evidence: |
  At startup the terminal fills with 8-10 lines of WARN tracing from
  `SkillRegistry` (invalid skill names) and `Cron` (legacy jobs.json format),
  visually burying the MCP startup messages and the REPL prompt. Additionally,
  when the npx child MCP process prints its usage text to stderr
  (`Usage: mcp-server-filesystem [allowed-directory] ...`), that output
  appears directly in the parent terminal — in the observed session it landed
  on the `You:` prompt line, corrupting it.
fix_direction: |
  (a) In `hermes` CLI init (main.rs near the `tracing_subscriber` setup),
      default the EnvFilter to `error` when running interactive chat/REPL
      unless RUST_LOG is set. Keep the current default for `hermes agent` /
      gateway commands where operators want diagnostics.
  (b) In `ironhermes-mcp` server_task, capture the child's stderr into a
      per-server log buffer or a rotating log file rather than inheriting
      the parent's stderr fd. Surface only a `failure_reason` (already
      sanitized) on the control-plane channel. `rmcp::transport::child_process`
      or `TokioChildProcess` configuration may need to set `stderr(Stdio::piped())`
      and spawn a reader task.

### GAP-7: `McpManager` reports `connected` for servers whose child exited before MCP initialize
severity: high
source_test: 4 (live) — observed "MCP: 0 tool(s) ready from 1 server(s)."
evidence: |
  When the npx child exits shortly after launch (e.g. missing required
  directory argument causes `mcp-server-filesystem` to print usage and exit),
  the dimmed startup message still reports
  `MCP: 0 tool(s) ready from 1 server(s).` instead of
  `MCP: all servers failed to connect.` or
  `MCP: 0 tool(s) ready. 1 server(s) failed.`. The `names.len() == n` branch
  in `spawn_mcp_background` at `crates/ironhermes-cli/src/main.rs:1423`
  fires because `connected_server_names()` returns the server name even
  though no tool ever registered. The "connected" signal is set too early
  (e.g. at transport-connect time) rather than at "initialize succeeded AND
  list_tools returned".
fix_direction: |
  In `ironhermes-mcp/src/manager.rs`, redefine `connected_server_names` (or
  the bit it reads) so a server is considered connected only after both:
  (a) the rmcp `initialize` handshake completed AND (b) `list_all_tools`
  returned without error. `server_task::connect_and_serve` should flip a
  `connected: true` flag on the shared task state ONLY on the success path,
  not at the "process spawned" or "transport open" step. Then the `else if
  names.len() < n` branch in `spawn_mcp_background` will fire correctly and
  the user will see `MCP: 0 tool(s) ready. 1 server(s) failed.`. Consider
  also logging the per-server `failure_reason` so operators can diagnose
  without `RUST_LOG=debug`.

### GAP-8: `ironhermes gateway` does not exit on Ctrl+C once MCP servers are connected
severity: high
source_test: 5 (live terminal, 2026-04-22, operator ran `ironhermes gateway` after MCP add → Telegram round-trip worked → Ctrl+C did not terminate the process)
evidence: |
  In `crates/ironhermes-cli/src/main.rs` `run_gateway` (line 1204-1349),
  `build_mcp_manager(&config, registry.clone()).await` is called at line 1291
  and its return value is bound to the local `mcp_manager` — BUT this value
  is never passed to `GatewayRunner`. The runner at `crates/ironhermes-gateway/src/runner.rs:556-566`
  handles `tokio::signal::ctrl_c()` by cancelling its own `self.cancel`
  token; it has no knowledge of the MCP manager and no hook to call
  `shutdown_all()` on exit. When `runner.start().await` returns, the
  `mcp_manager` local drops, which fires `Drop for McpManager` at
  `manager.rs:258-265`. That Drop impl cancels the per-server cancellation
  tokens synchronously inside a sync context, but it does NOT `await` the
  tokio `JoinHandle` in `tasks` and does NOT explicitly kill the stdio
  child processes. The rmcp transport closes the parent→child pipe on
  cancel, but the child may keep running, and tokio's process reaper
  keeps the runtime alive until the child is reaped. Net effect: Ctrl+C
  is acknowledged but the process hangs until SIGKILL or child self-exit.
root_cause_summary: |
  Two defects compound:
  1. `run_gateway` holds the MCP manager but never wires its `shutdown_all`
     into the gateway's shutdown path.
  2. `Drop for McpManager` fires tokens but doesn't ensure child-process
     termination — async cleanup work can't happen from a sync Drop.
fix_direction: |
  (a) In `crates/ironhermes-gateway/src/runner.rs`: add a setter on
      `GatewayRunner` (e.g. `set_mcp_manager(Option<Arc<McpManager>>)`)
      that stores the manager on the runner. In `run`, after the
      `tokio::select!` returns from SIGINT or cancel, call
      `if let Some(ref mgr) = self.mcp_manager { let _ = mgr.shutdown_all().await; }`
      BEFORE `self.cancel.cancel()` so stdio children are explicitly
      cancelled and (if GAP-7's Arc<AtomicBool> work is in) flipped to
      `connected=false`, and BEFORE `while join_set.join_next().await.is_some() {}`.
  (b) In `crates/ironhermes-cli/src/main.rs` `run_gateway`: after
      `runner.set_active_skills(active_skills)` (line 1347) and before
      `runner.start().await` (line 1348), add
      `runner.set_mcp_manager(mcp_manager.clone());` so the runner
      owns a clone.
  (c) In `crates/ironhermes-mcp/src/manager.rs` `shutdown_all`: after
      `cancel.cancel()`, explicitly drop/close the child handles so
      SIGKILL is issued rather than relying on pipe-EOF. The rmcp
      `TokioChildProcess` or the underlying `tokio::process::Child` has
      a `start_kill()` / `kill().await` method — call it so the process
      is guaranteed to terminate, then await the JoinHandle with a
      short bounded timeout (e.g. 2s). After that, if the handle still
      has not joined, log a warning but proceed — do not block exit.
  (d) Regression: a new integration test in ironhermes-gateway that
      spawns a dummy long-lived child process as an MCP server, sends
      a simulated SIGINT via `cancel.cancel()`, and asserts the runner's
      `start()` returns within N seconds AND the child is reaped
      (via `Child::try_wait().is_some()`).
