# Phase 21.2: Code Review Report

**Reviewed:** 2026-04-22T00:00:00Z
**Depth:** standard
**Files Reviewed:** 25
**Status:** issues_found

## Summary

Phase 21.2 delivers a well-structured MCP client crate (`ironhermes-mcp`) with good separation of concerns: `config`, `security`, `transport`, `tool`, `server_task`, `manager`, `sampling`, plus a `/reload-mcp` slash command and `hermes mcp` CLI subcommands. The trait-object `McpReloader` pattern (D-15) cleanly avoids a circular dependency between `ironhermes-core` and `ironhermes-mcp`, matching the established `MemoryManagerHandle` pattern.

Security primitives (`build_safe_env`, `sanitize_error`, `${ENV_VAR}` interpolation) are well-tested and correctly port the hermes-agent behavior. Write-lock hold duration across `.await` was consciously minimized (see `server_task.rs:160` comment), and rmcp 1.5 transport wiring looks correct.

However, two critical correctness bugs exist around the `server__tool` naming scheme when server names contain hyphens or dots: the tool-discovery presence heuristic and `unregister_by_prefix` both compare against the ORIGINAL (un-sanitized) server name, but tools are registered under SANITIZED prefixes. This means servers named `my-server` or `github.com` (any name with `-` or `.`) will never be detected as "connected" and their tools will never be unregistered on reload — the registry accumulates duplicates every reload. Additionally, `cmd_add` interpolates `${ENV_VAR}` placeholders BEFORE persisting to `config.yaml`, losing the indirection and writing resolved secrets to disk.

The retry-count off-by-one means servers get 6 attempts instead of the documented 5. A number of warnings around silent error swallowing (try_lock failures, unwrap_or_default on config parse errors, read_line().ok() on credential input) would surprise users in failure modes.

## Critical Issues

### CR-01: `unregister_by_prefix` and connection heuristic use un-sanitized server names

**File:** `crates/ironhermes-mcp/src/manager.rs:117, 152`
**File:** `crates/ironhermes-mcp/src/tool.rs:79-83`

**Issue:** Tools are registered using `make_prefixed_name(server, tool)` which replaces `-` and `.` with `_` before joining with `__`. So a server named `my-server` registers tools as `my_server__tool_x`. However:

1. `McpManager::start_all_and_wait` (line 117) checks `t.function.name.starts_with(&format!("{name}__"))` using the RAW server name (`my-server__`), which will never match the registered prefix (`my_server__`). Consequence: the `StartResult.connected` list remains empty for every server whose name contains `-` or `.`; reload UI reports "0 connected" even when servers are healthy.
2. `McpManager::shutdown_all` (line 152) calls `guard.unregister_by_prefix(&name)` — again with the RAW name. For server `my-server`, this tries to remove tools starting with `my-server__`, but the registered tools are `my_server__tool_x`, so NOTHING is removed. Every `/reload-mcp` leaks the previous generation's tools into the registry, producing duplicate / stale tool entries over time (D-09 contract violation).
3. `ToolRegistry::unregister_by_prefix` (`ironhermes-tools/src/registry.rs:53`) has the same issue: it takes `server_name`, formats `{server_name}__`, but the stored keys were produced with the sanitized name.

**Fix:**
```rust
// In ironhermes-mcp/src/manager.rs, reuse the sanitizer:
use crate::tool::make_prefixed_name;

// Line 117 replacement:
let sanitized_prefix = format!("{}__",
    make_prefixed_name(name, "").trim_end_matches("__"));
// Or simpler: expose a `sanitize_server_name(name) -> String` from tool.rs.

let has_tools = guard
    .get_definitions(None)
    .iter()
    .any(|t| t.function.name.starts_with(&sanitized_prefix));

// Line 152 replacement:
let sanitized = name.replace('-', "_").replace('.', "_");
let removed = guard.unregister_by_prefix(&sanitized);
```

Also add a regression test with a hyphenated server name (e.g. `my-server`) that asserts `shutdown_all` removes the tools and `start_all_and_wait` reports it as connected.

### CR-02: `cmd_add` persists interpolated (resolved) secrets to `config.yaml`, losing `${ENV_VAR}` indirection

**File:** `crates/ironhermes-cli/src/mcp_config.rs:232, 284`

**Issue:** `cmd_add` constructs `server_config` and immediately calls `interpolate_config(&mut server_config)` (line 232) BEFORE calling `save_mcp_server_config(&name, &server_config)` (line 284). This means that if a user enters `${GITHUB_TOKEN}` at the `auth_token` prompt, or `${HOME}/.npm` at the command prompt, the RESOLVED value (the actual token or path) is what gets written to `config.yaml`. The `${ENV_VAR}` indirection is silently lost on first save.

This has two distinct downsides:
1. **Data loss / surprise**: A user who wants their config to reference `${GITHUB_TOKEN}` so they can commit `config.yaml` to git will find their actual token persisted on disk instead. Next time they run the tool, the env var is ignored in favor of the hardcoded value.
2. **Security**: Secrets that the user never intended to be written to plaintext on disk (they were hoping to keep them in env vars) are now persisted.

The same `interpolate_config` call is correctly used before `attempt_connect_and_list` (so the connection test uses real values), but that should be a local copy, not a mutation of the value destined for disk.

**Fix:**
```rust
// Build the config as user entered it (with ${VAR} placeholders intact):
let server_config = McpServerConfig {
    command: final_command,
    args: final_args,
    url: final_url,
    env: env_vars,
    auth: auth_token,
    ..McpServerConfig::default()
};

// Test connection with a LOCAL interpolated copy — do not mutate the one we save:
let mut test_config = server_config.clone();
interpolate_config(&mut test_config);
let tools = attempt_connect_and_list(&test_config).await;
println!();

// ... enabled_tools logic unchanged ...

// Save the ORIGINAL (uninterpolated) config so ${VAR} placeholders survive:
let mut final_config = server_config;
final_config.enabled_tools = enabled_tools;
save_mcp_server_config(&name, &final_config)?;
```

Add a unit test that saves a config with `auth: Some("${GITHUB_TOKEN}".into())` and asserts the written YAML contains the literal `${GITHUB_TOKEN}`, not its resolved value.

### WR-01: Retry-count off-by-one — servers get 6 attempts instead of the documented 5

**File:** `crates/ironhermes-mcp/src/server_task.rs:13, 65-76`

**Issue:** `MAX_RETRIES: u32 = 5` and the doc comment says "Maximum number of reconnection retries before giving up". The loop pre-increments `retries += 1` on each error (line 65), then breaks only when `retries > MAX_RETRIES` (line 67). So the sequence is: attempts 1,2,3,4,5,6 — with retries={1,2,3,4,5,6} — and break triggers when retries becomes 6. That's 6 connection attempts total, or 5 retries after the initial attempt. The D-05 contract (hermes-agent parity) should be verified, but the intra-code expectation (`"exhausted {MAX_RETRIES} retries"` log) is inconsistent with observable behavior.

**Fix:** Either change the comparison to `retries >= MAX_RETRIES`, or document MAX_RETRIES as "maximum reconnection ATTEMPTS" rather than "retries":
```rust
// Option A: make retries==5 the terminating value (5 total attempts)
if retries >= MAX_RETRIES {
    failure_reason = Some(sanitized);
    break;
}
```
Add a unit test that counts actual invocations of `connect_and_serve` under a mock transport that always fails.

### WR-02: `connected_server_names()` silently returns empty Vec on Mutex contention

**File:** `crates/ironhermes-mcp/src/manager.rs:184-190`

**Issue:** `connected_server_names` uses `self.tasks.try_lock()` and returns `Vec::new()` if the mutex is already locked (e.g., by a concurrent `start_all_and_wait` or `shutdown_all` call). This is called from the `/reload-mcp` UI path (`main.rs:897`) to capture `old_names` for add/remove diffs BEFORE calling `reload()`. If contention occurs, the diff shows every server as "added" (since old_names is empty), producing misleading UI output like `Added: server1, server2` even when nothing changed.

**Fix:** Either block (use `lock().await` — but that requires making the method async, which conflicts with the sync `McpReloader::connected_server_names` trait method), or log a debug line on contention so operators can diagnose spurious UI. Because the trait requires sync, consider refactoring the trait method to `async fn` on the next major revision; for now log the contention:
```rust
pub fn connected_server_names(&self) -> Vec<String> {
    match self.tasks.try_lock() {
        Ok(tasks) => tasks.keys().cloned().collect(),
        Err(_) => {
            tracing::debug!(
                "connected_server_names: tasks mutex contended -- returning empty Vec"
            );
            Vec::new()
        }
    }
}
```

### WR-03: `McpReloader::reload` silently treats config-load failures as "no servers"

**File:** `crates/ironhermes-mcp/src/manager.rs:215-229`

**Issue:** When `Config::load()` fails (e.g., malformed YAML), the match arm returns `HashMap::new()`, which is then passed to `reload_and_report`. The effect is that `/reload-mcp` will:
1. Shut down all currently-running MCP servers.
2. Start zero new ones (because the new config is empty).
3. Return `connected=[]`, `failed=[]`, `tool_count=0` — a completely "clean" looking result.

The user is told "0 tool(s) from 0 server(s)" with no indication that config parsing failed. They will assume their config is empty when in fact it's broken.

**Fix:** Log the error and surface it to the user via the `failed` vector (synthetic entry):
```rust
let new_configs: HashMap<String, McpServerConfig> = match ironhermes_core::Config::load() {
    Ok(config) => /* ... as before ... */,
    Err(e) => {
        tracing::error!(error = %e, "McpReloader::reload failed to load config");
        let mut result = self.reload_and_report(HashMap::new()).await;
        result.failed.push((
            "<config>".to_string(),
            sanitize_error(&format!("failed to load config.yaml: {}", e)),
        ));
        return ironhermes_core::commands::context::McpReloadResult {
            connected: result.connected,
            failed: result.failed,
            tool_count: result.tool_count,
        };
    }
};
```

Same concern applies to the `filter_map(|(name, val)| serde_yaml::from_value::<...>().ok())` chain on lines 222-226: entries with malformed per-server config are silently dropped. Surface those as `failed` entries too.

### WR-04: `build_mcp_manager` silently drops malformed per-server config entries

**File:** `crates/ironhermes-cli/src/main.rs:1383-1391`

**Issue:** Similar to WR-03, `build_mcp_manager` uses `.filter_map` to silently discard any `mcp_servers.<name>` entry whose YAML doesn't deserialize to `McpServerConfig`. A user with one typo'd entry will see their server just ... not appear in `hermes mcp list` and get no startup warning. Given that this runs at `run_chat`/`run_single`/`run_gateway` startup, the failure is completely invisible.

**Fix:**
```rust
let mut mcp_configs: HashMap<String, ironhermes_mcp::McpServerConfig> = HashMap::new();
for (name, val) in &config.mcp_servers {
    match serde_yaml::from_value::<ironhermes_mcp::McpServerConfig>(val.clone()) {
        Ok(c) => { mcp_configs.insert(name.clone(), c); }
        Err(e) => {
            eprintln!(
                "{} MCP server '{}' has invalid config: {}",
                "Warning:".yellow().bold(),
                name,
                e
            );
        }
    }
}
```

### WR-05: Credential prompt echoes API key / Bearer token in plaintext

**File:** `crates/ironhermes-cli/src/mcp_config.rs:207-222`

**Issue:** `cmd_add` prompts for an auth token with a plain `read_line(&mut line)` — the typed characters echo to the terminal. Users entering long-lived GitHub or API tokens at this prompt will see them on screen, leak them to scroll-back / ttyrec logs, and may be caught on-camera in screen shares.

**Fix:** Use the `rpassword` crate (or similar) for no-echo input:
```rust
// Cargo.toml (dev-dependency → regular dependency for cli)
// rpassword = "7"

if needs_auth {
    let token = rpassword::prompt_password("  API key / Bearer token (input hidden): ")
        .context("failed to read token")?;
    let trimmed = token.trim();
    if trimmed.is_empty() { None } else { Some(trimmed.to_string()) }
} else {
    None
}
```
Same fix applies to the `Value for {}` prompt in the env-vars loop (line 202) since users will likely enter tokens there too.

### WR-06: `Drop` for `McpManager` uses `try_lock` — tasks may not be cancelled

**File:** `crates/ironhermes-mcp/src/manager.rs:251-260`

**Issue:** On drop, if the `tasks` mutex is contended (e.g., another task is still running `start_all_and_wait`), `try_lock` returns `Err` and the Drop silently skips cancellation. All background `tokio::spawn`'d server tasks then continue running as orphans with no way to signal them. The comment acknowledges the risk ("to avoid orphaned background tasks") but the implementation doesn't actually deliver that guarantee.

A proper fix requires using `std::sync::Mutex` (not tokio's) for the tasks map — or storing the cancellation tokens in a separate `std::sync::Mutex<Vec<CancellationToken>>` alongside the tokio mutex, so Drop can cancel them without needing to await.

**Fix (option A, simplest):** Store cancellation tokens separately:
```rust
pub struct McpManager {
    // ...
    all_cancel_tokens: std::sync::Mutex<Vec<CancellationToken>>,
}
// On every insert into tasks, also push into all_cancel_tokens.
// On Drop, iterate all_cancel_tokens.lock().unwrap() and cancel each.
```
Or document explicitly that callers MUST call `shutdown_all().await` before dropping.

### WR-07: Interpolation regex recompiled on every `interpolate_env` call

**File:** `crates/ironhermes-mcp/src/config.rs:84-90`

**Issue:** `interpolate_env` calls `regex::Regex::new(...)` on every invocation. The regex is a simple, fixed pattern that cannot fail, yet it is compiled from source each call. When `interpolate_config` runs it calls `interpolate_env` for each arg, env var, header, plus command/url/auth — potentially 10+ compilations per server. This matches an established anti-pattern that the codebase otherwise avoids (see `CREDENTIAL_PATTERN` in `security.rs:34` which correctly uses `LazyLock`).

**Fix:**
```rust
use std::sync::LazyLock;

static ENV_INTERP_RE: LazyLock<regex::Regex> =
    LazyLock::new(|| regex::Regex::new(r"\$\{([^}]+)\}").unwrap());

pub fn interpolate_env(value: &str) -> String {
    ENV_INTERP_RE.replace_all(value, |caps: &regex::Captures| {
        std::env::var(&caps[1]).unwrap_or_else(|_| caps[0].to_string())
    })
    .into_owned()
}
```

### WR-08: `read_line().ok()` swallows stdin errors in interactive prompts

**File:** `crates/ironhermes-cli/src/mcp_config.rs:212, 583, 597`

**Issue:** `prompt_line`, `prompt_yn`, and the auth-token prompt all use `io::stdin().lock().read_line(&mut line).ok()`. If stdin is closed (EOF — e.g. `hermes mcp add foo < /dev/null`), `read_line` returns `Ok(0)` but any actual I/O error returns `Err`, which is discarded. The prompt appears to "succeed" with an empty line, defaults are used silently, and the user never learns why the wizard skipped their input.

**Fix:** Propagate EOF / errors as an early `anyhow::bail!`:
```rust
fn prompt_line(prompt: &str, default: &str) -> anyhow::Result<String> {
    // ... print prompt ...
    let mut line = String::new();
    let bytes_read = io::stdin().lock().read_line(&mut line)
        .context("failed to read from stdin")?;
    if bytes_read == 0 {
        anyhow::bail!("stdin closed before input received");
    }
    let trimmed = line.trim();
    Ok(if trimmed.is_empty() { default.to_string() } else { trimmed.to_string() })
}
```

This is a warning rather than info because silent input-consumption produces baffling UX (the wizard just loops or completes with garbage defaults).

### IN-04: `configure` subcommand clobbers the `enabled_tools` list without showing current state

**File:** `crates/ironhermes-cli/src/mcp_config.rs:408-469`

**Issue:** `cmd_configure` discovers tools, prompts "Enable tool 'X'?" for each, and writes a fresh `selected_tools` list. If a user previously enabled only 2 of 10 tools and now wants to add a third, they have to confirm all existing selections plus any new ones. Worse: the default for each prompt is `true` (line 450), so a user who just hits Enter through the wizard will end up with ALL tools enabled, silently overriding their prior selection.

**Fix:** Load the existing `enabled_tools` list first and pre-select those as the default for each prompt:
```rust
let existing: std::collections::HashSet<String> = server.enabled_tools
    .as_ref()
    .cloned()
    .unwrap_or_default()
    .into_iter()
    .collect();

for (tool_name, _description) in &tools {
    let was_enabled = existing.is_empty() || existing.contains(tool_name);
    let enable = prompt_yn(&format!("Enable tool '{}'?", tool_name), was_enabled);
    if enable { selected_tools.push(tool_name.clone()); }
}
```

### IN-05: `unwrap_or_default()` on `Config::load` loses error visibility

**File:** `crates/ironhermes-cli/src/mcp_config.rs:72, 135, 303, 323, 409`

**Issue:** Every `McpAction` handler opens with `let config = Config::load().unwrap_or_default();` — if the config file is malformed, the user silently operates on defaults. `hermes mcp list` will say "No MCP servers configured" even when their `config.yaml` is chock full of entries (just unparseable). This is consistent with other codebase paths but especially harmful for `hermes mcp remove` (which will "succeed" by writing a default config, overwriting the broken one with empty content).

**Fix:** Use `Config::load().context("failed to load config.yaml; fix YAML errors or re-run")` and bubble up the error. Alternatively, log a warning and continue with defaults only in idempotent read-only paths like `list`.

### IN-06: `cmd_remove` writes config even when nothing changed

**File:** `crates/ironhermes-cli/src/mcp_config.rs:553-569`

**Issue:** `remove_mcp_server_config` always calls `std::fs::write(&config_path, serde_yaml::to_string(&doc)?)`, even if the server name wasn't present under `mcp_servers` or the file had no `mcp_servers` mapping at all. This rewrites `config.yaml` through the serde_yaml round-trip, potentially normalizing formatting (indent, quote style, comment loss) unnecessarily. Low impact, but losing YAML comments on an unrelated operation is user-hostile.

**Fix:** Short-circuit if the target was not present:
```rust
let mut modified = false;
if let Some(mapping) = doc.as_mapping_mut() {
    if let Some(servers) = mapping.get_mut(&serde_yaml::Value::String("mcp_servers".to_string())) {
        if let Some(s) = servers.as_mapping_mut() {
            if s.remove(&serde_yaml::Value::String(name.to_string())).is_some() {
                modified = true;
            }
        }
    }
}
if modified {
    std::fs::write(&config_path, serde_yaml::to_string(&doc)?)?;
}
```
More broadly: `serde_yaml` round-tripping drops comments by design. If YAML comment preservation matters, consider switching to `yaml-rust2` or the newer `serde_yml` (a maintained fork) which has better round-tripping.
