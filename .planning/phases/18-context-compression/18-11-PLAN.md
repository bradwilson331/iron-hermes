Close 18-UAT.md high-severity gap "Default compression.protect_first_n=3 is safe for single-tool-pair conversations" by making protect_first_n tool-pair aware.

Live UAT 2026-04-13T23:44 passed Tests 5 & 6 only after lowering `compression.protect_first_n` from the documented default `3` to `2`. Under the default with message shape `[sys, user, asst(tool_use), tool_result]`, the 18-10 post-ship front-straddle guard correctly detects that `asst` sits inside the front-protected region while its `tool_result` sits in the prune range, and pushes `prune_start = max(prune_start, max(tool_result_idx) + 1) = 4`. Because `prune_end = protect_start = 3`, `prune_start (4) >= prune_end (3)` → the range collapses, `pair_atomicity_collapsed_range` warns every turn, zero compression lands, and the agent hits `MAX_COMPRESSION_PASSES=10` with unbounded token growth.

Root cause: `protect_first_n` is a pure index count. It does not consider tool-pair atomicity when selecting its boundary. The documented default (3) is unsafe for the most common real shape.

Fix direction (safety > recovery per 18-10 precedent): a deterministic helper `compute_effective_protect_first_n(messages, configured_first_n, tool_pairs)` runs BEFORE `compute_protect_start` inside `SummarizingEngine::compress`. When an asst(tool_use) pinned in the front-protected region has any tool_result outside it, the helper auto-shrinks `protect_first_n` to `asst_idx` so the whole pair falls into the prunable range. The 18-10 front-straddle guard then finds no conflict and the normal prune path runs.

Purpose: make `protect_first_n = 3` (documented default) safe for single-tool-pair conversations, unblocking UAT Tests 7 (aux fallback) and 8 (memory flush ordering) for live re-run under default config.
