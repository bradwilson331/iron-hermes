## Warnings

### WR-01: sqlite test sets wrong env var — does NOT isolate to tempdir

**File:** `crates/ironhermes-agent/src/memory/factory.rs:112-115`
**Issue:** The test sets `HERMES_HOME` to the tempdir, but `ironhermes_core::constants::get_hermes_home()` (constants.rs:30-37) reads `IRONHERMES_HOME`, not `HERMES_HOME`. Consequently the test does NOT redirect `memory.db` into the tempdir — it creates/opens `~/.ironhermes/memory.db` (or `./.ironhermes/memory.db` if no HOME is set) on the developer's real filesystem every time `cargo test --features memory-sqlite` runs. The tempdir is also dropped immediately after `set_var` (the `_tmp` binding stays alive until end-of-scope, but since nothing actually writes there, the bind is decorative).

Two concrete consequences:
1. **The test is a false-positive regression guard.** It claims to prove `run_gateway` no longer bails on the "not enabled" path, but because it writes to the real home, it could also pass if the factory silently ignored the `provider` field and always built a file store somewhere — the isolation promise in the comment is not delivered.
2. **Side effect on developer machines.** Each test run touches (and possibly creates) a real `~/.ironhermes/memory.db`. This is minor but violates D-13 filesystem invariants used elsewhere in the codebase (see `hub_search_test.rs:249` which explicitly asserts no HERMES_HOME mutation leaks).

Note that the project uses TWO distinct env var names inconsistently across the codebase: `IRONHERMES_HOME` is the real one (constants.rs, prompt_builder tests, cron tests), while `HERMES_HOME` is used by skills/hub code paths (skills_tool.rs, hub/paths.rs) via separate `std::env::var("HERMES_HOME")` reads. The factory reaches `memory.db` through `get_hermes_home()` which only honors `IRONHERMES_HOME`.

**Fix:**
```rust
#[cfg(feature = "memory-sqlite")]
#[test]
fn sqlite_provider_with_feature_returns_ok() {
    let tmp = tempfile::TempDir::new().expect("tempdir");
    let prev = std::env::var("IRONHERMES_HOME").ok();
    // SAFETY: test-only env mutation.
    unsafe { std::env::set_var("IRONHERMES_HOME", tmp.path()); }

    let result = build_memory_provider(&cfg("sqlite"));

    // Restore previous value before asserting so a panic does not leak state.
    unsafe {
        match prev {
            Some(v) => std::env::set_var("IRONHERMES_HOME", v),
            None => std::env::remove_var("IRONHERMES_HOME"),
        }
    }
    assert!(
        result.is_ok(),
        "sqlite provider must build when memory-sqlite feature is enabled, got {:?}",
        result.err(),
    );
    // Optional: also assert the db was created inside the tempdir.
    assert!(tmp.path().join("memory.db").exists());
}
```

Additional consideration: once the test uses the correct env var, it will race against `ironhermes_agent::prompt_builder` tests that also mutate `IRONHERMES_HOME` (prompt_builder.rs:569 notes "Tests that manipulate IRONHERMES_HOME must hold this lock"). You can either (a) reuse that existing `env_lock()` from prompt_builder in the factory test (if made accessible via a shared test helper), or (b) accept the race since `cargo test -p ironhermes-agent` runs lib tests single-threaded by default per cargo's test harness — but call that out in the test's SAFETY comment instead of the current incorrect claim that parallelism is not an issue because the cfg gate is only compiled once.

### WR-02: sqlite test does not restore prior env-var value — leaks process state

**File:** `crates/ironhermes-agent/src/memory/factory.rs:115`
**Issue:** Even accepting the `HERMES_HOME` name (which WR-01 shows is inert for this code path anyway), the test sets the variable once and never restores it. If a future test in the same `ironhermes-agent` lib-test binary reads `HERMES_HOME` (e.g., new skills test, new personality test), it will observe the tempdir path — which by then points to a deleted directory (`_tmp` drops at end of this test). Compare the careful save/restore pattern used elsewhere: `hub_search_test.rs:224-244`, `skills_cmd_test.rs:17-27`, `prompt_builder.rs:590-601`.

This is primarily a hygiene issue today but becomes a genuine correctness risk once WR-01 is addressed (because then a stale `IRONHERMES_HOME` pointing at a deleted tempdir would break everything relying on `get_hermes_home()`).

**Fix:** Apply the save/restore pattern shown in WR-01's fix (also capture `prev` via `std::env::var(...).ok()` and restore in both success and failure paths — a `scopeguard::defer!` or explicit match after the assertion works).

### IN-01: Test helper `cfg(provider)` uses explicit mutable construction despite struct being trivial

**File:** `crates/ironhermes-agent/src/memory/factory.rs:79-83`
**Issue:** The helper writes:
```rust
fn cfg(provider: &str) -> MemoryConfig {
    let mut c = MemoryConfig::default();
    c.provider = provider.to_string();
    c
}
```
This is fine but slightly less idiomatic than struct-update syntax. Also worth noting: `MemoryConfig` currently has only one field (`provider`), so `..Default::default()` is trivially equivalent, but if more fields are added in the future the current pattern still compiles correctly — this is defensively correct. Clippy may flag `field_reassign_with_default`; if so, silence it explicitly rather than "fix" it.

**Fix:** Optional cosmetic:
```rust
fn cfg(provider: &str) -> MemoryConfig {
    MemoryConfig { provider: provider.to_string(), ..MemoryConfig::default() }
}
```

### IN-02: Error-message assertion on `unknown_provider_returns_err_with_message` pins a loose invariant

**File:** `crates/ironhermes-agent/src/memory/factory.rs:94-103`
**Issue:** The test asserts `msg.contains("Unknown memory provider 'totally-unknown'")` and `msg.contains("file")`. The second assertion is weak — the word "file" can appear elsewhere in the message (e.g., in a future error hint). It also does NOT pin the availability list (which varies by feature flags, lines 64-68 in the factory). A future refactor that changes the "Available providers:" prefix or the enumeration order would still pass the test.

Given the factory explicitly enumerates `file{, sqlite}{, grafeo}{, duckdb}` conditionally, consider asserting the prefix:

**Fix:**
```rust
assert!(
    msg.contains("Available providers:"),
    "message must list available providers, got: {msg}",
);
assert!(msg.contains("file"), "file is always available, got: {msg}");
```
Or lock to a specific regex. Not a must-fix; flagging because the comment block in the plan's task 3 specifically required the test to "pin the correct invariant".

### IN-03: Inline comment about "test-only env mutation acceptable since tests run single-threaded" is incorrect

**File:** `crates/ironhermes-agent/src/memory/factory.rs:113-114`
**Issue:** The SAFETY comment reads:
```
// SAFETY: test-only env mutation; acceptable since tests run single-threaded
// per cargo's default for lib tests.
```
Cargo's default is the OPPOSITE: lib tests run in parallel across `#[test]` functions within the same binary (via `libtest`'s threadpool). The only way tests run single-threaded is with `--test-threads=1` or `#[cfg(test)]` using `serial_test`. The comment is factually wrong and could mislead future contributors into assuming env-var mutation is safe here. Combined with WR-01 (wrong var name), the risk is low today, but the justification should be corrected.

**Fix:** Either (a) acquire a shared test lock like `prompt_builder::env_lock()` and update the comment to reflect that, or (b) add `serial_test` as a dev-dep and annotate with `#[serial]`, or (c) accept the race and document: "This test may race with other tests in the same binary that mutate IRONHERMES_HOME. No other ironhermes-agent lib tests currently do so [verify: grep]; if that changes, wrap this test with the shared env lock."
