Close the D-24 hysteresis-across-turns gap surfaced by live UAT on 2026-04-14T16:12–16:17.

Three consecutive user prompts in the same CLI session (`session_id=4c3bda53-0acf-45c3-88dd-b8560a8526f9`) all WARNed for the same crossing:

- 16:12:53 — WARN fires mid-turn at ratio=0.0639 (first crossing — correct).
- 16:16:12 — WARN fires again at turn START at ratio=0.0433 (should be suppressed; `above_threshold` should still be `true`).
- 16:17:04 — WARN fires again at turn START at ratio=0.0434 (same issue).

Root cause (verified, from `crates/ironhermes-cli/src/main.rs:529` + `crates/ironhermes-agent/src/agent_wiring.rs:39`):

```rust
// main.rs — called per user prompt in the interactive REPL
agent = ironhermes_agent::attach_context_engine(agent, config, resolver, session_id, None);

// agent_wiring.rs:39 — inside attach_context_engine
let tracker = Arc::new(PressureTracker::new()); // FRESH Arc every turn
```

Every REPL turn constructs a brand-new `AgentLoop` and a brand-new `PressureTracker`. `PressureTracker` owns the hysteresis map (`above_threshold`, `pending_transient` per session_id), so the state is dropped at end of turn and the next turn starts over. Consequences:

1. WARN re-fires on every turn that lands in the band — hysteresis contract broken.
2. `pending_transient` set during turn N is lost before turn N+1 runs, so the user-facing transient `[CONTEXT PRESSURE HIGH — earlier history may soon be summarized]` system message **never reaches the LLM in the CLI**, even though the drain site (`agent_loop.rs:294`) is wired correctly.
3. `compression_count` on `AgentLoop::new` also resets to 0 each turn — minor secondary issue that distorts the summarizing engine's prior-summary bookkeeping.

Fix direction (minimal, surgical, preserves 18-13):

- Hoist tracker ownership out of `attach_context_engine` into the CLI session scope.
- Extend `attach_context_engine` with an optional caller-provided tracker. When `None`, behavior is unchanged (one-shot path at `main.rs:298` and any tests still work). When `Some`, it is used verbatim — no `PressureTracker::new()` call.
- In the interactive REPL (`main.rs:341..491`), allocate `Arc<PressureTracker>` and a `compression_count` carrier alongside `session_id`, then thread them into `run_agent_turn` for every prompt.
- Add a compression_count carryover path: either return the final `compression_count` from `AgentLoop::run` (or the `AgentRunResult` already exposed) and pass it back in on the next turn via a new `with_compression_count` builder, OR share a single `Arc<AtomicU32>` that `AgentLoop` increments.

Purpose: make PressureTracker's D-24 hysteresis actually observable in a real CLI session, and make the transient user-facing warning actually show up. Close UAT Test 4's remaining edge-case behavior.

Output:
- `agent_wiring.rs`: `attach_context_engine` accepts an optional `Arc<PressureTracker>`; creates a fresh one only when caller passes `None`.
- `main.rs` (interactive REPL): one tracker, one compression_count carrier, constructed once at session start, reused by every `run_agent_turn`. One-shot path (`main.rs:298` area) unchanged.
- `agent_loop.rs`: `with_compression_count(u32) -> Self` builder (or equivalent carryover mechanism) so the summarizing engine's `prior_summary` chain is continuous across turns.
- RED integration test: simulate 3 consecutive turns in the same session with an in-memory tracker; assert WARN counter increments exactly once across 3 crossings that never descend.
- GREEN: same test passes after the fix.
- Live CLI re-run (Task 5): 3 consecutive turns in the band produce exactly ONE `WARN context pressure warning` log line, and turn 2's LLM-bound message vector contains a system message whose body starts with `[CONTEXT PRESSURE HIGH`.
