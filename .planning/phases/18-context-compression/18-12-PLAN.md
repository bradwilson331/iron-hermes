<objective>
Close 18-UAT.md medium-severity gap "After compression, the model recognizes its prior tool call completed and does not retry".

Live UAT 2026-04-13T23:44: the agent re-called `web_read` on every turn for 10 consecutive turns, never emitting a final summary, until `MAX_COMPRESSION_PASSES=10` tripped. Each compression pass correctly replaced the asst(tool_use)+tool_result pair with a `[CONTEXT HISTORY]` summary — but the summary body did not carry enough signal for the model to recognize the task was completed. The model interpreted every turn as a fresh request.

Root cause hypothesis (from UAT):
1. The summary prompt / aux-model output does not preserve tool-call outcome markers ("Agent called X with args Y and received Z — task completed"), OR
2. The summary positioned as a system message competes with the original user message which still reads as an open request.

Fix direction:
- Enrich the `SummarizingEngine` prompt template (lines 434-445) to require structured tool-call outcome lines.
- Prepend a stable sentinel (`COMPLETED_TOOLS_SENTINEL`) inside the produced `[CONTEXT HISTORY]` body such as `"Tool executions already completed; do NOT re-call unless the user explicitly asks again:"` so the model has an unambiguous "done" signal.
- Preserve PRMT-15 `NewSummary = Summarize(OldSummary + NewPrunedBlocks)` exactly — the enrichment is ADDITIVE (stronger directive in the prompt, one new sentinel line in the pinned body), not a replacement of the iterative formula.

Depends on 18-11 because the live UAT re-run verifying this fix requires default-config compression to work first (UAT Tests 7/8 unblocked).

Output:
- `pub const COMPLETED_TOOLS_SENTINEL` in `summarizing_engine.rs`.
- Enriched prompt template in `compress()`.
- When pruned_blocks contain any assistant-with-tool_calls messages, pinned body starts with `COMPLETED_TOOLS_SENTINEL` + tool-name list + model-generated summary, wrapped by the existing `[CONTEXT HISTORY]` sentinel.
- Test double `PromptCapturingSummarizer` that records the outgoing prompt string for assertions.
- RED-then-GREEN unit tests covering both the prompt directive and the pinned body sentinel.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| aux-model output → pinned system message | Untrusted model-generated text is embedded in a system-role message read by the main model on the next turn |
| pruned_blocks tool_calls → summary prompt | Tool-call arguments (potentially user-influenced, e.g. URLs) are interpolated into the summary prompt |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-12-01 | Tampering | completion_header construction | mitigate | Tool names come from structured `ToolCall.function.name` (type `String`, no parsing). Header template uses only the name list, not the args — args are NOT interpolated into the sentinel (they pass through the aux model's summary which has existing HISTORY_SUMMARY_MAX_CHARS truncation). |
