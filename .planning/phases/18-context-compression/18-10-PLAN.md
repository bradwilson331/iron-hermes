The atomic-rollback safety net (commit `883f8ce`) masks the failure by restoring the pre-compression vec, so no Anthropic 400 reaches the user — but no actual compression ever lands. Token usage grows unbounded across turns.

**Root-cause hypothesis (primary arm — boundary arithmetic):** In `summarizing_engine.rs:285-299`, the engine computes `protect_start` once, then calls `apply_adaptive_shift` in a `let _ = ...` pattern that **discards the returned adjusted boundary**. The engine subsequently uses the original `protect_start` as `prune_end` at line 323 — so even when the shift says "expand boundary backward to keep the pair whole," that instruction is thrown away and the prune range still cuts through the pair. Result: the assistant's `tool_calls` message is placed into `pruned_blocks` (summarized into `[CONTEXT HISTORY]`), while the `tool_result` with its `tool_call_id` remains live in `kept_after` → `check_orphan_invariant` fails → rollback.

**Secondary arm (large-body branch):** For tool_result bodies > 500 tokens, `apply_adaptive_shift` rewrites the result in place but returns the unchanged `protect_start`, so the assistant goes into the pruned range while a rewritten-but-still-live tool_result referencing it stays after `prune_end` — same orphan shape.

This plan fixes the boundary arithmetic so pairs are kept atomic across the prune boundary, adds the reproducing test matrix for all 4 UAT token counts, and tightens observability so a future regression surfaces from logs alone. Phase invariants (atomic rollback, pinned `[CONTEXT HISTORY]` segment, aux-model fallback) are preserved.

       // apply_adaptive_shift may have rewritten tool_result content (but not
       // indices).

       for pair in &pairs_after_shift {
           let asst_in = pair.assistant_idx >= prune_start && pair.assistant_idx < prune_end;
           let all_results_in = pair
               .tool_result_indices
               .iter()
               .all(|&i| i >= prune_start && i < prune_end);
           let any_result_in = pair
               .tool_result_indices
               .iter()
               .any(|&i| i >= prune_start && i < prune_end);
           let fully_in = asst_in && all_results_in;
           let fully_out = !asst_in && !any_result_in;
           if fully_in || fully_out {

           // Split detected — expand prune_end to include all results, OR
           // pull prune_end back to exclude the assistant. Prefer pulling back
           // (keeps more live) unless that would drop below protect_first_n.

               // assistant is in prune range but some tool_result is outside.
               // Pull prune_end back to exclude the assistant so both stay live.
               prune_end = prune_end.min(pair.assistant_idx);

                   prune_end = pair.assistant_idx; // pull back to before assistant
