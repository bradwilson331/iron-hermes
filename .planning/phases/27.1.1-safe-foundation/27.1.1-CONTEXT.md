# Phase 27.1.1: Safe Foundation - Context

**Gathered:** 2026-05-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Build a new `hexapod_tcp` Rust tool in the IronHermes repo (`~/code/ironhermes`) that lets an agent issue walk, stop, read_battery, read_distance, and relax_servos commands to the Freenove hexapod robot over TCP — with every unsafe command blocked before any bytes reach the hardware.

The Freenove repo is **read-only reference** for this phase. No Python server code is modified.

</domain>

<decisions>
## Implementation Decisions

### Project Location
- **D-01:** All Phase 1 code is written in `~/code/ironhermes`, not in this repo. Target file: `crates/ironhermes-tools/src/hexapod_tcp.rs`. Skill doc is `skills/hexapod/DESCRIPTION.md` (Phase 3).
- **D-02:** This repo (`Freenove_Big_Hexapod_Robot_Kit_for_Raspberry_Pi`) serves as the protocol reference only. Nothing here is modified.

### Tool Schema
- **D-03:** A single `hexapod_tcp` tool with an `action` enum field — not multiple separate tools. The LLM calls it once per intent. Modeled on the existing `terminal.rs` single-tool pattern.
- **D-04:** Phase 1 action enum values: `walk | stop | read_battery | read_distance | relax_servos`
- **D-05:** Walk action takes `direction: forward | backward | left | right` and `speed: integer (2–10)`. All other Phase 1 actions take no extra parameters.

### Wire Protocol Translation
- **D-06:** Walk direction maps to CMD_MOVE with ±25 displacement:
  - `forward`  → `CMD_MOVE#1#0#25#<speed>#0\n`
  - `backward` → `CMD_MOVE#1#0#-25#<speed>#0\n`
  - `left`     → `CMD_MOVE#1#-25#0#<speed>#0\n`
  - `right`    → `CMD_MOVE#1#25#0#<speed>#0\n`
  - Mode=1 (tripod gait), angle=0. Speed is direct pass-through, clamped to 2–10.
- **D-07:** Stop maps to `CMD_MOVE#0#0#0#0#0\n` (mode=0 halts motion, returns to neutral).
- **D-08:** Relax servos maps to `CMD_RELAX\n`.
- **D-09:** Read battery maps to `CMD_POWER\n`; tool reads response `CMD_POWER#<v1>#<v2>\n` and formats as `"Battery: <v1>V / <v2>V (<OK|LOW>)"`. Low threshold: v1 < 5.5V or v2 < 6.0V.
- **D-10:** Read distance maps to `CMD_SONIC\n`; tool reads response `CMD_SONIC#<dist>\n` and formats as `"Distance: <dist> cm"`.

### TCP Connection
- **D-11:** One fresh TCP connection per tool call (stateless). No persistent socket. Avoids the hexapod server's TCP stream-splitting fragility.
- **D-12:** Robot IP from `HEXAPOD_IP` env var, port 5002. Missing env var returns `Ok("Error: HEXAPOD_IP env var not set — cannot connect to robot")`.

### Session End (HXP-TOOL-06)
- **D-13:** Add a default no-op `fn on_session_end(&self) {}` to the `Tool` trait in `ironhermes-tools/src/registry.rs`. All existing tools get the no-op default; no breaking changes.
- **D-14:** `HexapodTcpTool` overrides `on_session_end` to fire-and-forget `tokio::spawn(send_stop(...))` — sends `CMD_MOVE#0#0#0#0#0\n` then `CMD_RELAX\n` to halt and relax servos.
- **D-15:** The REPL/CLI shutdown path calls `on_session_end()` on all registered tools after the agent loop returns.

### Error Feedback Design
- **D-16:** Blocked commands (CMD_CALIBRATION, CMD_SERVOPOWER, CMD_LED_MOD modes 2–5) return `Ok("Action '<name>' is blocked — not permitted via hexapod_tcp. Never send this command.")`. Returns `Ok()` so the LLM sees it inline and learns not to retry.
- **D-17:** Robot unreachable → `Ok("Error: cannot connect to robot at <ip>:5002 — is HEXAPOD_IP set and the robot powered on?")`.
- **D-18:** Read timeout (3s) → `Ok("Error: read timed out after 3s waiting for robot response")`.
- **D-19:** Fire-and-forget commands (walk, stop, relax) that send successfully return `Ok("OK")` — no response expected from the robot for those commands.

### Allowlist
- **D-20:** The allowlist is a compile-time `match` on the `action` field — exhaustive Rust enum or `match` arms. No runtime config. Unrecognized actions return the blocked-command error string (D-16).

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `~/code/ironhermes/crates/ironhermes-tools/src/terminal.rs`: Complete reference implementation — copy the struct shape, `Tool` trait impl, `anyhow::Result<String>` return convention, `tokio::time::timeout` usage, and test structure. `HexapodTcpTool` will be approximately 150–200 lines following this pattern.
- `~/code/ironhermes/crates/ironhermes-core/src/` `ToolSchema` — already used for schema registration; same type as terminal.rs uses.

### Established Patterns
- **One connection per call**: `terminal.rs` spawns a fresh `tokio::process::Command` per execute — `hexapod_tcp.rs` opens a fresh `tokio::net::TcpStream` per execute. Same stateless pattern.
- **Timeout with `tokio::time::timeout`**: terminal.rs wraps the async operation in `timeout(Duration::from_secs(...), fut)`. Use identical pattern for the 3-second read timeout on sensor commands.
- **`anyhow::anyhow!` for errors**: All tool errors use `anyhow::anyhow!("message")`. Error messages become the tool result string the LLM sees.
- **`serde_json::json!` for schema**: ToolSchema constructed with `json!({...})` inline in `schema()` — matches terminal.rs exactly.

### Integration Points
- `~/code/ironhermes/crates/ironhermes-tools/src/lib.rs` — register `HexapodTcpTool` here, same as other tools
- `~/code/ironhermes/crates/ironhermes-tools/src/registry.rs` `Tool` trait — add `on_session_end` default method (D-13)
- REPL/CLI shutdown path in `~/code/ironhermes` — needs to call `on_session_end()` on registered tools; check `crates/ironhermes-core/src/commands/` for the shutdown sequence

### Known Fragilities to Work Around (from CONCERNS.md / PROJECT.md)
- TCP recv may split commands across packets — send one command per connection (D-11 already handles this)
- `condition_monitor` has no malformed-command handling — always send complete, valid command strings with `\n` terminator
- No response framing on command channel — use a fixed read buffer and the 3-second timeout (D-18) rather than reading until EOF

</code_context>

<specifics>
## Specific Ideas

- The tool description string (for the LLM) should include the full list of Phase 1 actions and the blocked commands, so Gemma 4 knows what's available and what's forbidden without needing to call the skill doc.
- The `prerequisites()` method on `HexapodTcpTool` should declare `HEXAPOD_IP` as a required env var — this integrates with IronHermes' setup wizard if it checks prerequisites at startup.
- Error strings are `Ok(...)` not `Err(...)` for all non-programming errors (D-16, D-17, D-18) so the LLM sees them inline in the tool result rather than as a system error.
