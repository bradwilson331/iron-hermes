# Phase 27.1.1 Gap-02: set_toolset_config Startup Enforcement Summary

Enforce `tools.toolsets` from `config.yaml` at session start across all production entry points; populate `PromptBuilder.active_toolsets` so the system-prompt tool catalog matches the API schemas.

## What Was Done

### Root Cause Closed

`ToolRegistry.toolset_config` was `None` at session startup because no production code called `registry.set_toolset_config(...)`. The filter in `get_definitions()`:

```rust
.filter(|t| {
    self.toolset_config
        .as_ref()
        .map(|cfg| cfg.is_toolset_enabled(t.toolset()))
        .unwrap_or(true)   // None ⇒ every toolset passes — was always hit
})
```

was a no-op. `web: { enabled: false }` in `config.yaml` had zero effect. `/toolset list` showed the right state; the LLM's actual schema did not match it.

Simultaneously, `PromptBuilder.active_toolsets` was `HashSet::new()` at every production call site — the "Phase 20 wires real toolset state" TODO from 2024 was never closed.

### Merge Semantics

**`with_default_toolsets_merged()` on `ToolsConfig`:**
- For every name in `ALL_TOOLSETS` (8 total): if absent from the map, insert `ToolsetEntry { enabled: true }`.
- If present (enabled or disabled), leave untouched. `web: { enabled: false }` stays false.
- Idempotent: applying twice produces the same result.
- Back-compat preserved: old configs that predate a toolset keep full access; explicit disables are respected.

**Single source-of-truth for toolset names:** `constants::ALL_TOOLSETS` (`["memory", "session", "agent", "skills", "robotics", "web", "code", "browser"]`). `toolset_cmd.rs::KNOWN_TOOLSETS` and `toolset_session.rs::members_map()` hold separate copies for historical reasons; both are documented as pointing at `ALL_TOOLSETS`.

### Wiring Points

| Entry point | set_toolset_config called | set_active_toolsets called |
|---|---|---|
| `build_app_runtime_bundle` | Yes — after all `register_*` calls, before Arc wrap | n/a (registry level) |
| TUI `event_loop.rs` | Yes — on locally-built registry before Arc wrap | Yes — from `merged_tools` |
| `run_chat` (main.rs) | Via bundle | Yes — from `bundle.merged_tools` |
| `run_single` (main.rs) | Via bundle | Yes — from `bundle.merged_tools` |
| `iron_hermes_ui` state.rs | Via bundle | Yes — from `bundle.merged_tools` |

### RegistryToolsetSession baseline

All 3 `RegistryToolsetSession::new(...)` call sites (run_chat, run_gateway in main.rs; TUI event_loop.rs) now pass `bundle.merged_tools.clone()` instead of raw `config.tools.clone()`. This ensures `/toolset enable|disable` mutates from the same baseline the registry started with.

## Subagent / RPC Sub-Registry Isolation Finding

**Confirmed isolated — no changes required.**

- **RPC sub-registry** (`build_rpc_registry`): Hand-rolled safe subset for execute_code sandbox. Never receives `set_toolset_config` — no call in the function. The sandbox's restricted tool list is maintained by explicit registration, not toolset filtering.

- **Subagent registry** (`delegate_task.rs`): Zero `set_toolset_config` calls. Subagent tool subsets are governed independently by `subagent.default_toolsets` config, which controls which tool *names* a subagent receives. The main registry's `toolset_config` does not bleed into subagent or RPC sub-registries.
