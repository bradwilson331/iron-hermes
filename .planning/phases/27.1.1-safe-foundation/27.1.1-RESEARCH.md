## Summary

Phase 27.1.1 adds a single new Rust tool (`hexapod_tcp`) to `crates/ironhermes-tools/` and wires
two small changes: a default `on_session_end` method on the `Tool` trait, and a shutdown hook in
the REPL/CLI paths. All twenty implementation decisions (D-01..D-20) are locked; this document
provides the execution-grounding details the planner needs to write task-level plans.

The canonical model for `hexapod_tcp.rs` is `terminal.rs` — struct shape, `Tool` trait impl,
`anyhow::Result<String>` return, `tokio::time::timeout` pattern, and `#[tokio::test]` inline
tests all transfer verbatim. The TCP layer is the only genuinely new element, and it is the
simplest possible form: open, write, optionally read with timeout, close.

**Primary recommendation:** Write `hexapod_tcp.rs` first by cloning the terminal.rs structure.
Then splice `on_session_end` into the `Tool` trait (two lines, no breaking changes). Then wire
shutdown in `run_chat` and `run_single`. Register last. Tests inline.

## Tool Trait Extension Plan

[VERIFIED: `crates/ironhermes-tools/src/registry.rs` lines 24-71]

### Current Tool trait surface (lines 24-70)

```rust
#[async_trait]
pub trait Tool: Send + Sync {
    fn name(&self) -> &str;
    fn toolset(&self) -> &str;
    fn description(&self) -> &str;
    fn schema(&self) -> ToolSchema;
    fn is_available(&self) -> bool { ... }    // defaulted
    fn prerequisites(&self) -> Vec<Prerequisite> { vec![] }  // defaulted
    fn redact_args(&self, raw: &serde_json::Value) -> serde_json::Value { raw.clone() } // defaulted
    async fn execute(&self, args: serde_json::Value) -> anyhow::Result<String>;
}
```

### D-13 Splice: Where to Insert `on_session_end`

Insert immediately after `redact_args` (line 70) and before the `async fn execute` line.

**Exact insertion point:** `registry.rs` line 70, before `async fn execute`.

```rust
    // D-13 (Phase 27.1.1): called on REPL/CLI session end. Default is a
    // synchronous no-op — all 18+ existing tools get this for free.
    // Override: fire-and-forget tokio::spawn (sync trait method per D-14).
    fn on_session_end(&self) {}

    async fn execute(&self, args: serde_json::Value) -> anyhow::Result<String>;
```

**Why sync, not async:** D-14 says fire-and-forget `tokio::spawn` — the override internally
spawns an async task. The trait method itself stays `fn` (not `async fn`) so the trait remains
object-safe and no existing implementors need to add `async`. The tokio runtime is already
running at shutdown time (CLI shutdown is async context), so `tokio::spawn` works from a sync
trait method.

**Zero breaking changes:** All 18+ existing `Tool` implementors get the no-op default. Rust
traits with defaulted methods do not require existing implementors to add anything.

### HexapodTcpTool override (D-14)

```rust
fn on_session_end(&self) {
    // Fire-and-forget safety halt: send stop + relax to prevent robot
    // running indefinitely after the agent session closes.
    tokio::spawn(async move {
        if let Ok(ip) = std::env::var("HEXAPOD_IP") {
            let addr = format!("{}:5002", ip);
            let _ = send_stop_and_relax(&addr).await;
        }
    });
}
```

Where `send_stop_and_relax` is a private async fn that opens a TCP connection, sends
`CMD_MOVE#0#0#0#0#0\n` then `CMD_RELAX\n`, and swallows all errors (fire-and-forget).

## TCP / Timeout Pattern

[VERIFIED: terminal.rs lines 195-231; tokio in Cargo.toml line 25]

### Existing TcpStream usage in codebase

There is no existing `tokio::net::TcpStream` usage in the tools or agent crates.
[VERIFIED: grep of all crates — only `TcpListener` in `iron_hermes_ui/src/main.rs:46` for the
HTTP UI server; `UnixListener`/`UnixStream` in the exec sandbox]. This is the first TCP client.

`tokio::net::TcpStream` is available via the `tokio = { workspace = true }` dependency that
`ironhermes-tools/Cargo.toml` already declares. No new dependency needed.

### Fire-and-forget commands (walk, stop, relax_servos) — D-19

```rust
// Open connection, send command bytes, close. No read.
async fn send_command(addr: &str, cmd: &str) -> anyhow::Result<()> {
    use tokio::io::AsyncWriteExt;
    let mut stream = tokio::net::TcpStream::connect(addr).await?;
    stream.write_all(cmd.as_bytes()).await?;
    stream.flush().await?;
    Ok(())
}
```

Return `Ok("OK")` on success (D-19). On connect failure, return `Ok("Error: cannot connect...")` (D-17).

### Sensor commands (read_battery, read_distance) — D-18, 3s timeout

```rust
// Open connection, send command, read response with 3s timeout.
async fn send_and_read(addr: &str, cmd: &str) -> anyhow::Result<String> {
    use tokio::io::{AsyncReadExt, AsyncWriteExt};
    let mut stream = tokio::net::TcpStream::connect(addr).await?;
    stream.write_all(cmd.as_bytes()).await?;
    stream.flush().await?;
    let mut buf = vec![0u8; 256];
    let n = stream.read(&mut buf).await?;
    Ok(String::from_utf8_lossy(&buf[..n]).trim().to_string())
}

// At call site in execute():
let read_fut = send_and_read(&addr, "CMD_POWER\n");
match tokio::time::timeout(Duration::from_secs(3), read_fut).await {
    Err(_) => return Ok("Error: read timed out after 3s waiting for robot response".to_string()),
    Ok(Err(e)) => return Ok(format!("Error: {}", e)),
    Ok(Ok(raw)) => { /* parse raw response */ }
}
```

### TCP recv split risk (from CONTEXT known fragilities)

D-11 (one connection per call) completely eliminates the split-packet risk because the server
sends exactly one response per command on a fresh connection. The 256-byte read buffer is
sufficient for all sensor responses (CMD_POWER and CMD_SONIC responses are under 40 bytes).

## Blocked-Command Surface Pattern

[VERIFIED: CONTEXT D-16, D-20; registry.rs execute() pattern]

### How blocked commands are surfaced

The project's convention is `Ok(String)` for all non-programming errors (confirmed by terminal.rs
returning `Ok(...)` for all user-visible results). Blocked commands follow D-16:

```rust
// In execute(), the exhaustive match:
let action = args["action"].as_str().unwrap_or("");
match action {
    "walk" => { /* D-06 */ }
    "stop" => { /* D-07 */ }
    "read_battery" => { /* D-09 */ }
    "read_distance" => { /* D-10 */ }
    "relax_servos" => { /* D-08 */ }
    // Blocked or unknown:
    other => return Ok(format!(
        "Action '{}' is blocked — not permitted via hexapod_tcp. Never send this command.",
        other
    )),
}
```

This is a compile-time exhaustive Rust `match` (D-20). The `other` catch-all handles both
explicitly blocked commands (CMD_CALIBRATION, CMD_SERVOPOWER, CMD_LED_MOD modes 2-5 that a
confused LLM might construct as action names) and any unknown string.

**Consistency check:** The `Ok(...)` return for blocked commands is consistent with the project
pattern. `Err(...)` would cause the agent loop to surface a system-level error rather than
an inline tool result, which would NOT give the LLM the "never retry" guidance.

## Pitfalls

### Pitfall 1: Trait `on_session_end` must be sync, not async

**What goes wrong:** If `on_session_end` is declared `async fn`, the trait is no longer
object-safe (`dyn Tool` breaks). Existing code uses `Box<dyn Tool>` extensively.

**Why it happens:** `async fn` in traits requires `async_trait` macro, which works for
object-safe traits, but the tokio::spawn fire-and-forget pattern doesn't need async at the
trait level — the spawned task is the async boundary.

**How to avoid:** Declare `fn on_session_end(&self) {}` (sync). The override internally calls
`tokio::spawn(...)`. No `async fn` in the trait method.

**Warning signs:** Compiler error `the trait Tool cannot be made into an object` or
`the size for values of type Self cannot be known at compilation time`.

### Pitfall 2: `tokio::spawn` in `on_session_end` requires `'static` bounds

**What goes wrong:** If `on_session_end` captures `&self` references inside the spawned future,
the compiler rejects it because `Box<dyn Tool>` is not `'static` in the captured context.

**Why it happens:** `tokio::spawn` requires `'static` lifetime on the future.

**How to avoid:** Read `HEXAPOD_IP` inside the `async move` block (via `std::env::var`, which
needs no captures from `self`). The struct has no fields, so there is nothing to capture.

### Pitfall 3: TCP recv may return partial response

**What goes wrong:** `stream.read(&mut buf)` may return fewer bytes than the complete response
line for CMD_POWER or CMD_SONIC.

**Why it happens:** TCP is a stream protocol; `read()` returns whatever is buffered.

**How to avoid:** D-11 (one connection per call) makes this unlikely on a local network, and
the 3-second timeout (D-18) handles the failure case. For extra safety, use
`BufReader::lines()` or read until `\n` is in the buffer. The protocol doc confirms responses
end with `\n`, so reading until newline is safe and correct.

**Recommended:** Use `tokio::io::BufReader` + `read_line()`:
```rust
use tokio::io::BufReader;
let reader = BufReader::new(stream);
let mut line = String::new();
reader.read_line(&mut line).await?;
```
This is more robust than fixed-size `read()` and stays well within the 3s timeout.

### Pitfall 4: Speed parameter must be validated, not trusted

**What goes wrong:** LLM passes `speed: 1` or `speed: 11`; robot receives out-of-range value.

**Why it happens:** JSON schema `minimum`/`maximum` are advisory to the LLM, not enforced by
the tool.

**How to avoid:** Clamp speed in `execute()` before building the wire string:
```rust
let speed = args["speed"].as_u64().unwrap_or(5).clamp(2, 10);
```

### Pitfall 5: `CMD_RELAX` is a toggle, not a one-shot relax

**What goes wrong:** According to the protocol doc (Section 10): "Each time the command is
received, it toggles the servo enable/disable state." The doc even notes this is "not
recommended; please use `CMD_SERVOPOWER` instead."

**Why it matters:** If the agent calls `relax_servos` twice, the second call re-enables servos
(robot is no longer relaxed). For Phase 1 this is acceptable behavior — the phase goal is
to send the command, not to guarantee idempotency.

**How to avoid (Phase 1):** Document in code that CMD_RELAX is a toggle. D-08 is locked; do
not change the mapping. Add a comment in the tool.

**Note:** `CMD_SERVOPOWER#0\n` would be more reliable, but it is in the blocked list (D-16).
Phase 27.1.2+ can revisit if needed.

### Pitfall 6: `call_session_end_hooks` must not deadlock on the registry RwLock

**What goes wrong:** If `call_session_end_hooks` is called while holding a write lock on the
registry, calling `tool.on_session_end()` from inside the lock causes a deadlock if `on_session_end`
tries to acquire a read lock.

**Why it happens:** `tokio::spawn` is fire-and-forget and does NOT acquire the registry lock,
so in practice this is not a problem for the hexapod implementation. But the pattern matters.

**How to avoid:** Call `call_session_end_hooks` only after releasing any registry write lock.
In `run_chat` the registry is accessed via `Arc<RwLock<ToolRegistry>>`; call
`registry.read().await.call_session_end_hooks()` — read lock is released after the call.

### Pitfall 7: Battery response framing — two voltages, not one

**What goes wrong:** Parsing `CMD_POWER#7.2#8.1\n` assuming one voltage field returns wrong data.

**Why it happens:** The protocol returns TWO voltage readings (load battery + Pi battery).

**How to avoid:** Split on `#` and index fields correctly:
```rust
// Response: "CMD_POWER#<v1>#<v2>\n"
let parts: Vec<&str> = raw.trim().split('#').collect();
// parts[0] = "CMD_POWER", parts[1] = v1, parts[2] = v2
let v1: f32 = parts.get(1).unwrap_or(&"0").parse().unwrap_or(0.0);
let v2: f32 = parts.get(2).unwrap_or(&"0").parse().unwrap_or(0.0);
let status = if v1 < 5.5 || v2 < 6.0 { "LOW" } else { "OK" };
```

Thresholds [VERIFIED against adc.py]: v1 (load/channel 0) < 5.5V = LOW; v2 (Pi/channel 4) < 6.0V = LOW.
