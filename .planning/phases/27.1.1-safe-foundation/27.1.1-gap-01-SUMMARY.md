# Phase 27.1.1 Plan gap-01: Canonical Tool Registration Entry Point — Summary

Eliminate systemic tool-registration drift by collapsing 7 hand-rolled registration sites into one canonical entry point. Adding a tool to `register_defaults_except` now makes it visible across all production binaries automatically.

## Design Pattern Chosen: Option A (Skip-List)

`pub fn register_defaults_except(&mut self, skip: &[&str])` in `crates/ironhermes-tools/src/registry.rs`.

**Rationale:** Simplest approach with zero new types, zero new traits, and minimal conceptual overhead. The pattern matches existing Rust idioms for "register everything except X". Production paths that need a process-registry-wired terminal call:

```rust
registry.register_defaults_except(&["terminal"]);
registry.register_terminal_tool_with_process_registry(process_registry);
```

`register_defaults()` is kept as a thin wrapper over `register_defaults_except(&[])` for callers that want the plain terminal.

## rpc_registry Decision: Keep Hand-Rolled Safe-Subset

`build_rpc_registry` (in `app_runtime_factory.rs`) and the rpc_registry block in `event_loop.rs` are intentionally kept hand-rolled with an explicit SAFETY doc-comment:

> SAFETY: this sub-registry intentionally does NOT inherit from register_defaults().
> execute_code runs agent-generated code in a sandboxed context and must NOT gain
> new capabilities silently when tools are added to the default set.
> - terminal: excluded — nested shell execution from within execute_code is not permitted
> - execute_code: excluded — recursive execute_code is not permitted
> - hexapod_tcp: excluded — hexapod hardware control must not be accessible from within
>   the execute_code sandbox; hardware commands require explicit top-level tool calls

Adding a new tool to `register_defaults_except` does NOT silently add it to the RPC sandbox. That's the intended security property.

## Gateway Production Path Finding

The gateway was shown as "NOT fixed" in the plan's entry-point map, but investigation revealed it was already fixed via `build_app_runtime_bundle` inheritance:

- `run_gateway` (in `main.rs:2441`) calls `build_app_runtime_bundle` which calls `build_registry_with_process_registry`
- `build_registry_with_process_registry` now delegates to `register_defaults_except`
- `GatewayRunner::new` receives `runtime_bundle.registry.clone()` — the fully-wired registry
- The gateway's handler.rs:1157/1220 and runner.rs:1958/2071 are ALL test-only (empty `ToolRegistry::new()` in test scaffolding, not production paths)
- No separate gateway fix needed — it inherits the fix automatically

This confirms the cascade: `register_defaults_except` → `build_registry_with_process_registry` → `build_app_runtime_bundle` → CLI REPL, CLI batch, iron_hermes_ui, AND gateway all fixed at once.

## set_toolset_config Latent Bug: Resolved in gap-02

The `set_toolset_config` bug (config.yaml's `tools.toolsets` section had no effect at session start because `set_toolset_config` was never called from production startup) has been fixed in `27.1.1-gap-02`.

See `27.1.1-gap-02-SUMMARY.md` for full details. The fix:
- Adds `with_default_toolsets_merged()` to `ToolsConfig` (back-compat: absent entries → enabled=true)
- Calls `registry.set_toolset_config(Some(merged_tools))` in `build_app_runtime_bundle` after all registrations
- Exposes `merged_tools` in `AppRuntimeBundle` for use by `RegistryToolsetSession::new` call sites
- Wires `PromptBuilder.set_active_toolsets(merged_tools.enabled_toolset_names())` at all 4 production construction sites
- Confirmed RPC sub-registry and subagent registry are unaffected (isolated by design)
