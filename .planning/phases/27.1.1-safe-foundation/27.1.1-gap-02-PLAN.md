<objective>
Make the `tools.toolsets` section of `~/.ironhermes/config.yaml` actually take effect at session start.

Root cause (discovered during Phase 27.1.1 UAT, documented in `27.1.1-gap-01-PLAN.md`):
`ToolRegistry.toolset_config` is `None` at session startup because nothing calls
`registry.set_toolset_config(...)` from production code — only the runtime
`/toolset enable|disable` commands (via `RegistryToolsetSession`) ever set it. So the
toolset filter in `ToolRegistry::get_definitions()`:

```rust
.filter(|t| {
    self.toolset_config
        .as_ref()
        .map(|cfg| cfg.is_toolset_enabled(t.toolset()))
        .unwrap_or(true)          // None ⇒ every toolset passes
})
```

is a no-op until the user runs a `/toolset` command. Consequences:
1. `web: { enabled: false }` in `config.yaml` does NOT hide web tools at startup.
2. `/toolset list` (reads `config.yaml` directly via `toolset_cmd.rs::load_tools_config`)
   shows the "right" state, but the LLM's actual tool schema doesn't match it. The UI lies.
3. It "works by accident" — everything is on regardless of config — which is why nobody
   noticed, and why the `49e35eec` fix (adding `"robotics"` to `DEFAULT_TOOLSETS`) was a
   no-op for the live UAT.

There is also a parallel third tracker: `PromptBuilder.active_toolsets` (drives the
system-prompt tool *catalog text*, separate from the API tool schemas). It's
`HashSet::new()` by default with a code comment "Phase 20 wires real toolset state" —
never done. So even where the schema is right, the prompt catalog can be empty.

Output: a back-compat-preserving merge helper on `ToolsConfig`, a single
`set_toolset_config` call wired into `build_app_runtime_bundle`, the same merged config
handed to `RegistryToolsetSession`, `PromptBuilder.active_toolsets` populated at all
production construction sites, and tests proving the config now actually filters.

Non-goal: redesigning the toolset model, the `/toolset` command UX, or the subagent
toolset subsets. This plan makes the existing config field do what it already claims to do.
</objective>
