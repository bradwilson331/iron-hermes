# Phase 27.1.1: Code Review Report

**Reviewed:** 2026-05-11
**Depth:** standard
**Files Reviewed:** 11
**Status:** issues_found

## Summary

This phase adds the hexapod TCP tool, session-end safety hooks (stop + relax), canonical tool registration via `register_defaults_except`, and startup enforcement of `tools.toolsets` config. The overall structure is sound and the design decisions (stateless tool, fire-and-forget session-end hook, allowlist before I/O, `with_default_toolsets_merged` back-compat) are sensible.

Three blockers were found:

1. `ToolsConfig::default()` is **missing "robotics"** in its hard-coded init loop, so a brand-new install with no `config.yaml` silently excludes the hexapod tool until `with_default_toolsets_merged()` is called — which is only called on production entry points, not on `ToolsConfig::default()` standalone (e.g. tests that construct `Config::default()` and then call `get_definitions()` without going through the factory will see the tool filtered out even with `HEXAPOD_IP` set).

2. `iron_hermes_ui/src/server/state.rs::AppState::init()` builds an `AppRuntimeBundle` through the factory (which calls `set_toolset_config(Some(merged_tools))`) but **never calls `call_session_end_hooks()`** at any web-UI session teardown site. The hexapod safety-halt guarantee introduced by D-14/D-15 does not cover the web-UI entry point.

3. `event_loop.rs::build_app_deps` calls `std::fs::create_dir_all` for all 8 subdirectories **then immediately calls `ensure_home_dirs()`**, which iterates the same 8 subdirectories again. This is not a security issue, but the doubled work is symptomatic of a merge conflict leftover and will create `hermes_home` subdirs under the **wrong home** if `IRONHERMES_HOME` changes between the two calls (not possible in practice, but the structural redundancy is a defect).

## Critical Issues

### CR-01: `ToolsConfig::default()` excludes "robotics" — hexapod tool invisible without factory path

**File:** `crates/ironhermes-core/src/config.rs:107-119`
**Issue:** `ToolsConfig::default()` hard-codes only `["memory", "session", "agent", "skills"]` as enabled toolsets and `["web", "code", "browser"]` as disabled. The "robotics" toolset is not present in the map at all. `is_toolset_enabled("robotics")` returns `false` (the `unwrap_or(false)` fallback at line 127) for any `ToolsConfig::default()` instance that hasn't been through `with_default_toolsets_merged()`.

The production factory (`build_app_runtime_bundle`) calls `with_default_toolsets_merged()` immediately, so the live product is unaffected. However:
- `constants::DEFAULT_TOOLSETS` explicitly includes "robotics" (line 39 of `constants.rs`) as an intended default, and the comment there says the toolset "is enabled by default."
- The `Config::default()` path (used pervasively in tests and in `Config::load().unwrap_or_default()`) yields a `ToolsConfig` where robotics is absent, and `is_toolset_enabled("robotics")` returns `false`. Any test or code path that constructs a registry from `Config::default()` without going through the factory will silently hide `hexapod_tcp`.
- The Phase 27.1.1-gap-02 test `test_toolset_config_robotics_default_enabled` explicitly removes "robotics" to simulate a pre-gap-02 config and relies on `with_default_toolsets_merged()` restoring it — but this means `ToolsConfig::default()` itself is inconsistent with `DEFAULT_TOOLSETS`.

**Fix:** Add "robotics" to the enabled set in `ToolsConfig::default()`, consistent with `DEFAULT_TOOLSETS`:
```rust
impl Default for ToolsConfig {
    fn default() -> Self {
        let mut toolsets = HashMap::new();
        for name in ["memory", "session", "agent", "skills", "robotics"] {
            toolsets.insert(name.to_string(), ToolsetEntry { enabled: true });
        }
        for name in ["web", "code", "browser"] {
            toolsets.insert(name.to_string(), ToolsetEntry { enabled: false });
        }
        Self {
            toolsets,
            skip_prompts: vec![],
            disabled: vec![],
        }
    }
}
```

### CR-02: Web-UI entry point (`iron_hermes_ui`) never calls `call_session_end_hooks()` — hexapod safety halt has no effect on the web path

**File:** `crates/iron_hermes_ui/src/server/state.rs:49-110`
**Issue:** `AppState::init()` wires the factory correctly and `build_app_runtime_bundle` populates the registry with `HexapodTcpTool` (including `on_session_end`). But `AppState` has no shutdown path that calls `registry.read().await.call_session_end_hooks()`. The CLI (`run_chat`, `run_single`) and the ratatui TUI (`run_with_deps`) both call this at session teardown. The web-UI is the only production entry point that does not. If the robot is walking when the web-UI process terminates or the session ends, the safety stop is never sent.

**Fix:** Add a teardown method (or `Drop` impl) on `AppState` that fires the hooks, and call it at the end of the request handler or server shutdown:
```rust
pub async fn teardown(&self) {
    self.runtime_bundle.registry.read().await.call_session_end_hooks();
}
```
Call `app_state.teardown().await` at the end of a web-session lifecycle or in the server's shutdown handler. Alternatively, `impl Drop for AppState` with a `tokio::spawn` equivalent to how `HexapodTcpTool::on_session_end` works.

### CR-03: Duplicate subdirectory creation in `build_app_deps` — double `create_dir_all` for all 8 dirs before `ensure_home_dirs` is called

**File:** `crates/ironhermes-cli/src/tui_rata/event_loop.rs:150-162`
**Issue:** Lines 150-161 loop over the same 8 subdirectories with `std::fs::create_dir_all`, and then line 162 calls `ensure_home_dirs(&hermes_home)`, which loops over them again. This is a clear copy-paste duplication introduced when the phase landed (one block was the original code, the other was added to mirror the factory). Beyond wasted work, there is a subtle correctness risk: if `ensure_home_dirs` is ever updated to add or remove a subdirectory, the in-line block above it will silently diverge.

**Fix:** Remove the inline `create_dir_all` loop at lines 150-161 and rely solely on the `ensure_home_dirs` call at line 162:
```rust
// Remove lines 150-161 (the manual loop)
ensure_home_dirs(&hermes_home)?;
```
The `ensure_home_dirs` helper at line 598 already covers all 8 subdirectories.

## Warnings

### WR-01: `CMD_RELAX` toggle behavior not communicated to the LLM tool description

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:21-25` and `41-45`
**Issue:** The code comments at line 22-23 correctly note that `CMD_RELAX` is a toggle (each call flips servo-enabled state). However, the `DESCRIPTION` constant (used as the LLM-visible tool description, lines 41-45) only describes `relax_servos` as a safe action. It does not mention the toggle semantics. An LLM calling `relax_servos` twice in quick succession to "make sure servos are off" will instead re-enable them — a physically dangerous outcome for a robotics tool. The tool schema also has no way for the LLM to query the current servo state before calling relax.

**Fix:** Add the toggle warning to the LLM-visible description:
```rust
const DESCRIPTION: &str = "Control the Freenove hexapod robot over TCP. \
    Phase 1 actions: walk (with direction and speed), stop, read_battery, \
    read_distance, relax_servos. \
    NOTE: relax_servos is a TOGGLE — calling it twice re-enables servos. \
    Always call stop before relax_servos to halt motion first. \
    BLOCKED (do not attempt): calibration, servo_power, led_mode 2-5 — \
    these return a block error and are never permitted via this tool.";
```

### WR-02: `state_store.lock().unwrap()` in async context can panic and poison the mutex

**File:** `crates/iron_hermes_ui/src/server/state.rs:115, 150, 228`
**Issue:** `self.state_store.lock().unwrap()` is called at three sites inside async methods (`ensure_web_session`, `run_web_turn`, and `build_messages_for_turn`). Using `std::sync::Mutex` with `.unwrap()` inside an async context means: (a) if a previous holder panicked while holding the lock, every subsequent call panics too (poisoned mutex); (b) holding a `std::sync::MutexGuard` across an `.await` point would deadlock — which is avoided here by scoping carefully, but the `build_messages_for_turn` usage at line 228 holds the lock across the `store.add_message` call inside a block that also calls `store.get_messages`, while the enclosing function is async and lines 222-224 have an `.await` above the lock acquisition. This is safe as written because the lock is not held across the `.await`, but the pattern is fragile.

**Fix:** Use `tokio::sync::Mutex` for `state_store` in `AppState`, or use `expect` with a descriptive message, and add a lock-poisoning recovery:
```rust
let mut store = self.state_store.lock()
    .unwrap_or_else(|e| e.into_inner()); // recover from poisoned lock
```

### WR-03: `HexapodTcpTool::on_session_end` spawns a task but the runtime may not be present

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:310-320`
**Issue:** `on_session_end` calls `tokio::spawn(async move { ... })`. This works correctly in the CLI and TUI code paths where a tokio runtime is always active. However, `on_session_end` is called from `call_session_end_hooks()` which is a synchronous method. If `call_session_end_hooks()` is ever called from a context without an active tokio runtime (e.g., during a test that doesn't use `#[tokio::test]`, or from a sync drop site), `tokio::spawn` will panic with "no current runtime". The existing tests avoid this because they use `#[tokio::test]`, but the trait contract for `on_session_end` (which is defined as `fn on_session_end(&self) {}` — a sync fn) does not state that a tokio runtime is required.

**Fix:** Document the runtime requirement in the trait method comment, and add a guard:
```rust
fn on_session_end(&self) {
    if let Ok(handle) = tokio::runtime::Handle::try_current() {
        handle.spawn(async move {
            if let Ok(ip) = env::var("HEXAPOD_IP") {
                let addr = format!("{ip}:5002");
                send_stop_and_relax(&addr).await;
            }
        });
    }
    // If no runtime is active, skip the safety halt — best-effort only.
}
```

### WR-04: `with_default_toolsets_merged()` inserts absent entries as `enabled: true`, including future toolsets that may be high-blast-radius

**File:** `crates/ironhermes-core/src/config.rs:142-149`
**Issue:** The `with_default_toolsets_merged()` method inserts `enabled: true` for every name in `ALL_TOOLSETS` that is not already in the map. The comment justifies this as "back-compat: upgrading users don't silently lose access to new toolsets." However, `ALL_TOOLSETS` currently contains "browser", "web", and "code" — all of which are marked as disabled-by-default in the code and in comments because they require external keys or have high blast radius. If a user's old config predates these toolsets (no explicit entry), `with_default_toolsets_merged()` will enable them on upgrade. This contradicts the stated intent in `DEFAULT_TOOLSETS` and in the `ToolsConfig::default()` impl, which explicitly disabled those three.

The back-compat default should be `enabled: false` for toolsets that `ToolsConfig::default()` explicitly disables, and `enabled: true` only for those that are enabled by default.

**Fix:** Use `DEFAULT_TOOLSETS` as the source of truth for "absent = enabled" semantics, not `ALL_TOOLSETS`:
```rust
pub fn with_default_toolsets_merged(mut self) -> Self {
    for &name in crate::constants::ALL_TOOLSETS {
        let default_enabled = crate::constants::DEFAULT_TOOLSETS.contains(&name);
        self.toolsets
            .entry(name.to_string())
            .or_insert(ToolsetEntry { enabled: default_enabled });
    }
    self
}
```
This means absent "browser"/"web"/"code" entries default to `enabled: false` (as intended), while absent "memory"/"session"/"agent"/"skills"/"robotics" default to `enabled: true`.

### IN-02: `HEXAPOD_IP` port is hardcoded in two separate places with no shared constant

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:237, 316`
**Issue:** The port `5002` is assembled via `format!("{ip}:5002")` at line 237 in `execute()` and again at line 316 in `on_session_end()`. There is no shared `HEXAPOD_PORT` constant. If the port changes, both sites must be updated manually. The wire-protocol constants at the top of the file define all other protocol values; the port should join them.

**Fix:**
```rust
/// TCP port for the Freenove hexapod server (Code/Server/server.py default). Per D-11.
pub(crate) const HEXAPOD_PORT: u16 = 5002;

// Then at call sites:
let addr = format!("{ip}:{HEXAPOD_PORT}");
```
