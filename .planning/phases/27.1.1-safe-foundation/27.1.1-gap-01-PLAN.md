<objective>
Eliminate the systemic tool-registration drift bug discovered during Phase 27.1.1 UAT.

Root cause: `crates/ironhermes-tools/src/registry.rs::register_defaults()` is the
canonical default-registration method (Plan 04 of Phase 27.1.1 added HexapodTcpTool
to it), but the live production code path bypasses it entirely. All three production
entries (`run_chat` / `run_single` in `main.rs`, the iron_hermes_ui server) call
`build_app_runtime_bundle`, which calls a LOCAL `build_registry_with_process_registry`
in `crates/ironhermes-agent/src/app_runtime_factory.rs:142` that hand-rolls the tool
list (Terminal+ReadFile+WriteFile+PatchFile+SearchFiles+WebSearch+WebRead) and never
calls `register_defaults()`. Result: any tool added to `register_defaults()` is
invisible at runtime until someone manually duplicates the registration into the
hand-rolled list.

This bit Plan 04 — UAT failure logged at 2026-05-11 ~00:23 EDT, surfaced via gap fix
commit 49e35eec (constants.rs DEFAULT_TOOLSETS — itself a misdiagnosis: the toolset
filter doesn't even fire at startup because nothing calls `set_toolset_config` from
production code) and then 89XX (app_runtime_factory.rs HexapodTcpTool registration —
the actual unblock).

Known duplicate registration sites (5 total):
1. `crates/ironhermes-tools/src/registry.rs::register_defaults()` — canonical
2. `crates/ironhermes-agent/src/app_runtime_factory.rs::build_registry_with_process_registry()` line 142 — production CLI/TUI/UI path (HOT-PATCHED at 27.1.1-89XX, still drifts for next tool)
3. `crates/ironhermes-agent/src/app_runtime_factory.rs::build_rpc_registry()` line 162 — RPC sub-registry (intentionally restricted; review case-by-case)
4. `crates/ironhermes-cli/src/main.rs:2592 (build_registry_with_process_registry)` — looks like dead code; verify and delete if unused
5. `crates/ironhermes-cli/src/tui_rata/event_loop.rs:291, :296` — TUI path with separate `registry` and `rpc_registry` lists

Bonus latent bug to consider:
- `set_toolset_config(Some(config.tools))` is NEVER called from production startup code
  (grep returned only test code and the runtime `/toolset enable/disable` mutation path).
  This means the `tools.toolsets.<X>.enabled` section of `~/.ironhermes/config.yaml`
  has NO effect at session start — the toolset filter is `None` so all toolsets pass.
  The user-facing `/toolset list` command shows enabled-state from config, but the
  schema gate doesn't enforce it. This is a separate gap; this plan need not fix it
  unless the refactor naturally subsumes it.

Output: One canonical entry point, all production paths delegate to it, hand-rolled
duplicates removed (or replaced with `register_defaults_except(&[...])` pattern), and
a regression test that fails if a tool is added to `register_defaults` but bypassed
by `build_app_runtime_bundle`.
</objective>
