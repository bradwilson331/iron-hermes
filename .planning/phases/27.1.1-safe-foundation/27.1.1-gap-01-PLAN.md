<objective>
Eliminate the systemic tool-registration drift bug discovered during Phase 27.1.1 UAT.

Root cause: `crates/ironhermes-tools/src/registry.rs::register_defaults()` is the
canonical default-registration method (Plan 04 of Phase 27.1.1 added HexapodTcpTool
to it), but the live production code paths bypass it entirely. There is NO single
source of truth for "which tools an IronHermes session exposes" — every entry point
hand-rolls its own list.

This bit Plan 04 — UAT failure logged at 2026-05-11 ~00:23 EDT. Misdiagnosis chain:
- `49e35eec` fix(27.1.1): added "robotics" to constants.rs `DEFAULT_TOOLSETS` — turned
  out IRRELEVANT: `set_toolset_config` is never called from production startup, so the
  toolset filter is `None` and never fires at session start (latent bug, see below).
- `16ea01ee` fix(27.1.1): added `HexapodTcpTool` to `app_runtime_factory.rs::build_registry_with_process_registry`
  — fixed the CLI REPL (`run_chat`), CLI batch (`run_single -e`), and `iron_hermes_ui` web server,
  because all three call `build_app_runtime_bundle`. Verified live: agent read battery
  `CMD_POWER#7.94#7.47 → "Battery: 7.94V / 7.47V (OK)"` from the rebuilt debug binary.
- TUI fix (this plan's commit): added `HexapodTcpTool` to `tui_rata/event_loop.rs` —
  the ratatui `chat` TUI builds its OWN registry at event_loop.rs:229 and DISCARDS
  `build_app_runtime_bundle`'s result (`let _runtime_bundle = build_app_runtime_bundle(...)`),
  so the factory registration never reaches that path.

## Latent bug surfaced during this investigation (separate, may be subsumed)

`set_toolset_config(Some(config.tools))` is NEVER called from production startup code
(grep found only test code and the runtime `/toolset enable/disable` mutation path).
This means `~/.ironhermes/config.yaml`'s `tools.toolsets.<X>.enabled` section has NO
effect at session start — the registry's `toolset_config` stays `None`, so the toolset
filter at `registry.rs:280` (`unwrap_or(true)`) lets every toolset through. The
user-facing `/toolset list` command displays enabled-state from config, but the schema
gate doesn't enforce it. (This is also why the `49e35eec` constants.rs change was a
no-op for the live UAT.) The refactor in this plan may naturally subsume this if the
canonical entry point also wires `set_toolset_config`; if not, open a SECOND gap plan.

Output: One canonical entry point, all production paths delegate to it, hand-rolled
duplicates removed (or replaced with `register_defaults_except(&[...])` pattern), and
a regression test that fails if a tool is added to `register_defaults` but bypassed
by `build_app_runtime_bundle`.
