## Summary

Phase 22.4.2 replaces 30 `Phase 22.4.1 stub:` arms in `tui_rata/commands.rs::invoke_handler` with a single one-line delegation to `core::handlers::dispatch`, then pushes real handler bodies into `core::commands::handlers.rs` for the ~15 currently-stubbed commands. The plumbing work (Plan 00) adds 8 `Option<Arc<...>>` fields to `CommandContext` and 10 new fields to `App` / `AppDeps`. The actual subsystems are already built and in the workspace — the research task is confirming their exact type signatures so Plan 00 compiles on the first try.

**Critical shape finding:** `StateStore` is NOT wrapped in `Arc<RwLock<StateStore>>` anywhere in the codebase. Every existing call site (AgentLoop, gateway handler) wraps it in `Arc<std::sync::Mutex<StateStore>>` because `StateStore` is synchronous rusqlite and its `&mut self` methods require exclusive access. The CONTEXT.md D-04 proposal of `Arc<RwLock<StateStore>>` must be corrected to `Arc<Mutex<StateStore>>` (std::sync, not tokio) to match the existing pattern.

**Critical shape finding:** `PersonalityOverlay` does NOT exist as a named type. The actual type is `PersonalityRegistry` in `ironhermes_agent::personality`. It is not wrapped in any Arc or Mutex currently — it is a pure value type. There is no `PersonalityOverlay` in the workspace.

**Critical shape finding:** `ProviderResolver` derives `Clone` and is used as a plain value (cloned) at all construction sites — never wrapped in Arc. The existing `event_loop.rs` already builds a `ProviderResolver` (line 140) and clones it into `AgentSubagentRunner` (line 229) but does NOT store it in `AppDeps`. `App` does not currently hold a `ProviderResolver`.

**Primary recommendation:** Plan 00 must reconcile three type mismatches before writing any of Plans 01-04: `StateStore` wrapper is `Arc<Mutex<StateStore>>` (std::sync), `PersonalityOverlay` does not exist (use `PersonalityRegistry`), and `ProviderResolver` needs to be added to `AppDeps` (it is currently local to `build_app_deps` and discarded).

### P0.8 — history (App.history)

**Current field on App:** `pub history: Vec<ChatMessage>` — plain `Vec`, NOT wrapped in Arc.
**Access pattern:** Direct field access `app.history` in all current code.

**CONTEXT.md D-04 proposes:** `history: Option<Arc<RwLock<Vec<ChatMessage>>>>`

**CRITICAL MISMATCH:** `App.history` is a plain `Vec<ChatMessage>`, not an `Arc<RwLock<Vec<ChatMessage>>>`. Upgrading `App.history` to `Arc<RwLock<...>>` would be a significant refactor — all existing direct field accesses (`self.history.push(...)`, `app.history.clone()` in spawn_turn, etc.) would need updating.

**Recommended approach:** Do NOT wrap `App.history` in Arc<RwLock> in Plan 00. Instead, for commands that need to read history (`/history` display, `/retry`, `/undo`), pass a **snapshot clone** when building CommandContext:
```rust
// in build_command_context(app):
let history_snap = Arc::new(std::sync::RwLock::new(app.history.clone()));
ctx = ctx.with_history(history_snap);
```
For `/retry` and `/undo` that mutate history, the mutation happens in the **post-router hook** (tui_rata's `handle_session_control`) which has `&mut App` access — core handler reads the snapshot for display, post-router hook applies the mutation to `app.history` directly.

**Recommended ctx field type:** `history: Option<Arc<std::sync::RwLock<Vec<ChatMessage>>>>` — populated with a clone snapshot at `build_command_context` time; mutations apply post-router on App directly.

## Async Bridge Verification (P2)

**Established pattern** (VERIFIED in handlers.rs):
```rust
// Source: handlers.rs:129-131 (cmd_stop), :516-519 (cmd_models_refresh)
tokio::task::block_in_place(|| {
    tokio::runtime::Handle::current().block_on(async { ... })
});
```

**Runtime guarantee:** `App` is constructed under `#[tokio::main]` multi-threaded runtime (main.rs). `block_in_place` is safe on multi-threaded runtime. All handler dispatch occurs inside `block_in_place` in `dispatch_slash_blocking` (app.rs:356-360) which itself calls `block_in_place`. Double-nesting `block_in_place` is allowed on tokio multi-threaded runtime.

**Per-handler async bridge analysis:**

| Handler | Async Need | Bridge Required | Notes |
|---------|-----------|----------------|-------|
| `cmd_sessions` | `StateStore.list_sessions()` — sync | `spawn_blocking` (can't hold std Mutex across await) | Use `spawn_blocking` or call sync inside `block_in_place` |
| `cmd_resume` | `StateStore.get_messages()` — sync | Same as above | |
| `cmd_save` | `StateStore.export_session()` — sync | Same as above | |
| `cmd_history` | Read `ctx.history` snapshot — sync | NONE — RwLock::read() is sync | |
| `cmd_title` | `StateStore.update_session_title(&mut self)` — sync | `spawn_blocking` | `&mut self` needs exclusive Mutex lock |
| `cmd_memory` | `MemoryManager.to_memory_entries()` — async | `block_in_place + block_on` | |
| `cmd_mcp` | `McpManager.connected_server_names()` — sync | NONE | |
| `cmd_compress` | `ContextEngine.compress()` — async | `block_in_place + block_on` | |
| `cmd_personality` | `PersonalityRegistry.get()` — sync | NONE | |
| `cmd_model` | `ProviderResolver.resolve()` — sync | NONE | |
| `cmd_provider` | `ProviderResolver.resolve_for_main()` — sync | NONE | |
| `cmd_retry` / `cmd_undo` | Read/write `ctx.history` snapshot | NONE for read; post-router hook for write | |
| `cmd_rollback` | Read `ctx.history`, truncate | Post-router hook handles mutation | |
| `cmd_background` / `cmd_btw` / `cmd_queue` | Spawn new AgentLoop — async | Post-router hook (has App access for spawn) | Core returns Output("Background task queued."); post-router spawns |

**StateStore-specific pattern (new — different from cmd_stop):**
```rust
// Correct pattern for sync Mutex<StateStore> inside sync handler:
let store = match &ctx.state_store {
    Some(s) => s.clone(),
    None => return CommandResult::Output("Session storage not configured.".to_string()),
};
// Inside block_in_place (already in effect from dispatch_slash_blocking):
let sessions = {
    let guard = store.lock().expect("StateStore lock poisoned");
    guard.list_sessions(None, 20, 0)?
};
```
The outer `block_in_place` from `dispatch_slash_blocking` covers sync Mutex operations — no additional `spawn_blocking` needed since `StateStore` methods complete quickly (no network I/O).

**Stream-returning methods:** None of the required subsystem methods return streams or iterators. All return single values (sync) or single Futures (async). No different bridge pattern needed.

## Open Questions / Risks

### OQ-1: `Arc<Mutex<StateStore>>` vs `Arc<RwLock<StateStore>>` in CommandContext

**Risk:** CONTEXT.md D-04 says `state_store: Option<Arc<RwLock<StateStore>>>` but every existing call site uses `Arc<std::sync::Mutex<StateStore>>`. If Plan 00 implements RwLock, it creates a second incompatible wrapper type alongside the existing Mutex wrapper in AgentLoop and gateway.

**Recommendation:** Use `Arc<std::sync::Mutex<StateStore>>` to match all existing call sites. This is a type correction from CONTEXT.md, not a new design choice. The planner should lock this as `Arc<std::sync::Mutex<ironhermes_state::StateStore>>`.

### OQ-2: `PersonalityOverlay` type name

**Risk:** `PersonalityOverlay` does not exist. Using this name in Plan 00 causes a compile error.

**Recommendation:** Use `PersonalityRegistry` (`ironhermes_agent::personality::PersonalityRegistry`). Since it has no `&mut self` methods, use `Arc<PersonalityRegistry>` (no RwLock needed). Field name in CommandContext and App can stay `personality_overlay` for naming consistency with CONTEXT.md, but the type must be `Arc<PersonalityRegistry>`.

### OQ-3: `ProviderResolver` not in AppDeps

**Risk:** Plan 00 adds `provider_resolver` to CommandContext, but `build_command_context` reads it from `app.provider_resolver` — which doesn't exist yet.

**Recommendation:** Plan 00 must simultaneously: (a) add `resolver: ProviderResolver` to `AppDeps`, (b) add `pub resolver: ProviderResolver` to `App`, (c) propagate from `build_app_deps` (where it's already built at line 140) into `AppDeps` and then `App::new`. This is a 3-site change all in Plan 00.

### OQ-4: `App.history` type upgrade

**Risk:** CONTEXT.md D-04 proposes `history: Option<Arc<RwLock<Vec<ChatMessage>>>>` in CommandContext, but `App.history` is `Vec<ChatMessage>` (plain, not Arc-wrapped). Upgrading `App.history` itself to `Arc<RwLock<...>>` requires changing all current access patterns throughout `app.rs` (push, clear, clone in spawn_turn, etc.).

**Recommendation:** Do NOT upgrade `App.history` to Arc<RwLock>. Instead, in `build_command_context`, create a snapshot clone wrapped in Arc<RwLock> for CommandContext use. Mutations (for `/retry`, `/undo`, `/rollback`) happen in the post-router hook via `&mut app.history` directly. This avoids a large refactor and is semantically correct — handlers read a snapshot, post-router hook applies mutations.

### OQ-5: `/rollback` split between Plan 03 and Plan 04

**Risk:** `/rollback` is assigned to both Plan 03 (compressor rollback) and Plan 04 (session-history rollback). Research confirms ContextEngine has NO public rollback API. Plan 03's ownership of "compressor rollback" cannot be implemented as specified.

**Recommendation:** Consolidate `/rollback` entirely in Plan 04. The implementation is: read `ctx.history` snapshot to show current length, then in post-router hook, truncate `app.history` by removing the last N (user, assistant) pairs. No ContextEngine API change needed. Plan 03 should remove `/rollback` from its scope; Plan 04 owns it entirely.

### OQ-6: `McpManager` does not implement `McpReloader` trait

**Risk:** `build_command_context` already calls `ctx.with_mcp_reloader(mgr.clone())` when `app.mcp_manager` is Some. The `with_mcp_reloader` builder expects `Arc<dyn McpReloader>`. If `McpManager` implements `McpReloader`, this works. If not, the cast would fail.

**Verification needed:** Confirm `impl McpReloader for McpManager` exists in ironhermes-mcp. The existing working code at `commands.rs:177-180` implies it does (otherwise it wouldn't compile today). [ASSUMED based on existing compiling code — medium risk]

### OQ-7: `ironhermes-state` as dev-dependency of `ironhermes-core`

**Risk:** `handlers_state_store.rs` is in `crates/ironhermes-core/tests/` but `ironhermes-state` may not be a dev-dependency of `ironhermes-core`. Adding it creates a potential circular dependency if `ironhermes-state` depends on `ironhermes-core`.

**Verification:** `ironhermes-state` imports `ironhermes_core` (lib.rs line 11: `use ironhermes_core::{get_hermes_home, ChatMessage, Role}`). Adding `ironhermes-state` as a dev-dep of `ironhermes-core` would create a circular dependency. **This is a BLOCKER for placing `handlers_state_store.rs` in `ironhermes-core/tests/`.**

**Alternative:** Place `handlers_state_store.rs` in `crates/ironhermes-cli/tests/` (which already depends on both crates) instead. OR create a separate integration test crate. The planner must resolve this before Plan 01 is written.

### OQ-8: INV-22.4-33 test will fail immediately after Plan 00 unless pre-emptively updated

**Risk:** INV-22.4-33 currently asserts that stub markers EXIST for the 28 named commands (`stub_count >= expected_arms.len()`). After Plan 00 collapses `invoke_handler` to one line, ALL stub markers disappear — making INV-22.4-33 fail. But INV-22.4-33's inversion (to "markers must be ABSENT") is specified for Phase 22.4.2. Plan 00 must invert INV-22.4-33 AS PART OF the Plan 00 commit, not in a separate plan.

**Recommendation:** Plan 00 tasks must include inverting INV-22.4-33 in `invariants_22_4.rs` as part of the same commit that collapses `invoke_handler`.
