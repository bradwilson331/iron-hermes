## Summary

Reviewed the three files involved in the gateway active-skills wiring fix (phase 10.1). The core fix -- ensuring the shared `Arc<Mutex<Vec<SkillRecord>>>` flows from `main.rs` through `GatewayRunner` into `GatewayMessageHandler` and then into `AgentLoop` -- is correctly implemented. The `set_active_skills()` setter pattern properly overwrites the default standalone Arc created in `GatewayMessageHandler::new()`, and regression tests verify Arc identity. The cron path in `execute_cron_job` also correctly wires active skills from the job's skill list.

Three warnings relate to error handling consistency and a potential whitelist bypass. Two informational items note silent-failure patterns with empty API keys.

## Warnings

### WR-01: Mutex lock panics on poison in production code path

**File:** `crates/ironhermes-gateway/src/runner.rs:679`
**Issue:** `active_skills.lock().unwrap()` in `execute_cron_job` will panic if the mutex is poisoned. This is production code that runs in the cron tick task. A panic here would crash the entire tick task, preventing all future cron jobs from running. Other production code in this file (e.g., `fast_forward_backlog` at line 534) correctly uses `.map_err()` for graceful error handling.
**Fix:**
```rust
let mut skills_guard = active_skills
    .lock()
    .map_err(|e| anyhow::anyhow!("active_skills lock poisoned: {}", e))?;
```

### WR-02: Whitelist bypass when sender_id fails to parse

**File:** `crates/ironhermes-gateway/src/runner.rs:265`
**Issue:** `event.sender_id.parse::<i64>().unwrap_or(0)` converts an unparseable sender ID to `0`. If `0` is present in the whitelist configuration, any message with a malformed sender ID would pass the whitelist check. While unlikely in practice (Telegram uses positive integer IDs), the `unwrap_or(0)` sentinel value is not a safe default for a security-sensitive check.
**Fix:**
```rust
let sender_id: i64 = match event.sender_id.parse() {
    Ok(id) => id,
    Err(_) => {
        warn!(sender_id = %event.sender_id, "Unparseable sender_id, ignoring");
        continue;
    }
};
```

### WR-03: Retry loop discards the final attempt's actual error

**File:** `crates/ironhermes-gateway/src/handler.rs:40`
**Issue:** `with_rate_limit_retry` runs up to 3 attempts for 429 errors but if all 3 fail due to rate limiting, the actual error from the third attempt is discarded and replaced with a generic `"Bot is being rate limited, please wait"` bail message. This loses diagnostic information (e.g., the specific Telegram API error, retry-after header value) that would help debug persistent rate limiting.
**Fix:**
```rust
// Track the last error and return it instead of a generic message
let mut last_err = None;
for attempt in 0..3u64 {
    match f().await {
        Ok(v) => return Ok(v),
        Err(e) => {
            let err_str = e.to_string();
            if err_str.contains("429") || err_str.contains("Too Many Requests") {
                let wait = (attempt + 1) * 2;
                warn!("Telegram rate limit hit, retrying in {}s (attempt {})", wait, attempt + 1);
                last_err = Some(e);
                tokio::time::sleep(std::time::Duration::from_secs(wait)).await;
                continue;
            }
            return Err(e);
        }
    }
}
Err(last_err.unwrap_or_else(|| anyhow::anyhow!("Bot is being rate limited, please wait")))
```

## Info

### IN-01: Silent empty API key in gateway path

**File:** `crates/ironhermes-gateway/src/handler.rs:353`
**Issue:** `self.config.resolve_api_key().unwrap_or_default()` silently produces an empty string when no API key is configured. This leads to a confusing downstream API error instead of a clear "no API key" message. The same pattern appears in `crates/ironhermes-gateway/src/runner.rs:647` and `crates/ironhermes-cli/src/main.rs:459`.
**Fix:** Consider logging a warning when falling back to an empty API key, or failing early with a descriptive error message as `build_client()` does at main.rs:548 with `.context("No API key configured...")`.

### IN-02: Disconnected Arc if set_active_skills is never called

**File:** `crates/ironhermes-gateway/src/handler.rs:72`
**Issue:** `GatewayMessageHandler::new()` creates a standalone `Arc<Mutex<Vec<SkillRecord>>>` at line 72. If `set_active_skills()` is never called, the handler silently uses this disconnected Arc, meaning skills activated via `SkillsTool` would never reach `AgentLoop` enforcement. The current call chain in `runner.rs:151-153` correctly calls the setter, but the API design relies on callers remembering to wire it. This is documented and tested (the regression test at line 489), so this is informational only.
**Fix:** Consider a builder pattern or a constructor that requires the shared Arc, making the wiring compile-time enforced rather than runtime-dependent.
