# Phase 10.1: Gateway active_skills Wiring Fix - Research

**Researched:** 2026-04-10
**Domain:** Rust Arc sharing, gateway handler/runner wiring, skills enforcement
**Confidence:** HIGH

## Summary

This phase is a pure wiring fix — no new types, no new semantics, no enforcement logic changes. The bug is precisely located: `GatewayMessageHandler::new()` creates its own fresh `Arc<Mutex<Vec<SkillRecord>>>` at line 72 of handler.rs, independent of the one passed to `SkillsTool` in main.rs. When a skill is activated via the skills tool, it updates the SkillsTool's Arc, but the handler's Arc (passed to AgentLoop) is a separate allocation — so enforcement never sees the activation.

The fix follows an already-established pattern in the codebase: `set_memory_store()`, `set_hook_registry()`, and `set_skill_registry()` all use the exact same "optional field + setter" approach on both `GatewayRunner` and `GatewayMessageHandler`. The `active_skills` field needs the same treatment. Four touch points, one line each: `GatewayMessageHandler` setter, `GatewayRunner` field + setter, `runner.start()` propagation, and `main.rs run_gateway()` call site.

The enforcement logic in `agent_loop.rs:339-387` is already correct and tested from Phase 07.5 — it just needs the right Arc to read from. The substantial work in this phase is writing a regression test that covers both Arc identity (ptr_eq) and behavioral enforcement.

**Primary recommendation:** Add `set_active_skills()` setter mirroring `set_skill_registry()` on both `GatewayRunner` and `GatewayMessageHandler`, wire it through `runner.start()`, and call it in `main.rs run_gateway()` using the existing `active_skills` Arc from line 437.

### The Established Setter Pattern (template for this fix)

All optional shared state in `GatewayRunner` and `GatewayMessageHandler` follows this pattern: [VERIFIED: runner.rs lines 22-65, handler.rs lines 44-90]

```

  set_field(&mut self, v: Arc<T>) { self.field = Some(v); }

The `active_skills` field differs slightly — it's `Arc<Mutex<Vec<T>>>` rather than `Option<Arc<T>>` on `GatewayMessageHandler` (because a default empty Arc is always valid). The setter just overwrites the default.

### Pattern: Arc Identity Test (Arc::ptr_eq)

The D-07 test requirement calls for proving the handler and SkillsTool share the same allocation:

```rust
// Source: Rust stdlib documentation [ASSUMED — standard Rust pattern]
let shared: Arc<Mutex<Vec<SkillRecord>>> = Arc::new(Mutex::new(Vec::new()));
let handler_arc = handler.active_skills(); // getter (if added) or test field access
assert!(Arc::ptr_eq(&shared, &handler_arc), "handler must share the same Arc as SkillsTool");
```

`Arc::ptr_eq` compares the raw data pointer — two independently-created Arcs with the same content would fail this check, which is exactly what catches the bug.

### Pattern: Behavioral Enforcement Test

The existing test pattern in `agent_loop.rs` (hooks_ordering_tests) provides the template: [VERIFIED: agent_loop.rs lines 461-539]

```rust
// Pattern: mock tool + mock LlmClient + populated active_skills Arc
// 1. Create Arc<Mutex<Vec<SkillRecord>>> with a skill that has allowed_tools = Some(vec!["skills"])
// 2. Create AgentLoop with that Arc via with_active_skills()
// 3. Call execute_tool_call() with a blocked tool name
// 4. Assert result contains the enforcement error message substring
```

The enforcement error message format (from agent_loop.rs:366-373):
```
"Tool '{name}' is not permitted by the active skill set. Allowed tools: [{list}]. ..."
```

### Anti-Patterns to Avoid

- **Creating a new Arc in the test and passing it separately to handler and SkillsTool:** This recreates the bug. The test must pass the SAME Arc instance to both.
- **Testing enforcement without first verifying Arc identity:** If the identity test fails, the behavioral test may pass for the wrong reason (e.g., if the test constructs the scenario differently than production does).
- **Modifying `GatewayMessageHandler::new()` to remove the default empty Arc:** The default must remain for backward compat (D-04). The setter is additive.

## Common Pitfalls

### Pitfall 1: Active Skills Type Mismatch — `std::sync::Mutex` vs `tokio::sync::Mutex`

**What goes wrong:** `AgentLoop` uses `std::sync::Mutex<Vec<SkillRecord>>` (not `tokio::sync::Mutex`). If a developer adds imports from `tokio::sync` while editing gateway files (which use both sync types heavily), they might accidentally use the wrong Mutex type.
**Why it happens:** `runner.rs` imports `tokio::sync::{mpsc, RwLock, Semaphore}` alongside `std::sync::{Arc, Mutex}`. The active_skills field uses `std::sync::Mutex`.
**How to avoid:** The type signature is already correct in handler.rs line 51 — copy it exactly.
**Warning signs:** Compiler error "expected std::sync::Mutex, found tokio::sync::Mutex" on the `with_active_skills()` call.

### Pitfall 2: Forgetting to Add SkillRecord Import to runner.rs

**What goes wrong:** `runner.rs` does not currently import `SkillRecord` from `ironhermes_core` — it only imports `SkillRegistry`. Adding `active_skills: Option<Arc<Mutex<Vec<SkillRecord>>>>` requires adding `SkillRecord` to the `use ironhermes_core::` import.
**Why it happens:** `GatewayRunner` previously had no direct use of `SkillRecord` (only `SkillRegistry`). [VERIFIED: runner.rs line 7]
**How to avoid:** Update the `use ironhermes_core::{...}` import line in runner.rs to include `SkillRecord`.
**Warning signs:** `error[E0412]: cannot find type 'SkillRecord' in this scope`

### Pitfall 3: main.rs Call Order — active_skills Must Be Cloned Before Passing to register_skills_tool

**What goes wrong:** Line 439 in main.rs calls `registry.register_skills_tool(skill_registry.clone(), active_skills)` — note it moves `active_skills` (no `.clone()`). If the `runner.set_active_skills()` call happens after line 439, `active_skills` has been moved and is no longer available.
**Why it happens:** The current code at line 439 moves `active_skills` into `register_skills_tool()`. The fix (D-03) requires calling `runner.set_active_skills(active_skills.clone())` — the `.clone()` must happen before OR the line 439 call must also be updated to clone.
**How to avoid:** Change line 439 to `registry.register_skills_tool(skill_registry.clone(), active_skills.clone())`, then add `runner.set_active_skills(active_skills)` (or another clone) at the runner setup block.
**Warning signs:** `error[E0382]: use of moved value: 'active_skills'`

### Pitfall 4: Test Requires a Getter or Field Access for Arc Identity Check

**What goes wrong:** The D-07 test needs to call `Arc::ptr_eq` on the handler's internal `active_skills`. Since it's a private field, the test (which is in `handler.rs #[cfg(test)]`) can access it directly as `handler.active_skills` (same module). No getter required.
**Why it happens:** Module-level test access is different from external access. Tests in `#[cfg(test)] mod tests` inside `handler.rs` have access to private fields of the module's types.
**How to avoid:** Put the test in `handler.rs #[cfg(test)] mod tests` per D-08. No getter needed.

## Open Questions

1. **Does `GatewayMessageHandler::new()` need an `active_skills` getter for the Arc identity test?**
   - What we know: Tests in `#[cfg(test)] mod tests` inside `handler.rs` can access private fields directly (same module). No getter needed for the test itself.
   - What's unclear: Whether a public `active_skills()` getter would be useful for external callers (e.g., runner.rs verifying the wire-up in its own tests).
   - Recommendation: Add getter only if needed by runner.rs tests; otherwise access the field directly from handler.rs tests (Claude's Discretion per CONTEXT.md).

2. **How to construct the handler + SkillsTool for the behavioral enforcement test?**
   - What we know: The test needs to call `execute_tool_call()` (or equivalent) with an active skill that restricts tools. The agent_loop enforcement is already unit-tested in agent_loop.rs. The handler test only needs to prove the Arc reaches AgentLoop correctly.
   - What's unclear: Whether to test via `AgentLoop::execute_tool_call()` directly (using the handler's Arc) or whether a higher-level integration path is needed.
   - Recommendation: Direct `AgentLoop` test using the shared Arc is simpler and sufficient — it proves the Arc flows from handler to loop correctly without needing a full Telegram message dispatch.
