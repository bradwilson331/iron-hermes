# Phase 32.3: delegation-agent-runaway - Context

**Gathered:** 2026-05-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 32.3 closes a runaway-child bug in delegation and ships the "shrike service" — the operator-facing termination surface that guarantees a delegated subagent can be reliably stopped and cleaned up from any UI.

**The bug.** Live repro (gemma4 via ollama, captured in the TUI chat scroll): a research subagent (`sub_20667cb71808`) wrote its output files at 02:52 AM and finished its work, but the `SubagentRegistry::list()` snapshot kept reporting it as `Active` for 24,150s ≈ **6.7 hours**. The agent loop returned, the output landed on disk, but `SubagentRegistry::unregister` never fired — the registry entry leaked. From the user's perspective the agent "never terminated."

**Five capability areas in scope:**

1. **RAII registry guard** — `SubagentRegistry::register` returns a `RegistrationGuard` whose `Drop` impl calls `unregister`. Cleanup is automatic on natural completion, timeout, cancel, panic, and any early-return path. This fixes the 6.7-hour ghost class of bugs at the type level.

2. **Stale-child detection (heartbeat)** — use the existing `AgentLoop::ActivityTracker` (landed in Phase 32.1) to expose `last_activity` per subagent. Surface in tree view + log a `tracing::warn!` when `now - last_activity > stale_warn_seconds` (default 120s, per-call overridable). Hard-kill ceiling reuses `child_timeout_seconds`.

3. **Shrike service** — termination/cleanup operations exposed as a coherent subcommand family on `/agents`, with the same four operations available across all three surfaces (tui_rata, iron_hermes_ui web, gateway/Telegram):
   - `/agents kill <id>` — hard-kill (interrupt + drop JoinHandle + force unregister)
   - `/agents interrupt <id>` — soft cancel (signals cancel_token, child finalizes current iteration)
   - `/agents prune` — sweep registry entries with `now - last_activity > stale_warn_seconds`, deregister them (one-shot manual override that fixes already-leaked ghosts without restart)
   - `/agents status <id>` — diagnostic view: uptime, last_activity, turns_used, tool calls, transcript path, parent_id chain

4. **Hard-kill guarantee on timeout/cancel** — when `tokio::time::timeout` fires or cancel_token cancels in `delegate_task.rs`, the contract is: future dropped, semaphore released, transcript closed, registry deregistered — all within ~1s. RAII guard (area 1) plus an explicit `JoinHandle::abort()` on the spawned child task close the remaining gap.

5. **`/help` audit (audit-only this phase, fix in follow-up)** — inventory `/help` output in all three surfaces (tui_rata, iron_hermes_ui, gateway/Telegram), confirm the `/agents` family is discoverable and accurately documented, and file gaps as follow-up todos in `32.3-SUMMARY.md`. Doc fixes themselves are out of scope for 32.3 to keep this phase focused on the runaway-fix.

**Does NOT deliver:**

- TUI overlay with pause-controls per subtree (deferred from Phase 32.2 — pause vs kill is a separate UX)
- Per-branch cost/token rollup in the tree view (still deferred from 32.2)
- Slash-help documentation rewrite (audit only — actual rewrite is a follow-up doc-polish phase)
- Memory-side runaway hooks (memory writes during runaway children — deferred to Phase 34b+ memory work)
- Cron-runner runaway protection (cron has its own runner crate from Phase 32.1; runaway protection there is a separate concern even though it shares `AgentLoop`)

</domain>

<decisions>
## Implementation Decisions

### Registry Cleanup Contract

- **D-01:** `SubagentRegistry::register(info)` returns a `RegistrationGuard` value. The guard holds a `Weak<RwLock<SubagentRegistry>>` (or equivalent — exact handle shape is Claude's Discretion) plus the registered `SubagentId`. The guard's `Drop` impl calls `unregister(id)` synchronously. Cleanup is automatic on any exit path: natural completion, error return, timeout-induced drop, panic, cancel, JoinHandle::abort. No callers may call `unregister` directly — that method becomes `pub(crate)` (or private) and is invoked only by the guard.

- **D-02:** The guard is held in the spawn task that runs the child agent — i.e., inside the future passed to `tokio::time::timeout` in `delegate_task.rs::execute_single` / `execute_batch`. When timeout drops the future, the guard's Drop runs and deregisters. This is the single source of truth for "child is alive."

- **D-03:** Existing `register` / `unregister` call sites are migrated in one atomic refactor — every caller of `register` is updated to bind the returned guard. No backward-compat shim. Tests assert `register` count == `unregister` count at the end of every delegation test (a `RegistrationCounter` in `#[cfg(test)]` is acceptable).

### Stale-Child Detection

- **D-04:** Reuse Phase 32.1's `AgentLoop::ActivityTracker`. The tracker already bumps on tool calls; verify it also bumps on LLM-API responses (read the wiring before planning — may need a separate bump point). The `SubagentInfo` struct grows a `last_activity: Instant` field, populated from the child's `ActivityTracker` on each registry read.

- **D-05:** Add `stale_warn_seconds: Option<u64>` to the `delegate_task` JSON schema (single-task and per-task-in-batch). Default 120s (2 min). Falls back to a new `delegation.stale_warn_seconds` config field (also default 120s). Per-call overrides config, same pattern as `max_iterations`/`timeout_seconds` from Phase 32.2 D-08.

- **D-06:** When `now - last_activity > stale_warn_seconds`, the tree-view status pill shifts to `Stale` (new variant) — visible in `/agents` output across all surfaces. A `tracing::warn!` is emitted exactly once per child crossing the threshold (don't spam every render). No automatic kill at the warn threshold — that's user-driven via `/agents kill`.

- **D-07:** The hard-kill ceiling stays at `child_timeout_seconds` (default 300s) — this is already enforced by `tokio::time::timeout` in delegate_task.rs. The bug is that the leak survives the timeout, not that the timeout never fires. D-01's RAII guard fixes that without raising/lowering the ceiling.

### Shrike Service (Slash Subcommands)

- **D-08:** Four new `/agents` subcommands ship in 32.3 across all three surfaces with parity (same arguments, same output shape, same return codes):
  - `/agents kill <id>` — calls `AgentLoop::interrupt(reason="user kill")` to cancel the token, then `JoinHandle::abort()` on the spawned task. Guard Drop deregisters. Returns `Killed sub_xxx (ran NNNs, NN turns)`.
  - `/agents interrupt <id>` — calls `AgentLoop::interrupt(reason="user interrupt")` only — no abort. Child gets to finalize the current iteration before cancel-checked exit. Returns `Interrupted sub_xxx — finalizing...`.
  - `/agents prune` — iterates `registry.list()`, deregisters every entry where `now - last_activity > stale_warn_seconds` (uses each entry's per-call threshold, falls back to config default). Returns `Pruned N stale entries: [id1, id2, ...]`.
  - `/agents status <id>` — returns `SubagentInfo { id, parent_id, task_summary, role, depth, uptime, last_activity, turns_used, transcript_path, status }` rendered as a key/value block (TUI/gateway) or JSON (iron_hermes_ui).

- **D-09:** `kill` and `prune` are destructive — both require an explicit second-arg confirmation token on **gateway/Telegram** specifically (`/agents kill sub_xxx confirm` or `/agents prune confirm`). Rationale: Telegram messages can come from spoofed message-edit replays; the rest of the surfaces have synchronous user presence. TUI and web do NOT require confirmation — the user is clearly present. This is the only behavioral divergence between surfaces.

- **D-10:** Whether the "shrike service" is a dedicated module (`crates/ironhermes-agent/src/shrike.rs` containing `ShrikeService` with `kill/interrupt/prune/status` methods called by all three surfaces) vs scattered through `cmd_agents` handlers is **Claude's Discretion**. The functional contract is fixed; the file layout is researcher/planner's call. Recommendation: a single module is easier to test and audit. The name `shrike` is the operator-facing public name; the code module can carry it or use a more conventional name like `SubagentControl` — researcher decides.

### `/help` Discoverability Audit

- **D-11:** A dedicated plan task audits `/help` output in tui_rata, iron_hermes_ui, and gateway. For each surface, it (a) runs `/help` (or the equivalent), (b) checks whether each of `/agents`, `/agents kill|interrupt|prune|status` is listed with a one-line description, (c) writes findings as a "Help discoverability gaps" section in `32.3-SUMMARY.md` with structured entries: `surface | command | listed? | accurate? | gap_severity`.

- **D-12:** Gaps are **not fixed in 32.3**. They become follow-up todos written to `.planning/.tickets/` or the project todo system (whichever exists — check during planning) so a future doc-polish phase can address them. The new subcommands added in 32.3 (D-08) DO need help entries — those are written as part of the implementation, not the audit.

### Claude's Discretion

- **RegistrationGuard handle shape** — `Arc<RwLock<SubagentRegistry>>` weak-ref vs an mpsc channel back to a central reaper task vs an owned `Inner` with shared state. Pick the cleanest fit for the existing `SubagentRegistry::active: HashMap<...>` storage.
- **Where ActivityTracker reads happen** — push (tracker writes into a shared field on `SubagentInfo`) vs pull (registry queries the tracker via Weak ref) vs both. Pick whichever is correct given the existing Phase 32.1 plumbing.
- **`JoinHandle` retention** — the registry currently does NOT hold `JoinHandle`s for spawned children, so `kill`'s `JoinHandle::abort` needs a place to live. New `active_handles: HashMap<SubagentId, JoinHandle<()>>` on `SubagentRegistry`, or a parallel `SubagentReaper` actor, or a different shape — researcher's call.
- **`Stale` status pill rendering** — pick a color/symbol consistent with existing status pills in the tree view (probably yellow `⚠ Stale (Ns idle)`).
- **Telegram-side confirmation UX** — inline keyboard with Confirm/Cancel buttons vs require user to retype with `confirm` token. Default to retype (D-09), but inline keyboard is acceptable if it's already a pattern in the gateway.
- **What counts as a Telegram surface** — confirm during research whether `gateway` means the Telegram-specific code path or any non-TUI/non-web entry. Existing 32.2-05 work shipped `iron_hermes_ui subagent_tree_json adapter` for web; gateway carries Telegram. Use both.
- **Whether `/agents` itself (list/tree) needs any changes** — if the existing render in `cmd_agents` doesn't expose `last_activity` / `Stale` pill, extend it minimally. Don't redesign the layout.
- **Diagnostic timeout dump from 32.2 deferred list** — `~/.ironhermes/logs/subagent-timeout-<session>-<ts>.log` on child timeout with zero API calls. Optional in 32.2, optional here. Skip unless it falls naturally out of the new kill/timeout paths.

</decisions>

<specifics>
## Specific Ideas

- **The 6.7-hour ghost is the canonical bug.** Any plan must include a regression test that registers a child, simulates natural completion / timeout / panic / cancel, and asserts the registry has 0 active entries afterward. The screenshot evidence is the user's contract.
- **"Shrike service" is the operator-facing name.** It's the bird-of-prey metaphor for the termination surface — runaway agents are impaled (terminated cleanly). Use the name in user-facing help text and the DELEGATION.md section heading. The internal Rust module can use a more conventional name (researcher's call — `SubagentControl` is fine if shrike feels off in code).
- **Three-surface parity is non-negotiable.** TUI, web, and Telegram gateway all ship the same four subcommands in the same plan wave. No surface lags. The one allowed behavioral divergence is D-09 (Telegram requires `confirm` for destructive ops).
- **Reuse `AgentLoop::interrupt` from Phase 32.1 / 32.2 rather than inventing a new cancel path.** The Phase 32.2 D-21 cancel token + Phase 32.1 interrupt are the existing primitives. D-08 wraps them in user-facing commands.
- **Per-call `stale_warn_seconds` enables long-running research tasks.** The default 120s is aggressive for noisy tool-call loops; research tasks like the LoRA repro should set `stale_warn_seconds: 1800` (30 min) to silence false warnings. The kill ceiling stays at `child_timeout_seconds` — that's the actual safety net.

</specifics>
