# Phase 32.3 Plan 04: Three-surface parity for kill | interrupt | prune | status Summary

Ship the web (iron_hermes_ui REST endpoints) and gateway (Telegram registry attach + D-09 confirm gate) surfaces of the three-surface parity contract. TUI auto-inherits structurally from Plan 03's cmd_agents extension via the existing `with_subagent_registry` block in tui_rata/commands.rs:573-582 — no TUI code changes required.

After this plan, operators can kill / interrupt / prune / status from any of the three surfaces (tui_rata, iron_hermes_ui, ironhermes-gateway), with the gateway's destructive ops gated by a retype-`confirm` token (D-09).

## What Changed

### Task 1 — iron_hermes_ui REST endpoints + ShrikeService wiring

**AppState extensions (`crates/iron_hermes_ui/src/server/state.rs`):**
- New field `subagent_registry: Arc<RwLock<SubagentRegistry>>` — the same Arc threaded into `AgentSubagentRunner` via `with_subagent_registry` (state.rs:91+), now also held on AppState so the REST endpoints reach the live registry.
- New field `shrike: Option<Arc<ShrikeService>>` — constructed once in `init()` from the same registry Arc. The `Option` shape lets the endpoints degrade gracefully if a future code path leaves shrike unconfigured (currently always `Some`).
- Four free fns `api_agents_kill / interrupt / prune / status` over `Option<&ShrikeService>` + JSON body/id. Free-fn shape lets tests construct ShrikeService directly without spinning up a full AppState.
- Four thin AppState methods of the same names — pure delegation to the free fns.

**Dioxus server-fn endpoints (`crates/iron_hermes_ui/src/server/api.rs`):**
- `POST /api/agents/kill`      body `{"id": "sub_xxx"}` → `KillResult` JSON or `{"killed": false, "id": ...}`
- `POST /api/agents/interrupt` body `{"id": "sub_xxx"}` → `{"interrupted": bool, "id": ...}`
- `POST /api/agents/prune`     body `{"stale_secs": 120}` (optional) → `{"pruned": [ids...]}`
- `GET  /api/agents/status?id=sub_xxx` → `SubagentStatusInfo` JSON or 404 via `ServerFnError`

These follow the established `#[get] / #[post]` pattern in `api.rs` (matching `list_slash_commands`, `list_sessions`, `list_tools`). No separate `routes.rs` was needed because iron_hermes_ui's REST surface is Dioxus server functions — `main.rs` has only one Axum mount point (`serve_dioxus_application`).

**No confirm token on the web surface** (D-09 — only the Telegram gateway enforces `confirm` because of spoof-replay risk; TUI + web both have synchronous user presence).

**Tests (7 new in `plan_32_3_04_tests` test module):**
- `test_api_agents_kill_returns_killed_true_for_active_id` (positive path with uptime + turns shape assertions)
- `test_api_agents_kill_returns_killed_false_for_missing_id` (negative path)
- `test_api_agents_interrupt_returns_true_for_active_id`
- `test_api_agents_prune_returns_pruned_array` (empty-registry edge case)
- `test_api_agents_status_returns_some_for_active_id` (status field == "running")
- `test_api_agents_status_returns_none_for_missing_id` (404 path)

All six round-trip tests use `flavor = "multi_thread"` per Plan 03 Pitfall 1 (ShrikeService's methods bridge async→sync via `block_in_place + block_on`).

### Task 2 — Gateway subagent_registry attach + D-09 confirm gate

**`handle_slash_command` extensions (`crates/ironhermes-gateway/src/handler.rs`):**
1. **Pitfall 3 fix:** New `with_subagent_registry` attach block after the existing `with_skill_registry` block. Mirrors the pattern of all the other `with_*` attachments. The gateway has had `self.subagent_registry` since Plan 21.7-07, but `handle_slash_command` never called `with_subagent_registry` — so /agents subcommands hit the "subagent registry not wired" fallback at handlers.rs:171. Now the four arms (list / kill / interrupt / prune / status) reach core unchanged.
2. **D-09 confirm-token gate** BEFORE `handlers::dispatch`. Matches `def.name == "agents"` AND `args[0]` ∈ `{kill, prune}` AND no `confirm` arg → emits a refusal message and returns. `interrupt` and `status` pass through unchanged. Human-readable retype hint:
   ```
   Destructive op `/agents kill`. Re-run as:
     `/agents kill sub_xxx confirm`
   ```

**New module-level fn `requires_confirm`:**
```rust
pub(crate) fn requires_confirm(subcommand: &str, args: &[&str]) -> bool {
    let is_destructive = matches!(subcommand, "kill" | "prune");
    if !is_destructive {
        return false;
    }
    !args.iter().any(|a| *a == "confirm")
}
```
Tolerant `confirm` position (any arg after the subcommand) — operators can type `/agents kill sub_xxx confirm` OR `/agents kill confirm sub_xxx`.

**Tests (`crates/ironhermes-gateway/tests/agents_confirm.rs` — 7 new):**

Pure-function contract tests (4):
- `test_kill_requires_confirm_on_gateway` — kill without confirm refused; with confirm anywhere allowed
- `test_kill_with_confirm_on_gateway` — three positive positions including operator's "extra" arg
- `test_prune_requires_confirm` — both branches plus the `<stale_secs> confirm` operator path
- `test_interrupt_status_never_require_confirm` — D-09 carve-out for non-destructive subcommands

Static source-grep invariants (3, matches `skill_registry_wiring.rs` pattern):
- `requires_confirm_definition_lives_in_handler` — grep-locks the production fn body
- `confirm_gate_fires_before_dispatch_in_handle_slash_command` — byte-position assertion that the gate call comes BEFORE the dispatch call (locks ordering against silent reorder)
- `subagent_registry_attached_to_command_context_in_handle_slash_command` — locks the Pitfall 3 fix against silent removal

The integration test uses a **mirror-fn pattern**: a private `mirror_requires_confirm` in the test file with identical logic to the production fn, exercised by the unit tests. The grep invariants then assert the production fn AND call site both exist verbatim — so any divergence between the mirror and production is caught immediately by the grep invariants failing.

## Deviations from Plan

### Rule 1 — REST endpoints in api.rs (Dioxus server fns) instead of a new routes.rs

The plan's <action> step 7 instructed editing `crates/iron_hermes_ui/src/server/routes.rs` to add Axum `.route(...)` chains. **No such file exists** — iron_hermes_ui's REST surface is Dioxus server functions in `api.rs` (the entire file is `#[get]` / `#[post]` macros), and `main.rs` has only one Axum mount point (`serve_dioxus_application`). Adding a parallel routes.rs would diverge from the established crate convention.

**Fix:** Implemented the four endpoints as `#[get] / #[post]` Dioxus server fns in `api.rs` next to `list_slash_commands` / `list_sessions` / `list_tools`. This matches the crate's idiomatic pattern verbatim.

Surfaced for awareness — not a behavior change vs the plan's intent.

### Rule 2 — Free fn shape over Option<&ShrikeService> (not &AppState)

The plan's <action> step 4 had the four endpoints take `&AppState`. **AppState is too heavy to construct in a unit test** — `AppState::init()` requires Config, StateStore, ProviderResolver, MemoryManager (with optional providers), and a fully-built `AppRuntimeBundle` with semaphore + delegate-task wiring. Building this in a `#[test]` would either run an actual init (which depends on `$HERMES_HOME` + on-disk providers + LLM API keys) or require a parallel mock harness.

**Fix:** Extracted the four endpoint bodies as free fns over `Option<&ShrikeService>` + JSON body. AppState methods are thin one-liner wrappers. Tests construct ShrikeService directly from a SubagentRegistry — covers the load-bearing endpoint contract without the heavyweight init.

The free-fn surface IS the contract — server fns in `api.rs` call the AppState method which delegates to the free fn. Zero behavior change vs the plan's intent.
