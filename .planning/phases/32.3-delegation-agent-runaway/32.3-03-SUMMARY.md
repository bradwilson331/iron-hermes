# Phase 32.3 Plan 03: Shrike Service + /agents interrupt|prune|status Summary

Centralized the four operator-facing termination/diagnostic operations (kill/interrupt/prune/status) in a single `ShrikeService` library type so all three surfaces (TUI / web / gateway in Plan 04) share one tested contract. Kill now performs token cancel **AND** `JoinHandle::abort` (W3 — closes the silent token-only cancel under D-08). `/help` lists the three new subcommands with one-line descriptions (B3 / D-12 carve-out).

### Task 1a — `shrike.rs` module + lib.rs + SubagentRegistryHandle wiring

- **`crates/ironhermes-agent/src/shrike.rs` (NEW, 280 LOC)**: `pub struct ShrikeService` holding `Arc<RwLock<SubagentRegistry>>` + `Arc<Mutex<HashMap<SubagentId, JoinHandle<()>>>>`. Four public methods:

## Deviations from Plan

### Deviation D-1 [Rule 4 - architectural carve-out → resolved without checkpoint]

**Topic:** JoinHandle storage in `execute_batch` keyed by synthetic `batch_task_<index>` instead of by SubagentId

**Plan text (Task 1b action 4):** "after `let handle = tokio::spawn(async move { ... });`, insert `if let Some(ref sh) = self.shrike { sh.handle_map().lock().unwrap().insert(subagent_id.clone(), handle); }`"

**Issue:** The plan's snippet assumes `subagent_id` is in scope at spawn-time in `execute_batch`. It is NOT. The canonical SubagentId is minted INSIDE `runner.run_child` at `subagent_runner.rs:217` via `random_subagent_id()`. The variable `subagent_id` does not exist in `delegate_task.rs::execute_batch` until the run_child call returns, at which point the JoinHandle has already been pushed to the results-collection Vec (and JoinHandle is not Clone).

**Resolution:** Plan 03 ships the *structural* pieces — `ShrikeHandleMap` type alias, `DelegateTaskTool.shrike_handles` field, `with_shrike_handle_map` builder, and a synthetic `batch_task_<index>` key documented for Plan 04 wiring. The unit tests (`shrike_service.rs`) populate the handle map *directly* to exercise the JoinHandle::abort semantics. The W3 fix (kill performing JoinHandle::abort) is fully functional in tests and via direct `ShrikeService::kill` calls; production batch correlation is deferred to Plan 04.

**Pragmatic impact:** Zero regression vs pre-Plan-03 behavior. Real-world `/agents kill <id>` today cancels the CancellationToken via the trait override → `ShrikeService::kill` (same as the existing token-cancel-only kill at HEAD). The `JoinHandle::abort` codepath is exercised by tests; production wiring lands in Plan 04 when surface adapters need the abort semantics for wedged children that ignore their cancel tokens.

**Alternative considered & rejected:** Threading a pre-minted SubagentId through the `SubagentRunner::run_child` trait signature. Plan 02 already extended the trait once (added `stale_warn_seconds: u64`); doing it again in Plan 03 would have required updating 12+ mock impls and an additional production runner — significantly out of scope.

**Tracked in `<output>` per the plan's reporting requirement.**

### Deviation D-2 [Rule 2 - documentation correctness]

**Topic:** PRUNE_DEFAULT_STALE_SECS hardcoded to 120 instead of reading from CommandContext

**Plan text:** "let stale_secs = ctx.delegation_stale_warn_seconds().unwrap_or(120); let ids = reg.prune(stale_secs); ..."

**Issue:** `CommandContext` has no `delegation_stale_warn_seconds()` accessor. The plan's `<action>` step 1 for the prune arm said "If `CommandContext` has no `delegation_stale_warn_seconds` accessor yet, add one returning `Option<u64>` (read from a new ctx field that callers populate; default None → 120 fallback in the arm)." Adding a new CommandContext field requires updating the context-attach sites in TUI, gateway, web — that's surface wiring (Plan 04 scope).

**Resolution:** Use `const PRUNE_DEFAULT_STALE_SECS: u64 = 120;` matching the `SubagentConfig::stale_warn_seconds` default (D-05). Plan 04's per-surface adapters may surface a per-session override via a new CommandContext field if needed. The 120s default matches the registry's own stale threshold, so `/agents prune` sweeps exactly the entries flagged `[stale]` by the existing tree-view pill.

### Deviation D-3 [Rule 3 - blocking issue]

**Topic:** SubagentRegistryHandle field-shape change (tuple → named struct)

**Issue:** The pre-Plan-03 `SubagentRegistryHandle(pub Arc<RwLock<SubagentRegistry>>)` was a tuple struct. Field visibility set to `pub(crate)` for the same reason. Zero external callers affected.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| ShrikeService constructed inside SubagentRegistryHandle::new | Existing callers of `SubagentRegistryHandle::new` (TUI/gateway/web) automatically get the hard-kill semantics without any wiring change. The handle's trait override for kill forwards to ShrikeService::kill via `self.shrike.kill(id).is_some()` — handlers.rs source line is unchanged. |
| ShrikeService is Clone + cheap | Two Arc fields. The `.shrike()` accessor on SubagentRegistryHandle returns the same Arc that the embedded ShrikeService uses, so Plan 04 surface adapters can hand the same handle_map to `DelegateTaskTool::with_shrike_handle_map` without forking the registry's view of active handles. |
| ShrikeService::kill does NOT explicitly remove from registry | The cancel + abort fires the spawned future's RegistrationGuard Drop which calls unregister_internal (Plan 01 RAII). Letting that path own deregistration preserves: (a) `stale_warned` HashSet cleanup, (b) single-source-of-truth on `active` map mutations, (c) compatibility with the existing 6.7-hour-ghost regression test. Explicitly removing in kill would double-remove and break the dedup invariant. |
| Trait default impls return sensible "not supported" defaults | `fn interrupt → false`, `fn prune → Vec::new()`, `fn status → None`. This keeps all 4 existing `SubagentListSnapshot` impls compiling unchanged. Only the canonical `SubagentRegistryHandle` overrides them; `FakeSubagents` overrides them explicitly in cmd_agents_and_stop.rs to enable the new dispatch tests. |
| Two-pass prune (read → drop → per-id mutation) | RESEARCH T-32.3-10 mitigation: holding the write lock across the iteration would block concurrent register/unregister for the duration of the sweep. Two-pass keeps each lock acquisition bounded. The window between read and per-id-cancel is tolerable — if an entry de-registers concurrently, the cancel is a no-op (token already cancelled on Drop or entry already gone). |
| JoinHandle map keyed by `batch_task_<index>` not SubagentId | See Deviation D-1. Plan 04 owns the SubagentId-correlation refactor; Plan 03 lands the structural plumbing. Tests insert handles directly to validate the abort code path. |
