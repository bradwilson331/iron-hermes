# Phase 32.3 Plan 02: Stale-child detection + tree-view [stale] pill Summary

Activity-tracker wiring (Phase 32.1's `AgentLoop::activity_last`) flows through to `SubagentInfo` via a new `activity_last_arc()` accessor, the registry derives `Stale` status by comparing `elapsed > stale_warn_seconds` at render time, and once-per-id `tracing::warn!` is gated by the `stale_warned` HashSet that Plan 01 reserved.

## What Changed

### Task 1 — `activity_last_arc` accessor + delegate_task schema + SubagentInfo extension

- **`AgentLoop::activity_last_arc(&self) -> Arc<std::sync::Mutex<std::time::Instant>>`** added (agent_loop.rs:617). Returns a clone of the SAME Arc bumped by the four existing `mark_activity` sites — no new bump points (RESEARCH A1 confirmation: field exists named exactly `activity_last`, is exactly `Arc<Mutex<Instant>>`).
- **`SubagentInfo.stale_warn_seconds: u64`** field added (D-05). Stored alongside `activity_last` so the registry render path computes `elapsed > stale_warn_seconds` without re-reading config.
- **`SubagentRegistry.stale_warned`** type changed from bare `HashSet<SubagentId>` → `std::sync::Mutex<HashSet<SubagentId>>` (Rule 2 critical-correctness fix). Plan 01 reserved the field bare; Task 2's once-per-child gate fires from `tree_summary()` (which takes `&self`) so interior mutability via `Mutex` is the correct pick (HashSet doesn't impl `Copy` so `Cell` is not an option).
- **`SubagentRunner::run_child` trait signature** extended with `stale_warn_seconds: u64` parameter. Threaded through `DelegateTaskTool::execute` (single-task path) and `execute_batch` (per-task spawn block) with the same fallback chain as `max_iterations`: `task_obj.get("stale_warn_seconds") → config.stale_warn_seconds → 120`.
- **`AgentSubagentRunner::run_child`** reordered to construct `AgentLoop` FIRST (so `activity_last_arc()` can be cloned), then build `SubagentInfo` with both new fields populated, then `register_guarded`. The order is: `agent.new → wire fallback + cancel + progress + budget → activity_last_arc.clone() → register_guarded → agent.run`.
- **JSON schema** gained `stale_warn_seconds: Option<u64>` at both the top-level (single-task) and `tasks.items.properties` (batch) locations. Mirrors the existing `max_iterations` schema entries verbatim.
- **12 SubagentInfo construction sites** updated to set the new `stale_warn_seconds: 120` field; **12 mock `SubagentRunner` impls** updated to receive the new `_stale_warn_seconds: u64` parameter.

### Task 2 — Stale derivation in `flatten_tree` + render pill + tests

- **`flatten_tree`** in `subagent_registry.rs` extended with the priority chain: `killed > stale > running`. Stale derivation reads `info.activity_last.lock().elapsed().as_secs()` and compares to `info.stale_warn_seconds`. When the threshold is crossed AND the id is NOT in `stale_warned`, `tracing::warn!(subagent_id, idle_secs, stale_warn_seconds, "subagent stale threshold crossed")` fires and the id is inserted into the set. Free function parameterised on `&std::sync::Mutex<HashSet<SubagentId>>` for clean separation.
- **`SubagentRegistryHandle::tree_summary`** passes `&guard.stale_warned` into `flatten_tree` so the read lock and dedup set are scoped together.
- **`render_agent_tree` flat-list fallback** in `handlers.rs` appends `[{status}]` pill when status is NOT `"running"` — preserves byte-identical legacy output for the dominant non-orchestrated session case while surfacing `[stale]`/`[killed]` when relevant. Tree mode already rendered the pill verbatim (Phase 32.2 Plan 04 D-12), so no change there.
- **2 new tests in `cmd_agents_and_stop.rs`**:
  - `test_render_agent_tree_shows_stale_pill` — dispatch `/agents` against `FakeSubagentsWithTree` returning `status="stale"`; output contains `[stale]`.
  - `test_stale_warn_fires_once` — smoke test against the trait surface (the canonical contract test lives in `ironhermes-agent/tests/stale_warn_once.rs` because `ironhermes-core` cannot dev-dep on `ironhermes-agent` — circular).
- **4 new tests in NEW `crates/ironhermes-agent/tests/stale_warn_once.rs`** — canonical exercise of the registry-side stale + warn-gate contract:
  - `test_tree_summary_derives_stale_status` (elapsed > threshold → `"stale"`)
  - `test_tree_summary_running_below_threshold` (elapsed <= threshold → `"running"`)
  - `test_stale_warn_fires_once_across_repeated_tree_summary_calls` (3x `tree_summary` calls; status remains `"stale"`; `stale_warned` HashSet semantics are deterministic on id membership and the registry's `active_count` stays exactly 1)
  - `test_killed_status_takes_precedence_over_stale` (D-12 priority order: cancelled token + idle clock → `"killed"`, not `"stale"`)

## stale_warned Mutex<HashSet> Wrapping (per <output> requirement)

**Required:** `stale_warned` field needed to support `&self` mutation because `flatten_tree` is called from `tree_summary()` which takes `&self` via the `SubagentListSnapshot` trait.

**Resolution:** wrapped in `std::sync::Mutex<HashSet<SubagentId>>` (Rule 2 fix surface from Plan 01). Both `Default::default()` and the field's initial state are unchanged structurally (a Mutex around an empty HashSet is equivalent to an empty HashSet for read semantics).

`unregister_internal` was updated to lock the mutex before calling `remove` (silently no-ops if the lock is poisoned — the entry is still removed from `active`). The lock is held briefly (a single membership check + insert) so contention is structurally impossible in practice — `flatten_tree` is called from a single tokio task at a time (the slash dispatch path).
