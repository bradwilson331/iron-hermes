## Summary

Phase 32.3 fixes a confirmed registry-leak bug where a subagent's `run_child` future is dropped by `tokio::time::timeout` (or by cancel-token abort) before reaching its `unregister` call on line 331 of `subagent_runner.rs`. The fix is RAII: `SubagentRegistry::register` returns a `RegistrationGuard` whose `Drop` impl calls `unregister`. When `tokio::time::timeout` drops the future mid-execution, Rust's drop semantics guarantee the guard fires ﻗ no code path can bypass it. The canonical live repro is `sub_20667cb71808` remaining Active for 24,150s (6.7 hours) after its output files were written at 02:52 AM.

On top of the RAII fix, Phase 32.3 ships the "shrike service": four new `/agents` subcommands (`kill`, `interrupt`, `prune`, `status`) available across all three surfaces (tui_rata, iron_hermes_ui web, ironhermes-gateway Telegram). Stale-child detection using Phase 32.1's `ActivityTracker` adds a `Stale` status pill and `tracing::warn!` when a subagent has been idle longer than `stale_warn_seconds`.

The architecture is cleanly layered: the guard fix lives in `ironhermes-agent` (subagent_runner.rs + subagent_registry.rs), the schema changes live in `ironhermes-tools` (delegate_task.rs + SubagentConfig in ironhermes-core), and the command surface changes live in `ironhermes-core` (handlers.rs cmd_agents expansion) with thin per-surface adapters in ironhermes-cli (tui_rata), iron_hermes_ui, and ironhermes-gateway.

**Primary recommendation:** Implement the RAII guard in Wave 1, wire `last_activity` + `Stale` detection in Wave 2, implement the four shrike subcommands in Wave 3 (all three surfaces simultaneously ﻗ three-surface parity is non-negotiable per D-08), write regression tests in Wave 4, and audit `/help` discoverability in Wave 5.

## Architecture Patterns

### The Exact Bug Path [VERIFIED: live source code]

```
delegate_task.rs execute() line 803:
  tokio::time::timeout(
    effective_timeout_secs,
    self.runner.run_child(...)   // <-- this Future is DROPPED on timeout
  ).await
  
  On timeout: run_child Future dropped mid-execution.
  subagent_runner.rs line 331: reg.write().await.unregister(&subagent_id)
  -- THIS CODE NEVER RUNS because the future is dropped before reaching it.
  
  Result: SubagentRegistry::active still contains the entry.
  The agent appears Active for 6.7 hours after natural completion.
```

The same path exists in `execute_batch`'s `tokio::spawn` inner future.

### Pattern 1: RAII RegistrationGuard

**What:** `SubagentRegistry::register` returns an owned guard that calls `unregister` in its `Drop` impl. The guard is placed at the top of the `run_child` async body, inside the timeout future.

**When to use:** Any time you need "cleanup on all exit paths" in async Rust, including future-drop.

**Recommended handle shape:** `Weak<tokio::sync::RwLock<SubagentRegistry>>`

The existing `SubagentRegistryHandle` wraps `Arc<RwLock<SubagentRegistry>>`. The guard should hold a `Weak` to avoid a reference cycle (guard ﻗ registry ﻗ guard). `Weak::upgrade()` in Drop is safe: if the Arc is gone (session shutdown), the unregister is a no-op, which is correct.

```rust
// Source: subagent_registry.rs (proposed addition)
pub struct RegistrationGuard {
    registry: std::sync::Weak<tokio::sync::RwLock<SubagentRegistry>>,
    id: SubagentId,
}

impl Drop for RegistrationGuard {
    fn drop(&mut self) {
        if let Some(arc) = self.registry.upgrade() {
            // block_in_place bridge ﻗ same pattern as SubagentRegistryHandle::kill
            tokio::task::block_in_place(|| {
                tokio::runtime::Handle::current()
                    .block_on(async { arc.write().await.unregister_internal(&self.id) })
            });
        }
    }
}

impl SubagentRegistry {
    // New: returns guard; replaces direct register call
    pub fn register_guarded(
        &mut self,
        info: SubagentInfo,
        weak: std::sync::Weak<tokio::sync::RwLock<SubagentRegistry>>,
    ) -> RegistrationGuard {
        let id = info.id.clone();
        self.active.insert(id.clone(), info);
        RegistrationGuard { registry: weak, id }
    }

    // Becomes pub(crate) ﻗ only called by RegistrationGuard::drop
    pub(crate) fn unregister_internal(&mut self, id: &str) -> Option<SubagentInfo> {
        self.active.remove(id)
    }
}
```

**Caller side (subagent_runner.rs):** The runner receives a `Weak` from its `Arc<RwLock<SubagentRegistry>>` caller. Inside `run_child`, the guard is the first binding after registration, and is `let _guard = ...` so it lives for the full future scope.

**Why NOT mpsc reaper:** The mpsc approach requires the receiver to be running for unregister to fire. If the receiver is also dropped (session shutdown), the send silently fails. The `Weak`-based Drop is strictly simpler and doesn't require a separate actor.

**Why NOT `Arc<Inner>` with shared state:** The registry is already `Arc<RwLock<SubagentRegistry>>`. Adding another Arc layer around an inner struct would require restructuring the existing `SubagentRegistryHandle` and all callers. The `Weak` approach is a surgical addition.

### Pattern 2: JoinHandle Retention for kill

**What:** `SubagentRegistry` (or a parallel `ShrikeService`) holds `JoinHandle`s for spawned children so `kill` can call `JoinHandle::abort()`.

**Current state:** The registry holds `SubagentInfo` but NOT `JoinHandle`s. The `kill` method on the registry (line 62-70, subagent_registry.rs) cancels the `CancellationToken` but cannot `abort()` the task.

**Recommended shape:** Add `active_handles: HashMap<SubagentId, tokio::task::JoinHandle<()>>` to `SubagentRegistry`. Populated by the spawner (`delegate_task.rs execute_batch` spawns tokio tasks; the single-task path does NOT spawn ﻗ it awaits directly via `tokio::time::timeout`). The `kill` subcommand: cancel token + `handle.abort()` + remove from `active_handles`.

**Key insight:** Only `execute_batch` uses `tokio::spawn`. The single-task path (`execute`) awaits inline ﻗ there is no `JoinHandle` to abort for single tasks. The RAII guard covers the single-task path. For batch tasks, `JoinHandle::abort()` is needed because the task is running concurrently.

**Shrike module vs scattered:** Recommended to add a `shrike.rs` module in `ironhermes-agent` with `ShrikeService { registry: Arc<RwLock<SubagentRegistry>> }` implementing `kill/interrupt/prune/status`. This keeps the logic auditable in one place and matches D-10's guidance. The internal Rust name `SubagentControl` is also acceptable ﻗ researcher recommends `shrike.rs` to match the operator-facing name.

### Pattern 3: ActivityTracker ﻗ SubagentInfo Wiring

**What:** Each child's `AgentLoop` has an `ActivityTracker` (Phase 32.1). `SubagentInfo` needs `last_activity: Instant`. The question is how the registry gets updated.

**ActivityTracker bump points confirmed [VERIFIED: agent_loop.rs lines 636-639, 946, 1151, 1404, 1412]:**
- Line 946: Before each LLM API call (`ApiCall, None`)
- Line 1151: On each streamed content delta (`StreamToken, None`)
- Line 1404: Before each tool dispatch (`ToolCall, Some(name)`)
- Line 1412: After each tool dispatch (`ToolCall, None`)

The tracker bumps on BOTH LLM responses (streaming) and tool calls. No new bump point is needed ﻗ the existing four sites cover all activity.

**Recommended wiring (pull model):** Add `last_activity: Instant` to `SubagentInfo`. In `subagent_runner.rs::run_child`, after `agent.run()` completes (or just before unregister), update the registry entry with `agent.activity_summary().last_kind` timing. For the stale detection path in `list()` / `tree_summary()`, compute `now - info.last_activity`.

**Alternative (push model):** The child `AgentLoop` could hold a reference to the `SubagentInfo` and update it on each `mark_activity` call. This is more complex ﻗ it requires giving the AgentLoop a mutable handle to registry data, which breaks the existing separation. Pull is cleaner.

**For `/agents status` real-time accuracy:** The pull model has one limitation: `last_activity` is only updated when `run_child` returns or at periodic checkpoints. For a running child, the registry entry's `last_activity` is stale until the next update. The simplest fix: store the child's `ActivitySummary` source (an `Arc<Mutex<Instant>>`) in `SubagentInfo` so reads are always live. This is exactly the `activity_last` mutex already on `AgentLoop` ﻗ expose it via a `last_activity_instant()` accessor.

**Recommended:** Add `activity_last: Option<Arc<std::sync::Mutex<std::time::Instant>>>` to `SubagentInfo`. Wire it from `agent.activity_last.clone()` before `agent.run(...)`. Registry reads call `.lock().elapsed()` for live staleness. This is the cleanest zero-copy solution ﻗ no new bump point, no periodic push.

### Pattern 4: SubagentInfo.status Variants

**Current state [VERIFIED: subagent_registry.rs]:**

`SubagentInfo` has NO `status` field. Status is DERIVED in `flatten_tree` (subagent_registry.rs line 146-150):
```rust
let status = if node.info.cancel.is_cancelled() {
    "killed".to_string()
} else {
    "running".to_string()

```

And in `SubagentTreeEntry.status: String` (context.rs ﻗ passed as a string, not an enum).

**For D-06 Stale:** Either:
1. Add a `status: SubagentStatus` enum field to `SubagentInfo` with variants `Running`, `Stale`, `Killed` ﻗ mutated when stale threshold crossed.
2. Keep status as derived: compute stale in `flatten_tree` by checking `last_activity.elapsed() > stale_warn_seconds`.

**Recommended: option 2 (derive at read time).** Avoids mutable state churn in the registry and keeps the registry as a pure data store. `flatten_tree` gets the stale threshold as a parameter. The `tracing::warn!` for once-per-child crossing is handled by a separate `HashSet<SubagentId>` tracking which IDs have already been warned ﻗ stored on the `ShrikeService` or as a `Mutex<HashSet>` alongside the registry.

### Pattern 6: cmd_agents Extension

The current `cmd_agents` (handlers.rs line 168-255) has arms for `list`, `kill`, `logs`, and unknown. The four new subcommands extend this match:

```rust
Some("interrupt") => { /* cancel token only, no abort */ }
Some("prune") => { /* sweep stale entries */ }
Some("status") => { /* diagnostic key/value block */ }
```

The `kill` arm currently calls `reg.kill(id)` which only cancels the CancellationToken (subagent_registry.rs line 62-70). Phase 32.3 must extend this to also call `JoinHandle::abort()`. The cleanest path: `SubagentListSnapshot` trait grows `interrupt(id)` and `prune()` and `status(id)` methods (alongside the existing `kill`), and `SubagentRegistryHandle` implements them. Alternatively, `kill` in the trait is extended to also abort the handle. Researcher recommendation: extend `SubagentListSnapshot` with new methods for the new operations ﻗ keeps the trait as the full surface abstraction.

### Anti-Patterns to Avoid

- **`unregister` after `await`:** The existing pattern in `subagent_runner.rs` ﻗ register, run agent, then unregister after `agent.run()` ﻗ is WRONG because the future can be dropped before `agent.run()` returns. The RAII guard fixes this structurally.
- **Async Drop:** Rust does not support async Drop. The guard's Drop must use `block_in_place` + `block_on` bridge (same pattern as `SubagentRegistryHandle::kill` at line 187-190). This is safe on the tokio multi-thread runtime.
- **Unregister in timeout arm only:** Only cancelling the token in the timeout arm (current code) is insufficient ﻗ if the future is already dropped by `tokio::time::timeout`, the code after `timeout.await` runs, but the inner future (which would have called unregister) is gone. The guard fixes this at the language level.
- **Blocking the registry lock in a hot path:** The `block_in_place` bridge in Drop is fine for the (rare) drop-on-timeout path. Don't add it to frequently-called paths.
- **warn! spam:** The stale warn should fire exactly once per child ID crossing the threshold. Use a `warned_ids: Mutex<HashSet<SubagentId>>` or equivalent to gate the emit.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Cleanup on async future drop | Manual cleanup in timeout arm | RAII `RegistrationGuard` with `Drop` | Rust guarantees Drop runs on future-drop; manual cleanup in match arms can be skipped by early returns |
| Child task cancellation | New signal channel | `CancellationToken::cancel()` (32.1) + `JoinHandle::abort()` | Both already exist; abort is the hard-kill guarantee |
| Elapsed time since last activity | New timestamp field managed manually | `Arc<Mutex<Instant>>` shared with AgentLoop's `activity_last` | Direct reference to the live counter; zero serialization overhead |
| Subcommand confirmation | Session state machine for pending confirmations | `args[2] == "confirm"` parse pattern | Stateless; Telegram replay-attack resistance at zero cost |

## Common Pitfalls

### Pitfall 1: block_in_place Inside tokio::spawn

**What goes wrong:** Calling `block_in_place` inside a `tokio::spawn` task requires the multi-thread runtime. On the current-thread runtime (some tests), this panics.

**Why it happens:** `block_in_place` is only safe on multi-thread. The guard's Drop will call `block_in_place` when a batch task's JoinHandle is aborted.

**How to avoid:** Tests that test the guard Drop must use `#[tokio::test(flavor = "multi_thread")]`. The SubagentRegistryHandle tests already use this pattern (subagent_registry.rs line 311: `#[tokio::test]` with `spawn_blocking`).

**Warning signs:** `Cannot block_in_place inside a current_thread runtime` panic in tests.

### Pitfall 2: Double-unregister if Guard and Explicit unregister Both Run

**What goes wrong:** If any code path calls `unregister` explicitly AND the guard also runs, the registry gets a double-remove (benign ﻗ `HashMap::remove` on absent key returns None) but the logic is wrong.

**Why it happens:** Migrating existing call sites. The comment at subagent_runner.rs line 331 (`reg.write().await.unregister(&subagent_id)`) must be REMOVED when the guard is introduced.

**How to avoid:** D-03 mandates one atomic refactor. `unregister` (renamed `unregister_internal`) becomes `pub(crate)`. The compiler will catch any remaining external callers.

### Pitfall 3: Gateway /agents Subcommands Return "Subagent registry not wired"

**What goes wrong:** `handle_slash_command` builds a `CommandContext` without attaching the subagent registry, so `cmd_agents` hits the `None` arm at handlers.rs line 171.

**Why it happens:** The gateway has `self.subagent_registry` (line 116 of handler.rs) but `handle_slash_command` (line 364+) does NOT call `ctx.with_subagent_registry(...)`. The TUI does this at commands.rs line 574-582.

**How to avoid:** In `handle_slash_command`, after building `ctx`, add:
```rust
let ctx = if let Some(ref reg) = self.subagent_registry {
    use ironhermes_agent::subagent_registry::SubagentRegistryHandle;
    use ironhermes_core::commands::context::SubagentListSnapshot;
    let handle: Arc<dyn SubagentListSnapshot> =
        Arc::new(SubagentRegistryHandle::new(reg.clone()));
    ctx.with_subagent_registry(handle)
} else {
    ctx
};
```

### Pitfall 4: Stale Warn Fires Every Render Cycle

**What goes wrong:** The stale check runs every time `tree_summary()` is called (which can be every TUI render frame, ~100ms). Without a warn-dedup gate, `tracing::warn!` fires hundreds of times per second.

**Why it happens:** D-06 says warn "exactly once per child crossing the threshold" ﻗ but the threshold crossing is checked at read time.

**How to avoid:** The warn-dedup state is a `Arc<Mutex<HashSet<SubagentId>>>` on the registry (or ShrikeService). On threshold cross: check if ID is in set. If not: warn + insert. On unregister: remove from set. This is the same "exactly-once" pattern used in budget tier warnings (agent_loop.rs `last_pressure_tier_seen`).

### Pitfall 5: JoinHandle Abort Does Not Guarantee Instant Cleanup

**What goes wrong:** `JoinHandle::abort()` schedules the task for cancellation but does not block until the task has exited. The registry may be cleaned up (guard Drop) while the task is still completing its current `await` point.

**Why it happens:** tokio's abort is cooperative at `await` points ﻗ it cancels the task at the next poll. The guard Drop runs when the future is dropped (which happens when abort completes the future), so the timing is correct ﻗ but callers should not assume the task is gone immediately.

**How to avoid:** The `kill` subcommand returns `Killed sub_xxx` immediately after calling abort + cancel. This is correct ﻗ the cleanup (guard Drop) is guaranteed to run, but the caller doesn't need to await it.

### Pitfall 6: execute_batch Spawns Tasks ﻗ Guard Must Be INSIDE the Spawned Future

**What goes wrong:** If the guard is placed in `execute_batch` outside the `tokio::spawn(async move { ... })` block, it drops when `execute_batch` returns, not when the individual task completes.

**Why it happens:** The batch loop spawns tasks and collects handles. The guard for each task must be inside the `async move { ... }` closure.

**How to avoid:** For batch tasks, the guard is created inside the spawned async block, immediately after acquiring the semaphore permit and registering in the registry.

### Key Test Design: Guard Drop on Future-Drop

```rust
// Pattern for testing timeout-induced drop:
#[tokio::test(flavor = "multi_thread", worker_threads = 2)]
async fn test_guard_deregisters_on_timeout() {
    let registry = Arc::new(RwLock::new(SubagentRegistry::new()));
    
    // Spawn a future that registers and then hangs
    let reg_clone = registry.clone();
    let result = tokio::time::timeout(
        Duration::from_millis(50),
        async move {
            let weak = Arc::downgrade(&reg_clone);
            let mut w = reg_clone.write().await;
            let _guard = w.register_guarded(make_info("sub_test"), weak);
            drop(w);
            // Hang forever ﻗ timeout will drop this future
            tokio::time::sleep(Duration::from_secs(9999)).await;
        }
    ).await;
    
    assert!(result.is_err(), "should have timed out");
    // Guard must have fired ﻗ registry must be empty
    assert_eq!(registry.read().await.active_count(), 0,
        "guard must deregister when future is dropped by timeout");
}
```
