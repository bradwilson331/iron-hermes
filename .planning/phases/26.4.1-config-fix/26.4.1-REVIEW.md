# Phase 26.4.1: Code Review Report

**Reviewed:** 2026-05-07T00:00:00Z
**Depth:** standard
**Files Reviewed:** 3
**Status:** issues_found

## Summary

Three files were reviewed: the setup wizard (`setup.rs`), the CLI entry point (`main.rs`), and the CFG-03 regression test (`invariants_26_4_1_cfg_03.rs`). The CFG-01 (.env write) and CFG-03 (preflight gate widening) implementations are structurally correct. Two security issues were found: secret input is never actually masked despite the comment and prompt claiming it is (rustyline 15's `Builder::new().build()` has no masking capability), and the dotenv upsert writes unsanitized newlines from user input into the `.env` file, enabling line injection. Four additional quality/robustness warnings are documented below.

---

## Critical Issues

### CR-01: Secret input is NOT masked — `Builder::new().build()` has no masking support in rustyline 15

**File:** `crates/ironhermes-cli/src/setup.rs:459-465`

**Issue:** The code creates a `masked_cfg` from `Builder::new().build()` and claims this causes typed characters to be echoed as asterisks or suppressed (the comment says "so typed chars aren't echoed" and the prompt label says "(masked)"). Rustyline 15's `Builder` / `Config` struct has no masking field whatsoever — confirmed by inspecting `rustyline-15.0.0/src/config.rs`, which contains no `mask` option. The temporary `masked_rl` editor behaves identically to a plain editor: every character is echoed to the terminal in plain text. Any operator typing an API key at a `"kind": "env_var"` prerequisite prompt will have their secret echoed on-screen, contradicting the T-25-02 security requirement.

**Fix:** Replace the `Builder` approach with rustyline's actual masking mechanism. In rustyline 15 the correct path is `Editor::new()` followed by calling `readline` via the `rpassword` crate (or the `rpassword::read_password()` / `rpassword::prompt_password()` API), which disables TTY echo at the OS level and is the standard Rust approach for secrets. Alternatively, use `crossterm`'s raw-mode echo disable if rustyline is required. The simplest drop-in that works today:

```rust
// Replace the masked_rl block entirely:
let raw = rpassword::prompt_password(format!("{}: ", prompt))
    .map_err(|e| anyhow!("readline error: {}", e))?;
```

Add `rpassword` to `Cargo.toml` (it is a standard dependency for this purpose). The non-secret path can keep rustyline unchanged.

### CR-02: Newline injection into `.env` via unsanitized user input in `write_env_var_to_dotenv`

**File:** `crates/ironhermes-cli/src/setup.rs:510-566`

**Issue:** `write_env_var_to_dotenv` joins lines with `\n` and appends a trailing `\n`, but neither `name` nor `value` is checked for embedded newline characters before formatting them into `format!("{}={}", name, value)`. If a user (or a test fixture) passes a value containing a literal `\n`, the resulting `.env` file gains an injected line. For example, `value = "sk-real\nEVIL_VAR=injected"` writes:

```
OPENROUTER_API_KEY=sk-real
EVIL_VAR=injected
```

This is a config injection vulnerability — an operator entering a maliciously crafted API key (e.g. pasted from a compromised clipboard) can silently write arbitrary variables into the `.env` file that will be loaded by `dotenvy` on next startup. The upsert match (`starts_with(&prefix)`) also only replaces the first line matching the key prefix, so the injected line survives future upserts of the original key.

The `name` parameter comes from `provider_env_var_name()` (controlled), but the `prereq.name` path in `apply_prereq_value` → `write_env_var_to_dotenv` passes `prereq.name` directly, which is sourced from tool registry definitions and not operator-controlled in normal use. The `value` parameter is always operator-supplied input.

**Fix:** Sanitize both parameters before writing. Reject (return an error) if either contains `\n` or `\r`:

```rust
pub(crate) fn write_env_var_to_dotenv(hermes_home: &Path, name: &str, value: &str) -> Result<()> {
    if name.contains('\n') || name.contains('\r') {
        anyhow::bail!("env var name must not contain newline characters");
    }
    if value.contains('\n') || value.contains('\r') {
        anyhow::bail!("env var value must not contain newline characters; strip or quote before passing");
    }
    // ... rest of existing logic unchanged
```

## Warnings

### WR-01: `run_setup` error message omits `agent` from the valid-section list

**File:** `crates/ironhermes-cli/src/setup.rs:123-126`

**Issue:** The `Some(other)` catch-all arm prints `"valid: model, memory, gateway, tools"` but the match block actually handles six named sections: `model`, `memory`, `gateway`, `tools`, `agent`, and `skills` (the last as a deferred bail). A user typing `hermes setup agent` succeeds, but `hermes setup foo` produces a message that misleads them into thinking `agent` does not exist.

**Fix:**
```rust
Some(other) => anyhow::bail!(
    "unknown setup section: {} (valid: model, memory, gateway, tools, agent)",
    other
),
```

### WR-02: `apply_minimum_viable_answers` public seam still calls the deprecated `apply_api_key_answer`

**File:** `crates/ironhermes-cli/src/setup.rs:634-645`

**Issue:** CFG-01 intentionally removes the `apply_api_key_answer` call from `run_minimum_viable_flow` to stop writing the deprecated `model.api_key` field. However, the public testability seam `apply_minimum_viable_answers` (line 642) still calls `apply_api_key_answer`. Any test or external caller using `apply_minimum_viable_answers` will write the deprecated field, contradicting the CFG-01 invariant. The test `cfg_01_does_not_write_legacy_model_api_key` exercises the helpers directly and will pass, but a test using `apply_minimum_viable_answers` would silently produce the banned field. The seam's doc comment does not warn callers of this discrepancy.

**Fix:** Either remove the `apply_api_key_answer` call from `apply_minimum_viable_answers` to align the seam with the new flow, or add a prominent `// NOTE: this seam deliberately preserves the legacy behaviour for backward-compat tests ONLY` comment and consider whether any existing callers should be migrated to the new helpers.

### WR-03: D-19 ordering invariant test (`run_setup_appends_optional_tool_prereq_stage_d19`) fires on the first occurrence of `apply_minimum_viable_answers` in the source file, which is a doc comment at line 5 — not a call site

**File:** `crates/ironhermes-cli/src/setup.rs:789-797`

**Issue:** The test calls `source.find("apply_minimum_viable_answers")` which returns the byte offset of the **first** occurrence. The first occurrence is line 5 (the module doc comment: `"The `apply_minimum_viable_answers` function"`), not the function definition at line 634 or a call site inside `run_minimum_viable_flow`. Because the comment appears near the top of the file (byte offset ~200) and the D-19 prompt appears much later (~line 231 in the function body), the ordering assertion `opt_pos > apply_pos` is trivially satisfied for the wrong reason. If someone renames or removes the function but leaves the comment, the test still passes.

**Fix:** Use a search string that uniquely identifies the function definition or a genuine call site:
```rust
let apply_pos = source
    .find("pub fn apply_minimum_viable_answers(")
    .expect("apply_minimum_viable_answers function def must exist");
```

### WR-04: CFG-03 INV-03 amendment doc comment check uses a 1500-character lookback window that is brittle if the gate comment grows

**File:** `crates/ironhermes-cli/tests/invariants_26_4_1_cfg_03.rs:72-85`

**Issue:** The test looks back 1500 bytes before the gate pattern to verify that a `"Phase 26.4.1"` or `"CFG-03"` comment is present. The gate's comment block (lines 293–312 in main.rs) is currently ~900 bytes, so 1500 is safe today. However this is a magic constant with no margin documentation, and any refactor that adds prose to the comment block or moves the gate further down could silently push the comment outside the window, causing a false-negative (test passes even when the amendment comment is gone).

**Fix:** Either increase the window to a well-documented value (e.g. 3000) with a comment explaining why, or anchor the search by finding the start of the enclosing comment block rather than using a fixed lookback:
```rust
// Look for the comment start marker instead of a fixed offset
let comment_start = MAIN_RS[..gate_idx].rfind("// Phase 26.4.1")
    .or_else(|| MAIN_RS[..gate_idx].rfind("// CFG-03"))
    .expect("Phase 26.4.1 CFG-03: amendment doc comment must precede the gate");
let _ = comment_start; // presence check is sufficient
```

## Info

### IN-01: `apply_tool_prereq_answers` ignores the `tool_name` component of its input tuples

**File:** `crates/ironhermes-cli/src/setup.rs:622-630`

**Issue:** The function signature is `answers: &[(&str, &str, &str)]` documented as `(tool_name, prereq_name, value)`, but the loop destructures as `(_tool, prereq_name, value)` and discards `_tool`. If a caller intends the `Skip` behaviour (writing to `tools.skip_prompts`) to be driven by this seam, it cannot — the seam only writes env-vars. This is currently fine because tests only need the env-var write path, but it is misleading API surface.

**Fix:** Either document explicitly that `tool_name` is reserved for future use and `Skip` is not exercisable via this seam, or add a variant to handle the skip path.

### IN-02: `prompt_for_prereq_value` has unreachable `let _ = tool_name` suppression in the non-secret branch

**File:** `crates/ironhermes-cli/src/setup.rs:472`

**Issue:** `tool_name` is a parameter of `prompt_for_prereq_value` but is only used via `let _ = tool_name` in the non-secret branch to suppress an "unused variable" warning. The secret branch also does not use it. The parameter exists on the public-facing signature but has no purpose in either code path.

**Fix:** Remove the parameter from the function signature. All current call sites pass `tool_name` only to satisfy this parameter; removing it simplifies both the function and its callers.
