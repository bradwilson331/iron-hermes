# Phase 26.4.1: config fix - Context

**Gathered:** 2026-05-07
**Status:** Ready for planning
**Source:** User-reported bugs + direct code analysis

<domain>
## Phase Boundary

Fix three setup/config bugs discovered after Phase 26.4:

1. **CFG-01 — API key in config.yaml instead of .env**: `run_minimum_viable_flow` calls `apply_api_key_answer()` which writes the key to `config.model.api_key` (deprecated field in config.yaml). The canonical path is `.env` (as `{PROVIDER_UPPER}_API_KEY=value`) + `providers.{provider}.api_key_env = "{PROVIDER_UPPER}_API_KEY"` in config.yaml. Current behaviour causes a deprecation warning on every gateway run and means gateway never gets a `providers:` map entry for the provider it needs.

2. **CFG-02 — Auxiliary provider crash**: The wizard's optional auxiliary prompt (`Auxiliary provider (cheaper helper-task model — Enter to skip)`) accepts any freeform string. If the user types "local" (thinking it means skip or local LLM), `apply_auxiliary_answer()` saves it literally. `ProviderResolver::build()` then fails with `"auxiliary.provider 'local' is not a known provider — define it in providers: first"`. Fix: validate the entered provider name against the known providers list before saving; if unrecognised, warn and skip (don't persist it).

3. **CFG-03 — Gateway has no preflight guard**: `run_preflight` in `main.rs` only fires for `Chat` subcommand or bare invocation. `gateway` has no setup guard — if config.yaml is absent or invalid, gateway immediately errors instead of running the wizard.

</domain>

<decisions>
## Implementation Decisions

### CFG-01: API key → .env + providers map
- **Locked:** Change `run_minimum_viable_flow` to write the API key to `.env` using existing `write_env_var_to_dotenv(hermes_home, env_var_name, &api_key)`, where `env_var_name = format!("{}_API_KEY", provider.to_uppercase().replace('-', '_'))`.
- **Locked:** After writing to `.env`, call `config_setter::config_set(hermes_home, "providers.{provider}.api_key_env", &env_var_name)` to write the `providers:` map entry.
- **Locked:** Do NOT call `apply_api_key_answer(config, &api_key)` in the minimum-viable flow anymore (it writes the deprecated `model.api_key` field). Leave `apply_api_key_answer` as-is for `run_model_section` re-configuration (backward compat, fix-mode).
- **Locked:** The `config.validate()` check in `preflight.rs` already accepts EITHER `model.api_key` OR `providers.{provider}.api_key_env` — no change needed there.

### CFG-02: Auxiliary provider validation
- **Locked:** In `run_minimum_viable_flow`, after getting the `aux_provider` input, check it against known built-in provider names (`["openrouter", "anthropic", "openai", "google", "mistral", "groq"]`) before calling `apply_auxiliary_answer`. If the entered value is non-empty but not in the list, print a warning and skip saving: `"Unknown provider '{name}' — skipping auxiliary setup. Use 'hermes setup agent' to configure later."`.
- **Locked:** Same guard in `run_agent_section` (the standalone re-config path).
- **Claude's Discretion:** Whether the known-provider list is a static const or derives from config at runtime (static const is simpler, fine for now).

### CFG-03: Gateway preflight guard
- **Locked:** Extend the `run_preflight` gate in `main.rs` to also cover `Commands::Gateway { .. }`. New condition: `matches!(cli.command, Some(Commands::Chat { .. }) | Some(Commands::Gateway { .. }) | None) && cli.execute.is_none()`.
- **Locked:** The existing `preflight::run_preflight_check` logic (check config.yaml exists → load → validate → wizard if any fail) is correct; no changes needed there.

### Claude's Discretion
- Exact env var name derivation for providers with hyphens (e.g., `open-router` → `OPEN_ROUTER_API_KEY` or `OPENROUTER_API_KEY`) — prefer uppercase + replace hyphen with underscore
- Whether `run_model_section` (the re-config path) should also migrate to .env (out of scope for this phase — keep it simple)
- Test strategy for wizard integration tests (use existing `apply_minimum_viable_answers` testability seam pattern from `setup.rs`)

</decisions>
