<objective>
CFG-01: Fix the first-run wizard so the API key is persisted to `$HERMES_HOME/.env` (as `{PROVIDER}_API_KEY=value`) AND `$HERMES_HOME/config.yaml` is augmented with `providers.{provider}.api_key_env: "{PROVIDER}_API_KEY"` — instead of the current behaviour which writes the literal key to the deprecated `model.api_key` field.

Purpose: After this fix, a fresh `hermes setup` produces config that matches the canonical `providers:` map shape gateway/agent code expect, and `hermes gateway` no longer emits the `[provider:openrouter] using deprecated env var OPENROUTER_API_KEY — set providers.openrouter.api_key_env in config.yaml` warning on every launch.

Output: Modified `crates/ironhermes-cli/src/setup.rs::run_minimum_viable_flow` + accompanying unit tests verifying the new persistence shape.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator → wizard stdin | Operator types the API key at the rustyline prompt; the value crosses into our process and must NOT be echoed, logged, or persisted in cleartext outside the 0600-mode .env file. |
| wizard process → filesystem | The wizard writes both `.env` (secret value) and `config.yaml` (env-var-name reference, NOT the secret). Mode-0600 enforcement on `.env` is preserved by the existing `write_env_var_to_dotenv` helper. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-26.4.1-01 | Information Disclosure | `.env` file containing API key | mitigate | Reuse existing `write_env_var_to_dotenv` which sets mode 0600 atomically via tempfile + persist (unchanged from Phase 25). No new disclosure surface. |
| T-26.4.1-02 | Information Disclosure | stdout/stderr during wizard | mitigate | `write_env_var_to_dotenv` prints only `"  Saved."` — never echoes the value. We do NOT add any new println! that prints `api_key` content. |
| T-26.4.1-03 | Tampering | config.yaml `providers.{provider}.api_key_env` field | accept | The value written is a fixed-shape env-var name derived from the user-supplied provider string. Worst case the operator types a non-provider — gateway then fails to resolve the env var with a clear error. No injection vector: `config_setter::config_set` walks dotted keys safely; provider name is treated as a literal map key, not interpreted as YAML. |
| T-26.4.1-04 | Repudiation / Mis-config | Operator types provider with hyphens (e.g. `open-router`) | accept | Derivation `to_uppercase + replace('-', '_')` is documented and unit-tested; resulting env var is what gateway will look up. If the operator's actual provider key in `endpoints` config differs, gateway will surface a clear error at use-time. Out-of-scope to validate the provider string against a registry here (deferred per CONTEXT). |
| T-26.4.1-05 | Denial of Service | tempfile creation in HERMES_HOME | accept | Pre-existing surface (Phase 25) — unchanged by this plan. |
</threat_model>
