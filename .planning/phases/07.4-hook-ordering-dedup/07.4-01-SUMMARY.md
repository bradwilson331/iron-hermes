# Phase 07.4 Plan 01: Hook Ordering & Duplicate Event Fixes Summary

**One-liner:** Eliminated duplicate MessageReceived/ResponseSent events by making the platform layer the sole owner, and moved ToolCalled to fire post-guardrail via a new `dispatch_with_hook()` API.

## What Was Built

This plan closed two "warning" severity integration issues from the v1.1 audit:

### Issue #3 Fix — ToolCalled fires after guardrail check

**Problem:** `agent_loop.rs::execute_tool_call()` fired `ToolCalled` unconditionally before calling `registry.dispatch()`. Since guardrails run inside `dispatch()`, blocked tool calls still emitted `ToolCalled` events — incorrect observability.

**Fix:** Added `ToolRegistry::dispatch_with_hook()` method that accepts an optional `post_guardrail_hook: Option<F>` closure called only after ALL guardrails return `Allow` or `Warn`, immediately before `tool.execute()`. Updated `execute_tool_call` to use `dispatch_with_hook()`, moving the `ToolCalled` fire into the callback. Blocked tools now never emit `ToolCalled`.

The original `dispatch()` is preserved as a zero-overhead wrapper calling `dispatch_with_hook(name, args, None::<fn(&str, &str)>)` — no API breakage.

### Issue #4 Fix — Duplicate MessageReceived/ResponseSent events

**Problem:** `agent_loop.rs::run()` fired `MessageReceived` at loop start and `ResponseSent` at loop end. `handler.rs::run_agent()` also fired both events with real Telegram `platform`/`chat_id`. This produced 2x events per Telegram message — one with real metadata, one with empty `platform="agent"` and `chat_id=""`.

**Fix:** Removed both hook fires from `agent_loop.rs`. The platform layer (handler.rs for Telegram, runner.rs for cron) is the sole owner of these lifecycle events, as it has the real `platform` and `chat_id` values. Added comments in `agent_loop.rs` documenting why these are intentionally absent.

## Tests Added

Two new regression tests in `crates/ironhermes-gateway/src/runner.rs`:

1. **`test_telegram_roundtrip_produces_exactly_one_message_received_and_response_sent`** — asserts a canonical Telegram round-trip produces exactly 1 `MessageReceived` + 1 `ResponseSent` with correct platform metadata.

2. **`test_tool_called_not_emitted_for_blocked_tools`** — asserts:
   - Post-guardrail hook is NOT called when a tool is blocked
   - Post-guardrail hook IS called when a tool is allowed
   - Verifies the `dispatch_with_hook()` API contract directly
