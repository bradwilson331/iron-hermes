# Phase 07.4 Plan 02: Execute_tool_call D-05 Ordering Summary

**One-liner:** Rewrote execute_tool_call to use split guardrail API (check_guardrails + execute_tool) with D-05 check-before-fire ordering; blocked tools emit zero ToolCalled and exactly one ToolCompleted{success:false}.

## What Was Built

### Split API Added to ToolRegistry

Three new public methods added to `crates/ironhermes-tools/src/registry.rs`:

1. **`check_guardrails(name, args) -> GuardrailDecision`** — runs the full guardrail chain and returns the decision without executing the tool. Warn decisions are logged and returned (caller decides). Block decisions return immediately on first block.

2. **`execute_tool(name, args) -> anyhow::Result<String>`** — executes the tool WITHOUT running guardrails. Callers must check guardrails first. This is the execution-only half of the D-05 split.

3. **`guardrail_error_detail() -> &ErrorDetailLevel`** — exposes the registry's configured error detail level so callers can format block errors consistently with `format_guardrail_error`.

### execute_tool_call Rewritten (D-05 Ordering)

`agent_loop.rs::execute_tool_call` was rewritten from using `dispatch_with_hook()` to the explicit D-05 three-step ordering:

**Step 1 — Check guardrails first (no execution):**
```
let decision = self.registry.check_guardrails(name, &args);
```

**Step 2 — On Block:** fire `ToolCompleted{success:false}` ONLY. Never fire `ToolCalled`. Format the error via `format_guardrail_error` using the registry's `ErrorDetailLevel` so the LLM sees the same error string as pre-07.4. Return `duration_ms: 0` (no execution occurred).

**Step 3 — On Allow or Warn:** fire `ToolCalled` first, then call `execute_tool()`, then fire `ToolCompleted` with success/failure from the execution result. Warn counts as Allow for event firing (D-08).

The old `dispatch_with_hook()` call site is gone — zero uses of `self.registry.dispatch(` remain in `agent_loop.rs`.

### Deleted Duplicate Event Fires (already done in Wave 1)

The Wave 1 commit (`8dff834`) already removed `MessageReceived` and `ResponseSent` fires from `agent_loop.rs`. This plan's source-text regression test (`test_agent_loop_source_has_no_message_received_or_response_sent_fires`) guards against reintroduction.

## Tests Added

Five new tests in `crates/ironhermes-agent/src/agent_loop.rs` (`hooks_ordering_tests` module):

1. **`test_blocked_tool_no_tool_called_event`** — BlocklistGuardrail registered against "mock". Assert: 0 ToolCalled, 1 ToolCompleted{success:false} with block-reason text, returned tool_result contains block error.

2. **`test_allowed_tool_fires_tool_called_then_tool_completed`** — No guardrails. Assert: 2 events in order [ToolCalled, ToolCompleted{success:true}], result == "mock result".

3. **`test_warn_guardrail_fires_tool_called_and_tool_completed`** — WarnGuardrail registered. Assert: 2 events [ToolCalled, ToolCompleted{success:true}]. Proves D-08 (warn = allow for events).

4. **`test_allowed_tool_execution_failure_still_fires_tool_called`** — FailMockTool (returns Err). Assert: 2 events [ToolCalled, ToolCompleted{success:false}]. Pre-07.4 observability preserved for execution errors.

5. **`test_agent_loop_source_has_no_message_received_or_response_sent_fires`** — Uses `include_str!` + `concat!` to search for the fire_hook call patterns. Guards against future reintroduction of forbidden events. The `concat!` trick avoids the test text itself triggering the assertion (since the assertion strings would otherwise contain the forbidden patterns literally).
