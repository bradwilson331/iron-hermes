<decisions>
## Implementation Decisions

### Protocol scope
- **D-01:** MVP subset of Agent Protocol — core chat loop only: session create/list/get, send message, stream events, permission requests. Skip task endpoints, artifact management, agent discovery. Enough for a working VS Code agent.
- **D-02:** Use a jsonrpc crate (e.g., jsonrpsee, tower-jsonrpc) for spec-compliant JSON-RPC 2.0 framing, error codes, and batch support. Do not reuse the hand-rolled JSON-RPC in ironhermes-exec.
- **D-03:** Researcher must investigate the current Agent Protocol spec version at agentprotocol.ai and map CLI-03..08 requirements to specific protocol endpoints. Do not assume spec contents — fetch and verify.

### Session lifecycle
- **D-04:** 1:1 session mapping — each ACP session creates a corresponding StateStore session. Messages persist to SQLite. Reuses existing persistence, FTS5 search, session lineage, and export infrastructure.
- **D-05:** Fork = copy conversation. Creates a new session with a copy of all messages up to the fork point. Parent-child linked via session lineage (SESS-03 `parent_session_id`). New session diverges from there.
- **D-06:** Explicit cleanup only. Sessions persist until the editor explicitly removes them or user runs cleanup. No idle timeout. Editors manage their own lifecycle.
- **D-07:** Unlimited concurrent sessions. No hard cap. Each session is lightweight (AgentLoop + conversation state). Trust the editor to manage resources.

### Permission bridge
- **D-08:** Reuse BlocklistGuardrail from ironhermes-hooks to classify dangerous tool calls. If a call would be blocked/flagged by the guardrail, route it through the permission bridge instead of blocking outright.
- **D-09:** allow_always is per-session only. When the session ends, all permissions reset. No persistent permission grants without explicit config.
- **D-10:** Block indefinitely on unanswered permission requests. The agent turn pauses until the editor responds with approve/reject. No timeout. The editor owns the user-facing approval UX.

### Tool rendering
- **D-11:** Structured content by tool type. File tools → file path + content (editor diffs against local). Terminal → command + output. Web → markdown summary. Protocol carries type hints for editor rendering.
- **D-12:** Strip ANSI escape codes from terminal output before sending to editor. Clean content — the editor controls its own styling.
- **D-13:** File write/patch results send path + full new content. The editor diffs against its local copy. Simple for the adapter, leverages editor's native diff capability.

### Crate structure (from Phase 22 D-10)
- **D-14:** New crate `ironhermes-acp` — separate from CLI, clean dependency boundary. Mirrors `ironhermes-gateway` as a top-level consumer crate.

### Claude's Discretion
- Choice of specific jsonrpc crate (jsonrpsee vs alternatives) based on research findings
- Internal event bridge architecture (channels, callbacks, or trait objects)
- Whether the ACP binary is a separate binary target or a subcommand of the existing ironhermes CLI
- Error mapping between AgentLoop errors and JSON-RPC error codes
- CWD binding implementation (session metadata field vs constructor parameter)

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `AgentLoop` (ironhermes-agent) — Core agent loop with StreamCallback and ToolProgressCallback. ACP event bridge wraps these callbacks to emit protocol events.
- `StateStore` (ironhermes-state) — SQLite session persistence with CRUD, FTS5, lineage. ACP SessionManager delegates to this for all persistence.
- `BlocklistGuardrail` (ironhermes-hooks) — Dangerous command classification. ACP permission bridge reuses this to decide what needs editor approval.
- `ToolRegistry` (ironhermes-tools) — Tool registration and dispatch. ACP sessions get their own registry instance (same as gateway pattern).
- `GatewayRunner` / `PlatformAdapter` (ironhermes-gateway) — Architectural pattern for a top-level consumer crate wrapping AgentLoop with platform-specific I/O.
- `RpcServer` (ironhermes-exec) — JSON-RPC 2.0 request/response pattern (reference only — ACP uses a crate instead).

### Established Patterns
- Top-level consumer crates (CLI, gateway) each build their own ToolRegistry, wire AgentLoop, and manage sessions independently
- `Arc<ToolRegistry>` shared across agent turns within a session
- `StreamCallback` and `ToolProgressCallback` closures for real-time event forwarding
- `SessionStore` (gateway) as write-through cache over StateStore — ACP can follow same pattern or use StateStore directly

### Integration Points
- New crate `ironhermes-acp` in workspace `Cargo.toml`
- Binary entry point reads stdio, dispatches JSON-RPC methods to SessionManager
- SessionManager creates AgentLoop instances per session, wires callbacks to event bridge
- Event bridge serializes AgentLoop events to JSON-RPC notifications on stdout
- Permission bridge intercepts tool dispatch, sends JSON-RPC request to editor, blocks on response

</code_context>

<specifics>
## Specific Ideas

- The ACP adapter is architecturally a "headless gateway" — same AgentLoop + ToolRegistry + StateStore wiring as the gateway, but with JSON-RPC stdio I/O instead of platform adapters.
- The existing `ironhermes-exec/src/rpc_server.rs` is ~100 lines of hand-rolled JSON-RPC. Using a proper crate (per D-02) gets batch support, proper error codes, and notification support for free — important for the streaming event bridge.
- Fork (D-05) leverages SESS-03's `parent_session_id` field — the StateStore schema already supports session lineage.
- Permission bridge (D-08) sits between AgentLoop's tool dispatch and the actual Tool::execute() call — similar to how BlocklistGuardrail already intercepts in agent_loop.rs lines 772-787.

</specifics>
