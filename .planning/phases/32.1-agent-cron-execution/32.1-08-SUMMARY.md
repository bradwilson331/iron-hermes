# Phase 32.1 Plan 08: Tick Idle Label Fix Summary

## One-liner

Renamed `TickResult.jobs_skipped` to `jobs_idle` with clarifying doc comment; updated `cmd_tick` println from `{} errors` to `{} idle` to prevent operator alarm on healthy ticks.

## What Was Built

This plan closed the cosmetic UX gap recorded in 32.1-HUMAN-UAT.md §Gaps: the `cmd_tick` summary line printed "errors" for what was actually a count of enabled-but-not-due jobs. Observed during UAT Test 3: `0 due, 0 ran, 12 errors` actually meant "12 enabled jobs that weren't due this tick, with zero actual failures."

### Changes Made

**`crates/ironhermes-cron/src/tick.rs`:**
- Renamed `pub jobs_skipped: usize` → `pub jobs_idle: usize` in `TickResult` struct
- Added doc comment: "Number of enabled jobs that were NOT due this tick. This is `jobs_checked - jobs_run`; it is NOT a failure count."
- Updated early-return branch: `jobs_skipped: 0` → `jobs_idle: 0`
- Updated calculation block: `let jobs_skipped = ...` → `let jobs_idle = ...`; struct init updated accordingly
- Added regression test `tick_result_idle_counts_enabled_but_not_due` that constructs a `TickResult` and asserts `jobs_idle == jobs_checked - jobs_run`

**`crates/ironhermes-cli/src/cron.rs`:**
- Updated `cmd_tick` println: `"Tick complete. {} due, {} ran, {} errors."` → `"Tick complete. {} due, {} ran, {} idle."`
- Updated field reference: `tick_result.jobs_skipped` → `tick_result.jobs_idle`

### Decision: `jobs_idle` vs `jobs_not_due`

Chose **`jobs_idle`** for brevity. The Python `scheduler.py::tick` equivalent uses idle phrasing in its summary log. The println token `idle` is concise and unambiguous — it clearly describes jobs that were present but had nothing to do this tick.

### Final println String

```
Tick complete. {} due, {} ran, {} idle.
```
