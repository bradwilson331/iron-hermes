## Decisions by Area

### Area 1: Scope & deferral line

**Question:** Where do we draw the 32.1 / follow-up line, given Agent execution is now in scope?

**Options presented:**
1. Full Python parity in 32.1 — everything in PARITY.md §10 + §11.
2. Behavioral parity, defer per-job runtime knobs.
3. Minimum useful execution polish.

**Chosen:** Option 1 — Full Python parity.

**Implication:** ~8–12 plans. The executor's first plan must amend PARITY.md §10 to reflect the scope shift (move §10 items into §11's gap inventory).

### Area 2: Where the runner lives

**Question 1:** Crate location for `execute_cron_job` (currently `ironhermes-gateway::runner`)?

**Options presented:**
1. Extract to new `ironhermes-cron-runner` crate.
2. Keep in `ironhermes-gateway`.
3. Fold into `ironhermes-cron`.

**Chosen:** Option 1 — new `ironhermes-cron-runner` crate.

**Rationale:** Decouples cron execution from the gateway crate; gives CLI standalone tick a shared code path; avoids forcing `ironhermes-cron` (a scheduling library) to depend on `ironhermes-agent`/`ironhermes-tools`/`ironhermes-hooks`.

**Question 2:** Should CLI run cron jobs standalone (without the gateway daemon)?

**Options presented:**
1. Yes — `hermes cron tick` + `hermes cron daemon`.
2. No — gateway-only.

**Chosen:** Option 1.

**Implication:** Tick file-lock (`ironhermes-cron::acquire_tick_lock`) already prevents overlap across gateway + `tick` + `daemon` running against the same `HERMES_HOME`.

### Area 3: Delivery routing parity

**Question 1:** Multi-target delivery (comma-split, `all` token, home-channel fallback, origin-fallback)?

**Options presented:**
1. Full parity — comma-split + `all` + home-channel + origin-fallback.
2. Multi-target + home-channel, defer `all`.
3. Multi-target only.

**Chosen:** Option 1 — full parity.

**Implication:** `resolve_delivery_target` becomes `resolve_delivery_targets -> Vec<DeliveryTarget>`. `_KNOWN_DELIVERY_PLATFORMS` allowlist added. Legacy env-var fallback (`QQBOT_HOME_CHANNEL` → `QQ_HOME_CHANNEL`) included. Thread-id env vars included.

**Question 2:** `[SILENT]` matching semantics (Python `in upper(content)` vs Rust `starts_with` prefix)?

**Options presented:**
1. Align with Python — substring `in` match.
2. Keep Rust strict — prefix-only.

**Chosen:** Option 2 — keep Rust strict.

**Rationale:** Lower false-positive risk; the system prompt instructs the agent to respond with *exactly* `[SILENT]` so prefix matching is sufficient.

### Area 4: Timeout & resilience

**Question 1:** Cron job timeout mechanism?

**Options presented:**
1. Inactivity-tracked (Python parity).
2. Wall-clock timeout via `tokio::time::timeout`.
3. Both — inactivity primary, wall-clock as hard ceiling.

**Chosen:** Option 3 — both.

**Implication:** `AgentLoop` gains an activity tracker API (`activity_summary()`). Cron runner polls every 5s; idle > `IRONHERMES_CRON_TIMEOUT` (default 600s) ⇒ `agent.interrupt()`. Outer `tokio::time::timeout` at `IRONHERMES_CRON_WALL_TIMEOUT_SECS` (default 4h).

**Question 2:** `JobState::Error` variant for recurring jobs whose `compute_next_run` returns `None`?

**Options presented:**
1. Add `JobState::Error` + leave-enabled semantics.
2. Keep current behavior — recurring jobs auto-complete.

**Chosen:** Option 2 — keep current.

**Rationale:** The Rust `cron` crate always succeeds for syntactically valid expressions, so Python's #16265 failure mode (missing-croniter ⇒ silent disable) has no analogue. If a Rust-side failure mode emerges that needs leave-enabled-but-error semantics, add the variant then.
