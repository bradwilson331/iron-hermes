# Phase 32.1 Plan 02: Activity Tracker API on AgentLoop Summary

Added the activity-tracking API and interruption hook to `AgentLoop` so the upcoming `ironhermes-cron-runner` crate can implement the inactivity-tracked timeout: `activity_summary()` returns a live snapshot of the most-recent LLM/tool/stream event with sub-ms latency, and `interrupt(reason)` cancels the agent's CancellationToken for a clean early return.

## What Was Built

### Task 1 — Types and API

Added to `crates/ironhermes-agent/src/agent_loop.rs`:

- `pub enum ActivityKind { ApiCall, ToolCall, StreamToken }` — Copy+Clone+Debug+PartialEq+Eq
- `pub struct ActivitySummary { seconds_since: f64, last_kind: ActivityKind, current_tool: Option<String> }` — Clone+Debug
- Three `Arc<std::sync::Mutex<>>` tracker fields on `AgentLoop` struct (mirrors `active_skills` pattern):
  - `activity_last: Arc<std::sync::Mutex<std::time::Instant>>`
  - `activity_kind: Arc<std::sync::Mutex<ActivityKind>>`
  - `current_tool: Arc<std::sync::Mutex<Option<String>>>`
- Fields initialized in both `AgentLoop::new()` (with ApiCall sentinel) and `for_tests()`
- `pub fn activity_summary(&self) -> ActivitySummary` — non-async, reads all three fields
- `pub fn interrupt(&self, reason: &str)` — calls `cancel_token.cancel()` if Some, warns if None
- `pub fn is_cancelled(&self) -> bool` — delegates to `cancel_token.is_cancelled()`, false if None
- `fn mark_activity(&self, kind: ActivityKind, current_tool: Option<String>)` — private internal bump
- `#[cfg(test)] pub fn mark_activity_for_test(...)` — test-only shim

Test module `activity_tracker_tests` with 5 tests all passing.

### Task 2 — Wiring

Wired `self.mark_activity(...)` at three observable sites:

1. **API-call site** (`run()`, before retry loop): `mark_activity(ApiCall, None)` — before the `if let Some(ref token) = self.cancel_token` LLM dispatch block. Placed outside the retry loop so each attempt bumps the tracker.

2. **Stream-delta site** (`call_llm_streaming()`, ContentDelta arm): `mark_activity(StreamToken, None)` — one bump per delta event, before the stream callback. Not per-byte.

3. **Tool-call site** (`execute_tool_call()`, Allow/Warn branch): `mark_activity(ToolCall, Some(name.clone()))` immediately before `execute_tool()`, then `mark_activity(ToolCall, None)` immediately after the await returns (post-dispatch reset). This clears `current_tool` so `activity_summary()` reflects finished-dispatch state for both Ok and Err paths (both flow through the same `match dispatch_result`).

Test module `activity_tracker_run_wiring` with 6 tests (4 source-text assertions + 2 behavioral):
- Source assertions verify all three `ActivityKind` variants appear at wiring sites
- `execute_tool_call_bumps_activity_tracker`: behavioral test calls `execute_tool_call` directly and asserts post-dispatch state
- `activity_summary_seconds_since_bounded_after_bump`: verifies seconds_since is < 5.0 after a fresh bump

## Key Output Facts (for CONTEXT.md §Output)

- **Cancellation token field name:** `cancel_token: Option<CancellationToken>` (the field is optional — `interrupt()` is a no-op + logs warn when None)
- **CancellationToken shape:** `Option<CancellationToken>` — NOT a plain non-Optional field. This plan did NOT add a new field; it uses the existing `cancel_token` field set via `with_cancellation_token()`.
- **Streaming-delta site choice:** per-delta (one `mark_activity` call per `StreamEvent::ContentDelta` event). Line: in `call_llm_streaming()` inside the `ContentDelta` match arm, before the stream callback invocation.
- **Deferred items:** None. All three mark_activity sites are wired. The `parking_lot::Mutex` optimization (T-32.1.02-01 note in threat model) was not pursued — `std::sync::Mutex` matches the existing `active_skills` pattern and is sufficient for sub-microsecond critical sections.
