<objective>
Close the cosmetic UX gap recorded in `32.1-HUMAN-UAT.md` ┬¦Gaps: the `cmd_tick` summary line prints "errors" for what is actually a count of enabled-but-not-due jobs. Observed during UAT Test 3: `0 due, 0 ran, 12 errors` actually meant "12 enabled jobs that weren't due this tick, with zero actual failures."

Purpose: prevent operator alarm when a healthy tick lists "errors". The underlying tick logic is correct ŌĆö only the label is misleading. This is a pure rename + label fix; no behavior change.

Output:
- `TickResult.jobs_skipped` renamed to `jobs_idle` (or `jobs_not_due` ŌĆö see Decision below) with a clarifying doc comment.
- `cmd_tick` println updated to read the renamed field and print `idle` (or `not-due`) instead of `errors`.
- All call sites of the renamed field updated; workspace compiles clean.

**Decision (executor): `jobs_idle` vs `jobs_not_due`.** Prefer `jobs_idle` for brevity (Python `scheduler.py::tick` uses the equivalent phrasing in its summary log). The println token in the user-facing line should match the field's intent: `"X idle"`. If `idle` reads ambiguously in the doc comment, fall back to `jobs_not_due` and `"X not-due"` ŌĆö but commit to ONE choice consistently across both files.
</objective>
