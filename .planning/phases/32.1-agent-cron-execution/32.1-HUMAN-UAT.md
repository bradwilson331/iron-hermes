## Gaps

### UX: misleading `cron tick` summary line label (RESOLVED — Plan 32.1-08)
The third number in `Tick complete. {due} due, {ran} ran, {errors} errors.`
is sourced from `tick_result.jobs_skipped = total_enabled - jobs_run`, which
counts enabled-but-not-due jobs, NOT errors. Observed during UAT Test 3:
`0 due, 0 ran, 12 errors` simply meant 12 enabled jobs that weren't due this
tick, with zero actual failures. Rename the field to `idle` or `not-due` in
`cmd_tick`'s println and update the underlying `TickResult` doc comment.
Non-blocking — purely a CLI display issue; the underlying tick logic is
correct.

**Resolved by Plan 32.1-08** (commits c0b1d449, bf7cec18): `TickResult.jobs_skipped`
renamed to `jobs_idle` with explicit doc comment that it is NOT a failure
count. `cmd_tick` now prints `{} idle`. Regression test
`tick_result_idle_counts_enabled_but_not_due` locks the field name + semantics.

### Regression: telegram home-channel resolution lost the config.yaml whitelist fallback (RESOLVED — Plan 32.1-09)
`resolve_delivery_targets` (Plan 04) reads `TELEGRAM_HOME_CHANNEL` from env
vars only. Pre-32.1 gateway also used the single entry of
`config.yaml::gateway.telegram.whitelist` as the implicit home channel — the
`cron create --deliver` help text still documents this behavior:
"defaults to 'origin' routing to the configured Telegram chat if the
gateway has exactly one authorized chat in config.yaml's whitelist".
Observed during UAT Test 5: `deliver=telegram` with config.yaml whitelist
configured but `TELEGRAM_HOME_CHANNEL` unset emits
`WARN deliver=telegram has no home channel configured platform=telegram`
and no delivery occurs.

Fix path (gap-closure plan): in `resolve_delivery_targets` / `lookup_home_channel`
(crates/ironhermes-cron/src/delivery.rs), after the `<PLATFORM>_HOME_CHANNEL`
env-var lookup returns None for telegram, fall back to
`Config::load()?.gateway.telegram.whitelist` and use the single whitelist
entry if `whitelist.len() == 1`. Cron-runner already depends on
ironhermes-core::Config (used via build_cron_runner_ctx), so the fallback
adds no new dependency.

Severity: regression but workaroundable by `export TELEGRAM_HOME_CHANNEL=<id>`.

**Resolved by Plan 32.1-09** (commits 05d827da, f7d54dec): added
`lookup_telegram_whitelist_fallback()` helper in
`crates/ironhermes-cron/src/delivery.rs`. `lookup_home_channel` falls back
to `Config::gateway.platforms.telegram.whitelist` when env var is unset
AND platform is "telegram" AND whitelist has exactly one entry. Env var
still wins when set; non-telegram platforms unaffected. 12 new
`telegram_whitelist_fallback_tests` cover env-wins, single-entry success,
zero/multi-entry unchanged, telegram disabled, thread_id env preserved,
explicit chat_id bypass, origin routing precedence. Additionally fixed
3 pre-existing tests that were leaking into developer's real
`~/.ironhermes/config.yaml` (now properly isolated via `IRONHERMES_HOME`
tempdir).
