<objective>
Add the activity-tracking API and interruption hook to
`crates/ironhermes-agent/src/agent_loop.rs` so the upcoming
`ironhermes-cron-runner` crate can implement the inactivity-tracked timeout
locked in CONTEXT.md §Timeout & resilience. The tracker observes three events
inside `AgentLoop::run`: every LLM API call, every tool dispatch, and every
streamed token. The `interrupt` method gives the cron runner a way to
unblock a runaway agent without panicking the task.

Purpose: the cron runner needs `agent.activity_summary()` and
`agent.interrupt(reason)` to wrap `AgentLoop::run` in a `tokio::select!`
inactivity-poller. RESEARCH.md Pattern 3 and CONTEXT.md Specific Ideas §3
both depend on this API. Phase 32.1 Plan 05 cannot start without it. By
landing it in Wave 1 (parallel with Plan 01's `CronJob` field additions),
we keep the dependency graph flat and the runner crate (Plan 05–06)
unblocked.

Output: an extended `AgentLoop` with append-only public methods, no
behavioral change to existing call sites (gateway, CLI, tests), and a
test suite that proves the three observable events bump the tracker.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `AgentLoop` internal state ↔ external pollers | The cron runner will read `activity_summary()` from a separate tokio task while `run()` is in flight; the `std::sync::Mutex` guards prevent torn reads but block the awaiting task briefly |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-32.1.02-01 | Denial of Service | `Mutex::lock()` contention in `mark_activity` on hot streaming path | mitigate | `std::sync::Mutex` is fine for sub-microsecond critical sections (3 stores). `Arc<Mutex<...>>` matches the existing `active_skills` pattern from `with_active_skills` — proven non-pathological in production. If `mark_activity` shows up on a flamegraph later, swap to `parking_lot` or `AtomicU64` storing `Instant::now().elapsed().as_nanos() as u64`. Out of scope here. |
| T-32.1.02-02 | Tampering | Public `interrupt(reason)` could be called by malicious callers in the gateway | accept | Cancelling an already-cancelled `CancellationToken` is a no-op; the only effect of unauthorized `interrupt` is graceful shutdown — not data exfiltration or corruption. The risk is acceptable. |
| T-32.1.02-03 | Information Disclosure | `activity_summary().current_tool` exposes the in-flight tool name to any caller that has `&AgentLoop` | accept | `&AgentLoop` callers are already trusted in-process code (gateway, CLI). Tool names are not secrets. |
| T-32.1.02-SC | Tampering | npm/pip/cargo installs | mitigate | Zero new crates; only existing `tokio_util::sync::CancellationToken` (already imported per `grep`). No supply-chain footprint. |
</threat_model>
