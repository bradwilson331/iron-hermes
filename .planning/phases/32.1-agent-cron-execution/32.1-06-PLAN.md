<objective>
Land the three orchestration modules of the `ironhermes-cron-runner` crate:
`runner.rs` (the `run_cron_job` entry point + `CronRunnerContext`),
`delivery.rs` (multi-target dispatch loop + wrap_response + MEDIA tag
routing + per-target error accumulation), and `tick_loop.rs` (the 60s
tick scaffold the gateway and CLI daemon share). The leaves from Plans 05a + 05b
(`script_runner`, `prompt_builder`, `timeout`) plus the cron crate's
already-hardened `resolve_delivery_targets`, `complete_job_run`,
`trigger_job`, and storage primitives are the dependencies — none of the
inter-task communication leaves the runner crate.

Purpose: this plan is the agent-execution porting work that PARITY.md §10
documents. After it lands, the gateway still uses its own
`execute_cron_job` (deletion happens in Plan 07) but the new
`run_cron_job` is an equivalent — with full Python parity — that any
caller can switch to.

Output: a complete runner crate with end-to-end execution from "tick fires"
to "delivery dispatched, store updated, errors accumulated".

**Note on Telegram-only `tg_client` (§Claude's Discretion):** This plan ships
`CronRunnerContext.tg_client: Option<Arc<dyn TgSendApi>>` — a Telegram-only
adapter type — intentionally. CONTEXT.md §Live-adapter mentions a future
platform-generic `AdapterRegistry` abstraction, but for Phase 32.1 the
gateway only has a working Telegram path. `dispatch_all_targets` therefore
handles the `"telegram"` arm and returns `"unsupported platform"` errors
for every other platform name resolved by Plan 04's multi-target router.
The generic `AdapterRegistry` is **deferred** to a follow-up phase once a
second platform's send adapter exists (Discord, Matrix, etc.). When that
phase lands, swap `tg_client: Option<Arc<dyn TgSendApi>>` for
`adapter_registry: Option<Arc<dyn AdapterRegistry>>` and replace the
`"telegram"` match arm with a registry lookup.
</objective>
