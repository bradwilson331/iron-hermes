## Implementation Notes

### grace_seconds field on JobStore
The `pub grace_seconds: i64` field was present in the original store.rs (line 99, initialized to `3600`). It has been **removed**. `get_due_jobs` now calls `compute_grace_seconds(&job.schedule)` per job. There is one pre-existing test (`get_due_jobs_fast_forwards_stale_jobs`) that backdated a job by 7200s — it continues to pass because the 60-min interval has a dynamic grace of 1800s, so 7200s is well beyond stale.

### Control-char repair match arm (for Plan 04 reference)
The detection condition used in `parse_jobs_with_repair`:
```rust
fn raw_has_bare_ctrl_chars(raw: &str) -> bool {
    raw.bytes().any(|b| b < 0x20 && !matches!(b, b'\n' | b'\r' | b'\t'))
}
```
This is the canonical check. Plan 04 can mirror this exact helper for `save_job_output` recovery. The repair strips characters with `(c as u32) < 0x20 && !matches!(c, '\n' | '\r' | '\t')` → replaced with space.

### compute_next_run external call sites
`compute_next_run` (no `_from`) is called in these files outside parser.rs:
- `store.rs::add_job` (line ~225) — `compute_next_run(&schedule, now)`
- `store.rs::update_job` (line ~296) — `compute_next_run(&schedule, now)`
- `store.rs::toggle_job` (line ~394) — `compute_next_run(&job.schedule, now)`
- `store.rs::mark_job_run` (line ~419) — `compute_next_run(&job.schedule, now)`
- `tick.rs` (at least one call site for complete_job_run)

All continue to compile via the thin wrapper. Gradual migration to `compute_next_run_from` can happen in later plans.

### jobs_mut() confirmed shipped
`pub fn jobs_mut(&mut self) -> &mut Vec<CronJob>` is present at line 380 of store.rs. Plan 06 Task 2 (`complete_and_dispatch`) can call `store.jobs_mut()[idx].last_delivery_error = Some(...)` followed by `store.save()`.

## Deviations from Plan

**1. [Rule 2 - Missing critical functionality] `parse_jobs_with_repair` applied to legacy migration write paths too**
- **Found during:** Task 2 implementation
- **Issue:** Plan specified repair only on the primary parse path, but the legacy migration write in both `open()` and `reload()` lacked `sync_all()`. Adding it here was a correctness requirement (PARITY §11.18 applies to all writes, not just `save()`).
- **Fix:** Added `f.sync_all()?` and chmod 0600 to both the `open()` legacy-migration write and `reload()` legacy-migration write, mirroring `save()` exactly.
- **Files modified:** store.rs

**2. [Rule 3 - Blocking] ParseResult enum to resolve borrow-checker conflict**
- **Found during:** Task 2 GREEN phase
- **Issue:** Inline `if let Ok(jobs)` chains in `open()` and `reload()` required repeating the control-char fallback. Extracting to `parse_jobs_with_repair()` with a `ParseResult` enum eliminated duplication and resolved the borrow-checker conflict between mutable `path` capture and the existing format-fallback ladder.
- **Fix:** Created private `enum ParseResult { NewFormat, Legacy, Empty }` and `fn parse_jobs_with_repair()`.
- **Files modified:** store.rs

All other changes implemented exactly as specified in the plan.
