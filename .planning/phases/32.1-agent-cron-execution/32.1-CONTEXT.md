# Phase 32.1: Agent cron execution — Context

**Gathered:** 2026-05-15
**Status:** Ready for planning
**Source:** /gsd:discuss-phase 32.1

<domain>
## Phase Boundary

Port the **agent-runner side** of Python `hermes-agent/cron/scheduler.py::run_job`
(approximately 600 LOC, lines 1013–1656 of `scheduler.py`) into the IronHermes
Rust workspace at full feature parity, and close all incidental parity gaps
catalogued in `crates/ironhermes-cron/PARITY.md` §11.

What this phase delivers:

- A new `ironhermes-cron-runner` crate that owns end-to-end execution of a due
  `CronJob`: prompt assembly, script execution, agent loop invocation, output
  capture, delivery dispatch.
- `hermes cron tick` (single-shot) and `hermes cron daemon` (long-running) CLI
  commands so cron can run without the Telegram gateway.
- Full parity with Python's `run_job` behavior: script + `no_agent` short-
  circuit, wake-gate JSON parsing, `context_from` chain injection, assembled-
  prompt threat rescan, per-job model/provider/toolsets/workdir overrides, MCP
  tool discovery per tick, inactivity-tracked timeout with a wall-clock
  ceiling, live-adapter delivery with standalone fallback, MEDIA tag routing,
  multi-target delivery (comma-separated + `all` token + home-channel
  fallback), `wrap_response` header/footer, and `last_delivery_error`
  tracking.
- All incidental parity gaps in `crates/ironhermes-cron/PARITY.md` §11 closed
  except #2 `JobState::Error` (see Decisions below).

What this phase does **not** deliver:

- New cron *features* beyond Python parity. The reference implementation is
  authoritative; novel behavior belongs in a later phase.
- Removal of the existing `ironhermes-gateway::runner::execute_cron_job` until
  the new crate has feature parity and tests pass. The migration is
  call-site-by-call-site; both implementations coexist during transition.

</domain>

<decisions>
## Implementation Decisions

### Scope

- **Full Python parity is in scope.** Both PARITY.md §10 (Agent execution —
  previously marked out-of-scope) and PARITY.md §11 (incidental gaps) are
  pulled into 32.1. The executor's first plan must amend PARITY.md §10 to
  reflect this scope shift.
- `JobState::Error` variant is **deferred**. Rationale: the Rust `cron` crate
  always succeeds for syntactically valid expressions, so Python's
  `compute_next_run`-returns-`None`-for-recurring failure mode (#16265) has no
  analogue in the Rust port. The existing `JobState::Completed` path on
  `next_run_at = None` is the correct behavior here. If a future failure mode
  emerges that needs leave-enabled-but-error semantics, add the variant then.

### Crate layout

- A **new workspace crate** `ironhermes-cron-runner` owns agent execution.
  Justification:
  - Decouples cron-job execution from the gateway crate so CLI standalone
    tick mode can share the same code path.
  - Co-locating with `ironhermes-cron` would force `ironhermes-cron` to depend
    on `ironhermes-agent` + `ironhermes-tools` + `ironhermes-hooks`, thickening
    the dep graph for any consumer that only needs CRUD.
  - Mirrors Python's `cron/scheduler.py` vs `cron/jobs.py` separation.
- The new crate depends on: `ironhermes-cron` (job model + storage + tick
  lock), `ironhermes-core`, `ironhermes-agent` (`AgentLoop`, `PromptBuilder`),
  `ironhermes-tools` (`ToolRegistry`), `ironhermes-hooks` (`HookRegistry`),
  `ironhermes-skills`/whatever crate holds `SkillRegistry`,
  `ironhermes-state` if needed for `SessionDB` parity.
- `ironhermes-gateway::runner::execute_cron_job` and friends move into the
  new crate; gateway calls the new crate's public `run_cron_job(...)` entry
  point. Gateway retains the tick-task plumbing (interval, cancel token,
  fast-forward-on-first-tick guard, MCP orphan reaper).

### CLI standalone execution

- Add **`hermes cron tick`** — runs `acquire_tick_lock()` + `run_tick_check`
  + `run_cron_job` for each due job once, then exits. Useful for cron-in-cron
  setups (system cron invokes `hermes cron tick` every minute).
- Add **`hermes cron daemon`** — long-running tokio process that ticks every
  60s using the same loop the gateway runs. Useful for headless deployments
  without a Telegram bot.
- The existing tick file-lock (`~/.hermes/cron/.tick.lock`, PID-based stale
  recovery in `ironhermes-cron::lib::acquire_tick_lock`) already prevents
  overlap between gateway, `tick`, and `daemon` running on the same
  `HERMES_HOME`.
- CRUD commands (`hermes cron list/create/edit/remove/show/pause/resume`)
  already exist and need no changes for this phase.

### Delivery routing (PARITY.md §7)

- **Full multi-target parity.** `resolve_delivery_target` becomes
  `resolve_delivery_targets` returning `Vec<DeliveryTarget>`. The current
  single-target call site (`dispatch_delivery`) becomes a per-target loop
  with per-target error accumulation written into `last_delivery_error`.
- **Comma-split** the `deliver` field: `"telegram,discord"` ⇒ two targets.
- **`all` routing-intent token** expands to every platform that has a
  configured home-channel env var. Mirrors Python `_expand_routing_tokens`.
- **`deliver=platform` without colon** resolves to that platform's
  `<PLATFORM>_HOME_CHANNEL` env var (table mirrors
  `_HOME_TARGET_ENV_VARS`). Empty/unset ⇒ skip that target with a warning.
- **`deliver=origin` fallback**: when origin is missing, iterate home channels
  and use the first configured one. Mirrors Python's fallback in
  `_resolve_single_delivery_target`.
- **`_KNOWN_DELIVERY_PLATFORMS`** allowlist: a `const &[&str]` in the runner
  crate. Validates platform names before any env-var lookup to prevent
  env-var enumeration via crafted job records.
- **Legacy env-var fallback** (`QQBOT_HOME_CHANNEL` → `QQ_HOME_CHANNEL`):
  table-driven; mirror Python `_LEGACY_HOME_TARGET_ENV_VARS`.
- **Thread/topic env vars** (`<X>_HOME_CHANNEL_THREAD_ID`): supported.
- **Defensive `JobOrigin` parsing**: a malformed origin in `jobs.json`
  (non-object, missing platform/chat_id) should be treated as missing — not
  surface as a serde-fail and break the whole job. Add `#[serde(default)]`
  + a normalizer in the cron crate's deserializer.

### `[SILENT]` semantics

- **Keep Rust strict — `is_silent` stays `starts_with("[SILENT]")` after
  trim + uppercase.** Rationale: the cron-hint banner in the system prompt
  instructs the agent to respond with *exactly* `[SILENT]`. A substring
  match risks suppressing delivery when the agent legitimately discusses
  the marker (e.g., explaining cron behavior to the user). Strict matching
  is the safer default; if false-negatives appear in practice, revisit.

### Timeout & resilience

- **Both inactivity AND wall-clock** timeouts.
  - **Inactivity timeout (primary).** Add an activity tracker to `AgentLoop`
    in `ironhermes-agent`: timestamps for last-tool-call, last-stream-token,
    last-api-call. Cron runner polls every 5s; if idle exceeds
    `IRONHERMES_CRON_TIMEOUT` seconds (default 600), it calls
    `agent.interrupt(reason)` and the run resolves as an error.
  - **Wall-clock ceiling (secondary).** Wrap the run in
    `tokio::time::timeout` at a generous ceiling (default 4h, configurable
    via `IRONHERMES_CRON_WALL_TIMEOUT_SECS`). Catches edge cases where the
    activity tracker stays warm but the job has gone runaway.
  - `0` ⇒ unlimited for both env vars (matches Python convention).
- **Empty `final_response` ⇒ soft failure.** Match Python issue #8585: agent
  ran but produced no output; mark `last_status="error"` and
  `last_error="Agent completed but produced empty response"`. The strict
  `[SILENT]` semantics above mean a truly silent run uses `[SILENT]` and
  doesn't pass this gate.

### Per-job runtime overrides (PARITY.md §3)

- Add to `CronJob` (in `ironhermes-cron::job`): `model: Option<String>`,
  `provider: Option<String>`, `base_url: Option<String>`, `script:
  Option<String>`, `no_agent: bool` (default false), `context_from:
  Option<Vec<String>>`, `enabled_toolsets: Option<Vec<String>>`, `workdir:
  Option<String>`, `last_delivery_error: Option<String>`.
- Each field is `#[serde(default)]` so existing `jobs.json` records remain
  loadable.
- Cron-runner crate threads these into `AgentLoop` via the same constructor
  surface used by the gateway main path (model/provider/base_url ⇒
  `ProviderResolver`, `enabled_toolsets` ⇒ scoped tool registry, `workdir`
  ⇒ `TERMINAL_CWD` env var for the duration of the job).
- **Workdir jobs serialize.** `TERMINAL_CWD` is a process-global. The
  tick-task partitions due jobs into workdir vs non-workdir; workdir jobs
  run sequentially in a single tokio task, non-workdir jobs use
  `JoinSet` with a configurable concurrency limit
  (`IRONHERMES_CRON_MAX_PARALLEL`, default unbounded; set to `1` for
  Python-equivalent serial behavior).

### Script execution (`script`, `no_agent`, wake-gate)

- New module `script_runner.rs` in `ironhermes-cron-runner`.
- Scripts MUST resolve inside `${HERMES_HOME}/scripts/`. Use
  `Path::canonicalize` + `starts_with(scripts_dir)` to reject path traversal
  and absolute-path injection. Empty/missing dir ⇒ create with `0700`.
- Interpreter selection by extension: `.sh` / `.bash` ⇒
  `which("bash").unwrap_or("/bin/bash")`; everything else ⇒
  `which("python3")`. On Windows fall back gracefully with a clear error
  (Git Bash for `.sh`, or rewrite as `.py`).
- Per-script timeout: `IRONHERMES_CRON_SCRIPT_TIMEOUT` env var; default 120s.
- Secret redaction on stdout + stderr before any return path. Reuse
  whatever the project's existing redactor is (search `ironhermes-core` or
  the trajectory crate for the equivalent of Python's
  `agent.redact.redact_sensitive_text`); add a TODO if none exists yet.
- **`no_agent=True` short-circuit.** If set, the script's stdout IS the
  job — delivered verbatim, no LLM. Empty stdout ⇒ silent (no delivery,
  `last_status="ok"`). Non-zero exit / timeout ⇒ alert delivery with the
  formatted error doc.
- **Wake-gate** (`{"wakeAgent": false}` on the last non-empty stdout line of
  the pre-run script): parse it; if false, treat as silent — no agent run,
  no delivery, `last_status="ok"`.
- **`script` without `no_agent`**: stdout is injected into the prompt as a
  `## Script Output` block before the user prompt. Empty stdout ⇒ skip
  the agent run entirely (matches Python `_build_job_prompt` returning
  `None`).

### Prompt assembly (`build_job_prompt`)

- Order must match Python `_build_job_prompt`:
  1. Cron-hint banner (`[IMPORTANT: You are running as a scheduled cron
     job...]` — verbatim text from `scheduler.py` lines 921–932).
  2. Skill content (per-skill, in order; each loaded skill prefixed with
     `[IMPORTANT: The user has invoked the "{name}" skill...]`).
  3. `## Script Output` block from the pre-run script (when applicable).
  4. `## Output from job '{source_job_id}'` block for each `context_from`
     entry, reading the most recent `.md` from
     `${HERMES_HOME}/cron/output/{job_id}/`. **Truncate at 8000 chars**
     with a `[... output truncated ...]` suffix.
  5. The user-supplied `prompt`.
- **Assembled-prompt threat rescan** (Python #3968): after assembly, run
  `ironhermes_cron::scanner::scan_cron_prompt` on the full assembled
  prompt — not just the user prompt. If it fires, emit a `BLOCKED`
  delivery doc with the scanner's reason and refuse the agent run. The
  scanner module already exists; only the call-site is new.
- **Skip-missing-skill** behavior: a skill that fails to load is logged,
  added to a `skipped: Vec<String>` list, and the assembled prompt is
  prefixed with `[IMPORTANT: The following skill(s) were listed for this
  job but could not be found and were skipped: {names}.]`.

### MCP discovery per tick

- Per the gateway's existing pattern, call `discover_mcp_tools()` at the
  start of each tick. Cron-runner crate exposes a `prepare_mcp_for_tick()`
  helper; non-fatal on error. After each tick completes, the cron-runner
  invokes the MCP orphan reaper.

### Storage parity (PARITY.md §4)

- **`jobs.json` envelope**: keep the bare array (current Rust shape). Do
  NOT migrate to Python's `{ "jobs": [...], "updated_at": ... }`. The
  legacy migration path in `JobStore::open` already handles Python's
  shape if encountered on disk, but new writes stay flat for simplicity.
- **fsync**: add `f.sync_all()` after `flush()` in `JobStore::save` and
  `save_job_output`. Without it the atomic rename only guarantees that
  the new path points at the temp inode — not that the temp inode's data
  is durable.
- **Unix 0700/0600 chmod**: add `#[cfg(unix)] fs::set_permissions` after
  create for the cron dir, output dir, and `jobs.json`. Mirrors Python
  `_secure_dir`/`_secure_file`.
- **Bare-control-char auto-repair**: if `serde_json::from_str` fails with
  a control-char error, retry with a relaxed deserializer that strips
  ` `–`` from string values, rewrite `jobs.json`, log a warn.
- **One-shot grace window**: 120s. Add `ONESHOT_GRACE_SECONDS` const to
  `ironhermes-cron::job` and use it in `JobStore::get_due_jobs` so a
  `Once` job created right before its `run_at` still fires.
- **Dynamic per-schedule grace**: replace the fixed
  `JobStore::grace_seconds = 3600` with `compute_grace_seconds(schedule)`
  returning half-period clamped to `[120s, 7200s]`. Mirrors
  `_compute_grace_seconds`.
- **`compute_next_run` anchoring**: optional `last_run_at` parameter for
  Interval and Cron variants; when present, anchor at `last_run_at`
  instead of `now`. Prevents drift after crash/restart.
- **Recover missing `next_run_at`** in `get_due_jobs`: for recurring
  schedules, recompute from now and persist; for `Once` schedules, use
  the 120s grace window. Matches Python `_recoverable_oneshot_run_at`.
- **`trigger_job(id)`**: new `JobStore` method that sets `next_run_at =
  Utc::now()` so the job fires on the next tick. Matches Python
  `trigger_job`. CLI gets `hermes cron trigger <id-or-name>`.

### `HERMES_CRON_AUTO_DELIVER_*` equivalence

- Python uses ContextVars (per-task) to pass the resolved delivery target
  into the agent's tool layer so `send_message` can route correctly. In
  Rust, **use tokio task-locals** (`tokio::task_local!`) rather than env
  vars or thread-locals. Task-locals propagate across awaits naturally
  and survive `tokio::spawn` when wrapped explicitly.
- Variables: `CRON_AUTO_DELIVER_PLATFORM`, `CRON_AUTO_DELIVER_CHAT_ID`,
  `CRON_AUTO_DELIVER_THREAD_ID`. Set inside the per-job execution scope
  before `agent.run(messages)`; cleared on exit.
- A `HERMES_CRON_SESSION` env var flip (Python parity) is **not** needed
  in Rust — its only Python consumer was the approval system, which has
  no Rust analogue yet. Re-evaluate if/when an approval system phase
  lands.

### `wrap_response` config

- Add `cron.wrap_response: bool` (default `true`) to the YAML config.
  When true, prepend a `Cronjob Response: {name}\n(job_id: {id})\n---\n`
  header and append a `To stop or manage this job, send me a new
  message...` footer. Off ⇒ deliver raw output.

### MEDIA: tag routing

- Reuse whatever the gateway crate uses for `MEDIA: <path>` extraction
  (search for the existing `extract_media` function — likely in
  `ironhermes-gateway` or `ironhermes-tools`). The cron runner threads
  extracted file paths into the live-adapter delivery loop with the
  correct `send_voice` / `send_image_file` / `send_video` /
  `send_document` routing based on extension. Standalone-fallback path
  delivers media as documents.

### Live-adapter vs standalone delivery

- When invoked from the gateway, the cron-runner accepts an
  `Option<Arc<dyn AdapterRegistry>>` parameter — when `Some`, use the
  live adapter for delivery (supports E2EE matrix paths). Fall back to
  standalone HTTP send on failure or when `None` (CLI `tick`/`daemon`
  paths always pass `None`).

</decisions>

<specifics>
## Specific Ideas

- The new crate's public API is intentionally narrow: `pub async fn
  run_cron_job(job: &CronJob, ctx: &CronRunnerContext) -> Result<()>` and
  a thin `CronRunnerContext` struct holding `Arc`-shared dependencies
  (job_store, skill_registry, tool_registry, memory_manager, hook_registry,
  config, optional tg_client/adapter_registry). Both `gateway::runner` and
  the new CLI tick command instantiate `CronRunnerContext` and call this
  function.
- The CLI daemon mode (`hermes cron daemon`) should reuse the same tick
  scaffold the gateway uses — extract a `run_tick_loop(ctx, cancel_token)`
  helper into the runner crate so gateway and CLI share it byte-for-byte.
- Per-job inactivity timeout polling needs an `AgentLoop` activity API.
  Suggest: `agent.activity_summary() -> ActivitySummary { seconds_since:
  f64, last_kind: ActivityKind, current_tool: Option<String>, ... }`. The
  cron runner polls this every 5s while waiting on the `agent.run` future
  via `tokio::select!`.
- `last_delivery_error` is set to `Some(comma-joined-per-target-errors)`
  when at least one target fails; `None` when all targets succeeded or
  when the job is `local`. This survives across runs so operators can
  see persistent delivery issues without grepping logs.
- The `_KNOWN_DELIVERY_PLATFORMS` allowlist should be a `pub const &[&str]`
  in the runner crate, NOT hard-coded into Python's frozenset shape.
  Plugin platforms (Python's `platform_registry.plugin_entries()`) have
  no Rust analogue yet — leave a `// TODO: plugin platforms` comment at
  the lookup site.

</specifics>

<deferred>
## Deferred Ideas

- **`JobState::Error` variant**: deferred per decisions above. Re-evaluate
  if a Rust-side failure mode emerges that needs leave-enabled-but-error
  semantics.
- **Plugin platform delivery registry**: Python's `platform_registry`
  with `cron_deliver_env_var` per-plugin. IronHermes has no plugin system
  yet. Add when/if a Rust plugin layer lands.
- **Channel directory resolution** (Python `resolve_channel_name`, e.g.
  `Alice (dm)` → real chat_id). Requires a channel-directory subsystem
  that doesn't exist yet. Defer.
- **Curator skill-ref rewrites** (`rewrite_skill_refs`). Belongs with the
  curator phase, not the cron-runner.
- **`HERMES_CRON_SESSION` env flag**: only consumer in Python was the
  approval system. No Rust approval system yet — defer until one lands.
- **{ "jobs": [...], "updated_at": ... } envelope migration**: keep the
  bare array. If a future phase needs the metadata, migrate then.
- **Pre-curation skill consolidation** in cron jobs: out of scope here.

</deferred>
