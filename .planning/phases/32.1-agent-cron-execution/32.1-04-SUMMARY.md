## What Was Built

### Task 1: Multi-Target Delivery Routing

Added to `crates/ironhermes-cron/src/delivery.rs`:

**`pub const KNOWN_DELIVERY_PLATFORMS`** — 7-entry allowlist: `telegram`, `discord`, `slack`, `matrix`, `whatsapp`, `webhook`, `qq`. Any deliver token not in this list never triggers an env-var read (closes STRIDE threat T-32.1.04-01).

**Five private helpers:**
- `home_channel_env_var(platform)` — maps platform to its primary `<PLATFORM>_HOME_CHANNEL` env var
- `legacy_home_channel_env_var(platform)` — maps `qq` to `QQBOT_HOME_CHANNEL` (only legacy alias)
- `home_channel_thread_env_var(platform)` — maps platform to `<PLATFORM>_HOME_CHANNEL_THREAD_ID`
- `lookup_home_channel(platform)` — reads primary + legacy env, populates thread_id, returns `Option<DeliveryTarget>`
- `expand_routing_token(token, job)` — dispatches a single token to `all`, `origin`, `platform:chat_id`, or bare platform

**`pub fn resolve_delivery_targets(job)`** — plural resolver: comma-splits `job.deliver`, calls `expand_routing_token` per part, deduplicates (same platform + chat_id + thread_id), returns `Vec<DeliveryTarget>`.

**`pub fn resolve_delivery_target(job)`** (singular) — deprecated; now delegates to `resolve_delivery_targets(job).into_iter().next()`. All existing callers (`tick.rs::complete_job_run`) continue to compile unchanged.

**16 unit tests** in `mod multi_target_tests` covering all routing behaviors.

### Task 2: save_job_output Hardening

Three changes inside `save_job_output`:
1. `f.sync_all()?` added immediately after `f.flush()?` — ensures data durability before atomic rename (closes PARITY.md §11 #18)
2. `#[cfg(unix)]` block after `create_dir_all` sets directory mode to `0o700` via `PermissionsExt::from_mode`
3. `#[cfg(unix)]` block after `fs::rename` sets output file mode to `0o600` (closes PARITY.md §11 #15, STRIDE T-32.1.04-02)

**4 unit tests** in `mod save_job_output_tests_phase_32_1` covering fsync regression, chmod 0600 on file, chmod 0700 on dir, atomic rename.

## Env-Mutex Pattern

Copied from `crates/ironhermes-agent/src/memory/factory.rs` (lines 391–396):

```rust
fn env_lock() -> MutexGuard<'static, ()> {
    static LOCK: OnceLock<Mutex<()>> = OnceLock::new();
    LOCK.get_or_init(|| Mutex::new(()))
        .lock()
        .unwrap_or_else(|e| e.into_inner())
}
```

Used in both `multi_target_tests` and `save_job_output_tests_phase_32_1` modules with `IRONHERMES_HOME` (the correct env var name, as confirmed from `ironhermes-core/src/constants.rs`; the delivery.rs comment mentions `HERMES_HOME` but the actual implementation reads `IRONHERMES_HOME`).

Some(DeliveryTarget {

## Plan 06 Hand-off

`resolve_delivery_targets(job: &CronJob) -> Vec<DeliveryTarget>` is the canonical entry point for the runner crate's per-target dispatch loop. The runner should iterate the returned `Vec<DeliveryTarget>` and dispatch each target independently, accumulating per-target errors into `job.last_delivery_error` (PARITY.md §10 `_deliver_result` multi-target loop).
