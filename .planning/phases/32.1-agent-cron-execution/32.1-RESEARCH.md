**Researched:** 2026-05-15

## Summary

Phase 32.1 ports the agent-runner side of Python `hermes-agent/cron/scheduler.py::run_job` (~600 LOC, lines 1013ŌĆō1656) into a new `ironhermes-cron-runner` Rust workspace crate, achieving full feature parity with the Python reference. The existing `ironhermes-cron` crate handles scheduling, storage, and tick orchestration; the new crate owns end-to-end execution: prompt assembly, script execution, agent loop invocation, output capture, and delivery dispatch.

The seed of the new crate already exists in `crates/ironhermes-gateway/src/runner.rs::execute_cron_job` (lines 1051ŌĆō1229). That function is a stripped-down version that lacks: script execution, `no_agent` short-circuit, wake-gate parsing, `context_from` chain injection, assembled-prompt threat rescan, per-job model/provider/toolsets/workdir overrides, inactivity timeout, multi-target delivery with `all`/comma-split/platform-without-chat_id, `wrap_response` header/footer, MEDIA tag routing, `last_delivery_error` tracking, and all the `CronJob` field additions required to carry the new data.

The work spans three scopes: (1) new fields on `CronJob` in `ironhermes-cron::job`; (2) storage parity fixes in `ironhermes-cron::store`, `delivery.rs`, and `parser.rs`; (3) the new `ironhermes-cron-runner` crate plus CLI commands `hermes cron tick` and `hermes cron daemon`.

**Primary recommendation:** Create `crates/ironhermes-cron-runner` as a thin orchestration crate. Its public surface is `run_cron_job(job, ctx)`. Gateway calls it instead of its own `execute_cron_job`. CLI tick and daemon commands construct `CronRunnerContext` and call it. Gate every plan on reading the canonical refs listed in CONTEXT.md.

### Pattern 2: tokio task-locals for per-job delivery context

**What:** Use `tokio::task_local!` to propagate delivery target into the agent's tool layer without env-var mutation. Set inside per-job scope before `agent.run(messages)`.
**When to use:** Every job execution. Required for correct parallel delivery routing.

### Pattern 3: Inactivity timeout via tokio::select!

**What:** Poll `agent.activity_summary()` every 5s while the agent runs; kill if idle exceeds `IRONHERMES_CRON_TIMEOUT`. Wrap whole run in wall-clock `tokio::time::timeout`.
**When to use:** Every LLM-path job run.

```rust
// Source: CONTEXT.md ┬¦Timeout & resilience
let activity_handle = /* Arc<Mutex<ActivitySummary>> shared with AgentLoop */;
let inactivity_limit = Duration::from_secs(cron_timeout_secs);
let wall_limit = Duration::from_secs(wall_timeout_secs);

tokio::time::timeout(wall_limit, async {
    tokio::select! {
        result = agent.run(messages) => result,
        _ = async {
            loop {
                tokio::time::sleep(Duration::from_secs(5)).await;
                let idle = activity_handle.lock().await.seconds_since;
                if idle > inactivity_limit.as_secs_f64() {
                    agent.interrupt("inactivity timeout").await;
                    break;
                }
            }
        } => Err(anyhow!("inactivity timeout"))
    }
}).await
```

### Pattern 4: Workdir job serialization

**What:** Partition due jobs by `job.workdir.is_some()`. Workdir jobs run sequentially in a single tokio task. Non-workdir jobs use `JoinSet` with configurable concurrency.
**When to use:** Tick orchestration in `run_tick_loop`.

```rust
// Source: CONTEXT.md ┬¦Per-job runtime overrides ŌĆö workdir serialization
let (workdir_jobs, parallel_jobs): (Vec<_>, Vec<_>) =
    due_jobs.iter().partition(|j| j.workdir.is_some());

// Sequential
for job in &workdir_jobs {
    run_cron_job(job, ctx).await?;
}

// Parallel
let max_parallel = std::env::var("IRONHERMES_CRON_MAX_PARALLEL")
    .ok()
    .and_then(|s| s.parse::<usize>().ok());
let mut set = tokio::task::JoinSet::new();
// semaphore if max_parallel is Some(n)
```

### Pattern 5: Wake-gate parsing

**What:** Parse the last non-empty stdout line of a pre-run script as JSON. If `{"wakeAgent": false}`, treat as silent.

```rust
// Source: CONTEXT.md ┬¦Script execution; hermes-agent/cron/scheduler.py _parse_wake_gate
fn parse_wake_gate(stdout: &str) -> bool {
    let last_line = stdout.lines()
        .rev()
        .find(|l| !l.trim().is_empty())
        .unwrap_or("");
    if let Ok(v) = serde_json::from_str::<serde_json::Value>(last_line) {
        if v.get("wakeAgent").and_then(|v| v.as_bool()) == Some(false) {
            return false; // do not wake agent
        }
    }
    true // default: wake agent
}
```

### Anti-Patterns to Avoid

- **Using `os.environ` / thread-locals for delivery context:** env vars are process-global and break parallel jobs. Use `tokio::task_local!` instead.
- **Running script without path traversal guard:** `scripts/` sandbox + `canonicalize + starts_with` is mandatory. Absolute paths and `..` components must be rejected.
- **Calling `AgentLoop::new` without `wire_fallback_if_configured`:** cron jobs must honor `PROV-07` fallback chain, same as gateway path.
- **Mutating `TERMINAL_CWD` env var for parallel workdir jobs:** workdir jobs MUST be serialized before setting `TERMINAL_CWD` since it is process-global.
- **Single-target delivery:** the new `resolve_delivery_targets` (plural) must be used, not the old `resolve_delivery_target` (singular) which returns `Option<DeliveryTarget>`.
- **Swallowing empty `final_response` silently:** empty response is a soft failure (PARITY.md ┬¦9 issue #8585), not a silent success.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Agent execution loop | Custom LLM loop | `ironhermes_agent::AgentLoop::run` | Already handles tools, compression, hooks, budget, fallback |
| System prompt assembly | Ad-hoc string concat | `ironhermes_agent::PromptBuilder` | Handles SOUL.md, memory, skills catalog, platform hint, cache layers |
| Prompt injection scan | Custom regex | `ironhermes_cron::scanner::scan_cron_prompt` | Already ported, already covers `_CRON_THREAT_PATTERNS` |
| Tool registry construction | Re-build tools inline | Pass `Arc<RwLock<ToolRegistry>>` from caller | ToolRegistry is expensive; gateway already has one |
| Secret redaction | Custom scrubber | Search `ironhermes-core` or trajectory crate for existing redactor | Python uses `agent.redact.redact_sensitive_text`; find Rust equivalent |
| Bash/python interpreter discovery | Hardcoded paths | `which::which("bash")` / `which::which("python3")` with fallback | Portable, handles custom installs |
| Process-global delivery context | `std::env::set_var` | `tokio::task_local!` | Task-locals are safe across awaits, survive `tokio::spawn` when wrapped |
| Tick loop | Duplicate gateway logic | `run_tick_loop(ctx, cancel_token)` in runner crate | Both gateway and CLI daemon share byte-for-byte identical tick scaffold |

**Key insight:** The gateway's `execute_cron_job` is the seed but is underspecified ŌĆö it lacks ~70% of the Python `run_job` behavior. The new crate is not a greenfield build; it is a systematic port of Python `run_job` (600 LOC) with the gateway seed as the starting skeleton.

## Common Pitfalls

### Pitfall 1: Circular dependency between crates

**What goes wrong:** `ironhermes-cron-runner` depends on `ironhermes-agent`; `ironhermes-agent` already depends on `ironhermes-cron`. Adding `ironhermes-cron-runner` to `ironhermes-agent`'s deps would create a cycle.
**Why it happens:** The dependency CONTEXT.md specifies is one-directional: `ironhermes-cron-runner` ŌåÆ `ironhermes-agent`, not the reverse. This is the correct topology.
**How to avoid:** `ironhermes-cron-runner` depends on `ironhermes-agent`; `ironhermes-agent` does NOT depend on `ironhermes-cron-runner`. Gateway depends on both.
**Warning signs:** `cargo build` emits "cyclic package dependency" error.

### Pitfall 2: `TERMINAL_CWD` race condition with parallel workdir jobs

**What goes wrong:** Two parallel jobs with different `workdir` values clobber each other's `TERMINAL_CWD` env var.
**Why it happens:** `std::env::set_var` is process-global. The partition + serialization pattern from CONTEXT.md exists precisely to prevent this.
**How to avoid:** Always partition `due_jobs` into workdir vs non-workdir groups before launching. Workdir group runs in a single sequential task.
**Warning signs:** Agent sees wrong working directory; `AGENTS.md` loaded from wrong project.

### Pitfall 3: `resolve_delivery_target` (singular) left at existing call sites

**What goes wrong:** Gateway's `complete_job_run` and `dispatch_delivery` continue using the old single-target `Option<DeliveryTarget>` instead of the new `Vec<DeliveryTarget>`.
**Why it happens:** The old function and new function coexist during transition. Code using the old function compiles fine.
**How to avoid:** The old `resolve_delivery_target` (singular) should be deprecated or made private once the new plural version lands. Check all call sites in `tick.rs` and `gateway/runner.rs`.
**Warning signs:** Multi-target `deliver = "telegram,discord"` only sends to first target.

### Pitfall 4: Missing `fsync` on `save_job_output`

**What goes wrong:** On a crash after `rename()` but before data durability, the output file has correct path but corrupt/empty content.
**Why it happens:** The existing `save_job_output` in `delivery.rs` calls `flush()` but not `sync_all()`. Atomic rename only guarantees the new path exists ŌĆö not that the inode's data is flushed to disk.
**How to avoid:** Add `f.sync_all()?` after `f.flush()?` in both `save_job_output` and `JobStore::save`.
**Warning signs:** Corrupted output files after hard shutdown; `last_status="ok"` but empty `.md` file.

### Pitfall 5: `context_from` job ID validation allows UUID format

**What goes wrong:** Python validates `context_from` IDs as 12-char hex strings (Python's uuid hex[:12]). Rust uses full UUIDs. The validation logic must match Rust's UUID format.
**Why it happens:** PARITY.md ┬¦3 notes that Python uses 12-char hex while Rust uses full UUID. The context_from validation in Python guards against path traversal with a 12-char hex check; Rust needs a UUID-compatible equivalent.
**How to avoid:** Validate that `source_job_id` matches UUID format (or at minimum contains no `/`, `\`, `..`, or empty). Use `uuid::Uuid::parse_str()` for strict validation.
**Warning signs:** `context_from` silently skips all jobs because UUID format fails the hex12 check.

### Pitfall 6: `tokio::task_local!` scoping with `tokio::spawn`

**What goes wrong:** Task-local values do NOT automatically propagate to `tokio::spawn` child tasks unless explicitly wrapped.
**Why it happens:** tokio task-locals propagate across `.await` within the same task but not across `spawn` boundaries.
**How to avoid:** If the agent run or any delivery helper spawns a separate task, ensure the task-local scope wraps the `spawn` call or the locals are passed explicitly.
**Warning signs:** `send_message` tool inside agent run cannot read `CRON_AUTO_DELIVER_PLATFORM`.

### Pitfall 7: `AgentLoop` activity tracker API does not exist yet

**What goes wrong:** The inactivity timeout polling requires `agent.activity_summary()` and `agent.interrupt(reason)`, but these methods do not exist on `AgentLoop` today.
**Why it happens:** This is new API that must be added to `ironhermes-agent::agent_loop.rs` as part of this phase.
**How to avoid:** Plan 1 or Plan 2 must add the activity tracker fields and public API to `AgentLoop` before the runner crate can be written.
**Warning signs:** Compiler error "no method named `activity_summary` found for struct `AgentLoop`".

```rust
    "telegram", "discord", "slack", "matrix", "whatsapp", "webhook", "qq",
    // TODO: plugin platforms
];

pub fn resolve_delivery_targets(job: &CronJob) -> Vec<DeliveryTarget> {
    if job.deliver == "local" { return vec![]; }

    job.deliver.split(',')
        .map(|s| s.trim())
        .filter(|s| !s.is_empty())
        .flat_map(|part| expand_routing_token(part, job))
        .collect::<IndexSet<_>>()  // dedup
        .into_iter()
        .collect()
}

fn expand_routing_token(token: &str, job: &CronJob) -> Vec<DeliveryTarget> {
    if token == "all" {
        return KNOWN_DELIVERY_PLATFORMS.iter()
            .filter_map(|p| lookup_home_channel(p))
            .collect();
    }
    resolve_single_delivery_target(token, job).into_iter().collect()
}
```

```rust
// Source: CONTEXT.md ┬¦Script execution; hermes-agent/cron/scheduler.py _run_job_script
pub async fn run_job_script(script_name: &str) -> Result<(bool, String)> {
    let scripts_dir = get_hermes_home().join("scripts");
    tokio::fs::create_dir_all(&scripts_dir).await?;

    let candidate = scripts_dir.join(script_name);
    let canonical = candidate.canonicalize()
        .context("script path canonicalization failed")?;
    let scripts_dir_canonical = scripts_dir.canonicalize()?;

    if !canonical.starts_with(&scripts_dir_canonical) {
        anyhow::bail!("script path escapes scripts_dir: {:?}", canonical);
    }

    let interpreter = select_interpreter(&canonical);
    let timeout_secs = parse_env_u64("IRONHERMES_CRON_SCRIPT_TIMEOUT", 120);

    let child = tokio::process::Command::new(&interpreter)
        .arg(&canonical)
        .stdout(Stdio::piped())
        .stderr(Stdio::piped())
        .spawn()?;

    match tokio::time::timeout(
        Duration::from_secs(timeout_secs),
        child.wait_with_output()
    ).await {
        Ok(Ok(out)) => {
            let stdout = redact(String::from_utf8_lossy(&out.stdout).into_owned());
            let stderr = redact(String::from_utf8_lossy(&out.stderr).into_owned());
            let ok = out.status.success();
            let combined = if stderr.is_empty() { stdout } else {
                format!("{}\n\nstderr:\n{}", stdout, stderr)
            };
            Ok((ok, combined))
        }
        Ok(Err(e)) => Err(e.into()),
        Err(_) => Ok((false, format!("script timed out after {}s", timeout_secs))),
    }
}
```
