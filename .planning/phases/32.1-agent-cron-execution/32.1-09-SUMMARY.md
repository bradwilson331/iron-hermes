# Phase 32.1 Plan 09: Telegram Whitelist Fallback (gap-closure) Summary

**One-liner:** Restored pre-32.1 parity in `lookup_home_channel` — falls back to `Config::gateway.platforms.telegram.whitelist` (single entry) when `TELEGRAM_HOME_CHANNEL` env var is unset; env var still wins when set.

## What Was Built

### New helper: `lookup_telegram_whitelist_fallback` (delivery.rs line 83)

A private function added directly above `lookup_home_channel`. It:

- Calls `ironhermes_core::config::Config::load()` (no new crate dep — this dep edge already existed)
- Reads `config.gateway.platforms.get("telegram")`
- Gates on `tg.enabled`
- Returns `Some(DeliveryTarget)` when `tg.whitelist.len() == 1`
- Returns `None` for zero or multiple whitelist entries (matching `OriginDecision::None/Multi` semantics)
- Returns `None` on `Config::load` failure (non-fatal, logs at `tracing::debug!`)

### Extended: `lookup_home_channel` (delivery.rs lines ~130-160)

Added a telegram-only fallback branch after both the primary and legacy env var lookups yield `None`:

```rust
if chat_id.is_none() && platform == "telegram" {
    if let Some(target) = lookup_telegram_whitelist_fallback() {
        return Some(target);
    }
}
```

Env var lookup still runs first — precedence is unchanged.

### Existing tests isolated (Rule 1 — auto-fix)

Three pre-existing tests were asserting empty results for `deliver=telegram` without `IRONHERMES_HOME` isolation. After the new fallback, they would fail on any developer machine with a real `~/.ironhermes/config.yaml` containing a single telegram whitelist entry:

- `multi_target_tests::test6_bare_platform_home_unset_returns_empty` — added `IRONHERMES_HOME` tempdir
- `multi_target_tests::test12_origin_no_origin_no_home_channel_returns_empty` — added `IRONHERMES_HOME` tempdir
- `tests::origin_no_origin_field_returns_none` — added `env_lock()` + `IRONHERMES_HOME` tempdir

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] YAML fixture format: `whitelist` is `Vec<i64>`, not `Vec<String>`**
- **Found during:** First test run (test2 and test10 failed with `0 != 1`)
- **Issue:** Plan's YAML examples used quoted strings (`"12345"`) but `PlatformGatewayConfig.whitelist` is `Vec<i64>`. serde_yaml deserializes quoted strings as strings but the field type is i64, causing parse failure and `Config::load` returning defaults.
- **Fix:** Changed all YAML whitelist fixtures to unquoted integers (`whitelist: [12345]`), matching the canonical format in `config.rs` tests (lines 1682, 1699, 1700).
- **Files modified:** `delivery.rs` (test fixtures only)
- **Commit:** `05d827da`

**2. [Rule 1 - Bug] `clear_telegram_env` function safety**
- **Found during:** First compile (warning about `unsafe_op_in_unsafe_fn`)
- **Issue:** Helper declared `unsafe fn` but internal `remove_var` calls weren't wrapped in `unsafe {}` blocks, triggering Rust 2024 lint.
- **Fix:** Changed to a safe `fn` with internal `unsafe {}` blocks.
- **Files modified:** `delivery.rs`
- **Commit:** `05d827da`

**3. [Rule 2 - Missing Critical Functionality] 3 pre-existing tests needed IRONHERMES_HOME isolation**
- **Found during:** Second test run (test `origin_no_origin_field_returns_none` failed after step 5 fix)
- **Issue:** The plan's Step 5 correctly identified `test6` and `test12` in `multi_target_tests` as needing isolation, but missed `origin_no_origin_field_returns_none` in the original `tests` module. On the developer's real machine (which has a single-entry telegram whitelist `7018949547` in `~/.ironhermes/config.yaml`), this test fails because `resolve_delivery_target("origin", None)` now falls back to the config whitelist.
- **Fix:** Added `env_lock()` infrastructure to the `tests` module and isolated `IRONHERMES_HOME` in that test.
- **Files modified:** `delivery.rs`
- **Commit:** `05d827da`

## UAT Test 5 Re-run Note

The gap is now code-closed. A repeat of UAT Test 5 ("deliver=telegram, env unset, single-entry whitelist") requires:
1. `~/.ironhermes/config.yaml` has `gateway.platforms.telegram.enabled: true` and `whitelist: [<your-chat-id>]`
2. `TELEGRAM_HOME_CHANNEL` is NOT set in the environment
3. A cron job with `deliver = "telegram"` fires and delivers successfully (no "WARN deliver=telegram has no home channel configured" message)

This is a runtime/operator validation — not blocking plan completion, but it is the ultimate proof that the regression is closed. The unit tests at `test2_config_fallback_single_whitelist_entry` provide the programmatic equivalent.
