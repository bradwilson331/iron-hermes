### Task 2 — CronJob extensions + defensive origin deserializer

Commit: `c415b101`

**`crates/ironhermes-cron/src/job.rs`:**

1. **JobOrigin** — added `Default` + `PartialEq` derives. Behavior of `JobOrigin`'s own `Deserialize` impl unchanged (the defensive handling lives on `CronJob.origin`, not on `JobOrigin`).

2. **`deserialize_origin_lenient`** — new private helper:
   ```rust
   fn deserialize_origin_lenient<'de, D: Deserializer<'de>>(
       d: D,
   ) -> Result<Option<JobOrigin>, D::Error>
   ```
   Behavior:
   - `Value::Null` → `Ok(None)` (existing behavior preserved)
   - `Value::Object(_)` → `serde_json::from_value::<JobOrigin>(...).map(Some).or(Ok(None))` (silently downgrade malformed objects to `None`)
   - Anything else (string, number, array, bool) → emits `tracing::warn!(origin = ?value, ...)` and returns `Ok(None)`

3. **CronJob** — `origin` field annotated `#[serde(deserialize_with = "deserialize_origin_lenient", default)]`. Type stays `Option<JobOrigin>`.

4. **CronJob** — 9 new fields appended after `last_error`, each `#[serde(default)]`:
   - `model: Option<String>`
   - `provider: Option<String>`
   - `base_url: Option<String>`
   - `script: Option<String>`
   - `no_agent: bool` (default `false`)
   - `context_from: Option<Vec<String>>`
   - `enabled_toolsets: Option<Vec<String>>`
   - `workdir: Option<String>`
   - `last_delivery_error: Option<String>`

5. **`#[cfg(test)] mod cronjob_serde_tests`** — 6 tests covering all behaviors in the plan's `<behavior>` block (test1..test6). All pass.

## Deviations from Plan

**1. [Rule 3 — Blocking compilation] JobOrigin needed PartialEq + Default**
- **Found during:** Task 2 first build attempt (RED phase implicit — TDD step revealed the gap)
- **Issue:** Test 1 and Test 3 use `assert_eq!` on `Option<JobOrigin>`, which requires `PartialEq` on `JobOrigin`. The PATTERNS.md §JobOrigin defensive-serde block also added `Default`, which is harmless to keep and matches the lenient deserializer's fallback semantics.
- **Fix:** Added `Default, PartialEq` to the `JobOrigin` derive list. Did NOT add `#[serde(default)]` to `JobOrigin` itself (the plan explicitly said "Keep the public `JobOrigin` derives ... and field signatures exactly as they are today — the defensive handling lives on `CronJob`, not on `JobOrigin`"). PATTERNS.md showed a `#[serde(default)]` on the struct as an alternative; the plan's `<action>` (b) language overrides that alternative.
- **Files modified:** `crates/ironhermes-cron/src/job.rs` (single line)
- **Commit:** `c415b101`

**2. [Rule 3 — Blocking compilation] Five fixture-builder sites needed the 9 new field defaults**
- **Found during:** Task 2 first build attempt
- **Issue:** After appending the 9 fields, every existing `CronJob { ... }` struct literal in the cron crate failed `E0063 missing fields`. Sites: `store.rs::LegacyCronJob::from` (production code path for legacy migration), `store.rs::add_job` (production code path for new jobs), `delivery.rs::tests::make_job`, `tick.rs::tests::make_job_with_deliver`, `display.rs::tests::make_job`.
- **Fix:** Appended the 9 default field values to each site (`model: None, provider: None, base_url: None, script: None, no_agent: false, context_from: None, enabled_toolsets: None, workdir: None, last_delivery_error: None`).
- **Files modified:** 4 files (store.rs/delivery.rs/tick.rs/display.rs)

## Hand-off Notes for Downstream Plans

### Plan 02 (or whichever plan needs runtime parsing)

`CronJob` now carries the 9 fields. They are populated by `serde` on read and serialized on write. Production-code `CronJob` constructors (`LegacyCronJob::from`, `JobStore::add_job`) default them to `None` / `false`. To set them programmatically, mutate the struct field directly (`job.model = Some("claude-3-opus".into())`) or expand `JobUpdate` to carry them in a follow-up plan.

- `LegacyCronJob::From` already migrates with the 9 fields defaulted — no extra work needed there.

- Defensive `origin` deserialization is structural — no Plan 03 work needed.

### Plan 04 (delivery routing)

- `last_delivery_error` field is in place. Plan 04 writes to it via `JobStore::update_job` (likely needs a new `JobUpdate.last_delivery_error: Option<Option<String>>` or a dedicated `set_last_delivery_error` method).
- `delivery.rs::tests::make_job` fixture already has the 9 fields zeroed — Plan 04 fixture additions just need to populate `last_delivery_error` where relevant.

### `deserialize_origin_lenient` signature (for cross-crate reference)

```rust
fn deserialize_origin_lenient<'de, D: Deserializer<'de>>(
    d: D,
) -> Result<Option<JobOrigin>, D::Error>
```

- Visibility: **private** (`fn`, not `pub fn`). If a downstream plan needs to invoke this from another module/crate, it can be promoted to `pub(crate)` or `pub` in a follow-up.
- Side effect: emits `tracing::warn!(origin = ?value, "CronJob origin field is not an object — treating as None")` on non-object/non-null inputs. No warn is emitted on the structurally-valid `null` and `object` branches.
