## Summary

This phase creates all deployment and first-run infrastructure files by porting hermes-agent's patterns to IronHermes. The work is primarily file authoring â€” no new Rust logic except a small first-run scaffolding function in `main.rs`. Every canonical pattern has been verified against the actual source files.

IronHermes is dramatically simpler than hermes-agent for Docker purposes: no Python venv, no Node.js/npm, no Playwright. The Dockerfile reduces to a two-stage Rust build (rust:latest builder â†’ debian-slim runtime) with only python3 added for the `execute_code` sandbox. Expected image size: 80-120MB versus hermes-agent's multi-GB image.

The biggest adaptation challenge is the install.sh: hermes-agent's 1,436-line script clones a git repo and sets up a Python venv. IronHermes's install.sh is fundamentally different â€” it downloads a prebuilt binary from GitHub Releases (standard Rust CLI pattern, e.g. cargo-binstall, ripgrep installers) or falls back to `cargo install`. The script structure can borrow hermes-agent's OS detection, interactive prompt helpers, PATH patching, and directory scaffolding logic while stripping all Python/venv/Node machinery.

**Primary recommendation:** Author the six files in dependency order â€” .env.example and cli-config.yaml.example first (no dependencies), then Dockerfile and docker/entrypoint.sh (depend on templates), then install.sh and setup-ironhermes.sh (depend on all of the above), then add the first-run `ensure_home_dirs()` call in main.rs.

### Pattern 1: Multi-Stage Rust Dockerfile

```dockerfile

COPY Cargo.toml Cargo.lock ./
COPY crates/ crates/
COPY providers/ providers/

        python3 python3-pip ca-certificates && \
```

### Pattern 2: entrypoint.sh Structure

```bash
#!/bin/bash
# Source: hermes-agent/docker/entrypoint.sh (direct port)
set -e
IRONHERMES_HOME="${IRONHERMES_HOME:-/opt/data}"
INSTALL_DIR="/opt/ironhermes"

# Privilege drop: run as root â†’ remap UID/GID â†’ re-exec as ironhermes
if [ "$(id -u)" = "0" ]; then
    [ -n "$IRONHERMES_UID" ] && usermod -u "$IRONHERMES_UID" ironhermes
    [ -n "$IRONHERMES_GID" ] && groupmod -o -g "$IRONHERMES_GID" ironhermes 2>/dev/null || true
    chown -R ironhermes:ironhermes "$IRONHERMES_HOME" 2>/dev/null || \
        echo "Warning: chown failed (rootless container?) â€” continuing anyway"
    exec gosu ironhermes "$0" "$@"
fi

# Running as ironhermes user from here
mkdir -p "$IRONHERMES_HOME"/{cron,sessions,logs,hooks,memories,skills,workspace}

# Copy templates only if they don't exist (preserve user edits)
[ ! -f "$IRONHERMES_HOME/.env" ]         && cp "$INSTALL_DIR/.env.example" "$IRONHERMES_HOME/.env"
[ ! -f "$IRONHERMES_HOME/config.yaml" ]  && cp "$INSTALL_DIR/cli-config.yaml.example" "$IRONHERMES_HOME/config.yaml"
[ ! -f "$IRONHERMES_HOME/SOUL.md" ]      && cp "$INSTALL_DIR/docker/SOUL.md" "$IRONHERMES_HOME/SOUL.md"

exec ironhermes "$@"
```

**Dropped from hermes-agent:** `.venv` activation (no venv) and `skills_sync.py` call (IronHermes skills are compiled-in, not file-synced). [VERIFIED: hermes-agent/docker/entrypoint.sh lines 40-70]

### Pattern 4: install.sh OS/Arch Detection and Binary Download

The standard Rust CLI installer pattern (used by cargo-binstall, ripgrep, fd, etc.):

```bash
detect_platform() {
    local os arch
    case "$(uname -s)" in
        Linux*)  os="linux"  ;;
        Darwin*) os="macos"  ;;
        *)       log_error "Unsupported OS"; exit 1 ;;
    esac
    case "$(uname -m)" in
        x86_64)        arch="x86_64"  ;;
        aarch64|arm64) arch="aarch64" ;;
        *)             arch=""; log_warn "Unknown arch, will fall back to cargo install" ;;
    esac
    PLATFORM="${os}-${arch}"
}

download_binary() {
    local version="$1"
    local url="https://github.com/OWNER/ironhermes/releases/download/${version}/ironhermes-${PLATFORM}.tar.gz"
    if curl -fsSL "$url" -o /tmp/ironhermes.tar.gz 2>/dev/null; then
        tar -xzf /tmp/ironhermes.tar.gz -C /tmp/
        install -m 755 /tmp/ironhermes "$INSTALL_BIN"
        return 0
    fi
    return 1  # fall through to cargo install
}
```

[ASSUMED] â€” the exact GitHub Releases URL format depends on release artifact naming decided in CI setup, which doesn't exist yet. The pattern is standard but the URL template is provisional.

### Anti-Patterns to Avoid

- **Don't copy hermes-agent's Dockerfile verbatim.** It installs Node.js, npm, Playwright, uv, and builds a web dashboard. IronHermes needs none of this â€” the binary is statically compiled.
- **Don't use `useradd` without `-d` pointing to IRONHERMES_HOME.** The home dir for the user must be the data volume, not /home/ironhermes (which won't persist).
- **Don't `apt install gosu`.** Use `COPY --from=tianon/gosu:...` in the Dockerfile to get a fixed-version binary without adding Debian package mirrors.
- **Don't attempt cargo install in Docker.** The multi-stage build produces the binary; the runtime image never needs Rust or cargo.

## Common Pitfalls

### Pitfall 1: IRONHERMES_HOME Env Var vs Binary Default Path
**What goes wrong:** `.env.example` documents `IRONHERMES_HOME` as an override. The binary reads it via `get_hermes_home()` in constants.rs (checks `IRONHERMES_HOME` env var, falls back to `~/.ironhermes`). If the entrypoint.sh hardcodes `/opt/data` but forgets to export `IRONHERMES_HOME`, the binary will write to `~/.ironhermes` inside the container (which is the user's home, `/opt/data`).
**Why it happens:** The Docker ENV sets `IRONHERMES_HOME=/opt/data`; the entrypoint also sets it in shell â€” but if the `exec ironhermes "$@"` replaces the shell, ENV should propagate correctly.
**How to avoid:** The Dockerfile must have `ENV IRONHERMES_HOME=/opt/data` (not just a comment). The entrypoint.sh should also set `export IRONHERMES_HOME=...` before `exec`.
**Warning signs:** On first run, `~/.ironhermes/` gets created inside the container's ephemeral layer instead of the volume.

### Pitfall 2: UID Remapping + Volume Ownership
**What goes wrong:** hermes-agent's entrypoint does `chown -R hermes:hermes "$HERMES_HOME"` but wraps it in `|| echo "Warning: chown failed (rootless container?) â€” continuing anyway"`. Copying this pattern exactly is critical â€” omitting the `|| true` causes the entrypoint to exit non-zero in rootless Podman environments.
**How to avoid:** Port the exact error-suppressed chown pattern from hermes-agent/docker/entrypoint.sh lines 31-32. [VERIFIED: hermes-agent/docker/entrypoint.sh]

### Pitfall 3: install.sh curl-pipe Non-Interactive Detection
**What goes wrong:** When running `curl ... | bash`, stdin is a pipe (not a TTY). Any `read -p` interactive prompt causes `set -e` to abort the entire script silently.
**Why it happens:** hermes-agent solves this with an `IS_INTERACTIVE` check (`[ -t 0 ]`) before every interactive prompt. Forgetting this causes the installer to silently do nothing when piped.
**How to avoid:** Port the `IS_INTERACTIVE` detection pattern from hermes-agent/scripts/install.sh lines 44-48. All prompts must check `$IS_INTERACTIVE` first or have a default fallback. [VERIFIED: hermes-agent/scripts/install.sh]

### Pitfall 4: Cargo Build Cache in Multi-Stage Docker
**What goes wrong:** Without layer-caching the dependency compile step, every `docker build` recompiles all of Tokio, reqwest, serde etc. (5-10 minute build).
**Why it happens:** Copying the full source tree before `cargo build` invalidates the cache on any file change.
**How to avoid:** Use the "fake Cargo.toml + cargo build --release (empty)" trick or use `cargo-chef` pattern: copy Cargo.toml/Cargo.lock first, compile dependencies only, then copy source and compile the binary. This is Claude's discretion per the decisions list.

### Pitfall 5: Binary Not Statically Linked â€” glibc Version Mismatch
**What goes wrong:** The Rust binary compiled on `rust:latest` (Ubuntu-based) links against a newer glibc than available on `debian:bookworm-slim`.
**Why it happens:** Rust binaries are dynamically linked to glibc by default; `rust:latest` and `debian:bookworm-slim` must be compatible.
**How to avoid:** Both `rust:latest` (based on Debian bookworm as of 2025) and `debian:bookworm-slim` use the same glibc. If the builder ever moves to a newer base, switch to `--target x86_64-unknown-linux-musl` for static linking. [ASSUMED] â€” glibc compatibility between current rust:latest and debian:bookworm-slim should be fine.

### Pitfall 6: setup-ironhermes.sh Without Rust Toolchain Check
**What goes wrong:** Developer runs `./setup-ironhermes.sh` on a machine without Rust installed; `cargo build` fails with an unhelpful error.
**How to avoid:** Add a `command -v cargo` check at the top with a helpful message pointing to rustup.rs. hermes-agent's `setup-hermes.sh` does a similar check for `uv`. [VERIFIED: setup-hermes.sh pattern, lines 62-100]

### Pitfall 7: Template Copy Overwrites User Edits
**What goes wrong:** Running `install.sh` again after configuration overwrites `.env` and `config.yaml`.
**How to avoid:** The `[ ! -f dest ] && cp src dest` pattern in both entrypoint.sh and install.sh is the correct guard. This is explicitly required by D-14. Never use `cp -f` for template seeding.

## Open Questions

1. **GitHub Releases artifact naming**
   - What we know: install.sh needs to download a prebuilt binary; pattern is standard
   - What's unclear: No GitHub Actions CI exists yet to publish releases; the exact artifact name isn't decided
   - Recommendation: Use `ironhermes-{os}-{arch}[-musl].tar.gz` convention; add a comment in install.sh noting the URL is a placeholder until CI is set up. The fallback to `cargo install` ensures the script works today.

2. **debian:bookworm-slim vs distroless**
   - What we know: D-07 says "debian-slim (or distroless)"; hermes-agent uses `debian:13.4` (trixie)
   - What's unclear: Distroless has no shell, which makes debugging harder; bookworm-slim has python3 in apt
   - Recommendation: Use `debian:bookworm-slim` â€” distroless has no python3 package and complicates the entrypoint shell script; bookworm-slim is the natural choice for a Rust binary + python3.

3. **Cargo dependency layer caching in Dockerfile**
   - What we know: Full recompile on source change is slow (5-10min); cargo-chef is the standard solution
   - What's unclear: Whether this phase should implement cargo-chef or just use a simple single-stage approach with a TODO comment
   - Recommendation: Implement simple two-stage without cargo-chef for now; add a comment explaining the optimization. The layer cache still works when only source files change if Cargo.toml/Cargo.lock are copied before source.
