# Phase 21.6: Port Deployment Setup Files from hermes-agent - Context

**Gathered:** 2026-04-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Port hermes-agent's deployment and setup infrastructure to IronHermes: `.env.example` env var template, `cli-config.yaml.example` config template, `Dockerfile` with multi-stage Rust build, `docker/entrypoint.sh` bootstrap script, `install.sh` curl-pipe installer, `setup-ironhermes.sh` post-clone dev setup, and first-run directory scaffolding. This phase creates deployment files and templates — it does NOT implement new sandbox backends or the interactive setup wizard (those are separate phases).

</domain>

<decisions>
## Implementation Decisions

### .env.example
- **D-01:** Full mirror of hermes-agent's `.env.example` — document all env vars IronHermes reads today plus placeholders for upcoming features (MCP, browser, additional providers). Keeps parity as features land.
- **D-02:** Organized by section: LLM providers (OpenRouter, Anthropic, Gemini, Ollama, etc.), tool API keys (Firecrawl, Exa), terminal/sandbox config, gateway platforms (Telegram), debug flags, compression settings.
- **D-03:** Commented-out by default (same pattern as hermes-agent) — users uncomment what they need.

### cli-config.yaml.example
- **D-04:** Full config template covering all Config struct sections: model, agent, terminal, web, gateway, cron, memory, compression, skills, exec, subagent, rate_limit, hooks.
- **D-05:** Self-documenting with inline comments explaining each field, valid values, and defaults. Same pattern as hermes-agent's `cli-config.yaml.example`.
- **D-06:** File lives at repo root. Copied to `~/.ironhermes/config.yaml` by entrypoint.sh, install.sh, and first-run scaffolding.

### Dockerfile
- **D-07:** Multi-stage build: `rust:latest` build stage for `cargo build --release`, debian-slim (or distroless) runtime stage with just the binary + Python3.
- **D-08:** Python3 included in runtime image so `execute_code` sandbox works out of the box.
- **D-09:** Target image size ~100-150MB (binary + python3 + minimal runtime deps). No Node.js, no npm, no Playwright in Docker image.
- **D-10:** Non-root runtime user (`ironhermes`) matching hermes-agent's security pattern.
- **D-11:** `IRONHERMES_HOME=/opt/data` as VOLUME for persistent state.

### docker/entrypoint.sh
- **D-12:** Port hermes-agent's entrypoint pattern: privilege dropping via gosu, UID/GID remapping, directory creation, template copying (.env, config.yaml, SOUL.md), then `exec ironhermes "$@"`.
- **D-13:** Create essential directories: cron/, sessions/, logs/, hooks/, memories/, skills/, workspace/.
- **D-14:** Copy `.env.example` → `.env` and `cli-config.yaml.example` → `config.yaml` only if they don't already exist (preserve user edits).

### Install/setup scripts
- **D-15:** Two scripts: `install.sh` (curl-pipe installer for end users) and `setup-ironhermes.sh` (post-clone dev setup).
- **D-16:** `install.sh` tries downloading prebuilt binary from GitHub Releases for OS/arch first, falls back to `cargo install ironhermes` if no binary available.
- **D-17:** `install.sh` scaffolds `~/.ironhermes/` directory structure, copies templates, symlinks binary to `~/.local/bin/ironhermes`, updates PATH in shell config.
- **D-18:** `setup-ironhermes.sh` for developers: `cargo build --release`, copy `.env.example` → `.env`, scaffold config, optionally run setup wizard.
- **D-19:** Skip Windows scripts for now — track as deferred idea.

### Directory bootstrapping
- **D-20:** Match hermes-agent's directory structure under `~/.ironhermes/`: cron/, sessions/, logs/, hooks/, memories/, skills/, workspace/. Skip skins/ (no skin engine) and plans/ (no trajectory saving).
- **D-21:** Belt-and-suspenders: binary itself creates missing dirs on first run (robust for all install methods), AND scripts scaffold on install.
- **D-22:** workspace/ directory serves as default CWD for agent tool operations. Document `terminal.cwd` config option.

### Sandboxing + workspaces (config only)
- **D-23:** Document local + Docker sandbox backends in `.env.example` and `cli-config.yaml.example` (terminal.backend: local | docker, TERMINAL_DOCKER_IMAGE, etc.). Do NOT implement Docker backend in this phase — just set up config documentation.
- **D-24:** Create `~/.ironhermes/workspace/` as default agent working directory. Document in config template. Actual per-session workspace isolation deferred to a future phase.

### Claude's Discretion
- Exact .env.example ordering and grouping of provider sections
- Dockerfile layer caching strategy and build optimization
- install.sh OS/arch detection logic and error handling
- Whether to include `.dockerignore` (yes, should exclude target/, .planning/, .git/)
- SOUL.md default template content for Docker seeding

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `Config` struct in `config.rs`: All config sections already defined with `#[serde(default)]` — cli-config.yaml.example should mirror these fields exactly
- `get_hermes_home()` in `constants.rs`: Resolves `IRONHERMES_HOME` env var or defaults to `~/.ironhermes/`
- `TerminalConfig`: Already has `backend: String`, `cwd: String`, `timeout: u64` — supports documenting docker backend in config template
- `ExecConfig`: Has `python_path`, `timeout_secs`, `max_rpc_calls` — map to .env.example terminal/exec section

### Established Patterns
- Config loading: YAML deserialization with `serde(default)` for backward compatibility
- Env var precedence: .env vars override config.yaml values (already implemented in provider resolution)
- `IRONHERMES_HOME` env var for data directory path (analogous to `HERMES_HOME`)

### Integration Points
- `main.rs` in ironhermes-cli: Where first-run directory scaffolding would be added
- `Cargo.toml` workspace root: Where Dockerfile would reference for build
- Repo root: Where .env.example, cli-config.yaml.example, Dockerfile, setup scripts live

</code_context>

<specifics>
## Specific Ideas

- Docker image should be dramatically smaller than hermes-agent's since there's no Python venv, npm, or Playwright — just the Rust binary + python3 for sandbox
- install.sh should detect linux/macos/arm64/x86_64 and download the right binary from GitHub Releases — standard pattern for Rust CLI tools
- The `.env.example` should be forward-looking: include commented sections for MCP, browser, additional gateways even if not implemented yet, so users see what's coming

</specifics>
