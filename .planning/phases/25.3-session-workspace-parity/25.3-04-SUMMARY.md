## Accomplishments

- `TrajectoryWriter::open(path)` creates parent directories (via `std::fs::create_dir_all`, analog to `StateStore::new`) and opens the file with `O_APPEND | O_CREAT` so re-opening preserves prior entries and writes are POSIX-line-atomic for payloads <= PIPE_BUF
- `TrajectoryWriter::append(&entry)` serializes one line, writes it with a trailing `\n`, then calls `self.file.sync_data()` — every entry hits storage before the next one starts, so a process kill BETWEEN lines cannot corrupt prior entries (RESEARCH.md §5 / Pitfall 3)
- `Drop for TrajectoryWriter` calls `self.file.sync_data().ok()` — best-effort flush on session end / panic / Ctrl+C; the `.ok()` honors the Drop-must-not-panic contract; `drop_flushes_final_entry` test reopens the file from cold and confirms the last entry persists
- `TrajectoryReader::open(path)` constructs a read-only handle (path may not exist yet); `read_all()` returns `Ok(Vec::new())` if the file is missing — Curator iteration is a defensive no-op for sessions that produced no trajectories
- `TrajectoryReader::read_all()` returns `Err` with `"parse trajectory line {idx} of {path}"` context on the first malformed line — loud failure surfaces format drift early; blank/whitespace lines are skipped (defensive against trailing newlines)
- 13 new tests green: 7 writer unit + 5 reader unit + 1 roundtrip integration. Plus the 8 pre-existing format tests from Plan 1 ⇒ total 21 tests for `cargo test -p ironhermes-trajectory` (5 format unit + 7 writer unit + 5 reader unit + 3 format integration + 1 roundtrip integration)
- `pub use writer::TrajectoryWriter` and `pub use reader::TrajectoryReader` re-exports added to `lib.rs` so downstream crates (Plan 8 CommandContext, Plan 9 AgentLoop, Plan 11 export, Phase 25.4 Curator) can import via the crate root

## Decisions Made

- **Multi-line `self.file.write_all(...)` + `self.file.sync_data()` formatting in `append()`** — chosen so the plan's literal `grep "self.file.sync_data"` >=2 acceptance contract holds (one match in `append`, one match in `Drop`) AND so each `Result` keeps its independent `.with_context()` telemetry. The alternative chained form `self.file.write_all(...).and_then(|_| self.file.sync_data())` would collapse the grep count to 1 and force a single `with_context` call covering both errors. Each fail-mode (write vs fsync) deserves its own context string.
- **`pub use reader::TrajectoryReader` re-export added in Task 2 GREEN, not Task 2 RED** — keeps the RED commit minimally intrusive (just `pub mod reader` so the test module is reachable). The integration test in `tests/roundtrip.rs` references `TrajectoryReader` via the crate root (`ironhermes_trajectory::TrajectoryReader`), so RED produces TWO E0432 errors (one in `reader::tests`, one in the integration test) — both confirm the missing API surface in different scopes.
- **`TrajectoryReader` stores only `path: PathBuf`, opens `File` per `read_all()` call** — matches D-T-2 ("auxiliary, never read at session-load"); no file-handle held between calls means Curator can construct/drop readers freely without resource-leak risk. Cost: a `read_all()` call re-opens the file each time. Benefit: simpler invariants, no `RefCell`/`Mutex` needed for interior mutability, matches the Plan 9/11/Curator usage shape (one read per session).
- **`#[allow(clippy::too_many_arguments)]` not needed in this plan** — `TrajectoryWriter::open` and `TrajectoryReader::open` are 1-arg functions; `append` is 2-arg; `read_all` is 0-arg. The 7-arg `TrajectoryEntry::success`/`::failure` constructors that needed the lint suppression live in Plan 1's `format.rs`, not in this plan's deliverables.

## Deviations from Plan

None — plan executed exactly as written.

The only minor implementation detail worth flagging: the plan's `<action>` block showed `append()` written as a chained `self.file.write_all(line.as_bytes())\n    .with_context(...)?; self.file.sync_data()\n    .with_context(...)?` form, which I initially wrote with a leading `self.file\n    .write_all(...)` newline-style. Rust's `cargo fmt` would have normalized it either way, but the plan's literal acceptance grep `self.file.sync_data` >=2 only counts whole-line substring matches — the multi-line form would have collapsed to a single matching line. Reformatted to keep `self.file.sync_data()` on one line (line 62 of writer.rs) to satisfy the literal acceptance contract. No semantic change. Counts as a no-op stylistic adjustment, not a deviation.
