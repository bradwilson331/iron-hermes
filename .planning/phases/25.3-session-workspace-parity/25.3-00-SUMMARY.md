## Decisions Made

- **session_from_row uses `.get(14).ok()`** — chose to tolerate narrower SELECT projections rather than force-update every internal SELECT. The 4 SELECTs that feed full Session structs project workspace_root explicitly (column 14); any future SELECT that doesn't project it gets `None` instead of an error. Belt-and-braces.
- **All 4 sessions SELECT projections updated** — would have been a real bug to leave them at 14 columns: workspace_root would silently always be `None` in returned Session structs because `get(14)` would fail on the row. Updating each of `get_session`, both branches of `list_sessions`, and `get_session_by_title` is required for correctness even though the fallible `.ok()` masks the error.
- **Plan 26-CONTEXT.md migration note placed verbatim** — used the exact wording from the plan's `<action>` Step 3 so the regression-grep `grep -c "Phase 25.3 update (2026-05-03)"` returns 1.
- **PROV-06 historical note** — added "(Phase 26 originally shipped with 5 roles; Phase 25.2 D-13 added summarization, Phase 25.3 D-P0-1 added curator)" to the Phase 26 PROV-06 scope description so future readers don't have to grep STATE.md to reconstruct the chain.
