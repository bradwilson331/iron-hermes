<domain>
## Phase Boundary

Bring IronHermes' session model up to parity with the [hermes-agent](https://github.com/NousResearch/hermes-agent) (Python) reference architecture, AND introduce a first-class `Workspace` abstraction that unifies the project-level rules layer (SOUL/AGENTS/MEMORY/skills/tools_config). The phase delivers three concrete subsystems plus one mechanical Plan 0:

1. **`ironhermes-trajectory` crate** — RL-grade tool-execution ledger (`trajectories.jsonl`) per session.
2. **Hybrid flat-file session export** — SQLite `StateStore` stays canonical; on-demand `hermes session export <id>` + `/export-session` slash produce hermes-agent's `~/.hermes/sessions/[id]/{messages,metadata,context}.json` folder layout for tooling/RL pipelines.
3. **`Workspace` newtype** — `Workspace { root, soul_path, agents_chain, memory_dir, skills_dir, tools_config }` in `ironhermes-core`. Sessions table gains `workspace_root TEXT` column. Workspace consumed by `/sessions` filter, system prompt slot, and trajectory directory scoping. (The `skills_dir` field is wired and documented for Phase 25.4's Curator output destination, but no curator code lands here.)
4. **Plan 0 — Phase 26 cascade prerequisite for 25.4** — extend `RESERVED_ROLE_NAMES` from 6 → 7 with `"curator"` AND update the validation test. Same shape as Phase 25.2 Plan 02's `summarization` addition. Mechanical change; lands as the first plan of 25.3 so 25.4 can `resolve_role("curator")` without forward references.

**Out of scope for 25.3 (capture as deferred ideas if they come up during planning):** Curator + Learning Loop subsystem (full content moved to Phase 25.4); RL training pipeline that consumes trajectories; multi-workspace switching mid-session; trajectory retention/eviction policy; daemon-based curator sweep across all surfaces; per-workspace skill registry (skills runtime is single-registry).

**Carved out to Phase 25.4:** `tools/curator.py` Rust port (heuristic gate, LLM educational-value scoring, skill emission, idle-timeout sweep, `/curate` slash + `hermes curate` CLI, `AnyClientCuratorHandle`, `CURATOR_PROMPT` authorship). 25.3 leaves these as forward references — every prerequisite (trajectory crate, Workspace newtype, RESERVED_ROLE_NAMES["curator"]) lands here so 25.4 has clean foundations.

</domain>

<decisions>
## Implementation Decisions

### Trajectories ledger (new `ironhermes-trajectory` crate)

- **D-T-1:** **Full RL-grade detail per tool call.** Each `trajectories.jsonl` line records `{ name, args (post-redact), result_or_error, duration_ms, impact_level (Read=0/Write=5/SystemChange=10), turn_index, tool_call_id, ts }`. Note: this is an IronHermes-original format — 25.3 RESEARCH.md confirmed the upstream Python `agent/trajectory.py` stores session-level ShareGPT, NOT per-tool-call records. Implement D-T-1 verbatim regardless. Cost: ~1–10 KB/call uncompressed; acceptable for typical sessions.
- **D-T-2:** **New `ironhermes-trajectory` crate** (separate workspace member, sibling to `ironhermes-state`). Contains `TrajectoryWriter` (append-only, fsync per line for crash safety) + `TrajectoryReader` (consumed by Phase 25.4 Curator). SQLite `state.db` stays canonical for messages; trajectories.jsonl is auxiliary, never read at session-load.
- **D-T-3:** **All four surfaces write trajectories** — CLI (`run_single`), classic-TUI REPL (`run_chat`), ratatui REPL (`run_chat_ratatui`), Telegram gateway (`run_gateway` per-message handler). Telegram is the primary user-facing surface, so excluding it would starve 25.4's training-data source. Same wireup pattern as Phase 25.2 Plan 15 + the post-UAT addendum (`tui_rata::build_app_deps` + `GatewayRunner::set_*`).
- **D-T-4:** **No automatic eviction in v1.** Trajectories are append-only and small relative to messages. Operator removes manually if needed. Retention policy (TTL, tiered compression) deferred to a follow-up phase once real disk-volume data exists.

### Flat-file session layout (hermes-agent compat — hybrid)

- **D-F-1:** **Hybrid: SQLite canonical + JSON files derived on demand.** `state.db` remains source of truth; no double-writes during normal session lifecycle (avoids consistency drift, no WAL-vs-fsync gymnastics). Add `hermes session export <id>` CLI + `/export-session` slash that produces `~/.ironhermes/sessions/<id>/{messages.json, metadata.json, context.json, trajectories.jsonl}` on demand. trajectories.jsonl is symlinked or copied from the trajectory crate's primary location.
- **D-F-2:** **Bulk-export via `hermes session export-all [--since YYYY-MM-DD]`.** Operator opts in; can scope by date. No automatic backfill on upgrade — existing sessions stay in SQLite until explicitly exported. Surprise-free; respects operators with thousands of sessions.

### Workspace abstraction (new `ironhermes-core` newtype)

- **D-W-1:** **`pub struct Workspace { root, soul_path, agents_chain, memory_dir, skills_dir, tools_config }` in `ironhermes-core`.** Resolved once at session start (frozen-snapshot pattern, mirrors Phase 18 D-04). The `sessions` table gains `workspace_root TEXT NULL` column (schema migration v8). Resolution: walk up from cwd looking for `.hermes/`, `.ironhermes/`, or fall back to `~/.ironhermes/` (global). Multi-workspace mid-session switching deferred. Aligns with Phase 24's `--profile` concept but is per-cwd, not per-HERMES_HOME-pivot (verified orthogonal in 25.3 RESEARCH.md).
- **D-W-2:** **Workspace consumed by three touchpoints in 25.3** (Curator output destination is wired here for 25.4 to consume but no curator code lands in 25.3):
  - **`/sessions` filter:** `/sessions --workspace .` and `/sessions --workspace <path>` filter session list by `workspace_root` column. Default `/sessions` lists current cwd's workspace. CLI subcommand parity: `hermes sessions list --workspace .`.
  - **System prompt slot:** new line `[Workspace: <root>]` in `prompt_builder.rs` (slot index TBD by planner — likely sub-slot inside Identity for cache-stability — see RESEARCH.md). Frozen at session start.
  - **Trajectory directory scoping:** trajectories live at `<workspace>/.ironhermes/sessions/<id>/trajectories.jsonl` instead of `~/.ironhermes/sessions/<id>/trajectories.jsonl` when a workspace is resolved. Multi-project users keep training-data segmented per project. Fallback to global path when no workspace.
  - **(Phase 25.4 forward-reference)** Curator skill output destination — `Workspace.skills_dir` is populated by 25.3 so 25.4's Curator emission code can read it without re-implementing resolution.

### Plan 0 — Phase 26 cascade prerequisite for Phase 25.4

- **D-P0-1:** **Extend `RESERVED_ROLE_NAMES` from 6 → 7 with `"curator"`** in `crates/ironhermes-core/src/config.rs`. Update `validate_role_name_accepts_all_reserved_roles` test (no body change needed — iterates the array — but verify the test exists per Phase 25.2 STATE.md log; create if missing). Update doc comments referencing "six roles" → "seven roles" (same three-site update as Phase 25.2 Plan 02). Lands as the first plan of 25.3 so all subsequent plans (and Phase 25.4) can `resolve_role("curator")` without forward references. Also update Phase 26 CONTEXT.md `AuxiliaryConfig` documentation to "seven roles" as part of this plan.

### Claude's Discretion

- **Slot index for the `[Workspace: <root>]` system prompt addition** — planner picks based on Phase 18 prompt-slot ordering. RESEARCH.md recommends: append to the Identity slot content (slot 1) — simplest, safest, cache-stable. Cache-awareness: this line MUST be cache-stable across the session (frozen-snapshot), otherwise it breaks the Anthropic prompt cache.
- **Trajectory `args` redaction layer** — `web_extract`'s D-19 `redact_secrets_in_url` is in scope but the trajectory writer also needs to redact secrets in OTHER tool args (e.g., `terminal` env vars, `write_file` content, MCP arg payloads). RESEARCH.md recommends: **Option B — add `Tool::redact_args(&self, raw: &Value) -> Value` method to the `Tool` trait** with default `{ raw.clone() }`. Tools that handle secrets override the default. Cleaner than a central map and scales naturally. Phase 25.2 Plan 16's `redact_secrets_in_url` is still callable from individual tool overrides for URL-typed args.

</decisions>

## Existing Code Insights

### Reusable Assets

- **`StateStore` (Phase 13)** — owns sessions/messages tables; SQLite + WAL + FTS5. Phase 25.3 adds `workspace_root` column via standard schema migration v8. No new persistence layer.
- **`SessionStore` (Phase 13) + `SessionKey`** — in-memory cache + key (platform, chat_id, user). Phase 25.4's idle-timeout sweep iterates over `SessionStore` entries; existing `last_message_at` column tracks idle. (25.3 doesn't ship the sweep itself — it just confirms `SessionStore` already exposes what 25.4 needs.)
- **Phase 26 cascade (`resolve_role`)** — drop-in: register `"curator"` as the 7th role in Plan 0. Operator routes to a cheap model. Same shape as Phase 25.2 Plan 02 added `"summarization"`.
- **`build_command_context` parity-guard pattern** — Phase 25.2 VERIFICATION addendum lesson: the workspace + trajectory wireups MUST land in all four CommandContext-construction sites simultaneously, locked by a parity-guard test that greps the workspace (not just `main.rs`).
- **`redact_secrets_in_url` (Phase 25.2 Plan 16)** — precedent shape for trajectory args redaction. Discretion D-2 chose the per-tool `Tool::redact_args` method approach.
- **Existing `SessionExport` / `StateStore::export_session` machinery** — `ironhermes-state` already has session-export primitives (`StateStoreHandle::export_session_text` per RESEARCH.md). D-F-1's `export <id>` reuses this as the data source and adds a 4-file directory writer on top.

### Established Patterns

- **Frozen-snapshot pattern** (Phase 17/18) — Workspace is resolved once at session start, captured into `Session.workspace_root`, never mutated mid-session. Cache-stable. Mid-session workspace switching is explicitly deferred.
- **Cross-surface wireup parity** (Phase 25.2 VERIFICATION addendum) — every new field threaded through `CommandContext` requires updates in `main.rs::build_cmd_ctx`, `tui_rata::build_command_context`, AND `gateway::handler::handle_command` (via `GatewayRunner::set_*` plumbing). Workspace + Trajectory each pay this 3-site tax in 25.3.
- **Append-only JSONL with fsync per line** — the trajectory crate follows this pattern. `Drop for TrajectoryWriter` must call `file.sync_data().ok()` to flush on session end (RESEARCH.md Pitfall 3).
- **`RESERVED_ROLE_NAMES` extension** — Phase 25.2 Plan 02 is the template: add the literal, update the doc comment, no validate_role_name body change (auto-iterates).
- **Hybrid SQLite + JSON files** — operator-on-demand export, not auto-mirror, avoids the consistency drift class of bugs (see Plan 21.7-11 / INV-21.7-08 lock-step parity discussion).
- **SQLite schema migration version-gating** — `if current < N { ALTER TABLE ...; UPDATE schema_version SET version = N }`. Do NOT use `let _ = ALTER TABLE` ignore-error pattern (RESEARCH.md Pitfall 5).

### Integration Points

- **`Workspace` resolution at session start** — three `run_*` functions in `main.rs` + `tui_rata::build_app_deps` + `GatewayMessageHandler::handle` all need the resolution helper. Single helper in `ironhermes-core::workspace::resolve_from_cwd` consumed by all four sites. Plus an `Option<Arc<Workspace>>` threaded through `CommandContext` for slash dispatch.
- **`TrajectoryWriter` lifecycle** — created at session start (alongside `StateStore` write), receives every tool-call result via a thin AgentLoop callback (similar to existing `StreamCallback` shape). Drop on session end. Per-surface AppDeps/Runner threading.
- **Flat-file export dispatch** — slash command `/export-session` lives in `commands/handlers.rs` (joining the existing 49-command catalog from Phase 21.1). CLI subcommand `hermes session export <id>` and `hermes session export-all` are `Subcommand` arms in `cli_args.rs` under a new `Session(SessionCommand)` group. Pattern: same as the Phase 25 `Toolset(ToolsetCommand)` group.
- **Plan 0 cascade extension** — adding `"curator"` to `RESERVED_ROLE_NAMES` is a Phase 26 contract change; the array literal lives in `ironhermes-core::config`. Validation test auto-covers it. Doc comment three-site update mirrors Phase 25.2 Plan 02.
