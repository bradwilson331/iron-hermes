# Phase 25.3: session-workspace parity - Context

**Gathered:** 2026-05-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Bring IronHermes' session model up to parity with the [hermes-agent](https://github.com/NousResearch/hermes-agent) (Python) reference architecture, and introduce a first-class `Workspace` abstraction that unifies the project-level rules layer (SOUL/AGENTS/MEMORY/skills/tools_config). The phase delivers four concrete subsystems:

1. **`ironhermes-trajectory` crate** — RL-grade tool-execution ledger (`trajectories.jsonl`) per session.
2. **Curator + Learning Loop** — bridges episodic memory (sessions/trajectories) to procedural memory (skills) via a heuristic gate + LLM educational-value scoring (Phase 26 cascade with a new `curator` RESERVED_ROLE).
3. **Hybrid flat-file session export** — SQLite `StateStore` stays canonical; on-demand `hermes session export <id>` + `/export-session` slash produce hermes-agent's `~/.hermes/sessions/[id]/{messages,metadata,context}.json` folder layout for tooling/RL pipelines.
4. **`Workspace` newtype** — `Workspace { root, soul_path, agents_chain, memory_dir, skills_dir, tools_config }` in `ironhermes-core`. Sessions table gains `workspace_root TEXT` column. Workspace consumed by Curator (output destination), `/sessions` filter, system prompt slot, and trajectory directory scoping.

**Out of scope (capture as deferred ideas if they come up during planning):** RL training pipeline that consumes trajectories; multi-workspace switching mid-session; trajectory retention/eviction policy; daemon-based curator sweep across all surfaces; per-workspace skill registry (skills runtime is single-registry).

**Scope concern (flagged for /gsd-plan-phase to evaluate):** Phase 25.3 spans four subsystems with multi-decisions each. The Curator alone is a non-trivial subsystem (heuristic + LLM-scored + skill-template generation + idle-timeout integration). Planner should consider whether a 2-phase split (25.3 = trajectories + flat-file export + Workspace newtype; 25.4 = Curator + Learning Loop) is more tractable than a single-phase landing. Recorded so the planner can size honestly rather than collapsing under hidden complexity.

</domain>

<decisions>
## Implementation Decisions

### Trajectories ledger (new `ironhermes-trajectory` crate)

- **D-T-1:** **Full RL-grade detail per tool call.** Each `trajectories.jsonl` line records `{ name, args (post-redact), result_or_error, duration_ms, impact_level (Read=0/Write=5/SystemChange=10), turn_index, tool_call_id, ts }`. Mirrors hermes-agent's trajectory format. Cost: ~1–10 KB/call uncompressed; acceptable for typical sessions.
- **D-T-2:** **New `ironhermes-trajectory` crate** (separate workspace member, sibling to `ironhermes-state`). Contains `TrajectoryWriter` (append-only, fsync per line for crash safety) + `TrajectoryReader` for the Curator. SQLite `state.db` stays canonical for messages; trajectories.jsonl is auxiliary, never read at session-load.
- **D-T-3:** **All four surfaces write trajectories** — CLI (`run_single`), classic-TUI REPL (`run_chat`), ratatui REPL (`run_chat_ratatui`), Telegram gateway (`run_gateway` per-message handler). Telegram is the primary user-facing surface, so excluding it would starve the Curator's training-data source. Same wireup pattern as Phase 25.2 Plan 15 + the post-UAT addendum (`tui_rata::build_app_deps` + `GatewayRunner::set_*`).
- **D-T-4:** **No automatic eviction in v1.** Trajectories are append-only and small relative to messages. Operator removes manually if needed. Retention policy (TTL, tiered compression) deferred to a follow-up phase once real disk-volume data exists.

### Curator + Learning Loop (`tools/curator.py` Rust port)

- **D-C-1:** **Two triggers:** (a) end-of-session auto-fire on `/new`, `/clear`, gateway idle-timeout — runs **heuristic gate only** (no LLM cost); (b) on-demand `/curate <session?>` slash + `hermes curate <session>` CLI — runs the **full pipeline** (heuristic + LLM scoring + skill generation). User can curate any session retroactively.
- **D-C-2:** **Heuristic gate + LLM educational-value score ≥ 7.** Heuristic: `tool_calls ≥ 3` AND at least one impactful tool (terminal, write_file, patch_file, mcp_call) AND `duration ≥ 30s`. If passes, build trajectory summary + send through Phase 26 cascade as a NEW RESERVED_ROLE `"curator"`. LLM returns 1–10 score; ≥7 promotes. Operator routes curator calls to a cheap model (Haiku/llama) via `model.roles["curator"]` or `auxiliary.curator` — same cascade as `summarization` (Phase 25.2 D-13). Phase 26 `RESERVED_ROLE_NAMES` extends from 6 → 7.
- **D-C-3:** **Generated skills use the existing Phase 19 SKILL.md format verbatim.** Curator is just another skill author. Output to `<workspace>/skills/<slug>/SKILL.md` (project-local) when a Workspace is active; falls back to `~/.ironhermes/skills/<slug>/SKILL.md` (global) otherwise. Frontmatter: `name`, `category`, `allowed_tools`, `env_requirements`. No curator-only fields. Clean integration with Phase 21.8 install/update/uninstall.
- **D-C-4:** **Gateway end-of-session = idle-timeout cutoff.** New `cfg.gateway.session_idle_minutes` config (default 30). Background tokio task in `GatewayRunner::start` sweeps `SessionStore` periodically; for any `SessionKey` idle ≥ threshold, fires the heuristic curator on its trajectories. Gateway users get automatic curation without manual `/curate`. CLI `/new` and `/clear` keep their per-session triggers.

### Flat-file session layout (hermes-agent compat — hybrid)

- **D-F-1:** **Hybrid: SQLite canonical + JSON files derived on demand.** `state.db` remains source of truth; no double-writes during normal session lifecycle (avoids consistency drift, no WAL-vs-fsync gymnastics). Add `hermes session export <id>` CLI + `/export-session` slash that produces `~/.ironhermes/sessions/<id>/{messages.json, metadata.json, context.json, trajectories.jsonl}` on demand. trajectories.jsonl is symlinked or copied from the trajectory crate's primary location.
- **D-F-2:** **Bulk-export via `hermes session export-all [--since YYYY-MM-DD]`.** Operator opts in; can scope by date. No automatic backfill on upgrade — existing sessions stay in SQLite until explicitly exported. Surprise-free; respects operators with thousands of sessions.

### Workspace abstraction (new `ironhermes-core` newtype)

- **D-W-1:** **`pub struct Workspace { root, soul_path, agents_chain, memory_dir, skills_dir, tools_config }` in `ironhermes-core`.** Resolved once at session start (frozen-snapshot pattern, mirrors Phase 18 D-04). The `sessions` table gains `workspace_root TEXT NULL` column. Resolution: walk up from cwd looking for `.hermes/`, `.ironhermes/`, or fall back to `~/.ironhermes/` (global). Multi-workspace mid-session switching deferred. Aligns with Phase 24's `--profile` concept but is per-cwd, not per-HERMES_HOME-pivot.
- **D-W-2:** **Workspace consumed by all four touchpoints** in Phase 25.3:
  - **Curator output:** writes `<workspace>/skills/` first, falls back to `~/.ironhermes/skills/` only when no workspace is active.
  - **`/sessions` filter:** `/sessions --workspace .` and `/sessions --workspace <path>` filter session list by `workspace_root` column. Default `/sessions` lists current cwd's workspace. CLI subcommand parity: `hermes sessions list --workspace .`.
  - **System prompt slot:** new line `[Workspace: <root>]` in `prompt_builder.rs` (slot index TBD by planner — likely between Identity and Tool-use). Frozen at session start.
  - **Trajectory directory scoping:** trajectories live at `<workspace>/.ironhermes/sessions/<id>/trajectories.jsonl` instead of `~/.ironhermes/sessions/<id>/trajectories.jsonl` when a workspace is resolved. Multi-project users keep training-data segmented per project. Fallback to global path when no workspace.

### Claude's Discretion

- **Slot index for the `[Workspace: <root>]` system prompt addition** — planner picks based on Phase 18 prompt-slot ordering. Recommendation: between Identity (slot 1) and Tool-use guidance (slot 2), or as a sub-slot inside Identity. Cache-awareness: this line MUST be cache-stable across the session (frozen-snapshot), otherwise it breaks the Anthropic prompt cache.
- **Trajectory `args` redaction layer** — `web_extract`'s D-19 `redact_secrets_in_url` is in scope but the trajectory writer also needs to redact secrets in OTHER tool args (e.g., `terminal` env vars, `write_file` content, MCP arg payloads). Planner should generalize the redactor or add a per-tool `redact_args(&self, raw) -> Value` method to the `Tool` trait. Phase 25.2 Plan 16's `redact_secrets_in_url` is the precedent shape.
- **Curator concurrency** — when multiple sessions reach idle-timeout simultaneously, do they curate in parallel or serialize? Recommendation: serialize via a single `Arc<Semaphore>(1)` so the curator role's LLM cost is bounded and predictable. Planner should confirm.
- **Phase 26 cascade `curator` role registration** — adding a new `RESERVED_ROLE_NAMES` entry is a Phase 26 decision; Phase 25.3 plan should include a Plan 0 task to extend the array AND update the validation test (`validate_role_name_accepts_all_reserved_roles`) before the curator code lands. Same shape as Phase 25.2 Plan 02's `summarization` addition.

</decisions>

## Existing Code Insights

### Reusable Assets

- **`StateStore` (Phase 13)** — owns sessions/messages tables; SQLite + WAL + FTS5. Phase 25.3 adds `workspace_root` column via standard schema migration. No new persistence layer.
- **`SessionStore` (Phase 13) + `SessionKey`** — in-memory cache + key (platform, chat_id, user). The Curator's idle-timeout sweep iterates over `SessionStore` entries; existing `last_message_at` column tracks idle.
- **Phase 26 cascade (`resolve_role`)** — drop-in: register `"curator"` as the 7th role. Operator routes to a cheap model. Same shape as Phase 25.2 Plan 02 added `"summarization"`.
- **`AnyClientSummarizationHandle` (Phase 25.2 Plan 14)** — port verbatim into a `AnyClientCuratorHandle` for the curator's LLM scoring. `any_client.rs` already has the helper pattern.
- **Phase 19 SKILL.md format + Phase 21.8 skill registry** — Curator emits skills into the existing pipeline; install/update/uninstall lifecycle stays untouched.
- **`build_command_context` parity-guard pattern** — Phase 25.2 VERIFICATION addendum lesson: the workspace + curator + trajectory wireups MUST land in all four CommandContext-construction sites simultaneously, locked by a parity-guard test that greps the workspace (not just `main.rs`).
- **`redact_secrets_in_url` (Phase 25.2 Plan 16)** — precedent shape for trajectory args redaction. Generalize OR keep narrow + per-tool redact methods (planner decides).

### Established Patterns

- **Frozen-snapshot pattern** (Phase 17/18) — Workspace is resolved once at session start, captured into `Session.workspace_root`, never mutated mid-session. Cache-stable. Mid-session workspace switching is explicitly deferred.
- **Cross-surface wireup parity** (Phase 25.2 VERIFICATION addendum) — every new field threaded through `CommandContext` requires updates in `main.rs::build_cmd_ctx`, `tui_rata::build_command_context`, AND `gateway::handler::handle_command` (via `GatewayRunner::set_*` plumbing). Workspace + Curator + Trajectory each pay this 3-site tax.
- **Append-only JSONL with fsync per line** — the trajectory crate follows this Pitfall pattern (used elsewhere in `ironhermes-state` for FTS5 batch flush).
- **`RESERVED_ROLE_NAMES` extension** — Phase 25.2 Plan 02 is the template: add the literal, update the test that asserts every reserved role auto-accepts in `validate_role_name`, no other changes.
- **Hybrid SQLite + JSON files** — operator-on-demand export, not auto-mirror, avoids the consistency drift class of bugs (see Plan 21.7-11 / INV-21.7-08 lock-step parity discussion).

### Integration Points

- **`Workspace` resolution at session start** — three `run_*` functions in `main.rs` + `tui_rata::build_app_deps` + `GatewayMessageHandler::handle` all need the resolution helper. Single helper in `ironhermes-core::workspace::resolve_from_cwd` consumed by all four sites. Plus an `Option<Arc<Workspace>>` threaded through `CommandContext` for slash dispatch.
- **`TrajectoryWriter` lifecycle** — created at session start (alongside `StateStore` write), receives every tool-call result via a thin AgentLoop callback (similar to existing `StreamCallback` shape). Drop on session end. Per-surface AppDeps/Runner threading.
- **Curator dispatch** — slash command `/curate` lives in `commands/handlers.rs` (joining the existing 49-command catalog from Phase 21.1). CLI subcommand `hermes curate <session>` is a separate `Subcommand` arm in `cli_args.rs`. Idle-timeout sweep is a `tokio::spawn` task in `GatewayRunner::start` sibling to the existing MCP shutdown task.
- **Skill emission** — Curator writes to `<workspace>/skills/<slug>/SKILL.md` and registers via the existing Phase 19 `SkillRegistry::register` path. Skills runtime is single-registry; the workspace-vs-global decision is purely about WRITE destination, not READ resolution.
- **Phase 26 `curator` role** — Plan 0 adds the literal to `RESERVED_ROLE_NAMES`, updates the validation test, and updates the `AuxiliaryConfig` cascade doc to "seven roles". Mechanical change; lands in Plan 0 of Phase 25.3 so all subsequent plans can `resolve_role("curator")` without forward-references.

## Specific Ideas

- User explicitly cited the hermes-agent reference architecture as the parity target. The Python files `agent/trajectory.py`, `tools/curator.py`, `trajectory_compressor.py`, `hermes_state.py` are the load-bearing references — researchers should read them before proposing alternative algorithms.
- User's "Educational Value" curator prompt template (verbatim from the Python source) should land in `crates/ironhermes-tools/src/curator.rs` as `const CURATOR_PROMPT: &str = ...` — frozen contract, lock-tested for byte equality in the unit tests.
- User's `Curatable` trait sketch (`fn educational_value(&self) -> Score; fn is_worthy_of_learning(&self) -> bool { self.educational_value().rank >= 7 }`) is the recommended Rust shape — researchers may adapt but should keep the trait pattern for testability.

<deferred>
## Deferred Ideas

- **RL training pipeline** that consumes `trajectories.jsonl` to produce a fine-tuning dataset (HuggingFace-format SFT/DPO). Phase 25.3 produces the data; the pipeline that consumes it is a future phase.
- **Multi-workspace mid-session switching** — explicit `/workspace switch <path>` slash + cache invalidation. Frozen-snapshot pattern locks this for v1.
- **Trajectory retention/eviction policy** — TTL or tiered compression. Defer until real-volume data exists.
- **Daemon-based curator sweep across all surfaces** (REPL + single-shot + Telegram) — Phase 25.3 only sweeps the gateway via idle-timeout. A unified daemon is a follow-up.
- **Per-workspace skill registry** — currently the Skills runtime is single-registry; workspace-local skills WRITE to `<workspace>/skills/` but the registry READS from a unified merge. True per-workspace registries (with isolation) is a future architecture decision.
- **Curator dedup against existing skills** — embedding-search or substring-match on existing `~/.ironhermes/skills/` before generating a new one. Useful but adds vector-store dependency. Defer to a follow-up that targets skill-folder bloat.
- **Curator-only frontmatter extension** (`source: curator`, `source_session_id`, `educational_score`) — kept explicitly out of v1 to keep Curator skills indistinguishable from hand-authored ones. Reconsider if the Curator generates a high enough volume that operators need a `--curated` filter on `/skills list`.

</deferred>
