### 4. Workspace Resolution Algorithm

**Resolution order (D-W-1, ASSUMED shape — no existing helper found):** [ASSUMED]

```rust
/// Resolve workspace root by walking up from `cwd` looking for markers.
/// Returns Some(root) if a workspace marker is found, None for global-only mode.
pub fn resolve_workspace_from_cwd(cwd: &Path) -> Option<PathBuf> {
    let mut current = cwd.to_path_buf();
    loop {
        // Check for .hermes/ or .ironhermes/ directory
        if current.join(".hermes").is_dir() || current.join(".ironhermes").is_dir() {
            return Some(current);
        }
        // Walk up to parent
        if !current.pop() {
            break; // reached filesystem root
        }
    }
    None // global fallback: caller uses get_hermes_home()
}
```

**Workspace struct shape (from D-W-1):**

```rust
pub struct Workspace {
    pub root: PathBuf,           // resolved directory containing .hermes/ or .ironhermes/
    pub soul_path: Option<PathBuf>,     // <root>/SOUL.md or <root>/.hermes/SOUL.md
    pub agents_chain: Vec<PathBuf>,     // AGENTS.md / CLAUDE.md / .hermes.md chain
    pub memory_dir: PathBuf,           // <root>/.ironhermes/memory/ or <root>/.hermes/memory/
    pub skills_dir: PathBuf,           // <root>/skills/ (write destination for Curator)
    pub tools_config: Option<PathBuf>, // <root>/.ironhermes/tools.yaml or similar
}
```

**Frozen-snapshot semantics:** Workspace is resolved once at session start, stored in `Session.workspace_root: Option<String>` (serialized as the root path), and frozen for the session lifetime. The `[Workspace: <root>]` system prompt line is injected at session start and never changes mid-session (same frozen-snapshot pattern as Phase 17/18 memory). [VERIFIED: pattern from StateStore Session struct + Phase 17 precedent]

**Orthogonality with Phase 24 `--profile` flag:** [VERIFIED: Phase 24 CONTEXT.md D-03]
- `--profile` pivots `IRONHERMES_HOME` (the global agent home directory)
- `Workspace` resolution walks up from `cwd` looking for project-local markers
- These are independent axes: a user can have `--profile work` AND a workspace at `/projects/myapp/`
- No interaction code needed; the resolution helper uses `cwd` (not `HERMES_HOME`)

**Trajectory directory scoping:**
- With workspace: `<workspace.root>/.ironhermes/sessions/<session_id>/trajectories.jsonl`
- Without workspace (global): `~/.ironhermes/sessions/<session_id>/trajectories.jsonl`
- Fallback path uses `get_hermes_home()` from `ironhermes-core::constants`

**System prompt slot:** PromptSlot enum currently has slots 1-10 (Identity=1, SystemMessage=2, ToolGuidance=3, Memory=4, Skills=5, ContextFiles=6, Timestamp=7, PlatformHints=8, SessionOverlay=9, UserMessage=10). [VERIFIED: prompt_builder.rs read]

Slots 1-6 are durable/cacheable; slots 7-10 are ephemeral. The `[Workspace: <root>]` line MUST go in a durable slot to be cache-stable. Recommendation: sub-slot inside Identity (slot 1) by appending to the identity string, OR introduce a new `PromptSlot::WorkspaceHint = 2` (shifting SystemMessage to 3, etc. — risky, breaks ordering). Simplest approach: append to the Identity slot content after the DEFAULT_AGENT_IDENTITY text. This is Claude's Discretion; planner decides.

**Cache-stability requirement:** The workspace root is frozen at session start and never changes, so it is safe to place in any durable slot (1-6). The key invariant is that the value cannot change mid-session, which the frozen-snapshot guarantees.

### 5. Trajectory Writer Crash-Safety

**Existing precedent for append-only JSONL with fsync:** [VERIFIED: ironhermes-state migration pattern]

The FTS5 batch-flush in `ironhermes-state` does `execute_batch` inside a transaction — different from fsync per-line. The trajectory writer has stricter requirements (crash safety at line granularity) because a process kill between lines must not corrupt existing entries.

**Recommended write path:**

```rust
pub struct TrajectoryWriter {
    file: std::fs::File,          // opened with OpenOptions::append(true)
    session_id: String,
    path: PathBuf,
}

impl TrajectoryWriter {
    pub fn append(&mut self, entry: &TrajectoryEntry) -> anyhow::Result<()> {
        use std::io::Write;
        let line = serde_json::to_string(entry)? + "\n";
        self.file.write_all(line.as_bytes())?;
        self.file.sync_data()?;   // fsync per line — crash safety
        Ok(())
    }
}
```

**Multi-process writer concern:** The IronHermes design has one process per session surface (gateway handles one bot token). The scenario of gateway + REPL writing to the same session's trajectories.jsonl simultaneously cannot happen in practice (a single session_id is owned by one surface at a time). [ASSUMED — no locking analysis found; verify if gateway and REPL can share session IDs]. If they can, `O_APPEND` POSIX semantics guarantee line-level atomicity for writes up to PIPE_BUF (4096 bytes on Linux) — a single JSONL line for a trajectory entry will typically be well under this limit.

**Per-tool args redaction — recommendation:**

CONTEXT.md flags this as Claude's Discretion. Two options:

| Option | Shape | Pros | Cons |
|--------|-------|------|------|
| A: Generalize `redact_secrets_in_url` | `redact_args(args: &Value, tool_name: &str) -> Value` in `sanitize.rs` | No Tool trait change | Requires a tool-name → redaction strategy map; hard to extend per-tool |
| B: `Tool::redact_args(&self, raw: &Value) -> Value` | Method on `Tool` trait with default `{ raw.clone() }` | Each tool controls its own redaction | Adds a method to the Tool trait (minor API surface change) |

**Recommendation: Option B.** The `Tool` trait already has `fn prerequisites() -> Vec<Prerequisite>` with a default no-op. Adding `fn redact_args(&self, raw: &Value) -> Value { raw.clone() }` follows the same pattern. Tools that handle secrets (web_extract, terminal, write_file, MCP tools) override the default. This is cleaner than a central map and scales naturally. The trajectory writer calls `tool.redact_args(&raw_args)` before serializing.

The `redact_secrets_in_url` from Phase 25.2 Plan 16 is still used for URL-typed args (web tools can call it from within their `redact_args` override).

### 7. Schema Migration for `workspace_root TEXT NULL`

**Existing migration runner shape:** [VERIFIED: ironhermes-state/src/lib.rs read]

```rust
// Current SCHEMA_VERSION = 7
// Migration pattern in run_migrations():
if current < 8 {
    // v8: add workspace_root to sessions
    let _ = self.conn.execute(
        "ALTER TABLE sessions ADD COLUMN workspace_root TEXT",
        []
    );
    self.conn.execute("UPDATE schema_version SET version = 8", [])?;
}
```

Key points:
- SQLite `ALTER TABLE ADD COLUMN` with no DEFAULT is valid — existing rows get NULL automatically (TEXT NULL semantics). [VERIFIED: SQLite documentation behavior, standard]
- The `let _ =` pattern (ignore AlreadyExists error) is used for CREATE INDEX idempotency but NOT for ALTER TABLE — ALTER TABLE is non-idempotent if the column already exists. The `if current < 8` version gate is the correct guard.
- The `SCHEMA_SQL` constant defines the fresh-install schema. Phase 25.3 should add `workspace_root TEXT` to the CREATE TABLE sessions DDL AND add the migration in `run_migrations` for existing databases.

## Risks and Pitfalls

### Pitfall 1: Parity-Guard Scope Blindness (LOAD-BEARING)
**What goes wrong:** Implementing Workspace, TrajectoryWriter, and CuratorHandle in `main.rs::build_cmd_ctx` but forgetting `tui_rata/commands.rs::build_command_context` and `gateway/handler.rs::handle_slash_command`. The ratatui REPL has been the DEFAULT `hermes chat` since Phase 22.4 — missing it means the features don't work for the primary user-facing surface.
**Why it happens:** Phase 25.2 fell into this trap exactly. The Phase 25.2 parity test only grep'd `main.rs`. The fix required two follow-up commits after UAT failure.
**How to avoid:** Write the workspace-wide parity-guard test FIRST (Plan 0 or Plan 1), before implementing any CommandContext field. Make it red, then green.
**Warning signs:** Any CommandContext field that has `.with_field()` in `main.rs` but the grep count for `with_field` across the workspace is only 1 (not 3+).

### Pitfall 2: Prompt Cache Breakage from Non-Frozen Workspace Line
**What goes wrong:** The `[Workspace: <root>]` system prompt line is placed in an ephemeral slot (>= Timestamp = slot 7), causing the Anthropic prompt cache to miss on every turn.
**Why it happens:** Slot placement is confusing when adding a new slot to the existing ordering.
**How to avoid:** Append to the Identity slot content (slot 1) — the simplest, safest approach. The workspace root is frozen at session start so it cannot change and is safe in a durable slot.
**Warning signs:** Prompt cache hit rate drops to 0% after Phase 25.3 lands.

### Pitfall 3: TrajectoryWriter File Handle Leak on Session End
**What goes wrong:** `TrajectoryWriter` holds an open `std::fs::File`. If the session ends via panic, Ctrl+C, or graceful shutdown without explicit `Drop`, the final fsync may not fire.
**Why it happens:** The `Drop` implementation must call `sync_data()`.
**How to avoid:** Implement `Drop for TrajectoryWriter` to call `file.sync_data().ok()`. The `ok()` prevents panic in Drop context.

### Pitfall 4: Curator LLM Call in Hot Path
**What goes wrong:** The curator's LLM scoring call (D-C-2) fires synchronously in the gateway message handler, blocking the next Telegram message from being processed.
**Why it happens:** Easy to accidentally `await` the curator in the per-message dispatch path.
**How to avoid:** The on-demand `/curate` and the idle-timeout sweep both run in background tasks (`tokio::spawn`), not inline in the message handler. The auto-fire heuristic-only path on `/new` and `/clear` is also async/spawned.

### Pitfall 5: schema_version Check Missing for v8 Migration
**What goes wrong:** Running `ALTER TABLE sessions ADD COLUMN workspace_root TEXT` on an existing database that already has the column (if migration was manually applied or if the test suite creates a fresh DB multiple times).
**Why it happens:** SQLite `ALTER TABLE ADD COLUMN` fails with "duplicate column name" if run twice.
**How to avoid:** Use the `if current < 8` version gate. The existing `run_migrations` pattern does this correctly. Do NOT use the `let _ =` ignore-error pattern for ALTER TABLE (it hides bugs). Use version gating exclusively.

### Pitfall 6: Multi-Source Session ID Collision
**What goes wrong:** A `session_id` from the ratatui REPL and a `session_id` from the gateway happen to collide (UUID4 collision is astronomically unlikely but worth documenting). More realistically: trajectory file path uses session_id as directory name; if two surfaces somehow create sessions with the same ID, their trajectories.jsonl files would be at the same path.
**Why it happens:** UUID4 is used — collision probability is negligible but the concern is the path-based scoping.
**How to avoid:** The `uuid::Uuid::new_v4()` used by StateStore is cryptographically random. No action needed; document as a non-issue.

### Pitfall 7: Python Reference Curator Prompt Not Available
**What goes wrong:** The CONTEXT.md says the CURATOR_PROMPT must land with byte-equality lock test and implies it can be copied from the Python reference. The Python code does NOT contain an educational-value scoring prompt.
**Why it happens:** The CONTEXT.md "Educational Value" prompt description does not match the actual `agent/curator.py` content. The curator.py is a skill-consolidation tool, not a session-scoring tool.
**How to avoid:** The implementer must AUTHOR the `CURATOR_PROMPT` — a prompt that asks an LLM to rate a session's educational value on a 1-10 scale. It must:
  - Describe the heuristic criteria that already passed (tool_calls >= 3, impactful tool, duration >= 30s)
  - Provide the trajectory summary as input
  - Return a JSON score `{"score": N, "reason": "..."}` or structured output
  - Define "educational value" as: "would a developer benefit from having a SKILL.md documenting this workflow?"
  The lock test captures the final byte count AFTER authoring.

   - Recommendation: Planner should verify `event_loop.rs::build_app_deps` creates a session in StateStore and add `workspace_root` to that call.

3. **Does the Workspace newtype need to extend `CommandContext`?**
   - `CommandContext` currently has no workspace field. The Workspace is used for: (1) system prompt slot injection (PromptBuilder level, not CommandContext), (2) Curator skill output destination (Curator level), (3) `/sessions --workspace` filter (StateStoreHandle level), (4) trajectory directory scoping (TrajectoryWriter level).
   - Recommendation: Add `workspace: Option<Arc<Workspace>>` to `CommandContext` for the `/sessions --workspace` filter and for the `/curate` slash handler to know where to write skills. PromptBuilder gets the workspace path from the session_id lookup (StateStore), not from CommandContext.

4. **Can the SkillRegistry scan `<workspace>/skills/` at session start for workspace-local skills?**
   - CONTEXT.md says runtime is single-registry but READ merges both. The existing `SkillRegistry` scans a single directory. Does it support multiple scan roots?
   - Recommendation: Research `ironhermes-core/src/skills.rs` before planning the Curator skill registration step to determine whether `SkillRegistry` needs to be extended to scan two roots or if the Curator simply calls `SkillRegistry::load_from_path(&skill_path)` for targeted registration.

5. **What is the exact trajectory summary format passed to the Curator LLM?**
   - D-C-2 says "build trajectory summary + send through Phase 26 cascade." The trajectory summary format (how tool calls and results are condensed for the scoring prompt) is not specified.
   - Recommendation: Define in the curator implementation plan. A reasonable format: `"Session <id> | Duration: Xs | Tool calls: N | Tools: [tool1, tool2, ...] | Last tool result excerpt: ..."`.

6. **`hermes session export` vs existing `StateStore::export_session`?**
   - `ironhermes-state` already has `SessionExport` struct and likely `export_session` method (seen in `StateStoreHandle::export_session_text`). The D-F-1 export writes to 4 separate files in a directory.
   - Recommendation: Reuse `StateStore::export_session` as the data source; add a new `export_to_directory(session_id, output_dir)` function that writes the 4-file layout.
