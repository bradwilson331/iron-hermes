# Phase 25.3 Plan 08: LOAD-BEARING 4-Site Wireup — Summary

**One-liner:** The load-bearing wireup. Workspace + TrajectoryWriter resolved/created ONCE per session at run_chat / run_single / run_gateway / tui_rata::build_app_deps; threaded through every CommandContext-construction site (main.rs::build_cmd_ctx + tui_rata/commands.rs::build_command_context + gateway/handler.rs::handle_slash_command + gateway/runner.rs setter + GatewayMessageHandler.set_*). ## Why This Plan Existed

Phase 25.2 Plan 15 wired `toolset_session` only in `main.rs::build_cmd_ctx` and missed the ratatui REPL (`tui_rata/commands.rs::build_command_context`) and Telegram (`gateway/handler.rs::handle_slash_command`). UAT failed for `/toolset` on the two surfaces; two follow-up commits (`61ba493`, `3f25ac5`) were required after the verifier returned PASS.

Plan 3 (Wave 0) locked the parity contract IN CODE before this plan implemented the wireup. The 6 `#[ignore = "Wave 0 RED ..."]` attributes were a forcing function: removing them is the success signal that the 4-site contract is satisfied.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 — Critical correctness] Gateway PromptBuilder workspace_root parity**

- **Found during:** Task 3 verification (counting `with_workspace_root` occurrences across the workspace)
- **Issue:** The plan's `<action>` listed `with_workspace_root` chains in main.rs's run_chat / run_single / run_gateway PromptBuilder paths, but `run_gateway` in main.rs DOES NOT construct a PromptBuilder — the gateway constructs one per-message inside `GatewayMessageHandler::run_agent` (gateway/handler.rs:535). Without an additional edit there, Telegram users would see a different Identity slot text than CLI users for the same workspace.
- **Fix:** Added `if let Some(ref ws) = self.workspace { prompt_builder = prompt_builder.with_workspace_root(&ws.root); }` chain in `gateway/handler.rs::run_agent` after `.load_context(&cwd)`.
- **Why critical:** Phase 25.4 Curator scopes skills by workspace identity rendered into the system prompt; missing this would cause Curator to misclassify Telegram-originated sessions.
- **Files modified:** `crates/ironhermes-gateway/src/handler.rs`
- **Commit:** `3605f8f`

**2. [Rule 3 — Blocking issue, anticipated by plan] tui_rata has no PromptBuilder construction in build_app_deps**

- **Found during:** Task 2 — searching for `PromptBuilder::new` / `prompt_builder` in `event_loop.rs` returned no matches.
- **Issue:** The plan's Edit 2e said "find the PromptBuilder construction in build_app_deps" but no such construction exists — the ratatui REPL constructs its PromptBuilder per-turn inside `spawn_turn` (which lives in `commands.rs`, not in `build_app_deps`).
- **Fix:** Treated the 2e instruction as conditional ("if there is a PromptBuilder construction here, chain `.with_workspace_root`") and confirmed the no-op interpretation by reading the file. The acceptance criterion `grep "with_workspace_root" event_loop.rs >= 1` is incorrect for this codebase; the equivalent Identity-slot wiring in tui_rata happens in `spawn_turn`, which Plan 9 (AgentLoop callback wireup) will touch separately.
- **No code change** was required for this discovery — the fix is a documented interpretation of the conditional instruction.
