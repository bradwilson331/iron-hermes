### `strip_line_comments` helper hardening

Defined as a private fn at the top of the gap-closure section. Strips:
- **Leading line comments** (`// foo` at line start with optional whitespace) — same as the inline filter pattern used by the existing 7 tests.
- **Inline trailing line comments** (`code, // foo`) — required to make INV-25.3-11 actually RED today, because gateway/session.rs:113 reads `None, // workspace_root: Plan 0 placeholder ...` with the asserted-on token in an inline trailing comment. Without this hardening, the bare `workspace_root` token would survive the strip and self-invalidate the gate.

This extends the planner's documented antipattern guard (`// safe-grep-comment-stripped` idiom) one step further. The helper is used by all 5 new tests (`grep -c 'strip_line_comments'` returns 8: 1 helper definition + 5 callers + 2 mentions in surrounding doc comments).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] Hardened `strip_line_comments` to also strip inline trailing `//` comments**
- **Found during:** Initial RED-verification run after first commit attempt.
- **Issue:** The plan's `strip_line_comments` helper as specified strips only leading line comments. INV-25.3-11 then incorrectly **passed** (1 passed, 4 failed when it should have been 0 passed, 5 failed) because gateway/session.rs:113 reads `None, // workspace_root: Plan 0 placeholder — Plan 8 wires resolved workspace` — an inline trailing comment that the bare leading-only filter does not remove. The bare `workspace_root` token in that comment survived and satisfied INV-25.3-11's `>=1` count, self-invalidating the gate.
- **Fix:** Extended `strip_line_comments` to also strip everything after the first `//` on each kept line (after leading-comment filter). Added explicit doc comment on the helper documenting the two strip layers, the specific Plan 0 placeholder this protects against, and the safety reasoning (no `//` appears inside any string literal we depend on in the asserted-on files).
- **Files modified:** `crates/ironhermes-cli/tests/invariants_25_3.rs` (helper body).
- **Commit:** `cf90262` (folded into the same commit as the test additions, since the helper and the tests were authored together).
- **Why this is Rule 1, not Rule 4:** This is the exact "self-invalidating gate" antipattern the plan body warns about. Failing to fix it would have left INV-25.3-11 silently GREEN when it should be RED — the plan's own success criteria (`grep -E "FAILED|invariant_25_3_(07|08|09|10|11)"` matches FAILED for all 5) would have been violated. Hardening the helper is the structural fix.

**2. [Rule 1 — Spec interpretation] Used `#[ignore]` gating instead of plain RED-on-default**
- **Found during:** Reading the spawn prompt's `<parallel_execution>` directive: "Gate them with `#[ignore]` so the wave doesn't break."
- **Issue:** The plan's `<verify>` block specifies running the tests directly and grepping for `FAILED`, suggesting they should fail on default. But the spawn prompt is authoritative and explicitly requires `#[ignore]` so the Wave 7 build/test run is not blocked.
- **Fix:** Each new test carries `#[ignore = "Phase 25.3 Plan NN (CR-XX) un-ignores after wiring ..."]` with a descriptive reason. The default `cargo test` run shows 7 passed + 5 ignored; `cargo test -- --ignored` shows all 5 RED.
- **Result:** Honors both directives — the plan's RED contract is enforceable via `--ignored` AND Wave 7 `cargo test --workspace` stays green so the wave can ship.
- **Why this is correct, not a deviation:** This is the same Wave 0 -> Plan 8 RED-then-GREEN pattern that the existing 6 parity-guards use (see the file's module docstring lines 19-27). Plans 25.3-13..16 are expected to remove `#[ignore]` for their respective test as the fix lands — exactly mirroring the existing pattern.
