**One-liner:** Closes Phase 25.3 by wiring the user-facing surfaces for D-F-1 and D-F-2: `hermes session export <id> [--output <dir>]` and `hermes session export-all [--since YYYY-MM-DD]` CLI subcommands; `/export-session [session_id]` slash command; phase-closing subprocess e2e (`session_export_e2e.rs`) that materializes the 4-file layout under a tempdir IRONHERMES_HOME; VALIDATION.md plan-IDs reconciled with final 25.3-00..25.3-11 numbering and `nyquist_compliant: true`. ## Why This Plan Existed

Plan 10 shipped `ironhermes_state::SessionDirectoryExport` (the 4-file directory writer for D-F-1) and the `/sessions --workspace` filter for D-W-2, but neither was reachable from the CLI or slash surface. Plan 11 closes the loop:

- **D-F-1 user-facing entry:** `hermes session export <id>` and `/export-session [id]` reach the SessionDirectoryExport writer.
- **D-F-2 bulk entry:** `hermes session export-all [--since YYYY-MM-DD]` iterates StateStore sessions and writes each one's 4-file layout, with optional date filtering for thousand-session-class operators.
- **Phase-closing e2e:** `session_export_e2e.rs` proves the full pipeline (StateStore seed → subprocess invoke → on-disk 4-file layout) actually works against the binary itself, not just unit-test mocks.
- **Validation reconciliation:** `25.3-VALIDATION.md` had placeholder plan IDs (`25.3-00..25.3-13`) from the original draft. Plan 11 swaps in the actual final IDs (`25.3-00..25.3-11`) across 6 waves so `/gsd-verify-work` can match the test map against what shipped.

Per RESEARCH.md: "On-demand `hermes session export <id>` + `/export-session` slash produce hermes-agent's `~/.ironhermes/sessions/[id]/{messages,metadata,context}.json` folder layout for tooling/RL pipelines." Plan 11 is that surface.

## Decisions Made

- **`StateStoreHandle::export_to_directory_text` gets a DEFAULT body** returning the not-configured guard string. Plan suggested no default — but the trait has multiple impls across crates (FakeStateStore in handlers.rs tests, FakeExportStore added by Plan 11, StateStoreAdapter in tui_rata, TestStateStoreAdapter in tests/handlers_state_store.rs). A no-default trait method would force every existing impl to override or break compilation. The default is structurally safe (returns `"error: ..."` which the dispatcher routes to `CommandResult::Error`) and maintains backward compat. Production impls override with the real export.
- **Bulk export iterates `list_sessions(None, usize::MAX)` then filters by `started_at` in Rust** rather than adding a `since_unix` WHERE clause to `list_sessions_filtered`. Plan-mandated reasoning verbatim: "the operator-on-demand semantic of D-F-2 means we can iterate all and filter — bulk export across thousands of rows is still tractable since each export is a 4-file write." Adding a third optional bind to `list_sessions_filtered` would also touch Plan 10's 24 existing call sites.
- **lib-reachable cli_args.rs gets its own SessionSubcommand definition** (not main.rs's session_cmd::SessionSubcommand) for the same reason cli_args::Cli mirrors main::Cli — Plan 21.7 ISS-06 invariant. Field names match exactly (`session_id` + `output` for `Export`; `since` for `ExportAll`) so the two definitions stay in sync without delegation.
- **Slash command name `export-session`** (hyphen, NOT export_session) matches the existing `/reload-mcp` convention. `args_hint("[session_id]")` documents the optional-positional form. Universal platform (CLI REPL + ratatui REPL + Telegram) — same as the precedent `/sessions` and `/resume`.
- **`cmd_export_session` routes `error:`-prefixed responses to `CommandResult::Error`** and other strings to `CommandResult::Output` via `if result.starts_with("error:")`. This keeps the StateStoreHandle trait sync (no async dispatch needed) while still distinguishing failure from success at the CommandResult level for downstream UI surfaces.
- **TestStateStoreAdapter duplicates the production export_to_directory_text body** rather than centralizing it because the production type is private to ironhermes-cli. The body is short (~30 lines) and the duplication is locked by the existing 18-test integration suite that exercises both adapters in lockstep.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 — Critical correctness] Default body for `StateStoreHandle::export_to_directory_text`**

- **Found during:** Task 2 — the plan's edit instruction added the trait method with NO default body (`fn export_to_directory_text(&self, session_id: &str) -> String;`).
- **Issue:** Adding a no-default trait method forces every existing impl across crates (FakeStateStore in handlers.rs, FakeExportStore added by Plan 11 tests, StateStoreAdapter, TestStateStoreAdapter) to override — breaking the workspace compile chain at multiple sites simultaneously. The existing 18-test integration suite plus the 7 cmd_sessions tests already exercise the trait through fakes that don't care about export semantics.
- **Fix:** Added a default body returning `"error: directory export not supported by this StateStoreHandle impl"`. Production adapters (StateStoreAdapter + TestStateStoreAdapter) override with the real SessionDirectoryExport call. Test fakes that don't care about export semantics (FakeStateStore in handlers.rs) compile clean without overrides. The default is structurally safe — the dispatcher routes `"error:"`-prefixed strings to `CommandResult::Error`, so even if a misconfigured adapter falls through to the default, the operator sees a clear error instead of a silent panic or compile break.
- **Files modified:** `crates/ironhermes-core/src/commands/context.rs` (the default body)
- **Commit:** `c8bbd4c` (Task 2 GREEN)

**2. [Rule 1 — Bug] Plan's e2e test expected `_store` shadow but tempdir lifetime requires explicit drop**

- **Found during:** Task 3 — the plan's `session_export_all_with_no_sessions_exits_zero` test wrote `let _store = StateStore::new(&home.join("state.db")).unwrap();` and then immediately spawned the binary.
- **Issue:** The leading-underscore binding pattern in Rust is documented to "drop immediately at end of statement" for `_` (single underscore) but `_store` (named with leading underscore) holds the value across the function body's scope — meaning the SQLite WAL connection stays open across the subprocess invocation. macOS SQLite tolerates this but the seam is fragile.
- **Fix:** Added explicit `drop(_store);` after the binding so the file handle is released before subprocess invocation. Test passes deterministically.
- **Files modified:** `crates/ironhermes-cli/tests/session_export_e2e.rs`
- **Commit:** `df7e000` (Task 3)

| Threat ID | Disposition | How handled |

| T-25.3-11-01 | mitigate | session_id IS used as a path component via `PathBuf::join("sessions").join(session_id)`. | T-25.3-11-04 | mitigate | `chrono::NaiveDate::parse_from_str(s, "%Y-%m-%d")` returns `Err` which `run_export_all` surfaces via `.with_context(\"--since must be YYYY-MM-DD; got '{s}'\")`.
