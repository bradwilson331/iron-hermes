## Why This Plan Existed

Phase 25.2 Plan 15's parity-guard test only grep'd `main.rs`. It missed:
- `crates/ironhermes-cli/src/tui_rata/commands.rs::build_command_context` — the DEFAULT `hermes chat` REPL since Phase 22.4.
- `crates/ironhermes-gateway/src/handler.rs::handle_slash_command` — Telegram per-message slash dispatch.

Result: Phase 25.2 UAT FAILED for `/toolset` in REPL + Telegram. Two follow-up commits (`61ba493`, `3f25ac5`) were needed after the verifier had returned PASS. See `.planning/phases/25.2-web-extract-tools/25.2-VERIFICATION.md` Addendum (2026-05-03).

This plan locks the parity contract IN CODE before Plan 8 implements the wireup, so the same scope-blindness regression cannot recur for Phase 25.3's new `Workspace` and `TrajectoryWriter` CommandContext fields.

### Comment-Stripping Defence (T-25.3-03-01 mitigation)

Each test pre-processes the source text with `.lines().filter(|l| !l.trim_start().starts_with("//"))` before the substring grep. This prevents documentation prose (like the file's own coverage matrix) from satisfying the gate via doc-comment text — the "self-invalidating grep" planner antipattern. The GREEN sanity test (`existing_toolset_session_parity_still_holds`) doubles as proof-of-method: it asserts on a fact already true in Phase 25.2-wired source, so a regression in the comment-stripping pass would surface as a sanity-test failure rather than silently passing the contract.
