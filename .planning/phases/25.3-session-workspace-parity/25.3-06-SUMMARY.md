## Decisions Made

- **Cycle-break direction is `trajectory -> core`, NOT the reverse.** Verified by `grep "path = " crates/ironhermes-core/Cargo.toml` returning empty (zero internal path deps — strict leaf-crate). The trait-object form is the documented convention across MemoryManagerHandle / ToolsetSessionHandle / SummarizationClientHandle. Inverting would force every CommandContext consumer (CLI, gateway, tools) to compile ironhermes-trajectory transitively, even those that never log trajectories.
- **`TrajectoryWriterHandle` trait def landed with Task 1 (not deferred to Task 2 Edit 0).** Reason: `handle.rs` imports the trait via `use ironhermes_core::commands::context::TrajectoryWriterHandle;`. Without the trait in core, Task 1's `cargo build -p ironhermes-trajectory` verify step fails with E0432 (unresolved import). Lifting the Edit 0 trait def into Task 1's commit unblocks Task 1's verify; Task 2 RED then exercises the missing fields + builders (E0609 + E0599) which is the canonical TDD-via-missing-symbol RED pattern.
- **`append_raw_line` conditional add fired** (Plan 04's `writer.rs` only shipped `append(&TrajectoryEntry)`). The trait impl writes a pre-serialized JSON line through the trait boundary — round-tripping through `append` would require parsing the line back into a `TrajectoryEntry` first, which is wasteful and error-prone. `append_raw_line` is a 17-line addition that mirrors `append`'s fsync-per-line + `.with_context()` telemetry exactly, with the only differences being (a) takes `&str` instead of `&TrajectoryEntry` and (b) skips the `serde_json::to_string` step.
- **In-test `NoopHandle` (4 lines) instead of dev-dep on ironhermes-trajectory.** The trait-object form makes inline test stubs trivial: any local impl works. Preserves leaf-crate purity even in test scope (no dev-dep arrow back to ironhermes-trajectory). Same pattern as plan_21_7_07_tests's `FakeSubagentList`, `FakeBudget`, and `FakeProc`.
- **Field placement order: workspace BEFORE trajectory_writer.** Both decisions in the plan's `must_haves.truths` list `workspace` first, and the docs reference each other in the `workspace -> Plan 7` and `trajectory_writer -> Plan 9` directions. Placing `workspace` first lets the trajectory_writer doc-comment reference workspace's role ("trajectory directory scoping in CommandContext.trajectory_writer") without forward-references.
- **Two new fields placed AFTER `toolset_session` (the LAST current field) on `CommandContext`.** Two new None initializers placed AFTER `toolset_session: None,`. Two new builder methods placed AFTER `with_toolset_session`. This co-locates Plan 25.3-06 additions at the end of struct/impl, making them easy to grep + visually identify; mirrors Plan 21.7-07 + Phase 22.4.2 Plan 00 + Phase 22.4.2.1 + Phase 25 Plan 04 conventions.
