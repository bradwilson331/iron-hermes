# Phase 25.3 Plan 15: Gateway Trajectory Canonical Session UUID (CR-02 Close-Out) Summary

**One-liner:** Replaces the gateway's process-wide `gateway-<random-uuid>` TrajectoryWriter with per-session writers cached in SessionStore and keyed by the canonical SQLite session UUID, so `hermes session export <id>` finds the trajectory file and per-Telegram-user tool calls land in per-session files instead of a single shared cross-user log.

## What was built

A single-task atomic close-out for 25.3-REVIEW.md CR-02. The previous implementation opened ONE TrajectoryWriter per gateway-process lifetime at `<workspace>/.ironhermes/sessions/gateway-<random-uuid>/trajectories.jsonl` and shared it across every Telegram message. CR-02 flagged three downstream consequences:

1. All Telegram users' tool calls land in a single shared file (chat_id boundaries erased in the trajectory ledger).
2. `session_cmd::resolve_trajectory_source` and `SessionDirectoryExport::write` look up trajectories at `<root>/sessions/<session_id>/trajectories.jsonl` keyed by the canonical SQLite session UUID — the gateway file at `gateway-<uuid>` is unreachable; `hermes session export <gw-session-uuid>` produces a 4-file directory with NO `trajectories.jsonl`.
3. Phase 25.4 Curator's per-session weighting is starved — gateway sessions appear empty at the canonical path.

The fix follows the CR-02 Option A recommendation:

### main.rs `run_gateway` (5 lines changed at the trajectory-open block, plus the runner setter call)

- Removed the `gateway_trajectory_id = format!("gateway-{}", uuid::Uuid::new_v4())` literal.
- Removed the process-wide `TrajectoryWriter::open(...)` block that produced the `Option<Arc<dyn TrajectoryWriterHandle>>` named `trajectory_writer`.
- Computes a `trajectory_root: PathBuf` instead — workspace-aware (`<workspace>/.ironhermes/sessions`) or hermes-home fallback (`~/.ironhermes/sessions`).
- Replaced `runner.set_trajectory_writer(handle)` with `runner.set_trajectory_root(trajectory_root)`.

### runner.rs `GatewayRunner`

- Replaced field `trajectory_writer: Option<Arc<dyn TrajectoryWriterHandle>>` with `trajectory_root: Option<PathBuf>`.
- Replaced setter `set_trajectory_writer` with `set_trajectory_root`. The new setter propagates the path into the inner `SessionStore` via `try_write` — mirrors the Plan 25.3-14 `set_workspace` propagation pattern. (`try_write` cannot legitimately fail during the gateway setup phase; on impossible-failure we `tracing::warn!` rather than panic so a future refactor that moves the call onto a contended path surfaces the misuse loudly.)
- Removed the `if let Some(ref tw) = self.trajectory_writer { handler.set_trajectory_writer(tw.clone()) }` clone in `build_gateway_handler` — per-session writers are now owned by SessionStore.

### handler.rs `GatewayMessageHandler`

- Removed the `trajectory_writer: Option<Arc<dyn TrajectoryWriterHandle>>` field.
- Removed the `set_trajectory_writer` setter.
- Removed the slash-dispatch `ctx.with_trajectory_writer(tw.clone())` attach (slash commands don't currently record per-tool trajectory; if Phase 25.4 needs that, the per-session handle is available via SessionStore).
- `run_agent` now:
  1. Reads the canonical SQLite session UUID off the `GatewaySession` just inserted/fetched by the prior `store.get_or_create(...)` call (same `&key` used to clone `session_messages`).
  2. Calls `store.get_or_create_trajectory_writer(&canonical_session_id)`.
  3. Passes the resulting handle to `agent.with_trajectory_writer(tw)` (the per-tool AgentLoop wireup path that INV-25.3-05 grep-locks — preserved).
- Per-message `session_id_str = format!("gw:{}:{}", event.chat_id, event.sender_id)` retained for AgentLoop telemetry / `on_session_end` hooks. It is now intentionally distinct from the canonical UUID used for trajectory file paths; comments document this distinction to prevent re-merging the two identifiers.

### session.rs `SessionStore`

- Added two new fields:
  - `trajectory_root: Option<PathBuf>` — gateway-wide root path under which per-session subdirs are nested.
  - `trajectory_writers: HashMap<String, Arc<dyn TrajectoryWriterHandle>>` — cache so the same chat reuses ONE writer across messages (no file-handle leak across long-running gateway sessions).
- Added two new methods:
  - `pub fn set_trajectory_root(&mut self, root: PathBuf)` — installer (called from `GatewayRunner::set_trajectory_root` via try_write).
  - `pub fn get_or_create_trajectory_writer(&mut self, session_id: &str) -> Option<Arc<dyn TrajectoryWriterHandle>>` — lazy-open on first call; cached on subsequent calls. Returns `None` if no `trajectory_root` was installed (gateway launched without trajectory dir configured) OR if `TrajectoryWriter::open` failed (logged + `None` per the existing best-effort pattern in main.rs:642 and run_single).

### tests/invariants_25_3.rs

- Un-ignored `invariant_25_3_10_gateway_trajectory_uses_canonical_session_uuid` and replaced the `#[ignore]` attribute with an explanatory comment documenting why it is now GREEN.
- Widened `trajectory_writer_wired_in_gateway_runner_setter` (INV-25.3-06) to accept EITHER `set_trajectory_writer(` OR `set_trajectory_root(` — the canonical form post-CR-02 close-out is `set_trajectory_root` (per-session lazy-open via SessionStore); the original setter shape is grandfathered for compatibility.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] `canonical_session_id` derivation must read from already-inserted session, not call `get_or_create` again**

- **Found during:** Implementing Edit 3 in handler.rs.
- **Issue:** The plan body proposes:
  ```rust
  let canonical_session_id = {
      let sess = self
          .session_store
          .write()
          .await
          .get_or_create(session_key.clone(), &model, "telegram")
          .session_id
          .clone();
      sess
  };
  ```
  But `get_or_create` requires a `&model` argument and the `source` arg in handler.rs is `&source` (built earlier in the function as `key.platform.to_string()`, equal to `"telegram"`). More importantly, `run_agent` ALREADY calls `store.get_or_create(key.clone(), &model, &source)` further up (line 527), and the GatewaySession instance just inserted carries the session_id we want. Calling `get_or_create` a second time is structurally redundant and would also re-acquire the write lock for no reason.
- **Fix:** Read `canonical_session_id` via `store.get(&key).map(|s| s.session_id.clone()).unwrap_or_default()` against the already-inserted session. Wraps in a `read()` lock since we don't mutate. The follow-on `get_or_create_trajectory_writer` requires `write()`; that lock is acquired separately to keep the critical section minimal.
- **Files modified:** `crates/ironhermes-gateway/src/handler.rs` (run_agent body).
- **Commit:** Folded into commit `c29377e`.
- **Why this is Rule 1, not Rule 4:** The plan's behavior contract ("opens or reuses a per-session trajectory writer keyed by the canonical SQLite session UUID") is preserved exactly. The structural change is from "redundant get_or_create call" to "read existing session" — the code is shorter, locks are tighter, and behavior is identical. No architectural decision is required.

**2. [Rule 1 — Bug] Defensive `is_empty()` guard on canonical_session_id**

- **Found during:** Same edit.
- **Issue:** If for some reason `store.get(&key)` returns `None` (e.g., the session row creation failed silently — which is best-effort per the existing `warn!("Failed to persist session to SQLite")` pattern in session.rs:134), `canonical_session_id` would be empty string. Calling `get_or_create_trajectory_writer("")` would attempt to open `<root>//trajectories.jsonl` which is a malformed path.
- **Fix:** Wrapped the writer lookup in `if !canonical_session_id.is_empty()`. On the empty-string fallback, the trajectory writer is simply not attached — best-effort behavior matching the existing `if let Some(ref tw) = self.trajectory_writer` pattern that this fix replaces.
- **Files modified:** `crates/ironhermes-gateway/src/handler.rs` (run_agent body).
- **Commit:** Folded into commit `c29377e`.
- **Why this is Rule 1, not Rule 4:** Defensive null-handling is correctness, not architecture. The plan body's phrasing ("get_or_create_trajectory_writer ... lazily opens (or returns cached) ... `<trajectory_root>/<session_id>/trajectories.jsonl`") implicitly assumes a non-empty session_id; the empty-string guard makes the implicit contract explicit.
