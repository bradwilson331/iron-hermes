## Phase Boundary

Extend the existing `hexapod_tcp` tool (built in Phase 27.1.1) with three new hardware capabilities: rotate in place by degrees (HXP-NAV-01), head pan and tilt control (HXP-NAV-03), and buzzer on/off (HXP-NAV-04).

All new actions follow the Phase 27.1.1 patterns: single `hexapod_tcp` tool with action enum, one fresh TCP connection per call, fire-and-forget returns `"OK"`, errors return `Ok("Error: ...")` strings. No new tools, no new crates.

The Freenove repo at `~/code/Freenove_Big_Hexapod_Robot_Kit_for_Raspberry_Pi` is read-only protocol reference. No Python server code is modified.

</domain>

<decisions>
## Implementation Decisions

### Rotate In Place (HXP-NAV-01)

- **D-01:** `rotate` action takes a single `degrees: integer` parameter. Positive = clockwise (right), negative = counterclockwise (left). Matches HXP-NAV-01 spec sign convention exactly.
- **D-02:** Speed is fixed internally at `speed=5` — not caller-configurable. Keeps the LLM interface minimal; speed tuning happens in code.
- **D-03:** Rotation uses `CMD_MOVE#1#0#0#5#±10\n` (angle field = ±10, max turn rate). `x=0, y=0` gives in-place rotation. Positive degrees → angle=+10, negative → angle=-10.
- **D-04:** Duration is computed as `|degrees| * ROTATE_MS_PER_DEGREE` milliseconds. Named constant `ROTATE_MS_PER_DEGREE = 20` (placeholder — tune on real robot). Same calibration-constant pattern as was used for BATTERY_LOW_V1/V2 in Phase 27.1.1.
- **D-05:** Sequence: send `CMD_MOVE#1#0#0#5#<±10>\n`, async-sleep for computed duration, then send `CMD_MOVE#0#0#0#0#0\n` (stop). All fire-and-forget; returns `"OK"` on success.

### Head Pan and Tilt (HXP-NAV-03)

- **D-06:** `head_pan` action: sends `CMD_HEAD#0#<angle>\n`. Clamped to `[-HEAD_PAN_MAX, HEAD_PAN_MAX]`.
- **D-07:** `head_tilt` action: sends `CMD_HEAD#1#<angle>\n`. Clamped to `[-HEAD_TILT_MAX, HEAD_TILT_MAX]`.
- **D-08:** Both axes use separate named constants: `HEAD_PAN_MAX = 90` and `HEAD_TILT_MAX = 90` (protocol maximum; tunable once live-tested). Separate constants because pan and tilt are mechanically different and may need independent calibration.
- **D-09:** Both are fire-and-forget (no response from robot). Return `"OK"` on successful send, D-17 error string on connection failure.

### Buzzer (HXP-NAV-04)

- **D-10:** `buzzer_on` action: sends `CMD_BUZZER#1\n`. Fire-and-forget, returns `"OK"`.
- **D-11:** `buzzer_off` action: sends `CMD_BUZZER#0\n`. Fire-and-forget, returns `"OK"`.
- **D-12:** Two discrete action names (not `buzzer` + state param) — matches the `walk`/`stop` precedent from Phase 27.1.1. Explicit intent in the action name; no secondary param needed.

### Tool Schema Extension

- **D-13:** The existing `hexapod_tcp` action enum grows from 5 to 10 values:
  `walk | stop | read_battery | read_distance | relax_servos | rotate | head_pan | head_tilt | buzzer_on | buzzer_off`
- **D-14:** New schema properties required by the new actions:
  - `degrees`: integer (used by `rotate`; positive=right, negative=left)
  - `angle`: integer (used by `head_pan` and `head_tilt`)
- **D-15:** The tool `DESCRIPTION` string must be updated to include the 5 new Phase 2 actions and clarify that `degrees` and `angle` are action-specific params.

### Inherited (carry-forward from Phase 27.1.1 — do NOT re-litigate)

- **D-11 (27.1.1):** One fresh TCP connection per tool call — stateless, no persistent socket.
- **D-12 (27.1.1):** Robot IP from `HEXAPOD_IP` env var, port 5002.
- **D-16 (27.1.1):** Blocked commands return `Ok("Action '<name>' is blocked — not permitted via hexapod_tcp. Never send this command.")`.
- **D-17 (27.1.1):** Robot unreachable → `Ok("Error: cannot connect to robot at <ip>:5002 — is HEXAPOD_IP set and the robot powered on?")`.
- **D-20 (27.1.1):** Allowlist enforced by exhaustive compile-time `match` on action — unknown actions hit catch-all and return blocked string without TCP.

### Claude's Discretion

- The `rotate` implementation sleeps the tool execution thread for `|degrees| * ROTATE_MS_PER_DEGREE` ms — the planner should verify this is acceptable in tokio's async context (use `tokio::time::sleep`, not `std::thread::sleep`).
- Whether to extract a shared `send_fire_and_forget_nav` helper for the new fire-and-forget actions or reuse the existing `send_fire_and_forget` from Phase 27.1.1 directly.
- Exact angle clamping behavior when caller passes out-of-range values (clamp silently vs. return an error string). Recommend: clamp silently and return `"OK"` — consistent with how `walk` clamps speed.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `send_fire_and_forget(addr, cmd)` in `hexapod_tcp.rs` — already handles fire-and-forget over a fresh TCP connection; reuse directly for buzzer_on, buzzer_off, head_pan, head_tilt.
- `build_walk_wire()` pattern — model the new `build_rotate_wire(degrees: i64) -> String` helper on this: pure function, no I/O, unit-testable without a robot.
- `ENV_LOCK` mutex + test pattern in the existing test suite — new tests for Phase 2 actions follow the same serialized env-var pattern (tests 14–20 or similar).

### Established Patterns
- **Compile-time exhaustive match** (D-20): add the 5 new action strings to the inner `match action { "walk" | "stop" | ... | "rotate" | "head_pan" | ... => { ... }` arm and the outer allowlist arm. The compiler catches any mismatch.
- **Named calibration constants** at module top: `ROTATE_MS_PER_DEGREE`, `HEAD_PAN_MAX`, `HEAD_TILT_MAX` follow `BATTERY_LOW_V1`/`BATTERY_LOW_V2` placement and naming style.
- **`tokio::time::sleep`** for the rotate duration — same crate already imported for `timeout`. Do NOT use `std::thread::sleep` (blocks the async executor).
- **`anyhow::Result<String>` return** with `Ok("OK")` for fire-and-forget success — existing walk/stop arms are the exact model.

### Integration Points
- `hexapod_tcp.rs` is the only file that changes for Phase 2. No new files, no new crate deps, no changes to `lib.rs` or `registry.rs`.
- The `DESCRIPTION` const string in `hexapod_tcp.rs` must be updated to list the 10 actions (was "Phase 1 actions: walk...").

</code_context>

<specifics>
## Specific Ideas

- Keep `ROTATE_MS_PER_DEGREE`, `HEAD_PAN_MAX`, and `HEAD_TILT_MAX` as `pub(crate)` constants at the top of the file alongside the existing constants — makes them easy to find and tune after live robot testing.
- The `rotate` action should be the only Phase 2 action that sleeps. All others (head_pan, head_tilt, buzzer_on, buzzer_off) are instantaneous sends.
- Test coverage should include: rotate wire builder (positive/negative degrees), head_pan/tilt clamping (out-of-range angle gets clamped), buzzer wire strings, and the new blocked-action catch-all (ensure the 5 new actions aren't accidentally caught by the blocked arm).
