## Summary

Phase 27.1.2 extends the existing `hexapod_tcp.rs` tool (built in Phase 27.1.1) with five new action arms: `rotate`, `head_pan`, `head_tilt`, `buzzer_on`, and `buzzer_off`. The Freenove wire protocol has been read directly from the canonical reference and all three command formats are confirmed. No new crates, no new files, no changes to `lib.rs` or `registry.rs` are needed.

The only substantive complexity in this phase is the `rotate` action, which requires a timed async sleep between the motion command and the stop command. The existing `tokio` dependency already has `features = ["full"]`, so `tokio::time::sleep` is available without any dependency change. The four remaining actions are straightforward fire-and-forget sends that reuse the existing `send_fire_and_forget` helper directly.

All protocol details (command strings, field layout, angle ranges, response behavior) are verified against the canonical Freenove protocol document. No assumed claims about wire format remain in this research.

**Primary recommendation:** Extend `hexapod_tcp.rs` with the five new match arms, three new pure builder/wire-string helpers, and three new named constants. Reuse `send_fire_and_forget` as-is. Add 7 new unit tests following the existing ENV_LOCK pattern.

### Pattern 1: Rotate — timed motion with stop

The rotate action is the only Phase 2 action that sleeps. The sleep must use `tokio::time::sleep` because `std::thread::sleep` blocks the tokio executor thread and would stall all other async tasks.

```rust
// Source: [VERIFIED: tokio::time::sleep — tokio is workspace dep with features=full]
// Source: Phase 27.1.1 hexapod_tcp.rs — tokio::time::Duration already imported

"rotate" => {
    let degrees = args["degrees"].as_i64().unwrap_or(0);
    let wire = build_rotate_wire(degrees);
    let duration = Duration::from_millis(
        (degrees.unsigned_abs()) * ROTATE_MS_PER_DEGREE
    );
    match send_fire_and_forget(&addr, &wire).await {
        Err(_) => return Ok(format!(
            "Error: cannot connect to robot at {addr} — is HEXAPOD_IP set and the robot powered on?"
        )),
        Ok(_) => {}
    }
    tokio::time::sleep(duration).await;
    let _ = send_fire_and_forget(&addr, STOP_CMD).await; // best-effort stop
    Ok("OK".to_string())
}
```

[VERIFIED: `Duration` already imported in hexapod_tcp.rs via `tokio::time::{Duration, timeout}`]

### Anti-Patterns to Avoid

- **`std::thread::sleep` in async context:** Blocks the tokio executor. Use `tokio::time::sleep(duration).await` exclusively. [VERIFIED: CONTEXT.md Claude's Discretion note]
- **Reading `degrees` as `u64` directly:** `degrees` is signed; use `as_i64()` then `unsigned_abs()` to get duration millis.
- **Omitting stop after rotate:** The robot will continue rotating at max angle rate until stopped. The stop send after sleep is mandatory.
- **Adding head_pan/head_tilt to the `send_and_read_line` path:** These are fire-and-forget, not sensor queries. Do not wrap in a timeout read loop.
- **Using a single `buzzer` action with a state param:** D-12 locks two discrete action names. The schema enum must list both `buzzer_on` and `buzzer_off` as separate values.

## Common Pitfalls

### Pitfall 1: `std::thread::sleep` in async code
**What goes wrong:** Blocks the tokio worker thread. All other tasks waiting on that thread stall for the full rotation duration.
**Why it happens:** Cargo doesn't warn; the code compiles and runs — it just silently degrades all async concurrency.
**How to avoid:** Always use `tokio::time::sleep(duration).await`.
**Warning signs:** CI passes but latency spikes when rotate is called concurrently with other tool calls.

### Pitfall 2: Forgetting the stop after rotate
**What goes wrong:** Robot continues rotating at max angular rate indefinitely after the timed sleep ends.
**Why it happens:** `CMD_MOVE#1#0#0#5#10` is a continuous motion command — the robot stays in that mode until told otherwise.
**How to avoid:** The stop send (`send_fire_and_forget(addr, STOP_CMD)`) must be unconditional after the sleep, even if it returns an error (best-effort).
**Warning signs:** Robot keeps spinning after the tool call returns.

### Pitfall 3: Allowlist omission — new actions caught by catch-all
**What goes wrong:** A new action string like `"buzzer_on"` is not added to the outer match arm (the allowlist). It falls into the `other =>` catch-all and returns a blocked-command error instead of executing.
**Why it happens:** The outer match arm and the inner match arm are separate; adding to the inner without the outer compiles fine but silently blocks.
**How to avoid:** The outer allowlist arm must list all 10 action strings. The compiler will warn if the inner arm has unreachable patterns, but NOT if the outer arm is missing a string.
**Warning signs:** `buzzer_on` returns `"Action 'buzzer_on' is blocked..."` — check the outer match arm first.

### Pitfall 4: Head angle range confusion — degrees vs protocol units
**What goes wrong:** The protocol says angle range is `-90` to `90` degrees. This is the actual servo degree value, not a scaled unit. Passing values outside this range likely causes the servo controller to clamp or error internally.
**Why it happens:** Some robot protocols use scaled or unsigned units; this one uses signed degrees directly.
**How to avoid:** Clamp with `HEAD_PAN_MAX = 90` and `HEAD_TILT_MAX = 90` before formatting the wire string. [VERIFIED: protocol doc §6]

### Pitfall 5: `degrees` overflow in duration calculation
**What goes wrong:** `degrees.unsigned_abs() * ROTATE_MS_PER_DEGREE` could overflow if `degrees` is very large (though `u64` overflow requires 2^64/20 degrees, which is not a realistic input).
**Why it happens:** `i64::unsigned_abs()` returns `u64`; multiplication with `ROTATE_MS_PER_DEGREE: u64` is safe for any realistic input.
**How to avoid:** No special handling needed for realistic inputs. Documented for awareness.

## Wire Protocol Verification

All wire formats verified against the canonical Freenove protocol reference. [VERIFIED: `/Users/twilson/code/Freenove_Big_Hexapod_Robot_Kit_for_Raspberry_Pi/Code/robot_control_communication_protocol.md`]

### CMD_MOVE (rotate in place) — §1

```
CMD_MOVE#<mode>#<x>#<y>#<speed>#<angle>\n
```

| Field | Rotate value | Notes |
|-------|-------------|-------|
| mode | 1 | Motion mode (not gait mode 2) |
| x | 0 | Zero = no lateral displacement → in-place |
| y | 0 | Zero = no forward displacement → in-place |
| speed | 5 | Fixed per D-02 |
| angle | +10 or -10 | Max turn rate; +10 = clockwise, -10 = counterclockwise |

[VERIFIED: protocol doc §1 — angle range `-10` to `10`, default `0` straight walking]

Stop command (already in Phase 27.1.1): `CMD_MOVE#0#0#0#0#0\n`

### CMD_HEAD — §6

```
CMD_HEAD#<axis>#<angle>\n
```

| Field | pan value | tilt value | Notes |
|-------|-----------|------------|-------|
| axis | 0 | 1 | 0=horizontal/pan, 1=vertical/tilt |
| angle | -90 to +90 | -90 to +90 | signed degrees |

[VERIFIED: protocol doc §6 — "Horizontal Rotation: CMD_HEAD#0#<angle>", "Vertical Rotation: CMD_HEAD#1#<angle>", "angle range is -90 to 90 degrees"]

**Response behavior:** No response from robot. Fire-and-forget. [ASSUMED — protocol doc does not explicitly state "no response"; consistent with movement commands which are also fire-and-forget]

### CMD_BUZZER — §5

```
CMD_BUZZER#1\n   ← on
CMD_BUZZER#0\n   ← off
```

[VERIFIED: protocol doc §5 — "Turn On Buzzer: CMD_BUZZER#1\n", "Turn Off Buzzer: CMD_BUZZER#0\n"]
