## Rotate Implementation

| Option | Description | Selected |
|--------|-------------|----------|
| Degrees only — fixed speed | Tool always rotates at speed=5, scales duration by degrees. One-param API. | ✓ |
| Degrees + speed | Two params like walk (direction + speed). More flexible but adds LLM burden. | |

**Follow-up — angle field value:**

| Option | Description | Selected |
|--------|-------------|----------|
| Max turn rate (±10) | CMD_MOVE angle=±10 always. Fastest, most predictable timing. | ✓ |
| Fixed moderate angle (±5) | Slower rotation. Potentially more precise but still requires calibration. | |

**Follow-up — calibration constant:**

| Option | Description | Selected |
|--------|-------------|----------|
| Placeholder (20 ms/degree), tune later | Ship ROTATE_MS_PER_DEGREE = 20; tune on real robot. | ✓ |
| Known value now | User has a real calibration number from testing. | |

**User's choice:** Degrees-based API, fixed speed=5 internally, angle=±10, ROTATE_MS_PER_DEGREE = 20 placeholder.
**Notes:** User initially redirected away from the philosophical question ("go back to rotate by degrees") — confirming degrees-based is the desired API regardless of protocol limitations. Time-based implementation is acceptable.

## Head Servo Safe Limits

| Option | Description | Selected |
|--------|-------------|----------|
| Trust full ±90° — protocol limits | Firmware enforces hardware limits; clamp at ±90. | |
| Conservative clamping | Smaller window (e.g. ±60° pan, ±45° tilt) based on physical intuition. | |
| You decide — protocol limits + calibration note | Ship ±90° with named tunable constants. | ✓ |

**Follow-up — single vs. separate constants:**

| Option | Description | Selected |
|--------|-------------|----------|
| Separate constants — HEAD_PAN_MAX and HEAD_TILT_MAX | Pan and tilt are mechanically different; tune independently. | ✓ |
| Single HEAD_ANGLE_MAX constant | Simpler; same limit for both axes. | |

**User's choice:** Full ±90° protocol limits with HEAD_PAN_MAX = 90 and HEAD_TILT_MAX = 90 as separate tunable constants.
**Notes:** User deferred to Claude for the range decision; separate constants chosen because pan/tilt may need independent calibration after live testing.

## Action Naming + Schema

| Option | Description | Selected |
|--------|-------------|----------|
| Two actions: head_pan + head_tilt | Parallel to walk/stop separation. One required param each. | ✓ |
| One action: set_head with axis param | Fewer enum values; LLM must pass axis string. | |

**Follow-up — buzzer:**

| Option | Description | Selected |
|--------|-------------|----------|
| Two actions: buzzer_on + buzzer_off | Matches walk/stop pattern. Explicit intent, no secondary param. | ✓ |
| One action: buzzer with state param | More compact; requires secondary param. | |

**Follow-up — rotate parameter:**

| Option | Description | Selected |
|--------|-------------|----------|
| Signed degrees — rotate(degrees: int) | positive=right, negative=left. Matches HXP-NAV-01 spec directly. | ✓ |
| rotate(direction: left\|right, degrees: int) | Two params like walk. More explicit but adds param. | |

**User's choice:** head_pan/head_tilt as separate actions; buzzer_on/buzzer_off as separate actions; rotate(degrees: int) signed.
**Notes:** All choices follow the walk/stop precedent from Phase 27.1.1 — explicit action names over parameterized state.
