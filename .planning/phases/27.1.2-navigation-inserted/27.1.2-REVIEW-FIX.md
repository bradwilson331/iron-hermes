### CR-01, CR-02, CR-03: rotate overflow, stop propagation, zero no-op

**Files modified:** `crates/ironhermes-tools/src/hexapod_tcp.rs`
**Commit:** 0849e0c1
**Applied fix:**
- CR-01: Added `const MAX_DEGREES: u64 = 3600` cap via `.min()` on `degrees.unsigned_abs()`, switched duration multiplication to `.saturating_mul(ROTATE_MS_PER_DEGREE)`. Added `"minimum": -3600, "maximum": 3600` to the `degrees` JSON schema property.
- CR-02: Replaced `let _ = send_fire_and_forget(&addr, STOP_CMD).await` with a match that emits `tracing::warn!` and returns a distinct warning string on stop failure instead of discarding the error silently.

Updated test 16 from a wire-builder boundary test (which documented the bug) to an async `execute()` no-op test that verifies `rotate(0)` returns `"OK"` without sending a motion command. Also updated the doc comment on `build_rotate_wire` to describe parameter semantics rather than wire-level field values.

### WR-01: walk direction strict validation

**Files modified:** `crates/ironhermes-tools/src/hexapod_tcp.rs`
**Commit:** 6a25acb3
**Applied fix:** Replaced `args["direction"].as_str().unwrap_or("forward")` with an explicit match using an or-pattern `("forward" | "backward" | "left" | "right")`. Invalid direction strings return a descriptive error; absent direction returns a required-parameter error. Unexpected forward motion on typos is no longer possible.

### WR-02: ENV_LOCK mutex poison recovery

**Files modified:** `crates/ironhermes-tools/src/hexapod_tcp.rs`
**Commit:** c338993a
**Applied fix:** Replaced all 7 occurrences of `ENV_LOCK.lock().unwrap()` in the test module with `ENV_LOCK.lock().unwrap_or_else(|p| p.into_inner())`. A panicking test no longer poisons the mutex and cascades failures to all subsequent tests.
