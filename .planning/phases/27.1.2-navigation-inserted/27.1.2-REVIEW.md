# Phase 27.1.2: Code Review Report

**Reviewed:** 2026-05-11
**Depth:** standard

## Summary

Reviewed the Phase 27.1.2 extension to `hexapod_tcp.rs`, which adds `rotate`, `head_pan`, `head_tilt`, `buzzer_on`, and `buzzer_off` actions plus tests 14–20. The structure and patterns are consistent with Phase 27.1.1. The wire constants, allowlist extension, and schema additions are correct. However, one behavioral defect with real physical consequences was found: passing `degrees=0` to `rotate` sends an actual rotation command to the robot before the zero-duration sleep fires and stop is issued. Four additional warning-level issues were found in test coverage quality and a physical safety gap.

## Critical Issues

### CR-01: `rotate` with `degrees=0` sends a motion command and physically jolts the robot

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:99,334`

**Issue:** `build_rotate_wire(0)` returns `"CMD_MOVE#1#0#0#5#10\n"` (the clockwise motion wire) because `0 >= 0` triggers the positive branch. In `execute()`, `degrees.unsigned_abs() * ROTATE_MS_PER_DEGREE` evaluates to `0 * 20 = 0`, so `Duration::from_millis(0)` is produced. The code then:

1. Sends `CMD_MOVE#1#0#0#5#10\n` — robot begins rotating clockwise.
2. Sleeps for 0 ms — yields to the executor but returns on the very next poll.
3. Sends `CMD_MOVE#0#0#0#0#0\n` (stop).

This means a caller passing `degrees=0` intending "no rotation" instead causes the robot to receive a clockwise motion command followed almost immediately by stop. On physical hardware this produces a servo jerk. The zero case should short-circuit before sending any wire command.

**Fix:**
```rust
"rotate" => {
    let degrees = args["degrees"].as_i64().unwrap_or(0);
    if degrees == 0 {
        return Ok("OK".to_string()); // no-op: zero rotation
    }
    let wire = build_rotate_wire(degrees);
    let duration = Duration::from_millis(degrees.unsigned_abs() * ROTATE_MS_PER_DEGREE);
    match send_fire_and_forget(&addr, &wire).await {
        Ok(_) => {
            tokio::time::sleep(duration).await;
            let _ = send_fire_and_forget(&addr, STOP_CMD).await;
            Ok("OK".to_string())
        }
        Err(_) => Ok(format!(
            "Error: cannot connect to robot at {addr} — is HEXAPOD_IP set and the robot powered on?"
        )),
    }
}
```

Additionally, test 16 (`test_rotate_wire_zero_degrees`) asserts that `build_rotate_wire(0)` produces the clockwise wire — this is actually documenting the buggy behavior. The test should instead verify the `execute()` path with `degrees=0` returns `"OK"` without sending any motion command (which requires either a mock or restructuring `build_rotate_wire` to return `None` for zero). At minimum, update the test comment to note that callers must guard against zero before calling the builder.

## Warnings

### WR-01: Stop command failure after rotation is silently discarded with no trace log

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:338`

**Issue:** After the async sleep in the `rotate` arm, the stop command is sent as `let _ = send_fire_and_forget(&addr, STOP_CMD).await;`. If this connection fails (e.g., the robot's Wi-Fi hiccups during the sleep window), the robot continues rotating at max angle rate indefinitely until a power cycle or physical intervention. The error is silently swallowed with no trace, log, or indication in the returned `"OK"` response that the stop failed. The research document acknowledges this as T-27.1.2-05 / "accept" risk, but the absence of even a debug-level trace means there is no operational visibility.

**Fix:**
```rust
let stop_result = send_fire_and_forget(&addr, STOP_CMD).await;
if let Err(e) = stop_result {
    tracing::warn!("hexapod_tcp: rotate stop command failed after sleep: {e}");
    // Return a degraded-OK so the caller knows the stop was not confirmed
    return Ok(format!(
        "Warning: rotate motion sent but stop command failed — robot may still be rotating: {e}"
    ));
}
Ok("OK".to_string())
```

At minimum, add `debug!` or `warn!` logging on stop failure, since `tracing` is already imported.

### WR-02: Tests 17 and 18 validate the clamp constant but not the actual `execute()` code path

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:679–694`

**Issue:** Tests 17 (`test_head_pan_clamps_high`) and 18 (`test_head_tilt_clamps_low`) exercise `i64.clamp(-HEAD_PAN_MAX, HEAD_PAN_MAX)` directly — they replicate the arithmetic inline in the test body rather than calling through `execute()` or through a named helper. If the clamp line were accidentally removed from `execute()` (lines 349 and 361), these tests would still pass because they test the Rust standard library's `clamp` method, not the application code. A regression removing `.clamp(...)` from the `head_pan` arm would go undetected by the test suite.

**Fix:** Either extract a `build_head_wire(axis: u8, angle_raw: i64, max: i64) -> String` pure helper (analogous to `build_walk_wire`) and test that, or restructure tests 17/18 to call `execute()` with an out-of-range angle and assert on the generated wire string indirectly (e.g., by checking the mock TCP payload). The simplest fix is to add a `build_head_wire` helper tested directly:

```rust
pub(crate) fn build_head_wire(axis: u8, angle_raw: i64, max: i64) -> String {
    let angle = angle_raw.clamp(-max, max);
    format!("CMD_HEAD#{axis}#{angle}\n")
}

// Test 17:
assert_eq!(build_head_wire(0, 180, HEAD_PAN_MAX), "CMD_HEAD#0#90\n");
// Test 18:
assert_eq!(build_head_wire(1, -180, HEAD_TILT_MAX), "CMD_HEAD#1#-90\n");
```

### WR-03: `walk` action silently accepts invalid `direction` and executes as `forward`

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:280,92`

**Issue:** `args["direction"].as_str().unwrap_or("forward")` at line 280 silently substitutes `"forward"` for any missing or unrecognised direction value. Combined with the `_` arm in `build_walk_wire` at line 92 (also silently producing forward), a caller that passes `"forwards"` or `"fwd"` receives `"OK"` and the robot moves forward with no indication that the input was invalid. The JSON schema `enum` constraint is advisory only — it is not enforced at runtime. This issue predates Phase 27.1.2 but the phase extends the tool without addressing it.

**Fix:** Validate the direction explicitly in the `walk` arm and return a user-facing error for unrecognised values:
```rust
"walk" => {
    let direction = args["direction"].as_str().unwrap_or("");
    if !matches!(direction, "forward" | "backward" | "left" | "right") {
        return Ok(format!(
            "Error: invalid direction '{direction}' — must be one of: forward, backward, left, right"
        ));
    }
    let speed = args["speed"].as_i64().unwrap_or(5);
    // ...
}
```

### WR-04: `ENV_LOCK` mutex poisoning causes cascading test failures on any panic

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:432,439,460,480,500,603,711`

**Issue:** All env-var-mutating tests acquire `ENV_LOCK` with `.lock().unwrap()`. If any test holding the lock panics (e.g., an assertion failure), the mutex is poisoned and every subsequent `.lock().unwrap()` call in other tests will also panic, producing a cascade of misleading failures that obscure the original failure. This is a standard Rust test-infrastructure hazard.

**Fix:**
```rust
// Replace all occurrences of:
let _guard = ENV_LOCK.lock().unwrap();

// With:
let _guard = ENV_LOCK.lock().unwrap_or_else(|p| p.into_inner());
```

This recovers the inner guard from a poisoned mutex, allowing subsequent tests to proceed normally. The poisoned state indicates a previous test panicked, but the remaining tests should run independently.

### IN-01: Comment on ENV_LOCK thread-safety exemption overstates the safety of pure tests

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:430–431`

**Issue:** The comment states "Tests that DO NOT call set_var/remove_var are exempt and can run concurrently as normal." Tests 14–19 are plain `#[test]` fns that do not hold `ENV_LOCK`. However, they run in the same process as the async `ENV_LOCK`-holding tests, so their safety depends on `--test-threads=1` being passed at the call site. The comment implies inherent thread-safety from not calling `set_var`, but the actual guarantee comes solely from the single-threaded test runner flag documented elsewhere. If someone runs the test binary without `--test-threads=1`, tests 14–19 could interleave with env-var mutation from other tests — the pure tests themselves are safe, but tests 1/2/3/4/12b/20 that mutate the env without holding the lock for the full execution window are not.

**Fix:** Tighten the comment to note that ENV_LOCK is only effective when `--test-threads=1` is used:
```rust
/// Serialize all env-var-mutating tests so parallel test threads cannot race on HEXAPOD_IP.
/// IMPORTANT: This mutex is only effective when the test binary is run with --test-threads=1.
/// Pure tests (14–19) do not mutate env vars and are safe in any thread count.
```
