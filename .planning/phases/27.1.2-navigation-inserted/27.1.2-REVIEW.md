# Phase 27.1.2: Code Review Report

**Reviewed:** 2026-05-11
**Depth:** standard
**Files Reviewed:** 1
**Status:** issues_found

## Summary

Reviewed `crates/ironhermes-tools/src/hexapod_tcp.rs`, which extends the Phase 1 hexapod TCP
tool with `rotate`, `head_pan`, `head_tilt`, `buzzer_on`, and `buzzer_off` actions. The wire
constants, allowlist extension, buzzer/head arms, and test infrastructure are structurally
sound. However, three critical blockers were found, all in the `rotate` action.

The most severe is a `u64` integer overflow in the rotate duration calculation: for large input
values (near `i64::MAX`), the product of `degrees.unsigned_abs() * ROTATE_MS_PER_DEGREE`
overflows `u64`. In debug builds this panics; in release builds wrapping arithmetic produces a
sleep duration of approximately 584 million years, leaving the robot in permanent continuous
rotation. The second blocker is that the stop command sent after rotation silently discards its
error — if the TCP connection fails during the sleep window the robot continues rotating and the
caller receives `"OK"`. The third is that `degrees=0` emits a live clockwise motion command
rather than a no-op, relying on the same failure-prone stop path.

Four warnings cover silent direction defaulting for `walk`, `ENV_LOCK` mutex-poison cascades
in tests, incomplete cross-module env-var race protection, and a malformed battery-response
silent parse failure that produces misleading output.

## Critical Issues

### CR-01: Integer overflow in `rotate` duration — robot rotates forever (release) or panics (debug)

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:334`

**Issue:** The rotate sleep duration is computed as:

```rust
let duration = Duration::from_millis(degrees.unsigned_abs() * ROTATE_MS_PER_DEGREE);
```

`degrees.unsigned_abs()` returns `u64`. `ROTATE_MS_PER_DEGREE` is `u64`. For
`degrees = i64::MAX` (9223372036854775807), the product overflows `u64`:

- **Debug build:** the `*` operator panics on overflow — the `execute()` future aborts.
- **Release build:** wrapping multiplication produces 18446744073709551596, i.e. a sleep
  of ~584 million years. The motion command is sent, the task parks indefinitely, and STOP
  is never transmitted. The robot rotates until powered off.

For `degrees = i64::MIN`, release-mode wrapping produces 0 ms — a different silent failure
(the stop arrives immediately but the motion command was already sent, per CR-03).

The JSON Schema for `degrees` has no `minimum`/`maximum` bounds, so any `i64` value is
valid input from the LLM. This is not a theoretical edge case.

**Fix:**

```rust
"rotate" => {
    let degrees = args["degrees"].as_i64().unwrap_or(0);
    if degrees == 0 {
        return Ok("OK".to_string()); // no-op
    }
    let wire = build_rotate_wire(degrees);
    // Cap absolute degrees to prevent overflow and runaway rotation.
    const MAX_DEGREES: u64 = 3600; // 10 full rotations
    let abs_degrees = degrees.unsigned_abs().min(MAX_DEGREES);
    let duration = Duration::from_millis(
        abs_degrees.saturating_mul(ROTATE_MS_PER_DEGREE)
    );
    match send_fire_and_forget(&addr, &wire).await {
        Ok(_) => {
            tokio::time::sleep(duration).await;
            match send_fire_and_forget(&addr, STOP_CMD).await {
                Ok(_) => Ok("OK".to_string()),
                Err(_) => Ok(format!(
                    "Warning: rotate completed but stop command failed at {addr} — robot may still be moving"
                )),
            }
        }
        Err(_) => Ok(format!(
            "Error: cannot connect to robot at {addr} — is HEXAPOD_IP set and the robot powered on?"
        )),
    }
}
```

Also add `"minimum": -3600, "maximum": 3600` to the `degrees` schema property.

### CR-02: Stop command after rotate silently discarded — robot continues rotating on TCP failure

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:338`

**Issue:**

```rust
let _ = send_fire_and_forget(&addr, STOP_CMD).await;
Ok("OK".to_string())
```

After the timed sleep the stop-command error is discarded with `let _`. If the robot's
network connection drops during the sleep window (Wi-Fi hiccup, robot reboot), the TCP
connection fails silently and the robot continues rotating at speed 5 indefinitely. The
caller receives `"OK"` with no indication that the stop failed. This is a physical safety
hazard on real hardware.

The research document notes this as an accepted risk (T-27.1.2-05) but the implementation
does not even emit a `warn!` trace for operational visibility. The `tracing` crate is already
imported.

**Fix:** Return a distinct error string and add a warning trace (see fix in CR-01 above):

```rust
let stop_result = send_fire_and_forget(&addr, STOP_CMD).await;
if let Err(e) = &stop_result {
    tracing::warn!("hexapod_tcp: rotate stop command failed: {e}");
    return Ok(format!(
        "Warning: rotate motion sent but stop command failed — robot may still be rotating"
    ));
}
Ok("OK".to_string())
```

### CR-03: `rotate` with `degrees=0` emits a live clockwise motion command instead of a no-op

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:99` and `334`

**Issue:** `build_rotate_wire(0)` returns `"CMD_MOVE#1#0#0#5#10\n"` (clockwise, speed 5)
because the branch is `if degrees >= 0`. In `execute()`, the sleep duration is 0 ms, so:

1. Robot receives `CMD_MOVE#1#0#0#5#10` — begins clockwise rotation.
2. Executor yields and immediately resumes (0 ms sleep).
3. Stop command is attempted — and silently dropped if it fails (CR-02).

A caller requesting `rotate(0)` intending "no rotation" instead causes a servo jerk and,
if the stop fails, continuous rotation. The zero case must short-circuit before any wire
command is sent. Test 16 (`test_rotate_wire_zero_degrees`) documents this buggy behavior
rather than guarding against it.

**Fix:** Guard zero in `execute()` (see CR-01 fix). Also update `build_rotate_wire` to
use a strict positive check so the function itself is correct as a pure helper:

```rust
pub(crate) fn build_rotate_wire(degrees: i64) -> String {
    // Precondition: degrees != 0 (callers must guard; execute() returns early for 0).
    let angle = if degrees > 0 { 10i64 } else { -10i64 };
    format!("CMD_MOVE#1#0#0#5#{angle}\n")
}
```

Update test 16 to verify the `execute()` early-return for `degrees=0`, not the wire builder.

## Warnings

### WR-01: `walk` action silently defaults to `"forward"` on missing or invalid `direction`

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:280` and `92`

**Issue:**

```rust
let direction = args["direction"].as_str().unwrap_or("forward");
```

If `direction` is absent, `null`, or a non-string (e.g., a typo like `"forwards"`), the
robot silently walks forward. The JSON Schema `enum` constraint is advisory only — it is not
enforced at runtime. A caller that omits `direction` receives `"OK"` and unexpected forward
motion rather than an error prompting a retry.

**Fix:**

```rust
let direction = match args["direction"].as_str() {
    Some(d @ ("forward" | "backward" | "left" | "right")) => d,
    Some(other) => return Ok(format!(
        "Error: invalid direction '{other}' — must be one of: forward, backward, left, right"
    )),
    None => return Ok(
        "Error: 'direction' parameter is required for the walk action".to_string()
    ),
};
```

### WR-02: `ENV_LOCK` mutex poisoning causes cascading test failures on any single panic

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:439,460,480,500,603,711`

**Issue:** All env-var-mutating tests acquire `ENV_LOCK` with `.lock().unwrap()`. If any
test holding the lock panics (assertion failure, etc.), the mutex is poisoned. Every
subsequent `.lock().unwrap()` call in other tests then also panics, producing a cascade of
misleading failures that obscure the root cause.

**Fix:** Replace all occurrences with poison recovery:

```rust
// Before:
let _guard = ENV_LOCK.lock().unwrap();

// After:
let _guard = ENV_LOCK.lock().unwrap_or_else(|p| p.into_inner());
```

### WR-03: ENV_LOCK serializes only this module — other test modules in the binary can still race on HEXAPOD_IP

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:429-432`

**Issue:** `ENV_LOCK` is a module-local `Mutex`. Other test modules in the same `cargo test`
binary that also read or mutate `HEXAPOD_IP` are not protected by it. The `unsafe` blocks
are correct per Rust 1.81+ (which requires `unsafe` to signal the inherent data-race hazard
on environment mutation), but the hazard persists across module boundaries within the same
test process. The comment implies broader safety than exists.

**Fix:** Either use a workspace-level env-var serialization crate (e.g., `serial_test`), or
document that the test binary must be run with `RUST_TEST_THREADS=1`:

```rust
/// Serialize env-var-mutating tests within this module.
/// NOTE: This mutex only protects against races within this module.
/// Run the full test binary with RUST_TEST_THREADS=1 to avoid races
/// with other modules that may also read HEXAPOD_IP.
static ENV_LOCK: Mutex<()> = Mutex::new(());
```

### WR-04: Malformed battery response silently reports `0V / 0V (LOW)` — no parse error surfaced

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:106-112`

**Issue:**

```rust
let v1: f32 = parts.get(1).unwrap_or(&"0").parse().unwrap_or(0.0);
let v2: f32 = parts.get(2).unwrap_or(&"0").parse().unwrap_or(0.0);
```

If the robot sends a malformed response (e.g., `"ERROR\n"` or an incomplete `"CMD_POWER\n"`),
both voltages default to `0.0`. The caller receives `"Battery: 0V / 0V (LOW)"`, which
looks like a genuine critical low-battery reading. An LLM agent may take incorrect follow-up
actions (emergency shutdown, user alert) based on fabricated data.

**Fix:** Detect missing fields and return an unambiguous error string:

```rust
pub(crate) fn parse_battery_response(raw: &str) -> String {
    let parts: Vec<&str> = raw.trim().split('#').collect();
    match (parts.get(1), parts.get(2)) {
        (Some(s1), Some(s2)) => {
            let v1: f32 = s1.parse().unwrap_or(0.0);
            let v2: f32 = s2.parse().unwrap_or(0.0);
            let status = if v1 < BATTERY_LOW_V1 || v2 < BATTERY_LOW_V2 { "LOW" } else { "OK" };
            format!("Battery: {v1}V / {v2}V ({status})")
        }
        _ => format!("Error: unexpected battery response from robot: {:?}", raw.trim()),
    }
}
```

### IN-01: `build_rotate_wire` doc comment describes wire-level field values, not the parameter semantics

**File:** `crates/ironhermes-tools/src/hexapod_tcp.rs:97`

**Issue:** The comment reads:

```
/// angle=+10 clockwise, angle=-10 counterclockwise. x=0, y=0 → in-place. Speed fixed at 5 per D-02.
```

This describes the wire-level `angle` field emitted in the CMD_MOVE string, not the `degrees`
parameter. A reader may conflate the two and incorrectly assume the function accepts wire-level
angle values directly.

**Fix:**

```rust
/// Converts a signed degree count to a CMD_MOVE in-place rotation wire string.
/// Positive degrees → wire angle field +10 (clockwise); negative → -10 (counterclockwise).
/// Precondition: degrees != 0 (execute() guards this; zero emits the positive branch spuriously).
/// Speed is fixed at 5 (D-02); caller is responsible for sleeping the appropriate duration
/// before sending STOP.
```
