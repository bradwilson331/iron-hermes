<objective>
Land D-05: skill-declared env vars bypass the Phase 8 sandbox secret-strip in `ironhermes-exec::sandbox::build_env`. This is the final wiring that closes the SKILL-11 loop — a user exports `TENOR_API_KEY=...`, activates the Tenor skill, and the Python child process launched by `execute_code` sees `os.environ["TENOR_API_KEY"]`.

Purpose: `SECRET_PATTERNS` in sandbox.rs currently strips any env var whose uppercase name contains `KEY`, `TOKEN`, `SECRET`, etc. — this correctly blocks accidental secret leakage but incorrectly blocks skill-declared keys too. D-05 adds a declared-and-present whitelist that bypasses the strip.

Output: `build_env` signature extension + call-site updates + integration test that Python child process reads the declared var + regression test that non-declared secrets still get stripped.

Wave 3 because this depends on Plan 01 (HermesMetadata + EnvVarEntry) AND Plan 03 (SkillsTool.active_skills list where the whitelist is sourced from).
</objective>

    /// Collect declared env var names across all currently-active skills.
    /// Returns only names declared by the skill; does NOT check parent env presence
    /// (build_env handles the presence check naturally — whitelisted names that aren't
    /// in std::env::vars() simply never enter the returned env Vec).
    pub fn active_skill_env_names(active: &[SkillRecord]) -> Vec<String> {
        let mut out: Vec<String> = Vec::new();
        for record in active {
            if let Some(meta) = &record.hermes_metadata {
                for entry in &meta.required_environment_variables {
                    if !out.contains(&entry.name) {
                        out.push(entry.name.clone());

                }
            }
        }
        out
