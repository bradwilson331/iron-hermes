## 1. System Classification

**System Type:** Hybrid — Multi-Agent (subagent delegation runtime polish) + Autonomous Agent (`--yolo` non-interactive mode) + Code Automation (background process registry / sandbox status).

**Description:**
IronHermes is a Rust coding-agent CLI (cargo workspace: `ironhermes-agent`, `ironhermes-tools`, `ironhermes-cli`, `ironhermes-exec`, `ironhermes-core`, `ironhermes-providers`, etc.) that speaks to multiple LLM providers (Anthropic / OpenAI / others) and executes tool-use loops driven by its own agent runtime. Phase 21.7 closes four v2.0 parity gaps against the Python `hermes-agent` reference: (a) surfaces the existing `delegate_task` multi-agent system with `/agents`, status-line pill, persistent JSONL transcripts, cascade cancellation, wall-clock timeout, concurrency surface, and parent/child iteration-budget inheritance; (b) adds `--yolo` non-interactive mode that bypasses dangerous-command approvals and continues past tool errors; (c) adds `hermes status [--all] [--deep] [--json]` component diagnostics; (d) adds an in-memory session-scoped `ProcessRegistry` for `terminal(background=true)` / `execute_code` with spawn/poll/wait/kill, 200KB rolling output buffer, `/stop` slash, watch patterns with rate-limited notifications, and cleanup on session end. "Good" = hermes-agent parity for these four surfaces, no regression in existing single-agent flow, and observable, interruptible behavior for autonomous / long-running work.

**Critical Failure Modes:**
1. **Orphaned subagents / leaked processes.** Parent ctrl-c (double-ctrl-c) fails to cascade-cancel active subagents or drain tracked background processes on `on_session_end`, leaving zombies holding PIDs, tokens, file handles, or network sockets across sessions.
2. **Unbounded autonomous runaway.** `--yolo` blanket approval bypass executes a destructive tool (e.g., `rm -rf`, force-push, drop table) without user consent because the iteration-budget hard-stop (100% tier) fails to fire, pressure advisories never reach the model, or the fatal-error halt path is bypassed.
3. **Silently dropped safety signals.** Watch-pattern rate limiter drops matches past the 8/10s cap without surfacing backpressure, transcripts fail to flush before cleanup, or `hermes status` reports stale / missing state — so operators cannot tell what the agent is doing.
4. **Shared-budget desync under concurrency.** Parent and subagent turns fail to decrement the same atomic iteration counter (PROV-10), causing the budget to read "fine" in `hermes status` while the system is deep into pressure / stop state — or vice-versa, halting legitimate work prematurely.
5. **False-negative `hermes status --deep` probes.** `--deep` probes (provider HEAD, FTS5 integrity, MCP tools/list) report "ok" when a component is actually broken, masking real production incidents.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Developer tooling — specifically, **autonomous / multi-agent coding-agent CLIs** (the same category as Claude Code, OpenAI Codex CLI, Aider, Cline, Cursor's agent mode, and the Python `hermes-agent` reference that IronHermes targets for parity). Not a regulated vertical; domain norms come from practitioner convention plus a small number of high-profile production failures.

**User Population:** Senior software engineers, SREs, and platform engineers who run IronHermes locally (or on a dev VM / over SSH) as their daily coding-agent CLI. They drive it through tool-use loops that read/write code, execute shell commands in the sandbox, spawn long-running background processes (build pipelines, test suites, `cargo watch`, dev servers), and delegate sub-tasks to subagents. Stakes escalate sharply when `--yolo` is enabled — in that mode, the CLI executes dangerous commands without per-command approval, so the operator has already ceded blast-radius decisions to the iteration budget, the cancellation tree, and the sandbox boundary. A secondary population is incident responders running `hermes status` against a running gateway to triage a misbehaving session.

**Stakes Level:** **High.** The four parity features in scope are specifically the lifecycle / safety / observability primitives that make autonomous operation safe to leave unattended. A defect in any one of them turns a well-behaved agent into an incident (leaked processes, runaway destructive commands, silently dropped safety signals, misread status output driving the wrong mitigation).

**Output Consequence:** Concrete downstream harms when the Phase 21.7 surfaces misbehave, each tied back to Section 1's failure modes:
- **Leaked PIDs / file handles / sockets** → developer laptop slowdown, "zombie hermes" processes consuming the provider API budget across sessions, port collisions on next `cargo run`.
- **`--yolo` approval bypass failing open** → irreversible destructive actions on the developer's own machine or, worse, on a production system the developer was connected to (the Replit July-2025 incident is the canonical public anti-pattern: agent ran destructive commands during a code-freeze window, wiped a production database holding 1,200+ users, then misled the operator about recoverability).
- **Dropped watch-pattern signal** → agent misses a build failure or test regression and keeps iterating against stale context, producing code that compiles against reality that no longer exists.
- **Stale / wrong `hermes status` output** → incident responder investigates the wrong component, wasting mean-time-to-diagnose on a live outage.
- **Shared-budget desync** → either the operator interrupts a legitimate long-running task early because status says "at pressure" when it isn't, or the agent burns through the cap without the operator seeing advisories in time to intervene.

### Known Failure Modes in This Domain

Concrete manifestations a senior-developer reviewer expects us to have defences against — each with a one-line mitigation hint tied back to CONTEXT.md / AI-SPEC.md.

- **Zombie / leaked child processes across sessions.** Orphan `python`, `node`, `cargo`, or `pytest` children linger after the agent exits, holding PIDs, file handles, and network ports. *Mitigation:* `ProcessRegistry::drain_and_kill()` on `on_session_end` (D-24), NOT on Drop; pair with integration test that asserts zero orphans after simulated ctrl-c.
- **Autonomous runaway under `--yolo`.** Agent loops on a destructive-adjacent task (force-push, migration rollback, `rm -rf` on a path it mis-parsed), each iteration widening blast radius. Compounds with deceptive post-hoc explanations (see Replit 2025 incident). *Mitigation:* iteration-budget 100% hard stop (D-15), wall-clock subagent timeout with real child-token cancel (D-08 + Pitfall 5), `/stop` slash kills background children (D-26), tests that prove budget-stop wins a race with an in-flight tool call.
- **Silent watch-pattern drop during a notification storm.** Build spews error lines faster than the 8/10s cap; the critical "tests failed" line lands in the dropped window. Agent continues iterating against stale context. *Mitigation:* `tracing::warn` on every cap drop with PID + dropped-count (Pitfall 7), 45-second sustained-overload disables watch **for that process only** with an explicit user-visible message.
- **Non-TTY deadlock in gateway or yolo mode.** Any stdin-reading prompt (clarify, approval, sudo, pairing) hangs the agent loop when there is no TTY. *Mitigation:* every interactive read must be gated behind `!config.autonomous.yolo && isatty(stdin)` (Pitfall 10); gateway mode rejects any request-level `yolo` field (D-12).
- **Shared-budget ordering bug.** `Relaxed` atomic ordering on `BudgetHandle` makes `hermes status` report a stale `remaining` value; operator thinks the agent has headroom it doesn't. *Mitigation:* SeqCst on all reads and decrements (Pitfall 9), unit test with concurrent parent + subagent loops hammering the same handle.
- **Three-site drift.** A new handle gets threaded into `run_chat` but not `run_single` or `run_gateway`; behaviour diverges between CLI and gateway. Users hit it only in production. *Mitigation:* static-grep regression test per Phase 22 precedent (Pitfall 1).

### Regulatory / Compliance Context

**No directly-applicable formal regulatory regime.** IronHermes is a developer-tooling CLI with local or on-dev-VM deployment — HIPAA, PCI-DSS, FCA, GDPR-for-data-processors, etc. do not apply to the runtime itself. The following **soft constraints** are the only compliance-adjacent considerations, and they are enumerated honestly so the eval-planner doesn't over-reach:

- **Enterprise developer-workstation logging (SOC2 / ISO 27001).** Operators running IronHermes inside SOC2-governed organisations need the subagent transcripts (`$HERMES_HOME/subagent-transcripts/<session>/<id>.jsonl`, D-05) and `hermes status --json` output (D-20) to be auditable — stable schema, timestamps, no PII-by-accident leakage into transcript filenames or status output. This is not a hard regulation, but it is a realistic deployment constraint for the target user population.
- **GDPR for on-disk session data.** Transcripts, `state.db`, and memory files may contain personal data if the operator's prompts mention it. Purge-on-session-end (D-05) must honour the "interrupted" flag cleanly so that right-to-erasure requests can be satisfied by deleting a session directory. Not a Phase 21.7 deliverable, but the transcript layout chosen here must not block it later.
- **Blast-radius containment is not a regulation but an industry expectation** after the July-2025 Replit incident. A coding-agent CLI that can run `--yolo` without a budgeted kill-switch is now considered reckless by the practitioner community; this phase's eval rubric should treat budget-stop correctness and cascade-cancel correctness as equivalent to a safety guardrail in a regulated domain.

No other regimes (HIPAA, PCI-DSS, FedRAMP, FINRA, etc.) apply. Do not fabricate coverage that isn't there.

## 2. Framework Decision

**Selected Framework:** IronHermes (Rust, in-house) — the existing runtime. No new framework is introduced by Phase 21.7.

**Version:** Current `develop` branch of this repo (tracked per-commit; not a pinned external dependency). Key crate boundaries: `ironhermes-agent`, `ironhermes-tools`, `ironhermes-cli`, `ironhermes-exec`, `ironhermes-core`, `ironhermes-providers`.

**Rationale:**
IronHermes *is* the framework — a Rust cargo-workspace coding-agent runtime with its own agent loop (`AgentLoop::run_agent_turn`), tool registry (`ironhermes-tools`, including `DelegateTaskTool` / `SubagentRunner`), CLI (`ironhermes-cli` with TUI pills + status line + three registration sites), exec sandbox (`ironhermes-exec`), and provider abstraction (`ironhermes-providers`). Phase 21.7 is lifecycle / visibility / safety polish on infrastructure that already exists (D-01, D-02: `delegate_task` is explicitly NOT being re-implemented). Selecting an external framework (Python or otherwise) would be category-error — the work is to extend existing traits (`SubagentRunner`), add wiring at existing registration sites, and surface existing state through new CLI and slash commands. Vendor-lock question is n/a for the same reason.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangGraph | Python, not Rust. Re-implementing IronHermes' agent loop on top of an external graph-based orchestrator would rewrite the project, not polish it. Violates D-01 (minimum viable parity cut) and D-02 (preserve existing runner). |
| CrewAI | Python, role-based multi-agent framework. IronHermes' delegation model (`SubagentRunner` trait, `AgentSubagentRunner` impl, `build_child_registry` toolset restrictions per AGENT-01..05) already differs in shape; swapping in CrewAI would be a full port, not a parity closeout. |
| LlamaIndex | Python, RAG-first framework. IronHermes is not a RAG system; memory/session storage already lives in `ironhermes-core` (state.db, FTS5). No fit. |
| AutoGen | Python, conversation-centric multi-agent. IronHermes' subagent model is delegation-centric (parent invokes `delegate_task`, child runs tool loop, returns structured result) — a different topology. Replacing it would invalidate existing CLI + gateway + toolset-restriction machinery. |
| Anthropic Agents SDK | Python. Same objection as above; also IronHermes is provider-agnostic via `ironhermes-providers` (Anthropic, OpenAI, others) and cannot be locked to a single provider's SDK without violating that abstraction. |
| Rust agent crates (e.g., `rig`, `genai`, `swiftide`) | Possible future reference points, but adopting any of them would mean re-homing the agent loop + tool registry + sandbox, which is explicitly out of scope. Re-evaluate for future major versions, not Phase 21.7. |

**Vendor Lock-In Accepted:** N/A — no new framework introduced. IronHermes remains provider-agnostic (multi-provider via `ironhermes-providers`) and has no external runtime-framework lock-in. Phase 21.7 retains that property.

### Common Pitfalls

Every item here is either documented in CONTEXT.md (D-XX) or surfaced by prior-phase RESEARCH / verification.

1. **Three registration sites must stay in sync.** `crates/ironhermes-cli/src/main.rs` calls `register_delegate_task_tool` at L370 (run_single), L683 (run_chat), and L1287 (run_gateway). Any new handle threaded through (ProcessRegistry, BudgetHandle, updated timeout) must touch ALL three or the CLI and gateway diverge. A static-grep regression test per the Phase 22 precedent is the standard gate. Caught in production before, will be caught again.
2. **`child_token()` must chain from the parent token, not be freshly constructed.** `CancellationToken::new()` inside `register_delegate_task_tool` yields an independent token; ctrl-c on the parent does not cascade. The existing wiring at `main.rs:640-641` and `main.rs:688` is correct — the root parent lives the whole session and `chat_cancel_parent.child_token()` is what gets passed in. Re-entrancy bug: `DelegateTaskTool` internally re-derives `parent_token.child_token()` (see `delegate_task.rs:517-519`); breaking that chain silently disables cascade cancel. Phase 21 RESEARCH §Pitfall 2 + §Pitfall 7 call this out and drive the 3rd-ctrl-c emergency escape at `main.rs:646-651`.
3. **Fire-and-forget writes must swallow errors.** Subagent-transcript writes (D-05) run inside `tokio::spawn` and must NOT `.unwrap()` or bubble errors — a failed JSONL write cannot stall or kill the turn. Log at `tracing::warn` and drop. Same rule as Phase 21.2 JSONL event log and the existing `SubagentProgressCallback` (which only emits `eprintln!`, never returns a Result).
4. **`--yolo` in gateway mode reads config only (D-12).** The gateway must NOT read a per-message `--yolo` field from the inbound request; that would let any connected platform bypass safety. CLI flag wins if both config+flag are set, but gateway has no flag — `config.autonomous.yolo` is the only path. Write a test for `run_gateway` that asserts it rejects/ignores request-level yolo fields.
5. **`tokio::time::timeout` does not cancel the inner future.** When the timeout fires, the wrapper Err's, but the spawned child agent keeps running in the background unless its `CancellationToken` is cancelled. `DelegateTaskTool::execute_batch` relies on the `.await` point returning — but for D-08 hard-cancel, the wrapper must also call `.cancel()` on the child token before returning the timeout error. Current code at `delegate_task.rs:265-283,547-579` returns a Rust `anyhow::Error` on timeout but does NOT cancel the child token; 21.7 adds that cancel.
6. **`ProcessRegistry` cleanup must happen on `on_session_end`, NOT on Drop (D-24).** Async drop is a trap — `tokio::spawn` from a Drop impl requires a runtime that may be shutting down, and blocking kills inside Drop panic on the reactor. Use the Phase 21.4 GAP-5 `on_session_end` hook pattern instead: read the registry, `drain_and_kill()` each `ProcessHandle`, await completion, then clear.
7. **Watch-pattern rate limiter silently dropping is a critical failure (Section 1 item 3).** The 8-notifications-per-10s cap (D-27, `WATCH_MAX_PER_WINDOW=8`, `WATCH_WINDOW_SECONDS=10`) prevents notification storms, but a silent drop loses safety signal. Every capped drop MUST emit a `tracing::warn!("watch pattern backpressure: dropped N matches in last 10s for pid={}", pid)` and, after 45s sustained at cap (`WATCH_OVERLOAD_KILL_SECONDS`), disable watch for that process (not the whole registry). Mirror hermes `process_registry.py` semantics exactly.
8. **Status-line pill must hide at zero active subagents (D-04).** Showing `agents: 0/4` at idle is pure visual noise and fights the existing pills' density. The render path must drop the pill from the `Vec<String>` before calling `rotate_pill_colors` when `active_subagents == 0`. Add a unit test in `pills.rs` mirroring the existing `empty_hint_omits_trailing_hint_pill` (`status_line.rs:167-181`).
9. **Shared-budget atomic ordering (Section 1 item 4).** `AtomicUsize::fetch_sub` with `Ordering::Relaxed` is wrong for a shared parent/child counter — the reader-side `hermes status` read must see the writer-side decrement. Use `Ordering::SeqCst` (or paired `Release`/`Acquire`) on both sites. The existing `AgentSubagentRunner` holds the `Arc<AtomicUsize>` (`subagent_runner.rs:30`) but does not yet enforce ordering; 21.7 standardises on `SeqCst` for correctness over microbenchmark gains.
10. **Non-TTY subagent dispatch.** `--yolo` non-interactive mode + gateway mode both run without a TTY. The `SubagentProgressCallback` in `main.rs:654-681` writes to `eprintln!`, which is fine, but any approval prompt or confirmation read from stdin would deadlock. D-11 blanket bypass dodges this for dangerous commands, but any NEW prompt added to the subagent path must be gated behind `!config.autonomous.yolo && isatty(stdin)`.

## 4. Implementation Guidance

**Model Configuration:**
**N/A for Phase 21.7.** Provider/model selection flows through `ironhermes-providers` (Anthropic, OpenAI, others) and `ironhermes-core::ProviderResolver`, which this phase does not touch. No temperature, token-limit, or sampling parameters are introduced. What Phase 21.7 *does* configure is the **iteration budget** and **wall-clock subagent timeout**, which are provider-agnostic:

- `agent.max_iterations` — parent + shared-child iteration ceiling (D-15). Default: Claude's Discretion, likely 50.
- `subagent.default_timeout_seconds` — wall-clock cap on a single `delegate_task` call, overridable per-call via `timeout_seconds` arg (D-08). Default: Claude's Discretion, likely 300.
- Pressure tiers (D-15): **70%** → inject "consider consolidating" system-prompt advisory; **90%** → inject "respond now" directive; **100%** → terminate turn cleanly, flush memory, return to prompt. Tiers are exact; advisory text is Claude's Discretion (hardcoded is acceptable for v2.0).
- `config.autonomous.yolo: bool` (D-11/D-12). CLI `--yolo` flag wins over config. Gateway mode reads config only.

**Core Pattern:**
Rust tokio-native concurrent-state pattern. Three shared state objects live in `Arc<RwLock<_>>` (for reader-heavy surface reads from `/agents` and `hermes status`); the atomic counter lives bare inside `Arc<AtomicUsize>` because fast hot-path reads don't want a `RwLock::read().await`.

```rust
// Shared state: cheap clone, read-mostly, write on mutation.
let subagent_registry: Arc<RwLock<SubagentRegistry>> = Arc::new(RwLock::new(SubagentRegistry::default()));
let process_registry:  Arc<RwLock<ProcessRegistry>>  = Arc::new(RwLock::new(ProcessRegistry::new_for_session(session_id)));

// Concurrency bound: tokio::sync::Semaphore, sized from config.
let subagent_semaphore = Arc::new(Semaphore::new(config.subagent.max_subagents));

// Budget: bare atomic; parent and child decrement the SAME Arc (PROV-10, D-16).
let budget: Arc<AtomicUsize> = Arc::new(AtomicUsize::new(config.agent.max_iterations));

// Cancellation tree: one root per session, child tokens minted per-turn and per-subagent.
let session_cancel = CancellationToken::new();          // lives the session
let turn_token     = session_cancel.child_token();      // one per turn
let delegate_token = session_cancel.child_token();      // one for delegate_task's long-lived handle

// Status-line watch: render task polls `.borrow()`, no await in hot path.
let (status_tx, status_rx) = tokio::sync::watch::channel(StatusLineState::default());
```

Budget decrement is the critical correctness site — same handle from both parent and child:

```rust
// Parent turn (ironhermes-agent/src/agent_loop.rs::run_agent_turn):
let remaining = budget.fetch_sub(1, Ordering::SeqCst);
if remaining == 0 {
    // D-15 stop tier: terminate cleanly, do not panic.
    return Ok(AgentResult::budget_exhausted());
}
let pressure = pressure_tier(remaining, max_iterations); // 70/90/100
if let Some(advisory) = pressure.advisory_text() {
    messages.push(ChatMessage::system(advisory));       // inject before next turn
}

// Child turn (ironhermes-agent/src/subagent_runner.rs::run_child → AgentLoop):
// The child AgentLoop receives the SAME Arc<AtomicUsize> via
// .with_budget(self.budget.clone()) at subagent_runner.rs:76,
// so it decrements the parent's counter — PROV-10 / D-16.
```

**Tool Use:**
Tool registration extensions happen in `crates/ironhermes-tools/src/registry.rs:280-335`.

- **`delegate_task` (D-08):** add an optional `timeout_seconds: u32` input-schema field. At execute time, `args["timeout_seconds"].as_u64().unwrap_or(config.timeout_secs)` picks per-call override. Thread it into the existing `tokio::time::timeout(Duration::from_secs(effective_timeout), runner.run_child(...))` wrapper at `delegate_task.rs:265-276` and `:547-558`. On `Err(_)` from the timeout, **also call `child_cancel_token.cancel()`** before returning the structured error — see Pitfall 5.
- **`terminal` + `execute_code` (D-29):** add a new registration variant `register_*_with_process_registry(..., process_registry: Arc<RwLock<ProcessRegistry>>)` that mirrors the existing `register_execute_code_tool_with_active_skills` pattern at `registry.rs:321-334`. The tool's `execute` impl branches: `if args["background"].as_bool().unwrap_or(false)` → route through `ProcessRegistry::spawn`; else unchanged foreground path. Watch patterns (D-27) attach at spawn time.
- **Tool-error feedback (D-13):** tool failures under `--yolo` serialize to a structured JSON error (`{"error": "...", "code": "..."}`) and return via the standard tool-result channel — the agent sees it as normal content and keeps iterating. No per-tool retry counter. Hard-stops only on (a) budget 100%, (b) fatal framework panic / provider death, (c) user interrupt.

**State Management:**
Session-scoped, in-memory, keyed by `task_id` (hermes convention). Explicitly **no** SQLite table, **no** JSON checkpoint file (D-23 rejects persistence for process_registry; the existing `state.db` already owns persistent session metadata).

- `ProcessRegistry` lives for the life of one session, drained by the `on_session_end` hook (Phase 21.4 GAP-5) via `drain_and_kill()` (D-24). LRU-prune at 64 concurrent processes; finished processes kept 30min (`FINISHED_TTL_SECONDS`).
- `SubagentRegistry` is similarly in-memory, session-scoped. `/agents` reads a snapshot; active-count feeds the status-line pill watch channel.
- Subagent transcripts (D-05) are the only disk artefact: `$HERMES_HOME/subagent-transcripts/<session_id>/<subagent_id>.jsonl`, one line per turn (tool_call / tool_result / stream_delta / done). Written fire-and-forget via `tokio::spawn`; errors are logged at `tracing::warn` and swallowed (Pitfall 3). Purged on `on_session_end` unless the session was flagged "interrupted" (parity with session storage).
- Budget state is **not** persisted — a crashed session reboots to a fresh `max_iterations` counter. Same as hermes-agent.

**Context Window Strategy:**
IronHermes' "context window" in Phase 21.7 is the **iteration budget**, not an LLM token window. The 2-tier pressure model (D-15) is the single strategy:

- **Caution tier — 70% consumed:** before each subsequent turn, the agent loop injects a system message advisory: "You have used ~70% of your iteration budget; consider consolidating remaining work." Advisory-only; model chooses whether to respond.
- **Warning tier — 90% consumed:** injection text changes to directive: "You have used ~90% of your iteration budget; respond with your final answer now." Stronger signal; model is expected to terminate soon.
- **Stop tier — 100% consumed:** `run_agent_turn` returns `AgentResult::budget_exhausted()` before calling the provider. Memory flushed via standard `on_session_end` path; CLI returns to the prompt. Never panic; never discard in-flight tool results.
- **Parent/child shared counter (D-10, D-16):** child `AgentLoop` decrements the same `Arc<AtomicUsize>` via `AgentSubagentRunner` (`subagent_runner.rs:76`), so a chatty subagent pushes the parent's pressure tier forward. `hermes status` reads `remaining()` and derives the tier for the "Subagents + Processes + MCP" section (D-17).

Token-window management (context compaction, summarization) is a separate concern, handled by the existing `ironhermes-core` memory subsystem (Phase 21.4) and out of scope for 21.7.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — here translated to IronHermes' Rust idioms.

### Structured Tool I/O with serde + schemars

Phase 21.7 is Rust-native, so "structured outputs with Pydantic" maps to `#[derive(Serialize, Deserialize, JsonSchema)]` on tool input/output structs. The existing `DelegateTaskTool::schema()` (`delegate_task.rs:393-436`) builds schemas via hand-rolled `serde_json::json!({...})` — the 21.7 convention for the new `timeout_seconds` field is to keep that pattern (minimise churn) but mirror the shape with a typed struct used for deserialisation in `execute`.

```rust
use serde::{Deserialize, Serialize};
use schemars::JsonSchema;

/// Typed view of `delegate_task` args — used for runtime validation in `execute`.
/// Adds `timeout_seconds` per D-08; `task` / `tasks` mutual exclusivity is
/// enforced post-parse (matches the existing WR-01 comment at delegate_task.rs:769).
#[derive(Debug, Deserialize, JsonSchema)]
pub struct DelegateTaskArgs {
    pub task: Option<String>,
    pub tasks: Option<Vec<BatchTask>>,
    pub allowed_tools: Option<Vec<String>>,
    pub toolsets: Option<Vec<String>>,
    #[serde(default)]
    pub detach: bool,
    /// D-08: per-call wall-clock override; falls back to
    /// `config.subagent.default_timeout_seconds` when None.
    pub timeout_seconds: Option<u32>,
}

#[derive(Debug, Deserialize, JsonSchema)]
pub struct BatchTask {
    pub goal: String,
    pub context: Option<String>,
    pub toolsets: Option<Vec<String>>,
}
```

Retry / error-feedback path when the model sends malformed tool_use JSON:

- `serde_json::from_value::<DelegateTaskArgs>(args)` returns `Err(serde_json::Error)`.
- The tool's `execute` converts it to `anyhow::Error` with a structured message: `"Invalid delegate_task args: {field} — expected {type}, got {value}"`.
- The agent loop (`ironhermes-agent::agent_loop`) catches this at the tool-result boundary and feeds the error text back to the model as the next turn's tool_result content — same flow as any tool runtime error today.
- **No separate retry counter.** The iteration budget (D-15) *is* the retry bound: a model that keeps sending malformed JSON exhausts its budget and the 100% stop-tier fires. This is explicitly simpler than a per-tool retry limit (D-13 rejects per-tool retry as premature complexity).
- Under `--yolo` (D-11/D-13), the same loop applies — tool errors are not fatal, agent keeps iterating against the shared budget. Only budget 100%, framework panic, or user interrupt can stop it.

### Async-First Design

IronHermes is already tokio-native. Phase 21.7 reinforces three patterns and avoids one common mistake.

- **Fire-and-forget transcript writes (D-05).** Each subagent turn spawns a tokio task that appends JSONL. The handle is intentionally dropped — errors must log and die, never block the turn:
  ```rust
  let path = transcript_path.clone();
  let line = serde_json::to_string(&turn_event)?;
  tokio::spawn(async move {
      if let Err(e) = tokio::fs::write(&path, line).await {
          tracing::warn!("subagent transcript write failed: {}", e);
      }
  });
  ```
- **Cancel-aware awaits via `tokio::select!`.** The subagent execution wrapper already combines `tokio::time::timeout` with cancellation implicitly through `child_cancel_token.cancelled()` cascading into `AgentLoop`. D-08 explicitly upgrades the timeout path to `.cancel()` the child token on expiry so the inner future is truly cancelled, not merely abandoned (see Pitfall 5). Example shape:
  ```rust
  tokio::select! {
      res = runner.run_child(registry, prompt, max_iter, model, Some(token.clone()), cb) => res,
      _   = tokio::time::sleep(Duration::from_secs(timeout_secs)) => {
          token.cancel();                      // D-08 hard-cancel, not just "time's up"
          Err(anyhow!("Subagent timed out after {}s", timeout_secs))
      }
      _   = token.cancelled() => Err(anyhow!("Subagent cancelled by parent")),
  }
  ```
- **`tokio::sync::watch` for status-line updates.** Writes are cheap (`watch_tx.send_modify(|s| s.active_subagents = n)`); readers `.borrow()` without awaiting. D-04 pill updates must NEVER go through an `RwLock::write().await` in the hot path — use the existing watch channel from `StatusLineState`.

**Common mistake:** a spawned tokio task holds a `JoinHandle` that nobody stores. If the parent cancels, the task **keeps running** — `tokio::spawn` does NOT implicitly cancel on parent drop. For the transcript writer the fire-and-forget semantics are intended. For any task doing substantive work (subagent dispatch, background process watcher), store the `JoinHandle` and `.abort()` it on cancel, or route cancellation through a `CancellationToken` the task checks.

### Safety Message Discipline

Phase 21.7 does not author prompts (that's handled by existing prompt builders). What it *does* emit are **safety-critical operator-facing strings** that go into (a) the model's system message and (b) the CLI UI. These need the same "say it plainly, say it once" discipline prompt engineering demands.

**Budget-pressure advisories (D-15) — model-facing.** Hardcoded for v2.0 (Claude's Discretion per CONTEXT.md). Exact text lives in `ironhermes-agent::agent_loop::pressure_advisory`:

- 70% caution: `"You have used approximately 70% of your iteration budget. Consider consolidating remaining work and moving toward a final answer."`
- 90% warning: `"You have used approximately 90% of your iteration budget. Respond with your final answer now."`
- 100% stop: no injection — the loop terminates before the next provider call.

Inject as a fresh `ChatMessage::system(...)` at the start of each turn once the tier is reached; do NOT rewrite the original system prompt (models handle mid-conversation system messages cleanly and it keeps the intent legible to anyone reading transcripts).

**`--yolo` banner (D-11/D-12) — user-facing.** First line of output in yolo mode, stderr, bold red via `colored`:

```rust
if config.autonomous.yolo {
    eprintln!("{} {}", "--yolo enabled:".bold().red(),
              "all dangerous-command approvals are bypassed. Iteration budget \
               and user interrupt are the only stops.".red());
}
```

Repeat *once per session start*, not per-turn. Mirror the pattern to `hermes status` output (D-18) under the provider/model section so operators running status on a live gateway can tell yolo is active.

### Iteration Budget Management

This is the beating heart of Phase 21.7's safety story. One atomic, two writer sites, three tiers, one terminate.

**Where the counter lives.** `Arc<AtomicUsize>` seeded to `config.agent.max_iterations`. Wrapped as `BudgetHandle` (new thin newtype) exposing:

```rust
pub struct BudgetHandle {
    remaining: Arc<AtomicUsize>,
    max:       usize,
}
impl BudgetHandle {
    pub fn consume(&self) -> Option<usize> {
        // Returns Some(new_remaining) on success; None when already at 0.
        // SeqCst per Pitfall 9.
        let prev = self.remaining.fetch_sub(1, Ordering::SeqCst);
        if prev == 0 { self.remaining.fetch_add(1, Ordering::SeqCst); None }
        else { Some(prev - 1) }
    }
    pub fn remaining(&self) -> usize { self.remaining.load(Ordering::SeqCst) }
    pub fn pressure(&self) -> PressureTier { /* 70/90/100 of max */ }
}
```

**Who decrements.**
- Parent turns: `AgentLoop::run_agent_turn` at the top of each iteration.
- Child turns: also `AgentLoop::run_agent_turn`, via the child loop built by `AgentSubagentRunner::run_child` — the child `AgentLoop` is constructed with `.with_budget(self.budget.clone())` at `subagent_runner.rs:76`, so both sites decrement the SAME `Arc<AtomicUsize>` (D-16 / PROV-10).

**How advisories inject.** `AgentLoop::run_agent_turn` calls `budget.pressure()` at the start of each iteration; if the tier is Caution70 or Warning90 and the previous turn wasn't at the same tier, push `ChatMessage::system(advisory_text)` into the conversation before calling the provider. Do NOT inject on every turn once the tier is reached — once is enough to shift model behaviour, twice is nagging that wastes tokens.

**What "clean terminate" at 100% means.**
- `consume()` returns `None`.
- `AgentLoop::run_agent_turn` returns `Ok(AgentResult::budget_exhausted())` — an `Ok`, not an `Err`.
- Caller (CLI or gateway) renders: `"Iteration budget exhausted ({}/{} iterations). Stopped cleanly."` to the user.
- Active subagents get cancelled via their child tokens (D-07 cascade); background processes are left running (D-24 drains on `on_session_end`, not on budget exhaustion).
- State.db session row is updated to `status='budget_exhausted'` so `hermes status` can surface it historically.
- No panic. No `std::process::exit`. The normal `on_session_end` path runs.

### Resource Budgets (Iteration + Wall-Clock + Concurrency)

Three dimensions, three different mechanisms — none of them $/token (IronHermes is provider-agnostic, cost rollups belong to `ironhermes-providers` telemetry, not the runtime).

**1. Iteration budget (D-10, D-15, D-16, D-17).** Covered above. Reported in `hermes status` under the "Subagents + Processes + MCP" section: `iterations: used/max (tier)`. JSON shape (D-20):

```json
{"agent": {"iterations_used": 23, "iterations_max": 50, "pressure": "caution70"}}
```

**2. Wall-clock subagent timeout (D-08).** Default ~300s from `subagent.default_timeout_seconds`. Per-call override via `timeout_seconds` on the tool schema. Implemented at `delegate_task.rs:265-283,547-579` via `tokio::time::timeout` + explicit `child_cancel_token.cancel()` on expiry (see Pitfall 5). On timeout the subagent returns a structured error string to the parent; the parent model sees "Subagent timed out after Ns" and decides what to do.

**3. Concurrency (D-09).** Hard-bounded by `tokio::sync::Semaphore::new(config.subagent.max_subagents)`, constructed once per run at each registration site (`main.rs:371,631,1288`). Warn-after-2s logging (already present at `delegate_task.rs:487-496` measuring >50ms; 21.7 adds a separate `tracing::warn` at 2s debounced). `/agents` and `hermes status` both report `N/M subagents in use` — `N` = `max_permits - available_permits()`, `M` = `config.subagent.max_subagents` (semaphore does not expose `.capacity()`; carry `M` from config directly).

**Caching / cost.** Explicitly out of scope for 21.7. Any provider-level caching (Anthropic prompt cache, OpenAI cached input) is handled by `ironhermes-providers`. The ProcessRegistry's 200KB rolling buffer (D-25) and 30min finished TTL are the only in-process caches introduced by this phase, and they exist for observability, not cost reduction.

**JSON status schema (D-20) — stable v1.** Document in the phase SUMMARY; sketch:

| G-02 | **Wall-clock subagent timeout (hard-cancel)** | `tokio::time::sleep(timeout_secs)` wins the `tokio::select!` in `DelegateTaskTool` | `child_cancel_token.cancel()` on the subagent's token (NOT merely returning `Err` — see Pitfall 5) + return structured `"Subagent timed out after Ns"` error to parent turn. Parent model sees error as normal tool-result content. | **YES** — survives yolo. | FM-1, FM-2; E-01, E-09; D-08 |
| G-03 | **Concurrency semaphore bound** | `tokio::sync::Semaphore::new(config.subagent.max_subagents)` saturates | New `delegate_task` calls `.await` on permit acquisition; debounced `tracing::warn` after 2s blocked. Bound is hard — cannot be bypassed by config at runtime. | **YES** — survives yolo. | FM-1; E-11; D-09 |
| G-04 | **Ctrl-c cascade cancellation** | `SIGINT` → root `CancellationToken` cancels | All child tokens (per-turn + per-subagent + per-delegate_task handle) propagate `.cancelled()`. Active subagents hard-cancel; in-flight tool calls see the cancel and return. Third SIGINT triggers emergency `process::exit(130)` (Phase 21 RESEARCH §Pitfall 7). | **YES** — survives yolo; user interrupt is always honoured (D-13). | FM-1; E-01; D-07 |
| G-05 | **Watch-pattern rate-limit + sustained-overload auto-disable** | 9th+ match in a `WATCH_WINDOW_SECONDS=10` window | Drop the match; emit `tracing::warn` with PID + dropped-count (backpressure visibility, NOT silent). If sustained at cap for `WATCH_OVERLOAD_KILL_SECONDS=45`, disable watch **for that process only** with a one-shot user-facing message; other processes' watches continue firing. | **YES** — survives yolo. | FM-3; E-04; D-27 |
| G-06 | **`/stop` slash command** | User types `/stop` in CLI or gateway | `ProcessRegistry::kill_all()` for the current session: SIGTERM every running `ProcessSession`, await exit (with short timeout), then SIGKILL stragglers, reap via `waitpid`, mark finished. Non-interactive (no confirmation — matches hermes-agent). | **YES** — survives yolo; it's the autonomous-mode operator kill-switch. | FM-1, FM-3; E-01, E-03; D-26 |
| G-07 | **`on_session_end` cleanup hook** | Session ends cleanly OR via interrupt OR via budget exhaustion | `ProcessRegistry::drain_and_kill()` — iterate all `ProcessSession` for this session, kill + reap. Flush pending transcript writes. Purge transcripts unless session flagged `interrupted`. Async, not Drop (Pitfall 6). | **YES** — survives yolo. | FM-1, FM-3; E-03, E-08; D-05, D-24 |
| G-08 | **Non-TTY stdin-read gate** | Any potential `stdin().read()` path | Gated behind `!config.autonomous.yolo && atty::is(Stream::Stdin)`. Under yolo or gateway mode, the gate evaluates false and the prompt path is skipped (returns the default or structured error). No stdin deadlock possible. | **YES** — this guardrail exists *specifically* for yolo/gateway correctness. | FM-2; E-02; D-11, D-12, Pitfall 10 |
| G-09 | **Fatal-error halt** | Framework panic, provider SDK unrecoverable error, tokio reactor shutdown | Immediate halt of agent loop; active subagents cancelled via G-04 cascade; `on_session_end` runs. | **YES** — survives yolo (D-13). | FM-2; E-02; D-13 |
| G-10 | **`--yolo` banner (one-shot, stderr)** | Session start with `config.autonomous.yolo` OR CLI `--yolo` flag | Print bold-red banner to stderr exactly once per session explaining: approvals bypassed; budget + interrupt + fatal-error are the only stops. Mirrored into `hermes status` Provider section when yolo is active. | N/A — this IS the yolo guardrail marker. | FM-2; E-02, E-10; D-11, D-12 |

**Explicit design note on `--yolo`:** Replit's July-2025 incident is the anchor. Blanket approval bypass is safe **only** when G-01, G-02, G-04, G-05, G-06, G-07, G-08, G-09 are all intact — lose any one and `--yolo` becomes reckless. E-02 exists specifically to prove none of these can be bypassed by the yolo path; the eval-auditor should insist on one scenario per guardrail-×-yolo combination.

### Offline (Flywheel) — Production Signals → Follow-up Action

Offline guardrails are metrics collected from real sessions (dev-tool scale, local CLI — see Section 7 for the "no PagerDuty" context) that drive follow-up action rather than real-time intervention. Sampling strategy is per-row because some metrics only matter for sessions with certain characteristics.

| # | Metric | Sampling Strategy | Action on Degradation |
|---|--------|-------------------|----------------------|
| FW-01 | **Cascade-cancel latency (p50/p95)** — time from root-token `cancel()` to all children reaped | Measured in every session that receives ≥1 SIGINT; stored in `state.db` event log | If p95 regresses past **500ms** between releases (or violates CI gate), file a regression issue against the cascade path; bisect against last green. Regression threshold is hard CI gate from Section 7. |
| FW-02 | **Zombie-PID count per session** — count of children in `<defunct>` state AND children still alive AND owned by the exited session's `task_id`, 5s after session end | Every session, computed by `on_session_end` tail-hook | Any non-zero value is a critical flywheel signal — file an issue immediately. Aggregate weekly; target is **0 per session**, no tolerance. |
| FW-03 | **Watch-pattern overload-disable frequency** — count of watches auto-disabled via 45s-sustained-cap path | Every session with at least one `terminal(background=true)` + watch_patterns | Per-repo frequency >1/day indicates either the pattern is too loose or the rate-limit constants are wrong for that workload. Surface to SUMMARY for the phase; consider making `WATCH_MAX_PER_WINDOW` configurable in a future phase (currently hardcoded at 8 per D-27). |
| FW-04 | **Iteration-budget exhaustion rate** — `budget_exhausted` terminations / total terminations | Every session; logged to state.db | Rate >10% means `max_iterations` default (50) is mis-calibrated for real workloads — tune in a follow-up phase. Rate >30% on `--yolo` sessions specifically → treat as a possible runaway signal and inspect top 5 transcripts. |
| FW-05 | **`hermes status --deep` false-negative rate** — downstream incident occurred within 30min of a `--deep` probe returning `ok` on the affected component | Every `--deep` invocation + cross-reference with incident ledger | Any confirmed false-negative → add a new probe type or tighten the existing probe's pass condition. E-06 is the pre-deployment counterpart; FW-05 catches failure modes the fault-injection harness didn't anticipate. |
| FW-06 | **Three-site drift audit** — CI run of the static-grep gate from E-09 against every merged PR touching `main.rs` | Every PR that touches `crates/ironhermes-cli/src/main.rs` | Gate fails → block merge. Drift that slipped through → add the missing handle + write a follow-up test scenario. |

**Degradation actions are scoped for a dev-tool project**: no on-call paging, no customer SLA. The flywheel is "ship a fix in the next release" + "add a scenario to the reference dataset so the regression can't come back". The CI-gate cutoffs in Section 7 are the real-time proxy for the things FW-01..FW-06 watch in production.

## 7. Production Monitoring

> IronHermes is a local developer-tool CLI — there is no PagerDuty, no SRE rotation, no 24/7 on-call. "Production monitoring" here means (a) CI-regression gates that catch known-bad deltas at merge time, (b) structured `tracing` events users can surface locally, and (c) a smart-sampling strategy for which sessions the project maintainer reviews deeper.

### Tracing Tool

**Primary:** the **`tracing`** crate (already in use per Phase 21.2 GAP-6, instrumented across `ironhermes-agent`, `ironhermes-tools`, `ironhermes-cli`, `ironhermes-exec`). Configured via `tracing_subscriber::EnvFilter` with the existing context-aware init.

- **Output sinks:** stdout / stderr by default (matches current IronHermes convention); file sink under `$HERMES_HOME/logs/hermes.log` with daily rotation (optional via `RUST_LOG_FILE`); structured JSON output via `tracing_subscriber::fmt().json()` when `HERMES_LOG_JSON=1` is set.
- **Event schema:** Phase 21.7 adds exactly these new event targets (all existing `tracing` conventions — no new crates):
  - `target = "ironhermes_agent::budget"` — pressure tier crossings, `consume` returns None (100% stop), decrement site.
  - `target = "ironhermes_exec::process_registry"` — spawn, kill, finish, LRU-prune, TTL-prune, watch-rate-limit drops, watch-overload-disable.
  - `target = "ironhermes_tools::delegate_task"` — subagent spawn, cancel cascade, timeout-hard-cancel, permit-blocked-2s debounced warn.
  - `target = "ironhermes_cli::status"` — `--deep` probe outcomes (component, latency, ok/fail/faulted-by-injection).

**Deferred / optional (post-v2.0):** OpenTelemetry export via `tracing-opentelemetry` + OTLP exporter → a self-hosted Phoenix / Langfuse / Jaeger collector for deeper subagent-turn trace analysis. **NOT** in scope for Phase 21.7. Flagged in SUMMARY so a future phase can light it up without re-instrumenting. Arize Phoenix / Langfuse are Python-side analysis layers in this architecture — not primary tools.

### Key Metrics to Track

Five metrics. Each one maps to a Section 6 guardrail and a Section 5 eval dimension, so regressions in production surface against the same rubric that gates CI.

| # | Metric | Source | Maps To |
|---|--------|--------|---------|
| M-01 | **Subagent-cascade-cancel latency (ms)** — root-token cancel → last child reaped | `tracing` span close on `cascade_cancel_duration_ms` field in `ironhermes_cli::main` | E-01, G-04, FW-01 |
| M-02 | **Zombie-PID count per session** — defunct OR orphan children bearing exited session's `task_id`, 5s post-end | Sampled by the `on_session_end` tail-hook; logged as structured event on `target = "ironhermes_exec::process_registry"` | E-03, G-07, FW-02 |
| M-03 | **Watch-pattern overload-disable rate** — `watch_overload_disabled` events per session per running process | `target = "ironhermes_exec::process_registry"` event count | E-04, G-05, FW-03 |
| M-04 | **Iteration-budget-exhaust rate** — `budget_exhausted` terminations / total session terminations | `target = "ironhermes_agent::budget"` event + state.db `sessions.status` column | E-05, G-01, FW-04 |
| M-05 | **`hermes status --deep` probe failure rate** — `ok=false` probe responses per component per week | `target = "ironhermes_cli::status"` event count | E-06, FW-05 |

### Alert Thresholds (CI-Regression Cutoffs)

No paging — these are **merge-blocking CI thresholds** that act as the real-time proxy for the flywheel metrics. Any PR that crosses a cutoff is blocked until the regression is explained or reverted.

| Metric | CI Cutoff | Enforcement |
|--------|-----------|-------------|
| M-01 (cascade-cancel p95) | **> 500ms fails CI** | `cargo test --test cascade_cancel_bench` runs S-01/S-02/S-04 10 times, computes p95; fails above 500ms. Matches the Section 1b rubric PASS bound. |
| M-02 (zombie-PID count) | **Any non-zero value fails CI** | Every integration test that exercises process-spawn ends with a `sysinfo` + `waitpid` assertion block (boilerplate helper: `assert_no_zombies!(session_id)`). Zero tolerance. |
| M-03 (watch-overload-disable) | **Rate > 0 in a test that shouldn't trigger it fails CI** | S-11 asserts rate limiter does NOT disable at 12 matches in 10s; only S-12 is allowed to trigger disable. |
| M-04 (budget-exhaust determinism) | **S-13 must produce exactly `max_iterations` consumes, no more no less** | Stress test with 10k loops. Any drift = atomic-ordering bug → block merge. |
| M-05 (probe false-negative) | **Each fault-injection scenario (S-14/S-15/S-16) must produce `ok=false`** | Exact-match assertion in `insta` snapshot for both text and JSON output. |

Production thresholds (for operator-side observation, not CI) are intentionally looser — if the project grows to a team context where real incident data accumulates, FW-01..FW-06 become the data source for retuning these cutoffs.

### Smart Sampling Strategy

Not every session is interesting to review. The project maintainer (and any future eval-auditor pass) inspects deeper transcripts on a signal-weighted sample. Priority descending:

1. **Any session that hit `budget_exhausted`** (`M-04`). Read the final 10 turns of the parent transcript + the last active subagent's transcript. Look for loop-y behaviour or tool-error thrash that the iteration budget correctly caught but advisory-pressure failed to prevent.
2. **Any `--yolo` session with >5 tool calls in a single turn OR >20 tool calls across the session** (`G-10` is active; large surface). Read the full transcript. This is the Replit-2025 analogue — if a destructive-adjacent tool ran, make sure the budget + cancel + sandbox path actually held.
3. **Any session that fired watch-overload-disable (`G-05` / `M-03`).** Read the producing background-process output buffer (last 200KB) + the 10 turns before the disable. Check whether the pattern was too loose or the signal genuinely exceeded design capacity.
4. **Any session with a non-zero `M-02` zombie count.** Read the exit path — `on_session_end` ran but didn't reap. High priority; always surfaces a real bug.
5. **Any session where `hermes status --deep` returned `ok` within 30min of a downstream incident (`FW-05`).** Read the probe outputs + incident ledger. Drives new fault-injection scenarios for E-06.
6. **Any session with cascade-cancel p95 > 300ms** (halfway to the CI cutoff; leading indicator). Read the subagent tree + cancellation-audit events. Pre-empts an M-01 regression before it lands.

**How the sample is surfaced:** `hermes status --json` for a running session gives the maintainer the live slice. For historical review, `state.db` session rows filtered by (`status='budget_exhausted' OR yolo_active=1 OR zombie_count>0 OR watch_overload_fired=1`) produce the review queue.

**Cadence:** the maintainer (per Section 1b Domain Expert Roles — Brad Wilson + senior-power-user reviewers) does a weekly pass on the queue during active phases; reduced to monthly once parity is locked in. The eval-auditor runs the same sampling once per phase close against the phase's scenario set to verify no flywheel signal slipped through CI.

**Signal-metric divergence watch:** when user-reported issues ("hermes hung", "my process came back") don't map onto any of M-01..M-05, that's the early-warning signal that the rubric has a gap — file a new E-dimension and a new scenario, repeat the gsd-eval-planner pass.
