<objective>
Close GAP-21.7-02 — the REPL `You: ` prompt paints at a non-deterministic row after Plan 11 moved rustyline onto a `spawn_blocking` worker thread. Each submitted prompt ends up at the top or bottom of the terminal depending on how much stdout/stderr scrolled during the preceding agent turn and which TUI render-loop tick interleaved with the worker's `rl.readline` paint.

**Chosen approach: Hybrid E = (B-narrowed) + (C-hardened).** Neither candidate A/B/C/D alone is correct. Approach A (full ratatui-style coordinator) is a multi-phase refactor and rustyline's `!Send` constraint fights it. Approach B (suppress TUI repaints during readline) is half the answer but by itself creates flicker AND doesn't fix the underlying cursor-positioning race. Approach C (explicit per-prompt cursor positioning) is correct in spirit but was ALREADY landed in Plan 11 via `prepare_prompt_with_reserve` at the MAIN-task `request_prompt` site — and THAT is precisely what introduced the race: the cursor-move ANSI executes on the main task before the command even reaches the worker's `blocking_recv`, giving the TUI ticker (and any in-flight ExternalPrinter flush) a window to shift the cursor before rustyline paints.

The fix combines (1) moving the cursor-positioning ANSI INTO the rustyline worker thread so it is the last terminal write before `rl.readline(&prefix)` — closing the race window at the structural level — plus (2) a narrow TUI readline barrier (`AtomicBool` checked in the render loop's status-row paint) so the 100ms ticker cannot interleave `SavePosition` / `MoveTo(0, bottom)` / `RestorePosition` in the 1-10ms window between the worker's positioning write and its readline cursor query. The scanner + widget rows continue to paint normally (they live ABOVE the prompt row and cannot collide). No flicker on the status bar because it stops repainting only while readline is actively positioned — a window measured in the time it takes the user to type one line, during which the status content has no live reason to change.

**Why not candidate A (full TUI widget coordinator):** Rustyline `DefaultEditor` is `!Send`; hoisting its cursor state into the render task violates the Plan 11 threading invariant. Would require rewriting the input path and would almost certainly introduce new regressions in the mid-turn slash arm that Plan 11 took three revisions to stabilize (commits c6142fd, d709610, cb98ad6, 04ca60f, e5a3dc7 landed sequentially).

**Why not candidate B alone (suppress repaints):** Plan-11's `prepare_prompt_with_reserve` on the main task is still racing the ticker AND any concurrent stderr from the just-completed turn. Suppressing the ticker's status-row write helps but does nothing about the cross-thread positioning race — the main task emits `MoveTo(0, prompt_row)` + `Clear(CurrentLine)`, THEN hands off to the worker, THEN the worker's readline calls DSR `\x1b[6n` to read the cursor and paints based on that response. Between the main-task write and the worker's DSR, any stderr flush can move the cursor. Barrier alone doesn't close that window.

**Why not candidate C alone (cursor save/restore):** Already effectively present via Plan 11's `prepare_prompt_with_reserve`. The deeper issue is WHICH THREAD emits the positioning ANSI. Worker-thread emission is the structural fix; main-thread emission is what broke in Plan 11.

**Why not candidate D (alternate screen):** Nuclear refactor. Loses scrollback (a feature users rely on for auditing tool output post-turn). Wrong scope for a gap-closure plan.

Purpose: Close the final open UAT gap for Phase 21.7 (GAP-21.7-02) so the operator can re-run HUMAN-UAT test 1 on a live TTY and confirm the prompt anchors deterministically. No new features. No scope expansion beyond anchoring the prompt.

```rust
/// Compute the ANSI byte sequence that positions the cursor to the fixed
/// prompt row for a given reserved-row count + terminal row count. Pure
/// function: same inputs -> same bytes. Testable without a live TTY.
///
/// Returns None when positioning is not meaningful (rows < 4, reserved >= rows).
pub fn prompt_position_ansi(rows: u16, reserved: u16) -> Option<Vec<u8>> {
    if rows < 4 || reserved >= rows { return None; }
    let prompt_row = rows.saturating_sub(reserved);
    // ESC [ row+1 ; 1 H  (CUP) then ESC [ 2 K  (erase entire line — safer than CurrentLine
    // because the worker doesn't know what's on the line yet).
    let mut out = Vec::with_capacity(16);
    // crossterm uses 1-based rows; prompt_row here is 0-based like MoveTo.
    // Emit raw bytes to avoid pulling crossterm's queue machinery into a
    // function that the worker thread calls with its own &mut stderr.
    use std::io::Write as _;
    let _ = write!(&mut out, "\x1b[{};1H\x1b[2K", prompt_row + 1);
    Some(out)
}
```
