# Phase 21.7 Plan 12: REPL Prompt Anchoring Summary

Close GAP-21.7-02 — the REPL `You: ` prompt painting at a non-deterministic row after Plan 11 moved rustyline onto a dedicated worker thread. Root cause: the cursor-positioning ANSI (`prepare_prompt_with_reserve`) executed on the main task before the command reached the worker's `blocking_recv`, leaving a 1–10ms window in which the TUI ticker's `SavePosition`/`MoveTo(bottom)`/`RestorePosition` sequence and any stray stderr from the just-completed turn could shift the cursor. By the time rustyline's DSR query read the cursor position, it could be anywhere — and that's where the prompt painted.

Fix is a hybrid: (1) move the cursor-positioning ANSI INTO the rustyline worker thread so it is the last write before `rl.readline(&prefix)` — closing the race at the structural level — plus (2) a TUI readline barrier (`Arc<AtomicBool>`) that suppresses the 100ms render-loop status-row write during the inter-turn prompt window.

## What Shipped

### Task 12-01 — Worker-thread prompt positioning + `PromptRequest.reserved_rows` (commit `03ecd0f`)

`PromptRequest` gained `reserved_rows: Option<u16>`. When `Some(N)` AND `in_turn == false`, the rustyline worker emits absolute-positioning ANSI (`\x1b[{row};1H\x1b[2K`) IMMEDIATELY before `rl.readline(&prefix)` on the SAME thread as the readline paint. `None` OR `in_turn == true` skips positioning, preserving the invisible mid-turn prompt behavior.

New pure helper `tui::render::prompt_position_ansi(rows: u16, reserved: u16) -> Option<Vec<u8>>`:

- Pure function: same `(rows, reserved)` → identical bytes. Deterministic; no scroll state, no TTY dependency; testable without a live terminal.
- Returns `None` on invalid geometry (`rows < 4` or `reserved >= rows`) so the worker skips the write rather than emitting broken ANSI.
- Writes CUP + erase-line via `write!(&mut Vec<u8>, ...)`; the worker then `stderr.write_all(&bytes) + flush()` directly (no crossterm `queue!` buffering that might reorder with `rl.readline`).

All 7 `PromptRequest` call sites in `main.rs` updated:

- 1 prompt-time: `reserved_rows: Some(tui.reserved_row_count())`.
- 6 mid-turn: `reserved_rows: None` (initial prime + 5 re-prime sites: CancelTurn, ShowPromptHint, ReplLine::Error, empty input, post-dispatch).

5 new unit tests:
- `prompt_request_carries_reserved_rows`
- `prompt_request_none_reserved_rows_default_preserved`
- `external_printer_still_send_sync_clone`
- `prompt_position_ansi_is_deterministic`
- `prompt_position_ansi_refuses_invalid_geometry`

### Task 12-02 — TUI readline barrier + main-task call-site swap (commit `2dd0b60`)

`TuiHandle` gained `readline_active: Arc<AtomicBool>` field (initial `false`) + public `readline_active_handle()` accessor that returns a clone of the `Arc`.

Render-loop changes:
- `render_loop` signature gains `readline_active: Arc<AtomicBool>` parameter.
- Each frame snapshots the barrier via one `load(Ordering::Relaxed)` and passes the `bool` into `redraw_with_extensions`.
- `redraw_with_extensions` gates the `MoveTo(0, bottom) + Clear(CurrentLine) + Print(status_str)` triplet on `!readline_active`.
- `Show + RestorePosition` still fire unconditionally after the gate — cursor hygiene is independent of readline state (the scanner paint above would otherwise leave the cursor in the reserved area).

- REMOVED `finish_prompt_with_reserve(tui.reserved_row_count())` at the old L1076.
- WRAPPED `request_prompt` + `recv_line` in `readline_barrier.store(true/false, Relaxed)` toggles. Channel-error path explicitly clears the barrier before breaking.
- Dropped the now-unused imports `prepare_prompt_with_reserve, finish_prompt_with_reserve`.

Mid-turn `PromptRequest` sites left alone — they carry `reserved_rows: None` (from Task 12-01) and do NOT toggle the barrier (mid-turn is not an inter-turn gap).

## Final Shape — Worker Loop

```rust
// crates/ironhermes-cli/src/repl_input.rs
while let Some(cmd) = cmd_rx.blocking_recv() {
    match cmd {
        Command::Prompt(req, reply) => {
            // Plan 21.7-12 worker-thread positioning (same thread as rl.readline).
            if let Some(reserved) = req.reserved_rows
                && !req.in_turn
            {
                let mut err = std::io::stderr();
                if err.is_tty()
                    && let Ok((_cols, rows)) = crossterm::terminal::size()
                    && let Some(bytes) = crate::tui::prompt_position_ansi(rows, reserved)
                {
                    let _ = err.write_all(&bytes);
                    let _ = err.flush();
                }
            }
            let outcome = match rl.readline(&req.prefix) { /* ... */ };
            let _ = reply.send(outcome);
        }
        /* AddHistory, Shutdown ... */
    }
}
```

## GAP-21.7-02 Resolution

**Closed at the code level.** The structural race is eliminated by the combination of:
1. **Worker-thread emission** of positioning ANSI on the same thread as `rl.readline` — no cross-thread gap where stray stderr can shift the cursor.
2. **Render-loop barrier** suppressing the 100ms ticker's `SavePosition`/`MoveTo(bottom)`/`RestorePosition` during the inter-turn prompt window.
3. **Deterministic helper** (`prompt_position_ansi`) so the positioning is a pure function of `(rows, reserved_row_count())` and independent of the terminal's current scroll state.

INV-21.7-14 + INV-21.7-15 static-grep gates lock the structural fix against silent regression. `repl_prompt_anchoring.rs` (4 tests) locks the deterministic properties into a TTY-independent regression suite. Live end-to-end operator verification (observing the `You: ` prompt anchoring across 5+ consecutive turns) requires a real terminal and is routed to HUMAN-UAT re-run.

## Deviations from Plan

### Rule 3 (blocking issue) — `PromptRequest` field addition required updating ALL call sites together

The plan specified Task 12-01 adds the field and Task 12-02 updates the call sites. However, `reserved_rows: Option<u16>` is a required struct-literal field — omitting it from any of the 7 `PromptRequest` literals is a hard compile error. Attempting to split the change across two commits would have left an intermediate non-building state.

**Action (Rule 3):** All 7 `PromptRequest` call sites in `main.rs` updated in the Task 12-01 commit (1 prompt-time site receives `Some(tui.reserved_row_count())`; 6 mid-turn sites receive `None`). Task 12-02 then handles the orthogonal work (prepare/finish removal + barrier wiring) against an already-compiling baseline. Net logical content per task matches the plan; commit boundary shifted by one file edit.

### Rule 2 (missing critical functionality) — channel-error path clears barrier

The plan example showed the `if let Err(e) = repl_input.request_prompt(…)` branch calling `eprintln!` and `break`. Without also clearing `readline_barrier` on that path, a `request_prompt` failure would leave the barrier stuck in `true`, paralyzing the status bar for the rest of the process lifetime (T-21.7-12-02 DoS).

**Action (Rule 2):** Added `readline_barrier.store(false, Relaxed)` immediately before the `eprintln!` on the error branch. Documented in the Task 12-02 commit message. Confirmed via INV-21.7-15 (c) which now passes because both toggle directions appear in `main.rs`.

### Plan wording — `prepare_prompt_with_reserve` / `finish_prompt_with_reserve` functions retained

The plan said main.rs call sites are REMOVED; the plan did not explicitly require removing the helper functions themselves from `render.rs`. They are still exported from `tui/mod.rs` for backward compatibility (three `#[allow(dead_code)]`-equivalent warnings now emit in cargo build but are non-blocking). This avoids any potential external-consumer break (integration tests or future plans may still reference them). The plan's verification command `grep -c "prepare_prompt_with_reserve(" crates/ironhermes-cli/src/main.rs` returns 0 as required.

### No architectural deviations (Rule 4 — not triggered)

No changes beyond the plan's approved Hybrid-E scope. Rustyline stays on `std::thread::spawn`. No alternate-screen mode. No ratatui. No new dependencies.
