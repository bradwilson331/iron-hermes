### GAP-21.7-02: REPL prompt floats — not anchored to a fixed line

source: HUMAN-UAT test 1 (third re-run, during GAP-21.7-01 verification)
severity: blocking (user-facing visual chaos; hard to use)
status: open
observed: |
  User reports: "the user's command line is floating and should be locked
  and clear. Each time I hit enter it was at the top of the terminal and
  then the bottom." After submitting a slash command or a chat prompt and
  the agent turn completes, the next prompt (`You: `) paints at a
  non-deterministic row — sometimes at the top of the terminal, sometimes
  near the bottom, depending on how much stdout/stderr output scrolled
  during the turn and how the TUI status-line render loop repainted.

  Observed behavior sequence during UAT:
  1. User submits "I'll help you launch agents to research LoRA..."
  2. Agent streams subagent ticker output via ExternalPrinter + TUI status
     line updates via watch channel send_modify.
  3. Turn completes; run_chat's 'turn select loop exits and control
     returns to the outer while-loop at the `request_prompt(in_turn:
     false)` site.
  4. prepare_prompt_with_reserve(tui.reserved_row_count()) fires; rustyline
     worker receives PromptRequest and calls rl.readline("You: ").
  5. The prompt lands at wherever the terminal's current cursor position
     was left by the preceding output — NOT at a stable reserved row
     above the status line.
  6. User types, presses Enter. Next cycle: prompt may paint at a
     DIFFERENT row because terminal cursor state is again non-deterministic.

  This is a regression introduced when Plan 21.7-11 moved rustyline onto
  a spawn_blocking worker thread. Pre-plan-11, readline ran inline on the
  main task so prepare/finish_prompt_with_reserve executed on the same
  thread that issued the rustyline paint — the reserved-row math lined
  up. Post-plan-11, prepare_prompt_with_reserve runs on the main task
  (per plan-checker advisory) but rustyline paints from the worker thread
  where the terminal state is decoupled from that reserve math.

why_human_triggered: |
  TTY interaction is the only way to reproduce — scroll region (DECSTBM)
  interactions with rustyline's cursor positioning + the TUI status line's
  render-loop repaints are terminal-driver dependent and not observable
  in unit tests.
investigation_hints:
  - The DECSTBM scroll region is set ONCE at TUI startup
    (crates/ironhermes-cli/src/tui/render.rs:179-182). After that, the
    render loop repaints the status line at rows after scroll_end.
  - rustyline in the worker paints the prompt via crossterm/termion raw
    cursor moves — these bypass the scroll region constraint. If the
    cursor was left anywhere in the scroll region by the prior output,
    that's where the prompt lands.
  - prepare_prompt_with_reserve writes ANSI codes to position cursor
    before the prompt paint (tui/render.rs:343+). Since plan-11 these
    codes execute on the main task before the PromptRequest is sent to
    the worker — there's a race window where the worker's paint can land
    AFTER further scroll output arrives.
  - Candidate approaches (not conclusions — planner should pick):
    - A: Route rustyline's prompt render through the TUI's render loop
         (ratatui-style widget) so scroll region + prompt + status line
         share one paint coordinator. Largest scope.
    - B: Suppress TUI status-line repaints during active readline, restore
         after. Creates flicker but simpler.
    - C: On each prompt cycle, explicitly position the cursor to a fixed
         row (reserved row count + 1 from bottom) via cursor-save /
         cursor-restore ANSI, then have the worker paint at the current
         cursor position. Fragile.
    - D: Use crossterm's `terminal::SetSize` + alternate-screen mode for
         the TUI and keep the scrollback buffer for the agent output.
         Cleanest long-term but requires major refactor.
  - The plan should quantify: does the floating-prompt issue ALSO affect
    the mid-turn slash dispatch path, or only the inter-turn prompt? If
    only inter-turn, scope can be narrower.
  - Check whether the TUI status-line render loop has a frame-time
    invariant (e.g., repaints every N ms regardless of event) or is
    purely event-driven. A purely event-driven render loop may reduce
    collision surface.
