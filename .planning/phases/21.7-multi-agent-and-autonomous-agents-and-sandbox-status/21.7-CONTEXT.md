# Phase 21.7: Multi-agent and autonomous agents and sandbox status - Context

**Gathered:** 2026-04-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Close four v2.0 parity gaps against hermes-agent, scoped tight ("minimum viable parity"):

1. **Subagent polish / visibility** — surface the existing `delegate_task` system (already wired in CLI + gateway via `AgentSubagentRunner`) with a `/agents` slash command, a status-line pill, persistent transcripts, gateway progress echo, and missing lifecycle guards (cascade cancellation, wall-clock timeout, concurrency surface, parent/child budget inheritance).
2. **Autonomous mode (`--yolo`)** — non-interactive mode that bypasses all dangerous-command approvals, runs under a proper iteration budget (ports PROV-09 and PROV-10), continues past tool errors, and is invocable via CLI flag or config.
3. **`hermes status` CLI** — `hermes status [--all] [--deep] [--json]` component diagnostics covering provider/model, memory/sessions, gateway/platforms, and subagents/processes/MCP.
4. **Background process registry / sandbox status** — in-memory, session-scoped registry tracking processes spawned via `terminal(background=true)` / `execute_code`, with spawn/poll/wait/kill, 200KB rolling output buffer, `/stop` slash, watch patterns with rate-limited notifications, and surface in `hermes status`.

**Explicit exclusions (deferred to own phases):** worktree-parallel mode (`hermes -w`), toolsets grouping + per-platform tool enable/disable (TOOL-02), terminal backends (Docker/SSH/Modal/Singularity), full gateway session mirror (cross-session mirror.py port), plugin discovery sources (User/Project/Pip).

Trigger: v2.0 parity closeout.

</domain>

<decisions>
## Implementation Decisions

### Scope boundary

- **D-01:** Minimum viable parity cut — only the four anchors listed in Phase Boundary. Adjacent parity items (toolsets, terminal backends, gateway mirror, plugin discovery) are out of scope and become their own phases.
- **D-02:** `delegate_task` already exists (registered in `crates/ironhermes-cli/src/main.rs:370,683,1287` and in gateway) — 21.7 does NOT re-implement subagent dispatch. It polishes surfacing, lifecycle, and introspection around the existing runner.

### Subagent surface (multi-agent)

- **D-03:** `/agents` slash command registered in the unified `CommandRouter` (Phase 21.1) — available in both CLI and gateway. Subcommands: `/agents` (list active: id, task summary, duration, parent), `/agents kill <id>` (cancel one), `/agents logs <id>` (dump transcript).
- **D-04:** Status-line pill showing active-subagent count (e.g., `agents: 2/4`) when any subagent is running; hides at 0. Wires into `crates/ironhermes-cli/src/tui/pills.rs` via the existing `StatusLineState` watch channel.
- **D-05:** Persistent subagent transcripts written to `$HERMES_HOME/subagent-transcripts/<session_id>/<subagent_id>.jsonl` (one JSON line per turn: tool_call, tool_result, stream_delta, done). Cleanup policy: purge on `on_session_end` unless the session was "interrupted" (parity with session storage).
- **D-06:** Gateway subagent-progress echo — reuse existing `SubagentProgressCallback` (`crates/ironhermes-cli/src/main.rs:654`) pattern to route subagent tool progress into the gateway's outbound stream for the parent session. NOT a full `gateway/mirror.py` port (that's deferred); this only pipes subagent events into the parent's existing stream, no cross-session echo.
- **D-07:** Parent ctrl-c cancellation cascades to active subagents via the existing `child_token()` chain (Phase 21 Plan 03 RESEARCH §Pitfall 2). 21.7 adds an end-to-end regression test proving double-ctrl-c in the parent kills all active subagents cleanly before exit.
- **D-08:** Wall-clock subagent timeout — per-call `timeout_seconds` parameter on `delegate_task` tool schema, plus `subagent.default_timeout_seconds` config key (default Claude's Discretion, likely 300s). On timeout, hard-cancel via child token; subagent returns structured error to parent.
- **D-09:** Concurrency surface — `/agents` and `hermes status` both report current semaphore state (`N/M subagents in use`). Emit a single `tracing::warn` event (debounced) when `delegate_task` blocks waiting for a slot longer than 2s.
- **D-10:** Parent/child budget inheritance (PROV-10) — parent's iteration budget passed into `AgentSubagentRunner::spawn`; subagent consumes from the same counter. Flows through Area 3's budget machinery (D-15..D-17).

### Autonomous mode (`--yolo`)

- **D-11:** Blanket approval bypass — when yolo is active, every dangerous-command approval gate is skipped (sudo, destructive git, etc.). Matches hermes-agent `--yolo` exactly. No classification policy; no allow/deny regex list.
- **D-12:** Surface — both CLI flag (`hermes --yolo ...`, `hermes chat --yolo`) AND config key (`autonomous.yolo: true`). CLI flag wins if both set. Gateway mode reads config only (no per-message --yolo).
- **D-13:** Non-interactive error policy — tool failures feed back to the model as error text and the agent keeps iterating. Hard-stops only on (a) budget exhaustion, (b) fatal framework errors (panic, broken provider), (c) user interrupt. No per-tool retry counter (kept simple).
- **D-14:** `--yolo` is additive to existing dangerous-command prompts; when NOT in yolo, the existing approval flow (currently stubbed — parity gap item 6) remains untouched. 21.7 does NOT implement the full approval queue — only the bypass. Approval queue is deferred.

### Iteration budget (PROV-09 + PROV-10)

- **D-15:** Full 2-tier pressure model ported: caution at 70% (inject "consider consolidating" advisory into system prompt), warning at 90% (inject "respond now" directive), stop at 100% (terminate turn cleanly, flush memory, return to prompt). Defaults to Claude's Discretion (probably 50 iterations) but overridable via `agent.max_iterations` config.
- **D-16:** Parent/child shared budget (PROV-10) — subagent consumes from parent's counter, not its own. `AgentSubagentRunner::spawn` accepts a `BudgetHandle` so child turns decrement the same atomic.
- **D-17:** Budget state is part of status — reported under `subagents + processes + MCP` section of `hermes status` (current / max / pressure tier).

### `hermes status` CLI

- **D-18:** Four component sections reported: (a) Provider + model (active provider, resolved model, API mode, fallback chain), (b) Memory + sessions (memory provider, MEMORY.md/USER.md capacity, session count, state.db path + size), (c) Gateway + platforms (Telegram auth, bot token presence, gateway PID if running, enabled platforms, allowlist counts), (d) Subagents + processes + MCP (active subagent count + slot usage, background process count, MCP servers connected + tool counts).
- **D-19:** Default output: styled text via `colored` crate, matching existing patterns in `crates/ironhermes-cli/src/cron.rs` and `crates/ironhermes-cli/src/memory_cmd.rs`.
- **D-20:** `--json` flag emits machine-readable JSON (stable schema — document in phase SUMMARY). Used by status-bar integrations and CI health probes.
- **D-21:** `--deep` flag runs live probes: ping provider (cheap HEAD/OPTIONS), verify FTS5 integrity on state.db (PRAGMA integrity_check), test each MCP server handshake (rmcp tools/list with short timeout). Skipped in default mode — only on `--deep`.
- **D-22:** Command registered in `CommandRouter` as `status` + bound as `hermes status` CLI subcommand. Not a slash command (lives as `hermes` subcommand, not `/status`).

### Background process registry / sandbox status

- **D-23:** In-memory `Arc<RwLock<ProcessRegistry>>` living in `crates/ironhermes-exec/src/process_registry.rs` (new module). NOT persisted to disk — session-scoped only. No JSON checkpoint file, no SQLite table (simpler than hermes parity; can upgrade later if crash recovery becomes a requirement).
- **D-24:** Cleanup lifecycle — `on_session_end` drains and kills tracked processes belonging to that session. Session-scoping keyed by `task_id` (following hermes' `ProcessSession.task_id` convention).
- **D-25:** Core operations: `spawn(env, cmd, task_id)`, `poll(session_id)`, `wait(session_id, timeout)`, `kill(session_id)`, `logs(session_id, tail_bytes)`. 200KB rolling output buffer per process (matching hermes `MAX_OUTPUT_CHARS`). LRU-prune to 64 concurrent tracked processes (match `MAX_PROCESSES`). Finished processes kept 30min (`FINISHED_TTL_SECONDS`).
- **D-26:** `/stop` slash command (already stubbed in the command router) kills all running background processes in the current session. Non-interactive. Matches hermes' non-interactive kill-all semantics.
- **D-27:** Watch patterns + rate-limited notifications — `terminal(background=true)` accepts `watch_patterns: [regex]`; matches trigger a new agent turn via the hooks/agent-loop interface. Rate-limited at 8 notifications per 10s window (matching `WATCH_MAX_PER_WINDOW` / `WATCH_WINDOW_SECONDS`). Sustained overload (45s+ at limit) disables watch for that process (`WATCH_OVERLOAD_KILL_SECONDS`).
- **D-28:** Registry state surfaces in `hermes status` under the "Processes" subsection (count, per-process PID, uptime, task_id, exit_code if finished). Uses `format_uptime_short` helper (port from hermes `process_registry.py:56`).
- **D-29:** Terminal tool and execute_code tool both use the registry when `background=true`. Foreground execution path is unchanged.

### Claude's Discretion

- Exact defaults: `subagent.default_timeout_seconds` (likely 300), `agent.max_iterations` (likely 50), pressure thresholds fine-tuning (70/90/100 are the tiers — values are exact, just verify they feel right in live UAT).
- Status output layout details (column widths, color assignments) — follow `colored` conventions from cron.rs / memory_cmd.rs.
- JSON schema field names for `--json` — researcher/planner pick; document in SUMMARY.
- Exact transcript JSONL schema (which fields per line) — researcher/planner pick; must round-trip losslessly so /agents logs can replay cleanly.
- Whether the 2-tier pressure advisory text is hardcoded or a PromptBuilder slot — Claude's Discretion; hardcoded is fine for v2.0.

</decisions>
