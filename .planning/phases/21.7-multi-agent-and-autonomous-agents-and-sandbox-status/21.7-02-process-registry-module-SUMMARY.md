# Phase 21.7 Plan 02: ProcessRegistry Module Summary

End-to-end session-scoped in-memory background-process registry (D-23..D-29): types, hermes-agent parity constants (byte-for-byte match of `process_registry.py:57-64`), core ops (spawn/poll/wait/kill/logs/drain_and_kill), LRU-over-64 + 30min TTL pruning, 200KB UTF-8-safe rolling buffer helper, broadcast channel for watch events, and watch-pattern rate limiter with 8/10s cap + 45s sustained-overload auto-disable. Plan 06 will wire stdout-drain tasks that invoke `append_bounded_utf8` and route matched lines through `rate_limit_check` before `broadcast_watch_event`. No `Drop` impl anywhere (Pitfall 6 / G-07). 37 tests green (22 lib unit + 10 integration ops + 5 rate-limit).

### Task 2-03 — Rate limiter + sustained-overload auto-disable (commit 49abc2c)

`rate_limit_check(&mut self, id: &str) -> RateDecision` with full semantics:
- Window rolls at `WATCH_WINDOW_SECONDS = 10` based on `tokio::time::Instant::now()`.
- First `WATCH_MAX_PER_WINDOW = 8` calls in a window return `Fire`.
- 9th+ in the same window return `Drop` and emit `tracing::warn!(target: "ironhermes_exec::process_registry", pid, dropped_count, ...)` with message `"watch pattern backpressure: dropped N match(es) in last 10s"` (Pitfall 7 gate).
- `overload_since` latched on first Drop and **persists across window rolls** as long as the cap continues to be exceeded. When 45s of sustained cap elapses, `watch_state.disabled = true`, tracing::warn `"watch overload disabled after 45s sustained at cap"` emitted once, `AutoDisable` returned exactly once.
- Once disabled, every subsequent call returns `Drop` silently (no additional warns).
- Per-process scope — peers retain fresh rate-limit state (S-12).

5 integration tests (`tests/process_registry_watch_rate_limit.rs`):
| Test | Assertion |
|------|-----------|
| `twelve_matches_in_10s_fires_eight_drops_four_with_warn` | S-11 — exactly 8 Fire + 4 Drop; `logs_contain("watch pattern backpressure")` |
| `window_rolls_after_10s_and_allows_new_matches` | After burning window + 1s past boundary, next call is Fire |
| `sustained_overload_45s_disables_watch_for_that_process_only` | S-12 — AutoDisable triggered; peer (`proc_cool`) still Fires; `logs_contain("watch overload disabled")` |
| `already_disabled_returns_drop_every_time` | After reaching disabled, 5 straight calls all Drop |
| `unknown_id_returns_drop_silently` | Unknown IDs return Drop (no panic) |

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] `overload_since` reset-on-window-roll made 45s detection impossible**
- **Found during:** Task 2-03 first test run — `sustained_overload_45s_disables_watch_for_that_process_only` hit 46+ drop events but AutoDisable never fired.
- **Issue:** Plan sample code unconditionally reset `s.watch_state.overload_since = None` when the 10s window rolled. The test advances 1s per iteration and drops 1 event per iteration, so every 10 seconds the overload anchor was wiped and the 45s accumulating counter never accumulated.
- **Fix:** `overload_since` now only clears when a full window passes **without** exceeding the cap (i.e., `window_hits <= WATCH_MAX_PER_WINDOW` at roll time). Under sustained cap, it persists → the 45s threshold actually triggers.
- **Files modified:** `crates/ironhermes-exec/src/process_registry.rs`
- **Commit:** 49abc2c

**2. [Rule 3 - Blocking] `std::time::Instant` is not affected by `tokio::time::pause()`**
- **Found during:** Task 2-03 first test run — all paused-time rate-limit tests failed because `std::time::Instant::now()` returns real wall-clock time, unaffected by `tokio::time::pause()/advance()`.
- **Fix:** Migrated `WatchState.window_start` and `WatchState.overload_since` from `std::time::Instant` to `tokio::time::Instant`. `rate_limit_check` calls `tokio::time::Instant::now()`. `ProcessSession.started_at` / `finished_at` remain `std::time::Instant` (they're uptime display, not under test).
- **Files modified:** `crates/ironhermes-exec/src/process_registry.rs`
- **Commit:** 49abc2c

**3. [Rule 3 - Blocking] `tracing-test` default filter scopes captured events to the test's own crate**
- **Found during:** Task 2-03 — `logs_contain("watch pattern backpressure")` always false because the `warn!` call lives in `ironhermes_exec::process_registry` while the test binary is `process_registry_watch_rate_limit`. `tracing-test 0.2.6`'s `#[traced_test]` macro sets `EnvFilter` to `"{crate_name}=trace"` where `crate_name` is the test binary's own crate.
- **Fix:** Declared `tracing-test = { workspace = true, features = ["no-env-filter"] }` in `ironhermes-exec/Cargo.toml` `[dev-dependencies]`. The `no-env-filter` feature sets filter to `"trace"` (everything) so cross-crate warns reach the mock writer and `logs_contain` sees them.
- **Files modified:** `crates/ironhermes-exec/Cargo.toml`
- **Commit:** 49abc2c

**4. [Rule 2 - Missing] Integration tests can't construct `ProcessSession` due to pub(crate) fields**
- **Found during:** Task 2-02 first test build — `tests/process_registry_ops.rs` at the `finished_ttl_prunes_30min_old_entries` test hit `E0451` on `child` and `cancel` being private.
- **Fix:** Added `#[doc(hidden)] pub fn fake_process_session(...)` builder + `#[doc(hidden)] pub fn insert_fake_finished(ProcessSession)` / `insert_fake_running(ProcessSession)` / `insert_fake_running_minimal(&str, u32)` methods. These give integration tests a blessed route to pre-populate registry state without a real child, while keeping the runtime surface (`child`, `cancel` fields) `pub(crate)`.
- **Files modified:** `crates/ironhermes-exec/src/process_registry.rs`
- **Commit:** 4bc60b6 + 49abc2c (insert_fake_running_minimal added in 49abc2c)

## Known Stubs

- `append_bounded_utf8` emits a `dead_code` warning at build time (intentional — Plan 06 stdout-drain tasks consume it). No runtime call site lives in this plan.
- `stdout` / `stderr` are `take()`'n from `Child` at spawn time and immediately dropped. Plan 06 will redirect them into drain tasks that feed `append_bounded_utf8` and route matched lines through `rate_limit_check` before `broadcast_watch_event`. The current spawn path still sets `stdout(Stdio::piped())` / `stderr(Stdio::piped())` so the interface is Plan-06-ready and changing it is a zero-effort wiring task.

## Threat Flags

None new. The plan's six mitigate-disposition threats are all mitigated as specified:

- **T-21.7-02-01** (unbounded output buffer DoS) — `MAX_OUTPUT_CHARS = 200_000` + char-boundary truncation via `append_bounded_utf8`. Unit test `append_bounded_utf8_exceeds_cap_rolls_window` proves the 20-byte cap holds after 50 bytes written.
- **T-21.7-02-02** (notification-storm DoS) — 8/10s cap + 45s sustained auto-disable. 5 dedicated rate-limit tests lock the semantics.
- **T-21.7-02-03** (cross-session info disclosure) — `drain_and_kill_session(task_id)` task-id-scoped; LRU drops oldest-running; 30min TTL on finished; no disk persistence.
- **T-21.7-02-04** (zombie reaping) — `kill()` sequences `Child::kill()` + `child.wait()` for each child. Integration test `spawn_then_drain_kills_all_children` drives sleep processes through the drain path and confirms `running_count == 0`. Full `waitpid(WNOHANG)` zombie check deferred to Wave 4 S-01 harness as plan notes.
- **T-21.7-02-05** (regex ReDoS) — `regex` crate (linear-time, no backreferences) is used; invalid patterns rejected at spawn time. `spawn_with_bad_watch_pattern_errors` integration test locks the up-front validation.
- **T-21.7-02-06** (async drop race) — No `Drop` impl on `ProcessRegistry` or `ProcessSession`. Grep-gate locked in Task 2-02 acceptance criteria.
