## Decisions Made

1. **Subprocess binary (not assert_cmd) for CLI tests:** `skills_cmd_integration.rs` already used `CARGO_BIN_EXE_ironhermes` subprocess pattern from Plans 01-03. Consistent with that approach — no `assert_cmd` dep added, no cmd_* functions exposed as pub seams.

2. **AlwaysBlockedScanner from scanner.rs:** The plan mentioned defining it test-locally if absent. It was already defined as `pub struct AlwaysBlockedScanner` at `scanner.rs:97-118` with `new(reason: &str)` constructor and proper `[BLOCKED: reason]` verdict. Used directly via `ironhermes_hub::AlwaysBlockedScanner`.

3. **update() API takes skill_name, not identifier:** The hub `update()` function reads the identifier from the lock file using the skill name. Tests install first, then call `update("my-skill", ...)`. The identifier in the lock is automatically used by the update pipeline.

4. **root check via `id -u` subprocess:** No `libc` crate in workspace. Used `std::process::Command::new("id").arg("-u")` to check UID. Works on all unix platforms.

5. **WARN-BUT-LOAD aliasing-risk:** Asserted via `outcome.scan_verdict.contains("blocked")`. `InstallOutcome.scan_verdict` is a `String` containing the `ScanVerdict::summary()` output — exactly `"blocked: SKILL.md"` when AlwaysBlockedScanner is used.
