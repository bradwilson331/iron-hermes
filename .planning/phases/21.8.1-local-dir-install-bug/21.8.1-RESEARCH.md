## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-A1:** Explicit `local:<path>` prefix only. Dispatcher gains a fourth `else if identifier.starts_with("local:")` arm BEFORE the `contains('/')` GitHub arm.
- **D-A2:** `local:` accepts absolute, relative, and `~/` forms. Tilde via `dirs::home_dir()`; relative against CWD; final path canonicalized via `std::fs::canonicalize()` before any further I/O. Hard-fail if canonicalize fails.
- **D-B1:** Install mode is **copy** (full recursive directory copy into `~/.ironhermes/skills/<category>/<name>/`). Reuses 21.8 install pipeline. Source-dir edits do NOT auto-propagate.
- **D-B2:** Trust tier = `SkillSource::Trusted`. `CoreSkillScanner` runs with Phase 19 D-15 **WARN-BUT-LOAD** posture. Pre-install audit (`add-skill.vercel.sh/audit`) is **SKIPPED**.
- **D-C1:** New `source: "local-dir"` value in `SkillLockEntry.source`. `identifier` = canonicalized absolute path. `repoPath` = relative SKILL.md path. `snapshotHash` = `""`. `computedHash` = D-13 algorithm over installed folder.
- **D-C2:** `hermes skills update <name>` for local-installed skill re-runs full pipeline against source path in `identifier`. Source dir missing → hard-fail with typed error. Full recursive re-copy (every subfile and subdir). Bulk update iterates ALL skills including local-dir; failures reported but do not abort.
- **D-D1:** Pre-dispatch hint before `contains('/')` arm: if identifier has no known prefix AND starts with `/`, `./`, `../`, `~/` OR `fs::metadata(identifier).map(|m| m.is_dir()).unwrap_or(false)` is true → emit hint and exit 1.

## Summary

Phase 21.8.1 is a focused extension of the Phase 21.8 install pipeline. The entire install machinery (quarantine, scan, atomic rename, lock write, `compute_folder_hash`, `write_bundle_to_dir`, `sanitize_subpath`) already exists and is source-agnostic from quarantine onward. The work is: (1) add a `LocalDirSource` adapter that walks a directory and produces a `SkillBundle`, (2) wire a new `local:` dispatch arm in `cmd_install`, (3) add a pre-dispatch hint for path-shaped identifiers, (4) add `"local-dir"` handling in `cmd_update` and `recompute_trust_str`, (5) add a `HubErrorKind::LocalSourceMissing` variant, and (6) write tests.

The highest-leverage architectural finding: the `walk()` function inside `lock.rs` (used by `compute_folder_hash`) is **the canonical recursive walker** for this codebase, and `LocalDirSource::fetch()` must mirror it exactly — same `std::fs::read_dir` loop, same symlink skip, same `.git`/`node_modules` skip, same `Box<dyn Fn>` or recursive descent pattern. Do NOT introduce `walkdir` as a dependency; the existing hand-rolled walker in `lock.rs:129-161` is already correct and tested. The `LocalDirSource` source-dir walk and `compute_folder_hash` installed-dir walk use the same code path, which means hash consistency between "bytes we copied" and "bytes on disk" is guaranteed by construction.

**Primary recommendation:** Wave 0 = new `HubErrorKind::LocalSourceMissing` variant + `recompute_trust_str` patch (zero-risk, unblocks all consumers). Wave 1 = `LocalDirSource` adapter + lib.rs re-export. Wave 2 = CLI wiring (`cmd_install` arm + D-D1 pre-dispatch hint + `cmd_update` branch). Wave 3 = tests.

### Pattern 1: LocalDirSource struct shape
**What:** `HubSource` implementor that walks a local directory.
**When to use:** When identifier has `local:` prefix.
**Example:**
```rust
// Source: mirrors crates/ironhermes-hub/src/well_known.rs struct shape
// No HTTP client needed; no test_mode flag needed (no network to override).
pub struct LocalDirSource;

#[async_trait]
impl HubSource for LocalDirSource {
    fn source_id(&self) -> &str { "local-dir" }

    fn trust_level_for(&self, _identifier: &str) -> SkillSource {
        SkillSource::Trusted  // D-B2
    }

    async fn search(&self, _query: &str, _limit: usize)
        -> Result<Vec<SkillMeta>, HubError>
    {
        Ok(vec![])  // local dirs are not searchable
    }

    async fn fetch(&self, identifier: &str) -> Result<SkillBundle, HubError> {
        // identifier = canonical absolute path string (D-C1)
        let path = std::path::Path::new(identifier);
        let mut files: Vec<BundleFile> = Vec::new();
        walk_source(path, path, &mut files)?;
        files.sort_by(|a, b| a.path.cmp(&b.path));
        // find SKILL.md, parse name/category...
        Ok(SkillBundle {
            source_id: "local-dir".to_string(),
            snapshot_hash: None,  // no remote snapshot
            ..
        })
    }
}
```

**What:** Before the `contains('/')` arm, check if raw identifier looks like a path.
**When to use:** Always, at the top of the dispatch block.
**Example:**

    || identifier.starts_with("./")
    || identifier.starts_with("../")
    || identifier.starts_with("~/")

**What:** Re-run full pipeline against stored `identifier` path. Hard-fail if missing.
**When to use:** When `entry.source == "local-dir"` in update loop.

**IMPORTANT:** `LocalDirSource` must be added to the `build_sources()` return vec so `cmd_update`'s existing `sources.iter().find(|s| s.source_id() == entry.source.as_str())` dispatch finds it. This is the correct generalization — no special-casing needed in `cmd_update`.

### Anti-Patterns to Avoid
- **Using `walkdir` crate:** The hand-rolled `walk()` in `lock.rs:129-161` already exists, is tested, and has the exact skip semantics needed. Adding `walkdir` would be a redundant dependency.
- **Calling `fs::canonicalize` in the pre-dispatch hint:** D-D1 uses `fs::metadata` (cheaper, no resolution needed). `canonicalize` is only called after `local:` prefix is confirmed (D-A2).
- **Calling the audit endpoint with a local path:** `extract_owner_repo()` in `installer.rs` already returns `""` for identifiers that don't match GitHub format — and a canonical absolute path produces `""` from that function, so audit is automatically skipped with zero new code.
- **Storing the pre-canonicalized path in the lock entry:** D-C1 specifies the **canonicalized** absolute path. Storing the raw user input would break `cmd_update` on relative paths or tilde paths after CWD change.
- **Modifying `installer.rs::install()` or `installer.rs::update()`:** The pipeline is source-agnostic and must not be modified. `LocalDirSource` produces `SkillBundle` with `snapshot_hash: None`, and `update()` already handles `None` snapshot_hash correctly (falls through to `bundle_folder_hash` vs `entry.computed_hash` comparison).

## Cross-Platform / fs::canonicalize Landmines

[VERIFIED via Rust stdlib documentation knowledge; behavior is well-established]

### macOS / Linux
`std::fs::canonicalize` resolves all symlinks and returns an absolute path. On macOS, `/tmp` is a symlink to `/private/tmp` — so `canonicalize("/tmp/foo")` returns `/private/tmp/foo`. This matters for the D-20 containment check (`assert_temp_contained`) which uses `canonicalize` on both sides, so it remains consistent. For LocalDirSource this is not a D-20 concern (the source dir is read, not deleted), but the canonical path stored in the lock entry will be the fully-resolved form (e.g., `/private/tmp/...` on macOS). This is correct and expected.

### Windows — UNC path prefix [ASSUMED]
On Windows, `canonicalize` returns `\\?\` UNC extended-length paths (e.g., `\\?\C:\Users\...`). This prefix is stored in the lock entry and used as the identifier for re-canonicalize on update. `std::path::Path::new("\\\\?\\C:\\...")` parses correctly. `fs::metadata` and `fs::read_dir` handle UNC paths. The `is_path_safe` lexical fallback in `sanitize.rs:183-196` handles `Component::Prefix(_)` correctly (it is included in the normalized output, not rejected). **No code change needed; Windows should work.** [ASSUMED — project is currently macOS/Linux focused per CONTEXT.md; Windows not tested]

### Permission denied
`fs::canonicalize` returns `Err(PermissionDenied)` if the process cannot traverse a path component. D-A2 specifies hard-fail on canonicalize error — the error message should surface the OS error. The typed error should use `HubErrorKind::Io` (for the canonicalize failure) or `HubErrorKind::LocalSourceMissing` (for the missing-path case). Distinguish: missing path → `LocalSourceMissing`; permission denied → `Io` with message including `strip_terminal_escapes` of the path.

### Tilde expansion
`dirs::home_dir()` on Windows returns `None` when `USERPROFILE` and `HOMEDRIVE`+`HOMEPATH` are not set. The error path should be `anyhow::anyhow!("cannot expand ~: home directory not available")`. On macOS/Linux `dirs::home_dir()` uses `HOME` env var (or `getpwuid_r`). [VERIFIED: `dirs::home_dir()` is already used at `paths.rs:8` and `main.rs:521` — established pattern in the codebase]

### Relative path resolution

## Recursive Walk: Key Implementation Details

[VERIFIED: live read of lock.rs:129-161]

The existing `walk()` function in `lock.rs` is the canonical model. `LocalDirSource::fetch()` must use the **same pattern** for the source-dir walk:

```
walk(base, dir, out):
  for entry in read_dir(dir):
    if is_symlink → skip (defense against symlinks inside user-created skill dirs)
    if is_dir:
      if name == ".git" || name == "node_modules" → skip
      recurse into dir
    if is_file:
      rel = path.strip_prefix(base).replace('\\', '/')
      content = read(path)
      out.push((rel, content))
```

**Ordering for D-13 hash consistency:** The source walk result must be `sort_by path` before constructing `BundleFile` entries. `lock.rs::compute_folder_hash` also sorts alphabetically before hashing. Since `BundleFile` paths are forward-slash normalized, the sort order is the same regardless of OS path separator. [VERIFIED: `lock.rs:120` does `files.sort_by(|a, b| a.0.cmp(&b.0))`]

**Filter recommendation (Claude's discretion):** The source walk should also skip `target/` directories (Rust build artifacts — common in developer skill directories) and hidden files/directories starting with `.` (except files explicitly named `.gitignore` etc. that are part of the skill). However CONTEXT.md does not mandate this beyond `.git` and `node_modules`. The plan should use at minimum the same filters as `lock.rs::walk()` (`.git`, `node_modules`, symlinks) and document the decision about `target/` in the plan.

**Symlinks inside skill dir:** A malicious or misconfigured skill directory might contain symlinks pointing outside the directory. The `walk()` skip-symlinks rule (`if file_type.is_symlink() { continue; }`) ensures symlinks are never followed during the source walk. Combined with `sanitize_subpath()` in `write_bundle_to_dir()`, this provides defense-in-depth. The CONTEXT.md `<code_context>` section explicitly calls this out: "user trust ≠ filesystem trust."

**`target/` skip as D-B1 implication:** Skills like `paper2code`, `frontend-design`, `redesign-existing-projects` that CONTEXT.md mentions as carrying `references/`, helper scripts, and fixtures — these would NOT have a `target/` dir (they're skill dirs, not Rust project dirs). The `target/` skip is primarily a developer ergonomics concern for users who point `local:` at a Rust crate they're developing. Recommend skipping it; minimal risk.

## Bulk Update Failure-Isolation Pattern

[VERIFIED: skills_cmd.rs:461-524]

The existing `cmd_update` (no name) already uses the failure-isolation pattern:

```rust
let mut exit_code = 0i32;
for skill_name in &names_to_update {
    // ...
    let source = sources.iter().find(|s| s.source_id() == entry.source.as_str());

}

The `LocalSourceMissing` error from a missing source directory will surface through the `hub_update → LocalDirSource::fetch()` error path, be caught by the `Err(e)` arm above, print the error, set `exit_code = 1`, and `continue` to the next skill. **No code change needed in the failure-isolation loop itself** — it already generalizes.

The only change needed: `LocalDirSource` must be in `build_sources()` return vec, so the `sources.iter().find(...)` dispatch finds it for `"local-dir"` entries. The current `None` arm (unknown source → error + continue) would otherwise fire for every local-dir skill update.

## Common Pitfalls

### Pitfall 1: D-D1 hint fires false-positive for `owner/repo` identifiers that happen to exist as a directory
**What goes wrong:** User has a local directory named `anthropics` with a subdirectory `skills`. `fs::metadata("anthropics/skills")` returns `is_dir() = true`. The hint fires before routing to GitHub.
**Why it happens:** The D-D1 spec explicitly uses `fs::metadata` as one of the two probes.
**How to avoid:** CONTEXT.md D-D1 explicitly acknowledges this: "GitHub repo names that happen to also exist as a directory in the user's CWD are a degenerate edge case — we accept the false positive there because the hint is informational, not destructive." Implement exactly as specified; do not add logic to suppress the false positive.
**Warning signs:** Test case: `fs::metadata("foo/bar")` returns `is_dir() = true` — this is expected behavior, not a bug.

### Pitfall 2: `cmd_update` routing for `local-dir` source fails with "unknown source" before LocalDirSource is wired
**What goes wrong:** `LocalDirSource` is implemented but not added to `build_sources()`. Every `hermes skills update` for a local-dir skill hits the `None` arm and prints "unknown source 'local-dir'".
**Why it happens:** `build_sources()` is called once and returns `[github, well-known, skills-sh]`. `LocalDirSource` must be added.
**How to avoid:** `LocalDirSource` needs no config — it's stateless. `build_sources()` should unconditionally push it: `vec![Box::new(SharedGitHubSource(gh)), Box::new(wk), Box::new(sh), Box::new(LocalDirSource)]`.

### Pitfall 3: Source-dir walk follows symlinks, triggering sanitize_subpath violation
**What goes wrong:** User's skill dir contains a symlink. `walk()` encounters it. If NOT skipped, the symlink target path might contain `..` components after resolution, causing `sanitize_subpath` to reject the bundle.
**Why it happens:** Following symlinks during source walk is the default without explicit skip.
**How to avoid:** Mirror `lock.rs:141-143`: `if file_type.is_symlink() { continue; }`. CONTEXT.md explicitly calls this out.

### Pitfall 4: `strict_yaml_delimiter` not called on SKILL.md from local dir
**What goes wrong:** A locally-authored SKILL.md with `---js` frontmatter bypasses the D-17 guard (because the guard is not called during `LocalDirSource::fetch()`).
**Why it happens:** For remote sources, `strict_yaml_delimiter` is called in the blob fetch pipeline. `LocalDirSource::fetch()` must call it explicitly on the SKILL.md content before returning the bundle.
**How to avoid:** In `LocalDirSource::fetch()`, after reading the SKILL.md file bytes, call `crate::sanitize::strict_yaml_delimiter(content)?` before constructing the bundle.

### Pitfall 5: `recompute_trust_str` returns `"builtin"` for local-dir entries in `cmd_list`
**What goes wrong:** `cmd_list` shows `[builtin]` for local-dir skills because `recompute_trust_str` falls through to the `else { "builtin" }` arm.
**Why it happens:** No `"local-dir"` arm exists today. [VERIFIED: skills_cmd.rs:208-229]
**How to avoid:** Add `else if source == "local-dir" { "trusted" }` arm to `recompute_trust_str`.

### Pitfall 6: D-21 Line 1 still says "skills.sh/" for local: installs
**What goes wrong:** User sees `Resolving skills.sh/local:/home/user/skill...` which is confusing.
**Why it happens:** Line 1 print is emitted before the dispatch block with hardcoded "skills.sh/" prefix.
**How to avoid:** Either (a) move D-21 lines 1-3 inside each dispatch arm, or (b) add a "Resolving local:..." print inside the `local:` arm and suppress/override the pre-dispatch print. Simplest: put the print inside the `local:` arm body.

### Pitfall 7: `HubErrorKind::LocalSourceMissing` not added before Wave 2 code compiles
**What goes wrong:** `cmd_update` code references `HubErrorKind::LocalSourceMissing` but the variant doesn't exist yet; build fails.
**Why it happens:** Wave ordering — error variant must land before any consumer code.
**How to avoid:** Wave 0 adds the variant. All later waves depend on Wave 0.

### Pitfall 8: `assert_temp_contained` in `cleanup_quarantine_safely` rejects quarantine paths on macOS
**What goes wrong:** macOS resolves `/tmp` to `/private/tmp`. If `HERMES_HOME` is under `/Users/...`, the quarantine dir is under `<HERMES_HOME>/.hub/quarantine/`, not under `/private/tmp`. `is_in_accepted_cleanup_root` has a second check for `quarantine_dir()` containment. [VERIFIED: installer.rs:651-675 — this was already fixed in Phase 21.8 Plan 05]
**Why it happens:** This is already fixed — documented here so the test author knows to use `tempfile::tempdir_in(&quarantine_root)` in tests, not `tempfile::tempdir()`.

// Source: mirrors lock.rs:129-161 walk() pattern
async fn fetch(&self, identifier: &str) -> Result<SkillBundle, HubError> {
    // identifier is already canonicalized (D-C1)
    let base = std::path::Path::new(identifier);

    if !base.is_dir() {
        return Err(HubError::Typed {
            kind: HubErrorKind::LocalSourceMissing,
            message: format!(
                "source dir no longer exists at {}",
                identifier
            ),
            suggestion: Some("Run 'hermes skills remove <name>' to de-register, or restore the source directory.".to_string()),
            retry_after_s: None,
        });
    }

    let mut files: Vec<(String, Vec<u8>)> = Vec::new();
    walk_source_dir(base, base, &mut files)?;
    files.sort_by(|a, b| a.0.cmp(&b.0));

    // Find SKILL.md
    let skill_md_entry = files.iter().find(|(p, _)| {
        p == "SKILL.md" || p.ends_with("/SKILL.md")
    }).ok_or_else(|| HubError::Typed {
        kind: HubErrorKind::Parse,
        message: format!("no SKILL.md found in {}", identifier),
        suggestion: Some("Ensure the source directory contains a SKILL.md file.".to_string()),
        retry_after_s: None,
    })?;

    let skill_md_content = String::from_utf8_lossy(&skill_md_entry.1).into_owned();

    // D-17: strict YAML delimiter check
    crate::sanitize::strict_yaml_delimiter(&skill_md_content)?;

    let skill_md_path = skill_md_entry.0.clone();
    let metadata = parse_skill_metadata_from_frontmatter(&skill_md_content);
    let name = metadata.name.unwrap_or_else(|| {
        // Fallback: use last component of source dir name
        base.file_name()
            .map(|n| crate::sanitize::sanitize_name(&n.to_string_lossy()))
            .unwrap_or_else(|| "unnamed-skill".to_string())
    });
    let name = crate::sanitize::sanitize_name(&name);

    let bundle_files: Vec<BundleFile> = files.into_iter()
        .map(|(path, bytes)| BundleFile { path, bytes })
        .collect();

    Ok(SkillBundle {
        name,
        identifier: identifier.to_string(),
        source_id: "local-dir".to_string(),
        files: bundle_files,
        skill_md: skill_md_content,
        metadata: serde_json::json!({}),
        snapshot_hash: None,  // D-C1: no remote snapshot
    })
}
```

fn expand_path(raw: &str) -> anyhow::Result<std::path::PathBuf> {
    if let Some(rest) = raw.strip_prefix("~/") {
        let home = dirs::home_dir()
            .ok_or_else(|| anyhow::anyhow!("cannot expand ~: home directory not available"))?;
        Ok(home.join(rest))

    } else if std::path::Path::new(raw).is_relative() {

            .map_err(|e| anyhow::anyhow!("cannot resolve relative path: {e}"))?;
        Ok(cwd.join(raw))
    } else {
        Ok(std::path::PathBuf::from(raw))
    }
}

### Nyquist Sampling Rationale

**D-D1 hint prefix variants:** The Nyquist floor is one test per distinct path-shape trigger: `/`, `./`, `../`, `~/`, and the metadata-probe case (bare existing-dir). That is 5 distinct discriminants — all 5 must be covered to avoid aliasing. An implementation that only checks `starts_with('/')` would pass a `./` test silently if the `./` path doesn't start with `/`.

**Aliasing risk — WARN-BUT-LOAD vs hard-reject:** The `enforce_trust_gate` code (scanner.rs:127-166) uses `match trust { Community => Err(...), _ => Ok(...) }`. For `Trusted` tier, the test must assert that the install **succeeded** (returned `Ok`) AND that the scan verdict summary is NOT `"clean"` (to prove the scan actually ran and hit the pattern). A test that only asserts `Ok` on a WARN-BUT-LOAD case does not prove scan ran — use `AlwaysBlockedScanner` and assert `outcome.scan_verdict` contains `"blocked"` in the summary string. [VERIFIED: scanner.rs:29-42 — `summary()` returns `"blocked: <files>"` when `has_blocks()` is true]

**Aliasing risk — D-13 hash vs D-14 snapshot hash:** `computedHash` and `snapshotHash` are different fields. For local installs `snapshotHash = ""` always. A test that only asserts `entry.computed_hash != ""` doesn't prove `snapshotHash == ""`. Both must be explicitly asserted.

## Open Questions

1. **`target/` directory skip in source walk**
   - What we know: `lock.rs::walk()` only skips `.git` and `node_modules`.
   - What's unclear: Should `LocalDirSource` also skip `target/` (Rust build artifacts)?
   - Recommendation: Add `target/` to the skip list for `LocalDirSource` only (not for `compute_folder_hash` which runs on the installed dir). Document the divergence. This is Claude's discretion per CONTEXT.md.

2. **Hidden-file filtering in source walk**
   - What we know: `lock.rs::walk()` does not skip hidden files/directories (except `.git`).
   - What's unclear: Should `.DS_Store`, `.gitignore`, etc. be excluded from the installed bundle?
   - Recommendation: Mirror `lock.rs::walk()` exactly — skip nothing beyond `.git` and `node_modules`. If a user wants to exclude files, they should use a `.hermesignore` mechanism (future phase). Adding ad-hoc filtering now creates surprising behavior.
