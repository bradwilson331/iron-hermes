<objective>
Close gap-01 from `21.8.1-VERIFICATION.md`: skills installed via `ironhermes skills install` (any source — local-dir, skills-sh, GitHub) do not appear in chat `/skills`, gateway tool registry, or Telegram surfaces.

**Root cause** (verified by live source read + on-disk inspection):
- `crates/ironhermes-hub/src/installer.rs:271-272` writes installed skills to **two-level** layout: `<skills_root>/<category>/<name>/SKILL.md` (e.g. `~/.ironhermes/skills/creative/ascii-art/SKILL.md`). Verified in user's filesystem at the time of planning — entries at `general/code-review/SKILL.md`, `creative/ascii-art/SKILL.md`, `creative/excalidraw/SKILL.md`, etc.
- `crates/ironhermes-core/src/skills.rs::load_with_paths` (lines 577-606) scans only **one level deep**: `for entry in read_dir(search_path) { ... entry.path().join("SKILL.md") }`. Two-level entries are silently skipped because `creative/SKILL.md` does not exist.
- All three surfaces (chat /skills, gateway Telegram prompt, iron_hermes_ui webserver `/api/tools` adjacency) consume the **same** `Arc<SkillRegistry>` produced by `build_app_runtime_bundle`. Single-point fix.

**Why one plan covers all three surfaces:** `crates/ironhermes-agent/src/app_runtime_factory.rs:89-92` is the **only** place the SkillRegistry is loaded for production runtime. CLI run_chat / run_single / run_gateway and `crates/iron_hermes_ui/src/server/state.rs:83` all funnel through `build_app_runtime_bundle`. Telegram is a gateway message platform — gateway turns wire `skill_registry` into the prompt-builder at `crates/ironhermes-gateway/src/runner.rs:1093`. Fix the loader once; every surface gains visibility.

Purpose: Make installed skills (including the new `local-dir` source landing in 21.8.1) actually usable end-to-end across every chat surface IronHermes ships.

Output: Two-level-aware `SkillRegistry::load_with_paths` + three layered tests (unit / CLI integration / gateway integration) that lock the fix in place against future regression.
</objective>
