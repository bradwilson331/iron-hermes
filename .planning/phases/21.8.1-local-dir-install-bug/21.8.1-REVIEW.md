## Summary

Plan 05 extends `SkillRegistry::load_with_paths` from a one-level scan to a
bounded two-level scan and ships unit + integration tests proving the install
→ load → prompt-builder loop. The extraction of `try_register_skill_from_dir`
is clean and the dedup-by-name invariant is preserved. However, the new
two-level scan introduces a **symlink-based bypass of the `.hub` filter** that
the test suite does not exercise. The remaining findings are quality issues
(asymmetric filtering between levels, dead `ENV_LOCK` acquisition in a
subprocess test, and one inline comment that misrepresents reachable code).

The headline classification (BLOCKER / WARNING) summary:

| ID    | Severity | Area                                                          |
|-------|----------|---------------------------------------------------------------|
| CR-01 | BLOCKER  | `.hub` filter bypassable via symlinked directory entries      |
| WR-01 | WARNING  | `node_modules` filter only at level 2; asymmetric with L1     |
| WR-02 | WARNING  | First-path-wins same-name collision across L1/L2 untested     |
| WR-03 | WARNING  | Lock poisoning silently recovered via `into_inner()`          |
| WR-04 | WARNING  | Same-name collision within one root produces silent shadowing |

## Critical Issues

### CR-01: `.hub` filter bypassable via a symlinked directory entry (BLOCKER)

> **Disposition (2026-05-09):** **Deferred — do not fix in 21.8.1.** A more
> detailed, user-involved fix will land in a future security phase whose
> hardening behavior is driven by the project-wide DEFCON 5→1 strictness scale
> (DEFCON 5 = least extreme, DEFCON 1 = most severe). Symlink handling for the
> `.hub` filter — and any related path-trust decisions — should be expressed
> as DEFCON-conditional behavior, not as a one-shot patch. Tracked in project
> memory: `project_security_defcon_scale.md`. Decision is intentional;
> subsequent reviews must not silently re-flag this as a "missed fix."

**File:** `crates/ironhermes-core/src/skills.rs:724-778`

**Issue:** The two-level scan filters out `.hub` and dotfile dirs **by file
name only**, but uses `Path::is_dir()` (which follows symlinks) to decide
whether to descend. A malicious or accidentally-created symlink whose own
file name is innocuous (e.g. `creative`) but whose target is `.hub`
(or any other directory containing a `SKILL.md`) defeats both filters
at both levels.

Concrete path against the stated invariant
"NEVER descend into the hub state directory":

```
<skills_root>/
  .hub/
    lockfile-fake-skill/SKILL.md       ← T-21.8.1-08 says "must not load"
  innocuous -> .hub                    ← symlink, name = "innocuous"
```

Trace through the loop at line 722:

1. `entry.path()` returns `<root>/innocuous` — `is_dir()` returns `true`
   (follows the symlink to `.hub/`).
2. Filter at line 736: `dir_name_str == "innocuous"` — passes.
3. Level-1 SKILL.md check: `<root>/innocuous/SKILL.md` — does not exist on
   the link target, so skipped. OK.
4. Level-2 read_dir at line 749: now reads the contents of `.hub/`. The
   inner-name filter at line 770 sees `lockfile-fake-skill` — passes.
5. `try_register_skill_from_dir` is called with
   `<root>/innocuous/lockfile-fake-skill`, reads SKILL.md, and registers it.

The hub state directory has now been treated as a skill category. T-21.8.1-08
is bypassed. The same vector exists for any path-traversal target the
runtime user can read (e.g. a symlink to `~/.ssh` so an attacker-supplied
`SKILL.md` parent stages registration with a `path` claim that later flows
into `read_content`).

Setting aside intent, this is also a robustness concern: a user who runs
`ln -s .hub backup` inside their skills root for any reason will silently
start loading lock-file artifacts as skills.

The unit test `test_load_with_paths_two_level_skips_hub_dir` only proves
the **name-based** filter works; it does not exercise symlink targets, so
regressions are invisible.

**Fix:** Either reject symlinks entirely at both levels, or canonicalize
each candidate before applying the name filter, or use
`entry.file_type()` instead of `Path::is_dir()` and explicitly skip
`is_symlink()`. Minimal patch (level 1):

```rust
// Replace `if !subdir.is_dir() { continue; }` at line 724
let ft = match entry.file_type() {
    Ok(t) => t,
    Err(_) => continue,
};
if ft.is_symlink() || !ft.is_dir() {
    debug!(
        "SkillRegistry: skipping non-dir or symlinked entry {:?}",
        subdir
    );
    continue;
}
```

Apply the same change to the inner loop at line 761
(`!inner_subdir.is_dir()`). Add a regression test: create
`<root>/.hub/fake/SKILL.md` plus a symlink `<root>/innocuous -> .hub`,
then assert `registry.list().is_empty()`.

If symlinks are intentionally supported (e.g. for shared skill libraries),
canonicalize the target and re-apply the `.hub` / dotfile / `node_modules`
filter against the canonical path components, not just the link's own
file name.

## Warnings

### WR-01: `node_modules` filter is asymmetric between level 1 and level 2

**File:** `crates/ironhermes-core/src/skills.rs:736, 770`

**Issue:** The level-1 filter rejects only `.hub` and dotfile dirs:

```rust
if dir_name_str == ".hub" || dir_name_str.starts_with('.') {
    continue;
}
```

The level-2 filter additionally rejects `node_modules`:

```rust
if inner_name == ".hub" || inner_name.starts_with('.') || inner_name == "node_modules" {
    continue;
}
```

A `<skills_root>/node_modules/` directory is therefore treated as a
category candidate at level 1, which means its level-2 children are scanned
as skill dirs. Today this is mostly cosmetic (no `SKILL.md` inside published
npm packages), but a single supply-chain package that ships a top-level
`SKILL.md` would land in the registry. More importantly, the asymmetry
indicates the hardening was added in one place and not the other — exactly
the shape of mistake plan 05 was meant to lock down.

**Fix:** Lift the filter set into a single helper used at both levels.
Consider extending it (e.g. `target/`, `dist/`, `__pycache__/`) or
documenting the policy explicitly:

```rust
fn is_excluded_dir_name(name: &str) -> bool {
    name.is_empty()
        || name.starts_with('.')
        || name == "node_modules"
        || name == "target"
        || name == "__pycache__"
}
// then at both levels:
if is_excluded_dir_name(dir_name_str) { continue; }
```

### WR-02: First-path-wins same-name collision across L1/L2 in *one* root is undocumented and untested

**File:** `crates/ironhermes-core/src/skills.rs:740-778`,
`crates/ironhermes-core/src/skills.rs::tests::test_load_with_paths_two_level_dedup_first_path_wins`

**Issue:** Within one search path, `try_register_skill_from_dir` runs for
level 1 first, then for each level-2 subdir. The dedup-by-name guard
(line 568) silently drops the second hit. So if a developer stages BOTH:

```
<root>/cat/SKILL.md           (name: shared)
<root>/cat/inner/SKILL.md     (name: shared)
```

…the level-1 copy wins, the level-2 copy is silently dropped, and the
inline comment at line 747-748 is correct in passing — but no test pins
this behavior. The test that *is* present
(`test_load_with_paths_two_level_dedup_first_path_wins`) crosses search
*paths*, not levels within one path.

The headline test
`test_load_with_paths_handles_intermixed_skill_md_at_root_and_subdir`
exercises the *different-name* case (cat-as-skill + ascii-art) but not the
collision case. A future refactor that swaps loop order (level 2 before
level 1) would silently flip semantics with no test failure.

**Fix:** Add a unit test that hand-builds `<root>/cat/SKILL.md` and
`<root>/cat/inner/SKILL.md` with identical `name:` and asserts that the
level-1 description survives. Or, equivalently, pin the order in a
doc-test on `try_register_skill_from_dir`.

### WR-03: Lock poisoning silently recovered via `into_inner()` masks panics

**File:** `crates/ironhermes-cli/tests/skills_cmd_integration.rs:27, 1044`

**Issue:**

```rust
let _guard = ENV_LOCK.lock().unwrap_or_else(|e| e.into_inner());
```

A poisoned mutex means a previous test panicked while holding `ENV_LOCK`,
which means env vars were left in an inconsistent state (e.g. `HERMES_HOME`
pointing at a now-deleted tempdir). `into_inner()` ignores that signal and
proceeds. Subsequent tests then run in the leaked state and may pass or
fail nondeterministically.

This is not new in plan 05 — line 27 has the same pattern — but plan 05's
new test at line 1044 inherits it (and arguably should not have been
copying that pattern at all; see IN-01).

**Fix:** Either propagate the poison
(`.expect("ENV_LOCK poisoned — a prior test panicked while holding the env mutex")`)
so the offending test is surfaced, or unconditionally restore env vars in a
`Drop` impl rather than at the end of `with_hermes_home`. The current
pattern fails open in exactly the case where you most want to fail closed.

### WR-04: Same-name collision within one root produces silent shadowing without warning

**File:** `crates/ironhermes-core/src/skills.rs:566-572`
(behavior reached from plan 05's two-level scan)

**Issue:** The dedup guard inside `try_register_skill_from_dir` uses
`debug!` for duplicates:

```rust
if seen_names.contains(&name_lower) {
    debug!(
        "SkillRegistry: skipping duplicate skill '{}' at {:?}",
        frontmatter.name, skill_md_path
    );
    return;
}
```

Plan 05 raises the probability of in-root collisions because the same
search path now contributes from two depths. A user who installs two
skills into different categories that happen to share a frontmatter
`name` will see one silently disappear, with the diagnostic emitted only
at `debug` level — which is filtered out by default in release builds.

The pre-21.8.1 behavior was similar (level-1-only could still collide
across `cwd/.ironhermes/skills/<name>/` and `~/.agents/skills/<name>/`),
but the user-visible failure mode now includes "I installed two skills
into different categories within the same root and one is gone."

**Fix:** Promote the dedup-suppression message to `warn!` when the
shadowed and shadowing skills come from the *same* search root
(unintentional collision, vs. the intentional first-path-wins case
across roots). Threading `current_root: &Path` through the helper
signature would let callers distinguish.

## Info

### IN-01: `ENV_LOCK.lock()` in a subprocess-only test does nothing useful

**File:** `crates/ironhermes-cli/tests/skills_cmd_integration.rs:1044`

**Issue:** The other subprocess test in the same file (line 196-198)
explicitly documents:

> This test bypasses ENV_LOCK because it spawns subprocesses — subprocess
> env is copy-on-spawn and does NOT mutate the parent.

`installed_local_dir_skill_appears_in_skill_registry_catalog` only ever
mutates env via `Command::env(...)` (lines 1057-1058), which is also
copy-on-spawn — so taking the lock at line 1044 serializes this test
against `with_hermes_home` callers for no functional reason. It also
inherits the poisoning trap from WR-03 for no benefit.

**Fix:** Remove the `let _guard = ENV_LOCK.lock()...` line, mirroring the
pattern used by `install_list_remove_round_trip` and the other in-file
subprocess tests. If the intent was to serialize cargo-test workers
because the test is otherwise heavy, document that explicitly.

### IN-02: Misleading inline comment in the level-2 fallback branch

**File:** `crates/ironhermes-core/src/skills.rs:776-777`

**Issue:** Comment reads:

```rust
// No `else { recurse }`: if there's no SKILL.md at level 2, this entry
// is just clutter inside a category dir (notes/, README.md/, etc.).
```

`README.md/` is not a directory, it is a file — and the surrounding loop
already filtered to `is_dir()` at line 761. The example mis-states what
actually reaches this branch. The intent ("we do not recurse a third
time, bounded depth 2") is correct but the parenthetical is wrong, and
the misleading wording invites a future contributor to "fix" the wrong
thing.

**Fix:**

```rust
// No `else { recurse }`: if there's no SKILL.md at level 2, this entry
// is just a clutter directory (e.g. `notes/`, `references/`,
// `helpers/`). Depth is bounded at 2 — see
// test_load_with_paths_does_not_descend_three_levels.
```

### IN-03: `inner.flatten()` silently drops per-entry I/O errors

**File:** `crates/ironhermes-core/src/skills.rs:759`

**Issue:** `for inner_entry in inner.flatten()` discards `Err` results
from the directory iterator (e.g. permission denied on a single child).
The outer loop at line 722 has the same pattern, so this is consistent
with existing behavior, but the level-1 *opener* logs at `debug!` on
failure (lines 752-755) while the *iterator's* per-entry failures pass
without any log. A skill that should have been discovered but is hidden
by EACCES on a single child will simply not appear, with no diagnostic.

**Fix:** Replace `flatten()` with an explicit `match` that emits
`debug!("inner read_dir entry error: {}", err)` on each `Err`. Cheap and
makes future "why isn't my skill loading" debugging tractable.
