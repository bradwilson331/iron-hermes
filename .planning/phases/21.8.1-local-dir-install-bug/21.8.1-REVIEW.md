# Phase 21.8.1: Code Review Report

**Reviewed:** 2026-05-08T00:00:00Z
**Depth:** standard
**Files Reviewed:** 7
**Status:** issues_found

## Summary

Reviewed the Phase 21.8.1 local-dir install adapter plus associated CLI plumbing,
lock file changes, error variant, and integration tests. The submitted
implementation is largely sound, with no security-critical findings: the
symlink-skip walker, canonicalization-at-CLI-layer pattern, and `[local]`
list-output annotation are all wired correctly. The error variant
`LocalSourceMissing` flows through serialization and `envelope()` cleanly.

The defects below cluster around three themes:

1. **Audit-skip for local installs is incidental, not explicit.** The audit
   endpoint is bypassed for `local-dir` only as a side effect of
   `extract_owner_repo` returning an empty string when the bundle identifier
   starts with `/`. The integration test that asserts this property
   (`local_dir_audit_skipped`) therefore exercises a fragile invariant — any
   future change to canonical-path formatting (Windows `C:\...`, UNC `\\?\...`,
   or a future "audit by path-segment" feature) silently re-enables the audit
   call and leaks local paths.

2. **A test assertion is logically tautological** (always passes regardless
   of the implementation). This was the regression-shield test for bulk
   update failure isolation — it currently shields nothing.

3. **YAML frontmatter `name:` parsing is brittle.** Line-comments
   (`name: foo # bar`), quoted-empty (`name: ""`), and post-sanitize empty
   strings can produce wrong skill names or pass through unchecked.

No BLOCKER findings; the headline UAT regression (`local:` prefix) does work
end-to-end. Recommend resolving the WARNING items before tagging the release.

## Warnings

### WR-01: Audit-skip for local installs depends on fragile `extract_owner_repo` empty-return path

**File:** `crates/ironhermes-hub/src/installer.rs:154-174` (called from `crates/ironhermes-hub/src/installer.rs:237-253` and `:424-440`)
**Issue:** The integration test `local_dir_audit_skipped`
(`crates/ironhermes-hub/tests/local_dir_adapter_test.rs:352-395`) asserts the
audit endpoint receives zero requests for local installs. This currently holds
because `bundle.identifier` for local installs is a Unix-style canonical
absolute path (e.g. `/private/tmp/foo`), and `extract_owner_repo`'s
`splitn(3, '/')` yields `parts[0] == ""`, returning an empty owner_repo. There
is **no explicit `local-dir` source-id skip** in `installer::install` /
`installer::update`. Failure modes:

- On Windows, a canonical path like `C:\Users\me\skill` produces
  `parts = ["C:\\Users\\me\\skill"]` (length 1) — empty return — OK by accident.
- A canonical UNC path `\\?\C:\foo\bar` would `splitn` to a single segment
  starting with `\\?\C:\foo\bar`. Length 1 → empty return — OK by accident.
- BUT: any future canonicalization tweak that stops the path with `/` — e.g.
  switching to a `file://` URI, or stripping the leading `/` for display —
  silently re-enables `fetch_audit` for local paths and leaks the absolute
  filesystem path of the user's source tree to the third-party audit endpoint.
**Fix:** Add an explicit skip at the top of the audit block in both `install`
and `update`:
```rust
// ── Step 3: Audit (D-19 soft-fail; skipped if --skip-audit; never run for local-dir) ───
if !skip_audit && source.source_id() != "local-dir" {
    let owner_repo = extract_owner_repo(&bundle);
    // ...
}
```
This makes the local-install privacy invariant structural rather than
incidental, and the existing `local_dir_audit_skipped` test continues to pass.

### WR-02: Tautological assertion in `cmd_update_local_dir_missing_source_isolates_failure`

**File:** `crates/ironhermes-cli/tests/skills_cmd_integration.rs:641-644`
**Issue:** The "skill-b succeeds even though skill-a failed" assertion is:
```rust
assert!(
    update_stdout.contains("skill-b") || !update_stderr.contains("skill-b"),
    ...
);
```
The second disjunct `!update_stderr.contains("skill-b")` is true whenever
skill-b is not mentioned on stderr. Since skill-b is the *succeeding* skill,
its progress goes to stdout (line 592 in `skills_cmd.rs`) and it is never
written to stderr, so the second branch is **always true** regardless of
implementation. The assertion accepts any outcome — even one where bulk update
silently dropped skill-b entirely. The regression shield this test was meant
to provide does not exist.
**Fix:** Tighten to a strict-AND that proves skill-b actually ran:
```rust
assert!(
    update_stdout.contains("skill-b"),
    "skill-b must succeed and appear in stdout even though skill-a failed; \
     stdout={update_stdout} stderr={update_stderr}"
);
// Optionally also verify the on-disk content reflects the v2 source:
let installed = std::fs::read_to_string(/* find skill-b SKILL.md */).unwrap();
assert!(installed.contains("updated-b"));
```

### WR-03: `parse_skill_name_from_frontmatter` mishandles inline YAML comments

**File:** `crates/ironhermes-hub/src/local_dir.rs:172-189`
**Issue:** YAML allows trailing line comments: `name: my-skill # the production name`.
The current parser does:
```rust
let name = rest.trim().trim_matches('"').trim_matches('\'').to_string();
```
For `rest = " my-skill # the production name"`, `name` becomes
`"my-skill # the production name"`. That string then flows into
`sanitize::sanitize_name`, which may strip `#` and spaces, yielding e.g.
`my-skill-the-production-name` — a name the user did not intend. The lock
entry, install dir, and CLI output all silently use the corrupted name.
**Fix:** Strip a `#` comment before quote-trimming:
```rust
let mut value = rest.trim();
// Strip YAML line comment ` # ...`. Note: `#` inside a quoted value is legal
// YAML; the existing parser is non-quote-aware so we mirror that limitation.
if let Some(idx) = value.find(" #") {
    value = &value[..idx];
}
let name = value.trim().trim_matches('"').trim_matches('\'').to_string();
```
For full correctness, switch to `serde_yaml` for the frontmatter block — the
hand-rolled scanner already loses to multi-line values, anchor refs, and
quoted-`#`. The repo already depends on `serde_yaml` (see `lib.rs`
`HubError::Yaml` and `manifest.rs`).

### WR-04: `parse_skill_name_from_frontmatter` returns empty string after `sanitize_name` strips all chars

**File:** `crates/ironhermes-hub/src/local_dir.rs:181-186`
**Issue:** The empty-name guard runs **before** `sanitize_name`:
```rust
let name = rest.trim().trim_matches('"').trim_matches('\'').to_string();
if !name.is_empty() {
    return Some(sanitize::sanitize_name(&name));   // could still return ""
}
```
If `name = "@@@"` (all sanitize-stripped chars), the function returns
`Some("")` — a non-`None` empty string. Downstream in `fetch()`:
```rust
let name = parse_skill_name_from_frontmatter(&skill_md_content)
    .or_else(|| base.file_name().map(...))
    .unwrap_or_else(|| "unnamed-skill".to_string());
```
`Some("")` short-circuits the `or_else` fallback to the directory basename,
and the bundle is built with `name = ""`. That empty string then feeds
`parse_skill_identity` and the lock entry — likely producing a confusing
"installed at `skills/<category>/`" with a trailing slash and a `name == ""`
lock entry that breaks subsequent `update` / `remove` lookups by name.
**Fix:** Sanitize-first, then check non-empty:
```rust
let sanitized = sanitize::sanitize_name(&name);
if !sanitized.is_empty() {
    return Some(sanitized);
}
// fall through to next line of frontmatter / fallback
```

### WR-05: `update()` hard-codes `install_path` category as `"general"` for cleanup

**File:** `crates/ironhermes-hub/src/installer.rs:387-393` (used at `:462`)
**Issue:** Pre-existing in 21.8 but inherits into the local-dir update path
hot in this PR. `install_path` is constructed as
`skills_root.join("general").join(&entry.name)` regardless of the *actual*
installed category (which is recorded in `entry.repo_path` as
`"<category>/<skill>/SKILL.md"`). The cleanup loop:
```rust
for candidate in [&install_path, &resolved_final] {
    if candidate.exists() { cleanup_final_path_safely(candidate); }
}
```
only cleans `general/<name>`, leaving the real old install (e.g.
`skills/test/<name>` for a `category: test` skill — exactly the fixture used by
`cmd_update_local_dir_recopies_from_source`) **orphaned on disk** if the
bundle's category changes between versions, or never cleans the old dir at all
if `general` was never the source's category. The integration test
`cmd_update_local_dir_recopies_from_source` (uses `category: test`) only
passes because `resolved_final` happens to equal the original install path
(category unchanged), so the old dir is removed via the `resolved_final`
candidate, masking the bug.
**Fix:** Derive `install_path` from `entry.repo_path` (which already encodes
the category):
```rust
let old_category = std::path::Path::new(&entry.repo_path)
    .components()
    .next()
    .map(|c| c.as_os_str().to_string_lossy().into_owned())
    .unwrap_or_else(|| "general".to_string());
let install_path = skills_root.join(&old_category).join(&entry.name);
```

### WR-06: Pre-dispatch `Resolving skills.sh/<id>...` line printed for unknown identifiers that later route to `skills-sh`

**File:** `crates/ironhermes-cli/src/skills_cmd.rs:265-296`
**Issue:** The "Resolving skills.sh/..." line is printed *before* the
`looks_like_local_path_with_probe` hint check. If the identifier looks like a
local path (e.g. `./my-skill`) but the user did NOT prefix `local:`, the user
sees:
```
Resolving skills.sh/./my-skill...
error: ./my-skill looks like a local path — did you mean ...?
```
The first line is misleading (we never actually try to resolve via skills.sh
— we exit immediately afterwards). The output mixes a false success-path
prefix with an error.
**Fix:** Move the unconditional `Resolving skills.sh/...` print *after* the
D-D1 path-shape probe, or skip it when the probe fires:
```rust
let has_known_prefix = /* ... */;
if !has_known_prefix {
    let looks_like_path = looks_like_local_path_with_probe(...);
    if looks_like_path {
        eprintln!("error: ...");
        return Ok(1);
    }
}
// Only NOW emit the generic resolving line.
if !identifier.starts_with("local:") {
    println!("Resolving skills.sh/{}...", strip_terminal_escapes(identifier));
}
```

### WR-07: Test `local_dir_perms_denied` parses `id -u` output and silently defaults UID to 1000 on failure

**File:** `crates/ironhermes-hub/tests/local_dir_adapter_test.rs:898-913`
**Issue:**
```rust
let uid_output = std::process::Command::new("id").arg("-u").output()
    .ok().and_then(|o| String::from_utf8(o.stdout).ok())
    .and_then(|s| s.trim().parse::<u32>().ok())
    .unwrap_or(1000);
if uid_output == 0 { ... return; }
```
If `id` is missing from PATH, fails to spawn, or returns non-UTF8/non-numeric
output, the test silently assumes UID=1000 and runs the chmod-000 assertion.
On a sandbox that runs `id` differently or has `id` masked, this could either
skip a real-root run (false negative — root passes the chmod and the test
asserts an error that never fires) OR run on root and corrupt the tempdir.
**Fix:** Either use `nix::unistd::Uid::current()` (already an indirect dep
via `tempfile`?), or make the fallback skip rather than run:
```rust
let uid = std::process::Command::new("id").arg("-u").output()
    .ok().and_then(|o| String::from_utf8(o.stdout).ok())
    .and_then(|s| s.trim().parse::<u32>().ok());
match uid {
    Some(0) | None => {
        eprintln!("Skipping local_dir_perms_denied: cannot determine UID or running as root");
        return;
    }
    Some(_) => {}
}
```

## Info

### IN-01: `expand_path` does not handle Windows `~\path` form

**File:** `crates/ironhermes-cli/src/skills_cmd.rs:771-786`
**Issue:** Only `~/` (forward-slash) and bare `~` are expanded. On Windows,
users frequently type `~\Documents\skill`. The current code falls into the
`is_relative()` branch, which joins CWD — producing a literal `~\Documents`
subdirectory that almost certainly does not exist, and the user sees a
confusing "cannot resolve local path" instead of a tilde-expansion.
**Fix:** Also strip `~\\` on Windows builds (or unconditionally — `\` is not
a legal path separator on Unix, so the strip is a no-op there):
```rust
if let Some(rest) = raw.strip_prefix("~/").or_else(|| raw.strip_prefix("~\\")) {
    let home = dirs::home_dir().ok_or_else(...)?;
    Ok(home.join(rest))
}
```

### IN-02: `looks_like_local_path_with_probe` does not detect Windows absolute paths

**File:** `crates/ironhermes-cli/src/skills_cmd.rs:800-809`
**Issue:** The five "trigger shapes" do not include `[A-Z]:\` (Windows drive
letter) or `\\?\` UNC. A Windows user typing
`hermes skills install C:\Users\me\skill` does not get the helpful
"did you mean local:..." hint. They get whatever the bare-name path produces
downstream (probably an `InvalidIdentifier` or skills-sh registry miss).
**Fix:** Add a Windows-aware check, gated on cfg or unconditional since `:` in
position 1 is unambiguous as a drive letter:
```rust
fn looks_windows_abs(s: &str) -> bool {
    let bytes = s.as_bytes();
    bytes.len() >= 3
        && bytes[0].is_ascii_alphabetic()
        && bytes[1] == b':'
        && (bytes[2] == b'\\' || bytes[2] == b'/')
}
// ... || looks_windows_abs(identifier) || identifier.starts_with("\\\\")
```

### IN-03: `compute_folder_hash` uses `to_string_lossy()` on relative paths — non-UTF8 paths could collide

**File:** `crates/ironhermes-hub/src/lock.rs:150-156`, mirrored at `crates/ironhermes-hub/src/local_dir.rs:151-156`
**Issue:** `path.strip_prefix(base)?.to_string_lossy().replace('\\', "/")` —
`to_string_lossy` replaces invalid UTF-8 sequences with U+FFFD. Two distinct
non-UTF8 file names that differ only in the invalid byte sequence both
become the same `"\u{FFFD}foo"` and produce identical hashes. Probability is
low (most filesystems people install skills from are UTF-8), but it weakens
the D-13 deterministic-hash guarantee.
**Fix:** Either reject non-UTF8 names with a typed error, or hash the raw
`OsStr` bytes via `as_bytes()` on Unix / `encode_wide()` on Windows. The
simplest correct fix:
```rust
use std::os::unix::ffi::OsStrExt;
let rel_os = path.strip_prefix(base)?;
let rel = rel_os.to_str().ok_or_else(|| typed(
    HubErrorKind::Io, format!("non-UTF8 path: {}", rel_os.display()),
))?.replace('\\', "/");
```

### IN-04: `cmd_install` swallows migration errors with `let _ = migrate_from_hub_manifest()`

**File:** `crates/ironhermes-cli/src/skills_cmd.rs:255` (also `:524`, `:622`, `:655`)
**Issue:** Documented as best-effort by the comment, and that's correct
policy, but the swallow happens silently. A future operator debugging
"why is my .hub/lock.json still on disk" gets no signal. Consider at least a
`tracing::warn!` log:
```rust
if let Err(e) = migrate_from_hub_manifest() {
    tracing::warn!(error = %e, "best-effort .hub/lock.json migration failed");
}
```
This is a one-line behavioral upgrade with no contract change.

### IN-05: `cmd_install` prints `Downloading 0 bytes...` line as fixed placeholder

**File:** `crates/ironhermes-cli/src/skills_cmd.rs:387`
**Issue:** `println!("Downloading {size} bytes...", size = 0);` — the `0` is
hard-coded. This is documented as preserving the D-21 5-line shape without a
real size probe, but `Downloading 0 bytes...` reads like an error to the
user (the install obviously transfers more than zero bytes). Recommend
either removing the byte count or using a non-numeric placeholder:
```rust
println!("Downloading...");
```
Especially noisy for `local:` installs where there is no network download —
the line is a literal lie.

### IN-06: `parse_skill_name_from_frontmatter` does not handle multi-line / quoted-pipe YAML values

**File:** `crates/ironhermes-hub/src/local_dir.rs:172-189`
**Issue:** Companion to WR-03. YAML scalars like:
```yaml
name: |
  my-skill
```
or
```yaml
name: >-
  my-skill
```
are silently treated as `name = "|"` or `name = ">-"`, then sanitize-stripped
to the empty string and falling back to the directory basename. Documented as
acceptable in the comment ("hand-rolled scan avoids pulling in serde_yaml"),
but is worth flagging since the test suite does not exercise these forms and
a real skill author may hit it.
**Fix:** See WR-03 — switch to `serde_yaml` for the frontmatter block.
