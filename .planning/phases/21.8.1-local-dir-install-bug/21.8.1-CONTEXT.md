# Phase 21.8.1: local-dir-install bug - Context

**Gathered:** 2026-05-08
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix the bug surfaced in 21.8 post-completion UAT: `hermes skills install <path>` rejects local directory identifiers (e.g., `bradwilson/download/ascii-art/`) because the dispatcher in `crates/ironhermes-cli/src/skills_cmd.rs:261-290` routes any identifier containing `/` to `GitHubSource`, which then fetches `api.github.com/repos/<dir>/tarball/HEAD` and hard-fails with `tarball not found`. There is no local-directory source variant in `ironhermes-hub`.

**In scope (Phase 21.8.1):**
- Add `local:<path>` identifier prefix to the `cmd_install` dispatch table.
- New `LocalDirSource` adapter implementing `HubSource` (sibling to `GitHubSource` / `WellKnownSkillSource` / `SkillsShBlobSource`).
- New `Source::LocalDir` variant in the lock file schema.
- Wire `install`, `update`, `remove`, `list` to handle the new source variant.
- Pre-dispatch hint: when a non-prefixed identifier looks like a path (leading `/`, `./`, `../`, `~/`, or resolves to an existing directory on disk), emit a single error line suggesting `local:<path>` and exit 1 — before routing to GitHub.
- Reuse existing security primitives: `CoreSkillScanner` over content; `sanitize_subpath` / path-traversal guards; terminal escape stripping on user-facing errors.

**Out of scope for this phase:**
- Symlink install mode (always copy).
- In-place register mode (no copy/symlink, would bypass install pipeline — rejected for security).
- Auto-detect local installs without `local:` prefix (only the pre-dispatch hint, never silent routing).
- Adding a new `Self-created` trust tier (would scope-creep into Phase 28 / SKILL-09 work).
- Watch-and-auto-refresh on source-dir edits (refresh requires explicit `hermes skills update`).
- Telemetry for local installs.
- Publishing local skills to skills.sh (deferred from 21.8 already).
- New schema migration concerns — `LocalDir` is a forward-additive variant; lock file v1 stays at v1 with WARN-BUT-LOAD on unknown sources for older readers.

</domain>

<decisions>
## Implementation Decisions

### Identifier format

- **D-A1:** Explicit `local:<path>` prefix only. Sibling to `skills-sh:` and `well-known:` in the dispatcher at `crates/ironhermes-cli/src/skills_cmd.rs:261-290`. No magic auto-detect; no path-shape heuristics dispatch silently. The dispatcher gains a fourth `else if identifier.starts_with("local:")` arm before the `contains('/')` GitHub arm.
- **D-A2:** `local:` accepts absolute, relative, and `~/` forms. Tilde expanded via `dirs::home_dir()`; relative paths resolved against the process CWD; final path canonicalized via `std::fs::canonicalize()` BEFORE any further I/O. If canonicalize fails (path missing, permission denied, not a directory), hard-fail with a typed error before the install pipeline begins.

### Install semantics, trust, audit

- **D-B1:** Install mode is **copy** — full directory contents (SKILL.md plus every subfile and subdirectory, recursively) copied into `~/.ironhermes/skills/<category>/<name>/`. Reuses the 21.8 install pipeline: walk source → build `SkillBundle` → quarantine → scan → atomic `rename` to final path → lock entry. Source-dir edits do NOT auto-propagate; user must run `hermes skills update <name>` to refresh. No symlink mode. No in-place mode.
- **D-B2:** Trust tier = `SkillSource::Trusted`. Local installs are user-authored content from the operator's own filesystem. The `CoreSkillScanner` runs over SKILL.md and content (D-17 frontmatter delimiter check, threat patterns) using the Phase 19 D-15 **WARN-BUT-LOAD** posture for Trusted skills (not Community hard-reject). Pre-install audit (`add-skill.vercel.sh/audit`) is **SKIPPED** — there is no remote URL for the audit endpoint to resolve; calling audit on a local path is meaningless and would leak the path to a third-party endpoint.

### Lock entry shape

- **D-C1:** New `source: "local-dir"` value in the `SkillLockEntry.source` discriminator (`crates/ironhermes-hub/src/lock.rs`). Field semantics:
  - `identifier` = `fs::canonicalize`'d absolute path of the source directory (symlinks resolved, `~/` expanded). Stored as a `String`.
  - `repoPath` = relative path of `SKILL.md` inside the source dir (e.g., `SKILL.md` or `nested/SKILL.md` if the user pointed at a parent dir). Same convention as remote sources.
  - `snapshotHash` = empty string (`""`) — there is no remote snapshot to mirror.
  - `computedHash` = SHA-256 over the **installed folder** (not the source) using the D-13 algorithm: sort by path, concatenate `path_bytes + content_bytes` with NO separators. Drift detection in `cmd_list` reads this exactly as it does for remote sources.
  - `installedAt` = ISO-8601 timestamp at install time. Outside the hashed region per Phase 21.8 D-12.

### Update + drift behavior

- **D-C2:** `hermes skills update <name>` for a local-installed skill re-runs the FULL install pipeline against the source path stored in `identifier`:
  1. canonicalize `identifier`; if the source dir is missing, hard-fail with `source dir no longer exists at <path>` (typed `HubError::Network`-style envelope is wrong here — add or reuse a `HubErrorKind::LocalSourceMissing` variant).
  2. recursively re-walk the source — **all SKILL.md updates AND every other subfile/subdirectory must be picked up so refreshed assets land**. No file in the on-disk install survives a stale copy from a prior version.
  3. quarantine → scan → atomic move (replacing the existing install) → recompute `computedHash` → write the lock back.
  4. Update `installedAt` to the refresh time.
- Bulk `hermes skills update` (no name) iterates ALL skills in the lock, including `local-dir` entries. Each local skill that fails (missing source, scan reject) is reported but does not abort the bulk run.
- `hermes skills list` already verifies `computedHash` against the live folder per Phase 21.8 D-13. For `local-dir` entries this surfaces drift between the installed copy and the lock — it does NOT compare against the source path. The user reads "drifted" and chooses to `update`.
- `hermes skills remove <name>` deletes only the install at `~/.ironhermes/skills/<cat>/<name>/`. The user's source directory is **never touched** (sacrosanct — it's outside `skills_root`).

### Error UX (pre-dispatch hint)

- **D-D1:** Before the existing `contains('/')` GitHub-routing arm in `cmd_install`, run a cheap pre-check on the raw identifier:
  - If it has no recognized prefix (`well-known:`, `skills-sh:`, `local:`) AND
  - either (a) starts with `/`, `./`, `../`, or `~/`, OR (b) `fs::metadata(identifier).map(|m| m.is_dir()).unwrap_or(false)` returns true,
  - then emit `error: looks like a local path — did you mean 'hermes skills install local:<path>'?` (passed through `strip_terminal_escapes` for D-16 parity even though it's all our own text, to keep the print-boundary contract structural) and exit 1 with no GitHub fetch attempted.
- The probe is one `fs::metadata` call; zero false positives for legitimate `owner/repo` identifiers (GitHub repo names that happen to also exist as a directory in the user's CWD are a degenerate edge case — we accept the false positive there because the hint is informational, not destructive).

### Claude's Discretion

- Exact module layout: a new `local_dir.rs` in `ironhermes-hub` mirroring `well_known.rs` shape vs folding the local adapter into an existing module — planner's call. The `HubSource` trait + lock-file schema constrain the public surface; internal organization is open.
- Whether `LocalDirSource` walks the directory using `walkdir` (already a transitive dep) or hand-rolled recursion via `std::fs::read_dir`. Either is fine; pick the one that matches the existing pattern in `installer.rs::compute_folder_hash` (which already walks the installed folder).
- The exact `HubErrorKind` variant name for source-dir-missing (`LocalSourceMissing`, `SourceMissing`, `PathNotFound`) — planner picks consistent with existing variants.
- Whether the pre-dispatch hint uses `eprintln!` directly or threads through a typed `HubError::Hint` variant — the existing dispatcher already returns `i32` exit codes, so a direct `eprintln!` + `Ok(1)` is consistent with the current shape.
- Test harness layout: extend `crates/ironhermes-hub/tests/` with `local_dir_adapter_test.rs` mirroring `skills_sh_adapter_test.rs` (uses `tempfile::tempdir` for fixture source dirs; no `wiremock` needed since there's no HTTP).

</decisions>

## Existing Code Insights

### Reusable Assets

- **`HubSource` trait** (`crates/ironhermes-hub/src/source.rs`): new `LocalDirSource` implements this trait verbatim — `source_id()` returns `"local-dir"`, `fetch(identifier)` walks the canonicalized path and returns a `SkillBundle`. Dispatch in `cmd_install` is one new `else if` arm.
- **`SkillBundle` shape** (`crates/ironhermes-hub/src/source.rs`): the existing `Vec<BundleFile>` representation is source-agnostic. Local adapter produces it from a directory walk; rest of the pipeline is unchanged.
- **`CoreSkillScanner`** (`crates/ironhermes-hub/src/scanner.rs`): scan gate works over the bundle's bytes regardless of provenance. WARN-BUT-LOAD posture for Trusted is already wired (Phase 19 D-15).
- **`compute_folder_hash`** (`crates/ironhermes-hub/src/installer.rs:96-118`): D-13 algorithm. Reused for `computedHash` post atomic-move; produces identical hash regardless of source variant.
- **`atomic_move` + `cleanup_quarantine_safely`** (`crates/ironhermes-hub/src/installer.rs`): atomic install primitive. Source-agnostic.
- **`sanitize::sanitize_subpath`** (`crates/ironhermes-hub/src/sanitize.rs`): Phase 21.8 D-18 path-traversal guard. Apply per-file from the source walk.
- **`SkillLock` / `SkillLockEntry` schema** (`crates/ironhermes-hub/src/lock.rs`): `source` field is a `String` with WARN-BUT-LOAD parsing for unknowns (Phase 19.1 D-08). Forward-additive — existing readers won't reject `local-dir`.
- **`save_config_atomic`** (`crates/ironhermes-cli/src/skills_cmd.rs`): tmp + rename for lock-file writes (Phase 21.8 D-25).
- **`strip_terminal_escapes`** (`crates/ironhermes-hub/src/sanitize.rs`): D-16 — applies to every error message printed, including the new pre-dispatch hint and `LocalSourceMissing` errors.

### Established Patterns

- **Atomic install pipeline** (Phase 19.1 D-11, Phase 21.8 D-25): fetch → quarantine → scan → atomic move → lock update. Local install plugs into step 1 (fetch becomes "walk source dir") and reuses steps 2-5 unchanged.
- **Trust recomputation per Phase 19.1 D-08**: trust level is NEVER frozen in the lock file; `cmd_list` recomputes from the live config + source-id mapping. Local-dir source maps to `Trusted` per D-B2; the lock file does not need a trust field.
- **Frozen-snapshot prompt building** (Phase 15 PRMT-02): D-23 from 21.8 carries forward — local installs land on disk; activation requires next-session restart.
- **Source-id-based dispatch in `cmd_install`** (Phase 21.8): identifier prefix → `source_id` → adapter lookup. New `local:` prefix follows the same pattern.
- **Pre-existing dispatcher pattern**: `cmd_install` already routes by prefix-then-fallback (`well-known:`, `skills-sh:`, `/`-contains, bare). The new `local:` arm fits between the `skills-sh:` arm and the `/`-contains arm.

### Integration Points

- `cmd_install` (`crates/ironhermes-cli/src/skills_cmd.rs:245-338`):
  - Insert pre-dispatch hint at the top (after escape-strip echo, before `build_sources`): if identifier looks like a path AND has no known prefix, emit hint and return 1.
  - Add `else if identifier.starts_with("local:")` arm.
  - D-21 progress lines: line 1 (`Resolving local:<path>...`) prints the canonicalized path; line 2 (`Discovering skills in <path>...`) prints the same; line 3 (`Downloading <size> bytes...`) reports the total bytes walked from the source dir; lines 4-5 unchanged.
- `cmd_list` (`crates/ironhermes-cli/src/skills_cmd.rs:~388`): `recompute_trust_str` for `local-dir` returns `Trusted`. JSON output gains a `local-dir` source value; human output gains a `[local]` or `[trusted/local]` annotation.
- `cmd_update`: branch on `e.source`; `local-dir` → re-fetch from `e.identifier` (canonicalize, walk, scan, move).
- `cmd_remove`: source-agnostic — only deletes from `skills_root` and lock. The user's source path is never touched.
- `migrate_from_hub_manifest()`: no-op for local installs (Phase 19.1's `.hub/*.json` never carried local entries; nothing to migrate).
- `paths::skills_root()`: unchanged; local installs land in the same dir tree as remote.

<specifics>
## Specific Ideas

- **Primary user story:** `hermes skills install local:~/Downloads/my-skill/` should Just Work — canonicalize, walk dir, scan SKILL.md, copy into `~/.ironhermes/skills/<category>/<name>/`, write lock entry, print the D-21 5-line progress shape, exit 0. Restart to use.
- **Pre-dispatch hint is the bug-fix anchor.** Today the user gets `tarball not found for bradwilson/download` from GitHub. Tomorrow they get `looks like a local path — did you mean 'hermes skills install local:bradwilson/download/ascii-art/'?`. That single hint line is the headline UX deliverable.
- **The "ENTIRE skill dir tree" rule (D-C2)** is load-bearing: skills like `paper2code`, `frontend-design`, `redesign-existing-projects` (visible in this session's available-skills list) carry `references/`, helper scripts, fixtures. A `local:` install or `update` that copied only SKILL.md would silently break those skills at runtime. Walk recursively; copy everything.
- **Trust = Trusted, not Self-created.** `Self-created` is reserved for Phase 28 / SKILL-09 work (autonomous skill authoring per LEARN-04). Trying to land it here would scope-creep; punt to Phase 28.
- **Audit endpoint silence is intentional security.** Calling `add-skill.vercel.sh/audit` with a local path leaks the path to a third party. Skip the call entirely; do not transform local paths into pseudo-identifiers for the audit endpoint.
- **The pre-dispatch hint uses `fs::metadata`, not `fs::canonicalize`.** Canonicalize requires the path to exist; metadata is the cheaper "does this resolve to a directory" probe. Keep the hint cheap and never fail-noisy.
- **Source dir is sacrosanct.** `hermes skills remove` only ever deletes from `skills_root`. Even if a future feature wanted to "uninstall and clean source", that is out of scope here — we do not touch the user's filesystem outside `~/.ironhermes/skills/`.

</specifics>

<deferred>
## Deferred Ideas

### Captured for later phases

- **Symlink install mode** (live-edit dev loop): rejected for security in 21.8.1 (scanner can't validate once-and-trust; lockfile drift becomes "always drifted"). Revisit only if a documented "skill development mode" feature emerges with explicit cache/scan-bypass semantics. Candidate: future `hermes skills dev <path>` subcommand.
- **In-place register** (no copy, no symlink): rejected — bypasses install pipeline, breaks `skills_root` containment guarantee.
- **Auto-detect local installs without `local:` prefix**: rejected as silent magic. Pre-dispatch hint (D-D1) explicitly tells the user to use the prefix; we never silently route to local.
- **`Self-created` trust tier**: out of scope — coordinated with Phase 28 (SKILL-09 trust tiers) and LEARN-04 (autonomous skill creation in v2.1).
- **Watch + auto-refresh on source-dir edits**: `hermes skills update` is manual. Mirrors the Phase 19.1 D-10 manual-only update stance.
- **Telemetry for local installs**: opt-out and irrelevant — no remote endpoint involved. No telemetry surface to consider.
- **Publishing local skills to skills.sh** (push flow): already deferred from Phase 19.1 D-12; remains deferred.
- **Lock file v2 / additional fields per source variant**: not needed; `source: "local-dir"` plus the existing fields covers it. WARN-BUT-LOAD on unknown fields preserves forward compat for any future schema additions.
- **`hermes skills dev <path>` development mode**: candidate for a future phase — would couple symlink install + scan-bypass + dev-only trust tier. Out of scope here.
- **GitLab / arbitrary-git source types**: still deferred from Phase 21.8.
