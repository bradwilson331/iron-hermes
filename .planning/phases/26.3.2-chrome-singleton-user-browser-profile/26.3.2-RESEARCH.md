## Summary

This phase adds a shared helper in `ironhermes-core` that, before chromiumoxide launches, reads `<user_data_dir>/SingletonLock` and determines whether it represents a stale or live Chromium process. If stale (dead pid, or a pid from a different hostname), the helper removes all three sentinel files and returns `UseProfile` so the launch proceeds with the persistent profile. If the lock is held by a live process, the helper returns `UseEphemeral` and the caller skips `.user_data_dir(...)`, silently degrading to a fresh ephemeral profile. If no lock exists, the helper returns `UseProfile` immediately (the common happy path).

The fix prevents the `Failed to create a ProcessSingleton for your profile directory. Aborting now.` crash that permanently breaks authenticated browser automation after any unclean Chromium exit (crash, kill -9, OS reboot, or two overlapping sessions).

The helper lives in `ironhermes-core/src/browser_profile.rs` (a new module, ~80 lines) and is called from both `browser_session.rs` copies immediately after the `user_data_dir` resolution block that Phase 26.3 added. The helper is a pure synchronous function over `&Path` — no chromiumoxide dependency, fully unit-testable without launching a browser.

**Primary recommendation:** Add `pub mod browser_profile;` to `ironhermes-core/src/lib.rs`, implement `reconcile_singleton_lock(profile_dir: &Path) -> SingletonOutcome` there, call it from both `browser_session.rs` spawn() functions right after the `user_data_dir` is resolved and before `builder.user_data_dir(...)`.

## Open Question Resolutions

### OQ-1: Dep-free Process-Liveness Probe (macOS + Linux)

**Recommendation: dual-path — `/proc/<pid>` on Linux, `Command::new("kill")` on macOS/other-Unix**

#### `/proc/<pid>` on Linux
`Path::new(&format!("/proc/{pid}")).exists()` — no subprocess, pure filesystem check. Returns `true` for live AND zombie processes (both have a `/proc` entry). Zombie is acceptable: a zombie chromium means its parent is still running and tracking it — not a stale lock scenario. Returns `false` when the process is truly gone (ESRCH). No EPERM issue whatsoever.
[VERIFIED: Linux procfs behavior — process entries exist until fully reaped]

#### `Command::new("kill") / /bin/kill` on macOS and other Unix
`std::process::Command::new("kill").args(["-0", &pid_str]).status()` calls `/bin/kill` (the Rust `Command` API goes through `execvp`, not the shell, so it finds `/bin/kill` via PATH, not the shell builtin).

**CRITICAL EPERM FINDING (macOS-specific, VERIFIED on this macOS Darwin 25.4.0 system):**
On macOS, `/bin/kill -0 <pid>` returns:
- Exit 0 — process exists AND is owned by the current user (alive, same user)
- Exit 1 — EITHER no such process (ESRCH / dead) OR process exists but owned by different user (EPERM)

This means on macOS, `/bin/kill -0` CANNOT distinguish "dead" from "alive but owned by different user". **This is acceptable for this use case** because:
1. Chromium is always spawned by the hermes user (`Browser::launch` in spawn()). The pid in `SingletonLock` will always be a same-user process. EPERM (different-user ownership) is essentially impossible for a legitimate Chromium lock from this application.
2. Even if EPERM were somehow triggered (shared filesystem with another user running Chrome), treating it as "dead" would be wrong. But `/proc/<pid>` on Linux handles EPERM correctly (returns `true` regardless of ownership).

On Linux, bash's kill builtin returns exit 0 for EPERM (process exists, confirms alive). But `Command::new("kill")` in Rust executes `/bin/kill` or `/usr/bin/kill`, NOT the shell builtin — behavior varies. The `/proc/<pid>` approach on Linux sidesteps this entirely.

**CONTRAST with `gateway/pid.rs`:** The existing gateway PID liveness probe uses `nix::sys::signal::kill(pid, None)` which properly distinguishes EPERM (`LiveOtherUser`) from ESRCH (`Stale`). `nix` is in the workspace and in `ironhermes-gateway/Cargo.toml`, but **NOT** in `ironhermes-core/Cargo.toml`. Adding `nix = { workspace = true }` to `ironhermes-core` would violate D-02 (no new crates). The dep-free approach is mandated.
[VERIFIED: `ironhermes-core/Cargo.toml` has no `nix` dep; `ironhermes-gateway/Cargo.toml` has `nix = { workspace = true, features = ["signal"] }`]

**Zombie process edge case:** `kill -0` returns exit 0 for zombie processes (they remain in the process table). `/proc/<pid>` also returns `true` for zombies. Treating a zombie chromium as "alive" is correct: its parent (the tokio async runtime) is still tracking it and will reap it. A zombie chromium holding a `SingletonLock` is not a stale-lock scenario.

**The recommended implementation:**

```rust
// Source: derived from gateway/pid.rs pattern, adapted to dep-free std-only
#[cfg(unix)]
fn is_pid_alive(pid: u32) -> bool {
    // Linux: use /proc/<pid> filesystem check — no subprocess, no EPERM issue.
    #[cfg(target_os = "linux")]
    {
        std::path::Path::new(&format!("/proc/{pid}")).exists()
    }
    // macOS and other Unix: shell out to /bin/kill -0.
    // Exit 0 = alive (same-user process). Exit 1 = dead (ESRCH).
    // EPERM (different-user alive process) also returns exit 1 on macOS /bin/kill,
    // so would be treated as dead — acceptable because chromium is always same-user.
    #[cfg(not(target_os = "linux"))]
    {
        std::process::Command::new("kill")
            .args(["-0", &pid.to_string()])
            .status()
            .map(|s| s.success())
            .unwrap_or(false) // spawn failure → treat as dead (safe conservative choice)
    }
}

#[cfg(not(unix))]
fn is_pid_alive(_pid: u32) -> bool {
    true // Windows: no-op, always treat as alive (D-09)
}
```

[VERIFIED: `/bin/kill -0 $$` returns exit 0 on this macOS system; `/bin/kill -0 99999999` returns exit 1; `/bin/kill -0 1` (launchd/root) returns exit 1 — confirms EPERM=exit 1 on macOS]

### OQ-2: Dep-free Hostname Read

**Recommendation: `Command::new("hostname")` with fallback to pid-liveness-only if it fails**

Testing on macOS Darwin 25.4.0:
- `Command::new("hostname")` — works, returns `Brads-MacBook-Pro.local` [VERIFIED: `hostname` command available on macOS]
- `/proc/sys/kernel/hostname` — NOT available on macOS [VERIFIED: macOS has no /proc]
- `HOSTNAME` env var — NOT set by default on macOS [VERIFIED: `HOSTNAME` env is unset in this shell]
- `uname -n` — works on macOS [VERIFIED: returns hostname]

On Linux: all three work. `/proc/sys/kernel/hostname` is Linux-only. `Command::new("hostname")` works on both.

**Chosen approach:** `Command::new("hostname")` stdout trimmed. If the command fails or returns empty string, fall back to **pid-liveness-only** (ignore the hostname segment of the symlink target). This matches D-03's stated fallback.

The hostname check is an optimization for network-filesystem scenarios where two different machines share a profile directory. For the common single-machine case, pid-liveness alone is sufficient. If `hostname` fails (possible in locked-down containers), the helper gracefully degrades to pid-only — still covers the primary use case (local stale lock after crash).

```rust
fn current_hostname() -> Option<String> {
    std::process::Command::new("hostname")
        .output()
        .ok()
        .and_then(|o| String::from_utf8(o.stdout).ok())
        .map(|s| s.trim().to_string())
        .filter(|s| !s.is_empty())
}
```

**D-03 compliance:** When `current_hostname()` returns `None`, the hostname segment from the symlink target is ignored and the liveness check (OQ-1 approach) is used as the sole discriminator. This is explicitly the D-03 fallback.
[VERIFIED: hostname command works on macOS and Linux]

### OQ-3: Does chromiumoxide 0.9 / Chrome Skip ProcessSingleton in Headless vs Headed Mode?

**Answer: No — ProcessSingleton is NOT skipped in either headless or headed mode.**

**Evidence from chromiumoxide 0.9.1 source** (`~/.cargo/registry/src/.../chromiumoxide-0.9.1/src/browser/config.rs`):
[VERIFIED: read directly from cargo registry]

- `HeadlessMode` is an enum with variants `False` (headed), `True` (old headless, the **default**), and `New` (`--headless=new`)
- The default `BrowserConfigBuilder::default()` sets `headless: HeadlessMode::True` → passes `--headless` flag (old headless mode)
- `.with_head()` sets `headless: HeadlessMode::False` → no headless flag at all
- `.new_headless_mode()` sets `headless: HeadlessMode::New` → passes `--headless=new`

None of these modes adds `--disable-extensions` or any other flag that would bypass `ProcessSingleton`. The `DEFAULT_ARGS` array in chromiumoxide does NOT include any singleton-disabling flag.

**From Chromium `process_singleton_posix.cc`:** No headless bypass code exists in the ProcessSingleton implementation. ProcessSingleton operates regardless of headless/headed mode.
[CITED: https://chromium.googlesource.com/chromium/src/+/refs/heads/main/chrome/browser/process_singleton_posix.cc]

**Production log confirmation:** The actual crash (`Failed to create .../browser-profile/SingletonLock: File exists (17)`) occurred in production with the current default config (which uses `HeadlessMode::True` since `.with_head()` is not called unless `config.headed = true`). This proves ProcessSingleton fires in both modes.

**ironhermes current default:** `config.headed` defaults to `false` → `HeadlessMode::True` → old headless mode (`--headless`). The fix covers this. When the operator sets `browser.headed = true`, the fix also covers that. Both paths hit ProcessSingleton.

### OQ-4: Symlink Creation in TempDir for Unit Tests

**Answer: Yes — `std::os::unix::fs::symlink` works in TempDir on macOS + Linux CI with no elevated privileges.**

[VERIFIED: manually tested on macOS Darwin 25.4.0 — `ln -s "testhost-12345" SingletonLock` in a temp directory succeeds without sudo]

Symlink creation on POSIX filesystems requires no special permissions — only write permission on the containing directory, which `TempDir` provides.

**Exact test fixture cookbook:**

```rust
use std::os::unix::fs::symlink;
use tempfile::TempDir;

// Fixture A: stale lock — pid 99999999 is virtually guaranteed to be dead
// (Linux max PID is 4,194,304; macOS max is ~99,998; this is far above both)
fn make_stale_lock(dir: &TempDir) {
    symlink("somehost-99999999", dir.path().join("SingletonLock")).unwrap();
}

// Fixture B: live lock — our own process pid is guaranteed alive
fn make_live_lock(dir: &TempDir) {
    let target = format!("somehost-{}", std::process::id());
    symlink(target, dir.path().join("SingletonLock")).unwrap();
}

// Fixture C: unparseable lock — regular file (not a symlink)
// read_link() on a non-symlink returns Err on Linux (EINVAL) and Err on macOS
fn make_unparseable_lock_regular_file(dir: &TempDir) {
    std::fs::write(dir.path().join("SingletonLock"), b"not-a-symlink").unwrap();
}

// Fixture D: unparseable lock — symlink with no dash-digits suffix
fn make_unparseable_lock_bad_target(dir: &TempDir) {
    symlink("no-pid-here", dir.path().join("SingletonLock")).unwrap();
    // "no-pid-here" → split on last '-' → "here" → parse::<u32>() fails → unparseable
    // Wait — this DOES have a dash. The pid portion "here" is not a number.
    // That is the correct unparseable case.
}

// Fixture E: stale lock with hostname mismatch (for D-03 test)
fn make_stale_lock_wrong_host(dir: &TempDir) {
    // Use a hostname that cannot be our machine's hostname
    symlink("definitely-not-this-machine-99999", dir.path().join("SingletonLock")).unwrap();
    // pid 99999 is also dead on most systems; the hostname mismatch fires first
}
```

**Assert strategy:** `reconcile_singleton_lock(dir.path())` is a pure function returning `SingletonOutcome` — no chromiumoxide needed. Assertions:
- For stale: returns `UseProfile`, and `dir.path().join("SingletonLock")` no longer exists
- For live: returns `UseEphemeral`, and `dir.path().join("SingletonLock")` still exists
- For unparseable: returns `UseProfile`, and `dir.path().join("SingletonLock")` still exists (D-01: do not delete unparseable)
- For absent: returns `UseProfile`, no files touched

**NOTE on pid 99999999 guard:** Use `i32::MAX as u32` (2_147_483_647) for the guaranteed-stale PID in tests, matching the `gateway/pid.rs` precedent. `u32::MAX` wraps to -1 as i32, which is the POSIX "send to all processes" sentinel and must NOT be used. Linux PID max is 4,194,304; macOS PID max is approximately 99,998 on modern systems — `i32::MAX as u32` is safely above both.

## Chromium Singleton Sentinel Files: Confirmed Facts

### Three Sentinel Files (and only three)

From Chromium `process_singleton_posix.cc`:
[CITED: https://chromium.googlesource.com/chromium/src/+/refs/heads/main/chrome/browser/process_singleton_posix.cc]

1. **`SingletonLock`** — POSIX symlink whose target is `<hostname>-<pid>`. Created via `symlink()` syscall in the profile root. Removed only on clean exit.
2. **`SingletonSocket`** — symlink to the actual Unix domain socket path under `/tmp/...`. The socket itself lives in `/tmp`; this symlink in the profile points at it.
3. **`SingletonCookie`** — symlink carrying a random per-session validation cookie string.

There is NO separate `lockfile` or `lock_file` plain-file entry. The lock is entirely symlink-based on POSIX. [CITED: chromium source]

### `SingletonLock` Symlink Target Format

```
<hostname>-<pid>
```

The delimiter is **a single hyphen** (`-`). The Chromium source uses `base::StringPrintf("%s%c%u", net::GetHostName().c_str(), kProcessSingletonLockDelimiter, current_pid_)`. Based on the documented format `example.com-9156` and the `%c` format specifier, `kProcessSingletonLockDelimiter` is `-`. [CITED: chromium source, WebFetch result]

**CRITICAL:** Hostnames legitimately contain hyphens (e.g., `Brads-MacBook-Pro.local-20704`). The pid is the substring after the **LAST** hyphen. Correct parsing:

```rust
fn parse_singleton_target(target: &str) -> Option<u32> {
    let last_dash = target.rfind('-')?;
    let pid_str = &target[last_dash + 1..];
    pid_str.parse::<u32>().ok()
}

fn parse_hostname_from_target(target: &str) -> Option<&str> {
    let last_dash = target.rfind('-')?;
    Some(&target[..last_dash])
}
```

[VERIFIED: production log shows `Brads-MacBook-Pro.local-20704` as real target; `rfind` is the correct split]

### `SingletonSocket` Removal: No Dangling Socket Risk

`SingletonSocket` is a **symlink** pointing to a path like `/tmp/SingletonSocket12345`. The socket file itself is in `/tmp`. `std::fs::remove_file` on a symlink removes the symlink itself, NOT the target. So:
- Removing `SingletonSocket` from the profile dir: removes only the symlink pointer — the socket in `/tmp` is unaffected
- The socket file in `/tmp` is either already cleaned up (Chromium removed it on clean exit) or it is a dangling socket that will be cleaned up by the OS on next reboot, or simply occupies a trivial amount of space
- This does NOT cause any user-visible issue [ASSUMED — but follows directly from POSIX symlink semantics which are well-established]

```rust
//! Phase 26.3.2 — Chrome SingletonLock resilience.
//!
//! Provides `reconcile_singleton_lock(profile_dir: &Path) -> SingletonOutcome`
//! called from both `BrowserSession::spawn()` copies before profile selection.

use std::path::Path;

/// The outcome of the singleton lock check.
/// Returned by `reconcile_singleton_lock`.
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum SingletonOutcome {
    /// Profile is available: either no lock, or the stale lock was cleaned.
    /// Caller MUST call `builder.user_data_dir(profile_dir)`.
    UseProfile,
    /// Lock is held by a live process. Caller MUST NOT call `builder.user_data_dir(...)`.
    /// Chromiumoxide will use a fresh ephemeral temp dir.
    UseEphemeral,
}

/// Check and optionally repair the `SingletonLock` in `profile_dir`.
///
/// Logic (D-01..D-06, D-09):
///  - No lock → `UseProfile` (debug log)
///  - Lock present, stale → remove sentinels, `UseProfile` (warn log)
///  - Lock present, live → `UseEphemeral` (warn log)
///  - Lock unparseable → `UseProfile`, do NOT delete (debug log)
///
/// On Windows this is a no-op returning `UseProfile` (D-09).
pub fn reconcile_singleton_lock(profile_dir: &Path) -> SingletonOutcome {
    #[cfg(unix)]
    {
        reconcile_unix(profile_dir)
    }
    #[cfg(not(unix))]
    {
        let _ = profile_dir;
        SingletonOutcome::UseProfile
    }
}
```

The `reconcile_unix` function implements D-01 through D-06. Full shape in Code Examples section.

### Anti-Patterns to Avoid

- **Using `nix`, `libc`, `sysinfo`, or `hostname` crates** — D-02 hard constraint. The existing `gateway/pid.rs` uses `nix` and must be treated as a pattern that is NOT available in `ironhermes-core`.
- **Unconditional delete** — D-06. Never `remove_file` without first checking liveness.
- **Deleting on unparseable target** — D-01. If the lock is not a symlink or the target has no `-<digits>` suffix, leave it alone.
- **Using `u32::MAX` as the dead-pid sentinel in tests** — `u32::MAX` wraps to -1 as i32, which is POSIX "send to all processes" and returns `Ok(())` on macOS. Use `i32::MAX as u32` (2_147_483_647) instead — established by `gateway/pid.rs` tests.
- **`println!` / `eprintln!`** — D-10. All output through `tracing`.
- **Adding `pub use` to the `ironhermes-core` root for internal-only types** — `SingletonOutcome` only needs re-export if the planner decides callers use the full path `ironhermes_core::browser_profile::SingletonOutcome`; the qualified path is fine.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| EPERM/ESRCH discrimination on macOS | Custom libc binding | `/proc/<pid>` on Linux + `kill -0` on macOS | macOS chromium is same-user; EPERM cannot occur in practice |
| Cross-process lock serialization | A mutex or flock-based lock | Ephemeral fallback (D-05) | D-05 explicitly chosen as interim mitigation; mutex deferred |
| Socket file cleanup in /tmp | Walk /tmp for orphaned sockets | Do nothing | `remove_file` on `SingletonSocket` symlink removes only the symlink; the socket in /tmp is OS-managed |
| Hostname parsing from DNS/mDNS | Reverse DNS lookup | `Command::new("hostname")` stdout | Synchronous, dep-free, returns the same string Chrome used |

```rust
// Source: derived from D-01..D-10, OQ-1..OQ-4 resolutions, chromiumoxide source analysis
// Location: crates/ironhermes-core/src/browser_profile.rs

use std::path::Path;

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum SingletonOutcome {
    UseProfile,
    UseEphemeral,
}

pub fn reconcile_singleton_lock(profile_dir: &Path) -> SingletonOutcome {
    #[cfg(unix)]
    return reconcile_unix(profile_dir);
    #[cfg(not(unix))]
    {
        let _ = profile_dir;
        return SingletonOutcome::UseProfile;
    }
}

#[cfg(unix)]
fn reconcile_unix(profile_dir: &Path) -> SingletonOutcome {
    use tracing::{debug, warn};

    let lock_path = profile_dir.join("SingletonLock");

    // D-01: read_link; if absent or not-a-symlink, proceed normally.
    let target = match std::fs::read_link(&lock_path) {
        Ok(t) => t,
        Err(e) if e.kind() == std::io::ErrorKind::NotFound => {
            debug!(path = %lock_path.display(), "SingletonLock absent, launching normally");
            return SingletonOutcome::UseProfile;
        }
        Err(_) => {
            // Not a symlink or I/O error — unparseable per D-01, do not delete
            debug!(path = %lock_path.display(), "SingletonLock present but not a symlink, treating as absent");
            return SingletonOutcome::UseProfile;
        }
    };

    let target_str = target.to_string_lossy();

    // D-01: parse pid from last '-' segment
    let Some(pid) = parse_singleton_target(&target_str) else {
        debug!(
            path = %lock_path.display(),
            target = %target_str,
            "SingletonLock target unparseable (no trailing -<pid>), proceeding without cleanup"
        );
        return SingletonOutcome::UseProfile;
    };

    // D-03: hostname check — if different host, it's stale regardless of pid
    if let Some(lock_host) = parse_hostname_from_target(&target_str) {
        if let Some(our_host) = current_hostname() {
            if lock_host != our_host {
                warn!(
                    path = %lock_path.display(),
                    lock_host = %lock_host,
                    our_host = %our_host,
                    "SingletonLock written by different host (shared profile dir?); cleaning stale sentinels"
                );
                cleanup_sentinels(profile_dir);
                return SingletonOutcome::UseProfile;
            }
        }
        // If current_hostname() returns None: fall through to pid-liveness check (D-03 fallback)
    }

    // D-02: pid liveness check
    if is_pid_alive(pid) {
        warn!(
            path = %profile_dir.display(),
            pid = pid,
            "browser profile locked by live process (pid {pid}); falling back to ephemeral profile (no cookie/login persistence for this session)"
        );
        return SingletonOutcome::UseEphemeral;
    }

    // Stale lock — clean sentinels then launch with persistent profile (D-04)
    warn!(
        path = %profile_dir.display(),
        pid = pid,
        "SingletonLock is stale (pid {pid} no longer running); cleaning sentinel files"
    );
    cleanup_sentinels(profile_dir);
    SingletonOutcome::UseProfile
}

#[cfg(unix)]
fn parse_singleton_target(target: &str) -> Option<u32> {
    let last_dash = target.rfind('-')?;
    target[last_dash + 1..].parse::<u32>().ok()
}

#[cfg(unix)]
fn parse_hostname_from_target(target: &str) -> Option<&str> {
    let last_dash = target.rfind('-')?;
    if last_dash == 0 { return None; }
    Some(&target[..last_dash])
}

#[cfg(unix)]
fn cleanup_sentinels(profile_dir: &Path) {
    use tracing::warn;
    // D-04: remove in order — SingletonLock, SingletonSocket, SingletonCookie
    // Each removal is best-effort.
    for name in &["SingletonLock", "SingletonSocket", "SingletonCookie"] {
        let path = profile_dir.join(name);
        if let Err(e) = std::fs::remove_file(&path) {
            if e.kind() != std::io::ErrorKind::NotFound {
                warn!(path = %path.display(), error = %e, "failed to remove singleton sentinel (best-effort, continuing)");
            }
        }
    }
}

// OQ-1 implementation
#[cfg(unix)]
fn is_pid_alive(pid: u32) -> bool {
    #[cfg(target_os = "linux")]
    {
        std::path::Path::new(&format!("/proc/{pid}")).exists()
    }
    #[cfg(not(target_os = "linux"))]
    {
        std::process::Command::new("kill")
            .args(["-0", &pid.to_string()])
            .status()
            .map(|s| s.success())
            .unwrap_or(false)
    }
}

// OQ-2 implementation
#[cfg(unix)]
fn current_hostname() -> Option<String> {
    std::process::Command::new("hostname")
        .output()
        .ok()
        .and_then(|o| String::from_utf8(o.stdout).ok())
        .map(|s| s.trim().to_string())
        .filter(|s| !s.is_empty())
}
```

## Common Pitfalls

### Pitfall 1: Using `u32::MAX` as the Dead-PID Sentinel in Tests
**What goes wrong:** `u32::MAX` (4_294_967_295) when cast to `i32` becomes -1. POSIX `kill(-1, 0)` means "send signal to all processes I have permission to send to" and returns `Ok(())` — a false "alive" result. The test appears to pass locally but the liveness probe returns "live" for a supposedly dead pid.
**How to avoid:** Always use `i32::MAX as u32` (2_147_483_647) as the guaranteed-dead sentinel. This follows `gateway/pid.rs` precedent. [VERIFIED: `gateway/pid.rs` tests use this exact value]

### Pitfall 2: Putting the `#[cfg(unix)]` Guards Wrong
**What goes wrong:** `std::os::unix::fs::symlink` does not exist on Windows. Test code using it inside `#[cfg(test)]` without `#[cfg(unix)]` will fail to compile on Windows.
**How to avoid:** Wrap unix-only test fixtures in `#[cfg(unix)]`. The production code already has the `#[cfg(unix)]`/`#[cfg(not(unix))]` split.

### Pitfall 3: Calling `remove_file` on a Symlink's TARGET
**What goes wrong:** Confusing "remove the symlink" with "remove what the symlink points to". `std::fs::remove_file` on a symlink path removes the symlink itself, NOT the target. This is the desired behavior for `SingletonLock` and `SingletonSocket` cleanup.
**Confirmation:** POSIX `unlink()` (which Rust's `remove_file` uses) removes the directory entry (the symlink), not the referent. [VERIFIED: POSIX semantics]
**How to avoid:** Use `std::fs::remove_file(&path)` directly — it does the right thing.

### Pitfall 4: Inserting the Helper AFTER `builder.user_data_dir()`
**What goes wrong:** If `reconcile_singleton_lock` is called after `builder.user_data_dir(&user_data_dir)`, the chromiumoxide builder already has the profile dir set. Even if `UseEphemeral` is returned, there is no way to "unset" `user_data_dir` from the builder — the launch will still use the profile dir.
**How to avoid:** The `reconcile_singleton_lock` call must precede `builder.user_data_dir(...)`. The match block either calls or skips `builder.user_data_dir(...)`. The `create_dir_all` block (Phase 26.3) can remain after the match — it does not affect the builder.

### Pitfall 5: TOCTOU Window Between Liveness Check and Launch
**What goes wrong (Security):** Between `is_pid_alive(pid)` returning false (stale) and the three sentinel files being removed, a real chromium process could start up and re-create the `SingletonLock`. The helper then removes the new `SingletonLock`, causing profile corruption — exactly what Chromium was trying to prevent.
**Why the risk is small:** The TOCTOU window is approximately the time to execute three `remove_file` calls — microseconds. A real Chromium startup takes several seconds to reach the point where it creates `SingletonLock`. An attacker would need to time a new Chromium launch to complete within this microsecond window.
**How to handle:** No mitigation is needed in code. Document the window. The alternative — doing nothing — would leave the stale lock permanently blocking browser automation. The production failure is worse than this theoretical race. Include a `<threat_model>` note in each plan.

### Pitfall 6: `SingletonSocket` Cleanup Leaves Dangling `/tmp` Socket
**What goes wrong:** After removing `SingletonSocket` (the symlink in the profile), the actual socket file at the symlink target (e.g., `/tmp/SingletonSocket.XXXXXX`) remains on disk.
**Why it doesn't matter:** Chromium will create a new socket at a new path when it next launches. The old socket file in `/tmp` takes trivial space and will be cleaned up by the OS on reboot or by Chromium's own cleanup logic. Do NOT attempt to follow the symlink and delete the target — that would be unsafe (symlink could point anywhere).
**How to avoid:** Just call `remove_file(&socket_path)` as D-04 specifies. Do nothing about `/tmp`.

### Known Threat Patterns for This Stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| TOCTOU: stale-check-then-delete race | Tampering | Window is microseconds; Chromium startup takes seconds; risk is negligible; document and accept |
| Symlink-following attack: attacker creates `SingletonLock` → `/etc/passwd` | Tampering | `remove_file` on a symlink removes the symlink, NOT the target — no arbitrary file deletion possible |
| Attacker pre-creates `SingletonLock` with live pid to deny browser access | Denial of Service | Ephemeral fallback (D-05) ensures browser still launches; only persistence is lost |
| Pid recycling: dead chromium's pid reused by a new process | Spoofing | `is_pid_alive` returns true → `UseEphemeral` (conservative; browser still works) |
