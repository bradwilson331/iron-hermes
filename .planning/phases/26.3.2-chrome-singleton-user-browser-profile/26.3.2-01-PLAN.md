<objective>
Make chromiumoxide browser launch resilient to a stale Chromium `SingletonLock` left behind in `$HERMES_HOME/browser-profile` after a crash, `kill -9`, OS reboot, or two overlapping browser sessions. Today such a stale lock makes **every** `browser_*` tool call abort with `Failed to create a ProcessSingleton for your profile directory. Aborting now to avoid profile corruption.` until the file is hand-deleted.

This plan adds a shared, dep-free helper `ironhermes_core::browser_profile::reconcile_singleton_lock(profile_dir: &Path) -> SingletonOutcome` and calls it from both `BrowserSession::spawn()` copies right before the `.user_data_dir(...)` decision: a stale lock (dead pid, or a pid from a different host) → remove `SingletonLock` + `SingletonSocket` + `SingletonCookie` and use the persistent profile; a live lock → skip `.user_data_dir(...)` and fall back to an ephemeral profile; no lock or an unparseable lock → proceed unchanged.

Purpose: Authenticated browser automation must not stay permanently broken after an unclean Chromium exit. The fix is one implementation called from two crates so the existing `browser_session.rs` copy-paste drift is not extended.

Output: New file `crates/ironhermes-core/src/browser_profile.rs` (~80 lines + 7 unit tests), `pub mod browser_profile;` in `crates/ironhermes-core/src/lib.rs`, and a 3-line match-block call site replacing the single `builder.user_data_dir(...)` line in each of the two `browser_session.rs` files. No new dependencies; no `config.yaml` schema change.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| filesystem → ironhermes process | `$HERMES_HOME/browser-profile/SingletonLock` is read (`read_link`) and possibly removed (`remove_file`). On a single-user machine the only writer is the operator's own Chromium / hermes; on a shared/network filesystem another machine or user could plant a `SingletonLock`. |
| profile dir state → chromiumoxide launch | The helper's outcome decides whether `builder.user_data_dir(...)` is set, i.e. whether Chromium is launched against the persistent profile or an ephemeral temp dir. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-26.3.2-01 | Tampering (TOCTOU) | `reconcile_unix` → `is_pid_alive` then `cleanup_sentinels` in `browser_profile.rs` | accept | A new Chromium could start between the stale-check returning false and the three `remove_file` calls. Window is ~microseconds (three unlink syscalls); a real Chromium takes several seconds to reach the point where it creates `SingletonLock`. The alternative — doing nothing — leaves the stale lock permanently blocking authenticated browser automation, which is the worse failure. Documented; no code mitigation. |
| T-26.3.2-02 | Tampering (symlink following) | `cleanup_sentinels` → `std::fs::remove_file` in `browser_profile.rs` | accept | A malicious `SingletonLock` symlink pointing at e.g. `/etc/passwd` does NOT cause that target to be deleted: POSIX `unlink()` (Rust `remove_file`) removes the directory entry (the symlink itself), never the referent. Verified POSIX semantics — no arbitrary-file-deletion possible. Additionally, D-01 means an unparseable target is left untouched, so a symlink whose target is not `<host>-<pid>` is never even a removal candidate. No mitigation needed beyond this fact. |
| T-26.3.2-03 | Denial of Service | `reconcile_unix` live-pid branch → `SingletonOutcome::UseEphemeral` | accept | An attacker who can write into the profile dir could plant a `SingletonLock` whose target is a *live* pid (e.g. their own long-running process) to deny use of the persistent profile. Result: `reconcile_singleton_lock` returns `UseEphemeral`, browser launch still succeeds against an ephemeral temp profile — only cookie/login persistence for that session is lost, and a `warn!` line records it. Acceptable; a cross-process browser mutex is explicitly deferred (CONTEXT § Deferred Ideas). |

None of these are `high` severity — the security gate does not block.
</threat_model>
