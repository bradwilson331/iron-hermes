# Phase 26.3.2 — Security

> Per-phase security contract: threat register, accepted risks, and audit trail.

---

## Trust Boundaries

| Boundary | Description | Data Crossing |
|----------|-------------|---------------|
| filesystem → ironhermes process | `$HERMES_HOME/browser-profile/SingletonLock` is read (`std::fs::read_link`) and possibly removed (`std::fs::remove_file`). On a single-user machine the only writer is the operator's own Chromium / hermes; on a shared or network filesystem another machine or user could plant a `SingletonLock`. | Symlink-target string `<hostname>-<pid>` (no secrets); deletion of Chromium sentinel files (`SingletonLock`, `SingletonSocket`, `SingletonCookie`) |
| profile dir state → chromiumoxide launch | The helper's `SingletonOutcome` decides whether `builder.user_data_dir(...)` is set — i.e. whether Chromium launches against the persistent (authenticated) profile or a fresh ephemeral temp dir. | Profile-selection decision; affects whether cookies / login state persist for the session |

## Threat Register

| Threat ID | Category | Component | Disposition | Mitigation | Status |
|-----------|----------|-----------|-------------|------------|--------|
| T-26.3.2-01 | Tampering (TOCTOU) | `reconcile_unix` → `is_pid_alive` then `cleanup_sentinels` (`crates/ironhermes-core/src/browser_profile.rs:94-109`) | accept | No code mitigation — see Accepted Risks **AR-1**. The window between the liveness check returning dead and the three `remove_file` calls is ~3 syscalls; a real Chromium needs several seconds to reach `SingletonLock` creation. Doing nothing leaves the stale lock permanently aborting authenticated browser automation — strictly worse. | closed |
| T-26.3.2-02 | Tampering (symlink following) | `cleanup_sentinels` → `std::fs::remove_file` (`crates/ironhermes-core/src/browser_profile.rs:138-151`) | accept | Verified in code: cleanup uses only `std::fs::remove_file` — POSIX `unlink()` removes the directory entry (the symlink itself), never the referent, so a `SingletonLock` symlink pointing at e.g. `/etc/passwd` cannot cause that target's deletion. Additionally D-01 (`browser_profile.rs:47-74`) leaves any non-`<host>-<pid>` target untouched, so a crafted symlink is never even a removal candidate. Tests SL-02 / SL-03 assert no deletion on unparseable locks. | closed |
| T-26.3.2-03 | Denial of Service | `reconcile_unix` live-pid branch → `SingletonOutcome::UseEphemeral` (`crates/ironhermes-core/src/browser_profile.rs:94-101`) | accept | An attacker able to write the profile dir could plant a `SingletonLock` targeting a live pid to force the ephemeral-profile fallback. Result: browser launch still succeeds (chromiumoxide's own temp dir), only this session's cookie / login persistence is lost, and a `warn!` line records the locked profile dir + pid. A cross-process browser mutex / lock-takeover protocol is explicitly deferred (CONTEXT § Deferred Ideas). See Accepted Risks **AR-2**. | closed |

*Status: open · closed*
*Disposition: mitigate (implementation required) · accept (documented risk) · transfer (third-party)*

## Accepted Risks Log

| Risk ID | Threat Ref | Rationale | Accepted By | Date |
|---------|------------|-----------|-------------|------|
| AR-1 | T-26.3.2-01 | TOCTOU between `is_pid_alive(pid)` returning dead and the subsequent `remove_file` calls in `cleanup_sentinels`. The exploit window is ~microseconds (three `unlink` syscalls); a freshly-launched Chromium takes several seconds to reach the point where it creates `SingletonLock`. The only alternative — never cleaning the stale lock — leaves authenticated browser automation permanently broken after a crash / `kill -9` / reboot. No code mitigation; documented and accepted. | Brad Wilson (plan-time threat model, `26.3.2-01-PLAN.md`) | 2026-05-12 |
| AR-2 | T-26.3.2-03 | A writer of `$HERMES_HOME/browser-profile` can plant a live-pid `SingletonLock` and force the ephemeral-profile fallback — loss of cookie / login persistence for that session only; the browser still launches and a `warn!` is logged. On a single-user machine the only such writer is the operator. A cross-process browser mutex / lock-takeover protocol is out of scope for this phase and explicitly deferred. Documented and accepted. | Brad Wilson (plan-time threat model, `26.3.2-01-PLAN.md`) | 2026-05-12 |
