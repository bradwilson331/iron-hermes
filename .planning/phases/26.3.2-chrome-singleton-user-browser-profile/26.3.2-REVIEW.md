# Phase 26.3.2: Code Review Report

**Reviewed:** 2026-05-12T17:38:13Z
**Depth:** standard
**Files Reviewed:** 4
**Status:** issues_found

## Summary

This phase adds `ironhermes_core::browser_profile::reconcile_singleton_lock(&Path) -> SingletonOutcome`, a dep-free helper that detects and cleans a stale Chromium `SingletonLock` symlink in `$HERMES_HOME/browser-profile`, and wires it into both `BrowserSession::spawn()` call sites before `builder.user_data_dir(...)`. The `ironagent-tools-api/src/browser_session.rs` twin was confirmed byte-identical to `ironhermes-tools/src/browser_session.rs` (verified via `diff`), consistent with its intentional exclusion from the cargo workspace.

The module is well-documented, the test fixtures are thoughtful (e.g. the `i32::MAX as u32` vs `u32::MAX` PID-wrap reasoning), and `read_link`/symlink semantics are handled correctly (no follow, broken-symlink-as-normal). No security vulnerabilities or hard crashes were found.

The findings below center on the **failure-on-error** behavior of the two external-command probes (`kill`, `hostname`) and one cross-host edge case: in each of those paths, an unexpected failure or environment quirk can cause a **live** `SingletonLock` to be deleted, which defeats the very purpose of Chromium's singleton mechanism (concurrent use of one profile dir → potential `Local State` / `Cookies` SQLite corruption). These are environment-dependent and the design notes (D-02/D-03, OQ-1) explicitly accepted some of this, but they warrant hardening.

## Warnings

### WR-01: `hostname` CLI output diverging from Chromium's `gethostname()` deletes a live lock

**File:** `crates/ironhermes-core/src/browser_profile.rs:77-91`
**Issue:** The D-03 host check treats `lock_host != our_host` as definitively stale and immediately calls `cleanup_sentinels(profile_dir)` — before any liveness check. Chromium writes the lock host from `net::GetHostName()` (i.e. `gethostname(2)`). `current_hostname()` shells out to the bare `hostname` command. In the common case these agree, but they can diverge: a shell function/alias named `hostname`, a containerized environment where `/etc/hostname` is FQDN while `gethostname()` returns the short name (or vice versa), or a trailing-dot FQDN. When they diverge on the *same* machine, a chrome process that is alive and holding the lock has its `SingletonLock`/`SingletonSocket`/`SingletonCookie` deleted out from under it, the new chrome binds the same `user_data_dir`, and two chromiums race on the same profile DB — exactly the corruption the lock exists to prevent. Note the test fixtures cannot catch this because `host()` is built from `current_hostname()` itself.
**Fix:** Do not let a host *mismatch* alone trigger cleanup. Either (a) only treat a mismatch as informative and still fall through to the pid-liveness check, or (b) when a mismatch is detected, require the pid-liveness check to also report dead before cleaning:
```rust
if let (Some(lock_host), Some(our_host)) = (parse_hostname_from_target(&target_str), current_hostname()) {
    if lock_host != our_host.as_str() {
        // Different host *or* hostname-CLI/gethostname disagreement — only safe to
        // clean if the pid is also dead. A cross-host pid number is meaningless
        // locally, but a same-host disagreement must not delete a live lock.
        if is_pid_alive(pid) {
            warn!(/* live lock, host string mismatch — fall back to ephemeral */);
            return SingletonOutcome::UseEphemeral;
        }
        warn!(/* stale, cross-host */);
        cleanup_sentinels(profile_dir);
        return SingletonOutcome::UseProfile;
    }
}
```

### WR-02: external-command probes resolve via `PATH`; spawn failure → live lock treated as dead

**File:** `crates/ironhermes-core/src/browser_profile.rs:158-189`
**Issue:** `is_pid_alive` (non-Linux) does `Command::new("kill")` and `current_hostname()` does `Command::new("hostname")` — both resolved through the inherited `PATH`. The doc comment on `is_pid_alive` even claims it "shells out to `/bin/kill`", but the code uses the bare name. If `PATH` is empty/unusual (cron, restricted systemd unit, sandbox), `kill` fails to spawn → `.unwrap_or(false)` → the pid is reported **dead** → `cleanup_sentinels` deletes a live `SingletonLock`. Same exposure exists for `hostname` returning `None`, which then *also* drops into the pid-liveness branch (and from there into the same `unwrap_or(false)` path). The "spawn failure → treat as dead" comment calls this a "safe conservative choice", but for *this* function dead is the dangerous direction (it triggers deletion), not the safe one.
**Fix:** Use absolute paths (`/bin/kill`, and `/bin/hostname` — falling back to `uname -n` / reading `/etc/hostname` is unnecessary if you keep it dep-free, but at minimum pin the binary). More importantly, on spawn/probe failure prefer the **non-destructive** outcome — return `UseEphemeral` (or treat the lock as "live/unknown" and skip cleanup) rather than proceeding to delete:
```rust
fn is_pid_alive(pid: u32) -> Option<bool> { // None = "could not determine"
    #[cfg(target_os = "linux")]
    { Some(std::path::Path::new(&format!("/proc/{pid}")).exists()) }
    #[cfg(not(target_os = "linux"))]
    {
        match std::process::Command::new("/bin/kill").args(["-0", &pid.to_string()]).status() {
            Ok(s) => Some(s.success()),
            Err(_) => None, // caller: treat as live → UseEphemeral, do NOT clean
        }
    }
}
```

### WR-03: `kill -0` PID-reuse + EPERM (non-Linux) — false "live" wastes the profile, false "dead" deletes a live lock

**File:** `crates/ironhermes-core/src/browser_profile.rs:159-176`
**Issue:** Two distinct edge cases share this code path on macOS/BSD:
1. **PID reuse → false live:** the original chrome exited and the kernel recycled its PID for an unrelated process; `kill -0 <pid>` succeeds → `UseEphemeral`. Harmless (no corruption) but the persistent profile is needlessly abandoned and the stale sentinels are never cleaned, so the condition recurs on every spawn until something else clears them.
2. **EPERM → false dead:** if the lock was written by a chrome owned by a *different* user (e.g. a `sudo`-launched run, or a shared `$HERMES_HOME`), `/bin/kill -0` returns exit 1 (EPERM) which this code maps to "dead" → `cleanup_sentinels` deletes a live lock. The inline comment cites RESEARCH OQ-1 ("Chromium is always spawned by the same user") as the reason this is acceptable; that assumption is the only thing standing between this code and profile corruption, and it is not enforced anywhere.
**Fix:** Distinguish ESRCH (dead) from EPERM (alive-but-other-user) instead of collapsing both to "dead". `/bin/kill` does not expose errno cleanly, so capture stderr or use `kill -0` semantics carefully; alternatively, on any non-zero exit other than a recognized ESRCH signature, treat as **alive/unknown → `UseEphemeral`** (non-destructive). For case 1, consider also clearing `SingletonSocket`/`SingletonCookie` when falling back to ephemeral so the directory does not accumulate cruft — or at least document that ephemeral fallback is sticky until manual cleanup.

## Info

### IN-01: `create_dir_all` runs even on the `UseEphemeral` branch

**File:** `crates/ironhermes-tools/src/browser_session.rs:86-92` (and the identical twin `crates/ironagent-tools-api/src/browser_session.rs:86-92`)
**Issue:** When `reconcile_singleton_lock` returns `UseEphemeral`, the code intentionally does *not* bind `user_data_dir` to the builder — yet the unconditional `std::fs::create_dir_all(&user_data_dir)` immediately below still runs against that directory. It is harmless (the directory must already exist for a lock to have been found there) but it is misleading: a reader sees "we're going ephemeral" then "create the persistent dir anyway".
**Fix:** Move the `create_dir_all` into the `UseProfile` match arm so directory creation is co-located with the decision to use it.

### IN-02: doc comment says `/bin/kill` but code uses bare `kill`

**File:** `crates/ironhermes-core/src/browser_profile.rs:155, 170`
**Issue:** The function-level doc says "On other Unix: shells out to `/bin/kill -0 <pid>`" but the implementation is `Command::new("kill")`. Same drift between the module-level doc claim and the actual call. (Functionally folded into WR-02; noted separately so the comment gets corrected even if WR-02 is deferred.)
**Fix:** Either change the call to `Command::new("/bin/kill")` (preferred — see WR-02) or fix the comments to say "the `kill` command resolved via `PATH`".

### IN-03: `lib.rs` re-export placement breaks the alphabetical grouping

**File:** `crates/ironhermes-core/src/lib.rs:51`
**Issue:** `pub use browser_profile::{SingletonOutcome, reconcile_singleton_lock};` is inserted between the `skills::build_skill_search_paths` re-export and the `ssrf` re-export, rather than near the top with the other early-alphabet re-exports. The `pub mod browser_profile;` declaration on line 1 *is* correctly first. Purely cosmetic; flagged only because the file otherwise keeps a consistent ordering.
**Fix:** Move the `pub use browser_profile::...` line up next to `pub use config::...` / before `pub use commands::...` to match the module-declaration ordering.
