<domain>
## Phase Boundary

Phase 26.3 made `BrowserSession::spawn()` launch chromiumoxide against a **persistent** profile directory (`$HERMES_HOME/browser-profile`, or `browser.user_data_dir` when set). Chromium's `ProcessSingleton` machinery writes three sentinel files into that directory at launch and removes them only on a **clean** exit:

- `SingletonLock` — a POSIX symlink whose target is `<hostname>-<pid>` (the host + process that holds the lock)
- `SingletonSocket` — the singleton IPC socket (symlink to a socket under `/tmp/...`)
- `SingletonCookie` — a symlink carrying a random validation cookie

After a crash, `kill -9`, an OS reboot, or two overlapping browser sessions (e.g. cron agent + interactive agent), a **stale** `SingletonLock` is left behind. From then on **every** `browser_*` tool call aborts:

```
[ERROR:process_singleton_posix.cc:345] Failed to create $HERMES_HOME/browser-profile/SingletonLock: File exists (17)
[ERROR:chrome_main_delegate.cc:515] Failed to create a ProcessSingleton for your profile directory. This means that
running multiple instances would start multiple browser processes rather than opening a new window in the existing
process. Aborting now to avoid profile corruption.
```

Observed in production: a cron agent's `browser_navigate` calls failed twice ~7s apart with exactly this error against `/Users/twilson/.ironhermes/browser-profile/SingletonLock`. Authenticated browser automation is permanently broken until someone hand-deletes the file.

**This phase makes browser launch resilient to a stale singleton lock.** Before launching chromiumoxide:
1. If `SingletonLock` exists and is **stale** (no live process on this host owns the pid) → remove `SingletonLock` + `SingletonSocket` + `SingletonCookie`, log a `warn!`, then launch normally against the persistent profile.
2. If `SingletonLock` exists and is held by a **live** process → do NOT touch it. Fall back to a process-scoped ephemeral profile (the pre-26.3 behavior: don't call `.user_data_dir(...)` at all), log a `warn!` explaining the session has no persistence, and launch.
3. If `SingletonLock` does not exist → launch normally (no-op, current happy path preserved).

The fix lands in both copies of `spawn()` — `crates/ironhermes-tools/src/browser_session.rs` and `crates/ironagent-tools-api/src/browser_session.rs` — via a shared helper in `ironhermes-core` (both crates already depend on `ironhermes-core`; this avoids the existing copy-paste drift between the two `browser_session.rs` files).

### Out of scope
- Pointing the agent's browser at the operator's **real** Chrome profile (`~/Library/Application Support/Google/Chrome/Default`) so it inherits their logins — that is a separate, larger feature (profile mixing, Chrome version coupling, "close your Chrome first" UX). Deferred.
- A global mutex serializing browser tool calls across hermes processes so two agents never race for the profile. The live-lock ephemeral fallback (decision D-05) is the chosen mitigation for now; a cross-process mutex is deferred.
- Periodic background cleanup of stale locks. Cleanup happens lazily, at spawn time only.
- Windows `ProcessSingleton` (named-mutex based, not a symlink). On Windows the helper is a best-effort no-op — see D-09.
- New config keys. No `config.yaml` schema change.

</domain>

<decisions>
## Implementation Decisions

These decisions D-01..D-10 are the requirements set for this phase (no entry in REQUIREMENTS.md).

### Stale-lock detection
- **D-01:** Before `Browser::launch(...)`, inspect `<user_data_dir>/SingletonLock`. On POSIX it is a symlink — read its target with `std::fs::read_link`. The target is `<hostname>-<pid>`; the pid is the substring after the last `-`. If the path does not exist, or is not a symlink, or the target does not parse into a pid → treat the lock as **absent/unparseable** and proceed to launch normally (do not delete anything in the unparseable case — log `debug!` and move on).
- **D-02:** "Stale" = the pid is **not a live process on this host**. Liveness check must be **dependency-free** (no new crates added to either crate's Cargo.toml). Acceptable approach: shell out via `std::process::Command::new("kill").args(["-0", &pid])` and treat exit-0 as alive, any non-zero / spawn failure as dead. The researcher may propose an equivalent dep-free check; whatever is chosen must compile cleanly on macOS + Linux and be a no-op on Windows.
- **D-03:** Hostname mismatch is **also stale**: if the symlink target's host segment does not match the current machine's hostname, the lock was written by a different machine sharing the profile dir (e.g. on a network FS) — treat as stale and clean it. (Dep-free hostname read: `std::process::Command::new("hostname")` or read `/proc/sys/kernel/hostname` on Linux; researcher to confirm. If the hostname can't be determined, fall back to the pid-liveness check only.)

### Cleanup behaviour
- **D-04:** When the lock is stale, remove all three sentinels in this order — `SingletonLock`, then `SingletonSocket`, then `SingletonCookie` — using `std::fs::remove_file` (works on symlinks). Each removal is **best-effort**: a failure on any one is logged at `warn!` and does not abort the launch. After cleanup, proceed to launch against the persistent profile.
- **D-05:** When the lock is held by a **live** process, do NOT delete it. Fall back to a process-scoped ephemeral profile by **not** calling `builder.user_data_dir(...)` at all (chromiumoxide then uses its own temp dir — the pre-26.3 behavior). Emit one `warn!` line that names the locked profile dir and states the session will not persist cookies/login state. The launch must still succeed.
- **D-06:** Cleanup is **never unconditional** — a stale check always runs first. Blindly deleting `SingletonLock` while a real chromium owns the profile is exactly the "profile corruption" scenario Chromium's abort is protecting against, so the live-process branch must be the gate.

### Where the code lives
- **D-07:** Add a shared helper to `ironhermes-core` — e.g. `ironhermes_core::browser_profile::reconcile_singleton_lock(profile_dir: &Path) -> SingletonOutcome` where `SingletonOutcome` is `UseProfile` (no lock, or stale lock cleaned) or `UseEphemeral` (live lock — caller must skip `.user_data_dir`). Both `browser_session.rs` files call it right before the `.user_data_dir(...)` decision and branch on the result. Exact names/signature are the planner's call; the constraint is **one implementation, called from both crates** — no third copy-paste.
- **D-08:** The `$HERMES_HOME/browser-profile` default-path resolution stays exactly where it is in each `spawn()` (the `match config.user_data_dir.as_deref()` block, Phase 26.3). The helper takes the already-resolved `&Path`; it does not re-derive the default.

### Portability & logging
- **D-09:** On Windows, the helper is a best-effort no-op that returns `UseProfile` unconditionally (Windows `ProcessSingleton` uses a named mutex, not a `SingletonLock` symlink; the failure mode this phase fixes does not occur there). Guard the POSIX-specific symlink/`kill -0` logic behind `#[cfg(unix)]` (or `#[cfg(not(windows))]`).
- **D-10:** All operator-visible outcomes go through `tracing` at the right level: `warn!` for "cleaned a stale lock" and for "fell back to ephemeral profile (locked by live pid N)"; `debug!` for "no lock present" and "lock unparseable, proceeding". No `println!`/`eprintln!`.

### Claude's Discretion
- The exact module path / struct / enum names for the helper (`browser_profile.rs` vs adding to `constants.rs`; `SingletonOutcome` vs `ProfileMode`; etc.).
- Whether the liveness probe is `kill -0` via `Command`, or a `#[cfg(target_os="linux")]` `/proc/<pid>` check with a `kill -0` fallback elsewhere — pick the simplest dep-free thing that works on macOS + Linux.
- Whether to also remove `lockfile`/`SingletonLock`-adjacent leftovers Chromium sometimes writes; minimum is the three named in D-04.
- Test mechanics: how to fabricate a "stale lock" in a `tempfile::TempDir` (create a symlink named `SingletonLock` pointing at `<hostname>-<a-pid-that-is-definitely-dead>` such as `99999999`) and a "live lock" (symlink pointing at `<hostname>-<std::process::id()>`). Researcher/planner choose; see Open Questions.

</decisions>

## Existing Code Insights

- Both `spawn()` functions are identical through the `user_data_dir` resolution + `builder.user_data_dir(...)` + `create_dir_all(...)` block. The new branch must be applied to both, which is exactly why D-07 puts the logic in `ironhermes-core` (one impl, two call sites) instead of pasting it twice.
- `find_chromium_binary()` is *also* duplicated across the two crates — evidence that copy-paste drift is a real risk here; the shared-helper decision (D-07) is deliberate.
- chromiumoxide 0.9: omitting `.user_data_dir(...)` makes it create a fresh temp profile per launch (the pre-26.3 default) — that is the "ephemeral fallback" of D-05; nothing extra is needed to opt into it, you just don't call the setter.
- `tracing::{debug, warn}` are already imported in both `browser_session.rs` files.
- `get_hermes_home()` reads `IRONHERMES_HOME` env or `~/.ironhermes` — tests that want a throwaway profile dir should use `tempfile::TempDir`, not muck with `IRONHERMES_HOME`.

</code_context>

<specifics>
## Specific Ideas

- The exact production failure to reproduce in a regression test (paraphrased): `browser_navigate` → `chromium launch failed: Browser process exited with status ... stderr: "Failed to create .../browser-profile/SingletonLock: File exists (17) ... Failed to create a ProcessSingleton ... Aborting now to avoid profile corruption."` — after this phase, a stale `SingletonLock` must NOT produce this; either it gets cleaned (persistent profile used) or the launch silently degrades to an ephemeral profile.
- `SingletonLock` symlink target format on macOS/Linux: `"<hostname>-<pid>"`, e.g. `"Brads-MacBook-Pro.local-20704"` (note: hostnames can contain `-`, so split on the **last** `-` to get the pid).
- Three sentinels to clean: `SingletonLock`, `SingletonSocket`, `SingletonCookie` (Chromium creates exactly these in the profile root).

</specifics>

<open_questions>
## Open Questions (for the researcher)

- **OQ-1:** Cleanest dep-free liveness probe on macOS + Linux that doesn't add `libc`/`nix`/`sysinfo`. Candidates: `Command::new("kill").args(["-0", pid]).status()`; `#[cfg(target_os="linux")] Path::new("/proc/<pid>").exists()`. Confirm one, note the fallback.
- **OQ-2:** Dep-free hostname read. Candidates: `Command::new("hostname")`; `std::fs::read_to_string("/proc/sys/kernel/hostname")` (Linux only); env `HOSTNAME` (not always set). If none is reliable, D-03 degrades to "pid-liveness only" — confirm that's acceptable.
- **OQ-3:** Does chromiumoxide ever pass `--headless=new` in a way that *skips* `ProcessSingleton`? The production log proves it does NOT skip it for the current config, but confirm there's no `headed`-vs-`headless` divergence so the fix covers both modes.
- **OQ-4:** Can a test reliably create a symlink named `SingletonLock` in a `TempDir` on the CI OS (`std::os::unix::fs::symlink`)? If symlink creation needs privileges anywhere, propose a regular-file fallback for the "lock present but unparseable" test path.

</open_questions>
