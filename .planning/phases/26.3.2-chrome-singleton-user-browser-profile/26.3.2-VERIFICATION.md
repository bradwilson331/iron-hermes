# Phase 26.3.2: Chrome SingletonLock Resilience — Verification Report

**Phase Goal:** Make browser launch resilient to a stale `SingletonLock` in `$HERMES_HOME/browser-profile`. Before launching chromiumoxide, detect whether the lock is stale (no live process owns the pid on this host) and if so clear `SingletonLock` + `SingletonSocket` + `SingletonCookie`; if held by a live process, fall back to a process-scoped ephemeral profile and emit a clear warn!. Apply the fix to both copies of `spawn()` via a shared helper in `ironhermes-core`.

**Accepted warnings from code review (non-blocking per REVIEW.md):**
- WR-01: `hostname` CLI divergence from Chromium's `gethostname()` could trigger cleanup of a live lock on the same machine when hostnames disagree (container FQDN vs short name). Design-accepted; D-03 + fallback-to-pid logic is explicit in the spec.
- WR-02: `kill` and `hostname` resolved via PATH; spawn failure defaults to "dead" (destructive direction). Design-accepted per CONTEXT.md OQ-1 and OQ-2.
- WR-03: EPERM from `kill -0` on macOS treated as dead (same as ESRCH). Design-accepted; RESEARCH OQ-1 assumes same-user Chromium.
