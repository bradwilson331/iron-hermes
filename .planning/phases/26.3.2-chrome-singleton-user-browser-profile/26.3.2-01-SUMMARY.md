## What Was Built

### New: `crates/ironhermes-core/src/browser_profile.rs`

`pub enum SingletonOutcome { UseProfile, UseEphemeral }` with `#[derive(Debug, Clone, Copy, PartialEq, Eq)]` and `pub fn reconcile_singleton_lock(profile_dir: &Path) -> SingletonOutcome`.

The function dispatches to a `#[cfg(unix)]` `reconcile_unix` private implementation and a `#[cfg(not(unix))]` no-op returning `UseProfile` (D-09: Windows is silent pass-through, not a panic — unlike gateway/pid.rs which panics).

Unix implementation follows D-01..D-06:
- `std::fs::read_link` on `SingletonLock` — NotFound or non-symlink error → `debug!` + `UseProfile` (no delete)
- Parse pid via `rfind('-')` on symlink target — parse failure → `debug!` + `UseProfile` (no delete)  
- D-03 hostname check via `Command::new("hostname")` — mismatch → `warn!` + `cleanup_sentinels` + `UseProfile`
- D-02 pid liveness: Linux uses `/proc/<pid>` (no subprocess), other Unix uses `Command::new("kill").args(["-0", pid])` — alive → `warn!` + `UseEphemeral` (no delete, D-05/D-06)
- Stale pid → `warn!` + `cleanup_sentinels` + `UseProfile` (D-04)

`cleanup_sentinels` loops over `["SingletonLock", "SingletonSocket", "SingletonCookie"]` calling `std::fs::remove_file` best-effort, logging `warn!` on non-NotFound errors.

8 unit tests (SL-01..SL-07 + parse_hostname_from_target extra):
- SL-01: no lock → UseProfile
- SL-02: unparseable regular file → UseProfile, not deleted
- SL-03: unparseable symlink bad target → UseProfile, not deleted
- SL-04: stale lock (dead pid) → UseProfile + SingletonLock removed
- SL-05: stale lock → all 3 sentinels removed
- SL-06: live lock (current pid) → UseEphemeral + symlink preserved
- SL-07: parse_singleton_target handles hostnames with embedded dashes (e.g. `Brads-MacBook-Pro.local-20704`)
- Extra: parse_hostname_from_target handles dashes

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Test fixtures used hardcoded "testhost" hostname causing D-03 host mismatch in all tests**

- **Found during:** Task 1 verification (cargo test run)
- **Issue:** The initial test fixtures used `format!("testhost-{}", ...)` as symlink targets. The D-03 hostname check in `reconcile_unix` compared the lock's hostname segment against the real machine hostname (`Brads-MacBook-Pro.local`). Since `"testhost" != "Brads-MacBook-Pro.local"`, D-03 always fired, treating every lock as "stale from different host". This caused: (a) `live_lock` test to return `UseProfile` instead of `UseEphemeral`, (b) `unparseable_bad_target` test to trigger cleanup of `"nodash-nopid"` (because `"nodash"` was parsed as a hostname mismatch before the unparseable check ran).
- **Fix:** Changed fixture helpers to call `current_hostname()` (the same function `reconcile_unix` uses) to get the real hostname, prepending it to the symlink target. This ensures the D-03 check passes and tests fall through to D-02 pid-liveness.
- **Files modified:** `crates/ironhermes-core/src/browser_profile.rs` (test fixtures only)
- **Commit:** e0760ff7

**2. [Rule 1 - Bug] Test assertions used Path::exists() which follows symlinks and returns false for dangling symlinks**

- **Found during:** Task 1 verification (after fix 1, SL-06 still failed)
- **Issue:** `dir.path().join("SingletonLock").exists()` follows the symlink and checks if the *target* exists as a real file. The symlink target `"Brads-MacBook-Pro.local-28327"` is a relative path that doesn't point to any real file, so `exists()` returned `false` even though the symlink directory entry was still present.
- **Fix:** Changed symlink-preservation assertions (SL-03, SL-06) to use `std::fs::symlink_metadata(...).is_ok()` (lstat) which checks the symlink entry itself without following it. Deletion assertions (SL-04, SL-05) keep `!dir.path().join(...).exists()` since after `remove_file` the entry is gone regardless.
- **Files modified:** `crates/ironhermes-core/src/browser_profile.rs` (test assertions only)
- **Commit:** e0760ff7
