## A · TERMINATED-HOLD specifics

### A.1 — HOLD duration

| Option | Description | Selected |
|--------|-------------|----------|
| 5 seconds | Roadmap entry's proposed N=5s. Long enough to register visually for a user glancing at the page; short enough that stale cards don't clutter. | ✓ |
| 3 seconds | Faster cycling. Less screen clutter when many agents end at once. Risk: user may miss short-lived agents. | |
| 10 seconds | Heavier residency. Good for slow human cadence; bad if dozens of agents fire and end in rapid succession. | |

**User's choice:** 5 seconds (Recommended)
**Notes:** Locks D-01.

### A.2 — Termination kinds covered

| Option | Description | Selected |
|--------|-------------|----------|
| All terminations — natural + kill + interrupt | Consistent UX. Kill click → ENDED card stays for N seconds with KILLED status; same fade as natural exit. The diff-based implementation captures all three uniformly with no extra branching. | ✓ |
| Only natural / interrupt-driven — kill is instant | Kill button clears the card immediately because the user just explicitly killed it. Natural and interrupt-then-finalize paths still HOLD. Requires per-action branching in the diff logic. | |
| Only natural terminations | HOLD only for agents that ended on their own (Max iterations, self-finish). Both kill and interrupt clear instantly. Loses the 'something just ended' feedback for the non-natural paths. | |

**User's choice:** All terminations — natural + kill + interrupt (Recommended)
**Notes:** Locks D-02. Drives D-12's PRUNE behavior and D-09's single-pill decision.

## B · Poll cadence + visibility

### B.1 — Plan 01 poll interval

| Option | Description | Selected |
|--------|-------------|----------|
| 1500 ms | Roadmap entry's proposed cadence. ~0.67 fetches/sec/tab. New agents visible in ≤1.5s. Server cost is trivial for the read-only api_agents_list. | ✓ |
| 1000 ms | Snappier perceived liveness. ~1 fetch/sec/tab. Doubles server-side request volume vs 1500ms; still negligible for local app. | |
| 2500 ms | Lighter touch. Better for many simultaneous tabs / weak hardware. Trade: short-lived agents (<2.5s lifetime) more likely to be missed entirely if Plan 02 isn't merged yet. | |

**User's choice:** 1500 ms (Recommended)
**Notes:** Locks D-03.

### B.2 — Page visibility integration

| Option | Description | Selected |
|--------|-------------|----------|
| Yes — pause on hidden, resume on visible | Hook the Page Visibility API. Eliminates pointless fetches when the user isn't looking; standard web practice. Adds ~10 LoC. | ✓ |
| No — poll continuously regardless of tab state | Simpler (no visibility wiring). Pages stays warm when refocused. Trade: every backgrounded tab keeps hitting the server forever. | |

**User's choice:** Yes — pause on hidden, resume on visible (Recommended)
**Notes:** Locks D-04. Uses `web_sys::Document::hidden()` + visibilitychange listener.

### B.3 — Action-pause during RPCs

| Option | Description | Selected |
|--------|-------------|----------|
| Yes — pause while killing()/interrupting() is true | Prevents the poll from clobbering per-card transient signal state mid-RPC. Already implicit in the roadmap entry's 'paused while a per-card kill/interrupt is in flight'. | ✓ |
| No — keep polling; rely on signal-set ordering | Simpler. Trusts that .restart() doesn't actually mutate per-card local signals (true today). Risk: future-change race. | |

**User's choice:** Yes — pause while killing()/interrupting() is true (Recommended)
**Notes:** Locks D-05.

## C · Event-driven path scope (PIVOT)

### C.1 — Plan 02 transport channel

| Option | Description | Selected |
|--------|-------------|----------|
| Reuse chat ws — plumb SubagentProgressCallback | Extend the existing /ws chat connection with a new event type. Wires state.rs:124 progress_callback to a callback that forwards to the ws send half. One ws connection, no new server route, ~80 LoC. | ✓ |
| Dedicated /api/agents/events ws | New server route + new client ws_use. Isolated from chat lifecycle. Trade: second ws connection per tab, duplicate keepalive/reconnect logic, ~150 LoC. | |
| Server-Sent Events (SSE) | GET /api/agents/stream with text/event-stream. Simpler than ws (one-way push). Trade: less reuse — adds a third event-transport pattern. | |
| Drop Plan 02 — ship Plan 01 only | 1500ms poll is good enough; defer event-driven to a later phase. Trade: <1s update latency target not met. | |

**User's choice:** Reuse chat ws — plumb SubagentProgressCallback (Recommended)
**Notes:** Locks D-06. Critical: this fork came from the discovery that the roadmap entry's claim "SubagentProgress events already flow through the chat-page WebSocket" was wrong — `state.rs:124` is `progress_callback: None` and `ws.rs:195` is `ToolProgressCallback`. The plumbing work is now correctly in scope.

### C.2 — Event payload shape

| Option | Description | Selected |
|--------|-------------|----------|
| Counter only — Signal<u64> increments on any subagent event | Smallest contract: server sends `{"kind": "subagent_event"}`; client increments counter; ScreenAgents use_effect calls .restart() on increment. Forces fresh GET — stays consistent with poll path. ~5 byte event. | ✓ |
| Full AgentInfo on Started, id on Completed | Server sends complete state inline; client updates in-memory list directly. Faster perceived (no extra GET). Trade: two divergent code paths. | |
| Event kind + id only — client re-fetches | `{"kind": "started"|"completed", "id": "sub_xxx"}`. Lets the client log which agent changed. Still triggers re-fetch. Middle ground. | |

**User's choice:** Counter only — Signal<u64> increments on any subagent event (Recommended)
**Notes:** Locks D-07. Keeps poll and event paths converging on the same GET — no diff-logic divergence.

### C.3 — Ws-disconnect fallback

| Option | Description | Selected |
|--------|-------------|----------|
| Widen poll to 5s while ws connected; tighten back to 1500ms when ws drops | When ws is alive, events drive most updates so poll is the slow fallback. When ws breaks, poll becomes the only signal so it speeds up. State via is_ws_connected: Signal<bool>. | ✓ |
| Keep 1500ms poll always; events are pure additive | Poll stays at Plan-01 cadence always. Events just bump the resource an extra time when they fire. Simpler. Trade: 2x request volume on happy path. | |
| Drop poll entirely when ws connected; reconnect-or-bust | Ws is the source of truth, poll is OFF while connected. Trade: missed Started event = silent failure. | |

**User's choice:** Widen poll to 5s safety net while ws connected; tighten back to 1500ms when ws drops (Recommended)
**Notes:** Locks D-08. Requires `is_ws_connected: Signal<bool>` driven by the chat-ws lifecycle handler in `warp_hermes.rs`.

## D · Terminal visual state

### D.1 — Status pill labeling

| Option | Description | Selected |
|--------|-------------|----------|
| Single 'ENDED' pill for all terminations | One label, simple. `.pill.gray` (or new `.pill.ended`). Matches D-02. Doesn't distinguish natural/kill/interrupt — visual sameness is the point. | ✓ |
| Distinguish: ENDED / KILLED / INTERRUPTED | Three pills with different copy + maybe different colors. More information. Trade: requires tracking termination kind client-side and recording in `recently_terminated`. | |
| ENDED + a small icon hint | ENDED pill plus a tiny icon (✘ killed, ⏸ interrupted, none natural). Lightweight differentiation. | |

**User's choice:** Single 'ENDED' pill for all terminations (Recommended)
**Notes:** Locks D-09.

### D.2 — Body + buttons

| Option | Description | Selected |
|--------|-------------|----------|
| Dimmed body (opacity ~0.55), action buttons hidden | Clear visual fade signal. Card stays visible to confirm 'this just ended' but doesn't invite interaction. Buttons removed (not just disabled) cuts visual noise. | ✓ |
| Dimmed body, buttons disabled (greyed) | Buttons remain visible but inert. More cluttered than hidden; some users prefer the affordance memory. | |
| No dim, just the pill change | Card looks identical except for ENDED pill. Trade: easy to miss. | |

**User's choice:** Dimmed body (opacity ~0.55), action buttons hidden (Recommended)
**Notes:** Locks D-10.

### D.3 — uptime_secs handling

| Option | Description | Selected |
|--------|-------------|----------|
| Freeze at termination time | Snapshot uptime when the diff captures the agent into recently_terminated. Card shows e.g. '12s' frozen for the 5s HOLD. | ✓ |
| Keep ticking (wall-clock) until card disappears | Uptime increases for 5 more seconds while the card holds. Trade: misleading — agent isn't running anymore. | |

**User's choice:** Freeze at termination time (Recommended)
**Notes:** Locks D-11.

## E · Edge cases (added round)

### E.1 — PRUNE + HOLD interaction

| Option | Description | Selected |
|--------|-------------|----------|
| Yes — PRUNE clears both server-side ended agents AND local recently_terminated map | Consistent UX: PRUNE ENDED means 'remove all ended cards from view' regardless of source. The button empties `recently_terminated.write() = HashMap::new()` plus the existing POST. | ✓ |
| No — PRUNE only affects server-side; HOLD cards expire naturally | PRUNE behavior unchanged. Trade: user clicks PRUNE expecting all ENDED cards gone but a freshly-HOLDed card lingers up to 5s after. | |
| PRUNE accelerates HOLD — truncates remaining time to 500ms | Compromise: speeds up the fade. Adds animation polish at the cost of one more code path. | |

**User's choice:** Yes — PRUNE clears both server-side ended agents AND local recently_terminated map (Recommended)
**Notes:** Locks D-12.

### E.2 — ID replay edge case

| Option | Description | Selected |
|--------|-------------|----------|
| Re-running wins — immediately remove from recently_terminated, show as live card | On next fetch, the diff sees the id is in the new list. The existing recently_terminated entry is dropped without animation. Card appears as running again. Avoids confusing 'ended card next to running card with same id' state. | ✓ |
| Treat as new agent — keep the ENDED HOLD AND show the new running card | Both visible side-by-side for up to 5s. Trade: same id rendered twice — Dioxus key collision risk. | |
| Keep ENDED — ignore the new running entry until HOLD expires | Stickier: HOLD wins. Trade: user sees a 'running' agent the server reports but it's not visible on the page for up to 5s. | |

**User's choice:** Re-running wins — immediately remove from recently_terminated, show as live card (Recommended)
**Notes:** Locks D-13.

### E.3 — CSS class strategy

| Option | Description | Selected |
|--------|-------------|----------|
| New modifier classes — .card.is-ended, .pill.ended | Adds two well-named state modifiers. `.card.is-ended` sets the opacity + hides `.card-footer` buttons via CSS. `.pill.ended` is the gray ENDED variant. Matches existing `.is-active`/`.is-cmd`/`.is-ai` modifier convention. | ✓ |
| Inline style attrs only — no new CSS | Zero CSS file changes. Tighter diff. Trade: visual logic scattered across the rsx! macro; harder to theme/restyle later. | |
| Reuse existing .pill.gray + a generic .is-dim class | If a `.pill.gray` already exists in design tokens. Trade: planner needs to grep CSS to confirm; semantic meaning of 'gray pill' may already be claimed. | |

**User's choice:** New modifier classes — .card.is-ended, .pill.ended (Recommended)
**Notes:** Locks D-14.

## Discovery during discussion

**Critical correction recorded:** The original roadmap entry for Phase 26.7.1 claimed that `SubagentProgress::Started`/`Completed` events already flow through the chat-page WebSocket. Investigation during this discussion confirmed they do NOT — `crates/iron_hermes_ui/src/server/state.rs:124` explicitly sets `progress_callback: None`, and `crates/iron_hermes_ui/src/server/ws.rs:195` only wires `ToolProgressCallback`. Plan 02 must therefore plumb a `SubagentProgressCallback` end-to-end (server → ws → client). This finding is locked as D-06 and added the chat-ws plumbing work back into Plan 02's scope. The fix vs the entry is approximately one paragraph of effort; the alternative (dedicated /api/agents/events ws) was considered and rejected for transport-reuse cleanliness.
