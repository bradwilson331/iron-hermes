# Phase 26.7.1: Agents page live updates - Context

**Gathered:** 2026-05-18
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 26.7.1 makes the web UI Agents screen reflect live `SubagentRegistry` state in real time, with terminated agents shown for **N = 5 seconds** before removal. Today (post-26.7 close), `crates/iron_hermes_ui/src/components/hermes_app/screens/agents.rs::ScreenAgents` calls `use_server_future(api_agents_list)` exactly once and only re-fetches when the user clicks PRUNE/KILL — short-lived subagents register, run, and unregister entirely between renders, so they never appear on screen (user-observed during the Phase 26.7-07 UAT).

**In scope — 2 plans, 1 wave:**
1. **Plan 01 — periodic-poll baseline + client-side TERMINATED-HOLD-N fade** (~80 LoC): `use_future` interval calling `agents_resource.restart()`; `recently_terminated: Signal<HashMap<String, (AgentInfo, Instant)>>` populated by next-fetch diff; parallel decay-future drops entries past 5s; new `.card.is-ended` + `.pill.ended` CSS modifiers; uptime frozen at termination time.
2. **Plan 02 — chat-ws subagent event plumbing + dynamic poll cadence** (~120 LoC): wire `SubagentProgressCallback` from `state.rs:124` (currently `None`) through to the chat WebSocket; new ws event type (kind only, no payload); client `subagent_events: Signal<u64>` counter; `ScreenAgents` `use_effect` triggers `.restart()` on increment; poll widens to 5s while ws connected, tightens back to 1500ms on ws disconnect.

**Closes with:** `checkpoint:human-verify` UAT exercising (i) live-spawn <1s after `delegate_task` post-Plan-02; (ii) TERMINATED-HOLD-N (ENDED card visible 5s after natural termination, then disappears cleanly); (iii) kill/interrupt regression (26.7-06's 200 responses still hold; HOLD card also shows for killed agents); (iv) ws-disconnected fallback (closing chat ws leaves page working on 1500ms poll).

**Out of scope (own follow-up phases):**
- Tree view rendering of `AgentInfo.parent_id` (today rendered flat).
- `+ NEW AGENT` button — own "spawn agent from UI" phase.
- Changes to any of the four `/api/agents/*` endpoints — 26.7-06's `spawn_blocking` + 26.7-07's `Drop` fix carry forward unchanged.
- Wrapping `api_agents_status` in `spawn_blocking` (latent, UI-unexercised; carry from 26.7's deferred list).
- Server-side `recently_terminated` buffer (rejected — see D-02 below); revisit only if multi-tab consistency becomes a goal.
- Per-termination-kind distinguishing visuals (kept simple — single ENDED pill regardless of how the agent ended; see D-09).

</domain>

<decisions>
## Implementation Decisions

### TERMINATED-HOLD fade (Area A)
- **D-01:** `N = 5 seconds`. The HOLD card persists for 5 seconds in the recently-terminated state before being removed from view.
- **D-02:** HOLD applies to **all terminations uniformly** — natural exit (Max iterations, self-finish), kill button, interrupt-then-finalize. No per-termination-kind branching in the diff logic. The PRUNE behavior in D-12 is the one exception to "uniformity."

### Poll cadence (Area B)
- **D-03:** Plan 01 baseline poll interval = **1500 ms**. Implemented via a `use_future` loop in `ScreenAgents` (or a sibling subcomponent) calling `agents_resource.restart()`.
- **D-04:** Poll **pauses when the tab is hidden** (`web_sys::window().unwrap().document().unwrap().hidden()` via Page Visibility API). On visibility return, fire an immediate fetch then resume the interval. ~10 LoC additional vs always-polling.
- **D-05:** Poll **pauses while any per-card `killing()` or `interrupting()` signal is true**, to prevent the poll from racing the in-flight RPC's signal-set ordering. Resumes when both signals are false again. Today's component already tracks these signals — the poll loop just consults them.

### Event-driven path (Area C — PIVOT vs roadmap entry)
- **D-06:** **Plan 02 reuses the existing chat WebSocket.** The roadmap entry's assumption that `SubagentProgress::Started`/`Completed` already flow through chat ws was wrong — today `state.rs:124` sets `progress_callback: None` and `ws.rs:195` only wires `ToolProgressCallback`. Plan 02 must: (a) construct a `SubagentProgressCallback` in `state.rs` that forwards events to the ws send half; (b) extend the chat-ws message shape with a new event kind; (c) parse the new event on the client side in `warp_hermes.rs`.
- **D-07:** **Event payload = counter only.** Server sends a minimal event (e.g. `{"kind": "subagent_event"}`); client increments `subagent_events: Signal<u64>` on `WarpHermes`. `ScreenAgents` `use_effect` watches this counter and calls `agents_resource.restart()` on increment. This forces a fresh `GET /api/agents/list` — same code path as the poll, no divergent diff logic. ~5-byte event over the wire.
- **D-08:** **Dynamic poll cadence**: while `is_ws_connected: Signal<bool>` is `true`, poll widens to **5s** (safety net for missed/dropped events); while `false`, poll tightens to the Plan-01 **1500ms** baseline. State tracked via the chat-ws lifecycle handler in `warp_hermes.rs`.

### Terminal visual state (Area D)
- **D-09:** **Single `ENDED` pill for all terminations.** No `KILLED`/`INTERRUPTED` differentiation in pill text. The visual sameness is deliberate — matches D-02's "all terminations uniform."
- **D-10:** HOLD card body uses **opacity ~0.55**, **action buttons hidden** (not just disabled). CHAT, INTERRUPT, and KILL buttons all removed from view during the HOLD state. Cuts visual noise; agent is gone, no actions are valid.
- **D-11:** **`uptime_secs` frozen at termination time.** When the diff captures an agent into `recently_terminated`, the `AgentInfo` snapshot is stored as-is (including its last-observed `uptime_secs`). The HOLD card renders that frozen value for 5s, not a wall-clock ticker.

### Edge cases (Area E)
- **D-12:** **PRUNE ENDED clears both server-side and client-side state.** The existing PRUNE click already calls `api_agents_prune(None)`; the new behavior also calls `recently_terminated.write().clear()` synchronously. User clicks PRUNE → all ENDED cards gone, both kinds.
- **D-13:** **Re-running id wins on replay edge case.** If a terminated agent's id appears in a subsequent fetch as running again (registry register-with-same-id), the diff drops the entry from `recently_terminated` immediately. Card renders as live, no animation. Avoids ENDED-and-running-with-same-id rendering collision.
- **D-14:** **New CSS modifier classes** — `.card.is-ended` (sets `opacity: 0.55`, hides `.card-footer` buttons via CSS) and `.pill.ended` (gray variant of the existing `.pill` family). Matches the existing `.is-active` / `.is-cmd` / `.is-ai` modifier convention in `app.html` / `warp-ih.css`.

### Claude's Discretion
- Exact opacity value for `.card.is-ended` (D-10 proposes ~0.55; planner may tune to 0.5 / 0.6 to match the existing design system's "dim" tokens if one exists).
- Exact `.pill.ended` background color (D-14 says gray variant — picker depends on whether the existing tokens already have a "neutral/disabled" pill color; researcher / planner confirm against `colors_and_type.css`).
- Whether `subagent_events: Signal<u64>` lives directly on `WarpHermes` or in a smaller dedicated context provider (D-07 says `WarpHermes`; if `WarpHermes` already has too many signals, planner may slot it elsewhere).
- Whether the chat-ws lifecycle handler at `warp_hermes.rs:135` exposes `is_ws_connected` via a new signal or whether an existing connection-status signal already exists (D-08 needs one — planner confirms).
- Whether the dimming + button-hiding for `.card.is-ended` is pure CSS (selectors that match `.card-footer > .btn` and `display: none`) or a conditional in the RSX macro. Either is acceptable; CSS-only is preferred for "one source of truth."
- Whether the recently-terminated decay future runs at 1s intervals checking each entry's `Instant::elapsed()` or uses per-entry `gloo_timers::TimeoutFuture` cancellation. Both work; 1s sweep is simpler.

</decisions>

## Existing Code Insights

### Reusable Assets
- `use_server_future` + `.restart()` pattern: `agents.rs` already uses both. The poll loop just needs to call `.restart()` on a timer — no new fetch infrastructure required.
- `gloo_timers::future::TimeoutFuture` — already imported in `agents.rs:170, 199` for the KILL/INTERRUPT visual delays. Reuse for the poll interval and decay sweeps (or `gloo_timers::callback::Interval` for the periodic case).
- `dioxus_fullstack::use_websocket` — already used in `warp_hermes.rs:135` for chat. Plan 02 extends the message-parse arm in the existing `use_future` reader at `warp_hermes.rs:145`, not a new ws.
- `ToolProgressCallback` wiring at `ws.rs:195-217` — direct template for `SubagentProgressCallback`. Construct the callback in `ws.rs`, plumb to `run_web_turn` via `Some(callback)` replacing `state.rs:124`'s `None`.
- `.card`, `.pill`, `.is-active` CSS class family — extend with `.is-ended` and `.pill.ended` modifiers; same conventions as existing modifier classes.

### Established Patterns
- **Signal-borrow discipline**: every `.read()` returns a `GenerationalRef` that must drop before any `.await`. The poll loop's read-current-list step inside the `use_future` must collect into an owned `Vec` and drop the read guard before awaiting `TimeoutFuture`. Plan 01 must add a clippy-clean test demonstrating this.
- **`recently_terminated: Signal<HashMap<String, (AgentInfo, Instant)>>`**: signal containing a HashMap is fine in Dioxus 0.7 — the signal-write replaces the whole map; the read is by reference (GenerationalRef). Standard pattern; just don't hold the borrow across awaits.
- **`use_effect` for cross-signal reactivity**: Plan 02's "increment counter → call restart" is exactly the use_effect idiom (subscribe to one signal, mutate another).
- **`web_sys` Page Visibility integration**: web target only; no #[cfg(feature = "web")] gate needed in `iron_hermes_ui` since the crate IS the web target (per `iron_hermes_ui/CLAUDE.md` constraints).

### Integration Points
- **Plan 01 → Plan 02 hand-off**: Plan 01's `recently_terminated` signal + diff logic is unchanged in Plan 02 — Plan 02 only adds an additional `restart()` trigger (ws-event-driven) on top of the existing poll-driven trigger. The diff sees the same dataset either way.
- **Plan 02 → existing chat ws**: the `SubagentProgressCallback` lives next to the existing `ToolProgressCallback`. Both are constructed in `ws.rs`, both forward to the same ws send half. Different event kinds in the message shape distinguish them on the client.
- **`is_ws_connected: Signal<bool>`**: set inside the chat-ws lifecycle handler in `warp_hermes.rs`. Existing handler may already have a connection-status local — confirm with a grep before adding a new signal.

<specifics>
## Specific Ideas

- The roadmap entry's assumption that `SubagentProgress` flows through chat ws today was WRONG — `state.rs:124` is `None`, `ws.rs:195` is `ToolProgressCallback`. Plan 02 must plumb. This is the single biggest correction from the original draft.
- The user's UAT against `dx serve` confirmed the symptom: `delegate_task` fired, the tool call streamed in chat, but the Agents page never populated. Plan 01's 1500ms poll alone closes this for any agent that lives >1.5s; Plan 02 closes it for any agent regardless of lifetime.
- The HOLD-N fade exists primarily for user-facing visual feedback ("I saw something happen"). Without it, the natural-termination path is invisible — the agent registers, runs, unregisters, and the next poll shows an empty list with no indication anything occurred.
- Multi-tab consistency is explicitly NOT a goal (each tab's `recently_terminated` is independent). The user accepted this trade-off at roadmap-entry time.
- `is_ws_connected` is the dynamic-poll-cadence trigger. Without it, the poll defaults to 1500ms always — which works but doubles request volume when Plan 02's event path is live.

</specifics>

<deferred>
## Deferred Ideas

- **Tree view of subagent hierarchy** — `AgentInfo.parent_id` exists today but is rendered flat. Future phase: indent child cards or draw connection lines parent→child. Useful when subagents start spawning their own subagents.
- **`+ NEW AGENT` button wiring** — own "spawn agent from UI" phase. Requires a new server fn that takes a task + agent profile, calls `delegate_task` server-side, and returns the spawned id.
- **`api_agents_status` in `spawn_blocking`** — same latent `block_in_place` bug pattern as 26.7-06 kill/interrupt/prune. Currently UI-unexercised; carry from 26.7's deferred list. Could be bundled with this phase opportunistically but kept out of scope to preserve plan size.
- **Server-side `recently_terminated` buffer** — explicitly rejected for this phase. Revisit only if multi-tab consistency becomes a requirement (which it isn't today).
- **Per-termination-kind visual differentiation** — single `ENDED` pill for all terminations is the locked decision (D-09). Future polish phase could re-introduce `KILLED` (red-dim) / `INTERRUPTED` (amber-dim) labels if user feedback warrants.
- **Loading spinner during first page render** — today the page renders empty until `use_server_future` resolves (per 26.7 D-13). No spinner added in this phase; revisit if "is the page broken?" feedback emerges.
- **Telemetry on ws event drops** — Plan 02 doesn't log dropped subagent events (e.g. ws send-half buffer full). Could be added; deferred unless drops are observed in practice.
- **Cap on `recently_terminated` HashMap size** — DoS guardrail in case a script spawns 10k agents fast. Not addressed; in practice the 5s decay window bounds the map size to roughly (terminations per second × 5) which is fine for human-cadence agents.

</deferred>
