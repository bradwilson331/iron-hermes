## Summary

Phase 26.7.1 adds real-time live updates to the Agents screen in two plans. Plan 01 is a polling baseline with a terminated-agent HOLD card. Plan 02 wires a WebSocket push notification through the existing chat ws path to provide sub-second refresh latency.

The critical correction from the previous attempt: all integration happens in `crates/iron_hermes_ui/src/components/hermes_app/` (the active root under the default `web` feature). `WarpHermes` is legacy-shell dead code that compiles only when `--features legacy-shell` is passed ŌĆö it never mounts in `dx serve` or `cargo build --features web`.

The two new signals (`subagent_events: Signal<u64>`, `is_ws_connected: Signal<bool>`) are declared in `HermesApp`'s function body and exposed via `use_context_provider`, exactly like the existing nine signals at lines 58-61 and 302-306 of `mod.rs`. The ws event arm for `ChatStreamEvent::SubagentEvent {}` is added inside the existing `use_future` recv loop at `mod.rs:156`, not as a new ws connection. `ScreenAgents` consumes both signals via `use_context`.

**Primary recommendation:** Declare both new signals in `HermesApp` body before line 302, add `use_context_provider` calls after line 306, add the `SubagentEvent` match arm and connection-lifecycle sets in the recv loop, then update `ScreenAgents` to consume the context signals.

### Pattern 4: Wiring progress_callback in state.rs

**The constraint:** `DelegateTaskWiring` is constructed once in `AppState::init()` via `build_app_runtime_bundle`. The `progress_callback` field is `Option<SubagentProgressCallback>` (an `Arc<dyn Fn(...)>`) baked at construction time into the `AppRuntimeBundle`, which is then Arc-wrapped. It cannot be swapped per-turn without reconstructing the bundle.

**Recommended approach (Path A ŌĆö Global channel in AppState):**

1. In `AppState::init()`, create an `mpsc::unbounded_channel::<ChatStreamEvent>()`.
2. Store the `Sender` half on `AppState` as `pub subagent_event_tx: tokio::sync::mpsc::UnboundedSender<ChatStreamEvent>`.
3. Set `progress_callback: Some(Arc::new(move |_idx, _ev| { let _ = tx.send(ChatStreamEvent::SubagentEvent {}); }))` using a clone of the sender.
4. In `ws.rs`, before starting the turn's tokio::spawn, clone `app_state.subagent_event_tx` and subscribe via a relay task that forwards events to the per-turn `tx` channel.

**Alternative (Path B ŌĆö per-turn injection):** Add a `run_web_turn_with_subagent_cb` method that passes a fresh callback into `build_agent_loop`. This is more surgical but requires changing how `build_agent_loop` wires the delegate task's tool registration. Higher risk.

**Recommendation: Path A** ŌĆö consistent with CONTEXT.md's "reuse existing ws send half" principle and lower implementation risk for Phase 26.7.1's single-session use case.

**Multi-session note:** Path A broadcasts subagent events from all sessions to all connected ws clients. This is acceptable per CONTEXT.md's explicit out-of-scope decision on multi-tab consistency.

### Pattern 5: ScreenAgents Poll Loop with use_future

**What:** Screen-level `use_future` loop checking visibility, RPC guards, then calling `agents_resource.restart()`, sleeping on a dynamic interval.

```rust
// Consume context signals

let subagent_events = use_context::<Signal<u64>>();    // subagent_events from HermesApp

// D-05: screen-level RPC guard counter (see Pitfall 3 + Open Question 1)
let mut rpc_in_flight = use_signal(|| 0u32);

// Poll loop
use_future(move || async move {
    loop {
        // D-04: skip while tab is hidden
        #[cfg(target_arch = "wasm32")]
        {
            let hidden = web_sys::window()
                .and_then(|w| w.document())
                .map(|d| d.hidden())
                .unwrap_or(false);
            // Drop borrow before await (clippy rule ŌĆö hidden is bool, already dropped)
            if hidden {
                gloo_timers::future::TimeoutFuture::new(500).await;
                continue;

        // D-05: skip while a kill/interrupt RPC is in flight
        let in_flight = *rpc_in_flight.read();   // Copy ŌĆö ref dropped at ;
        if in_flight > 0 {
            gloo_timers::future::TimeoutFuture::new(200).await;
            continue;

        agents_resource.restart();

        // D-08: dynamic cadence
        let interval_ms: u32 = if *ws_connected.read() { 5_000 } else { 1_500 };
        // Drop borrow before await
        gloo_timers::future::TimeoutFuture::new(interval_ms).await;

// D-07 Plan 02: counter-driven restart via use_effect
use_effect(move || {
    subagent_events();        // subscribe ŌĆö Dioxus re-runs this effect on change
    agents_resource.restart();
});
```

### Pattern 6: recently_terminated Diff Logic (Plan 01)

**What:** After each `agents_resource.restart()` resolves, compare the new list against the previous list to detect terminations. Move newly-absent ids into `recently_terminated` (with snapshot + `Instant::now()`). Remove re-running ids from the map (D-13).

```rust
let mut recently_terminated =
    use_signal(|| std::collections::HashMap::<String, (crate::server::api::AgentInfo, std::time::Instant)>::new());

// After agents_list is materialized (existing pattern in agents.rs:33-36):
// Capture snapshot of previous list into owned Vec before await (clippy rule)
let prev_ids: Vec<String> = recently_terminated.read().keys().cloned().collect();
// prev_ids: local Vec<String>, borrow dropped

// Diff: ids in previous live list that are gone from new list
for id in prev_ids {
    if !agents_list.iter().any(|a| a.id == id) {
        // Already in recently_terminated ŌĆö keep it (decay future will remove it)
    }
}
// Ids in old live list absent from new live list ŌåÆ move to recently_terminated
// (requires tracking previous live list as a Signal<Vec<AgentInfo>> or equivalent)

// D-13: re-running id wins ŌĆö remove from recently_terminated if reappears
for agent in &agents_list {
    let in_map = recently_terminated.read().contains_key(&agent.id);  // bool ŌĆö drop ref
    if in_map {
        recently_terminated.write().remove(&agent.id);
    }
}
```

**Pure-fn extraction (required for Wave 0 unit test):** Extract the diff logic as:
```rust
pub fn diff_terminations(
    live_prev: &[crate::server::api::AgentInfo],
    live_next: &[crate::server::api::AgentInfo],
) -> Vec<crate::server::api::AgentInfo> {
    // Returns agents present in live_prev but absent in live_next
    let next_ids: std::collections::HashSet<&str> =
        live_next.iter().map(|a| a.id.as_str()).collect();
    live_prev.iter().filter(|a| !next_ids.contains(a.id.as_str())).cloned().collect()
}
```

### Pattern 7: JsCast Import Convention (project-established)

**What:** `JsCast` is a `wasm_bindgen` trait. In this codebase, it is imported at item scope, gated by `#[cfg(target_arch = "wasm32")]`. It is NOT re-exported by `dioxus::prelude::*`.

**Confirmed pattern (theme_effects.rs:24, wheel.rs:179):**
```rust
#[cfg(target_arch = "wasm32")]
use wasm_bindgen::JsCast;
```

Or inside a cfg block:
```rust
#[cfg(target_arch = "wasm32")]
{
    use wasm_bindgen::JsCast as _;
    // .unchecked_ref::<T>() and .dyn_into::<T>() now resolve
}
```

**Failure mode from prior attempt:** Calling `.unchecked_ref()` without `JsCast` in scope gives "no method named `unchecked_ref` found". The prelude does not help.

### Pattern 8: use_drop Import Path

**What:** `use_drop` for cleanup on unmount must be explicitly imported from `dioxus::core`.

```rust
use dioxus::core::use_drop;
```

It is NOT available via `dioxus::prelude::*`.

**Failure mode from prior attempt:** Compiles on WASM (`--features web`) but errors on native (`--features server`) when the wrong path is assumed.

## Common Pitfalls

### Pitfall 1: Targeting WarpHermes Instead of HermesApp
**What goes wrong:** Changes to `warp_hermes.rs` compile cleanly (the file is always compiled). But at runtime (`dx serve`), only `HermesApp` mounts. Context providers added to `WarpHermes` are never called, so any `use_context` downstream panics: `Could not find context dioxus_signals::signal::Signal<bool>`.
**Why it happens:** Both components are in the binary. The cfg gate is at the `root_shell()` call site, not at the component definition. Grep hits on `warp_hermes.rs` look like live code.
**How to avoid:** Always verify the active root by reading `app.rs:root_shell()` and `Cargo.toml:default`. For any default build, the active root is `HermesApp`.
**Warning signs:** Any plan that edits `warp_hermes.rs` for Phase 26.7.1. Compile-time success + runtime context panic.

### Pitfall 2: Signal Borrow Across Await (clippy.toml enforced)
**What goes wrong:** Holding a `GenerationalRef` (from `.read()`) or `GenerationalRefMut` (from `.write()`) or `dioxus_signals::WriteLock` across an `.await` point causes a compile error via `clippy.toml`'s `await-holding-invalid-types` configuration (lines 1-8 of `clippy.toml`).
**Why it happens:** These types are not `Send`; the lint is configured at the project level and is treated as an error via `-D warnings`.
**How to avoid:** Read into a primitive or owned value, end the statement (dropping the ref), then await.
```rust
// WRONG ŌĆö GenerationalRef held across await
let interval = if *ws_connected.read() { 5_000 } else { 1_500 };
some_future().await;

// CORRECT ŌĆö read into Copy value, ref dropped at ;
let interval: u32 = if *ws_connected.read() { 5_000 } else { 1_500 };
// ws_connected GenerationalRef dropped here (end of statement)
gloo_timers::future::TimeoutFuture::new(interval).await;
```
**Warning signs:** `cargo clippy -p iron_hermes_ui --features web -- -D warnings` fails with "this type is not allowed to be held across an await point".

### Pitfall 3: D-05 ŌĆö killing/interrupting Signals Are Per-Card Local
**What goes wrong:** The poll loop in `ScreenAgents` cannot directly observe `killing: Signal<bool>` or `interrupting: Signal<bool>` because those are declared with `use_signal(|| false)` inside `AgentCard`, making them component-local.
**Why it happens:** `use_signal` inside a child component creates state scoped to that component instance, invisible to the parent.
**How to avoid:** Introduce a `rpc_in_flight: Signal<u32>` counter at `ScreenAgents` scope. Pass it as a prop to `AgentCard`. AgentCard closures increment it at RPC start and decrement at RPC end. The poll loop checks `*rpc_in_flight.read() == 0`. This is a minimal-scope change ŌĆö no context provider needed.
**Warning signs:** A plan that says "check `killing()` from the poll loop" without describing how ScreenAgents accesses AgentCard's local signal.

### Pitfall 4: JsCast Trait Not in Prelude
**What goes wrong:** Methods `.unchecked_ref::<T>()` and `.dyn_into::<T>()` from `wasm_bindgen` fail with "no method named `unchecked_ref` found" if `JsCast` is not in scope.
**Why it happens:** `JsCast` is not re-exported by `dioxus::prelude::*`. It must be imported explicitly.
**How to avoid:** `#[cfg(target_arch = "wasm32")] use wasm_bindgen::JsCast;` at module or block scope. See established pattern in `theme_effects.rs:24` and `wheel.rs:179`.
**Warning signs:** Compile error on `.unchecked_ref()` or `.dyn_into()` with no method found.

### Pitfall 5: use_drop Import Path
**What goes wrong:** Cleanup hooks using `use_drop` compile on WASM but fail on native builds.
**Why it happens:** `use_drop` lives at `dioxus::core::use_drop`, not in `dioxus::prelude::*`. On native, the prelude path resolution differs.
**How to avoid:** Always write `use dioxus::core::use_drop;` explicitly.
**Warning signs:** Compilation succeeds for `--features web` but fails for `--features server` or native builds.

### Pitfall 6: cfg-Gated Dead Code Trap
**What goes wrong:** Editing `warp_hermes.rs` or any `#[cfg(feature = "legacy-shell")]`-gated code compiles cleanly but has zero effect at runtime in default builds.
**Why it happens:** Cargo silently skips cfg-disabled blocks. There is no runtime warning that changed code is never executed.
**How to avoid:** Before any edit, check the feature gate in `Cargo.toml` and confirm `app.rs:root_shell()`. The active root in default builds is always `HermesApp`.

### Pitfall 7: Cargo.lock / dx Version Drift
**What goes wrong:** `dioxus = "0.7"` in `Cargo.toml` resolves to any 0.7.x patch. If `dx` CLI was built at a different 0.7.x than Cargo resolves, API signatures can diverge.
**Why it happens:** `Cargo.lock` is not committed (per `CLAUDE.md`: "Lockfile: `Cargo.lock` is generated by Cargo but not committed"). SemVer allows silent 0.7.x patch bumps.
**How to avoid:** If a mysterious API mismatch appears, run `cargo update -p dioxus --precise <exact-version>` to pin. Verify versions match: `dx --version` and `cargo metadata --format-version 1 | grep '"dioxus"'`.
**Warning signs:** "method not found" on a documented Dioxus API. `dx serve` build errors that don't match `cargo check` errors.

### Pitfall 8: UAT Must Use the Correct Absolute Worktree Path
**What goes wrong:** Running `dx serve` from the main checkout when changes are in a git worktree serves the wrong binary ŌĆö the worktree's commits are not reflected.
**Why it happens:** `dx serve` resolves `Dioxus.toml` from the current working directory. If cwd is the main checkout, Dioxus.toml and the source files there are used.
**How to avoid:** Always run UAT from the absolute worktree path: `cd /absolute/path/to/worktree/crates/iron_hermes_ui && dx serve`. Specify the path explicitly in UAT instructions.

### Pitfall 9: DelegateTaskWiring.progress_callback Is a Singleton (Plan 02)
**What goes wrong:** The `SubagentProgressCallback` set in `AppState::init()` is shared across all sessions via the Arc-wrapped `AppRuntimeBundle`. Events from any subagent in any session are broadcast to every connected ws client.
**Why it happens:** `build_app_runtime_bundle` wraps the result in `Arc<AppRuntimeBundle>`. The callback inside cannot be swapped per-turn without rebuilding the bundle.
**How to avoid:** This is acceptable for Phase 26.7.1 (single-session assumption consistent with CONTEXT.md's out-of-scope decision on multi-tab consistency). Document it as a known limitation. Path A (global channel) makes this explicit.
**Warning signs:** If multi-session or multi-tab support is introduced in a future phase, the callback architecture must be revisited.
