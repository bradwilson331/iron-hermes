## User Constraints (from CONTEXT.md)

### Locked Decisions

**TERMINATED-HOLD fade (Area A)**
- **D-01:** `N = 5 seconds`. The HOLD card persists for 5 seconds in the recently-terminated state before being removed from view.
- **D-02:** HOLD applies to **all terminations uniformly** — natural exit (Max iterations, self-finish), kill button, interrupt-then-finalize. No per-termination-kind branching in the diff logic. The PRUNE behavior in D-12 is the one exception to "uniformity."

**Poll cadence (Area B)**
- **D-03:** Plan 01 baseline poll interval = **1500 ms**. Implemented via a `use_future` loop in `ScreenAgents` (or a sibling subcomponent) calling `agents_resource.restart()`.
- **D-04:** Poll **pauses when the tab is hidden** (`web_sys::window().unwrap().document().unwrap().hidden()` via Page Visibility API). On visibility return, fire an immediate fetch then resume the interval. ~10 LoC additional vs always-polling.
- **D-05:** Poll **pauses while any per-card `killing()` or `interrupting()` signal is true**, to prevent the poll from racing the in-flight RPC's signal-set ordering. Resumes when both signals are false again. Today's component already tracks these signals — the poll loop just consults them.

**Event-driven path (Area C — PIVOT vs roadmap entry)**
- **D-06:** **Plan 02 reuses the existing chat WebSocket.** The roadmap entry's assumption that `SubagentProgress::Started`/`Completed` already flow through chat ws was wrong — today `state.rs:124` sets `progress_callback: None` and `ws.rs:195` only wires `ToolProgressCallback`. Plan 02 must: (a) construct a `SubagentProgressCallback` in `state.rs` that forwards events to the ws send half; (b) extend the chat-ws message shape with a new event kind; (c) parse the new event on the client side in `warp_hermes.rs`.
- **D-07:** **Event payload = counter only.** Server sends a minimal event (e.g. `{"kind": "subagent_event"}`); client increments `subagent_events: Signal<u64>` on `WarpHermes`. `ScreenAgents` `use_effect` watches this counter and calls `agents_resource.restart()` on increment. This forces a fresh `GET /api/agents/list` — same code path as the poll, no divergent diff logic. ~5-byte event over the wire.
- **D-08:** **Dynamic poll cadence**: while `is_ws_connected: Signal<bool>` is `true`, poll widens to **5s** (safety net for missed/dropped events); while `false`, poll tightens to the Plan-01 **1500ms** baseline. State tracked via the chat-ws lifecycle handler in `warp_hermes.rs`.

**Terminal visual state (Area D)**
- **D-09:** **Single `ENDED` pill for all terminations.** No `KILLED`/`INTERRUPTED` differentiation in pill text. The visual sameness is deliberate — matches D-02's "all terminations uniform."
- **D-10:** HOLD card body uses **opacity ~0.55**, **action buttons hidden** (not just disabled). CHAT, INTERRUPT, and KILL buttons all removed from view during the HOLD state. Cuts visual noise; agent is gone, no actions are valid.
- **D-11:** **`uptime_secs` frozen at termination time.** When the diff captures an agent into `recently_terminated`, the `AgentInfo` snapshot is stored as-is (including its last-observed `uptime_secs`). The HOLD card renders that frozen value for 5s, not a wall-clock ticker.

**Edge cases (Area E)**
- **D-12:** **PRUNE ENDED clears both server-side and client-side state.** The existing PRUNE click already calls `api_agents_prune(None)`; the new behavior also calls `recently_terminated.write().clear()` synchronously. User clicks PRUNE → all ENDED cards gone, both kinds.
- **D-13:** **Re-running id wins on replay edge case.** If a terminated agent's id appears in a subsequent fetch as running again (registry register-with-same-id), the diff drops the entry from `recently_terminated` immediately. Card renders as live, no animation. Avoids ENDED-and-running-with-same-id rendering collision.
- **D-14:** **New CSS modifier classes** — `.card.is-ended` (sets `opacity: 0.55`, hides `.card-footer` buttons via CSS) and `.pill.ended` (gray variant of the existing `.pill` family). Matches the existing `.is-active` / `.is-cmd` / `.is-ai` modifier convention in `app.html` / `warp-ih.css`.

## Summary

This is a Dioxus 0.7.1 WASM UI phase with a small Rust server-side companion edit. Plan 01 is a pure client-side fix to the Agents screen — add a periodic poll loop, a recently-terminated HashMap signal, a decay sweep future, a Page Visibility listener, and two CSS modifier classes. Plan 02 plumbs an existing-but-unused server-side callback (`SubagentProgressCallback`) through the chat WebSocket so the client gets push-style invalidation events and can widen its poll cadence.

The codebase already has every primitive needed: `use_server_future` with `.restart()` is in production use at `agents.rs:29,64,192`; `gloo_timers::future::TimeoutFuture` is at `agents.rs:171,199`; `web_sys::window().add_event_listener_with_callback` with `Closure<dyn FnMut(...)>` storage in a `Signal<Option<Closure>>` and `use_drop` cleanup is at `wheel.rs:189-280` and `warp_hermes.rs:505-552`. The `SubagentProgressCallback` type alias is defined at `crates/ironhermes-tools/src/delegate_task.rs:38` and is already plumbed through `DelegateTaskWiring` (`app_runtime_factory.rs:31`) into the tool registry — it's just being passed `None` at the construction site. The chat ws send half is captured by a `tokio::sync::mpsc::UnboundedSender<ChatStreamEvent>` in `ws.rs:185-235`.

**Primary recommendation:** Implement Plan 01 with one screen-level `recently_terminated: Signal<HashMap<String, (AgentInfo, Instant)>>` (with `Instant` = `std::time::Instant` since `gloo_timers` runs on JS event loop and `Instant::now()` IS supported on wasm32-unknown-unknown via wasm-bindgen-time shim — see Open Q1); one `use_future` poll loop reading both `killing()/interrupting()` and `document.hidden()` to decide whether to fire; one `use_future` decay sweep that reads `Instant::elapsed()` at 1s intervals; one Page Visibility listener stored in a signal and cleaned up in `use_drop`. Plan 02 mirrors `ws.rs:195-217`'s `ToolProgressCallback` construction: capture `tx: mpsc::UnboundedSender<ChatStreamEvent>` in an `Arc<dyn Fn>` closure, push the wiring into the per-turn `state.rs` call so the callback lifetime matches the turn (NOT the session — `state.rs:124` is the per-session slot but the tx half is per-turn; this is the critical insight in Risk #1). Add a `ChatStreamEvent::SubagentEvent {}` variant. Client `WarpHermes` exposes `subagent_events: Signal<u64>` via `use_context_provider`; `ScreenAgents` reads it via `use_context` and watches with `use_effect`.

### Pattern 1: WASM Event Listener with `use_drop` cleanup (D-04 Page Visibility)

**What:** Install a `visibilitychange` listener on `document`, store the `Closure` in a `Signal<Option<Closure<dyn FnMut(_)>>>`, and remove it in `use_drop`. The signal storage is mandatory — `cb.forget()` leaks AND prevents cleanup.

**When to use:** Plan 01 D-04 — exactly this pattern.

**Example (lifted verbatim from `crates/iron_hermes_ui/src/components/hermes_app/wheel.rs:189-280`):**
```rust
// Source: crates/iron_hermes_ui/src/components/hermes_app/wheel.rs:184-280

#[cfg(target_arch = "wasm32")]
{
    use web_sys::Event as WebEvent;
    type VisibilityListener = Closure<dyn FnMut(WebEvent)>;
    let mut listener_slot: Signal<Option<VisibilityListener>> = use_signal(|| None);

    let mut tab_hidden = use_signal(|| false);

    use_effect(move || {
        let Some(window) = web_sys::window() else { return; };
        let Some(document) = window.document() else { return; };

        // Mirror initial state into the signal.
        tab_hidden.set(document.hidden());

        let cb = Closure::<dyn FnMut(WebEvent)>::new(move |_e: WebEvent| {
            if let Some(window) = web_sys::window() {
                if let Some(doc) = window.document() {
                    tab_hidden.set(doc.hidden());
                }
            }
        });
        let _ = document
            .add_event_listener_with_callback("visibilitychange", cb.as_ref().unchecked_ref());
        listener_slot.set(Some(cb));
    });

    use_drop(move || {
        if let Some(cb) = listener_slot.write().take() {
            if let Some(window) = web_sys::window() {
                if let Some(doc) = window.document() {
                    let _ = doc.remove_event_listener_with_callback(
                        "visibilitychange",
                        cb.as_ref().unchecked_ref(),
                    );
                }
            }
        }
    });
}
```

**Cargo.toml dependency:** `web-sys` already has `Window`, `Document`, `EventTarget`, `KeyboardEvent` enabled; planner must verify whether `Event` is needed (it is — Page Visibility events are bare `Event`s, not specialized). Likely needs `"Event"` added to the `web-sys` features list at `Cargo.toml:19-32`.

### Pattern 2: `use_future` poll loop with signal-borrow discipline (D-03, D-05)

**What:** A `use_future` that loops forever, awaits a `TimeoutFuture`, then conditionally restarts the resource. All signal `.read()`s must drop before `.await`.

**When to use:** Plan 01 D-03 poll baseline; Plan 01 D-05 kill/interrupt pause; Plan 02 D-08 cadence widen.

**Critical idiom:** Read each signal via call-syntax (`some_signal()` for `Copy` types like `bool`, `u64`) so it returns by value, not by `GenerationalRef`. Where the value is non-Copy (a `HashMap`), `.read().clone()` into an owned `Vec`/`HashMap` and let the read guard drop before awaiting — see `agents.rs:33-37` for the existing pattern.

**Example:**
```rust
// Source: synthesis from agents.rs:29-37,171 + warp_hermes.rs:145
use_future(move || async move {
    loop {
        // All reads happen BEFORE the await — none span the .await point.
        let hidden = tab_hidden();        // Copy bool
        let busy = killing_screen_aggregate() || interrupting_screen_aggregate(); // Copy bool
        let connected = is_ws_connected(); // Copy bool — Plan 02
        let interval_ms = if connected { 5_000 } else { 1_500 };

        if !hidden && !busy {
            agents_resource.restart();
        }
        // No signal borrow held across this await — clippy.toml satisfied.
        gloo_timers::future::TimeoutFuture::new(interval_ms).await;
    }
});
```

**Aggregating per-card busy state:** Plan 01 must lift `killing` / `interrupting` from per-AgentCard local signals to **screen-level aggregates** so the poll loop can consult them. Options:
1. Promote to `use_signal` on `ScreenAgents`, pass mutable down via props (works; pollutes AgentCard signature).
2. `use_context_provider` a `Signal<HashSet<String>>` of "busy ids"; cards insert/remove their id on entry/exit. Cleaner; per Phase 26.7 D-02 hybrid-state model the planner should pick whichever is simpler in the AgentCard refactor.

### Pattern 3: `SubagentProgressCallback` construction mirroring `ToolProgressCallback` (Plan 02 D-06)

**What:** Construct the callback inside the ws upgrade handler so the `Arc<dyn Fn>` captures the per-turn `tx: UnboundedSender<ChatStreamEvent>`. Thread the callback into the run-turn call so it overrides the per-session `None` default at `state.rs:124`.

**Type signature (verified at `crates/ironhermes-tools/src/delegate_task.rs:38`):**
```rust
pub type SubagentProgressCallback = Arc<dyn Fn(usize, SubagentProgress) + Send + Sync>;
```

**Emission sites (verified):**
- `delegate_task.rs:368-374` — `SubagentProgress::Started { task_summary }` when child task begins (inside `tokio::spawn`)
- `delegate_task.rs:292-296` — `SubagentProgress::ToolCall { tool_name }` per tool call inside child
- `delegate_task.rs:446-448` — `SubagentProgress::Completed` after child finishes (success OR timeout — both paths)

**Construction template (lifted from `ws.rs:194-201` for ToolProgressCallback, adapted):**
```rust
// Source: synthesis from ws.rs:194-218 + state.rs:124 + delegate_task.rs:38
let tx_subagent = tx.clone();
let subagent_progress_callback: ironhermes_tools::delegate_task::SubagentProgressCallback =
    std::sync::Arc::new(move |_index: usize, _event: ironhermes_tools::delegate_task::SubagentProgress| {
        // D-07: counter-only — no payload shape decisions to make per event.
        // Send fires on Started + ToolCall + Completed alike; client diff via
        // /api/agents/list does the actual state reconciliation.
        let _ = tx_subagent.send(crate::protocol::ChatStreamEvent::SubagentEvent {});
    });
```

**Plumbing into run_web_turn:** `AppState::run_web_turn` already takes `tool_progress_callback: Option<ToolProgressCallback>` (state.rs:174). Add a parallel `subagent_progress_callback: Option<SubagentProgressCallback>` parameter. The `progress_callback: None` at state.rs:124 is per-session (constructed once in `init()`), but we need a **per-turn** callback that captures the turn's `tx`. The cleanest path is to make `DelegateTaskWiring` clonable per-turn and rebuild the tool registry; that's invasive. The simpler path: convert `DelegateTaskWiring.progress_callback` into an `Arc<RwLock<Option<SubagentProgressCallback>>>` that `run_web_turn` swaps for the turn's duration. See Risk #1 — this is the single biggest lifecycle decision in Plan 02.

### Pattern 4: Cross-signal reactivity via `use_effect` (D-07)

**What:** Subscribe to one signal (the counter), trigger another action (`.restart()`) when it changes.

**Verified working in this codebase** at `warp_hermes.rs:104-128, 400, 555-557`, e.g.:


### Pattern 5: Recently-terminated diff with frozen snapshot (D-02, D-11, D-13)

```rust
// Source: synthesis from agents.rs:33-37 + CONTEXT.md D-02/D-11/D-13
let prev_ids: HashSet<String> = recently_terminated.read().keys().cloned().collect();
let current_ids: HashSet<String> = agents_list.iter().map(|a| a.id.clone()).collect();

// D-13: any current id that was in recently_terminated → drop (replay wins)

    if prev_ids.contains(id) {
        recently_terminated.write().remove(id);
    }
}

// D-02 + D-11: any id that disappeared since last fetch → enter HOLD with frozen snapshot
// (need prev_agent_list signal to diff against; not just prev_ids)
let prev_agents: Vec<AgentInfo> = prev_agents_signal.read().clone();
for prev in &prev_agents {
    if !current_ids.contains(&prev.id) && !recently_terminated.read().contains_key(&prev.id) {
        // Frozen uptime_secs (D-11) — `prev` is a snapshot captured at last poll
        recently_terminated.write().insert(prev.id.clone(), (prev.clone(), Instant::now()));
    }
}
prev_agents_signal.set(agents_list.clone());
```

### Anti-Patterns to Avoid

- **Anti-pattern:** Holding `recently_terminated.read()` across `.await` (clippy will panic at runtime per `clippy.toml:1-13`). Always `.read().clone()` into an owned collection before awaiting.
- **Anti-pattern:** Constructing the `SubagentProgressCallback` once at app init and storing it in `DelegateTaskWiring` (per-session). The callback must capture the **per-turn** `tx: UnboundedSender`. Per-session capture leaks the channel from turn N into turn N+1's lifetime and breaks ordering. See Risk #1.
- **Anti-pattern:** Using `cb.forget()` to "drop" a wasm-bindgen Closure (leaks the JS-side reference AND prevents `remove_event_listener_with_callback` from working). Always store in `Signal<Option<Closure<_>>>` and clean up in `use_drop`. Verified at `wheel.rs:262-279` and `warp_hermes.rs:541-551`.
- **Anti-pattern:** Per-AgentCard `use_future` decay timers — would spawn N futures per render. Use one screen-level decay sweep.
- **Anti-pattern:** Reading `subagent_events` inside the rsx! block of ScreenAgents (would force a re-render of the entire grid on every event). Read it ONLY inside `use_effect` and trigger `.restart()`.

## Common Pitfalls

### Pitfall 1: Signal borrow across `.await` in poll loop
**What goes wrong:** Runtime panic ("already mutably borrowed") because the `.read()` `GenerationalRef` held during `TimeoutFuture::new().await` collides with a `.write()` from a kill/interrupt RPC completing.
**Why it happens:** Clippy lint flags this but only on explicit `.read()` calls; harder to spot when calling signal as function `recently_terminated()` returns a Copy of small types but a HashMap is not Copy.
**How to avoid:** Always `.read().clone()` into owned `HashMap` before any await; or use call-syntax `signal()` only on `Copy` types (bool, u64). The `agents.rs:33-37` pattern is the established idiom.
**Warning signs:** clippy.toml will catch most cases at compile; runtime panic with `borrow already taken` if it slips through.

### Pitfall 2: Per-session vs per-turn callback lifetime mismatch (Plan 02)
**What goes wrong:** Building `SubagentProgressCallback` once inside `AppState::init` captures the wrong `tx` — there is no `tx` at init time. Building per-turn requires either threading the callback through `run_web_turn` (clean) or making `DelegateTaskWiring.progress_callback` mutable+swappable (invasive).
**Why it happens:** `progress_callback: None` at state.rs:124 is set during `init()`. The natural read of D-06 is "fix that None" — but you can't fix it there because there's no `tx` to capture.
**How to avoid:** Add a `subagent_progress_callback: Option<SubagentProgressCallback>` parameter to `run_web_turn` (mirroring the existing `tool_progress_callback` parameter at state.rs:174). Then inside `run_web_turn`, either (a) clone the runtime bundle's `DelegateTaskWiring` and swap in the per-turn callback before building the registry, or (b) reach into the existing registry and update the captured callback via interior mutability. Option (a) is cleaner but requires re-building part of `AppRuntimeBundle` per turn. Option (b) requires changing `DelegateTaskWiring.progress_callback` to `Arc<RwLock<Option<SubagentProgressCallback>>>`.
**Warning signs:** If a ws event fires for the previous turn's session_id, you've captured the wrong tx.

### Pitfall 3: `is_ws_connected` initial value race (D-08)
**What goes wrong:** Component mounts with `is_ws_connected = false` (initial value) and immediately starts polling at 1500ms. Then a few hundred ms later the ws opens and the poll widens to 5s — net result: 4 wasted requests in the first 6s. Or the opposite: initial value `true` and a slow ws handshake leaves 5s gap before fallback.
**Why it happens:** `dioxus_fullstack::use_websocket` doesn't expose a synchronous "connecting" state.
**How to avoid:** Initial value should be `false` (assume disconnected → fast poll = safe default). Set `true` on first Text/Binary frame in `warp_hermes.rs:174-187`; set `false` on `Message::Close` (line 188), `ws.is_err()` (line 150), or recv error (line 372).
**Warning signs:** Devtools shows >1 fetch/second consistently — `is_ws_connected` never went `true`.

### Pitfall 4: `recently_terminated` HashMap grows during burst of terminations
**What goes wrong:** A `delegate_task` with 100 batch tasks completing in <5s puts 100 entries into the map. No DoS guardrail per CONTEXT.md "deferred" section.
**Why it happens:** Decay sweep runs every 1s, so the map peaks at ~100 entries for ~5s.
**How to avoid:** Acceptable for human-cadence operation; document the bound. Future phase if it becomes a problem. Be aware that the rsx! render iterates BOTH `agents_list` AND `recently_terminated` entries — 100 dim cards is visually noisy but technically fine.
**Warning signs:** Visible slowdown only above 1000 simultaneous HOLD cards.

### Pitfall 5: Page Visibility return resumes poll mid-tick
**What goes wrong:** User leaves tab for 30s, returns. The poll loop was awaiting `TimeoutFuture::new(1500)` when `hidden` became true — it will sleep its full 1500ms before re-checking. Result: up to 1.5s gap between tab-return and first fetch.
**Why it happens:** `TimeoutFuture` is not cancellable from outside the future.
**How to avoid:** On `visibilitychange` listener firing (when `!hidden`), call `agents_resource.restart()` immediately from the listener. The poll loop will then re-arm naturally on its next iteration. CONTEXT.md D-04 explicitly notes: "On visibility return, fire an immediate fetch then resume the interval."
**Warning signs:** Returning to tab shows stale data for >2s.

### Pitfall 6: Cleaning up the per-card `killing/interrupting` aggregator on AgentCard drop
**What goes wrong:** If aggregation uses a HashSet<String>-via-context (Pattern 2 option 2 above), the AgentCard must remove its id when it unmounts — otherwise the screen-level "busy" state stays true forever.
**Why it happens:** Dioxus does not auto-clean signal state owned by parent context when a child unmounts.
**How to avoid:** Use `use_drop` in AgentCard to remove its id from the busy-set. Same pattern as `warp_hermes.rs:541-551`.

### Pitfall 7: `web-sys` feature flag missing for `Event`
**What goes wrong:** `Closure::<dyn FnMut(WebEvent)>::new(...)` won't compile because `web_sys::Event` is gated behind the `"Event"` feature.
**Why it happens:** Current `Cargo.toml:19-32` enables `KeyboardEvent`, `PointerEvent`, but not bare `Event`.
**How to avoid:** Add `"Event"` to the web-sys features list. Verify before coding by attempting a minimal compile.

### Pitfall 8: Subagent very-short-lived (<1500ms) still missed by Plan 01
**What goes wrong:** A subagent that registers, runs, and unregisters in 800ms never appears in any `api_agents_list` snapshot polled at 1500ms intervals. Plan 01 alone CANNOT capture this — it requires Plan 02's event-driven trigger.
**Why it happens:** The poll cadence is the lower bound on observability.
**How to avoid:** This IS the explicit motivation for Plan 02 per CONTEXT.md "specifics" section. Plan 01 ships with the documented limitation; Plan 02 closes it. Verifier must exercise both with delegate_task at different durations.

```rust
        }
    }
});
```

## Open Questions

1. **Does `std::time::Instant::now()` work on `wasm32-unknown-unknown` in the project's toolchain?**
   - What we know: as of Rust 1.78+, `std::time::Instant` is supported on `wasm32-unknown-unknown` via panicking-on-`SystemTime::now()` but **monotonic** `Instant::now()` IS supported. The codebase uses `std::time::Instant` heavily in non-WASM crates (e.g. `state.rs:534`).
   - What's unclear: the iron_hermes_ui WASM target hasn't been observed using `Instant::now()` in any current file (grep finds zero matches in `crates/iron_hermes_ui/src/`).
   - Recommendation: Planner should attempt with `std::time::Instant` first. If runtime panics under `dx serve` (or compile fails), add `web-time = "1.1"` as a dep and replace with `web_time::Instant`. The API surface is identical.

2. **Is there an existing `is_ws_connected` signal already exposed on `WarpHermes` that the planner missed?**
   - What we know: `warp_hermes.rs:135` declares `let mut ws = dioxus_fullstack::use_websocket(...)`. The ws handle has `is_err()` and `connect()` accessors. There's NO existing `is_connected` `Signal<bool>`.
   - What's unclear: nothing — the planner must add one.
   - Recommendation: Add `let mut is_ws_connected = use_signal(|| false);` next to the existing ws declaration, then `is_ws_connected.set(true)` inside the Text/Binary arm and `.set(false)` inside Close/error arms. Provide via `use_context_provider`.

3. **Should the planner aggregate per-card `killing`/`interrupting` via `use_context_provider<Signal<HashSet<String>>>` or via screen-level props passed down?**
   - What we know: Both patterns work. The Phase 26.7 D-02 hybrid-state model leaves this to the planner.
   - What's unclear: which is fewer LoC and lower-coupling.
   - Recommendation: Use context provider — keeps AgentCard signature unchanged, allows cleanup via `use_drop` on the card, mirrors how `ShellSettings` is provided in `warp_hermes.rs` (CONTEXT D-02).

4. **Does Plan 02 need to add the new event variant to the EXISTING `ChatStreamEvent` enum, or invent a separate event channel?**
   - What we know: CONTEXT.md D-06 says "reuses the existing chat WebSocket"; one channel is the contract.
   - What's unclear: whether extending the enum risks breaking any other parser.
   - Recommendation: Extend the enum. Search confirmed two parse sites (`warp_hermes.rs:210-355` and `hermes_app/mod.rs:166-225`) — BOTH must be updated to match-arm the new variant (or use `_` catch-all that already exists if `Continue` semantics). Verify both before commit.

## Risk Register (top 3)

### Risk #1: Per-turn vs per-session callback lifetime mismatch (HIGH)
**What can go wrong:** Building the `SubagentProgressCallback` once in `AppState::init` (the obvious read of "fix `progress_callback: None` at state.rs:124") fails because the per-turn `tx` doesn't exist at init time. Naive workarounds (storing a global `tx`, using a session-scoped channel) leak the channel from turn N into turn N+1.
**Detection:** Server-side unit test (Wave 0) that verifies the callback receives the **current** turn's `tx`, not a stale one. Or: a ws event arriving on a turn AFTER the turn that spawned the subagent.
**Mitigation:** Add `subagent_progress_callback: Option<SubagentProgressCallback>` parameter to `run_web_turn`. Inside `run_web_turn`, EITHER (a) build a per-turn `DelegateTaskWiring` clone and rebuild the per-turn portion of the tool registry, OR (b) swap into an `Arc<RwLock<Option<SubagentProgressCallback>>>` field on the runtime bundle. Option (b) is one line of plumbing; option (a) is cleaner architecturally. Planner picks; researcher recommends (b) as the lower-LoC path with a comment documenting the swap discipline.
**Severity:** HIGH — gets baked into the public surface of `run_web_turn` and the runtime bundle.

### Risk #2: Subagent that lives <1500ms still missed by Plan 01 (MEDIUM, by-design)
**What can go wrong:** Plan 01 ships, user runs `delegate_task` with a fast child (e.g. simple read_file in <800ms), expects to see ENDED card, sees nothing.
**Detection:** UAT scenario "live-spawn <1s" — listed in CONTEXT.md "Closes with" but ONLY tested against Plan 02.
**Mitigation:** This is by design — Plan 02 closes it. Plan 01 UAT should explicitly use a `delegate_task` that takes >2s (e.g. a model call to a real LLM) so the agent is present in at least one poll snapshot before disappearing.
**Severity:** MEDIUM — accepted limitation, but UAT script must reflect it.

### Risk #3: WASM `Instant::now()` panics or returns garbage in `wasm32-unknown-unknown` target (LOW, fallback-cheap)
**What can go wrong:** Decay sweep computes `ts.elapsed()` → panic, or always returns 0, or returns a huge constant.
**Detection:** First UAT run will panic visibly under `dx serve` browser devtools. Hard to miss.
**Mitigation:** If `std::time::Instant` doesn't work, add `web-time = "1.1"` (single-file substitution — API is identical). 5-minute fix.
**Severity:** LOW — discovered immediately, fix is trivial.

## Sequencing Note (Plan 01 → Plan 02 hand-off)

The phase is one wave with two ordered plans. Some Plan 01 decisions should anticipate Plan 02's needs:

1. **Add `use_context_provider<Signal<u64>>` for `subagent_events` in Plan 01.** Even though Plan 01 never increments it, providing the context slot in `WarpHermes` early lets Plan 02 just `use_context::<...>()` in `ScreenAgents` without touching `WarpHermes` again. Plan 02 then ONLY adds the increment site in the ws match arm and the `use_effect` watcher.

2. **Add `use_context_provider<Signal<bool>>` for `is_ws_connected` in Plan 01.** Same rationale — initial value `false` is safe (Plan 01 polls at 1500ms baseline regardless; the `false` value never widens the poll).

3. **Lift `killing`/`interrupting` to context in Plan 01.** Plan 02's `use_effect` over `subagent_events` SHOULD also check the busy-set — otherwise a ws event arriving during a kill RPC would race the kill's own `agents_resource.restart()` call. Centralizing the busy-set in Plan 01 means Plan 02's `use_effect` body is just `if !is_busy() { agents_resource.restart(); }`.

4. **Name the poll-loop's helper functions with verbs that Plan 02 can naturally extend.** E.g., `should_poll()` → `should_poll(connected: bool) -> Duration` extended in Plan 02. Avoid baking `1500` as a literal in the poll loop body; declare `POLL_INTERVAL_BASELINE_MS: u64 = 1500` so Plan 02 just adds `POLL_INTERVAL_WS_CONNECTED_MS: u64 = 5000` and the loop pivots on the bool.

5. **Plan 01 does NOT extend `ChatStreamEvent`.** That's a Plan 02 edit. Plan 01's ws parse arm is untouched.

6. **Document the Plan 01 → Plan 02 contract surface as one line in the Plan 01 SUMMARY:** "Context slots `subagent_events: Signal<u64>` and `is_ws_connected: Signal<bool>` provided but unused; Plan 02 wires the increment site." This prevents future-Claude from concluding the unused signals are dead code and removing them.
