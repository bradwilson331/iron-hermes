## UAT-1 Iteration (2026-05-18)

First UAT pass (against commit `603b819c`) returned partial fail:
- **Pass:** Scenarios 1, 2, 3, 4, 7, 8, 9, 10 (SSR/context, polling rate, tab-hidden pause, natural-termination HOLD, regression checks, build hygiene).
- **Fail:** Scenarios 5 (KILL HOLD card) and 6 (INTERRUPT HOLD card) — the agent card disappeared immediately without showing the 5s ENDED/gray pill state. Server-side RPCs returned 200 normally; only the client-side diff path was failing to capture the termination on the RPC-driven termination paths (the natural-termination path worked, so the diff logic itself was sound).

**Hotfix commit `121db4ec`** ("fix(26.7.1-01): kill/interrupt HOLD card now renders via diff path"):

**Root cause:** The diff and `prev_live` snapshot ran unconditionally even when `agents_resource` was in the suspended (`None`) state during a `.restart()` refetch. The suspended state fell through to the `Vec::new()` fallback at line 64, which caused two problems:

1. `prev_live` got overwritten with `[]` during the suspended window. When the real fetch later resolved with `[]`, the diff saw `prev=[]` vs `curr=[]` and produced zero terminations — the `[A]→[]` signal was silently lost.
2. Even when the diff DID produce a termination on a suspended-state render, a subsequent render would re-snapshot `prev_live = []` before the operator could see anything.

This affected the kill/interrupt paths because both fired `agents_resource.restart()` from their async handlers, opening the suspended-state window. The natural-termination path didn't suffer because terminations were caught by the periodic poll's restart, which timed itself between renders rather than during synchronous handler chains.

**Fix applied:**

- Gate both the `diff_terminations` call AND `prev_live.set()` on `matches!(agents_resource(), Some(Ok(_)))`. The suspended state is now a no-op for the diff path — `prev_live` only updates on resolved data, so the `[A]→[]` transition is preserved across the restart window.
- **INTERRUPT handler** now calls `agents_resource.restart()` (parity with KILL). Previously interrupt relied on the next periodic poll (1.5s later) to detect termination, which left a visible gap.
- **KILL handler** now does: kill RPC → 100ms wait (for server `RegistrationGuard::drop`) → `agents_resource.restart()` → decrement `rpc_in_flight`. The decrement-after-restart ordering prevents the periodic poll from racing the kill-driven restart.
- **INTERRUPT handler** now does: interrupt RPC → 500ms visual hold (also gives server time to finalize) → restart → decrement.
- Added wasm-gated `web_sys::console::log_1` instrumentation in the diff path so the UAT operator can trace `resolved=X prev=N curr=M recently_terminated=K` per render.

**Build status post-hotfix:** Native + wasm32 builds green; all 4 `agents_diff` unit tests still pass; the original Plan 01 verification grep counts remain valid.

**Re-test required:** Scenarios 5 (KILL) and 6 (INTERRUPT). All other scenarios already passed and are not affected by this hotfix.

## UAT-2 Iteration (2026-05-18)

UAT-2 (against commit `4c6535c0`) revealed a regression introduced by the UAT-1 hotfix and confirmed the original kill/interrupt HOLD failure was still present:

- **Finding A — BASELINE POLLING REGRESSION:** zero `/api/agents/list` calls over 30 seconds on a fresh Agents page navigation. UAT-1 scenario 2 had passed; UAT-2 confirmed it broke after the UAT-1 hotfix. The expected ~20 fetches at 1.5s baseline did not appear.
- **Finding B — KILL/INTERRUPT HOLD STILL FAILS:** scenarios 5 and 6 still failed because the poll loop wasn't running at all to refresh the resource.

**Hotfix commit `e72b8bdd`** ("fix(26.7.1-01): swap use_server_future to use_resource — fixes baseline polling regression"):

**Root cause (the user's H5 hypothesis was correct):** `use_server_future(fn)?` returns `Result<Resource<T>, RenderError>`. The `?` operator early-returns when the resource is in the loading (`None`) state, propagating `RenderError` up the component tree.

- In Dioxus 0.7, hooks must register in the same order on every render. When `?` fires its early-return before reaching the `use_future` hook registrations, the poll loop and decay sweep never get spawned.
- Even after the resource first resolved and the `use_future`s DID spawn, every subsequent `agents_resource.restart()` cycled the resource back into the loading state. The next render hit `?` early-return again, violating the hook-ordering invariant.
- The UAT-1 hotfix made this dramatically worse by gating `prev_live.set()` on `resource_resolved`. While the gate was correct in principle, it also meant that the kill/interrupt-triggered `restart()` calls produced longer suspended-state windows during which `?` killed off later hooks.

**Fix:** swap `use_server_future` for `use_resource`.

- `use_resource(|| async move { api_agents_list().await })` returns `Resource<Result<T, ServerFnError>>` with no SSR suspension wrapper.
- The resource's value via `()` returns `Option<Result<T>>` where `None` = loading — same semantics as before but no `?` early-return.
- All hooks below the `use_resource` call now register on every render regardless of resource state.
- Drop in SSR suspension is acceptable here: `ScreenAgents` is class-toggle-mounted (always present in DOM, `is-active` class gates visibility), so first-paint suspension never blocked the user from seeing other screens. The empty-list fallback during initial load is visually indistinguishable from an empty registry.

**Plus: per-iteration poll-loop instrumentation** (so future UAT iterations can localize failures fast):
- `[26.7.1-01 poll] tick hidden=X in_flight=Y` every iteration so UAT can see the loop is alive and which gate (if any) is suppressing fetches.
- `[26.7.1-01 poll] restart() fired` immediately after `.restart()` so we can correlate with the network tab.

**Build status post-hotfix:** Native + wasm32 builds green; all 4 `agents_diff` unit tests pass.

**Re-test required:** Scenarios 1 (SSR no panic), 2 (baseline polling — the regression), 5 (KILL HOLD), and 6 (INTERRUPT HOLD).

## UAT-3 Iteration (2026-05-19)

UAT-3 (against commit `74fded98`) revealed a new failure mode: the app FROZE. DevTools Console showed the `[26.7.1-01 diff]` log line repeating dozens of times at the same wall-clock timestamp (`resolved=true prev=0 curr=0 recently_terminated=0 × N`) — an infinite render loop, not a polling issue.

**Hotfix commit `5d2f51ca`** ("fix(26.7.1-01): gate prev_live.set on content change — fixes infinite render loop"):

**Root cause:** `prev_live.set(agents_list.clone())` was called unconditionally on every resolved render, even when the value didn't change.

- Dioxus 0.7 signals do NOT deduplicate identical writes — every `.set()` schedules a re-render.
- The diff block then re-runs, calls `.set()` again, schedules another re-render, ad infinitum.
- The `[26.7.1-01 diff]` log fires each loop iteration because the `println` is BEFORE the set, hence the repeated identical lines.

**Why didn't this loop manifest in UAT-1?** Because `use_server_future`'s suspended-state `?` early-return on every alternate render skipped the diff block entirely, naturally pacing the loop. UAT-2's swap to `use_resource` removed that accidental pacing, exposing the latent bug.

**Fix:** gate `prev_live.set()` on actual content change.

- `AgentInfo` derives `PartialEq` (verified at `src/server/api.rs` line 360).
- `prev_snapshot` is already cloned from `prev_live.read()` earlier in the render, so it represents the CURRENT value of `prev_live` at the start of this render. Comparing against it is equivalent to checking whether the upcoming `.set()` would be a no-op.
- Now: `if resource_resolved && prev_snapshot != agents_list { prev_live.set(...) }`
- Identical-content renders are now no-ops on the diff path; only real state transitions (agent appeared / disappeared / changed) trigger the snapshot update.
- No additional `.read()` held across `.set()` — signal-borrow discipline preserved (clippy.toml).

**Build status post-hotfix:** Native + wasm32 builds green; all 4 `agents_diff` unit tests pass.

**Re-test required:** Scenarios 1 (SSR no panic), 2 (baseline polling — should now fire ~1x per ~1.5s, NOT dozens of times per second), 5 (KILL HOLD), and 6 (INTERRUPT HOLD).

## UAT-4 Iteration (2026-05-19)

UAT-4 (against commit `40c15d98`) confirmed the diff path is correct, baseline polling is sane (~1.5s), `use_resource` is the right hook, and the `prev_live` dedup gate works. But the kill/interrupt HOLD still failed — DevTools Console showed:

```
04:06:36 [web] panicked at .../library/std/src/sys/pal/wasm/../unsupported/time.rs:13:9:
time not implemented on this platform
```

…followed by a wasm-bindgen WARN about an "imported JS function that was not marked as `catch` threw an error". The wasm render task died from that point — that's why `curr` stayed at 1 across subsequent polls and `recently_terminated` never grew.

**Hotfix commit `a0532c38`** ("fix(26.7.1-01): swap std::time::Instant → web_time::Instant for wasm32"):

**Root cause:** `std::time::Instant` is unimplemented on `wasm32-unknown-unknown`. rustc's stdlib for that target has no monotonic clock — calling `Instant::now()` literally hits an `unreachable!()` in `library/std/src/sys/pal/wasm/../unsupported/time.rs:13:9`. This was documented as a possible failure mode in the original Plan 01 checkpoint scenario 9 but had been masked by earlier failures (SSR panic, render loop).

**Fix:** swap to `web_time::Instant`.

- Added `web-time = "1.1"` to `crates/iron_hermes_ui/Cargo.toml` as a client-only WASM dep (placed next to `gloo-timers`).
- `web_time::Instant` is API-identical to `std::time::Instant`: it's a `pub use` of `std::time::Instant` on native targets, and a `Performance.now()`-backed shim on wasm32. Same `Duration` arithmetic, same `.elapsed()` method.
- `agents.rs` line 76 (HashMap value type) and line 135 (`Instant::now()` call in the insert) now use `web_time::Instant`.
- The decay sweep's `std::time::Duration::from_secs(5)` is unchanged — `Duration` is a plain struct that works on both targets. `ts.elapsed()` returns `std::time::Duration` on native and `web_time::Duration` (= `std::time::Duration`) on wasm.
- `Cargo.lock` regenerated to pin the new transitive deps; committed alongside since the lockfile is now tracked at the workspace root (per `0b92d420`).

## Deviations from Plan

### Auto-applied (Rule 2 / pattern decisions)

**1. [Pattern Decision] Poll interval stored in variable, not direct literal**
- Found during: Task 4
- Issue: Signal-borrow discipline (clippy.toml) requires releasing the `ws_connected_ctx.read()` borrow before calling `TimeoutFuture::new()`. The plan's verify grep for `TimeoutFuture::new(1_500)` assumed the literal would be inlined.
- Fix: `let interval_ms: u32 = if *ws_connected_ctx.read() { 5_000 } else { 1_500 };` followed by `TimeoutFuture::new(interval_ms).await`. The `1_500` literal IS present in the file; the verify grep `grep -c 'd.hidden()\|document().*hidden'` matches with count ≥ 1.
- Impact: Semantically correct, clippy-clean. Plan's intent fully met.

**2. [Pattern Decision] document.hidden() per-iteration (no visibilitychange listener)**
- Found during: Task 4
- Issue: The plan explicitly recommended the simpler per-iteration approach: "if you take the simpler 'poll-loop-checks-document.hidden() each iteration' approach (no listener at all — recommended per D-04 simplicity), this import is unnecessary."
- Fix: Used `web_sys::window().and_then(|w| w.document()).map(|d| d.hidden())` inline in the poll loop. No `use_drop`, no `JsCast`, no listener teardown complexity.
- Impact: Simpler, fewer lines, no memory management edge cases.

- Fix: None — out of scope. Logged to deferred items.
