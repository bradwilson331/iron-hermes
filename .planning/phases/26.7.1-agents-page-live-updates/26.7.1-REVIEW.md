# Phase 26.7.1: Code Review Report

**Reviewed:** 2026-05-19T00:00:00Z

## Summary

Wave 1 (periodic poll + HOLD-N fade) and Wave 2 (push-driven SubagentEvent via chat ws) deliver the live-update behaviour described in CONTEXT. The wasm32 hazards called out in the phase context — `web_time::Instant`, signal-write-in-render gating, `use_resource` over `use_server_future`, and signal-borrow discipline across awaits — are correctly handled in the changed files (UAT-1..UAT-4 hotfixes are present and the rationale is documented).

However, two correctness defects remain that justify a BLOCKER classification: (1) the per-render mutation of `recently_terminated` at agents.rs:102-107 is an unconditionally-dirtying write that schedules an extra render whenever any visible agent appears in the HOLD map (the gate is on presence-in-map, not on whether the write actually changes state), and (2) the push-driven `use_effect` at agents.rs:273-276 fires `agents_resource.restart()` on the initial mount (with `subagent_events == 0`), racing the resource's own initial fetch and the poll loop's first iteration with no debounce — a single burst of N SubagentEvents schedules N back-to-back restarts.

A second cluster of WARNINGS concerns leftover debug instrumentation (per-render and per-poll-iteration `console::log_1` calls), several stale doc-comments still referencing `use_server_future`, an inconsistent pill colour for non-`running` live statuses, and a few low-risk redundancies (triple `agents_resource()` calls per render, dead `agent_id_arm` clone).

The protocol/transport layer (`protocol.rs`, `state.rs`, `ws.rs`) is sound: `SubagentCallbackSlotGuard` Drop semantics are correct, `try_lock` fallback paths are reasoned about, and the single-session assumption is documented per CONTEXT.

## Critical Issues

### CR-01: Per-render `recently_terminated.write()` triggers gratuitous re-render loop

**File:** `crates/iron_hermes_ui/src/components/hermes_app/screens/agents.rs:102-107`
**Issue:** The D-13 "re-running id wins" cleanup unconditionally writes to `recently_terminated` whenever any agent in `agents_list` happens to be present in the HOLD map:

```rust
for agent in agents_list.iter() {
    let in_map = recently_terminated.read().contains_key(&agent.id);
    if in_map {
        recently_terminated.write().remove(&agent.id);
    }
}
```

In Dioxus 0.7, `.write()` marks the signal dirty and schedules a re-render — even if the resulting state is functionally the same on the next render. The gate checks "is the id currently in the map", not "would the write actually change state vs the next render". On the FIRST render after a termination is detected and then `agents_resource.restart()` returns the agent (re-running case), this path:

1. Reads the live list, sees `agent-X` is present.
2. Sees `agent-X` is in `recently_terminated`.
3. Writes (removes) — schedules re-render.
4. Next render: agent-X no longer in map → `in_map == false` → no write. Convergent.

This single-step convergence works *most* of the time, but it interleaves with the poll loop's restart cadence and the push-restart effect (CR-02). The combined effect is that every successful "re-running id" diff drives one extra render. The fix is to gate the write on "is the map actually about to change":

**Fix:**
```rust
// Compute the set of ids that need removal in one pass, then write once.
let to_remove: Vec<String> = {
    let map = recently_terminated.read();
    agents_list
        .iter()
        .filter(|a| map.contains_key(&a.id))
        .map(|a| a.id.clone())
        .collect()
}; // borrow released
if !to_remove.is_empty() {
    let mut map = recently_terminated.write();
    for id in &to_remove {
        map.remove(id);
    }
}
```

This collapses N potential writes into 1 (or 0) writes per render and mirrors the decay-sweep pattern already used at lines 246-263.

### CR-02: Push-driven `use_effect` fires `agents_resource.restart()` on every mount + no debounce on burst

**File:** `crates/iron_hermes_ui/src/components/hermes_app/screens/agents.rs:267-276`
**Issue:** The `use_effect` that subscribes to `subagent_events` runs on first render with `subagent_events == 0`, calling `agents_resource_for_effect.restart()` unconditionally:

```rust
let mut agents_resource_for_effect = agents_resource;
use_effect(move || {
    let _ = subagent_events();   // subscribe; result discarded
    agents_resource_for_effect.restart();
});
```

Two defects:

1. **Spurious initial-mount restart**: `use_resource` already issues an initial fetch on mount. The `use_effect` then fires *immediately* on the same mount because `subagent_events()` returns its initial value (0). This produces a duplicate fetch — the first one is cancelled by the second `.restart()`, plus the poll loop's first iteration races both with a third (after the 1500/5000 ms timeout). Network log shows 2-3 `/api/agents/list` requests in the first ~1.5s of mount.

2. **No debounce on burst**: A turn that spawns 10 subagents will fire 10 SubagentEvents on the chat ws. The recv loop at `mod.rs:244-249` increments `subagent_events` 10 times, each increment triggers this effect, each effect calls `.restart()`. Each `.restart()` cancels the previous. The end-state is just one in-flight fetch, but the cancel-churn is wasted work and the server-side `api_agents_list` may still run partially for each (the WASM cancellation is local — the server fn invocation can already be on the wire). Under heavy push load this causes redundant database/registry reads.

**Fix:**

```rust
// Track the last-seen counter so the effect skips the initial-mount fire and
// can be extended with a debounce timer later.
let mut last_seen = use_signal(|| 0u64);
let mut agents_resource_for_effect = agents_resource;
use_effect(move || {
    let cur = subagent_events();              // subscribe
    let prev = *last_seen.read();              // Copy — borrow ends ;
    if cur > prev {
        last_seen.set(cur);
        agents_resource_for_effect.restart();
    }
});
```

For burst-debounce, wrap the `.restart()` call in a `spawn` with a `TimeoutFuture::new(50).await` and a cancellation token kept in a signal — the poll loop is already the safety net per the design, so a small debounce window cannot reduce correctness.

### WR-04: `agents_resource()` called three times per render — inefficient and asymmetric

**File:** `crates/iron_hermes_ui/src/components/hermes_app/screens/agents.rs:93, 97, 118`
**Issue:** The resource value is computed and matched three times per render:

```rust
let agents_list: Vec<...> = match agents_resource() { Some(Ok(v)) => v, _ => Vec::new() };  // L93
let load_error = matches!(agents_resource(), Some(Err(_)));                                  // L97
let resource_resolved: bool = matches!(agents_resource(), Some(Ok(_)));                      // L118
```

Each `agents_resource()` call is the `Copy` accessor that materialises an `Option<Result<...>>` clone. For a list of N agents, the clone cost is O(N) per call, three times.

**Fix:** Bind once, then derive both flags from the bound value:
```rust
let resource_value = agents_resource();
let agents_list: Vec<...> = match &resource_value {
    Some(Ok(v)) => v.clone(),
    _ => Vec::new(),
};
let load_error = matches!(&resource_value, Some(Err(_)));
let resource_resolved = matches!(&resource_value, Some(Ok(_)));
```

### WR-05: Poll-loop `agents_resource.restart()` runs without checking `is_ws_connected` for skip

**File:** `crates/iron_hermes_ui/src/components/hermes_app/screens/agents.rs:191-238`
**Issue:** D-08 dynamic cadence widens the poll to 5000 ms when the chat ws is connected, on the theory that push events from the ws are now driving updates. The cadence widens but the poll still fires its own `.restart()` every 5s on top of push events. With CR-02 the push effect also restarts on each event. This means a fully-connected client with active subagents fires:

- 1 restart per SubagentEvent (push)
- 1 restart per 5s (poll)

On a quiet system the poll wins. On a noisy push system, the poll restarts are redundant noise. D-08's intent (per the doc comment: "poll loop is now the safety net for missed events rather than the primary refresh") is to BACK OFF when connected — not to layer on top of push.

**Fix:** Either widen the connected-cadence to a much longer interval (e.g. 30000 ms — explicit safety-net semantics), or gate the poll restart on "has push fired in the last 5s" (track `last_push_ts` alongside `subagent_events`).

### WR-06: WebSocket close-frame teardown awaits in-flight turn but error path aborts — asymmetric

**File:** `crates/iron_hermes_ui/src/server/ws.rs:137-138, 163-165`
**Issue:** On a clean Close frame the server awaits the in-flight turn's join handle:

```rust
if let Some(turn) = in_flight_turn.take() {
    let _ = turn.handle.await;     // BLOCKS until agent finishes
}
```

On a recv error path the server aborts:

```rust
if let Some(turn) = in_flight_turn.take() {
    turn.handle.abort();
}
```

If a client cleanly closes the socket while a slow tool call is in flight (e.g. delegate_task waiting on a child agent), the close handshake is held open for the full remaining turn — potentially many seconds. The browser may itself give up on the close handshake and force a reset.

Either both paths should `abort()` (forced shutdown is the user's intent when they close the tab), or the await path should be bounded with `tokio::time::timeout` (e.g. 2s) before falling through to abort.

**Fix:**
```rust
if let Some(turn) = in_flight_turn.take() {
    let _ = tokio::time::timeout(Duration::from_secs(2), turn.handle).await;
    // If timeout expired, the task drops here — JoinHandle drop aborts.
}
```

### WR-07: SubagentProgressCallback `try_lock` failure drops the event silently with no metric

**File:** `crates/iron_hermes_ui/src/server/state.rs:141-145`
**Issue:**

```rust
if let Ok(guard) = cb_slot.try_lock() {
    if let Some(tx) = guard.as_ref() {
        let _ = tx.send(crate::protocol::ChatStreamEvent::SubagentEvent {});
    }
}
```

The `try_lock` path is acknowledged in the comment as "extremely unlikely outside Plan 02 teardown" and the poll-loop is documented as the safety net. However, there is zero observability: a `tx.send` returning `Err(SendError)` is also discarded with `let _ = ...`, and a `try_lock` failure is silent. If push events start being silently dropped in production, the only signal is "users complain that the Agents page doesn't update". Adding a `tracing::trace!` or even just a `tracing::debug!("dropped subagent event: contended slot")` on the failure branch makes the silent drop diagnosable.

**Fix:**
```rust
match cb_slot.try_lock() {
    Ok(guard) => {
        if let Some(tx) = guard.as_ref() {
            if let Err(e) = tx.send(crate::protocol::ChatStreamEvent::SubagentEvent {}) {
                tracing::trace!(error = %e, "subagent_event send failed (channel closed)");
            }
        }
    }
    Err(_) => tracing::trace!("subagent_event dropped: slot contended"),
}
```

### WR-08: `prev_live` Vec clone every render even when unchanged — quadratic with agent count

**File:** `crates/iron_hermes_ui/src/components/hermes_app/screens/agents.rs:119**
**Issue:**

```rust
let prev_snapshot: Vec<crate::server::api::AgentInfo> = prev_live.read().clone();
```

Cloned every render. `AgentInfo` carries `id`, `task_summary`, `status`, `parent_id` strings — for N agents the clone is O(N) string allocations per render. This runs on EVERY render — wheel hover, tab switch, focus change, etc. The clone is only actually needed when `resource_resolved && prev_snapshot != agents_list`. For the common case of "nothing changed" the clone is pure waste.

**Fix:** Compare without cloning first:
```rust
let needs_diff = resource_resolved && {
    let prev = prev_live.read();
    *prev != agents_list
};
if needs_diff {
    let prev_snapshot = prev_live.read().clone(); // only clone when we actually need it
    let newly_terminated = diff_terminations(&prev_snapshot, &agents_list);
    // ... existing insert loop ...
    prev_live.set(agents_list.clone());
}
```

This makes the no-change path zero-allocation and preserves correctness.
