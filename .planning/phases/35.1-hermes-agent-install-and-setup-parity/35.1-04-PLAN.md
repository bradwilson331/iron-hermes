<objective>
Gap A closure — backfill `providers.{provider}.api_key_env` for existing configs that are missing it.

Manual smoke testing revealed: the test config at `/Users/twilson/test/config.yaml` has `model.provider: openrouter` but `providers:` is empty. The doctor check reports `providers.openrouter.api_key_env` as missing. This happens because older setup paths (or setup paths that went through `run_model_section` with its backward-compat `apply_api_key_answer` call) never wrote the canonical providers map entry.

Fix: add `fn backfill_providers_api_key_env(config: &Config, hermes_home: &Path) -> Result<()>` that silently writes the missing key if it can be inferred from an existing env var in the .env file or process environment. Call it at the top of `run_minimum_viable_flow` (before the Welcome banner) and in `run_model_section` (after config loads). This ensures any setup invocation on an existing config heals the providers map as a side effect.

No prompting — backfill is silent and skipped if the key is already present OR if no env var is found.
</objective>

       if in_dotenv || in_env {
