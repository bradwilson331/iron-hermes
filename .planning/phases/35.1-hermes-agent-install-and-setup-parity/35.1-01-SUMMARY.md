## What Was Built

### run_skills_section

`pub async fn run_skills_section(rl: &mut rustyline::DefaultEditor, hermes_home: &Path) -> Result<()>`

- Checks `hermes_home/skills/` — returns `Ok(())` with "No skills installed" if absent
- Loads `SkillRegistry::load_with_paths(&[skills_dir])` (single-path, RESEARCH Pitfall 5)
- For each skill's `hermes_metadata`:
  - `required_environment_variables` → `env_var_exists_in_dotenv_setup` check → `prompt_for_prereq_value` (secret masking via `is_secret_prereq_name`) → `write_env_var_to_dotenv`
  - `config` fields → `config_setter::config_get` check → `prompt_for_prereq_value` → `config_setter::config_set`
- Both paths reuse existing `Prerequisite` + `prompt_for_prereq_value` machinery (no new UX code)

### apply_skills_prereq_answers (testability seam)

`pub fn apply_skills_prereq_answers(hermes_home: &Path, answers: &[(&str, &str, &str, &str)]) -> Result<()>`

Tuple shape: `(skill_name, kind, prereq_name, value)`. Dispatches:
- `"env_var"` → `write_env_var_to_dotenv` (0600 mode, atomic write)
- `"config_field"` → `config_setter::config_set`
- Unknown kind → `anyhow::bail!`

### run_terminal_section

`pub async fn run_terminal_section(config: &mut Config, hermes_home: &Path, rl: &mut rustyline::DefaultEditor) -> Result<()>`

- Prompts once for `cwd` with current value (or ".") as default via `prompt_with_default`
- Assigns `config.terminal.cwd` only — `config.terminal.backend` not touched (D-02)
- Saves config exactly once via `config.save_to(&hermes_home.join("config.yaml"))`

### apply_terminal_answer (testability seam)

`pub fn apply_terminal_answer(hermes_home: &Path, cwd: &str) -> Result<()>`

Loads config from disk (or default), sets `terminal.cwd`, saves. Test seam bypasses rustyline.

### Dispatch wiring

Both arms wired in `run_setup` match:
```rust
Some("skills") => run_skills_section(&mut rl, &hermes_home).await?,
Some("terminal") => run_terminal_section(&mut config, &hermes_home, &mut rl).await?,
```

Unknown-section error message updated to include `skills` and `terminal` in the valid list.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] env_var_exists_in_dotenv: config_cli is binary-only, not accessible from lib**
- **Found during:** Task 1 GREEN build
- **Issue:** Plan suggested making `config_cli::env_var_exists_in_dotenv` pub(crate) for reuse in setup.rs. However, `config_cli` is declared in `main.rs` (`mod config_cli;`) — it is part of the binary crate, not the lib crate. setup.rs compiles as part of the lib crate (`pub mod setup;` in lib.rs). Accessing `crate::config_cli` from lib context fails: "could not find `config_cli` in the crate root".
- **Fix:** Inlined an identical 6-line helper as `fn env_var_exists_in_dotenv_setup` in setup.rs. Logic is byte-for-byte identical to config_cli.rs. config_cli.rs visibility left unchanged (still private `fn`).
- **Files modified:** crates/ironhermes-cli/src/setup.rs (added helper)
- **Commit:** ba523962

**2. [Rule 1 - Bug] setup_skills_section_errors_with_deferred_message test was asserting old bail! behavior**
- **Found during:** Full test suite run after Task 2
- **Issue:** The test at line 182 was checking that `ironhermes setup skills` fails with "Phase 28" in stderr. After implementing D-01, the skills section succeeds — the test was testing the old deferred stub behavior.
- **Fix:** Renamed test to `setup_skills_section_succeeds_with_no_skills_installed` and updated assertion to check `success()` + stdout contains "nothing to configure" / "No skills installed" / "prerequisites satisfied".
- **Files modified:** crates/ironhermes-cli/tests/setup_wizard.rs
- **Commit:** e35a4d3e

**3. [Rule 2 - Scope] apply_terminal_answer was implemented in Task 1 GREEN commit**
- **Found during:** Task 2 RED phase
- **Issue:** The `run_terminal_section` + `apply_terminal_answer` functions were added alongside `run_skills_section` in the Task 1 GREEN commit (same batch edit). When Task 2's RED test was added, it compiled and passed immediately — the traditional RED→COMMIT→GREEN sequence couldn't be observed. The function was part of the same `setup.rs` edit as the skills code.
- **Fix:** Documented as deviation. The TDD contract is satisfied: the Wave 0 stub was the RED commit (55c4d769 from Plan 00), and the GREEN is the Task 2 commit. Both D-01 and D-02 tests pass.
