# Phase 35.1: hermes-agent install and setup parity - Context

**Gathered:** 2026-05-22
**Status:** Ready for planning

<domain>
## Phase Boundary

**This phase makes `ironhermes setup` result in a fully working agent — matching the hermes-agent end-to-end setup experience.** The goal is that a new user who runs `ironhermes setup` (or who installs the binary fresh) ends up with a running, usable agent, the same way `hermes setup` does in the Python reference.

**In scope:**
- Skills wizard section in `setup.rs` (currently `bail!("deferred to Phase 28")`) — Phase 28 has shipped; unblock it
- Terminal section in `setup.rs` — add minimal cwd config prompt
- Wizard exit: run `ironhermes doctor` automatically after wizard completes
- `ironhermes config set/get/show` subcommands (CFG-02)
- `ironhermes config migrate` subcommand (CFG-03): scan installed skills for unconfigured prereqs
- First-run auto-detection: binary auto-launches setup when no runnable LLM config found
- Wizard fast-path vs full-path choice at wizard start

**Out of scope:**
- Full profile lifecycle CLI (list/create/use/delete) — CFG-04 already complete, full PROF-01..N is a future phase
- Skin/theme engine — separate phase
- Tab autocomplete (SlashCommandCompleter) — separate phase
- ACP adapter, execute_code/hooks in CLI — cli-feature-parity.md, different phase
- install.sh post-install prompt — first-run detection in the binary is sufficient

</domain>

<decisions>
## Implementation Decisions

### Wizard coverage gaps

- **D-01 (Skills wizard section):** Implement the skills wizard section in `setup.rs` — scan all installed skills that declare config requirements (`metadata.hermes.config`), prompt user for each unconfigured prereq (API keys / env vars). Mirrors the existing `run_tools_section` prereq-scan pattern exactly. Write answers to `~/.ironhermes/.env` via `write_env_var_to_dotenv`.

- **D-02 (Terminal section):** Add a minimal terminal config section to the wizard — prompt for the working directory (`cwd`) default only. Maps to `TerminalConfig.cwd` in `config.rs`.

- **D-03 (Wizard exit):** After all wizard sections complete, run `ironhermes doctor` automatically as a full preflight check. Doctor output serves as the final verification before declaring setup complete.

### Config commands (CFG-02, CFG-03)

- **D-04 (Config set/get — dot-notation):** `ironhermes config set <key> <value>` and `ironhermes config get <key>` use dot-notation addressing the existing `cli-config.yaml.example` key structure (e.g., `model.default`, `agent.max_iterations`, `gateway.telegram.enabled`). The config structure is already fully defined — these commands read/write to `~/.ironhermes/config.yaml` using that structure.

- **D-05 (Config show):** `ironhermes config show` dumps the merged config (file + env var overrides applied) as formatted YAML to stdout. No pretty table — YAML for scriptability.

- **D-06 (Config migrate — CFG-03):** `ironhermes config migrate` scans installed skills for unconfigured prereqs and prompts the user to fill them in. Matches hermes-agent's reference behavior. Reuses the prereq discovery and `write_env_var_to_dotenv` machinery already in `setup.rs`.

### First-run auto-detection

- **D-07 (Auto-launch wizard):** When the binary starts (chat or gateway mode), if no runnable LLM config is detected, auto-launch `ironhermes setup` before proceeding. After wizard completes, continue to the original command.

- **D-08 (Detection signal):** Runnable = has a non-empty API key in `~/.ironhermes/.env` (any of: `OPENROUTER_API_KEY`, `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`) **OR** has a local endpoint configured (`model.base_url` set to localhost/127.0.0.1 in `config.yaml`). **Both** must be absent to trigger setup. Ollama users who configure a local base_url are NOT prompted.

- **D-09 (install.sh stays as-is):** The install script does not explicitly prompt to run setup. First-run detection in the binary is sufficient — user gets prompted automatically the first time they run `ironhermes`.

### Setup end state and UX

- **D-10 (Minimum completion bar):** Working LLM provider (API key or Ollama endpoint) is the minimum to declare setup "done". Gateway, tools, skills, memory sections are optional and can be revisited via `ironhermes setup <section>` at any time.

- **D-11 (Fast-path vs full):** Wizard starts by asking: "Quick setup (provider + model only) or full setup (all sections)?" Fast-path: only `run_minimum_viable_flow`. Full path: all sections in sequence. The `run_minimum_viable_flow` function already exists in `setup.rs`.

- **D-12 (Completion summary):** After wizard + doctor run, print a summary: configured provider and model, which platforms are enabled, then: "Run `ironhermes` to start chatting" or "Run `ironhermes gateway` to start the Telegram bot." Clear next-step instruction.

### Folded Todos

- **"hermes-agent feature parity gaps"** (2026-04-18, area: parity) — comprehensive gap list; items 9 (setup wizard), 10 (tools config CLI), 11 (skills config CLI), 13 (config migration) all folded into this phase's scope. Items 1-8, 12, 14-20 remain for future phases.
- **"Configuration and setup wizard improvements"** (2026-04-17, area: cli) — CFG-01..04; CFG-01 (setup wizard) already largely implemented; this phase completes it. CFG-02/03 (config set/get/show/migrate) are the new implementations. CFG-04 already shipped.
- **"Add setup wizard and config scaffolding for gateway testing"** (2026-04-02, area: tooling) — install experience gaps; first-run auto-detection (D-07/D-08) addresses the core ask.

</decisions>
