## Summary

Phase 35.1 is a **brownfield completion phase** вЂ” most of the underlying infrastructure already exists and is wired. The planner must avoid rebuilding what is already shipped.

**The critical discovery:** `Commands::Config` is already implemented in `main.rs` and fully wired to `config_cli::handle_config_command`. `config_cli.rs` contains complete implementations of `Set`, `Get`, `Show`, `Migrate`, `Path`, and `EnvPath`. The `config migrate` command already scans skill frontmatter for `hermes_metadata.config` fields and `required_environment_variables`. The actual gaps remaining for this phase are:

2. No `run_terminal_section` вЂ” needs adding

**Primary recommendation:** Plan in 3 focused waves: (1) unblock skills section + add terminal section, (2) add fast-path choice + doctor call + completion summary, (3) add D-07/D-08 first-run LLM detection to preflight.

### Pattern 1: Skills Section (mirrors run_tools_section exactly)

`run_tools_section` is the canonical prereq-scanning pattern. `run_skills_section` follows the same structure but sources prereqs from `SkillRegistry` instead of `ToolRegistry`:

```rust
// Source: crates/ironhermes-cli/src/setup.rs (existing run_tools_section pattern)
pub async fn run_skills_section(
    rl: &mut rustyline::DefaultEditor,
    hermes_home: &Path,
) -> Result<()> {
    use ironhermes_core::SkillRegistry;

    let skills_dir = hermes_home.join("skills");
    if !skills_dir.exists() {
        println!("No skills installed вЂ” nothing to configure.");
        return Ok(());
    }

    let registry = SkillRegistry::load_with_paths(&[skills_dir]);
    let mut had_any = false;

    for skill in registry.list() {
        if let Some(hm) = &skill.hermes_metadata {
            // env var gaps
            for env_entry in &hm.required_environment_variables {
                if !env_var_is_set(&hermes_home.join(".env"), &env_entry.name) {
                    had_any = true;
                    // ... prompt_for_prereq_value / apply_prereq_value pattern
                }
            }
            // config field gaps
            for cfg_field in &hm.config {
                if ironhermes_core::config_setter::config_get(hermes_home, &cfg_field.key)?.is_none() {
                    had_any = true;
                    // ... prompt + config_setter::config_set
                }
            }
        }
    }

    if !had_any {
        println!("All skill prerequisites satisfied.");
    }
    Ok(())
}
```

### Pattern 4: Doctor call at wizard exit (D-03)

`cmd_doctor()` is a synchronous `fn` in `main.rs`. At wizard exit, call it directly:

```rust
// In run_setup after all sections complete:
println!("\nRunning doctor check...\n");
// cmd_doctor() is not pub; options:
//   A) make cmd_doctor pub and call from setup.rs
//   B) extract doctor logic to a shared fn in ironhermes-cli lib
//   C) call `std::process::Command::new("ironhermes").arg("doctor")` (heavyweight, wrong for sub-process)
// Option A is cleanest вЂ” make cmd_doctor() pub in main.rs and call from run_setup.
// But setup.rs is in the binary crate (not lib), so it can access main.rs functions
// directly since they share the same crate.
```

**The integration point:** `setup.rs` and `main.rs` are in the same binary crate (`ironhermes-cli/src/`). `run_setup` can call `cmd_doctor()` directly if `cmd_doctor` is accessible within the crate. Currently `cmd_doctor` is a crate-private `fn`. The planner should either:
- Make `cmd_doctor` crate-private callable from setup.rs (they are in the same crate вЂ” this works by moving doctor logic to a shared module or by extracting to a `fn run_doctor_check()` in a dedicated `doctor.rs`), OR
- Inline the doctor check call via the `preflight` module's probe functions.

### Pattern 5: D-08 Runnable LLM Detection

The current `preflight.rs` checks only for `config.yaml` existence and `config.validate()`. It does NOT check for runnable LLM keys. The D-08 signal needs to be added:

```rust
// In preflight.rs run_preflight_check, AFTER config validates successfully:
fn has_runnable_llm(config: &Config, hermes_home: &Path) -> bool {
    // Check 1: any API key in .env
    let env_path = hermes_home.join(".env");
    if env_path.exists() {
        if let Ok(text) = std::fs::read_to_string(&env_path) {
            for key in &["OPENROUTER_API_KEY=", "ANTHROPIC_API_KEY=", "OPENAI_API_KEY="] {
                if text.lines().any(|l| l.starts_with(key) && l.len() > key.len()) {
                    return true;
                }
            }
        }
    }
    // Check 2: env vars loaded into process
    for var in &["OPENROUTER_API_KEY", "ANTHROPIC_API_KEY", "OPENAI_API_KEY"] {
        if std::env::var(var).map(|v| !v.is_empty()).unwrap_or(false) {
            return true;
        }
    }
    // Check 3: local endpoint in config (Ollama users)
    if let Some(ref base_url) = config.model.base_url {
        let lower = base_url.to_lowercase();
        if lower.contains("localhost") || lower.contains("127.0.0.1") {
            return true;
        }
    }
    false
}
```

## Common Pitfalls

### Pitfall 1: "config_cli.rs is already complete"
**What goes wrong:** Planner allocates tasks to re-implement config set/get/show/migrate.
**Why it happens:** CONTEXT.md lists these as deliverables (D-04/D-05/D-06) but they are already shipped.
**How to avoid:** The planner MUST read `config_cli.rs` first. CFG-02 and CFG-03 are satisfied. The only work in config_cli is: nothing вЂ” it is done.
**Warning signs:** Any task that says "implement ConfigSubcommand::Set" or "wire config migrate" is a duplicate.

### Pitfall 2: "skills section line 122 bail!"
**What goes wrong:** Developer sees `Some("skills") => anyhow::bail!("section deferred to Phase 28")` and adds a new section subcommand instead of replacing the bail.
**Why it happens:** Misreading the dispatch pattern.
**How to avoid:** Replace the bail line with `Some("skills") => run_skills_section(rl, hermes_home).await?`.
**Warning signs:** Any plan that creates a new enum variant instead of replacing the bail.

### Pitfall 3: Preflight gate condition widening
**What goes wrong:** D-07 auto-launch is implemented by widening the `run_preflight` match to include more command variants.
**Why it happens:** D-07 says "auto-launch when no runnable LLM". Developer adds more arms.
**How to avoid:** The gate condition (`Chat | Gateway | None`) is PHASE-23-LOCKED per the comment at main.rs line 323. D-08 detection adds a new check inside the existing valid-config branch of `run_preflight_check()`, not by widening the outer gate.
**Warning signs:** Any diff that touches the `matches!(cli.command, ...)` line in main.rs.

### Pitfall 4: cmd_doctor is not pub
**What goes wrong:** setup.rs calls `cmd_doctor()` but it doesn't compile because the function is crate-private in main.rs.
**Why it happens:** Rust binary crate вЂ” all functions in main.rs are the binary, but setup.rs is a module within the same binary crate, so they can share code. However, `cmd_doctor` is defined in main.rs directly, not in a module with `pub`. Within the same crate, `pub(crate)` or just moving it to a separate `mod doctor` makes it accessible from setup.rs.
**How to avoid:** Extract doctor check logic to `crates/ironhermes-cli/src/doctor.rs` as a `pub fn` module, matching the `cron.rs`, `config_cli.rs` pattern. Then `mod doctor` in main.rs and call `doctor::cmd_doctor()` from setup.rs.
**Warning signs:** Compile error "function `cmd_doctor` is private".

### Pitfall 5: SkillRegistry load path
**What goes wrong:** `run_skills_section` scans only `hermes_home/skills/` but misses `optional-skills/`.
**Why it happens:** The skills registry has multiple scan paths.
**How to avoid:** Per SKILL-02 and the existing `cmd_config_migrate` implementation in config_cli.rs (line 176: `SkillRegistry::load_with_paths(&[skills_dir.clone()])`), the wizard section should scan `hermes_home/skills/` only (user-installed). The optional-skills directory is for Phase 25.7. Follow the same single-path pattern as `cmd_config_migrate`.
**Warning signs:** Any plan that passes multiple paths when they don't exist yet.

### Pitfall 6: D-08 check must run AFTER dotenvy loads
**What goes wrong:** First-run check fires before `.env` is loaded, sees no API keys even though they are configured.
**Why it happens:** main.rs loads `.env` at line 273-276. preflight runs at line 327. D-08 check is inside preflight вЂ” the ordering is already correct.
**How to avoid:** Verify the D-08 env-var check reads `std::env::var()` (which reflects post-dotenvy state) NOT raw file scan. The raw file scan is a belt-and-suspenders backup only.
**Warning signs:** D-08 always triggers setup even when API keys are configured.

### Pitfall 7: Skills env vars vs config fields are different write paths
**What goes wrong:** `run_skills_section` writes all prereqs to `.env` regardless of type.
**Why it happens:** `required_environment_variables` в†’ `.env`. `config` fields в†’ `config.yaml` via config_setter. These are different.
**How to avoid:** Check `prereq.kind` ("env_var" vs "config_field") exactly as `apply_prereq_value` does in setup.rs line 492. For skills, check `hm.required_environment_variables` (в†’ .env) vs `hm.config` (в†’ config_setter::config_set). Do not conflate.
**Warning signs:** config.yaml keys written to .env or vice versa.

fn env_var_exists_in_dotenv(env_path: &Path, var: &str) -> Result<bool> {
    if !env_path.exists() {
        return Ok(false);
    }
    let text = std::fs::read_to_string(env_path)?;
    Ok(text.lines().any(|l| {
        let trimmed = l.trim_start();
        trimmed.starts_with(&format!("{}=", var))
    }))
}
