## What Was Built

### run_setup None arm — D-11 fast/full choice

Replaced the single-line `run_minimum_viable_flow(...)` dispatch with a block that:

1. Prompts `"Full setup (all sections)? [y = full, N = quick provider+model only]"` via existing `prompt_yes_no` (default=false, quick path)
2. Always runs `run_minimum_viable_flow` unconditionally (provider+model mandatory for both paths)
3. If `full_setup` is true: runs `run_skills_section` → `run_terminal_section` → `run_tools_section` → `run_memory_section` → `run_gateway_section` → `run_agent_section` in order
4. After the if block (both paths): calls `run_doctor_check()` then `print_setup_completion_summary()`

Per-section dispatch arms (`Some("model") => ...` etc.) are unchanged — D-11/D-03/D-12 fire only for bare `ironhermes setup` (section=None).

### run_doctor_check — D-03 inline implementation

`fn run_doctor_check() -> Result<()>` added to `setup.rs` — a lib-safe equivalent of `doctor::run_doctor_check()`. Checks: home directory, config file, .env file, OpenRouter API key, Anthropic API key, state database, gateway PID liveness. Omits the profile line (doctor.rs binary-only feature). Returns `Ok(())` — issues shown as MISSING, not errors.

### print_setup_completion_summary — D-12 helper

`fn print_setup_completion_summary(config: &Config, _hermes_home: &Path)` — prints:
- Bold-green "Setup complete!" header
- `"  Provider: {}"` from `config.model.provider`
- `"  Model:    {}"` from `config.model.default`
- `"  Telegram: enabled"` if `config.gateway.platforms.get("telegram").map(|p| p.enabled)` is true
- Next-step hint: `"  Run \`ironhermes gateway\` to start the Telegram bot."` if telegram enabled, else `"  Run \`ironhermes\` to start chatting."`

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] doctor::run_doctor_check() not callable from lib context**
- **Found during:** Task 1 build
- **Issue:** `setup.rs` compiles as part of the lib crate (`pub mod setup` in lib.rs). `doctor.rs` is declared via `mod doctor;` in `main.rs` (binary crate only), because it calls `ironhermes_cli::status_cmd::current_profile()` — a self-reference that fails from lib context. Attempts to add `pub mod doctor` to lib.rs and use `crate::status_cmd` failed in binary context (status_cmd not in main.rs), and `ironhermes_cli::doctor` failed (lib doesn't depend on itself).
- **Fix:** Added an inline `fn run_doctor_check() -> Result<()>` directly in `setup.rs` that performs the same health checks without the profile line. The binary-crate `doctor::run_doctor_check()` (dispatched from `main.rs`) is unchanged. The wizard exit call goes to the local inline version.
- **Files modified:** crates/ironhermes-cli/src/setup.rs
- **Commit:** d182090d

**2. [Rule 2 - Scope] D-12 telegram detection uses typed access, not config_setter fallback**
- **Found during:** Task 2 implementation
- **Issue:** RESEARCH Open Question 2 recommended `config_setter::config_get(hermes_home, "gateway.platforms.telegram.enabled")` as a fallback if the platforms type was opaque. Direct inspection showed `GatewayConfig.platforms` is `HashMap<String, PlatformGatewayConfig>` and `PlatformGatewayConfig.enabled` is a plain `bool`.
- **Fix:** Used direct typed access `config.gateway.platforms.get("telegram").map(|p| p.enabled).unwrap_or(false)` — simpler, no string parsing, no fallback needed.
- **Files modified:** crates/ironhermes-cli/src/setup.rs
- **Commit:** d182090d
