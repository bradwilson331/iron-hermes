# Phase 26.2: Code Review Report

**Reviewed:** 2026-05-04
**Depth:** standard
**Files Reviewed:** 5
**Status:** issues_found

## Summary

This phase adds interactive session tabs to the TitleBar, wires up three tab event handlers in `WarpHermes`, and introduces a WebSocket keepalive + structured teardown to the server-side `ws_chat` handler. The structural shape is solid: the test suite is well-anchored to real code rather than mocks, the WR-01/WR-02 fixes for streaming state and the busy-gate are correct, and the close-frame helper is cleanly factored.

Two blockers exist. First, the `on_tab_close` handler removes a tab then reads back the updated `tabs` signal in a separate `.read()` call, creating a race where the intermediate state can be observed by another concurrent scheduler tick, and in the specific case of the active-tab index arithmetic the write and read are not performed atomically — a concurrent `on_tab_new` spawn can interleave. Second, when serialization of `ChatRequest` silently produces an empty string (via `unwrap_or_default()`), the WebSocket `send_raw` call still fires — the server receives an empty text frame, cannot parse it as `ChatRequest`, and emits an error event back to the client that surfaces as a user-visible error block for what is actually a client-side programming fault.

Five warnings cover: the RSX tab key being the iterator index (breaks keying on tab reorder or mid-list close), the `on_tab_new` handler not guarding against concurrent streaming (D-02 gap), a TOCTOU window between `tabs.write().remove(idx)` and the immediately following `tabs.read().is_empty()` check, the test assertion for `send_close_frame` invocations counting the function definition as a call site (making the `>= 5` guard pass with only 4 real call sites), and `streaming_block_id` not being cleared when `on_tab_click` switches sessions (leaving stale delta accumulation possible on reconnect).

## Critical Issues

### CR-01: Silent ChatRequest serialization failure sends empty WebSocket frame

**File:** `crates/iron_hermes_ui/src/components/warp_hermes.rs:517` and `:676`
**Issue:** Both the `on_rerun` handler (line 517) and the `submit` handler (line 676) call `serde_json::to_string(&req).unwrap_or_default()`. If serialization fails (which it can if `ChatRequest` contains non-serializable fields introduced later, or if `serde_json` encounters an IO error in a future version), `unwrap_or_default()` silently produces an empty `String`. The subsequent `ws.send_raw(dioxus_fullstack::Message::Text(json))` sends that empty frame to the server. The server's `serde_json::from_str("")` fails and emits `ChatStreamEvent::Error { message: "Invalid request: ..." }` — which the client renders as a user-visible error block. The user submitted a message; instead they get a cryptic protocol error with no indication their message was lost. The message is also silently dropped (never retried, never reflected in the block stream as failed).

**Fix:** Fail loudly on the client side before sending. Surface a disconnect-style error block instead of swallowing the fault silently:

```rust
let json = match serde_json::to_string(&req) {
    Ok(j) => j,
    Err(e) => {
        // Serialization is a programmer error; surface it to the block stream
        // rather than sending an empty frame and getting a confusing protocol error back.
        #[cfg(target_arch = "wasm32")]
        web_sys::console::error_1(&format!("ChatRequest serialize failed: {e}").into());
        #[cfg(not(target_arch = "wasm32"))]
        eprintln!("ChatRequest serialize failed: {e}");
        return; // or push an Err block
    }
};
```

### CR-02: `on_tab_close` removes tab then reads `is_empty` in a separate borrow — stale active_tab index possible

**File:** `crates/iron_hermes_ui/src/components/warp_hermes.rs:562-593`
**Issue:** The close handler does:

```rust
tabs.write().remove(idx);          // line 562 — write lock dropped immediately
let now_empty = tabs.read().is_empty();  // line 563 — separate read lock
```

Between those two statements any other Dioxus scheduler work that reads `tabs` sees the post-remove state. More concretely, the `active_tab` index can become stale: if the tab being closed is index 2 in a 3-tab list (indices 0,1,2) and `active_tab` is 2, `len` is now 2 after remove, and the arithmetic `cur.saturating_sub(1).min(len - 1)` = `1.min(1)` = `1` is correct — but the `session_id` lookup at line 588 uses `new_idx` against the already-modified `tabs` signal in a third separate borrow. There are now three separate read/write windows. If a concurrent `on_tab_new` spawn resolves between line 562 and line 586, `len` and the index can be wrong.

Additionally, `on_tab_close` does not guard against streaming (`scanner_active()` check), so it is callable while streaming is active. Closing the active tab mid-stream leaves `streaming_block_id` pointing at a block in the now-cleared `blocks` vec — the next delta will find no matching entry and silently open a new block with id derived from `next_id`, which will never receive a `Finished` event because the server's turn is still running against the old session.

**Fix:** Wrap the entire close-tab mutation in a single write-scope and add the streaming guard:

```rust
let mut on_tab_close = move |idx: usize| {
    if scanner_active() { return; }   // D-02 guard — same as on_tab_click
    streaming_block_id.set(None);     // clear stale streaming anchor

    let (now_empty, new_idx, new_sid) = {
        let mut ts = tabs.write();
        ts.remove(idx);
        if ts.is_empty() {
            (true, 0, None)
        } else {
            let cur = active_tab();
            let len = ts.len();
            let new_idx = if idx < cur {
                cur - 1
            } else if idx == cur {
                cur.saturating_sub(1).min(len - 1)
            } else {
                cur
            };
            let sid = ts.get(new_idx).map(|t| t.session_id.clone());
            (false, new_idx, sid)
        }
    }; // write lock released once here

    if now_empty {
        spawn(async move { /* create_session ... */ });
    } else {
        active_tab.set(new_idx);
        if let Some(sid) = new_sid { session_id.set(sid); }
        blocks.set(Vec::new());
        messages.write().clear();
    }
};
```

## Warnings

### WR-01: Tab RSX key is iterator index, not session_id — breaks keying on close

**File:** `crates/iron_hermes_ui/src/components/shell/title_bar.rs:57`
**Issue:** `key: "{i}"` uses the iteration index as the RSX reconciliation key. When a tab is removed from the middle of the list, Dioxus sees all subsequent keys shift by one — it will re-render and potentially re-mount every tab after the removed one rather than the minimal diff. More critically, if tab order is ever made reorderable, index keys will cause Dioxus to confuse element identity entirely. The session_id is already available in `t.session_id` and is stable.

**Fix:**
```rust
key: "{t.session_id}",
```

### WR-02: `on_tab_new` does not guard against concurrent streaming (D-02 gap)

**File:** `crates/iron_hermes_ui/src/components/warp_hermes.rs:539-558`
**Issue:** `on_tab_click` (line 528) correctly checks `if scanner_active() { return; }` before switching sessions. `on_tab_new` has no such guard. During active streaming a user can click `+`, which spawns `create_session().await`, appends a new tab, calls `active_tab.set(new_idx)`, `session_id.set(sid)`, and clears `blocks`/`messages`. The in-flight WebSocket turn's remaining `Delta` and `Finished` events will still arrive and be applied against the new empty block list (or open a new block in the new session's context), corrupting the AI response.

**Fix:** Add the same streaming guard at the top of `on_tab_new`:
```rust
let on_tab_new = move |_: ()| {
    if scanner_active() { return; }
    spawn(async move { /* ... */ });
};
```

### WR-03: `streaming_block_id` not cleared on `on_tab_click` session switch

**File:** `crates/iron_hermes_ui/src/components/warp_hermes.rs:527-535`
**Issue:** `on_tab_click` correctly clears `blocks` and `messages`, and the streaming guard means it can only fire when `scanner_active` is false. However `streaming_block_id` is not reset to `None`. If streaming completed but `streaming_block_id` was not yet cleared (a narrow race between `Finished` processing and the tab click), the stale `streaming_block_id` will cause the next submitted message's first delta to try to `find` a block with the old id in the (now empty) `blocks` vec, silently fail, and leave `streaming_block_id` set to a nonexistent id. All subsequent deltas then accumulate into the same nonexistent block until a new first-delta path fires.

**Fix:**
```rust
let mut on_tab_click = move |idx: usize| {
    if scanner_active() { return; }
    let sid = { let ts = tabs.read(); ts.get(idx).map(|t| t.session_id.clone()) };
    if let Some(sid) = sid {
        active_tab.set(idx);
        session_id.set(sid);
        blocks.set(Vec::new());
        messages.write().clear();
        streaming_block_id.set(None);  // clear stale streaming anchor
    }
};
```

### WR-04: Test assertion on `send_close_frame` call count includes the function definition

**File:** `crates/iron_hermes_ui/tests/websocket_lifecycle_parity.rs:195-203`
**Issue:** The test counts `ws.matches("send_close_frame(").count()` and asserts `>= 5`, commenting that "Definition + 4 call sites". This means the assertion passes with only 4 real call sites because the function definition `async fn send_close_frame(` contributes 1 to the match count. If a future refactor removes one call site (say the clean-recv branch), the count drops to 4 (3 real + 1 definition) — still `>= 5` is false, so the test *would* catch a removal. Wait — actually `4 >= 5` is false, so it would correctly fail. However the current production code has exactly 4 real call sites (lines 111, 137, 271, 315), and definition at line 39 = 5 total occurrences. The assertion passes at exactly `>= 5`. This means the test has zero headroom: removing *any one* real call site drops the count to 4, failing correctly — but adding any new occurrence of the string `send_close_frame(` anywhere (e.g., in a comment, a test log) inflates the count and silently weakens the regression. The comment itself in the test file (`// send_close_frame(...)`) does not exist right now but the pattern is fragile.

**Fix:** Anchor the count more precisely using a pattern that cannot match the definition:
```rust
// Count only call-site invocations: `send_close_frame(&mut socket,`
let call_site_count = ws.matches("send_close_frame(&mut socket,").count();
assert_eq!(
    call_site_count, 4,
    "expected exactly 4 send_close_frame call sites; found {call_site_count}"
);
```

### WR-05: `on_tab_close` new-session spawn does not reset `streaming_block_id`

**File:** `crates/iron_hermes_ui/src/components/warp_hermes.rs:564-582`
**Issue:** In the `now_empty` branch, `on_tab_close` spawns an async block that, on success, calls `blocks.set(Vec::new())` and `messages.write().clear()`, but does not reset `streaming_block_id`. The same stale-anchor issue described in WR-03 applies here. If the closed tab had been streaming (which CR-02's fix would prevent, but defensively), `streaming_block_id` would remain set to an id that no longer exists in the new empty block list.

**Fix:** After `blocks.set(Vec::new())` inside the spawn, add:
```rust
streaming_block_id.set(None);
```

### IN-01: `pulse_scanner` redundantly sets `scanner_active` to `true` then immediately lets `on_rerun` also set it

**File:** `crates/iron_hermes_ui/src/components/warp_hermes.rs:508-509`
**Issue:** `on_rerun` calls `pulse_scanner(2000, scanner_active)` which sets `scanner_active` to `true` inside `pulse_scanner`, then the very next line is `scanner_active.set(true)` again (line 509). The double-set is harmless but the intent of `pulse_scanner` as a "set true for N ms then set false" utility is partially undermined: `pulse_scanner` takes `scanner_active` by value (it is `Signal<bool>` which is `Copy`), sets it true internally, and then 2000ms later sets it false. The explicit `scanner_active.set(true)` on line 509 is therefore redundant dead code.

**Fix:** Remove line 509:
```rust
// Before:
pulse_scanner(2000, scanner_active);
scanner_active.set(true);  // redundant — pulse_scanner already sets true

// After:
pulse_scanner(2000, scanner_active);
```
