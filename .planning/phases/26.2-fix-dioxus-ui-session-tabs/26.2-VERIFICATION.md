### Human Verification Required

#### 1. Tab click visual switching

**Test:** Run `dx serve`, open the app in a browser. Click a non-active tab in the title bar.
**Expected:** The clicked tab becomes highlighted with the active CSS class; the blocks area and messages area clear; any new message goes to the switched session.
**Why human:** Dioxus RSX signal → DOM update → CSS class application cannot be verified by source-text grep alone.

#### 2. D-02 streaming disabled state

**Test:** Submit a chat message and observe the tab strip while scanner_active is true (during the streaming turn).
**Expected:** The tab strip appears at 0.5 opacity with cursor interactions suppressed (pointer-events: none); clicking tabs during streaming has no effect.
**Why human:** The disabled prop is wired in source but the live CSS application and click suppression require an active streaming turn in the browser.

#### 3. D-03/D-04 new tab creation

**Test:** Click the + button.
**Expected:** A new tab labeled "New Session" appears in the tab strip, becomes the active tab, and the chat area clears.
**Why human:** The on_tab_new spawn(async) path calls create_session() — verifying the server function completes and the Signal<Vec<Tab>> update renders correctly requires a running dx serve instance.

#### 4. D-05/D-06 close button behavior

**Test:** Click the x (close) button on a tab that is NOT the last tab.
**Expected:** The tab is removed; an adjacent tab becomes active; clicking x does NOT trigger a tab switch on the same tab (stop_propagation working). Then close until one tab remains — it should auto-create a new session.
**Why human:** DOM event bubbling suppression by evt.stop_propagation() can only be confirmed in a live browser; the last-tab auto-creation requires a running server.

#### 5. D-07 tab list sync from list_sessions

**Test:** Restart the app. Observe the tab strip after sessions() resolves.
**Expected:** Pre-existing sessions from StateStore are seeded as tabs via the sessions-seeding use_effect; no duplicate tabs for a session that was also created by the mount effect.
**Why human:** The already_present deduplication guard and the async resolution ordering of use_server_future vs mount use_effect require a running app with pre-populated StateStore to observe.
