# Phase 26.2: fix-dioxus-ui-session-tabs - Context

**Gathered:** 2026-05-04
**Status:** Ready for planning

<domain>
## Phase Boundary

Make the `TitleBar` session tab strip fully interactive: clicking a tab switches the active session, "+" creates a new session and switches to it, "×" removes a tab, and the tabs list stays in sync with session lifecycle events (mount creation, new tab, close).

Out of scope:
- Loading session message history when switching tabs (no `list_session_messages` server function needed this phase)
- Deleting sessions from StateStore (tabs are UI-removed only, sessions persist)
- Any WebSocket/streaming protocol changes beyond what's needed for tab gating

Also in scope (code quality from Phase 26.1 review, same file surface):
- WR-01: Fix `Finished` event appending stale AI output from older turn
- WR-02: Fix busy-gate false rejection after turn completion
- WR-03: Anchor parity assertion strings to branch context
- IN-01: Remove duplicate `#[allow(unused_mut)]` on `mode` binding

</domain>

<decisions>
## Implementation Decisions

### Tab Switching Behavior
- **D-01:** Clicking a tab changes `session_id` for future messages and clears `blocks` + `messages` signals (fresh start). No session history loading. Keep it minimal.
- **D-02:** Tab clicks are disabled (no-op) when `scanner_active` is true (streaming turn in progress). Grey-out or ignore — no mid-turn teardown complexity needed.

### New Tab ("+" button)
- **D-03:** "+" calls `create_session()`, appends a new `TabEntry` to the local tabs signal, sets `active_tab` to the new tab's index, and clears `blocks` + `messages`. Immediate switch — same flow as D-01 tab click.
- **D-04:** New tabs get a `"New Session"` placeholder label. No prompt, no UUID display.

### Close Tab ("×" button)
- **D-05:** "×" removes the tab from the local tabs signal only. The underlying session stays in StateStore. No `delete_session` server function needed this phase.
- **D-06:** When closing the active tab: auto-switch to the previous tab (index - 1), or next tab if it's the first one. If closing the last remaining tab, auto-create a new session (same as D-03) so there's always at least one tab.

### Session List Sync
- **D-07:** Tabs live in a `Signal<Vec<TabEntry>>` (not re-derived from `use_server_future` on each change). `use_server_future(list_sessions)` seeds the initial list on mount. Create/close operations mutate the signal directly — no server re-fetch needed for tab state.
- **D-08:** On mount, after `create_session()` succeeds, push a `TabEntry` for the newly created session into the tabs signal and set it as the active tab. This ensures the live session is always visible even if `list_sessions()` ran before the creation completed.

### TitleBar Callbacks
- **D-09:** `TitleBar` gains three `EventHandler` callbacks: `on_tab_click: EventHandler<usize>`, `on_tab_close: EventHandler<usize>`, `on_tab_new: EventHandler<()>`. All three are wired to the corresponding closures in `WarpHermes`. The `active_tab` signal becomes `mut`.

### Phase 26.1 Code Review Fixes (WR-01, WR-02, WR-03, IN-01)
- **D-10:** WR-01 fix — On `ChatStreamEvent::Finished`, use the active `streaming_block_id` to find the completed block by id instead of scanning all blocks in reverse. Clear `streaming_block_id` before appending to messages.
- **D-11:** WR-02 fix — Before the busy-gate check in `ws_chat`, opportunistically clear `in_flight_turn` if `turn.handle.is_finished()`. This prevents false "another request in progress" errors when a frame arrives just after the turn channel closes.
- **D-12:** WR-03 fix — Replace global `ws.contains("continue;")` / `ws.contains("abort()")` parity assertions with branch-anchored checks (e.g., assert the malformed-request block contains both the error-log string and a nearby `continue;`).
- **D-13:** IN-01 fix — Remove the duplicate `#[allow(unused_mut)]` on the `mode` binding in `WarpHermes`.

### Claude's Discretion
- `TabEntry` struct shape — whether to extend the existing `state::Tab` with a `session_id: String` field, or introduce a new `TabEntry` wrapper. Planner decides; `Tab` should carry the session key so tab click can look up the right `session_id`.
- Whether "×" close button uses `onclick` with `stopPropagation` or a separate `button` element to avoid triggering `on_tab_click` — planner picks the RSX approach that avoids event bubbling.
- Exact CSS for greyed-out tabs during streaming — planner can use `opacity` or `pointer-events: none` inline style on the tab div.

</decisions>

<specifics>
## Specific Ideas

- The `Tab` struct in `state.rs` currently has `label: String` and `live: bool` but no `session_id` field. The planner should add `session_id: String` to `Tab` (or create a `TabEntry` wrapper) so `on_tab_click` can look up which session to switch to.
- The "×" close button in `title_bar.rs` is currently a bare `span` with no onclick. To avoid triggering the parent tab's onclick, it should become a `button` with `onclick: move |evt| { evt.stop_propagation(); on_tab_close.call(i); }`.
- Per D-02, the simplest greying approach: pass `disabled: bool` to `TitleBar` (or derive it inline from `scanner_active` at the call site) and apply `pointer-events: none; opacity: 0.5` to the tabs container when active.

</specifics>

- Tab renaming (double-click to edit label) — new capability, separate phase.
- Tab reordering (drag-and-drop) — new capability, separate phase.

</deferred>
