# Phase 26.2 Plan 01: WebSocket Quality Fixes (WR-01/WR-02/WR-03/IN-01) Summary

Four surgical code-quality fixes from the Phase 26.1 review applied to stabilize the WebSocket chat path before the Phase 26.2 tab-strip changes: id-anchored Finished arm (WR-01), opportunistic busy-gate clear via JoinHandle::is_finished() (WR-02), branch-anchored parity test assertions (WR-03), and duplicate allow(unused_mut) removed (IN-01).

## What Was Built

### WR-01 (warp_hermes.rs Finished arm)

Replaced a reverse-scan `iter().rev().find_map()` that could pick up stale AI blocks from prior turns with an id-anchored lookup. The active `streaming_block_id` is now captured into `completed_id` *before* `streaming_block_id.set(None)` is called. The blocks vec is searched by exact id match (`entry.id != id` guard). If `streaming_block_id` was `None` when `Finished` arrives, no message push happens (silent — same as prior no-Ai-block behavior).

### IN-01 (warp_hermes.rs signal declarations)

Removed the duplicate `#[allow(unused_mut)]` attribute preceding `let mut mode = use_signal(|| Mode::Shell)`. Only one attribute remains.

### WR-02 (ws.rs busy-gate)

Inserted an opportunistic clear block immediately before the `if in_flight_turn.is_some()` busy-gate. Uses `in_flight_turn.as_ref()` + `turn.handle.is_finished()` to detect a completed turn and sets `in_flight_turn = None` before the guard runs. Prevents false "Another request is already in progress" errors when a frame arrives after the prior tokio task completes but before its full tear-down.

### WR-03 (websocket_lifecycle_parity.rs assertions)

Replaced two over-broad global substring assertions with branch-anchored proximity checks:
- `continue;` is now checked within a 500-char window starting at the `Invalid request:` marker (not globally)
- `abort()` replaced with verbatim `turn.handle.abort();` + `aborting in-flight turn` log message assertions

Added new test `busy_gate_opportunistically_clears_finished_turn` that asserts the byte offset of `turn.handle.is_finished()` precedes the byte offset of `if in_flight_turn.is_some() {` in ws.rs, proving the WR-02 structural placement is correct.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] WR-03 proximity window widened from 200 to 500 chars**
- **Found during:** Task 3 verification
- **Issue:** Plan specified 200-char window, but the actual distance from `Invalid request:` to `continue;` in ws.rs is 454 chars (the `send_raw` call in between is longer than anticipated)
- **Fix:** Window widened to 500 chars with a comment explaining the reason
- **Files modified:** `crates/iron_hermes_ui/tests/websocket_lifecycle_parity.rs`
- **Commit:** 44daf55

**2. [Rule 1 - Bug] busy_gate test used specific "if in_flight_turn.is_some() {" anchor**
- **Found during:** Task 3 verification
- **Issue:** Plan specified `ws.find("in_flight_turn.is_some()")` but ws.rs has multiple earlier occurrences of `in_flight_turn.is_some()` in telemetry lines (byte offset 4089) that appear before `turn.handle.is_finished()` (byte offset 7901). Using bare `is_some()` made `finished_pos < busy_pos` false.
- **Fix:** Changed to `ws.find("if in_flight_turn.is_some() {")` which anchors to the actual busy-gate conditional at byte offset 8084, after `is_finished()` at 7901
- **Files modified:** `crates/iron_hermes_ui/tests/websocket_lifecycle_parity.rs`
- **Commit:** 44daf55
