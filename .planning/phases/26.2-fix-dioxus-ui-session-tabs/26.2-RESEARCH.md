### Pattern 1: EventHandler Props in Dioxus 0.7

**What:** Child component declares `EventHandler<T>` in its prop signature. Parent passes a closure. Child calls `.call(value)` inside an RSX event handler.

**When to use:** Any time a child needs to notify the parent of a user action without owning the state.

**Example:**
```rust
// Source: verified against existing BlockStream + InputBox patterns in codebase
#[component]
pub fn TitleBar(
    tabs: Vec<Tab>,
    active_tab: usize,
    show_traffic_lights: bool,
    on_tab_click: EventHandler<usize>,
    on_tab_new: EventHandler<()>,
    on_tab_close: EventHandler<usize>,
) -> Element {
    rsx! {
        // ...
        for (i, t) in tabs.iter().enumerate() {
            div {
                onclick: move |_| on_tab_click.call(i),
                // ...
                button {
                    onclick: move |evt| {
                        evt.stop_propagation();
                        on_tab_close.call(i);
                    },
                    "×"
                }
            }
        }
        button {
            onclick: move |_| on_tab_new.call(()),
            "+"
        }
    }
}
```

### Pattern 2: Signal<Vec<T>> Direct Mutation (no server re-fetch)

**What:** For tab create/close, mutate the `Signal<Vec<Tab>>` directly rather than re-querying the server. `use_server_future` seeds the initial list; subsequent mutations are local.

**When to use:** State changes that the server doesn't need to be queried for. CONTEXT.md D-07 explicitly mandates this approach.

**Example:**
```rust
// Tab signal declaration — local, not re-derived from server future each change
let mut tabs = use_signal(Vec::<Tab>::new);
let mut active_tab = use_signal(|| 0_usize);  // mut (was immutable before)

// Seed on mount from server future result
// (separate use_server_future call; read once and copy into tabs signal)

// on_tab_click closure (wired as EventHandler<usize> to TitleBar)
let on_tab_click = move |idx: usize| {
    if scanner_active() { return; }  // D-02: no tab switch during streaming
    active_tab.set(idx);
    let sid = tabs.read()[idx].session_id.clone();
    session_id.set(sid);
    blocks.set(Vec::new());
    messages.write().clear();
};
```
[VERIFIED: `blocks.set(Vec::new())` pattern from `/clear` palette handler, warp_hermes.rs line 609]

### Pattern 3: Seeding a Signal from use_server_future

**What:** `use_server_future` is reactive and re-runs on dependency change. For tabs, we want a one-shot seed. The approach: read the future's result in the component body, copy into the `Signal<Vec<Tab>>` using a `use_effect` that fires when the future resolves.

**When to use:** When you need one-time initialization of a locally-owned signal from a server fetch.

**Constraint:** `use_server_future` returns `Option<Result<T>>` — `None` while loading. The component already uses this for `sessions` (line 338, warp_hermes.rs). The existing `tabs` local variable is derived from `sessions()` — this becomes the seed for `Signal<Vec<Tab>>`.

**Implementation note:** The simplest approach that avoids re-derives: keep the existing `let sessions = use_server_future(...)` call; after it resolves, push into `Signal<Vec<Tab>>` once. A `use_effect` watching `sessions()` achieves this. Alternatively, use the initial value in the `use_signal` closure — but since server futures are async, the signal must start empty and be populated via effect.

### Pattern 4: New Session via spawn(async)

**What:** The "+" handler calls `create_session()` inside a `spawn(async move { ... })` closure (same pattern as mount session creation in warp_hermes.rs lines 72-84).

**Example:**

### Anti-Patterns to Avoid

- **Holding `Signal::read()` or `Signal::write()` across `.await` points:** WASM + Dioxus clippy rule — will cause runtime borrow panics. In async closures (`spawn(async move {...})`), always extract values before the `.await`, never hold guards across it.
- **Using `cx`, `Scope`, `use_state`:** Dioxus 0.6 APIs, removed in 0.7. Forbidden by CLAUDE.md.
- **Re-fetching `list_sessions()` on every tab change:** D-07 explicitly forbids this — local signal mutation only after initial seed.
- **Calling `on_tab_close` from within the tab `div`'s `onclick` without stop_propagation:** Will trigger both close and click handlers. Must use `evt.stop_propagation()` on the close button.
- **Declaring `active_tab` without `mut`:** Current code has `let active_tab = use_signal(|| 0_usize)` (no `mut`). Writing to it requires `let mut active_tab`. This is the IN-01 companion change — `active_tab` needs `mut` too, not just `mode`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Async server call in UI event | Ad-hoc thread / callback chain | `spawn(async move { ... })` | Dioxus WASM-compatible async executor; already used for WS sends and mount session creation |
| Event bubbling prevention | Manual event coordinate checks | `evt.stop_propagation()` | Dioxus wraps the DOM event; `stop_propagation()` is on `Event<T>` and works across platforms |
| UUID generation | Custom UUID impl | `uuid::Uuid::new_v4()` in `create_session()` server function | Already done server-side; client receives the string ID |

**Key insight:** All complexity for UUID generation, session storage, and StateStore interaction lives server-side in `create_session()`. The client only calls the server function and appends the returned `String` session key to its local tab signal.

## Common Pitfalls

### Pitfall 1: Reading Signal Inside spawn/async After Close

**What goes wrong:** Closure captures `tabs` by copy (signals are `Copy`), reads inside `spawn` after `tabs.write().push(...)` — works fine. But: reading `tabs.read().len()` immediately after `tabs.write().push(...)` in the same sync scope with both guards alive causes a panic.

**Why it happens:** Dioxus signals behave like `RefCell` — you cannot hold a `.write()` guard and call `.read()` on the same signal in the same scope.

**How to avoid:** Drop the write guard first (use a block `{}`), then read. Example: `let new_idx = { tabs.write().push(new_tab); tabs.read().len() - 1 };` will panic. Correct: push in a block, then read separately.

**Warning signs:** Panic at runtime with "already borrowed" message.

### Pitfall 2: Closing Last Tab Without Creating New Session

**What goes wrong:** If the user closes the last remaining tab and no new session is created, `active_tab` index 0 is out of bounds on a now-empty `tabs` vec.

**Why it happens:** D-06 specifies: closing the last tab should auto-create a new session. If the close handler only removes the entry without the "is last tab?" guard, the UI breaks.

**How to avoid:** In `on_tab_close`, after removal, check `if tabs.read().is_empty()` and call the same new-session logic as `on_tab_new`.

**Warning signs:** Index out of bounds panic or blank tab strip.

### Pitfall 3: active_tab Index Out of Bounds After Close

**What goes wrong:** Closing a tab shifts indices. If `active_tab` index pointed to the last position and is now past the end, tab strip renders incorrectly.

**Why it happens:** `Vec::remove(i)` collapses indices ≥ i by 1.

**How to avoid:** After removal, clamp `active_tab` to `min(active_tab, tabs.len().saturating_sub(1))`, or use the D-06 switching logic (switch to idx-1 or idx+1).

### Pitfall 4: Seeding tabs Signal Twice (mount + server future)

**What goes wrong:** Both `use_effect` (mount session creation) and a `use_effect` watching `sessions()` try to push a tab entry, resulting in duplicate tabs.

**Why it happens:** Mount creates a new session and pushes a tab. If the server future also seeds tabs from `list_sessions()`, the newly created session appears in both.

**How to avoid:** Seed tabs from `list_sessions()` first (initial mount), then in the mount `create_session()` effect, only push the tab if the session key isn't already in the list. CONTEXT.md D-08 specifies this: push the newly created session tab explicitly after creation, regardless of whether `list_sessions()` already ran.

**Simpler approach (recommended):** Seed tabs only from the create_session mount call (one guaranteed tab). Skip seeding from `list_sessions()` if that causes ordering ambiguity — CONTEXT.md D-08 says "after `create_session()` succeeds, push a `TabEntry`". The existing `sessions` server future can be used for initial tabs only if the create_session result isn't already in the list.

### Pitfall 5: WR-03 Over-Broad Assertion Replacement

**What goes wrong:** WR-03 asks for branch-anchored assertions. The fix must not break the other 6 passing tests in `websocket_lifecycle_parity.rs`. Each test reads the full file as a string and checks for substrings.

**How to avoid:** The anchored assertions should check for a multi-line string that includes both the branch marker (e.g., `"Invalid request:"`) and the `continue;` on a nearby line, or check that the two tokens appear in close proximity in the file. Do not remove the existing assertions entirely — anchor them more tightly.
