1. **Add converter functions** in `state.rs` to bridge server API types to UI types:
   ```rust
   impl From<crate::server::api::SlashCommandInfo> for PaletteItem {
       fn from(cmd: crate::server::api::SlashCommandInfo) -> Self {
           PaletteItem {
               section: "slash".into(),
               cmd: format!("/{}", cmd.name),
               label: cmd.description,
               kbd: vec![], // Keyboard shortcuts not stored in CommandRouter
           }
       }
   }

   impl From<crate::server::api::SessionInfo> for Tab {
       fn from(session: crate::server::api::SessionInfo) -> Self {
           Tab {
               label: session.title.unwrap_or_else(|| session.id.clone()),
               live: true, // All active sessions are "live"
           }
       }
   }
   ```
   Gate these impls behind `#[cfg(feature = "server")]` since the server types don't exist on the WASM client. On the client, these types are received as deserialized JSON.

   Actually — since Dioxus fullstack handles serialization transparently, the server function return types (SlashCommandInfo, etc.) ARE available on both client and server (they derive Serialize+Deserialize). So the From impls can be unconditional. But since `crate::server` is `#[cfg(feature = "server")]`, the From impls should be on the server types which are also available on the client via generated code.

   **Better approach:** Don't use From impls. Instead, map inside the WarpHermes component when the server function returns. This avoids cross-module coupling.

2. **Update WarpHermes** (`warp_hermes.rs`):

   a. Replace `demo_palette_items()` with a `use_server_future` (or `use_resource`) call:
      ```rust
      // Fetch real slash commands from backend
      let slash_commands = use_server_future(move || {
          crate::server::api::list_slash_commands()
      });

      // Convert to PaletteItem when data arrives
      let palette_items: Vec<PaletteItem> = match slash_commands() {
          Some(Ok(cmds)) => cmds.into_iter().map(|cmd| PaletteItem {
              section: "slash".into(),
              cmd: format!("/{}", cmd.name),
              label: cmd.description,
              kbd: vec![],
          }).collect(),
          _ => demo_palette_items(), // Fallback to demo data while loading
      };
      ```

   b. Replace `demo_tabs()` with:
      ```rust
      let sessions = use_server_future(move || {
          crate::server::api::list_sessions()
      });

      let tabs: Vec<Tab> = match sessions() {
          Some(Ok(sessions)) if !sessions.is_empty() => sessions.into_iter().map(|s| Tab {
              label: s.title.unwrap_or(s.id),
              live: true,
          }).collect(),
          _ => vec![Tab { label: "New Session".into(), live: true }],
      };
      ```

   c. Replace the hardcoded status bar props with:
      ```rust
      let config_summary = use_server_future(move || {
          crate::server::api::get_config_summary()
      });

      let (model_name, provider_name, context_len) = match config_summary() {
          Some(Ok(cfg)) => (cfg.model, cfg.provider, cfg.context_length),
          _ => ("loading...".to_string(), "...".to_string(), 128_000),
      };
      ```
      Then pass these to the `StatusBar` component:
      ```rust
      StatusBar {
          mode: "Chat".to_string(),
          model: model_name,
          provider: provider_name,
          tokens: tokens, // Keep existing signal for now — Plan 04 wires real token tracking
          scanner_active: scanner_active,
          hint: "/help · ⌃C cancel · ⌘K palette".to_string(),
      }
      ```

   d. Keep the existing `demo_block_entries()` and `demo_messages()` calls — these are replaced in Plan 04 which handles the interactive/streaming components.

   e. If `use_server_future` is not available in Dioxus 0.7.1 (it may be 0.6 API), use `use_resource` with the server function call inside an async block instead:
      ```rust
      let slash_commands = use_resource(move || async move {
          list_slash_commands().await
      });
      ```
