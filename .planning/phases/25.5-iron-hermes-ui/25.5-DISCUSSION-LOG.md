## Backend Communication

| Option | Description | Selected |
|--------|-------------|----------|
| Fullstack Dioxus | Server functions (#[server]) for commands/config, WebSocket for streaming chat. One crate, works web + desktop. | ✓ |
| Separate API crate | New ironhermes-api with REST + WS. UI stays a pure client. More flexible, more plumbing. | |
| Direct imports first | Desktop-only wiring via direct Rust calls. Web deferred. Fastest to ship but limits initial target. | |

**User's choice:** Fullstack Dioxus through existing IronHermes gateway
**Notes:** Server functions for commands/config, WebSocket for streaming chat responses. Desktop mode: one process, no HTTP layer needed. Routes through existing gateway infrastructure.

## Interface Relationship

### Coexistence model

| Option | Description | Selected |
|--------|-------------|----------|
| Additive — both live permanently | Web UI is another gateway surface. TUI stays as primary developer interface. | ✓ |
| Additive now, replace later | Web UI additive in v2.1, TUI deprecated and eventually removed. | |
| Web UI replaces TUI immediately | Ratatui REPL removed. Web UI takes over. CLI stays for one-shot only. | |

**User's choice:** Additive — both live permanently
**Notes:** Web UI is like Telegram — another surface to the same agent. TUI remains the primary developer interface.

### Session sharing

| Option | Description | Selected |
|--------|-------------|----------|
| Shared sessions | Same state.db, same session history. Web can pick up where TUI left off. | |
| Separate sessions | Each interface gets its own session namespace. Simpler, no cross-contamination. | ✓ |

**User's choice:** Separate sessions
**Notes:** Each interface (TUI, web, Telegram) gets its own session namespace following the existing session key pattern.

## Crate Architecture

### Git integration

| Option | Description | Selected |
|--------|-------------|----------|
| Fold into monorepo | Remove .git from crates/iron_hermes_ui/. Single git history. | ✓ |
| Git submodule | Keep as separate repo linked via submodule. Two repos, two histories. | |
| Keep as-is | Nested .git stays. Technically separate repo inside workspace. | |

**User's choice:** Fold into monorepo
**Notes:** Remove .git, commit source directly. Simplest ongoing workflow.

### Dependency wiring

| Option | Description | Selected |
|--------|-------------|----------|
| Import core + agent directly | Server functions call AgentLoop::run() directly. Heavy deps, maximum power. | ✓ |
| Import through gateway | Reuse gateway session management. UI as another PlatformAdapter. | |
| New ironhermes-api crate | Thin adapter between UI and backend. Clean boundary, more indirection. | |

**User's choice:** Import ironhermes-core + ironhermes-agent directly
**Notes:** Full dependency tree (core, agent, state, tools) behind #[cfg(feature = "server")].

### Build and binary

| Option | Description | Selected |
|--------|-------------|----------|
| Separate binary, separate build | Own binary via dx build. Two processes. cargo build skips UI. | ✓ |
| Integrated into main binary | ironhermes serve-ui subcommand. Single process but bloats CLI. | |
| Feature-gated in main binary | Behind a ui cargo feature flag. Enabled explicitly. | |

**User's choice:** Separate binary, separate build
**Notes:** iron_hermes_ui produces its own binary. Runs alongside ironhermes. cargo build at workspace root skips it.
