# Phase 25.5: iron-hermes-ui - Context

**Gathered:** 2026-05-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Wire the existing Dioxus 0.7.1 UI crate (`crates/iron_hermes_ui`) — which has a complete terminal-style shell UI with 13 mock components — to the live IronHermes backend. Every mock data source (`demo_blocks()`, `demo_messages()`, `demo_palette_items()`, `demo_tabs()`) is replaced with real data from the IronHermes agent runtime, session store, slash command registry, and tool registry.

The phase delivers:

1. **Fullstack Dioxus wiring** — Server functions (`#[server]`) connect UI components to IronHermes backend crates. WebSocket channel delivers streaming agent responses to the block stream.
2. **Full component wiring** — All 13 shell components receive real data: block stream (live agent output), input box (real message submission), command palette (49 real slash commands), agent panel (real tool calls and messages), title bar (real session tabs), status bar (real model/provider/token metrics), scanner (real agent activity).
3. **Separate binary** — `iron_hermes_ui` produces its own binary via `dx build`. Runs alongside the main `ironhermes` binary as a two-process deployment (backend + UI server).
4. **Git integration** — UI crate's standalone `.git` removed; source committed directly into the main ironhermes monorepo.

**Out of scope:**
- New UI components or design changes — the existing shell design is final. This phase only wires mock data to real data.
- Desktop or iOS targets — web is the only target for this phase. Desktop/mobile deferred to future phase.
- Replacing or deprecating the terminal REPL — TUI stays permanently as the primary developer interface.
- Multi-user authentication — single-operator deployment, same as existing gateway.
- Admin/monitoring dashboards beyond what the existing components already show (status bar, agent panel).
- Cross-interface session sharing — web UI gets its own session namespace, separate from TUI and Telegram.

</domain>

<decisions>
## Implementation Decisions

### Backend Communication
- **D-01:** **Fullstack Dioxus through existing IronHermes gateway.** Server functions (`#[server]`) for commands, config queries, session listing, and one-shot operations. WebSocket for streaming chat responses (agent `StreamCallback` output). Desktop mode runs as one process with no HTTP layer — Dioxus fullstack handles this natively. The Dioxus server-side code calls directly into `ironhermes-agent`, `ironhermes-core`, `ironhermes-state`, and `ironhermes-tools` via Rust function calls (no HTTP intermediary on the server side).

### MVP Feature Scope
- **D-02:** **Full wiring — every mock component gets real data.** No partial migration. All 13 shell components are connected to live backend sources in this phase:
  - **Block stream** (`block_stream.rs`, `block.rs`) — real agent output replacing `demo_block_entries()`
  - **Input box** (`input_box.rs`) — real message submission to `AgentLoop`
  - **Command palette** (`command_palette.rs`) — connected to the existing 49-command `CommandRouter`
  - **Agent panel** (`agent_panel.rs`) — real tool calls and conversation messages replacing `demo_messages()`
  - **Title bar** (`title_bar.rs`) — real session tabs from `StateStore` replacing `demo_tabs()`
  - **Status bar** (`status_bar.rs`) — real model name, provider, token budget from `ProviderResolver` / `AgentResult`
  - **Scanner** (`scanner.rs`) — activated by real agent activity (tool dispatch, streaming)
  - **Tool call cards** (`tool_call.rs`) — real `ToolCall` data from agent loop
  - **Sigil** (`sigil.rs`) — real mode state (Shell vs Agent)
  - **Command line** (`command_line.rs`) — real parsed command tokens
  - **Markdown** (`markdown.rs`) — real agent markdown responses

### Interface Relationship
- **D-03:** **Additive — web UI is another gateway surface (like Telegram).** The terminal REPL (rustyline Phase 22.3 + ratatui Phase 22.4) stays permanently as the primary developer interface. The web UI is an additional access point to the same IronHermes agent. Both interfaces coexist indefinitely. The web UI uses a `Platform::Web` variant (or similar) in the existing `Platform` enum for session key construction.
- **D-04:** **Separate sessions — each interface gets its own session namespace.** Web UI sessions are keyed separately from TUI and Telegram sessions (following the existing `agent:main:{platform}:{chat_type}:{chat_id}` pattern). No cross-interface session continuity. Each interface maintains its own conversation history. This avoids cross-contamination and simplifies state management.

### Crate Architecture
- **D-05:** **Fold into monorepo.** Remove `.git` from `crates/iron_hermes_ui/`, commit the UI source directly into the main ironhermes repo. Single git history going forward. The standalone development history is preserved as an archive branch or discarded. Simplest ongoing workflow — standard `git log`, standard PRs, no submodule dance.
- **D-06:** **Import ironhermes-core + ironhermes-agent directly.** Server functions in `iron_hermes_ui` call `AgentLoop::run()` directly on the server side. UI crate's `Cargo.toml` gains dependencies on:
  - `ironhermes-core` — types (`ChatMessage`, `Config`, `Platform`, `ToolSchema`), slash command definitions
  - `ironhermes-agent` — `AgentLoop`, `PromptBuilder`, `LlmClient`, `ContextCompressor`, streaming
  - `ironhermes-state` — `StateStore` (session CRUD, message history, search)
  - `ironhermes-tools` — `ToolRegistry`, tool schemas for command palette
  These deps are behind `#[cfg(feature = "server")]` — the WASM client binary does not compile backend crates.
- **D-07:** **Separate binary, separate build.** `iron_hermes_ui` produces its own binary via `dx build` / `dx serve`. It runs alongside the main `ironhermes` binary as a two-process setup. `cargo build` at workspace root does NOT build the UI (excluded from default workspace members or feature-gated). The UI is built explicitly with `dx serve` for dev or `dx build --release` for production. This keeps the main `ironhermes` CLI binary lean — no Dioxus/WASM deps in the standard build.

### Claude's Discretion

- **Session key format for web UI** — The `Platform` enum already has 16 variants; add `Web` (or reuse `API` if it exists) for session key construction. Session keys follow `agent:main:web:{chat_id}` pattern. Planner decides the exact variant name.
- **WebSocket implementation** — Dioxus 0.7 has `use_websocket` + `dioxus_fullstack::Websocket`. Whether to use Dioxus's built-in WebSocket support or a custom `axum` WebSocket route is planner's choice. The built-in is recommended for simplicity.
- **Server function granularity** — How many `#[server]` functions vs. bundling operations. Planner decides the API surface shape. Minimum needed: create session, send message (returns stream handle), list sessions, get session messages, execute slash command, get config, get tool schemas.
- **Mock data removal strategy** — Whether `demo_*()` functions are deleted, moved behind `#[cfg(test)]`, or kept as dev fixtures. Planner decides.
- **`dx serve` port and dev workflow** — Default port, proxy configuration for local development. Planner decides.
- **Workspace member exclusion** — Whether to use `default-members` in root `Cargo.toml` (excluding `iron_hermes_ui`) or a `[features]` gate. Planner decides the mechanism.

</decisions>

- **State types** (`crates/iron_hermes_ui/src/state.rs`) — `Block`, `BlockEntry`, `CommandLine`, `Token`, `ToolCall`, `Message`, `Tab`, `PaletteItem`, `ShellSettings`, `Personality`, `Mode`, `TokenBudget`. These may need adapters to/from `ironhermes-core` types.
- **Design system** — CSS design tokens (`design-tokens.css`), Warp-IH styles (`warp-ih.css`), scanner animations (`scanner-anim.css`), custom font (Ioskeley Mono). All ported from the React prototype. No changes needed.
- **Mock data** (`demo_block_entries()`, `demo_messages()`, `demo_palette_items()`, `demo_tabs()`) — Source of truth for what data each component expects. Replace call sites, preserve type signatures where possible.
- **Platform abstraction** (`src/platform/timer.rs`) — WASM/native timer abstraction. Pattern for future platform-specific code.

### Established Patterns
- **Dioxus 0.7 signals** — All state is `use_signal` / `use_memo` / `use_context_provider`. No legacy `use_state` / `cx` / `Scope`.
- **`ShellSettings` bag-of-signals** — Cross-cutting state provided via context. Individual `Signal<T>` fields so writes to one don't invalidate others. Extend for backend-connected state.
- **`BlockEntry` with stable IDs** — Blocks carry `id: u64` for RSX key stability across clear/append cycles. Runtime-appended entries start at ID 1000.
- **CSS class mapping** — `Block::kind_class()` and `Token::kind_class()` map to `warp-ih.css` classes. Pattern: Rust enums → CSS class strings.
- **Feature gates** — `web` (default), `desktop`, `mobile` features. Server-side deps will go behind `server` feature.

### Integration Points
- **`fn main()` → `dioxus::launch(App)`** — Entry point. Fullstack changes to `dioxus::LaunchBuilder` or conditional `#[cfg(feature = "server")]` main.
- **`App` component** (`src/app.rs`) — Root component. Currently links CSS and renders `WarpHermes {}`. Will gain context providers for backend connections (session handle, WebSocket, command router proxy).
- **`WarpHermes` component** (`src/components/warp_hermes.rs`) — Main shell layout. Currently calls `demo_*()` fixtures. All mock call sites are in this file and `state.rs`. These are the primary edit targets.
- **`Cargo.toml`** — Needs `dioxus = { features = ["fullstack"] }`, plus `ironhermes-*` crate deps behind `#[cfg(feature = "server")]`.
- **Root `Cargo.toml`** — May need `default-members` adjustment to exclude `iron_hermes_ui` from default `cargo build`.

<specifics>
## Specific Ideas

- User emphasized "the crate is already built, installed and mocked up — it needs to be wired into IronHermes." This is a connect-the-wires phase, not a design phase. No new components, no visual changes. Replace mock data sources with real backend calls.
- The existing `warp2ironhermes/` directory in the UI crate is a React prototype reference — read-only, not compiled. It stays as design reference.
- Dioxus 0.7's `#[server]` functions compile to API endpoints on the server and HTTP calls on the client — zero manual endpoint plumbing needed. This is why Fullstack Dioxus was chosen over a separate API crate.
- The UI currently uses `gloo-timers` for WASM timer abstraction and `web-sys` for browser APIs. These stay for client-side code; server functions add the `ironhermes-*` deps.
- PROJECT.md currently lists "Web UI" as Out of Scope — this needs updating to reflect that a Web UI now exists as Phase 25.5.

</specifics>

<deferred>
## Deferred Ideas

- **Desktop and iOS targets** — Dioxus supports `desktop` and `mobile` features. Web is first; native targets are a follow-up phase after web wiring is proven.
- **Cross-interface session sharing** — Starting a conversation in TUI and continuing in web UI. Technically possible via shared `state.db` but explicitly deferred. Each interface has its own session namespace.
- **Multi-user authentication** — Web UI is single-operator, same as existing gateway. Multi-user auth is out of scope per PROJECT.md constraints.
- **Admin dashboard panels** — The existing agent panel and status bar provide basic monitoring. Dedicated admin/analytics dashboards are future scope.
- **Real-time collaboration** — Multiple users viewing/interacting with the same session. Not in scope.
- **PROJECT.md update** — Remove "Web UI" from Out of Scope, add to Active requirements. (This is a metadata task, not a code task — can be done during planning or execution.)
- **TUI deprecation** — Explicitly rejected. TUI stays permanently alongside web UI.

</deferred>
