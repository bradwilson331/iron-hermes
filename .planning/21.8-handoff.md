## Security primitives ŌĆö non-negotiable

These must come over from the TypeScript codebase verbatim (semantics, not literal code):

1. **Terminal escape stripping** (`src/sanitize.ts::stripTerminalEscapes`) ŌĆö strip CSI, OSC, DCS, PM, APC, simple ESC, C1, raw control chars from every string emitted to terminal that came from skills.sh metadata.
2. **YAML-only frontmatter** ŌĆö reject `---js` and any non-YAML frontmatter delimiter.
3. **Path traversal guards** ŌĆö `sanitizeName`, `sanitizeSubpath`, `isPathSafe`, `isSubpathSafe`, `isContainedIn`. Use Rust's `path-clean` + manual prefix checks; never trust skill name or subpath from the manifest.
4. **Temp dir containment** ŌĆö if any extraction temp dir is created, validate it lives under `std::env::temp_dir()` before any cleanup `rm`.
5. **Pre-install audit** ŌĆö TypeScript fetches `https://add-skill.vercel.sh/audit` for partner risk data shown pre-install. Decide whether to port (probably yes ŌĆö same endpoint).

## Scope to confirm during questioning

**In scope (proposed):**
- `hermes skill install <skill-name>` ŌĆö pull from skills.sh blob API, validate, extract to ironhermes' skills dir, update local lock
- `hermes skill list` ŌĆö read installed skills from local lock + filesystem
- `hermes skill remove <skill-name>` ŌĆö remove + update lock
- `hermes skill update [<skill-name>]` ŌĆö diff lock's snapshot hash against skills.sh, re-install if changed
- Lock file at `~/.ironhermes/skills-lock.json` (mirror TypeScript schema)

- Telemetry endpoint (`add-skill.vercel.sh/t`) ŌĆö opt out by default; ironhermes has its own observability story

## Reference: skills.sh blob API contract

From `src/blob.ts` (skills CLI):
- `GET https://skills.sh/api/download?repo=<owner>/<repo>&path=<subpath>&ref=<sha>` ŌåÆ returns archive
- Pre-flight via GitHub Trees API for SHA + file list
- Override base URL via `SKILLS_DOWNLOAD_URL` env var
- Snapshot hash returned in response ŌĆö use as the version key in the local lock
