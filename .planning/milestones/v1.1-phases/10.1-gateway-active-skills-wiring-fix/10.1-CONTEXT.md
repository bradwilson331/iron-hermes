# Phase 10.1: Gateway active_skills Wiring Fix - Context

**Gathered:** 2026-04-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix the active_skills Arc mismatch so skill activation via the skills tool in Telegram conversations correctly feeds into AgentLoop's allowed_tools enforcement. The bug: `GatewayMessageHandler::new()` creates its own `Arc<Mutex<Vec<SkillRecord>>>` at `handler.rs:72` instead of receiving the one already shared with `SkillsTool`. Result: two isolated Arcs — skill activation never reaches enforcement.

**Not in scope:** Enforcement semantics changes (already locked by Phase 07.5), per-user skill isolation, new SkillsTool actions, or any Phase 8+ features.

</domain>

<decisions>
## Implementation Decisions

### Arc sharing strategy
- **D-01:** Add `set_active_skills(&mut self, skills: Arc<Mutex<Vec<SkillRecord>>>)` setter to `GatewayMessageHandler`, following the same pattern as `set_memory_store()` / `set_hook_registry()` / `set_skill_registry()`.
- **D-02:** Add `active_skills: Option<Arc<Mutex<Vec<SkillRecord>>>>` field to `GatewayRunner` with a `set_active_skills()` setter. In `start()`, pass it to `handler.set_active_skills()` if present. Mirrors how `memory_store`, `hook_registry`, and `skill_registry` flow through `GatewayRunner → GatewayMessageHandler`.
- **D-03:** In `main.rs run_gateway()`, call `runner.set_active_skills(active_skills.clone())` after creating the Arc on line 437-438, passing the same Arc that was given to `register_skills_tool()` on line 439.
- **D-04:** `GatewayMessageHandler::new()` still initializes `active_skills` with an empty Arc as the default (backward compat for any path that doesn't call the setter). The setter overwrites it with the shared one.

### Per-user vs global scope
- **D-05:** Keep the current global-shared design — one `Arc<Mutex<Vec<SkillRecord>>>` shared by all users. This matches the single-operator deployment model (SEC-03). User A activating a skill affects User B's tool restrictions; acceptable for current use case.
- **D-06:** Add a brief comment at the `active_skills` field in `handler.rs` noting it's global-shared and would need per-session isolation for true multi-user support.

### Test approach
- **D-07:** Two-part regression test: (1) assert that the handler's `active_skills` and `SkillsTool`'s `active_skills` are the same Arc allocation (Arc::ptr_eq), and (2) activate a skill with restrictive `allowed_tools` via the handler path, then verify a blocked tool call returns the enforcement error.
- **D-08:** Test lives in `handler.rs #[cfg(test)]` module — this is where the bug manifests (handler creates the wrong Arc).

### Claude's Discretion
- Whether `GatewayMessageHandler` needs a getter for `active_skills` (for test assertions) or if the test should verify through behavior only
- Whether to add a `with_active_skills()` builder method in addition to the setter
- Internal naming for the setter parameter
- Exact test fixture design (how to construct handler with ToolRegistry containing SkillsTool for the integration portion)

</decisions>

### Integration Points
- `GatewayRunner` — gains `active_skills` field + `set_active_skills()` setter
- `GatewayMessageHandler` — gains `set_active_skills()` setter
- `main.rs run_gateway()` — calls `runner.set_active_skills(active_skills.clone())` with the Arc from line 437

<specifics>
## Specific Ideas

- This is a pure wiring fix — no new types, no new semantics, no enforcement changes. The active_skills enforcement logic in `agent_loop.rs:339-370` is already correct and tested (Phase 07.5). The only problem is that the handler's Arc isn't the same allocation as the SkillsTool's Arc.
- The fix is 4-5 lines of setter code + 1 line in `main.rs` + 1 line in `runner.rs start()`. The test is the substantial part.

</specifics>
