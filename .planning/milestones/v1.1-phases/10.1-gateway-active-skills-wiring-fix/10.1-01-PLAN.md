<objective>
Fix the active_skills Arc mismatch so skill activation via SkillsTool in Telegram conversations correctly feeds into AgentLoop's allowed_tools enforcement.

Purpose: GatewayMessageHandler::new() creates its own fresh Arc<Mutex<Vec<SkillRecord>>> at handler.rs:72 instead of receiving the one shared with SkillsTool. When a user activates a skill, the SkillsTool updates its Arc but the handler passes a different (empty) Arc to AgentLoop — so enforcement never sees the activation. This is the last gap in SKILL-06 enforcement completeness.

Output: Wired setter on handler + runner, call site in main.rs, two-part regression test proving Arc identity and behavioral enforcement.
</objective>

**Test 1: `test_active_skills_arc_shared`**

```rust
#[test]
fn test_active_skills_arc_shared() {
    // The bug: handler.new() created its own Arc, so skills activated via SkillsTool
    // never reached AgentLoop enforcement. The fix: set_active_skills() overwrites
    // the default with the shared Arc.
    let config = Config::default();
    let session_store = Arc::new(RwLock::new(SessionStore::new()));
    let tool_registry = Arc::new(ToolRegistry::new());
    let mut handler = GatewayMessageHandler::new(config, session_store, tool_registry);

    let shared: Arc<std::sync::Mutex<Vec<ironhermes_core::SkillRecord>>> =
        Arc::new(std::sync::Mutex::new(Vec::new()));
    handler.set_active_skills(shared.clone());

    assert!(
        Arc::ptr_eq(&shared, &handler.active_skills),
        "handler.active_skills must be the same Arc allocation as the one passed to set_active_skills"
    );
}
```

**Test 2: `test_active_skills_enforcement_fires`**

```rust
#[tokio::test]
async fn test_active_skills_enforcement_fires() {
    // Prove: when active_skills Arc contains a skill with restrictive allowed_tools,
    // AgentLoop's execute_tool_call rejects tools not in the allowed set.
    // This is the behavioral half of the regression — if the Arc were a separate
    // allocation (the bug), this test would pass vacuously (empty skills = no restriction).

    let shared: Arc<std::sync::Mutex<Vec<ironhermes_core::SkillRecord>>> =
        Arc::new(std::sync::Mutex::new(Vec::new()));

    // Populate with a restrictive skill
    {
        let mut skills = shared.lock().unwrap();
        skills.push(ironhermes_core::SkillRecord {
            name: "restrictive-skill".to_string(),
            description: "test skill".to_string(),
            body: String::new(),
            allowed_tools: Some(vec!["skills".to_string()]),
        });
    }

    // Create AgentLoop with the shared Arc (same one handler would pass after fix)
    let client = ironhermes_agent::LlmClient::new(
        "http://localhost:0".to_string(),
        "test-key".to_string(),
        "test-model".to_string(),
    );
    let loop_instance = ironhermes_agent::AgentLoop::new(client)
        .with_active_skills(shared.clone());

    // Try to call a tool not in allowed_tools
    let result = loop_instance
        .execute_tool_call("read_file", &serde_json::json!({"path": "/tmp/test"}))
        .await;

    let err_msg = result.unwrap_or_else(|e| e.to_string());
    assert!(
        err_msg.contains("not permitted by the active skill set"),
        "Expected enforcement error, got: {err_msg}"
    );
}
```

Adjust imports at the top of the test module as needed — ensure `ToolRegistry`, `Config`, `SessionStore`, `Arc`, `RwLock` are available. The test module is inside handler.rs so it has access to private fields (per D-08, Research Pitfall 4).
