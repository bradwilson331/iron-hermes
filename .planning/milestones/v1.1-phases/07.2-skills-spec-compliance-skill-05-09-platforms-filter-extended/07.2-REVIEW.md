# Phase 07.2: Code Review Report

**Reviewed:** 2026-04-09
**Depth:** standard

## Summary

Phase 07.2 adds the SKILL-05..08 spec compliance work: `SkillFrontmatter` /
`SkillRecord` gain `platforms`, `compatibility`, `allowed-tools`, and `metadata`
fields; `parse_skill_md` enforces strict name validation and warn-but-load
description length; a platform filter runs before dedup (pinned by D-06 test);
and `SkillsConfig` introduces `enabled` / `extra_paths` with defaults-first
append ordering (D-19). The three CLI entry points (`run_single`, `run_chat`,
`run_gateway`) all migrated to `SkillRegistry::load_with_config(&cwd,
&config.skills)`, consistently passing the parsed config through.

The core behavioral changes in `skills.rs`, `config.rs`, and `main.rs` are
well-structured, thoroughly tested (parse, platform filter, validation, config
round-trip, legacy parity), and backward-compatible. `serde(default)` guards
both `SkillsConfig` itself and its fields, so pre-existing `config.yaml` files
load unchanged.

The clippy-cleanup changes across the 12 other files are mechanical
transformations (let-chain collapse, `.or_default()`, `is_some_and`,
`#[derive(Default)]`, `RangeInclusive::contains`, `#[allow(clippy::
too_many_arguments)]`). Each one preserves behavior exactly; the let-chain
collapses in particular are semantically identical to the nested `if let`
forms they replace. No behavioral regressions were introduced.

Findings below are minor — two warnings about latent edge-cases in skill
path resolution and double-lookup, plus a handful of info-level
observations.

## Warnings

### WR-01: HOME env fallback silently resolves to cwd

**File:** `crates/ironhermes-core/src/skills.rs:208-215`

**Issue:** `build_skill_search_paths` resolves `$HOME` as follows:

```rust
let home = std::env::var("HOME")
    .map(PathBuf::from)
    .unwrap_or_else(|_| dirs::home_dir().unwrap_or_else(|| PathBuf::from(".")));
```

On the (rare but possible) edge case where both `HOME` is unset AND
`dirs::home_dir()` returns `None` (e.g., certain sandboxed CI runners, systemd
units with `ProtectHome=true`, Windows environments where `USERPROFILE` is
stripped), `home` silently falls back to `PathBuf::from(".")`. This causes
the third default search path to resolve to `./.agents/skills` (relative to
the process cwd), which is (a) unexpected, (b) could accidentally shadow a
legitimate relative directory in the working tree, and (c) not a useful
scan target.

Since this is the third default priority and extras are appended after, a
skill in `./.agents/skills` that happens to exist could still be loaded
(violating user intent), and any debug log saying "scanning ~/.agents/skills"
would be misleading.

Low-severity because the env is almost never both-missing in practice, but
the silent fallback has no warning and no test.

**Fix:** Either warn loudly and skip the path, or drop the fallback and
return just the first two defaults when HOME is unresolvable:

```rust
let home = std::env::var("HOME")
    .ok()
    .map(PathBuf::from)
    .or_else(dirs::home_dir);
let mut paths = vec![
    cwd.join(".ironhermes/skills"),
    get_hermes_home().join("skills"),
];
if let Some(home) = home {
    paths.push(home.join(".agents/skills"));
} else {
    tracing::debug!("SkillRegistry: HOME unresolvable, skipping ~/.agents/skills default");
}
paths.extend(config.extra_paths.iter().cloned());
paths
```

### WR-02: `extra_paths` are not validated or canonicalized before scanning

**File:** `crates/ironhermes-core/src/skills.rs:207-219` and
`crates/ironhermes-core/src/config.rs:195-211`

**Issue:** User-supplied `extra_paths` in `SkillsConfig` are cloned directly
into the search list without any validation. Specifically:

1. They are not canonicalized — two entries pointing at the same dir via
   different paths (e.g., `/opt/skills` and `/opt/../opt/skills`) would both
   be scanned; first-path-wins dedup is on the skill *name*, not the path,
   so this only wastes a directory read, but it also means a user can
   accidentally shadow a default by listing a shorter symlink-resolved form
   of one of the three defaults into `extra_paths` (the filter is
   name-based, so the default still wins — so this is latent, not
   exploitable).
2. There is no size/count cap. A pathological config with thousands of
   `extra_paths` entries would cause thousands of `read_dir` syscalls on
   every scan.
3. Relative paths are interpreted relative to cwd at scan time. Since the
   config is loaded from `$IRONHERMES_HOME/config.yaml`, users may
   reasonably expect relative paths to be anchored to the config file's
   location or to `$IRONHERMES_HOME`. No test pins this behavior and the
   SkillsConfig doc comment does not clarify.

None of these are bugs, but #3 in particular is a footgun — a user writes
`extra_paths: ["./my-skills"]` in `~/.ironhermes/config.yaml` and is
surprised when it depends on where they launched `ironhermes` from.

**Fix:** Document the relative-path semantics in the `SkillsConfig`
doc-comment and consider resolving relative `extra_paths` against
`get_hermes_home()` (or refusing them) at load time:

```rust
// In build_skill_search_paths, after extending with extras:
for p in config.extra_paths.iter() {
    if p.is_relative() {
        tracing::warn!(
            "SkillsConfig.extra_paths entry {:?} is relative; \
             resolving against cwd — prefer absolute paths",
            p
        );
    }
}
```

Add a test pinning the intended semantics, and update the `SkillsConfig`
rustdoc to state "absolute paths recommended; relative paths are resolved
against the process cwd at scan time, not against the config file location."

## Info

### IN-01: `seen_names` uses `contains` + `insert` instead of `insert`-returns-bool

**File:** `crates/ironhermes-core/src/skills.rs:322-331`

**Issue:** The dedup check in the registry loop does two `HashSet` operations
where one would suffice:

```rust
let name_lower = frontmatter.name.to_lowercase();
if seen_names.contains(&name_lower) {
    debug!(...);
    continue;
}
seen_names.insert(name_lower);
```

`HashSet::insert` already returns `false` if the key was present. This is
purely stylistic — not a bug, no performance issue at skill-catalog scale.

**Fix:**

```rust
let name_lower = frontmatter.name.to_lowercase();
if !seen_names.insert(name_lower.clone()) {
    debug!(
        "SkillRegistry: skipping duplicate skill '{}' at {:?}",
        frontmatter.name, skill_md_path
    );
    continue;
}
```

### IN-02: Description length limit uses char count; spec-matching but undocumented

**File:** `crates/ironhermes-core/src/skills.rs:142-148`

**Issue:** `parse_skill_md` checks `frontmatter.description.chars().count()`
against 1..=1024. This is the correct choice for a human-facing length
(grapheme-adjacent), but it disagrees with Rust's common `str::len()` (byte
count) and with the `SKILL_DESC_MAX_LEN: usize = 1024` constant whose name
does not disclose the unit.

Not a bug — tests pin chars-based behavior at the boundaries. Just worth a
comment documenting that the unit is Unicode scalars, not bytes, so a future
maintainer doesn't "optimize" it to `description.len()`.

**Fix:** Rename or comment the constant:

```rust
/// Maximum description length in Unicode scalar values (chars), not bytes.
/// Matches the agentskills.io spec.
const SKILL_DESC_MAX_LEN: usize = 1024;
```

### IN-04: `SkillFrontmatter.metadata` is `serde_yaml::Value` but never surfaced beyond mirroring
