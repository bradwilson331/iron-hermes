## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SKILL.md file → SkillFrontmatter parse | User-authored YAML crosses into typed structs via serde_yaml |
| `metadata: serde_yaml::Value` opaque blob | Arbitrary YAML shapes accepted without schema validation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07.2-01-01 | Tampering | SkillFrontmatter deserialization | mitigate | serde_yaml rejects malformed YAML; parse_skill_md already returns None on failure. No additional work — existing parse-or-skip behavior covers malformed extended frontmatter. |
| T-07.2-01-02 | Denial of Service | `metadata: Option<serde_yaml::Value>` opaque blob — unbounded depth/size | accept | serde_yaml is not known to have stack-overflow or unbounded-allocation vulnerabilities for YAML within normal file-size limits. SKILL.md files are loaded from trusted local directories (user's own filesystem). Risk is low-severity. Documented trust boundary: SKILL.md is user-owned content. |
| T-07.2-01-03 | Information Disclosure | New fields leaking into system prompt via catalog | mitigate | Catalog format is explicitly unchanged per D-12 and D-26. Test `test_catalog_text_format` pins the `"- name: description"` format. Verified by not modifying `catalog_text()`. |
| T-07.2-01-04 | Elevation of Privilege | `allowed_tools` misinterpreted as enforcement | mitigate | D-10 explicitly locks allowed_tools as "parsed but NOT enforced". Doc comment on the field states this. Enforcement is a future security feature with its own design. |
</threat_model>
