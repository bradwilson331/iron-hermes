<decisions>
## Implementation Decisions

### Activation State Tracking
- **D-01:** Add `active_skills: Arc<Mutex<Vec<SkillRecord>>>` field to `AgentLoop`. This tracks which skills are currently active for the duration of the agent run. Natural cleanup — the vec is dropped when the AgentLoop is dropped.
- **D-02:** `SkillsTool` receives the same `Arc<Mutex<Vec<SkillRecord>>>` in its constructor. On `activate`, it clones the `SkillRecord` and pushes it into the shared vec. Same shared-state pattern as `MemoryStore`/`JobStore`.
- **D-03:** This introduces session state for skill activation, superseding Phase 7 D-07's "no session state tracking" for the specific purpose of `allowed_tools` enforcement. The `activate` action still returns the full SKILL.md body as the tool result (D-07 behavior preserved); the state tracking is a side-effect for enforcement only.

### Enforcement Semantics
- **D-04:** When `active_skills` is empty, ALL tools are allowed. The check only activates when at least one skill with `allowed_tools: Some(...)` is present. Backward-compatible — existing behavior is unchanged unless a skill explicitly restricts.
- **D-05:** When multiple skills are active, their `allowed_tools` lists combine as a **union**. A tool is permitted if ANY active skill allows it. Activating more skills expands the available toolset.
- **D-06:** Skills with `allowed_tools: None` (field absent or null) are **ignored** in enforcement. Only skills that explicitly declare `allowed_tools` participate in the filtering. A skill without the field is "open" and does not restrict anything.
- **D-07:** The `skills` tool itself is **always allowed**, regardless of any active skill's `allowed_tools` list. This prevents a restrictive skill from locking the agent out of activating additional skills or deactivating the restricting skill.
- **D-08:** Enforcement check runs in `execute_tool_call` in `agent_loop.rs`, AFTER arg parsing and BEFORE the guardrail check. If the tool is not in the allowed union, return an error immediately without firing `ToolCalled` (same pattern as guardrail blocks from 07.4 D-05).

### Error Message Design
- **D-09:** When a tool is blocked by `allowed_tools` enforcement, the error message names the restriction and lists allowed tools so the agent can self-correct:
  ```
  Tool '{name}' is not permitted by the active skill set. Allowed tools: [{union list}].
  Activate a skill that permits '{name}' or deactivate the restricting skill.
  ```
  This is NOT routed through the guardrail error formatting (`format_guardrail_error`) — it's a separate skill-enforcement path with its own message shape.

### Deactivation
- **D-10:** Add a `deactivate` action to `SkillsTool`. It removes the named skill from `active_skills` via `vec.retain(|s| s.name != name)`. Returns confirmation: `"Skill '{name}' deactivated."` If the skill wasn't active, returns: `"Skill '{name}' is not currently active."` Update the tool's JSON schema to include the new action.

### Cron Job Enforcement
- **D-11:** When `execute_cron_job` in `runner.rs` builds an `AgentLoop` for a cron-triggered run, it resolves each attached skill name to a `SkillRecord` and pre-populates `active_skills` before the loop starts. Same enforcement path as conversation mode — consistent behavior.
- **D-12:** Skills with `allowed_tools` attached to cron jobs restrict which tools the cron-triggered agent can use. This is the same union semantics as D-05.

### Documentation Cleanup
- **D-13:** Fix the ROADMAP Phase 07.2 title/description to say "SKILL-05..08" instead of "SKILL-05..09". The phase directory name (`07.2-skills-spec-compliance-skill-05-09-...`) is left as-is to avoid breaking references in existing planning docs.
- **D-14:** REQUIREMENTS.md is already correct — SKILL-09 is in v2 (line 120), SKILL-13 is in Backlog (line 83). No changes needed there.

### Claude's Discretion
- Whether `active_skills` check in `execute_tool_call` goes before or after the guardrail check (D-08 says before guardrails, but planner can adjust if there's a good reason)
- Whether to add an `active` or `list_active` action to SkillsTool for introspection (showing which skills are currently active)
- Internal naming: `active_skills` vs `activated_skills` vs `skill_state`
- Whether to fire a hook event on skill activation/deactivation (leveraging Phase 6 hooks)
- Test fixture design for integration tests

</decisions>

<specifics>
## Specific Ideas

- The `allowed_tools` field is already `Option<Vec<String>>` on both `SkillFrontmatter` and `SkillRecord` (Phase 07.2 D-08/D-11). No schema changes needed — this phase is pure enforcement wiring.
- Follow the `Arc<Mutex<T>>` sharing pattern already used by `MemoryStore` and `JobStore` — no new concurrency primitives.
- The error message should be actionable for the LLM — listing allowed tools lets it choose an alternative or ask the user.
- The `deactivate` action keeps the tool interface symmetric: list/view/activate/deactivate.

</specifics>

## Existing Code Insights

### Reusable Assets
- `Arc<Mutex<Vec<SkillRecord>>>` — follows the same pattern as `Arc<Mutex<MemoryStore>>` and `Arc<Mutex<JobStore>>`. No new concurrency patterns needed.
- `SkillRecord.allowed_tools: Option<Vec<String>>` — already parsed and stored per 07.2. Enforcement reads this field.
- `execute_tool_call` D-05 ordering — the check→fire→execute pattern from 07.4 provides a clean insertion point for the skill enforcement check.
- `resolve_skill_context` in `runner.rs` — already resolves skill names to content for cron. Extending to also resolve SkillRecords for enforcement is straightforward.

### Established Patterns
- Shared mutable state via `Arc<Mutex<T>>` passed through constructors (MemoryStore, JobStore)
- Action-based tool with JSON args (CronjobTool, SkillsTool)
- Guardrail-style blocking in `execute_tool_call` with structured error return

### Integration Points
- `AgentLoop::new()` — gains `active_skills` parameter
- `SkillsTool::new()` — gains `active_skills` parameter
- `execute_tool_call()` — new skill enforcement check before guardrail check
- `execute_cron_job()` — pre-populate `active_skills` from job's skill list
- CLI `run_single`/`run_chat`/`run_gateway` — create and wire `active_skills` Arc
