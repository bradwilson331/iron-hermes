# Phase 07.4 Plan 03: Cron Path Exactly-One Integration Tests Summary

**One-liner:** Integration-level proof that execute_cron_job fires MessageReceived and ResponseSent exactly once per cron run, via direct invocation with an unreachable LLM and a source-text regression guard.

## What Was Built

This plan adds integration-level tests proving exactly-one hook event counts for the cron path, complementing the Telegram path test added in Wave 1.

### Context

After Waves 1 and 2:
- Wave 1 added `test_telegram_roundtrip_produces_exactly_one_message_received_and_response_sent` in runner.rs
- Wave 0 (07.3) added `test_cron_hook_registry_receives_events` which manually fires events without calling `execute_cron_job`
- Missing: a test that calls `execute_cron_job` directly and asserts exactly-one counts end-to-end

### New Tests

**`test_cron_path_fires_exactly_one_message_received_and_response_sent`**

Calls `execute_cron_job` directly with:
- A real `CronJob` from a temp `JobStore`
- A capturing `HookRegistry` with a listener that records `HookEventKind` values
- A `Config` pointing to `http://127.0.0.1:1` (port 1 — always connection refused)
- No LLM required: agent.run() fails fast, the Err arm fires ResponseSent per D-04

Asserts:
- Exactly 1 `MessageReceived` event (fired before `agent.run()`)
- Exactly 1 `ResponseSent` event (fired in Err arm)
- Both events have `platform = "cron"`

This proves that even in the error path, the cron wrapper fires the lifecycle pair exactly once — and that the agent loop does NOT fire additional events (confirming Issue #4 fix from Wave 1 covers the cron path).

**`test_runner_source_execute_cron_job_fires_events_exactly_as_expected`**

Source-text regression guard that:
- Uses `include_str!("runner.rs")` to load the source at test time
- Locates the `execute_cron_job` function body (between its signature and `resolve_token`)
- Counts `HookEventKind::MessageReceived` occurrences: asserts exactly 1
- Counts `HookEventKind::ResponseSent` occurrences: asserts exactly 2 (Ok arm + Err arm)
- Uses `concat!` split strings so the test's own assertion text does not self-match (same pattern as agent_loop.rs test 5)

Any future addition of a duplicate `MessageReceived` fire or removal of the error-arm `ResponseSent` will fail CI immediately.

## Tests Added
