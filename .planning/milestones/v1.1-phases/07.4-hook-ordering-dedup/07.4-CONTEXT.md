# Phase 07.4: Hook Ordering & Duplicate Event Fixes - Context

**Gathered:** 2026-04-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Resolve the two **warning-severity** integration issues from the v1.1 milestone audit so hook event streams are accurate and single-source:

1. **Hook ordering** — `agent_loop.rs:317` fires `ToolCalled` BEFORE `ToolRegistry::dispatch()` runs the guardrail chain at `registry.rs:87-109`. JSONL logs and webhook endpoints currently record `tool_called` for tool invocations that are immediately blocked by guardrails.

2. **Duplicate MessageReceived/ResponseSent** — Each Telegram message produces 2× `MessageReceived` and 2× `ResponseSent` events with mismatched metadata:
   - `gateway/handler.rs:217-229` and `:385-397` fire with `platform="telegram"` + real `chat_id`
   - `agent_loop.rs:131` and `:200` fire with `platform="agent"` + empty `chat_id`

   Post-07.3, the cron path has the **same latent duplicate**: `gateway/runner.rs:620+` fires manually with `platform="cron"` + real chat_id, AND `runner.rs:669` passes `with_hook_registry()` to `AgentLoop` which re-fires with `platform="agent"` + empty chat_id. 07.4 must fix both Telegram and cron paths — not Telegram alone.

**Cosmetic, not functional:** Neither bug breaks the agent loop. Both corrupt the observability contract promised by HOOK-01 and HOOK-02.

**Out of scope:**
- Adding new hook event kinds (Phase 06 D-01 locked the 4-event model; 07.4 does not change it)
- Hot-reload of hook listeners (deferred Phase 06 tech debt)
- CLI direct-mode hook wiring (CLI `main.rs:237/363` does not pass `with_hook_registry()` to `AgentLoop` today — removing fires from AgentLoop is not a regression for that path because CLI emits no hook events today regardless)
- Platform label for AgentLoop-standalone callers (see D-06 — standalone AgentLoop callers now emit zero lifecycle events, by design)

</domain>

<decisions>
## Implementation Decisions

### Dedup Ownership (fixes audit warning #4)

- **D-01: Remove `MessageReceived` and `ResponseSent` firing from `agent_loop.rs`.** Delete the `fire_hook(HookEventKind::MessageReceived {...})` block at `agent_loop.rs:128-136` and the `fire_hook(HookEventKind::ResponseSent {...})` block at `:199-205`. The `agent_loop.rs` file no longer fires either event kind after this phase.
  - Evidence: audit `gaps.integration[3]` (severity: warning), roadmap SC#2, Phase 07.3 D-12 explicitly warned against re-introducing empty `(platform, chat_id)` values

- **D-02: Edges (gateway + cron) are the sole firing sites for `MessageReceived`/`ResponseSent`.** `gateway/handler.rs:217-229 / :385-397` (Telegram path) and `gateway/runner.rs:620+ / :685+` (cron path) keep their existing manual fires with real `(platform, chat_id)`. No changes to their firing logic except to confirm they still fire after D-01 lands.
  - Evidence: 07.3 D-12 ("new cron path should NOT re-introduce empty values"); gateway/handler already has correct metadata

- **D-03: Tool lifecycle events (`ToolCalled`, `ToolCompleted`) remain owned by `AgentLoop`.** Only the agent's tool-dispatch loop knows when a tool is called. Edges do not see tool calls, so edges cannot fire these. `agent_loop.rs` remains the sole firing site for tool events (subject to the ordering fix in D-04/D-05).
  - Evidence: current architecture; edges have no visibility into tool dispatch

- **D-06: AgentLoop callers that do not go through gateway/cron edges will emit zero hook lifecycle events.** This is the accepted trade-off of D-01/D-02. CLI `main.rs:237/363` already does not wire `with_hook_registry()` to its `AgentLoop` instances, so no regression today. If Phase 9 (Subagent Delegation) or Phase 10 (Batch Processing) later need lifecycle events for AgentLoop-standalone callers, those phases must fire the events themselves at their own edge — do not reopen D-01.
  - Evidence: `cli/src/main.rs:237, :363` AgentLoop::new calls without with_hook_registry

### ToolCalled Ordering (fixes audit warning #3)

- **D-04: Split `ToolRegistry::dispatch` into an explicit guardrail check + execution.** Add a new public method `ToolRegistry::check_guardrails(name: &str, args: &serde_json::Value) -> GuardrailDecision` that runs the existing guardrail chain at `registry.rs:87-109` WITHOUT executing the tool. The existing `dispatch()` method may either (a) internally call `check_guardrails` first for backwards compatibility, or (b) be renamed/split into `execute_unchecked` — planner's discretion as long as the new API surface is cohesive. The critical requirement: agent_loop must be able to get a `GuardrailDecision` without triggering tool execution.
  - Evidence: roadmap SC#1 exact wording: "fires `ToolCalled` only **after** the guardrail chain has permitted the call"

- **D-05: `agent_loop.rs::execute_tool_call` runs check → fires `ToolCalled` on `Allow` → executes → fires `ToolCompleted`.** The new ordering inside `execute_tool_call`:
  1. Call `self.registry.check_guardrails(name, &args)` — returns `Allow | Warn | Block`.
  2. On `Block { reason }`: fire `ToolCompleted { success: false, result_preview: "<block reason>", ... }`. Do NOT fire `ToolCalled`. Return the formatted block error string as the tool result (preserving today's behavior where blocked tools return an error-shaped string that the LLM sees).
  3. On `Allow` or `Warn`: fire `ToolCalled` FIRST (this is the post-fix ordering), then call the execution entrypoint (`dispatch` or `execute_unchecked` — whichever D-04 ends up with), then fire `ToolCompleted` with success/failure based on the execution result. An execution error on the allowed path still emits both `ToolCalled` and `ToolCompleted{success:false}` — matches today's observability for execution failures.
  - Evidence: roadmap SC#1; Phase 06 D-01 (4-event model); preserves execution-error observability

- **D-07: Blocked-tool observability is via `ToolCompleted{success:false, result_preview: <reason>}`.** No new event kind is added. Phase 06 D-01's 4-event model (`message_received`, `tool_called`, `tool_completed`, `response_sent`) stays unchanged. Blocked calls are fully observable in JSONL logs and webhook endpoints via the `ToolCompleted` entry whose `result_preview` contains the guardrail reason string (formatted via existing `format_guardrail_error`).
  - Evidence: Phase 06 D-01; planner may re-use `ironhermes_hooks::format_guardrail_error` already called inside `dispatch()`

- **D-08: Warn-level guardrail decisions count as Allow for event firing.** A `GuardrailDecision::Warn { reason }` is still a permitted call — tool executes, `ToolCalled` fires, `ToolCompleted` fires. The `tracing::warn!` side-effect at `registry.rs:90-97` remains. This matches the existing semantics of `dispatch()` which continues past warn-level guardrails to execute the tool.
  - Evidence: `registry.rs:90-97` current warn-proceeds behavior; existing test `test_dispatch_warn_guardrail_proceeds`

### Test Coverage (roadmap SC#3)

- **D-09: One canonical Telegram round-trip test must assert event counts.** A single Telegram message (no tool calls) must produce exactly one `MessageReceived` and one `ResponseSent` event in the shared `HookRegistry` — not two of each, not zero. Test subscribes a capture listener (pattern copied from `registry.rs` tests and the 07.3 `test_cron_hook_registry_receives_events` test at `runner.rs:942+`) and asserts `events.iter().filter(|e| matches!(kind, MessageReceived)).count() == 1` and similarly for `ResponseSent`.
  - Evidence: roadmap SC#2 exact wording ("A single Telegram message produces exactly one `MessageReceived` and one `ResponseSent` event")

- **D-10: A second test must assert `ToolCalled` is NOT emitted for a guardrail-blocked tool call.** Wire a `BlocklistGuardrail` to block a mock tool. Run a canonical round-trip that attempts to call the blocked tool. Assert the event capture contains zero `ToolCalled` events AND one `ToolCompleted{success: false}` event whose `result_preview` includes the block reason.
  - Evidence: roadmap SC#1 exact wording ("blocked tools never emit `tool_called` events")

- **D-11: A third test must assert the cron path is also deduplicated.** Expand or duplicate the existing `test_cron_hook_registry_receives_events` at `gateway/src/runner.rs:942+` to assert `MessageReceived` and `ResponseSent` are each fired exactly ONCE per cron tick. Today (pre-07.4) that test would see 2 of each because `runner.rs:620+` fires manually AND `:669` passes `with_hook_registry()` to AgentLoop. After D-01, the AgentLoop side stops firing them — the test must prove it.
  - Evidence: 07.3 introduced the same latent duplicate; audit warning #4 applies to cron too

- **D-12: Full `cargo test --workspace` stays green.** No regression on existing tests. In particular:
  - `crates/ironhermes-tools/src/registry.rs` tests (`test_dispatch_with_no_guardrails_passes`, `test_dispatch_blocked_by_guardrail`, `test_dispatch_allowed_by_guardrail`, `test_dispatch_warn_guardrail_proceeds`, `test_guardrail_error_detail_minimal`) must all still pass, possibly with API updates if D-04 renames `dispatch`.
  - 07.3's `test_cron_hook_registry_receives_events` must still pass — it may need its assertions updated from "events observed" to "exactly one of each event kind."
  - Evidence: Phase 07.3 D-18; audit tech_debt: 306 tests green baseline

### Claude's Discretion

The audit and roadmap lock the WHAT. Claude/planner picks the HOW for:

- Exact API shape for D-04 — whether to (a) rename `dispatch` to `execute_unchecked` and add a new `dispatch` that wraps `check_guardrails + execute_unchecked`, (b) keep `dispatch` as-is and add a parallel `check_guardrails` that agent_loop calls before `dispatch` (resulting in guardrails running twice on the happy path — acceptable cost for backwards compat?), or (c) split cleanly into `check_guardrails` + `execute_tool` and update every caller of `dispatch` in the workspace. Planner reads every call site of `ToolRegistry::dispatch` before deciding.
- Exact wording of the blocked-tool `result_preview` — re-use `format_guardrail_error` or construct a similar string locally in agent_loop
- Whether the agent_loop Err path on a blocked-but-unexecuted tool still returns a result string to push as a `ChatMessage::tool_result` to the LLM, or short-circuits differently. Current behavior (blocked tool → Err from dispatch → error message becomes the tool result so the LLM sees it) should be preserved unless the planner has strong reason to change it.
- Whether to consolidate the three test cases (D-09, D-10, D-11) into one parameterized test or keep them separate
- Module/file placement of the test helpers for the capture listener (shared test utility vs. duplicated across crates)

## Specific Ideas

1. **Grep-verifiable acceptance:** After 07.4, `grep -n "fire_hook(HookEventKind::MessageReceived" crates/ironhermes-agent/src/agent_loop.rs` must return zero matches, same for `fire_hook(HookEventKind::ResponseSent`. `grep -n "fire_hook(HookEventKind::ToolCalled" crates/ironhermes-agent/src/agent_loop.rs` must return exactly one match, and that match must be inside the post-check-guardrails Allow/Warn branch.

2. **Audit text the tests should echo:** Use the exact phrase "no duplicate MessageReceived" or "exactly one MessageReceived" in test names or doc-comments so future audits can mechanically map test → audit warning #4. Same for "ToolCalled after guardrail allow" → audit warning #3.

3. **Warn-level semantics are NOT changing.** A warn-level guardrail today proceeds to execute the tool and fires `ToolCompleted` on the normal path. Post-07.4 it must ALSO fire `ToolCalled` before execution (the reorder fix applies to warn the same as to allow). Do not accidentally skip `ToolCalled` on warn.

4. **Cron path is latent-broken today but silent.** No cron-specific test currently asserts exactly-one event counts, which is why the duplicate slipped past Phase 07.3's gate. 07.4 strengthens that test (D-11) alongside the Telegram one (D-09). The audit warning #4 only called out the Telegram path explicitly, but the same bug exists in cron — 07.4 fixes both.

5. **Preserve the `_full_input` rename invariant from 07.3 D-02.** 07.3 renamed `_full_input` → `full_input`. 07.4 does not touch that area of `runner.rs`. If the planner needs to edit `runner.rs` for tests, do not revert the rename.
