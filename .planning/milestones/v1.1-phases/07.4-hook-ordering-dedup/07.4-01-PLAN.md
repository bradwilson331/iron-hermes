<objective>
Split `ToolRegistry::dispatch` into an explicit guardrail check step and an execution step so that callers (specifically `agent_loop.rs::execute_tool_call`) can decide whether to fire `ToolCalled` hooks BETWEEN the guardrail check and the tool execution.

Purpose: Fixes audit integration warning #3 by enabling D-05's ordering (check -> fire ToolCalled -> execute -> fire ToolCompleted). Without this refactor, `agent_loop.rs` cannot observe the guardrail decision without also executing the tool.

Output: New public method `ToolRegistry::check_guardrails(name, &args) -> GuardrailDecision` and a new public method `ToolRegistry::execute_tool(name, args) -> Result<String>` that runs a tool WITHOUT re-checking guardrails. The existing `dispatch()` stays as a thin wrapper `check_guardrails(..) -> match { Block -> Err, Allow|Warn -> execute_tool(..) }` so any unknown caller keeps working. Existing registry tests continue to pass unchanged.
</objective>

Step 1. Add the following two new public methods to `impl ToolRegistry` in `crates/ironhermes-tools/src/registry.rs`, placed ABOVE the existing `pub async fn dispatch`. Signatures and bodies:

```rust
/// Run the registered guardrail chain against `(name, args)` WITHOUT executing the tool.
///
/// Returns the first non-Allow decision encountered (short-circuiting on Block), or
/// `GuardrailDecision::Allow` if all guardrails allow.
///
/// `Warn` decisions emit a `tracing::warn!` side-effect identical to `dispatch()`'s
/// existing behavior and then continue the loop — a `Warn` is only returned if it is
/// the final decision (i.e. no subsequent guardrail blocks).
///
/// Used by `agent_loop.rs::execute_tool_call` (phase 07.4 D-04/D-05) so that hook
/// events can be fired in the correct order relative to guardrail evaluation.
pub fn check_guardrails(
    &self,
    name: &str,
    args: &serde_json::Value,
) -> ironhermes_hooks::GuardrailDecision {
    use ironhermes_hooks::GuardrailDecision;

    let mut final_decision = GuardrailDecision::Allow;
    for guardrail in &self.guardrails {
        match guardrail.check(name, args) {
            GuardrailDecision::Allow => {}
            GuardrailDecision::Warn { reason } => {
                tracing::warn!(
                    tool = %name,
                    guardrail = %guardrail.name(),
                    reason = %reason,
                    "Guardrail warning (proceeding)"
                );
                // Per D-08: warn does not block — continue to next guardrail but
                // remember that at least one warned so callers can observe it.
                final_decision = GuardrailDecision::Warn { reason };
            }
            GuardrailDecision::Block { reason } => {
                // Per D-07: return the raw reason — callers format the error message
                // themselves via `format_guardrail_error` using the registry's
                // `error_detail` level (exposed via `guardrail_error_detail()` below).
                return GuardrailDecision::Block { reason };
            }
        }
    }
    final_decision
}

/// Accessor for the configured error detail level so callers (agent_loop) can format
/// block errors using the same `format_guardrail_error` path that `dispatch()` uses.
pub fn guardrail_error_detail(&self) -> &ironhermes_hooks::ErrorDetailLevel {
    &self.error_detail
}

/// Execute a tool WITHOUT running the guardrail chain.
///
/// Callers that want the full check+execute flow should use `dispatch` (or call
/// `check_guardrails` first, act on the decision, and then call `execute_tool`).
///
/// Returns an error if the tool is not registered or is unavailable.
pub async fn execute_tool(
    &self,
    name: &str,
    args: serde_json::Value,
) -> anyhow::Result<String> {
    let tool = self
        .tools
        .get(name)
        .ok_or_else(|| anyhow::anyhow!("Tool not found: {}", name))?;

    if !tool.is_available() {
        return Err(anyhow::anyhow!("Tool '{}' is not available", name));
    }

    tool.execute(args).await
}
```

Step 2. Refactor `pub async fn dispatch` so its body becomes a thin composition of the two new methods (so every existing dispatch caller still sees identical semantics):

```rust
pub async fn dispatch(
    &self,
    name: &str,
    args: serde_json::Value,
) -> anyhow::Result<String> {
    use ironhermes_hooks::GuardrailDecision;

    // Pre-check tool existence so the error message matches the previous behavior
    // (Err: "Tool not found" BEFORE guardrail chain runs). This preserves the
    // pre-07.4 ordering of errors for any external caller of dispatch().
    if !self.tools.contains_key(name) {
        return Err(anyhow::anyhow!("Tool not found: {}", name));
    }

    match self.check_guardrails(name, &args) {
        GuardrailDecision::Allow | GuardrailDecision::Warn { .. } => {
            self.execute_tool(name, args).await
        }
        GuardrailDecision::Block { reason } => {
            // Find the guardrail that blocked to preserve the "guardrail_name" in the
            // formatted error message. Re-walk the chain (cheap — guardrails are few)
            // and stop at the first Block, mirroring pre-07.4 behavior exactly.
            let guardrail_name = self
                .guardrails
                .iter()
                .find_map(|g| match g.check(name, &args) {
                    GuardrailDecision::Block { .. } => Some(g.name().to_string()),
                    _ => None,
                })
                .unwrap_or_else(|| "unknown".to_string());

            let error_msg = ironhermes_hooks::format_guardrail_error(
                name,
                &reason,
                &guardrail_name,
                &self.error_detail,
            );
            Err(anyhow::anyhow!("{}", error_msg))
        }
    }
}
```

NOTE: The re-walk to find the guardrail name is a deliberate backwards-compat choice so `test_guardrail_error_detail_minimal` (which depends on `format_guardrail_error` being called with the correct guardrail name) continues to pass unchanged. It is O(n_guardrails) — guardrail chains are configured at startup and typically hold 1-3 entries, so cost is negligible.
