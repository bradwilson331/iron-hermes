## Spacing Scale

Terminal output uses character-count spacing, not pixel spacing. The following column convention maps to the 8-point spirit:

| Token | Characters | Usage |
|-------|-----------|-------|
| xs | 2 spaces | Inline label-value gap (e.g., `  [OK] name`) |
| sm | 4 spaces | Nested indentation (e.g., subagent tree indent) |
| md | 1 blank line | Between output blocks in a single response |
| lg | 2 blank lines | Between major sections (banner → chat loop) |
| separator | `"─".repeat(40)` | Section dividers (matches existing `cmd_status` pattern) |

Source: `main.rs` lines 135, 180 — existing `"─".repeat(40)` separator convention.

Exceptions: none.

## Typography

Terminal output has no font sizes. The following role-to-style mapping serves the same purpose:

| Role | Style | Usage |
|------|-------|-------|
| Body | plain (no modifier) | Normal output text, agent responses |
| Label | `.bold()` | Section headings, prefixes (`"Commands:"`, `"API Keys"`) |
| Secondary | `.dimmed()` | Stats lines, hints, goodbye messages, command descriptions |
| Display | `.bold().cyan()` | App name, section titles (`"IronHermes Status"`) |

Source: `main.rs` lines 122–127, 133, 179, 213–215 — established bold/dimmed/cyan conventions.

Line-height equivalent: one `\n` between items; blank `println!()` between blocks.

## Color

The existing project color contract (enforced by `colored` v3):

| Role | Color | Usage |
|------|-------|-------|
| Dominant (60%) | default terminal foreground | All body text, plain output |
| Secondary (30%) | `.dimmed()` | Stats, hints, secondary labels, progress suffixes |
| Accent — brand (10%) | `.cyan()` | App name, `"Hermes:"` speaker label, banner, section titles |
| Accent — user | `.green()` | `"You:"` speaker label, `[OK]` status, configured/success states |
| Accent — tool/command | `.yellow()` | Tool names in progress lines, slash commands in help, `[MISSING]` state |
| Destructive / error | `.red()` | `"not set"` API key state, error conditions |

**Accent reserved for (exhaustive list):**
- `.cyan()`: IronHermes brand name, `"Hermes:"` response prefix, CLI banner, section title text only
- `.green()`: `"You:"` prompt prefix, `[OK]` doctor checks, `"configured"` API key state
- `.yellow()`: tool name in `"Running: <tool>..."` progress, slash commands in `/help` output, `[MISSING]` doctor checks, `"WAITING"` slot message prefix

**Phase 9 additions (subagent delegation):**
- Subagent slot-wait message: `"[Waited Nms for subagent slot]"` — rendered `.yellow()` when prepended to result
- Subagent tree node (CLI tree-view per D-19): `"  [subagent-N]"` prefix in `.cyan().dimmed()`, tool calls under it in `.yellow().dimmed()`
- Semaphore limit hit log: `tracing::info!` only (no color — structured log) per existing `info!()` pattern

Source: `main.rs` lines 122–175, 317, 327, 399–401 — all color assignments confirmed.

## Copywriting Contract

This phase adds one new tool (`delegate_task`) and one new in-flight state message. All other copy inherits from the existing CLI.

| Element | Copy |
|---------|------|
| Tool progress prefix | `Running: delegate_task...` (matches existing pattern: `"Running:".dimmed()` + `name.yellow()`) |
| Slot wait prefix | `[Waited Nms for subagent slot]` (prepended to result string when wait > 50ms; N = actual milliseconds) |
| Slot wait log | `Acquired subagent slot after waiting Nms` (tracing::info!, not user-visible) |
| Subagent no-response | `(no response)` (returned by child when final_response is None) |
| Unknown tool error | `Unknown tool in allowed_tools: <name>` (returned as tool error string to LLM) |
| Timeout error | `Subagent timed out after Ns` (N = config timeout_secs; returned as tool error string to LLM) |
| Semaphore closed error | `Semaphore closed: <reason>` (unexpected; returned as tool error) |
| Missing task param error | `Missing required parameter: task` (returned as tool error string to LLM) |
| Task description CTA | `delegate_task(task="...")` — no new user-facing CTA; this is an agent-to-agent tool |
| Empty state | Not applicable — delegate_task always has a task input; no list view |
| Destructive actions | None in this phase |

Source: `delegate_task.rs` lines 174, 236, 240, 247 — copy already implemented, contract documents it.

## Interaction Patterns

### Tool Progress Display (CLI chat mode)

Pattern already established in `main.rs` line 399–401:

```
\r Running: delegate_task...
```

- Written to `stderr` via `eprint!`
- `\r` carriage-return overwrites the line on each tool call
- `"Running:"` is `.dimmed()`, tool name is `.yellow()`
- No newline held — next output overwrites

### Slot-Wait Inline Notification (tool result prefix)

When `wait_duration > 50ms`, the tool result is prefixed:

```
[Waited 1240ms for subagent slot]
<child agent response here>
```

- Brackets and milliseconds are literal text
- This appears inside the tool result returned to the LLM, not printed directly
- The LLM may relay it to the user or process silently

### Subagent Tree-View (CLI, D-19 — Claude's Discretion)

When multiple subagents run, each active subagent should display as a tree line in stderr. Recommended format (consistent with existing progress pattern):

```
  [subagent-1] Running: read_file...
  [subagent-2] Running: web_search...
  [subagent-1] Done.
```

- `[subagent-N]` prefix: `.cyan().dimmed()`
- `Running: <tool>`: existing `.dimmed()` + `.yellow()` pattern
- `Done.`: `.dimmed()`
- Written to `stderr`, overwritten with `\r` per task, cleared on completion
- Index N is 1-based, assigned at task spawn time

### Gateway Progress (D-20 — Claude's Discretion)

In gateway (Telegram) mode, subagent progress is batched via `tracing::info!`. No in-chat progress messages are sent to Telegram during subagent execution. The final result only is delivered.
