## Medium

### MD-01: `execute_cron_job` always returns `Ok(())` even when `complete_job_run` succeeds on a failed agent run — but error-path `return Err(ce)` on line 624 is an inconsistent control-flow choice

**File:** `crates/ironhermes-gateway/src/runner.rs:596-628`
**Issue:** The `Err` arm of `agent.run().await` sanitizes the error, fires `ResponseSent`, and then calls `complete_job_run(..., success=false)`. If `complete_job_run` itself fails, line 624 returns `Err(ce)` — correct. But note the asymmetry: in the `Ok` arm, line 598 also returns `Err(e)` if `complete_job_run` fails. This is consistent between branches for persistence errors, but the overall contract is confusing because the outer tick task only logs `execute_cron_job` errors and continues (runner.rs:412-417). That means a failure to persist a completed-run record is logged but silently tolerated, while the underlying agent success/failure is already reflected in the persistence call's `success` arg. Consider whether `complete_job_run` persistence failures should escalate (e.g., mark the job in an error state, alert via hook) rather than only being logged. Today a cascading store failure loses the fact that the agent actually ran.

**Fix:** Either (a) document explicitly in the function comment that `complete_job_run` persistence failures are best-effort and only logged, or (b) fire an additional hook event (e.g., a new `JobPersistFailed` kind, or re-use `ResponseSent` with a sentinel) so observers can notice. At minimum, add a tracing `error!` field like `job_id = %job.id` on line 597/623 so structured log aggregators can filter. Today the `error!` calls format `job_id` into a plain string, which is harder to index downstream.

```rust
// Suggested: structured fields for observability
error!(
    job_id = %job.id,
    job_name = %job.name,
    error = %e,
    "complete_job_run persistence failure on success path"
);
```

## Low

### LW-01: `memory_store_tick` is captured but `PromptBuilder::set_memory_store` is the only consumer — the underlying Mutex is held by tokio but PromptBuilder uses `std::sync::Mutex`

**File:** `crates/ironhermes-gateway/src/runner.rs:376, 494, 543-545`
**Issue:** `memory_store` is `Option<Arc<Mutex<MemoryStore>>>` where `Mutex` is `std::sync::Mutex` (re-exported at line 1 via `use std::sync::{Arc, Mutex};`). This is consistent with the rest of the codebase. However, `execute_cron_job` is an async function and `set_memory_store` presumably locks the std Mutex internally when building the system message. If `build_system_message` holds the lock across an `.await`, this is a `Send` problem. I did not verify `PromptBuilder::set_memory_store` internals (out of file scope), but the pattern is worth flagging — the Telegram path has the same shape so any issue would be pre-existing, not introduced by this phase.

**Fix:** No action required in this phase. If a future audit finds `PromptBuilder` holding a `std::sync::Mutex` lock across `.await`, migrate to `tokio::sync::Mutex` or restructure `build_system_message` to take a snapshot. Out of scope for Phase 07.3.

### LW-02: `request_id` generated per cron job is a fresh UUID — not correlated to anything upstream

**File:** `crates/ironhermes-gateway/src/runner.rs:516`
**Issue:** `let request_id = uuid::Uuid::new_v4().to_string();` creates a brand-new UUID for each cron tick. This is fine for matching `MessageReceived` → `ResponseSent` inside one run, but a cron job that retries across ticks (missed tick / next schedule iteration) will get a new `request_id` each time, making it harder for JSONL/webhook consumers to correlate all runs of a single logical job. The job's own `id` (stable) is available on `job.id` and is already used as the fallback `chat_id`.

**Fix:** Consider seeding `request_id` with the job id for traceability, e.g., `format!("cron-{}-{}", job.id, Uuid::new_v4())` or using a deterministic `request_id` derived from job id + tick timestamp. Not a correctness issue; purely an observability improvement. Defer if `request_id` semantics are already specified in the hook protocol (they appear to be opaque per the `HookEvent::new(&request_id, ...)` signature).

### LW-03: `test_placeholder_string_absent` relies on `#[cfg(test)]` being the first occurrence of that literal in the file

**File:** `crates/ironhermes-gateway/src/runner.rs:673-683`
**Issue:** The test slices source up to the first `#[cfg(test)]` occurrence and asserts the placeholder is absent from the slice. If a future edit adds a `#[cfg(test)]` attribute earlier in the file (e.g., on a module, inline test helper, or `#[cfg(test)] use` statement above the tests module), the production slice will be truncated too early and the test will stop catching regressions without any failure signal. This is a silent-weakening risk, not a current bug.

**Fix:** Use a more specific marker like `"#[cfg(test)]\nmod tests"` or walk lines and gate on start-of-line matches. Alternatively, use `build.rs` to emit the production source into `OUT_DIR` and `include_str!` that — unambiguous boundary.

```rust
// Minimal fix: match the exact module opening
let test_marker = "#[cfg(test)]\nmod tests";
```

### LW-04: `tool_registry_tick` is captured but never used inside the tick task closure — only forwarded to `execute_cron_job` as a reference-to-Arc

**File:** `crates/ironhermes-gateway/src/runner.rs:375, 406`
**Issue:** Minor consistency note: `tool_registry_tick` is cloned via `self.tool_registry.clone()` on line 375 (cheap Arc clone), then passed by reference on line 406 into `execute_cron_job` which takes `&Arc<ToolRegistry>`. Inside `execute_cron_job` at line 563, it's cloned again via `tool_registry.clone()`. This is correct but slightly wasteful — the outer clone exists only so the closure owns a handle. Since Arc clones are cheap, this is informational only. The same pattern applies to `hook_registry_tick`, `memory_store_tick`, and `config_tick` (the last is a full `Config::clone()` which may be more expensive depending on `Config`'s size).

**Fix:** No action required. If `Config` is large (`.clone()` is not a cheap Arc-clone), consider wrapping it in `Arc<Config>` at the `GatewayRunner` level so the tick-task capture is Arc-cheap. Out of scope for this phase.

### IN-02: `info!` log on line 576 (successful cron agent completion) uses plain format string instead of structured fields

**File:** `crates/ironhermes-gateway/src/runner.rs:575-578`
**Issue:** `info!("Cron agent completed for job {} ({}), turns_used={}", job.name, job.id, result.turns_used);` uses positional formatting. The rest of this file (e.g., lines 90-95, 274) uses tracing's structured field syntax (`field = %value`). Structured fields are queryable in downstream log aggregators; plain format strings are not.

**Fix:**
```rust
info!(
    job_id = %job.id,
    job_name = %job.name,
    turns_used = result.turns_used,
    "Cron agent completed"
);
```

Applies to `error!` calls on lines 597, 604, 623 as well — inconsistent structured-vs-unstructured logging is a minor observability smell.
