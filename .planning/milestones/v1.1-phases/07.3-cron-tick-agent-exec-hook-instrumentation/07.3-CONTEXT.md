<domain>
## Phase Boundary

This phase closes two **critical** cross-phase integration gaps identified by the v1.1 milestone audit:

1. **Cron tick ↔ AgentLoop** — The cron tick task in `crates/ironhermes-gateway/src/runner.rs` currently resolves skill content via `resolve_skill_context()` into a `_full_input` local (prefix underscore = intentional discard) and then calls `complete_job_run()` with the literal placeholder string `"[Tick runner: agent execution pending full integration]"`. No `AgentLoop` is constructed at tick time, no LLM call happens, no tool dispatch runs, and skills never reach the model. This silently breaks SCHED-01, SCHED-03, and SCHED-04 end-to-end because the delivery routing carries placeholder text.

2. **Cron tick ↔ HookRegistry** — The tick task closure captures `job_store_tick` and `skill_registry_tick` but never captures `hook_registry`. The entire `ironhermes-cron` crate has zero references to `HookRegistry` / `HookEvent`. Scheduled-job lifecycle produces zero hook events, meaning HOOK-01's spec ("every message received, tool called, and response sent") is only satisfied for Telegram-triggered runs.

Phase 07.3 replaces the placeholder with a real `AgentLoop` construction inside the tick task closure, passes the already-resolved `full_input` (skill content + user prompt) as the message, delivers the agent's actual response through the existing `complete_job_run()` + `resolve_delivery_target()` routing, and fires `MessageReceived` / `ToolCalled` / `ResponseSent` hook events for every cron-triggered run — to the same `HookRegistry` instance that Telegram-triggered runs use.

**Out of scope (belongs to adjacent phases):**
- Hook ordering fix (ToolCalled fires before guardrail check) → Phase 07.4
- Duplicate MessageReceived/ResponseSent events between gateway handler and AgentLoop → Phase 07.4
- SKILL-09 (plugin/extension skills) → Phase 07.5
- SKILL-13 (backlog)
- Phases 8, 9, 10 (EXEC-*, AGENT-*, BATCH-* orphaned requirements) — tracked separately

**Verification side-effect (explicit in roadmap success criteria #5):** After this phase lands, a retroactive `/gsd-verify-work 05` must be able to produce Phase 05 VERIFICATION.md with all of SCHED-01..04 marked satisfied. Phase 05 was never formally verified — landing 07.3 unblocks that gate.

<decisions>
## Implementation Decisions

All items below are LOCKED decisions derived from the audit findings and ROADMAP Phase 07.3 success criteria. Every item has an evidence line pointing at audit source.

### Scope (from audit + ROADMAP)

- **D-01 Tick task must construct a real `AgentLoop`.** No more `_full_input` discard, no more placeholder string passed to `complete_job_run()`. The tick task per-job branch must build an `AgentLoop` configured like a Telegram-triggered run.
  - Evidence: audit `gaps.integration[0]` (severity: critical); ROADMAP success criterion #1

- **D-02 `full_input` (skill content + user prompt) is the agent's input.** The variable currently named `_full_input` must be renamed to `full_input` (no underscore) and passed into the `AgentLoop` as the message to run. Skill context is already being resolved correctly via `resolve_skill_context()` — only the wiring to the agent is missing.
  - Evidence: audit `gaps.integration[0]` issue text ("_full_input discarded"); runner.rs:393-413

- **D-03 Agent output replaces the placeholder in `complete_job_run()`.** The real LLM response string produced by `AgentLoop::run(...)` is what flows into `complete_job_run(store, job, output, success)`. The literal `"[Tick runner: agent execution pending full integration]"` is deleted from the codebase.
  - Evidence: audit `gaps.flows[0]` breaks_at; ROADMAP success criterion #1

- **D-04 The tick task captures the `HookRegistry` handle.** The tick-task closure in `runner.rs` (alongside `job_store_tick`, `skill_registry_tick`) must clone and capture an `Arc<HookRegistry>` as `hook_registry_tick` BEFORE `join_set.spawn(async move { ... })`.
  - Evidence: audit `gaps.integration[1]` (severity: critical); ROADMAP success criterion #4

- **D-05 `ironhermes-cron` crate must reference `HookRegistry`/`HookEvent` OR the wiring must live in `ironhermes-gateway`.** The audit's exact wording is "`ironhermes-cron` crate captures a `HookRegistry` handle in the tick task closure." Reading this against the actual code layout (the tick task closure lives in `crates/ironhermes-gateway/src/runner.rs`, not inside `ironhermes-cron`), the planner should interpret this as: the tick-task closure (wherever it physically lives in the workspace) must capture and use a `HookRegistry`. If a refactor is needed to move or expose the tick loop from `ironhermes-cron`, call it out in the plan — do not silently drop the "crate captures" wording. Prefer the smaller change: capture the registry in `runner.rs` since the closure already lives there.
  - Evidence: ROADMAP success criterion #4 + `gaps.integration[1]` affects HOOK-01; audit evidence `_full_input stub in cron tick runner — cross-phase tech debt that blocks Phase 5 delivery flow`

- **D-06 Cron-triggered runs fire the same three hook event kinds as Telegram-triggered runs: `MessageReceived`, `ToolCalled`, `ResponseSent`.** These are the three named in ROADMAP success criterion #3. The `AgentLoop` already fires these internally via `fire_hook()` (agent_loop.rs:131, :317, :200) when constructed with `with_hook_registry(...)`. Passing the captured `hook_registry_tick` into `AgentLoop::with_hook_registry(...)` is the mechanism.
  - Evidence: ROADMAP success criterion #3; agent_loop.rs `fire_hook` call sites

- **D-07 Hook events MUST flow to the same `HookRegistry` instance as Telegram-triggered runs.** Not a new registry. The tick task must clone the same `Arc<HookRegistry>` that `handler.rs` holds on its `Handler` struct (handler.rs:49 field `hook_registry: Option<Arc<ironhermes_hooks::HookRegistry>>`). Event receivers (JSONL, webhooks) already subscribe through this single registry.
  - Evidence: ROADMAP success criterion #3 ("same registry as Telegram-triggered runs"); handler.rs:49, :81

- **D-08 HookRegistry capture is conditional, not required to start the tick task.** If `self.hook_registry` is `None` (e.g., in test or minimal config), the tick task must still run and execute jobs — the `with_hook_registry(...)` call becomes a no-op in the `None` branch. This mirrors how `handler.rs` treats `hook_registry: Option<...>`.
  - Evidence: handler.rs:49 uses `Option<Arc<...>>`; conservative default — do not introduce a new hard dependency

### AgentLoop construction inside tick task

- **D-09 The tick task builds a FRESH `AgentLoop` per due job.** Do not attempt to cache or share AgentLoop across ticks or jobs. Build per-iteration, run once, drop. This matches the intent commented in runner.rs:404-406 ("involves building a fresh agent per job") and avoids mutable-state concerns inside the spawned closure.
  - Evidence: existing runner.rs comment at lines 404-406

- **D-10 AgentLoop configuration mirrors the Telegram path, except for platform metadata.** Whatever `handler.rs run_agent()` passes to its AgentLoop (system prompt from PromptBuilder, tool registry, guardrails, memory store, LLM config, hook registry) must also be passed in the tick task. The planner MUST read handler.rs:run_agent (or its equivalent AgentLoop construction path) during planning and copy the construction call verbatim into the plan's `<action>` text.
  - Evidence: audit `flows[passing #8]` shows handler wires AgentLoop correctly; D-07 requires "same registry"

- **D-11 The tick task needs access to everything `handler.rs` has.** This likely means promoting fields on the gateway/runner struct that are currently only held by `Handler`: `tool_registry`, `prompt_builder` (or `skill_registry` + a way to build prompts), `memory_store`, `llm_client`, `config`. The plan should enumerate exactly which fields need to be cloned into the tick-task closure and how they are made accessible (either via the gateway's top-level `Runner` struct already holding them, or by introducing clones before `spawn`).
  - Evidence: agent_loop.rs constructor fields; handler.rs already holds these

- **D-12 Per-job `chat_id` / `user_id` / `platform` for hook events comes from the scheduled `Job` record.** The `Job` struct stores the delivery target (Telegram chat, CLI, webhook URL) — use the same values for hook event metadata so downstream consumers (JSONL, webhooks) see a coherent `(platform, chat_id)` tuple. For cron-originated runs the platform label should be `"cron"` (or the delivery target's platform if that makes more sense — planner decides, but pick one consistently).
  - Evidence: audit `gaps.integration[3]` on duplicate events notes gateway uses real platform/chat_id vs AgentLoop using `"agent"`/`""` — the new cron path should NOT re-introduce empty values

### Delivery routing (unchanged semantics, real payload)

- **D-13 Delivery routing (`resolve_delivery_target` + `complete_job_run`) stays unchanged.** This phase does not touch delivery. It only changes WHAT flows into `complete_job_run` as the `output` parameter — from a placeholder string to the real agent response.
  - Evidence: audit `flows[passing]` — webhook/JSONL chains are working; only the input to `complete_job_run` is stubbed

- **D-14 `complete_job_run` success flag must reflect actual agent success.** If `AgentLoop::run()` returns `Err`, `complete_job_run` is called with `success=false` and an error string (or the error stringified) as output. Do not lose errors. Logs go to the existing `error!(...)` macro at runner.rs:412.
  - Evidence: runner.rs:410 currently passes hardcoded `true`

### Testing requirements (ROADMAP success criterion #2 + Nyquist)

- **D-15 Integration test required: scheduled job with attached skill produces an LLM response that reflects the skill content.** This is ROADMAP success criterion #2 verbatim. The test should:
  - Create a scheduled job with a known skill attached (e.g., a skill whose content instructs the agent to respond with a specific token)
  - Use a mock / stub LLM client that echoes its input or can be asserted against
  - Run one tick iteration (or call the tick execution function directly)
  - Assert that the output stored in `complete_job_run` contains content derived from the skill (NOT the placeholder string)
  - Evidence: ROADMAP success criterion #2 ("integration test")

- **D-16 Hook event test: cron-triggered runs emit MessageReceived + ResponseSent to the shared registry.** Subscribe a test consumer to the `HookRegistry`, run one tick iteration with a stub LLM, assert the consumer observes both events with `(platform, chat_id)` populated from the Job record. `ToolCalled` is conditional on the stub LLM requesting a tool — the test should optionally include a tool-dispatch scenario.
  - Evidence: ROADMAP success criterion #3; audit `gaps.requirements[HOOK-01]`

- **D-17 The placeholder string `"[Tick runner: agent execution pending full integration]"` must not appear anywhere in the codebase after this phase.** Grep-verifiable acceptance criterion.
  - Evidence: audit `gaps.flows[0]`; ROADMAP success criterion #1

- **D-18 No regression on the 306-test Telegram handler path.** `cargo test --workspace` must remain green. The Telegram path shares the same `HookRegistry`, `ToolRegistry`, and `AgentLoop` — no changes to handler.rs beyond what's required for field access.
  - Evidence: audit `tech_debt.07.2` notes 306 tests green

### Claude's Discretion

The audit and ROADMAP lock the WHAT. The following HOW details are Claude's discretion during planning:

- Whether to extract a helper function `build_agent_loop_for_cron(...)` vs inlining construction in the tick closure
- Whether `ironhermes-cron` gains a new `hook_registry` field on its tick/run signature vs. capturing in runner.rs (D-05 documents the reasoning; planner picks the minimal change)
- Number of integration tests beyond D-15 and D-16's minimums
- How to stub the LLM client for integration tests (mock trait impl vs feature-gated test harness)
- Platform label for hook event metadata — `"cron"` vs `"scheduled"` vs the underlying delivery platform (D-12 says pick one consistently)
- Number of PLAN.md files — roadmap says "single phase per user direction — combine AgentLoop wiring + hook capture in one plan if practical". Planner may split if complexity warrants, but default is one plan.
