### GAP-01: `platforms` Filter (SKILL-05)

**Severity:** HIGH | **Effort:** S | **Milestone:** v1.1

**Spec quote** (agentskills.io / `skills_tool.py:34-36`):
```
platforms: [macos]  # Optional — restrict to specific OS platforms
                    # Valid: macos, linux, windows
                    # Omit to load on all platforms (default)
```

Python implementation: `skills_tool.py:97-135` — `_PLATFORM_MAP` dict + `skill_matches_platform(frontmatter)` returns `False` when current OS not in platforms list. Called at discovery time: `skills_tool.py:532`, `655`, `891`.

**Current state:** `SkillFrontmatter` in `crates/ironhermes-core/src/skills.rs:14-20` does **not** include a `platforms` field. `load_with_paths()` at `skills.rs:109-173` performs no platform check. Skills with `platforms: [macos]` are loaded on Linux without filtering.

**Test coverage:** No test exercises platform filtering. The 14 `skills.rs` unit tests cover parse/load/dedup/catalog but not OS filtering.

**Delta:** Add `platforms: Option<Vec<String>>` to `SkillFrontmatter`. Add platform-check helper (maps `"macos"→"darwin"`, `"linux"→"linux"`, `"windows"→"win32"`) in `load_with_paths()`. Skip skills whose `platforms` list does not contain the current OS. Tests needed: platform matches, platform excluded, no platforms field (pass-through).

**Migration risk:** None — additive field on `SkillFrontmatter`. Existing skills without `platforms` continue to load.

**Recommended scope:** ~40 LOC Rust (Python proxy: `skills_tool.py:97-135` ≈ 38 lines). S effort.
**Proposed REQ-ID:** SKILL-05
**Proposed phase:** Phase 07.2 (skills parity — spec compliance basics)

### GAP-03: SKILL.md Validation Rules (SKILL-07)

**Severity:** MEDIUM | **Effort:** S | **Milestone:** v1.1

**Spec quote** (agentskills.io/specification — from `.planning/research/SKILLS.md:19-23`):
```
name:        1-64 chars, lowercase alphanumeric + hyphens only,
             no leading/trailing/consecutive hyphens,
             must match parent directory name
description: 1-1024 chars
```

Python reference: `skill_manager_tool.py:102-116` — `_validate_name()` applies regex `^[a-z0-9]([a-z0-9-]*[a-z0-9])?$` and checks length. `skill_manager_tool.py:141-178` — `_validate_frontmatter()` validates name, description length, and directory match.

**Current state:** `parse_skill_md()` (`skills.rs:44-77`) performs no validation on `name` or `description` beyond successful YAML parse. `load_with_paths()` (`skills.rs:109-173`) does not check that the skill directory name matches `frontmatter.name`. A skill named `"My Skill!"` in a directory named `"different-dir"` would load without warning.

**Test coverage:** No validation tests. `test_parse_skill_md_valid_frontmatter` only checks that valid content parses. No test for name length, regex, or dir-name mismatch.

**Delta:**
1. After successful YAML parse in `parse_skill_md()`: validate `name` against regex `^[a-z0-9]([a-z0-9-]*[a-z0-9])?$`, length 1-64. Return `None` on failure (or emit `tracing::warn!` and continue — warn-but-load is spec-compliant for non-critical issues per `.planning/research/SKILLS.md:118`).
2. In `load_with_paths()`: check that directory name matches `frontmatter.name` (lowercase), emit `tracing::warn!` on mismatch (warn-but-load per spec).
3. Validate description length 1-1024, emit warn if exceeded.

**Migration risk:** Low. If strict rejection is chosen, any malformed user skill is silently skipped (already the behavior for parse failures). Warn-but-load is safer.

**Recommended scope:** ~30 LOC. S effort.
**Proposed REQ-ID:** SKILL-07
**Proposed phase:** Phase 07.2

Skills Guard — Security scanner for externally-sourced skills.
Trust levels:
  - builtin:   Never scanned, always trusted.

**Current state:** No security scanning in ironhermes for skills. `load_with_paths()` (`skills.rs:109-173`) reads SKILL.md files without any content scanning. The `activate` action returns body content without threat checks. Note: ironhermes does have `scan_context_content()` for SOUL.md/AGENTS.md (`prompt_builder.rs:74`, `crates/ironhermes-core/src/security.rs`) — this is adjacent infrastructure, but it is not wired into skill loading.

**Test coverage:** Absent for skill scanning.

2. Apply at hub install time (not at every load — builtin/local skills are implicitly trusted).

4. Enforce `INSTALL_POLICY` matrix in hub install flow.

**Migration risk:** Medium. Scanning is only needed at install time for externally-sourced skills. Local skills (`~/.ironhermes/skills/`) can be treated as `builtin` trust, no scan required. Changes would only affect the hub install path (GAP-06).

**Recommended scope:** L (Python proxy: `skills_guard.py` 1,105 LOC; Rust estimate 400-600 LOC with regex crate). Target v2.

### Directory Depth / Name-Collision Precedence (not a gap)

Phase 7's `load_with_paths()` already implements first-path-wins deduplication (`skills.rs:154-162`) via `HashSet<String>`. The research doc (`.planning/research/SKILLS.md:118`) recommends warn-but-load on non-critical issues — current code uses `debug!` on duplicates. GAP-03 (SKILL-07) adds a `warn!` upgrade for name-collision messages. No separate gap required.

Directory depth: Phase 7's scanner walks exactly one level (`read_dir` + `is_dir` check, `skills.rs:127-133`) — only `{skills_dir}/{skill-name}/SKILL.md` is found. The agentskills.io spec permits nested category directories (`{skills_dir}/category/skill-name/SKILL.md`). This is a partial gap but not blocking — skills that use nested categories simply won't be discovered. Covered by SKILL-08's `extra_paths` as a workaround; a dedicated depth-walk enhancement can be tracked under SKILL-07 or a new SKILL-14 if desired by the planning team. Out of scope for this audit to prescribe.
