## Cross-Phase Integration Findings

### Medium

1. **active_skills Arc mismatch (Telegram gateway):** `main.rs::run_gateway` creates one `Arc<Mutex<Vec<SkillRecord>>>` for `SkillsTool`, but `GatewayMessageHandler` creates its own separate Arc at construction. Skills activated via the tool in Telegram don't reach AgentLoop's enforcement check. Fix: pass the shared Arc via a `set_active_skills()` setter on the handler, mirroring `set_hook_registry`/`set_skill_registry` pattern.
   → Affects **SKILL-06** (enforcement logic correct, wiring incomplete in Telegram path)

### Low (Architectural Decisions)

2. **execute_code gateway-only:** Tool registered only in `run_gateway`. CLI `run_single`/`run_chat` use `build_registry()` which omits it. Consistent with pattern where gateway is full-featured.
   → Affects **EXEC-01** (gateway mode works, CLI modes don't have it)

3. **Hooks/guardrails gateway-only:** `HookRegistry`, `BlocklistGuardrail`, JSONL logging, webhook delivery — all wired only in `run_gateway`. CLI modes bypass all hook infrastructure.
   → Affects **HOOK-01, HOOK-02** (gateway mode works, CLI modes don't have it)
